# 11.05 — Sicurezza nei microservizi

Nei capitoli precedenti abbiamo visto gateway, service discovery, resilienza, messaging, saghe, outbox, inbox e consistenza eventuale.

Ora affrontiamo un punto ancora più delicato: **la sicurezza nei microservizi**.

In una singola applicazione, spesso abbiamo:

```text id="i4e9pe"
un solo processo
un solo database
un solo sistema di autenticazione
un solo punto principale di autorizzazione
```

In un sistema distribuito, invece, abbiamo:

```text id="b9us6n"
più servizi
più API
più database
più segreti
più comunicazioni interne
più token
più punti di accesso
più confini da proteggere
```

Il punto centrale è questo:

```text id="l1igkr"
Nei microservizi la sicurezza non può dipendere solo dal perimetro.
Ogni servizio deve autenticare, autorizzare, validare e proteggere
le proprie risorse come se la rete interna non fosse automaticamente fidata.
```

Questo è coerente con il modello **Zero Trust**, che sposta la difesa dal perimetro statico di rete verso utenti, asset e risorse. NIST descrive la Zero Trust Architecture come un modello in cui l’accesso alle risorse viene concesso per sessione, valutando identità, contesto e policy, invece di fidarsi implicitamente della rete. ([NIST Computer Security Resource Center][1])

---

# Il problema della sicurezza distribuita

In un monolite, se un utente è autorizzato, tutto avviene nello stesso processo.

In un sistema a microservizi:

```text id="k89oiq"
Frontend
  ↓
API Gateway
  ↓
Ordini Service
  ↓
Pagamenti Service
  ↓
Notifiche Service
```

Domande da risolvere:

```text id="z14w5z"
Chi è l’utente originale?

Il servizio chiamante è autorizzato?

Il token ricevuto è valido?

Il token è destinato a questo servizio?

Il servizio A può chiamare il servizio B?

Il servizio B deve agire per conto dell’utente o per conto proprio?

Dove controllo i permessi?

Come ruoto i segreti?

Cosa succede se un token viene rubato?

Come traccio chi ha fatto cosa?
```

Queste domande non si risolvono con un semplice `[Authorize]` messo in un punto solo.

# Identity provider nel dettaglio

Un **Identity Provider**, spesso abbreviato in IdP, è il sistema responsabile dell’identità.

Si occupa di:

```text id="c94hji"
autenticazione utenti
login
MFA
password
federazione
emissione token
claims
ruoli
scopes
client applicativi
sessioni
logout
rotazione chiavi
```

Esempi:

```text id="qh7hv8"
Microsoft Entra ID
Duende IdentityServer
Keycloak
Okta
Auth0
OpenIddict
IdentityServer aziendale
```

In un’architettura a microservizi, l’identity provider diventa un componente centrale perché emette i token usati da gateway, API e servizi interni.

L’IdP può gestire:

```text id="qt44iz"

API resource

chiavi di firma

```

# JWT

**JWT**, JSON Web Token, è un formato di token compatto e firmato.

Una API può ricevere un JWT nell’header HTTP:

```http id="u9j6et"
Authorization: Bearer eyJhbGciOi...
```

Un JWT contiene tipicamente:

```text id="vb3g2h"
header
payload con claims
signature
```

Claims comuni:

```text id="w2h5b3"
iss
  issuer, chi ha emesso il token.

aud
  audience, per chi è destinato il token.

sub
  subject, identificativo del soggetto.

exp
  scadenza.

iat
  data emissione.

scope / scp
  scope autorizzativi.

roles
  ruoli.

client_id / azp
  client applicativo.
```

In ASP.NET Core, una API che usa JWT bearer authentication valida l’access token ricevuto; Microsoft sottolinea che l’API valida il token, non il modo in cui il client lo ha ottenuto. L’acquisizione del token viene gestita tramite protocolli come OAuth2 e OpenID Connect. ([Microsoft Learn][2])

---

# Validare un JWT

Una API non deve fidarsi di un token solo perché “sembra” un JWT.

Deve validare:

```text id="iu3szs"
firma
issuer
audience
scadenza
algoritmo
chiave di firma
eventuali scope
eventuali ruoli o permission claims
```

Esempio ASP.NET Core:

```csharp id="t6mgzz"
builder.Services
    .AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.Authority = "https://identity.example.com";
        options.Audience = "ordini-api";
    });

builder.Services.AddAuthorization(options =>
{
    options.AddPolicy("Ordini.Read", policy =>
    {
        policy.RequireAuthenticatedUser();
        policy.RequireClaim("scope", "ordini.read");
    });
});
```

Pipeline:

```csharp id="d4la1j"
app.UseAuthentication();
app.UseAuthorization();
```

Endpoint:

```csharp id="k1vwz0"
app.MapGet("/api/ordini", () =>
{
    return Results.Ok();
})
.RequireAuthorization("Ordini.Read");
```

# JWT: errori comuni

## Non controllare audience

Pericoloso:

```text id="utxtkj"
un token emesso per Clienti API
viene accettato anche da Ordini API
```

Ogni API deve controllare che `aud` sia corretto.

---

## Token troppo lunghi di durata

Se un token dura troppo:

```text id="avrn0j"
un token rubato resta valido troppo a lungo
```

Meglio access token brevi e rinnovo controllato.

---

## Mettere troppi dati nel token

Sbagliato:

```text id="r4t4em"
mettere dati personali, autorizzazioni enormi,
informazioni variabili o dati sensibili nel JWT
```

Il token può essere letto dal client se non cifrato. La firma garantisce integrità, non segretezza.

---

## Usare JWT come sessione mutabile

Un JWT già emesso resta valido fino a scadenza, salvo meccanismi di revoca o controllo aggiuntivo.

Se i permessi cambiano, un token vecchio potrebbe contenere claims non aggiornate.

# OAuth2

**OAuth2** è un framework di autorizzazione.

Serve a permettere a un client di ottenere un access token per chiamare una risorsa protetta.

OAuth 2.0 è definito dall’RFC 6749 ed è uno standard IETF per autorizzazione delegata. ([datatracker.ietf.org][3])

Esempio:

```text id="spccsd"
Frontend ottiene access token
Frontend chiama Ordini API
Ordini API valida token
Ordini API controlla scope ordini.read
```

OAuth2 risponde soprattutto alla domanda:

```text id="y0oqld"
Questo client può ottenere un token per accedere a questa API?
```

---

# OpenID Connect

**OpenID Connect**, abbreviato OIDC, è un layer di identità sopra OAuth2.

Serve per autenticare l’utente e ottenere informazioni sulla sua identità.

Microsoft descrive OIDC come un protocollo di autenticazione costruito sopra OAuth2, utilizzabile per Single Sign-On tramite un ID token. ([Microsoft Learn][4])

Distinzione pratica:

```text id="p1fgp8"
OAuth2
  accesso alle API.

OpenID Connect
  login e identità dell’utente.
```

OIDC introduce l’`id_token`.

OAuth2 usa `access_token`.

Regola pratica:

```text id="ov74l7"
Usa l’id_token per sapere chi è l’utente nel client.
Usa l’access_token per chiamare le API.
```

Una API non dovrebbe autorizzare richieste usando un `id_token` al posto di un access token.

---

# Flusso tipico web app + API

Schema:

```text id="xo34ie"
Browser
  ↓ login OIDC
Identity Provider
  ↓ id_token + access_token
Web App / BFF
  ↓ Authorization: Bearer access_token
API
```

La web app usa OIDC per autenticare l’utente.

La API valida l’access token per autorizzare l’accesso alla risorsa.

---

# Token delegation

La **token delegation** serve quando un servizio deve chiamare un altro servizio **per conto dell’utente originale**.

Scenario:

```text id="t7qf44"
Utente chiama Frontend.
Frontend chiama Gateway.
Gateway chiama Ordini API.
Ordini API deve chiamare Report API per conto dello stesso utente.
```

Domanda:

```text id="ig0vj3"
Report API deve sapere che la richiesta riguarda l’utente originale?
```

Se sì, non basta usare un token applicativo generico.

Serve un flusso delegato.

---

# On-Behalf-Of flow

Nel mondo Microsoft identity platform, il flusso tipico per la delega è l’**OAuth2 On-Behalf-Of flow**.

Microsoft lo descrive come un flusso per service-to-service authentication in cui un servizio usa un token ricevuto per ottenere un nuovo token verso una API downstream, agendo per conto dell’utente. ([Microsoft Learn][5])

Schema:

```text id="wysc9n"
Client ottiene token per API A
  ↓
API A riceve token utente
  ↓
API A chiede all’IdP un token per API B
  usando On-Behalf-Of
  ↓
API A chiama API B
  con token valido per API B
```

Punto fondamentale:

```text id="ednf50"
Non passare lo stesso token a qualunque servizio.
Ogni API dovrebbe ricevere un token con audience corretta.
```

---

# Perché non riusare sempre lo stesso token

Errore comune:

```text id="j6jr7x"
Il frontend manda un token a Ordini API.
Ordini API passa lo stesso token a Pagamenti API.
Pagamenti API lo accetta.
```

Problema:

```text id="kcqd8a"
il token potrebbe avere audience per Ordini API
non per Pagamenti API
```

Un token deve essere destinato al servizio che lo riceve.

Regola pratica:

```text id="hgluqj"
Ogni API deve validare audience e scope propri.
Non accettare token destinati ad altri servizi.
```

---

# Client credentials flow

Il **Client Credentials flow** serve quando non c’è un utente umano.

È usato per machine-to-machine.

Esempi:

```text id="ra2hko"
Worker Service chiama Report API
Ordini Service chiama Fatturazione API per attività tecnica
job notturno esegue sincronizzazione
microservizio chiama altro microservizio per conto proprio
pipeline chiama API interna
```

L’RFC 6749 definisce il client credentials grant per casi in cui il client agisce per conto proprio o accede a risorse sotto il suo controllo. ([datatracker.ietf.org][3])

Microsoft descrive il client credentials flow come un flusso in cui un confidential client usa le proprie credenziali, invece di impersonare un utente, per ottenere un access token e chiamare un’altra risorsa web. ([Microsoft Learn][6])

---

# Esempio client credentials

Richiesta token concettuale:

```http id="s7lmng"
POST /connect/token
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials&
client_id=ordini-worker&
client_secret=...&
scope=report.write
```

Risposta:

```json id="hho1rb"
{
  "access_token": "...",
  "token_type": "Bearer",
  "expires_in": 3600
}
```

Chiamata API:

```http id="yulrle"
POST /api/report/aggiorna
Authorization: Bearer eyJhbGciOi...
```

---

# Delegation vs client credentials

Distinzione importantissima:

```text id="kx9l75"
Delegation / On-Behalf-Of
  il servizio agisce per conto dell’utente originale.

Client credentials
  il servizio agisce come applicazione, non come utente.
```

Esempio:

```text id="bczygn"
Utente Mario esporta un report:
  usare delega, se i permessi dipendono da Mario.

Worker notturno rigenera tutti i report:
  usare client credentials, se è un’attività di sistema.
```

Errore comune:

```text id="m9cr2i"
usare client credentials per bypassare le autorizzazioni utente
in operazioni che invece dovrebbero rispettarle.
```

---

# Service-to-service authentication

La **service-to-service authentication** protegge le chiamate tra servizi.

Possibili approcci:

```text id="a7784q"
OAuth2 client credentials
token delegation / On-Behalf-Of
mTLS
managed identity in cloud
certificati client
API key, solo per casi limitati
service mesh con identità workload
```

Obiettivo:

```text id="saij6p"
il servizio chiamato deve sapere:
  chi sta chiamando
  se il chiamante è autorizzato
  per quale scopo
  con quali limiti
```

Esempio:

```text id="izypz5"
Ordini Service può chiamare Pagamenti API per creare autorizzazioni.
Report Service può leggere eventi ordini.
Notifiche Service non può cancellare pagamenti.
```

---

# API key: perché non basta spesso

Le API key sono semplici.

Esempio:

```http id="up840h"
X-API-Key: abc123
```

Ma hanno limiti:

```text id="c246bh"
spesso non hanno scadenza
sono difficili da ruotare se condivise
non contengono claims
non rappresentano bene identità e scope
se rubate, danno accesso finché non revocate
```

Possono andare bene per:

```text id="hut9c5"
integrazioni legacy
webhook semplici
ambienti molto controllati
prototipi interni
```

Ma per microservizi moderni, meglio usare token, certificati, managed identity o service mesh.

# mTLS

**mTLS**, mutual TLS, è TLS con autenticazione reciproca.

Nel TLS classico:

```text id="z9952a"
client verifica il server
```

Con mTLS:

```text id="h47wm8"
client verifica il server
server verifica il client
```

Entrambi presentano certificati.

Schema:

```text id="c46qh8"
Servizio A
  presenta certificato client
  ↓
Servizio B
  verifica certificato A

Servizio B
  presenta certificato server
  ↓
Servizio A
  verifica certificato B
```

---

# Quando usare mTLS

mTLS è utile per:

```text id="l05zvu"
comunicazione service-to-service
ambienti Zero Trust
protezione forte del canale
identità dei workload
service mesh
settori regolati
API interne critiche
```

Vantaggi:

```text id="ezmwjx"
autenticazione forte tra servizi
protezione del canale
riduce rischio di chiamanti non autorizzati
può essere gestito centralmente da service mesh
```

Svantaggi:

```text id="ca78ry"
gestione certificati
rotazione
scadenze
configurazione complessa
debug più difficile
```

Regola pratica:

```text id="qtvl58"
mTLS autentica il canale e il workload.
Non sostituisce sempre l’autorizzazione applicativa.
```

Un servizio può essere autenticato via certificato, ma devi comunque decidere cosa può fare.

# Authorization centralizzata

Con **autorizzazione centralizzata**, un componente centrale decide se una richiesta è consentita.

Esempi:

```text id="sb6u7d"
API Gateway applica policy comuni
Policy Decision Point esterno
OPA / Open Policy Agent
servizio autorizzativo centrale
identity provider con ruoli e claims
```

Vantaggi:

```text id="lb8xko"
policy uniformi
gestione centralizzata
meno duplicazione
audit più semplice
modifiche policy concentrate
```

Svantaggi:

```text id="zdeql8"
rischio collo di bottiglia
dipendenza critica
policy troppo lontane dal dominio
difficoltà con regole resource-based
possibile latenza
```

Esempio:

```text id="zel81u"
Gateway blocca richieste senza scope ordini.read.
```

Questo è utile, ma non basta sempre.

---

# Authorization decentrata

Con **autorizzazione decentrata**, ogni servizio protegge le proprie risorse.

Esempio:

```text id="tevtu6"
Ordini Service decide se l’utente può vedere l’ordine 123.
Documenti Service decide se l’utente può approvare il documento 456.
Pagamenti Service decide se un servizio può creare un rimborso.
```

Vantaggi:

```text id="v0493w"
regole vicine al dominio
resource-based authorization più naturale
servizi autonomi
meno dipendenza da un punto centrale
```

Svantaggi:

```text id="dvxmi8"
possibile duplicazione
policy incoerenti se non governate
audit più distribuito
serve disciplina
```

Regola pratica:

```text id="wkpvdd"
Centralizza le policy trasversali.
Decentra le decisioni di dominio e resource-based.
```

---

# Modello consigliato: difesa a più livelli

In pratica, spesso si combina:

```text id="ewuzzp"
Gateway
  controlla autenticazione base, token valido, rate limit,
  scope generali, routing.

Servizio
  controlla audience, scope specifici, ruoli,
  permessi e regole sulla risorsa.

Dominio
  protegge invarianti e operazioni consentite.
```

Esempio:

```text id="hxwck5"
Gateway:
  token valido e scope ordini.access

Ordini Service:
  scope ordini.read
  utente appartiene al tenant corretto
  ordine visibile all’utente

Domain model:
  ordine annullabile solo se non spedito
```

Non basta fermarsi al gateway.

# Rotazione dei segreti

Un segreto che non può essere ruotato è un rischio.

Devi progettare:

```text id="z0xm6b"

revoca
audit accessi
segreti per ambiente

```

Esempio:

```text id="mipz3n"

Se comprometto Report Service, non posso accedere come Ordini Service.
```


# Zero Trust

Zero Trust non significa “non fidarsi mai di nessuno” in modo assoluto.

Significa:

```text id="i4rgzr"
non fidarsi implicitamente della rete
verificare esplicitamente
applicare least privilege
assumere la compromissione
controllare continuamente accessi e contesto
```

NIST definisce Zero Trust come un insieme di paradigmi di cybersecurity che spostano le difese da perimetri statici basati sulla rete verso utenti, asset e risorse. ([NIST Computer Security Resource Center][1])

Applicato ai microservizi:

```text id="k9f0ax"
non perché una chiamata viene dalla rete interna è autorizzata
ogni servizio valida token/certificato
ogni servizio applica autorizzazione minima
ogni segreto è limitato
ogni chiamata è tracciata
ogni accesso è osservabile
```

---

# Zero Trust in pratica

Pratiche concrete:

```text id="a3gdi7"
autenticare ogni chiamata service-to-service
validare audience e issuer dei token
usare mTLS dove opportuno
limitare scope e permessi
usare identità per workload
evitare segreti condivisi
ruotare credenziali
centralizzare audit e logging
segmentare rete
limitare egress
applicare rate limiting
monitorare comportamenti anomali
```

Zero Trust non è un prodotto singolo. È un approccio architetturale.

# Flusso completo: utente → gateway → servizio

Scenario:

```text id="ld2a90"
Utente visualizza un ordine.
```

Flusso:

```text id="lx48r1"
1. Utente fa login tramite OIDC.
2. Il client ottiene access token per API Gateway o BFF.
3. Gateway valida token.
4. Gateway inoltra richiesta a Ordini Service.
5. Ordini Service valida token o token trasformato con audience corretta.
6. Ordini Service controlla policy Ordini.Read.
7. Ordini Service controlla che l’ordine appartenga al tenant/cliente consentito.
8. Ordini Service restituisce dati.
9. Log e trace mantengono UserId, CorrelationId e decisioni rilevanti.
```

Punti di sicurezza:

```text id="sm0q42"
token valido
audience corretta
scope corretto
resource authorization
logging sicuro
correlation ID
least privilege
```

---

# Flusso completo: servizio → servizio senza utente

Scenario:

```text id="hv9tde"
Worker notturno aggiorna report.
```

Flusso:

```text id="zh54kv"
1. ReportWorker usa client credentials.
2. Ottiene access token per Report API.
3. ReportWorker chiama Report API.
4. Report API valida issuer, audience, scadenza e scope.
5. Report API controlla che client_id sia autorizzato.
6. Report API esegue operazione.
```

Policy:

```csharp id="n2ii8p"
builder.Services.AddAuthorization(options =>
{
    options.AddPolicy("Report.WriteFromWorker", policy =>
    {
        policy.RequireAuthenticatedUser();
        policy.RequireClaim("scope", "report.write");
        policy.RequireClaim("client_id", "report-worker");
    });
});
```

Endpoint:

```csharp id="kezn0z"
app.MapPost("/api/report/refresh", () =>
{
    return Results.Accepted();
})
.RequireAuthorization("Report.WriteFromWorker");
```

# Flusso completo: servizio → servizio per conto utente

Scenario:

```text id="udl9u7"
Ordini API deve chiamare Documenti API per mostrare documenti
accessibili all’utente originale.
```

Flusso corretto:

```text id="q83qo5"
1. Frontend ottiene token per Ordini API.
2. Ordini API riceve richiesta utente.
3. Ordini API usa On-Behalf-Of flow.
4. IdP emette token per Documenti API,
   mantenendo delega utente.
5. Ordini API chiama Documenti API.
6. Documenti API valida token con audience Documenti API.
7. Documenti API applica policy e resource authorization sui documenti.
```

Errore da evitare:

```text id="i4lzk6"
Ordini API usa client credentials per accedere a tutti i documenti,
anche quando l’operazione dovrebbe rispettare i permessi dell’utente.
```

# Sicurezza del gateway

Il gateway può fare:

```text id="f6ay7m"
validazione token iniziale
rate limiting
CORS
TLS termination
header normalization
correlation ID
basic authorization su scope generali
routing sicuro
protezione endpoint interni
```

Ma non deve essere l’unico punto di controllo.

Perché?

```text id="gj36ji"
un servizio potrebbe essere chiamato bypassando il gateway
un errore di configurazione può esporre un endpoint
servizi interni possono chiamarsi tra loro
le regole resource-based vivono nel dominio del servizio
```

Regola pratica:

```text id="ep76su"
Gateway come prima linea di difesa.
```

# Sicurezza dei messaggi

Nei sistemi event-driven, non proteggi solo HTTP.

Devi proteggere anche:

```text id="uwyih9"
broker
code
topic
consumer
producer
event payload
dead-letter queue
management UI
connection string
```

Esempi:

```text id="ulzqiy"
Ordini Service può pubblicare OrdineConfermato.
Report Service può consumare OrdineConfermato.
Notifiche Service non deve poter pubblicare PagamentoAutorizzato.
```

Buone pratiche:

```text id="kfq8pg"
credenziali broker per servizio
permessi minimi su exchange/queue/topic
TLS verso broker
segreti broker protetti
payload senza dati sensibili inutili
idempotenza
audit messaggi critici
```

# Token e messaggi asincroni

Errore frequente:

```text id="nzu6il"
metto il JWT dell’utente dentro il messaggio
così il consumer sa chi era l’utente
```

Problemi:

```text id="t8w4ps"
il token può scadere prima del consumo
il messaggio può restare in coda
il token può essere esposto
il consumer potrebbe usare un token non più valido
```

Meglio mettere nel messaggio informazioni minime e stabili:

```text id="zl813r"
UserId
TenantId
CorrelationId
causation id
dati necessari al processo
```

Poi il consumer applica le proprie policy o usa identità applicativa se il processo è di sistema.

# Authorization centralizzata/decentrata: esempio pratico

Scenario:

```text id="cn5k90"
Approvazione documento.
```

Controlli centralizzati al gateway:

```text id="wro9d1"
utente autenticato
token valido
scope documenti.access
rate limit
```

Controlli nel Documenti Service:

```text id="ub1bne"
permission Documenti.Approve
documento esiste
documento è in stato Bozza
utente appartiene al reparto corretto
utente non è lo stesso che ha creato il documento, se la regola lo vieta
```

Controlli nel dominio:

```csharp id="z0trh2"
public void Approva(UserId approvatore)
{
    if (Stato != StatoDocumento.Bozza)
    {
        throw new DomainException(
            "Solo un documento in bozza può essere approvato.");
    }

    Stato = StatoDocumento.Approvato;
    ApprovatoDa = approvatore;
}
```

# Multi-tenant security

Se il sistema è multi-tenant, ogni servizio deve rispettare il tenant.

Esempio claim:

```text id="zyxjz0"
tenant_id = casino-aosta
```

Ogni query deve filtrare correttamente.

Esempio:

```csharp id="v5skdn"
Ordine? ordine = await dbContext.Ordini
    .Where(o => o.Id == ordineId)
    .Where(o => o.TenantId == currentTenant.Id)
    .FirstOrDefaultAsync(cancellationToken);
```

Errore grave:

```csharp id="xpp88z"
Ordine? ordine = await dbContext.Ordini
    .FindAsync(ordineId);
```

se non controlla tenant o ownership.

Regola pratica:

```text id="l8kwhi"
Il tenant boundary è un confine di sicurezza,
non solo un filtro funzionale.
```

# Difesa dai lateral movement

Nei microservizi devi considerare cosa succede se un servizio viene compromesso.

Domanda:

```text id="fzxn8t"
Se Notifiche Service viene compromesso,
può leggere pagamenti?
può cancellare ordini?
può accedere a tutti i database?
può pubblicare eventi falsi?
```

Least privilege significa:

```text id="dscbho"
ogni servizio ha solo i permessi necessari
ogni servizio ha credenziali proprie
ogni servizio accede solo ai propri dati
ogni servizio può pubblicare/consumare solo messaggi autorizzati
```

# Errori comuni

## Fidarsi della rete interna

Sbagliato:

```text id="r0n523"
Se arriva dalla rete interna, allora è sicuro.
```

No. Nei microservizi bisogna autenticare e autorizzare anche chiamate interne.

---

## Validare token solo al gateway

Il gateway è importante, ma i servizi devono proteggere le proprie risorse.

---

## Non controllare audience

Accettare token destinati ad altri servizi è un errore grave.

---

## Confondere id_token e access_token

`id_token` serve al client per conoscere l’identità dell’utente.

Le API devono ricevere e validare access token.

---

## Usare client credentials al posto della delega utente

Pericoloso quando l’operazione deve rispettare i permessi dell’utente originale.

---

## Segreti condivisi tra servizi

Se più servizi usano lo stesso secret, comprometterne uno compromette tutti.

---

## JWT nei messaggi asincroni

Spesso è una cattiva idea per scadenza, esposizione e semantica.

---

## API key come unica sicurezza interna

Troppo debole per molti scenari distribuiti moderni.

---

## Nessun audit

Senza audit, ricostruire incidenti e accessi diventa difficile.

---

# Linee guida pratiche

```text id="fgk57b"
Usa un identity provider
  per centralizzare login, token, client e claims.

Usa OpenID Connect
  per autenticazione utente.

Usa OAuth2
  per accesso alle API.

Usa JWT access token
  per proteggere API, validando issuer, audience e scadenza.

Usa token delegation
  quando un servizio agisce per conto dell’utente originale.

Usa client credentials
  quando un servizio agisce come applicazione.

Usa mTLS
  quando vuoi autenticazione forte tra workload o servizi.

Non fidarti della rete interna
  ogni servizio deve validare chiamanti e permessi.

Combina autorizzazione centralizzata e decentrata
  gateway per policy generali, servizio per regole di dominio.

Gestisci segreti con secret manager
  non nel codice, non nelle immagini, non nei log.

Applica least privilege
  a database, broker, API, token e segreti.

Progetta per Zero Trust
  verifica esplicitamente, limita accessi, monitora continuamente.

Non mettere JWT utente nei messaggi asincroni
  usa identificatori, correlation ID e dati minimi necessari.

Audita operazioni critiche
  con UserId, ClientId, ResourceId, CorrelationId e outcome.
```

# Riepilogo

In questo capitolo abbiamo studiato la sicurezza nei microservizi.

Abbiamo visto:

```text id="fyhpwp"
Identity provider
JWT
OAuth2
OpenID Connect
Token delegation
Client credentials flow
Service-to-service authentication
mTLS
Authorization centralizzata/decentrata
Secrets
Zero Trust
```

La distinzione principale è questa:

```text id="rsco4k"
OpenID Connect
  autentica l’utente.

OAuth2
  autorizza l’accesso alle API.

JWT
  è un formato di token.

Client credentials
  rappresenta un’applicazione o servizio.

Token delegation / On-Behalf-Of
  rappresenta un servizio che agisce per conto dell’utente.

mTLS
  autentica reciprocamente i workload a livello di trasporto.
```

La regola principale è questa:

```text id="kb6heu"
In un sistema distribuito non esiste più un solo punto sicuro.
Ogni servizio è un confine di sicurezza:
valida il chiamante, controlla l’audience,
applica autorizzazione, protegge i propri dati,
usa segreti minimi e produce audit osservabile.
```

La sicurezza nei microservizi richiede più disciplina rispetto alla sicurezza di una singola app: identity provider, token corretti, service-to-service authentication, least privilege, gestione segreti, audit e Zero Trust devono essere progettati insieme, non aggiunti alla fine.

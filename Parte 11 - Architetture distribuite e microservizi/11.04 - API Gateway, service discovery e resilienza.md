# 11.04 — API Gateway, service discovery e resilienza

Nei capitoli precedenti abbiamo progettato i confini dei servizi e abbiamo visto messaging, eventi, outbox, inbox, saga, retry e consistenza eventuale.

Ora guardiamo l’infrastruttura logica intorno ai servizi.

In un sistema distribuito non basta avere tanti servizi. Bisogna anche decidere:

```text id="d5xyrq"
come entra il traffico nel sistema
come i client chiamano le API
come i servizi si trovano tra loro
come si distribuisce il carico
come si gestiscono errori, timeout e sovraccarichi
come si evita che un guasto locale diventi un guasto globale
```

Il punto centrale è questo:

```text id="mvys93"
L’architettura a microservizi non è fatta solo dai servizi.
È fatta anche dal modo in cui il traffico entra, viene instradato,
bilanciato, protetto, limitato, osservato e reso resiliente.
```

---

# API Gateway

Un **API Gateway** è un componente posto davanti a uno o più servizi backend.

Schema:

```text id="a75t6q"
Client web/mobile/esterno
        ↓
   API Gateway
        ↓
 ┌──────┼────────┬────────┐
 │      │        │        │
Clienti Ordini  Report   Pagamenti
Service Service Service  Service
```

Il gateway riceve le richieste dei client e le instrada verso i servizi corretti.

Esempio:

```text id="o2e27t"
/api/clienti/*
  → Clienti Service

/api/ordini/*
  → Ordini Service

/api/report/*
  → Report Service
```

---

# A cosa serve un API Gateway

Un gateway può occuparsi di:

```text id="xn9sjs"
routing
terminazione TLS
autenticazione iniziale
autorizzazione di bordo
rate limiting
logging delle richieste
correlation ID
header forwarding
load balancing
request/response transformation
aggregazione di risposte
protezione dei servizi interni
versionamento o routing per versione
```

Non deve però diventare il posto dove mettere tutta la logica di business.

Regola pratica:

```text id="djd7jo"
Il gateway deve gestire traffico e policy trasversali.
Non deve diventare il nuovo monolite applicativo.
```

---

# Senza gateway

Senza gateway, il client conosce direttamente molti servizi:

```text id="gvlmzg"
Frontend
  chiama Clienti API
  chiama Ordini API
  chiama Report API
  chiama Pagamenti API
```

Problemi:

```text id="ya3dz0"
il client conosce la topologia interna
ogni servizio deve gestire esposizione pubblica
CORS e sicurezza si duplicano
cambiare indirizzo di un servizio rompe i client
versionamento più difficile
troppi endpoint pubblici
```

---

# Con gateway

Con gateway:

```text id="om7g25"
Frontend
  chiama solo Gateway
```

Il gateway conosce la topologia interna.

Vantaggi:

```text id="ccwzkw"
un solo ingresso pubblico
servizi interni più protetti
routing centralizzato
policy uniformi
meno dettagli interni esposti ai client
più libertà di riorganizzare servizi interni
```

Svantaggi:

```text id="p1aj52"
componente critico
può diventare collo di bottiglia
può accumulare logica impropria
richiede alta disponibilità
va osservato e scalato
```

---

# API Gateway non è sempre obbligatorio

Per sistemi piccoli, può bastare:

```text id="znwb85"
un reverse proxy
una singola API
un monolite modulare
un ingress controller
```

Un API Gateway ha senso quando:

```text id="k3g96y"
hai più servizi esposti ai client
vuoi un ingresso unico
devi centralizzare policy trasversali
hai client diversi
vuoi nascondere la topologia interna
devi fare routing dinamico o aggregazioni leggere
```

---

# BFF — Backend for Frontend

Il **BFF**, Backend for Frontend, è una variante del gateway pensata per un tipo specifico di client.

Esempio:

```text id="juqwmk"
Web BFF
  ottimizzato per frontend web.

Mobile BFF
  ottimizzato per app mobile.

Admin BFF
  ottimizzato per backoffice.
```

Schema:

```text id="h4n9o3"
Web App
  ↓
Web BFF
  ↓
servizi interni

Mobile App
  ↓
Mobile BFF
  ↓
servizi interni
```

---

# Perché usare un BFF

Client diversi hanno esigenze diverse.

Esempio:

```text id="pzc4et"
Mobile
  vuole meno chiamate
  payload piccoli
  risposte aggregate

Web admin
  vuole più dettagli
  filtri avanzati
  dati gestionali

Public API
  vuole contratti stabili e documentati
```

Senza BFF, rischi di creare API backend troppo generiche o troppo orientate a un solo client.

Con BFF:

```text id="k71z7i"
ogni frontend ha un backend adatto
riduci logica nel client
puoi aggregare chiamate
puoi adattare payload
puoi gestire autenticazione specifica del client
```

---

# Rischi del BFF

Un BFF può degenerare se contiene troppa logica.

Deve occuparsi di:

```text id="hatn9m"
orchestrazione leggera
adattamento dati
aggregazione
sicurezza del frontend
routing
```

Non dovrebbe contenere:

```text id="tmz37d"
regole di dominio principali
transazioni business critiche
logica duplicata dei servizi
accesso diretto ai database interni dei servizi
```

Regola pratica:

```text id="zok1es"
Il BFF adatta l’esperienza del client.
Non sostituisce i servizi di dominio.
```

# YARP

**YARP**, Yet Another Reverse Proxy, è una libreria .NET per costruire reverse proxy e gateway personalizzati in ASP.NET Core. Microsoft lo descrive come un framework reverse proxy altamente personalizzabile, robusto, flessibile, scalabile e pensato per creare soluzioni proxy adatte alle proprie esigenze. ([Microsoft Learn][1])

YARP è utile quando vuoi costruire un gateway in .NET e avere controllo su:

```text id="fpilzq"
routing
cluster
destinazioni
load balancing
transform request/response
health checks
rate limiting
autenticazione/autorizzazione ASP.NET Core
integrazione con middleware esistenti
```

---

## Esempio YARP minimale

Pacchetto:

```bash id="rn7avm"
dotnet add package Yarp.ReverseProxy
```

`Program.cs`:

```csharp id="qu2i6p"
var builder = WebApplication.CreateBuilder(args);

builder.Services
    .AddReverseProxy()
    .LoadFromConfig(
        builder.Configuration.GetSection("ReverseProxy"));

var app = builder.Build();

app.MapReverseProxy();

app.Run();
```

`appsettings.json`:

```json id="jminwr"
{
  "ReverseProxy": {
    "Routes": {
      "clienti-route": {
        "ClusterId": "clienti-cluster",
        "Match": {
          "Path": "/api/clienti/{**catch-all}"
        }
      },
      "ordini-route": {
        "ClusterId": "ordini-cluster",
        "Match": {
          "Path": "/api/ordini/{**catch-all}"
        }
      }
    },
    "Clusters": {
      "clienti-cluster": {
        "Destinations": {
          "clienti-1": {
            "Address": "http://clienti-service:8080/"
          }
        }
      },
      "ordini-cluster": {
        "Destinations": {
          "ordini-1": {
            "Address": "http://ordini-service:8080/"
          }
        }
      }
    }
  }
}
```

Questo crea un proxy che instrada richieste diverse verso servizi diversi.

---

# YARP e rate limiting

YARP può applicare rate limiting prima di inoltrare le richieste ai backend; questo riduce il carico sui servizi di destinazione, aggiunge protezione e permette policy coerenti tra applicazioni. ([Microsoft Learn][2])

Esempio concettuale:

```text id="mkr1vo"
Client
  ↓
YARP applica rate limit
  ↓
se consentito, inoltra al servizio
```

Questo è utile perché blocchi parte del traffico indesiderato prima che raggiunga i microservizi interni.

---

# YARP e health checks

YARP supporta health check attivi e passivi sulle destinazioni: il proxy può smettere di inviare traffico verso nodi non sani finché non recuperano. ([Microsoft Learn][3])

Schema:

```text id="myqvxc"
Cluster Ordini
  ordini-1 healthy
  ordini-2 unhealthy
  ordini-3 healthy

YARP invia traffico solo a ordini-1 e ordini-3
```

Questo è importante quando hai più istanze dello stesso servizio.

# Service discovery

Il **service discovery** è il meccanismo con cui un servizio o un gateway trova gli indirizzi delle istanze disponibili di un altro servizio.

In un ambiente statico potresti configurare:

```text id="b387l0"
OrdiniServiceUrl = http://10.0.0.12:8080
```

Ma in ambienti moderni le istanze possono cambiare:

```text id="qxsgnc"
container ricreati
pod Kubernetes spostati
scaling orizzontale
deploy rolling
istanze che muoiono e ripartono
```

Non puoi basarti sempre su IP fissi.

---

# Tipi di service discovery

Due modelli principali:

```text id="tf0ego"
Client-side discovery
  il client conosce un registry e sceglie l’istanza.

Server-side discovery
  il client chiama un indirizzo stabile,
  poi proxy/load balancer instrada verso un’istanza.
```

In Kubernetes, il modello usuale per molti casi è server-side tramite Service e DNS.

Kubernetes Services forniscono un modo per esporre applicazioni in esecuzione come uno o più Pod; la documentazione Kubernetes collega Services, networking e load balancing come meccanismi fondamentali per connettere applicazioni nel cluster. ([Kubernetes][4])

---

# Service discovery in Docker Compose

In Docker Compose, i servizi si risolvono per nome.

Esempio:

```yaml id="xyj7xt"
services:
  gateway:
    build: ./Gateway
    ports:
      - "8080:8080"

  ordini-service:
    build: ./OrdiniService

  clienti-service:
    build: ./ClientiService
```

Il gateway può chiamare:

```text id="ru87wv"
http://ordini-service:8080
http://clienti-service:8080
```

Non deve usare `localhost`.

Dentro un container:

```text id="ge2139"
localhost = il container stesso
```

---

# Service discovery in Kubernetes

In Kubernetes, un Service può esporre un insieme di Pod con un nome DNS stabile.

Esempio:

```text id="qt8pu5"
http://ordini-service
```

può indirizzare verso più Pod di Ordini.

Schema:

```text id="nffikv"
Ordini Service DNS
       ↓
Kubernetes Service
       ↓
Pod ordini-1
Pod ordini-2
Pod ordini-3
```

Il client non conosce i singoli Pod.

# Load balancing

Il **load balancing** distribuisce il traffico tra più istanze.

Esempio:

```text id="pqrivl"
Ordini Service
  istanza 1
  istanza 2
  istanza 3
```

Il bilanciatore decide a quale inviare ogni richiesta.

Può basarsi su:

```text id="bf5kfc"
round robin
least requests
random
hash
affinità/sessione
health status
peso delle destinazioni
```

---

# Dove avviene il load balancing

Può avvenire in vari punti:

```text id="n34fc3"
API Gateway
reverse proxy
Kubernetes Service
Ingress controller
cloud load balancer
service mesh
client HTTP con service discovery
```

Esempio:

```text id="h29kzn"
Internet
  ↓
Cloud Load Balancer
  ↓
Ingress / Gateway
  ↓
Kubernetes Service
  ↓
Pod
```

Ogni livello può avere un ruolo diverso.

# Health check

Gli **health check** indicano se un servizio o una sua istanza può ricevere traffico.

Tipi principali:

```text id="dw8uug"
liveness
  il processo è vivo?

readiness
  il servizio è pronto a ricevere traffico?

startup
  il servizio ha completato l’avvio iniziale?
```

Esempio ASP.NET Core:

```csharp id="xxc2iy"
builder.Services.AddHealthChecks();

var app = builder.Build();

app.MapHealthChecks("/health/live");
app.MapHealthChecks("/health/ready");
```

Gli health check sono fondamentali perché gateway, orchestratori e load balancer possano togliere dal traffico istanze non sane.

---

# Health check non è diagnostica completa

Un health check deve essere rapido.

Sbagliato:

```text id="vugg9d"
eseguire report complessi
chiamare dieci servizi esterni
fare query pesanti
modificare dati
```

Meglio:

```text id="x3om4x"
controlli brevi
timeout brevi
dipendenze essenziali
distinzione tra live e ready
```

Regola pratica:

```text id="kpzv3t"
Un health check deve aiutare a decidere se inviare traffico,
non sostituire gli strumenti di diagnosi.
```

# Resilienza

In un sistema distribuito, le chiamate falliscono.

Cause tipiche:

```text id="xc2vom"
timeout
rete instabile
servizio non disponibile
deploy in corso
istanza riavviata
rate limit
database saturo
coda piena
latenza elevata
```

La resilienza serve a evitare che un problema locale diventi una cascata.

Pattern principali:

```text id="crqw0d"
timeout
retry
circuit breaker
bulkhead
rate limiting
fallback
health check
```

Microsoft documenta Polly come libreria .NET per resilienza e transient fault handling, con strategie come retry, circuit breaker, timeout, bulkhead isolation, rate limiting, fallback e hedging. ([Microsoft Learn][5])

---

# Timeout

Il **timeout** impedisce di aspettare all’infinito.

Senza timeout:

```text id="b7099w"
servizio remoto non risponde
thread/connessioni restano occupati
le richieste si accumulano
il sistema degrada
```

Con timeout:

```text id="ahl1e4"
dopo un tempo massimo, la chiamata fallisce
il chiamante può gestire l’errore
```

Esempio con `HttpClient`:

```csharp id="us3bbr"
builder.Services.AddHttpClient<PagamentiClient>(client =>
{
    client.BaseAddress = new Uri("https://pagamenti.example.com");
    client.Timeout = TimeSpan.FromSeconds(3);
});
```

Regola pratica:

```text id="g7vwvo"
Ogni chiamata remota deve avere un timeout esplicito e ragionevole.
```

---

# Retry

Il **retry** riprova un’operazione fallita per un errore temporaneo.

Esempi di errori transitori:

```text id="f0nmkp"
timeout breve
rete instabile
HTTP 503
HTTP 429
database momentaneamente occupato
```

Microsoft descrive il retry pattern come una strategia per mitigare errori transitori, riprovando l’operazione fallita un numero configurabile di volte, spesso con attesa crescente. ([Microsoft Learn][6])

Esempio concettuale:

```text id="dqt4fn"
tentativo 1 fallisce
aspetto 200 ms
tentativo 2 fallisce
aspetto 1 s
tentativo 3 riesce
```

---

# Retry: attenzione

Retry non significa “riprovare sempre”.

È pericoloso quando:

```text id="ljhkg5"
l’operazione non è idempotente
il servizio remoto è davvero giù
il retry amplifica il carico
tutti i client riprovano nello stesso momento
```

Esempio pericoloso:

```text id="z0cw38"
invio pagamento
timeout
retry
il primo pagamento era già riuscito
secondo addebito
```

Regola pratica:

```text id="lkl8ch"
Retry solo per errori transitori,
con backoff,
jitter,
limite massimo
e idempotenza dove serve.
```

---

# Circuit breaker

Il **Circuit Breaker** impedisce di continuare a chiamare un servizio che sta fallendo.

Microsoft spiega che il Circuit Breaker pattern impedisce a un’applicazione di eseguire un’operazione che probabilmente fallirà; a differenza del retry, non riprova sperando che l’operazione riesca, ma blocca temporaneamente le chiamate verso una dipendenza problematica. ([Microsoft Learn][7])

Stati:

```text id="c8c33t"
Closed
  le chiamate passano normalmente.

Open
  le chiamate vengono bloccate subito.

Half-open
  il sistema prova poche chiamate per vedere se il servizio è tornato.
```

Schema:

```text id="gh4oqv"
troppi errori
  ↓
circuito aperto
  ↓
fail fast / fallback
  ↓
dopo un intervallo
  ↓
half-open
  ↓
se ok, closed
```

---

# Perché serve il circuit breaker

Senza circuit breaker:

```text id="cks7kh"
servizio pagamenti è giù
tutte le richieste continuano a chiamarlo
ognuna aspetta timeout
le risorse si saturano
anche Ordini Service diventa lento
```

Con circuit breaker:

```text id="rwrdwj"
dopo molti errori, le chiamate falliscono subito
le risorse restano disponibili
puoi mostrare stato temporaneo o fallback
```

Regola pratica:

```text id="kx9l1e"
Circuit breaker protegge il chiamante e il sistema complessivo,
non ripara il servizio remoto.
```

---

# Bulkhead

Il **Bulkhead pattern** isola risorse per evitare che un problema in una parte del sistema affondi tutto.

Azure Architecture Center spiega che il pattern Bulkhead isola elementi di un’applicazione in pool, così che se uno fallisce gli altri continuino a funzionare; raccomanda anche di combinarlo con retry, circuit breaker e throttling per fault handling più sofisticato. ([Microsoft Learn][8])

Esempio:

```text id="fry5kl"
Report Service è lento.
Non deve consumare tutte le connessioni/thread
e bloccare anche Login, Ordini e Pagamenti.
```

Tecniche:

```text id="piyjep"
limite concorrenza per dipendenza
pool separati
code separate
istanze separate
semafori
limiti per endpoint
limiti per tenant
```

Esempio con `SemaphoreSlim`:

```csharp id="oz4rkh"
public class ReportClient
{
    private readonly HttpClient httpClient;
    private readonly SemaphoreSlim bulkhead = new(5, 5);

    public ReportClient(HttpClient httpClient)
    {
        this.httpClient = httpClient;
    }

    public async Task<string> GetReportAsync(
        int reportId,
        CancellationToken cancellationToken)
    {
        await bulkhead.WaitAsync(cancellationToken);

        try
        {
            return await httpClient.GetStringAsync(
                $"/api/report/{reportId}",
                cancellationToken);
        }
        finally
        {
            bulkhead.Release();
        }
    }
}
```

# Rate limiting

Il **rate limiting** limita quante richieste un client, utente, IP, tenant o API key può fare in un certo periodo.

Serve per:

```text id="v8b1ro"
proteggere da abuso
ridurre sovraccarico
garantire equità tra utenti
difendere endpoint costosi
limitare traffico anonimo
proteggere servizi interni
```

ASP.NET Core include middleware di rate limiting: le app configurano policy e le associano agli endpoint; Microsoft raccomanda di testare attentamente gli endpoint con rate limiting prima del deploy. ([Microsoft Learn][9])

---

## Rate limiting in ASP.NET Core

Esempio:

```csharp id="xk2u99"
using System.Threading.RateLimiting;

builder.Services.AddRateLimiter(options =>
{
    options.AddFixedWindowLimiter("api", limiterOptions =>
    {
        limiterOptions.PermitLimit = 100;
        limiterOptions.Window = TimeSpan.FromMinutes(1);
        limiterOptions.QueueProcessingOrder =
            QueueProcessingOrder.OldestFirst;
        limiterOptions.QueueLimit = 0;
    });
});

var app = builder.Build();

app.UseRateLimiter();

app.MapGet("/api/report", () =>
{
    return Results.Ok();
})
.RequireRateLimiting("api");
```

In caso di superamento limite, il client riceve tipicamente un errore HTTP `429 Too Many Requests`.

---

# Dove applicare rate limiting

Può essere applicato:

```text id="fjzszk"
nel gateway
nel reverse proxy
nel servizio applicativo
nell’ingress controller
nel service mesh
nel cloud load balancer
```

Applicarlo al gateway è utile perché blocchi traffico prima che raggiunga i servizi interni.

Applicarlo anche nei servizi può essere utile per proteggere endpoint particolarmente costosi.

Regola pratica:

```text id="a7ap8r"
Rate limit al bordo per protezione generale.
Rate limit nel servizio per regole specifiche di dominio o endpoint.
```

# Resilienza con HttpClientFactory e Polly

In .NET moderno, le chiamate HTTP dovrebbero usare `IHttpClientFactory`.

Esempio:

```csharp id="t5qzuq"
builder.Services.AddHttpClient<PagamentiClient>(client =>
{
    client.BaseAddress =
        new Uri("https://pagamenti.example.com");

    client.Timeout = TimeSpan.FromSeconds(3);
});
```

Per retry, circuit breaker e altre policy, l’ecosistema .NET usa spesso Polly o le resilience pipeline integrate nei pacchetti Microsoft.Extensions.Http.Resilience. Microsoft documenta i pacchetti di resilienza .NET costruiti sopra Polly. ([Microsoft Learn][5])

Esempio concettuale:

```csharp id="egl9ot"
builder.Services.AddHttpClient<PagamentiClient>()
    .AddStandardResilienceHandler();
```

Questo applica strategie standard di resilienza alle chiamate HTTP, ma va sempre configurato e testato rispetto al dominio.

# Fallback

Il **fallback** fornisce una risposta alternativa quando una dipendenza fallisce.

Esempi:

```text id="um23cs"
mostrare dati in cache
mostrare messaggio “servizio temporaneamente non disponibile”
accettare richiesta in stato pending
disabilitare temporaneamente una feature
usare provider secondario
```

Attenzione:

```text id="p6cgwa"
fallback non deve nascondere errori critici
non deve restituire dati sbagliati come se fossero certi
```

Esempio:

```text id="ocq48q"
Il servizio report è giù.
Mostro ultimo report generato con timestamp:
  “Dati aggiornati al 25/04/2026 08:00”
```

Questo è meglio che mostrare dati vecchi senza indicarlo.

# Service mesh

Una **service mesh** è un livello infrastrutturale che gestisce comunicazione service-to-service.

In genere usa proxy sidecar o componenti di rete dedicati.

Può offrire:

```text id="pmy1jj"
mTLS tra servizi
traffic routing
retry
timeout
circuit breaking
load balancing
observability
policy
canary release
rate limiting
```

Esempi:

```text id="ra1zhs"
Istio
Linkerd
Consul Connect
Cilium Service Mesh
Open Service Mesh
```

---

# Quando valutare una service mesh

Ha senso quando:

```text id="t57jr7"
hai molti servizi
hai Kubernetes
serve mTLS uniforme
serve policy centralizzata
serve traffic management avanzato
vuoi osservabilità di rete standard
team diversi gestiscono molti servizi
```

Può essere eccessiva quando:

```text id="sju2zv"
hai pochi servizi
non hai Kubernetes
il team non è pronto a gestirla
hai già sufficiente controllo con gateway e librerie applicative
```

Regola pratica:

```text id="tbm084"
Service mesh sposta alcune responsabilità dalla libreria applicativa
all’infrastruttura, ma aggiunge complessità operativa.
```

---

# API Gateway vs Service Mesh

```text id="knfd8c"
API Gateway
  traffico nord-sud:
  client esterni → sistema interno.

Service Mesh
  traffico est-ovest:
  servizio interno → servizio interno.
```

Schema:

```text id="aqfrc3"
Internet
  ↓
API Gateway
  ↓
Servizi interni
  ↔ comunicazione gestita da service mesh
```

Non sono la stessa cosa.

Possono convivere.

# Ingress, Gateway, Reverse Proxy, API Gateway

I nomi possono confondere.

```text id="cgqh2q"
Reverse proxy
  inoltra richieste a server backend.

Ingress
  in Kubernetes, gestisce ingresso HTTP/HTTPS verso servizi.

API Gateway
  reverse proxy specializzato per API, con policy applicative.

BFF
  backend dedicato a uno specifico frontend.

Service mesh
  livello per comunicazione service-to-service interna.
```

Esempio pratico:

```text id="txts9u"
Nginx Ingress
  porta traffico nel cluster.

YARP API Gateway
  gestisce routing API e policy .NET.

Servizi interni
  comunicano tra loro, eventualmente tramite service mesh.
```

# Aggregazione nel gateway

Un gateway può aggregare dati da più servizi.

Esempio:

```text id="yqh3ij"
GET /api/mobile/home

Gateway chiama:
  Catalogo Service
  Promozioni Service
  Ordini Service

Restituisce una risposta ottimizzata per mobile.
```

Utile, ma attenzione:

```text id="r9ib5h"
aggregazioni lente
catene sincrone
logica business nel gateway
fallimenti parziali
timeout multipli
```

Regola pratica:

```text id="dakxcy"
Aggrega nel gateway solo per esigenze di esperienza client.
Non spostare nel gateway le regole di business dei servizi.
```

---

# Fallimenti parziali

In un sistema distribuito, un servizio può funzionare mentre un altro è giù.

Esempio:

```text id="h6mzqa"
Clienti Service ok
Ordini Service ok
Report Service lento
Pagamenti Service giù
```

Il sistema deve poter degradare.

Possibili comportamenti:

```text id="exxr46"
disabilitare solo pagamenti
mostrare report non aggiornati
accettare ordini in stato pending
mostrare messaggio temporaneo
usare provider alternativo
```

Questo è meglio che far fallire tutto.

---

# Esempio: gateway + resilienza

Scenario:

```text id="u5hquw"
Frontend chiama Gateway.
Gateway instrada a Ordini Service.
Ordini Service chiama Pagamenti Service.
Pagamenti Service è instabile.
```

Misure:

```text id="okh4m2"
Gateway:
  rate limiting
  autenticazione
  correlation ID
  routing
  health check destinazioni

Ordini Service:
  timeout breve verso Pagamenti
  retry con backoff solo su errori transitori
  circuit breaker
  stato ordine InAttesaPagamento
  messaggio asincrono se possibile

Pagamenti Service:
  health check
  metriche
  circuit breaker verso provider esterno
```

Risultato:

```text id="yz1c1i"
l’instabilità dei pagamenti non deve bloccare tutto il sistema.
```

# Anti-pattern

## Gateway monolitico

Sospetto:

```text id="h61e41"
Gateway contiene regole business
Gateway accede direttamente ai database
Gateway sa troppo dei modelli interni
```

Il gateway deve instradare e applicare policy, non diventare il cuore del dominio.

---

## Retry ovunque

Troppi retry possono amplificare un guasto.

Meglio:

```text id="nke1w2"
timeout
retry limitato
circuit breaker
backoff
jitter
idempotenza
```

---

## Timeout troppo lunghi

Timeout lunghi occupano risorse e peggiorano la cascata.

---

## Nessun timeout

Ancora peggio: il chiamante può restare bloccato.

---

## Rate limiting solo dopo il servizio

Se il servizio riceve comunque tutto il traffico, lo proteggi tardi.

---

## Health check pesanti

Un health check pesante può diventare esso stesso un problema.

---

## Service discovery manuale

IP hard-coded e configurazioni statiche fragili non scalano bene.

---

## Service mesh troppo presto

Una service mesh senza maturità operativa può aumentare complessità invece di ridurla.

---

# Linee guida pratiche

```text id="yym9n3"
Usa API Gateway
  per un ingresso unico, routing e policy trasversali.

Usa BFF
  quando frontend diversi hanno esigenze diverse.

Usa YARP
  se vuoi costruire gateway/reverse proxy in .NET con alta personalizzazione.

Usa service discovery
  per evitare IP statici e configurazioni fragili.

Usa load balancing
  per distribuire traffico tra istanze sane.

Usa health check
  per togliere dal traffico istanze non pronte o non sane.

Imposta timeout
  su ogni chiamata remota.

Usa retry
  solo per errori transitori, con backoff e idempotenza.

Usa circuit breaker
  per evitare chiamate continue verso dipendenze guaste.

Usa bulkhead
  per isolare risorse e limitare effetti domino.

Usa rate limiting
  al bordo e, se necessario, sui singoli servizi.

Valuta service mesh
  solo quando numero di servizi e requisiti operativi la giustificano.

Non mettere business logic nel gateway
  il dominio resta nei servizi proprietari.

Progetta fallimenti parziali
  un servizio giù non deve far crollare tutto.
```

# Riepilogo

In questo capitolo abbiamo studiato l’infrastruttura logica intorno ai servizi.

Abbiamo visto:

```text id="qlhxyc"
API Gateway
BFF
Service discovery
Load balancing
Circuit breaker
Retry
Timeout
Bulkhead
Rate limiting
Health check
Service mesh
YARP
```

La distinzione principale è questa:

```text id="vso2aa"
API Gateway
  gestisce l’ingresso e le policy verso il sistema.

BFF
  adatta il backend a uno specifico frontend.

Service discovery
  permette di trovare istanze dinamiche dei servizi.

Load balancing
  distribuisce traffico tra istanze sane.

Resilienza
  impedisce che errori locali diventino guasti a cascata.

Service mesh
  gestisce comunicazione service-to-service a livello infrastrutturale.
```

La regola principale è questa:

```text id="lt35fb"
Nei sistemi distribuiti il problema non è solo chiamare un servizio.
È chiamarlo sapendo che può essere lento, irraggiungibile,
sovraccarico, sostituito, scalato, riavviato o temporaneamente guasto.
```

API Gateway, service discovery, load balancing, timeout, retry, circuit breaker, bulkhead, rate limiting e health check servono proprio a questo: rendere il sistema capace di funzionare anche quando una parte dell’infrastruttura non si comporta come previsto.

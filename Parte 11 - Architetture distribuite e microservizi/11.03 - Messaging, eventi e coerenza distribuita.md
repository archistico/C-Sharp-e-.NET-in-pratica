# 11.03 — Messaging, eventi e coerenza distribuita

Nel capitolo precedente abbiamo visto come progettare i confini dei servizi: bounded context, autonomia, database-per-service, contratti, versionamento e scelta tra comunicazione sincrona e asincrona.

Ora entriamo nel nucleo tecnico delle architetture distribuite robuste: **messaging, eventi e coerenza distribuita**.

Il punto centrale è questo:

```text id="p7daqx"
In un sistema distribuito non puoi affidarti sempre
a una transazione unica e immediata.

Devi progettare messaggi, eventi, retry, idempotenza,
outbox, inbox, dead-letter queue e consistenza eventuale.
```

Microsoft descrive la comunicazione basata su eventi tra microservizi come uno strumento per realizzare transazioni business distribuite tramite consistenza eventuale: ogni microservizio aggiorna la propria entità e pubblica un evento che attiva il passo successivo, ma bisogna gestire esplicitamente l’idempotenza perché la transazione non copre insieme persistenza locale ed event bus. ([Microsoft Learn][1])

---

# Perché serve il messaging

In un monolite, un caso d’uso può spesso essere gestito così:

```text id="jh51d8"
inizio transazione database
  creo ordine
  aggiorno magazzino
  registro pagamento
  salvo notifica
commit
```

In un sistema a microservizi, invece:

```text id="if0yqb"
Ordini Service
  possiede DB Ordini

Magazzino Service
  possiede DB Magazzino

Pagamenti Service
  possiede DB Pagamenti

Notifiche Service
  possiede DB Notifiche
```

Non vuoi una transazione distribuita tra tutti questi database.

Allora devi coordinare il processo con messaggi ed eventi.

---

# Esempio: conferma ordine

Flusso distribuito:

```text id="ycxk92"
1. Ordini Service crea ordine in stato InAttesaPagamento.
2. Ordini Service pubblica OrdineCreato.
3. Pagamenti Service riceve OrdineCreato.
4. Pagamenti Service autorizza il pagamento.
5. Pagamenti Service pubblica PagamentoAutorizzato.
6. Ordini Service riceve PagamentoAutorizzato.
7. Ordini Service passa ordine a Confermato.
8. Notifiche Service invia email.
9. Report Service aggiorna il read model.
```

Non tutto avviene nello stesso istante.

Per un breve periodo il sistema è in uno stato intermedio.

Questo è normale, se è progettato.

# Eventi di dominio

Un **evento di dominio** rappresenta qualcosa di significativo accaduto nel dominio.

Esempi:

```text id="b5adhx"
OrdineCreato
OrdineConfermato
PagamentoAutorizzato
ClienteRegistrato
DocumentoApprovato
PrenotazioneAnnullata
```

Un evento di dominio nasce dentro il modello di dominio.

Esempio:

```csharp id="nrv9wg"
public record OrdineConfermatoDomainEvent(
    int OrdineId,
    int ClienteId,
    DateTimeOffset DataConferma);
```

Aggregate:

```csharp id="n0fonv"
public class Ordine
{
    private readonly List<object> domainEvents = [];

    public IReadOnlyCollection<object> DomainEvents =>
        domainEvents.AsReadOnly();

    public int Id { get; private set; }
    public int ClienteId { get; private set; }
    public StatoOrdine Stato { get; private set; }

    public void Conferma(DateTimeOffset dataConferma)
    {
        if (Stato != StatoOrdine.InAttesaPagamento)
        {
            throw new DomainException(
                "L'ordine non può essere confermato nello stato corrente.");
        }

        Stato = StatoOrdine.Confermato;

        domainEvents.Add(
            new OrdineConfermatoDomainEvent(
                Id,
                ClienteId,
                dataConferma));
    }

    public void ClearDomainEvents()
    {
        domainEvents.Clear();
    }
}
```

Microsoft distingue chiaramente il ruolo dei domain events: possono generare integration events da pubblicare fuori dai confini del microservizio, ma non sono necessariamente la stessa cosa. ([Microsoft Learn][2])

---

# A cosa servono i domain events

Servono a separare:

```text id="d0ohmk"
azione principale
  confermare ordine

reazioni secondarie
  aggiornare statistiche
  preparare notifica
  generare evento di integrazione
  avviare processo successivo
```

Il dominio non deve sapere come inviare email, pubblicare su RabbitMQ o aggiornare un sistema esterno.

Il dominio dice solo:

```text id="l6syeh"
È accaduto OrdineConfermato.
```

Poi l’application layer o infrastructure decide cosa farne.

# Eventi di integrazione

Un **evento di integrazione** è un messaggio pubblicato verso altri servizi o bounded context.

Esempio:

```csharp id="sh7j3v"
public record OrdineConfermatoIntegrationEvent(
    Guid MessageId,
    int OrdineId,
    int ClienteId,
    DateTimeOffset DataConferma);
```

Differenza pratica:

```text id="d2cpxs"
Domain event
  interno al servizio.
  Esprime qualcosa accaduto nel dominio.

Integration event
  esterno al servizio.
  È un contratto pubblico verso altri sistemi.
```

Non conviene pubblicare direttamente il domain event così com’è.

Meglio trasformarlo:

```text id="xprfsu"
OrdineConfermatoDomainEvent
  ↓ mapping
OrdineConfermatoIntegrationEvent
```

Perché?

```text id="m60285"
il domain event può cambiare con il modello interno
l’integration event è un contratto esterno
non vuoi esporre dettagli interni
puoi controllare versionamento e compatibilità
```

---

# Domain event vs integration event

```text id="zr5dp9"
Domain event
  vive dentro il bounded context
  può contenere concetti interni
  non è necessariamente stabile per altri servizi
  può essere gestito in memoria nella stessa transazione applicativa

Integration event
  attraversa confini di servizio
  deve essere stabile e versionato
  deve essere serializzabile
  deve essere idempotente lato consumer
  viene pubblicato su broker, bus o stream
```

Regola pratica:

```text id="u6v9d7"
Un evento di dominio racconta al tuo servizio cosa è successo.
Un evento di integrazione racconta agli altri servizi cosa possono sapere.
```

# Command message ed event message

Nel messaging è importante distinguere tra **comando** ed **evento**.

## Command message

Un comando chiede a qualcuno di fare qualcosa.

```text id="a7fj8c"
AutorizzaPagamento
RiservaMagazzino
InviaEmailConferma
GeneraFattura
```

Ha un destinatario logico.

```text id="m4w9oq"
Pagamenti Service, autorizza questo pagamento.
```

## Event message

Un evento comunica che qualcosa è già accaduto.

```text id="puqe27"
OrdineCreato
PagamentoAutorizzato
MagazzinoRiservato
EmailInviata
```

Non ordina direttamente cosa fare.

```text id="s9r2ga"
OrdineCreato è accaduto.
Chi è interessato reagisca.
```

Regola pratica:

```text id="f6h92r"
Command:
  fai questa cosa.

Event:
  questa cosa è successa.
```

# RabbitMQ

**RabbitMQ** è un message broker molto usato per code e publish/subscribe.

Concetti base:

```text id="gn68fj"
Producer
  pubblica messaggi.

Exchange
  riceve messaggi dal producer e li instrada.

Queue
  conserva messaggi fino al consumo.

Consumer
  legge messaggi dalla queue.

Binding
  collega exchange e queue con regole di routing.
```

Schema:

```text id="nub0uu"
Producer
  ↓
Exchange
  ↓ binding/routing
Queue
  ↓
Consumer
```

---

# Exchange RabbitMQ

Tipi comuni:

```text id="w3kmz3"
direct
  instrada in base a routing key esatta.

fanout
  invia a tutte le code collegate.

topic
  instrada usando pattern su routing key.

headers
  instrada in base agli header.
```

Esempio:

```text id="a8a83g"
Ordini Service pubblica evento su exchange ordini.events

Queue pagamenti.ordine-creato
  riceve OrdineCreato

Queue report.ordine-creato
  riceve OrdineCreato

Queue notifiche.ordine-confermato
  riceve OrdineConfermato
```

RabbitMQ documenta anche il concetto di dead-letter exchange: un messaggio può essere “dead-lettered”, cioè ripubblicato verso un altro exchange, quando viene rifiutato, scade, supera un limite di lunghezza della coda o supera un limite di consegne in quorum queue. ([rabbitmq.com][3])

# Perché non basta `PublishAsync`

Pubblicare un messaggio sembra semplice:

```csharp id="y0n9lp"
await dbContext.SaveChangesAsync(cancellationToken);

await publishEndpoint.Publish(
    evento,
    cancellationToken);
```

Ma c’è un problema.

Scenario 1:

```text id="k38l8z"
SaveChanges riesce.
Publish fallisce.
Il dato è salvato, ma l’evento non parte.
```

Scenario 2:

```text id="xckbc7"
Publish riesce.
SaveChanges fallisce.
L’evento parte, ma il dato non esiste davvero.
```

Questo è il classico problema risolto dall’**Outbox pattern**.

# Outbox pattern

L’**Outbox pattern** salva il messaggio da pubblicare nello stesso database e nella stessa transazione dei dati applicativi.

Schema:

```text id="co4zqu"
Transazione locale:
  salva ordine
  salva outbox message
commit

Processo separato:
  legge outbox message non pubblicati
  pubblica sul broker
  marca come pubblicato
```

Tabella:

```text id="fw7vr8"
OutboxMessages
  Id
  Type
  Payload
  OccurredAt
  PublishedAt
  RetryCount
  Error
```

L’obiettivo è evitare di perdere eventi dopo un commit del database.

Microsoft descrive il transactional outbox come pattern per reliable messaging e guaranteed delivery degli eventi, parte importante del supporto a processamenti idempotenti. ([Microsoft Learn][5])

---

# Esempio Outbox manuale

Entity:

```csharp id="b4e63c"
public class OutboxMessage
{
    public Guid Id { get; set; }
    public string Type { get; set; } = "";
    public string Payload { get; set; } = "";
    public DateTimeOffset OccurredAt { get; set; }
    public DateTimeOffset? PublishedAt { get; set; }
    public int RetryCount { get; set; }
    public string? Error { get; set; }
}
```

Durante il caso d’uso:

```csharp id="l2p0k9"
public async Task ConfermaOrdineAsync(
    int ordineId,
    CancellationToken cancellationToken)
{
    Ordine ordine = await dbContext.Ordini
        .FirstOrDefaultAsync(
            o => o.Id == ordineId,
            cancellationToken)
        ?? throw new NotFoundException("Ordine non trovato.");

    ordine.Conferma(clock.Now);

    OrdineConfermatoIntegrationEvent evento = new(
        MessageId: Guid.NewGuid(),
        OrdineId: ordine.Id,
        ClienteId: ordine.ClienteId,
        DataConferma: clock.Now);

    dbContext.OutboxMessages.Add(new OutboxMessage
    {
        Id = evento.MessageId,
        Type = nameof(OrdineConfermatoIntegrationEvent),
        Payload = JsonSerializer.Serialize(evento),
        OccurredAt = clock.Now
    });

    await dbContext.SaveChangesAsync(cancellationToken);
}
```

Worker:

```csharp id="otk5qx"
public class OutboxWorker : BackgroundService
{
    private readonly IServiceScopeFactory scopeFactory;
    private readonly ILogger<OutboxWorker> logger;

    public OutboxWorker(
        IServiceScopeFactory scopeFactory,
        ILogger<OutboxWorker> logger)
    {
        this.scopeFactory = scopeFactory;
        this.logger = logger;
    }

    protected override async Task ExecuteAsync(
        CancellationToken stoppingToken)
    {
        using PeriodicTimer timer =
            new(TimeSpan.FromSeconds(5));

        while (await timer.WaitForNextTickAsync(stoppingToken))
        {
            await ProcessAsync(stoppingToken);
        }
    }

    private async Task ProcessAsync(
        CancellationToken cancellationToken)
    {
        using IServiceScope scope =
            scopeFactory.CreateScope();

        AppDbContext dbContext =
            scope.ServiceProvider.GetRequiredService<AppDbContext>();

        IPublishEndpoint publishEndpoint =
            scope.ServiceProvider.GetRequiredService<IPublishEndpoint>();

        List<OutboxMessage> messages = await dbContext.OutboxMessages
            .Where(m => m.PublishedAt == null)
            .OrderBy(m => m.OccurredAt)
            .Take(50)
            .ToListAsync(cancellationToken);

        foreach (OutboxMessage message in messages)
        {
            try
            {
                if (message.Type == nameof(OrdineConfermatoIntegrationEvent))
                {
                    OrdineConfermatoIntegrationEvent? evento =
                        JsonSerializer.Deserialize<OrdineConfermatoIntegrationEvent>(
                            message.Payload);

                    if (evento is not null)
                    {
                        await publishEndpoint.Publish(
                            evento,
                            cancellationToken);
                    }
                }

                message.PublishedAt = DateTimeOffset.UtcNow;
                message.Error = null;
            }
            catch (Exception ex)
            {
                message.RetryCount++;
                message.Error = ex.Message;

                logger.LogError(
                    ex,
                    "Errore pubblicazione outbox message {MessageId}",
                    message.Id);
            }
        }

        await dbContext.SaveChangesAsync(cancellationToken);
    }
}
```

# Outbox in MassTransit

MassTransit supporta diversi approcci di outbox. La documentazione distingue l’in-memory outbox, che bufferizza messaggi nel processo fino al completamento del consumer, dalla transactional outbox, che persiste i messaggi nel database per consegnarli successivamente anche dopo un riavvio del processo. ([MassTransit Documentation][6])

MassTransit descrive anche che l’outbox trattiene i messaggi e li consegna dopo il completamento della parte transazionale; con una saga, per esempio, i messaggi vengono consegnati dopo che l’istanza della saga è stata salvata correttamente. ([MassTransit Documentation][7])

Concetto:

```text id="fsdc72"
handler/consumer
  aggiorna database
  produce messaggi

MassTransit outbox
  salva messaggi nel database
  consegna dopo commit riuscito
```

Questo riduce molto il codice infrastrutturale manuale.

# Inbox pattern

L’**Inbox pattern** protegge il consumer da messaggi duplicati.

In un sistema distribuito, devi assumere:

```text id="j2e4n9"
lo stesso messaggio può arrivare più di una volta
il consumer può fallire dopo aver elaborato ma prima di fare ack
il broker può riconsegnare
un retry può ripetere l’operazione
```

L’inbox salva gli ID dei messaggi già processati.

Tabella:

```text id="btg95z"
InboxMessages
  MessageId
  ConsumerName
  ProcessedAt
```

Consumer:

```csharp id="ganp4i"
public class OrdineConfermatoConsumer
    : IConsumer<OrdineConfermatoIntegrationEvent>
{
    private readonly AppDbContext dbContext;
    private readonly ILogger<OrdineConfermatoConsumer> logger;

    public OrdineConfermatoConsumer(
        AppDbContext dbContext,
        ILogger<OrdineConfermatoConsumer> logger)
    {
        this.dbContext = dbContext;
        this.logger = logger;
    }

    public async Task Consume(
        ConsumeContext<OrdineConfermatoIntegrationEvent> context)
    {
        OrdineConfermatoIntegrationEvent message =
            context.Message;

        string consumerName =
            nameof(OrdineConfermatoConsumer);

        bool alreadyProcessed = await dbContext.InboxMessages
            .AnyAsync(
                m => m.MessageId == message.MessageId &&
                     m.ConsumerName == consumerName,
                context.CancellationToken);

        if (alreadyProcessed)
        {
            logger.LogInformation(
                "Messaggio {MessageId} già processato da {ConsumerName}",
                message.MessageId,
                consumerName);

            return;
        }

        // logica idempotente del consumer
        // esempio: aggiorna read model, invia notifica, ecc.

        dbContext.InboxMessages.Add(new InboxMessage
        {
            MessageId = message.MessageId,
            ConsumerName = consumerName,
            ProcessedAt = DateTimeOffset.UtcNow
        });

        await dbContext.SaveChangesAsync(
            context.CancellationToken);
    }
}
```

Regola pratica:

```text id="nvge3z"
Outbox:
  non perdere messaggi in uscita.

Inbox:
  non processare due volte lo stesso messaggio in ingresso.
```

---

# Idempotenza

Un consumer è **idempotente** se può ricevere lo stesso messaggio più volte senza produrre effetti indesiderati.

Esempio non idempotente:

```csharp id="itp5ts"
cliente.Punti += evento.Punti;
```

Se il messaggio arriva due volte, i punti vengono aggiunti due volte.

Esempio più sicuro:

```csharp id="ku2kdz"
bool giaApplicato = await dbContext.EventiPuntiApplicati
    .AnyAsync(e => e.MessageId == evento.MessageId);

if (!giaApplicato)
{
    cliente.Punti += evento.Punti;

    dbContext.EventiPuntiApplicati.Add(new EventoPuntiApplicato
    {
        MessageId = evento.MessageId,
        AppliedAt = DateTimeOffset.UtcNow
    });

    await dbContext.SaveChangesAsync();
}
```

---

# Perché l’idempotenza è obbligatoria

Nel messaging reale puoi avere duplicati per:

```text id="ns42zi"
retry applicativo
consumer crash
ack non inviato
timeout
broker riconsegna
outbox ripubblicata
errore dopo side effect
```

Regola pratica:

```text id="asbn5a"
Non chiederti se un messaggio arriverà due volte.
Progetta cosa succede quando arriverà due volte.
```

# Retry

Il **retry** riprova una elaborazione fallita.

È utile per errori temporanei:

```text id="w650l9"
database momentaneamente non disponibile
rete instabile
servizio esterno lento
lock temporaneo
timeout transitorio
```

Non è utile per errori permanenti:

```text id="y71jkw"
JSON non valido
campo obbligatorio mancante
business rule violata
messaggio incompatibile
```

---

# Retry con MassTransit

Esempio:

```csharp id="j5esxi"
builder.Services.AddMassTransit(x =>
{
    x.AddConsumer<OrdineConfermatoConsumer>();

    x.UsingRabbitMq((context, cfg) =>
    {
        cfg.Host("rabbitmq", "/", h =>
        {
            h.Username("app");
            h.Password("app");
        });

        cfg.UseMessageRetry(retry =>
        {
            retry.Interval(
                retryCount: 3,
                interval: TimeSpan.FromSeconds(5));
        });

        cfg.ConfigureEndpoints(context);
    });
});
```

Strategie comuni:

```text id="jjxw1n"
immediate retry
  riprova subito.

interval retry
  riprova dopo intervallo fisso.

exponential backoff
  aumenta progressivamente l’attesa.

redelivery ritardata
  rimanda il messaggio in coda per un tentativo futuro.
```

Regola pratica:

```text id="e9k2mk"
Retry solo se l’errore può davvero risolversi riprovando.
```

---

# Retry: rischio amplificazione

Il retry può peggiorare un problema.

Scenario:

```text id="ck12po"
servizio pagamenti è giù
100 richieste falliscono
ognuna fa 5 retry
diventano 500 chiamate verso un servizio già in crisi
```

Per questo si combinano:

```text id="zbjfx7"
retry con backoff
circuit breaker
bulkhead
timeout
dead-letter queue
monitoraggio
```

# Dead-letter queue

Una **dead-letter queue**, spesso abbreviata DLQ, raccoglie messaggi che non possono essere processati correttamente.

Cause comuni:

```text id="b1hm1r"
retry esauriti
messaggio non valido
consumer sempre in errore
schema incompatibile
errore permanente
TTL scaduto
messaggio rifiutato senza requeue
```

RabbitMQ chiama questo meccanismo dead-lettering: un messaggio viene ripubblicato a un exchange dedicato quando viene rifiutato, scade, supera limiti di coda o, nel caso delle quorum queue, supera il limite di consegne. ([rabbitmq.com][3])

---

# A cosa serve una DLQ

Una DLQ serve a:

```text id="p39hku"
non perdere il messaggio
non bloccare la coda principale
analizzare il problema
correggere dati o codice
ripubblicare il messaggio se opportuno
creare alert operativo
```

Non deve diventare un cimitero ignorato.

Regola pratica:

```text id="xvi06g"
Ogni DLQ deve avere monitoraggio e procedura operativa.
```

---

# Poison message

Un **poison message** è un messaggio che fallisce sempre.

Esempio:

```json id="uqwtu3"
{
  "ordineId": null,
  "clienteId": 123
}
```

Se `ordineId` è obbligatorio, ogni retry fallirà.

Strategia:

```text id="bte3gu"
non riprovare all’infinito
spostare in DLQ
loggare motivo
notificare operatori
correggere o scartare consapevolmente
```

# Saga

Una **Saga** coordina un processo distribuito composto da più passi.

Esempio:

```text id="ebcueu"
1. Crea ordine
2. Riserva magazzino
3. Autorizza pagamento
4. Prepara spedizione
5. Conferma ordine
```

Se un passo fallisce, servono compensazioni.

Esempio:

```text id="r58fz7"
Pagamento fallisce
  ↓
rilascia magazzino
  ↓
annulla ordine
  ↓
notifica cliente
```

Una saga non è una transazione ACID distribuita.

È un workflow distribuito con stati, eventi e compensazioni.

---

# Stato della saga

Una saga spesso ha uno stato persistente:

```csharp id="z5p76u"
public class OrdineSagaState
{
    public Guid CorrelationId { get; set; }
    public int OrdineId { get; set; }
    public bool MagazzinoRiservato { get; set; }
    public bool PagamentoAutorizzato { get; set; }
    public bool SpedizionePreparata { get; set; }
    public string Stato { get; set; } = "";
}
```

Serve a sapere:

```text id="w39vkx"
dove siamo nel processo
quali passi sono completati
quali compensazioni servono
quali messaggi sono già arrivati
```

---

# Coreografia

Nella **coreografia**, non c’è un coordinatore centrale.

Ogni servizio reagisce agli eventi e pubblica nuovi eventi.

Schema:

```text id="exhayu"
Ordini Service pubblica OrdineCreato

Magazzino Service riceve OrdineCreato
  pubblica MagazzinoRiservato

Pagamenti Service riceve MagazzinoRiservato
  pubblica PagamentoAutorizzato

Spedizioni Service riceve PagamentoAutorizzato
  pubblica SpedizionePreparata

Ordini Service riceve eventi e aggiorna stato
```

---

## Pro della coreografia

```text id="dpy678"
servizi molto disaccoppiati
nessun orchestratore centrale
facile aggiungere nuovi subscriber
buona aderenza a event-driven architecture
```

## Contro della coreografia

```text id="uaawxq"
flusso difficile da vedere
debug più complesso
rischio catene di eventi non documentate
logica distribuita tra servizi
difficile capire chi controlla il processo
```

La coreografia va bene quando il processo è semplice o naturalmente event-driven.

---

# Orchestrazione

Nell’**orchestrazione**, un componente centrale coordina il processo.

Schema:

```text id="ifh0ua"
OrderSaga
  invia RiservaMagazzino
  attende MagazzinoRiservato
  invia AutorizzaPagamento
  attende PagamentoAutorizzato
  invia PreparaSpedizione
  attende SpedizionePreparata
  conclude ordine
```

---

## Pro dell’orchestrazione

```text id="x8ub5m"
flusso esplicito
più facile capire lo stato del processo
compensazioni centralizzate
debug più diretto
adatta a workflow complessi
```

## Contro dell’orchestrazione

```text id="kkri1l"
orchestratore più importante e complesso
rischio di accoppiamento verso molti servizi
richiede stato persistente affidabile
può diventare un punto centrale di coordinamento
```

Regola pratica:

```text id="rlg58p"
Coreografia per processi semplici e reazioni indipendenti.
Orchestrazione per workflow lunghi, critici e con compensazioni chiare.
```

---

# Saga con MassTransit: concetto

MassTransit supporta saghe e state machine.

Esempio concettuale di eventi:

```csharp id="n6wpnx"
public record OrdineCreato(
    Guid CorrelationId,
    int OrdineId);

public record MagazzinoRiservato(
    Guid CorrelationId,
    int OrdineId);

public record PagamentoAutorizzato(
    Guid CorrelationId,
    int OrdineId);

public record PagamentoFallito(
    Guid CorrelationId,
    int OrdineId,
    string Motivo);
```

State machine concettuale:

```csharp id="gtlld7"
public class OrdineState :
    SagaStateMachineInstance
{
    public Guid CorrelationId { get; set; }
    public string CurrentState { get; set; } = "";
    public int OrdineId { get; set; }
}
```

L’idea è:

```text id="ni7jjq"
ogni evento correlato allo stesso processo usa lo stesso CorrelationId
la saga carica il proprio stato
decide il prossimo passo
pubblica comandi o eventi
salva lo stato aggiornato
```

---

# Correlation ID nei messaggi

Ogni processo distribuito deve avere un identificativo di correlazione.

Esempio:

```csharp id="waqq3t"
public record OrdineCreatoIntegrationEvent(
    Guid MessageId,
    Guid CorrelationId,
    int OrdineId,
    int ClienteId);
```

Distinzione:

```text id="xkczl3"
MessageId
  identifica il singolo messaggio.
  Serve per inbox/idempotenza.

CorrelationId
  identifica il processo o flusso distribuito.
  Serve per collegare messaggi, log e trace.
```

Regola pratica:

```text id="qkpfk2"
MessageId per deduplicare.
CorrelationId per seguire il flusso.
```

# Consistenza eventuale

La **consistenza eventuale** significa che il sistema non è immediatamente allineato ovunque, ma converge nel tempo.

Esempio:

```text id="f03mle"
Ordini Service:
  ordine confermato alle 10:00:00

Report Service:
  vede l’ordine nel report alle 10:00:05
```

Questo può essere accettabile se il business lo sa.

Microsoft sottolinea che, quando si usa consistenza eventuale guidata da integration events, questo approccio deve essere chiaro anche all’utente e al business owner: il sistema non deve fingere di essere immediatamente consistente se non lo è. ([Microsoft Learn][8])

---

# Rendere visibile la consistenza eventuale

Esempi UI/API:

```text id="zf2j09"
Ordine ricevuto. Stiamo verificando il pagamento.
```

```text id="flj1ph"
Import in elaborazione.
Riceverai una notifica al completamento.
```

```text id="qrzo4s"
Report aggiornato ogni 5 minuti.
Ultimo aggiornamento: 10:05.
```

Meglio essere espliciti che mostrare dati apparentemente definitivi ma non ancora aggiornati.

# Stati intermedi

Per usare consistenza eventuale bene, servono stati intermedi nel dominio.

Esempio ordine:

```text id="awx1id"
Bozza
InAttesaPagamento
PagamentoAutorizzato
InPreparazione
Spedito
Annullato
PagamentoFallito
```

Non ridurre tutto a:

```text id="esth5v"
Confermato / Non confermato
```

se il processo reale ha più fasi.

Gli stati intermedi permettono di spiegare al sistema e all’utente che cosa sta accadendo.

# Esempio completo: ordine distribuito robusto

## 1. Ordini Service crea ordine

```text id="p6m8ui"
Ordine:
  Stato = InAttesaPagamento

Outbox:
  OrdineCreatoIntegrationEvent
```

Transazione locale:

```text id="ko4ik4"
salva ordine
salva outbox message
commit
```

## 2. Outbox pubblica evento

```text id="pdbn40"
OutboxWorker pubblica OrdineCreato su RabbitMQ
```

## 3. Pagamenti Service consuma evento

```text id="wzw6ac"
Inbox controlla MessageId
se già processato:
  ignora

altrimenti:
  tenta autorizzazione pagamento
  salva esito
  salva outbox PagamentoAutorizzato o PagamentoFallito
  salva inbox message
  commit
```

## 4. Ordini Service consuma esito

```text id="w728zq"
se PagamentoAutorizzato:
  Stato = Confermato

se PagamentoFallito:
  Stato = PagamentoFallito
```

## 5. Notifiche e Report reagiscono

```text id="iadxne"
Notifiche:
  invia email

Report:
  aggiorna read model
```

Questo flusso usa:

```text id="qlqkzr"
eventi di integrazione
outbox
inbox
idempotenza
retry
consistenza eventuale
stati intermedi
```

# Dead-letter e operatività

Quando un messaggio finisce in DLQ, serve una procedura.

Domande operative:

```text id="ny02fi"
Quale consumer ha fallito?
Qual è il MessageId?
Qual è il CorrelationId?
Qual è l’errore?
Quanti retry sono stati fatti?
Il messaggio è correggibile?
Va ripubblicato?
Va scartato?
Serve correggere codice?
Serve correggere dati?
```

Dashboard utili:

```text id="y6w1lj"
numero messaggi in DLQ
età massima messaggio in DLQ
numero retry
consumer più fallimentari
messaggi non processati
latenza code
```

Regola pratica:

```text id="avvon0"
Una DLQ senza alert è solo un posto dove gli errori vanno a nascondersi.
```

# Versionamento eventi

Gli eventi di integrazione sono contratti.

Devono essere versionati con attenzione.

Esempio v1:

```csharp id="ibz9d5"
public record OrdineConfermatoV1(
    Guid MessageId,
    int OrdineId,
    int ClienteId);
```

v2 compatibile:

```csharp id="jrmt9b"
public record OrdineConfermatoV2(
    Guid MessageId,
    int OrdineId,
    int ClienteId,
    DateTimeOffset DataConferma);
```

Aggiungere un campo opzionale è spesso compatibile.

Breaking change:

```text id="l1ed74"
rinominare OrdineId
cambiare tipo di ClienteId
rimuovere campi
cambiare significato
```

Regola pratica:

```text id="pmyg9t"
Gli eventi pubblicati sono API.
Cambiali con la stessa cautela con cui cambieresti una API pubblica.
```

---

# Eventi troppo generici

Sbagliato:

```text id="a2pucj"
EntityUpdated
RecordChanged
DataModified
```

Perché non raccontano il business.

Meglio:

```text id="hf6z7j"
ClienteRegistrato
EmailClienteAggiornata
OrdineConfermato
PagamentoRifiutato
DocumentoApprovato
```

Un buon evento dice:

```text id="gs00sa"
che cosa è successo
nel linguaggio del dominio
```

---

# Eventi troppo dettagliati

Anche l’opposto è un rischio.

Sospetto:

```text id="jmy7js"
CampoEmailTabellaClientiAggiornato
FlagXImpostatoA1
RigaOrdiniAggiornata
```

Questi sono eventi tecnici, non eventi di business.

Regola pratica:

```text id="g8n0fn"
Pubblica eventi utili ai consumer,
non il log tecnico di ogni modifica interna.
```

# Retry, Outbox e duplicati

Anche con outbox, possono esserci duplicati.

Scenario:

```text id="xnvi0g"
Outbox pubblica messaggio.
Broker riceve.
Processo muore prima di marcare PublishedAt.
Outbox riprova.
Il messaggio arriva due volte.
```

Quindi:

```text id="j28jce"
Outbox riduce il rischio di perdita.
Non elimina la necessità di idempotenza.
```

Per questo serve Inbox o consumer idempotenti.

---

# Exactly once: attenzione

Spesso si desidera:

```text id="yy8asf"
voglio processare ogni messaggio esattamente una volta
```

Nella pratica dei sistemi distribuiti, è più realistico progettare per:

```text id="kbcn5m"
at-least-once delivery
  il messaggio arriva almeno una volta, forse più di una.

consumer idempotente
  se arriva più volte, l’effetto finale resta corretto.
```

Regola pratica:

```text id="z3eh1q"
Non basare la correttezza del sistema sull’idea che i duplicati
non esisteranno.
```

# Checklist per un consumer robusto

```text id="pegulq"
validare messaggio
controllare MessageId
gestire idempotenza
usare transazione locale
salvare inbox record
loggare con CorrelationId
distinguere errori temporanei e permanenti
configurare retry sensato
mandare in DLQ dopo retry esauriti
emettere metriche
non bloccare la coda per sempre
```

---

# Checklist per un publisher robusto

```text id="h8zv8j"
creare integration event esplicito
assegnare MessageId
assegnare CorrelationId
salvare evento in outbox nella stessa transazione dei dati
pubblicare da worker o outbox integrata
gestire retry pubblicazione
marcare PublishedAt solo dopo successo
monitorare outbox arretrata
non pubblicare eventi prima del commit
```

# Errori comuni

## Pubblicare eventi dentro il dominio

Sbagliato:

```csharp id="dxe5f3"
public void Conferma()
{
    Stato = StatoOrdine.Confermato;
    rabbitMq.Publish(...);
}
```

Il dominio non deve conoscere RabbitMQ.

---

## Confondere domain event e integration event

Un domain event interno non dovrebbe diventare automaticamente contratto pubblico.

---

## Pubblicare prima del commit

Pericoloso:

```text id="so5eyr"
pubblico evento
poi il database fallisce
```

---

## Nessuna idempotenza

Errore grave:

```text id="k2g8gq"
assumo che il messaggio arrivi una sola volta
```

---

## Retry infinito

Un poison message può bloccare o intasare il sistema.

---

## DLQ non monitorata

Una dead-letter queue senza alert non serve abbastanza.

---

## Eventi generici

`EntityUpdated` non è un buon evento di integrazione.

---

## Coreografia senza visibilità

Troppi eventi senza documentazione rendono il processo invisibile.

---

## Orchestratore onnisciente

Un orchestratore che conosce troppo di tutti i servizi può diventare un nuovo monolite logico.

---

# Linee guida pratiche

```text id="pqh08a"
Usa domain events
  per esprimere fatti importanti dentro il dominio.

Usa integration events
  per comunicare fatti stabili ad altri servizi.

Non pubblicare direttamente dal dominio
  passa da application/infrastructure layer.

Usa RabbitMQ o broker equivalenti
  per disaccoppiare producer e consumer.

Usa MassTransit
  per semplificare consumer, retry, saghe e outbox in .NET.

Usa Outbox pattern
  per non perdere messaggi dopo il commit dei dati.

Usa Inbox pattern
  per evitare doppia elaborazione.

Rendi i consumer idempotenti
  perché i duplicati accadranno.

Usa retry con criterio
  solo per errori transitori.

Usa dead-letter queue
  per messaggi non processabili.

Scegli coreografia
  per reazioni semplici e indipendenti.

Scegli orchestrazione
  per workflow lunghi, critici e con compensazioni.

Accetta consistenza eventuale
  ma rendila esplicita nel dominio e nella UI.

Versiona gli eventi
  come contratti pubblici.

Monitora code, outbox, inbox e DLQ
  non solo le API HTTP.
```

# Riepilogo

In questo capitolo abbiamo studiato il nucleo tecnico delle architetture distribuite robuste.

Abbiamo visto:

```text id="f1ez2v"
eventi di dominio
eventi di integrazione
RabbitMQ
MassTransit
Outbox pattern
Inbox pattern
Saga
coreografia
orchestrazione
idempotenza
retry
dead-letter queue
consistenza eventuale
```

La distinzione più importante è questa:

```text id="qtb3r8"
Domain event
  fatto interno al dominio.

Integration event
  contratto esterno verso altri servizi.

Outbox
  protegge la pubblicazione dei messaggi.

Inbox
  protegge il consumer dai duplicati.

Saga
  coordina processi distribuiti con stati e compensazioni.

DLQ
  isola messaggi non processabili dopo retry.
```

La regola principale è questa:

```text id="ui52y6"
Nei sistemi distribuiti non progetti solo il caso felice.
Progetti soprattutto cosa succede quando qualcosa fallisce,
arriva in ritardo, arriva due volte, non arriva subito
o deve essere compensato.
```

Messaging ed eventi permettono servizi più autonomi e resilienti, ma richiedono disciplina: contratti chiari, idempotenza, outbox, inbox, retry, DLQ, osservabilità e accettazione esplicita della consistenza eventuale.

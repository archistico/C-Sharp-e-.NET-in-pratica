# 11.06 — Kubernetes e orchestrazione

Nei capitoli precedenti abbiamo visto Docker, CI/CD, osservabilità, microservizi, messaging, sicurezza distribuita e resilienza. Ora possiamo parlare di **Kubernetes** nel punto giusto del percorso.

Kubernetes non serve per imparare C# o .NET.
Non è nemmeno obbligatorio per usare i container.

Kubernetes diventa utile quando devi eseguire molte applicazioni containerizzate, scalarle, aggiornarle, monitorarle e mantenerle disponibili in ambienti complessi.

Il punto centrale è questo:

```text id="v5x1j4"
Kubernetes non è il punto di partenza.
È uno strumento di orchestrazione avanzato per gestire container
quando singoli server, Docker manuale o Docker Compose non bastano più.
```

La documentazione ufficiale definisce Kubernetes come un motore open source di orchestrazione per automatizzare deployment, scaling e gestione di applicazioni containerizzate. ([Kubernetes][1])

---

# Perché Kubernetes

Con Docker puoi eseguire un container:

```bash id="cw5y7c"
docker run miaapp-api:1.0.0
```

Con Docker Compose puoi eseguire più container su una macchina:

```bash id="u57uy2"
docker compose up -d
```

Ma in produzione potresti avere bisogno di:

```text id="eibv48"
più server
più istanze della stessa API
scaling automatico
rolling update
rollback
health check
riavvio automatico dei container
service discovery
load balancing interno
gestione configurazioni
gestione segreti
ingresso HTTP/HTTPS
deploy dichiarativo
monitoraggio centralizzato
```

Kubernetes nasce per questi scenari.

---

# Che cosa fa Kubernetes

Kubernetes gestisce lo stato desiderato.

Tu dichiari:

```text id="e002yh"
voglio 3 repliche della mia API
voglio che usino questa image
voglio esporle con questo Service
voglio queste variabili d’ambiente
voglio questi limiti CPU/memoria
```

Kubernetes cerca continuamente di portare lo stato reale verso quello desiderato.

Esempio:

```text id="blhh8x"
Desiderato:
  3 Pod attivi

Reale:
  2 Pod attivi perché uno è morto

Kubernetes:
  crea un nuovo Pod
```

Questo è il cuore del modello dichiarativo.

---

# Quando Kubernetes ha senso

Kubernetes ha senso quando:

```text id="v39apx"
hai molte applicazioni containerizzate
hai più ambienti
devi scalare orizzontalmente
devi fare rolling update frequenti
hai bisogno di self-healing
hai team diversi che distribuiscono servizi
hai microservizi
hai esigenze cloud-native
hai osservabilità e CI/CD già mature
```

Esempi:

```text id="cnzml2"
piattaforma SaaS
sistema a microservizi
cluster di API e worker
ambiente multi-team
applicazioni con picchi di carico
servizi containerizzati distribuiti su più nodi
```

---

# Quando Kubernetes è eccessivo

Kubernetes può essere eccessivo quando:

```text id="zkycny"
hai una singola web app
hai un piccolo gestionale interno
hai un solo server
non hai competenze DevOps
non hai osservabilità
non hai pipeline CI/CD
non hai bisogno di scaling automatico
Docker Compose o IIS/systemd bastano
```

Regola pratica:

```text id="cv6v7u"
Non usare Kubernetes per sentirti moderno.
Usalo quando il problema operativo giustifica la complessità.
```

# Cluster, nodi e control plane

Un cluster Kubernetes è composto da:

```text id="janwr1"
Control Plane
  decide e coordina.

Worker Nodes
  eseguono i container.
```

Schema:

```text id="gvb0h1"
┌─────────────────────────────┐
│        Control Plane         │
│  API Server                  │
│  Scheduler                   │
│  Controller Manager          │
│  etcd                        │
└──────────────┬──────────────┘
               │
        ┌──────┴───────┐
        │              │
┌───────▼──────┐ ┌─────▼───────┐
│ Worker Node  │ │ Worker Node │
│ Pod, Pod     │ │ Pod, Pod    │
└──────────────┘ └─────────────┘
```

Per un libro su .NET non serve entrare subito in tutti i dettagli interni, ma il lettore deve capire che Kubernetes non “esegue container a caso”: coordina risorse dichiarate tramite API.

# Pod

Il **Pod** è l’unità minima eseguibile in Kubernetes.

Un Pod contiene uno o più container che condividono:

```text id="thghu8"
rete
IP
volumi
lifecycle
contesto di esecuzione
```

Di solito, per applicazioni .NET, un Pod contiene un container principale:

```text id="qigv6a"
Pod
  container: miaapp-api
```

Esempio Pod minimale:

```yaml id="rfwc29"
apiVersion: v1
kind: Pod
metadata:
  name: miaapp-api
spec:
  containers:
    - name: miaapp-api
      image: registry.example.com/miaapp-api:1.0.0
      ports:
        - containerPort: 8080
```

Questo crea un singolo Pod.

Ma in produzione raramente crei Pod direttamente.
Di solito usi un **Deployment**.

---

# Perché non creare Pod direttamente

Se crei un Pod direttamente e il Pod muore, hai poco controllo evoluto.

Meglio usare un controller come Deployment, che mantiene il numero desiderato di repliche.

Regola pratica:

```text id="t5pwqi"
Il Pod è l’unità eseguita.
Il Deployment è il modo normale per gestire Pod applicativi.
```

---

# Deployment

Un **Deployment** gestisce un insieme di Pod per eseguire un workload applicativo, di solito stateless. La documentazione ufficiale spiega che un Deployment descrive uno stato desiderato e il Deployment Controller modifica lo stato reale a un ritmo controllato, creando e aggiornando ReplicaSet e Pod. ([Kubernetes][2])

Esempio per una API .NET:

```yaml id="v0ncwl"
apiVersion: apps/v1
kind: Deployment
metadata:
  name: miaapp-api
spec:
  replicas: 3
  selector:
    matchLabels:
      app: miaapp-api
  template:
    metadata:
      labels:
        app: miaapp-api
    spec:
      containers:
        - name: miaapp-api
          image: registry.example.com/miaapp-api:1.0.0
          ports:
            - containerPort: 8080
          env:
            - name: ASPNETCORE_URLS
              value: "http://+:8080"
            - name: ASPNETCORE_ENVIRONMENT
              value: "Production"
```

Questo dice a Kubernetes:

```text id="m1wctg"
voglio 3 repliche di miaapp-api
con questa image
esposte internamente sulla porta 8080
```

Se un Pod muore, Kubernetes ne crea un altro.

# Deployment e applicazioni stateless

Le API ASP.NET Core sono spesso buone candidate per Deployment stateless.

```text id="l6228y"
API stateless
```

Attenzione:

```text id="m9kkws"
Se salvi file caricati dentro il container,
```

Per file e dati persistenti usa:


# Service

Un **Service** fornisce un indirizzo stabile per raggiungere uno o più Pod.

I Pod sono effimeri:

```text id="xwxawc"
nascono
muoiono
cambiano IP
vengono spostati tra nodi
```

Il Service dà un nome stabile.

Schema:

```text id="qxqcen"
Service miaapp-api
        ↓
  Pod miaapp-api-1
  Pod miaapp-api-2
  Pod miaapp-api-3
```

Esempio:

```yaml id="e43t5d"
apiVersion: v1
kind: Service
metadata:
  name: miaapp-api
spec:
  type: ClusterIP
  selector:
    app: miaapp-api
  ports:
    - port: 80
      targetPort: 8080
```

Dentro il cluster, altri servizi possono chiamare:

```text id="t1rvvw"
http://miaapp-api
```

Kubernetes bilancia verso i Pod corrispondenti.

---

# Tipi di Service

Tipi comuni:

```text id="kk36g9"
ClusterIP
  espone il servizio solo dentro il cluster.

NodePort
  espone il servizio su una porta dei nodi.

LoadBalancer
  chiede al cloud provider un load balancer esterno.

ExternalName
  mappa il servizio a un nome DNS esterno.
```

Per servizi interni:

```text id="s56kop"
ClusterIP
```

Per esporre verso l’esterno in cloud:

```text id="b2gqp0"
LoadBalancer
```

In molti ambienti web reali si usa:

```text id="rrt0ne"
Ingress + Service ClusterIP
```

# Ingress

Un **Ingress** gestisce l’ingresso HTTP/HTTPS verso servizi nel cluster.

Esempio:

```text id="eqwras"
https://api.example.com/clienti
  → clienti-service

https://api.example.com/ordini
  → ordini-service
```

Kubernetes documenta Ingress come risorsa per gestire accesso esterno HTTP/HTTPS ai Service nel cluster. ([Kubernetes][3])

Esempio:

```yaml id="oek3z5"
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: miaapp-ingress
spec:
  rules:
    - host: api.example.com
      http:
        paths:
          - path: /clienti
            pathType: Prefix
            backend:
              service:
                name: clienti-api
                port:
                  number: 80
          - path: /ordini
            pathType: Prefix
            backend:
              service:
                name: ordini-api
                port:
                  number: 80
```

Nota importante:

```text id="yk527u"
Ingress è la risorsa.
Ingress Controller è il componente che la implementa.
```

Esempi di Ingress Controller:

```text id="lus7og"
NGINX Ingress Controller
Traefik
HAProxy Ingress
Azure Application Gateway Ingress Controller
AWS Load Balancer Controller
```

# ConfigMap

Una **ConfigMap** contiene configurazione non sensibile.

Esempi:

```text id="j30l2t"
ASPNETCORE_ENVIRONMENT
URL di servizi interni
livelli di log
feature flag non sensibili
nomi code
timeout
```

Esempio:

```yaml id="bsbyl9"
apiVersion: v1
kind: ConfigMap
metadata:
  name: miaapp-config
data:
  ASPNETCORE_ENVIRONMENT: "Production"
  Logging__LogLevel__Default: "Information"
  ExternalServices__ReportUrl: "http://report-api"
```

Uso nel Deployment:

```yaml id="cvmt56"
envFrom:
  - configMapRef:
      name: miaapp-config
```

Regola pratica:

```text id="l9ui2h"
ConfigMap per configurazione non sensibile.
Secret per dati sensibili.
```

# Secret

Un **Secret** contiene dati sensibili, come password, token o chiavi. La documentazione ufficiale spiega che usare un Secret evita di inserire informazioni confidenziali direttamente nel Pod specification o nell’immagine container. ([Kubernetes][4])

Esempio:

```yaml id="xabh2x"
apiVersion: v1
kind: Secret
metadata:
  name: miaapp-secrets
type: Opaque
stringData:
  ConnectionStrings__Default: "Server=sql;Database=MiaApp;User Id=app;Password=..."
  Jwt__SigningKey: "super-secret"
```

Uso:

```yaml id="s43glv"
envFrom:
  - secretRef:
      name: miaapp-secrets
```

Attenzione:

```text id="oisdom"
I Secret Kubernetes non devono essere trattati come cassaforte assoluta
senza configurazioni adeguate di sicurezza, RBAC, encryption at rest
e accesso controllato.
```

Per produzione, spesso conviene integrarli con:

```text id="icdw6r"
Azure Key Vault
AWS Secrets Manager
Google Secret Manager
HashiCorp Vault
External Secrets Operator
Sealed Secrets
CSI Secret Store Driver
```

# Resource requests e limits

In Kubernetes è importante dichiarare le risorse.

```yaml id="voskph"
resources:
  requests:
    cpu: "100m"
    memory: "128Mi"
  limits:
    cpu: "500m"
    memory: "512Mi"
```

Significato:

```text id="tvn81p"
requests
  risorse richieste per schedulare il Pod.

limits
  massimo che il container può usare.
```

Esempio completo:

```yaml id="d4aozl"
containers:
  - name: miaapp-api
    image: registry.example.com/miaapp-api:1.0.0
    resources:
      requests:
        cpu: "100m"
        memory: "128Mi"
      limits:
        cpu: "500m"
        memory: "512Mi"
```

Regola pratica:

```text id="xaxjqm"
Senza requests e limits,
Kubernetes non può prendere buone decisioni di scheduling e protezione.
```

# Limiti risorse e .NET

Per app .NET, i limiti di memoria sono particolarmente importanti.

Il Garbage Collector deve lavorare dentro il limite imposto dal container.

Linee guida:

```text id="sy0l4g"
misura memoria reale
imposta requests realistici
imposta limits non troppo stretti
monitora OOMKilled
monitora GC e allocation rate
testa sotto carico
```

Sintomo:

```text id="tqj2m3"
Pod si riavvia improvvisamente
Reason: OOMKilled
```

Significa che il container ha superato il limite di memoria.

# Probes: liveness, readiness, startup

Le probe aiutano Kubernetes a capire lo stato dell’app.

## Liveness probe

```text id="eu5zs3"
Il processo è vivo?
Se fallisce, Kubernetes può riavviare il container.
```

## Readiness probe

```text id="gq0dce"
Il Pod è pronto a ricevere traffico?
Se fallisce, il Service non dovrebbe inviare traffico a quel Pod.
```

## Startup probe

```text id="bhf2q2"
L’app ha completato l’avvio iniziale?
Utile per app lente ad avviarsi.
```

Esempio:

```yaml id="el7r0q"
livenessProbe:
  httpGet:
    path: /health/live
    port: 8080
  initialDelaySeconds: 10
  periodSeconds: 10

readinessProbe:
  httpGet:
    path: /health/ready
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 5
```

ASP.NET Core:

```csharp id="opdwh4"
builder.Services.AddHealthChecks();

var app = builder.Build();

app.MapHealthChecks("/health/live");
app.MapHealthChecks("/health/ready");
```

# Scaling orizzontale

Lo **scaling orizzontale** aumenta il numero di repliche.

Esempio:

```bash id="zvu3su"
kubectl scale deployment miaapp-api --replicas=5
```

Stato desiderato:

```text id="mgy2rg"
replicas: 5
```

Kubernetes crea o rimuove Pod per arrivare a quel numero.

YAML:

```yaml id="rxmju7"
spec:
  replicas: 5
```

Scaling orizzontale funziona bene se l’app è stateless.

Se l’app conserva stato locale nel Pod, scalare diventa più complicato.

---

# HPA — Horizontal Pod Autoscaler

L’**HorizontalPodAutoscaler** aggiorna automaticamente una risorsa, come Deployment o StatefulSet, per adattare la capacità al carico. La documentazione ufficiale spiega che scaling orizzontale significa aumentare il numero di Pod, a differenza dello scaling verticale che modifica le risorse dei Pod già esistenti. ([Kubernetes][5])

Esempio:

```yaml id="sqzc8v"
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: miaapp-api-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: miaapp-api
  minReplicas: 2
  maxReplicas: 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
```

Significa:

```text id="q2a064"
mantieni almeno 2 repliche
non superare 10 repliche
scala in base alla CPU media al 70%
```

---

# HPA: attenzione

HPA non è magia.

Funziona bene se:

```text id="daq73h"
hai metriche disponibili
requests CPU configurate
app stateless
startup abbastanza rapido
dipendenze esterne scalano anch’esse
```

Può non bastare se:

```text id="a1j1vv"
il collo di bottiglia è il database
il problema è una API esterna lenta
le richieste sono bloccate da lock
la coda cresce più velocemente della capacità
i Pod impiegano troppo ad avviarsi
```

Regola pratica:

```text id="khdokt"
Scalare l’API non risolve un database saturo.
```

# Rolling update

Il **rolling update** aggiorna gradualmente i Pod da una versione a un’altra.

Kubernetes documenta il rolling update come meccanismo che consente di aggiornare un Deployment con zero downtime, sostituendo progressivamente i vecchi Pod con nuovi Pod e aspettando che i nuovi siano avviati prima di rimuovere i vecchi. ([Kubernetes][6])

Esempio:

```bash id="kv7bj3"
kubectl set image deployment/miaapp-api \
  miaapp-api=registry.example.com/miaapp-api:1.1.0
```

Kubernetes passa da:

```text id="fzlrwd"
miaapp-api:1.0.0
```

a:

```text id="wwc0jr"
miaapp-api:1.1.0
```

gradualmente.

---

# Strategia rolling update

Nel Deployment puoi configurare:

```yaml id="jo6nqg"
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxUnavailable: 1
    maxSurge: 1
```

Significato:

```text id="p29n1y"
maxUnavailable: 1
  al massimo 1 Pod non disponibile durante update.

maxSurge: 1
  Kubernetes può creare 1 Pod in più temporaneamente.
```

Questo aiuta a mantenere disponibilità durante il deploy.

Se una release fallisce:

```bash id="hcfrwh"
kubectl rollout undo deployment/miaapp-api
```

Controllare stato:

```bash id="xt5pj8"
kubectl rollout status deployment/miaapp-api
```

Storico:

```bash id="awx9bz"
kubectl rollout history deployment/miaapp-api
```

Attenzione:

```text id="pz1s3m"
Il rollback dell’app può essere semplice.

```

# Blue-Green deployment

Nel **Blue-Green deployment** hai due ambienti/versioni:

```text id="nob4h9"
Blue
  versione attualmente attiva.

Green
  nuova versione pronta.
```

Schema:

```text id="cu0zdc"
Service
  → Blue Pods

Green Pods
  pronti ma non ricevono traffico

Switch:
Service
  → Green Pods
```

Esempio concettuale:

```text id="x14f33"
miaapp-api-blue
  image 1.0.0

miaapp-api-green
  image 1.1.0
```

Il Service decide a quale set di Pod inviare traffico tramite selector.

Vantaggi:

```text id="j3hl7y"
switch rapido
rollback rapido
test green prima dello switch
```

Svantaggi:

```text id="et5ydo"
costo doppio temporaneo
gestione database delicata
configurazione più complessa
```

---

# Canary deployment

Nel **Canary deployment** rilasci la nuova versione a una piccola percentuale di traffico.

Esempio:

```text id="bfn1zm"
95% traffico → versione 1.0.0
5% traffico  → versione 1.1.0
```

Poi osservi:

```text id="na8jvd"
error rate
latenza
metriche business
log
trace
feedback utenti
```

Se va bene, aumenti:

```text id="vbir51"
5%
25%
50%
100%
```

Canary richiede strumenti di routing più avanzati, spesso tramite:

```text id="nl00xg"
Ingress controller evoluto
service mesh
progressive delivery controller
Argo Rollouts
Flagger
gateway avanzato
```

Regola pratica:

```text id="s2zljn"
Canary senza osservabilità è pericoloso.
Non puoi aumentare traffico se non misuri effetti.
```

# Helm

**Helm** è il package manager per Kubernetes. Helm Charts permettono di definire, installare e aggiornare anche applicazioni Kubernetes complesse. ([helm.sh][7])

Un **Chart** è un pacchetto di risorse Kubernetes parametrizzabili.

La documentazione Helm spiega che un chart è una raccolta di file che descrive un set correlato di risorse Kubernetes e può essere impacchettato in archivi versionati. ([helm.sh][8])

---

# Perché usare Helm

Senza Helm potresti avere molti file YAML:

```text id="vzof23"
deployment.yaml
service.yaml
ingress.yaml
configmap.yaml
secret.yaml
hpa.yaml
```

Con Helm puoi creare template e valori per ambiente:

```text id="x6mebm"
values-dev.yaml
values-staging.yaml
values-prod.yaml
```

Vantaggi:

```text id="n4klvq"
riuso manifest
parametri per ambiente
versionamento release
install/upgrade/rollback
pacchetti condivisibili
meno copia-incolla YAML
```

---

# Struttura di un Helm Chart

```text id="h8z58j"
miaapp-chart/
  Chart.yaml
  values.yaml
  templates/
    deployment.yaml
    service.yaml
    ingress.yaml
    configmap.yaml
    hpa.yaml
```

`Chart.yaml`:

```yaml id="mpw60d"
apiVersion: v2
name: miaapp
description: Chart per MiaApp API
type: application
version: 0.1.0
appVersion: "1.0.0"
```

`values.yaml`:

```yaml id="eafnj3"
image:
  repository: registry.example.com/miaapp-api
  tag: "1.0.0"

replicaCount: 3

service:
  port: 80
  targetPort: 8080

resources:
  requests:
    cpu: "100m"
    memory: "128Mi"
  limits:
    cpu: "500m"
    memory: "512Mi"
```

Template Deployment:

```yaml id="ibfeq0"
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ .Release.Name }}-api
spec:
  replicas: {{ .Values.replicaCount }}
  selector:
    matchLabels:
      app: {{ .Release.Name }}-api
  template:
    metadata:
      labels:
        app: {{ .Release.Name }}-api
    spec:
      containers:
        - name: api
          image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
          ports:
            - containerPort: {{ .Values.service.targetPort }}
```

Deploy:

```bash id="nrwrpu"
helm install miaapp ./miaapp-chart
```

Upgrade:

```bash id="c8q4tb"
helm upgrade miaapp ./miaapp-chart \
  --set image.tag=1.1.0
```

Esempio shutdown graceful in worker:

```csharp id="pfh0xq"
protected override async Task ExecuteAsync(
    CancellationToken stoppingToken)

    {
        await DoWorkAsync(stoppingToken);
```

# Rolling update e migrazioni database

Una delle parti più delicate è il database.

Scenario problematico:

```text id="q8wydv"
Versione 1.0.0 usa colonna Nome.
Versione 1.1.0 rinomina Nome in NomeCompleto.
Durante rolling update convivono Pod 1.0.0 e 1.1.0.
```

Risultato:

```text id="szb3ca"
alcuni Pod leggono vecchio schema
altri nuovo schema
errori intermittenti
```

Strategia più sicura:

```text id="xuwk85"
1. Aggiungi nuova colonna senza rimuovere la vecchia.
2. Pubblica app compatibile con entrambe.
3. Migra/backfill dati.
4. Pubblica app che usa solo la nuova.
5. Solo dopo rimuovi vecchia colonna.
```

Regola pratica:

```text id="o1z0fc"
In Kubernetes possono convivere temporaneamente versioni diverse.
Le migrazioni devono essere compatibili con questa realtà.
```

# Blue-Green e Canary con database

Blue-Green e Canary sono potenti, ma il database resta critico.

Domande:

```text id="yg33r9"
La nuova versione è compatibile con lo schema attuale?
La vecchia versione può funzionare dopo la migrazione?
Il rollback è possibile?
I dati scritti dalla nuova versione sono leggibili dalla vecchia?
```

Se la risposta è no, il rollback può essere solo apparente.

# Errori comuni

## Usare Kubernetes troppo presto

Kubernetes non sostituisce architettura, test, osservabilità e CI/CD.

---

## Pod senza readiness probe

Il Service può mandare traffico a un Pod non ancora pronto.

---

## Nessun resource limit

Rischi:

```text id="mxkylc"
un Pod consuma troppe risorse
impatti sugli altri workload
scheduling inefficiente
```

---

## Salvare file nel container

Quando il Pod viene ricreato, perdi i dati.

---

## Configurazione hard-coded nell’image

L’immagine deve essere uguale tra ambienti.
La configurazione deve arrivare da ConfigMap, Secret o sistemi esterni.

---

## Secret Kubernetes trattati come soluzione definitiva

Vanno protetti con RBAC, encryption at rest e, spesso, integrati con vault esterni.

---

## HPA senza requests

HPA basato su CPU richiede requests sensati.

---

## Rolling update senza compatibilità database

Errore molto comune e molto pericoloso.

---

## Nessun monitoraggio del cluster

Senza osservabilità, Kubernetes rende solo più difficile capire dove sia il problema.

# Linee guida pratiche

```text id="jbgz4h"
Usa Kubernetes
  quando hai bisogno reale di orchestrazione, scaling e self-healing.

Non usarlo come prerequisito per imparare .NET
  prima capisci app, container, CI/CD e osservabilità.

Usa Deployment
  per API e worker stateless.

Usa Service
  per dare indirizzi stabili ai Pod.

Usa Ingress
  per esporre traffico HTTP/HTTPS verso l’esterno.

Usa ConfigMap
  per configurazioni non sensibili.

Usa Secret o vault
  per dati sensibili.

Imposta requests e limits
  per CPU e memoria.

Configura liveness e readiness probe
  con endpoint ASP.NET Core dedicati.

Usa HPA
  solo con metriche affidabili e app stateless.

Preferisci rolling update
  per aggiornamenti ordinari.

Valuta blue-green o canary
  per release ad alto rischio.

Usa Helm
  per pacchettizzare manifest Kubernetes e gestire ambienti.

Monitora cluster e applicazione
  con metriche, log, trace e alert.

Progetta migrazioni database compatibili
  perché durante rollout possono convivere versioni diverse.
```

# Riepilogo

In questo capitolo abbiamo introdotto Kubernetes come strumento di orchestrazione avanzato per applicazioni containerizzate.

Abbiamo visto:

```text id="mw3zhl"
perché Kubernetes

Service
Ingress
ConfigMap
Secret

scaling orizzontale
HPA
limiti risorse

```

La distinzione principale è questa:

```text id="ckjo76"
Pod
  esegue uno o più container.

Deployment
  mantiene il numero desiderato di Pod e gestisce rollout.

Service
  fornisce un indirizzo stabile e bilanciato verso i Pod.

Ingress
  espone HTTP/HTTPS verso l’esterno.

ConfigMap
  contiene configurazione non sensibile.

Secret
  contiene dati sensibili.

Helm
  pacchettizza e parametrizza manifest Kubernetes.
```

La regola principale è questa:

```text id="ae7ww0"
Kubernetes non rende automaticamente un’applicazione scalabile,
sicura o affidabile.
Fornisce strumenti per orchestrarla;
sta all’architettura dell’applicazione usarli correttamente.
```

Per applicazioni .NET, Kubernetes diventa davvero utile quando il progetto ha già container puliti, configurazione esterna, health check, logging su console, osservabilità, CI/CD e una strategia chiara di deploy e rollback.

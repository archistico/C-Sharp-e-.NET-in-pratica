# 11.02 — Progettare i confini dei servizi

Nel capitolo precedente abbiamo chiarito una cosa fondamentale: i microservizi non sono automaticamente “meglio” di un monolite o di un monolite modulare.

Ora arriviamo al problema più difficile:

```text id="zjq65b"
Dove taglio il sistema?
```

Questa è la domanda centrale dei microservizi.

Creare tanti progetti, tante API o tanti container è facile. Progettare confini corretti è difficile.

Il punto centrale è questo:

```text id="nzgta8"
Un buon confine di servizio non nasce dalla tabella del database,
ma dal dominio, dalle responsabilità, dai dati posseduti
e dal grado di autonomia richiesto.
```

Se tagli male, non ottieni microservizi: ottieni dipendenze distribuite, chiamate fragili, database condivisi, deploy coordinati e un sistema più difficile del monolite di partenza.

---

# Il problema dei confini

In un monolite mal organizzato, le dipendenze sono dentro lo stesso processo.

Esempio:

```text id="kc2fsu"
OrdiniService chiama ClientiRepository
ClientiService legge tabelle Ordini
ReportService interroga tutto
DocumentoService aggiorna dati Cliente
```

In un microservizio mal progettato, lo stesso accoppiamento rimane, ma attraversa la rete:

```text id="yp4y4d"
Ordini API chiama Clienti API
Clienti API chiama Documenti API
Documenti API chiama Report API
Report API legge database Ordini
```

Il problema non è risolto. È solo diventato remoto.

Regola pratica:

```text id="buwwlq"
Prima devi ridurre l’accoppiamento concettuale.
Solo dopo ha senso separare fisicamente i servizi.
```

---

# Bounded context

Il concetto più importante per progettare confini è il **Bounded Context**, che viene da Domain-Driven Design.

Un bounded context è un confine entro cui un modello di dominio ha un significato preciso e coerente.

Martin Fowler descrive il Bounded Context come un pattern centrale del DDD strategico: serve a gestire modelli grandi dividendo il dominio in contesti separati e rendendo esplicite le relazioni tra loro. ([martinfowler.com][1])

Esempio: la parola “Cliente” può significare cose diverse in contesti diversi.

```text id="nwiz29"
Contesto Marketing
  Cliente = persona da profilare, segmentare, contattare.

Contesto Fatturazione
  Cliente = soggetto fiscale a cui emettere documenti.

Contesto Accessi
  Cliente = persona fisica identificata all’ingresso.

Contesto Supporto
  Cliente = soggetto che apre ticket.
```

Usare una sola classe `Cliente` per tutti questi significati può creare confusione.

---

# Stesso nome, modelli diversi

Esempio:

```csharp id="p824cj"
public class Cliente
{
    public int Id { get; set; }
    public string Nome { get; set; } = "";
    public string Email { get; set; } = "";
    public string CodiceFiscale { get; set; } = "";
    public string SegmentoMarketing { get; set; } = "";
    public decimal FatturatoAnnuo { get; set; }
    public DateTime? UltimoAccesso { get; set; }
    public bool Bloccato { get; set; }
}
```

Questa classe sembra comoda, ma rischia di diventare un “modello universale” usato da tutti.

Problemi:

```text id="zbid8d"
contiene campi che non servono a tutti
mescola regole di contesti diversi
ogni modifica impatta molti moduli
diventa difficile capire chi possiede cosa
```

Meglio riconoscere modelli separati:

```text id="pzczuk"
Marketing.ClienteProfilato
Fatturazione.SoggettoFiscale
Accessi.Visitatore
Supporto.Richiedente
```

Non è duplicazione sbagliata.
È separazione semantica.

---

# Bounded context non significa automaticamente microservizio

Un bounded context può essere implementato come:

```text id="eub2yz"
modulo in un monolite modulare
progetto separato nella stessa solution
servizio indipendente
microservizio
bounded context distribuito su più componenti
```

Non c’è obbligo di rapporto 1:1 tra bounded context e microservizio.

La guida Microsoft sui microservizi .NET collega il concetto di bounded context alla sovranità dei dati e osserva che ogni bounded context o sottosistema autonomo dovrebbe possedere il proprio domain model, cioè dati, logica e comportamento. ([Microsoft Learn][2])

Regola pratica:

```text id="o31bg6"
Il bounded context è un confine di modello.
Il microservizio è un confine di deploy.
```

A volte coincidono.
A volte no.

---

# Autonomia del servizio

Un microservizio deve avere autonomia reale.

Autonomia significa:

```text id="rxf7b0"
può essere sviluppato indipendentemente
può essere testato indipendentemente
può essere distribuito indipendentemente
possiede i propri dati
espone contratti chiari
non richiede modifiche coordinate continue con altri servizi
può continuare a funzionare, almeno parzialmente, se altri servizi falliscono
```

Se un servizio non può fare nulla senza chiamare altri tre servizi, probabilmente il confine è troppo piccolo o tagliato male.

---

# Segnali di bassa autonomia

```text id="x3x7tg"
ogni use case richiede molte chiamate sincrone tra servizi
ogni deploy deve essere coordinato con altri servizi
i servizi condividono lo stesso database
ogni servizio espone CRUD interno agli altri
le modifiche a una tabella rompono più servizi
i test end-to-end sono l’unico modo per validare il sistema
```

Esempio sospetto:

```text id="au4fqq"
Ordini Service
  per creare un ordine deve chiamare:
    Clienti Service
    Listino Service
    Sconti Service
    Disponibilità Service
    Pagamenti Service
```

Una parte di queste chiamate può essere inevitabile, ma se ogni operazione critica dipende da una catena lunga, il servizio non è davvero autonomo.

---

# Database-per-service

Uno dei principi più importanti nei microservizi è: ogni servizio deve possedere i propri dati.

La guida classica di Fowler sui microservizi nota che i microservizi preferiscono lasciare che ogni servizio gestisca il proprio database, anche usando tecnologie diverse quando opportuno. ([martinfowler.com][3])

Schema corretto:

```text id="rvwkg2"
Clienti Service
  DB Clienti

Ordini Service
  DB Ordini

Pagamenti Service
  DB Pagamenti

Report Service
  DB Report / Read Model
```

Il servizio è proprietario del proprio schema dati.

---

# Database condiviso: perché è rischioso

Schema problematico:

```text id="eh7fef"
Clienti Service
Ordini Service
Pagamenti Service
Report Service
        │
        ▼
  Database condiviso
```

Problemi:

```text id="ffbdq4"
lo schema diventa un contratto nascosto
i servizi non sono indipendenti
una modifica al database richiede coordinamento
è facile violare le regole di dominio di un altro servizio
il deploy indipendente diventa illusorio
```

Esempio:

```text id="ia9r62"
Ordini Service aggiorna direttamente la tabella Clienti
per modificare il campo UltimoOrdine.
```

Questo rompe l’ownership del servizio Clienti.

Meglio:

```text id="p5gv4d"
Ordini Service pubblica OrdineCreato
Clienti Service aggiorna la propria proiezione interna, se serve
Report Service aggiorna il proprio read model
```

---

# Database-per-service non significa per forza server diversi

Database-per-service può voler dire:

```text id="rovmpn"
database fisico separato
schema separato nello stesso server
istanza separata
storage diverso
read model dedicato
```

La cosa importante non è sempre l’infrastruttura fisica.

La cosa importante è l’ownership:

```text id="h6xkg1"
Solo il servizio proprietario modifica direttamente i propri dati.
Gli altri passano da API, messaggi o proiezioni consentite.
```

---

# Coerenza eventuale

Quando ogni servizio possiede i propri dati, non puoi più fare sempre una transazione ACID unica su tutto.

Esempio:

```text id="dj9stl"
Ordini Service crea ordine.
Pagamenti Service autorizza pagamento.
Magazzino Service riserva stock.
Spedizioni Service prepara consegna.
```

In un monolite con un database unico potresti tentare una transazione unica.

In un sistema distribuito, devi accettare spesso la **coerenza eventuale**.

```text id="xcn9zh"
Per un breve periodo, servizi diversi possono vedere stati diversi.
Dopo la propagazione degli eventi, il sistema converge.
```

Esempio:

```text id="omc42t"
OrdineCreato
  subito: Ordini vede ordine creato
  dopo poco: Magazzino vede richiesta riserva
  dopo poco: Pagamenti vede richiesta pagamento
  dopo poco: Report aggiorna statistiche
```

---

# Coerenza eventuale non significa caos

Coerenza eventuale non vuol dire:

```text id="czq7j2"
ognuno fa quello che vuole
i dati sono sempre sbagliati
gli errori si ignorano
```

Significa progettare esplicitamente:

```text id="fy4nrk"
eventi
stati intermedi
compensazioni
idempotenza
retry
outbox
saga
monitoraggio
tempi di convergenza accettabili
```

Esempio stato ordine:

```text id="hpj3ux"
Bozza
In attesa pagamento
Pagamento autorizzato
In preparazione
Spedito
Annullato
Pagamento fallito
```

Gli stati intermedi sono parte del dominio, non un dettaglio tecnico.

# CAP theorem

Il **CAP theorem** riguarda i sistemi distribuiti e il trade-off tra:

```text id="gwcwts"
Consistency
  ogni lettura vede il dato più recente o riceve errore.

Availability
  ogni richiesta riceve una risposta, anche se non necessariamente aggiornata.

Partition tolerance
  il sistema continua a operare nonostante problemi di comunicazione tra nodi.
```

Nel caso di partizione di rete, un sistema distribuito deve scegliere se privilegiare consistenza o disponibilità. La formulazione classica del CAP theorem dice che un distributed data store può garantire al massimo due tra consistency, availability e partition tolerance. ([Wikipedia][4])

---

# CAP nella pratica

In un sistema distribuito reale, le partizioni di rete non sono teoriche.

Possono essere:

```text id="oq2tfi"
timeout
servizio non raggiungibile
messaggi ritardati
packet loss
DNS temporaneamente non disponibile
broker lento
database remoto irraggiungibile
```

Esempio:

```text id="u152e2"
Ordini Service non riesce a contattare Pagamenti Service.
```

Scelte possibili:

```text id="u6fd2i"
rifiutare l’ordine
  maggiore consistenza, minore disponibilità

accettare ordine in stato In attesa pagamento
  maggiore disponibilità, consistenza eventuale

mettere richiesta in coda
  disponibilità del front-end, elaborazione asincrona
```

Non esiste una risposta valida sempre.

Dipende dal dominio.

---

# Esempio: prenotazione vs report

Per un report:

```text id="ctlt2u"
accettare dati vecchi di 2 minuti può essere ok
```

Per un pagamento:

```text id="y96339"
accettare doppio addebito non è ok
```

Per una prenotazione:

```text id="rzm28f"
vendere due volte lo stesso posto può essere inaccettabile
```

Per una dashboard:

```text id="jvn4ys"
mostrare conteggi leggermente ritardati è spesso accettabile
```

Regola pratica:

```text id="wrcp1i"
Non decidere coerenza e disponibilità in astratto.
Decidile per caso d’uso.
```

# Contratti API

Un servizio comunica con il mondo esterno tramite contratti.

Un contratto può essere:

```text id="cfl2sj"
endpoint REST
schema JSON
contratto gRPC/protobuf
messaggio su broker
evento di integrazione
OpenAPI document
SDK client
```

Il contratto è più importante dell’implementazione interna.

Esempio REST:

```http id="fykpru"
GET /api/clienti/123
```

Risposta:

```json id="tl3g2i"
{
  "id": 123,
  "nome": "Mario Rossi",
  "email": "mario@example.com"
}
```

Se altri servizi dipendono da questo formato, il formato è un contratto.

---

# Contratti stabili

Un contratto dovrebbe essere:

```text id="wfrtzu"
esplicito
documentato
versionato
retrocompatibile quando possibile
testabile
non troppo legato al database interno
```

Errore comune:

```text id="b68c47"
esporre direttamente entity EF Core come risposta API
```

Esempio pericoloso:

```csharp id="mr4lcf"
app.MapGet("/api/clienti/{id:int}", async (
    int id,
    AppDbContext dbContext) =>
{
    return await dbContext.Clienti.FindAsync(id);
});
```

Meglio DTO esplicito:

```csharp id="qvnt6m"
public record ClienteResponse(
    int Id,
    string Nome,
    string? Email);
```

Endpoint:

```csharp id="rl89ps"
app.MapGet("/api/clienti/{id:int}", async (
    int id,
    AppDbContext dbContext,
    CancellationToken cancellationToken) =>
{
    ClienteResponse? cliente = await dbContext.Clienti
        .AsNoTracking()
        .Where(c => c.Id == id)
        .Select(c => new ClienteResponse(
            c.Id,
            c.Nome,
            c.Email))
        .FirstOrDefaultAsync(cancellationToken);

    return cliente is null
        ? Results.NotFound()
        : Results.Ok(cliente);
});
```

# Versionamento

Prima o poi un contratto cambia.

Esempio versione 1:

```json id="qk123d"
{
  "id": 123,
  "nome": "Mario Rossi"
}
```

Versione 2:

```json id="ry54gs"
{
  "id": 123,
  "nomeCompleto": "Mario Rossi",
  "email": "mario@example.com"
}
```

Se rimuovi o rinomini campi, puoi rompere i consumer.

---

# Strategie di versionamento API

Possibili strategie:

```text id="saw8ie"
URL versioning
  /api/v1/clienti
  /api/v2/clienti

Header versioning
  api-version: 1.0

Media type versioning
  Accept: application/vnd.miaapp.clienti.v1+json

Versionamento per endpoint nuovo
  /api/clienti/search
  /api/clienti/advanced-search
```

Per molte API aziendali, URL versioning è semplice e comprensibile.

Esempio:

```csharp id="xrx2f0"
app.MapGet("/api/v1/clienti/{id:int}", ...);

app.MapGet("/api/v2/clienti/{id:int}", ...);
```

Regola pratica:

```text id="h36kpa"
Prima prova a fare modifiche retrocompatibili.
Versiona quando una modifica rompe i consumer.
```

# Modifiche compatibili e incompatibili

Di solito sono compatibili:

```text id="gmkl8q"
aggiungere un campo opzionale nella response
aggiungere un endpoint
aggiungere un valore opzionale
rendere più permissiva una validazione
```

Di solito sono breaking change:

```text id="w2job4"
rimuovere un campo
rinominare un campo
cambiare tipo di un campo
cambiare significato di un campo
rendere obbligatorio un campo prima opzionale
cambiare codice HTTP atteso
cambiare semantica di un endpoint
```

Esempio:

```text id="ih4l4q"
email: string
```

diventa:

```text id="y6z5rz"
email: object
```

Probabile breaking change.

# Granularità

La granularità indica quanto grande o piccolo è un servizio.

Servizio troppo grande:

```text id="woz9n0"
fa troppe cose
ha troppi modelli
ha troppi team che lo modificano
diventa un mini-monolite
```

Servizio troppo piccolo:

```text id="dv24zv"
non è autonomo
deve chiamare altri servizi continuamente
ha poco valore business
aumenta latenza e complessità
```

Esempio troppo piccolo:

```text id="g2khgj"
EmailClienteService
TelefonoClienteService
IndirizzoClienteService
```

Esempio più ragionevole:

```text id="mkhd35"
Clienti Service
```

---

# Criteri per scegliere la granularità

Chiediti:

```text id="odbgy9"
Il servizio rappresenta una capacità di business?
Possiede dati e regole proprie?
Può essere deployato indipendentemente?
Può essere testato in isolamento?
Ha un team o ownership chiara?
Ha bisogno di scalare diversamente?
Ha un ciclo di vita diverso dagli altri?
Riduce accoppiamento o lo aumenta?
```

Regola pratica:

```text id="kt6q1a"
Un servizio deve essere abbastanza grande da essere autonomo
e abbastanza piccolo da avere una responsabilità chiara.
```

# Comunicazione sincrona

La comunicazione sincrona avviene quando un servizio chiama un altro e attende risposta.

Esempi:

```text id="fc1qfe"
HTTP REST
gRPC
query diretta a un endpoint
```

Schema:

```text id="ssf79b"
Ordini Service
  ↓ richiesta
Clienti Service
  ↓ risposta
Ordini Service continua
```

Vantaggi:

```text id="qc8v3i"
semplice da capire
risposta immediata
adatta a query
adatta a interazioni utente dirette
debug più intuitivo
```

Svantaggi:

```text id="u6rdqy"
accoppiamento temporale
latenza
fallimenti di rete
timeout
cascate di errori
riduzione disponibilità complessiva
```

---

# Quando usare comunicazione sincrona

È adatta quando:

```text id="vqis5d"
serve una risposta immediata
il chiamante non può procedere senza risposta
la chiamata è veloce e affidabile
il servizio chiamato è stabile
il caso d’uso è una lettura
```

Esempio:

```text id="n599qv"
UI chiama API Clienti per mostrare dettaglio cliente
```

Oppure:

```text id="xu7du7"
API Gateway chiama servizio Catalogo per leggere prodotti
```

---

# Rischio: catene sincrone

Catena fragile:

```text id="hzfihh"
A → B → C → D → E
```

Se ogni servizio ha disponibilità 99.9%, la disponibilità della catena peggiora.

In più:

```text id="py96nj"
latenze si sommano
timeout si propagano
errori diventano difficili da isolare
```

Regola pratica:

```text id="wu179n"
Evita chiamate sincrone profonde tra servizi.
```

La guida Microsoft sui microservizi .NET suggerisce di minimizzare la comunicazione tra microservizi e, quando serve integrazione, preferire la propagazione asincrona dei dati invece di dipendere da altri servizi interni durante la richiesta HTTP iniziale. ([Microsoft Learn][5])

---

# Comunicazione asincrona

La comunicazione asincrona avviene quando un servizio pubblica un messaggio o evento e non attende necessariamente che il destinatario completi il lavoro.

Esempi:

```text id="efcr21"
message queue
event bus
publish/subscribe
integration event
stream
```

Schema:

```text id="ctrapn"
Ordini Service
  pubblica OrdineCreato
        ↓
     Message Broker
        ↓
  Magazzino Service
  Pagamenti Service
  Report Service
```

Vantaggi:

```text id="jqa12b"
minore accoppiamento temporale
migliore resilienza
picchi assorbiti da code
servizi destinatari indipendenti
adatto a eventi di dominio e integrazione
```

Svantaggi:

```text id="ivpqki"
coerenza eventuale
debug più difficile
messaggi duplicati
ordine dei messaggi da gestire
idempotenza necessaria
monitoraggio code necessario
```

---

# Quando usare comunicazione asincrona

È adatta quando:

```text id="kdsuel"
il chiamante non ha bisogno di risposta immediata
più servizi devono reagire allo stesso evento
serve resilienza ai picchi
il processo è lungo
accetti coerenza eventuale
vuoi ridurre accoppiamento
```

Esempio:

```text id="ol05bx"
OrdineConfermato
  Magazzino aggiorna disponibilità
  Report aggiorna statistiche
  Email invia conferma
```

L’utente non deve aspettare che tutte queste operazioni finiscano.

# REST

**REST** è lo stile più comune per API HTTP.

Esempi:

```http id="s1gqjd"
GET /api/clienti/123
POST /api/ordini
PUT /api/clienti/123/email
DELETE /api/documenti/456
```

Vantaggi:

```text id="or6lxe"
semplice
ampiamente supportato
facile da testare
ottimo per API pubbliche e web
compatibile con JSON/OpenAPI
facile da consumare da browser e client diversi
```

Svantaggi:

```text id="xsegdg"
può essere verboso
contratti meno rigidi rispetto a protobuf
performance inferiori a gRPC in scenari ad alta efficienza
versionamento da progettare bene
```

Usa REST quando:

```text id="xd4qx4"
devi esporre API a client diversi
hai web/mobile/frontend
vuoi semplicità
vuoi documentazione OpenAPI
hai operazioni CRUD o resource-oriented
```

# gRPC

**gRPC** è un framework RPC moderno basato su HTTP/2 e Protocol Buffers.

Esempio contratto `.proto`:

```proto id="k2qzpo"
syntax = "proto3";

service ClientiService {
  rpc GetCliente (GetClienteRequest)
      returns (ClienteReply);
}

message GetClienteRequest {
  int32 id = 1;
}

message ClienteReply {
  int32 id = 1;
  string nome = 2;
  string email = 3;
}
```

Vantaggi:

```text id="ga8c2p"
contratti fortemente tipizzati
performance elevate
generazione client/server
streaming
adatto a comunicazione service-to-service
```

Svantaggi:

```text id="b40p37"
meno immediato da testare manualmente
meno naturale per browser puri senza supporti specifici
versionamento protobuf da conoscere
debug meno semplice di JSON/REST
```

Usa gRPC quando:

```text id="k3jn1x"
comunicazione interna tra servizi
alte prestazioni
contratti forti
streaming
ambienti controllati
```

# Command message vs Event message

Distinzione importante.

## Command message

Chiede a qualcuno di fare qualcosa.

```text id="i8mwml"
RiservaMagazzino
InviaEmailConferma
GeneraFattura
```

Ha un destinatario logico.

## Event message

Comunica che qualcosa è accaduto.

```text id="muvpgs"
OrdineConfermato
ClienteRegistrato
PagamentoAutorizzato
```

Non comanda direttamente cosa fare.

Regola pratica:

```text id="ldpkja"
Command:
  fai questa cosa.

Event:
  questa cosa è successa.
```

---

# Messaging: problemi da progettare

Con il messaging devi gestire:

```text id="qxhmiu"
duplicati
ordine dei messaggi
retry
dead-letter queue
idempotenza
schema dei messaggi
versionamento eventi
monitoraggio code
poison messages
backpressure
outbox
```

Esempio consumer idempotente:

```csharp id="q4taai"
public async Task HandleAsync(
    OrdineConfermatoIntegrationEvent evento,
    CancellationToken cancellationToken)
{
    bool giaProcessato = await dbContext.ProcessedMessages
        .AnyAsync(m => m.MessageId == evento.MessageId, cancellationToken);

    if (giaProcessato)
    {
        return;
    }

    // esegui logica

    dbContext.ProcessedMessages.Add(new ProcessedMessage
    {
        MessageId = evento.MessageId,
        ProcessedAt = DateTimeOffset.UtcNow
    });

    await dbContext.SaveChangesAsync(cancellationToken);
}
```

# Sincrono vs asincrono: confronto

```text id="fssj9g"
Sincrono
  semplice
  risposta immediata
  accoppiamento temporale
  più fragile se usato in catene lunghe

Asincrono
  più resiliente
  disaccoppia
  adatto a eventi
  introduce coerenza eventuale e complessità operativa
```

Scelta pratica:

```text id="is5g73"
Query utente immediata
  spesso REST/gRPC sincrono.

Evento di business
  spesso messaging asincrono.

Processo lungo
  spesso comando + stato + notifiche/eventi.

Aggiornamento read model
  spesso evento asincrono.
```

# Progettare un servizio: domande guida

Per ogni candidato servizio chiediti:

```text id="obioiv"
Quale capacità di business rappresenta?

Quali dati possiede?

Quali regole protegge?

Quali operazioni espone?

Quali eventi pubblica?

Di quali eventi ha bisogno?

Quali chiamate sincrone sono davvero necessarie?

Può funzionare se un altro servizio non risponde?

Che cosa succede se riceve lo stesso messaggio due volte?

Qual è il suo contratto pubblico?

Come versiono il contratto?

Chi è il team owner?

Come faccio logging, metriche e tracing?
```

Se non sai rispondere, il servizio è prematuro.

# Esempio di flusso asincrono

```text id="o923l4"
1. Ordini Service crea ordine in stato InAttesaPagamento.
2. Ordini Service pubblica OrdineCreato.
3. Pagamenti Service riceve OrdineCreato.
4. Pagamenti Service tenta autorizzazione.
5. Pagamenti Service pubblica PagamentoAutorizzato o PagamentoFallito.
6. Ordini Service aggiorna stato ordine.
7. Notifiche Service invia email.
8. Report Service aggiorna read model.
```

Questo flusso evita una transazione distribuita unica, ma richiede:

```text id="s42nyp"
idempotenza
outbox
retry
gestione stati intermedi
monitoraggio
```

# Antipattern nei confini

## Servizio per tabella

Sbagliato:

```text id="dibfyx"
ClientiTableService
OrdiniTableService
RigheOrdineTableService
PagamentiTableService
```

I servizi non dovrebbero nascere dalle tabelle, ma dalle capacità di business.

---

## CRUD remoto

Sospetto:

```text id="c7h4xm"
un servizio espone solo Get/Add/Update/Delete
e altri servizi lo usano come repository remoto
```

Questo crea accoppiamento forte.

Meglio esporre operazioni di dominio:

```text id="p52lpc"
ConfermaOrdine
AnnullaOrdine
AutorizzaPagamento
RegistraCliente
```

---

## Servizio troppo piccolo

```text id="w8ag7q"
EmailService remoto chiamato sincronicamente a ogni operazione
```

Se l’invio email non è critico per completare la richiesta utente, meglio evento asincrono.

---

## Contratti basati su entity interne

Se esponi direttamente le entity, ogni modifica interna diventa breaking change.

---

## Eventi troppo generici

Sbagliato:

```text id="dxy1y1"
EntityUpdated
DataChanged
ClienteModificato
```

se non dicono cosa è successo.

Meglio:

```text id="mcy0r4"
ClienteRegistrato
EmailClienteAggiornata
ClienteDisattivato
```

---

## Eventi troppo tecnici

Sbagliato:

```text id="ru5t4p"

```

Meglio evento di business:

# Linee guida pratiche

```text id="l2m6dk"
Parti dal dominio
  non dalle tabelle.

Usa bounded context
  per capire dove un modello ha significato coerente.

Non trasformare automaticamente ogni bounded context in microservizio
  può restare modulo di un monolite modulare.

Progetta servizi autonomi
  con dati, regole e responsabilità proprie.

Evita database condivisi
  sono accoppiamento nascosto.

Accetta coerenza eventuale
  quando attraversi confini distribuiti.

Usa comunicazione sincrona
  per letture immediate e operazioni semplici.

Usa comunicazione asincrona
  per eventi, processi lunghi e integrazione resiliente.

Progetta contratti espliciti
  DTO, OpenAPI, protobuf o message schema.

Versiona i contratti
  e privilegia modifiche retrocompatibili.

Scegli granularità pragmatica
  abbastanza grande per autonomia,
  abbastanza piccola per responsabilità chiara.

Non creare servizi CRUD remoti
  crea servizi orientati a capacità di business.

Progetta idempotenza
  soprattutto con messaging e retry.

Osserva il sistema
  log, metriche, trace e correlation ID sono obbligatori.
```

# Riepilogo

In questo capitolo abbiamo affrontato il problema più importante dei microservizi: progettare i confini.

Abbiamo visto:

```text id="uexp0x"
bounded context
autonomia del servizio
database-per-service
coerenza eventuale
CAP theorem
contratti API
versionamento
granularità
comunicazione sincrona
comunicazione asincrona
REST
gRPC
messaging
```

La distinzione più importante è questa:

```text id="b3fyod"
Bounded context
  confine semantico del modello.

Microservizio
  confine di deploy, ownership e runtime.

Database-per-service
  confine di ownership dei dati.

Contratto API o messaggio
  confine di comunicazione.
```

La regola principale è questa:

```text id="kafxz7"
Un buon microservizio non è piccolo perché ha poche classi.
È ben tagliato perché possiede una responsabilità di business,
un modello coerente, dati propri, contratti chiari
e un livello reale di autonomia.
```

Progettare i confini è difficile perché richiede dominio, architettura, organizzazione e operatività. Se il confine è sbagliato, REST, gRPC, Docker e Kubernetes non risolvono il problema: lo rendono solo più costoso da diagnosticare.

# 11.01 — Monolite, monolite modulare e microservizi

Dopo aver parlato di architettura, pattern, sicurezza, container, CI/CD e osservabilità, possiamo affrontare i microservizi nel modo corretto: non come moda, ma come scelta architetturale.

Il punto centrale è questo:

```text id="oc0rf1"
I microservizi non sono automaticamente migliori di un monolite.
Sono una scelta che risolve alcuni problemi,
ma ne introduce molti altri.
```

Un sistema distribuito può dare scalabilità, autonomia e separazione, ma porta anche complessità operativa, coerenza dei dati più difficile, deploy più articolati, osservabilità obbligatoria e costi maggiori.

Prima di scegliere i microservizi, bisogna capire bene tre modelli:

```text id="od73ra"
monolite
monolite modulare
microservizi
```

---

# Monolite

Un **monolite** è un’applicazione distribuita come una singola unità.

Esempio:

```text id="z05p6z"
MiaApp.Web
  Controllers
  Services
  Repositories
  DbContext
  Business logic
  UI
```

Il deploy produce una sola applicazione:

```text id="ude6rv"
MiaApp.dll
```

oppure un solo sito IIS, un solo container o un solo servizio.

Schema:

```text id="jkl8cn"
┌─────────────────────────────┐
│        Applicazione          │
│                             │
│  UI / API                   │
│  Application services       │
│  Domain logic               │
│  Data access                │
│                             │
└──────────────┬──────────────┘
               │
               ▼
        ┌─────────────┐
        │  Database   │
        └─────────────┘
```

---

# Monolite non significa codice disordinato

Un errore comune è pensare:

```text id="hnlbie"
monolite = brutto
microservizi = bello
```

Non è così.

Un monolite può essere:

```text id="olxftm"
ben organizzato
testabile
modulare
facile da distribuire
economico da gestire
perfettamente adatto al dominio
```

Oppure può essere:

```text id="kdrc52"
caotico
accoppiato
difficile da modificare
pieno di dipendenze incrociate
fragile
```

Il problema non è il monolite in sé.

Il problema è il **monolite disordinato**.

---

# Pro del monolite

```text id="ugj56f"
deploy semplice
debug più diretto
transazioni locali più semplici
meno complessità infrastrutturale
meno problemi di rete
meno duplicazione
meno necessità di osservabilità avanzata
sviluppo iniziale più rapido
adatto a team piccoli
adatto a domini non ancora stabilizzati
```

Esempio:

```text id="l3n36b"
Una singola app ASP.NET Core con EF Core e SQL Server
è spesso più semplice da sviluppare, testare e rilasciare
rispetto a cinque servizi separati.
```

---

# Contro del monolite

```text id="z3qtob"
tutto viene rilasciato insieme
scalabilità meno granulare
rischio di accoppiamento interno
build e test possono diventare lenti
il codice può diventare difficile da navigare
un errore può impattare tutta l’app
team diversi possono pestarsi i piedi
```

Il monolite diventa problematico soprattutto quando:

```text id="n4nkh6"
il dominio cresce molto
i moduli hanno cicli di vita diversi
team diversi lavorano su aree indipendenti
alcune parti richiedono scalabilità molto diversa
il deploy unico rallenta il rilascio
```

---

# Monolite modulare

Il **monolite modulare** è una via intermedia molto importante.

È ancora una singola applicazione distribuita come unità unica, ma internamente è organizzata in moduli con confini chiari.

Schema:

```text id="vjt6md"
┌──────────────────────────────────────────┐
│              Monolite modulare           │
│                                          │
│  ┌────────────┐ ┌────────────┐           │
│  │ Clienti    │ │ Ordini     │           │
│  │ Module     │ │ Module     │           │
│  └────────────┘ └────────────┘           │
│                                          │
│  ┌────────────┐ ┌────────────┐           │
│  │ Documenti  │ │ Report     │           │
│  │ Module     │ │ Module     │           │
│  └────────────┘ └────────────┘           │
│                                          │
└──────────────────────┬───────────────────┘
                       │
                       ▼
                ┌─────────────┐
                │  Database   │
                └─────────────┘
```

Il deploy resta semplice:

```text id="dzudll"
una sola applicazione
un solo processo
un solo rilascio
```

Ma il codice è organizzato meglio.

---

# Esempio di struttura modulare

```text id="lfo3tz"
src
  MiaApp.Web
    Program.cs

  MiaApp.Modules.Clienti
    ClientiModule.cs
    Features
      CreaCliente
      CercaClienti
    Domain
      Cliente.cs
    Infrastructure
      ClientiDbContext.cs

  MiaApp.Modules.Ordini
    OrdiniModule.cs
    Features
      CreaOrdine
      ConfermaOrdine
    Domain
      Ordine.cs
    Infrastructure
      OrdiniDbContext.cs

  MiaApp.SharedKernel
    Result.cs
    IClock.cs
    DomainEvent.cs
```

Ogni modulo contiene:

```text id="wfwkfz"
feature
domain model
use case
validazioni
persistenza interna
endpoint
eventuali eventi interni
```

---

# Confini nel monolite modulare

Un monolite modulare funziona solo se i confini sono rispettati.

Esempio corretto:

```text id="k9gk7x"
Modulo Ordini
  non modifica direttamente le tabelle interne del modulo Clienti.

Modulo Clienti
  espone contratti o eventi interni controllati.

Modulo Report
  può leggere viste/proiezioni dedicate,
  ma non deve diventare il proprietario delle regole degli altri moduli.
```

Esempio sbagliato:

```text id="zs3ohk"
OrdiniService accede direttamente a ClientiDbContext,
modifica Cliente,
legge dettagli interni,
chiama classi private del modulo Clienti.
```

In quel caso hai cartelle modulari, ma non un’architettura modulare.

---

# Monolite modulare: pro

```text id="yem7nr"
deploy ancora semplice
confini più chiari
meno accoppiamento
più facile evolvere verso microservizi
buon equilibrio tra semplicità e ordine
adatto a team piccoli o medi
testabilità migliore
modularità senza complessità distribuita
```

È spesso la scelta migliore per molte applicazioni aziendali.

---

# Monolite modulare: contro

```text id="r1xkw9"
richiede disciplina architetturale
i confini non sono protetti dalla rete
il database può diventare ancora condiviso male
il deploy resta unico
la scalabilità resta prevalentemente unitaria
serve controllo sulle dipendenze tra moduli
```

Il rischio principale è che il monolite modulare degeneri in monolite tradizionale.

---

# Microservizi

I **microservizi** sono servizi piccoli o medi, indipendenti, distribuiti separatamente e organizzati intorno a capacità di business.

Schema:

```text id="tj2k60"
┌──────────────┐      ┌──────────────┐
│ Clienti API  │      │ Ordini API   │
│              │      │              │
│ DB Clienti   │      │ DB Ordini    │
└──────┬───────┘      └──────┬───────┘
       │                     │
       └───────┬─────────────┘
               │
        ┌──────▼──────┐
        │ Message Bus │
        └──────┬──────┘
               │
       ┌───────▼───────┐
       │ Pagamenti API │
       │ DB Pagamenti  │
       └───────────────┘
```

Ogni servizio dovrebbe avere:

```text id="mhubqu"
responsabilità chiara
deploy indipendente
dati posseduti dal servizio
API o messaggi ben definiti
team o ownership chiara
osservabilità autonoma
```

---

# Microservizi non significa “tante API”

Un errore comune è pensare:

```text id="gklsz4"
Ho diviso il sistema in tante API,
quindi ho microservizi.
```

Non basta.

Se le API:

```text id="nkc65z"
condividono lo stesso database
devono essere deployate insieme
si chiamano in modo sincrono per ogni operazione
non hanno autonomia reale
rompono tutto se una cambia
```

potresti avere solo un **monolite distribuito**.

---

# Pro dei microservizi

```text id="au6fh6"
deploy indipendente
scalabilità indipendente
ownership per team
isolamento tecnologico
confini più forti
possibilità di evolvere servizi separatamente
resilienza se progettati bene
adatti a domini grandi e organizzazioni grandi
```

Esempio:

```text id="ccgilq"
Il servizio Report richiede molta CPU.
Il servizio Login ha traffico costante.
Il servizio Ordini ha picchi durante campagne promozionali.

Con microservizi, puoi scalare solo ciò che serve.
```

---

# Contro dei microservizi

```text id="a3mod9"
complessità operativa alta
rete fallibile
latenza
versionamento API
tracing distribuito necessario
logging centralizzato obbligatorio
deploy più complesso
test più difficili
coerenza dati più difficile
transazioni distribuite da evitare
duplicazione dati
gestione eventi
sicurezza service-to-service
maggiore costo infrastrutturale
```

La complessità non sparisce.

Si sposta:

```text id="nvox9i"
dal codice interno
alla comunicazione tra servizi
alla piattaforma operativa
alla consistenza dei dati
all’osservabilità
al coordinamento dei team
```

---

# Quando evitare i microservizi

Evitali quando:

```text id="n3tope"
il dominio è piccolo
il team è piccolo
non hai DevOps maturo
non hai osservabilità
non hai CI/CD affidabile
non sai ancora dove sono i confini del dominio
il prodotto è in fase esplorativa
hai bisogno di cambiare rapidamente modello dati
non hai competenze su sistemi distribuiti
un monolite modulare risolve già il problema
```

Regola pratica:

```text id="ki6klq"
Se non riesci a costruire un buon monolite modulare,
probabilmente non riuscirai a costruire buoni microservizi.
```

I microservizi amplificano sia le buone pratiche sia i problemi.

---

# Complessità operativa

Con un monolite devi gestire:

```text id="c2kn8i"
una app
un database
un deploy
un set di log
una configurazione principale
```

Con microservizi devi gestire:

```text id="rdxktu"
molte app
molti database
molti deploy
molti log
molte configurazioni
molte versioni API
molti health check
molti scaling rule
molte pipeline
molti segreti
molti failure mode
```

Questo richiede:

```text id="fywlr4"
CI/CD
container
orchestrazione
logging centralizzato
metriche
tracing distribuito
service discovery
gateway
security service-to-service
monitoraggio
strategie di rollback
```

Senza questi elementi, i microservizi diventano fragili.

---

# Team e organizzazione

I microservizi non sono solo scelta tecnica. Sono anche scelta organizzativa.

Funzionano meglio quando:

```text id="wo0i28"
ci sono team autonomi
ogni team possiede uno o più servizi
i confini del dominio sono chiari
ogni team può deployare senza coordinare tutto
ci sono standard comuni su logging, sicurezza, CI/CD
```

Se invece hai un solo piccolo team, i microservizi possono creare più coordinamento, non meno.

---

# Conway’s Law

In modo semplificato:

```text id="rc95se"
L’architettura del software tende a riflettere
la struttura comunicativa dell’organizzazione.
```

Se l’organizzazione è divisa in team autonomi per dominio, i microservizi possono avere senso.

Se tutto passa da un unico team centrale, potresti ottenere molti servizi ma poca autonomia reale.

---

# Database condiviso: rischio

Uno dei problemi più gravi nei falsi microservizi è il database condiviso.

Scenario:

```text id="dgbmtv"
Clienti API
Ordini API
Report API
Pagamenti API

tutti leggono e scrivono le stesse tabelle
```

Schema:

```text id="fwcdcm"
┌────────────┐
│ Clienti    │
└─────┬──────┘
      │
┌─────▼──────┐
│ Database   │
└─────▲──────┘
      │
┌─────┴──────┐
│ Ordini     │
└────────────┘
```

Problema:

```text id="orct9x"
i servizi non sono davvero indipendenti
lo schema database diventa contratto nascosto
ogni modifica rompe altri servizi
deploy indipendente diventa falso
le transazioni attraversano confini non dichiarati
```

---

# Database-per-service

In microservizi veri, ogni servizio possiede i propri dati.

```text id="lthzwy"
Clienti Service
  DB Clienti

Ordini Service
  DB Ordini

Pagamenti Service
  DB Pagamenti
```

Questo non significa necessariamente server fisici diversi.

Può significare:

```text id="iwzbrw"
database separati
schema separati
ownership chiara
accesso controllato
nessuna scrittura diretta sulle tabelle di altri servizi
```

Regola pratica:

```text id="x28p86"
Un servizio può esporre dati tramite API, eventi o proiezioni.
Non dovrebbe permettere ad altri servizi di modificare direttamente
le sue tabelle.
```

---

# Ma allora come faccio i report?

Domanda comune:

```text id="lip70t"
Se ogni servizio ha il proprio database,
come faccio report e query trasversali?
```

Possibili soluzioni:

```text id="ry5uv8"
proiezioni dedicate
data warehouse
read model
eventi di integrazione
ETL
repliche controllate
CQRS
materialized views
```

Esempio:

```text id="rnmslr"
Ordini Service pubblica OrdineConfermato
Clienti Service pubblica ClienteAggiornato
Report Service costruisce una propria tabella di lettura
```

Il Report Service non modifica i dati degli altri servizi.
Mantiene una proiezione ottimizzata per letture.

---

# Distributed monolith

Il **distributed monolith** è uno dei peggiori risultati possibili.

Sembra una architettura a microservizi, ma ha i difetti del monolite e dei sistemi distribuiti insieme.

Sintomi:

```text id="p5xh7s"
servizi deployati separatamente ma obbligati a uscire insieme
database condiviso
chiamate sincrone a catena
nessun servizio può funzionare senza gli altri
cambiamenti coordinati tra molti repository
assenza di ownership chiara
test end-to-end fragili
log sparsi
debug difficile
```

Esempio chiamata fragile:

```text id="dt2wfm"
API Gateway
  ↓
Ordini Service
  ↓
Clienti Service
  ↓
Sconti Service
  ↓
Pagamenti Service
  ↓
Notifiche Service
```

Se una richiesta utente richiede cinque servizi sincroni per completarsi, la disponibilità complessiva peggiora.

---

# Perché il distributed monolith è pericoloso

Hai i costi dei microservizi:

```text id="epd3p3"
rete
latenza
deploy multipli
monitoraggio complesso
configurazione distribuita
sicurezza service-to-service
```

ma non hai i benefici:

```text id="k4r2nb"
autonomia
deploy indipendente
scalabilità indipendente
ownership chiara
isolamento reale
```

Regola pratica:

```text id="egfj9e"
Se ogni servizio deve conoscere troppo degli altri,
probabilmente i confini sono sbagliati.
```

# Strangler Fig pattern

Lo **Strangler Fig pattern** è un approccio per migrare gradualmente da un sistema legacy o monolitico verso una nuova architettura.

L’idea è:

```text id="y1dpsk"
non riscrivo tutto in una volta.
costruisco nuove funzionalità intorno al sistema esistente.
sposto gradualmente traffico e responsabilità.
alla fine il vecchio sistema si riduce o viene spento.
```

Schema:

```text id="ffb2y0"
Client
  ↓
Router / Gateway
  ├── vecchio monolite
  └── nuovo servizio/modulo
```

Fase iniziale:

```text id="mtxzh7"
tutto va al monolite
```

Fase intermedia:

```text id="gpl8ky"
alcune rotte vanno al nuovo servizio
altre restano sul monolite
```

Fase finale:

```text id="mgn6er"
il monolite è stato svuotato o ridotto
```

---

# Quando usare Strangler Fig

È utile quando:

```text id="k2b4y1"
il sistema esistente è troppo grande per una riscrittura completa
serve ridurre rischio
devi mantenere il business operativo
puoi isolare funzionalità una alla volta
puoi instradare traffico verso nuovo codice
```

Esempio:

```text id="v1kpv2"
Vecchio gestionale monolitico
  continua a gestire tutto.

Nuovo modulo documenti
  gestisce solo documenti nuovi.

Gateway
  manda /documenti al nuovo sistema,
  il resto al monolite.
```

---

# Rischi dello Strangler Fig

```text id="xbs5ia"
doppia logica temporanea
sincronizzazione dati complessa
routing complicato
periodo ibrido lungo
necessità di compatibilità
costi temporanei più alti
```

Regola pratica:

```text id="gydy2p"
Lo Strangler Fig riduce il rischio della riscrittura totale,
ma richiede disciplina per non restare per sempre a metà.
```

---

# Matrice rischio/valore

Prima di estrarre un modulo o creare un microservizio, conviene valutare rischio e valore.

Matrice:

```text id="v04jhi"
                 Valore
              Basso     Alto
Rischio  Basso   A        B
         Alto    C        D
```

Interpretazione:

```text id="o901hm"
A - Basso rischio, basso valore
  non prioritario.

B - Basso rischio, alto valore
  ottimo candidato.

C - Alto rischio, basso valore
  da evitare.

D - Alto rischio, alto valore
  valutare con attenzione, prototipi, migrazione graduale.
```

---

# Esempio matrice

Possibili candidati:

```text id="cl82qt"
Reportistica
  valore alto
  rischio medio
  spesso buon candidato perché è lettura e può usare proiezioni.

Pagamenti
  valore alto
  rischio alto
  serve molta attenzione.

Anagrafica clienti
  valore alto
  rischio alto
  dati centrali, molte dipendenze.

Notifiche email
  valore medio
  rischio basso
  buon candidato per servizio separato.

Configurazioni statiche
  valore basso
  rischio basso
  non prioritario.
```

Scelta ragionevole:

```text id="sz5h1q"
iniziare da notifiche o reportistica,
non dal cuore transazionale più rischioso.
```

---

# Quando scegliere monolite

Scegli un monolite semplice quando:

```text id="q6lp0k"
il progetto è piccolo
il dominio è poco complesso
il team è piccolo
serve andare velocemente
i requisiti sono ancora instabili
non hai bisogno di deploy indipendente
non hai esigenze di scala differenziata
```

Esempi:

```text id="vlikqx"
prototipo
gestionale interno piccolo
MVP
app dipartimentale
servizio con pochi use case
```

---

# Quando scegliere monolite modulare

Scegli un monolite modulare quando:

```text id="urh45o"
il dominio è medio o grande
vuoi ordine architetturale
hai più aree funzionali
il deploy unico va ancora bene
vuoi evitare complessità distribuita
potresti voler estrarre servizi in futuro
vuoi confini chiari senza microservizi subito
```

Esempi:

```text id="oqix7s"
gestionale aziendale
backoffice
sistema documentale
CRM interno
applicazione business con più moduli
```

Per molti sistemi aziendali, questa è la scelta più equilibrata.

---

# Quando scegliere microservizi

Scegli microservizi quando:

```text id="uordmq"
il dominio è grande e ben compreso
i bounded context sono chiari
team diversi devono lavorare autonomamente
servono deploy indipendenti
servono scaling indipendenti
hai DevOps maturo
hai osservabilità solida
hai competenze su sistemi distribuiti
il costo operativo è giustificato dal valore
```

Esempi:

```text id="eqm1t5"
piattaforme SaaS grandi
e-commerce ad alto traffico
sistemi con team multipli
sistemi con domini molto separati
piattaforme cloud-native mature
```

---

# Domande decisionali

Prima di scegliere microservizi, chiediti:

```text id="tu4blf"
Abbiamo confini di dominio chiari?

I servizi potranno essere deployati indipendentemente?

Ogni servizio può possedere i propri dati?

Abbiamo CI/CD maturo?

Abbiamo logging centralizzato?

Abbiamo tracing distribuito?

Abbiamo metriche e alert?

Abbiamo gestione segreti e configurazione affidabile?

Abbiamo competenze per gestire errori di rete, retry, circuit breaker,
idempotenza e consistenza eventuale?

Il team è organizzato per ownership autonome?

Il valore supera la complessità?
```

Se molte risposte sono no, meglio partire da monolite modulare.

---

# Evoluzione consigliata

Per molti progetti il percorso migliore è:

```text id="d7jlvc"
1. Monolite semplice
   se il progetto è piccolo o esplorativo.

2. Monolite modulare
   quando il dominio cresce.

3. Estrazione selettiva di servizi
   solo dove esiste un beneficio chiaro.

4. Microservizi
   solo per parti che richiedono autonomia reale.
```

Non partire dai microservizi solo perché il sistema potrebbe crescere.

Regola pratica:

```text id="l0k02u"
Parti semplice, ma non disordinato.
```

Un monolite modulare ben fatto lascia aperta la strada a evoluzioni future.

---

# Esempio pratico: gestionale aziendale

Scenario:

```text id="z7mg4e"
Clienti
Documenti
Ordini
Report
Notifiche
Import dati
```

Scelta iniziale ragionevole:

```text id="axbr9f"
monolite modulare
```

Moduli:

```text id="sj09m8"
Clienti
Documenti
Ordini
Report
Notifiche
Import
```

Possibili estrazioni future:

```text id="y5x3ib"
Notifiche
  servizio separato se deve gestire code, retry, provider email/SMS.

Report
  servizio separato se richiede carichi pesanti o database read model.

Import
  worker separato se elabora file grandi o batch schedulati.
```

Non estrarre subito:

```text id="p1k2ah"
Clienti
Ordini
Documenti
```

se sono fortemente collegati e il dominio non è ancora stabile.

# Esempio pratico: e-commerce

Possibili bounded context:

```text id="pl8l1r"
Catalogo
Carrello
Ordini
Pagamenti
Spedizioni
Clienti
Promozioni
```

Microservizi possono avere senso se:

```text id="t8q1th"
il traffico è alto
team diversi possiedono aree diverse
catalogo e ordini hanno esigenze diverse
pagamenti richiedono isolamento
spedizioni integra provider esterni
```

Ma per un e-commerce piccolo:

```text id="n3nv28"
un monolite modulare è probabilmente più sano
```

# Anti-pattern frequenti

## Microservizi per moda

```text id="g2tzd9"
Scegliamo microservizi perché sono moderni.
```

No.

Sceglili per problemi concreti.

---

## Database condiviso

```text id="rrs7bd"
ogni servizio accede allo stesso database
```

È un segnale forte di monolite distribuito.

---

## Servizi troppo piccoli

```text id="pqgdcx"
ClienteNomeService
ClienteEmailService
ClienteTelefonoService
```

Troppa granularità aumenta comunicazione e complessità.

---

## Chiamate sincrone a catena

```text id="bq01gs"
A chiama B
B chiama C
C chiama D
D chiama E
```

Latenza e fragilità aumentano.

---

## Nessuna ownership

```text id="n6ucdb"
tutti modificano tutti i servizi
nessuno è responsabile davvero
```

Microservizi senza ownership diventano caos distribuito.

---

## Nessuna osservabilità

Microservizi senza tracing, metriche e log centralizzati sono difficili da gestire.

---

## Estrarre prima di capire

Se non conosci il dominio, rischi di tagliare nei punti sbagliati.

# Linee guida pratiche

```text id="d96fg8"
Non associare microservizi a modernità
  valuta costi e benefici.

Usa monolite semplice
  per progetti piccoli, MVP e domini instabili.

Usa monolite modulare
  per molte applicazioni business reali.

Usa microservizi
  solo quando serve autonomia reale di team, deploy, dati e scalabilità.

Evita database condivisi tra servizi
  sono un forte accoppiamento nascosto.

Evita distributed monolith
  ha i costi dei microservizi senza i benefici.

Applica Strangler Fig
  per migrazioni graduali da sistemi legacy o monoliti grandi.

Valuta rischio e valore
  prima di estrarre un servizio.

Non iniziare dal cuore più rischioso
  scegli candidati con buon valore e rischio controllato.

Investi prima in DevOps e osservabilità
  senza questi, i microservizi sono pericolosi.

Parti semplice, ma modulare
  così puoi evolvere senza riscrivere tutto.
```

# Riepilogo

In questo capitolo abbiamo confrontato monolite, monolite modulare e microservizi.

Abbiamo visto:

```text id="qbf14b"
monolite
monolite modulare
microservizi
pro e contro
quando evitare i microservizi
complessità operativa
team e organizzazione
database condiviso
distributed monolith
Strangler Fig pattern
matrice rischio/valore
```

La distinzione più importante è questa:

```text id="h9j7dt"
Monolite
  una sola unità di deploy, spesso più semplice.

Monolite modulare
  una sola unità di deploy, ma codice organizzato per moduli.

Microservizi
  più unità autonome, con deploy, dati e responsabilità separati.
```

La regola principale è questa:

```text id="jpe1cd"
I microservizi non risolvono il disordine.
Lo distribuiscono.
```

Per molte applicazioni reali, soprattutto aziendali, un monolite modulare ben progettato è una scelta più equilibrata: mantiene semplicità operativa, permette buona separazione interna e lascia aperta la possibilità di estrarre servizi solo quando il valore supera davvero il costo.

Il messaggio corretto non è:

```text id="tr4vnr"
microservizi = moderno
```

ma:

```text id="cg585t"
scegli l’architettura più semplice che soddisfa i requisiti attuali
e che può evolvere senza diventare ingestibile.
```

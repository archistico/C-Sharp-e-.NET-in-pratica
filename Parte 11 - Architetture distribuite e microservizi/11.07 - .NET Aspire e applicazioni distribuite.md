# 11.07 — .NET Aspire e applicazioni distribuite

Nei capitoli precedenti abbiamo parlato di Docker, Kubernetes, gateway, microservizi, osservabilità, sicurezza e resilienza.

Ora introduciamo **.NET Aspire**, che si colloca in un punto molto interessante: non sostituisce tutto ciò che abbiamo visto, ma aiuta a sviluppare e coordinare applicazioni distribuite in modo più semplice, soprattutto in locale.

Il punto centrale è questo:

```text id="6dzuis"
.NET Aspire è un modello code-first per descrivere, avviare,
osservare e collegare applicazioni distribuite durante lo sviluppo.

Non è Kubernetes.
Non è Docker Compose.
Non è un cloud provider.
È un layer di orchestrazione e osservabilità pensato per semplificare
lo sviluppo di sistemi composti da più servizi.
```

La documentazione ufficiale Aspire descrive Aspire come un layer code-first di orchestrazione e osservabilità per applicazioni distribuite; il progetto `AppHost` modella applicazioni, container, database, cache e altre risorse, rendendole più facili da eseguire e diagnosticare in locale. ([Aspire][1])

---

# Che cos’è .NET Aspire

**.NET Aspire** è uno stack e un insieme di strumenti per costruire applicazioni distribuite cloud-ready.

Serve a gestire in modo più semplice problemi come:

```text id="72eknu"
avviare più progetti insieme
collegare API, worker, database e cache
gestire connection string tra servizi
fare service discovery in locale
vedere log, trace e metriche
avviare container di supporto
configurare Redis, database, broker e servizi esterni
osservare l’applicazione tramite dashboard
```

Aspire è particolarmente utile quando una solution contiene più componenti:

```text id="gzkg0k"
Frontend
API
Worker
Redis
Database
Message broker
Servizio esterno simulato
```

Invece di avviarli uno per uno, Aspire permette di descriverli in un progetto orchestratore.

# Il problema che risolve Aspire

Senza Aspire, durante lo sviluppo di un’app distribuita potresti dover fare:

```text id="3xomq8"
avviare API Clienti
avviare API Ordini
avviare Worker Notifiche
avviare Redis
avviare SQL Server o PostgreSQL
configurare connection string
configurare porte
ricordare URL dei servizi
aprire terminali diversi
controllare log separati
capire quale servizio non è partito
```

Con Aspire, l’obiettivo è arrivare a un comando simile:

```bash id="p3a4cb"
dotnet run --project MiaApp.AppHost
```

e vedere l’intero sistema avviarsi insieme.

# I progetti tipici di una soluzione Aspire

Una solution Aspire contiene spesso almeno due progetti speciali:

```text id="fcwovr"
MiaApp.AppHost
  descrive l’applicazione distribuita:
  servizi, container, database, cache, dipendenze.

MiaApp.ServiceDefaults
  contiene configurazioni comuni:
  OpenTelemetry, health check, service discovery,
  resilienza HTTP, logging.
```

Poi ci sono i progetti applicativi veri:

```text id="nrn4et"
MiaApp.Api
MiaApp.Worker
MiaApp.Web
MiaApp.BackOffice
```

Schema:

```text id="qn4al6"
MiaApp.AppHost
  ├── MiaApp.Api
  ├── MiaApp.Worker
  ├── Redis container
  ├── SQL/PostgreSQL container
  └── Dashboard Aspire

MiaApp.ServiceDefaults
  └── configurazione comune usata da Api, Worker, Web
```

# AppHost

Il progetto **AppHost** è il cuore dell’orchestrazione Aspire.

Contiene codice C# che descrive le risorse dell’applicazione.

Esempio concettuale:

```csharp id="v3b8sf"
var builder = DistributedApplication.CreateBuilder(args);

var redis = builder.AddRedis("cache");

var api = builder.AddProject<Projects.MiaApp_Api>("api")
    .WithReference(redis);

builder.AddProject<Projects.MiaApp_Worker>("worker")
    .WithReference(redis);

builder.Build().Run();
```

Qui stiamo dicendo:

```text id="40h2tu"
crea una risorsa Redis chiamata cache
avvia il progetto API
avvia il progetto Worker
collega API e Worker a Redis
```

L’AppHost non è il codice business dell’applicazione.
È il modello della composizione distribuita.

---

# Orchestrazione locale

Uno dei punti forti di Aspire è l’**orchestrazione locale**.

Aspire può:

```text id="bz5xel"
avviare progetti .NET
avviare container necessari
iniettare connection string
assegnare porte
configurare variabili d’ambiente
mostrare lo stato delle risorse
raccogliere log e trace
aprire dashboard
```

Esempio:

```csharp id="l3uqf3"
var builder = DistributedApplication.CreateBuilder(args);

var postgres = builder.AddPostgres("postgres")
    .AddDatabase("miaappdb");

var redis = builder.AddRedis("redis");

builder.AddProject<Projects.MiaApp_Api>("api")
    .WithReference(postgres)
    .WithReference(redis);

builder.AddProject<Projects.MiaApp_Worker>("worker")
    .WithReference(postgres);

builder.Build().Run();
```

Questo consente a uno sviluppatore di eseguire un ambiente distribuito senza configurare manualmente tutti i pezzi.

# Aspire non è Docker Compose

Aspire e Docker Compose possono sembrare simili perché entrambi avviano più componenti.

Ma hanno scopi diversi.

```text id="pssewm"
Docker Compose
  descrive container.

Aspire
  descrive un’applicazione distribuita:
  progetti .NET, container, database, cache, collegamenti,
  service discovery e telemetria.
```

Aspire può usare container sotto il cofano, ma il modello è più vicino all’applicazione .NET.

Regola pratica:

```text id="m6vxi1"
Docker Compose ragiona in termini di container.
Aspire ragiona in termini di risorse applicative.
```

# Service discovery

In un’app distribuita, i servizi devono trovarsi.

Senza service discovery, potresti scrivere:

```json id="wk7ewi"
{
  "Services": {
    "OrdiniApi": "http://localhost:5012"
  }
}
```

Ma porte e indirizzi possono cambiare.

Aspire aiuta a usare nomi logici.

Esempio AppHost:

```csharp id="l5aqrc"
var ordiniApi = builder.AddProject<Projects.Ordini_Api>("ordini-api");

builder.AddProject<Projects.BackOffice_Web>("backoffice")
    .WithReference(ordiniApi);
```

Nel progetto `backoffice`, puoi usare HTTP client e service discovery per chiamare il servizio per nome logico, invece di fissare manualmente la porta.

La prima introduzione ufficiale di .NET Aspire sottolineava proprio che l’orchestrazione Aspire semplifica aspetti come service discovery, variabili d’ambiente e configurazione dei container senza gestire direttamente i dettagli più bassi. ([Microsoft for Developers][2])

# Configurazione

Aspire semplifica la configurazione tra risorse.

Esempio:

```csharp id="rzw5l6"
var sql = builder.AddSqlServer("sql")
    .AddDatabase("miaappdb");

builder.AddProject<Projects.MiaApp_Api>("api")
    .WithReference(sql);
```

Il progetto API riceve la configurazione necessaria per collegarsi al database.

Nel codice applicativo, continui a usare il normale sistema di configurazione .NET:

```csharp id="gmnumg"
string? connectionString =
    builder.Configuration.GetConnectionString("miaappdb");
```

Aspire non elimina `appsettings.json`, variabili d’ambiente o secrets.
Li coordina meglio in un contesto distribuito.

# ServiceDefaults

Il progetto `ServiceDefaults` contiene configurazioni comuni per tutti i servizi.

Tipicamente include:

```text id="gsdbkd"
OpenTelemetry
health checks
service discovery
resilienza HTTP
logging
metriche
endpoint di management
```

Esempio concettuale in un progetto API:

```csharp id="cw6iyp"
var builder = WebApplication.CreateBuilder(args);

builder.AddServiceDefaults();

var app = builder.Build();

app.MapDefaultEndpoints();

app.MapGet("/api/ping", () => "pong");

app.Run();
```

L’idea è evitare di ripetere la stessa configurazione tecnica in ogni progetto.

# Telemetria

Aspire è molto orientato all’osservabilità.

Raccoglie e mostra:

```text id="6c9c93"
log strutturati
trace distribuite
metriche
stato risorse
endpoint
variabili/configurazioni rilevanti
```

La telemetria di Aspire si basa su OpenTelemetry. Aspire fornisce logging strutturato, tracing distribuito e metriche, esportati tramite protocollo OTLP. ([Mintlify][3])

Questo si collega direttamente al capitolo sull’osservabilità:

```text id="o6lpgg"
Log
  cosa è successo.

Trace
  percorso di una richiesta tra servizi.

Metriche
  andamento numerico nel tempo.
```

# Worker

Anche un Worker Service si integra bene con Aspire.

Esempio:

```csharp id="v406oc"
var builder = Host.CreateApplicationBuilder(args);

builder.AddServiceDefaults();

builder.Services.AddHostedService<Worker>();

var host = builder.Build();

host.Run();
```

Worker:

```csharp id="atzdz5"
public class Worker : BackgroundService
{
    private readonly ILogger<Worker> logger;

    public Worker(ILogger<Worker> logger)
    {
        this.logger = logger;
    }

    protected override async Task ExecuteAsync(
        CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            logger.LogInformation(
                "Worker attivo alle {Time}",
                DateTimeOffset.Now);

            await Task.Delay(
                TimeSpan.FromSeconds(10),
                stoppingToken);
        }
    }
}
```

AppHost:

```csharp id="shj1c4"
builder.AddProject<Projects.MiaApp_Worker>("worker");
```

La dashboard mostrerà anche i log del worker.

# Aspire e Kubernetes

Aspire non sostituisce Kubernetes.

Confronto:

```text id="l68sji"
Aspire
  modella e orchestra l’applicazione soprattutto in sviluppo.
  aiuta con service discovery, configurazione e telemetria.
  può generare/deployare verso ambienti cloud tramite strumenti dedicati.

Kubernetes
  orchestra container in produzione.
  gestisce Pod, Deployment, Service, Ingress, scaling e rollout.
```

Aspire può aiutare nel percorso verso il cloud e i container, ma Kubernetes resta una piattaforma di orchestrazione production-grade.

Per Azure Container Apps, la documentazione Aspire indica che `aspire deploy` può distribuire applicazioni Aspire e provisionare risorse come Container Apps environment, Azure Container Registry, managed identity e container app necessari alle risorse definite nell’AppHost. ([Aspire][7])


# Aspire Dashboard standalone

La dashboard Aspire può essere usata anche come strumento di osservabilità locale o in ambienti controllati.

L’immagine Docker della dashboard espone un endpoint OTLP/gRPC per ricevere telemetria; quando viene lanciata dall’AppHost, l’indirizzo è protetto con HTTPS, e la documentazione dell’immagine raccomanda di proteggere la dashboard con HTTPS. ([hub.docker.com][9])

Questo è utile per capire che la dashboard riceve dati OpenTelemetry.

Schema:

```text id="wwxgit"
API / Worker
  ↓ OTLP
Aspire Dashboard
  ↓
visualizza log, trace, metriche
```

# Errori comuni

## Pensare che Aspire sia Kubernetes

No.

Aspire aiuta a modellare e orchestrare lo sviluppo di applicazioni distribuite. Kubernetes orchestra container in produzione.

---

## Pensare che Aspire renda superfluo Docker

No.

Aspire può usare container e si integra con essi, ma bisogna comunque capire immagini, container, porte, volumi e sicurezza.

---

## Usarlo per una sola app semplicissima

Possibile, ma spesso inutile.

---

## Mettere logica business nell’AppHost

L’AppHost descrive risorse e collegamenti.
Non deve contenere regole di dominio.

---

## Ignorare la configurazione production

Aspire semplifica molto il locale, ma la produzione richiede comunque gestione seria di segreti, reti, database, scaling e monitoraggio.

---

## Confondere ServiceDefaults con architettura

`ServiceDefaults` standardizza configurazioni tecniche.
Non decide i confini del dominio.

# Linee guida pratiche

```text id="dohli4"
Usa Aspire
  quando hai più progetti o risorse da coordinare.

Usa AppHost
  per descrivere progetti, container, database, cache e dipendenze.

Usa ServiceDefaults
  per configurare osservabilità, health check e service discovery
  in modo uniforme.

Usa la dashboard
  per vedere log, trace, metriche e stato delle risorse.

Usa integrazioni Redis/database
  per semplificare lo sviluppo locale.

Non trattare Aspire come sostituto di Kubernetes
  sono strumenti diversi.

Non trattare Aspire come sostituto dell’architettura
  devi comunque progettare bene confini, dati e sicurezza.

Usa Aspire come ponte
  tra sviluppo locale, container, osservabilità e cloud.

Valuta Aspire anche per monoliti modulari
  se hanno worker, cache, database e risorse esterne.

Mantieni separati sviluppo locale e produzione
  Aspire aiuta, ma non cancella le differenze operative.
```

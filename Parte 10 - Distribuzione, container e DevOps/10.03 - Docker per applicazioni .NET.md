# 10.03 — Docker per applicazioni .NET

Nel capitolo precedente abbiamo visto NuGet e packaging: come distribuire librerie, analyzer, source generator e tipi riutilizzabili.

Ora passiamo a Docker, cioè a un modo standard e ripetibile per impacchettare ed eseguire applicazioni.

Il punto centrale è questo:

```text id="d8x2qv"
Un container non è una macchina virtuale leggera.
È un processo isolato che esegue un’applicazione con filesystem,
rete, variabili d’ambiente e dipendenze controllate.
```

Docker è diventato uno standard perché permette di eseguire la stessa applicazione in modo più prevedibile su:

```text id="g0jgo6"
computer dello sviluppatore
server di test
server di staging
produzione
pipeline CI/CD
cloud
Kubernetes
```

Microsoft descrive i container come infrastruttura immutabile, portabile e scalabile; un’immagine può essere usata per creare container in sviluppo locale, cloud privato o cloud pubblico. ([Microsoft Learn][1])

---

# Concetto di container

Un **container** esegue un’applicazione partendo da una **image**.

Distinzione fondamentale:

```text id="pbsq97"
Image
  modello immutabile.
  Contiene filesystem, runtime, dipendenze e comando di avvio.

Container
  istanza in esecuzione di una image.
```

Analogia:

```text id="e2eo3c"
Image
  classe

Container
  oggetto creato da quella classe
```

Esempio:

```bash id="dnf97n"
docker run --rm mcr.microsoft.com/dotnet/samples
```

Questo comando scarica, se necessario, una image .NET di esempio e crea un container temporaneo.

---

# Container non è VM

Una virtual machine contiene un sistema operativo completo.

Un container condivide il kernel dell’host e isola il processo tramite funzionalità del sistema operativo.

Differenze pratiche:

```text id="aahx4k"
VM
  più pesante
  sistema operativo completo
  avvio più lento
  isolamento più forte

Container
  più leggero
  processo isolato
  avvio rapido
  ideale per deploy ripetibili
```

Per .NET, questo significa che l’app viene pubblicata dentro una image che contiene il runtime necessario o usa una base image già predisposta.

# Dockerfile

Un **Dockerfile** descrive come costruire una image.

Esempio minimale per una API già pubblicata:

```dockerfile id="hasy62"
FROM mcr.microsoft.com/dotnet/aspnet:10.0

WORKDIR /app

COPY ./publish .

ENTRYPOINT ["dotnet", "MiaApp.Api.dll"]
```

Build:

```bash id="zxubn5"
docker build -t miaapp-api .
```

Run:

```bash id="hnyrdt"
docker run --rm -p 8080:8080 miaapp-api
```

La documentazione Microsoft mostra che per containerizzare un’app ASP.NET Core si può pubblicare l’app e poi creare una image Docker, oppure usare Dockerfile multi-stage per compilare e pubblicare direttamente dentro la build dell’immagine. ([Microsoft Learn][2])

# Immagini base .NET

Microsoft pubblica immagini ufficiali .NET ottimizzate e disponibili sul Microsoft Artifact Registry. ([Microsoft Learn][3])

Le immagini più comuni sono:

```text id="vx09sj"
mcr.microsoft.com/dotnet/sdk
  contiene SDK, CLI, runtime e strumenti di build.
  Serve per compilare, testare e pubblicare.

mcr.microsoft.com/dotnet/aspnet
  contiene runtime ASP.NET Core.
  Serve per eseguire web app e API.

mcr.microsoft.com/dotnet/runtime
  contiene runtime .NET.
  Serve per console app e worker non web.

mcr.microsoft.com/dotnet/runtime-deps
  contiene dipendenze native minime.
  Utile per self-contained, trimming e Native AOT.
```

Regola pratica:

```text id="i4fpib"
SDK image per build.
Runtime/ASP.NET image per esecuzione.
```

Non usare l’immagine SDK in produzione se non serve: è più grande e contiene strumenti non necessari.

# Multi-stage build

Il **multi-stage build** permette di usare una image per compilare e una image diversa, più piccola, per eseguire.

Docker raccomanda i multi-stage build perché separano ambiente di build e runtime, riducendo dimensione finale e superficie d’attacco. ([Docker Documentation][4])

Schema:

```text id="p8dm1i"
Stage build
  usa dotnet/sdk
  restore
  build
  publish

Stage runtime
  usa dotnet/aspnet o dotnet/runtime
  copia solo output pubblicato
  esegue app
```

# Dockerfile per API ASP.NET Core

Esempio tipico:

```dockerfile id="zz6a5x"
FROM mcr.microsoft.com/dotnet/sdk:10.0 AS build
WORKDIR /src

COPY ["MiaApp.Api/MiaApp.Api.csproj", "MiaApp.Api/"]
RUN dotnet restore "MiaApp.Api/MiaApp.Api.csproj"

COPY . .
WORKDIR "/src/MiaApp.Api"

RUN dotnet publish "MiaApp.Api.csproj" \
    -c Release \
    -o /app/publish \
    /p:UseAppHost=false

FROM mcr.microsoft.com/dotnet/aspnet:10.0 AS final
WORKDIR /app

COPY --from=build /app/publish .

ENV ASPNETCORE_URLS=http://+:8080

EXPOSE 8080

ENTRYPOINT ["dotnet", "MiaApp.Api.dll"]
```

Build:

```bash id="qrauzs"
docker build -t miaapp-api:1.0.0 .
```

Run:

```bash id="x8ugej"
docker run --rm -p 8080:8080 miaapp-api:1.0.0
```

Apri:

```text id="f5o0kn"
http://localhost:8080
```

# Perché copiare prima il `.csproj`

Nel Dockerfile precedente:

```dockerfile id="w2qqlu"
COPY ["MiaApp.Api/MiaApp.Api.csproj", "MiaApp.Api/"]
RUN dotnet restore "MiaApp.Api/MiaApp.Api.csproj"
```

poi:

```dockerfile id="tm0u49"
COPY . .
```

Questo sfrutta la cache Docker.

Se cambi solo un file `.cs`, Docker può riusare lo stage `restore`.

Se copiassi subito tutto, ogni modifica invaliderebbe più facilmente la cache.

Regola pratica:

```text id="sjwrc6"
Copia prima i file progetto.
Esegui restore.
Poi copia il resto del codice.
```

# Container per API

Una API in container deve normalmente configurare:

```text id="sn11iu"
porta interna
variabili d’ambiente
connection string
logging verso stdout/stderr
health checks
reverse proxy o orchestratore davanti
```

Esempio:

```bash id="m50ldo"
docker run --rm \
  -p 8080:8080 \
  -e ASPNETCORE_ENVIRONMENT=Production \
  -e ConnectionStrings__Default="Server=db;Database=MiaApp;User Id=app;Password=secret" \
  miaapp-api:1.0.0
```

In ASP.NET Core, le variabili d’ambiente possono sovrascrivere la configurazione dei file JSON; quando si usa `docker run`, si può impostare `ASPNETCORE_ENVIRONMENT` con `-e`, e un valore impostato a runtime sovrascrive quello definito nel Dockerfile. ([Microsoft Learn][5])

# Container per worker

Un worker in container è adatto a:

```text id="xsimvc"
elaborazione code
job schedulati
outbox worker
import/export
sincronizzazioni
processi batch
```

Esempio:

```bash id="rtngq9"
docker run --rm \
  -e DOTNET_ENVIRONMENT=Production \
  -e ConnectionStrings__Default="Server=db;Database=MiaApp;User Id=app;Password=secret" \
  miaapp-worker:1.0.0
```

Per worker è importante gestire correttamente:

```text id="f56q47"
CancellationToken
shutdown graceful
log su console
retry controllati
idempotenza
configurazione via environment
```

Esempio worker:

```csharp id="edxvwq"
public class Worker : BackgroundService
{
    private readonly ILogger<Worker> logger;

    public Worker(ILogger<Worker> logger)
    {
        this.logger = logger;
    }

    protected override async Task ExecuteAsync(
        CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            logger.LogInformation(
                "Worker attivo alle {Time}",
                DateTimeOffset.Now);

            await Task.Delay(
                TimeSpan.FromSeconds(10),
                stoppingToken);
        }
    }
}
```

Nei container, la configurazione deve arrivare dall’esterno.

Esempio:

```bash id="fs64ia"

  -e ASPNETCORE_ENVIRONMENT=Staging \

  -e Email__Port=587 \
```

In .NET:

```text id="lz0pkb"

```

```text id="jltr9c"

```

Esempio `appsettings.json`:

```json id="h262o5"
{
  "Email": {
    "SmtpServer": "localhost",
    "Port": 2525
  }
}
```

Variabile d’ambiente:

```bash id="p2qpv6"
Email__SmtpServer=smtp-prod.example.com
```

sovrascrive il valore.

# Volumi

Un container è pensato per essere sostituibile.

Il filesystem interno non deve essere considerato persistente.

I **volumi** servono per conservare dati fuori dal ciclo di vita del container.

Esempi:

```text id="cyp6j2"
file caricati dagli utenti
database locale in sviluppo
log file, se proprio necessari
certificati
cartelle di import/export
```

Esempio bind mount:

```bash id="fwdlr4"
docker run --rm \
  -v C:\Dati\MiaApp:/app/data \
  miaapp-worker:1.0.0
```

Su Linux:

```bash id="vfm83c"
docker run --rm \
  -v /srv/miaapp/data:/app/data \
  miaapp-worker:1.0.0
```

Named volume:

```bash id="d7a2qw"
docker volume create miaapp-data

docker run --rm \
  -v miaapp-data:/app/data \
  miaapp-worker:1.0.0
```

Regola pratica:

```text id="zilnh1"
Il container deve essere eliminabile.
I dati importanti devono stare fuori dal container.
```

# Docker Compose

**Docker Compose** permette di descrivere più servizi in un file.

Esempio:

```text id="fgfc5g"

```

File:

```yaml id="l1bvo5"
services:
  api:
    build:
      context: .
      dockerfile: MiaApp.Api/Dockerfile
    ports:
      - "8080:8080"
    environment:
      ASPNETCORE_ENVIRONMENT: Development
      ASPNETCORE_URLS: http://+:8080

      - db

      - sql-data:/var/opt/mssql

  sql-data:
```

Avvio:

```bash id="akpzek"
docker compose up -d
```

```bash id="q8fvsa"

```


# Attendere il database

`depends_on` garantisce l’ordine di avvio dei container, ma non sempre garantisce che il database sia già pronto ad accettare connessioni.

Problemi tipici:

```text id="as3g1t"
API parte
database non è ancora pronto
migrazione fallisce
prima request fallisce
```

Soluzioni:

```text id="3y9krl"
retry sulla connessione
healthcheck Docker Compose
script di attesa
resilienza lato applicazione
migrazioni eseguite separatamente
```

Esempio healthcheck:

```yaml id="pjbr3i"
services:
  db:
    image: mcr.microsoft.com/mssql/server:2022-latest
    environment:
      ACCEPT_EULA: "Y"
      SA_PASSWORD: ${DB_PASSWORD}
    healthcheck:
      test: /opt/mssql-tools18/bin/sqlcmd -S localhost -U sa -P "$$SA_PASSWORD" -C -Q "SELECT 1"
      interval: 10s
      timeout: 5s
      retries: 10
```

# Health check

Una API in container dovrebbe esporre health check.

In ASP.NET Core:

```csharp id="vqyv52"
builder.Services.AddHealthChecks();

var app = builder.Build();

app.MapHealthChecks("/health");
```

Dockerfile:

```dockerfile id="dq9eec"
HEALTHCHECK --interval=30s --timeout=3s --retries=3 \
  CMD wget -qO- http://localhost:8080/health || exit 1
```

Oppure Docker Compose:

```yaml id="hxfk6t"
services:
  api:
    healthcheck:
      test: ["CMD", "wget", "-qO-", "http://localhost:8080/health"]
      interval: 30s
      timeout: 3s
      retries: 3
```

Nota: non tutte le immagini includono `wget` o `curl`. In alcuni casi conviene gestire health check a livello di orchestratore.

# Errori comuni

## Usare l’immagine SDK in produzione

Sbagliato:

```dockerfile id="q20ix2"
FROM mcr.microsoft.com/dotnet/sdk:10.0
```

nello stage finale.

Meglio:

```dockerfile id="gi11bk"
FROM mcr.microsoft.com/dotnet/aspnet:10.0
```

per API, oppure:

```dockerfile id="eevsmc"
FROM mcr.microsoft.com/dotnet/runtime:10.0
```

per worker.

---

## Mettere segreti nel Dockerfile

Pericoloso:

```dockerfile id="ik4l6s"
ENV ApiKey=supersecret
```

I layer dell’immagine possono conservare informazioni sensibili.

---

## Usare localhost per parlare con un altro container

Dentro un container:

```text id="guco35"
localhost = il container stesso
```

In Docker Compose usa il nome del servizio:

```text id="jvedqo"
Server=db
```

non:

```text id="xohb2s"
Server=localhost
```

---

## Non usare volumi per dati persistenti

Se il database salva dati solo nel filesystem del container, perdi tutto quando elimini il container.

Usa volumi.

---

## Non avere `.dockerignore`

Rischi:

```text id="roux4c"
build lente
contesto enorme
file inutili nell’immagine
segreti copiati per errore
```

---

## Affidarsi a `latest`

`latest` non dice quale versione stai eseguendo.

In produzione usa tag espliciti.

---

## Fare migrazioni automaticamente senza controllo

Eseguire `Database.Migrate()` all’avvio può essere comodo in sviluppo, ma in produzione può essere rischioso.

Valuta:

```text id="b63s9h"
migrazioni in pipeline
script approvati
backup
rollback
finestre di manutenzione
```

# Linee guida pratiche

```text id="fdxuaj"
Usa container
  per rendere l’esecuzione ripetibile e portabile.

Usa Dockerfile multi-stage
  SDK per build, runtime per esecuzione.

Usa immagini ufficiali Microsoft
  per SDK, ASP.NET runtime e runtime.

Configura tramite variabili d’ambiente
  non hard-coded nel Dockerfile.

Non mettere segreti nella image
  passali a runtime tramite sistemi sicuri.

Usa volumi
  per dati persistenti.

Usa Docker Compose
  per ambienti locali con API, worker, database e servizi.

Usa nomi servizio Compose
  per comunicare tra container.

Aggiungi .dockerignore
  per ridurre contesto e rischi.

Scrivi log su console
  così Docker e orchestratori li raccolgono.

Usa tag immutabili
  non latest in produzione.

Riduci superficie d’attacco
  niente SDK nello stage finale, immagini piccole, utente non root.

Scansiona immagini
  e aggiorna regolarmente immagini base.
```

# Riepilogo

In questo capitolo abbiamo introdotto Docker per applicazioni .NET.

Abbiamo visto:

```text id="g2al7w"
concetto di container
Dockerfile
multi-stage build
immagini base .NET
container per API
container per worker
variabili d’ambiente
volumi
Docker Compose
database in container
debug in container
sicurezza immagini
```

La distinzione fondamentale è:

```text id="v8ekcu"
Image
  pacchetto immutabile dell’applicazione.

Container
  istanza in esecuzione di quell’immagine.
```

Per .NET, la struttura consigliata è quasi sempre multi-stage: build con `mcr.microsoft.com/dotnet/sdk`, esecuzione con `mcr.microsoft.com/dotnet/aspnet` per API/web app o `mcr.microsoft.com/dotnet/runtime` per worker. Le immagini ufficiali .NET sono create e ottimizzate da Microsoft e pubblicate sul Microsoft Artifact Registry. ([Microsoft Learn][3])

La regola pratica finale è questa:

```text id="r9lncg"
Un buon container .NET contiene solo ciò che serve per eseguire l’app,
riceve configurazione dall’esterno, non contiene segreti,
scrive log su console, usa volumi per dati persistenti
ed è costruito in modo ripetibile tramite Dockerfile e pipeline.
```

# 10.02 — NuGet e packaging

Nel capitolo precedente abbiamo visto come pubblicare e distribuire applicazioni .NET. Ora affrontiamo un tema collegato ma diverso: **distribuire codice riutilizzabile**.

Non tutto ciò che scriviamo deve diventare un’applicazione. A volte vogliamo creare una libreria condivisa da più progetti:

```text id="q3elpm"
libreria di utility
client per una API interna
componenti comuni di dominio
validazioni condivise
source generator
analyzer
SDK aziendale
libreria per logging o auditing
wrapper verso sistemi legacy
```

Il formato standard per distribuire librerie .NET è il pacchetto **NuGet**.

Il punto centrale è questo:

```text id="g54xww"
NuGet non serve solo a consumare librerie di terzi.
Serve anche a distribuire in modo controllato codice riutilizzabile
tra applicazioni, team e ambienti.
```

La documentazione Microsoft descrive `dotnet pack` e `dotnet nuget push` come i passaggi principali per creare un pacchetto `.nupkg` da una libreria .NET e pubblicarlo su nuget.org o su un feed. ([learn.microsoft.com](https://learn.microsoft.com/en-us/nuget/quickstart/create-and-publish-a-package-using-the-dotnet-cli))

---

# Creare una libreria

Una libreria .NET si crea con:

```bash id="t7vxmx"
dotnet new classlib -n MiaAzienda.Clienti
```

Struttura:

```text id="uzdxtt"
MiaAzienda.Clienti
  MiaAzienda.Clienti.csproj
  Class1.cs
```

Rinominiamo `Class1.cs` in qualcosa di utile.

Esempio:

```csharp id="x0dswi"
namespace MiaAzienda.Clienti;

public class CodiceClienteFormatter
{
    public string Format(int codice)
    {
        return codice.ToString("D10");
    }
}
```

La libreria può essere referenziata da un altro progetto:

```bash id="qfuzfu"
dotnet add MiaApp.Web reference MiaAzienda.Clienti
```

Questo riferimento va bene dentro la stessa solution.

Ma se vogliamo riusare la libreria in molte solution diverse, conviene creare un pacchetto NuGet.

---

# Il file `.csproj` di una libreria pacchettizzabile

Esempio:

```xml id="h78og6"
<Project Sdk="Microsoft.NET.Sdk">

  <PropertyGroup>
    <TargetFramework>net10.0</TargetFramework>
    <ImplicitUsings>enable</ImplicitUsings>
    <Nullable>enable</Nullable>

    <PackageId>MiaAzienda.Clienti</PackageId>
    <Version>1.0.0</Version>
    <Authors>Mia Azienda</Authors>
    <Company>Mia Azienda</Company>
    <Description>Libreria condivisa per funzionalità clienti.</Description>
    <PackageTags>clienti;utility;miaazienda</PackageTags>
    <RepositoryUrl>https://github.com/miaazienda/miaazienda-clienti</RepositoryUrl>
    <PackageLicenseExpression>MIT</PackageLicenseExpression>
  </PropertyGroup>

</Project>
```

Campi importanti:

```text id="l2ch6o"
PackageId
  nome del pacchetto.

Version
  versione del pacchetto.

Authors
  autore o organizzazione.

Description
  descrizione leggibile.

PackageTags
  tag utili per ricerca e classificazione.

RepositoryUrl
  repository sorgente.

PackageLicenseExpression
  licenza.
```

# Multi-targeting

Una libreria riutilizzabile può supportare più target framework.

Esempio:

```xml id="gjkt51"
<Project Sdk="Microsoft.NET.Sdk">

  <PropertyGroup>
    <TargetFrameworks>net8.0;net10.0</TargetFrameworks>
    <ImplicitUsings>enable</ImplicitUsings>
    <Nullable>enable</Nullable>

    <PackageId>MiaAzienda.Clienti</PackageId>
    <Version>1.0.0</Version>
    <Authors>Mia Azienda</Authors>
    <Description>Libreria condivisa per funzionalità clienti.</Description>
  </PropertyGroup>

</Project>
```

Quando usare multi-targeting:

```text id="j6mwuq"
devi supportare applicazioni con versioni diverse di .NET
hai clienti interni ancora su .NET 8
vuoi usare API nuove solo su target recenti
```

Per codice condizionale:

```csharp id="q2gyg0"
#if NET10_0
    // codice specifico per .NET 10
#else
    // fallback per target precedenti
#endif
```

Regola pratica:

```text id="k3h7kd"
Supporta solo i target che ti servono davvero.
Ogni target aggiunge test, manutenzione e compatibilità da verificare.
```

# Creare un pacchetto NuGet

Per creare il pacchetto:

```bash id="tv3pvo"
dotnet pack -c Release
```

Output tipico:

```text id="go4fpd"
bin/Release/MiaAzienda.Clienti.1.0.0.nupkg
```

Oppure puoi indicare una cartella:

```bash id="e6ksse"
dotnet pack -c Release -o ./packages
```

Il file `.nupkg` è il pacchetto NuGet.

È uno zip con metadati, assembly, dipendenze e contenuti del pacchetto.

---

# Generare pacchetto al build

Puoi configurare il progetto per generare il pacchetto ad ogni build Release.

```xml id="jtunom"
<PropertyGroup>
  <GeneratePackageOnBuild>true</GeneratePackageOnBuild>
</PropertyGroup>
```

Da usare con cautela: in progetti grandi può rallentare il ciclo di sviluppo.

Più spesso si preferisce generare pacchetti in pipeline CI/CD.

# Versionamento semantico

NuGet supporta Semantic Versioning 2.0.0; la documentazione NuGet spiega che un pacchetto ha un numero di versione con eventuale suffisso pre-release e che i consumer possono specificare una versione precisa o un intervallo di versioni accettabili. ([learn.microsoft.com](https://learn.microsoft.com/en-us/nuget/concepts/package-versioning))

Formato:

```text id="yt8hp6"
MAJOR.MINOR.PATCH
```

Esempio:

```text id="uvbpco"
1.4.2
```

Significato tipico:

```text id="sww5e9"
MAJOR
  modifiche incompatibili.

MINOR
  nuove funzionalità compatibili.

PATCH
  correzioni compatibili.
```

---

## Esempi

```text id="y8o3uk"
1.0.0
  prima versione stabile.

1.0.1
  bug fix compatibile.

1.1.0
  nuova funzionalità compatibile.

2.0.0
  breaking change.
```

Pre-release:

```text id="czy8kq"
1.2.0-alpha.1
1.2.0-beta.1
1.2.0-rc.1
```

Uso tipico:

```text id="i31x5n"
alpha
  molto sperimentale.

beta
  funzionalità quasi complete ma non stabili.

rc
  release candidate.
```

---

# Versionamento e compatibilità

Quando cambi un pacchetto, chiediti:

```text id="g7v61p"
Ho rimosso API pubbliche?
Ho cambiato firma di metodi pubblici?
Ho cambiato comportamento in modo incompatibile?
Ho modificato DTO pubblici?
Ho cambiato target framework?
Ho introdotto nuove dipendenze obbligatorie?
```

Esempio di breaking change:

```csharp id="jduhm4"
public string Format(int codice)
```

diventa:

```csharp id="pt87l5"
public string Format(long codice, string prefisso)
```

Questo rompe i consumer.

Versione consigliata:

```text id="nr1bps"
da 1.x.y a 2.0.0
```

Esempio di modifica compatibile:

```csharp id="w7qjbs"
public string Format(int codice)
```

aggiungi overload:

```csharp id="ixwdga"
public string Format(int codice, string prefisso)
```

Versione consigliata:

```text id="envjkd"
da 1.2.0 a 1.3.0
```

# Pubblicazione pacchetti

Per pubblicare su nuget.org serve una API key.

Comando:

```bash id="aqiewx"
dotnet nuget push ./packages/MiaAzienda.Clienti.1.0.0.nupkg \
  --api-key <API_KEY> \
  --source https://api.nuget.org/v3/index.json
```

Microsoft documenta che, dopo aver creato il file `.nupkg`, la pubblicazione su nuget.org può essere fatta con .NET CLI o NuGet CLI usando una API key generata da nuget.org. ([learn.microsoft.com](https://learn.microsoft.com/en-us/nuget/quickstart/create-and-publish-a-package-using-the-dotnet-cli))

---

## Attenzione alla pubblicazione pubblica

Prima di pubblicare su nuget.org, verifica:

```text id="kvcpfe"
nome pacchetto corretto
nessun segreto incluso
nessun file interno
licenza corretta
README presente
versione corretta
test passati
API pubblica stabile
```

Una volta pubblicata una versione, non va trattata come un file modificabile.
Se devi correggere, pubblica una nuova versione.

# Feed privati

Non tutti i pacchetti devono essere pubblici.

Per librerie aziendali si usano spesso feed privati:

```text id="sz70wo"
Azure Artifacts
GitHub Packages
GitLab Package Registry
ProGet
NuGet Server interno
file share aziendale
feed privato su Nexus/Artifactory
```

Aggiungere un source:

```bash id="g8uqm6"
dotnet nuget add source "https://nuget.example.com/v3/index.json" \
  --name "MiaAzienda"
```

Con credenziali:

```bash id="p1n6l6"
dotnet nuget add source "https://nuget.example.com/v3/index.json" \
  --name "MiaAzienda" \
  --username "utente" \
  --password "password" \
  --store-password-in-clear-text
```

Attenzione: `--store-password-in-clear-text` è comodo ma sensibile. In CI/CD è meglio usare secret store della pipeline.

## `nuget.config`

Un progetto può avere un file `nuget.config`:

```xml id="h2st5x"
<?xml version="1.0" encoding="utf-8"?>
<configuration>
  <packageSources>
    <clear />
    <add key="nuget.org"
         value="https://api.nuget.org/v3/index.json" />

         value="https://nuget.example.com/v3/index.json" />
```

Vantaggi:

```text id="x9fwtz"
source espliciti

configurazione condivisa nel repository
```

Non inserire password nel `nuget.config` committato.

# Central Package Management

In solution grandi, gestire versioni pacchetto in ogni `.csproj` diventa scomodo.

Con Central Package Management puoi mettere le versioni in `Directory.Packages.props`.

Esempio:

```xml id="y5k9de"
<Project>
  <ItemGroup>
    <PackageVersion Include="MiaAzienda.Clienti" Version="1.2.0" />
    <PackageVersion Include="Microsoft.EntityFrameworkCore" Version="10.0.0" />
  </ItemGroup>
</Project>
```

Nel `.csproj`:

```xml id="njn5m6"
<ItemGroup>
  <PackageReference Include="MiaAzienda.Clienti" />
  <PackageReference Include="Microsoft.EntityFrameworkCore" />
</ItemGroup>
```

Questo evita versioni duplicate in molti progetti.

# Distribuzione di tipi riutilizzabili

Un pacchetto NuGet dovrebbe distribuire **tipi riutilizzabili**, non pezzi casuali di applicazione.

Buoni candidati:

```text id="v8f04n"
Value Object comuni
DTO di contratti pubblici
client SDK per API
validazioni condivise
abstractions
middleware riutilizzabili
extension methods
source generator
analyzer
```

Candidati sospetti:

```text id="jmvkxy"
entità EF Core specifiche di una sola app
DbContext applicativo completo
servizi con troppe dipendenze ambientali
logica business non realmente condivisa
classi usate da un solo progetto
```

Regola pratica:

```text id="kj2j33"
Un pacchetto deve avere un confine chiaro.
Se non sai spiegare perché esiste, forse non deve essere un pacchetto.
```

# Abstractions package

Spesso conviene separare:

```text id="fb5p2s"
MiaAzienda.Notifiche.Abstractions
MiaAzienda.Notifiche.Smtp
MiaAzienda.Notifiche.SendGrid
```

`Abstractions` contiene interfacce e tipi leggeri:

```csharp id="b657dp"
namespace MiaAzienda.Notifiche;

public interface IEmailSender
{
    Task SendAsync(
        EmailMessage message,
        CancellationToken cancellationToken);
}

public record EmailMessage(
    string To,
    string Subject,
    string Body);
```

Implementazione SMTP in un pacchetto separato:

```csharp id="s66rp6"
public class SmtpEmailSender : IEmailSender
{
    public Task SendAsync(
        EmailMessage message,
        CancellationToken cancellationToken)
    {
        // invio SMTP
        return Task.CompletedTask;
    }
}
```

Vantaggi:

```text id="ethcat"
i consumer possono dipendere solo dai contratti
le implementazioni restano opzionali
meno dipendenze transitive
più flessibilità
```

# Pacchetti e Dependency Injection

Un pacchetto riutilizzabile per ASP.NET Core spesso espone un extension method per registrare i servizi.

Esempio:

```csharp id="fmquxj"
namespace Microsoft.Extensions.DependencyInjection;

public static class NotificheServiceCollectionExtensions
{
    public static IServiceCollection AddNotificheSmtp(
        this IServiceCollection services,
        Action<SmtpOptions> configure)
    {
        services.Configure(configure);

        services.AddScoped<IEmailSender, SmtpEmailSender>();

        return services;
    }
}
```

Uso nel progetto consumer:

```csharp id="cmdzq7"
builder.Services.AddNotificheSmtp(options =>
{
    options.Host = builder.Configuration["Smtp:Host"]!;
    options.Port = 587;
});
```

Questo rende il pacchetto facile da usare.

# Source generators

Un **source generator** è un componente Roslyn che genera codice C# durante la compilazione.

Può essere distribuito come pacchetto NuGet.

Esempi d’uso:

```text id="v0mc8n"
generare mapping
generare serializer
generare codice ripetitivo
validare attributi a compile-time
generare endpoint o registrazioni
generare strongly-typed IDs
```

Microsoft descrive i source generator come generatori di codice sorgente eseguiti durante la compilazione, capaci di analizzare il codice utente e aggiungere nuovi file sorgente alla compilazione. ([learn.microsoft.com](https://learn.microsoft.com/en-us/dotnet/csharp/roslyn-sdk/source-generators-overview))

---

## Esempio minimale di source generator

Progetto:

```bash id="ui25o5"
dotnet new classlib -n MiaAzienda.Generators
```

`.csproj` tipico:

```xml id="ouwu46"
<Project Sdk="Microsoft.NET.Sdk">

  <PropertyGroup>
    <TargetFramework>netstandard2.0</TargetFramework>
    <Nullable>enable</Nullable>
    <LangVersion>latest</LangVersion>

    <PackageId>MiaAzienda.Generators</PackageId>
    <Version>1.0.0</Version>
    <Description>Source generator di esempio.</Description>
  </PropertyGroup>

  <ItemGroup>
    <PackageReference Include="Microsoft.CodeAnalysis.CSharp"
                      Version="4.13.0"
                      PrivateAssets="all" />
  </ItemGroup>

</Project>
```

Esempio generator:

```csharp id="m5jwo4"
using Microsoft.CodeAnalysis;
using Microsoft.CodeAnalysis.Text;
using System.Text;

namespace MiaAzienda.Generators;

[Generator]
public class HelloGenerator : ISourceGenerator
{
    public void Initialize(GeneratorInitializationContext context)
    {
    }

    public void Execute(GeneratorExecutionContext context)
    {
        string source = """
        namespace MiaAzienda.Generated;

        public static class GeneratedHello
        {
            public static string Message => "Hello from source generator";
        }
        """;

        context.AddSource(
            "GeneratedHello.g.cs",
            SourceText.From(source, Encoding.UTF8));
    }
}
```

---

## Consumare un source generator

Nel progetto consumer:

```xml id="b1d5h0"
<ItemGroup>
  <PackageReference Include="MiaAzienda.Generators"
                    Version="1.0.0"
                    OutputItemType="Analyzer"
                    ReferenceOutputAssembly="false"
                    PrivateAssets="all" />
</ItemGroup>
```

In molti pacchetti generator, il packaging è configurato in modo che il consumer debba solo aggiungere il `PackageReference`.

Regola pratica:

```text id="zptqrf"
Un source generator deve essere trattato come strumento di compilazione,
non come normale libreria runtime.
```

# Analyzer

Un **analyzer** Roslyn analizza il codice e produce diagnostiche: warning, errori o suggerimenti.

Esempi:

```text id="vi6x5f"
vietare DateTime.Now in certi progetti
imporre naming convention
segnalare uso di API obsolete interne
validare attributi custom
imporre regole architetturali
```

Esempio concettuale:

```text id="c9h8e3"
Warning:
  Non usare DateTime.Now. Usa IClock.
```

Gli analyzer sono molto utili in aziende e team perché trasformano convenzioni in controlli automatici.

---

## Analyzer come pacchetto NuGet

Un analyzer può essere distribuito con NuGet e referenziato così:

```xml id="rwe384"
<ItemGroup>
  <PackageReference Include="MiaAzienda.Analyzers"
                    Version="1.0.0"
                    PrivateAssets="all" />
</ItemGroup>
```

`PrivateAssets="all"` è importante perché evita che l’analyzer diventi una dipendenza transitiva runtime dei progetti consumer.

---

## Analyzer e qualità architetturale

Esempi di regole aziendali:

```text id="m5hpzc"
il Domain Layer non può referenziare Infrastructure
non usare DateTime.Now direttamente
non usare Thread.Sleep in ASP.NET Core
non usare FromSqlRaw con stringhe interpolate
non loggare proprietà chiamate Password o Token
```

Queste regole possono essere controllate automaticamente.

Regola pratica:

```text id="f6wh96"
Un analyzer è utile quando una regola è importante,
ripetitiva e verificabile staticamente.
```

# Pacchetti analyzer/source generator: attenzione

Source generator e analyzer sono potenti, ma hanno costi:

```text id="md6asg"
aumentano complessità build
possono rallentare compilazione
devono essere compatibili con versioni Roslyn/SDK
possono generare diagnostiche rumorose
richiedono test specifici
```

Usali quando il beneficio supera il costo.

# CI/CD per pacchetti

In un team professionale, i pacchetti non dovrebbero essere creati manualmente dal PC dello sviluppatore.

Pipeline tipica:

```text id="b1e4te"
restore
build
test
pack
publish artifact
push su feed
tag versione
```

Esempio concettuale:

```bash id="hjd393"
dotnet restore
dotnet build -c Release --no-restore
dotnet test -c Release --no-build
dotnet pack -c Release --no-build -o ./artifacts
dotnet nuget push ./artifacts/*.nupkg --source <feed> --api-key <token>
```

Regola pratica:

```text id="iqj0vn"
Il pacchetto ufficiale dovrebbe essere prodotto dalla pipeline,
non a mano.
```

# Separare versioni di assembly e package

In progetti semplici puoi usare solo `Version`.

In progetti più maturi puoi distinguere:

```text id="xe87x5"
PackageVersion
  versione del pacchetto NuGet.

AssemblyVersion
  versione assembly per binding/compatibilità.

FileVersion
  versione file.

InformationalVersion
  versione informativa, spesso include commit SHA.
```

Esempio:

```xml id="z3s4uk"
<PropertyGroup>
  <Version>1.4.0</Version>
  <AssemblyVersion>1.0.0.0</AssemblyVersion>
  <FileVersion>1.4.0.0</FileVersion>
  <InformationalVersion>1.4.0+abc1234</InformationalVersion>
</PropertyGroup>
```

Per molte librerie moderne basta iniziare semplice.
La distinzione diventa importante in ambienti enterprise e librerie con forte compatibilità binaria.

# Esempio completo: libreria pacchettizzata

## Creazione

```bash id="jvw3ec"
dotnet new classlib -n MiaAzienda.Codici
cd MiaAzienda.Codici
```

## Codice

```csharp id="vo30ky"
namespace MiaAzienda.Codici;

public sealed class CodiceClienteFormatter
{
    public string Format(int codice)
    {
        if (codice <= 0)
        {
            throw new ArgumentOutOfRangeException(
                nameof(codice),
                "Il codice deve essere positivo.");
        }

        return codice.ToString("D10");
    }
}
```

## `.csproj`

```xml id="zv0sax"
<Project Sdk="Microsoft.NET.Sdk">

  <PropertyGroup>
    <TargetFrameworks>net8.0;net10.0</TargetFrameworks>
    <ImplicitUsings>enable</ImplicitUsings>
    <Nullable>enable</Nullable>

    <PackageId>MiaAzienda.Codici</PackageId>
    <Version>1.0.0</Version>
    <Authors>Mia Azienda</Authors>
    <Company>Mia Azienda</Company>
    <Description>Utility aziendali per formattazione codici.</Description>
    <PackageTags>miaazienda;codici;utility</PackageTags>
    <PackageReadmeFile>README.md</PackageReadmeFile>
    <RepositoryUrl>https://github.com/miaazienda/codici</RepositoryUrl>
    <PackageLicenseExpression>MIT</PackageLicenseExpression>
```

# Errori comuni

## Pacchettizzare troppo presto

Sospetto:

```text id="t9kbdz"
creo pacchetti per codice non ancora stabile
e poi devo cambiare continuamente API pubbliche
```

Prima stabilizza il confine.

---

## Pacchetti troppo grandi

Sospetto:

```text id="gbrkwv"
MiaAzienda.Common
```

con dentro:

```text id="p2hb1l"
email
database
clienti
ordini
logging
date
file
sicurezza
validazioni
```

Meglio pacchetti più coesi.

---

## Dipendenze transitive indesiderate

Se un pacchetto piccolo porta dentro molte dipendenze, i consumer pagano il costo.

Controlla sempre:

```text id="vlzcue"
dipendenze dirette
dipendenze transitive
target framework
licenze
dimensione
compatibilità
```

---

## Breaking change senza cambiare major

Se rompi API pubbliche, incrementa la major version.

---

## Pubblicare segreti nel pacchetto

Controlla sempre il contenuto del `.nupkg`.

Puoi aprirlo come zip o ispezionarlo con strumenti NuGet.

---

## Usare sempre repository pubblici per pacchetti aziendali

Pacchetti interni devono stare su feed privati.

---

## Analyzer troppo rumorosi

Un analyzer che produce troppi warning inutili verrà ignorato.

Meglio poche regole importanti e chiare.

# Linee guida pratiche

```text id="xj76vu"
Crea una libreria
  quando hai codice realmente riutilizzabile.

Crea un pacchetto NuGet
  quando quel codice deve uscire dalla solution.

Usa PackageId chiaro
  stabile e coerente con namespace/azienda.

Usa SemVer
  patch per bug fix, minor per funzionalità compatibili,
  major per breaking change.

Pubblica pacchetti pubblici su nuget.org
  solo se sono davvero destinati all’esterno.

Usa feed privati
  per librerie aziendali.

Non committare credenziali NuGet
  usa secret store o variabili sicure della pipeline.

Usa README e metadati
  per rendere il pacchetto comprensibile.

Valuta Abstractions package
  per separare contratti da implementazioni.

Usa source generators
  per generare codice ripetitivo a compile-time.

Usa analyzer
  per imporre regole importanti automaticamente.

Produci pacchetti ufficiali da CI/CD
  non dal PC dello sviluppatore.

Testa compatibilità
  su tutti i target framework dichiarati.
```

# Riepilogo

In questo capitolo abbiamo visto come passare da codice riutilizzabile a pacchetti distribuibili.

Abbiamo trattato:

```text id="y8wtti"
creare una libreria
creare un pacchetto NuGet
versionamento semantico
pubblicazione pacchetti
feed privati
source generators
analyzer
distribuzione di tipi riutilizzabili
```

La distinzione più importante è questa:

```text id="l8ohej"
Project reference
  riuso dentro la stessa solution.

NuGet package
  riuso versionato tra solution, team e ambienti.
```

NuGet usa versioni in stile Semantic Versioning 2.0.0, e la scelta della versione comunica ai consumer se stanno installando una correzione, una nuova funzionalità compatibile o una modifica potenzialmente incompatibile. ([learn.microsoft.com](https://learn.microsoft.com/en-us/nuget/concepts/package-versioning))

La regola principale è questa:

```text id="e6qc6l"
Un pacchetto NuGet è un contratto.
Quando lo pubblichi, altri progetti dipendono dalla sua API,
dalla sua versione, dalle sue dipendenze e dalla sua compatibilità.
```

Questo capitolo collega modularità, riuso e distribuzione: non stai solo “impacchettando DLL”, stai definendo componenti riutilizzabili, versionati e mantenibili nel tempo.

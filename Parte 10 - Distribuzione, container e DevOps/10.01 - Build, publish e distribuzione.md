# 10.01 — Build, publish e distribuzione

Fino a questo punto abbiamo costruito applicazioni, API, architetture, pattern e configurazioni production-ready. Ora il progetto deve uscire dal computer dello sviluppatore.

Distribuire un’applicazione .NET significa rispondere a domande molto pratiche:

```text id="nhp8c4"
Come compilo?
Che file devo copiare?
Sul server serve installare .NET?
L'app gira su Windows o Linux?
La pubblico come servizio?
La metto dietro IIS?
La metto dietro Nginx?
La voglio single-file?
Posso usare trimming o Native AOT?
```

Il punto centrale è questo:

```text id="wzno1n"
Build e publish non sono la stessa cosa.

Build:
  compila il progetto.

Publish:
  produce il pacchetto di file pronto per essere distribuito.
```

La documentazione Microsoft descrive `dotnet publish` come il comando che compila l’applicazione, legge le dipendenze dal progetto e pubblica il set di file risultante in una directory pronta per la distribuzione. ([Microsoft Learn][1])

---

# Build Debug e Release

Quando compili un progetto .NET, puoi usare configurazioni diverse.

Le due principali sono:

```text id="jw8zpc"
Debug
  pensata per sviluppo e diagnostica.

Release
  pensata per distribuzione e prestazioni.
```

---

## Debug

```bash id="wfhc91"

```

Caratteristiche tipiche:

```text id="mp2uso"
simboli di debug
ottimizzazioni ridotte
debug più agevole
output pensato per sviluppo
```

Si usa mentre sviluppi.

---

## Release

```bash id="pmlbli"
dotnet build -c Release
```

Caratteristiche tipiche:

```text id="gtoyr3"
ottimizzazioni abilitate
output più adatto alla produzione
prestazioni migliori
debug meno diretto
```

Per pubblicare, quasi sempre si usa Release:

```bash id="zgyks5"
dotnet publish -c Release
```

Regola pratica:

```text id="d15rqi"
Debug per sviluppare.
Release per distribuire.
```

# `dotnet publish`

Il comando fondamentale per preparare una distribuzione è:

```bash id="i1d3p6"
dotnet publish -c Release -o ./publish
```

Questo produce una cartella `publish` con i file da copiare sul server.

Esempio:

```text id="gk2xkd"
publish/
  MiaApp.dll
  MiaApp.exe, su Windows in certi casi
  appsettings.json
  web.config, se ASP.NET Core per IIS
  dipendenze .dll
  file statici wwwroot
  runtimeconfig.json
  deps.json
```

Per eseguire una app framework-dependent:

```bash id="epm13k"
dotnet MiaApp.dll
```

Oppure, se è stato prodotto un eseguibile:

```bash id="ou2g0q"
./MiaApp
```

o su Windows:

```powershell id="h2d3ei"
.\MiaApp.exe
```

# Runtime Identifier

Quando pubblichi per uno specifico sistema operativo e architettura, usi un **Runtime Identifier**, spesso abbreviato in RID.

Esempi:

```text id="st9akw"
win-x64
linux-x64
linux-arm64
osx-x64
osx-arm64
```

Esempio:

```bash id="r5mulv"
dotnet publish -c Release -r win-x64
```

Oppure:

```bash id="bv1e58"
dotnet publish -c Release -r linux-x64
```

Il RID diventa importante per self-contained, single-file, Native AOT e distribuzioni specifiche per piattaforma.

# Framework-dependent deployment

Nel **framework-dependent deployment**, l’applicazione pubblicata non include il runtime .NET.

Sul server deve essere installato il runtime compatibile.

Comando:

```bash id="tvkxak"
dotnet publish -c Release -o ./publish
```

Output tipico:

```text id="rjjlsy"
MiaApp.dll
MiaApp.deps.json
MiaApp.runtimeconfig.json
dipendenze
```

Esecuzione:

```bash id="kqkvhu"
dotnet MiaApp.dll
```

---

## Vantaggi

```text id="usv3lt"
pacchetto più piccolo
aggiornamento runtime centralizzato
ideale quando controlli il server
più app possono condividere lo stesso runtime
```

## Svantaggi

```text id="nlbvnw"
sul server deve esserci .NET installato
possibili problemi se manca la versione corretta
dipendenza dall’ambiente target
```

Quando usarlo:

```text id="riytwn"
server aziendale controllato
IIS con Hosting Bundle
Linux server con runtime installato
ambienti dove gestisci bene le versioni .NET
```

# Self-contained deployment

Nel **self-contained deployment**, l’applicazione pubblicata include anche il runtime .NET.

Comando:

```bash id="v4d0xi"
dotnet publish -c Release -r win-x64 --self-contained true -o ./publish
```

Per Linux:

```bash id="nny5zo"
dotnet publish -c Release -r linux-x64 --self-contained true -o ./publish
```

La documentazione Microsoft spiega che le modalità di pubblicazione possono produrre applicazioni dipendenti dal framework o self-contained, influenzando quali file vengono distribuiti e quali prerequisiti servono sul sistema target. ([Microsoft Learn][2])

---

## Vantaggi

```text id="qq9gzz"
non richiede runtime installato sul server
versione .NET controllata dall’app
distribuzione più autonoma
utile per server non gestiti direttamente
```

## Svantaggi

```text id="hawep9"
pacchetto più grande
ogni app include il proprio runtime
aggiornamenti runtime da gestire ripubblicando l’app
serve pubblicare per ogni RID
```

Quando usarlo:

```text id="fvjcwi"
server dove non vuoi installare .NET
tool distribuiti a utenti finali
servizi standalone
ambienti con runtime non controllabile
```

# Single-file app

La pubblicazione **single-file** impacchetta l’applicazione in un singolo file eseguibile, o comunque riduce molto il numero di file da distribuire.

Esempio Windows:

```bash id="ygy1kf"
dotnet publish -c Release -r win-x64 \
  --self-contained true \
  -p:PublishSingleFile=true \
  -o ./publish
```

Esempio Linux:

```bash id="ic58ze"
dotnet publish -c Release -r linux-x64 \
  --self-contained true \
  -p:PublishSingleFile=true \
  -o ./publish
```

Microsoft documenta la pubblicazione single-file e mostra esempi con `dotnet publish -r <RID>`, sia self-contained sia framework-dependent; il single-file è controllato da proprietà come `PublishSingleFile`, `SelfContained` e `RuntimeIdentifier`. ([Microsoft Learn][3])

---

## Vantaggi

```text id="pjm034"
distribuzione più semplice
meno file da copiare
comodo per tool console
comodo per piccoli servizi
```

## Svantaggi

```text id="ygm5ze"
non sempre davvero un solo file in tutti gli scenari
attenzione a file nativi e contenuti esterni
startup e diagnostica possono cambiare
alcune librerie possono generare avvisi
```

Regola pratica:

```text id="x8kedr"
Single-file è ottimo per tool e servizi semplici.
Per web app grandi, valuta se semplifica davvero la gestione.
```

# Trimming

Il **trimming** rimuove codice non usato per ridurre la dimensione del deployment.

Comando:

```bash id="thwclg"
dotnet publish -c Release -r linux-x64 \
  --self-contained true \
  -p:PublishTrimmed=true
```

Microsoft descrive il trimming come un modello specializzato per app self-contained che riduce la dimensione del deployment rimuovendo parti non necessarie delle dipendenze e del framework. ([Microsoft Learn][4])

---

## Vantaggi

```text id="sn6p0p"
pacchetto più piccolo
utile in container
utile in ambienti con limiti di spazio
utile per app client e tool
```

## Rischi

Il trimming può rompere librerie che usano:

```text id="v0x7aj"
reflection
serializzazione dinamica
plugin caricati a runtime
dependency injection molto dinamica
codice generato
scansione assembly
```

Esempio sospetto:

```csharp id="n7cij1"
Type.GetType("MiaApp.Plugin.ClientePlugin")
```

Il trimmer potrebbe non capire che quel tipo serve.

Regola pratica:

```text id="pe0rsf"
Il trimming va testato.
Non attivarlo alla cieca su applicazioni grandi o molto dinamiche.
```

# Native AOT

**Native AOT** compila l’applicazione ahead-of-time in codice nativo.

Comando:

```bash id="o0z26s"
dotnet publish -c Release -r linux-x64 -p:PublishAot=true
```

Oppure Windows:

```bash id="rgnkcg"
dotnet publish -c Release -r win-x64 -p:PublishAot=true
```

Microsoft spiega che la pubblicazione Native AOT produce un’app self-contained compilata ahead-of-time in codice nativo, con startup più rapido e footprint di memoria minore, eseguibile anche su macchine senza runtime .NET installato. ([Microsoft Learn][5])

---

## Vantaggi

```text id="eqabhx"
startup molto rapido
minore uso memoria in certi scenari
nessun JIT a runtime
distribuzione self-contained
utile per CLI, serverless, microservizi piccoli
```

## Limiti

Native AOT richiede più attenzione con:

```text id="qkx1xb"
reflection
serializzazione dinamica
runtime code generation
Expression.Compile
plugin dinamici
alcune librerie non compatibili
alcuni scenari ASP.NET Core avanzati
```

Regola pratica:

```text id="j1j7ng"
Native AOT è una scelta progettuale e di deployment.
Va valutata all’inizio, testata e supportata dalle librerie usate.
```

# Servizi Windows

Un worker service può essere eseguito come servizio Windows.

Nel progetto:

```bash id="zsgz2r"
dotnet add package Microsoft.Extensions.Hosting.WindowsServices
```

In `Program.cs`:

```csharp id="psjv1u"
HostApplicationBuilder builder =
    Host.CreateApplicationBuilder(args);

builder.Services.AddWindowsService(options =>
{
    options.ServiceName = "MiaApp Worker";
});

builder.Services.AddHostedService<Worker>();

IHost host = builder.Build();

host.Run();
```

```bash id="y7gzu2"

```

Creazione servizio:

```powershell id="j02pdd"

```

Avvio:

```powershell id="hj5sqp"
sc.exe start "MiaApp Worker"
```

Stop:

```powershell id="zr5zpi"
sc.exe stop "MiaApp Worker"
```

Eliminazione:


# IIS

IIS è una scelta molto comune per ospitare ASP.NET Core su Windows Server.

Con ASP.NET Core, IIS lavora insieme all’**ASP.NET Core Module**. Microsoft spiega che, negli scenari IIS, l’app ASP.NET Core viene gestita dal modulo ASP.NET Core e, in hosting out-of-process, IIS inoltra le richieste a Kestrel su una porta interna; per usare IIS è necessario installare il .NET Hosting Bundle. ([Microsoft Learn][6])

---

## Passi principali per IIS

```text id="q7y6pg"
1. Installare .NET Hosting Bundle sul server.
2. Pubblicare l’app.
3. Copiare i file pubblicati sul server.
4. Creare sito o applicazione in IIS.
5. Configurare application pool.
6. Configurare binding HTTP/HTTPS.
7. Verificare permessi cartella.
8. Riavviare sito/app pool.
```

Microsoft ha una guida specifica per pubblicare un’app ASP.NET Core in IIS: installare il .NET Hosting Bundle, creare il sito IIS e distribuire l’app. ([Microsoft Learn][7])

---

## Publish per IIS

```bash id="mqbtmd"
dotnet publish -c Release -o ./publish
```

La cartella publish conterrà anche `web.config` per IIS.

Esempio concettuale:

```xml id="m4rzwu"
<configuration>
  <system.webServer>
    <handlers>
      <add name="aspNetCore"
           path="*"
           verb="*"
           modules="AspNetCoreModuleV2"
           resourceType="Unspecified" />
    </handlers>
    <aspNetCore processPath="dotnet"
                arguments=".\MiaApp.dll"
                hostingModel="inprocess" />
  </system.webServer>
</configuration>
```

Normalmente non devi scriverlo a mano: viene generato dal publish.

---

## In-process e out-of-process

Con IIS esistono due modelli:

```text id="dr6iz9"
In-process
  l’app ASP.NET Core gira nello stesso processo del worker process IIS.

Out-of-process
  IIS fa da reverse proxy verso Kestrel.
```

Microsoft indica che l’hosting in-process offre prestazioni migliori rispetto all’out-of-process perché le richieste non vengono proxate tramite loopback. ([Microsoft Learn][8])

Regola pratica:

```text id="l59ncs"
Per IIS moderno, usa in-process salvo esigenze specifiche.
```

---

## Application Pool

Per ASP.NET Core:

```text id="mmxkeg"
.NET CLR Version:
  No Managed Code
```

Questo può sembrare strano, ma è corretto: l’app gira con il runtime .NET moderno, non con il CLR classico caricato da IIS come per ASP.NET Framework.

# Systemd

Su Linux, per far partire un’app ASP.NET Core o Worker Service come servizio, si usa spesso **systemd**.

File:

```bash id="vyr0ro"

```

Esempio:

```ini id="am5o2x"

After=network.target

SyslogIdentifier=miaapp
User=www-data
Environment=ASPNETCORE_ENVIRONMENT=Production
Environment=ASPNETCORE_URLS=http://localhost:5000

WantedBy=multi-user.target
```



# Kestrel

Kestrel è il web server cross-platform integrato in ASP.NET Core.

In sviluppo può essere esposto direttamente.

In produzione spesso viene messo dietro un reverse proxy:

```text id="yymks1"
Apache

```

```bash id="l49q7k"

```

Oppure in `appsettings.Production.json`:

```json id="mzm8oy"
{
  "Kestrel": {
    "Endpoints": {
      "Http": {
        "Url": "http://localhost:5000"
      }
    }
  }
}
```

Regola pratica:

```text id="cy1zsb"

```

# Reverse proxy

Un **reverse proxy** riceve le richieste dal client e le inoltra all’applicazione interna.

Schema:

```text id="u2po2l"
Browser
  ↓ HTTPS 443
Nginx / IIS / Apache
  ↓ HTTP localhost:5000
Kestrel / ASP.NET Core
```

Vantaggi:

```text id="pftvk8"
terminazione TLS
compressione
static file caching
load balancing
routing tra più app
header forwarding
rate limiting
logging accessi
gestione certificati
```

Attenzione agli header inoltrati:

```text id="kft2pm"
X-Forwarded-For
X-Forwarded-Proto
Host
```

In ASP.NET Core spesso serve configurare Forwarded Headers quando l’app è dietro proxy, per ricostruire schema HTTPS e IP reale del client.

---

# Nginx

Nginx è una scelta comune su Linux come reverse proxy davanti a Kestrel.

Esempio configurazione:

```nginx id="votx7s"
server {
    listen 80;
    server_name example.com;

    location / {
        proxy_pass         http://localhost:5000;
        proxy_http_version 1.1;

        proxy_set_header   Upgrade $http_upgrade;
        proxy_set_header   Connection keep-alive;
        proxy_set_header   Host $host;
        proxy_cache_bypass $http_upgrade;

        proxy_set_header   X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header   X-Forwarded-Proto $scheme;
    }
}
```

Test configurazione:

```bash id="z6frob"
sudo nginx -t
```

Reload:

```bash id="cv161k"
sudo systemctl reload nginx
```

La guida Microsoft per Linux con Nginx mostra proprio la configurazione di Nginx come reverse proxy che inoltra il traffico HTTP a un’app ASP.NET Core in esecuzione su Kestrel. ([Microsoft Learn][9])

---

# Configurazione ASP.NET Core dietro proxy

In `Program.cs`:

```csharp id="x7q0cu"
using Microsoft.AspNetCore.HttpOverrides;

builder.Services.Configure<ForwardedHeadersOptions>(options =>
{
    options.ForwardedHeaders =
        ForwardedHeaders.XForwardedFor |
        ForwardedHeaders.XForwardedProto;
});

var app = builder.Build();

app.UseForwardedHeaders();

app.UseHttpsRedirection();
```

Questo è importante quando Nginx termina HTTPS ma l’app riceve traffico HTTP da `localhost`.

Senza forwarded headers, l’app potrebbe pensare che la richiesta originale sia HTTP e generare URL o redirect sbagliati.

# Strategia di aggiornamento

Distribuire non significa solo copiare file.

Bisogna pensare anche agli aggiornamenti.

## Stop/copy/start

Semplice:

```text id="e3gkrg"
ferma app
copia file
avvia app
```

Va bene per app interne con downtime accettabile.

## Blue/green

```text id="tl4wrg"
ambiente blue attivo
ambiente green aggiornato
switch traffico verso green
rollback tornando a blue
```

## Rolling deployment

```text id="a55at2"
aggiorni un’istanza alla volta
il servizio resta disponibile
```

## Canary

```text id="tmjkas"
rilasci a una piccola percentuale di utenti
osservi metriche/errori
poi aumenti progressivamente
```

Questi modelli saranno approfonditi nella parte DevOps/CI-CD.

## Pubblicare Debug

Sbagliato:

```bash id="ybt72e"
dotnet publish -c Debug
```

In produzione usare Release.

---

## Non specificare RID quando serve

Per self-contained, single-file e Native AOT, specifica il runtime target:

```bash id="ctsvmq"
-r linux-x64
-r win-x64
```

---

## Attivare trimming senza test

Il trimming può rompere codice dinamico. Testare sempre.

---

## Pensare che self-contained aggiorni da solo il runtime

No.

Se distribuisci self-contained, l’app include il runtime.
Per aggiornare il runtime devi ripubblicare l’app.

---

## Esporre Kestrel direttamente senza motivo

In molti scenari production è meglio usare un reverse proxy davanti.

---

## Dimenticare systemd o servizio Windows

Se avvii l’app manualmente da terminale, al logout o al riavvio si ferma.

Serve un service manager.

---

## Non configurare forwarded headers

Dietro reverse proxy, potresti avere problemi con HTTPS, redirect, URL assoluti e IP client.

---

# Linee guida pratiche

```text id="p0hb4a"
Usa Debug
  per sviluppo.

Usa Release
  per distribuzione.

Usa dotnet publish
  per creare l’output da consegnare.

Usa framework-dependent
  se il server ha runtime .NET gestito correttamente.

Usa self-contained
  se vuoi distribuire runtime insieme all’app.

Usa single-file
  per semplificare tool e servizi piccoli.

Usa trimming
  solo dopo test accurati.

Usa Native AOT
  quando startup, memoria e self-contained nativo sono importanti
  e le librerie sono compatibili.

Su Windows web
  IIS + Hosting Bundle è una scelta comune.

Su Linux web
  Kestrel + systemd + Nginx è una scelta comune.

Per worker
  usa Windows Service o systemd.

Dietro reverse proxy
  configura forwarded headers.

Non distribuire sorgenti o segreti
  distribuisci solo output publish.
```

# Riepilogo

In questo capitolo abbiamo visto come portare un’applicazione .NET fuori dal computer dello sviluppatore.

Abbiamo trattato:

```text id="f749me"
Build Debug/Release
dotnet publish
Framework-dependent deployment
Self-contained deployment
Single-file app
Trimming
Native AOT
Distribuzione Windows
Distribuzione Linux
Servizi Windows
Systemd
IIS
Reverse proxy
Nginx
```

La distinzione più importante è questa:

```text id="f16p3h"
Build
  produce output compilato per sviluppo o verifica.

Publish
  produce il pacchetto pronto per essere distribuito.
```

La scelta del modello di pubblicazione dipende dal controllo che hai sull’ambiente target: framework-dependent richiede un runtime installato, self-contained include il runtime, single-file semplifica la consegna, trimming riduce le dimensioni ma richiede test, Native AOT produce codice nativo con startup rapido ma impone vincoli di compatibilità. ([Microsoft Learn][2])

La regola pratica finale è questa:

```text id="qlcno4"
Una distribuzione professionale non è “copiare qualche DLL”.
È scegliere un modello di publish, configurare l’ambiente,
proteggere i segreti, gestire il processo applicativo,
mettere l’app dietro un hosting affidabile e prevedere aggiornamento,
diagnostica e rollback.
```

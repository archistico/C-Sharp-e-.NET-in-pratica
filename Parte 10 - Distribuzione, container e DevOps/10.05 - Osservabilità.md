# 10.05 — Osservabilità

Nei capitoli precedenti abbiamo visto build, publish, Docker e CI/CD. Ora affrontiamo un tema indispensabile per qualunque applicazione reale: **osservabilità**.

Un’applicazione può compilare, passare i test e fare deploy correttamente, ma questo non basta. Una volta in produzione, dobbiamo poter rispondere a domande come:

```text id="m6acb1"
L'applicazione è viva?
Sta rispondendo lentamente?
Quale endpoint sta generando errori?
Il database è raggiungibile?
Quale richiesta ha causato questa eccezione?
Quali servizi sono coinvolti?
Il problema è nell'API, nel database, nella rete o in un servizio esterno?
```

Il punto centrale è questo:

```text id="v26t7n"
L’osservabilità è la capacità di capire cosa sta succedendo
dentro un sistema osservando i segnali che produce:
log, metriche, trace e health check.
```

OpenTelemetry definisce sé stesso come un framework open source e vendor-neutral per instrumentare, generare, raccogliere ed esportare dati di telemetria come trace, metriche e log. ([OpenTelemetry][1])

---

# Monitoring e observability

I due termini sono collegati, ma non identici.

```text id="mgmgs6"
Monitoring
  controlla valori noti:
  CPU alta, errori HTTP 500, servizio down, disco pieno.

Observability
  permette di investigare problemi non previsti:
  perché questa richiesta è lenta?
  quale dipendenza esterna sta fallendo?
  perché solo alcuni utenti hanno errori?
```

Il monitoring risponde a:

```text id="n6wrz3"
Il sistema sta bene?
```

L’osservabilità aiuta a rispondere a:

```text id="qoq1y8"
Perché il sistema non sta bene?
```

---

# I tre pilastri classici

Tradizionalmente si parla di tre pilastri:

```text id="btxas4"
Logging
  eventi testuali o strutturati.

Metrics
  numeri misurabili nel tempo.

Tracing
  percorso di una richiesta attraverso componenti e servizi.
```

A questi, nelle applicazioni operative, aggiungiamo anche:

```text id="ea9fdm"
Health checks
  stato di salute dell’applicazione e delle sue dipendenze.

Correlation ID
  identificativo comune per collegare log, trace e richieste.
```

# Logging

Il **logging** registra eventi accaduti nell’applicazione.

Esempi:

```text id="pgql04"
utente autenticato
ordine creato
errore durante chiamata a servizio esterno
timeout database
job completato
file importato
validazione fallita
```

ASP.NET Core supporta logging strutturato ad alte prestazioni tramite `ILogger`, con provider configurabili verso destinazioni diverse. ([Microsoft Learn][2])

---

## Logging con `ILogger`

Esempio:

```csharp id="tu2s9u"
public class OrdineService
{
    private readonly ILogger<OrdineService> logger;

    public OrdineService(ILogger<OrdineService> logger)
    {
        this.logger = logger;
    }

    public async Task ConfermaAsync(
        int ordineId,
        CancellationToken cancellationToken)
    {
        logger.LogInformation(
            "Conferma ordine {OrdineId} iniziata",
            ordineId);

        try
        {
            // logica applicativa

            logger.LogInformation(
                "Ordine {OrdineId} confermato",
                ordineId);
        }
        catch (Exception ex)
        {
            logger.LogError(
                ex,
                "Errore durante conferma ordine {OrdineId}",
                ordineId);

            throw;
        }
    }
}
```

Nota importante:

```text id="iv2svf"
Non concatenare stringhe nei log.
Usa template strutturati con proprietà.
```

Meglio:

```csharp id="csqyfe"
logger.LogInformation(
    "Cliente {ClienteId} aggiornato",
    clienteId);
```

Peggio:

```csharp id="mywwzd"
logger.LogInformation(
    "Cliente " + clienteId + " aggiornato");
```

Con logging strutturato, `ClienteId` diventa una proprietà interrogabile.

---

# Livelli di log

I livelli più comuni sono:

```text id="e7iw3w"
Trace
  dettaglio massimo, raramente in produzione.

Debug
  utile in sviluppo o diagnosi mirata.

Information
  eventi significativi del flusso normale.

Warning
  situazione anomala ma non bloccante.

Error
  errore che impedisce una specifica operazione.

Critical
  errore grave che compromette l’applicazione o il sistema.
```

Esempi:

```csharp id="c2g617"
logger.LogInformation(
    "Import clienti completato. Righe elaborate: {Count}",
    count);

logger.LogWarning(
    "Cliente {ClienteId} non trovato durante sincronizzazione",
    clienteId);

logger.LogError(
    exception,
    "Errore durante chiamata al servizio pagamenti");
```

Regola pratica:

```text id="s6vk4w"
Un log deve aiutare una persona a diagnosticare un problema.
Se non aiuta, è rumore.
```

# Logging centralizzato

In produzione non basta scrivere log su file locali o console.

Con più istanze, container o server, i log devono essere raccolti centralmente.

Esempi di destinazioni:

```text id="o279c1"
Application Insights
ELK / Elastic Stack
Seq
Grafana Loki
Splunk
Datadog
New Relic
CloudWatch
Azure Monitor
```

Schema:

```text id="kt0z7y"
Applicazione
  produce log

Collector / Agent
  raccoglie log

Sistema centralizzato
  indicizza, cerca, filtra, correla, genera alert
```

Vantaggi:

```text id="e6lg2t"
ricerca su più istanze
dashboard
alert
correlazione con trace
storico
audit tecnico
diagnosi incidenti
```

# Logging sicuro

Non loggare:

```text id="x7cyyx"
password
token
cookie
Authorization header
API key
codici OTP
dati di carte
documenti completi
connection string
payload sensibili integrali
```

Esempio sbagliato:

```csharp id="uplxcb"
logger.LogInformation(
    "Login request: {@Request}",
    request);
```

se `request` contiene password.

Meglio:

```csharp id="vcz9nl"
logger.LogInformation(
    "Tentativo login per utente {UserName}",
    request.UserName);
```

Regola pratica:

```text id="qbl441"
Loggare abbastanza per diagnosticare,
ma non abbastanza da esporre dati sensibili.
```

# HTTP logging

ASP.NET Core include un middleware di HTTP logging che può registrare informazioni su request e response, inclusi header, proprietà comuni e, se configurato, body. Microsoft avverte implicitamente che bisogna filtrare con attenzione cosa viene loggato. ([Microsoft Learn][3])

Esempio:

```csharp id="y6sg66"
builder.Services.AddHttpLogging(options =>
{
    options.LoggingFields =
        HttpLoggingFields.RequestMethod |
        HttpLoggingFields.RequestPath |
        HttpLoggingFields.ResponseStatusCode;
});

var app = builder.Build();

app.UseHttpLogging();
```

Attenzione:

```text id="cdwkqy"
Loggare body di request/response può esporre dati sensibili
e aumentare molto volume e costi.
```

# Metriche

Le **metriche** sono misure numeriche raccolte nel tempo.

Esempi:

```text id="dta9uj"
numero richieste HTTP
durata media delle richieste
percentuale errori
numero ordini creati
tempo query database
numero messaggi in coda
memoria usata
CPU
GC collections
thread pool queue length
```

Le metriche servono per:

```text id="irbas6"
dashboard
alert
analisi trend
capacity planning
SLO/SLA
diagnosi di regressioni
```

OpenTelemetry .NET include supporto per generare e raccogliere metriche, oltre a log e trace. ([OpenTelemetry][4])

---

## Tipi di metriche

Tipi frequenti:

```text id="eg1qrk"
Counter
  valore che cresce.
  Esempio: numero richieste.

Gauge
  valore istantaneo.
  Esempio: messaggi in coda.

Histogram
  distribuzione di valori.
  Esempio: durata richieste.

UpDownCounter
  valore che può salire o scendere.
  Esempio: connessioni attive.
```

---

## Metriche custom in .NET

Esempio con `Meter`:

```csharp id="e863ni"
using System.Diagnostics.Metrics;

public class OrdiniMetrics
{
    private readonly Counter<long> ordiniCreati;

    public OrdiniMetrics(IMeterFactory meterFactory)
    {
        Meter meter = meterFactory.Create("MiaApp.Ordini");

        ordiniCreati = meter.CreateCounter<long>(
            "ordini_creati");
    }

    public void OrdineCreato(string canale)
    {
        ordiniCreati.Add(
            1,
            new KeyValuePair<string, object?>("canale", canale));
    }
}
```

Registrazione:

```csharp id="jzfgji"
builder.Services.AddSingleton<OrdiniMetrics>();
```

Uso:

```csharp id="lyyqaf"
metrics.OrdineCreato("web");
```

---

# Attenzione alla cardinalità

Le metriche hanno label/tag.

Esempio:

```text id="g4o4sh"
endpoint = /api/ordini
status_code = 200
method = GET
```

Attenzione a non usare tag con cardinalità enorme:

```text id="dw9ayw"
clienteId
email
ordineId
codice fiscale
requestId
```

Perché ogni valore distinto può creare una nuova serie temporale.

Regola pratica:

```text id="hlic6e"
Nelle metriche usa tag con pochi valori possibili.
Non usare identificativi unici come label.
```

# Tracing distribuito

Il **tracing distribuito** ricostruisce il percorso di una richiesta attraverso più componenti.

Esempio:

```text id="ykpmh5"
Browser
  ↓
API Gateway
  ↓
Ordini API
  ↓
Database
  ↓
Pagamenti API
  ↓
Message broker
```

Un trace è composto da più **span**.

```text id="a4af8d"
Trace
  rappresenta l’intera richiesta.

Span
  rappresenta una singola operazione dentro il trace.
```

Esempio:

```text id="mh8wck"
TraceId = abc123

Span 1: HTTP POST /api/ordini
Span 2: query SQL Clienti
Span 3: query SQL Ordini
Span 4: HTTP POST /pagamenti
Span 5: publish messaggio OrdineCreato
```

OpenTelemetry .NET fornisce API e SDK per tracing, log e metriche, permettendo di capire comportamento e performance del software. ([OpenTelemetry][5])

---

# Perché il tracing è fondamentale

Senza tracing:

```text id="nmzdb0"
vedo che una richiesta è lenta
ma non so dove perde tempo
```

Con tracing:

```text id="lesmj6"
vedo che:
  API = 80 ms
  query clienti = 15 ms
  query ordini = 30 ms
  servizio pagamenti = 2200 ms
```

Risultato:

```text id="dhvbd7"
il problema non è l’API,
è la dipendenza esterna pagamenti.
```

# Correlation ID

Il **Correlation ID** è un identificativo usato per collegare log, request e trace.

Esempio:

```text id="p1u30j"
X-Correlation-ID: 7f6d8c2e2f6144df9d2c7b4d79d2cf35
```

Middleware semplice:

```csharp id="fjxfr0"
app.Use(async (context, next) =>
{
    string correlationId =
        context.Request.Headers.TryGetValue(
            "X-Correlation-ID",
            out var value)
            ? value.ToString()
            : Guid.NewGuid().ToString("N");

    context.Response.Headers["X-Correlation-ID"] = correlationId;

    using (logger.BeginScope(
        new Dictionary<string, object>
        {
            ["CorrelationId"] = correlationId
        }))
    {
        await next();
    }
});
```

In ambienti moderni, il tracing W3C usa identificativi come `traceparent`, ma un correlation ID applicativo resta spesso utile per supporto, log e comunicazione con utenti o operatori.

Regola pratica:

```text id="w29m2j"
Ogni richiesta importante deve essere rintracciabile.
```

# OpenTelemetry

**OpenTelemetry**, spesso abbreviato OTel, è lo standard moderno per raccogliere log, metriche e trace in modo vendor-neutral.

Vantaggio principale:

```text id="icyfns"
strumenti l’app una volta
poi puoi esportare verso backend diversi:
Application Insights, Jaeger, Prometheus, Grafana, Datadog, New Relic, ecc.
```

OpenTelemetry .NET introduce l’uso di API e SDK per generare e raccogliere telemetry data come metrics, logs e traces. ([OpenTelemetry][4])

---

## Pacchetti tipici

Per una API ASP.NET Core:

```bash id="r0m2s2"
dotnet add package OpenTelemetry.Extensions.Hosting
dotnet add package OpenTelemetry.Instrumentation.AspNetCore
dotnet add package OpenTelemetry.Instrumentation.Http
dotnet add package OpenTelemetry.Instrumentation.Runtime
dotnet add package OpenTelemetry.Exporter.OpenTelemetryProtocol
```

Per EF Core o SQL, a seconda del caso, si possono aggiungere instrumentation specifiche disponibili nell’ecosistema OpenTelemetry.

---

## Configurazione base OpenTelemetry

```csharp id="z9k40f"
using OpenTelemetry.Metrics;
using OpenTelemetry.Resources;
using OpenTelemetry.Trace;

builder.Services.AddOpenTelemetry()
    .ConfigureResource(resource =>
    {
        resource.AddService(
            serviceName: "MiaApp.Api",
            serviceVersion: "1.0.0");
    })
    .WithTracing(tracing =>
    {
        tracing
            .AddAspNetCoreInstrumentation()
            .AddHttpClientInstrumentation()
            .AddOtlpExporter();
    })
    .WithMetrics(metrics =>
    {
        metrics
            .AddAspNetCoreInstrumentation()
            .AddHttpClientInstrumentation()
            .AddRuntimeInstrumentation()
            .AddOtlpExporter();
    });
```

Questa configurazione invia trace e metriche a un collector OTLP, se configurato.

# Health checks

Gli **health checks** indicano se l’applicazione e le sue dipendenze sono in salute.

Esempio base:

```csharp id="m4ws7a"
builder.Services.AddHealthChecks();

var app = builder.Build();

app.MapHealthChecks("/health");
```

Endpoint:

```text id="oehf58"
/health
```

ASP.NET Core include middleware Health Checks e permette di registrare controlli per verificare stato dell’app e dipendenze; ad esempio, una probe verso database può eseguire una query come `SELECT 1` per verificare che il database risponda. ([Microsoft Learn][9])

---

## Liveness e readiness

In produzione conviene distinguere:

```text id="vi41ev"
Liveness
  il processo è vivo?
  Se fallisce, l’orchestratore può riavviare il container.

Readiness
  l’app è pronta a ricevere traffico?
  Se fallisce, l’orchestratore non dovrebbe mandarle richieste.
```

Esempio:

```csharp id="ajwp86"
builder.Services.AddHealthChecks()
    .AddCheck("self", () =>
        HealthCheckResult.Healthy(),
        tags: ["live"]);

var app = builder.Build();

app.MapHealthChecks("/health/live", new HealthCheckOptions
{
    Predicate = check =>
        check.Tags.Contains("live")
});

app.MapHealthChecks("/health/ready", new HealthCheckOptions
{
    Predicate = check =>
        !check.Tags.Contains("live")
});
```

Microsoft documenta la possibilità di filtrare health checks tramite `Predicate`, così da eseguire solo un sottoinsieme di controlli registrati. ([Microsoft Learn][10])

---

## Health check per database

Esempio concettuale con check custom:

```csharp id="zjkv27"
public class DatabaseHealthCheck : IHealthCheck
{
    private readonly AppDbContext dbContext;

    public DatabaseHealthCheck(AppDbContext dbContext)
    {
        this.dbContext = dbContext;
    }

    public async Task<HealthCheckResult> CheckHealthAsync(
        HealthCheckContext context,
        CancellationToken cancellationToken = default)
    {
        try
        {
            bool canConnect = await dbContext.Database
                .CanConnectAsync(cancellationToken);

            return canConnect
                ? HealthCheckResult.Healthy()
                : HealthCheckResult.Unhealthy("Database non raggiungibile.");
        }
        catch (Exception ex)
        {
            return HealthCheckResult.Unhealthy(
                "Errore connessione database.",
                ex);
        }
    }
}
```

Registrazione:

```csharp id="ntb55t"
builder.Services.AddHealthChecks()
    .AddCheck<DatabaseHealthCheck>("database");
```

# Health checks: attenzione

Un health check non deve diventare troppo pesante.

Sbagliato:

```text id="ks6i9w"
health check che esegue report complessi
health check che modifica dati
health check che chiama 15 servizi esterni con timeout lunghi
```

Meglio:

```text id="qqpity"
controlli rapidi
timeout brevi
separare liveness e readiness
verificare solo dipendenze necessarie
```

Regola pratica:

```text id="royo2q"
Un health check deve dire se l’app può funzionare,
non deve fare diagnostica completa.
```

# Diagnosi di problemi in produzione

Quando arriva un problema in produzione, non bisogna partire a caso.

Procedura ragionata:

```text id="vap2zx"
1. Capire impatto
   quanti utenti? quali endpoint? da quando?

2. Controllare health check
   app viva? database raggiungibile? dipendenze ok?

3. Guardare metriche
   error rate, latenza, traffico, CPU, memoria, GC.

4. Cercare log correlati
   usando correlation ID, user id tecnico, request path, timestamp.

5. Analizzare trace
   capire dove la richiesta perde tempo o fallisce.

6. Verificare deploy recenti
   versione, configurazione, feature flag, migrazioni.

7. Isolare dipendenza
   database, API esterna, code, storage, rete.

8. Mitigare
   rollback, feature flag off, scaling, circuit breaker, fix configurazione.

9. Correggere
   patch, test, deploy controllato.

10. Post-mortem
   cosa mancava a log, metriche, alert, test o processo?
```

# Osservabilità in microservizi

Nei microservizi l’osservabilità diventa indispensabile.

Senza tracing distribuito:

```text id="e9y56u"
una richiesta attraversa 8 servizi
fallisce il quinto
i log sono sparsi
nessuno sa dove guardare
```

Con tracing:

```text id="ggwvfl"
un unico TraceId collega tutti gli span
vedo tempi, errori e dipendenze
posso isolare il servizio problematico
```

Regola pratica:

```text id="ex3yvz"
Prima di fare microservizi seriamente,
devi avere logging centralizzato,
metriche, tracing distribuito e correlation ID.
```

# Esempio completo: setup osservabilità base

## Pacchetti

```bash id="hsclsu"
dotnet add package OpenTelemetry.Extensions.Hosting
dotnet add package OpenTelemetry.Instrumentation.AspNetCore
dotnet add package OpenTelemetry.Instrumentation.Http
dotnet add package OpenTelemetry.Instrumentation.Runtime
dotnet add package OpenTelemetry.Exporter.OpenTelemetryProtocol
dotnet add package OpenTelemetry.Exporter.Prometheus.AspNetCore
```

## Program.cs

```csharp id="jtckgp"
using OpenTelemetry.Metrics;
using OpenTelemetry.Resources;
using OpenTelemetry.Trace;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddHealthChecks();

builder.Services.AddOpenTelemetry()
    .ConfigureResource(resource =>
    {
        resource.AddService(
            serviceName: builder.Environment.ApplicationName,
            serviceVersion: "1.0.0",
            serviceInstanceId: Environment.MachineName);
    })
    .WithTracing(tracing =>
    {
        tracing
            .AddAspNetCoreInstrumentation()
            .AddHttpClientInstrumentation()
            .AddOtlpExporter();
    })
    .WithMetrics(metrics =>
    {
        metrics
            .AddAspNetCoreInstrumentation()
            .AddHttpClientInstrumentation()
            .AddRuntimeInstrumentation()
            .AddPrometheusExporter()
            .AddOtlpExporter();
    });

var app = builder.Build();

app.MapHealthChecks("/health");
app.MapPrometheusScrapingEndpoint("/metrics");

app.MapGet("/api/ping", () =>
{
    return Results.Ok(new
    {
        Message = "pong",
        Time = DateTimeOffset.UtcNow
    });
});

app.Run();
```

Endpoint disponibili:

```text id="yzy03b"
/health
/metrics
/api/ping
```

# Log, metriche e trace insieme

I tre segnali devono collaborare.

Esempio:

```text id="c6b5xf"
Metriche
  dicono che la latenza è aumentata.

Trace
  mostrano quale dipendenza è lenta.

Log
  spiegano l’errore specifico o il contesto applicativo.
```

Se hai solo log:

```text id="us7a03"
rischi di cercare ago nel pagliaio.
```

Se hai solo metriche:

```text id="s5gnd4"
sai che c’è un problema, ma non sai perché.
```

Se hai solo trace:

```text id="juligx"
vedi percorsi tecnici, ma potresti perdere contesto business.
```

# Errori comuni

## Loggare troppo

Troppi log:

```text id="a2ivgs"
aumentano costi
nascondono segnali importanti
rallentano diagnosi
possono esporre dati sensibili
```

---

## Loggare troppo poco

Pochi log:

```text id="n8yykf"
non si capisce cosa è successo
gli errori non hanno contesto
supporto e sviluppatori lavorano alla cieca
```

---

## Usare log come unico strumento

I log non sostituiscono metriche e trace.

---

## Non avere correlation ID

Senza correlation ID, collegare eventi della stessa richiesta diventa difficile.

---

## Health check troppo pesanti

Un health check non deve stressare il sistema.

---

## Metriche con cardinalità enorme

Non usare `UserId`, `Email`, `OrderId` come label.

---

## Nessun alert

Una dashboard senza alert richiede che qualcuno la guardi sempre.

---

## Alert rumorosi

Alert inutili portano a ignorare anche quelli importanti.

---

## Osservabilità solo dopo l’incidente

Aggiungere osservabilità dopo il problema è sempre più costoso.

---

# Linee guida pratiche

```text id="mfg9st"
Usa logging strutturato
  con proprietà interrogabili.

Centralizza i log
  non lasciarli solo sui singoli server.

Non loggare segreti
  né payload sensibili completi.

Raccogli metriche tecniche e business
  non solo CPU e memoria.

Usa tracing distribuito
  soprattutto con API, microservizi e dipendenze esterne.

Propaga correlation ID o trace context
  tra servizi.

Usa OpenTelemetry
  per non legarti subito a un solo vendor.

Esponi health checks
  separando liveness e readiness.

Usa Prometheus e Grafana
  per metriche e dashboard, se adatti al tuo stack.

Usa Jaeger o strumenti equivalenti
  per visualizzare trace distribuiti.

Usa Application Insights
  se sei in ecosistema Azure.

Configura alert azionabili
  pochi, chiari e legati a impatti reali.

Progetta osservabilità prima della produzione
  non dopo il primo incidente.
```

# Riepilogo

In questo capitolo abbiamo visto perché l’osservabilità è un requisito fondamentale per applicazioni production-ready.

Abbiamo trattato:

```text id="sq2h1s"
logging centralizzato
metriche
tracing distribuito
OpenTelemetry
health checks
Prometheus
Grafana
Application Insights
Jaeger
Correlation ID
diagnosi di problemi in produzione
```

La distinzione principale è questa:

```text id="daeeve"
Log
  raccontano eventi e contesto.

Metriche
  misurano andamento e salute nel tempo.

Trace
  mostrano il percorso di una richiesta.

Health checks
  indicano se l’app è viva e pronta.

Correlation ID
  collega segnali diversi della stessa operazione.
```

OpenTelemetry permette di instrumentare applicazioni .NET e raccogliere log, metriche e trace con un approccio standard e vendor-neutral; Microsoft mostra anche un esempio pratico con OpenTelemetry, Prometheus, Grafana e Jaeger. ([OpenTelemetry][4])

La regola pratica finale è questa:

```text id="nnf4x1"
In produzione non basta sapere che qualcosa non funziona.
Bisogna sapere dove, quando, per chi, con quale impatto
e attraverso quale catena di componenti.
```

Senza osservabilità, ogni incidente diventa investigazione manuale. Con osservabilità ben progettata, il sistema produce gli indizi necessari per intervenire rapidamente e con meno rischio.

# 10.04 — CI/CD

Nei capitoli precedenti abbiamo visto build, publish, distribuzione, NuGet, packaging e Docker. Ora uniamo questi concetti in un flusso professionale: **CI/CD**.

CI/CD significa automatizzare il percorso che porta il codice dal repository a un artifact verificato e, quando opportuno, a un ambiente di esecuzione.

Il punto centrale è questo:

```text id="st6ay5"
CI/CD non è solo “fare deploy automatico”.
È rendere ripetibile, verificabile e tracciabile tutto il processo:
build, test, packaging, pubblicazione, deploy e rollback.
```

---

# Che cosa significa CI/CD

CI/CD può indicare due o tre concetti collegati:

```text id="upx5d7"
Continuous Integration
  integrare spesso il codice e verificarlo automaticamente.

Continuous Delivery
  produrre sempre un artifact pronto per essere rilasciato.

Continuous Deployment
  distribuire automaticamente in produzione ogni modifica valida.
```

Nel lavoro quotidiano, la differenza pratica è questa:

```text id="o02g7j"
CI
  build e test automatici a ogni push o pull request.

CD, delivery
  artifact pronto e deploy verso ambienti controllati, spesso con approvazione.

CD, deployment
  deploy automatico fino alla produzione.
```

Regola pratica:

```text id="hme361"
Non partire dal deploy automatico in produzione.
Parti da build e test automatici affidabili.
```

---

# Pipeline

Una **pipeline** è la descrizione automatizzata dei passaggi necessari per validare, impacchettare e distribuire il software.

Esempio concettuale:

```text id="v0a42x"
Commit
  ↓
Restore
  ↓
Build
  ↓
Test
  ↓
Publish
  ↓
Create artifact
  ↓
Deploy to staging
  ↓
Manual approval
  ↓
Deploy to production
```

In GitHub Actions, un workflow è un processo automatizzato configurato in YAML e composto da uno o più job; il file YAML viene salvato nel repository. ([GitHub Docs][1])

In Azure Pipelines, le pipeline YAML permettono di implementare CI/CD e descrivono build, test e deploy usando stage, job e step. ([Microsoft Learn][2])

# Build automatica

La build automatica deve rispondere a una domanda semplice:

```text id="fvy75f"
Il codice compila in un ambiente pulito?
```

Comandi tipici per .NET:

```bash id="jvygh7"
dotnet restore
dotnet build -c Release --no-restore
```

Una build in pipeline è diversa dalla build sul PC dello sviluppatore.

Sul PC potresti avere:

```text id="y6d13j"
file non committati
pacchetti già in cache
variabili d’ambiente locali
SDK installati a mano
configurazioni residue
```

In pipeline l’ambiente dovrebbe essere il più possibile pulito e ripetibile.

---

## Esempio script build

```bash id="knu0f9"
dotnet restore
dotnet build -c Release --no-restore
```

Se il progetto usa warning importanti, puoi decidere di trattarli come errori:

```bash id="xuv4o7"
dotnet build -c Release --no-restore -warnaserror
```

Questo rende la pipeline più severa.

# Test automatici

Dopo la build, la pipeline deve eseguire i test.

```bash id="m5zt1p"
dotnet test -c Release --no-build
```

Con output test:

```bash id="k8dix2"
dotnet test -c Release --no-build \
  --logger "trx" \
  --results-directory ./TestResults
```

Obiettivo:

```text id="aeb3jd"
se i test falliscono,
il codice non deve avanzare verso il deploy.
```

Tipi di test:

```text id="pc9lbu"
unit test
integration test
test API
test database
test containerizzati
test end-to-end, se presenti
```

Regola pratica:

```text id="cxxcyr"
La pipeline deve fallire presto.
Prima fallisce, meno costa correggere.
```

# Pubblicazione artifact

Un **artifact** è il risultato prodotto dalla pipeline: può essere una cartella publish, un pacchetto NuGet, una image Docker, un file zip o un report.

Esempi:

```text id="ia61db"
publish.zip
MiaApp.Api.1.4.0.nupkg
miaapp-api:1.4.0
coverage-report
test-results.trx
```

GitHub Actions definisce gli artifact come file o raccolte di file prodotti durante un workflow, utili per conservare output di build/test o condividerli tra job dello stesso workflow. ([GitHub Docs][3])

---

## Publish .NET come artifact

```bash id="h420nf"
dotnet publish src/MiaApp.Api/MiaApp.Api.csproj \
  -c Release \
  -o ./artifacts/publish
```

Creazione zip:

```bash id="ecx1ff"
cd ./artifacts/publish
zip -r ../miaapp-api.zip .
```

L’artifact deve essere:

```text id="mmpv9e"
versionato
tracciabile
riproducibile
associato a commit e build
```

## Pipeline CI per .NET

```yaml id="fl327p"
name: CI

on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main

permissions:
  contents: read

jobs:
  build-test:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup .NET
        uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '10.0.x'

      - name: Restore
        run: dotnet restore

      - name: Build
        run: dotnet build -c Release --no-restore

      - name: Test
        run: dotnet test -c Release --no-build --logger "trx" --results-directory TestResults

      - name: Publish test results
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: test-results
          path: TestResults
```

Questo workflow:

```text id="v3t4jx"
parte su push e pull request
installa .NET
ripristina pacchetti
compila
esegue test
pubblica risultati come artifact
```

## Build + publish artifact

```yaml id="fe4ltc"
name: Build Artifact

on:
  push:
    branches:
      - main

permissions:
  contents: read

jobs:
  publish:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '10.0.x'

      - name: Restore
        run: dotnet restore

      - name: Test
        run: dotnet test -c Release --no-restore

      - name: Publish
        run: |
          dotnet publish src/MiaApp.Api/MiaApp.Api.csproj \
            -c Release \
            -o ./publish

      - name: Upload artifact
        uses: actions/upload-artifact@v4
        with:
          name: miaapp-api
          path: ./publish
```

# Deploy automatico

Il deploy automatico deve essere trattato con più prudenza della build automatica.

Possibili strategie:

```text id="mrb08d"
deploy automatico su Development
deploy automatico su Staging
deploy su Production con approvazione manuale
deploy automatico su Production solo con forte maturità tecnica
```

Esempio concettuale GitHub Actions con environment:

```yaml id="t47njr"
name: Deploy

on:
  workflow_dispatch:

permissions:
  contents: read

jobs:
  deploy-staging:
    runs-on: ubuntu-latest
    environment: staging

    steps:
      - uses: actions/checkout@v4

      - name: Deploy to staging
        run: echo "Deploy staging"

  deploy-production:
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment: production

    steps:
      - name: Deploy to production
        run: echo "Deploy production"
```

Gli environment GitHub possono essere configurati con segreti specifici e regole di protezione, come approvazioni. I secret GitHub Actions possono essere definiti a livello di repository, environment o organizzazione. ([GitHub Docs][4])

# Azure DevOps

Azure DevOps usa spesso un file:

```text id="xkm86h"
azure-pipelines.yml
```

Pipeline base:

```yaml id="m4elnk"
trigger:
  branches:
    include:
      - main

pool:
  vmImage: ubuntu-latest

variables:
  buildConfiguration: Release

stages:
  - stage: Build
    displayName: Build and Test
    jobs:
      - job: Build
        steps:
          - task: UseDotNet@2
            inputs:
              packageType: sdk
              version: 10.0.x

          - script: dotnet restore
            displayName: Restore

          - script: dotnet build -c $(buildConfiguration) --no-restore
            displayName: Build

          - script: dotnet test -c $(buildConfiguration) --no-build --logger trx
            displayName: Test

          - script: |
              dotnet publish src/MiaApp.Api/MiaApp.Api.csproj \
                -c $(buildConfiguration) \
                -o $(Build.ArtifactStagingDirectory)/publish
            displayName: Publish

          - task: PublishPipelineArtifact@1
            inputs:
              targetPath: $(Build.ArtifactStagingDirectory)/publish
              artifact: miaapp-api
```

Azure Pipelines espone molte variabili predefinite come `Build.ArtifactStagingDirectory`, utili per memorizzare output temporanei e artifact durante una pipeline. ([Microsoft Learn][5])

# Stage e deployment job in Azure DevOps

Una pipeline Azure DevOps può essere organizzata in stage:

```text id="elw1fl"
Build
Test
Package
Deploy_Staging
Deploy_Production
```

Esempio:

```yaml id="n5j4gr"
stages:
  - stage: Build
    jobs:
      - job: Build
        steps:
          - script: dotnet build -c Release

  - stage: Deploy_Staging
    dependsOn: Build
    jobs:
      - deployment: DeployStaging
        environment: staging
        strategy:
          runOnce:
            deploy:
              steps:
                - script: echo "Deploy staging"
```

Azure DevOps raccomanda i deployment job per le fasi di deploy; questi job sono associati a un environment e supportano strategie come `runOnce`, rolling e canary. ([Microsoft Learn][6])

# Secret nelle pipeline

Le pipeline hanno spesso bisogno di segreti:

```text id="aytlag"
connection string
token deploy
API key
password registry Docker
client secret OAuth
chiave SSH
NuGet API key
```

Regola principale:

```text id="nqri78"
Mai scrivere segreti in chiaro nel file YAML.
```

GitHub Actions usa secrets definiti a livello repository, environment o organizzazione; il workflow può leggerli solo se li referenzi esplicitamente. ([GitHub Docs][7])

Esempio GitHub Actions:

```yaml id="n3ix7q"
- name: Push NuGet package
  run: dotnet nuget push ./artifacts/*.nupkg --source https://api.nuget.org/v3/index.json --api-key "${{ secrets.NUGET_API_KEY }}"
```

Azure DevOps raccomanda di impostare secret variables tramite UI, variable group o variable group collegato a Key Vault; non bisogna salvare valori sensibili in chiaro in `azure-pipelines.yml`. ([Microsoft Learn][8])

Esempio Azure DevOps:

```yaml id="d4yt7i"
- script: dotnet nuget push ./artifacts/*.nupkg --source $(NuGetFeed) --api-key $(NuGetApiKey)
  displayName: Push NuGet package
```

dove `NuGetApiKey` è una secret variable.

# Regole di sicurezza per secret

```text id="ke8z2u"
non stamparli nei log
non passarli come argomenti se possono finire in output
non committarli
ruotarli periodicamente
limitarli per ambiente
usare secret diversi tra staging e production
usare approval per environment sensibili
revocarli se compromessi
```

Attenzione:

```text id="s4t1hd"
Mascherare un secret nei log non significa che sia impossibile esporlo.
Se uno script lo trasforma, lo concatena o lo codifica, può sfuggire al masking.
```

# Versionamento release

Una release deve essere identificabile.

Possibili versioni:

```text id="e5qvje"
1.4.2
1.4.2-beta.1
1.4.2+sha.abc123
2026.04.25.5
main-abc1234
```

Per librerie NuGet conviene usare Semantic Versioning:

```text id="w2fz78"
MAJOR.MINOR.PATCH
```

Per immagini Docker:

```text id="f32ilx"
miaapp-api:1.4.2
miaapp-api:1.4.2-abc1234
miaapp-api:sha-abc1234
```

Evita di dipendere solo da:

```text id="v4zz5g"
latest
```

in produzione.

---

## Versionare da tag Git

Esempio:

```bash id="hwmkfu"
git tag v1.4.2
git push origin v1.4.2
```

Workflow GitHub:

```yaml id="fkpk9w"
on:
  push:
    tags:
      - 'v*'
```

Questo permette di produrre release solo da tag.

---

## Versione nel `.csproj`

```xml id="tygo2a"
<PropertyGroup>
  <Version>1.4.2</Version>
  <AssemblyVersion>1.4.0.0</AssemblyVersion>
  <FileVersion>1.4.2.0</FileVersion>
  <InformationalVersion>1.4.2+$(SourceRevisionId)</InformationalVersion>
</PropertyGroup>
```

In pipeline puoi passare la versione:

```bash id="a0k8c6"
dotnet pack -c Release -p:PackageVersion=1.4.2
```

Oppure:

```bash id="thq3p6"
dotnet publish -c Release -p:Version=1.4.2
```

# Rollback

Il rollback è la capacità di tornare a una versione precedente se qualcosa va male.

Il rollback non si improvvisa durante l’incidente.

Va progettato prima.

Strategie:

```text id="e71vvs"
ridistribuire artifact precedente
ripuntare a image Docker precedente
blue/green switch
rollback Kubernetes
ripristino servizio Windows/Linux precedente
feature flag off
migrazione database reversibile o compatibile
```

---

## Rollback applicativo

Se hai artifact versionati:

```text id="ehpqxb"
miaapp-api-1.4.1.zip
miaapp-api-1.4.2.zip
```

puoi tornare a `1.4.1`.

Se hai Docker image versionate:

```text id="h2kitf"
miaapp-api:1.4.1
miaapp-api:1.4.2
```

puoi ridistribuire `1.4.1`.

---

## Rollback e database

Il database rende il rollback più difficile.

Scenario:

```text id="kxwae5"
rilascio 1.4.2
migrazione aggiunge colonna obbligatoria
app 1.4.1 non sa gestirla
rollback applicativo fallisce
```

Strategia migliore:

```text id="dlqws6"
migrazioni compatibili in avanti e indietro
deploy in più fasi
feature flag
non rimuovere colonne subito
backfill controllato
rollback plan
backup prima delle migrazioni rischiose
```

Regola pratica:

```text id="i31h9v"
Il rollback del codice è facile.
Il rollback dei dati è difficile.
```


# GitHub Actions: esempio completo API .NET + Docker

```yaml id="vygrbg"
name: CI-CD

on:
  push:
    branches:
      - main
    tags:
      - 'v*'
  pull_request:
    branches:
      - main

permissions:
  contents: read
  packages: write
  security-events: write

env:
  DOTNET_VERSION: '10.0.x'
  IMAGE_NAME: ghcr.io/${{ github.repository }}/miaapp-api

jobs:
  build-test:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-dotnet@v4
        with:
          dotnet-version: ${{ env.DOTNET_VERSION }}

      - name: Restore
        run: dotnet restore

      - name: Build
        run: dotnet build -c Release --no-restore

      - name: Test
        run: dotnet test -c Release --no-build --logger "trx" --results-directory TestResults

      - name: Upload test results
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: test-results
          path: TestResults

  docker:
    needs: build-test
    runs-on: ubuntu-latest
    if: github.event_name != 'pull_request'

    steps:
      - uses: actions/checkout@v4

      - name: Login registry
        run: echo "${{ secrets.GITHUB_TOKEN }}" | docker login ghcr.io -u ${{ github.actor }} --password-stdin

      - name: Build image
        run: docker build -t $IMAGE_NAME:${{ github.sha }} .

      - name: Push image
        run: docker push $IMAGE_NAME:${{ github.sha }}
```

Nota:

```text id="qvzs4n"
La build e i test girano anche sulle pull request.
Il push image avviene solo dopo merge/push, non sulle pull request.
```

# Errori comuni

## Fare deploy senza test

Sbagliato:

```text id="u41iol"
push su main
deploy immediato
nessun test
```

Prima test, poi deploy.

---

## Build diversa dal deploy

Sbagliato:

```text id="k3v9jn"
compilo una volta per test
poi ricompilo un’altra volta per produzione
```

Meglio:

```text id="gpyhe2"
build una volta
crea artifact
promuovi lo stesso artifact tra ambienti
```

---

## Segreti nel YAML

Pericoloso:

```yaml id="rg3jmk"
password: "P@ssw0rd!"
```

Usa secrets, variable groups o Key Vault.

---

## Usare sempre `latest`

In produzione devi sapere esattamente quale versione stai eseguendo.

---

## Nessun rollback

Se non sai tornare indietro, il deploy è incompleto.

---

## Migrazioni database non pianificate

Il database deve essere parte del piano di release.

---

## Pipeline troppo permissive

Attenzione a:

```text id="m35qlj"
secrets disponibili su pull request non fidate
permessi GITHUB_TOKEN troppo ampi
script che eseguono codice esterno
azioni non pinningate
deploy da branch non protetti
```

GitHub documenta pratiche di uso sicuro per Actions, inclusa l’attenzione ai workflow non fidati e alla superficie di attacco dei workflow. ([GitHub Docs][9])

# Linee guida pratiche

```text id="lce3n2"
Automatizza la build
  ogni push e pull request devono compilare.

Esegui test automatici
  prima di creare artifact o fare deploy.

Produci artifact versionati
  zip, NuGet package o Docker image.

Promuovi lo stesso artifact
  da staging a produzione.

Non mettere segreti nel repository
  usa secrets, variable groups o vault.

Limita i permessi della pipeline
  principio del minimo privilegio.

Scansiona vulnerabilità
  pacchetti, codice e immagini container.

Usa versioni tracciabili
  SemVer, tag Git, commit SHA, build number.

Proteggi produzione
  branch protection, environment approval, audit.

Prevedi rollback
  artifact precedente, image precedente, blue/green o feature flag.

Controlla il deploy
  health check e smoke test post-deploy.

Tratta il database con cautela
  migrazioni compatibili, backup e piano di rollback.
```

# Riepilogo

In questo capitolo abbiamo visto come trasformare build, test, publish e deploy in un processo automatico e professionale.

Abbiamo trattato:

```text id="b6q41r"
pipeline
build automatica
test automatici
pubblicazione artifact
GitHub Actions
Azure DevOps
deploy automatico
secret nelle pipeline
scansione vulnerabilità
versionamento release
rollback
```

La distinzione più importante è questa:

```text id="xql796"
CI
  verifica automaticamente che il codice compili e passi i test.

CD
  produce artifact pronti e li distribuisce in modo controllato.
```

GitHub Actions e Azure DevOps permettono entrambi di definire pipeline YAML, pubblicare artifact, gestire segreti e modellare stage di deploy; Azure DevOps, in particolare, offre deployment job collegati a environment e strategie come runOnce, rolling e canary. ([GitHub Docs][1])

La regola pratica finale è questa:

```text id="q3xfzq"
Una pipeline CI/CD professionale non serve solo a distribuire più velocemente.
Serve a distribuire con meno rischio:
stesso processo, stessi controlli, artifact tracciabili,
segreti protetti, vulnerabilità controllate e rollback possibile.
```

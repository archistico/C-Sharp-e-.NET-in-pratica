# 09.03 — Configurazione, segreti e ambienti

Nel capitolo precedente abbiamo parlato di autenticazione e autorizzazione. Ora affrontiamo un tema molto pratico e spesso sottovalutato: **configurazione, segreti e ambienti**.

Un’applicazione production-ready non deve avere valori hard-coded nel codice. Deve poter cambiare comportamento tra sviluppo, test, staging e produzione senza ricompilare.

Il punto centrale è questo:

```text id="ck1w1b"
Il codice dovrebbe restare lo stesso.
La configurazione cambia in base all’ambiente.
I segreti non devono mai stare nel repository.
```

ASP.NET Core ha un sistema di configurazione basato su provider: può leggere valori da file JSON, variabili d’ambiente, user secrets, argomenti da riga di comando, Azure Key Vault e altri provider. La documentazione Microsoft descrive la configurazione come un sistema estendibile composto da coppie chiave-valore lette da più origini. ([Microsoft Learn][1])

---

# Perché la configurazione è importante

In sviluppo potresti usare:

```text id="tnzjwk"
SQLite locale
logging dettagliato
servizi fake
email disabilitate
feature sperimentali attive
```

In produzione invece:

```text id="zb379j"
SQL Server reale
logging controllato
servizi esterni reali
segreti protetti
HTTPS obbligatorio
feature rilasciate gradualmente
```

Se questi valori sono scritti nel codice, ogni ambiente richiede modifiche e ricompilazioni.

Esempio sbagliato:

```csharp id="w8ure6"
string connectionString =
    "Server=PROD-SQL;Database=MiaApp;User Id=sa;Password=Password123!";
```

Esempio corretto:

```csharp id="uk0vjy"
string connectionString =
    builder.Configuration.GetConnectionString("Default")
    ?? throw new InvalidOperationException(
        "Connection string 'Default' mancante.");
```

Il codice legge la configurazione.
L’ambiente decide il valore.

# Development, Staging e Production

ASP.NET Core usa il concetto di **runtime environment**.

Gli ambienti più comuni sono:

```text id="yf6nym"
Development
Staging
Production
```

Microsoft documenta che l’ambiente runtime viene usato per controllare il comportamento dell’applicazione e viene impostato tramite variabili come `ASPNETCORE_ENVIRONMENT` o `DOTNET_ENVIRONMENT`. ([Microsoft Learn][2])

---

## Development

Ambiente di sviluppo locale.

Caratteristiche tipiche:

```text id="ktsths"
errori dettagliati
database locale
user secrets
logging verboso
strumenti di debug
Hot Reload
feature sperimentali
```

Esempio:

```csharp id="jgpjm5"
if (app.Environment.IsDevelopment())
{
    app.UseDeveloperExceptionPage();
}
```

---

## Staging

Ambiente il più possibile simile alla produzione, ma non usato dagli utenti finali.

Caratteristiche tipiche:

```text id="xw1ktk"
database separato
servizi esterni sandbox
logging simile alla produzione
test pre-rilascio
verifica configurazioni
test migrazioni
test performance
```

Staging serve a ridurre il rischio del rilascio.

---

## Production

Ambiente reale.

Caratteristiche tipiche:

```text id="ukzrmk"
errori non dettagliati verso l’utente
segreti protetti
logging strutturato
monitoraggio attivo
HTTPS
health checks
backup
least privilege
configurazione stabile
```

In produzione non bisogna mostrare stack trace o dettagli interni agli utenti.

Su Windows PowerShell:

```powershell id="stdiwq"

```

Su Linux/macOS:

```bash id="zz45mt"

```

```yaml id="ebgrba"

  - ASPNETCORE_ENVIRONMENT=Production
```

In Azure App Service o altri ambienti cloud, normalmente si imposta tra le configurazioni applicative del servizio.

# Ordine dei provider di configurazione

ASP.NET Core carica la configurazione da più fonti. In generale, i provider caricati dopo possono sovrascrivere valori caricati prima.

Un flusso tipico è:

```text id="dfq19w"
appsettings.json
appsettings.{Environment}.json
User Secrets, in Development
variabili d’ambiente
argomenti command-line
```

Questo significa che una variabile d’ambiente può sovrascrivere un valore scritto in `appsettings.json`.

Esempio:

```json id="p9trdy"
{
  "Email": {
    "SmtpServer": "smtp-dev.example.com"
  }
}
```

Variabile d’ambiente:

```bash id="k9qbpo"
Email__SmtpServer=smtp-prod.example.com
```

Il valore finale sarà:

```text id="z2u4nj"
smtp-prod.example.com
```

# Variabili d’ambiente

Le **variabili d’ambiente** sono uno dei modi più comuni per configurare applicazioni in produzione, container e pipeline CI/CD.

Esempio:

```bash id="cinrqu"
ConnectionStrings__Default="Server=sql;Database=MiaApp;User Id=app;Password=..."
```

In .NET, il doppio underscore `__` rappresenta la separazione gerarchica.

Questa chiave:

```text id="lrntqa"
ConnectionStrings__Default
```

corrisponde a:

```json id="se9dms"
{
  "ConnectionStrings": {
    "Default": "..."
  }
}
```

Uso:

```csharp id="5lwttc"
string? connectionString =
    builder.Configuration.GetConnectionString("Default");
```

---

## Quando usare variabili d’ambiente

Sono adatte per:

```text id="k5e7dd"
container
CI/CD
ambienti cloud
configurazione per ambiente
override rapidi
valori non committati
```

Possono contenere segreti, ma in ambienti production è meglio usare un secret manager o un vault quando disponibile.

# User Secrets

**User Secrets** è uno strumento pensato per lo sviluppo locale.

Serve a evitare di mettere segreti nel repository durante lo sviluppo.

Microsoft è esplicita: password e dati sensibili non devono essere salvati nel codice sorgente o nei file di configurazione, i segreti di produzione non devono essere usati per sviluppo o test, e i segreti di produzione non devono essere distribuiti con l’applicazione. ([Microsoft Learn][3])

---

## Inizializzare User Secrets

Nel progetto:

```bash id="t3rwtl"
dotnet user-secrets init
```

Questo aggiunge nel `.csproj` un identificatore:

```xml id="miz9vg"
<PropertyGroup>
  <UserSecretsId>f5acb3a2-....</UserSecretsId>
</PropertyGroup>
```

Aggiungere un segreto:

```bash id="f9w3e4"
dotnet user-secrets set "ConnectionStrings:Default" "Server=.;Database=MiaApp;Trusted_Connection=True;"
```

Leggerlo nel codice:

```csharp id="gn7d8s"
string? connectionString =
    builder.Configuration.GetConnectionString("Default");
```

L’app non deve sapere se il valore viene da `appsettings.json`, user secrets, variabili d’ambiente o Key Vault.

---

## Cosa sono davvero User Secrets

User Secrets non cifra magicamente i segreti. Li salva in un file JSON nel profilo utente locale.

Quindi:

```text id="k6huen"
va bene per sviluppo locale
non va bene come cassaforte di produzione
non va committato
non sostituisce un secret manager production-grade
```

Regola pratica:

```text id="7fqa4l"
User Secrets per sviluppo.
Secret manager o vault per produzione.
```

# Azure Key Vault

**Azure Key Vault** è il servizio cloud di Microsoft per proteggere segreti, chiavi e certificati.

Il provider di configurazione Azure Key Vault per ASP.NET Core permette di caricare valori di configurazione direttamente dai segreti salvati in Key Vault. Microsoft descrive Key Vault come un servizio cloud per proteggere chiavi crittografiche e segreti usati da app e servizi. ([Microsoft Learn][4])

---

## Pacchetti

```bash id="bp6316"
dotnet add package Azure.Extensions.AspNetCore.Configuration.Secrets
dotnet add package Azure.Identity
```

Il pacchetto `Azure.Extensions.AspNetCore.Configuration.Secrets` permette di usare Azure Key Vault come provider di configurazione per `Microsoft.Extensions.Configuration`. ([Microsoft Learn][5])

---

## Configurazione con DefaultAzureCredential

```csharp id="w928iy"
using Azure.Identity;

var builder = WebApplication.CreateBuilder(args);

if (!builder.Environment.IsDevelopment())
{
    Uri keyVaultUri = new(
        builder.Configuration["KeyVault:Uri"]
        ?? throw new InvalidOperationException(
            "KeyVault:Uri mancante."));

    builder.Configuration.AddAzureKeyVault(
        keyVaultUri,
        new DefaultAzureCredential());
}
```

`DefaultAzureCredential` è utile perché in sviluppo può usare credenziali locali, mentre in produzione può usare managed identity, se configurata correttamente.

---

## Nomi dei segreti

In Key Vault i nomi dei segreti non usano `:`.
Spesso si usa `--` per rappresentare la gerarchia.

Esempio Key Vault secret:

```text id="juobd7"
ConnectionStrings--Default
```

diventa:

```text id="2eus8p"
ConnectionStrings:Default
```

nel sistema di configurazione.

---

## Quando usare Key Vault

Usalo quando:

```text id="d2mcu0"
sei su Azure
hai segreti production
vuoi access control centralizzato
vuoi audit
vuoi separare segreti dal deployment
vuoi usare managed identity
devi ruotare segreti
```

Regola pratica:

```text id="yk9sf2"
Il codice non deve contenere il segreto.
Il deployment non dovrebbe trasportare il segreto.
L’app deve ottenere il segreto tramite identità autorizzata.
```

# Options pattern per configurazione tipizzata

Invece di leggere stringhe sparse nel codice:

```csharp id="eqmxky"
string smtpServer = configuration["Email:SmtpServer"];
```

meglio creare classi di configurazione.

```csharp id="dsqkwq"
public class EmailOptions
{
    public const string SectionName = "Email";

    public string SmtpServer { get; set; } = "";
    public int Port { get; set; }
    public string Sender { get; set; } = "";
}
```

Registrazione:

```csharp id="bdfc1z"
builder.Services
    .AddOptions<EmailOptions>()
    .Bind(builder.Configuration.GetSection(EmailOptions.SectionName))
    .ValidateDataAnnotations()
    .ValidateOnStart();
```

Con Data Annotations:

```csharp id="mzmb4w"
public class EmailOptions
{
    public const string SectionName = "Email";

    [Required]
    public string SmtpServer { get; set; } = "";

    [Range(1, 65535)]
    public int Port { get; set; }

    [Required]
    public string Sender { get; set; } = "";
}
```

Uso:

```csharp id="rl6dky"
public class EmailService
{
    private readonly EmailOptions options;

    public EmailService(IOptions<EmailOptions> options)
    {
        this.options = options.Value;
    }
}
```

---

## Perché `ValidateOnStart`

Senza validazione all’avvio, potresti scoprire una configurazione mancante solo quando un utente usa una certa funzionalità.

Con `ValidateOnStart()` l’app fallisce subito se la configurazione non è valida.

Regola pratica:

```text id="lbecdq"
Meglio fallire all’avvio per configurazione errata
che fallire in produzione durante una richiesta utente.
```

# Feature flags

Le **feature flags** permettono di abilitare o disabilitare funzionalità senza ridistribuire codice.

Servono per:

```text id="kdzbat"
rilasci graduali
A/B test
abilitare funzioni per specifici utenti
disattivare rapidamente una feature problematica
separare deploy da release
sperimentazione controllata
```

Microsoft descrive le feature flag come una tecnica moderna che permette di distribuire nuove funzionalità in produzione ma limitarne la disponibilità, attivandole per utenti specifici senza riavviare l’app o fare nuovo deployment. ([Microsoft Learn][6])

---

## Microsoft.FeatureManagement

Pacchetto:

```bash id="j8la68"
dotnet add package Microsoft.FeatureManagement.AspNetCore
```

Configurazione:

```json id="hlhd6v"
{
  "FeatureManagement": {
    "NuovaDashboard": true,
    "ExportExcel": false
  }
}
```

Registrazione:

```csharp id="ja7ibf"
builder.Services.AddFeatureManagement();
```

Uso:

```csharp id="kyx2cr"
public class DashboardService
{
    private readonly IFeatureManager featureManager;

    public DashboardService(IFeatureManager featureManager)
    {
        this.featureManager = featureManager;
    }

    public async Task<string> GetDashboardAsync()
    {
        if (await featureManager.IsEnabledAsync("NuovaDashboard"))
        {
            return "Nuova dashboard";
        }

        return "Dashboard classica";
    }
}
```

La libreria Microsoft Feature Management fornisce un modo per esporre funzionalità in base a feature flag e si integra con i pattern comuni di .NET. ([Microsoft Learn][7])

---

## Feature flag su endpoint

Esempio:

```csharp id="scsiae"
app.MapGet("/api/dashboard-nuova", () =>
{
    return Results.Ok("Nuova dashboard");
})
.RequireAuthorization()
.RequireHost("*");
```

Per filtrare davvero l’accesso in modo elegante si può usare un endpoint filter o controllare `IFeatureManager` dentro l’handler.

Esempio semplice:

```csharp id="1u4okf"
app.MapGet("/api/dashboard", async (
    IFeatureManager featureManager) =>
{
    if (await featureManager.IsEnabledAsync("NuovaDashboard"))
    {
        return Results.Ok("Nuova dashboard");
    }

    return Results.Ok("Dashboard classica");
});
```

# Separare configurazione e segreti

È importante distinguere:

```text id="kj78hm"
configurazione non sensibile
  log level
  URL pubblici
  feature flag non sensibili
  nomi code
  timeout
  dimensioni batch

segreti
  password
  token
  API key
  connection string con credenziali
  client secret
  certificati privati
```

La configurazione non sensibile può stare in `appsettings.json`.

I segreti no.

# Esempio completo: configurazione Email

## appsettings.json

```json id="mkg31g"
{
  "Email": {
    "Enabled": true,
    "SmtpServer": "smtp.example.com",
    "Port": 587,
    "Sender": "noreply@example.com"
  }
}
```

## User secrets

```bash id="e90d73"
dotnet user-secrets set "Email:Password" "password-locale"
```

## Classe options

```csharp id="jjj7qd"
public class EmailOptions
{
    public const string SectionName = "Email";

    [Required]
    public string SmtpServer { get; set; } = "";

    [Range(1, 65535)]
    public int Port { get; set; }

    [Required]
    public string Sender { get; set; } = "";

    public bool Enabled { get; set; }

    public string? Password { get; set; }
}
```

## Registrazione

```csharp id="t8331x"
builder.Services
    .AddOptions<EmailOptions>()
    .Bind(builder.Configuration.GetSection(EmailOptions.SectionName))
    .ValidateDataAnnotations()
    .Validate(options =>
    {
        if (options.Enabled &&
            string.IsNullOrWhiteSpace(options.Password))
        {
            return false;
        }

        return true;
    }, "Email:Password obbligatoria quando Email:Enabled è true.")
    .ValidateOnStart();
```

## Uso

```csharp id="n7i1xc"
public class EmailService
{
    private readonly EmailOptions options;
    private readonly ILogger<EmailService> logger;

    public EmailService(
        IOptions<EmailOptions> options,
        ILogger<EmailService> logger)
    {
        this.options = options.Value;
        this.logger = logger;
    }

    public Task SendAsync(
        string to,
        string subject,
        string body,
        CancellationToken cancellationToken)
    {
        if (!options.Enabled)
        {
            logger.LogInformation(
                "Invio email disabilitato. Destinatario: {To}",
                to);

            return Task.CompletedTask;
        }

        // invio reale

        return Task.CompletedTask;
    }
}
```

# Esempio completo: configurazione per servizi esterni

## Options

```csharp id="yzgwvd"
public class ExternalApiOptions
{
    public const string SectionName = "ExternalApi";

    [Required]
    public string BaseUrl { get; set; } = "";

    [Range(1, 300)]
    public int TimeoutSeconds { get; set; } = 30;

    [Required]
    public string ApiKey { get; set; } = "";
}
```

## Configurazione

```json id="l591vn"
{
  "ExternalApi": {
    "BaseUrl": "https://api.example.com",
    "TimeoutSeconds": 30
  }
}
```

Il valore `ApiKey` arriva da User Secrets, variabile d’ambiente o Key Vault.

## Registrazione HttpClient

```csharp id="j4qs3q"
builder.Services
    .AddOptions<ExternalApiOptions>()
    .Bind(builder.Configuration.GetSection(ExternalApiOptions.SectionName))
    .ValidateDataAnnotations()
    .ValidateOnStart();

builder.Services.AddHttpClient<ExternalApiClient>((sp, client) =>
{
    ExternalApiOptions options =
        sp.GetRequiredService<IOptions<ExternalApiOptions>>().Value;

    client.BaseAddress = new Uri(options.BaseUrl);
    client.Timeout = TimeSpan.FromSeconds(options.TimeoutSeconds);
    client.DefaultRequestHeaders.Add("X-API-Key", options.ApiKey);
});
```

# Linee guida pratiche

```text id="humgde"
Usa ambienti separati
  Development, Staging, Production, Test.

Non mettere segreti nel repository
  mai.

Usa User Secrets
  solo per sviluppo locale.
```

# Riepilogo

In questo capitolo abbiamo visto come gestire configurazione, segreti e ambienti in applicazioni ASP.NET Core e .NET.

Abbiamo trattato:

```text id="ye0cll"
Development/Staging/Production
variabili d’ambiente
User Secrets
secret manager
Azure Key Vault
configurazione per container
feature flags

```

La distinzione più importante è questa:

```text id="qgx5uo"
Configurazione
  valori che cambiano tra ambienti ma non sono necessariamente segreti.

Segreti
  valori sensibili che non devono stare nel codice,
  nel repository o nel pacchetto distribuito.
```

ASP.NET Core permette di combinare più fonti di configurazione, mentre User Secrets serve per lo sviluppo locale e Azure Key Vault o altri secret manager sono più adatti agli ambienti di test e produzione. ([Microsoft Learn][3])

La regola principale è questa:

```text id="c150qf"
Una stessa build dovrebbe poter girare in ambienti diversi.
Cambiano configurazione, segreti e risorse esterne;
non dovrebbe cambiare il codice.
```

# 09.02 — Autenticazione e autorizzazione in ASP.NET Core

Nel capitolo precedente abbiamo visto le basi della sicurezza applicativa: validazione input, output encoding, SQL injection, XSS, CSRF, gestione segreti, HTTPS, CORS, logging sicuro, password e least privilege.

Ora entriamo nel tema centrale delle applicazioni web e API: **autenticazione** e **autorizzazione**.

Il punto centrale è questo:

```text id="b5cavd"
Autenticazione e autorizzazione non sono la stessa cosa.

Authentication:
  chi sei?

Authorization:
  cosa puoi fare?
```

ASP.NET Core separa chiaramente questi due concetti: l’autenticazione costruisce l’identità dell’utente o del servizio, mentre l’autorizzazione decide se quell’identità può accedere a una risorsa o compiere un’azione. ASP.NET Core supporta sia autorizzazione basata su ruoli sia un modello più ricco basato su policy, requirements e handler. ([Microsoft Learn][1])

---

# Authentication

L’**authentication** verifica l’identità del chiamante.

Il chiamante può essere:

```text id="bisz7d"
utente umano
applicazione web
app mobile
servizio backend
job schedulato
microservizio
client esterno
```

Esempi di meccanismi di autenticazione:

```text id="jtdvnt"
cookie authentication
JWT bearer token
OpenID Connect
certificati client
API key, se usate con attenzione
Windows Authentication
```

In ASP.NET Core, dopo l’autenticazione, l’utente viene rappresentato principalmente tramite:

```csharp id="f1pemd"
HttpContext.User
```

che è un `ClaimsPrincipal`.

---

# Authorization

L’**authorization** decide se un’identità autenticata può fare qualcosa.

Esempi:

```text id="rv8itp"
può vedere questa pagina?
può chiamare questa API?
può cancellare questo cliente?
può approvare questo documento?
può leggere i dati di un altro reparto?
può operare su questa risorsa specifica?
```

Esempio Minimal API:

```csharp id="rdwj5o"
app.MapDelete("/api/clienti/{id:int}", async (
    int id,
    ClienteService service,
    CancellationToken cancellationToken) =>
{
    await service.EliminaAsync(id, cancellationToken);

    return Results.NoContent();
})
.RequireAuthorization("Clienti.Delete");
```

Qui l’endpoint richiede la policy `Clienti.Delete`.

---

# Middleware di autenticazione e autorizzazione

In un’app ASP.NET Core, autenticazione e autorizzazione entrano nella pipeline HTTP.

Schema tipico:

```csharp id="m67b4f"
app.UseAuthentication();
app.UseAuthorization();
```

Ordine importante:

```text id="94tmuj"
UseAuthentication()
  prima costruisce HttpContext.User.

UseAuthorization()
  poi controlla policy, ruoli o attributi.
```

Se inverti l’ordine, l’autorizzazione potrebbe non trovare un utente autenticato.

# Cookie authentication

La **cookie authentication** è tipica delle applicazioni web server-side:

```text id="p88jcd"
Razor Pages
MVC
Blazor Server
applicazioni intranet
backoffice
gestionali web tradizionali
```

Il flusso base è:

```text id="m5eoit"
1. L’utente fa login.
2. Il server verifica credenziali o riceve identità da un provider esterno.
3. Il server crea un cookie di autenticazione.
4. Il browser invia automaticamente il cookie nelle richieste successive.
5. ASP.NET Core legge il cookie e ricostruisce l’identità.
```

Esempio configurazione semplificata:

```csharp id="b6r66h"
builder.Services
    .AddAuthentication("AppCookie")
    .AddCookie("AppCookie", options =>
    {
        options.LoginPath = "/login";
        options.AccessDeniedPath = "/access-denied";
        options.Cookie.Name = "MiaApp.Auth";
        options.Cookie.HttpOnly = true;
        options.Cookie.SecurePolicy = CookieSecurePolicy.Always;
        options.Cookie.SameSite = SameSiteMode.Lax;
    });

builder.Services.AddAuthorization();
```

Uso:

```csharp id="j0tw1n"
app.UseAuthentication();
app.UseAuthorization();
```

## Login manuale con cookie

Esempio didattico:

```csharp id="tb5s5m"
app.MapPost("/login", async (
    LoginRequest request,
    HttpContext httpContext) =>
{
    if (request.UserName != "admin" ||
        request.Password != "password-demo")
    {
        return Results.Unauthorized();
    }

    List<Claim> claims =
    [
        new Claim(ClaimTypes.Name, request.UserName),
        new Claim(ClaimTypes.Role, "Admin"),
        new Claim("permission", "Clienti.Read"),
        new Claim("permission", "Clienti.Delete")
    ];

    ClaimsIdentity identity = new(
        claims,
        authenticationType: "AppCookie");

    ClaimsPrincipal principal = new(identity);

    await httpContext.SignInAsync(
        "AppCookie",
        principal);

    return Results.Ok();
});
```

Questo esempio serve solo a capire il meccanismo. In produzione non si devono verificare password hard-coded e conviene usare ASP.NET Core Identity, OpenID Connect o un identity provider.

app.MapPost("/logout", async (HttpContext httpContext) =>

    await httpContext.SignOutAsync("AppCookie");

## Cookie authentication e CSRF

Con i cookie, il browser allega automaticamente le credenziali. Questo rende importante la protezione CSRF nelle app basate su form e cookie.

Regola pratica:

```text id="lv733g"
Cookie authentication:
  ottima per web app server-side,
  ma va abbinata a protezioni CSRF, cookie sicuri e HTTPS.
```

# JWT bearer

JWT significa **JSON Web Token**.

La **JWT bearer authentication** è molto usata per API.

Il flusso base è:

```text id="8m26tw"
1. Il client ottiene un access token da un identity provider.
2. Il client chiama l’API con header Authorization.
3. L’API valida firma, issuer, audience e scadenza.
4. Se il token è valido, ASP.NET Core costruisce HttpContext.User.
```

Header:

```http id="ea06mi"
Authorization: Bearer eyJhbGciOi...
```

Configurazione semplificata:

```csharp id="lfpezf"
builder.Services
    .AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.Authority = "https://identity.example.com";
        options.Audience = "mia-api";
    });

builder.Services.AddAuthorization();
```

Endpoint:

```csharp id="b2d5tc"
app.MapGet("/api/profilo", (ClaimsPrincipal user) =>
{
    string? name = user.Identity?.Name;

    return Results.Ok(new
    {
        Name = name
    });
})
.RequireAuthorization();
```

## Cosa deve validare una API

Una API non deve “fidarsi” del token perché sembra corretto.

Deve validare:

```text id="42z0x3"
firma
issuer
audience
scadenza
algoritmo
eventuali scope/claims necessari
```

Regola pratica:

```text id="oex5d0"
JWT bearer è adatto alle API.
Cookie authentication è adatta alle web app server-side.
```

# Claims

Una **claim** è un’affermazione sull’identità.

Esempi:

```text id="rwj557"
name = emilie
sub = 123456
email = emilie@example.com
role = Admin
department = Finance
permission = Clienti.Read
scope = api.read
```

In ASP.NET Core, un’identità contiene claims e può avere claims multiple dello stesso tipo. La documentazione Microsoft mostra come usare le claims per costruire controlli di autorizzazione più ricchi rispetto al solo controllo del login. ([Microsoft Learn][2])

Esempio lettura claims:

```csharp id="jz0ga7"
app.MapGet("/api/me", (ClaimsPrincipal user) =>
{
    string? subject = user.FindFirstValue(ClaimTypes.NameIdentifier);
    string? email = user.FindFirstValue(ClaimTypes.Email);

    List<string> permissions = user
        .FindAll("permission")
        .Select(c => c.Value)
        .ToList();

    return Results.Ok(new
    {
        Subject = subject,
        Email = email,
        Permissions = permissions
    });
})
.RequireAuthorization();
```

# Roles

Un **role** rappresenta un gruppo funzionale.

Esempi:

```text id="yw9rcd"
Admin
Manager
Operator
Supervisor
BackOffice
Reader
```

Autorizzazione per ruolo:

```csharp id="wtal8h"
app.MapGet("/api/admin", () =>
{
    return Results.Ok("Area admin");
})
.RequireAuthorization(policy =>
{
    policy.RequireRole("Admin");
});
```

Oppure con attributi:

```csharp id="br8hug"
[Authorize(Roles = "Admin")]
public class AdminController : Controller
{
}
```

ASP.NET Core supporta autorizzazione basata su ruoli tramite il parametro `Roles` di `[Authorize]` e componenti come `AuthorizeView` in Blazor. ([Microsoft Learn][3])

---

## Limite dei ruoli

I ruoli sono semplici, ma possono diventare troppo grossolani.

Esempio:

```text id="g2b2qh"
Admin
```

Che cosa può fare esattamente?

```text id="gg3qct"
leggere clienti?
modificare clienti?
cancellare clienti?
approvare documenti?
vedere report?
gestire utenti?
```

Per sistemi semplici, i ruoli bastano.
Per sistemi complessi, meglio affiancarli a policy e permission claims.

# Policies

Le **policy** sono il modello più flessibile di autorizzazione in ASP.NET Core.

Una policy può richiedere:

```text id="wnpfnq"
utente autenticato
ruolo
claim
più claims
requirement personalizzati
logica su risorsa
```

La documentazione Microsoft descrive l’autorizzazione policy-based come un modello basato su requirements e handler che valutano le claims dell’utente rispetto ai requisiti richiesti. ([Microsoft Learn][4])

Esempio:

```csharp id="vezb5g"
builder.Services.AddAuthorization(options =>
{
    options.AddPolicy("Clienti.Read", policy =>
    {
        policy.RequireAuthenticatedUser();
        policy.RequireClaim("permission", "Clienti.Read");
    });

    options.AddPolicy("Clienti.Delete", policy =>
    {
        policy.RequireAuthenticatedUser();
        policy.RequireClaim("permission", "Clienti.Delete");
    });
});
```

Uso:

```csharp id="de07x3"
app.MapGet("/api/clienti", () =>
{
    return Results.Ok();
})
.RequireAuthorization("Clienti.Read");

app.MapDelete("/api/clienti/{id:int}", (int id) =>
{
    return Results.NoContent();
})
.RequireAuthorization("Clienti.Delete");
```

# Policy con requirement personalizzato

A volte una claim non basta.

Esempio:

```text id="n3ax2i"
Può approvare documenti solo se:
  ha permission Documenti.Approve
  e appartiene allo stesso reparto del documento
  e il documento è in stato Bozza
```

Requirement:

```csharp id="e55imv"
public class ApprovaDocumentoRequirement
    : IAuthorizationRequirement
{
}
```

Handler:

```csharp id="mj5pk8"
public class ApprovaDocumentoHandler
    : AuthorizationHandler<ApprovaDocumentoRequirement, Documento>
{
    protected override Task HandleRequirementAsync(
        AuthorizationHandlerContext context,
        ApprovaDocumentoRequirement requirement,
        Documento documento)
    {
        bool haPermesso = context.User.HasClaim(
            "permission",
            "Documenti.Approve");

        string? repartoUtente = context.User.FindFirstValue("department");

        if (haPermesso &&
            repartoUtente == documento.Reparto &&
            documento.Stato == StatoDocumento.Bozza)
        {
            context.Succeed(requirement);
        }

        return Task.CompletedTask;
    }
}
```

Registrazione:

```csharp id="cu1h0e"
builder.Services.AddAuthorization(options =>
{
    options.AddPolicy("Documenti.Approve", policy =>
    {
        policy.Requirements.Add(
            new ApprovaDocumentoRequirement());
    });
});

builder.Services.AddScoped<
    IAuthorizationHandler,
    ApprovaDocumentoHandler>();
```

Uso imperativo:

```csharp id="z6w8gg"
public class DocumentoService
{
    private readonly IAuthorizationService authorizationService;

    public DocumentoService(
        IAuthorizationService authorizationService)
    {
        this.authorizationService = authorizationService;
    }

    public async Task ApprovaAsync(
        ClaimsPrincipal user,
        Documento documento)
    {
        AuthorizationResult result =
            await authorizationService.AuthorizeAsync(
                user,
                documento,
                "Documenti.Approve");

        if (!result.Succeeded)
        {
            throw new UnauthorizedAccessException();
        }

        documento.Approva();
    }
}
```

Questo è un esempio di autorizzazione basata sulla risorsa. Microsoft documenta l’uso di resource-based authorization quando la decisione dipende dalla risorsa concreta e non può essere espressa solo con un attributo statico. ([Microsoft Learn][5])

# RBAC

**RBAC** significa Role-Based Access Control.

L’utente riceve uno o più ruoli e i ruoli danno accesso a funzioni.

Esempio:

```text id="ip0inn"
Admin
  può gestire utenti
  può vedere report
  può cancellare clienti

Operator
  può creare clienti
  può modificare clienti

Reader
  può solo leggere
```

Implementazione semplice:

```csharp id="m26sbv"
builder.Services.AddAuthorization(options =>
{
    options.AddPolicy("AdminOnly", policy =>
    {
        policy.RequireRole("Admin");
    });
});
```

Uso:

```csharp id="u8jcby"
app.MapGet("/api/admin/report", () =>
{
    return Results.Ok();
})
.RequireAuthorization("AdminOnly");
```

---

## Quando usare RBAC

RBAC è adatto quando:

```text id="bdd7p3"
i permessi sono stabili
i ruoli sono pochi
le regole sono semplici
il dominio autorizzativo è comprensibile
```

Limite:

```text id="bwx6sw"
se i ruoli aumentano troppo,
diventano difficili da gestire.
```

Esempio di esplosione:

```text id="7kmw2n"
AdminNord
AdminSud
AdminReport
AdminReportNord
AdminReportSud
ManagerConExport
ManagerSenzaExport
```

Quando succede, conviene passare a policy e permission claims.

# PBAC

**PBAC** significa Policy-Based Access Control.

In ASP.NET Core, PBAC è naturale perché il framework ha un modello policy-based.

Una policy può combinare:

```text id="o84qx2"
ruoli
claims
permessi
reparto
tenant
proprietà della risorsa
stato della risorsa
orario
livello di rischio
```

Esempio:

```csharp id="ohy587"
builder.Services.AddAuthorization(options =>
{
    options.AddPolicy("Report.View", policy =>
    {
        policy.RequireAuthenticatedUser();
        policy.RequireClaim("permission", "Report.View");
        policy.RequireClaim("department", "Finance", "Marketing");
    });
});
```

PBAC è più flessibile di RBAC.

Regola pratica:

```text id="pok6el"
Usa RBAC per casi semplici.
Usa PBAC quando la decisione dipende da permessi,
claims o proprietà della risorsa.
```

# OpenID Connect

**OpenID Connect**, spesso abbreviato OIDC, è un protocollo di autenticazione costruito sopra OAuth 2.0.

Serve a fare login tramite un identity provider.

Esempi di identity provider:

```text id="b7fzpc"
Microsoft Entra ID
Duende IdentityServer
Auth0
Okta
Keycloak
Google
provider aziendale
```

Flusso tipico web app:

```text id="oy6pk5"
1. L’utente accede alla web app.
2. La web app redirige all’identity provider.
3. L’utente fa login.
4. L’identity provider restituisce una risposta OIDC.
5. La web app crea la sessione locale, spesso con cookie.
```

Configurazione semplificata:

```csharp id="fqvgyt"
builder.Services
    .AddAuthentication(options =>
    {
        options.DefaultScheme = "AppCookie";
        options.DefaultChallengeScheme = "oidc";
    })
    .AddCookie("AppCookie")
    .AddOpenIdConnect("oidc", options =>
    {
        options.Authority = "https://identity.example.com";
        options.ClientId = "mia-web-app";
        options.ClientSecret = builder.Configuration["Oidc:ClientSecret"];
        options.ResponseType = "code";
        options.SaveTokens = true;

        options.Scope.Add("openid");
        options.Scope.Add("profile");
        options.Scope.Add("email");
    });
```

OpenID Connect viene usato per autenticare l’utente; Duende documenta, ad esempio, scenari in cui una Razor Pages app usa IdentityServer per l’autenticazione interattiva tramite OIDC. ([Duende Software Docs][6])

# OAuth2

**OAuth2** è principalmente un framework di autorizzazione delegata.

Serve a consentire a un client di ottenere accesso a una risorsa protetta.

Esempio:

```text id="r2k7kt"
Una web app vuole chiamare una API per conto dell’utente.
Ottiene un access token.
Chiama l’API con Bearer token.
L’API valida il token.
```

Differenza pratica:

```text id="hm5zwx"
OpenID Connect
  login e identità dell’utente.

OAuth2
  accesso delegato a risorse/API.
```

Duende sottolinea che OpenID Connect e OAuth si combinano bene: in uno stesso flusso si può ottenere autenticazione utente e accesso a una API tramite scopes e token. ([Duende Software Docs][7])

# Microsoft Entra ID

**Microsoft Entra ID** è l’identity provider cloud Microsoft, molto usato in contesti aziendali.

È utile quando:

```text id="aswt4r"
gli utenti sono aziendali
hai Microsoft 365/Azure
vuoi Single Sign-On
vuoi MFA e Conditional Access
vuoi integrazione con gruppi e directory
devi proteggere API aziendali
```

In una web app o API ASP.NET Core, Entra ID viene normalmente usato tramite OpenID Connect e OAuth2 sulla Microsoft identity platform. La documentazione Microsoft Identity Platform copre gli scenari OIDC/OAuth per autenticare utenti e proteggere API. ([Microsoft Learn][4])

Esempio concettuale per API:

```csharp id="e1gqip"
builder.Services
    .AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.Authority =
            "https://login.microsoftonline.com/{tenant-id}/v2.0";

        options.Audience = "api://mia-api";
    });
```

In progetti reali si usa spesso anche `Microsoft.Identity.Web`, ma il concetto resta: l’API valida token emessi da Entra ID.

## Client credentials flow

Scenario tipico OAuth2:

```text id="qa7qkd"
1. Il servizio client si autentica presso l’identity provider.
2. Ottiene un access token per una API.
3. Chiama l’API con Authorization: Bearer.
4. L’API valida token, audience, issuer e scope/role applicativo.
```

Esempio concettuale token request:

```http id="mw8kg4"
POST /connect/token
Content-Type: application/x-www-form-urlencoded

client_id=report-worker&
client_secret=...&
scope=report-api
```

API:

```csharp id="tglmsw"
app.MapPost("/api/report/genera", () =>
{
    return Results.Accepted();
})
.RequireAuthorization("Report.Generate");
```

Policy:

```csharp id="q2e5ox"
builder.Services.AddAuthorization(options =>
{
    options.AddPolicy("Report.Generate", policy =>
    {
        policy.RequireAuthenticatedUser();
        policy.RequireClaim("scope", "report-api");
    });
});
```

# API key

Le API key sono semplici, ma meno robuste di OAuth2/OIDC.

Possono andare bene per:

```text id="7s12jw"
integrazioni interne semplici
webhook controllati
servizi legacy
ambienti non complessi
```

Ma hanno limiti:

```text id="lfd8lq"
spesso non identificano bene il chiamante
sono difficili da ruotare se usate male
non hanno scadenza naturale
non portano claims ricche
se rubate, danno accesso finché non revocate
```

Regola pratica:

```text id="8bkntz"
Per service-to-service moderno, preferisci OAuth2 client credentials,
managed identity o certificati rispetto a semplici API key.
```

# Autenticazione multipla

A volte una stessa app supporta più schemi:

```text id="cpe8kl"
cookie per pagine web
JWT per API
OpenID Connect per login
```

ASP.NET Core permette di limitare l’autorizzazione a uno schema specifico quando sono presenti più metodi di autenticazione. Microsoft documenta l’uso di schemi specifici per controllare quale identità viene usata per autorizzare una risorsa. ([Microsoft Learn][11])

Esempio:

```csharp id="ikbeto"
app.MapGet("/api/data", () =>
{
    return Results.Ok();
})
.RequireAuthorization(new AuthorizeAttribute
{
    AuthenticationSchemes =
        JwtBearerDefaults.AuthenticationScheme
});
```

In pratica, conviene progettare bene:

```text id="m2x6h4"
quale schema usa la UI
quale schema usano le API
quale challenge parte per richieste non autenticate
quali endpoint accettano quali credenziali
```

# Autenticazione vs autorizzazione nella UI

Non basta nascondere pulsanti.

Esempio Razor/Blazor:

```text id="cr3kah"
se l’utente non è Admin,
nascondo il pulsante Cancella.
```

Questo migliora l’esperienza utente, ma non è sicurezza sufficiente.

Devi proteggere anche l’endpoint/server action:

```csharp id="pcyjht"
app.MapDelete("/api/clienti/{id:int}", ...)
   .RequireAuthorization("Clienti.Delete");
```

La documentazione Microsoft mostra come usare l’autorizzazione anche nelle view, ma la logica server-side resta indispensabile. ([Microsoft Learn][12])

Regola pratica:

```text id="9rnn9a"
La UI può nascondere ciò che l’utente non può fare.
Il server deve impedire ciò che l’utente non può fare.
```

# Esempio completo: API con JWT, claims e policy

Configurazione:

```csharp id="a6qcsd"
using Microsoft.AspNetCore.Authentication.JwtBearer;

builder.Services
    .AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.Authority = builder.Configuration["Auth:Authority"];
        options.Audience = builder.Configuration["Auth:Audience"];
    });

builder.Services.AddAuthorization(options =>
{
    options.AddPolicy("Clienti.Read", policy =>
    {
        policy.RequireAuthenticatedUser();
        policy.RequireClaim("permission", "Clienti.Read");
    });

    options.AddPolicy("Clienti.Write", policy =>
    {
        policy.RequireAuthenticatedUser();
        policy.RequireClaim("permission", "Clienti.Write");
    });
});
```

Pipeline:

```csharp id="w2d2px"
app.UseAuthentication();
app.UseAuthorization();
```

Endpoint:

```csharp id="sxu7uu"
app.MapGet("/api/clienti", async (
    ClienteService service,
    CancellationToken cancellationToken) =>
{
    List<ClienteDto> clienti =
        await service.ListaAsync(cancellationToken);

    return Results.Ok(clienti);
})
.RequireAuthorization("Clienti.Read");

app.MapPost("/api/clienti", async (
    CreaClienteRequest request,
    ClienteService service,
    CancellationToken cancellationToken) =>
{
    ClienteDto cliente =
        await service.CreaAsync(request, cancellationToken);

    return Results.Created(
        $"/api/clienti/{cliente.Id}",
        cliente);
})
.RequireAuthorization("Clienti.Write");
```

# Errori comuni

## Confondere autenticazione e autorizzazione

Sbagliato:

```text id="p354s3"
L’utente è loggato, quindi può fare tutto.
```

Corretto:

```text id="04wu0p"
Login dimostra chi sei.
Le policy decidono cosa puoi fare.
```

---

## Usare solo ruoli generici

Sospetto:

```text id="6ndlhm"
Admin ovunque
```

Meglio definire permessi/policy più specifiche:

```text id="u3lzy9"
Clienti.Read
Clienti.Write
Documenti.Approve
Report.Export
```

---

## Mettere logica di autorizzazione solo nella UI

Nascondere il pulsante non basta.
Proteggi endpoint, handler o risorsa lato server.

---

## Fidarsi di claims senza validare token

Le claims sono affidabili solo se il token/cookie è stato validato correttamente.

---

## Salvare token nei posti sbagliati

Attenzione soprattutto alle SPA:

```text id="zbvsor"
localStorage
sessionStorage
JavaScript accessibile
```

Espongono rischi in caso di XSS. Per scenari SPA/Blazor WASM moderni, considera pattern BFF o configurazioni che riducono l’esposizione dei token nel browser. Duende, ad esempio, documenta anche un approccio BFF per proteggere SPA e Blazor WASM senza conservare token direttamente nel browser. ([Duende Software Docs][13])

---

## API key usate come autenticazione enterprise

Le API key possono bastare in scenari semplici, ma non sostituiscono sempre OAuth2/OIDC, mTLS o managed identity.

---

## Non gestire scadenza e revoca

Token e sessioni devono avere:

```text id="iak0w2"
scadenza
rotazione
revoca dove possibile
logout
gestione compromissione
```

---

# Linee guida pratiche

```text id="volwaq"
Distingui sempre authentication e authorization
  chi sei, cosa puoi fare.

Usa cookie authentication
  per web app server-side.

Usa JWT bearer
  per API chiamate da client o servizi.

Usa OpenID Connect
  per login federato con identity provider.

Usa OAuth2
  per accesso delegato alle API.

Usa claims
  per rappresentare attributi e permessi.

Usa ruoli
  solo quando il modello autorizzativo è semplice.

Usa policy
  per autorizzazione seria e manutenibile.

Usa resource-based authorization
  quando la decisione dipende dalla risorsa concreta.

Proteggi sempre lato server
  non solo nella UI.

Per service-to-service
  preferisci client credentials, managed identity o certificati.

Non costruire identity provider da zero
  se puoi usare Identity, Entra ID, Duende, Keycloak o altri provider consolidati.

Mantieni token, cookie e segreti protetti
  e configura HTTPS sempre.
```

# Riepilogo

In questo capitolo abbiamo studiato autenticazione e autorizzazione in ASP.NET Core.

Abbiamo visto:

```text id="oxd5dv"
Authentication
Authorization
Cookie authentication
JWT bearer
Claims
Roles
Policies
RBAC
PBAC
OpenID Connect
OAuth2
Identity provider
Duende IdentityServer
Entra ID
Service-to-service authentication
```

La distinzione fondamentale è:

```text id="l3elnx"
Authentication
  costruisce l’identità.

Authorization
  valuta se quell’identità può accedere a una risorsa o azione.
```

I ruoli sono semplici e utili, ma ASP.NET Core offre un modello policy-based più ricco, basato su requirements e handler, adatto a regole reali e resource-based authorization. ([Microsoft Learn][4])

La regola principale è questa:

```text id="ujq7pj"
Non progettare la sicurezza intorno a “utente loggato sì/no”.
Progettala intorno a identità, claims, policy, risorse,
contesto applicativo e principio del minimo privilegio.
```

Questo capitolo prepara il terreno per la parte production-ready: hardening, configurazioni per ambiente, health checks, observability, secrets management, rate limiting, audit, deployment sicuro e protezione delle API in scenari reali.

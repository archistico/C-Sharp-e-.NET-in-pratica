# 09.01 — Sicurezza applicativa di base

Finora abbiamo parlato di linguaggio, librerie, dati, web, architettura, pattern e performance. Ma un’applicazione reale non è completa se non affronta la sicurezza.

La sicurezza non riguarda solo login, JWT o microservizi. Riguarda ogni punto in cui il sistema riceve input, produce output, salva dati, legge configurazioni, espone API o registra informazioni nei log.

Il punto centrale è questo:

```text id="v0beqx"
La sicurezza applicativa non è una funzionalità separata.
È una proprietà trasversale del sistema.
```

ASP.NET Core offre strumenti per autenticazione, autorizzazione, protezione dati, HTTPS, gestione segreti, CORS, prevenzione CSRF/XSRF e mitigazioni contro XSS e SQL injection, ma questi strumenti devono essere configurati e usati correttamente dallo sviluppatore. ([Microsoft Learn][1])

---

# Principi base di sicurezza

Prima delle singole vulnerabilità, conviene fissare alcuni principi.

```text id="jtwp10"
Non fidarsi mai dell’input esterno.
Validare lato server.
Codificare l’output.
Usare query parametrizzate.
Proteggere i segreti.
Limitare i permessi.
Loggare senza esporre dati sensibili.
Aggiornare dipendenze e runtime.
Usare HTTPS.
Ridurre la superficie di attacco.
```

La sicurezza non è “mettere un controllo alla fine”.
È progettare ogni confine dell’applicazione come potenziale punto di rischio.

Confini tipici:

```text id="x37n22"
request HTTP
form web
query string
header
file caricati
JSON ricevuto da API
messaggi da code
dati letti da database legacy
risposte da servizi esterni
parametri da riga di comando
configurazioni
```

Regola pratica:

```text id="s0afzi"
Tutto ciò che arriva dall’esterno va trattato come non attendibile
finché non viene validato, normalizzato e gestito correttamente.
```

# Validazione input

La validazione dell’input serve a verificare che i dati ricevuti siano compatibili con ciò che l’applicazione si aspetta.

Esempio request:

```csharp id="dwy60e"
public record CreaClienteRequest(
    string Nome,
    string? Email,
    DateOnly? DataNascita);
```

Validazione manuale:

```csharp id="hcvw9x"
public static List<string> Valida(CreaClienteRequest request)
{
    List<string> errors = [];

    if (string.IsNullOrWhiteSpace(request.Nome))
    {
        errors.Add("Il nome è obbligatorio.");
    }

    if (request.Nome.Length > 100)
    {
        errors.Add("Il nome non può superare 100 caratteri.");
    }

    if (request.Email is not null &&
        !request.Email.Contains('@'))
    {
        errors.Add("Email non valida.");
    }

    if (request.DataNascita is not null &&
        request.DataNascita > DateOnly.FromDateTime(DateTime.Today))
    {
        errors.Add("La data di nascita non può essere futura.");
    }

    return errors;
}
```

Endpoint:

```csharp id="42ccml"
app.MapPost("/api/clienti", async (
    CreaClienteRequest request,
    ClienteService service,
    CancellationToken cancellationToken) =>
{
    List<string> errors = Valida(request);

    if (errors.Count > 0)
    {
        return Results.BadRequest(errors);
    }

    ClienteDto cliente = await service.CreaAsync(
        request,
        cancellationToken);

    return Results.Created(
        $"/api/clienti/{cliente.Id}",
        cliente);
});
```

---

## Validazione non significa sicurezza completa

La validazione è necessaria, ma non basta.

Esempio:

```text id="u83qtv"
Validare che il campo Nome sia lungo massimo 100 caratteri
non protegge automaticamente da XSS se poi stampi quel valore
senza encoding nel punto sbagliato.
```

La validazione risponde a:

```text id="i4wng3"
Questo dato è accettabile per il mio dominio?
```

La codifica dell’output risponde a:

```text id="eq77ff"
Come mostro questo dato in modo sicuro nel contesto HTML, JavaScript,
URL, attributo o JSON?
```

Sono due difese diverse.

# Output encoding

L’**output encoding** serve a impedire che dati non attendibili vengano interpretati come codice.

Esempio di input malevolo:

```html id="pvrtws"
<script>alert('xss')</script>
```

Se questo valore viene stampato in una pagina HTML senza codifica, il browser può eseguirlo come JavaScript.

ASP.NET Core e Razor codificano l’output lato server per impostazione predefinita; Microsoft avverte però che usare `Html.Raw` o codice personalizzato con input non attendibile può aumentare il rischio di XSS. ([Microsoft Learn][2])

Esempio Razor sicuro:

```cshtml id="uif6cg"
<p>@Model.Nome</p>
```

Razor codifica il contenuto.

Esempio pericoloso:

```cshtml id="toopxm"
@Html.Raw(Model.Nome)
```

Se `Model.Nome` contiene HTML o script non attendibile, può diventare vulnerabile.

---

## Encoding dipendente dal contesto

Il modo corretto di codificare dipende dal contesto:

```text id="7typ22"
HTML body
HTML attribute
JavaScript
URL
CSS
JSON
```

OWASP sottolinea che non esiste una singola tecnica che risolva tutta la prevenzione XSS: bisogna usare la combinazione corretta di difese e codifica in base al contesto. ([cheatsheetseries.owasp.org][3])

Regola pratica:

```text id="70o34i"
Non concatenare input utente dentro HTML, JavaScript o SQL.
Usa i meccanismi del framework e codifica nel contesto corretto.
```

---

# SQL injection

La **SQL injection** avviene quando input non attendibile viene concatenato dentro una query SQL.

Esempio vulnerabile:

```csharp id="jrn92n"
string sql =
    "SELECT * FROM Clienti WHERE Nome = '" + nome + "'";

SqlCommand command = new(sql, connection);
```

Se `nome` contiene:

```sql id="og10v4"
' OR 1=1 --
```

la query cambia significato.

OWASP indica la SQL injection come una vulnerabilità comune e critica perché il database contiene spesso dati sensibili o fondamentali; le difese principali includono query parametrizzate, stored procedure sicure, allow-list validation e, in casi specifici, escaping corretto. ([cheatsheetseries.owasp.org][4])

---

## Query parametrizzata con ADO.NET

Corretto:

```csharp id="65u23h"
using SqlCommand command = new(
    "SELECT CLIE_CODI, NOME FROM CLIENTI_GEN WHERE NOME = @Nome",
    connection);

command.Parameters.Add(
    "@Nome",
    SqlDbType.VarChar,
    100).Value = nome;

using SqlDataReader reader =
    await command.ExecuteReaderAsync(cancellationToken);
```

Il parametro non viene interpretato come parte del comando SQL.

---

## EF Core e query sicure

Con LINQ, EF Core parametrizza normalmente i valori:

```csharp id="s4iqkr"
Cliente? cliente = await dbContext.Clienti
    .FirstOrDefaultAsync(
        c => c.Nome == nome,
        cancellationToken);
```

Attenzione però alle query raw.

Sicuro:

```csharp id="w0e8x7"
var clienti = await dbContext.Clienti
    .FromSqlInterpolated(
        $"SELECT * FROM Clienti WHERE Nome = {nome}")
    .ToListAsync(cancellationToken);
```

Pericoloso:

```csharp id="2zq83q"
string sql =
    $"SELECT * FROM Clienti WHERE Nome = '{nome}'";

var clienti = await dbContext.Clienti
    .FromSqlRaw(sql)
    .ToListAsync(cancellationToken);
```

Regola pratica:

```text id="em871s"
Mai concatenare input utente dentro SQL.
Usa sempre parametri.
```

# XSS

**XSS**, Cross-Site Scripting, si verifica quando un’applicazione accetta input utente e lo restituisce in una pagina senza validazione, encoding o escaping adeguati; Microsoft descrive proprio questa dinamica come causa generale delle vulnerabilità XSS. ([Microsoft Learn][5])

Esempio:

```text id="b2jd6d"
Un utente inserisce un commento.
Il commento contiene JavaScript.
L’app mostra il commento ad altri utenti.
Il browser esegue lo script.
```

---

## Esempio vulnerabile

```cshtml id="o4gzpd"
<div>
    @Html.Raw(Model.Commento)
</div>
```

Se `Commento` viene da un utente, è pericoloso.

Meglio:

```cshtml id="p6qsqw"
<div>
    @Model.Commento
</div>
```

Razor codifica l’output.

---

## Difese contro XSS

```text id="x89r4g"
output encoding
validazione input
sanitizzazione HTML quando accetti HTML
Content Security Policy
HttpOnly cookie
evitare Html.Raw con input non attendibile
evitare concatenazione di stringhe in JavaScript
```

Attenzione:

```text id="ihvywj"
Se permetti agli utenti di inserire HTML,
non basta codificare tutto.
Devi sanitizzare con una libreria affidabile e limitare i tag permessi.
```

# CSRF

**CSRF**, Cross-Site Request Forgery, avviene quando un sito malevolo induce il browser di un utente autenticato a eseguire un’azione indesiderata su un sito affidabile; OWASP spiega che, se l’utente è autenticato, il sito vulnerabile può non distinguere una richiesta legittima da una forgiata. ([cheatsheetseries.owasp.org][6])

Scenario:

```text id="z47z8z"
L’utente è loggato su banca.example.
Visita sito-malevolo.example.
Il sito malevolo invia una POST a banca.example.
Il browser allega automaticamente i cookie.
L’azione viene eseguita come se fosse l’utente.
```

---

## Quando è rilevante

CSRF è particolarmente rilevante quando usi autenticazione basata su cookie.

Esempi:

```text id="hdqrbp"
Razor Pages
MVC
Blazor Server con cookie
app web tradizionale
```

È meno tipico per API chiamate con Bearer token in header, perché il browser non allega automaticamente il token come fa con i cookie. Ma la configurazione concreta conta.

---

## Difese contro CSRF

```text id="xvd4jr"
anti-forgery token
SameSite cookie
controllo Origin/Referer dove appropriato
non usare GET per azioni che modificano stato
autenticazione corretta
protezione XSS
```

ASP.NET Core include supporto anti-forgery/XSRF, con particolare importanza negli scenari basati su cookie e form server-side. ([Microsoft Learn][1])

Esempio concettuale Razor:

```cshtml id="rwic7z"
<form method="post">
    @Html.AntiForgeryToken()

    <input name="Nome" />
    <button type="submit">Salva</button>
</form>
```

# CORS

**CORS**, Cross-Origin Resource Sharing, controlla quali origini esterne possono chiamare una API dal browser.

Esempio di origine:

```text id="klc5pv"
https://app.example.com
```

Diversa da:

```text id="wtrror"
https://api.example.com
```

Configurazione permissiva e pericolosa:

```csharp id="lpagc1"
builder.Services.AddCors(options =>
{
    options.AddPolicy("Open", policy =>
    {
        policy
            .AllowAnyOrigin()
            .AllowAnyHeader()
            .AllowAnyMethod();
    });
});
```

Meglio limitare:

```csharp id="cq1604"
builder.Services.AddCors(options =>
{
    options.AddPolicy("Frontend", policy =>
    {
        policy
            .WithOrigins("https://app.example.com")
            .AllowAnyHeader()
            .AllowAnyMethod();
    });
});
```

Uso:

```csharp id="t0sme6"
app.UseCors("Frontend");
```

---

## Attenzione a CORS e credenziali

Se usi cookie o credenziali:

```csharp id="z3speb"
policy
    .WithOrigins("https://app.example.com")
    .AllowCredentials();
```

Non combinare in modo superficiale credenziali e origine aperta.

Regola pratica:

```text id="exszmg"
CORS non è autenticazione.
CORS limita quali browser possono chiamare la tua API da certe origini.
Non sostituisce autorizzazione, token o policy server-side.
```

# Logging sicuro

I log sono fondamentali per diagnosi e audit, ma possono diventare una perdita di dati.

Non loggare:

```text id="po9dyo"
password
token
cookie
authorization header
codici OTP
dati di carte
segreti
documenti completi
dati personali non necessari
connection string
payload sensibili integrali
```

Sbagliato:

```csharp id="4anq8k"
logger.LogInformation(
    "Login request: {@Request}",
    request);
```

se `request` contiene password.

Meglio:

```csharp id="mj8mlz"
logger.LogInformation(
    "Tentativo login per utente {UserName}",
    request.UserName);
```

---

## Logging strutturato

Preferisci logging strutturato:

```csharp id="e3qf1t"
logger.LogInformation(
    "Ordine {OrdineId} confermato per cliente {ClienteId}",
    ordine.Id,
    ordine.ClienteId);
```

Non:

```csharp id="yf6q1m"
logger.LogInformation(
    "Ordine " + ordine.Id + " confermato per cliente " + ordine.ClienteId);
```

Vantaggi:

```text id="o10qd2"
query sui log
correlation ID
filtri
dashboard
alert
osservabilità migliore
```

---

## Correlation ID

Per tracciare una richiesta:

```csharp id="a64fw1"
app.Use(async (context, next) =>
{
    string correlationId =
        context.Request.Headers.TryGetValue(
            "X-Correlation-ID",
            out var value)
            ? value.ToString()
            : Guid.NewGuid().ToString();

    using (logger.BeginScope(
        new Dictionary<string, object>
        {
            ["CorrelationId"] = correlationId
        }))
    {
        context.Response.Headers["X-Correlation-ID"] = correlationId;

        await next();
    }
});
```

# Password e hashing

Le password non devono mai essere salvate in chiaro.

Non salvare:

```text id="p2mig4"
Password = "ciao123"
```

Non basta nemmeno un semplice hash veloce:

```text id="oab6zt"
SHA256(password)
MD5(password)
```

Le password devono essere protette con algoritmi appositi, lenti e salati.

OWASP raccomanda di proteggere le password anche nel caso in cui applicazione o database vengano compromessi e osserva che molti framework moderni forniscono funzionalità integrate per conservarle in sicurezza. ([cheatsheetseries.owasp.org][7])

---

## In ASP.NET Core Identity

Se usi ASP.NET Core Identity, il framework gestisce hashing e verifica password.

Esempio concettuale:

```csharp id="qnfkj8"
var result = await userManager.CreateAsync(
    user,
    password);
```

Non devi implementare tu l’hashing.

Se devi farlo manualmente, usa primitive appropriate, non hash generici.

Algoritmi comunemente usati:

```text id="cblgqg"
Argon2
bcrypt
PBKDF2
scrypt
```

Regola pratica:

```text id="er1jd6"
Non inventare un algoritmo di hashing password.
Usa librerie/framework affidabili.
```

# Least privilege

**Least privilege** significa concedere solo i permessi necessari.

Esempi:

```text id="dxod7a"
l’utente database dell’app non deve essere sysadmin
un servizio che legge non deve poter cancellare
un token API deve avere scope limitati
un account di servizio deve accedere solo alle risorse necessarie
un utente applicativo deve vedere solo le funzioni permesse
```

---

## Database

Sbagliato:

```text id="l3xk52"
applicazione connessa al database come sa/sysadmin
```

Meglio:

```text id="d1z4ji"
utente applicativo dedicato
permessi solo su schema/tabelle necessarie
permessi EXECUTE solo sulle stored procedure richieste
niente privilegi amministrativi
```

OWASP include linee guida specifiche per configurare in modo sicuro i database, inclusi Microsoft SQL Server e altri database relazionali. ([cheatsheetseries.owasp.org][8])

---

## File system

Sbagliato:

```text id="doow5v"
application pool identity con accesso completo a tutto il disco
```

Meglio:

```text id="y8o9k5"
accesso solo alle cartelle necessarie
permessi di sola lettura dove basta leggere
cartelle upload separate
niente esecuzione da cartelle upload
```

## Applicazione

Sbagliato:

```csharp id="r3gtwh"
[Authorize]

{
}
```

se ogni utente autenticato accede a funzioni amministrative.

```csharp id="1twbsw"
[Authorize(Policy = "AdminOnly")]
```

# Esempio completo: endpoint più sicuro

Request:

```csharp id="jalydh"
public record AggiornaEmailRequest(
    string Email);
```

Endpoint:

```csharp id="d1rrvc"
app.MapPut("/api/clienti/{id:int}/email", async (
    int id,
    AggiornaEmailRequest request,
    ClienteService service,
    ILogger<Program> logger,
    CancellationToken cancellationToken) =>
{
    if (id <= 0)
    {
        return Results.BadRequest("Id cliente non valido.");
    }

    if (string.IsNullOrWhiteSpace(request.Email) ||
        !request.Email.Contains('@') ||
        request.Email.Length > 254)
    {
        return Results.BadRequest("Email non valida.");
    }

    Result result = await service.AggiornaEmailAsync(
        id,
        request.Email,
        cancellationToken);

    if (result.IsFailure)
    {
        logger.LogWarning(
            "Aggiornamento email fallito per cliente {ClienteId}: {Error}",
            id,
            result.Error);

        return Results.BadRequest(result.Error);
    }

    logger.LogInformation(
        "Email aggiornata per cliente {ClienteId}",
        id);

    return Results.NoContent();
})
.RequireAuthorization("Clienti.Update");
```

Service:

```csharp id="mow10c"
public async Task<Result> AggiornaEmailAsync(
    int clienteId,
    string email,
    CancellationToken cancellationToken)
{
    Cliente? cliente = await dbContext.Clienti
        .FirstOrDefaultAsync(
            c => c.Id == clienteId,
            cancellationToken);

    if (cliente is null)
    {
        return Result.Failure("Cliente non trovato.");
    }

    bool emailGiaUsata = await dbContext.Clienti
        .AnyAsync(
            c => c.Email == email && c.Id != clienteId,
            cancellationToken);

    if (emailGiaUsata)
    {
        return Result.Failure("Email già utilizzata.");
    }

    cliente.Email = email;

    await dbContext.SaveChangesAsync(cancellationToken);

    return Result.Success();
}
```

Questo esempio mostra:

```text id="kig1gl"
validazione input
autorizzazione
Result pattern
query parametrizzate tramite EF Core
logging senza dati sensibili
nessuna password o token nei log
```

# Checklist minima di sicurezza per app ASP.NET Core

```text id="oi3pma"
Input
  validare lato server
  limitare lunghezze
  controllare formato
  normalizzare quando serve

Output
  usare encoding automatico
  evitare Html.Raw con input utente
  sanitizzare HTML se accetti HTML

Database
  usare parametri
  evitare SQL concatenato
  applicare least privilege
  gestire transazioni correttamente

Web
  usare HTTPS
  configurare CORS in modo restrittivo
  proteggere CSRF se usi cookie/form
  usare cookie Secure, HttpOnly, SameSite

Segreti
  non committare segreti
  usare user-secrets solo in sviluppo
  usare secret store in produzione

Log
  non loggare password, token, cookie o dati sensibili
  usare correlation ID
  loggare eventi di sicurezza rilevanti

Password
  non salvare in chiaro
  non usare hash veloci generici
  usare Identity o librerie affidabili

Permessi
  applicare least privilege
  usare policy di autorizzazione
  separare ruoli e permessi
```

# Errori comuni

## Fidarsi della validazione client-side

La validazione JavaScript migliora la UX, ma non è una difesa.

Il client può essere modificato.

La validazione deve essere lato server.

---

## Concatenare SQL

Sbagliato:

```csharp id="tco0uc"
"WHERE Nome = '" + nome + "'"
```

Usa parametri.

---

## Usare Html.Raw per comodità

Pericoloso se il contenuto non è completamente attendibile.

---

## Aprire CORS a tutti

Sospetto:

```csharp id="k1b84o"
AllowAnyOrigin()
```

soprattutto se l’API gestisce dati sensibili.

---

## Loggare payload completi

Pericoloso con dati personali, password, token o documenti.

---

## Salvare segreti in appsettings.json versionato

Non farlo.

---

## Usare account database privilegiati

L’applicazione non deve usare account amministrativi.

---

## Confondere autenticazione e autorizzazione

```text id="6hwyrj"
Autenticazione
  chi sei?

Autorizzazione
  cosa puoi fare?
```

Un utente autenticato non deve poter fare tutto.

---

# Linee guida pratiche

```text id="ys2319"
Valida sempre lato server
  anche se hai validazione client.

Codifica sempre l’output
  nel contesto corretto.

Usa query parametrizzate
  sempre.

Evita Html.Raw
  con dati non attendibili.

Proteggi CSRF
  nelle app basate su cookie e form.

Configura CORS in modo restrittivo
  solo per origini necessarie.

Non mettere segreti nel codice
  né nel repository.

Forza HTTPS
  e configura cookie sicuri.

Logga abbastanza per diagnosticare
  ma mai segreti o dati sensibili inutili.

Usa hashing password affidabile
  preferibilmente tramite ASP.NET Core Identity.

Applica least privilege
  su database, file system, API e ruoli applicativi.
```

```text id="pc1n85"
  impedisce che il dato venga interpretato come codice nel contesto sbagliato.

Parametrizzazione SQL
  impedisce che il dato venga interpretato come comando SQL.

Autorizzazione
  decide se l’utente può eseguire l’azione.

Least privilege
  limita i danni se qualcosa viene compromesso.
```

La regola principale è questa:

```text id="l2mrvz"
La sicurezza non si aggiunge alla fine.
Si progetta in ogni confine dell’applicazione:
input, output, database, configurazione, log, identità,
rete, permessi e integrazioni esterne.
```

Questo capitolo prepara il terreno per i successivi: autenticazione, autorizzazione, Identity, cookie, JWT, policy, ruoli, claims, sicurezza delle API e configurazione production-ready.

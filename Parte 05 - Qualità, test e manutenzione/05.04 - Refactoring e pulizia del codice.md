# 05.04 — Refactoring e pulizia del codice

Scrivere codice che “funziona” è solo una parte del lavoro.

In un progetto reale il codice deve anche essere:

```text id="3l2z4c"
leggibile
modificabile
testabile
estendibile
diagnosticabile
comprensibile da altre persone
comprensibile da noi stessi tra sei mesi
```

Il **refactoring** è l’attività con cui miglioriamo la struttura interna del codice senza cambiarne il comportamento osservabile.

Il punto centrale del capitolo è questo:

```text id="6r9c1v"
Il refactoring non serve a rendere il codice più elegante in astratto.
Serve a rendere il software più facile da capire, modificare e mantenere.
```

---

# Che cos’è il refactoring

Rifattorizzare significa modificare il codice per migliorarne la struttura, mantenendo invariato il comportamento.

Esempio semplice.

Prima:

```csharp id="mpzqu8"
public decimal CalcolaTotale(decimal prezzo, int quantita, decimal scontoPercentuale)
{
    decimal lordo = prezzo * quantita;
    decimal sconto = lordo * scontoPercentuale / 100m;
    decimal totale = lordo - sconto;

    return totale;
}
```

Dopo:

```csharp id="tb9t4t"
public decimal CalcolaTotale(decimal prezzo, int quantita, decimal scontoPercentuale)
{
    decimal lordo = CalcolaLordo(prezzo, quantita);
    decimal sconto = CalcolaSconto(lordo, scontoPercentuale);

    return lordo - sconto;
}

private static decimal CalcolaLordo(decimal prezzo, int quantita)
{
    return prezzo * quantita;
}

private static decimal CalcolaSconto(decimal lordo, decimal scontoPercentuale)
{
    return lordo * scontoPercentuale / 100m;
}
```

Il risultato non cambia.
Cambia la leggibilità.

---

# Refactoring non è riscrittura

Refactoring:

```text id="td6a5p"
modifica graduale
piccoli passi
comportamento invariato
test esistenti ancora validi
rischio controllato
```

Riscrittura:

```text id="extqqj"
cambio ampio
nuova implementazione
rischio maggiore
comportamento da riconfermare
possibile cambio architetturale
```

Riscrivere tutto da zero può sembrare più pulito, ma spesso è rischioso.

Regola pratica:

```text id="t8k5me"
Rifattorizza quando puoi migliorare per piccoli passi.
Riscrivi solo quando la struttura esistente impedisce davvero l’evoluzione.
```

---

# Code smell

Un **code smell** è un segnale che il codice potrebbe avere un problema di progettazione.

Non è una prova assoluta di errore.

È un indizio.

Esempi:

```text id="qcp2kc"
metodi troppo lunghi
classi troppo grandi
duplicazione
nomi poco chiari
troppi parametri
troppi if/switch
dipendenze statiche
codice difficile da testare
commenti che spiegano codice confuso
logica di business mischiata alla UI
accesso ai dati mischiato al dominio
```

Un code smell non significa automaticamente:

```text id="j1cf5c"
questo codice è sbagliato
```

Significa:

```text id="hug4mp"
questo codice merita attenzione
```

---

# Metodi troppo lunghi

Un metodo troppo lungo tende a fare troppe cose.

Esempio:

```csharp id="a88xve"
public void RegistraDocumento(DocumentoDto dto)
{
    if (dto.ClienteId <= 0)
    {
        throw new ArgumentException("Cliente non valido.");
    }

    if (dto.Righe.Count == 0)
    {
        throw new ArgumentException("Il documento deve avere almeno una riga.");
    }

    decimal totale = 0m;

    foreach (RigaDocumentoDto riga in dto.Righe)
    {
        if (riga.Quantita <= 0)
        {
            throw new ArgumentException("Quantità non valida.");
        }

        if (riga.PrezzoUnitario < 0)
        {
            throw new ArgumentException("Prezzo non valido.");
        }

        decimal subtotale = riga.Quantita * riga.PrezzoUnitario;
        decimal sconto = subtotale * riga.ScontoPercentuale / 100m;

        totale += subtotale - sconto;
    }

    Documento documento = new Documento
    {
        ClienteId = dto.ClienteId,
        Data = DateTime.Today,
        Totale = totale
    };

    dbContext.Documenti.Add(documento);
    dbContext.SaveChanges();

    logger.LogInformation("Documento registrato {DocumentoId}", documento.Id);
}
```

Questo metodo fa molte cose:

```text id="s2z4t2"
valida il documento
valida le righe
calcola il totale
crea l’entità
salva su database
scrive log
```

## Refactoring con Extract Method

Possiamo estrarre metodi più piccoli.

```csharp id="nmk4t5"
public void RegistraDocumento(DocumentoDto dto)
{
    ValidaDocumento(dto);

    decimal totale = CalcolaTotale(dto.Righe);

    Documento documento = CreaDocumento(dto, totale);

    dbContext.Documenti.Add(documento);
    dbContext.SaveChanges();

    logger.LogInformation("Documento registrato {DocumentoId}", documento.Id);
}

private static void ValidaDocumento(DocumentoDto dto)
{
    if (dto.ClienteId <= 0)
    {
        throw new ArgumentException("Cliente non valido.");
    }

    if (dto.Righe.Count == 0)
    {
        throw new ArgumentException("Il documento deve avere almeno una riga.");
    }
}

private static decimal CalcolaTotale(IEnumerable<RigaDocumentoDto> righe)
{
    decimal totale = 0m;

    foreach (RigaDocumentoDto riga in righe)
    {
        ValidaRiga(riga);

        decimal subtotale = riga.Quantita * riga.PrezzoUnitario;
        decimal sconto = subtotale * riga.ScontoPercentuale / 100m;

        totale += subtotale - sconto;
    }

    return totale;
}

private static void ValidaRiga(RigaDocumentoDto riga)
{
    if (riga.Quantita <= 0)
    {
        throw new ArgumentException("Quantità non valida.");
    }

    if (riga.PrezzoUnitario < 0)
    {
        throw new ArgumentException("Prezzo non valido.");
    }
}

private static Documento CreaDocumento(DocumentoDto dto, decimal totale)
{
    return new Documento
    {
        ClienteId = dto.ClienteId,
        Data = DateTime.Today,
        Totale = totale
    };
}
```

Il metodo principale ora racconta il flusso.

I dettagli sono spostati in metodi con nomi espliciti.

# Classi troppo grandi

Una classe troppo grande tende ad avere troppe responsabilità.

Esempio:

```csharp id="l7a6ns"
public class ClienteService
{
    public void RegistraCliente() { }
    public void ModificaCliente() { }
    public void CancellaCliente() { }
    public void EsportaClientiCsv() { }
    public void InviaEmailBenvenuto() { }
    public void CalcolaPunteggioMarketing() { }
    public void GeneraReportPdf() { }
    public void SincronizzaConCrm() { }
}
```

Questa classe sembra un contenitore generico per tutto ciò che riguarda i clienti.

Potrebbe essere meglio separare:

```text id="8q2l1y"
ClienteRegistrationService
ClienteExportService
ClienteNotificationService
ClienteScoringService
ClienteReportService
CrmClienteSyncService
```

---

## Segnali di classe troppo grande

```text id="v8j5za"
molti campi privati
molte dipendenze nel costruttore
metodi che usano solo una parte dei campi
nome generico come Manager, Helper, Utility, Service
modifiche frequenti per motivi diversi
test difficili da scrivere
```

Se una classe cambia per tanti motivi diversi, probabilmente ha troppe responsabilità.

# Duplicazione

La duplicazione è uno dei code smell più comuni.

Esempio:

```csharp id="xzclmf"
public decimal CalcolaTotalePreventivo(Documento documento)
{
    decimal totale = 0m;

    foreach (RigaDocumento riga in documento.Righe)
    {
        decimal subtotale = riga.Quantita * riga.PrezzoUnitario;
        decimal sconto = subtotale * riga.ScontoPercentuale / 100m;

        totale += subtotale - sconto;
    }

    return totale;
}

public decimal CalcolaTotaleFattura(Documento documento)
{
    decimal totale = 0m;

    foreach (RigaDocumento riga in documento.Righe)
    {
        decimal subtotale = riga.Quantita * riga.PrezzoUnitario;
        decimal sconto = subtotale * riga.ScontoPercentuale / 100m;

        totale += subtotale - sconto;
    }

    return totale;
}
```

La logica è duplicata.

Meglio:

```csharp id="xqlmcj"
public decimal CalcolaTotale(Documento documento)
{
    return documento.Righe.Sum(CalcolaSubtotale);
}

private static decimal CalcolaSubtotale(RigaDocumento riga)
{
    decimal subtotale = riga.Quantita * riga.PrezzoUnitario;
    decimal sconto = subtotale * riga.ScontoPercentuale / 100m;

    return subtotale - sconto;
}
```

---

## Duplicazione reale e duplicazione apparente

Non tutta la somiglianza è duplicazione da eliminare subito.

Due pezzi di codice possono sembrare simili oggi, ma evolvere in direzioni diverse domani.

Esempio:

```text id="g8vcsq"
calcolo prezzo preventivo
calcolo prezzo fattura fiscale
```

All’inizio potrebbero essere identici, ma poi una fattura può avere regole fiscali diverse.

Regola pratica:

```text id="lggkfz"
Elimina duplicazione quando rappresenta la stessa regola.
Non unificare troppo presto codice che ha motivi diversi per cambiare.
```

# Nomi poco chiari

I nomi sono una parte fondamentale del design.

Esempio debole:

```csharp id="nbdt48"
public decimal Calc(decimal a, int b, decimal c)
{
    return a * b - c;
}
```

Esempio migliore:

```csharp id="dcqacs"
public decimal CalcolaTotaleRiga(
    decimal prezzoUnitario,
    int quantita,
    decimal importoSconto)
{
    return prezzoUnitario * quantita - importoSconto;
}
```

Il codice migliore non richiede al lettore di indovinare.

# Separazione delle responsabilità

La separazione delle responsabilità significa che ogni parte del codice dovrebbe avere un motivo chiaro per esistere.

Esempio sbagliato:

```csharp id="jpfcbz"
public class DocumentoService
{
    public void RegistraDocumento(DocumentoDto dto)
    {
        // validazione
        // calcolo totale
        // accesso database
        // generazione PDF
        // invio email
        // logging tecnico
    }
}
```

Meglio separare:

```text id="8f9ox7"
DocumentoValidator
DocumentoTotaleCalculator
DocumentoRepository
DocumentoPdfGenerator
EmailSender
DocumentoApplicationService
```

Il servizio applicativo può orchestrare, ma non deve fare tutto direttamente.

---

## Esempio

```csharp id="3de8b8"
public class DocumentoApplicationService
{
    private readonly DocumentoValidator validator;
    private readonly DocumentoTotaleCalculator totaleCalculator;
    private readonly IDocumentoRepository repository;
    private readonly IEmailSender emailSender;

    public DocumentoApplicationService(
        DocumentoValidator validator,
        DocumentoTotaleCalculator totaleCalculator,
        IDocumentoRepository repository,
        IEmailSender emailSender)
    {
        this.validator = validator;
        this.totaleCalculator = totaleCalculator;
        this.repository = repository;
        this.emailSender = emailSender;
    }

    public async Task RegistraAsync(DocumentoDto dto)
    {
        validator.Valida(dto);

        decimal totale = totaleCalculator.Calcola(dto.Righe);

        Documento documento = Documento.Crea(dto.ClienteId, totale);

        await repository.SalvaAsync(documento);

        await emailSender.InviaConfermaDocumentoAsync(documento.Id);
    }
}
```

Questa classe orchestra il caso d’uso.

Le singole responsabilità sono delegate.

# Refactoring sicuro

Il refactoring deve essere fatto in modo controllato.

Sequenza consigliata:

```text id="i3q8no"
1. Capire il comportamento attuale.
2. Aggiungere o verificare test automatici.
3. Fare un cambiamento piccolo.
4. Eseguire i test.
5. Fare commit.
6. Ripetere.
```

Il refactoring senza test è possibile, ma più rischioso.

In quel caso bisogna procedere con ancora più cautela:

```text id="v6usbn"
piccoli passi
debug mirato
confronto output prima/dopo
log temporanei
review accurata
```

## Refactoring con test

Supponiamo di voler rifattorizzare:

```csharp id="0028v7"
public decimal CalcolaTotale(IEnumerable<RigaDocumento> righe)
{
    return righe
        .Where(r => r.Attiva)
        .Sum(r => r.Quantita * r.PrezzoUnitario);
}
```

Prima scriviamo test:

```csharp id="6etxzw"
[Fact]
public void CalcolaTotale_ignoraRigheNonAttive()
{
    DocumentoTotaleCalculator calculator = new DocumentoTotaleCalculator();

    List<RigaDocumento> righe =
    [
        new RigaDocumento("Libro", 2, 30m, true),
        new RigaDocumento("Corso", 1, 100m, false)
    ];

    decimal totale = calculator.CalcolaTotale(righe);

    Assert.Equal(60m, totale);
}
```

Poi possiamo cambiare implementazione:

```csharp id="t7sw5u"
public decimal CalcolaTotale(IEnumerable<RigaDocumento> righe)
{
    decimal totale = 0m;

    foreach (RigaDocumento riga in righe)
    {
        if (!riga.Attiva)
        {
            continue;
        }

        totale += riga.Quantita * riga.PrezzoUnitario;
    }

    return totale;
}
```

Se il test passa, il comportamento è preservato.

# Introduzione ai principi SOLID

I principi SOLID sono cinque linee guida per progettare codice più manutenibile.

```text id="1vhj2v"
S - Single Responsibility Principle
O - Open/Closed Principle
L - Liskov Substitution Principle
I - Interface Segregation Principle
D - Dependency Inversion Principle
```

Non sono leggi matematiche.

Sono criteri di progettazione da usare con buon senso.

---

## S — Single Responsibility Principle

Una classe dovrebbe avere una responsabilità principale e un solo motivo significativo per cambiare.

Classe sospetta:

```csharp id="k5pmfj"
public class ReportService
{
    public IEnumerable<DatoReport> CaricaDati() { }
    public byte[] GeneraPdf() { }
    public void InviaEmail() { }
    public void SalvaSuDisco() { }
}
```

Possibile separazione:

```text id="6a66gg"
ReportDataProvider
ReportPdfGenerator
EmailSender
FileStorage
ReportApplicationService
```

---

## O — Open/Closed Principle

Il codice dovrebbe essere aperto all’estensione ma chiuso alla modifica.

Esempio con `switch` fragile:

```csharp id="ercib8"
public decimal CalcolaSpedizione(string tipo, decimal importo)
{
    return tipo switch
    {
        "Standard" => 5m,
        "Express" => 10m,
        "Premium" => 0m,
        _ => throw new NotSupportedException()
    };
}
```

Se aggiungiamo un nuovo tipo, modifichiamo il metodo.

Alternativa con strategia:

```csharp id="jrxu3v"
public interface ISpedizioneStrategy
{
    string Tipo { get; }
    decimal Calcola(decimal importo);
}
```

```csharp id="s4zxuu"
public class SpedizioneStandardStrategy : ISpedizioneStrategy
{
    public string Tipo => "Standard";

    public decimal Calcola(decimal importo)
    {
        return 5m;
    }
}
```

Aggiungere una nuova strategia richiede una nuova classe, non la modifica del metodo centrale.

---

## L — Liskov Substitution Principle

Una classe derivata deve poter sostituire la classe base senza rompere il comportamento atteso.

Esempio problematico:

```csharp id="p8unog"
public class Uccello
{
    public virtual void Vola()
    {
    }
}

public class Pinguino : Uccello
{
    public override void Vola()
    {
        throw new NotSupportedException();
    }
}
```

Se il codice si aspetta che ogni `Uccello` voli, `Pinguino` rompe la sostituibilità.

Meglio modellare diversamente:

```csharp id="q2r48h"
public interface IAnimale
{
}

public interface IVolante
{
    void Vola();
}
```

Solo gli animali che volano implementano `IVolante`.

---

## I — Interface Segregation Principle

Meglio interfacce piccole e specifiche che interfacce enormi.

Sospetto:

```csharp id="48kl12"
public interface IClienteService
{
    void Registra();
    void EsportaCsv();
    void InviaEmail();
    void GeneraPdf();
    void SincronizzaCrm();
}
```

Meglio separare:

```csharp id="j2d8ry"
public interface IClienteRegistrationService
{
    void Registra();
}

public interface IClienteExporter
{
    void EsportaCsv();
}

public interface IClienteNotifier
{
    void InviaEmail();
}
```

Una classe non dovrebbe essere costretta a implementare metodi che non le servono.

---

## D — Dependency Inversion Principle

Il codice ad alto livello dovrebbe dipendere da astrazioni, non da dettagli concreti.

Sbagliato:

```csharp id="7960vz"
public class OrdineService
{
    private readonly SmtpEmailSender emailSender = new SmtpEmailSender();
}
```

Meglio:

```csharp id="shnp1s"
public class OrdineService
{
    private readonly IEmailSender emailSender;

    public OrdineService(IEmailSender emailSender)
    {
        this.emailSender = emailSender;
    }
}
```

Questo principio è strettamente collegato alla Dependency Injection.

# Tecnica: Extract Method

**Extract Method** consiste nel prendere un blocco di codice e spostarlo in un metodo con un nome chiaro.

Prima:

```csharp id="zhu2gj"
public decimal CalcolaTotale(Documento documento)
{
    decimal totale = 0m;

    foreach (RigaDocumento riga in documento.Righe)
    {
        decimal lordo = riga.Quantita * riga.PrezzoUnitario;
        decimal sconto = lordo * riga.ScontoPercentuale / 100m;
        totale += lordo - sconto;
    }

    return totale;
}
```

Dopo:

```csharp id="7fuc62"
public decimal CalcolaTotale(Documento documento)
{
    return documento.Righe.Sum(CalcolaSubtotale);
}

private static decimal CalcolaSubtotale(RigaDocumento riga)
{
    decimal lordo = riga.Quantita * riga.PrezzoUnitario;
    decimal sconto = lordo * riga.ScontoPercentuale / 100m;

    return lordo - sconto;
}
```

Vantaggio:

```text id="6wre9l"
il metodo principale diventa più leggibile
la logica estratta può essere testata o riusata
il nome del metodo documenta l’intenzione
```

# Tecnica: Extract Class

**Extract Class** consiste nel separare una parte di responsabilità in una nuova classe.

Prima:

```csharp id="m6m1wm"
public class DocumentoService
{
    public decimal CalcolaTotale(Documento documento)
    {
        // calcolo totale
    }

    public void Salva(Documento documento)
    {
        // accesso database
    }

    public byte[] GeneraPdf(Documento documento)
    {
        // generazione PDF
    }
}
```

Dopo:

```csharp id="5y049p"
public class DocumentoTotaleCalculator
{
    public decimal Calcola(Documento documento)
    {
        return documento.Righe.Sum(r =>
        {
            decimal lordo = r.Quantita * r.PrezzoUnitario;
            decimal sconto = lordo * r.ScontoPercentuale / 100m;

            return lordo - sconto;
        });
    }
}
```

```csharp id="4buza2"
public class DocumentoPdfGenerator
{
    public byte[] Genera(Documento documento)
    {
        // generazione PDF
        return [];
    }
}
```

```csharp id="ja59w7"
public class DocumentoApplicationService
{
    private readonly DocumentoTotaleCalculator calculator;
    private readonly DocumentoPdfGenerator pdfGenerator;

    public DocumentoApplicationService(
        DocumentoTotaleCalculator calculator,
        DocumentoPdfGenerator pdfGenerator)
    {
        this.calculator = calculator;
        this.pdfGenerator = pdfGenerator;
    }
}
```

# Tecnica: Introduce Parameter Object

Un metodo con troppi parametri è difficile da leggere.

Sospetto:

```csharp id="40l06m"
public Cliente RegistraCliente(
    string nome,
    string cognome,
    string email,
    string telefono,
    string indirizzo,
    string citta,
    string cap,
    string nazione)
{
    // ...
}
```

Meglio introdurre un oggetto parametro.

```csharp id="bgtjlx"
public class RegistraClienteCommand
{
    public string Nome { get; init; } = "";
    public string Cognome { get; init; } = "";
    public string Email { get; init; } = "";
    public string? Telefono { get; init; }
    public IndirizzoDto Indirizzo { get; init; } = new();
}
```

```csharp id="q0w1v5"
public class IndirizzoDto
{
    public string Via { get; init; } = "";
    public string Citta { get; init; } = "";
    public string Cap { get; init; } = "";
    public string Nazione { get; init; } = "";
}
```

Metodo:

```csharp id="x4f8wb"
public Cliente RegistraCliente(RegistraClienteCommand command)
{
    // ...
}
```

Vantaggi:

```text id="gmijim"
firma più leggibile
parametri raggruppati per significato
più semplice aggiungere campi
più semplice validare input
più adatto ad API e casi d’uso
```

# Tecnica: Replace Conditional with Polymorphism

Quando un `switch` o una catena di `if` cresce molto, può essere utile sostituirla con polimorfismo.

Prima:

```csharp id="mcdz8o"
public decimal CalcolaSconto(Cliente cliente, decimal importo)
{
    if (cliente.Tipo == "Standard")
    {
        return importo * 0.05m;
    }

    if (cliente.Tipo == "Gold")
    {
        return importo * 0.10m;
    }

    if (cliente.Tipo == "Platinum")
    {
        return importo * 0.15m;
    }

    return 0m;
}
```

Dopo:

```csharp id="cbpjew"
public interface IScontoClientePolicy
{
    string TipoCliente { get; }
    decimal CalcolaSconto(decimal importo);
}
```

```csharp id="jz0euj"
public class ScontoStandardPolicy : IScontoClientePolicy
{
    public string TipoCliente => "Standard";

    public decimal CalcolaSconto(decimal importo)
    {
        return importo * 0.05m;
    }
}
```

```csharp id="rnfh45"
public class ScontoGoldPolicy : IScontoClientePolicy
{
    public string TipoCliente => "Gold";

    public decimal CalcolaSconto(decimal importo)
    {
        return importo * 0.10m;
    }
}
```

```csharp id="9bvzqx"
public class ScontoClienteService
{
    private readonly Dictionary<string, IScontoClientePolicy> policies;

    public ScontoClienteService(IEnumerable<IScontoClientePolicy> policies)
    {
        this.policies = policies.ToDictionary(p => p.TipoCliente);
    }

    public decimal CalcolaSconto(string tipoCliente, decimal importo)
    {
        if (!policies.TryGetValue(tipoCliente, out IScontoClientePolicy? policy))
        {
            return 0m;
        }

        return policy.CalcolaSconto(importo);
    }
}
```

Ora aggiungere un nuovo tipo cliente significa aggiungere una nuova classe policy.

---

## Quando non usare questa tecnica

Non ogni `switch` va eliminato.

Questo va benissimo:

```csharp id="xqecp2"
string descrizione = stato switch
{
    StatoDocumento.Bozza => "Bozza",
    StatoDocumento.Confermato => "Confermato",
    StatoDocumento.Annullato => "Annullato",
    _ => "Sconosciuto"
};
```

Il polimorfismo serve quando:

```text id="j61b7z"
ogni ramo contiene logica significativa
i casi crescono spesso
ogni caso cambia per motivi diversi
il metodo centrale diventa difficile da mantenere
```

---

# Quando non rifattorizzare

Il refactoring è utile, ma non va fatto sempre.

Non rifattorizzare quando:

```text id="zjeea8"
non capisci ancora il comportamento
non hai test né modo di verificare
sei in emergenza produzione
il codice è stabile e non deve cambiare
il costo supera il beneficio
il refactoring è solo gusto personale
stai vicino a una scadenza critica senza margine
```

---

## Codice brutto ma stabile

A volte un pezzo di codice è brutto, ma:

```text id="4w6z2h"
funziona da anni
non viene quasi mai modificato
non ha bug noti
non blocca nuove funzionalità
```

Rifattorizzarlo può non essere prioritario.

Meglio concentrarsi su codice:

```text id="nepe8q"
che cambia spesso
che genera bug
che impedisce nuove funzionalità
che è difficile da testare
che è difficile da capire
```

---

## Refactoring opportunistico

Una buona regola pratica:

```text id="7r2q77"
Lascia il codice un po’ migliore di come lo hai trovato.
```

Questo non significa riscrivere tutto.

Significa che, mentre lavori su una parte, puoi migliorare nomi, estrarre un metodo, rimuovere duplicazione evidente o aggiungere un test.

---

# Esempio completo: da codice confuso a codice pulito

Partiamo da un metodo troppo grande:

```csharp id="sreskt"

{

    {
        decimal totale = 0m;

        foreach (RigaOrdine r in righe)
        {
            if (r.Qta <= 0)
            {
                throw new Exception("Qta errata");
            }

            if (r.Prezzo < 0)
            {
                throw new Exception("Prezzo errato");
            }

            decimal subtot = r.Qta * r.Prezzo;

            if (tipoCliente == "GOLD")
            {
                subtot = subtot - subtot * 0.10m;
            }
            else if (tipoCliente == "SILVER")
            {
                subtot = subtot - subtot * 0.05m;
            }

            totale += subtot;
        }

        return totale;
    }
}
```

```text id="35p31r"
nome metodo generico
nomi poco chiari
validazione mischiata al calcolo
sconti hard-coded
eccezioni generiche
if su tipo cliente
metodo difficile da estendere
```

## Secondo passo: Extract Method

```csharp id="cmttyn"
public class OrdineService
{

        foreach (RigaOrdine riga in righe)
```

```csharp id="edb3lh"
    private readonly RigaOrdineValidator validator;
    private readonly RigaOrdineCalculator rigaCalculator;
    private readonly ScontoClienteCalculator scontoCalculator;

    public OrdineService(
        RigaOrdineValidator validator,

        this.scontoCalculator = scontoCalculator;

    {
        decimal totale = 0m;

        foreach (RigaOrdine riga in righe)

            validator.Valida(riga);
```

# Errori comuni nel refactoring

## Cambiare comportamento senza accorgersene

Rifattorizzare significa mantenere lo stesso comportamento.

Se cambi anche la regola di business, non è più solo refactoring.

Meglio separare:

```text id="c2xcn9"
commit 1
  refactoring senza cambio comportamento

commit 2
  modifica funzionale
```

---

## Fare troppi cambiamenti insieme

Sbagliato:

```text id="oanfqe"
rinomino classi
cambio logica
sposto file
modifico database
cambio interfacce
ottimizzo performance
```

Tutto nello stesso intervento.

Meglio piccoli passi verificabili.

---

## Creare astrazioni premature

Sospetto:

```text id="srljvk"
introduco 5 interfacce per una sola implementazione
creo factory, strategy, provider e resolver senza bisogno reale
```

Le astrazioni hanno un costo.

Usale quando:

```text id="69n25j"
riducano complessità
permettano test
isolino infrastruttura
gestiscano variabilità reale
```

---

## Spezzare troppo

Anche metodi troppo piccoli possono peggiorare la leggibilità se costringono a saltare continuamente da un file all’altro.

Regola pratica:

```text id="f2ph3g"
Estrai quando il nome del nuovo metodo aggiunge significato.
Non estrarre solo per ridurre il numero di righe.
```

---

## Applicare SOLID in modo dogmatico

SOLID è utile, ma non deve trasformare codice semplice in un labirinto di classi.

Per un’applicazione piccola, un metodo chiaro può essere meglio di dieci classi astratte.

---

# Linee guida pratiche

```text id="bvrkpl"
Rifattorizza per ridurre rischio futuro
  non solo per gusto estetico.

Cerca code smell
  ma non trattarli come errori automatici.

Estrai metodi
  quando un blocco ha un’intenzione nominabile.

Estrai classi
  quando una classe ha più responsabilità.

Rimuovi duplicazione
  quando rappresenta la stessa regola.

Migliora i nomi
  perché il nome è parte del design.

Usa test automatici
  per rendere il refactoring più sicuro.

Applica SOLID con buon senso
  non come rituale.

Non rifattorizzare in emergenza
  se non puoi verificare il comportamento.

Fai piccoli passi
  e verifica dopo ogni passo.
```

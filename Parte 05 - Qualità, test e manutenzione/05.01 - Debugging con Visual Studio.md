# 05.01 — Debugging con Visual Studio

Fino a questo punto abbiamo studiato linguaggio, librerie, LINQ, ADO.NET ed EF Core. Ora entriamo in una parte fondamentale per lo sviluppo reale: **qualità, test e manutenzione**.

Prima ancora di scrivere test automatici avanzati, uno sviluppatore deve saper diagnosticare un problema mentre il programma è in esecuzione.

Il debugging serve a rispondere a domande come:

```text id="jwaawu"
Perché questa variabile ha questo valore?
Perché questo ramo di codice viene eseguito?
Dove nasce questa eccezione?
Perché questa query LINQ restituisce meno dati del previsto?
Perché questo metodo async non torna?
Perché questa operazione è lenta?
```

Visual Studio offre un debugger molto ricco. La documentazione Microsoft lo descrive come uno strumento per osservare il comportamento runtime dell’applicazione e capire cosa sta facendo il codice mentre viene eseguito. ([Microsoft Learn][1])

Il punto centrale del capitolo è questo:

```text id="op8b6x"
Il debugger non serve solo a fermare il programma.
Serve a formulare ipotesi, verificarle e capire il comportamento reale del codice.
```

---

# Debugging: che cosa significa davvero

Fare debug non significa “cliccare a caso finché funziona”.

Un buon processo di debugging segue un ragionamento:

```text id="osezj3"
1. Osservo il problema.
2. Formulo un’ipotesi.
3. Metto un breakpoint nel punto giusto.
4. Controllo valori, stack e flusso.
5. Confermo o scarto l’ipotesi.
6. Correggo il codice.
7. Verifico che il problema non si ripresenti.
```

Il debugger è uno strumento di indagine.

---

# Modalità Debug e modalità Release

In Visual Studio possiamo compilare in modalità:

```text id="mkihej"
Debug
Release
```

La modalità `Debug` è pensata per sviluppare e diagnosticare.

La modalità `Release` è pensata per distribuire l’applicazione.

Differenze tipiche:

```text id="s5qr8i"
Debug
  simboli di debug più completi
  ottimizzazioni ridotte
  esperienza di debug più chiara

Release
  codice ottimizzato
  performance migliori
  debug talvolta meno lineare
```

Per imparare e diagnosticare, normalmente si usa `Debug`.

# Breakpoint

Un **breakpoint** è un punto di interruzione.

Quando il programma raggiunge quella riga, Visual Studio sospende l’esecuzione e permette di ispezionare lo stato.

Esempio:

```csharp id="pi6byt"
public decimal CalcolaTotale(decimal prezzo, int quantita)
{
    decimal totale = prezzo * quantita;
    return totale;
}
```

Possiamo mettere un breakpoint sulla riga:

```csharp id="a5ipmx"
decimal totale = prezzo * quantita;
```

Quando il programma si ferma, possiamo vedere:

```text id="lo6cch"
prezzo
quantita
totale
call stack
thread corrente
eccezioni
oggetti in memoria
```

Microsoft documenta diversi tipi di breakpoint in Visual Studio: breakpoint standard, condizionali, tracepoint, data breakpoint, dependent breakpoint, temporary breakpoint e function breakpoint. ([Microsoft Learn][2])

---

## Impostare un breakpoint

In Visual Studio:

```text id="ore991"
clic sul margine sinistro della riga
oppure F9
```

Un pallino rosso indica che il breakpoint è attivo.

Durante il debug, possiamo usare:

```text id="d4pj6p"
F5
  continua l’esecuzione.

F10
  esegue la prossima istruzione senza entrare nei metodi.

F11
  entra nel metodo chiamato.

Shift + F11
  esce dal metodo corrente.
```

---

## Esempio pratico

```csharp id="snzs17"
public class ScontoService
{
    public decimal ApplicaSconto(decimal importo, decimal percentuale)
    {
        decimal sconto = importo * percentuale / 100m;
        decimal totale = importo - sconto;

        return totale;
    }
}
```

Se il totale non è quello atteso, possiamo mettere breakpoint su:

```csharp id="hynfwp"
decimal sconto = importo * percentuale / 100m;
```

Poi controllare:

```text id="ntbvof"
importo
percentuale
sconto
totale
```

# Conditional breakpoint

Un **conditional breakpoint** si attiva solo se una condizione è vera.

È molto utile quando un ciclo gira migliaia di volte, ma vogliamo fermarci solo in un caso specifico.

Esempio:

```csharp id="a439pm"
foreach (Cliente cliente in clienti)
{
    CalcolaPunteggio(cliente);
}
```

Vogliamo fermarci solo per il cliente con `Id == 123`.

Impostiamo un breakpoint sulla riga:

```csharp id="j7m4zv"
CalcolaPunteggio(cliente);
```

Poi aggiungiamo la condizione:

```csharp id="g62aks"
cliente.Id == 123
```

Visual Studio consente di usare breakpoint condizionali, filtri e hit count per fermare il codice solo quando viene raggiunto uno stato desiderato, per esempio quando una variabile assume un valore problematico. ([Microsoft Learn][3])

---

## Condizioni utili

Esempi:

```csharp id="s9skci"
cliente.Id == 123
```

```csharp id="xjc3z2"
totale < 0
```

```csharp id="q0hg2u"
nome == "Emilie"
```

```csharp id="cr9hhd"
ordine.Righe.Count == 0
```

```csharp id="j998cv"
exceptionMessage.Contains("timeout")
```

---

## Hit count

Un breakpoint può fermarsi solo dopo essere stato raggiunto un certo numero di volte.

Esempio:

```text id="h8jzlw"
fermati alla centesima esecuzione di questa riga
```

È utile quando un bug compare dopo molte iterazioni.

---

# Tracepoint

Un **tracepoint** è simile a un breakpoint, ma invece di fermare il programma scrive un messaggio nell’output.

È molto utile quando vogliamo osservare il flusso senza interrompere l’esecuzione.

Esempio messaggio:

```text id="k7u2cq"
Cliente elaborato: {cliente.Id} - {cliente.Nome}
```

Visual Studio include i tracepoint tra i tipi di breakpoint utili per registrare informazioni e seguire modifiche di stato senza sospendere necessariamente l’esecuzione del codice. ([Microsoft Learn][4])

---

## Quando usare tracepoint

Usa un tracepoint quando:

```text id="rmjaaf"
il problema è temporale
fermare il programma cambia il comportamento
vuoi vedere molte iterazioni
stai seguendo un flusso complesso
non vuoi modificare il codice con Console.WriteLine
```

Esempio pratico:

```csharp id="4h3ewa"
foreach (Ordine ordine in ordini)
{
    ProcessaOrdine(ordine);
}
```

Tracepoint:

```text id="6tkhsc"
Processo ordine {ordine.Id} cliente {ordine.ClienteId}
```

# Watch

La finestra **Watch** permette di osservare variabili ed espressioni durante il debug.

Esempi di espressioni utili:

```csharp id="2bv9wd"
cliente.Nome
```

```csharp id="48nqey"
ordine.Righe.Count
```

```csharp id="9mqx7t"
totale > 1000
```

```csharp id="v3abb7"
clienti.Where(c => c.Attivo).Count()
```

La Watch window è utile quando vogliamo tenere sotto controllo alcuni valori mentre facciamo step nel codice.

---

## Watch e proprietà con effetti collaterali

Attenzione: valutare un’espressione nella Watch può eseguire codice.

Esempio:

```csharp id="xm3b9j"
ordine.CalcolaTotale()
```

Se `CalcolaTotale()` modifica stato interno, chiamarla dalla Watch può alterare il comportamento.

Regola pratica:

```text id="g2y6bp"
Nella Watch preferisci proprietà semplici ed espressioni senza effetti collaterali.
```

---

# Immediate Window

La **Immediate Window** permette di eseguire espressioni mentre il programma è fermo in debug.

Esempi:

```csharp id="sf9yu7"
cliente.Nome
```

```csharp id="cut5ob"
cliente.Attivo = false
```

```csharp id="k4knyf"
ordini.Count
```

```csharp id="n0yj94"
CalcolaTotale(100m, 22m)
```

È utile per:

```text id="gg5rrn"
testare espressioni
verificare condizioni
modificare temporaneamente variabili
capire il comportamento di un metodo
preparare una condizione per un breakpoint condizionale
```

Un uso pratico molto efficace è provare nella Immediate Window l’espressione che poi useremo come condizione di un conditional breakpoint.

# Call stack

La finestra **Call Stack** mostra la catena di chiamate che ha portato il programma al punto corrente.

Esempio:

```text id="rywzuw"
ClienteController.Post
  ClienteService.Registra
    ClienteRepository.Salva
      AppDbContext.SaveChanges
```

Il call stack risponde alla domanda:

```text id="ucl1g3"
Come siamo arrivati qui?
```

È fondamentale quando un metodo viene chiamato da molti punti diversi.

---

## Esempio

```csharp id="wi1xap"
public void RegistraCliente(ClienteDto dto)
{
    ValidaCliente(dto);
    SalvaCliente(dto);
}
```

Se il debugger si ferma dentro `ValidaCliente`, il call stack ci mostra chi ha chiamato quel metodo.

Questo è utile quando:

```text id="rg6rbh"
un metodo è condiviso
un errore nasce lontano dal punto in cui esplode
una funzione viene chiamata in modo inatteso
stiamo analizzando codice legacy
```

Visual Studio consente anche di impostare breakpoint dalla finestra Call Stack quando il debugger è in pausa. ([Microsoft Learn][5])

---

# Exception settings

La finestra **Exception Settings** permette di decidere quando il debugger deve fermarsi in presenza di eccezioni.

Ci sono due momenti importanti:

```text id="7x1kfv"
thrown
  l’eccezione viene lanciata.

user-unhandled
  l’eccezione non viene gestita dal codice utente.
```

Di solito il debugger si ferma sulle eccezioni non gestite. Ma possiamo chiedergli di fermarsi appena una certa eccezione viene lanciata.

---

## Perché è utile

Supponiamo di avere:

```csharp id="zbfqal"
try
{
    ImportaFile(path);
}
catch (Exception ex)
{
    logger.LogError(ex, "Errore import");
}
```

L’eccezione viene catturata, quindi il programma non si ferma.

Ma se vogliamo scoprire il punto esatto in cui nasce, possiamo configurare Visual Studio per fermarsi quando quella specifica eccezione viene lanciata.

Esempi:

```text id="owjezi"
NullReferenceException
InvalidOperationException
SqlException
DbUpdateException
DbUpdateConcurrencyException
IOException
```

---

## Errori da non nascondere

Durante il debug è importante evitare catch troppo generici che ingoiano errori.

Sconsigliato:

```csharp id="s544rb"
try
{
    EseguiOperazione();
}
catch
{
}
```

Questo rende il debug molto più difficile.

Meglio:

```csharp id="budqtx"
try
{
    EseguiOperazione();
}
catch (Exception ex)
{
    logger.LogError(ex, "Errore durante l'operazione");
    throw;
}
```

# Debug di codice async

Il codice `async/await` è leggibile come codice sequenziale, ma viene eseguito tramite `Task`, continuazioni e scheduler.

Esempio:

```csharp id="ygo6lm"
public async Task<Cliente?> TrovaClienteAsync(int id)
{
    Cliente? cliente = await repository.TrovaAsync(id);

    return cliente;
}
```

Quando facciamo debug di codice async, dobbiamo ricordare che:

```text id="93m1hj"
il metodo può sospendersi su await
l’esecuzione può riprendere più tardi
lo stack fisico non sempre coincide con il flusso logico
più Task possono essere attivi contemporaneamente
```

Microsoft descrive il debugging di applicazioni async in Visual Studio usando la vista Tasks della finestra Parallel Stacks, utile per capire il comportamento runtime del codice basato su `async/await` e TAP. ([Microsoft Learn][6])

---

## Breakpoint dopo await

```csharp id="ihe2ct"
public async Task ElaboraAsync()
{
    Console.WriteLine("Prima");

    await Task.Delay(1000);

    Console.WriteLine("Dopo");
}
```

Possiamo mettere breakpoint:

```text id="5gv2gr"
prima dell'await
dopo l'await
dentro il metodo chiamato
```

Il debugger permette di seguire il flusso logico, ma è importante sapere che dopo `await` l’esecuzione può riprendere in un momento successivo.

---

## Parallel Stacks e Tasks

Per debug async più complesso, Visual Studio offre strumenti come:

```text id="aq0sp4"
Parallel Stacks
Tasks view
Parallel Watch
Threads
```

La documentazione Microsoft indica che la finestra Parallel Stacks è utile per applicazioni multithread e che, per app .NET basate su `async/await`, la vista Tasks è la più utile per visualizzare call stack centrati sulle attività. ([Microsoft Learn][7])

---

## Errori comuni nel debug async

### Dimenticare `await`

Sbagliato:

```csharp id="k9sb6q"
repository.SalvaAsync(cliente);
```

Corretto:

```csharp id="f95so0"
await repository.SalvaAsync(cliente);
```

Senza `await`, l’operazione può proseguire in modo inatteso e le eccezioni possono emergere altrove.

---

### Usare `.Result` o `.Wait()`

Sospetto:

```csharp id="2cat3s"
var cliente = repository.TrovaAsync(id).Result;
```

Meglio:

```csharp id="u3ju1m"
var cliente = await repository.TrovaAsync(id);
```

`.Result` e `.Wait()` possono creare deadlock o bloccare thread inutilmente, soprattutto in applicazioni UI o web.

# Debug di LINQ

LINQ è molto espressivo, ma può essere meno immediato da debuggare rispetto a un ciclo `foreach`.

Esempio:

```csharp id="9t1kb4"
var risultato = clienti
    .Where(c => c.Attivo)
    .Where(c => c.Citta == "Aosta")
    .Select(c => new ClienteDto(c.Id, c.Nome))
    .ToList();
```

Per capire dove si perde un dato, possiamo spezzare la query.

---

## Spezzare una query LINQ

Invece di:

```csharp id="n0p9dc"
var risultato = clienti
    .Where(c => c.Attivo && c.Citta == "Aosta")
    .Select(c => new ClienteDto(c.Id, c.Nome))
    .ToList();
```

Possiamo scrivere temporaneamente:

```csharp id="vfbh08"
var clientiAttivi = clienti
    .Where(c => c.Attivo)
    .ToList();

var clientiAosta = clientiAttivi
    .Where(c => c.Citta == "Aosta")
    .ToList();

var risultato = clientiAosta
    .Select(c => new ClienteDto(c.Id, c.Nome))
    .ToList();
```

Ora possiamo mettere breakpoint tra una fase e l’altra.

---

## Debug con lambda più leggibile

Possiamo espandere la lambda:

```csharp id="bb0sz8"
var clientiFiltrati = clienti
    .Where(c =>
    {
        bool attivo = c.Attivo;
        bool cittaCorretta = c.Citta == "Aosta";

        return attivo && cittaCorretta;
    })
    .ToList();
```

Questo permette di mettere breakpoint dentro la lambda e vedere i valori intermedi.

---

## Deferred execution e debug

Ricorda: molte query LINQ sono eseguite solo quando enumerate.

```csharp id="6ln8yz"
var query = clienti.Where(c => c.Attivo);
```

Qui la query non è ancora eseguita.

Viene eseguita qui:

```csharp id="ixol7j"
var lista = query.ToList();
```

oppure:

```csharp id="qcoq56"
foreach (Cliente cliente in query)
{
}
```

Durante il debug, questo è fondamentale.

# Hot Reload

**Hot Reload** permette di modificare il codice mentre l’applicazione è in esecuzione e applicare alcune modifiche senza riavviare completamente.

È utile per:

```text id="fg7bqn"
correggere piccoli bug
modificare logica locale
cambiare testo o condizioni
iterare più velocemente durante sviluppo UI o API
```

Visual Studio include strumenti di Hot Reload per applicare modifiche al codice durante il debug o l’esecuzione, nei limiti supportati dal tipo di modifica e dal progetto. ([Microsoft Learn][1])

---

## Quando Hot Reload aiuta

Esempio:

```csharp id="co8628"
if (cliente.Attivo)
{
    Console.WriteLine("Cliente attivo");
}
```

Possiamo modificare:

```csharp id="ds2mwi"
if (cliente.Attivo && cliente.Email is not null)
{
    Console.WriteLine("Cliente attivo con email");
}
```

e provare subito, se la modifica è supportata.

---

## Limiti di Hot Reload

Non tutte le modifiche possono essere applicate al volo.

In alcuni casi serve riavviare l’applicazione, per esempio quando cambiano parti strutturali non aggiornabili a runtime.

Regola pratica:

```text id="andfe2"
Hot Reload accelera il ciclo di sviluppo,
ma non sostituisce build pulita, test e riavvio quando il cambiamento è strutturale.
```

# Esempio completo: debug di un bug reale

Supponiamo di avere questo codice:

```csharp id="dpezer"
public class DocumentoService
{
    public decimal CalcolaTotale(Documento documento)
    {
        decimal totale = documento.Righe
            .Where(r => r.Attiva)
            .Sum(r => r.Quantita * r.PrezzoUnitario);

        return totale;
    }
}
```

Il totale risulta troppo basso.

Modelli:

```csharp id="ir25o4"
public class Documento
{
    public List<RigaDocumento> Righe { get; set; } = new();
}
```

```csharp id="xb5pzg"
public class RigaDocumento
{
    public string Descrizione { get; set; } = "";
    public int Quantita { get; set; }
    public decimal PrezzoUnitario { get; set; }
    public bool Attiva { get; set; }
}
```

Dati:

```csharp id="ve67a6"
Documento documento = new Documento
{
    Righe =
    [
        new RigaDocumento
        {
            Descrizione = "Libro C#",
            Quantita = 2,
            PrezzoUnitario = 30m,
            Attiva = true
        },
        new RigaDocumento
        {
            Descrizione = "Corso .NET",
            Quantita = 1,
            PrezzoUnitario = 120m,
            Attiva = false
        }
    ]
};
```

Risultato:

```text id="yfsxz8"
Totale calcolato: 60
Totale atteso: 180
```

---

## Strategia di debug

1. Mettere breakpoint su:

```csharp id="y7kk3e"
decimal totale = documento.Righe
```

2. Controllare in Watch:

```csharp id="n4dnzv"
documento.Righe.Count
```

3. Controllare:

```csharp id="d6aqfj"
documento.Righe.Where(r => r.Attiva).Count()
```

4. Espandere le righe nella finestra Locals o Watch.

Si scopre che:

```text id="ddaxg0"
la riga "Corso .NET" ha Attiva = false
```

Il bug non è nella somma, ma nel dato di input o nella regola di filtro.

---

## Versione debug-friendly

Per diagnosticare meglio:

```csharp id="b3vzsd"
public decimal CalcolaTotale(Documento documento)
{
    List<RigaDocumento> righeAttive = documento.Righe
        .Where(r => r.Attiva)
        .ToList();

    decimal totale = righeAttive
        .Sum(r => r.Quantita * r.PrezzoUnitario);

    return totale;
}
```

Ora possiamo ispezionare `righeAttive`.

# Errori comuni nel debugging

## Mettere breakpoint troppo tardi

Se ti fermi dove l’errore esplode, potresti essere già lontano dalla causa.

Esempio:

```text id="hrpjt9"
NullReferenceException in fase di salvataggio
```

La causa può essere:

```text id="0vv02g"
mapping errato
DTO incompleto
servizio chiamato con parametro null
validazione saltata
```

Meglio risalire il flusso.

---

## Guardare solo il messaggio dell’eccezione

Un’eccezione contiene:

```text id="g0ij6o"
tipo
messaggio
stack trace
inner exception
dati aggiuntivi
```

Per EF Core, SQL Server, file system e HTTP, l’`InnerException` è spesso decisiva.

---

## Modificare codice senza capire

Sbagliato:

```text id="6q4tiu"
Cambio righe finché il bug sparisce.
```

Meglio:

```text id="7p9do3"
Capisco perché succede.
Correggo la causa.
Aggiungo un test se possibile.
```

---

## Debuggare dati vecchi o ambiente sbagliato

Prima di indagare:

```text id="vp1alr"
sto usando il database giusto?
sono in Debug o Release?
sto leggendo l'appsettings corretto?
l'ambiente è Development o Production?
il file è quello aggiornato?
sto chiamando l'endpoint giusto?
```

Molti bug apparenti sono errori di ambiente.

---

## Non controllare il call stack

Se un metodo è chiamato da più punti, il call stack è essenziale.

Non basta sapere dove siamo.
Bisogna sapere come ci siamo arrivati.

---

## Ignorare codice generato o librerie

In alcune situazioni il problema sembra essere in una libreria, ma nasce dal nostro input.

Esempio:

```text id="4jyf3j"
errore in System.Text.Json
```

Possibile causa reale:

```text id="pp9isl"
DTO non compatibile
JSON malformato
proprietà required mancante
conversione non valida
```

---

```text id="nu8iup"
Usa breakpoint

  per monitorare variabili ed espressioni.

  per testare espressioni e condizioni.

Controlla il Call Stack
  per capire il percorso della chiamata.

Configura Exception Settings
  per fermarti quando l’eccezione nasce.

Nel codice async
  controlla Task, await e Parallel Stacks.

Nel LINQ
  spezza query complesse e ricorda la deferred execution.

Usa Hot Reload
  per iterazioni rapide, ma non abusarne.

Misura le performance
  prima di ottimizzare.
```

# Riepilogo

In questo capitolo abbiamo introdotto il debugging con Visual Studio.

Abbiamo visto:

```text id="k6zv7r"
Breakpoint
  fermano il programma in un punto preciso.

Conditional breakpoint
  fermano solo quando una condizione è vera.

Tracepoint
  scrivono informazioni senza interrompere.

Watch
  permette di osservare variabili ed espressioni.

Immediate Window
  permette di valutare espressioni durante il debug.

Call Stack
  mostra il percorso delle chiamate.

Exception Settings
  controlla quando il debugger si ferma sulle eccezioni.

Debug async
  richiede attenzione a Task, await e flusso logico.

Debug LINQ
  richiede attenzione a deferred execution e query complesse.

Hot Reload
  accelera alcune modifiche durante l’esecuzione.

Diagnostica performance
  aiuta a capire CPU, memoria e colli di bottiglia.
```

La regola principale è questa:

```text id="az92m8"
Un buon debug non è casuale:
parte da un’ipotesi, osserva dati reali e porta a una correzione verificabile.
```

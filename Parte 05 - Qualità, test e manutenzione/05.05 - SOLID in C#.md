# 05.05 — SOLID in C#

Nel capitolo precedente abbiamo introdotto il refactoring e la pulizia del codice. Abbiamo parlato di code smell, metodi troppo lunghi, classi troppo grandi, duplicazione, nomi poco chiari e separazione delle responsabilità.

Ora possiamo affrontare i principi **SOLID** con il contesto giusto.

SOLID non va studiato come un insieme di regole astratte. Ha senso solo dopo aver capito:

```text id="8140tz"
classi e oggetti
ereditarietà e interfacce
polimorfismo
Dependency Injection
unit test
refactoring
responsabilità del codice
```

Il punto centrale del capitolo è questo:

```text id="w96rxr"
SOLID non serve a scrivere più classi.
Serve a rendere il codice più comprensibile, testabile ed estendibile quando il dominio lo richiede.
```

---

# Che cos’è SOLID

SOLID è un acronimo che raccoglie cinque principi di progettazione object-oriented:

```text id="i07av7"
S — Single Responsibility Principle
O — Open/Closed Principle
L — Liskov Substitution Principle
I — Interface Segregation Principle
D — Dependency Inversion Principle
```

Sono linee guida per ridurre:

```text id="x8h0sb"
accoppiamento eccessivo
classi troppo grandi
dipendenze rigide
modifiche rischiose
codice difficile da testare
effetti collaterali inattesi
```

Non sono leggi assolute.

Non devono essere applicati meccanicamente.

---

# Perché SOLID va studiato qui

Se introduci SOLID troppo presto, il lettore rischia di vedere solo astrazioni:

```text id="wwjokd"
interfacce
classi separate
factory
strategy
dependency injection
```

senza capire quale problema risolvano.

Dopo OOP, DI, test e refactoring, invece, SOLID diventa concreto.

Il lettore ha già visto problemi reali:

```text id="5d4dfw"
classi difficili da testare
metodi troppo lunghi
dipendenze statiche
servizi che fanno troppe cose
if/switch che crescono continuamente
interfacce troppo grandi
```

SOLID dà un linguaggio per descrivere e correggere questi problemi.

---

# S — Single Responsibility Principle

Il **Single Responsibility Principle**, o SRP, dice che una classe dovrebbe avere una responsabilità principale.

Una formulazione molto usata è:

```text id="rccyxe"
Una classe dovrebbe avere un solo motivo significativo per cambiare.
```

Questo non significa che una classe debba avere un solo metodo.

Significa che non dovrebbe mescolare responsabilità diverse.

---

## Esempio sbagliato

```csharp id="c98xky"
public class DocumentoService
{
    public void RegistraDocumento(DocumentoDto dto)
    {
        // Validazione input
        if (dto.ClienteId <= 0)
        {
            throw new ArgumentException("Cliente non valido.");
        }

        // Calcolo totale
        decimal totale = 0m;

        foreach (RigaDocumentoDto riga in dto.Righe)
        {
            totale += riga.Quantita * riga.PrezzoUnitario;
        }

        // Salvataggio database
        Documento documento = new Documento
        {
            ClienteId = dto.ClienteId,
            Totale = totale
        };

        dbContext.Documenti.Add(documento);
        dbContext.SaveChanges();

        // Generazione PDF
        byte[] pdf = GeneraPdf(documento);

        // Invio email
        smtpClient.Send(
            "cliente@example.com",
            "Documento registrato",
            pdf);
    }

    private byte[] GeneraPdf(Documento documento)
    {
        return [];
    }
}
```

Questa classe ha troppi motivi per cambiare:

```text id="9e4hfh"
cambiano le regole di validazione
cambia il calcolo totale
cambia il database
cambia il formato PDF
cambia il sistema email
cambia il caso d’uso
```

---

## Versione migliore

Separiamo le responsabilità:

```csharp id="ku6fog"
public class DocumentoValidator
{
    public void Valida(DocumentoDto dto)
    {
        if (dto.ClienteId <= 0)
        {
            throw new ArgumentException("Cliente non valido.");
        }

        if (dto.Righe.Count == 0)
        {
            throw new ArgumentException("Il documento deve avere almeno una riga.");
        }
    }
}
```

```csharp id="4n5f64"
public class DocumentoTotaleCalculator
{
    public decimal Calcola(IEnumerable<RigaDocumentoDto> righe)
    {
        return righe.Sum(r => r.Quantita * r.PrezzoUnitario);
    }
}
```

```csharp id="bqdswm"
public interface IDocumentoRepository
{
    Task SalvaAsync(Documento documento);
}
```

```csharp id="yqigxl"
public interface IDocumentoPdfGenerator
{
    byte[] Genera(Documento documento);
}
```

```csharp id="ej0rtj"
public interface IEmailSender
{
    Task InviaDocumentoAsync(string destinatario, byte[] allegato);
}
```

Servizio applicativo:

```csharp id="356ifx"
public class DocumentoApplicationService
{
    private readonly DocumentoValidator validator;
    private readonly DocumentoTotaleCalculator totaleCalculator;
    private readonly IDocumentoRepository repository;
    private readonly IDocumentoPdfGenerator pdfGenerator;
    private readonly IEmailSender emailSender;

    public DocumentoApplicationService(
        DocumentoValidator validator,
        DocumentoTotaleCalculator totaleCalculator,
        IDocumentoRepository repository,
        IDocumentoPdfGenerator pdfGenerator,
        IEmailSender emailSender)
    {
        this.validator = validator;
        this.totaleCalculator = totaleCalculator;
        this.repository = repository;
        this.pdfGenerator = pdfGenerator;
        this.emailSender = emailSender;
    }

    public async Task RegistraDocumentoAsync(DocumentoDto dto)
    {
        validator.Valida(dto);

        decimal totale = totaleCalculator.Calcola(dto.Righe);

        Documento documento = Documento.Crea(dto.ClienteId, totale);

        await repository.SalvaAsync(documento);

        byte[] pdf = pdfGenerator.Genera(documento);

        await emailSender.InviaDocumentoAsync(dto.EmailCliente, pdf);
    }
}
```

Ora ogni classe ha una responsabilità più chiara.

---

## Attenzione: non spezzare troppo

SRP non significa creare una classe per ogni riga di codice.

Questo sarebbe eccessivo:

```text id="wgatul"
ClienteNomeValidator
ClienteEmailValidator
ClienteTelefonoValidator
ClienteCapValidator
ClienteCittaValidator
```

se il dominio è semplice.

Meglio una classe coerente:

```csharp id="ox4zvg"
public class ClienteValidator
{
    public void Valida(ClienteDto dto)
    {
        // regole di validazione del cliente
    }
}
```

Regola pratica:

```text id="s07z38"
SRP aiuta quando separa responsabilità reali.
Danneggia quando frammenta il codice senza ridurre la complessità.
```

# O — Open/Closed Principle

L’**Open/Closed Principle**, o OCP, dice che il codice dovrebbe essere:

```text id="4mqu8p"
aperto all’estensione
chiuso alla modifica
```

Significa che dovremmo poter aggiungere nuovi comportamenti senza modificare continuamente codice già stabile.

---

## Esempio sbagliato

```csharp id="tns0gn"
public class ScontoService
{
    public decimal CalcolaSconto(string tipoCliente, decimal importo)
    {
        if (tipoCliente == "Standard")
        {
            return importo * 0.05m;
        }

        if (tipoCliente == "Gold")
        {
            return importo * 0.10m;
        }

        if (tipoCliente == "Platinum")
        {
            return importo * 0.15m;
        }

        return 0m;
    }
}
```

Ogni volta che arriva un nuovo tipo cliente, dobbiamo modificare `ScontoService`.

Questo può andare bene all’inizio.
Ma se i casi aumentano spesso, il metodo diventa fragile.

---

Contratto:

Implementazioni:


```csharp id="k755kq"
public class ScontoGoldPolicy : IScontoPolicy
{
    public string TipoCliente => "Gold";

    public decimal Calcola(decimal importo)
    {
        return importo * 0.10m;
    }
}
```

```csharp id="96iptd"
public class ScontoPlatinumPolicy : IScontoPolicy
{
    public string TipoCliente => "Platinum";

    public decimal Calcola(decimal importo)
    {
        return importo * 0.15m;
    }
}
```

Servizio:

```csharp id="ex7ew8"
public class ScontoService
{
    private readonly Dictionary<string, IScontoPolicy> policies;

    public ScontoService(IEnumerable<IScontoPolicy> policies)
    {
        this.policies = policies.ToDictionary(p => p.TipoCliente);
    }

    public decimal CalcolaSconto(string tipoCliente, decimal importo)
    {
        if (!policies.TryGetValue(tipoCliente, out IScontoPolicy? policy))
        {
            return 0m;
        }

        return policy.Calcola(importo);
    }
}
```

Ora per aggiungere un nuovo tipo cliente creiamo una nuova classe:

```csharp id="u1y41l"
public class ScontoVipPolicy : IScontoPolicy
{
    public string TipoCliente => "Vip";

    public decimal Calcola(decimal importo)
    {
        return importo * 0.20m;
    }
}
```

Il servizio centrale non cambia.

---

## OCP non significa eliminare ogni switch

Questo `switch` è perfettamente accettabile:

```csharp id="n8gpf5"
public string DescriviStato(StatoOrdine stato)
{
    return stato switch
    {
        StatoOrdine.Bozza => "Bozza",
        StatoOrdine.Confermato => "Confermato",
        StatoOrdine.Annullato => "Annullato",
        _ => "Sconosciuto"
    };
}
```

Qui non c’è una vera famiglia di comportamenti complessi.

Applicare Strategy sarebbe probabilmente eccessivo.

Regola pratica:

```text id="xanma5"
Se i casi sono pochi, stabili e semplici, uno switch va bene.
Se i casi crescono spesso e contengono logiche diverse, valuta polimorfismo o strategy.
```

# L — Liskov Substitution Principle

Il **Liskov Substitution Principle**, o LSP, dice che un tipo derivato deve poter sostituire il tipo base senza rompere le aspettative del codice chiamante.

In parole semplici:

```text id="d0u2a1"
Se una funzione accetta una classe base o un’interfaccia,
deve poter lavorare correttamente con qualunque implementazione valida.
```

---

## Esempio sbagliato classico

```csharp id="p7tr0c"
public class Rettangolo
{
    public virtual int Larghezza { get; set; }
    public virtual int Altezza { get; set; }

    public int Area => Larghezza * Altezza;
}
```

```csharp id="7hnx5d"
public class Quadrato : Rettangolo
{
    public override int Larghezza
    {
        get => base.Larghezza;
        set
        {
            base.Larghezza = value;
            base.Altezza = value;
        }
    }

    public override int Altezza
    {
        get => base.Altezza;
        set
        {
            base.Altezza = value;
            base.Larghezza = value;
        }
    }
}
```

Sembra naturale dire che un quadrato è un rettangolo, ma il comportamento rompe le aspettative.

Codice chiamante:

```csharp id="d4qio7"
public int CalcolaAreaDopoResize(Rettangolo rettangolo)
{
    rettangolo.Larghezza = 10;
    rettangolo.Altezza = 5;

    return rettangolo.Area;
}
```

Con `Rettangolo`, l’area è 50.

Con `Quadrato`, l’area diventa 25 perché impostare altezza modifica anche larghezza.

La sostituibilità è rotta.

---

## Versione migliore

Meglio modellare diversamente:

```csharp id="8nhmmk"
public interface IFiguraGeometrica
{
    int Area { get; }
}
```

```csharp id="pswp4e"
public class Rettangolo : IFiguraGeometrica
{
    public int Larghezza { get; }
    public int Altezza { get; }

    public int Area => Larghezza * Altezza;

    public Rettangolo(int larghezza, int altezza)
    {
        Larghezza = larghezza;
        Altezza = altezza;
    }
}
```

```csharp id="ozd67y"
public class Quadrato : IFiguraGeometrica
{
    public int Lato { get; }

    public int Area => Lato * Lato;

    public Quadrato(int lato)
    {
        Lato = lato;
    }
}
```

Ora non fingiamo che `Quadrato` sia un `Rettangolo` modificabile.

Entrambi sono figure geometriche.

---

## Esempio più pratico: repository read-only

Interfaccia problematica:

```csharp id="s49fp1"
public interface IRepository<T>
{
    T? GetById(int id);
    void Add(T entity);
    void Delete(T entity);
}
```

Implementazione read-only:

```csharp id="jfz4vu"
public class ReadOnlyClienteRepository : IRepository<Cliente>
{
    public Cliente? GetById(int id)
    {
        // lettura
        return null;
    }

    public void Add(Cliente entity)
    {
        throw new NotSupportedException();
    }

    public void Delete(Cliente entity)
    {
        throw new NotSupportedException();
    }
}
```

Questa implementazione non rispetta il contratto.

Se il codice riceve `IRepository<Cliente>`, si aspetta che `Add` e `Delete` funzionino.

---

## Versione migliore

Separiamo le interfacce:

```csharp id="t7z0ly"
public interface IReadRepository<T>
{
    T? GetById(int id);
}
```

```csharp id="qg0pi3"
public interface IWriteRepository<T>
{
    void Add(T entity);
    void Delete(T entity);
}
```

Ora un repository read-only implementa solo ciò che può fare:

```csharp id="85az54"
public class ReadOnlyClienteRepository : IReadRepository<Cliente>
{
    public Cliente? GetById(int id)
    {
        return null;
    }
}
```

Questo migliora sia LSP sia ISP.

# I — Interface Segregation Principle

L’**Interface Segregation Principle**, o ISP, dice che è meglio avere interfacce piccole e specifiche invece di interfacce grandi e generiche.

Una classe non dovrebbe essere costretta a dipendere da metodi che non usa.

---

## Esempio sbagliato

```csharp id="wpnzry"
public interface IDocumentoService
{
    void CreaDocumento();
    void AnnullaDocumento();
    byte[] GeneraPdf();
    void InviaEmail();
    void EsportaCsv();
    void Archivia();
}
```

Una classe che deve solo generare PDF è costretta a conoscere un’interfaccia enorme.

---

## Versione migliore

Separiamo le capacità:

```csharp id="xvlyxd"
public interface IDocumentoCreator
{
    void CreaDocumento();
}
```

```csharp id="80va4l"
public interface IDocumentoPdfGenerator
{
    byte[] GeneraPdf();
}
```

```csharp id="nnh42y"
public interface IDocumentoEmailSender
{
    void InviaEmail();
}
```

```csharp id="whocmk"
public interface IDocumentoExporter
{
    void EsportaCsv();
}
```

Ora ogni classe dipende solo da ciò che usa.

---

## Interfacce troppo piccole?

Anche ISP può essere abusato.

Sospetto:

```text id="soiijz"
IClienteNomeReader
IClienteEmailReader
IClienteTelefonoReader
```

Se le interfacce sono troppo frammentate, il codice diventa rumoroso.

Regola pratica:

```text id="j7isac"
Un’interfaccia dovrebbe rappresentare una capacità coerente.
Non deve essere enorme, ma nemmeno artificiosamente minuscola.
```

# D — Dependency Inversion Principle

Il **Dependency Inversion Principle**, o DIP, dice che i moduli ad alto livello non dovrebbero dipendere direttamente da dettagli di basso livello.

Dovrebbero dipendere da astrazioni.

In pratica:

```text id="y5vu9t"
La logica applicativa non dovrebbe dipendere direttamente da SQL, SMTP, file system, API esterne o classi concrete infrastrutturali.
```

---

## Esempio sbagliato

```csharp id="3ru2q2"
public class OrdineService
{
    private readonly SmtpEmailSender emailSender = new SmtpEmailSender();
    private readonly SqlOrdineRepository repository = new SqlOrdineRepository();

    public void ConfermaOrdine(Ordine ordine)
    {
        repository.Salva(ordine);
        emailSender.InviaConferma(ordine);
    }
}
```

`OrdineService` dipende da dettagli concreti:

```text id="a07kyk"
SMTP
SQL
classi specifiche
creazione interna con new
```

È difficile da testare e difficile da sostituire.

---

## Versione corretta

Astrazioni:

```csharp id="ceii0p"
public interface IOrdineRepository
{
    Task SalvaAsync(Ordine ordine);
}
```

```csharp id="i1n1tu"
public interface IEmailSender
{
    Task InviaConfermaOrdineAsync(Ordine ordine);
}
```

Servizio:

```csharp id="4xqxqv"
public class OrdineService
{
    private readonly IOrdineRepository repository;
    private readonly IEmailSender emailSender;

    public OrdineService(
        IOrdineRepository repository,
        IEmailSender emailSender)
    {
        this.repository = repository;
        this.emailSender = emailSender;
    }

    public async Task ConfermaOrdineAsync(Ordine ordine)
    {
        await repository.SalvaAsync(ordine);
        await emailSender.InviaConfermaOrdineAsync(ordine);
    }
}
```

Implementazioni concrete:

```csharp id="j1r2hz"
public class SqlOrdineRepository : IOrdineRepository
{
    public Task SalvaAsync(Ordine ordine)
    {
        // Salvataggio su database
        return Task.CompletedTask;
    }
}
```

```csharp id="y27s9u"
public class SmtpEmailSender : IEmailSender
{
    public Task InviaConfermaOrdineAsync(Ordine ordine)
    {
        // Invio SMTP reale
        return Task.CompletedTask;
    }
}
```

Registrazione DI:

```csharp id="3vu1xw"
builder.Services.AddScoped<IOrdineRepository, SqlOrdineRepository>();
builder.Services.AddScoped<IEmailSender, SmtpEmailSender>();
builder.Services.AddScoped<OrdineService>();
```

# SOLID e Dependency Injection

La Dependency Injection non è uno dei principi SOLID, ma è uno strumento pratico per applicare soprattutto il Dependency Inversion Principle.

Senza DI:

```csharp id="iuqjee"
public class ClienteService
{
    private readonly SqlClienteRepository repository = new();
}
```

Con DI:

```csharp id="s8n6ey"
public class ClienteService
{
    private readonly IClienteRepository repository;

    public ClienteService(IClienteRepository repository)
    {
        this.repository = repository;
    }
}
```

La classe dichiara ciò di cui ha bisogno.
Il container decide quale implementazione fornire.

---

## DI non basta per essere SOLID

Questo codice usa DI, ma non è necessariamente ben progettato:

```csharp id="dt3bqn"
public class MegaService
{
    public MegaService(
        IClienteRepository clienteRepository,
        IOrdineRepository ordineRepository,
        IEmailSender emailSender,
        IPdfGenerator pdfGenerator,
        ICsvExporter csvExporter,
        ICacheService cacheService,
        ILogger<MegaService> logger,
        IOptions<AppOptions> options)
    {
    }
}
```

Troppe dipendenze nel costruttore sono spesso un segnale di responsabilità eccessive.

La DI rende visibile il problema, ma non lo risolve da sola.

# SOLID e testabilità

SOLID aiuta la testabilità perché riduce l’accoppiamento.

Esempio non testabile:

```csharp id="qf2j7u"
public class ClienteService
{
    public void Registra(string nome)
    {
        SqlClienteRepository repository = new SqlClienteRepository();
        repository.Salva(nome);

        SmtpEmailSender sender = new SmtpEmailSender();
        sender.Invia(nome);
    }
}
```

Questo test richiede database e SMTP reali, oppure molta fatica.

Versione testabile:

```csharp id="zr2ckq"
public class ClienteService
{
    private readonly IClienteRepository repository;
    private readonly IEmailSender emailSender;

    public ClienteService(
        IClienteRepository repository,
        IEmailSender emailSender)
    {
        this.repository = repository;
        this.emailSender = emailSender;
    }

    public async Task RegistraAsync(string nome)
    {
        await repository.SalvaAsync(nome);
        await emailSender.InviaBenvenutoAsync(nome);
    }
}
```

Nel test possiamo usare fake:

```csharp id="hxe9lt"
public class FakeClienteRepository : IClienteRepository
{
    public List<string> Salvati { get; } = new();

    public Task SalvaAsync(string nome)
    {
        Salvati.Add(nome);
        return Task.CompletedTask;
    }
}
```

```csharp id="keklrj"
public class FakeEmailSender : IEmailSender
{
    public List<string> Destinatari { get; } = new();

    public Task InviaBenvenutoAsync(string nome)
    {
        Destinatari.Add(nome);
        return Task.CompletedTask;
    }
}
```

Test:

```csharp id="urllka"
[Fact]
public async Task RegistraAsync_salvaClienteEInviaEmail()
{
    FakeClienteRepository repository = new FakeClienteRepository();
    FakeEmailSender emailSender = new FakeEmailSender();

    ClienteService service = new ClienteService(
        repository,
        emailSender);

    await service.RegistraAsync("Emilie");

    Assert.Contains("Emilie", repository.Salvati);
    Assert.Contains("Emilie", emailSender.Destinatari);
}
```

# Limiti dei principi SOLID

SOLID è utile, ma ha limiti.

Non risolve automaticamente:

```text id="82f8aw"
modellazione del dominio
performance
query inefficienti
UX
sicurezza
concorrenza
distribuzione
consistenza dei dati
complessità organizzativa
```

Inoltre, i principi possono entrare in tensione tra loro.

Esempio:

```text id="0r6zyv"
Per rispettare Open/Closed introduco molte classi Strategy.
Ma questo può peggiorare la semplicità se i casi non cambiano mai.
```

Oppure:

```text id="gzg2cf"
Per rispettare Interface Segregation creo troppe interfacce minuscole.
Ma il codice diventa più difficile da navigare.
```

SOLID non sostituisce il giudizio tecnico.

# Over-engineering

L’**over-engineering** è la tendenza a progettare una soluzione troppo complessa rispetto al problema reale.

Esempio semplice:

```text id="4wxwmm"
Devo calcolare uno sconto fisso del 10%.
```

Soluzione ragionevole:

```csharp id="x1s5pl"
public decimal ApplicaSconto(decimal importo)
{
    return importo * 0.90m;
}
```

Soluzione sovra-ingegnerizzata:

```text id="triv6h"
IScontoPolicy
IScontoPolicyFactory
IScontoPolicyResolver
AbstractScontoProvider
ScontoPipeline
ScontoContext
ScontoStrategyRegistry
```

Se oggi esiste un solo sconto e non c’è variabilità reale, questa complessità non è giustificata.

---

## Segnali di over-engineering

```text id="6u4evc"
molte interfacce con una sola implementazione
factory senza logica
classi astratte inutili
pattern usati per abitudine
codice difficile da seguire
molte cartelle e layer per un caso semplice
test pieni di mock per logica banale
```

---

## KISS e YAGNI

Due principi pratici aiutano a bilanciare SOLID.

```text id="sgcphh"
KISS
  Keep It Simple, Stupid.
  Mantieni la soluzione semplice.

YAGNI
  You Aren’t Gonna Need It.
  Non costruire oggi flessibilità che non serve.
```

Questo non significa scrivere codice disordinato.

Significa non anticipare complessità non richiesta.

# Esempio completo: applicare SOLID con equilibrio

Scenario:

```text id="d85gh2"
Dobbiamo registrare un cliente.
La registrazione deve:
- validare i dati
- salvare il cliente
- inviare una email di benvenuto
```

## Versione troppo accoppiata

```csharp id="oz22sy"
public class ClienteService
{
    public async Task RegistraAsync(string nome, string email)
    {
        if (string.IsNullOrWhiteSpace(nome))
        {
            throw new ArgumentException("Nome obbligatorio.");
        }

        if (string.IsNullOrWhiteSpace(email) || !email.Contains('@'))
        {
            throw new ArgumentException("Email non valida.");
        }

        SqlConnection connection = new SqlConnection("...");
        await connection.OpenAsync();

        SqlCommand command = connection.CreateCommand();
        command.CommandText = """
        INSERT INTO Clienti (Nome, Email)
        VALUES (@Nome, @Email)
        """;

        command.Parameters.AddWithValue("@Nome", nome);
        command.Parameters.AddWithValue("@Email", email);

        await command.ExecuteNonQueryAsync();

        SmtpClient smtp = new SmtpClient("smtp.example.com");
        smtp.Send("noreply@example.com", email, "Benvenuto", "Ciao!");
    }
}
```

Problemi:

```text id="q3y65n"
validazione, database e email sono mischiati
connection string hard-coded
SMTP hard-coded
difficile da testare
difficile da modificare
responsabilità multiple
```

## Versione SOLID ragionevole

DTO:

```csharp id="cdy92z"
public record RegistraClienteCommand(
    string Nome,
    string Email);
```

Validator:

```csharp id="w0nx0f"
public class RegistraClienteValidator
{
    public void Valida(RegistraClienteCommand command)
    {
        if (string.IsNullOrWhiteSpace(command.Nome))
        {
            throw new ArgumentException("Nome obbligatorio.", nameof(command));
        }

        if (string.IsNullOrWhiteSpace(command.Email) ||
            !command.Email.Contains('@'))
        {
            throw new ArgumentException("Email non valida.", nameof(command));
        }
    }
}
```

Entità:

```csharp id="apfqw2"
public class Cliente
{
    public int Id { get; private set; }
    public string Nome { get; private set; }
    public string Email { get; private set; }

    private Cliente(string nome, string email)
    {
        Nome = nome;
        Email = email;
    }

    public static Cliente Crea(string nome, string email)
    {
        return new Cliente(nome, email);
    }
}
```

Repository:

```csharp id="3dnkb7"
public interface IClienteRepository
{
    Task SalvaAsync(Cliente cliente);
}
```

Email:

```csharp id="mcdkxc"
public interface IEmailSender
{
    Task InviaBenvenutoAsync(string email, string nome);
}
```

Servizio applicativo:

```csharp id="vwii92"
public class ClienteRegistrationService
{
    private readonly RegistraClienteValidator validator;
    private readonly IClienteRepository repository;
    private readonly IEmailSender emailSender;

    public ClienteRegistrationService(
        RegistraClienteValidator validator,
        IClienteRepository repository,
        IEmailSender emailSender)
    {
        this.validator = validator;
        this.repository = repository;
        this.emailSender = emailSender;
    }

    public async Task RegistraAsync(RegistraClienteCommand command)
    {
        validator.Valida(command);

        Cliente cliente = Cliente.Crea(
            command.Nome,
            command.Email);

        await repository.SalvaAsync(cliente);

        await emailSender.InviaBenvenutoAsync(
            cliente.Email,
            cliente.Nome);
    }
}
```

Registrazione DI:

```csharp id="1jfxvi"
builder.Services.AddScoped<RegistraClienteValidator>();
builder.Services.AddScoped<IClienteRepository, EfClienteRepository>();
builder.Services.AddScoped<IEmailSender, SmtpEmailSender>();
builder.Services.AddScoped<ClienteRegistrationService>();
```

Questa versione applica SOLID in modo pratico:

```text id="icyuyy"
SRP
  validazione, persistenza, email e orchestrazione sono separate.

OCP
  posso cambiare email sender o repository senza modificare il servizio.

LSP
  ogni implementazione di IEmailSender deve poter essere usata al posto di un’altra.

ISP
  le interfacce sono piccole e mirate.

DIP
  il servizio dipende da astrazioni, non da SQL o SMTP concreti.
```

# Come decidere quanta astrazione serve

Domande utili:

```text id="w1vz5o"
Questa parte cambia spesso?
Ho più implementazioni reali?
Devo testare isolando una dipendenza esterna?
Questa classe ha troppe responsabilità?
Il codice attuale è difficile da modificare?
Sto introducendo un’interfaccia per un motivo concreto?
Il pattern riduce o aumenta la complessità?
```

Se la risposta è vaga, probabilmente l’astrazione può aspettare.

# Errori comuni con SOLID

## Creare interfacce per ogni classe

Non serve automaticamente:

```csharp id="hcmnnf"
public interface IClienteNomeFormatter
{
    string Format(string nome);
}
```

se esiste una sola implementazione, è pura logica interna e non serve sostituirla.

---

## Pensare che SOLID significhi usare tanti pattern

SOLID e design pattern sono collegati, ma non sono la stessa cosa.

Puoi applicare SOLID anche con codice semplice.

---

## Confondere SRP con “una classe, un metodo”

Una classe può avere più metodi e rispettare SRP se quei metodi appartengono alla stessa responsabilità.

---

## Applicare DIP anche ai modelli semplici

Non serve astrarre tutto.

Questo va bene:

```csharp id="o58uvr"
public decimal CalcolaTotale(decimal prezzo, int quantita)
{
    return prezzo * quantita;
}
```

Non serve:

```text id="3kjfo8"
IPrezzoCalculatorStrategyFactoryProvider
```

---

## Usare il repository pattern in modo meccanico

EF Core ha già `DbSet` e `DbContext`.

Un repository ha senso se aggiunge semantica di dominio o isola una dipendenza. Non ha senso se replica passivamente `Add`, `Update`, `Delete`, `GetAll`.

# Linee guida pratiche

```text id="s2h5do"
Usa SRP
  per separare responsabilità che cambiano per motivi diversi.

Usa OCP
  quando una famiglia di comportamenti cresce nel tempo.

Usa LSP
  per evitare implementazioni che rompono il contratto atteso.

Usa ISP
  per evitare interfacce troppo grandi e dipendenze inutili.

Usa DIP
  per isolare logica applicativa da infrastruttura concreta.

Usa Dependency Injection
  per fornire implementazioni dall’esterno.

Usa test
  per verificare che le astrazioni siano sostituibili.

Evita over-engineering
  se la complessità non è ancora reale.

Preferisci semplicità
  finché non hai un motivo concreto per astrarre.
```

# Riepilogo

In questo capitolo abbiamo studiato SOLID in C#.

Abbiamo visto:

```text id="on9gam"
Single Responsibility Principle
  una classe dovrebbe avere una responsabilità principale.

Open/Closed Principle
  il codice dovrebbe essere estendibile senza modifiche continue al nucleo stabile.

Liskov Substitution Principle
  le implementazioni devono rispettare le aspettative del contratto.

Interface Segregation Principle
  meglio interfacce piccole e coerenti che interfacce enormi.

Dependency Inversion Principle
  il codice ad alto livello dovrebbe dipendere da astrazioni, non da dettagli concreti.
```

Abbiamo collegato SOLID a:

```text id="oc3x60"
Dependency Injection
testabilità
refactoring
repository
servizi applicativi
over-engineering
```

La regola principale è questa:

```text id="vhk0bi"
SOLID è utile quando riduce il costo del cambiamento.
Diventa dannoso quando introduce astrazioni senza una necessità reale.
```

SOLID prepara naturalmente il terreno per architettura e design pattern: prima impariamo a separare responsabilità e dipendenze, poi possiamo riconoscere quando un pattern risolve davvero un problema ricorrente.

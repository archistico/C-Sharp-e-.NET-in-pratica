# 05.02 — Unit test con xUnit

Nel capitolo precedente abbiamo visto il debugging con Visual Studio. Il debugger è indispensabile per capire che cosa sta succedendo mentre il programma gira.

Ma il debugging da solo non basta.

Se ogni volta che modifichiamo il codice dobbiamo aprire l’applicazione, cliccare schermate, inserire dati e controllare manualmente il risultato, lo sviluppo diventa lento e fragile.

Gli **unit test** servono a verificare automaticamente che piccole parti del codice continuino a comportarsi come previsto.

Il punto centrale del capitolo è questo:

```text id="kyki58"
Un test automatico è una verifica ripetibile.
Ogni volta che cambi il codice, puoi rieseguire i test e sapere se hai rotto qualcosa.
```

---

# Perché testare

Testare significa proteggere il codice dal cambiamento.

In un progetto reale il codice cambia continuamente:

```text id="5cgfd7"
aggiungiamo funzionalità
correggiamo bug
rifattorizziamo metodi
aggiorniamo librerie
modifichiamo regole di business
ottimizziamo performance
separiamo classi troppo grandi
```

Senza test automatici, ogni modifica è più rischiosa.

Con test buoni, possiamo cambiare il codice con più sicurezza.

Microsoft descrive i test unitari come uno strumento utile sia per garantire la funzionalità sia per supportare le attività di refactoring; il code coverage misura quanto codice viene esercitato dai test. ([Microsoft Learn][1])

---

## Cosa non sono i test

Un test non dimostra che il programma è perfetto.

Un test dimostra che:

```text id="e93h0v"
in un certo scenario
con certi input
il codice produce il risultato atteso
```

I test non eliminano la necessità di:

```text id="tkcwwm"
debugging
review del codice
test manuali esplorativi
test di integrazione
test end-to-end
monitoraggio in produzione
```

Ma riducono drasticamente il rischio di rompere funzionalità già verificate.

---

# Test unitari

Un **unit test** verifica una piccola unità di codice in isolamento.

L’unità può essere:

```text id="7yyx0l"
un metodo
una classe
una regola di business
un servizio applicativo semplice
un validatore
un calcolatore
un parser
```

Esempio di classe da testare:

```csharp id="p80vuc"
public class ScontoService
{
    public decimal ApplicaSconto(decimal importo, decimal percentuale)
    {
        if (importo < 0)
        {
            throw new ArgumentOutOfRangeException(nameof(importo));
        }

        if (percentuale < 0 || percentuale > 100)
        {
            throw new ArgumentOutOfRangeException(nameof(percentuale));
        }

        decimal sconto = importo * percentuale / 100m;

        return importo - sconto;
    }
}
```

Un test unitario dovrebbe verificare una regola precisa:

```text id="5ra9nk"
Dato un importo di 100 e uno sconto del 20%,
il totale deve essere 80.
```

# xUnit

**xUnit.net** è uno dei framework di test più usati nel mondo .NET.

È gratuito, open source e supporta test per C#, F# e Visual Basic. La pagina ufficiale di xUnit lo descrive come un tool di unit testing community-focused per linguaggi .NET; xUnit v3 supporta .NET 8 o successivo e .NET Framework 4.7.2 o successivo. ([xUnit.net][2])

Nel libro possiamo usare xUnit perché:

```text id="hqm7vs"
è molto diffuso
è semplice da leggere
si integra con dotnet test
si integra con Visual Studio Test Explorer
supporta test parametrizzati
funziona bene con pipeline CI/CD
```

# Arrange, Act, Assert

Un test unitario leggibile segue spesso il pattern **AAA**:

```text id="yco60v"
Arrange
  preparo dati, oggetti e condizioni iniziali.

Act
  eseguo l’azione da testare.

Assert
  verifico il risultato.
```

Esempio:

```csharp id="xvnth3"
public class ScontoServiceTests
{
    [Fact]
    public void ApplicaSconto_conPercentualeValida_restituisceImportoScontato()
    {
        // Arrange
        ScontoService service = new ScontoService();

        // Act
        decimal risultato = service.ApplicaSconto(100m, 20m);

        // Assert
        Assert.Equal(80m, risultato);
    }
}
```

Questo test è leggibile perché separa chiaramente:

```text id="cg2yhj"
preparazione
azione
verifica
```

---

# `Fact`

In xUnit, `[Fact]` indica un test senza parametri.

Esempio:

```csharp id="7basde"
[Fact]
public void Somma_dueNumeri_restituisceTotale()
{
    Calcolatrice calcolatrice = new Calcolatrice();

    int risultato = calcolatrice.Somma(2, 3);

    Assert.Equal(5, risultato);
}
```

xUnit distingue tra **facts** e **theories**: i facts sono test sempre veri, mentre le theories sono test veri per un insieme specifico di dati. ([xUnit.net][5])

---

## Esempio completo con Fact

Classe da testare:

```csharp id="mh4ggl"
public class Calcolatrice
{
    public int Somma(int a, int b)
    {
        return a + b;
    }

    public int Dividi(int dividendo, int divisore)
    {
        if (divisore == 0)
        {
            throw new DivideByZeroException();
        }

        return dividendo / divisore;
    }
}
```

Test:

```csharp id="2rsy8y"
public class CalcolatriceTests
{
    [Fact]
    public void Somma_conDueNumeriInteri_restituisceSomma()
    {
        // Arrange
        Calcolatrice calcolatrice = new Calcolatrice();

        // Act
        int risultato = calcolatrice.Somma(2, 3);

        // Assert
        Assert.Equal(5, risultato);
    }
}
```

# `Theory`

`[Theory]` indica un test parametrizzato.

Serve quando vogliamo eseguire lo stesso test con dati diversi.

Esempio:

```csharp id="l6zct7"
[Theory]
[InlineData(100, 20, 80)]
[InlineData(100, 0, 100)]
[InlineData(100, 100, 0)]
public void ApplicaSconto_conDatiValidi_restituisceRisultatoAtteso(
    decimal importo,
    decimal percentuale,
    decimal atteso)
{
    ScontoService service = new ScontoService();

    decimal risultato = service.ApplicaSconto(importo, percentuale);

    Assert.Equal(atteso, risultato);
}
```

Qui il test viene eseguito tre volte.

La logica è la stessa, cambiano solo i dati.

---

# `InlineData`

`InlineData` fornisce direttamente i dati alla theory.

Esempio:

```csharp id="xe54tg"
[Theory]
[InlineData(2, 3, 5)]
[InlineData(10, 5, 15)]
[InlineData(-1, 1, 0)]
public void Somma_conVariInput_restituisceRisultatoAtteso(
    int a,
    int b,
    int atteso)
{
    Calcolatrice calcolatrice = new Calcolatrice();

    int risultato = calcolatrice.Somma(a, b);

    Assert.Equal(atteso, risultato);
}
```

Usa `InlineData` quando i dati sono semplici:

```text id="v51uub"
int
string
bool
decimal
enum semplici
combinazioni piccole e leggibili
```

## Fact o Theory?

```text id="6ubc9h"
Fact
  uno scenario specifico senza parametri.

Theory
  stesso comportamento verificato con più input.
```

Esempio:

```text id="2th8qp"
Fact
  ApplicaSconto_conImportoNegativo_lanciaEccezione

Theory
  ApplicaSconto_conPercentualiValide_restituisceTotaleCorretto
```

Regola pratica:

```text id="8yep73"
Se il corpo del test è uguale e cambiano solo i dati, usa Theory.
Se lo scenario ha una logica specifica, usa Fact.
```

# Assert

Gli `Assert` verificano il risultato.

Esempi comuni:

```csharp id="jj4x35"
Assert.Equal(atteso, risultato);
Assert.NotEqual(nonAtteso, risultato);
Assert.True(condizione);
Assert.False(condizione);
Assert.Null(valore);
Assert.NotNull(valore);
Assert.Contains(elemento, collezione);
Assert.DoesNotContain(elemento, collezione);
Assert.Empty(collezione);
Assert.NotEmpty(collezione);
Assert.Single(collezione);
Assert.Throws<ExceptionType>(() => codice);
```

Esempio:

```csharp id="8yoono"
Assert.Equal(80m, risultato);
```

Significa:

```text id="m5d5l0"
Mi aspetto 80.
Il risultato deve essere 80.
```

Spesso i bug nascono ai bordi:

```text id="a7ni0t"

ultimo elemento

```

## Esempio: validatore

Classe:

```csharp id="18vl6i"
public class ClienteValidator
{
    public bool ÈValido(string? nome, string? email)
    {
        if (string.IsNullOrWhiteSpace(nome))
        {
            return false;
        }

        if (string.IsNullOrWhiteSpace(email))
        {
            return false;
        }

        return email.Contains('@');
    }
}
```

Test:

```csharp id="6rcrs4"
public class ClienteValidatorTests
{
    [Theory]
    [InlineData(null, "test@example.com")]
    [InlineData("", "test@example.com")]
    [InlineData("   ", "test@example.com")]
    [InlineData("Emilie", null)]
    [InlineData("Emilie", "")]
    [InlineData("Emilie", "email-non-valida")]
    public void ÈValido_conDatiNonValidi_restituisceFalse(
        string? nome,
        string? email)
    {
        ClienteValidator validator = new ClienteValidator();

        bool risultato = validator.ÈValido(nome, email);

        Assert.False(risultato);
    }

    [Fact]
    public void ÈValido_conNomeEdEmailValidi_restituisceTrue()
    {
        ClienteValidator validator = new ClienteValidator();

        bool risultato = validator.ÈValido(
            "Emilie",
            "emilie@example.com");

        Assert.True(risultato);
    }
}
```

# Test delle eccezioni

Un buon test verifica anche quando il codice deve rifiutare input errati.

Esempio:

```csharp id="pv0mne"
[Fact]
public void ApplicaSconto_conImportoNegativo_lanciaArgumentOutOfRangeException()
{
    ScontoService service = new ScontoService();

    Assert.Throws<ArgumentOutOfRangeException>(() =>
        service.ApplicaSconto(-1m, 10m));
}
```

---

## Verificare il parametro dell’eccezione

```csharp id="f8t0p4"
[Fact]
public void ApplicaSconto_conPercentualeMaggioreDi100_lanciaEccezioneSulParametroPercentuale()
{
    ScontoService service = new ScontoService();

    ArgumentOutOfRangeException exception =
        Assert.Throws<ArgumentOutOfRangeException>(() =>
            service.ApplicaSconto(100m, 101m));

    Assert.Equal("percentuale", exception.ParamName);
}
```

Questo rende il test più preciso.

## Test eccezioni async

Per metodi async si usa `Assert.ThrowsAsync`.

Regola pratica:

```text id="i4p4bw"

  Assert.ThrowsAsync
```

# Test e refactoring

Il refactoring modifica la struttura del codice senza modificarne il comportamento osservabile.

Esempio:

```text id="otg4yk"
estrarre un metodo
rinominare una classe
dividere un servizio
rimuovere duplicazione
semplificare una condizione
introdurre un’interfaccia
```

I test sono fondamentali perché verificano che il comportamento resti lo stesso.

Esempio:

```csharp id="vt99du"
public decimal CalcolaTotale(IEnumerable<RigaDocumento> righe)
{
    return righe
        .Where(r => r.Attiva)
        .Sum(r => r.Quantita * r.PrezzoUnitario);
}
```

Dopo refactoring:

```csharp id="ekf2yr"
public decimal CalcolaTotale(IEnumerable<RigaDocumento> righe)
{
    decimal totale = 0m;

    foreach (RigaDocumento riga in righe)
    {
        if (!riga.Attiva)
        {
            continue;
        }

        totale += CalcolaSubtotale(riga);
    }

    return totale;
}

private static decimal CalcolaSubtotale(RigaDocumento riga)
{
    return riga.Quantita * riga.PrezzoUnitario;
}
```

Se i test passano, abbiamo più fiducia che il comportamento non sia cambiato.

# Code coverage

Il **code coverage** misura quanto codice viene eseguito dai test.

Può misurare:

```text id="hpve7v"
righe
rami condizionali
metodi
classi
```

Microsoft definisce il code coverage come una misura della quantità di codice eseguita dagli unit test, per esempio righe, rami o metodi; se un’app ha due rami condizionali e i test ne coprono solo uno, la branch coverage sarà del 50%. ([Microsoft Learn][1])

---

## Eseguire coverage con Coverlet

Una modalità comune è usare Coverlet.

Microsoft documenta l’uso di Coverlet e ReportGenerator per generare report di code coverage in progetti .NET, incluso xUnit. ([Microsoft Learn][6])

Comandi tipici:

```bash id="klot4n"
dotnet add MiaApp.Tests package coverlet.collector
```

Poi:

```bash id="gh60nd"
dotnet test --collect:"XPlat Code Coverage"
```

Questo genera un file di coverage in formato Cobertura.

Per un report leggibile si può usare ReportGenerator:

```bash id="bic6hk"
dotnet tool install -g dotnet-reportgenerator-globaltool
```

```bash id="cn60zy"
reportgenerator ^
  -reports:**/coverage.cobertura.xml ^
  -targetdir:coveragereport ^
  -reporttypes:Html
```

Su PowerShell moderno, si può adattare il comando su una sola riga.

---

## Coverage alto non significa qualità alta

Il coverage è utile, ma non basta.

Un test può eseguire codice senza verificare davvero il comportamento.

Esempio debole:

```csharp id="utwd2f"
[Fact]
public void TestDebole()
{
    ScontoService service = new ScontoService();

    service.ApplicaSconto(100m, 20m);
}
```

Questo test aumenta la coverage, ma non controlla il risultato.

Meglio:

```csharp id="4n951q"
[Fact]
public void ApplicaSconto_con20Percento_restituisce80()
{
    ScontoService service = new ScontoService();

    decimal risultato = service.ApplicaSconto(100m, 20m);

    Assert.Equal(80m, risultato);
}
```

Regola pratica:

```text id="83s6w5"
Coverage bassa
  segnala aree non testate.

Coverage alta
  non garantisce che i test siano buoni.
```

# Esempio completo: testare un servizio di documento

Classe da testare:

```csharp id="u4nmjm"
public class RigaDocumento
{
    public string Descrizione { get; }
    public int Quantita { get; }
    public decimal PrezzoUnitario { get; }
    public decimal ScontoPercentuale { get; }
    public bool Attiva { get; }

    public RigaDocumento(
        string descrizione,
        int quantita,
        decimal prezzoUnitario,
        decimal scontoPercentuale,
        bool attiva = true)
    {
        ArgumentException.ThrowIfNullOrWhiteSpace(descrizione);

        if (quantita <= 0)
        {
            throw new ArgumentOutOfRangeException(nameof(quantita));
        }

        if (prezzoUnitario < 0)
        {
            throw new ArgumentOutOfRangeException(nameof(prezzoUnitario));
        }

        if (scontoPercentuale < 0 || scontoPercentuale > 100)
        {
            throw new ArgumentOutOfRangeException(nameof(scontoPercentuale));
        }

        Descrizione = descrizione;
        Quantita = quantita;
        PrezzoUnitario = prezzoUnitario;
        ScontoPercentuale = scontoPercentuale;
        Attiva = attiva;
    }

    public decimal CalcolaSubtotale()
    {
        decimal lordo = Quantita * PrezzoUnitario;
        decimal sconto = lordo * ScontoPercentuale / 100m;

        return lordo - sconto;
    }
}
```

```csharp id="0a3w4n"
public class DocumentoService
{
    public decimal CalcolaTotale(IEnumerable<RigaDocumento> righe)
    {
        ArgumentNullException.ThrowIfNull(righe);

        return righe
            .Where(r => r.Attiva)
            .Sum(r => r.CalcolaSubtotale());
    }
}
```

# Testare senza dipendenze esterne

Un unit test dovrebbe essere veloce e isolato.

Evita, nei test unitari puri:

```text id="be86s1"
database reale
file system reale
rete
servizi HTTP esterni
orologio di sistema non controllato
dati dipendenti dall’ambiente
```

Se una classe dipende da un database, probabilmente non stai scrivendo un unit test puro, ma un test di integrazione.

Questo non è sbagliato, ma è un tipo di test diverso.

---

# Esempio: rendere testabile il tempo

Codice difficile da testare:

```csharp id="7ns9kr"
public class ScadenzaService
{
    public bool ÈScaduto(DateTime dataScadenza)
    {
        return dataScadenza < DateTime.Now;
    }
}
```

Il risultato dipende dall’orologio reale.

Meglio introdurre un’astrazione:

```csharp id="3s8npd"
public interface IClock
{
    DateTime Now { get; }
}
```

Implementazione reale:

```csharp id="83os46"
public class SystemClock : IClock
{
    public DateTime Now => DateTime.Now;
}
```

Servizio:

```csharp id="wgr32y"
public class ScadenzaService
{
    private readonly IClock clock;

    public ScadenzaService(IClock clock)
    {
        this.clock = clock;
    }

    public bool ÈScaduto(DateTime dataScadenza)
    {
        return dataScadenza < clock.Now;
    }
}
```

Fake per test:

```csharp id="fc151c"
public class FakeClock : IClock
{
    public DateTime Now { get; set; }
}
```

Test:

```csharp id="f924gh"
[Fact]
public void ÈScaduto_conDataPrimaDiOggi_restituisceTrue()
{
    FakeClock clock = new FakeClock
    {
        Now = new DateTime(2026, 4, 25)
    };

    ScadenzaService service = new ScadenzaService(clock);

    bool risultato = service.ÈScaduto(new DateTime(2026, 4, 24));

    Assert.True(risultato);
}
```

Questo esempio mostra il legame tra testabilità e Dependency Injection.

# Test e refactoring: esempio pratico

Partiamo da questo metodo:

```csharp id="vxecj5"
public decimal CalcolaTotale(IEnumerable<RigaDocumento> righe)
{
    return righe
        .Where(r => r.Attiva)
        .Sum(r => r.CalcolaSubtotale());
}
```

I test verificano:

```text id="5d6k9d"
somma righe attive
ignora righe non attive
lista vuota restituisce zero
input null lancia eccezione
```

Ora possiamo rifattorizzare:

```csharp id="668nia"
public decimal CalcolaTotale(IEnumerable<RigaDocumento> righe)
{
    ArgumentNullException.ThrowIfNull(righe);

    decimal totale = 0m;

    foreach (RigaDocumento riga in righe)
    {
        if (!riga.Attiva)
        {
            continue;
        }

        totale += riga.CalcolaSubtotale();
    }

    return totale;
}
```

Se i test passano, abbiamo una conferma automatica che il comportamento essenziale è rimasto uguale.

# Errori comuni nei test

## Test senza assert

Sbagliato:

```csharp id="4hvqbv"
[Fact]
public void Test()
{
    ScontoService service = new ScontoService();

    service.ApplicaSconto(100m, 20m);
}
```

Questo test verifica solo che non venga lanciata un’eccezione.

Meglio:

```csharp id="ql5tb6"
Assert.Equal(80m, risultato);
```

---

## Test troppo grandi

Sospetto:

```text id="ncuv1m"
un test crea 20 oggetti
chiama 5 servizi
verifica 10 cose diverse
fallisce per motivi difficili da capire
```

Meglio test più piccoli e mirati.

---

## Test fragili

Un test è fragile quando fallisce per dettagli non importanti.

Esempio:

```csharp id="gixwxx"
Assert.Equal("Cliente salvato correttamente alle 10:32:11", messaggio);
```

Se l’orario cambia, il test fallisce.

Meglio verificare il comportamento essenziale.

---

## Test dipendenti dall’ordine

Sospetto:

```text id="mg6ykk"
Il test B passa solo se prima è stato eseguito il test A.
```

Ogni test dovrebbe essere indipendente.

---

## Test che usano dati condivisi modificabili

Sconsigliato:

```csharp id="cx2nrv"
private static List<Cliente> clienti = new();
```

Se un test modifica la lista, può influenzare gli altri.

Meglio creare i dati dentro ogni test o usare factory pulite.

---

## Testare l’implementazione invece del comportamento

Sospetto:

```text id="qizsup"
verifico che sia stato chiamato esattamente un metodo privato
verifico dettagli interni non osservabili
rompo i test a ogni refactoring
```

Meglio testare il comportamento pubblico.

---

# Linee guida pratiche

```text id="scuzcj"
Scrivi test piccoli
  uno scenario, una ragione principale di fallimento.

Usa Arrange, Act, Assert
  per rendere il test leggibile.

Usa Fact
  per scenari singoli.

Usa Theory e InlineData
  per lo stesso comportamento con dati diversi.

Testa casi limite
  zero, null, vuoto, massimo, minimo, confini.

Testa eccezioni
  quando il codice deve rifiutare input non validi.

Dai nomi descrittivi
  il nome del test deve spiegare scenario e risultato.

Evita dipendenze esterne
  nei test unitari puri.

Usa fake semplici
  quando vuoi isolare una dipendenza.

Non inseguire solo la coverage
  un test deve verificare comportamento, non solo eseguire righe.

Usa i test per refactoring
  prima proteggi il comportamento, poi migliora il codice.
```

# Riepilogo

In questo capitolo abbiamo introdotto gli unit test con xUnit.

Abbiamo visto:


Abbiamo chiarito che xUnit distingue tra facts e theories: i facts verificano condizioni invarianti, mentre le theories verificano lo stesso comportamento con dati diversi. ([xUnit.net][5])

Abbiamo visto che `dotnet test` permette di eseguire i test anche da riga di comando e in pipeline automatizzate. ([Microsoft Learn][4])

```text id="yrmasq"
Un buon unit test non verifica che il codice sia scritto in un certo modo.
Verifica che un comportamento importante resti corretto nel tempo.
```

# 05.03 — Test di integrazione

Nel capitolo precedente abbiamo introdotto gli unit test con xUnit. Abbiamo visto come testare una classe isolata, senza database, file system, rete o applicazione avviata.

Ora facciamo un passo ulteriore.

I **test di integrazione** verificano che più parti del sistema funzionino correttamente insieme.

Esempi:

```text id="y5jg6b"
servizio applicativo + DbContext + database
API ASP.NET Core + routing + model binding + validazione
controller/minimal API + dependency injection + middleware
repository + EF Core + provider reale
autenticazione + autorizzazione + endpoint
serializzazione JSON + risposta HTTP
```

Microsoft definisce i test di integrazione ASP.NET Core come test che verificano i componenti dell’app a un livello più ampio rispetto agli unit test, spesso includendo infrastruttura come database, file system e rete; ASP.NET Core supporta questi test tramite un test web host e un server di test in memoria. ([GitHub][1])

Il punto centrale del capitolo è questo:

```text id="0dcy7g"
Gli unit test verificano una piccola unità isolata.
I test di integrazione verificano che più componenti collaborino correttamente.
```

---

# Differenza tra unit test e integration test

Un **unit test** dovrebbe essere:

```text id="hkxl12"
veloce
isolato
deterministico
senza database reale
senza rete
senza file system reale
centrato su una regola precisa
```

Esempio:

```csharp id="1wqeoe"
[Fact]
public void CalcolaTotale_conRigheValide_restituisceSomma()
{
    DocumentoService service = new DocumentoService();

    decimal totale = service.CalcolaTotale([
        new RigaDocumento("Libro", 2, 30m),
        new RigaDocumento("Corso", 1, 100m)
    ]);

    Assert.Equal(160m, totale);
}
```

Un **integration test** invece verifica che più parti reali funzionino insieme.

Esempio:

```text id="x6w8hu"
Creo un cliente tramite API.
L’API valida il DTO.
Il servizio applicativo crea l’entità.
EF Core salva nel database.
La risposta HTTP restituisce 201 Created.
```

---

## Confronto pratico

```text id="o3pdky"
Unit test
  Testa DocumentoService.CalcolaTotale in isolamento.

Integration test
  Testa POST /api/documenti verificando routing, JSON, validazione,
  servizio applicativo, DbContext e salvataggio nel database.
```

---

## Quando usare l’uno o l’altro

```text id="44p1m7"
Unit test
  regole di business pure
  validatori
  calcoli
  parser
  classi senza infrastruttura

Integration test
  database
  API
  dependency injection
  configurazione
  middleware
  serializzazione
  autenticazione
  query EF Core reali
```

Regola pratica:

```text id="6x4c3s"
Non sostituire gli unit test con integration test.
Usali insieme, per livelli diversi di fiducia.
```

---

# Perché servono i test di integrazione

Gli unit test possono passare anche se l’applicazione reale non funziona.

Esempi di problemi che un unit test potrebbe non vedere:

```text id="8ssbr2"
connection string errata
mapping EF Core sbagliato
migration mancante
relazione configurata male
query non traducibile dal provider database
serializzazione JSON inattesa
endpoint non registrato
middleware in ordine sbagliato
dependency injection incompleta
vincolo unique violato
differenza tra SQLite e SQL Server
```

I test di integrazione servono a intercettare proprio questi problemi.

# Test con database

Testare codice che usa il database è uno degli scenari più importanti.

Possiamo usare approcci diversi:

```text id="ld92m2"
EF Core InMemory provider
SQLite in-memory
database reale locale
database in Docker con Testcontainers
database dedicato di test
```

Ogni scelta ha pro e contro.

# EF Core InMemory provider: attenzione

EF Core offre un provider InMemory, ma non è un vero database relazionale.

È comodo per alcuni test, ma può comportarsi in modo diverso da SQL Server, PostgreSQL o SQLite.

Microsoft indica che quando si testano applicazioni EF Core bisogna decidere se coinvolgere il database di produzione o usare un test double; tra i test double cita SQLite in-memory e InMemory provider, ma segnala anche differenze importanti tra i comportamenti. ([Microsoft Learn][2])

Esempi di differenze possibili:

```text id="0o8mhy"
vincoli relazionali non equivalenti
traduzione LINQ diversa
transazioni diverse
case sensitivity diversa
funzioni SQL non testate
indici non rilevanti
```

Regola pratica:

```text id="tv3o8n"
EF Core InMemory può andare bene per test molto semplici.
Per test di persistenza realistici, preferisci SQLite in-memory o un database reale.
```

---

# SQLite in-memory

SQLite in-memory è una scelta molto usata per test rapidi con EF Core.

Vantaggi:

```text id="bsmxks"
è veloce
non crea file
è relazionale
supporta SQL reale
isola facilmente i test
non richiede Docker
```

Microsoft segnala che SQLite può essere configurato come provider EF Core per test al posto del database di produzione e che la modalità in-memory è utile perché consente isolamento senza gestire file SQLite fisici. ([Microsoft Learn][3])

Attenzione però: SQLite non è SQL Server.

La documentazione EF Core italiana segnala che SQLite in-memory offre migliore compatibilità relazionale rispetto al provider InMemory, ma restano discrepanze importanti rispetto al database di produzione, per esempio su funzionalità specifiche del provider. ([Microsoft Learn][4])

## Esempio con SQLite in-memory

Pacchetti nel progetto di test:

```bash id="ngwqk1"
dotnet add package Microsoft.EntityFrameworkCore.Sqlite
dotnet add package Microsoft.NET.Test.Sdk
dotnet add package xunit
dotnet add package xunit.runner.visualstudio
```

Entità:

```csharp id="227t3c"
public class Cliente
{
    public int Id { get; set; }
    public string Nome { get; set; } = "";
    public string? Email { get; set; }
    public bool Attivo { get; set; } = true;
}
```

DbContext:

```csharp id="156kl4"
using Microsoft.EntityFrameworkCore;

public class AppDbContext : DbContext
{
    public DbSet<Cliente> Clienti => Set<Cliente>();

    public AppDbContext(DbContextOptions<AppDbContext> options)
        : base(options)
    {
    }

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        modelBuilder.Entity<Cliente>(entity =>
        {
            entity.ToTable("Clienti");

            entity.HasKey(c => c.Id);

            entity.Property(c => c.Nome)
                .IsRequired()
                .HasMaxLength(100);

            entity.HasIndex(c => c.Email)
                .IsUnique();
        });
    }
}
```

Test:

```csharp id="xj6khw"
using Microsoft.Data.Sqlite;
using Microsoft.EntityFrameworkCore;

public class ClienteDbTests : IDisposable
{
    private readonly SqliteConnection connection;
    private readonly DbContextOptions<AppDbContext> options;

    public ClienteDbTests()
    {
        connection = new SqliteConnection("DataSource=:memory:");
        connection.Open();

        options = new DbContextOptionsBuilder<AppDbContext>()
            .UseSqlite(connection)
            .Options;

        using AppDbContext dbContext = new AppDbContext(options);

        dbContext.Database.EnsureCreated();
    }

    [Fact]
    public async Task InserimentoCliente_salvaClienteNelDatabase()
    {
        await using AppDbContext dbContext = new AppDbContext(options);

        dbContext.Clienti.Add(new Cliente
        {
            Nome = "Emilie",
            Email = "emilie@example.com"
        });

        await dbContext.SaveChangesAsync();

        Cliente? cliente = await dbContext.Clienti
            .SingleOrDefaultAsync(c => c.Email == "emilie@example.com");

        Assert.NotNull(cliente);
        Assert.Equal("Emilie", cliente.Nome);
    }

    public void Dispose()
    {
        connection.Dispose();
    }
}
```

## Perché la connessione resta aperta

Con SQLite in-memory il database esiste finché la connessione resta aperta.

Se apriamo e chiudiamo la connessione, il database viene perso.

Per questo nel test:

```csharp id="l3qpn9"
connection.Open();
```

viene fatto nel costruttore, e la connessione viene chiusa solo in `Dispose`.

# Testcontainers

SQLite in-memory è pratico, ma non replica perfettamente SQL Server, PostgreSQL o altri database reali.

Per test più realistici possiamo usare **Testcontainers**.

Testcontainers permette di avviare dipendenze reali in container Docker durante i test.

Esempi:

```text id="ym7ss4"
SQL Server
PostgreSQL
MySQL
Redis
RabbitMQ
MongoDB
Azurite
Cosmos DB emulator, dove supportato
```

La guida ufficiale di Testcontainers per .NET mostra come usare container reali, per esempio PostgreSQL, nei test; evidenzia anche che i membri del team possono clonare il progetto ed eseguire i test senza installare localmente il database, purché abbiano Docker. ([Testcontainers][5])

---

## Quando usare Testcontainers

Usa Testcontainers quando vuoi testare:

```text id="bahspr"
provider reale del database
SQL specifico del database
stored procedure
collation e case sensitivity reali
vincoli e indici reali
migrazioni su database reale
messaggistica reale
Redis reale
dipendenze esterne containerizzabili
```

## Limiti di Testcontainers

```text id="qf72jx"
richiede Docker
è più lento di SQLite in-memory

richiede gestione immagini e porte

```

Regola pratica:

```text id="6x3cba"

  veloce e semplice.

  più realistico, ma più pesante.

```

# Test di API ASP.NET Core

I test di integrazione sulle API verificano l’applicazione quasi come farebbe un client HTTP.

Esempio di scenario:

```text id="i4z551"
Invio POST /api/clienti con JSON valido.
Mi aspetto HTTP 201.
Leggo il JSON di risposta.

```

ASP.NET Core supporta i test di integrazione con `WebApplicationFactory<TEntryPoint>`, che crea un `TestServer` per l’applicazione sotto test; `TEntryPoint` è di solito `Program.cs`. ([Microsoft Learn][7])

# `WebApplicationFactory`

`WebApplicationFactory<TEntryPoint>` permette di avviare l’app ASP.NET Core in memoria per testarla con un `HttpClient`.

Pacchetto:

```bash id="9ga4jz"
dotnet add package Microsoft.AspNetCore.Mvc.Testing
```

Il pacchetto `Microsoft.AspNetCore.Mvc.Testing` fornisce supporto per scrivere integration test per app ASP.NET Core che usano MVC o Minimal API. ([nuget.org][8])

## Rendere accessibile `Program`

Con Minimal API moderne, spesso `Program.cs` usa top-level statements.

Per permettere a `WebApplicationFactory<Program>` di trovare il punto di ingresso, possiamo aggiungere in fondo a `Program.cs`:

```csharp id="2rbfw8"
public partial class Program
{
}
```

La documentazione ASP.NET Core mostra questo approccio con `public partial class Program { }` negli esempi di test di integrazione. ([Microsoft Learn][9])

# Esempio Minimal API da testare

`Program.cs`:

```csharp id="q5s371"
using Microsoft.EntityFrameworkCore;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddDbContext<AppDbContext>(options =>
{
    options.UseSqlite(
        builder.Configuration.GetConnectionString("DefaultConnection"));
});

var app = builder.Build();

app.MapGet("/api/clienti", async (AppDbContext dbContext) =>
{
    List<ClienteDto> clienti = await dbContext.Clienti
        .AsNoTracking()
        .OrderBy(c => c.Nome)
        .Select(c => new ClienteDto(
            c.Id,
            c.Nome,
            c.Email))
        .ToListAsync();

    return Results.Ok(clienti);
});

app.MapPost("/api/clienti", async (
    CreaClienteRequest request,
    AppDbContext dbContext) =>
{
    if (string.IsNullOrWhiteSpace(request.Nome))
    {
        return Results.BadRequest("Nome obbligatorio.");
    }

    Cliente cliente = new Cliente
    {
        Nome = request.Nome,
        Email = request.Email
    };

    dbContext.Clienti.Add(cliente);

    await dbContext.SaveChangesAsync();

    ClienteDto dto = new ClienteDto(
        cliente.Id,
        cliente.Nome,
        cliente.Email);

    return Results.Created($"/api/clienti/{cliente.Id}", dto);
});

app.Run();

public partial class Program
{
}

public record CreaClienteRequest(
    string Nome,
    string? Email);

public record ClienteDto(
    int Id,
    string Nome,
    string? Email);
```

# Personalizzare WebApplicationFactory

Nei test spesso vogliamo sostituire servizi reali.

Esempi:

```text id="p2gi0h"
database reale → SQLite in-memory
servizio email reale → fake email sender
servizio pagamento reale → fake payment gateway
autenticazione reale → autenticazione di test
configurazione produzione → configurazione test
```

Esempio con SQLite in-memory:

```csharp id="8xnb0f"
using Microsoft.AspNetCore.Hosting;
using Microsoft.AspNetCore.Mvc.Testing;
using Microsoft.Data.Sqlite;
using Microsoft.EntityFrameworkCore;
using Microsoft.Extensions.DependencyInjection;
using Microsoft.Extensions.DependencyInjection.Extensions;

public class CustomWebApplicationFactory
    : WebApplicationFactory<Program>
{
    private SqliteConnection? connection;

    protected override void ConfigureWebHost(IWebHostBuilder builder)
    {
        builder.UseEnvironment("Testing");

        builder.ConfigureServices(services =>
        {
            services.RemoveAll<DbContextOptions<AppDbContext>>();

            connection = new SqliteConnection("DataSource=:memory:");
            connection.Open();

            services.AddDbContext<AppDbContext>(options =>
            {
                options.UseSqlite(connection);
            });

            ServiceProvider serviceProvider = services.BuildServiceProvider();

            using IServiceScope scope = serviceProvider.CreateScope();

            AppDbContext dbContext =
                scope.ServiceProvider.GetRequiredService<AppDbContext>();

            dbContext.Database.EnsureCreated();

            SeedData(dbContext);
        });
    }

    private static void SeedData(AppDbContext dbContext)
    {
        dbContext.Clienti.AddRange(
            new Cliente
            {
                Nome = "Emilie",
                Email = "emilie@example.com"
            },
            new Cliente
            {
                Nome = "Mario",
                Email = "mario@example.com"
            });

        dbContext.SaveChanges();
    }

    protected override void Dispose(bool disposing)
    {
        base.Dispose(disposing);

        if (disposing)
        {
            connection?.Dispose();
        }
    }
}
```

Nota: `BuildServiceProvider` dentro `ConfigureServices` va usato con attenzione. In questo caso è confinato al setup del test per creare e inizializzare il database. Nel codice applicativo ordinario non è una pratica da usare come scorciatoia.

# Isolamento dei test

Un test di integrazione deve essere ripetibile.

Questo significa:

```text id="idn8qc"
non deve dipendere dall’ordine di esecuzione
non deve dipendere da dati lasciati da altri test
non deve richiedere pulizia manuale
non deve modificare ambienti condivisi pericolosi
```

---

## Strategie di isolamento

```text id="uixcf7"
database nuovo per ogni test
transazione rollback a fine test
schema nuovo per ogni classe di test
container nuovo per suite o classe
pulizia tabelle tra test
nomi unici per dati creati
```

---

## Database nuovo per test

Con SQLite in-memory è semplice: ogni test può creare la propria connessione.

```csharp id="sm3zrr"
public ClienteDbTests()
{
    connection = new SqliteConnection("DataSource=:memory:");
    connection.Open();

    options = new DbContextOptionsBuilder<AppDbContext>()
        .UseSqlite(connection)
        .Options;

    using AppDbContext dbContext = new AppDbContext(options);
    dbContext.Database.EnsureCreated();
}
```

Vantaggio:

```text id="0m9tun"
isolamento massimo
```

Svantaggio:

```text id="ii8rec"
setup ripetuto
```

---

## Pulizia tra test

Se usi un database condiviso per una classe di test, devi pulire.

Esempio semplice:

```csharp id="kfe4hh"
private static async Task PulisciDatabaseAsync(AppDbContext dbContext)
{
    dbContext.Clienti.RemoveRange(dbContext.Clienti);
    await dbContext.SaveChangesAsync();
}
```

Per database più complessi, la pulizia manuale può diventare fragile.

In progetti reali si possono usare strategie dedicate, come:

```text id="r0kzz5"
ricreare lo schema
rollback transazionale
strumenti di reset database
container effimeri
```

## IAsyncLifetime

`IAsyncLifetime` è utile quando setup e teardown sono asincroni.

```csharp id="alfeq0"
public class DatabaseFixture : IAsyncLifetime
{
    public Task InitializeAsync()
    {
        // Avvia database, crea schema, seed dati
        return Task.CompletedTask;
    }

    public Task DisposeAsync()
    {
        // Pulisce risorse
        return Task.CompletedTask;
    }
}
```

Con Testcontainers è molto comune usare `IAsyncLifetime` perché l’avvio e lo stop dei container sono asincroni.

# Test di integrazione con transazione

Una strategia diffusa è aprire una transazione per ogni test e fare rollback alla fine.

Esempio concettuale:

```csharp id="m1kb2x"
await using var transaction =
    await dbContext.Database.BeginTransactionAsync();

try
{
    // esegui test
}
finally
{
    await transaction.RollbackAsync();
}
```

Questa strategia funziona bene in alcuni scenari, ma non sempre è semplice con:

```text id="clz45a"
più DbContext
API testate via HttpClient
background service
code messaggi
transazioni distribuite
provider con limiti specifici
```

Per le API, spesso è più semplice avere un database isolato per test o una pulizia affidabile tra test.

# Test di API con verifica database

A volte non basta verificare la risposta HTTP. Vogliamo anche verificare che il database sia cambiato.

Esempio:

```csharp id="yzc8ro"
[Fact]
public async Task PostCliente_salvaClienteNelDatabase()
{
    HttpClient client = factory.CreateClient();

    CreaClienteRequest request = new(
        "Laura",
        "laura@example.com");
```

# Test di API con autenticazione finta

Molte API richiedono autenticazione.

Nei test di integrazione si può configurare uno schema finto.

Handler:

```csharp id="c2t3t6"
using System.Security.Claims;
using System.Text.Encodings.Web;
using Microsoft.AspNetCore.Authentication;
using Microsoft.Extensions.Options;

public class TestAuthHandler
    : AuthenticationHandler<AuthenticationSchemeOptions>
{
    public const string SchemeName = "Test";

    public TestAuthHandler(
        IOptionsMonitor<AuthenticationSchemeOptions> options,
        ILoggerFactory logger,
        UrlEncoder encoder)
        : base(options, logger, encoder)
    {
    }

    protected override Task<AuthenticateResult> HandleAuthenticateAsync()
    {
        Claim[] claims =
        [
            new Claim(ClaimTypes.NameIdentifier, "test-user"),
            new Claim(ClaimTypes.Name, "Utente Test"),
            new Claim(ClaimTypes.Role, "Admin")
        ];

        ClaimsIdentity identity = new ClaimsIdentity(
            claims,
            SchemeName);

        ClaimsPrincipal principal = new ClaimsPrincipal(identity);

        AuthenticationTicket ticket = new AuthenticationTicket(
            principal,
            SchemeName);

        return Task.FromResult(AuthenticateResult.Success(ticket));
    }
}
```

Nella factory:

```csharp id="59c9fr"
builder.ConfigureServices(services =>
{
    services
        .AddAuthentication(TestAuthHandler.SchemeName)
        .AddScheme<AuthenticationSchemeOptions, TestAuthHandler>(
            TestAuthHandler.SchemeName,
            options => { });
});
```

Questo permette di testare endpoint protetti senza dipendere da un provider reale.

## `CustomWebApplicationFactory.cs`

```csharp id="zn3ih7"
using Microsoft.AspNetCore.Hosting;
using Microsoft.AspNetCore.Mvc.Testing;
using Microsoft.Data.Sqlite;
using Microsoft.EntityFrameworkCore;
using Microsoft.Extensions.DependencyInjection;
using Microsoft.Extensions.DependencyInjection.Extensions;

public class CustomWebApplicationFactory
    : WebApplicationFactory<Program>
{
    private SqliteConnection? connection;

    protected override void ConfigureWebHost(IWebHostBuilder builder)
    {
        builder.UseEnvironment("Testing");

        builder.ConfigureServices(services =>
        {
            services.RemoveAll<DbContextOptions<AppDbContext>>();

            connection = new SqliteConnection("DataSource=:memory:");
            connection.Open();

            services.AddDbContext<AppDbContext>(options =>
            {
                options.UseSqlite(connection);
            });

            using ServiceProvider serviceProvider =
                services.BuildServiceProvider();

            using IServiceScope scope =
                serviceProvider.CreateScope();

            AppDbContext dbContext =
                scope.ServiceProvider.GetRequiredService<AppDbContext>();

            dbContext.Database.EnsureCreated();

            dbContext.Clienti.Add(new Cliente
            {
                Nome = "Cliente Seed",
                Email = "seed@example.com"
            });

            dbContext.SaveChanges();
        });
    }

    protected override void Dispose(bool disposing)
    {
        base.Dispose(disposing);

        if (disposing)
        {
            connection?.Dispose();
        }
    }
}
```

# Problema: stato condiviso tra test

Nell’esempio precedente, la factory è condivisa tra i test della classe.

Questo significa che se un test inserisce dati, i test successivi potrebbero vederli.

Possibili soluzioni:

```text id="xvdo7e"
creare factory nuova per ogni test
pulire database prima di ogni test
usare nomi/email unici
non fare assunzioni sul numero totale di record
usare fixture per classe solo con dati stabili
```

Esempio pratico:

```csharp id="kkisft"
string email = $"cliente-{Guid.NewGuid():N}@example.com";
```

Questo evita collisioni tra test.

# Integration test e CI/CD

I test di integrazione sono molto utili in pipeline.

Ma bisogna distinguere:

```text id="efvzrj"
test veloci
  eseguiti a ogni commit

test più pesanti
  eseguiti prima del merge o del rilascio

test con Docker
  richiedono runner configurati con Docker

test con database reale
  richiedono credenziali e cleanup sicuri
```

Regola pratica:

```text id="rl34rs"
Più un test è realistico, più tende a essere lento e complesso.
Serve bilanciare fiducia, velocità e manutenzione.
```

# Errori comuni

## Usare il database di produzione

Sbagliato:

```text id="bo6egm"
I test puntano alla connection string reale.
```

Gravissimo.

I test devono usare:

```text id="se1y2x"
database locale di test
SQLite in-memory
container temporaneo
database dedicato e isolato
```

---

## Test non isolati

Sospetto:

```text id="rxbg1t"
Il test passa solo se eseguito da solo.
Fallisce quando eseguo tutta la suite.
```

Causa tipica:

```text id="6o45sy"
stato condiviso
dati non puliti
ordine di esecuzione implicito
database persistente sporco
```

---

## Seed enorme e opaco

Sbagliato:

```text id="w314od"
Carico 10.000 righe di seed per ogni test.
```

Meglio dati minimi e mirati.

---

## Confondere integration test con end-to-end test

Integration test API con `WebApplicationFactory`:

```text id="tb9r2c"
testa server in memoria e pipeline ASP.NET Core
```

End-to-end test:

```text id="26w5mv"
testa browser reale, rete reale, deployment reale o quasi reale
```

Sono livelli diversi.

---

## Usare SQLite quando serve testare SQL Server

SQLite è ottimo, ma non sostituisce sempre SQL Server.

Se devi testare:

```text id="ax9t0q"
stored procedure SQL Server
T-SQL specifico
collation SQL Server
rowversion
schema complessi specifici SQL Server
funzioni EF.Functions specifiche
```

meglio un SQL Server reale, spesso via Testcontainers.

---

## Non verificare il database

Un test che controlla solo status code può essere insufficiente.

Esempio debole:

```csharp id="eud5f5"
Assert.Equal(HttpStatusCode.Created, response.StatusCode);
```

Meglio, quando ha senso:

```csharp id="54uxop"
Assert.True(await dbContext.Clienti.AnyAsync(c => c.Email == email));
```

# Linee guida pratiche

```text id="l5a5bo"
Usa unit test
  per regole pure e comportamento isolato.

Usa integration test
  per verificare collaborazione tra componenti reali.

Per EF Core semplice
  valuta SQLite in-memory.

Per provider specifici
  usa database reale o Testcontainers.

Per API ASP.NET Core
  usa WebApplicationFactory.

Per endpoint protetti
  configura autenticazione finta di test.

Prepara seed piccoli
  chiari e mirati.

Isola i test
  con database separati, cleanup o dati unici.

Pulisci sempre le risorse
  connessioni, file temporanei, container, dati.

Non usare mai produzione
  come ambiente di test.

Mantieni i test leggibili
  un test deve raccontare lo scenario.
```

# Riepilogo

In questo capitolo abbiamo visto i test di integrazione.

Abbiamo distinto:

```text id="6pk3vd"
unit test
  verifica una piccola unità isolata.

integration test
  verifica più componenti che collaborano.
```

Abbiamo studiato:

```text id="klg3n4"
test con database
SQLite in-memory
Testcontainers
test API ASP.NET Core
WebApplicationFactory
TestServer
seed dati
isolamento dei test
pulizia ambiente
```

Abbiamo visto che `WebApplicationFactory<TEntryPoint>` è lo strumento standard per creare un server di test in memoria per applicazioni ASP.NET Core e ottenere un `HttpClient` con cui chiamare l’applicazione. ([Microsoft Learn][7])

Abbiamo anche visto che SQLite in-memory è una buona soluzione intermedia per test EF Core veloci, ma che non sostituisce completamente il database di produzione; quando servono comportamenti realistici del provider, Testcontainers permette di usare servizi reali in Docker. ([Microsoft Learn][4])

La regola principale è questa:

```text id="3fz6gc"
Un buon test di integrazione deve essere abbastanza realistico da dare fiducia,
ma abbastanza isolato e controllato da essere ripetibile.
```

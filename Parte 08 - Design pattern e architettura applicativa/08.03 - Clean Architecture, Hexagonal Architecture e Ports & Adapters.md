# 08.03 — Clean Architecture, Hexagonal Architecture e Ports & Adapters

Nel capitolo precedente abbiamo introdotto Layered Architecture e concetti DDD come Application Layer, Domain Layer, Entity, Value Object, Aggregate, Repository, Domain Service, Application Service, Use Case, DTO, Mapping, linguaggio ubiquo e Bounded Context.

Ora facciamo un passo ulteriore.

Le architetture a layer tradizionali sono utili, ma possono degenerare facilmente in una struttura dove tutto finisce per dipendere dal database, dal framework web o da dettagli tecnici.

Clean Architecture e Hexagonal Architecture nascono per rispondere a questa domanda:

```text
Come posso progettare un’applicazione in modo che il cuore del sistema
non dipenda da database, UI, framework, file system, API esterne
o altri dettagli infrastrutturali?
```

Il punto centrale è questo:

```text
Clean Architecture e Hexagonal Architecture cercano di proteggere
il dominio e i casi d’uso dai dettagli esterni.
Il database, la UI e i servizi esterni diventano adapter,
non il centro dell’applicazione.
```

---

# Il problema della layered architecture classica

Una layered architecture classica spesso viene rappresentata così:

```text
Presentation
  ↓
Business Logic
  ↓
Data Access
  ↓
Database
```

Questa struttura sembra ordinata, ma nasconde un rischio:

```text
Il business layer tende a dipendere dal data access layer.
Il data access layer tende a modellare il sistema intorno al database.
Alla fine il database diventa il centro dell’architettura.
```

Esempio tipico:

```csharp
public class ClienteService
{
    private readonly AppDbContext dbContext;

    public ClienteService(AppDbContext dbContext)
    {
        this.dbContext = dbContext;
    }

    public async Task RegistraClienteAsync(CreaClienteRequest request)
    {
        Cliente cliente = new Cliente
        {
            Nome = request.Nome,
            Email = request.Email
        };

        dbContext.Clienti.Add(cliente);

        await dbContext.SaveChangesAsync();
    }
}
```

Questo codice è semplice, ma `ClienteService` conosce direttamente EF Core.

Per un’app CRUD piccola può andare bene.

Per un’app con dominio importante, il problema è che il caso d’uso dipende già da un dettaglio infrastrutturale:

```text
AppDbContext
EF Core
database relazionale
schema di persistenza
```

Clean Architecture e Hexagonal Architecture propongono un’inversione:

```text
Il cuore dell’applicazione definisce ciò di cui ha bisogno.
L’infrastruttura esterna implementa quei contratti.
```

---

# Clean Architecture

La **Clean Architecture** organizza il sistema in cerchi concentrici.

Al centro stanno le parti più importanti e più stabili:

```text
Domain
Use Cases / Application
```

All’esterno stanno i dettagli:

```text
Web
Database
File system
Email
Framework
UI
API esterne
```

Schema concettuale:

```text
┌─────────────────────────────────────┐
│ Frameworks & Drivers                │
│ ASP.NET Core, EF Core, UI, DB        │
│                                     │
│   ┌─────────────────────────────┐   │
│   │ Interface Adapters          │   │
│   │ Controllers, Presenters     │   │
│   │ Repositories concrete       │   │
│   │                             │   │
│   │   ┌─────────────────────┐   │   │
│   │   │ Use Cases           │   │   │
│   │   │ Application Layer   │   │   │
│   │   │                     │   │   │
│   │   │   ┌─────────────┐   │   │   │
│   │   │   │ Domain      │   │   │   │
│   │   │   │ Entities    │   │   │   │
│   │   │   └─────────────┘   │   │   │
│   │   └─────────────────────┘   │   │
│   └─────────────────────────────┘   │
└─────────────────────────────────────┘
```

La regola fondamentale è:

```text
Le dipendenze del codice devono puntare verso l’interno.
```

---

# Dipendenze verso l’interno

In Clean Architecture, i layer interni non devono conoscere i layer esterni.

Corretto:

```text
Web
  dipende da Application

Infrastructure
  dipende da Application e Domain

Application
  dipende da Domain

Domain
  non dipende da nessuno
```

Sbagliato:

```text
Domain
  dipende da EF Core

Application
  dipende da ASP.NET Core

Domain
  dipende da Newtonsoft.Json

Use case
  dipende da Controller MVC
```

Il dominio deve poter esistere senza sapere se verrà usato da:

```text
API REST
Blazor
Razor Pages
WPF
Console app
Worker Service
test automatici
```

E senza sapere se i dati verranno salvati in:

```text
SQL Server
SQLite
PostgreSQL
file JSON
Redis
servizio remoto
repository fake nei test
```

---

# Esempio di dipendenza sbagliata

```csharp
public class Ordine
{
    private readonly AppDbContext dbContext;

    public Ordine(AppDbContext dbContext)
    {
        this.dbContext = dbContext;
    }
}
```

Qui il dominio dipende da EF Core.

È una dipendenza verso l’esterno.

Meglio:

```csharp
public class Ordine
{
    private readonly List<OrdineRiga> righe = [];

    public int Id { get; private set; }
    public StatoOrdine Stato { get; private set; }

    public void Conferma()
    {
        if (righe.Count == 0)
        {
            throw new DomainException(
                "Non è possibile confermare un ordine senza righe.");
        }

        Stato = StatoOrdine.Confermato;
    }
}
```

Il dominio contiene regole, non dettagli di persistenza.

# Use case

In Clean Architecture, gli **use case** rappresentano le operazioni che l’applicazione offre.

Esempi:

```text
RegistraCliente
CreaOrdine
ConfermaOrdine
AnnullaPrenotazione
ApprovaDocumento
GeneraReport
ImportaFile
```

Uno use case coordina:

```text
input
validazioni applicative
caricamento dati
invocazione del dominio
salvataggio
eventuale invio notifiche
output
```

Esempio:

```csharp
public class ConfermaOrdineUseCase
{
    private readonly IOrdineRepository ordineRepository;
    private readonly IUnitOfWork unitOfWork;

    public ConfermaOrdineUseCase(
        IOrdineRepository ordineRepository,
        IUnitOfWork unitOfWork)
    {
        this.ordineRepository = ordineRepository;
        this.unitOfWork = unitOfWork;
    }

    public async Task EseguiAsync(
        int ordineId,
        CancellationToken cancellationToken)
    {
        Ordine ordine = await ordineRepository.GetByIdAsync(
            ordineId,
            cancellationToken)
            ?? throw new NotFoundException("Ordine non trovato.");

        ordine.Conferma();

        await unitOfWork.SaveChangesAsync(cancellationToken);
    }
}
```

Lo use case non sa se `IOrdineRepository` usa EF Core, ADO.NET, un file o un servizio remoto.

Sa solo che gli serve un ordine.

# Interfacce

Le interfacce sono il modo con cui il cuore dell’applicazione dichiara le sue necessità.

Esempio:

```csharp
public interface IOrdineRepository
{
    Task<Ordine?> GetByIdAsync(
        int id,
        CancellationToken cancellationToken);

    Task AddAsync(
        Ordine ordine,
        CancellationToken cancellationToken);
}
```

Oppure:

```csharp
public interface IEmailSender
{
    Task SendAsync(
        string destinatario,
        string oggetto,
        string corpo,
        CancellationToken cancellationToken);
}
```

Lo use case dipende dall’interfaccia:

```csharp
public class RegistraClienteUseCase
{
    private readonly IClienteRepository clienteRepository;
    private readonly IEmailSender emailSender;

    public RegistraClienteUseCase(
        IClienteRepository clienteRepository,
        IEmailSender emailSender)
    {
        this.clienteRepository = clienteRepository;
        this.emailSender = emailSender;
    }
}
```

L’infrastruttura fornisce l’implementazione concreta.

---

# Non tutte le interfacce sono utili

Clean Architecture non significa creare un’interfaccia per ogni classe.

Sospetto:

```csharp
public interface ICalcolatoreTotale
{
    decimal Calcola(...);
}

public class CalcolatoreTotale : ICalcolatoreTotale
{
}
```

Se non hai bisogno di invertire una dipendenza, sostituire un dettaglio esterno, testare in isolamento o rappresentare un vero contratto, l’interfaccia può essere inutile.

Interfacce utili:

```text
repository
servizi email
client API esterne
clock/time provider
file storage
message bus
unit of work
servizi di pagamento
generatori di ID, se serve controllo nei test
```

Interfacce spesso inutili:

```text
classi pure di dominio
value object
piccoli calcolatori senza dipendenze esterne
servizi interni mai sostituiti
classi usate solo in un punto
```

Regola pratica:

```text
Crea interfacce ai confini,
non automaticamente per ogni classe.
```

# Hexagonal Architecture

La **Hexagonal Architecture**, detta anche **Ports & Adapters**, usa una metafora diversa.

Al centro c’è l’applicazione.

Intorno ci sono porte e adattatori.

Schema:

```text
                 UI Web
                  │
             Web Adapter
                  │
                  ▼
        ┌─────────────────────┐
        │                     │
        │     Application     │
        │       Domain        │
        │                     │
        └─────────────────────┘
          ▲        ▲        ▲
          │        │        │
   DB Adapter  Email Adapter  API Adapter
          │        │        │
       Database   SMTP    Servizio esterno
```

L’idea è:

```text
L’applicazione non parla direttamente con il mondo esterno.
Parla attraverso porte.
Gli adapter traducono tra il mondo esterno e le porte.
```

---

# Port

Una **Port** è un punto di ingresso o uscita dell’applicazione.

Può essere:

```text
una interfaccia
un contratto
un use case
un comando
una query
```

Ci sono due grandi categorie:

```text
Inbound port
  permette al mondo esterno di chiamare l’applicazione.

Outbound port
  permette all’applicazione di usare il mondo esterno.
```

---

## Inbound Port

Una inbound port rappresenta ciò che l’applicazione offre.

Esempio:

```csharp
public interface ICreaOrdineUseCase
{
    Task<CreaOrdineResult> EseguiAsync(
        CreaOrdineCommand command,
        CancellationToken cancellationToken);
}
```

Un controller, una Minimal API, una CLI o una UI possono chiamare questa porta.

---

## Outbound Port

Una outbound port rappresenta ciò di cui l’applicazione ha bisogno.

Esempio:

```csharp
public interface IOrdineRepository
{
    Task<Ordine?> GetByIdAsync(
        int id,
        CancellationToken cancellationToken);

    Task AddAsync(
        Ordine ordine,
        CancellationToken cancellationToken);
}
```

Oppure:

```csharp
public interface IPaymentGateway
{
    Task<PaymentResult> AddebitaAsync(
        PaymentRequest request,
        CancellationToken cancellationToken);
}
```

L’applicazione non sa quale gateway concreto verrà usato.

---

# Adapter

Un **Adapter** collega una porta a una tecnologia concreta.

Esempi di adapter inbound:

```text
Minimal API endpoint
MVC Controller
Razor Page
Blazor component
SignalR Hub
CLI command
message consumer
```

Esempi di adapter outbound:

```text
EF Core repository
ADO.NET repository
SMTP email sender
HTTP client verso API esterna
file system storage
Redis cache
RabbitMQ publisher
Azure Service Bus sender
```

---

# Adapter inbound: Minimal API

```csharp
app.MapPost("/api/ordini", async (
    CreaOrdineRequest request,
    ICreaOrdineUseCase useCase,
    CancellationToken cancellationToken) =>
{
    CreaOrdineCommand command = new(
        request.ClienteId,
        request.Righe
            .Select(r => new CreaOrdineRigaCommand(
                r.ProdottoId,
                r.Quantita,
                r.PrezzoUnitario))
            .ToList());

    CreaOrdineResult result =
        await useCase.EseguiAsync(command, cancellationToken);

    return Results.Created(
        $"/api/ordini/{result.OrdineId}",
        result);
});
```

Questo endpoint è un adapter.

Traduce:

```text
HTTP request
  ↓
command applicativo
```

e poi:

```text
result applicativo
  ↓
HTTP response
```

# Adapter outbound: EF Core Repository

```csharp
public class EfOrdineRepository : IOrdineRepository
{
    private readonly AppDbContext dbContext;

    public EfOrdineRepository(AppDbContext dbContext)
    {
        this.dbContext = dbContext;
    }

    public async Task<Ordine?> GetByIdAsync(
        int id,
        CancellationToken cancellationToken)
    {
        return await dbContext.Ordini
            .Include(o => o.Righe)
            .FirstOrDefaultAsync(o => o.Id == id, cancellationToken);
    }

    public Task AddAsync(
        Ordine ordine,
        CancellationToken cancellationToken)
    {
        dbContext.Ordini.Add(ordine);

        return Task.CompletedTask;
    }
}
```

Questo adapter traduce:

```text
IOrdineRepository
  ↓
EF Core / database
```

# Infrastructure esterna

In Clean/Hexagonal Architecture, tutto ciò che è esterno al cuore applicativo è infrastructure o adapter.

Esempi:

```text
database
ORM
SMTP
file system
message broker
API esterne
cache distribuita
servizi cloud
framework web
librerie di pagamento
identity provider
```

L’infrastruttura è necessaria, ma non deve dominare il modello.

Il cuore deve parlare in termini di business:

```text
ordine
cliente
pagamento
documento
prenotazione
presenza
movimento
```

Non in termini di dettagli tecnici:

```text
DbSet
HttpClient
SmtpClient
SqlConnection
RedisDatabase
BlobClient
```

---

# Indipendenza dal database

Indipendenza dal database non significa che il database non conta.

Significa che le regole di business fondamentali non devono essere scritte in modo da dipendere direttamente da SQL Server, SQLite o EF Core.

Esempio sbagliato:

```csharp
public class ConfermaOrdineUseCase
{
    private readonly AppDbContext dbContext;

    public ConfermaOrdineUseCase(AppDbContext dbContext)
    {
        this.dbContext = dbContext;
    }
}
```

Esempio più pulito:

```csharp
public class ConfermaOrdineUseCase
{
    private readonly IOrdineRepository ordineRepository;
    private readonly IUnitOfWork unitOfWork;

    public ConfermaOrdineUseCase(
        IOrdineRepository ordineRepository,
        IUnitOfWork unitOfWork)
    {
        this.ordineRepository = ordineRepository;
        this.unitOfWork = unitOfWork;
    }
}
```

Ora lo use case dipende da contratti applicativi, non da EF Core.

---

## Indipendenza non significa sostituire database ogni settimana

Un’obiezione comune:

```text
Tanto non cambieremo mai database.
```

Il punto non è solo poter cambiare database.

Il punto è:

```text
testare senza database reale
tenere il dominio pulito
evitare che EF Core condizioni ogni classe
isolare query e mapping
ridurre accoppiamento
rendere esplicito cosa serve allo use case
```

Sostituire il database è solo uno dei possibili benefici, non l’unico.

```text
API REST

```

# Testabilità

Uno dei vantaggi più concreti di Clean/Hexagonal Architecture è la testabilità.

Se lo use case dipende da interfacce, puoi testarlo con implementazioni fake.

Esempio test:

```csharp
public class ConfermaOrdineUseCaseTests
{
    [Fact]
    public async Task ConfermaAsync_conOrdineInBozza_confermaOrdine()
    {
        FakeOrdineRepository repository = new();

        Ordine ordine = Ordine.Crea(clienteId: 10);
        ordine.AggiungiRiga(
            prodottoId: 1,
            quantita: 2,
            prezzoUnitario: 50m);

        repository.Ordine = ordine;

        FakeUnitOfWork unitOfWork = new();

        ConfermaOrdineUseCase useCase = new(
            repository,
            unitOfWork);

        await useCase.EseguiAsync(
            ordine.Id,
            CancellationToken.None);

        Assert.Equal(StatoOrdine.Confermato, ordine.Stato);
        Assert.True(unitOfWork.SaveChangesCalled);
    }
}
```

Fake repository:

```csharp
public class FakeOrdineRepository : IOrdineRepository
{
    public Ordine? Ordine { get; set; }

    public Task<Ordine?> GetByIdAsync(
        int id,
        CancellationToken cancellationToken)
    {
        return Task.FromResult(Ordine);
    }

    public Task AddAsync(
        Ordine ordine,
        CancellationToken cancellationToken)
    {
        Ordine = ordine;
        return Task.CompletedTask;
    }
}
```

Fake unit of work:

```csharp
public class FakeUnitOfWork : IUnitOfWork
{
    public bool SaveChangesCalled { get; private set; }

    public Task SaveChangesAsync(
        CancellationToken cancellationToken)
    {
        SaveChangesCalled = true;
        return Task.CompletedTask;
    }
}
```

Questo test non richiede:

```text
database
web server
HTTP
EF Core
file system
container Docker
```

# Esempio completo: Ports & Adapters

## Domain

```csharp
public class Ordine
{
    private readonly List<OrdineRiga> righe = [];

    public int Id { get; private set; }
    public int ClienteId { get; private set; }
    public StatoOrdine Stato { get; private set; }
    public IReadOnlyCollection<OrdineRiga> Righe => righe.AsReadOnly();

    private Ordine()
    {
    }

    private Ordine(int clienteId)
    {
        ClienteId = clienteId;
        Stato = StatoOrdine.Bozza;
    }

    public static Ordine Crea(int clienteId)
    {
        if (clienteId <= 0)
        {
            throw new DomainException("Cliente non valido.");
        }

        return new Ordine(clienteId);
    }

    public void AggiungiRiga(
        int prodottoId,
        int quantita,
        decimal prezzoUnitario)
    {
        if (Stato != StatoOrdine.Bozza)
        {
            throw new DomainException(
                "Non è possibile modificare un ordine non in bozza.");
        }

        if (quantita <= 0)
        {
            throw new DomainException(
                "La quantità deve essere maggiore di zero.");
        }

        righe.Add(new OrdineRiga(
            prodottoId,
            quantita,
            prezzoUnitario));
    }

    public void Conferma()
    {
        if (righe.Count == 0)
        {
            throw new DomainException(
                "Non è possibile confermare un ordine senza righe.");
        }

        Stato = StatoOrdine.Confermato;
    }
}
```

## Use case

```csharp
public class CreaOrdineUseCase : ICreaOrdineUseCase
{
    private readonly IOrdineRepository ordineRepository;
    private readonly IUnitOfWork unitOfWork;

    public CreaOrdineUseCase(
        IOrdineRepository ordineRepository,
        IUnitOfWork unitOfWork)
    {
        this.ordineRepository = ordineRepository;
        this.unitOfWork = unitOfWork;
    }

    public async Task<CreaOrdineResult> EseguiAsync(
        CreaOrdineCommand command,
        CancellationToken cancellationToken)
    {
        Ordine ordine = Ordine.Crea(command.ClienteId);

        foreach (CreaOrdineRigaCommand riga in command.Righe)
        {
            ordine.AggiungiRiga(
                riga.ProdottoId,
                riga.Quantita,
                riga.PrezzoUnitario);
        }

        ordine.Conferma();

        await ordineRepository.AddAsync(
            ordine,
            cancellationToken);

        await unitOfWork.SaveChangesAsync(cancellationToken);

        return new CreaOrdineResult(ordine.Id);
    }
}
```

# Struttura progetti possibile

Una struttura classica Clean Architecture:

```text
MiaApp.Domain
  Ordine.cs
  OrdineRiga.cs
  StatoOrdine.cs
  DomainException.cs

MiaApp.Application
  UseCases
    CreaOrdine
      CreaOrdineCommand.cs
      CreaOrdineResult.cs
      ICreaOrdineUseCase.cs
      CreaOrdineUseCase.cs
  Ports
    IOrdineRepository.cs
    IUnitOfWork.cs

MiaApp.Infrastructure
  Data
    AppDbContext.cs
    EfOrdineRepository.cs
    EfUnitOfWork.cs

MiaApp.Web
  Program.cs
  Endpoints
    OrdiniEndpoints.cs
  Contracts
    CreaOrdineRequest.cs
```

Dipendenze:

```text
MiaApp.Domain
  non dipende da altri progetti

MiaApp.Application
  dipende da Domain

MiaApp.Infrastructure
  dipende da Application e Domain

MiaApp.Web
  dipende da Application
  dipende da Infrastructure per registrare i servizi
```

Nota importante:

```text
Web può referenziare Infrastructure per comporre l’applicazione,
ma gli use case non devono dipendere da Web o Infrastructure.
```

# Variante modulare

In un monolite modulare puoi organizzare così:

```text
Modules
  Ordini
    Domain
      Ordine.cs
      OrdineRiga.cs
    Application
      CreaOrdineUseCase.cs
      IOrdineRepository.cs
    Infrastructure
      EfOrdineRepository.cs
    Presentation
      OrdiniEndpoints.cs
```

Questa struttura è spesso più vicina alle funzionalità reali.

Vantaggio:

```text
tutto ciò che riguarda Ordini è vicino
```

Svantaggio:

```text
serve disciplina per non duplicare troppo e per gestire dipendenze tra moduli
```

---

# Clean Architecture vs Hexagonal Architecture

Sono molto simili nello spirito.

Clean Architecture enfatizza:

```text
cerchi concentrici
dipendenze verso l’interno
entities
use cases
interface adapters
frameworks and drivers
```

Hexagonal Architecture enfatizza:

```text
applicazione al centro
ports
adapters
inbound e outbound
indipendenza dal mondo esterno
```

In pratica, in un progetto .NET, spesso si combinano.

Puoi dire:

```text
Uso Clean Architecture come regola di dipendenza
e Ports & Adapters come modo di modellare i confini.
```

---

# Differenza con layered architecture classica

Layered classica:

```text
Presentation
  ↓
Business
  ↓
Data Access
```

Clean/Hexagonal:

```text
Presentation Adapter
  ↓
Application / Domain
  ↑
Infrastructure Adapter
```

La differenza chiave è l’inversione della dipendenza.

Nella layered classica, spesso il business dipende dal data access.

Nella Clean/Hexagonal, l’application layer definisce il contratto e l’infrastruttura lo implementa.

---

# Pro

## 1. Testabilità

Gli use case possono essere testati senza database, web server o servizi esterni.

```text
Fake repository
Fake email sender
Fake clock
Fake payment gateway
```

---

## 2. Indipendenza dalla UI

La stessa logica può essere chiamata da:

```text
Minimal API
MVC Controller
Blazor
WPF
CLI
BackgroundService
SignalR Hub
```

---

## 3. Indipendenza dal database

Il dominio non dipende da EF Core o SQL.

Puoi cambiare o isolare:

```text
mapping
query
provider database
strategia di persistenza
```

---

## 4. Dominio più pulito

Le regole importanti vivono nel dominio, non in controller o query sparse.

---

## 5. Confini più chiari

Ports e adapters rendono più esplicito cosa entra e cosa esce dall’applicazione.

---

## 6. Manutenzione a lungo termine

Quando l’app cresce, la separazione aiuta a evitare che ogni modifica rompa parti lontane.

---

# Contro

## 1. Più codice

Servono più classi:

```text
command
result
use case
interfaccia
adapter
mapping
DTO
```

Per app piccole può sembrare eccessivo.

---

## 2. Maggiore curva di apprendimento

Il team deve capire:

```text
dipendenze verso l’interno
porte
adapter
use case
dominio
infrastruttura
```

Se non c’è disciplina, la struttura resta solo formale.

---

## 3. Rischio di over-engineering

Sospetto:

```text
un CRUD da 3 tabelle con 4 progetti,
30 interfacce e 50 classi di mapping
```

Non sempre serve.

---

## 4. Mapping aggiuntivo

Separare DTO, domain model ed entity persistence richiede mapping.

Questo può essere visto come costo.

---

## 5. Query complesse

Per query/report molto orientati ai dati, passare sempre dal dominio può essere inefficiente.

In questi casi ha senso separare comando e lettura:

```text
comandi
  dominio e aggregate

query
  proiezione diretta a DTO
```

Questo porta naturalmente verso CQRS leggero.

---

# Quando usarla

Clean/Hexagonal Architecture è adatta quando:

```text
il dominio è importante
l’app deve durare anni
servono test seri
ci sono più UI o canali di ingresso
ci sono integrazioni esterne
il database non deve dominare il modello
il team è abbastanza disciplinato
i casi d’uso sono più importanti del semplice CRUD
```

---

# Quando non usarla o usarla leggera

Usala con cautela quando:

```text
l’app è un prototipo
l’app è CRUD molto semplice
il team è piccolo e inesperto
il dominio è quasi inesistente
i requisiti cambieranno completamente a breve
il costo strutturale supera il beneficio
```

Versione leggera:

```text
un solo progetto
cartelle ben organizzate
use case chiari
DbContext isolato
dominio non troppo contaminato
interfacce solo ai confini necessari
```

Non è obbligatorio partire con quattro progetti.

---

# Errori comuni

## Pensare che Clean Architecture sia solo una struttura di cartelle

Sbagliato:

```text
Ho cartelle Domain, Application, Infrastructure,
quindi ho Clean Architecture.
```

No.

La domanda vera è:

```text
Le dipendenze puntano verso l’interno?
Il dominio è indipendente?
Gli use case dipendono da contratti?
I dettagli tecnici stanno fuori?
```

---

## Creare interfacce inutili

Non serve:

```text
IClienteService
ClienteService

IOrdineUseCase
OrdineUseCase

IMapper
Mapper
```

solo per “fare architettura”.

Crea interfacce quando rappresentano un confine o una sostituzione utile.

---

## Mettere EF Core nel dominio

Sbagliato:

```csharp
using Microsoft.EntityFrameworkCore;

public class Ordine
{
}
```

Il dominio non dovrebbe dipendere da EF Core.

---

## Mettere HTTP nello use case

Sbagliato:

```csharp
public async Task<IResult> EseguiAsync(...)
{
    return Results.Ok();
}
```

`IResult` è un dettaglio ASP.NET Core.

Lo use case dovrebbe restituire un result applicativo:

```csharp
public async Task<CreaOrdineResult> EseguiAsync(...)
{
}
```

---

## Rendere tutto troppo astratto

Sospetto:

```text
interfacce per ogni classe
factory per ogni oggetto
repository generico ovunque
service manager generici
DTO per ogni singolo passaggio senza motivo
```

L’architettura deve chiarire, non nascondere.

---

## Ignorare le query

Se per ogni schermata di elenco carichi aggregate complessi, le performance peggiorano.

Per le letture, spesso va bene fare query dirette a DTO.

---

# Linee guida pratiche

```text
Metti il dominio al centro
  e tienilo libero da framework e database.

Fai dipendere il codice verso l’interno
  non verso dettagli esterni.

Modella gli use case esplicitamente
  soprattutto quando il dominio è importante.

Usa interfacce come porte
  ai confini con infrastruttura e sistemi esterni.

Tratta controller, endpoint, UI e database come adapter
  non come cuore del sistema.

Non mettere tipi ASP.NET Core nell’Application Layer
  evita IResult, HttpContext, Controller, ClaimsPrincipal se non necessario.

Non mettere EF Core nel Domain Layer
  evita DbContext, DbSet e attributi invasivi.

Usa mapping consapevole
  tra request, command, domain e response.

Testa gli use case
  con fake adapter.

Non applicare Clean Architecture in modo dogmatico
  adatta la struttura alla complessità reale.
```

# Riepilogo

In questo capitolo abbiamo introdotto Clean Architecture, Hexagonal Architecture e Ports & Adapters.

Abbiamo visto:

```text
Clean Architecture
dipendenze verso l’interno
use case
interfacce
adapter
port
infrastructure esterna
testabilità
indipendenza dal database
indipendenza dalla UI
pro e contro
```

La regola fondamentale di Clean Architecture è:

```text
Il codice più importante deve dipendere dal codice più stabile,
non dai dettagli tecnici.
```

La regola fondamentale di Hexagonal Architecture è:

```text
L’applicazione sta al centro.
Il mondo esterno comunica con lei tramite porte e adapter.
```

La distinzione più utile è questa:

```text
Use case
  contiene il flusso applicativo.

Port
  definisce un contratto di ingresso o uscita.

Adapter
  collega quel contratto a una tecnologia concreta.

Infrastructure
  contiene dettagli tecnici sostituibili.

Domain
  contiene il cuore del business.
```

Il vantaggio principale è la protezione del cuore applicativo.

Il costo principale è maggiore struttura.

La regola pratica finale è questa:

```text
Clean Architecture e Hexagonal Architecture sono utili quando proteggono
il dominio e rendono i casi d’uso più chiari.
Diventano dannose quando producono astrazioni vuote,
mapping inutile e complessità superiore al problema reale.
```

Nel prossimo capitolo possiamo introdurre CQRS, MediatR, Command, Query e gestione esplicita dei casi d’uso, come evoluzione naturale dell’Application Layer.

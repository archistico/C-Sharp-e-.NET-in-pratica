# 08.02 — Layered Architecture, DDD e Application Layer

Nel capitolo precedente abbiamo fatto il passaggio dalla classe all’architettura. Abbiamo distinto layer e tier, monolite e monolite modulare, architettura a tre livelli e architettura a quattro layer.

Ora entriamo nel vocabolario più utile per progettare applicazioni aziendali reali.

Il punto centrale è questo:

```text id="df3nmk"
Layered Architecture e DDD non servono a complicare il progetto.
Servono a separare casi d’uso, regole di business e dettagli tecnici,
così che l’applicazione possa crescere senza diventare ingestibile.
```

In particolare, dobbiamo capire bene il ruolo dell’**Application Layer**.

È uno dei layer più fraintesi: spesso viene confuso con il dominio, con i service generici o con l’infrastruttura. In realtà è il layer che coordina i **casi d’uso** dell’applicazione.

---

# Il problema da risolvere

Immaginiamo un endpoint che registra un ordine.

Versione ingenua:

```csharp id="3uux4f"
app.MapPost("/api/ordini", async (
    CreaOrdineRequest request,
    AppDbContext dbContext,
    EmailService emailService) =>
{
    Cliente? cliente = await dbContext.Clienti
        .FirstOrDefaultAsync(c => c.Id == request.ClienteId);

    if (cliente is null)
    {
        return Results.NotFound();
    }

    Ordine ordine = new Ordine
    {
        ClienteId = cliente.Id,
        Data = DateTime.UtcNow,
        Stato = "Creato"
    };

    foreach (CreaOrdineRigaRequest riga in request.Righe)
    {
        ordine.Righe.Add(new OrdineRiga
        {
            ProdottoId = riga.ProdottoId,
            Quantita = riga.Quantita,
            PrezzoUnitario = riga.PrezzoUnitario
        });
    }

    dbContext.Ordini.Add(ordine);

    await dbContext.SaveChangesAsync();

    await emailService.InviaConfermaOrdineAsync(cliente.Email, ordine.Id);

    return Results.Created($"/api/ordini/{ordine.Id}", ordine.Id);
});
```

Funziona, ma mescola:

```text id="nir91s"
HTTP
accesso ai dati
regole di dominio
creazione oggetti
transazione
invio email
mapping
risposte API
```

Questo approccio sembra rapido all’inizio, ma con il tempo produce endpoint enormi e difficili da testare.

Una layered architecture aiuta a distribuire le responsabilità.

---

# I quattro layer principali

Una struttura molto usata nelle applicazioni aziendali distingue:

```text id="dxg760"
Presentation Layer
Application Layer
Domain Layer
Infrastructure Layer
```

Schema concettuale:

```text id="5ltlsy"
Presentation
  riceve input e restituisce output

Application
  coordina il caso d'uso

Domain
  contiene regole e concetti di business

Infrastructure
  implementa dettagli tecnici
```

# Presentation Layer

Il **Presentation Layer** è il punto di ingresso dell’applicazione.

Può essere:

```text id="23vyqv"
Minimal API
Controller MVC
Razor Pages
Blazor component
SignalR Hub
WPF ViewModel
Console command
Worker endpoint
```

Responsabilità:

```text id="thw21t"
ricevere request o input utente
fare model binding
validare aspetti superficiali dell’input
chiamare application layer
tradurre risultato in HTTP response o UI
gestire aspetti specifici del canale
```

Non dovrebbe contenere business logic profonda.

```text id="eeysj6"

```

# Application Layer

L’**Application Layer** coordina i casi d’uso.

È il layer che risponde alla domanda:

```text id="f4q8sa"
Che cosa deve fare l’applicazione quando l’utente o un sistema esterno
richiede questa operazione?
```

Esempi di use case:

```text id="4rudfy"

CreaOrdine
ConfermaOrdine
AnnullaPrenotazione
ApprovaDocumento
GeneraReport
InviaNotifica

```

Responsabilità dell’Application Layer:

```text id="bl86t6"
orchestrare il flusso
caricare dati necessari
chiamare il dominio
salvare modifiche
gestire transazioni applicative
chiamare repository tramite interfacce
chiamare servizi esterni tramite astrazioni
```

```text id="701joo"

SQL specifico

```

## Application Service

Un **Application Service** è un servizio che implementa uno o più casi d’uso.

Esempio:

```csharp id="z3hg52"
public class CreaOrdineUseCase
{
    private readonly IClienteRepository clienteRepository;
    private readonly IOrdineRepository ordineRepository;
    private readonly IUnitOfWork unitOfWork;
    private readonly IEmailService emailService;

    public CreaOrdineUseCase(
        IClienteRepository clienteRepository,
        IOrdineRepository ordineRepository,
        IUnitOfWork unitOfWork,
        IEmailService emailService)
    {
        this.clienteRepository = clienteRepository;
        this.ordineRepository = ordineRepository;
        this.unitOfWork = unitOfWork;
        this.emailService = emailService;
    }

    public async Task<CreaOrdineResult> EseguiAsync(
        CreaOrdineRequest request,
        CancellationToken cancellationToken)
    {
        Cliente cliente = await clienteRepository.GetByIdAsync(
            request.ClienteId,
            cancellationToken)
            ?? throw new NotFoundException("Cliente non trovato.");

        Ordine ordine = Ordine.Crea(cliente.Id);

        foreach (CreaOrdineRigaRequest riga in request.Righe)
        {
            ordine.AggiungiRiga(
                riga.ProdottoId,
                riga.Quantita,
                riga.PrezzoUnitario);
        }

        await ordineRepository.AddAsync(ordine, cancellationToken);

        await unitOfWork.SaveChangesAsync(cancellationToken);

        await emailService.InviaConfermaOrdineAsync(
            cliente.Email,
            ordine.Id,
            cancellationToken);

        return new CreaOrdineResult(ordine.Id);
    }
}
```

Nota importante:

```text id="23bya9"
Il caso d’uso coordina.
Non deve trasformarsi in una classe enorme con tutte le regole di business.
```

Le regole fondamentali devono stare nel dominio.

# Entity

Una **Entity** è un oggetto con identità.

Due entity possono avere gli stessi dati, ma essere diverse perché hanno identità diversa.

Esempio:

```csharp id="vv7up5"
public class Cliente
{
    public int Id { get; private set; }
    public string Nome { get; private set; }
    public string? Email { get; private set; }

    public Cliente(int id, string nome, string? email)
    {
        if (string.IsNullOrWhiteSpace(nome))
        {
            throw new DomainException("Il nome è obbligatorio.");
        }

        Id = id;
        Nome = nome;
        Email = email;
    }

    public void CambiaEmail(string? email)
    {
        Email = email;
    }
}
```

L’identità è `Id`.

Anche se due clienti hanno stesso nome e stessa email, non sono necessariamente lo stesso cliente.

---

# Value Object

Un **Value Object** è un oggetto senza identità propria, definito dai suoi valori.

Esempi:

```text id="ujy1fi"
Email
Indirizzo
Periodo
Importo
CodiceFiscale
Coordinate
Denaro
Percentuale
```

Esempio:

```csharp id="io4fnh"
public readonly record struct Periodo(
    DateOnly DataDa,
    DateOnly DataA)
{
    public Periodo : this()
    {
        if (DataA < DataDa)
        {
            throw new DomainException(
                "La data finale non può essere precedente alla data iniziale.");
        }
    }

    public bool Contiene(DateOnly data)
    {
        return data >= DataDa && data <= DataA;
    }
}
```

Oppure, più classicamente:

```csharp id="agtb17"
public sealed record Email
{
    public string Value { get; }

    public Email(string value)
    {
        if (string.IsNullOrWhiteSpace(value))
        {
            throw new DomainException("Email obbligatoria.");
        }

        if (!value.Contains('@'))
        {
            throw new DomainException("Email non valida.");
        }

        Value = value;
    }

    public override string ToString()
    {
        return Value;
    }
}
```

Due `Email` con lo stesso valore sono uguali.

```csharp id="aw62gd"
Email a = new("a@example.com");
Email b = new("a@example.com");

bool uguali = a == b;
```

Con un `record`, l’uguaglianza per valore è naturale.

---

# Entity vs Value Object

```text id="srn859"
Entity
  ha identità.
  Cambia nel tempo.
  Due oggetti con stessi dati possono essere diversi.

Value Object
  non ha identità.
  È definito dai suoi valori.
  Di solito è immutabile.
```

Esempio:

```text id="l8n1wb"
Cliente
  Entity

Indirizzo
  Value Object

Ordine
  Entity / Aggregate Root

RigaOrdine
  Entity o parte dell’Aggregate, secondo il dominio

Importo
  Value Object
```

# Aggregate

Un **Aggregate** è un gruppo di oggetti di dominio che viene trattato come una singola unità di consistenza.

Ha una radice, chiamata **Aggregate Root**.

Esempio:

```text id="ndmpsg"
Ordine
  Aggregate Root

OrdineRiga
  oggetto interno all’aggregate
```

L’esterno non dovrebbe modificare direttamente le righe senza passare da `Ordine`.

---

## Esempio Aggregate

```csharp id="0aewpw"
public class Ordine
{
    private readonly List<OrdineRiga> righe = [];

    public int Id { get; private set; }
    public int ClienteId { get; private set; }
    public StatoOrdine Stato { get; private set; }
    public IReadOnlyCollection<OrdineRiga> Righe => righe.AsReadOnly();

    private Ordine()
    {
    }

    private Ordine(int clienteId)
    {
        ClienteId = clienteId;
        Stato = StatoOrdine.Bozza;
    }

    public static Ordine Crea(int clienteId)
    {
        if (clienteId <= 0)
        {
            throw new DomainException("Cliente non valido.");
        }

        return new Ordine(clienteId);
    }

    public void AggiungiRiga(
        int prodottoId,
        int quantita,
        decimal prezzoUnitario)
    {
        if (Stato != StatoOrdine.Bozza)
        {
            throw new DomainException(
                "Non è possibile modificare un ordine non in bozza.");
        }

        if (quantita <= 0)
        {
            throw new DomainException(
                "La quantità deve essere maggiore di zero.");
        }

        if (prezzoUnitario < 0)
        {
            throw new DomainException(
                "Il prezzo non può essere negativo.");
        }

        righe.Add(new OrdineRiga(
            prodottoId,
            quantita,
            prezzoUnitario));
    }

    public void Conferma()
    {
        if (righe.Count == 0)
        {
            throw new DomainException(
                "Non è possibile confermare un ordine senza righe.");
        }

        Stato = StatoOrdine.Confermato;
    }
}
```

`OrdineRiga`:

```csharp id="etptjs"
public class OrdineRiga
{
    public int ProdottoId { get; private set; }
    public int Quantita { get; private set; }
    public decimal PrezzoUnitario { get; private set; }

    private OrdineRiga()
    {
    }

    internal OrdineRiga(
        int prodottoId,
        int quantita,
        decimal prezzoUnitario)
    {
        ProdottoId = prodottoId;
        Quantita = quantita;
        PrezzoUnitario = prezzoUnitario;
    }

    public decimal Totale => Quantita * PrezzoUnitario;
}
```

Nota:

```text id="7bxti9"
OrdineRiga non viene modificata liberamente dall’esterno.
Si passa da Ordine.
```

Questo protegge le invarianti dell’aggregate.

# Repository

Un **Repository** rappresenta una collezione concettuale di aggregate o entità.

Non è semplicemente una classe CRUD generica.

Il suo scopo è nascondere i dettagli di persistenza al dominio e all’application layer.

Esempio:

```csharp id="cyru01"
public interface IOrdineRepository
{
    Task<Ordine?> GetByIdAsync(
        int id,
        CancellationToken cancellationToken);

    Task AddAsync(
        Ordine ordine,
        CancellationToken cancellationToken);
}
```

Implementazione infrastrutturale:

```csharp id="ea11mz"
public class EfOrdineRepository : IOrdineRepository
{
    private readonly AppDbContext dbContext;

    public EfOrdineRepository(AppDbContext dbContext)
    {
        this.dbContext = dbContext;
    }

    public async Task<Ordine?> GetByIdAsync(
        int id,
        CancellationToken cancellationToken)
    {
        return await dbContext.Ordini
            .Include(o => o.Righe)
            .FirstOrDefaultAsync(o => o.Id == id, cancellationToken);
    }

    public Task AddAsync(
        Ordine ordine,
        CancellationToken cancellationToken)
    {
        dbContext.Ordini.Add(ordine);

        return Task.CompletedTask;
    }
}
```

---

Questo parla il linguaggio del dominio.

# Domain Service

Un **Domain Service** contiene logica di dominio che non appartiene naturalmente a una singola entity o value object.

Esempio:

```text id="mejom4"
calcolo tariffa che coinvolge cliente, contratto e periodo
verifica eleggibilità che coinvolge più aggregate
politica di sconto complessa
regola di assegnazione
```

Esempio:

```csharp id="mcfnhp"
public class CalcolatoreScontoCliente
{
    public decimal CalcolaSconto(
        Cliente cliente,
        decimal importoTotale,
        DateOnly data)
    {
        if (cliente.Tipo == TipoCliente.Vip &&
            importoTotale >= 1000m)
        {
            return importoTotale * 0.10m;
        }

        return 0m;
    }
}
```

Un Domain Service:

```text id="wgzj8a"
contiene regole di business
non coordina UI o database
non invia email
non gestisce HTTP
non è un application service
```

---

# Application Service vs Domain Service

Questa distinzione è fondamentale.

```text id="znxwy6"
Application Service
  coordina un caso d’uso.

Domain Service
  contiene una regola di dominio che non appartiene a una singola entity.
```

Esempio:

```text id="h7n4s3"
CreaOrdineUseCase
  Application Service

CalcolatoreScontoCliente
  Domain Service
```

Application Service:

```csharp id="qewoa2"
public class CreaOrdineUseCase
{
    public async Task<CreaOrdineResult> EseguiAsync(...)
    {
        // carica cliente
        // crea ordine
        // chiama dominio
        // salva
        // invia email/evento
    }
}
```

Domain Service:

```csharp id="cw1ref"
public class PoliticaSconto
{
    public decimal Calcola(Cliente cliente, Ordine ordine)
    {
        // regola di business pura
    }
}
```

## Use case come classe dedicata

```csharp id="gzri0m"
public class ConfermaOrdineUseCase
{
    private readonly IOrdineRepository ordineRepository;
    private readonly IUnitOfWork unitOfWork;

    public ConfermaOrdineUseCase(
        IOrdineRepository ordineRepository,
        IUnitOfWork unitOfWork)
    {
        this.ordineRepository = ordineRepository;
        this.unitOfWork = unitOfWork;

    public async Task ConfermaAsync(
        int ordineId,
        CancellationToken cancellationToken)

        await unitOfWork.SaveChangesAsync(cancellationToken);

}
```

# DTO

Un **DTO** è un oggetto usato per trasportare dati tra layer o attraverso confini esterni.

Esempi:

```csharp id="hucb7b"
public record CreaOrdineRequest(
    int ClienteId,
    List<CreaOrdineRigaRequest> Righe);

public record CreaOrdineRigaRequest(
    int ProdottoId,
    int Quantita,
    decimal PrezzoUnitario);

public record CreaOrdineResult(
    int OrdineId);
```

I DTO sono utili perché:

```text id="or0hg5"
separano API e dominio
evitano over-posting
rendono chiaro il contratto
permettono versionamento
semplificano serializzazione
evitano di esporre entità EF/Core Domain direttamente
```

---

## DTO non è Domain Model

Sbagliato:

```text id="br8hjd"
Uso lo stesso oggetto per:
  request API
  entity EF
  dominio
  view model
  response JSON
```

Può funzionare in demo piccole, ma in applicazioni reali crea accoppiamento.

Meglio distinguere:

```text id="5iq5na"
Request DTO
Response DTO
Domain Entity
EF mapping
ViewModel, se serve
```

# Mapping

Il **mapping** trasforma dati da una forma all’altra.

Esempi:

```text id="8mbdso"
Request DTO → Domain Entity
Domain Entity → Response DTO
Entity EF → DTO
DTO → ViewModel
```

Mapping manuale:

```csharp id="7211kv"
public static class OrdineMapping
{
    public static OrdineDto ToDto(this Ordine ordine)
    {
        return new OrdineDto(
            ordine.Id,
            ordine.ClienteId,
            ordine.Stato.ToString(),
            ordine.Righe
                .Select(r => new OrdineRigaDto(
                    r.ProdottoId,
                    r.Quantita,
                    r.PrezzoUnitario,
                    r.Totale))
                .ToList());
    }
}
```

DTO:

```csharp id="vpjp38"
public record OrdineDto(
    int Id,
    int ClienteId,
    string Stato,
    List<OrdineRigaDto> Righe);

public record OrdineRigaDto(
    int ProdottoId,
    int Quantita,
    decimal PrezzoUnitario,
    decimal Totale);
```

---

## Mapping nelle query

Quando usiamo EF Core per leggere dati, spesso conviene proiettare direttamente a DTO:

```csharp id="ph8m7q"
List<OrdineDto> ordini = await dbContext.Ordini
    .AsNoTracking()
    .Where(o => o.ClienteId == clienteId)
    .Select(o => new OrdineDto(
        o.Id,
        o.ClienteId,
        o.Stato.ToString(),
        o.Righe
            .Select(r => new OrdineRigaDto(
                r.ProdottoId,
                r.Quantita,
                r.PrezzoUnitario,
                r.Quantita * r.PrezzoUnitario))
            .ToList()))
    .ToListAsync(cancellationToken);
```

Questo può evitare di caricare aggregate completi se devi solo mostrare dati.

Regola pratica:

```text id="ivmj5b"
Per comandi che modificano stato:
  carica aggregate e usa il dominio.

Per query di sola lettura:
  spesso proietta direttamente a DTO.
```

Questa distinzione anticipa naturalmente CQRS.

# Linguaggio ubiquo

Il **linguaggio ubiquo** è il linguaggio condiviso tra sviluppatori ed esperti del dominio.

Significa che nomi di classi, metodi, concetti e conversazioni dovrebbero usare gli stessi termini del business.

Esempio generico e povero:

```csharp id="7we2x0"
public class DataManager
{
    public void Process(int id)
    {
    }
}
```

Esempio più espressivo:

```csharp id="fhdkuj"
public class ConfermaOrdineUseCase
{
    public Task ConfermaAsync(int ordineId, CancellationToken cancellationToken)
    {
        // ...
    }
}
```

Il codice deve raccontare il dominio.

---

```text id="xkwckw"
presenza
buono

```

allora il codice dovrebbe riflettere quei termini:

```csharp id="t21vf2"
buono.PuòEssereAssegnatoA(cliente)
```

Non:

```csharp id="esmhjv"
flag1
status = 3
Process()

Manager()
```

# Bounded Context

Un **Bounded Context** è un confine entro cui un modello e un linguaggio hanno significato preciso.

Lo stesso termine può voler dire cose diverse in contesti diversi.

Esempio:

```text id="x90bie"
Cliente nel contesto Marketing
  interessi, segmentazione, campagne, storico visite.

Cliente nel contesto Fatturazione
  dati fiscali, pagamenti, documenti contabili.

Cliente nel contesto Supporto
  ticket, problemi, SLA, contatti.
```

È sempre “cliente”, ma non è lo stesso modello.

---

## Perché serve

Senza bounded context, si tende a creare un unico modello enorme:

```csharp id="9iqage"
public class Cliente
{
    // dati anagrafici
    // dati marketing
    // dati fiscali
    // dati supporto
    // dati documento
    // dati privacy
    // dati contabili
    // dati notifiche
    // 200 proprietà
}
```

Questo modello diventa fragile.

Con bounded context:

```text id="v702zk"
Marketing.Cliente
Fatturazione.Cliente
Supporto.Cliente
Anagrafica.Cliente
```

Ogni contesto usa il modello adatto al proprio scopo.

---

# Bounded Context e moduli

In un monolite modulare, un bounded context può corrispondere a un modulo.

Esempio:

```text id="hz78lr"
Modules
  AnagraficaClienti
  Marketing
  Fatturazione
  Documenti
  Presenze
```

Ogni modulo ha:

```text id="22vmoy"
propri use case
propri DTO
proprio modello
proprie regole
proprie query
propri confini
```

Non sempre serve separare fisicamente in microservizi.

```csharp id="s8qffn"
public class Ordine
{
    private readonly List<OrdineRiga> righe = [];

    public int Id { get; private set; }
    public int ClienteId { get; private set; }
    public StatoOrdine Stato { get; private set; }
    public IReadOnlyCollection<OrdineRiga> Righe => righe.AsReadOnly();

    private Ordine()
    {
    }

    private Ordine(int clienteId)
    {
        ClienteId = clienteId;
        Stato = StatoOrdine.Bozza;
    }

    public static Ordine Crea(int clienteId)
    {
        if (clienteId <= 0)
        {
            throw new DomainException("Cliente non valido.");
        }

        return new Ordine(clienteId);
    }

    public void AggiungiRiga(
        int prodottoId,
        int quantita,
        decimal prezzoUnitario)
    {
        if (Stato != StatoOrdine.Bozza)
        {
            throw new DomainException(
                "Non è possibile modificare un ordine non in bozza.");
        }

        if (quantita <= 0)
        {
            throw new DomainException(
                "La quantità deve essere maggiore di zero.");
        }

        if (prezzoUnitario < 0)
        {
            throw new DomainException(
                "Il prezzo non può essere negativo.");
        }

        righe.Add(new OrdineRiga(
            prodottoId,
            quantita,
            prezzoUnitario));
    }

    public void Conferma()
    {
        if (righe.Count == 0)
        {
            throw new DomainException(
                "Non è possibile confermare un ordine senza righe.");
        }

        Stato = StatoOrdine.Confermato;
    }
}
```

```csharp id="b5dcl8"
public class OrdineRiga
{
    public int ProdottoId { get; private set; }
    public int Quantita { get; private set; }
    public decimal PrezzoUnitario { get; private set; }

    private OrdineRiga()
    {
    }

    internal OrdineRiga(
        int prodottoId,
        int quantita,
        decimal prezzoUnitario)
    {
        ProdottoId = prodottoId;
        Quantita = quantita;
        PrezzoUnitario = prezzoUnitario;
    }

    public decimal Totale => Quantita * PrezzoUnitario;
}
```

## Use case

```csharp id="xudqs3"
public class CreaOrdineUseCase
{
    private readonly IOrdineRepository ordineRepository;
    private readonly IUnitOfWork unitOfWork;

    public CreaOrdineUseCase(
        IOrdineRepository ordineRepository,
        IUnitOfWork unitOfWork)
    {
        this.ordineRepository = ordineRepository;
        this.unitOfWork = unitOfWork;
    }

    public async Task<CreaOrdineResult> EseguiAsync(
        CreaOrdineRequest request,
        CancellationToken cancellationToken)
    {
        Ordine ordine = Ordine.Crea(request.ClienteId);

        foreach (CreaOrdineRigaRequest riga in request.Righe)
        {
            ordine.AggiungiRiga(
                riga.ProdottoId,
                riga.Quantita,
                riga.PrezzoUnitario);
        }

        ordine.Conferma();

        await ordineRepository.AddAsync(
            ordine,
            cancellationToken);

        await unitOfWork.SaveChangesAsync(cancellationToken);

        return new CreaOrdineResult(ordine.Id);
    }
}
```

## Infrastructure EF Core

```csharp id="ipmkwb"
public class EfOrdineRepository : IOrdineRepository
{
    private readonly AppDbContext dbContext;

    public EfOrdineRepository(AppDbContext dbContext)
    {
        this.dbContext = dbContext;
    }

    public Task AddAsync(
        Ordine ordine,
        CancellationToken cancellationToken)
    {
        dbContext.Ordini.Add(ordine);

        return Task.CompletedTask;
    }

    public async Task<Ordine?> GetByIdAsync(
        int id,
        CancellationToken cancellationToken)
    {
        return await dbContext.Ordini
            .Include(o => o.Righe)
            .FirstOrDefaultAsync(o => o.Id == id, cancellationToken);
    }
}
```

# Dove mettere la validazione?

Ci sono più livelli di validazione.

## Validazione di input

Esempio:

```text id="a57k88"
ClienteId obbligatorio
Righe non vuote
Quantità formalmente valida
```

Può stare nel Presentation/Application Layer, spesso con Data Annotations o validator dedicati.

## Validazione applicativa

Esempio:

```text id="j0m5co"
cliente esiste
prodotto esiste
utente può creare ordini per quel cliente
```

Sta nell’Application Layer.

## Invarianti di dominio

Esempio:

```text id="l9nqwf"
un ordine confermato non può essere modificato
un ordine senza righe non può essere confermato
una quantità deve essere maggiore di zero
```

Devono stare nel Domain Layer.

Regola pratica:

```text id="ghn8i1"
Non affidare le invarianti fondamentali solo alla UI o all’endpoint.
Il dominio deve proteggere sé stesso.
```

# Letture e scritture non sono sempre uguali

Per un comando di scrittura:

```text id="f3axgr"
carica aggregate
chiama metodi di dominio
salva modifiche
```

Per una query di lettura:

```text id="dz6qfk"
puoi spesso proiettare direttamente a DTO
senza costruire tutto il dominio
```

Esempio query:

```csharp id="tukf5s"
public class ListaOrdiniQuery
{
    private readonly AppDbContext dbContext;

    public ListaOrdiniQuery(AppDbContext dbContext)
    {
        this.dbContext = dbContext;
    }

    public async Task<List<OrdineListItemDto>> EseguiAsync(
        int clienteId,
        CancellationToken cancellationToken)
    {
        return await dbContext.Ordini
            .AsNoTracking()
            .Where(o => o.ClienteId == clienteId)
            .OrderByDescending(o => o.Id)
            .Select(o => new OrdineListItemDto(
                o.Id,
                o.Stato.ToString(),
                o.Righe.Sum(r => r.Quantita * r.PrezzoUnitario)))
            .ToListAsync(cancellationToken);
    }
}
```

Questo non viola il dominio se stiamo solo leggendo dati per una vista.

# Errori comuni

## Mettere logica di dominio nell’Application Layer

Sospetto:

```text id="kobc7m"
CreaOrdineUseCase contiene tutti gli if sulle regole dell’ordine
e Ordine è solo una classe con proprietà pubbliche.
```

Meglio:

```text id="amf9q9"
Application coordina.
Domain decide.
```

---

## Mettere dettagli infrastrutturali nel dominio

Sbagliato:

```csharp id="ul9odl"
public class Ordine
{
    private readonly AppDbContext dbContext;
}
```

Il dominio non deve conoscere EF Core.

---

## Usare DTO come entità di dominio

Sbagliato:

```text id="tzjgpa"
CreaOrdineRequest viene passato ovunque,
anche dentro il dominio.
```

Meglio trasformare la request in operazioni sul dominio.

---

## Repository generici ovunque

Un repository generico può sembrare pulito ma spesso non esprime nulla del dominio.

Meglio repository specifici quando servono.

---

## Bounded context ignorati

Sospetto:

```text id="4wm2oa"
un’unica classe Cliente usata da marketing, fatturazione,
supporto, privacy, documenti e report.
```

Meglio modelli separati se i contesti hanno significati diversi.

---

## Mapping duplicato senza controllo

Se lo stesso mapping viene copiato in dieci punti, diventa fragile.

Soluzioni possibili:

```text id="81q5z4"
metodi di estensione
mapper dedicati
proiezioni centralizzate
librerie di mapping, se utili
```

# Linee guida pratiche

```text id="t2jo1x"
Tieni il Presentation Layer sottile
  deve ricevere input e restituire output.

Usa l’Application Layer
  per orchestrare casi d’uso.

Metti le invarianti nel Domain Layer
  il dominio deve proteggere sé stesso.

Tieni Infrastructure fuori dal dominio
  database, file, email e API esterne sono dettagli.

Usa Entity
  quando serve identità.

Usa Value Object
  quando conta il valore e vuoi immutabilità.

Usa Aggregate
  per proteggere gruppi di oggetti coerenti.

Usa Repository specifici
  quando esprimono concetti del dominio.

Distingui Domain Service e Application Service
  regola di dominio vs coordinamento use case.

Usa DTO
  per confini esterni e contratti.

Fai mapping esplicito
  almeno nei punti importanti.

Usa linguaggio ubiquo
  nomi del codice allineati al business.

Ragiona per bounded context
  lo stesso termine può avere modelli diversi.
```

# Riepilogo

In questo capitolo abbiamo introdotto il vocabolario architetturale più utile per applicazioni aziendali.

Abbiamo visto:

```text id="3f4qrf"
Presentation Layer
Application Layer
Domain Layer
Infrastructure Layer
Entity
Value Object
Aggregate
Repository
Domain Service
Application Service
Use case
DTO
Mapping
Linguaggio ubiquo
Bounded context
```

Abbiamo chiarito che:

```text id="vvd0uw"
Presentation
  gestisce il canale di ingresso e uscita.

Application
  coordina il caso d’uso.

Domain
  contiene regole, concetti e invarianti.

Infrastructure
  implementa dettagli tecnici.
```

La distinzione più importante è questa:

```text id="fiwa7e"
Application Service
  dice cosa deve succedere in un caso d’uso.

Domain Model
  decide se una regola di business è valida.

Infrastructure
  sa come salvare, inviare, leggere o integrare.
```

La regola principale è questa:

```text id="t1qkvq"
DDD non significa creare molti progetti e molte interfacce.
Significa costruire un modello che parla il linguaggio del dominio,
protegge le regole importanti e separa i dettagli tecnici
dal cuore dell’applicazione.
```

Questo capitolo prepara il terreno per i pattern successivi: non come catalogo astratto, ma come strumenti per risolvere problemi concreti di creazione, composizione, disaccoppiamento, notifiche, comandi, strategie e integrazione.

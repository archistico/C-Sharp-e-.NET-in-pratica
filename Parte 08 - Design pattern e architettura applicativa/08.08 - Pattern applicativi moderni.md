# 08.08 — Pattern applicativi moderni

Nei capitoli precedenti abbiamo studiato i pattern classici: creazionali, strutturali e comportamentali. Ora facciamo un passo più vicino alle applicazioni reali.

I pattern applicativi moderni non sono solo pattern “di codice”. Sono pattern che aiutano a risolvere problemi tipici di sistemi web, gestionali, API, microservizi e architetture distribuite.

Il punto centrale è questo:

```text
I pattern applicativi moderni collegano dominio, persistenza,
configurazione, eventi, resilienza e integrazione con sistemi esterni.
```

Alcuni sono semplici e utili anche in un monolite.
Altri diventano davvero importanti quando l’applicazione comunica con altri sistemi, usa code, pubblica eventi o deve gestire errori parziali.

# Result pattern

Il **Result pattern** rappresenta l’esito di un’operazione senza usare sempre eccezioni.

Invece di scrivere:

```csharp
throw new InvalidOperationException("Cliente non trovato.");
```

possiamo restituire un risultato esplicito:

```csharp
public class Result
{
    public bool IsSuccess { get; }
    public bool IsFailure => !IsSuccess;
    public string? Error { get; }

    protected Result(bool isSuccess, string? error)
    {
        IsSuccess = isSuccess;
        Error = error;
    }

    public static Result Success()
    {
        return new Result(true, null);
    }

    public static Result Failure(string error)
    {
        return new Result(false, error);
    }
}
```

Versione generica:

```csharp
public class Result<T>
{
    public bool IsSuccess { get; }
    public bool IsFailure => !IsSuccess;
    public T? Value { get; }
    public string? Error { get; }

    private Result(
        bool isSuccess,
        T? value,
        string? error)
    {
        IsSuccess = isSuccess;
        Value = value;
        Error = error;
    }

    public static Result<T> Success(T value)
    {
        return new Result<T>(true, value, null);
    }

    public static Result<T> Failure(string error)
    {
        return new Result<T>(false, default, error);
    }
}
```

---

## Esempio

```csharp
public async Task<Result<ClienteDto>> TrovaClienteAsync(
    int clienteId,
    CancellationToken cancellationToken)
{
    ClienteDto? cliente = await dbContext.Clienti
        .AsNoTracking()
        .Where(c => c.Id == clienteId)
        .Select(c => new ClienteDto(
            c.Id,
            c.Nome,
            c.Email))
        .FirstOrDefaultAsync(cancellationToken);

    if (cliente is null)
    {
        return Result<ClienteDto>.Failure(
            "Cliente non trovato.");
    }

    return Result<ClienteDto>.Success(cliente);
}
```

Uso nell’endpoint:

```csharp
app.MapGet("/api/clienti/{id:int}", async (
    int id,
    ClienteService service,
    CancellationToken cancellationToken) =>
{
    Result<ClienteDto> result =
        await service.TrovaClienteAsync(id, cancellationToken);

    if (result.IsFailure)
    {
        return Results.NotFound(result.Error);
    }

    return Results.Ok(result.Value);
});
```

---

## Quando usare Result pattern

Usalo per errori attesi:

```text
cliente non trovato
validazione fallita
operazione non consentita
stato non valido
regola di business non soddisfatta
```

Non usarlo per bug o errori imprevisti:

```text
NullReferenceException
database non raggiungibile
errore di configurazione
eccezione non prevista
```

Regola pratica:

```text
Result pattern per esiti previsti.
Eccezioni per errori eccezionali o imprevisti.
```

# Options pattern

L’**Options pattern** è il modo tipico in ASP.NET Core per leggere configurazioni fortemente tipizzate.

Invece di accedere direttamente a:

```csharp
configuration["Email:SmtpServer"]
```

creiamo una classe:

```csharp
public class EmailOptions
{
    public const string SectionName = "Email";

    public string SmtpServer { get; set; } = "";
    public int Port { get; set; }
    public string Sender { get; set; } = "";
}
```

`appsettings.json`:

```json
{
  "Email": {
    "SmtpServer": "smtp.example.com",
    "Port": 587,
    "Sender": "noreply@example.com"
  }
}
```

Registrazione:

```csharp
builder.Services.Configure<EmailOptions>(
    builder.Configuration.GetSection(EmailOptions.SectionName));
```

Uso:

```csharp
public class EmailService
{
    private readonly EmailOptions options;

    public EmailService(IOptions<EmailOptions> options)
    {
        this.options = options.Value;
    }

    public Task SendAsync(
        string to,
        string subject,
        string body,
        CancellationToken cancellationToken)
    {
        Console.WriteLine(
            $"Invio email tramite {options.SmtpServer}:{options.Port}");

        return Task.CompletedTask;
    }
}
```

---

## Varianti principali

```text
IOptions<T>
  configurazione letta normalmente, adatta a molti servizi.

IOptionsSnapshot<T>
  utile in ASP.NET Core scoped, rilegge configurazione per request.

IOptionsMonitor<T>
  permette osservazione di cambiamenti runtime.
```

Regola pratica:

```text
Usa Options pattern per configurazioni applicative.
Evita stringhe magiche sparse nel codice.
```

# Repository pattern

Il **Repository pattern** incapsula l’accesso ai dati dietro un’interfaccia orientata al dominio.

Esempio:

```csharp
public interface IOrdineRepository
{
    Task<Ordine?> GetByIdAsync(
        int id,
        CancellationToken cancellationToken);

    Task AddAsync(
        Ordine ordine,
        CancellationToken cancellationToken);
}
```

Implementazione EF Core:

```csharp
public class EfOrdineRepository : IOrdineRepository
{
    private readonly AppDbContext dbContext;

    public EfOrdineRepository(AppDbContext dbContext)
    {
        this.dbContext = dbContext;
    }

    public async Task<Ordine?> GetByIdAsync(
        int id,
        CancellationToken cancellationToken)
    {
        return await dbContext.Ordini
            .Include(o => o.Righe)
            .FirstOrDefaultAsync(o => o.Id == id, cancellationToken);
    }

    public Task AddAsync(
        Ordine ordine,
        CancellationToken cancellationToken)
    {
        dbContext.Ordini.Add(ordine);

        return Task.CompletedTask;
    }
}
```

---

## Quando è utile

Il Repository è utile quando:

```text
hai aggregate di dominio
vuoi isolare EF Core dall’application layer
vuoi testare use case con repository fake
hai query di caricamento complesse
vuoi esprimere metodi di dominio, non CRUD generico
```

Esempio buono:

```csharp
Task<Cliente?> TrovaPerEmailAsync(
    Email email,
    CancellationToken cancellationToken);
```

Esempio meno utile:

```csharp
Task<List<T>> GetAllAsync();
Task<T?> GetByIdAsync(int id);
Task AddAsync(T entity);
Task UpdateAsync(T entity);
Task DeleteAsync(T entity);
```

Il repository generico spesso aggiunge poco sopra EF Core.

# Unit of Work

Il **Unit of Work** rappresenta un’unità di lavoro transazionale.

In EF Core, `DbContext` è già, in parte, un Unit of Work:

```text
traccia modifiche
coordina entity
salva con SaveChangesAsync
gestisce transazione implicita
```

Possiamo però esporre un’interfaccia più pulita:

```csharp
public interface IUnitOfWork
{
    Task SaveChangesAsync(
        CancellationToken cancellationToken);
}
```

Implementazione:

```csharp
public class EfUnitOfWork : IUnitOfWork
{
    private readonly AppDbContext dbContext;

    public EfUnitOfWork(AppDbContext dbContext)
    {
        this.dbContext = dbContext;
    }

    public Task SaveChangesAsync(
        CancellationToken cancellationToken)
    {
        return dbContext.SaveChangesAsync(cancellationToken);
    }
}
```

Uso:

```csharp
public class ConfermaOrdineUseCase
{
    private readonly IOrdineRepository ordineRepository;
    private readonly IUnitOfWork unitOfWork;

    public ConfermaOrdineUseCase(
        IOrdineRepository ordineRepository,
        IUnitOfWork unitOfWork)
    {
        this.ordineRepository = ordineRepository;
        this.unitOfWork = unitOfWork;
    }

    public async Task EseguiAsync(
        int ordineId,
        CancellationToken cancellationToken)
    {
        Ordine ordine = await ordineRepository.GetByIdAsync(
            ordineId,
            cancellationToken)
            ?? throw new NotFoundException("Ordine non trovato.");

        ordine.Conferma();

        await unitOfWork.SaveChangesAsync(cancellationToken);
    }
}
```

---

## Quando serve esplicitarlo

Ha senso esplicitarlo quando:

```text
vuoi nascondere DbContext all’application layer
coordini più repository
vuoi rendere esplicito il commit
usi Clean Architecture
vuoi testare il salvataggio come dipendenza
```

Non serve sempre. In una piccola app, usare direttamente `DbContext.SaveChangesAsync` può essere sufficiente.

# Specification pattern

Lo **Specification pattern** incapsula una regola o un criterio di selezione in un oggetto riutilizzabile.

Esempio semplice:

```csharp
public interface ISpecification<T>
{
    bool IsSatisfiedBy(T item);
}
```

Specifica:

```csharp
public class ClienteAttivoSpecification
    : ISpecification<Cliente>
{
    public bool IsSatisfiedBy(Cliente cliente)
    {
        return cliente.Attivo;
    }
}
```

Uso:

```csharp
if (clienteAttivoSpecification.IsSatisfiedBy(cliente))
{
    // ...
}
```

---

## Specification con Expression

Con EF Core può essere utile rappresentare il criterio come espressione:

```csharp
public interface IQueryableSpecification<T>
{
    Expression<Func<T, bool>> ToExpression();
}
```

Esempio:

```csharp
public class ClientiAttiviSpecification
    : IQueryableSpecification<Cliente>
{
    public Expression<Func<Cliente, bool>> ToExpression()
    {
        return cliente => cliente.Attivo;
    }
}
```

Uso:

```csharp
IQueryableSpecification<Cliente> specification =
    new ClientiAttiviSpecification();

List<Cliente> clienti = await dbContext.Clienti
    .Where(specification.ToExpression())
    .ToListAsync(cancellationToken);
```

---

## Quando usarlo

Usalo quando:

```text
una regola è riutilizzata in più punti
vuoi dare un nome a un criterio complesso
vuoi combinare regole
vuoi evitare duplicazione di condizioni
```

Esempio:

```text
ClienteElegibilePerBuonoSpecification
DocumentoScadutoSpecification
OrdineConfermabileSpecification
```

Non usarlo per condizioni banali usate una sola volta.

# Domain events

I **Domain events** rappresentano qualcosa di significativo accaduto nel dominio.

Esempi:

```text
OrdineConfermato
ClienteRegistrato
DocumentoApprovato
PagamentoAutorizzato
PrenotazioneAnnullata
```

Evento:

```csharp
public record OrdineConfermatoDomainEvent(
    int OrdineId,
    int ClienteId,
    DateTimeOffset Data);
```

Aggregate:

```csharp
public class Ordine
{
    private readonly List<object> domainEvents = [];

    public IReadOnlyCollection<object> DomainEvents =>
        domainEvents.AsReadOnly();

    public int Id { get; private set; }
    public int ClienteId { get; private set; }
    public StatoOrdine Stato { get; private set; }

    public void Conferma(DateTimeOffset data)
    {
        if (Stato != StatoOrdine.Bozza)
        {
            throw new DomainException(
                "Solo un ordine in bozza può essere confermato.");
        }

        Stato = StatoOrdine.Confermato;

        domainEvents.Add(
            new OrdineConfermatoDomainEvent(
                Id,
                ClienteId,
                data));
    }

    public void ClearDomainEvents()
    {
        domainEvents.Clear();
    }
}
```

---

## A cosa servono

I domain events servono a disaccoppiare la regola principale dalle reazioni secondarie.

Esempio:

```text
Ordine.Conferma()
  cambia stato e registra evento OrdineConfermato

Handler evento
  invia email
  aggiorna statistiche
  pubblica integration event
  notifica dashboard
```

Il dominio non deve sapere come inviare email o pubblicare messaggi.

---

# Integration events

Gli **Integration events** sono eventi destinati ad altri sistemi o bounded context.

Esempi:

```text
OrdineConfermatoIntegrationEvent
ClienteRegistratoIntegrationEvent
PagamentoCompletatoIntegrationEvent
```

Differenza:

```text
Domain event
  evento interno al dominio/applicazione.

Integration event
  evento pubblicato verso l’esterno.
```

Esempio:

```csharp
public record OrdineConfermatoIntegrationEvent(
    int OrdineId,
    int ClienteId,
    DateTimeOffset DataConferma);
```

Questo evento può essere pubblicato su:

```text
RabbitMQ
Azure Service Bus
Kafka
Redis Streams
HTTP webhook
```

Regola pratica:

```text
Non esporre necessariamente i domain events verso l’esterno.
Trasformali in integration events espliciti.
```

# Outbox pattern

L’**Outbox pattern** risolve un problema molto importante:

```text
Come salvo dati sul database e pubblico un evento
senza rischiare che una delle due operazioni riesca e l’altra fallisca?
```

Scenario fragile:

```csharp
await dbContext.SaveChangesAsync(cancellationToken);

await messageBus.PublishAsync(
    new OrdineConfermatoIntegrationEvent(...),
    cancellationToken);
```

Problema:

```text
SaveChanges riesce.
Publish fallisce.

Il database dice che l’ordine è confermato,
ma nessun altro sistema riceve l’evento.
```

Oppure:

```text
Publish riesce.
SaveChanges fallisce.

Gli altri sistemi ricevono un evento per un ordine non salvato.
```

---

## Soluzione Outbox

L’idea è salvare l’evento da pubblicare dentro lo stesso database e nella stessa transazione dei dati applicativi.

Tabella:

```text
OutboxMessages
  Id
  Type
  Payload
  CreatedAt
  ProcessedAt
  Error
```

Entity:

```csharp
public class OutboxMessage
{
    public Guid Id { get; set; }
    public string Type { get; set; } = "";
    public string Payload { get; set; } = "";
    public DateTimeOffset CreatedAt { get; set; }
    public DateTimeOffset? ProcessedAt { get; set; }
    public string? Error { get; set; }
}
```

Quando confermo l’ordine:

```csharp
dbContext.Ordini.Update(ordine);

dbContext.OutboxMessages.Add(new OutboxMessage
{
    Id = Guid.NewGuid(),
    Type = nameof(OrdineConfermatoIntegrationEvent),
    Payload = JsonSerializer.Serialize(integrationEvent),
    CreatedAt = clock.Now
});

await dbContext.SaveChangesAsync(cancellationToken);
```

Poi un `BackgroundService` pubblica gli eventi non processati.

```csharp
public class OutboxWorker : BackgroundService
{
    private readonly IServiceScopeFactory scopeFactory;
    private readonly ILogger<OutboxWorker> logger;

    public OutboxWorker(
        IServiceScopeFactory scopeFactory,
        ILogger<OutboxWorker> logger)
    {
        this.scopeFactory = scopeFactory;
        this.logger = logger;
    }

    protected override async Task ExecuteAsync(
        CancellationToken stoppingToken)
    {
        using PeriodicTimer timer =
            new(TimeSpan.FromSeconds(10));

        while (await timer.WaitForNextTickAsync(stoppingToken))
        {
            await ProcessAsync(stoppingToken);
        }
    }

    private async Task ProcessAsync(
        CancellationToken cancellationToken)
    {
        using IServiceScope scope =
            scopeFactory.CreateScope();

        AppDbContext dbContext =
            scope.ServiceProvider.GetRequiredService<AppDbContext>();

        IMessageBus messageBus =
            scope.ServiceProvider.GetRequiredService<IMessageBus>();

        List<OutboxMessage> messages = await dbContext.OutboxMessages
            .Where(m => m.ProcessedAt == null)
            .OrderBy(m => m.CreatedAt)
            .Take(50)
            .ToListAsync(cancellationToken);

        foreach (OutboxMessage message in messages)
        {
            try
            {
                await messageBus.PublishAsync(
                    message.Type,
                    message.Payload,
                    cancellationToken);

                message.ProcessedAt = DateTimeOffset.UtcNow;
                message.Error = null;
            }
            catch (Exception ex)
            {
                message.Error = ex.Message;

                logger.LogError(
                    ex,
                    "Errore pubblicazione outbox message {MessageId}",
                    message.Id);
            }
        }

        await dbContext.SaveChangesAsync(cancellationToken);
    }
}
```

---

## Perché è importante

Outbox evita inconsistenze tra:

```text
database locale
event bus
sistemi esterni
```

Non garantisce che il messaggio venga processato una sola volta da tutto il mondo.
Garantisce che il messaggio da pubblicare non venga perso dopo il commit.

Per questo va abbinato a idempotenza lato consumer.

# Idempotenza

Un’operazione è **idempotente** se può essere eseguita più volte producendo lo stesso risultato finale.

Esempio idempotente:

```http
PUT /api/clienti/10/email
```

con body:

```json
{
  "email": "mario@example.com"
}
```

Ripetere la stessa richiesta lascia l’email uguale.

Esempio non idempotente:

```http
POST /api/pagamenti
```

Se ripetuto, potrebbe addebitare due volte.

---

## Idempotency key

Per operazioni critiche, il client può inviare una chiave:

```http
Idempotency-Key: 9f4e2b9c-0f7d-4c2d-ae2c-1f7b4c0b2f9a
```

Tabella:

```text
IdempotencyRecords
  Key
  RequestHash
  ResponsePayload
  CreatedAt
```

Logica:

```text
se la key non esiste:
  esegui operazione
  salva risultato
  restituisci risultato

se la key esiste:
  restituisci lo stesso risultato precedente
```

Idempotenza è fondamentale con:

```text
retry
pagamenti
messaggi da code
outbox
integration events
API pubbliche
```

# Retry

Il **Retry pattern** riprova un’operazione fallita per errore temporaneo.

Esempi di errori temporanei:

```text
timeout
rete instabile
database momentaneamente occupato
servizio esterno non disponibile per pochi secondi
rate limit temporaneo
```

Esempio concettuale:

```csharp
for (int attempt = 1; attempt <= 3; attempt++)
{
    try
    {
        return await httpClient.GetAsync(url, cancellationToken);
    }
    catch (HttpRequestException) when (attempt < 3)
    {
        await Task.Delay(
            TimeSpan.FromSeconds(attempt),
            cancellationToken);
    }
}
```

In progetti reali si usano librerie come Polly o le resilience pipeline di .NET.

---

## Attenzione al Retry

Retry è pericoloso se l’operazione non è idempotente.

Esempio:

```text
Invio richiesta di pagamento.
Timeout.
Riprovo.
Il primo pagamento era riuscito ma non ho ricevuto risposta.
Secondo addebito.
```

Regola pratica:

```text
Retry solo su errori transitori
e solo quando l’operazione è idempotente
o protetta da idempotency key.
```

# Circuit Breaker

Il **Circuit Breaker pattern** protegge il sistema quando un servizio esterno fallisce ripetutamente.

Senza circuit breaker:

```text
servizio esterno giù
ogni request prova comunque a chiamarlo
tutte aspettano timeout
thread e risorse si saturano
l’app peggiora ulteriormente
```

Con circuit breaker:

```text
dopo troppi errori:
  circuito aperto
  le chiamate falliscono subito o usano fallback

dopo un tempo:
  circuito half-open
  provo poche richieste

se tornano ok:
  circuito chiuso
```

Stati:

```text
Closed
  tutto normale.

Open
  chiamate bloccate.

Half-open
  test controllato per vedere se il servizio è tornato.
```

---

## Quando usarlo

Usalo per:

```text
API esterne
servizi remoti
microservizi
gateway di pagamento
servizi instabili
dipendenze con timeout frequenti
```

Non serve per chiamate locali semplici o codice in memoria.

# Bulkhead

Il **Bulkhead pattern** isola risorse per evitare che un problema in una parte del sistema affondi tutto.

Il nome viene dalle paratie delle navi: se una sezione si allaga, non deve affondare tutta la nave.

Esempio:

```text
le chiamate al servizio report sono lente
non devono bloccare anche login, ordini e pagamenti
```

Tecniche:

```text
limiti di concorrenza
thread pool separati
code separate
semafori
istanze separate
pool di connessioni separati
limiti per endpoint o servizio
```

Esempio con `SemaphoreSlim`:

```csharp
public class ReportClient
{
    private readonly HttpClient httpClient;
    private readonly SemaphoreSlim bulkhead = new(5, 5);

    public ReportClient(HttpClient httpClient)
    {
        this.httpClient = httpClient;
    }

    public async Task<string> GeneraAsync(
        int reportId,
        CancellationToken cancellationToken)
    {
        await bulkhead.WaitAsync(cancellationToken);

        try
        {
            return await httpClient.GetStringAsync(
                $"/api/report/{reportId}",
                cancellationToken);
        }
        finally
        {
            bulkhead.Release();
        }
    }
}
```

Massimo 5 chiamate concorrenti al servizio report.

# Saga

Una **Saga** coordina un processo distribuito composto da più passi, dove ogni passo può richiedere una compensazione se qualcosa fallisce.

Esempio ordine e-commerce:

```text
1. Crea ordine
2. Riserva magazzino
3. Autorizza pagamento
4. Prepara spedizione
```

Se il pagamento fallisce:

```text
devo rilasciare il magazzino
devo annullare l’ordine
```

In sistemi distribuiti non puoi sempre usare una transazione unica tra database diversi.

La Saga gestisce consistenza tramite passi e compensazioni.

---

## Tipi di Saga

```text
Orchestrata
  un orchestratore centrale decide il flusso.

Coreografata
  ogni servizio reagisce a eventi e pubblica altri eventi.
```

### Saga orchestrata

```text
OrderSaga
  invia comando a Inventory
  invia comando a Payment
  invia comando a Shipping
  decide compensazioni
```

### Saga coreografata

```text
OrdineCreato
  Inventory riserva stock e pubblica StockRiservato
  Payment autorizza e pubblica PagamentoAutorizzato
  Shipping prepara spedizione
```

---

## Quando usarla

Usa Saga quando:

```text
un processo attraversa più servizi
non puoi usare una transazione locale unica
servono compensazioni
accetti consistenza eventuale
il workflow è lungo o distribuito
```

Non usarla per una semplice operazione dentro lo stesso database.

# Event sourcing, introduzione

Con **Event Sourcing**, lo stato non viene salvato come ultimo valore corrente.
Viene salvata la sequenza degli eventi che hanno portato a quello stato.

Approccio tradizionale:

```text
Ordine
  Id = 10
  Stato = Spedito
```

Event sourcing:

```text
OrdineCreato
RigaAggiunta
OrdineConfermato
PagamentoRicevuto
OrdineSpedito
```

Lo stato si ricostruisce applicando gli eventi.

---

## Esempio concettuale

Eventi:

```csharp
public record OrdineCreato(int OrdineId, int ClienteId);

public record RigaAggiunta(
    int ProdottoId,
    int Quantita,
    decimal PrezzoUnitario);

public record OrdineConfermato(DateTimeOffset Data);
```

Aggregate:

```csharp
public class Ordine
{
    private readonly List<OrdineRiga> righe = [];

    public int Id { get; private set; }
    public int ClienteId { get; private set; }
    public StatoOrdine Stato { get; private set; }

    public void Apply(OrdineCreato evento)
    {
        Id = evento.OrdineId;
        ClienteId = evento.ClienteId;
        Stato = StatoOrdine.Bozza;
    }

    public void Apply(RigaAggiunta evento)
    {
        righe.Add(new OrdineRiga(
            evento.ProdottoId,
            evento.Quantita,
            evento.PrezzoUnitario));
    }

    public void Apply(OrdineConfermato evento)
    {
        Stato = StatoOrdine.Confermato;
    }
}
```

---

## Vantaggi

```text
audit completo
storico naturale
ricostruzione dello stato
debug temporale
integrazione con eventi
possibilità di proiezioni diverse
```

## Svantaggi

```text
maggiore complessità
versionamento eventi
query più difficili
eventual consistency
curva di apprendimento alta
debug diverso dal CRUD classico
```

Regola pratica:

```text
Event sourcing è potente,
ma non è la scelta predefinita per applicazioni CRUD.
```

# Pattern di resilienza insieme

Retry, Circuit Breaker e Bulkhead spesso lavorano insieme.

Esempio chiamata remota robusta:

```text
Bulkhead
  limita quante chiamate possono partire.

Timeout
  impedisce attese infinite.

Retry
  riprova errori temporanei.

Circuit Breaker
  interrompe chiamate verso servizio evidentemente guasto.

Fallback
  restituisce risposta alternativa, se possibile.
```

Ordine consigliato concettuale:

```text
limitare
chiamare con timeout
riprovare solo se sensato
aprire circuito se il servizio fallisce troppo
```

Attenzione:

```text
retry aggressivi possono peggiorare un outage.
circuit breaker senza fallback può comunque produrre errori.
bulkhead troppo stretto può creare colli di bottiglia.
```

# Errori comuni

## Usare Result pattern per tutto

Sbagliato:

```text
ogni errore tecnico diventa Result.Failure
e le eccezioni spariscono
```

Le eccezioni restano utili per errori imprevisti.

---

## Repository generico sopra EF Core senza valore

Sospetto:

```text
IRepository<T> che replica DbSet<T>
```

Aggiunge astrazione ma non dominio.

---

## Unit of Work duplicato male

Se usi EF Core, ricorda che `DbContext` già traccia modifiche e salva come unità.

Non creare un Unit of Work complesso se non serve.

---

## Specification per condizioni banali

Sbagliato:

```text
ClienteConIdMaggioreDiZeroSpecification
```

se usata una sola volta e senza valore semantico.

---

## Retry senza idempotenza

Pericoloso soprattutto con:

```text
pagamenti
ordini
scritture remote
invio messaggi
creazione risorse
```

---

## Outbox senza consumer idempotente

Outbox può pubblicare più volte lo stesso messaggio in caso di retry o errore dopo publish.

Il consumer deve gestire duplicati.

---

## Saga per operazioni locali

Se tutto avviene nello stesso database e nella stessa transazione, probabilmente non serve una Saga.

---

## Event sourcing per CRUD semplice

Event sourcing introduce complessità.
Va usato quando audit, storico e ricostruzione sono centrali.

---

# Linee guida pratiche

```text
Usa Result pattern
  per errori attesi e regole di business.

Usa Options pattern
  per configurazione tipizzata e validabile.

Usa Repository pattern
  quando protegge aggregate o isola persistenza reale.

Usa Unit of Work
  per rendere esplicito il commit di una transazione applicativa.

Usa Specification
  per regole o criteri complessi e riutilizzabili.

Usa Domain events
  per segnalare eventi interni al dominio.

Usa Integration events
  per comunicare con altri sistemi.

Usa Outbox
  quando devi salvare dati e pubblicare eventi in modo affidabile.

Usa Idempotenza
  quando richieste o messaggi possono essere ripetuti.

Usa Retry
  solo per errori temporanei e operazioni sicure da ripetere.

Usa Circuit Breaker
  per proteggerti da servizi esterni guasti.

Usa Bulkhead
  per isolare risorse e impedire effetti domino.

Usa Saga
  per processi distribuiti con compensazioni.

Usa Event sourcing
  solo quando lo storico degli eventi è parte centrale del dominio.
```

# Riepilogo

In questo capitolo abbiamo collegato i design pattern classici alle architetture moderne.

Abbiamo visto:

```text
Result pattern
Options pattern
Repository pattern
Unit of Work
Specification pattern
Outbox pattern
Saga
Circuit Breaker
Retry
Bulkhead
Idempotenza
Event sourcing
Domain events
Integration events
```

La distinzione più importante è questa:

```text
Pattern applicativi locali
  Result
  Options
  Repository
  Unit of Work
  Specification
  Domain events

Pattern di integrazione e distribuzione
  Integration events
  Outbox
  Idempotenza
  Retry
  Circuit Breaker
  Bulkhead
  Saga
  Event sourcing
```

La regola principale è questa:

```text
I pattern moderni non devono essere introdotti perché “fanno architettura”.
Devono risolvere un rischio reale:
errore atteso, configurazione fragile, persistenza complessa,
evento da non perdere, chiamata remota instabile,
duplicazione di messaggi, consistenza distribuita o workflow lungo.
```

Questo capitolo chiude idealmente la parte sui pattern mostrando una cosa fondamentale: i pattern non sono un catalogo da memorizzare, ma un linguaggio per ragionare sui problemi ricorrenti delle applicazioni reali.

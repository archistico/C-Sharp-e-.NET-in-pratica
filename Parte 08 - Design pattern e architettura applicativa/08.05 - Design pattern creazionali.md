# 08.05 — Design pattern creazionali

Nei capitoli precedenti abbiamo parlato di SOLID, Dependency Injection, Application Layer, Clean Architecture, Hexagonal Architecture, Vertical Slice Architecture e CQRS.

Ora possiamo introdurre i **design pattern creazionali** nel momento giusto.

I pattern creazionali rispondono a una domanda precisa:

```text id="7b2m4s"
Come creo oggetti in modo flessibile,
senza legare troppo il codice a classi concrete,
costruttori complessi o dettagli di configurazione?
```

Il punto centrale è questo:

```text id="xv304v"
I pattern creazionali non servono a evitare sempre new.
Servono quando la creazione di un oggetto contiene logica,
varianti, dipendenze, configurazione o regole che meritano
di essere isolate.
```

---

# Perché esistono i pattern creazionali

Creare un oggetto può sembrare banale:

```csharp id="waw5d9"
Cliente cliente = new Cliente();
```

Ma nelle applicazioni reali spesso la creazione non è così semplice.

Esempi:

```text id="rhjtlh"
devo scegliere una classe concreta in base a una configurazione
devo costruire un oggetto con molti parametri
devo garantire invarianti
devo creare famiglie di oggetti coerenti
devo evitare duplicazione nella costruzione
devo clonare una configurazione esistente
devo controllare il ciclo di vita di un servizio
```

Senza un pattern, il codice tende a riempirsi di:

```text id="tbptow"
new sparsi ovunque
if/switch ripetuti
costruttori enormi
oggetti creati in stato non valido
dipendenze concrete difficili da sostituire
configurazione duplicata
```

---

# Quando non serve un pattern creazionale

Prima regola:

```text id="0657us"
Se new è chiaro, diretto e non crea accoppiamento problematico,
puoi usare new.
```

Esempio perfettamente accettabile:

```csharp id="s9jnad"
ClienteDto dto = new ClienteDto(
    cliente.Id,
    cliente.Nome,
    cliente.Email);
```

Oppure:

```csharp id="izt2gy"
Periodo periodo = new Periodo(dataDa, dataA);
```

Non serve una factory per ogni oggetto.

Sospetto:

```csharp id="javgxs"
public class ClienteDtoFactory
{
    public ClienteDto Create(int id, string nome, string? email)
    {
        return new ClienteDto(id, nome, email);
    }
}
```

Se la factory non aggiunge logica, non aggiunge valore.

---

# Factory Method

Il **Factory Method** incapsula la creazione di un oggetto dietro un metodo.

È utile quando il chiamante non deve conoscere direttamente quale classe concreta creare o quando la creazione contiene regole.

Esempio semplice:

```csharp id="bggpug"
public static class NotificaFactory
{
    public static INotificaSender Crea(TipoNotifica tipo)
    {
        return tipo switch
        {
            TipoNotifica.Email => new EmailNotificaSender(),
            TipoNotifica.Sms => new SmsNotificaSender(),
            TipoNotifica.Push => new PushNotificaSender(),
            _ => throw new ArgumentOutOfRangeException(nameof(tipo))
        };
    }
}
```

Uso:

```csharp id="zq7lsx"
INotificaSender sender =
    NotificaFactory.Crea(TipoNotifica.Email);

await sender.InviaAsync(
    "Messaggio",
    cancellationToken);
```

---

## Problema risolto

Senza factory:

```csharp id="bixfij"
if (tipo == TipoNotifica.Email)
{
    sender = new EmailNotificaSender();
}
else if (tipo == TipoNotifica.Sms)
{
    sender = new SmsNotificaSender();
}
else if (tipo == TipoNotifica.Push)
{
    sender = new PushNotificaSender();
}
```

Questo codice rischia di essere copiato in più punti.

Con factory:

```text id="cnc9oy"
la scelta della classe concreta è centralizzata
il chiamante lavora con un’interfaccia
la creazione è più controllata
```

---

## Factory Method nel dominio

Una factory può anche servire per garantire invarianti.

Esempio:

```csharp id="dfwem4"
public class Ordine
{
    private readonly List<OrdineRiga> righe = [];

    public int ClienteId { get; private set; }
    public StatoOrdine Stato { get; private set; }
    public IReadOnlyCollection<OrdineRiga> Righe => righe.AsReadOnly();

    private Ordine(int clienteId)
    {
        ClienteId = clienteId;
        Stato = StatoOrdine.Bozza;
    }

    public static Ordine CreaPerCliente(int clienteId)
    {
        if (clienteId <= 0)
        {
            throw new DomainException("Cliente non valido.");
        }

        return new Ordine(clienteId);
    }
}
```

Uso:

```csharp id="azxkqk"
Ordine ordine = Ordine.CreaPerCliente(clienteId);
```

Qui il factory method non è una classe separata.
È un metodo statico nel modello di dominio.

Questo è spesso molto pulito.

---

## Quando usare Factory Method

Usalo quando:

```text id="b65098"
la creazione contiene regole
devi scegliere tra implementazioni diverse
vuoi nascondere classi concrete
vuoi dare un nome esplicito alla creazione
il costruttore non basta a esprimere l’intenzione
```

Esempi di nomi buoni:

```csharp id="pbbxty"
Ordine.CreaPerCliente(...)
Documento.CreaBozza(...)
Pagamento.CreaAutorizzazione(...)
Periodo.DaDate(...)
```

# Abstract Factory

L’**Abstract Factory** crea famiglie di oggetti correlati senza specificare le classi concrete.

È utile quando hai più varianti coerenti dello stesso sistema.

Esempio:

```text id="6c49d0"
Tema chiaro
  Button chiaro
  TextBox chiaro
  Dialog chiaro

Tema scuro
  Button scuro
  TextBox scuro
  Dialog scuro
```

Oppure:

```text id="kx779o"
Provider pagamento Stripe
  PaymentClient Stripe
  RefundClient Stripe
  WebhookValidator Stripe

Provider pagamento PayPal
  PaymentClient PayPal
  RefundClient PayPal
  WebhookValidator PayPal
```

---

## Esempio pratico: provider pagamento

Interfacce:

```csharp id="c87wz0"
public interface IPaymentClient
{
    Task<PaymentResult> PayAsync(
        PaymentRequest request,
        CancellationToken cancellationToken);
}

public interface IRefundClient
{
    Task<RefundResult> RefundAsync(
        RefundRequest request,
        CancellationToken cancellationToken);
}

public interface IWebhookValidator
{
    bool IsValid(string payload, string signature);
}
```

Abstract factory:

```csharp id="y438jf"
public interface IPaymentProviderFactory
{
    IPaymentClient CreatePaymentClient();

    IRefundClient CreateRefundClient();

    IWebhookValidator CreateWebhookValidator();
}
```

Implementazione Stripe:

```csharp id="18s6dk"
public class StripePaymentProviderFactory : IPaymentProviderFactory
{
    public IPaymentClient CreatePaymentClient()
    {
        return new StripePaymentClient();
    }

    public IRefundClient CreateRefundClient()
    {
        return new StripeRefundClient();
    }

    public IWebhookValidator CreateWebhookValidator()
    {
        return new StripeWebhookValidator();
    }
}
```

Implementazione PayPal:

```csharp id="l8q7rr"
public class PayPalPaymentProviderFactory : IPaymentProviderFactory
{
    public IPaymentClient CreatePaymentClient()
    {
        return new PayPalPaymentClient();
    }

    public IRefundClient CreateRefundClient()
    {
        return new PayPalRefundClient();
    }

    public IWebhookValidator CreateWebhookValidator()
    {
        return new PayPalWebhookValidator();
    }
}
```

Uso:

```csharp id="ojj37u"
public class PagamentoService
{
    private readonly IPaymentProviderFactory factory;

    public PagamentoService(IPaymentProviderFactory factory)
    {
        this.factory = factory;
    }

    public async Task<PaymentResult> PagaAsync(
        PaymentRequest request,
        CancellationToken cancellationToken)
    {
        IPaymentClient client = factory.CreatePaymentClient();

        return await client.PayAsync(request, cancellationToken);
    }
}
```

---

## Problema risolto

L’Abstract Factory evita di mescolare oggetti di famiglie diverse.

Senza factory potresti creare per errore:

```text id="70xgp3"
StripePaymentClient
PayPalRefundClient
StripeWebhookValidator
```

Con una factory per provider, la famiglia resta coerente.

---

## Quando usare Abstract Factory

Usala quando:

```text id="8twunj"
devi creare famiglie di oggetti correlate
le varianti devono restare coerenti
vuoi cambiare provider/tema/ambiente in blocco
il chiamante non deve conoscere classi concrete
```

Non usarla per creare un solo oggetto semplice.

Se devi creare una sola implementazione, basta spesso Factory Method o DI.

# Builder

Il **Builder** serve a costruire oggetti complessi passo dopo passo.

È utile quando:

```text id="7fcj7n"
ci sono molti parametri
alcuni parametri sono opzionali
la costruzione ha fasi
vuoi rendere il codice più leggibile
vuoi evitare costruttori enormi
```

---

## Problema: costruttore troppo grande

```csharp id="ns3j15"
Report report = new Report(
    titolo,
    dataDa,
    dataA,
    includeGrafici,
    includeDettaglio,
    formato,
    ordinamento,
    lingua,
    mostraTotali,
    mostraNote);
```

Il codice è fragile.

Non è immediato capire cosa significhi ogni parametro.

---

## Builder semplice

```csharp id="0tspzk"
public class ReportOptions
{
    public string Titolo { get; init; } = "";
    public DateOnly DataDa { get; init; }
    public DateOnly DataA { get; init; }
    public bool IncludeGrafici { get; init; }
    public bool IncludeDettaglio { get; init; }
    public string Formato { get; init; } = "PDF";
    public string Lingua { get; init; } = "it";
}
```

Builder:

```csharp id="440esq"
public class ReportOptionsBuilder
{
    private string titolo = "";
    private DateOnly dataDa;
    private DateOnly dataA;
    private bool includeGrafici;
    private bool includeDettaglio;
    private string formato = "PDF";
    private string lingua = "it";

    public ReportOptionsBuilder ConTitolo(string value)
    {
        titolo = value;
        return this;
    }

    public ReportOptionsBuilder PerPeriodo(
        DateOnly da,
        DateOnly a)
    {
        dataDa = da;
        dataA = a;
        return this;
    }

    public ReportOptionsBuilder ConGrafici()
    {
        includeGrafici = true;
        return this;
    }

    public ReportOptionsBuilder ConDettaglio()
    {
        includeDettaglio = true;
        return this;
    }

    public ReportOptionsBuilder InFormato(string value)
    {
        formato = value;
        return this;
    }

    public ReportOptions Build()
    {
        if (string.IsNullOrWhiteSpace(titolo))
        {
            throw new InvalidOperationException(
                "Il titolo è obbligatorio.");
        }

        if (dataA < dataDa)
        {
            throw new InvalidOperationException(
                "Periodo non valido.");
        }

        return new ReportOptions
        {
            Titolo = titolo,
            DataDa = dataDa,
            DataA = dataA,
            IncludeGrafici = includeGrafici,
            IncludeDettaglio = includeDettaglio,
            Formato = formato,
            Lingua = lingua
        };
    }
}
```

Uso:

```csharp id="c5p6po"
ReportOptions options = new ReportOptionsBuilder()
    .ConTitolo("Report vendite")
    .PerPeriodo(
        new DateOnly(2026, 1, 1),
        new DateOnly(2026, 1, 31))
    .ConGrafici()
    .ConDettaglio()
    .InFormato("PDF")
    .Build();
```

Il codice ora è più leggibile.

## Builder nei test

Il Builder è utilissimo nei test per creare oggetti con default sensati.

```csharp id="0esad3"
public class ClienteTestBuilder
{
    private string nome = "Mario Rossi";
    private string? email = "mario@example.com";
    private bool attivo = true;

    public ClienteTestBuilder ConNome(string value)
    {
        nome = value;
        return this;
    }

    public ClienteTestBuilder ConEmail(string? value)
    {
        email = value;
        return this;
    }

    public ClienteTestBuilder Disattivo()
    {
        attivo = false;
        return this;
    }

    public Cliente Build()
    {
        Cliente cliente = new Cliente(nome, email);

        if (!attivo)
        {
            cliente.Disattiva();
        }

        return cliente;
    }
}
```

Uso:

```csharp id="xihc2n"
Cliente cliente = new ClienteTestBuilder()
    .ConEmail(null)
    .Build();
```

Evita test pieni di dettagli irrilevanti.

---

## Quando usare Builder

Usalo quando:

```text id="nzm5ld"
l’oggetto è complesso da costruire
hai molti parametri opzionali
vuoi una costruzione fluente
vuoi migliorare i test
la creazione richiede validazione finale
```

Non usarlo quando il costruttore è semplice:

```csharp id="5bupte"
new Cliente(nome, email)
```

va benissimo.

# Prototype

Il **Prototype** crea nuovi oggetti copiando un oggetto esistente.

È utile quando:

```text id="duwrve"
creare da zero è costoso
vuoi partire da un modello base
vuoi duplicare una configurazione
vuoi modificare solo alcune proprietà
```

In C# moderno, spesso il pattern Prototype si esprime bene con i `record` e l’espressione `with`.

---

## Prototype con record

```csharp id="2s0pi2"
public record ReportTemplate(
    string Titolo,
    string Formato,
    bool IncludeGrafici,
    bool IncludeDettaglio);
```

Template base:

```csharp id="gb7w99"
ReportTemplate baseTemplate = new(
    Titolo: "Report standard",
    Formato: "PDF",
    IncludeGrafici: true,
    IncludeDettaglio: false);
```

Nuova variante:

```csharp id="j3wm7b"
ReportTemplate dettaglioTemplate = baseTemplate with
{
    Titolo = "Report dettagliato",
    IncludeDettaglio = true
};
```

Qui non costruiamo tutto da zero.
Partiamo da un prototipo.

---

## Shallow copy e deep copy

Attenzione: copiare un oggetto può essere superficiale o profondo.

### Shallow copy

Copia l’oggetto principale, ma i riferimenti interni restano condivisi.

Esempio:

```csharp id="xlztj5"
public record DocumentoTemplate(
    string Nome,
    List<string> Sezioni);
```

Uso:

```csharp id="by1asv"
DocumentoTemplate originale = new(
    "Contratto",
    ["Intro", "Privacy"]);

DocumentoTemplate copia = originale with
{
    Nome = "Contratto cliente"
};

copia.Sezioni.Add("Firma");
```

Problema:

```text id="rxgwcd"
originale.Sezioni e copia.Sezioni puntano alla stessa lista
```

---

### Deep copy

Copia anche gli oggetti interni.

```csharp id="fnqoxc"
DocumentoTemplate copiaSicura = originale with
{
    Nome = "Contratto cliente",
    Sezioni = originale.Sezioni.ToList()
};
```

Regola pratica:

```text id="ifpy8d"
Prototype è comodo,
ma devi sapere se stai facendo shallow copy o deep copy.
```

---

## Quando usare Prototype

Usalo quando:

```text id="uwpfz9"
hai template o configurazioni di partenza
vuoi creare varianti
gli oggetti sono immutabili o quasi
il costo di costruzione completa è alto
```

In C# moderno, spesso:

```text id="gxypd7"
record + with
```

copre molti casi semplici.

# Singleton

Il **Singleton** garantisce che esista una sola istanza di una classe e fornisce un punto di accesso globale.

Esempio classico:

```csharp id="d67hx9"
public sealed class AppSettingsSingleton
{
    private static readonly Lazy<AppSettingsSingleton> instance =
        new(() => new AppSettingsSingleton());

    public static AppSettingsSingleton Instance => instance.Value;

    private AppSettingsSingleton()
    {
    }

    public string ApplicationName { get; set; } = "Demo";
}
```

Uso:

```csharp id="2lgogu"
string name = AppSettingsSingleton.Instance.ApplicationName;
```

Questo è il Singleton classico.

Ma nelle applicazioni .NET moderne bisogna essere molto cauti.

---

# Singleton e DI

In .NET moderno, spesso non si implementa il Singleton manualmente.

Si registra un servizio con lifetime Singleton nel container DI.

```csharp id="7dvtgz"
builder.Services.AddSingleton<IClock, SystemClock>();
```

Implementazione:

```csharp id="nm9mjd"
public interface IClock
{
    DateTimeOffset Now { get; }
}

public class SystemClock : IClock
{
    public DateTimeOffset Now => DateTimeOffset.Now;
}
```

Uso:

```csharp id="p99139"
public class OrdineService
{
    private readonly IClock clock;

    public OrdineService(IClock clock)
    {
        this.clock = clock;
    }
}
```

Qui il container garantisce una sola istanza di `SystemClock`.

Vantaggi rispetto al Singleton statico:

```text id="vc6jqi"
dipendenze esplicite
testabilità migliore
lifetime gestito dal container
meno stato globale
possibilità di sostituzione nei test
```

---

# Singleton statico vs Singleton DI

```text id="cfgw3i"
Singleton statico
  accesso globale tramite Instance
  difficile da sostituire nei test
  può nascondere dipendenze
  rischia stato globale

Singleton DI
  accesso tramite costruttore
  dipendenza esplicita
  gestito dal container
  più testabile
```

Regola pratica:

```text id="pa0yjx"
In applicazioni .NET moderne,
preferisci Singleton tramite Dependency Injection
al Singleton statico classico.
```

---

# Quando un Singleton è utile

Un Singleton può essere adatto per servizi:

```text id="1lc4tp"
stateless
thread-safe
costosi da creare
condivisibili tra request
che non contengono dati utente
che non dipendono da servizi scoped
```

Esempi:

```text id="1hpah1"
cache globale thread-safe
servizio di configurazione immutabile
provider di clock
serializer configurato
client factory
servizio di mapping stateless
registry di metadati
```

# Quando un Singleton è pericoloso

Un Singleton è pericoloso quando contiene stato modificabile non protetto.

Esempio:

```csharp id="x3tg4n"
public class CurrentUserState
{
    public int UserId { get; set; }
}
```

Registrazione sbagliata:

```csharp id="o4hshr"
builder.Services.AddSingleton<CurrentUserState>();
```

Problema:

```text id="gbb0qo"
tutti gli utenti condividono la stessa istanza
```

In un’app web, questo può causare bug gravissimi.

---

## Singleton con dipendenze scoped

Errore comune:

```csharp id="lyd2m4"
public class ReportCache
{
    private readonly AppDbContext dbContext;

    public ReportCache(AppDbContext dbContext)
    {
        this.dbContext = dbContext;
    }
}
```

Registrazione:

```csharp id="irtg4r"
builder.Services.AddSingleton<ReportCache>();
```

`AppDbContext` normalmente è scoped.
Un singleton non dovrebbe dipendere direttamente da un servizio scoped.

Problemi:

```text id="yp6ydr"
lifetime sbagliato
DbContext riusato oltre lo scope
concorrenza non sicura
eccezioni runtime dal container
dati incoerenti
```

Se un singleton deve usare servizi scoped, deve creare uno scope esplicitamente tramite `IServiceScopeFactory`, ma va fatto con attenzione.

---

## Singleton e thread safety

Un singleton in ASP.NET Core può essere usato da più request contemporaneamente.

Quindi deve essere thread-safe.

Sbagliato:

```csharp id="t2iygk"
public class ContatoreVisite
{
    public int Count { get; private set; }

    public void Incrementa()
    {
        Count++;
    }
}
```

Se registrato singleton:

```csharp id="ri08f9"
builder.Services.AddSingleton<ContatoreVisite>();
```

`Count++` non è thread-safe.

Meglio:

```csharp id="jtzrgl"
public class ContatoreVisite
{
    private int count;

    public int Count => count;

    public void Incrementa()
    {
        Interlocked.Increment(ref count);
    }
}
```

Oppure evitare singleton con stato modificabile.

# Esempi pratici in .NET

## Factory con DI

Spesso una factory deve creare servizi già registrati nel container.

Esempio: scelta di strategia pagamento.

```csharp id="lb4lfb"
public interface IPaymentStrategy
{
    string Provider { get; }

    Task PagaAsync(
        PaymentRequest request,
        CancellationToken cancellationToken);
}
```

Implementazioni:

```csharp id="eww3q4"
public class StripePaymentStrategy : IPaymentStrategy
{
    public string Provider => "Stripe";

    public Task PagaAsync(
        PaymentRequest request,
        CancellationToken cancellationToken)
    {
        return Task.CompletedTask;
    }
}
```

```csharp id="t3lrdo"
public class PayPalPaymentStrategy : IPaymentStrategy
{
    public string Provider => "PayPal";

    public Task PagaAsync(
        PaymentRequest request,
        CancellationToken cancellationToken)
    {
        return Task.CompletedTask;
    }
}
```

Factory:

```csharp id="3aqubi"
public class PaymentStrategyFactory
{
    private readonly IEnumerable<IPaymentStrategy> strategies;

    public PaymentStrategyFactory(
        IEnumerable<IPaymentStrategy> strategies)
    {
        this.strategies = strategies;
    }

    public IPaymentStrategy GetByProvider(string provider)
    {
        return strategies.FirstOrDefault(s =>
            string.Equals(
                s.Provider,
                provider,
                StringComparison.OrdinalIgnoreCase))
            ?? throw new InvalidOperationException(
                $"Provider pagamento non supportato: {provider}");
    }
}
```

Registrazione:

```csharp id="b6labv"
builder.Services.AddScoped<IPaymentStrategy, StripePaymentStrategy>();
builder.Services.AddScoped<IPaymentStrategy, PayPalPaymentStrategy>();
builder.Services.AddScoped<PaymentStrategyFactory>();
```

Uso:

```csharp id="0xgxlw"
public class PagamentoService
{
    private readonly PaymentStrategyFactory factory;

    public PagamentoService(PaymentStrategyFactory factory)
    {
        this.factory = factory;
    }

    public async Task PagaAsync(
        string provider,
        PaymentRequest request,
        CancellationToken cancellationToken)
    {
        IPaymentStrategy strategy =
            factory.GetByProvider(provider);

        await strategy.PagaAsync(request, cancellationToken);
    }
}
```

Qui la factory non usa `new`.
Seleziona oggetti creati dal container DI.

Esempio tipico:

```csharp id="gdb876"
builder.Services.AddAuthentication()
```

Oppure:

```csharp id="ocaozu"
builder.Services.AddCors(options =>

```

Questo stile builder è leggibile perché costruisce configurazioni complesse passo dopo passo.

## Singleton per servizi stateless

```csharp id="4pwcyz"
builder.Services.AddSingleton<IClock, SystemClock>();
```

Oppure:

```csharp id="tmz8o0"
builder.Services.AddSingleton<SlugGenerator>();
```

Se `SlugGenerator` è stateless:

```csharp id="36d96i"
public class SlugGenerator
{
    public string Generate(string text)
    {
        return text
            .Trim()
            .ToLowerInvariant()
            .Replace(" ", "-");
    }
}
```

può essere singleton.

## Factory per oggetti di dominio

```csharp id="c6xub8"
public class DocumentoFactory
{
    private readonly IClock clock;

    public DocumentoFactory(IClock clock)
    {
        this.clock = clock;
    }

    public Documento CreaBozza(
        int clienteId,
        TipoDocumento tipo)
    {
        return Documento.CreaBozza(
            clienteId,
            tipo,
            clock.Now);
    }
}
```

Qui la factory serve perché la creazione richiede un servizio esterno al dominio: il tempo corrente.

Il dominio resta pulito:

```csharp id="gvjy32"
public static Documento CreaBozza(
    int clienteId,
    TipoDocumento tipo,
    DateTimeOffset dataCreazione)
{
    // regole di dominio
}
```

# Pattern creazionali e Dependency Injection

La DI container è già una grande factory.

Quando scrivi:

```csharp id="nwz3ad"
public ClienteService(
    IClienteRepository repository,
    ILogger<ClienteService> logger)
{
}
```

non sei tu a creare `ClienteService`.

Lo crea il container.

Quindi, prima di introdurre factory manuali, chiediti:

```text id="p9b57u"
Il container DI risolve già questo problema?
```

---

## Quando basta DI

Basta DI quando:

```text id="22j1jd"
la classe concreta è nota a startup
le dipendenze sono stabili
non devi scegliere a runtime tra molte varianti
non devi creare oggetti con parametri dinamici complessi
```

Esempio:

```csharp id="9abtw8"
builder.Services.AddScoped<IClienteRepository, EfClienteRepository>();
builder.Services.AddScoped<ClienteService>();
```

---

## Quando serve una factory oltre DI

Serve una factory quando:

```text id="4655fb"
devi scegliere implementazione a runtime
devi passare parametri dinamici
devi creare oggetti di dominio con regole
devi nascondere costruzione complessa
devi creare famiglie coerenti di oggetti
```

Esempio:

```csharp id="zt34eq"
IPaymentStrategy strategy =
    factory.GetByProvider(providerName);
```

---

# Anti-pattern: Service Locator

Attenzione a non trasformare la factory in un Service Locator.

Sospetto:

```csharp id="el6nx2"
public class OrdineService
{
    private readonly IServiceProvider serviceProvider;

    public OrdineService(IServiceProvider serviceProvider)
    {
        this.serviceProvider = serviceProvider;
    }

    public async Task EseguiAsync()
    {
        var repository =
            serviceProvider.GetRequiredService<IOrdineRepository>();

        var email =
            serviceProvider.GetRequiredService<IEmailService>();
    }
}
```

Problemi:

```text id="tawts3"
dipendenze nascoste
test più difficili
errori scoperti a runtime
classe meno leggibile
```

Meglio:

```csharp id="x17vgz"
public class OrdineService
{
    private readonly IOrdineRepository repository;
    private readonly IEmailService email;

    public OrdineService(
        IOrdineRepository repository,
        IEmailService email)
    {
        this.repository = repository;
        this.email = email;
    }
}
```

Usa `IServiceProvider` direttamente solo in casi specifici, come factory infrastrutturali o creazione di scope controllati.

---

# Confronto rapido

```text id="40vj8t"
Factory Method
  crea un oggetto scegliendo o nominando la creazione.

Abstract Factory
  crea famiglie di oggetti correlati.

Builder
  costruisce oggetti complessi passo dopo passo.

Prototype
  crea nuovi oggetti copiando un prototipo esistente.

Singleton
  garantisce una sola istanza, meglio tramite DI in .NET moderno.
```

# Esempio unico: sistema report

Immaginiamo un sistema di generazione report.

## Factory Method

Scegli il generatore in base al formato:

```csharp id="6agrjl"
public interface IReportGenerator
{
    Task<byte[]> GenerateAsync(
        ReportOptions options,
        CancellationToken cancellationToken);
}

public class ReportGeneratorFactory
{
    private readonly IEnumerable<IReportGenerator> generators;

    public ReportGeneratorFactory(
        IEnumerable<IReportGenerator> generators)
    {
        this.generators = generators;
    }

    public IReportGenerator Get(string formato)
    {
        return generators.FirstOrDefault(g =>
            g.GetType().Name.StartsWith(
                formato,
                StringComparison.OrdinalIgnoreCase))
            ?? throw new InvalidOperationException(
                $"Formato non supportato: {formato}");
    }
}
```

## Builder

Costruisci le opzioni:

```csharp id="q6p3so"
ReportOptions options = new ReportOptionsBuilder()
    .ConTitolo("Report mensile")
    .PerPeriodo(dataDa, dataA)
    .ConGrafici()
    .InFormato("PDF")
    .Build();
```

## Prototype

Parti da un template:

```csharp id="2tfdk2"
ReportTemplate templateMensile = templateBase with
{
    Titolo = "Report mensile",
    IncludeGrafici = true
};
```

## Singleton DI

Registra servizi stateless:

```csharp id="ov5pz9"
builder.Services.AddSingleton<ReportTemplateCatalog>();
```

Questo esempio mostra che i pattern possono collaborare.

# Errori comuni

## Creare factory inutili

Sbagliato:

```csharp id="2cwjo3"
public class ClienteFactory
{
    public Cliente Create(string nome)
    {
        return new Cliente(nome);
    }
}
```

se non aggiunge regole o scelta.

---

## Builder per oggetti semplici

Sbagliato:

```csharp id="7vnfu0"
new ClienteBuilder()
    .ConNome(nome)
    .Build();
```

quando basta:

```csharp id="ru5npj"
new Cliente(nome);
```

---

## Singleton con stato utente

Pericoloso:

```csharp id="w2tzl0"
builder.Services.AddSingleton<CarrelloUtente>();
```

In web app, utenti diversi condividerebbero lo stesso carrello.

---

## Singleton non thread-safe

Pericoloso:

```csharp id="47yw83"
public List<string> Messaggi { get; } = [];
```

in un singleton usato da più request.

---

## Abstract Factory troppo presto

Se hai una sola famiglia concreta, probabilmente non ti serve ancora Abstract Factory.

---

## Prototype con copie superficiali involontarie

Attenzione a liste, dizionari e oggetti mutabili interni.

---

## Usare il container come Service Locator

Sbagliato nella logica applicativa ordinaria:

```csharp id="u9pb9x"
serviceProvider.GetRequiredService<...>()
```

Meglio dipendenze esplicite nel costruttore.

# Linee guida pratiche

```text id="06mmbo"
Usa new
  quando la creazione è semplice e chiara.

Usa Factory Method
  quando la creazione ha un nome, una regola o una scelta.

Usa Abstract Factory
  quando devi creare famiglie coerenti di oggetti.

Usa Builder
  quando l’oggetto ha costruzione complessa o molti parametri opzionali.

Usa Prototype
  quando parti da template o configurazioni esistenti.

Usa Singleton tramite DI
  invece del Singleton statico classico.

Evita Singleton con stato utente
  soprattutto nelle web app.

Assicurati che i Singleton siano thread-safe
  o stateless.

Non creare interfacce/factory per ogni classe
  solo per “applicare pattern”.

Controlla sempre se DI risolve già il problema
  prima di introdurre una factory manuale.
```

# Riepilogo

In questo capitolo abbiamo introdotto i principali design pattern creazionali.

Abbiamo visto:

```text id="3k8dgn"
Factory Method
Abstract Factory
Builder
Prototype
Singleton
Singleton e DI
quando un Singleton è pericoloso
esempi pratici in .NET
```

La distinzione più importante è questa:

```text id="cow3cx"
Factory Method
  sceglie o nomina la creazione di un oggetto.

Abstract Factory
  crea famiglie coerenti di oggetti.

Builder
  costruisce oggetti complessi passo dopo passo.

Prototype
  crea varianti copiando un modello esistente.

Singleton
  controlla il ciclo di vita di una singola istanza.
```

La regola principale è questa:

```text id="zpopis"
I pattern creazionali servono quando la creazione è parte del problema.
Se l’oggetto è semplice da creare, un costruttore chiaro è spesso
la soluzione migliore.
```

Nel mondo .NET moderno, la Dependency Injection riduce molti usi classici di factory e singleton, ma non li elimina. Le factory restano utili per scelte runtime, costruzioni di dominio, provider multipli e oggetti complessi; i singleton restano utili per servizi stateless e thread-safe, ma diventano pericolosi quando nascondono stato globale o dati utente.

# 08.04 — Vertical Slice Architecture e CQRS

Nei capitoli precedenti abbiamo visto layered architecture, DDD, Application Layer, Clean Architecture, Hexagonal Architecture e Ports & Adapters.

Ora affrontiamo un approccio molto utile nelle applicazioni ASP.NET Core moderne: **Vertical Slice Architecture**, spesso usata insieme a **CQRS** e, in molti progetti .NET, a **MediatR**.

Il punto centrale è questo:

```text id="y0r3eo"
Invece di organizzare il codice prima per layer tecnici,
Vertical Slice Architecture lo organizza per funzionalità.
Ogni feature contiene tutto ciò che serve per realizzare uno specifico caso d’uso.
```

Questo approccio riduce il rischio di avere controller sottili ma service enormi, repository generici e cartelle tecniche piene di classi scollegate tra loro.

---

# Il problema dell’organizzazione per layer

Una struttura classica per layer può essere così:

```text id="l1sy9h"
Controllers
  ClientiController.cs
  OrdiniController.cs

Services
  ClienteService.cs
  OrdineService.cs

Repositories
  ClienteRepository.cs
  OrdineRepository.cs

DTOs
  CreaClienteRequest.cs
  ClienteDto.cs
  CreaOrdineRequest.cs
  OrdineDto.cs

Validators
  CreaClienteValidator.cs
  CreaOrdineValidator.cs
```

Questa organizzazione è comprensibile, ma quando l’app cresce ogni modifica a una funzionalità costringe a saltare tra molte cartelle.

Esempio: modificare “crea ordine”.

```text id="22mrbi"
Controllers/OrdiniController.cs
Services/OrdineService.cs
Repositories/OrdineRepository.cs
DTOs/CreaOrdineRequest.cs
DTOs/CreaOrdineResult.cs
Validators/CreaOrdineValidator.cs
Mappings/OrdineMapping.cs
```

La funzionalità è distribuita orizzontalmente.

Vertical Slice Architecture propone l’opposto.

---

# Organizzazione per feature

In Vertical Slice Architecture, il codice viene organizzato per **feature** o **caso d’uso**.

Esempio:

```text id="l56ziv"
Features
  Ordini
    CreaOrdine
      CreaOrdineEndpoint.cs
      CreaOrdineCommand.cs
      CreaOrdineHandler.cs
      CreaOrdineValidator.cs
      CreaOrdineResult.cs

    ConfermaOrdine
      ConfermaOrdineEndpoint.cs
      ConfermaOrdineCommand.cs
      ConfermaOrdineHandler.cs
      ConfermaOrdineValidator.cs

    ListaOrdini
      ListaOrdiniEndpoint.cs
      ListaOrdiniQuery.cs
      ListaOrdiniHandler.cs
      OrdineListItemDto.cs
```

Ogni slice contiene ciò che serve a una funzionalità.

La domanda non è più:

```text id="u4859r"
In quale layer tecnico metto questa classe?
```

ma:

```text id="9yw6ds"
A quale feature appartiene questa classe?
```

---

# Che cos’è una vertical slice

Una **vertical slice** è un taglio verticale attraverso l’applicazione.

Contiene, per una specifica funzionalità:

```text id="5kpxcd"
endpoint o controller action
request/command/query
validazione
handler
mapping
accesso ai dati necessario
response/result
test
```

Esempio:

```text id="xq6ch9"
Feature: CreaOrdine

HTTP endpoint
  ↓
Command
  ↓
Validator
  ↓
Handler
  ↓
Domain / DbContext / Repository
  ↓
Result
```

La slice attraversa i layer, ma resta confinata a un caso d’uso.

---

# Layered Architecture vs Vertical Slice Architecture

Layered Architecture:

```text id="nkmhd6"
Presentation
Application
Domain
Infrastructure
```

Vertical Slice Architecture:

```text id="j02u60"
CreaOrdine
ConfermaOrdine
AnnullaOrdine
ListaOrdini
DettaglioOrdine
```

Non sono per forza nemiche.

Puoi avere:

```text id="3f9wu9"
Domain condiviso
Infrastructure condivisa
Features organizzate per vertical slice
```

Esempio:

```text id="70xsg7"
Domain
  Ordine.cs
  Cliente.cs

Infrastructure
  AppDbContext.cs

Features
  Ordini
    CreaOrdine
    ConfermaOrdine
    ListaOrdini
```

Regola pratica:

```text id="doa9ri"
Layered Architecture organizza per responsabilità tecniche.
Vertical Slice Architecture organizza per funzionalità.
```

# Command

Un **Command** rappresenta un’operazione che modifica lo stato del sistema.

Esempi:

```text id="evgk7g"
CreaOrdine
ConfermaOrdine
AnnullaPrenotazione
RegistraCliente
AggiornaEmailCliente
ApprovaDocumento
```

Caratteristiche:

```text id="6u2fsu"
ha intenzione imperativa
può modificare dati
può fallire per regole di business
può generare eventi
di solito restituisce un risultato essenziale
```

Esempio:

```csharp id="fvojqf"
public record CreaOrdineCommand(
    int ClienteId,
    List<CreaOrdineRigaCommand> Righe);

public record CreaOrdineRigaCommand(
    int ProdottoId,
    int Quantita,
    decimal PrezzoUnitario);
```

Il nome è importante: un command dovrebbe essere espresso come azione.

```text id="52if84"
CreaOrdineCommand
ConfermaOrdineCommand
AnnullaOrdineCommand
```

Non:

```text id="xmlve9"
OrdineDto
OrdineModel
OrdineRequestGenerica
```

# Query

Una **Query** rappresenta un’operazione di lettura.

Esempi:

```text id="i8rkvw"
ListaOrdini
DettaglioOrdine
CercaClienti
ReportPresenze
ElencoDocumenti
```

Caratteristiche:

```text id="zpjrjt"
non modifica stato
restituisce dati
può essere ottimizzata per la lettura
può proiettare direttamente a DTO
può usare query specifiche
```

Esempio:

```csharp id="3r0qj4"
public record ListaOrdiniQuery(
    int ClienteId,
    int Page,
    int PageSize);
```

Risultato:

```csharp id="2c8c4m"
public record OrdineListItemDto(
    int Id,
    DateTime Data,
    string Stato,
    decimal Totale);
```

---

# Command vs Query

La distinzione è semplice ma potente.

```text id="8hct94"
Command
  cambia lo stato.

Query
  legge lo stato.
```

Esempi:

```text id="ryfeac"
CreaCliente
  Command

AggiornaEmailCliente
  Command

CercaClienti
  Query

DettaglioCliente
  Query
```

Regola pratica:

```text id="53gtks"
Non nascondere modifiche dentro una query.
Non usare un command solo per leggere dati.
```

# Handler

Un **Handler** contiene la logica applicativa del command o della query.

Esempio command handler:

```csharp id="48z8ta"
public class CreaOrdineHandler
{
    private readonly AppDbContext dbContext;

    public CreaOrdineHandler(AppDbContext dbContext)
    {
        this.dbContext = dbContext;
    }

    public async Task<CreaOrdineResult> HandleAsync(
        CreaOrdineCommand command,
        CancellationToken cancellationToken)
    {
        Ordine ordine = Ordine.Crea(command.ClienteId);

        foreach (CreaOrdineRigaCommand riga in command.Righe)
        {
            ordine.AggiungiRiga(
                riga.ProdottoId,
                riga.Quantita,
                riga.PrezzoUnitario);
        }

        ordine.Conferma();

        dbContext.Ordini.Add(ordine);

        await dbContext.SaveChangesAsync(cancellationToken);

        return new CreaOrdineResult(ordine.Id);
    }
}
```

Risultato:

```csharp id="q1wp9u"
public record CreaOrdineResult(
    int OrdineId);
```

Qui il handler è lo use case concreto.

# Query handler

Per una query, spesso non serve caricare l’aggregate completo.

```csharp id="cxxf6h"
public class ListaOrdiniHandler
{
    private readonly AppDbContext dbContext;

    public ListaOrdiniHandler(AppDbContext dbContext)
    {
        this.dbContext = dbContext;
    }

    public async Task<List<OrdineListItemDto>> HandleAsync(
        ListaOrdiniQuery query,
        CancellationToken cancellationToken)
    {
        int page = Math.Max(1, query.Page);
        int pageSize = Math.Clamp(query.PageSize, 1, 100);

        return await dbContext.Ordini
            .AsNoTracking()
            .Where(o => o.ClienteId == query.ClienteId)
            .OrderByDescending(o => o.Data)
            .Skip((page - 1) * pageSize)
            .Take(pageSize)
            .Select(o => new OrdineListItemDto(
                o.Id,
                o.Data,
                o.Stato.ToString(),
                o.Righe.Sum(r => r.Quantita * r.PrezzoUnitario)))
            .ToListAsync(cancellationToken);
    }
}
```

Questa è una lettura ottimizzata.

Non serve passare per metodi di dominio se non stiamo modificando invarianti.

# CQRS

CQRS significa:

```text id="sxfdl3"
Command Query Responsibility Segregation
```

L’idea è separare il modello di scrittura dal modello di lettura.

Nella forma più semplice:

```text id="si0fku"
Command
  operazioni che modificano stato.

Query
  operazioni che leggono stato.
```

Non significa obbligatoriamente:

```text id="veqo02"
due database
event sourcing
microservizi
code messaggi
architettura distribuita complessa
```

Può essere molto leggero.

Esempio CQRS leggero:

```text id="1nfcqo"
CreaOrdineCommandHandler
  usa dominio e salva modifiche.

ListaOrdiniQueryHandler
  fa proiezione diretta a DTO.
```

---

# Separazione lettura/scrittura

La separazione è utile perché lettura e scrittura hanno esigenze diverse.

## Scrittura

Quando modifichiamo stato, vogliamo:

```text id="1qr3nf"
proteggere invarianti
validare regole di business
usare aggregate
gestire transazioni
generare eventi
garantire consistenza
```

Esempio:

```text id="3qhx1f"
ConfermaOrdineCommand
  carica Ordine
  chiama ordine.Conferma()
  salva
```

## Lettura

Quando leggiamo dati, vogliamo:

```text id="5q9asx"
velocità
proiezioni leggere
DTO adatti alla UI
paginazione
filtri
ordinamenti
query ottimizzate
```

Esempio:

```text id="k84ed1"
ListaOrdiniQuery
  seleziona solo colonne necessarie
  usa AsNoTracking
  proietta a OrdineListItemDto
```

---

# CQRS leggero vs CQRS pesante

## CQRS leggero

```text id="wsq41c"
stesso database
stesso progetto
handler separati
DTO separati
query ottimizzate
command con dominio
```

È quello più adatto per iniziare.

## CQRS pesante

```text id="rfvq5q"
database separato per lettura e scrittura
sincronizzazione asincrona
event bus
eventual consistency
event sourcing
proiezioni dedicate
```

È utile solo in sistemi complessi.

Regola pratica:

```text id="5f7mzc"
Inizia con CQRS leggero.
Passa a CQRS distribuito solo se hai veri motivi di scala,
complessità o indipendenza dei modelli.
```

# MediatR

**MediatR** è una libreria .NET che implementa il pattern mediator.

Permette di inviare request a handler senza accoppiarli direttamente al chiamante.

Concetto:

```text id="6i66x3"
Endpoint
  invia Command o Query a MediatR

MediatR
  trova l’handler corretto

Handler
  esegue il caso d’uso
```

MediatR espone concetti come `IRequest<TResponse>`, `IRequestHandler<TRequest,TResponse>` e pipeline behavior, utili per implementare command/query handler e logica trasversale come validazione o logging; la registrazione moderna usa `RegisterServicesFromAssembly`. ([GitHub][1])

---

## Installazione

```bash id="ni8iy7"
dotnet add package MediatR
```

Registrazione tipica:

```csharp id="q62b1i"
builder.Services.AddMediatR(configuration =>
{
    configuration.RegisterServicesFromAssembly(
        typeof(Program).Assembly);
});
```

In una soluzione più strutturata, puoi registrare l’assembly Application:

```csharp id="wepv6u"
builder.Services.AddMediatR(configuration =>
{
    configuration.RegisterServicesFromAssembly(
        typeof(CreaOrdineCommand).Assembly);
});
```

# Validazione per use case

In Vertical Slice Architecture, la validazione può stare vicino al command o alla query.

Esempio:

```text id="ie91ok"
Features
  Ordini
    CreaOrdine
      CreaOrdineCommand.cs
      CreaOrdineValidator.cs
      CreaOrdineHandler.cs
```

Validator:

```csharp id="a07gxg"
public class CreaOrdineValidator
{
    public List<string> Validate(CreaOrdineCommand command)
    {
        List<string> errors = [];

        if (command.ClienteId <= 0)
        {
            errors.Add("Cliente non valido.");
        }

        if (command.Righe.Count == 0)
        {
            errors.Add("L'ordine deve contenere almeno una riga.");
        }

        foreach (CreaOrdineRigaCommand riga in command.Righe)
        {
            if (riga.Quantita <= 0)
            {
                errors.Add("La quantità deve essere maggiore di zero.");
            }

            if (riga.PrezzoUnitario < 0)
            {
                errors.Add("Il prezzo non può essere negativo.");
            }
        }

        return errors;
    }
}
```

In progetti reali si usa spesso FluentValidation, ma il concetto non dipende dalla libreria.

```csharp id="yuwj3j"
public async Task<CreaOrdineResult> Handle(
```

# Pipeline behavior

Un **pipeline behavior** è un componente che intercetta una request prima e/o dopo l’handler.

Concettualmente:

```text id="7kl4vq"
Endpoint
  ↓
MediatR
  ↓
LoggingBehavior
  ↓
ValidationBehavior
  ↓
TransactionBehavior
  ↓
Handler
```

I pipeline behavior di MediatR funzionano in modo simile a middleware per le request MediatR: permettono di eseguire logica prima e dopo l’handler, per esempio logging, validazione, performance tracking, gestione transazioni o error handling. ([DEV Community][2])

---

## Esempio ValidationBehavior

Esempio semplificato, senza FluentValidation:

```csharp id="mgfhez"
using MediatR;

public interface IRequestValidator<TRequest>
{
    List<string> Validate(TRequest request);
}
```

Behavior:

```csharp id="psejje"
using MediatR;

public class ValidationBehavior<TRequest, TResponse>
    : IPipelineBehavior<TRequest, TResponse>
    where TRequest : notnull
{
    private readonly IEnumerable<IRequestValidator<TRequest>> validators;

    public ValidationBehavior(
        IEnumerable<IRequestValidator<TRequest>> validators)
    {
        this.validators = validators;
    }

    public async Task<TResponse> Handle(
        TRequest request,
        RequestHandlerDelegate<TResponse> next,
        CancellationToken cancellationToken)
    {
        List<string> errors = validators
            .SelectMany(v => v.Validate(request))
            .ToList();

        if (errors.Count > 0)
        {
            throw new ValidationException(errors);
        }

        return await next();
    }
}
```

Registrazione:

```csharp id="cxh98v"
builder.Services.AddTransient(
    typeof(IPipelineBehavior<,>),
    typeof(ValidationBehavior<,>));
```

Oppure, nelle versioni moderne di MediatR, si usa anche la registrazione tramite configurazione:

```csharp id="x227yw"
builder.Services.AddMediatR(configuration =>
{
    configuration.RegisterServicesFromAssembly(
        typeof(CreaOrdineCommand).Assembly);

    configuration.AddOpenBehavior(
        typeof(ValidationBehavior<,>));
});
```

## Logging behavior

```csharp id="l0jg3a"
using MediatR;

public class LoggingBehavior<TRequest, TResponse>
    : IPipelineBehavior<TRequest, TResponse>
    where TRequest : notnull
{
    private readonly ILogger<LoggingBehavior<TRequest, TResponse>> logger;

    public LoggingBehavior(
        ILogger<LoggingBehavior<TRequest, TResponse>> logger)
    {
        this.logger = logger;
    }

    public async Task<TResponse> Handle(
        TRequest request,
        RequestHandlerDelegate<TResponse> next,
        CancellationToken cancellationToken)
    {
        string requestName = typeof(TRequest).Name;

        logger.LogInformation(
            "Esecuzione request {RequestName}",
            requestName);

        TResponse response = await next();

        logger.LogInformation(
            "Request {RequestName} completata",
            requestName);

        return response;
    }
}
```

## Performance behavior

```csharp id="lpp7hk"
using MediatR;
using System.Diagnostics;

public class PerformanceBehavior<TRequest, TResponse>
    : IPipelineBehavior<TRequest, TResponse>
    where TRequest : notnull
{
    private readonly ILogger<PerformanceBehavior<TRequest, TResponse>> logger;

    public PerformanceBehavior(
        ILogger<PerformanceBehavior<TRequest, TResponse>> logger)
    {
        this.logger = logger;
    }

    public async Task<TResponse> Handle(
        TRequest request,
        RequestHandlerDelegate<TResponse> next,
        CancellationToken cancellationToken)
    {
        Stopwatch stopwatch = Stopwatch.StartNew();

        TResponse response = await next();

        stopwatch.Stop();

        if (stopwatch.ElapsedMilliseconds > 500)
        {
            logger.LogWarning(
                "Request lenta {RequestName}: {Elapsed} ms",
                typeof(TRequest).Name,
                stopwatch.ElapsedMilliseconds);
        }

        return response;
    }
}
```

# Transaction behavior

Per command che modificano stato, può essere utile una pipeline transazionale.

Esempio concettuale:

```csharp id="3oik98"
public interface ICommand<TResponse> : IRequest<TResponse>
{
}
```

Command:

```csharp id="w5vhc2"
public record CreaOrdineCommand(
    int ClienteId,
    List<CreaOrdineRigaCommand> Righe)
    : ICommand<CreaOrdineResult>;
```

Transaction behavior solo per command:

```csharp id="g8nub9"
public class TransactionBehavior<TRequest, TResponse>
    : IPipelineBehavior<TRequest, TResponse>
    where TRequest : ICommand<TResponse>
{
    private readonly AppDbContext dbContext;

    public TransactionBehavior(AppDbContext dbContext)
    {
        this.dbContext = dbContext;
    }

    public async Task<TResponse> Handle(
        TRequest request,
        RequestHandlerDelegate<TResponse> next,
        CancellationToken cancellationToken)
    {
        await using var transaction =
            await dbContext.Database.BeginTransactionAsync(cancellationToken);

        TResponse response = await next();

        await dbContext.SaveChangesAsync(cancellationToken);

        await transaction.CommitAsync(cancellationToken);

        return response;
    }
}
```

Nota: se usi un behavior transazionale, gli handler non dovrebbero chiamare `SaveChangesAsync` direttamente, altrimenti la responsabilità è duplicata.

Regola pratica:

```text id="kcuxse"
O SaveChanges è nello handler,
o è in una pipeline transazionale.
Evita di fare entrambe le cose senza una ragione precisa.
```

# Minimal API e Vertical Slice

Vertical Slice Architecture si sposa bene con Minimal API.

Esempio file unico per feature:

```csharp id="x9yy4u"
public static class CreaOrdineEndpoint
{
    public static RouteGroupBuilder MapCreaOrdineEndpoint(
        this RouteGroupBuilder group)
    {
        group.MapPost("/", async (
            CreaOrdineRequest request,
            ISender sender,
            CancellationToken cancellationToken) =>
        {
            CreaOrdineCommand command = new(
                request.ClienteId,
                request.Righe
                    .Select(r => new CreaOrdineRigaCommand(
                        r.ProdottoId,
                        r.Quantita,
                        r.PrezzoUnitario))
                    .ToList());

            CreaOrdineResult result =
                await sender.Send(command, cancellationToken);

            return Results.Created(
                $"/api/ordini/{result.OrdineId}",
                result);
        });

        return group;
    }
}
```

Program.cs:

```csharp id="d8eerr"
RouteGroupBuilder ordini = app.MapGroup("/api/ordini");

ordini.MapCreaOrdineEndpoint();
ordini.MapListaOrdiniEndpoint();
ordini.MapConfermaOrdineEndpoint();
```

Questo evita un `Program.cs` enorme.

I filtri delle Minimal API possono eseguire logica prima e dopo l’endpoint, ispezionare o modificare i parametri e intercettare la risposta; sono quindi un’alternativa o complemento ai behavior quando la logica è specifica del livello HTTP. ([Microsoft Learn][3])

# VSA senza MediatR

Vertical Slice Architecture non richiede per forza MediatR.

Puoi chiamare direttamente l’handler.

```csharp id="ey4c8y"

    CreaOrdineRequest request,

    CreaOrdineCommand command = request.ToCommand();

    CreaOrdineResult result =
        await handler.HandleAsync(command, cancellationToken);

    return Results.Created(
        $"/api/ordini/{result.OrdineId}",
```

```csharp id="jsoxc0"
public class CreaOrdineHandler

        // ...
```

```text id="uvb995"
meno dipendenze
più esplicito
debug più diretto
meno magia
```

Svantaggi:

```text id="u2eqe6"
meno pipeline standard

endpoint accoppiato all’handler concreto
```

## Command

```csharp id="jh9tdy"
using MediatR;

public record CreaClienteCommand(
    string Nome,
    string? Email)
    : IRequest<CreaClienteResult>;
```

## Result

```csharp id="rvs66s"
public record CreaClienteResult(
    int ClienteId,
    string Nome,
    string? Email);
```

## Validator

```csharp id="6sjnhn"
public class CreaClienteValidator
    : IRequestValidator<CreaClienteCommand>
{
    public List<string> Validate(CreaClienteCommand request)
    {
        List<string> errors = [];

        if (string.IsNullOrWhiteSpace(request.Nome))
        {
            errors.Add("Il nome è obbligatorio.");
        }

        if (request.Nome.Length > 100)
        {
            errors.Add("Il nome non può superare 100 caratteri.");
        }

        if (request.Email is not null &&
            !request.Email.Contains('@'))
        {
            errors.Add("Email non valida.");
        }

        return errors;
    }
}
```

## Handler

```csharp id="jp1w6c"
using MediatR;

public class CreaClienteHandler
    : IRequestHandler<CreaClienteCommand, CreaClienteResult>
{
    private readonly AppDbContext dbContext;

    public CreaClienteHandler(AppDbContext dbContext)
    {
        this.dbContext = dbContext;
    }

    public async Task<CreaClienteResult> Handle(
        CreaClienteCommand command,
        CancellationToken cancellationToken)
    {
        bool emailEsistente = command.Email is not null &&
            await dbContext.Clienti.AnyAsync(
                c => c.Email == command.Email,
                cancellationToken);

        if (emailEsistente)
        {
            throw new BusinessRuleException(
                "Esiste già un cliente con questa email.");
        }

        Cliente cliente = new Cliente(
            command.Nome,
            command.Email);

        dbContext.Clienti.Add(cliente);

        await dbContext.SaveChangesAsync(cancellationToken);

        return new CreaClienteResult(
            cliente.Id,
            cliente.Nome,
            cliente.Email);
    }
}
```

## Endpoint

```csharp id="kmhttr"
public record CreaClienteRequest(
```

# Esempio completo: slice CercaClienti

```text id="q8gi4d"

  Clienti
    CercaClienti
      CercaClientiEndpoint.cs

      CercaClientiHandler.cs
      ClienteSearchItemDto.cs
```

## Query

```csharp id="3h9kzj"
using MediatR;

public record CercaClientiQuery(
    string? Search,
    int Page,
    int PageSize)
    : IRequest<List<ClienteSearchItemDto>>;
```

## DTO

```csharp id="yc9j2d"
public record ClienteSearchItemDto(
    int Id,
    string Nome,
    string? Email);
```

## Handler

```csharp id="dsmk8w"
using MediatR;

public class CercaClientiHandler
    : IRequestHandler<CercaClientiQuery, List<ClienteSearchItemDto>>
{
    private readonly AppDbContext dbContext;

    public CercaClientiHandler(AppDbContext dbContext)
    {
        this.dbContext = dbContext;
    }

    public async Task<List<ClienteSearchItemDto>> Handle(
        CercaClientiQuery query,
        CancellationToken cancellationToken)
    {
        int page = Math.Max(1, query.Page);
        int pageSize = Math.Clamp(query.PageSize, 1, 100);

        IQueryable<Cliente> clienti = dbContext.Clienti
            .AsNoTracking();

        if (!string.IsNullOrWhiteSpace(query.Search))
        {
            string search = query.Search.Trim();

            clienti = clienti.Where(c =>
                c.Nome.Contains(search) ||
                (c.Email != null && c.Email.Contains(search)));
        }

        return await clienti
            .OrderBy(c => c.Nome)
            .Skip((page - 1) * pageSize)
            .Take(pageSize)
            .Select(c => new ClienteSearchItemDto(
                c.Id,
                c.Nome,
                c.Email))
            .ToListAsync(cancellationToken);
    }
}
```

## Endpoint

```csharp id="z1hrc4"
public static class CercaClientiEndpoint

    public static RouteGroupBuilder MapCercaClientiEndpoint(

                pageSize);

            List<ClienteSearchItemDto> clienti =
```

# Pipeline behavior vs endpoint filters

In ASP.NET Core moderno hai due livelli possibili.

## Pipeline behavior MediatR

Adatto a logica applicativa trasversale:

```text id="gveix0"
validazione command/query
logging use case
misura durata handler
transazioni
authorization applicativa
idempotenza command
```

## Endpoint filter Minimal API

Adatto a logica HTTP trasversale:

```text id="aqbmkd"
lettura header
controllo parametri HTTP
trasformazione response
validazione specifica endpoint
rate limiting custom
logica prima/dopo endpoint
```

I filtri Minimal API sono documentati come meccanismo per eseguire codice prima e dopo l’endpoint, ispezionare parametri e intercettare il comportamento della risposta. ([Microsoft Learn][3])

Regola pratica:

```text id="oqunge"
Se riguarda HTTP, usa endpoint filter o middleware.
Se riguarda il caso d’uso, usa pipeline behavior.
```

---

# Validazione per use case

La validazione dovrebbe essere specifica del caso d’uso.

Esempio:

```text id="9x3ozt"
CreaCliente
  Nome obbligatorio
  Email valida

AggiornaEmailCliente
  ClienteId valido
  Email valida
  Email non già usata

CancellaCliente
  ClienteId valido
  Cliente cancellabile
```

Non creare un unico validator enorme per `Cliente`.

Meglio validator piccoli e vicini alla feature.

# Quando usare VSA

Vertical Slice Architecture è utile quando:

```text id="dx6qj7"
hai molte funzionalità distinte
i service stanno diventando troppo grandi
vuoi ridurre navigazione tra layer
usi Minimal API
usi CQRS leggero
vuoi testare use case specifici
vuoi avvicinare codice e caso d’uso
hai team che lavorano su feature diverse
```

Esempi adatti:

```text id="vmsi1k"
gestionale aziendale
API ASP.NET Core
backoffice
sistema con molti casi d’uso
monolite modulare
microservizio con feature interne ben definite
```

---

# Quando non usarla

Usala con cautela quando:

```text id="rr7jz2"
l’app è piccolissima
hai solo CRUD banale
il team non conosce command/query
la struttura per feature crea duplicazione inutile
le feature sono ancora instabili
il dominio è troppo semplice
```

Per una demo o una piccola app:

```text id="6ez4lc"
Controllers
Services
Data
```

può essere sufficiente.

---

# Pro di VSA

```text id="wd12se"
codice di una feature vicino
meno service giganteschi
meno repository generici
modifiche più localizzate
test più mirati
facile aggiungere nuovi use case
buona integrazione con CQRS
adatta a Minimal API
```

---

# Contro di VSA

```text id="cqbknm"
può duplicare piccoli DTO o mapping
richiede disciplina sui confini
può generare molti file
può essere dispersiva senza naming coerente
può nascondere logiche condivise se non gestite bene
MediatR può rendere il flusso meno esplicito
```

La duplicazione non è sempre negativa.

A volte due DTO simili ma usati da use case diversi dovrebbero restare separati.

Regola pratica:

```text id="1so83k"
Duplica quando i casi d’uso evolvono indipendentemente.
Condividi quando il concetto è davvero lo stesso.
```

# VSA e Domain Model

Vertical Slice Architecture non elimina il dominio.

Due modi possibili:

## Slice con dominio ricco

```text id="4d5249"
CreaOrdineHandler
  carica aggregate
  chiama ordine.AggiungiRiga()
  chiama ordine.Conferma()
  salva
```

Utile quando ci sono regole di business importanti.

## Slice CRUD diretta

```text id="427rs9"
AggiornaNomeClienteHandler
  carica entity EF
  modifica proprietà
  salva
```

Può andare bene per operazioni semplici.

Non serve forzare DDD ovunque.

# VSA e repository

Con VSA puoi scegliere.

## Repository

Utile se:

```text id="a2x97g"
devi proteggere aggregate
vuoi indipendenza da EF Core
hai logica di caricamento complessa
vuoi testare senza database
```

## DbContext diretto nello handler

Accettabile se:

```text id="p5rk2f"
la feature è semplice
usi EF Core consapevolmente
non vuoi repository pass-through inutili
la query è specifica del caso d’uso
```

Esempio:

```csharp id="j5bfc7"

    .AsNoTracking()
    .Where(...)
    .Select(...)
    .ToListAsync(cancellationToken);
```

Per query di lettura, il `DbContext` diretto nello handler è spesso pragmatico.

# Linee guida pratiche

```text id="vx9zee"
Organizza per feature
  quando i casi d’uso sono il centro dell’app.

Usa Command
  per operazioni che modificano stato.

Usa Query
  per operazioni di lettura.

Usa Handler
  per implementare il caso d’uso.

Usa CQRS leggero
  prima di pensare a CQRS distribuito.

Usa MediatR
  se vuoi disaccoppiamento e pipeline behavior.

Non usare MediatR
  se rende il progetto più difficile senza beneficio.

Metti la validazione vicino allo use case
  non in validator generici enormi.

Usa pipeline behavior
  per cross-cutting concerns applicativi.

Usa endpoint filter o middleware
  per cross-cutting concerns HTTP.

Per query
  proietta direttamente a DTO.

Per command complessi
  usa domain model e aggregate.

Evita handler enormi
  sposta regole nel dominio o in servizi dedicati.

Non essere dogmatica
  VSA è uno strumento, non una religione.
```

# Riepilogo

In questo capitolo abbiamo introdotto Vertical Slice Architecture e CQRS.

Abbiamo visto:

```text id="i4txug"
organizzazione per feature
Command
Query
Handler
MediatR
CQRS
separazione lettura/scrittura
validazione per use case
pipeline behavior
quando usare VSA
quando non usarla
```

La differenza più importante è questa:

```text id="5eux0m"
Layered Architecture
  organizza il codice per responsabilità tecniche.

Vertical Slice Architecture
  organizza il codice per funzionalità e casi d’uso.
```

CQRS aggiunge una separazione concettuale:

```text id="3145ec"
Command
  modifica stato.

Query
  legge stato.
```

MediatR può essere utile per inviare command/query agli handler e applicare pipeline behavior trasversali, ma non è obbligatorio per usare VSA o CQRS. I behavior sono particolarmente utili per validazione, logging, misurazione prestazioni e transazioni applicative. ([DEV Community][2])

La regola pratica finale è questa:

```text id="32b8kn"
Vertical Slice Architecture funziona bene quando ogni feature è piccola,
chiara e autonoma. Se le slice diventano solo mini-layer disordinati,
il problema non è risolto: è stato solo spostato.
```

Nel prossimo capitolo possiamo entrare nei design pattern più usati in applicazioni C# reali, ma con taglio pratico: non “catalogo GoF”, bensì quando un pattern risolve davvero un problema architetturale.

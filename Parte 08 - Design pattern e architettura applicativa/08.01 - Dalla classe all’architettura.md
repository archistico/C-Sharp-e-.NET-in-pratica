# 08.01 — Dalla classe all’architettura

Fino a questo punto abbiamo studiato il linguaggio, le librerie, l’accesso ai dati, ASP.NET Core, Blazor, async, concorrenza e performance.

Ora il problema cambia.

Non basta più chiedersi:

```text id="0wm78g"
Come scrivo questa classe?
Come implemento questo metodo?
Come salvo questo dato?
```

Bisogna iniziare a chiedersi:

```text id="y8gpgq"
Dove metto questa responsabilità?
Chi deve conoscere chi?
Che cosa può cambiare senza rompere tutto?
Come divido il sistema in parti comprensibili?
Come evito che l'applicazione diventi un groviglio?
```

Questo è il passaggio dalla programmazione alla progettazione applicativa.

Il punto centrale è questo:

```text id="h41an4"
L’architettura serve a organizzare il codice intorno alle responsabilità,
ai confini e alle dipendenze, in modo che il sistema resti comprensibile
anche quando cresce.
```

---

# Perché serve architettura

Una piccola applicazione può iniziare anche con pochi file:

```text id="geebhx"
Program.cs
Cliente.cs
AppDbContext.cs
```

All’inizio tutto sembra semplice.

Poi arrivano:

```text id="f3oc5m"
validazioni
autenticazione
autorizzazioni
logging
email
notifiche
database
API esterne
report
import/export
background job
test
regole di business
eccezioni
cache
interfacce utente diverse
```

Senza architettura, il codice tende a diventare così:

```text id="boz5fa"
controller che fanno tutto
servizi enormi
classi con troppe responsabilità
query sparse ovunque
logica duplicata
dipendenze circolari
business logic dentro la UI
business logic dentro SQL casuale
difficoltà a testare
paura di modificare
```

L’architettura serve a evitare che il sistema diventi fragile.

Non serve a rendere il codice “più elegante” in astratto.
Serve a renderlo più modificabile, testabile, comprensibile e coerente.

---

# Architettura non significa complicare

Un errore frequente è pensare che architettura significhi:

```text id="o05ooz"
più progetti
più interfacce
più pattern
più cartelle
più astrazioni
più framework
```

Non è così.

Una buona architettura può essere semplice.

Una cattiva architettura può essere molto sofisticata.

Esempio di sovra-ingegnerizzazione:

```text id="3c1olm"
IClienteAbstractFactoryProviderStrategy
ClienteDomainApplicationServiceManager
BaseGenericRepositoryOfTEntityWithUnitOfWorkFactory
```

Se l’applicazione ha tre tabelle e due form, probabilmente non serve tutto questo.

La regola è:

```text id="k6wba3"
L’architettura deve ridurre la complessità percepita,
non aumentarla.
```

---

# Complessità essenziale e complessità accidentale

Quando progettiamo software, incontriamo due tipi di complessità.

---

## Complessità essenziale

È la complessità propria del dominio.

Esempi:

```text id="s8n63e"
calcolare correttamente un premio
gestire regole fiscali
gestire autorizzazioni complesse
calcolare disponibilità di magazzino
coordinare un ordine con pagamento e spedizione
gestire presenze, costi, movimenti e rendicontazione
```

Questa complessità non può essere eliminata del tutto, perché appartiene al problema reale.

Possiamo solo rappresentarla bene.

---

## Complessità accidentale

È la complessità introdotta dal modo in cui costruiamo il software.

Esempi:

```text id="m7s6mc"
codice duplicato
nomi confusi
dipendenze circolari
classi troppo grandi
stato globale
layer mal separati
query copiate in punti diversi
configurazione disordinata
eccezioni nascoste
pattern usati senza motivo
```

Questa complessità possiamo e dobbiamo ridurla.

---

## Esempio

Regola di business:

```text id="068bpn"
Un cliente può ricevere un buono solo se ha almeno una presenza
nel periodo e non è inibito.
```

Questa è complessità essenziale.

Implementazione disordinata:

```csharp id="ht6fx3"
if (cliente != null && cliente.Seg != null && cliente.Seg.INIB_TIPO != "S"
    && db.Presenze.Where(x => x.ClieCodi == cliente.Id).Count() > 0)
{
    // logica copiata anche in altri 5 punti
}
```

Questa è complessità accidentale.

Meglio dare un nome alla regola:

```csharp id="d6hmkh"
public bool PuòRicevereBuono(Cliente cliente, Periodo periodo)
{
    return !cliente.IsInibito &&
           cliente.HaPresenzeNelPeriodo(periodo);
}
```

La regola resta complessa, ma diventa leggibile.

# Modularità

La **modularità** è la capacità di dividere un sistema in parti con responsabilità chiare.

Un modulo dovrebbe avere:

```text id="uijyp2"
uno scopo riconoscibile
confini chiari
dipendenze controllate
API interne comprensibili
bassa conoscenza dei dettagli degli altri moduli
```

Esempi di moduli:

```text id="6n45ud"
Clienti
Ordini
Pagamenti
Report
Autenticazione
Notifiche
Catalogo
Magazzino
Documenti
```

In .NET, un modulo può essere rappresentato in vari modi:

```text id="zzcdyb"
cartella
namespace
progetto
assembly
pacchetto NuGet
servizio separato
microservizio
```

Non bisogna confondere la modularità logica con la separazione fisica.

Un sistema può essere modulare anche dentro un singolo deploy.

---

# Separation of Concerns

**Separation of Concerns**, spesso abbreviato in SoC, significa separare le responsabilità.

Una classe, un metodo o un modulo non dovrebbe occuparsi di troppe cose diverse.

Esempio negativo:

```csharp id="48p5mp"
public class ClienteController : Controller
{
    public IActionResult Crea(CreaClienteInput input)
    {
        // valida input
        // controlla permessi
        // apre connessione SQL
        // costruisce query
        // salva cliente
        // invia email
        // scrive log
        // genera HTML
        // gestisce errori
    }
}
```

Questo codice mescola:

```text id="9q0egd"
UI
validazione
sicurezza
accesso ai dati
business logic
notifiche
logging
gestione errori
```

Separare le responsabilità significa distribuire meglio il lavoro:

```text id="jiunbl"
Controller / Endpoint
  riceve request e restituisce response.

Validator
  valida input.

Application Service
  coordina il caso d’uso.

Domain Model
  contiene regole di business.

Repository / DbContext
  gestisce persistenza.

EmailService
  invia email.

Logger
  registra eventi.
```

---

# Coesione

La **coesione** indica quanto gli elementi di una classe o di un modulo sono collegati tra loro.

Una classe è coesa quando i suoi metodi e dati lavorano per uno stesso scopo.

Esempio coeso:

```csharp id="qez8wh"
public class CalcolatoreScontoCliente
{
    public decimal CalcolaSconto(Cliente cliente, decimal importo)
    {
        // logica sconto cliente
    }
}
```

Responsabilità chiara:

```text id="p38gsy"
calcolare lo sconto di un cliente
```

Esempio poco coeso:

```csharp id="4ivbn6"
public class ClienteHelper
{
    public decimal CalcolaSconto(Cliente cliente, decimal importo) { ... }

    public void InviaEmailBenvenuto(Cliente cliente) { ... }

    public string GeneraHtmlTabella(List<Cliente> clienti) { ... }

    public void EsportaExcel(List<Cliente> clienti) { ... }

    public void PulisciCache() { ... }
}
```

Qui la classe fa troppe cose diverse.

Regola pratica:

```text id="92wqrs"
Alta coesione:
  una parte del sistema ha una responsabilità forte e riconoscibile.

Bassa coesione:
  una parte del sistema contiene cose messe insieme solo per comodità.
```

# Accoppiamento

L’**accoppiamento** indica quanto una parte del sistema dipende da un’altra.

Un certo accoppiamento è inevitabile.
Il problema è l’accoppiamento eccessivo o sbagliato.

Esempio di accoppiamento forte:

```csharp id="nf9led"
public class ClienteService
{
    private readonly SqlConnection connection;

    public ClienteService()
    {
        connection = new SqlConnection(
            "Server=.;Database=App;User Id=sa;Password=...");
    }
}
```

Problemi:

```text id="4q5b6v"
connection string hard-coded
difficile da testare
dipende direttamente da SQL Server
non usa configurazione
non usa DI
```

Meglio:

```csharp id="d7fbbs"
public class ClienteService
{
    private readonly IClienteRepository repository;

    public ClienteService(IClienteRepository repository)
    {
        this.repository = repository;
    }
}
```

Ora `ClienteService` dipende da un contratto, non da un dettaglio concreto.

---

## Accoppiamento buono e cattivo

Accoppiamento inevitabile:

```text id="3foccy"
un endpoint deve conoscere un servizio applicativo
un servizio applicativo deve conoscere il dominio
un repository deve conoscere il database
```

Accoppiamento problematico:

```text id="53qi2k"
la UI conosce dettagli SQL
il dominio conosce ASP.NET Core
le entità conoscono EF Core in modo invasivo
un servizio conosce classi concrete non necessarie
moduli diversi si chiamano direttamente in modo circolare
```

Regola pratica:

```text id="rg5k5z"
Le dipendenze devono andare verso ciò che è più stabile,
non verso ciò che cambia più spesso.
```

---

# Layer

Un **layer** è un livello logico dell’applicazione.

Esempio:

```text id="ut26ra"
Presentation Layer
Application Layer
Domain Layer
Infrastructure Layer
```

Un layer è una divisione logica.

Non implica necessariamente un server separato, un processo separato o un progetto separato.

Esempio in una singola solution:

```text id="uqk0v3"
MiaApp.Web
MiaApp.Application
MiaApp.Domain
MiaApp.Infrastructure
```

Ma potresti anche avere tutto in un solo progetto, almeno in una piccola applicazione, mantenendo comunque cartelle e namespace separati.

---

# Tier

Un **tier** è un livello fisico o di deployment.

Esempio:

```text id="631axz"
Browser
  tier client

Web server
  tier applicativo

Database server
  tier dati
```

Oppure:

```text id="511nob"
app desktop
  tier client

API server
  tier backend

SQL Server
  tier database
```

---

# Layer vs Tier

La differenza è fondamentale.

```text id="1v6o43"
Layer
  separazione logica del codice.

Tier
  separazione fisica/deployment.
```

Esempio:

```text id="e9spq8"
Presentation Layer
Application Layer
Domain Layer
Infrastructure Layer
```

possono vivere tutti nello stesso processo.

Oppure possono essere distribuiti su tier diversi.

Errore comune:

```text id="pu6h25"
Ho tre progetti, quindi ho tre tier.
```

No.

Hai tre progetti, quindi probabilmente hai tre separazioni logiche o assembly.
I tier dipendono da dove il codice viene eseguito fisicamente.

---

# Monolite

Un **monolite** è un’applicazione distribuita come una singola unità.

Esempio:

```text id="6g83fy"
un'app ASP.NET Core
  contiene UI, logica applicativa, dominio e accesso ai dati
  viene pubblicata come un singolo deployment
```

Monolite non significa automaticamente “codice brutto”.

Un monolite può essere:

```text id="2cpxwv"
semplice
efficiente
facile da distribuire
facile da debuggare
adatto a molti contesti aziendali
```

Il problema non è il monolite in sé.

Il problema è il **monolite disordinato**.

---

## Monolite disordinato

```text id="ncl8kd"
UI che chiama direttamente query SQL ovunque
business logic sparsa
classi enormi
nessun confine tra moduli
dipendenze circolari
database come unico punto di integrazione interna
test quasi impossibili
```

Questo è il monolite problematico.

---

# Monolite modulare

Un **monolite modulare** è un’applicazione distribuita come una singola unità, ma organizzata internamente in moduli ben separati.

Esempio:

```text id="7fs6o2"
MiaApp
  Modules
    Clienti
      ClientiEndpoints
      ClienteService
      ClienteRepository
      Cliente
    Ordini
      OrdiniEndpoints
      OrdineService
      OrdineRepository
      Ordine
    Pagamenti
      PagamentiEndpoints
      PagamentoService
      PagamentoRepository
      Pagamento
```

Oppure con progetti separati:

```text id="57lrsc"
MiaApp.Web
MiaApp.Modules.Clienti
MiaApp.Modules.Ordini
MiaApp.Modules.Pagamenti
MiaApp.SharedKernel
```

Il deploy resta unico, ma il codice è organizzato.

---

## Vantaggi del monolite modulare

```text id="sc3c5b"
deployment semplice
debug più facile
meno complessità distribuita
transazioni più semplici
refactoring più agevole rispetto ai microservizi
confini logici più chiari rispetto a un monolite casuale
```

È spesso una scelta eccellente per applicazioni aziendali.

Regola pratica:

```text id="e8yo0q"
Prima costruisci un buon monolite modulare.
Passa ai microservizi solo se hai reali motivi organizzativi,
tecnici e operativi.
```

# Architettura a tre livelli

L’architettura a tre livelli è una delle strutture più comuni.

Di solito distingue:

```text id="idw0g0"
Presentation
Business Logic
Data Access
```

Oppure:

```text id="51psy8"
UI
Logica applicativa/business
Persistenza
```

Schema:

```text id="adw685"
Presentation Layer
  ↓
Business Layer
  ↓
Data Access Layer
```

---

## Presentation Layer

Si occupa di interazione con l’utente o con il client.

Esempi:

```text id="8g92hx"
Razor Pages
MVC Controller
Minimal API endpoint
Blazor component
WinForms/WPF window
CLI command
```

Responsabilità:

```text id="93jkx4"
ricevere input
mostrare output
gestire request/response
fare model binding
gestire validazione superficiale
tradurre errori in messaggi utente
```

Non dovrebbe contenere business logic profonda.

---

## Business Layer

Contiene la logica applicativa e di business.

Esempi:

```text id="ndhmwj"
ClienteService
OrdineService
CalcolatoreSconto
GestorePrenotazione
GeneratoreReport
```

Responsabilità:

```text id="3ksas7"
regole applicative
coordinamento operazioni
validazioni di business
decisioni
calcoli
workflow
```

---

## Data Access Layer

Gestisce accesso ai dati.

Esempi:

```text id="fezqav"
DbContext EF Core
repository
ADO.NET
stored procedure wrapper
query SQL
client per storage esterni
```

Responsabilità:

```text id="wpxb8a"
leggere dati
scrivere dati
gestire query
gestire transazioni
tradurre database in oggetti
```

---

## Esempio pratico a tre livelli

Endpoint:

```csharp id="r6o2t1"
app.MapPost("/api/clienti", async (
    CreaClienteRequest request,
    ClienteService service,
    CancellationToken cancellationToken) =>
{
    ClienteDto cliente = await service.CreaAsync(
        request,
        cancellationToken);

    return Results.Created(
        $"/api/clienti/{cliente.Id}",
        cliente);
});
```

Service:

```csharp id="xq4qvp"
public class ClienteService
{
    private readonly ClienteRepository repository;

    public ClienteService(ClienteRepository repository)
    {
        this.repository = repository;
    }

    public async Task<ClienteDto> CreaAsync(
        CreaClienteRequest request,
        CancellationToken cancellationToken)
    {
        Cliente cliente = new Cliente
        {
            Nome = request.Nome,
            Email = request.Email,
            Attivo = true
        };

        await repository.SalvaAsync(cliente, cancellationToken);

        return new ClienteDto(
            cliente.Id,
            cliente.Nome,
            cliente.Email);
    }
}
```

Repository:

```csharp id="o9l2jf"
public class ClienteRepository
{
    private readonly AppDbContext dbContext;

    public ClienteRepository(AppDbContext dbContext)
    {
        this.dbContext = dbContext;
    }

    public async Task SalvaAsync(
        Cliente cliente,
        CancellationToken cancellationToken)
    {
        dbContext.Clienti.Add(cliente);

        await dbContext.SaveChangesAsync(cancellationToken);
    }
}
```

Separazione:

```text id="m2wuf5"
Endpoint
  HTTP

Service
  caso d'uso

Repository
  persistenza
```

# Limiti dell’architettura a tre livelli

Il modello a tre livelli è utile, ma può diventare troppo generico.

Problemi possibili:

```text id="zz4bwi"
Business Layer troppo vago
servizi enormi
repository generici inutili
dominio anemico
logica business sparsa tra service e database
modelli dati usati come modelli di dominio
nessuna distinzione tra caso d’uso e regola di dominio
```

Per sistemi più complessi, può essere utile passare a una separazione più ricca, come quella ispirata al Domain-Driven Design.

---

# Architettura a quattro layer DDD

Una struttura molto usata nelle architetture ispirate al DDD distingue quattro layer logici:

```text id="s2mowd"
Presentation Layer
Application Layer
Domain Layer
Infrastructure Layer
```

Schema:

```text id="8tr3it"
Presentation
  ↓
Application
  ↓
Domain
  ↑
Infrastructure
```

Più precisamente, l’infrastruttura implementa dettagli tecnici richiesti dall’application/domain, ma il dominio non dovrebbe dipendere dai dettagli infrastrutturali.

---

# Presentation Layer

Il Presentation Layer è il punto di ingresso.

Può essere:

```text id="6m5xs9"
Minimal API
Controller MVC
Razor Pages
Blazor components
WPF ViewModel
Console command
SignalR Hub
```

Responsabilità:

```text id="gkg8qk"
ricevere input esterno
tradurre request in comandi/query
chiamare application layer
restituire response o UI
gestire aspetti HTTP/UI
```

Esempio:

```csharp id="vlw8wr"
app.MapPost("/api/ordini", async (
    CreaOrdineRequest request,
    CreaOrdineHandler handler,
    CancellationToken cancellationToken) =>
{
    CreaOrdineResult result =
        await handler.HandleAsync(request, cancellationToken);

    return Results.Created(
        $"/api/ordini/{result.Id}",
        result);
});
```

---

# Application Layer

L’Application Layer coordina i casi d’uso.

Esempi:

```text id="n93x12"
CreaOrdine
ApprovaDocumento
RegistraCliente
GeneraReport
AnnullaPrenotazione
InviaNotifica
```

Responsabilità:

```text id="hxw3l2"
orchestrare il caso d’uso
caricare dati necessari
chiamare il dominio
salvare modifiche
gestire transazioni applicative
chiamare servizi esterni tramite interfacce
pubblicare eventi
```

Non dovrebbe contenere dettagli di UI, HTTP o database specifico.

Esempio:

```csharp id="p8r3b8"
public class CreaOrdineHandler
{
    private readonly IOrdineRepository ordineRepository;
    private readonly IClienteRepository clienteRepository;
    private readonly IUnitOfWork unitOfWork;

    public CreaOrdineHandler(
        IOrdineRepository ordineRepository,
        IClienteRepository clienteRepository,
        IUnitOfWork unitOfWork)
    {
        this.ordineRepository = ordineRepository;
        this.clienteRepository = clienteRepository;
        this.unitOfWork = unitOfWork;
    }

    public async Task<CreaOrdineResult> HandleAsync(
        CreaOrdineRequest request,
        CancellationToken cancellationToken)
    {
        Cliente cliente = await clienteRepository.GetByIdAsync(
            request.ClienteId,
            cancellationToken);

        Ordine ordine = Ordine.Crea(cliente.Id);

        foreach (CreaOrdineRigaRequest riga in request.Righe)
        {
            ordine.AggiungiRiga(
                riga.ProdottoId,
                riga.Quantita,
                riga.PrezzoUnitario);
        }

        await ordineRepository.AddAsync(ordine, cancellationToken);

        await unitOfWork.SaveChangesAsync(cancellationToken);

        return new CreaOrdineResult(ordine.Id);
    }
}
```

# Domain Layer

Il Domain Layer contiene il cuore del business.

Qui vivono:

```text id="3jct62"
entità
value object
aggregate
domain service
domain events
regole di business
invarianti
```

Il dominio deve essere il più possibile indipendente da:

```text id="4hrhoj"
ASP.NET Core
EF Core
database
file system
HTTP
UI
framework esterni
```

Esempio:

```csharp id="kfvbix"
public class Ordine
{
    private readonly List<OrdineRiga> righe = [];

    public int Id { get; private set; }
    public int ClienteId { get; private set; }
    public IReadOnlyList<OrdineRiga> Righe => righe;
    public StatoOrdine Stato { get; private set; }

    private Ordine()
    {
    }

    private Ordine(int clienteId)
    {
        ClienteId = clienteId;
        Stato = StatoOrdine.Bozza;
    }

    public static Ordine Crea(int clienteId)
    {
        if (clienteId <= 0)
        {
            throw new DomainException("Cliente non valido.");
        }

        return new Ordine(clienteId);
    }

    public void AggiungiRiga(
        int prodottoId,
        int quantita,
        decimal prezzoUnitario)
    {
        if (Stato != StatoOrdine.Bozza)
        {
            throw new DomainException(
                "Non è possibile modificare un ordine confermato.");
        }

        if (quantita <= 0)
        {
            throw new DomainException(
                "La quantità deve essere maggiore di zero.");
        }

        righe.Add(new OrdineRiga(
            prodottoId,
            quantita,
            prezzoUnitario));
    }

    public void Conferma()
    {
        if (righe.Count == 0)
        {
            throw new DomainException(
                "Non è possibile confermare un ordine senza righe.");
        }

        Stato = StatoOrdine.Confermato;
    }
}
```

Qui le regole importanti stanno nel dominio, non nel controller.

# Infrastructure Layer

L’Infrastructure Layer contiene i dettagli tecnici.

Esempi:

```text id="5o1qix"
EF Core DbContext
repository concreti
invio email SMTP
client HTTP esterni
file system
code messaggi
cache Redis
implementazioni logging specializzate
provider storage
servizi cloud
```

Esempio:

```csharp id="ytgy6m"
public class EfOrdineRepository : IOrdineRepository
{
    private readonly AppDbContext dbContext;

    public EfOrdineRepository(AppDbContext dbContext)
    {
        this.dbContext = dbContext;
    }

    public async Task AddAsync(
        Ordine ordine,
        CancellationToken cancellationToken)
    {
        dbContext.Ordini.Add(ordine);

        await Task.CompletedTask;
    }

    public async Task<Ordine?> GetByIdAsync(
        int id,
        CancellationToken cancellationToken)
    {
        return await dbContext.Ordini
            .Include(o => o.Righe)
            .FirstOrDefaultAsync(o => o.Id == id, cancellationToken);
    }
}
```

Il contratto può stare nell’Application o nel Domain Layer:

```csharp id="s6n6cc"
public interface IOrdineRepository
{
    Task AddAsync(
        Ordine ordine,
        CancellationToken cancellationToken);

    Task<Ordine?> GetByIdAsync(
        int id,
        CancellationToken cancellationToken);
}
```

L’implementazione concreta sta nell’infrastruttura.

---

# Dipendenze tra layer

Una regola importante:

```text id="a4b5hm"
I layer più interni non devono dipendere dai layer più esterni.
```

Schema consigliato:

```text id="nq42f7"
Presentation
  dipende da Application

Application
  dipende da Domain
  dipende da astrazioni

Infrastructure
  dipende da Application/Domain
  implementa astrazioni

Domain
  non dipende da Infrastructure
  non dipende da Presentation
```

In pratica:

```text id="agwiy9"
Il dominio non deve conoscere EF Core.
Il dominio non deve conoscere ASP.NET Core.
Il dominio non deve conoscere il database.
```

---

# Architettura e Dependency Injection

La Dependency Injection aiuta a rispettare le dipendenze.

Registrazione:

```csharp id="u356t7"
builder.Services.AddScoped<IOrdineRepository, EfOrdineRepository>();
builder.Services.AddScoped<CreaOrdineHandler>();
```

Application Layer usa l’interfaccia:

```csharp id="kvm1ce"
private readonly IOrdineRepository ordineRepository;
```

Infrastructure fornisce l’implementazione:

```csharp id="7cxwzv"
public class EfOrdineRepository : IOrdineRepository
{
}
```

Questo permette di:

```text id="ejcjex"
testare application layer con fake repository
cambiare persistenza
ridurre accoppiamento
rendere esplicite le dipendenze
```

# Monolite modulare con layer interni

Possiamo combinare monolite modulare e layer.

Esempio:

```text id="pr17lo"
MiaApp
  Modules
    Clienti
      Presentation
      Application
      Domain
      Infrastructure
    Ordini
      Presentation
      Application
      Domain
      Infrastructure
    Pagamenti
      Presentation
      Application
      Domain
      Infrastructure
```

Oppure una struttura più semplice:

```text id="xglowb"
MiaApp.Web
MiaApp.Application
MiaApp.Domain
MiaApp.Infrastructure
```

La prima organizza per modulo.
La seconda organizza per layer.

## Organizzare per layer

```text id="l9l8ok"
Application
  Clienti
  Ordini
  Pagamenti

Domain
  Clienti
  Ordini
  Pagamenti

Infrastructure
  Clienti
  Ordini
  Pagamenti
```

Vantaggi:

```text id="qpz48c"
chiara separazione tecnica
facile da capire all'inizio
molto comune
```

Svantaggi:

```text id="sumde1"
le funzionalità sono sparse in più progetti/cartelle
i confini di modulo possono indebolirsi
rischio di servizi generici enormi
```

## Organizzare per feature/modulo

```text id="kym60l"

  Clienti
    ClientiEndpoints
    CreaCliente

  Ordini
```

Vantaggi:

Svantaggi:

```text id="vty19r"

meno familiare per alcuni team
```

# Errori comuni

## Confondere architettura con cartelle

Avere cartelle chiamate:

```text id="zwcgpd"
Controllers
Services
Repositories
Models
```

non garantisce una buona architettura.

Conta cosa ci metti dentro e quali dipendenze permetti.

---

## Mettere tutta la logica nei controller

Sbagliato:

```text id="hukz80"
controller con validazioni business, query, calcoli, email, mapping e transazioni
```

Il controller dovrebbe essere un coordinatore HTTP leggero.

---

## Creare repository generici inutili

Sospetto:

```csharp id="nw45m4"
public interface IRepository<T>
{
    Task<T?> GetByIdAsync(int id);
    Task<List<T>> GetAllAsync();
    Task AddAsync(T entity);
    Task UpdateAsync(T entity);
    Task DeleteAsync(T entity);
}
```

Non sempre è utile.

Con EF Core, un repository generico spesso aggiunge poco e può nascondere male le capacità dell’ORM.

Meglio repository specifici quando c’è reale logica di accesso o confine di dominio.

---

## Dominio anemico

Un dominio anemico ha entità con solo proprietà e nessun comportamento.

```csharp id="sjcxrx"
public class Ordine
{
    public int Id { get; set; }
    public List<OrdineRiga> Righe { get; set; } = [];
    public string Stato { get; set; } = "";
}
```

Tutte le regole finiscono nei service.

Meglio spostare almeno le invarianti fondamentali dentro il modello:

```csharp id="t5s0kc"
ordine.Conferma();
ordine.AggiungiRiga(...);
ordine.Annulla();
```

---

## Infrastructure che invade il dominio

Sospetto:

```csharp id="t8gfg8"
public class Ordine
{
    private readonly AppDbContext dbContext;
}
```

Il dominio non dovrebbe conoscere il database.

## Troppi layer per app semplici

Sbagliato:

```text id="1cjml7"
per una tabella e due schermate creo 9 progetti,
20 interfacce e 15 pattern
```

La complessità architetturale deve essere proporzionata.

# Linee guida pratiche

```text id="2jnqah"
Usa architettura
  per controllare complessità, non per decorare il codice.

Distingui complessità essenziale
  da complessità accidentale.

Cerca alta coesione
  classi e moduli con responsabilità chiare.

Riduci accoppiamento
  soprattutto verso dettagli tecnici instabili.

Separa layer logici
  da tier fisici.

Non demonizzare il monolite
  il problema è il monolite disordinato.

Valuta il monolite modulare
  come scelta forte per molte app aziendali.

Usa tre livelli
  per app semplici o medie.

Usa quattro layer DDD
  quando il dominio e i casi d’uso diventano importanti.

Tieni il dominio indipendente
  da UI, database e framework.

Non creare astrazioni premature
  parti semplice e aumenta struttura quando serve.
```

# Riepilogo

In questo capitolo abbiamo fatto il passaggio dalla classe all’architettura.

Abbiamo visto:

```text id="dgf55o"
perché serve architettura
complessità accidentale e complessità essenziale
modularità
Separation of Concerns
coesione
accoppiamento
layer
tier
monolite
monolite modulare
architettura a tre livelli
architettura a quattro layer DDD
```

Abbiamo chiarito che:

```text id="sucw9e"
Layer
  è una separazione logica del codice.

Tier
  è una separazione fisica o di deployment.
```

Abbiamo distinto:

```text id="y6c929"
Monolite disordinato
  tutto dipende da tutto.

Monolite modulare
  deploy unico, ma confini interni chiari.
```

Abbiamo visto che una struttura a tre livelli può essere sufficiente per molte applicazioni, ma nei sistemi con dominio importante può essere utile distinguere meglio:

```text id="1td54a"
Presentation
Application
Domain
Infrastructure
```

La regola principale è questa:

```text id="atpwia"
Una buona architettura non nasce dal numero di pattern usati,
ma dalla chiarezza delle responsabilità e dalla direzione corretta
delle dipendenze.
```

Nel prossimo capitolo possiamo entrare nei pattern in modo pragmatico: non come catalogo astratto, ma come strumenti da usare quando risolvono un problema reale di progettazione.

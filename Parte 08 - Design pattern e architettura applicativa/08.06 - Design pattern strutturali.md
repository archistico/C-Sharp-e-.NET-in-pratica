# 08.06 — Design pattern strutturali

Nel capitolo precedente abbiamo visto i pattern creazionali, cioè pattern che aiutano a gestire la creazione degli oggetti.

Ora passiamo ai **design pattern strutturali**.

I pattern strutturali rispondono a una domanda diversa:

```text id="yw95sr"
Come organizzo classi e oggetti in modo che possano collaborare
senza creare dipendenze rigide, duplicazione o accoppiamento eccessivo?
```

Il punto centrale è questo:

```text id="sbvneu"
I pattern strutturali servono a comporre oggetti,
adattare interfacce, nascondere complessità,
aggiungere comportamento e rappresentare strutture gerarchiche.
```

Sono particolarmente utili quando dobbiamo:

```text id="rg9rde"
integrare sistemi legacy
semplificare API complesse
aggiungere funzionalità trasversali
proteggere accessi remoti
modellare alberi e gerarchie
separare astrazione e implementazione
```

---

# Adapter

L’**Adapter** permette a due parti incompatibili di collaborare.

Il problema tipico è questo:

```text id="5lz4el"
Il mio codice si aspetta una certa interfaccia,
ma un sistema esterno o legacy espone un’interfaccia diversa.
```

Invece di modificare tutto il codice applicativo, creiamo un adapter.

---

## Esempio: sistema legacy

Immaginiamo un vecchio servizio legacy:

```csharp id="wpkmmx"
public class LegacyCustomerService
{
    public LegacyCustomerData GetCustomerByCode(string customerCode)
    {
        // chiamata a sistema legacy
        return new LegacyCustomerData
        {
            Code = customerCode,
            FullName = "Mario Rossi",
            Mail = "mario@example.com"
        };
    }
}
```

DTO legacy:

```csharp id="g376eg"
public class LegacyCustomerData
{
    public string Code { get; set; } = "";
    public string FullName { get; set; } = "";
    public string? Mail { get; set; }
}
```

Il nostro codice moderno, però, vuole lavorare con questa interfaccia:

```csharp id="e0fcfs"
public interface IClienteReader
{
    Task<ClienteDto?> TrovaAsync(
        string codiceCliente,
        CancellationToken cancellationToken);
}
```

DTO moderno:

```csharp id="b0g4gl"
public record ClienteDto(
    string Codice,
    string Nome,
    string? Email);
```

Creiamo un adapter:

```csharp id="f6sy6i"
public class LegacyClienteReaderAdapter : IClienteReader
{
    private readonly LegacyCustomerService legacyService;

    public LegacyClienteReaderAdapter(
        LegacyCustomerService legacyService)
    {
        this.legacyService = legacyService;
    }

    public Task<ClienteDto?> TrovaAsync(
        string codiceCliente,
        CancellationToken cancellationToken)
    {
        LegacyCustomerData legacyCustomer =
            legacyService.GetCustomerByCode(codiceCliente);

        ClienteDto cliente = new(
            legacyCustomer.Code,
            legacyCustomer.FullName,
            legacyCustomer.Mail);

        return Task.FromResult<ClienteDto?>(cliente);
    }
}
```

Registrazione DI:

```csharp id="p5ijwq"
builder.Services.AddSingleton<LegacyCustomerService>();
builder.Services.AddScoped<IClienteReader, LegacyClienteReaderAdapter>();
```

Ora il resto dell’applicazione conosce solo `IClienteReader`.

---

## Problema risolto

Senza adapter, il codice applicativo conoscerebbe direttamente:

```text id="ldf8ax"
LegacyCustomerService
LegacyCustomerData
GetCustomerByCode
FullName
Mail
```

Con adapter, il legacy resta confinato.

```text id="beb342"
Application Layer
  usa IClienteReader

Infrastructure
  implementa IClienteReader tramite LegacyClienteReaderAdapter
```

Questo è perfettamente coerente con Clean Architecture e Ports & Adapters.

---

## Quando usare Adapter

Usalo quando:

```text id="jvyxjk"
integri sistemi legacy
usi librerie esterne con API scomode
devi tradurre DTO esterni in modelli interni
vuoi isolare un protocollo o formato
vuoi proteggere il dominio da dettagli tecnici
```

Esempi comuni in .NET:

```text id="nrnmxq"
adapter verso stored procedure legacy
adapter verso servizio SOAP
adapter verso API REST esterna
adapter verso file CSV di un fornitore
adapter verso SDK di terze parti
```

# Facade

La **Facade** fornisce un’interfaccia semplice sopra un sottosistema complesso.

Il problema tipico è questo:

```text id="8m5sx7"
Per fare un’operazione devo chiamare molte classi,
in un certo ordine, con dettagli che il chiamante non dovrebbe conoscere.
```

La facade nasconde questa complessità.

---

## Esempio: generazione documento

Senza facade:

```csharp id="twk5qk"
Template template = templateRepository.Get("privacy");
Documento documento = templateEngine.Render(template, dati);
byte[] pdf = pdfGenerator.Generate(documento);
storage.Save(path, pdf);
emailSender.Send(cliente.Email, "Documento", pdf);
```

Se questo flusso viene copiato in più punti, diventa fragile.

Creiamo una facade:

```csharp id="u0psui"
public class DocumentoFacade
{
    private readonly TemplateRepository templateRepository;
    private readonly TemplateEngine templateEngine;
    private readonly PdfGenerator pdfGenerator;
    private readonly FileStorage storage;
    private readonly EmailSender emailSender;

    public DocumentoFacade(
        TemplateRepository templateRepository,
        TemplateEngine templateEngine,
        PdfGenerator pdfGenerator,
        FileStorage storage,
        EmailSender emailSender)
    {
        this.templateRepository = templateRepository;
        this.templateEngine = templateEngine;
        this.pdfGenerator = pdfGenerator;
        this.storage = storage;
        this.emailSender = emailSender;
    }

    public async Task GeneraSalvaEInviaAsync(
        string templateName,
        object dati,
        ClienteDto cliente,
        CancellationToken cancellationToken)
    {
        Template template = await templateRepository.GetAsync(
            templateName,
            cancellationToken);

        Documento documento = templateEngine.Render(template, dati);

        byte[] pdf = pdfGenerator.Generate(documento);

        string path = $"documenti/{cliente.Id}/{templateName}.pdf";

        await storage.SaveAsync(path, pdf, cancellationToken);

        await emailSender.SendAsync(
            cliente.Email,
            "Documento",
            pdf,
            cancellationToken);
    }
}
```

Uso:

```csharp id="sx81kh"
await documentoFacade.GeneraSalvaEInviaAsync(
    "privacy",
    dati,
    cliente,
    cancellationToken);
```

---

## Problema risolto

La facade:

```text id="6r39k3"
semplifica il chiamante
centralizza il flusso
nasconde sottosistemi complessi
riduce duplicazione
rende più leggibile il caso d’uso
```

---

## Facade vs Application Service

Attenzione: una facade può assomigliare a un application service.

Differenza pratica:

```text id="l1q7f4"
Application Service
  rappresenta un caso d’uso dell’applicazione.

Facade
  semplifica l’accesso a un sottosistema complesso.
```

Esempio:

```text id="x5t7gq"
CreaOrdineUseCase
  application service

DocumentoFacade
  facade sopra template, PDF, storage ed email
```

In alcuni progetti i due concetti possono sovrapporsi. L’importante è non creare classi “facade” che diventano contenitori enormi di logica non correlata.

---

## Quando usare Facade

Usala quando:

```text id="jvflgy"
un sottosistema richiede molte chiamate coordinate
vuoi offrire un’API più semplice
vuoi nascondere dettagli tecnici
vuoi ridurre duplicazione
vuoi proteggere il chiamante da complessità interna
```

Non usarla come classe generica “Manager” dove finisce tutto.

---

# Decorator

Il **Decorator** aggiunge comportamento a un oggetto senza modificarne la classe originale.

Il problema tipico è questo:

```text id="9urlj4"
Ho un servizio esistente e voglio aggiungere logging, cache,
retry, misurazione performance o autorizzazione senza cambiare
la logica principale.
```

---

## Esempio base

Interfaccia:

```csharp id="t0ok0q"
public interface IClienteService
{
    Task<ClienteDto?> TrovaAsync(
        int id,
        CancellationToken cancellationToken);
}
```

Implementazione reale:

```csharp id="c2mwq5"
public class ClienteService : IClienteService
{
    private readonly AppDbContext dbContext;

    public ClienteService(AppDbContext dbContext)
    {
        this.dbContext = dbContext;
    }

    public async Task<ClienteDto?> TrovaAsync(
        int id,
        CancellationToken cancellationToken)
    {
        return await dbContext.Clienti
            .AsNoTracking()
            .Where(c => c.Id == id)
            .Select(c => new ClienteDto(
                c.Id,
                c.Nome,
                c.Email))
            .FirstOrDefaultAsync(cancellationToken);
    }
}
```

Decorator con logging:

```csharp id="u3iock"
public class LoggingClienteServiceDecorator : IClienteService
{
    private readonly IClienteService inner;
    private readonly ILogger<LoggingClienteServiceDecorator> logger;

    public LoggingClienteServiceDecorator(
        IClienteService inner,
        ILogger<LoggingClienteServiceDecorator> logger)
    {
        this.inner = inner;
        this.logger = logger;
    }

    public async Task<ClienteDto?> TrovaAsync(
        int id,
        CancellationToken cancellationToken)
    {
        logger.LogInformation(
            "Ricerca cliente {ClienteId}",
            id);

        ClienteDto? cliente = await inner.TrovaAsync(
            id,
            cancellationToken);

        if (cliente is null)
        {
            logger.LogWarning(
                "Cliente {ClienteId} non trovato",
                id);
        }

        return cliente;
    }
}
```

Il decorator implementa la stessa interfaccia e avvolge un altro oggetto della stessa interfaccia.

---

# Decorator con DI

In .NET, il pattern Decorator è molto utile con Dependency Injection.

Il container standard di Microsoft non ha sempre una sintassi nativa comodissima per decorare servizi, quindi nei progetti reali si usa spesso una libreria come Scrutor oppure una registrazione manuale.

## Registrazione manuale semplice

```csharp id="on14xz"
builder.Services.AddScoped<ClienteService>();

builder.Services.AddScoped<IClienteService>(sp =>
{
    ClienteService inner =
        sp.GetRequiredService<ClienteService>();

    ILogger<LoggingClienteServiceDecorator> logger =
        sp.GetRequiredService<ILogger<LoggingClienteServiceDecorator>>();

    return new LoggingClienteServiceDecorator(inner, logger);
});
```

Uso:

```csharp id="v6y3ar"
public class CercaClienteUseCase
{
    private readonly IClienteService clienteService;

    public CercaClienteUseCase(IClienteService clienteService)
    {
        this.clienteService = clienteService;
    }
}
```

Il caso d’uso riceve `IClienteService`, ma in realtà riceve il decorator.

---

## Decorator per cache

```csharp id="g7jgqf"
public class CachedClienteServiceDecorator : IClienteService
{
    private readonly IClienteService inner;
    private readonly IMemoryCache cache;

    public CachedClienteServiceDecorator(
        IClienteService inner,
        IMemoryCache cache)
    {
        this.inner = inner;
        this.cache = cache;
    }

    public async Task<ClienteDto?> TrovaAsync(
        int id,
        CancellationToken cancellationToken)
    {
        string cacheKey = $"cliente:{id}";

        if (cache.TryGetValue(cacheKey, out ClienteDto? cached))
        {
            return cached;
        }

        ClienteDto? cliente = await inner.TrovaAsync(
            id,
            cancellationToken);

        if (cliente is not null)
        {
            cache.Set(
                cacheKey,
                cliente,
                TimeSpan.FromMinutes(5));
        }

        return cliente;
    }
}
```

Possibile catena:

```text id="ohdk99"
LoggingDecorator
  → CacheDecorator
      → ClienteService reale
```

---

## Problema risolto

Il Decorator evita di mescolare logiche trasversali dentro il servizio principale.

Senza decorator:

```text id="x6pt58"
ClienteService
  query
  logging
  cache
  retry
  metriche
  autorizzazione
```

Con decorator:

```text id="amfvuh"
ClienteService
  logica principale

LoggingDecorator
  logging

CachedDecorator
  cache

RetryDecorator
  retry
```

Ogni classe ha una responsabilità più chiara.

---

## Quando usare Decorator

Usalo quando vuoi aggiungere:

```text id="wmf95m"
logging
cache
retry
metriche
validazione trasversale
autorizzazione
tracing
fallback
compressione
normalizzazione input/output
```

senza modificare la classe originale.

# Proxy

Il **Proxy** controlla l’accesso a un altro oggetto.

Sembra simile al Decorator, ma l’intenzione è diversa.

```text id="cmd46z"
Decorator
  aggiunge comportamento.

Proxy
  controlla l’accesso a qualcosa.
```

Esempi di proxy:

```text id="x0la05"
remote proxy
  rappresenta un servizio remoto.

virtual proxy
  ritarda la creazione di un oggetto costoso.

protection proxy
  controlla autorizzazioni.

cache proxy
  evita chiamate remote se ha già il dato.
```

---

# Proxy per chiamate remote

Immaginiamo un’interfaccia applicativa:

```csharp id="mdb6wm"
public interface IClienteRemoteApi
{
    Task<ClienteDto?> TrovaAsync(
        int id,
        CancellationToken cancellationToken);
}
```

Proxy HTTP:

```csharp id="utvlvn"
public class ClienteRemoteApiProxy : IClienteRemoteApi
{
    private readonly HttpClient httpClient;

    public ClienteRemoteApiProxy(HttpClient httpClient)
    {
        this.httpClient = httpClient;
    }

    public async Task<ClienteDto?> TrovaAsync(
        int id,
        CancellationToken cancellationToken)
    {
        HttpResponseMessage response =
            await httpClient.GetAsync(
                $"/api/clienti/{id}",
                cancellationToken);

        if (response.StatusCode == HttpStatusCode.NotFound)
        {
            return null;
        }

        response.EnsureSuccessStatusCode();

        return await response.Content
            .ReadFromJsonAsync<ClienteDto>(
                cancellationToken);
    }
}
```

Registrazione:

```csharp id="ol3nsv"
builder.Services.AddHttpClient<IClienteRemoteApi, ClienteRemoteApiProxy>(
    client =>
    {
        client.BaseAddress = new Uri("https://api.example.com");
    });
```

Uso:

```csharp id="u8c1ss"
public class ClienteSyncService
{
    private readonly IClienteRemoteApi clienteRemoteApi;

    public ClienteSyncService(IClienteRemoteApi clienteRemoteApi)
    {
        this.clienteRemoteApi = clienteRemoteApi;
    }
}
```

Il chiamante lavora con un’interfaccia locale.
Il proxy nasconde la chiamata HTTP.

---

## Proxy con autorizzazione

```csharp id="n8xmb2"
public class AuthorizedDocumentoServiceProxy : IDocumentoService
{
    private readonly IDocumentoService inner;
    private readonly IUserContext userContext;

    public AuthorizedDocumentoServiceProxy(
        IDocumentoService inner,
        IUserContext userContext)
    {
        this.inner = inner;
        this.userContext = userContext;
    }

    public async Task<DocumentoDto?> TrovaAsync(
        int id,
        CancellationToken cancellationToken)
    {
        if (!userContext.HasPermission("Documenti.Lettura"))
        {
            throw new UnauthorizedAccessException();
        }

        return await inner.TrovaAsync(id, cancellationToken);
    }
}
```

Qui il proxy controlla l’accesso.

---

## Proxy vs Adapter

```text id="9zmub9"
Adapter
  cambia interfaccia.

Proxy
  mantiene la stessa interfaccia ma controlla accesso,
  chiamata remota, caching, autorizzazione o lazy loading.
```

Esempio:

```text id="ea3gyl"
Adapter
  traduco LegacyCustomerService in IClienteReader.

Proxy
  implemento IClienteRemoteApi chiamando HTTP dietro le quinte.
```

# Composite

Il **Composite** permette di trattare oggetti singoli e gruppi di oggetti in modo uniforme.

È utile per strutture ad albero.

Esempi:

```text id="bhgswo"
menu
cartelle e file
alberi UI
categorie
organigrammi
componenti grafici
regole composte
permessi composti
```

---

## Esempio: menu

Interfaccia comune:

```csharp id="odf7k8"
public interface IMenuItem
{
    string Titolo { get; }

    void Render(StringBuilder builder, int livello);
}
```

Elemento singolo:

```csharp id="z7u8no"
public class MenuLink : IMenuItem
{
    public string Titolo { get; }
    public string Url { get; }

    public MenuLink(string titolo, string url)
    {
        Titolo = titolo;
        Url = url;
    }

    public void Render(StringBuilder builder, int livello)
    {
        string indent = new(' ', livello * 2);

        builder.AppendLine(
            $"{indent}- {Titolo} ({Url})");
    }
}
```

Elemento composto:

```csharp id="am3g0x"
public class MenuGroup : IMenuItem
{
    private readonly List<IMenuItem> children = [];

    public string Titolo { get; }

    public MenuGroup(string titolo)
    {
        Titolo = titolo;
    }

    public void Add(IMenuItem item)
    {
        children.Add(item);
    }

    public void Render(StringBuilder builder, int livello)
    {
        string indent = new(' ', livello * 2);

        builder.AppendLine($"{indent}+ {Titolo}");

        foreach (IMenuItem child in children)
        {
            child.Render(builder, livello + 1);
        }
    }
}
```

Uso:

```csharp id="p4bedx"
MenuGroup root = new("Menu principale");

root.Add(new MenuLink("Home", "/"));

MenuGroup clienti = new("Clienti");
clienti.Add(new MenuLink("Elenco clienti", "/clienti"));
clienti.Add(new MenuLink("Nuovo cliente", "/clienti/nuovo"));

root.Add(clienti);

StringBuilder builder = new();

root.Render(builder, 0);

Console.WriteLine(builder.ToString());
```

Output concettuale:

```text id="jsocwe"
+ Menu principale
  - Home (/)
  + Clienti
    - Elenco clienti (/clienti)
    - Nuovo cliente (/clienti/nuovo)
```

---

## Problema risolto

Senza Composite, il codice dovrebbe distinguere continuamente:

```text id="jxwpy9"
se è un link
  renderizza link

se è un gruppo
  cicla i figli

se è un gruppo dentro un gruppo
  ricorsione speciale
```

Con Composite, entrambi implementano `IMenuItem`.

Il chiamante può trattarli allo stesso modo.

---

## Composite per regole

Composite non serve solo per UI.

Esempio: regole componibili.

```csharp id="zv21en"
public interface ISpecification<T>
{
    bool IsSatisfiedBy(T item);
}
```

Specifica singola:

```csharp id="x4giwh"
public class ClienteAttivoSpecification
    : ISpecification<Cliente>
{
    public bool IsSatisfiedBy(Cliente cliente)
    {
        return cliente.Attivo;
    }
}
```

Composite AND:

```csharp id="oeshd7"
public class AndSpecification<T> : ISpecification<T>
{
    private readonly ISpecification<T> left;
    private readonly ISpecification<T> right;

    public AndSpecification(
        ISpecification<T> left,
        ISpecification<T> right)
    {
        this.left = left;
        this.right = right;
    }

    public bool IsSatisfiedBy(T item)
    {
        return left.IsSatisfiedBy(item) &&
               right.IsSatisfiedBy(item);
    }
}
```

Questo permette di costruire regole complesse combinando regole più semplici.

# Bridge

Il **Bridge** separa un’astrazione dalla sua implementazione, in modo che entrambe possano variare indipendentemente.

È meno usato quotidianamente rispetto ad Adapter, Facade e Decorator, ma è molto utile quando hai due dimensioni di variabilità.

Esempio:

```text id="m397zd"
Tipo notifica:
  conferma ordine
  reset password
  avviso sicurezza

Canale di invio:
  email
  SMS
  push
```

Senza Bridge, potresti creare molte classi:

```text id="i4tm4q"
EmailConfermaOrdine
SmsConfermaOrdine
PushConfermaOrdine
EmailResetPassword
SmsResetPassword
PushResetPassword
EmailAvvisoSicurezza
SmsAvvisoSicurezza
PushAvvisoSicurezza
```

Cresce combinatoriamente.

---

## Esempio Bridge

Implementazione variabile: canale.

```csharp id="tocu0v"
public interface INotificationChannel
{
    Task SendAsync(
        string destinatario,
        string titolo,
        string messaggio,
        CancellationToken cancellationToken);
}
```

Canale email:

```csharp id="n9n8c9"
public class EmailNotificationChannel : INotificationChannel
{
    public Task SendAsync(
        string destinatario,
        string titolo,
        string messaggio,
        CancellationToken cancellationToken)
    {
        Console.WriteLine(
            $"EMAIL a {destinatario}: {titolo} - {messaggio}");

        return Task.CompletedTask;
    }
}
```

Canale SMS:

```csharp id="hmh6jp"
public class SmsNotificationChannel : INotificationChannel
{
    public Task SendAsync(
        string destinatario,
        string titolo,
        string messaggio,
        CancellationToken cancellationToken)
    {
        Console.WriteLine(
            $"SMS a {destinatario}: {messaggio}");

        return Task.CompletedTask;
    }
}
```

Astrazzione: notifica.

```csharp id="mlsb34"
public abstract class Notification
{
    private readonly INotificationChannel channel;

    protected Notification(INotificationChannel channel)
    {
        this.channel = channel;
    }

    public async Task SendAsync(
        string destinatario,
        CancellationToken cancellationToken)
    {
        string titolo = GetTitle();
        string messaggio = GetMessage();

        await channel.SendAsync(
            destinatario,
            titolo,
            messaggio,
            cancellationToken);
    }

    protected abstract string GetTitle();

    protected abstract string GetMessage();
}
```

Notifica concreta:

```csharp id="c9ncna"
public class ConfermaOrdineNotification : Notification
{
    private readonly int ordineId;

    public ConfermaOrdineNotification(
        int ordineId,
        INotificationChannel channel)
        : base(channel)
    {
        this.ordineId = ordineId;
    }

    protected override string GetTitle()
    {
        return "Ordine confermato";
    }

    protected override string GetMessage()
    {
        return $"Il tuo ordine {ordineId} è stato confermato.";
    }
}
```

Uso:

```csharp id="ie2n3z"
INotificationChannel channel =
    new EmailNotificationChannel();

Notification notification =
    new ConfermaOrdineNotification(
        ordineId: 123,
        channel);

await notification.SendAsync(
    "cliente@example.com",
    cancellationToken);
```

Ora puoi variare indipendentemente:

```text id="88f2zw"
tipo di notifica
canale di invio
```

---

## Bridge vs Strategy

Bridge e Strategy possono sembrare simili.

```text id="flm8aj"
Strategy
  cambia un algoritmo o comportamento.

Bridge
  separa due gerarchie che possono evolvere indipendentemente.
```

Nel caso delle notifiche:

```text id="0to0ag"
gerarchia Notification
  tipo di messaggio

gerarchia INotificationChannel
  modo di invio
```

Questa è una situazione da Bridge.

---

# Confronto rapido

```text id="dbx1zq"
Adapter
  rende compatibile un’interfaccia diversa.

Facade
  semplifica un sottosistema complesso.

Decorator
  aggiunge comportamento mantenendo la stessa interfaccia.

Proxy
  controlla accesso, chiamata remota, lazy loading o protezione.

Composite
  tratta oggetti singoli e gruppi in modo uniforme.

Bridge
  separa astrazione e implementazione variabile.
```

---

# Esempio pratico: integrazione legacy completa

Scenario:

```text id="ddwxxs"
Un’app ASP.NET Core moderna deve leggere clienti da un sistema legacy,
arricchirli con dati remoti, applicare cache e mostrare un menu gerarchico.
```

Possibili pattern:

```text id="1cx2aq"
Adapter
  LegacyClienteReaderAdapter traduce il servizio legacy.

Proxy
  ClienteRemoteApiProxy nasconde chiamata HTTP.

Decorator
  CachedClienteReaderDecorator aggiunge cache.

Facade
  ClienteProfileFacade coordina legacy + API remota + mapping.

Composite
  MenuGroup/MenuLink rappresenta il menu UI.

Bridge
  Notifiche cliente separate dal canale email/SMS/push.
```

I pattern non sono “etichette da appiccicare”.
Sono strumenti che collaborano quando il problema lo richiede.

# Pattern strutturali e Dependency Injection

Molti pattern strutturali si integrano molto bene con DI.

## Adapter

```csharp id="8ufwzn"
builder.Services.AddScoped<IClienteReader, LegacyClienteReaderAdapter>();
```

## Proxy

```csharp id="evvucl"
builder.Services.AddHttpClient<IClienteRemoteApi, ClienteRemoteApiProxy>();
```

## Decorator

```csharp id="h94jxa"
builder.Services.AddScoped<ClienteService>();

builder.Services.AddScoped<IClienteService>(sp =>
{
    ClienteService inner =
        sp.GetRequiredService<ClienteService>();

    ILogger<LoggingClienteServiceDecorator> logger =
        sp.GetRequiredService<ILogger<LoggingClienteServiceDecorator>>();

    return new LoggingClienteServiceDecorator(inner, logger);
});
```

## Facade

```csharp id="p196mo"
builder.Services.AddScoped<DocumentoFacade>();
```

DI aiuta a comporre gli oggetti senza disseminare `new` ovunque.

# Errori comuni

## Usare Adapter come semplice rinomina

Sospetto:

```text id="e0wsxv"
una classe adapter che chiama un metodo con lo stesso nome
senza traduzione, isolamento o valore aggiunto
```

L’adapter deve realmente adattare interfacce o modelli diversi.

---

## Facade che diventa God Object

Pericoloso:

```text id="qgd0lo"
ApplicationFacade
  gestisce clienti
  ordini
  fatture
  email
  report
  import
  export
```

Una facade deve semplificare un sottosistema coerente, non diventare il contenitore di tutta l’applicazione.

---

## Decorator con troppe responsabilità

Sbagliato:

```text id="psh64c"
LoggingCachingRetryAuthorizationDecorator
```

Meglio decorator piccoli e componibili.

---

## Proxy che nasconde troppo

Un proxy remoto può far sembrare locale ciò che non lo è.

Attenzione:

```text id="txsxl0"
latenza
timeout
errori di rete
retry
idempotenza
autenticazione
serializzazione
versionamento API
```

Una chiamata remota non è mai davvero uguale a una chiamata in memoria.

---

## Composite troppo complicato

Se la struttura non è realmente gerarchica, Composite può essere eccessivo.

Non serve per una semplice lista piatta.

---

## Bridge usato dove basta Strategy

Se hai una sola dimensione di variabilità, spesso Strategy è più semplice.

Bridge serve quando ci sono due gerarchie che devono variare indipendentemente.

---

# Linee guida pratiche

```text id="rfuora"
Usa Adapter
  per isolare sistemi legacy, SDK esterni e formati incompatibili.

Usa Facade
  per semplificare sottosistemi complessi e ridurre duplicazione.

Usa Decorator
  per aggiungere logging, cache, retry, metriche o autorizzazione
  senza modificare il servizio principale.

Usa Proxy
  per rappresentare servizi remoti, lazy loading o controlli di accesso.

Usa Composite
  per menu, alberi, categorie, cartelle e strutture gerarchiche.

Usa Bridge
  quando astrazione e implementazione devono variare indipendentemente.

Non usare pattern per nome
  usali quando riducono accoppiamento o rendono il modello più chiaro.

Integra i pattern con DI
  per comporre oggetti in modo esplicito e testabile.

Mantieni classi piccole
  soprattutto decorator, adapter e facade.

Ricorda i costi nascosti
  proxy remoti, cache, retry e facade complesse possono introdurre bug.
```

# Riepilogo

In questo capitolo abbiamo studiato i principali design pattern strutturali.

Abbiamo visto:

```text id="dhxbc3"
Adapter
Facade
Decorator
Proxy
Composite
Bridge
```

La distinzione principale è questa:

```text id="7h3bde"
Adapter
  traduce un’interfaccia incompatibile.

Facade
  offre un accesso semplice a un sottosistema complesso.

Decorator
  aggiunge comportamento a un oggetto esistente.

Proxy
  controlla l’accesso a un oggetto o servizio.

Composite
  modella strutture ad albero trattando singoli e gruppi allo stesso modo.

Bridge
  separa due dimensioni di variabilità.
```

Abbiamo visto esempi pratici in .NET:

```text id="hyp4f8"
Adapter per sistemi legacy
Decorator con Dependency Injection
Proxy per chiamate remote
Composite per menu e alberi UI
Bridge per notifiche e canali di invio
```

La regola principale è questa:

```text id="qic75o"
I pattern strutturali servono a organizzare dipendenze e collaborazioni.
Sono utili quando rendono esplicito un confine,
nascondono una complessità reale o permettono di comporre comportamenti
senza accoppiare tutto direttamente.
```

Nel prossimo capitolo possiamo affrontare i pattern comportamentali: Strategy, Command, Observer, Mediator, State, Template Method e Chain of Responsibility, collegandoli a scenari reali come validazione, workflow, eventi, UI, MediatR e pipeline applicative.

# 08.07 — Design pattern comportamentali

Dopo i pattern creazionali e strutturali, arriviamo ai **design pattern comportamentali**.

Questi pattern rispondono a una domanda diversa:

```text id="l7n5wp"
Come organizzo il comportamento degli oggetti,
il flusso delle operazioni, le decisioni, gli eventi
e la collaborazione tra componenti?
```

Il punto centrale è questo:

```text id="j7ftyw"
I pattern comportamentali servono a separare decisioni, algoritmi,
messaggi, stati e flussi di elaborazione, evitando classi enormi
piene di if, switch e dipendenze incrociate.
```

Sono particolarmente importanti nelle applicazioni moderne perché ritornano continuamente in:

```text id="az07qu"
validazione
workflow
pipeline
eventi
MediatR
SignalR
UI
servizi applicativi
strategie di calcolo
gestione stati
notifiche
autorizzazione
```

---

# Strategy

Il pattern **Strategy** permette di definire una famiglia di algoritmi intercambiabili e scegliere quale usare senza modificare il codice chiamante.

Il problema tipico è questo:

```text id="ph6stc"
Ho più modi per fare la stessa cosa
e non voglio riempire il codice di if/switch.
```

Esempi:

```text id="x7x9sc"
calcolo sconto standard/VIP/promozionale
spedizione standard/express/ritiro in sede
esportazione PDF/Excel/CSV
validazione per tipo documento
algoritmo di scoring
calcolo commissioni
```

---

## Esempio senza Strategy

```csharp id="ydn5mj"
public decimal CalcolaSconto(
    Cliente cliente,
    decimal importo,
    TipoSconto tipoSconto)
{
    if (tipoSconto == TipoSconto.Standard)
    {
        return importo * 0.05m;
    }

    if (tipoSconto == TipoSconto.Vip)
    {
        return importo * 0.15m;
    }

    if (tipoSconto == TipoSconto.Promozionale)
    {
        return importo * 0.20m;
    }

    return 0m;
}
```

Funziona, ma cresce male.

Ogni nuova strategia modifica lo stesso metodo.

---

## Strategy con interfaccia

```csharp id="suo67g"
public interface IScontoStrategy
{
    TipoSconto Tipo { get; }

    decimal Calcola(
        Cliente cliente,
        decimal importo);
}
```

Strategia standard:

```csharp id="9rlgzx"
public class ScontoStandardStrategy : IScontoStrategy
{
    public TipoSconto Tipo => TipoSconto.Standard;

    public decimal Calcola(
        Cliente cliente,
        decimal importo)
    {
        return importo * 0.05m;
    }
}
```

Strategia VIP:

```csharp id="382l31"
public class ScontoVipStrategy : IScontoStrategy
{
    public TipoSconto Tipo => TipoSconto.Vip;

    public decimal Calcola(
        Cliente cliente,
        decimal importo)

        return importo * 0.15m;
    }
}
```

Strategia promozionale:

```csharp id="83v5r3"
public class ScontoPromozionaleStrategy : IScontoStrategy
{
    public TipoSconto Tipo => TipoSconto.Promozionale;

    public decimal Calcola(
        Cliente cliente,

```

## Resolver delle strategie

```csharp id="mfjpsr"
public class ScontoStrategyResolver
{
    private readonly IEnumerable<IScontoStrategy> strategies;

    public ScontoStrategyResolver(
        IEnumerable<IScontoStrategy> strategies)
    {
        this.strategies = strategies;
    }

    public IScontoStrategy Resolve(TipoSconto tipo)
    {
        return strategies.FirstOrDefault(s => s.Tipo == tipo)
            ?? throw new InvalidOperationException(
                $"Strategia sconto non supportata: {tipo}");
    }
}
```

Uso:

```csharp id="f0ebgp"
public class CalcolatoreSconto
{
    private readonly ScontoStrategyResolver resolver;

    public CalcolatoreSconto(ScontoStrategyResolver resolver)
    {
        this.resolver = resolver;
    }

    public decimal Calcola(
        Cliente cliente,
        decimal importo,
        TipoSconto tipoSconto)
    {
        IScontoStrategy strategy =
            resolver.Resolve(tipoSconto);

        return strategy.Calcola(cliente, importo);
    }
}
```

Registrazione DI:

```csharp id="bapbav"
builder.Services.AddScoped<IScontoStrategy, ScontoStandardStrategy>();
builder.Services.AddScoped<IScontoStrategy, ScontoVipStrategy>();
builder.Services.AddScoped<IScontoStrategy, ScontoPromozionaleStrategy>();
builder.Services.AddScoped<ScontoStrategyResolver>();
builder.Services.AddScoped<CalcolatoreSconto>();
```

---

## Quando usare Strategy

Usala quando:

```text id="bn0v5l"
hai algoritmi intercambiabili
aggiungi spesso nuove varianti
vuoi evitare switch lunghi
vuoi rispettare Open/Closed Principle
vuoi testare ogni algoritmo separatamente
```

Non usarla per due righe di codice che non cambieranno mai.

# Command

Il pattern **Command** incapsula una richiesta come oggetto.

Invece di chiamare direttamente un metodo con molti parametri, rappresentiamo l’azione con un oggetto.

Esempi:

```text id="k0n6f4"
CreaOrdineCommand
ConfermaOrdineCommand
AnnullaPrenotazioneCommand
InviaEmailCommand
ImportaClientiCommand
```

Un command esprime un’intenzione.

---

## Command semplice

```csharp id="g1qrwd"
public record ConfermaOrdineCommand(
    int OrdineId);
```

Handler:

```csharp id="v6mx61"
public class ConfermaOrdineHandler
{
    private readonly IOrdineRepository ordineRepository;
    private readonly IUnitOfWork unitOfWork;

    public ConfermaOrdineHandler(
        IOrdineRepository ordineRepository,
        IUnitOfWork unitOfWork)
    {
        this.ordineRepository = ordineRepository;
        this.unitOfWork = unitOfWork;
    }

    public async Task HandleAsync(
        ConfermaOrdineCommand command,
        CancellationToken cancellationToken)
    {
        Ordine ordine = await ordineRepository.GetByIdAsync(
            command.OrdineId,
            cancellationToken)
            ?? throw new NotFoundException("Ordine non trovato.");

        ordine.Conferma();

        await unitOfWork.SaveChangesAsync(cancellationToken);
    }
}
```

---

## Perché Command è utile

```text id="vtpcek"
dà un nome all’azione
rende il caso d’uso esplicito
facilita logging e validazione
permette pipeline behavior
isola handler e test
rende più semplice CQRS
```

Un command può essere validato, loggato, autorizzato e misurato prima dell’esecuzione.

---

## Command e undo

Nel pattern GoF originale, Command è spesso collegato anche ad annullamento e redo.

Esempio concettuale:

```csharp id="yfi45x"
public interface ICommand
{
    void Execute();

    void Undo();
}
```

In applicazioni business moderne, però, l’undo spesso non è un semplice “tornare indietro”.
Può richiedere:

```text id="ak3vfg"
evento compensativo
storno
annullamento ordine
nota di credito
ripristino stato
audit trail
```

Quindi nel business non bisogna confondere `Undo()` tecnico con compensazione di dominio.

# Observer

Il pattern **Observer** permette a un oggetto di notificare altri oggetti quando cambia qualcosa.

Il problema tipico è:

```text id="t95hiq"
Un componente produce un evento
e più componenti devono reagire
senza che il produttore li conosca direttamente.
```

Esempi:

```text id="lst43n"
ordine confermato
cliente creato
documento approvato
file importato
messaggio ricevuto
valore UI cambiato
```

---

## Observer con eventi C#

C# ha supporto nativo agli eventi.

```csharp id="e5gl01"
public class OrdineService
{
    public event EventHandler<OrdineConfermatoEventArgs>? OrdineConfermato;

    public void ConfermaOrdine(int ordineId)
    {
        // logica conferma

        OrdineConfermato?.Invoke(
            this,
            new OrdineConfermatoEventArgs(ordineId));
    }
}
```

Event args:

```csharp id="vpbgr3"
public class OrdineConfermatoEventArgs : EventArgs
{
    public int OrdineId { get; }

    public OrdineConfermatoEventArgs(int ordineId)
    {
        OrdineId = ordineId;
    }
}
```

Subscriber:

```csharp id="gmp6y1"
ordineService.OrdineConfermato += (sender, args) =>
{
    Console.WriteLine(
        $"Ordine confermato: {args.OrdineId}");
};
```

---

## Observer in UI

Nelle UI .NET il pattern Observer è ovunque.

Esempi:

```text id="yu6vpz"
eventi Button.Click
INotifyPropertyChanged in WPF
EventCallback in Blazor
eventi custom nei componenti
```

In WPF:

```csharp id="ehtcgx"
public class ClienteViewModel : INotifyPropertyChanged
{
    private string nome = "";

    public string Nome
    {
        get => nome;
        set
        {
            if (nome == value)
            {
                return;
            }

            nome = value;

            PropertyChanged?.Invoke(
                this,
                new PropertyChangedEventArgs(nameof(Nome)));
        }
    }

    public event PropertyChangedEventHandler? PropertyChanged;
}
```

La view osserva il ViewModel e si aggiorna quando cambia una proprietà.

## Domain events

Nel dominio possiamo usare eventi per segnalare qualcosa che è accaduto.

```csharp id="aj9oyk"
public record OrdineConfermatoDomainEvent(
    int OrdineId,
    int ClienteId);
```

L’aggregate può raccogliere eventi:

```csharp id="6vmwi6"
public class Ordine
{
    private readonly List<object> domainEvents = [];

    public IReadOnlyCollection<object> DomainEvents =>
        domainEvents.AsReadOnly();

    public int Id { get; private set; }
    public int ClienteId { get; private set; }
    public StatoOrdine Stato { get; private set; }

    public void Conferma()
    {
        if (Stato != StatoOrdine.Bozza)
        {
            throw new DomainException(
                "L'ordine non è in stato bozza.");
        }

        Stato = StatoOrdine.Confermato;

        domainEvents.Add(
            new OrdineConfermatoDomainEvent(
                Id,
                ClienteId));
    }

    public void ClearDomainEvents()
    {
        domainEvents.Clear();
    }
}
```

Poi l’application layer o infrastructure può pubblicarli dopo il salvataggio.

---

## Quando usare Observer

Usalo quando:

```text id="xc0a2l"
più componenti devono reagire allo stesso evento
il produttore non deve conoscere i destinatari
vuoi disaccoppiare azione principale da reazioni secondarie
hai UI reattive o notifiche
```

Attenzione però:

```text id="54zklw"
troppi eventi rendono il flusso difficile da seguire
gli eventi sincroni possono propagare errori
gli eventi asincroni richiedono gestione affidabile
```

# Mediator

Il pattern **Mediator** centralizza la comunicazione tra oggetti, evitando che si conoscano direttamente.

Il problema tipico:

```text id="jn8jll"
Molti componenti si chiamano tra loro direttamente,
creando una rete di dipendenze difficile da gestire.
```

Con un mediator:

```text id="hfx32b"
un componente invia una richiesta
il mediator la consegna all’handler corretto
```

---

## Mediator con MediatR

Abbiamo già visto MediatR con Command e Query.

Endpoint:

```csharp id="lq19ih"
await sender.Send(
    new ConfermaOrdineCommand(id),
    cancellationToken);
```

Il chiamante non conosce `ConfermaOrdineHandler`.

Questo riduce l’accoppiamento diretto.

---

## Mediator non deve nascondere tutto

Rischio:

```text id="kclyoa"
uso mediator per qualunque chiamata
handler che chiamano altri handler
flusso applicativo invisibile
catene difficili da debuggare
```

Esempio sospetto:

```csharp id="ag3ol6"
public class CreaOrdineHandler
{
    private readonly ISender sender;

    public async Task Handle(...)
    {
        await sender.Send(new VerificaClienteCommand(...));
        await sender.Send(new CalcolaScontoCommand(...));
        await sender.Send(new InviaEmailCommand(...));
    }
}
```

A volte ha senso.
Ma spesso rende il flusso meno chiaro.

Regola pratica:

```text id="0e3zmc"
Usa Mediator per disaccoppiare ingresso e handler.
Non usarlo per nascondere flussi che dovrebbero essere espliciti.
```

# Template Method

Il pattern **Template Method** definisce lo scheletro di un algoritmo in una classe base, lasciando alcuni passaggi alle sottoclassi.

Il problema tipico:

```text id="rnkewf"
Ho più processi simili con una struttura comune,
ma alcuni passaggi cambiano.
```

Esempi:

```text id="h9s83r"
import file CSV/Excel/XML
generazione report PDF/Excel
validazione documenti diversi
workflow con passi comuni
elaborazione batch
```

---

## Esempio: import file

Classe base:

```csharp id="hcawxl"
public abstract class FileImportTemplate
{
    public async Task ImportaAsync(
        string path,
        CancellationToken cancellationToken)
    {
        await VerificaFileAsync(path, cancellationToken);

        string contenuto = await LeggiFileAsync(
            path,
            cancellationToken);

        await ValidaContenutoAsync(
            contenuto,
            cancellationToken);

        await ProcessaContenutoAsync(
            contenuto,
            cancellationToken);

        await DopoImportAsync(
            path,
            cancellationToken);
    }

    protected virtual Task VerificaFileAsync(
        string path,
        CancellationToken cancellationToken)
    {
        if (!File.Exists(path))
        {
            throw new FileNotFoundException(
                "File non trovato.",
                path);
        }

        return Task.CompletedTask;
    }

    protected virtual Task<string> LeggiFileAsync(
        string path,
        CancellationToken cancellationToken)
    {
        return File.ReadAllTextAsync(path, cancellationToken);
    }

    protected abstract Task ValidaContenutoAsync(
        string contenuto,
        CancellationToken cancellationToken);

    protected abstract Task ProcessaContenutoAsync(
        string contenuto,
        CancellationToken cancellationToken);

    protected virtual Task DopoImportAsync(
        string path,
        CancellationToken cancellationToken)
    {
        return Task.CompletedTask;
    }
}
```

Implementazione CSV:

```csharp id="hvmtb0"
public class CsvClientiImport : FileImportTemplate
{
    protected override Task ValidaContenutoAsync(
        string contenuto,
        CancellationToken cancellationToken)
    {
        if (!contenuto.Contains(';'))
        {
            throw new InvalidOperationException(
                "Il file CSV non sembra valido.");
        }

        return Task.CompletedTask;
    }

    protected override Task ProcessaContenutoAsync(
        string contenuto,
        CancellationToken cancellationToken)
    {
        // parsing e import clienti
        return Task.CompletedTask;
    }
}
```

Uso:

```csharp id="hdrcpp"
FileImportTemplate import = new CsvClientiImport();

await import.ImportaAsync(
    "clienti.csv",
    cancellationToken);
```

---

## Template Method vs Strategy

Sono simili ma non uguali.

```text id="a6hsz5"
Template Method
  ereditarietà
  algoritmo base in una classe astratta
  sottoclassi ridefiniscono alcuni passaggi

Strategy
  composizione
  algoritmo intero intercambiabile tramite interfaccia
```

Regola moderna:

```text id="cbmfqs"
Preferisci Strategy quando vuoi composizione flessibile.
Usa Template Method quando hai davvero uno scheletro comune stabile.
```

# Chain of Responsibility

La **Chain of Responsibility** passa una richiesta attraverso una catena di handler.

Ogni handler può:

```text id="ez9ldp"
gestire la richiesta
modificarla
validarla
interrompere il flusso
passarla al prossimo
eseguire logica prima/dopo
```

È il pattern alla base di molte pipeline.

Esempi:

```text id="e32629"
middleware ASP.NET Core
pipeline MediatR
validazione
autorizzazione
logging
retry
rate limiting
filtri endpoint
```

---

## Esempio concettuale

Interfaccia:

```csharp id="f47862"
public interface IRequestHandlerStep<TRequest>
{
    Task HandleAsync(
        TRequest request,
        Func<Task> next);
}
```

Logging step:

```csharp id="ao9sq4"
public class LoggingStep<TRequest>
    : IRequestHandlerStep<TRequest>
{
    private readonly ILogger<LoggingStep<TRequest>> logger;

    public LoggingStep(ILogger<LoggingStep<TRequest>> logger)
    {
        this.logger = logger;
    }

    public async Task HandleAsync(
        TRequest request,
        Func<Task> next)
    {
        logger.LogInformation(
            "Inizio gestione {RequestType}",
            typeof(TRequest).Name);

        await next();

        logger.LogInformation(
            "Fine gestione {RequestType}",
            typeof(TRequest).Name);
    }
}
```

Validation step:

```csharp id="da8trd"
public class ValidationStep<TRequest>
    : IRequestHandlerStep<TRequest>
{
    private readonly IRequestValidator<TRequest> validator;

    public ValidationStep(
        IRequestValidator<TRequest> validator)
    {
        this.validator = validator;
    }

    public async Task HandleAsync(
        TRequest request,
        Func<Task> next)
    {
        List<string> errors = validator.Validate(request);

        if (errors.Count > 0)
        {
            throw new ValidationException(errors);
        }

        await next();
    }
}
```

---

# Chain of Responsibility nelle pipeline

## ASP.NET Core middleware

La pipeline HTTP di ASP.NET Core è una Chain of Responsibility.

```csharp id="na0sbl"
app.UseAuthentication();
app.UseAuthorization();

app.Use(async (context, next) =>
{
    // prima
    await next();
    // dopo
});

app.MapControllers();
```

Ogni middleware può:

```text id="wpp98z"
gestire la request
passarla al prossimo
interrompere la pipeline
modificare response
```

---

## Pipeline behavior MediatR

Anche i behavior MediatR sono una forma di Chain of Responsibility.

```csharp id="d2zgxc"
public class LoggingBehavior<TRequest, TResponse>
    : IPipelineBehavior<TRequest, TResponse>
    where TRequest : notnull
{
    public async Task<TResponse> Handle(
        TRequest request,
        RequestHandlerDelegate<TResponse> next,
        CancellationToken cancellationToken)
    {
        // prima

        TResponse response = await next();

        // dopo

        return response;
    }
}
```

Pipeline tipica:

```text id="ek6ys0"
LoggingBehavior
  ↓
ValidationBehavior
  ↓
AuthorizationBehavior
  ↓
TransactionBehavior
  ↓
Handler
```

---

## Quando usare Chain of Responsibility

Usala quando:

```text id="sttox5"
hai passaggi sequenziali
alcuni passaggi possono bloccare la richiesta
vuoi separare cross-cutting concerns
vuoi comporre una pipeline
vuoi evitare un metodo enorme con molte responsabilità
```

Non usarla se il flusso è semplice e lineare: rendere tutto pipeline può complicare inutilmente.

# State

Il pattern **State** permette a un oggetto di cambiare comportamento in base al suo stato interno.

Il problema tipico è questo:

```text id="pjfsrm"
Una classe ha molti if/switch sullo stato corrente.
Ogni stato permette alcune operazioni e ne vieta altre.
```

Esempi:

```text id="gtg1dm"
ordine: bozza, confermato, spedito, annullato
documento: bozza, approvato, archiviato
ticket: aperto, in lavorazione, risolto, chiuso
prenotazione: richiesta, confermata, annullata
pagamento: autorizzato, catturato, rimborsato
```

---

## Esempio senza State

```csharp id="p8ckl3"
public class Ordine
{
    public StatoOrdine Stato { get; private set; }

    public void Conferma()
    {
        if (Stato != StatoOrdine.Bozza)
        {
            throw new DomainException(
                "Solo un ordine in bozza può essere confermato.");
        }

        Stato = StatoOrdine.Confermato;
    }

    public void Spedisci()
    {
        if (Stato != StatoOrdine.Confermato)
        {
            throw new DomainException(
                "Solo un ordine confermato può essere spedito.");
        }

        Stato = StatoOrdine.Spedito;
    }

    public void Annulla()
    {
        if (Stato == StatoOrdine.Spedito)
        {
            throw new DomainException(
                "Un ordine spedito non può essere annullato.");
        }

        Stato = StatoOrdine.Annullato;
    }
}
```

Per pochi stati va benissimo.

Ma se le regole crescono, la classe può diventare piena di condizioni.

---

## State con classi separate

Interfaccia stato:

```csharp id="o5vpsm"
public interface IOrdineState
{
    StatoOrdine Stato { get; }

    void Conferma(Ordine ordine);

    void Spedisci(Ordine ordine);

    void Annulla(Ordine ordine);
}
```

Ordine:

```csharp id="ee4nqu"
public class Ordine
{
    private IOrdineState state;

    public StatoOrdine Stato => state.Stato;

    public Ordine()
    {
        state = new OrdineBozzaState();
    }

    internal void CambiaStato(IOrdineState nuovoStato)
    {
        state = nuovoStato;
    }

    public void Conferma()
    {
        state.Conferma(this);
    }

    public void Spedisci()
    {
        state.Spedisci(this);
    }

    public void Annulla()
    {
        state.Annulla(this);
    }
}
```

Stato Bozza:

```csharp id="phd3b2"
public class OrdineBozzaState : IOrdineState
{
    public StatoOrdine Stato => StatoOrdine.Bozza;

    public void Conferma(Ordine ordine)
    {
        ordine.CambiaStato(new OrdineConfermatoState());
    }

    public void Spedisci(Ordine ordine)
    {
        throw new DomainException(
            "Un ordine in bozza non può essere spedito.");
    }

    public void Annulla(Ordine ordine)
    {
        ordine.CambiaStato(new OrdineAnnullatoState());
    }
}
```

Stato Confermato:

```csharp id="guaeb1"
public class OrdineConfermatoState : IOrdineState
{
    public StatoOrdine Stato => StatoOrdine.Confermato;

    public void Conferma(Ordine ordine)
    {
        throw new DomainException(
            "L'ordine è già confermato.");
    }

    public void Spedisci(Ordine ordine)
    {
        ordine.CambiaStato(new OrdineSpeditoState());
    }

    public void Annulla(Ordine ordine)
    {
        ordine.CambiaStato(new OrdineAnnullatoState());
    }
}
```

Stato finale:

```csharp id="p7jl9e"
public class OrdineSpeditoState : IOrdineState
{
    public StatoOrdine Stato => StatoOrdine.Spedito;

    public void Conferma(Ordine ordine)
    {
        throw new DomainException(
            "Un ordine spedito non può essere riconfermato.");
    }

    public void Spedisci(Ordine ordine)
    {
        throw new DomainException(
            "L'ordine è già spedito.");
    }

    public void Annulla(Ordine ordine)
    {
        throw new DomainException(
            "Un ordine spedito non può essere annullato.");
    }
}
```

---

## Quando usare State

Usalo quando:

```text id="uegmj5"
hai molti stati
ogni stato ha regole diverse
le transizioni sono importanti
gli if/switch sullo stato stanno crescendo
vuoi testare ogni stato separatamente
```

Non usarlo se hai solo due o tre stati semplici.

Un enum con metodi nel dominio può essere sufficiente.

# Confronto tra pattern comportamentali

```text id="r45zzr"
Strategy
  sceglie un algoritmo intercambiabile.

Command
  rappresenta un’azione come oggetto.

Observer
  notifica eventi a più destinatari.

Mediator
  disaccoppia mittente e handler.

Template Method
  definisce uno scheletro di algoritmo con passi personalizzabili.

Chain of Responsibility
  passa una richiesta attraverso una pipeline di handler.

State
  cambia comportamento in base allo stato interno.
```

---

# Esempio integrato: conferma ordine

Una singola funzionalità può usare più pattern.

Scenario:

```text id="ahw0y3"
Confermare un ordine.
```

Possibili pattern:

```text id="4upft3"
Command
  ConfermaOrdineCommand

Mediator
  MediatR invia il command all’handler

Chain of Responsibility
  pipeline di validazione, logging e transazione

State
  Ordine permette Conferma solo se è in Bozza

Observer
  OrdineConfermatoDomainEvent notifica reazioni

Strategy
  calcolo sconto o politica commerciale

Template Method
  eventuale workflow di import/conferma con passaggi comuni
```

I pattern non vivono isolati.
Spesso si combinano nel design reale.

# Errori comuni

## Strategy per tutto

Sbagliato:

```text id="1by457"
Creo una strategia per ogni if.
```

Se la logica è semplice e stabile, un `if` è più leggibile.

---

## Command senza intenzione

Sbagliato:

```csharp id="fgcma3"
public record OrdineCommand(...);
```

Meglio:

```csharp id="b4ibp1"
public record ConfermaOrdineCommand(int OrdineId);
public record AnnullaOrdineCommand(int OrdineId);
```

Il command deve raccontare l’azione.

---

## Observer caotico

Troppi eventi possono rendere il flusso invisibile.

Sintomo:

```text id="zlwlyy"
succede qualcosa
parte un evento
che scatena altri eventi
che modificano dati
ma nessuno sa più dove avviene il cambiamento
```

Gli eventi vanno documentati e gestiti con disciplina.

---

## Mediator come nascondiglio

Sospetto:

```text id="8ybbqm"
handler che chiamano altri handler senza flusso chiaro
```

Il mediator non deve diventare un modo per nascondere dipendenze.

---

## Template Method troppo rigido

L’ereditarietà può rendere difficile cambiare comportamento dinamicamente.

Se le varianti devono essere intercambiabili a runtime, Strategy è spesso meglio.

---

## Pipeline troppo lunga

Una Chain of Responsibility con troppi step può diventare difficile da debuggare.

Ogni behavior deve avere uno scopo chiaro.

---

## State per workflow semplice

Se hai solo:

```text id="z43j17"
Attivo / Non attivo
```

probabilmente non serve il pattern State.

Un enum e due metodi chiari bastano.

---

# Linee guida pratiche

```text id="zcuh7u"
Usa Strategy
  per algoritmi intercambiabili e varianti di comportamento.

Usa Command
  per rendere esplicite le azioni applicative.

Usa Observer
  per notificare eventi senza accoppiare produttore e destinatari.

Usa Mediator
  per disaccoppiare request e handler, soprattutto con CQRS.

Usa Template Method
  quando hai uno scheletro di algoritmo stabile.

Usa Chain of Responsibility
  per pipeline di validazione, logging, autorizzazione e transazioni.

Usa State
  quando il comportamento cambia molto in base allo stato.

Non usare pattern meccanicamente
  usali quando migliorano chiarezza, testabilità o flessibilità.

Preferisci nomi di dominio
  ConfermaOrdineCommand è meglio di GenericCommand.

Mantieni visibile il flusso
  troppa astrazione rende il codice difficile da seguire.
```

# Riepilogo

In questo capitolo abbiamo studiato i principali design pattern comportamentali.

Abbiamo visto:

```text id="ha2xiz"
Strategy
Command
Observer
Mediator
Template Method
Chain of Responsibility
State
```

Abbiamo collegato i pattern a scenari concreti:

```text id="k69k76"
Strategy per algoritmi intercambiabili
Command con MediatR
Observer con eventi
Chain of Responsibility nelle pipeline
State per workflow con stati e transizioni
Template Method per processi con scheletro comune
Mediator per disaccoppiare mittente e handler
```

La distinzione principale è questa:

```text id="3otrg6"
Strategy
  quale algoritmo uso?

Command
  quale azione devo eseguire?

Observer
  chi deve reagire a un evento?

Mediator
  come evito dipendenze dirette tra mittente e handler?

Template Method
  quali passi comuni ha questo processo?

Chain of Responsibility
  quali passaggi attraversa questa richiesta?

State
  come cambia il comportamento in base allo stato?
```

La regola principale è questa:

```text id="6r7tog"
I pattern comportamentali servono a rendere espliciti flussi,
decisioni, eventi e stati. Se li usi bene, il codice racconta meglio
il comportamento del sistema. Se li usi per abitudine, lo nasconde.
```

Con questo capitolo il lettore ha una base solida per riconoscere i pattern non come teoria astratta, ma come strumenti pratici per progettare applicazioni web e business più manutenibili.

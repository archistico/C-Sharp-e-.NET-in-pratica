# 14.03 — Checklist di qualità

Questa appendice raccoglie checklist operative da usare durante sviluppo, revisione codice, test, rilascio e manutenzione.

Il punto centrale è questo:

```text id="g4z2h8"
Una checklist non sostituisce l’esperienza,
ma aiuta a non dimenticare controlli importanti
quando il progetto cresce.
```

Le checklist sono divise per area:

```text id="b4f8b1"
codice C#
API
EF Core
sicurezza
performance
deployment
logging/osservabilità
```

---

# Checklist codice C#

## Struttura generale

```text id="prw4bm"
[ ] Il nome di classi, metodi e variabili è chiaro.
[ ] Ogni classe ha una responsabilità comprensibile.
[ ] I metodi non sono troppo lunghi.
[ ] La logica duplicata è stata ridotta.
[ ] Le dipendenze sono esplicite tramite costruttore.
[ ] Non ci sono classi “manager” o “helper” troppo generiche.
[ ] Il codice è organizzato per modulo o responsabilità.
[ ] Il codice pubblico ha una firma stabile e comprensibile.
```

---

## Tipi e nullabilità

```text id="cq9c6w"
[ ] Nullable Reference Types è abilitato.
[ ] `string` e `string?` sono usati correttamente.
[ ] Non viene usato `!` per nascondere problemi senza motivo.
[ ] I parametri obbligatori vengono validati.
[ ] I valori opzionali sono espliciti.
[ ] Non si restituisce `null` dove una lista vuota è più chiara.
[ ] I DTO rappresentano correttamente dati obbligatori e opzionali.
```

Esempio preferibile:

```csharp id="ksazlg"
public IReadOnlyList<LibroListItemDto> Libri { get; init; } = [];
```

invece di:

```csharp id="u3lrfl"
public List<LibroListItemDto>? Libri { get; set; }
```

se la collezione non dovrebbe mai essere `null`.

---

## Eccezioni e risultati

```text id="e38jgm"
[ ] Le eccezioni non sono usate per normali errori di validazione.
[ ] Gli errori attesi usano `Result`, `ValidationProblem` o modelli simili.
[ ] Le eccezioni impreviste vengono loggate.
[ ] I messaggi di errore sono comprensibili.
[ ] Non vengono inghiottite eccezioni con `catch` vuoti.
[ ] I `catch` troppo generici sono giustificati.
```

Da evitare:

```csharp id="ymsbk0"
try
{
    await service.DoWorkAsync();
}
catch
{
}
```

Meglio:

```csharp id="yehs7h"
try
{
    await service.DoWorkAsync();
}
catch (Exception ex)
{
    logger.LogError(ex, "Errore durante l'elaborazione.");
    throw;
}
```

---

## Async

```text id="q3dnfw"
[ ] I metodi asincroni restituiscono `Task` o `Task<T>`.
[ ] Non viene usato `async void`, salvo event handler UI.
[ ] Le chiamate async vengono attese con `await`.
[ ] Non ci sono `.Result` o `.Wait()` su task asincroni.
[ ] I `CancellationToken` vengono propagati.
[ ] I timeout sono gestiti dove necessario.
[ ] Il fire-and-forget è evitato o gestito esplicitamente.
```

Da evitare:

```csharp id="plvfjr"
var result = httpClient.GetStringAsync(url).Result;
```

Meglio:

```csharp id="n8tnk2"
string result = await httpClient.GetStringAsync(
    url,
    cancellationToken);
```

---

## Testabilità

```text id="na24kl"
[ ] La business logic non è nascosta negli endpoint o nella UI.
[ ] I servizi dipendono da interfacce quando serve.
[ ] Le dipendenze esterne sono isolabili nei test.
[ ] Le funzioni pure sono testate direttamente.
[ ] La logica di dominio ha test unitari.
[ ] I casi d’uso principali hanno test.
```

---

# Checklist API

## Endpoint e routing

```text id="ww58vd"
[ ] Gli URL sono coerenti e prevedibili.
[ ] I metodi HTTP sono usati correttamente.
[ ] Gli endpoint sono organizzati per modulo.
[ ] Gli endpoint non contengono business logic complessa.
[ ] Le azioni di dominio sono modellate chiaramente.
[ ] I parametri route e query sono validati.
```

Esempi coerenti:

```text id="a491gf"
GET    /api/catalogo/libri
GET    /api/catalogo/libri/{id}
POST   /api/catalogo/libri
PUT    /api/catalogo/libri/{id}
DELETE /api/catalogo/libri/{id}

POST   /api/documenti/{id}/conferma
POST   /api/documenti/{id}/annulla
```

---

## DTO e contratti

```text id="e9x89x"
[ ] Le API non espongono entity EF Core direttamente.
[ ] Request e response usano DTO espliciti.
[ ] I DTO non contengono logica di dominio.
[ ] I campi obbligatori sono chiari.
[ ] I campi opzionali sono nullable.
[ ] I contratti sono stabili e documentati.
[ ] Le modifiche breaking sono evitate o versionate.
```

---

## Status code

```text id="cfv5mt"
[ ] `200 OK` per letture riuscite.
[ ] `201 Created` per creazioni riuscite.
[ ] `204 No Content` per update/delete senza body.
[ ] `400 Bad Request` per input non valido.
[ ] `404 Not Found` per risorse assenti.
[ ] `409 Conflict` per conflitti applicativi.
[ ] `401 Unauthorized` per utente non autenticato.
[ ] `403 Forbidden` per utente autenticato ma non autorizzato.
[ ] `500 Internal Server Error` solo per errori imprevisti.
```

---

## Validazione

```text id="av1vep"
[ ] La validazione non è solo lato client.
[ ] Le request API sono validate lato server.
[ ] Le regole applicative vengono verificate nel layer Application.
[ ] Le invarianti sono protette nel Domain.
[ ] Gli errori di validazione sono chiari.
[ ] Le validazioni ripetute sono centralizzate.
```

---

## Error handling

```text id="lvunua"
[ ] Gli errori previsti usano un modello coerente.
[ ] Gli errori imprevisti passano da exception handler globale.
[ ] Le response di errore non espongono stack trace.
[ ] Le response di errore non espongono dettagli sensibili.
[ ] `ProblemDetails` o formato equivalente è usato in modo coerente.
```

---

## OpenAPI

```text id="fbsw17"
[ ] OpenAPI è abilitato almeno in Development.
[ ] Gli endpoint hanno nomi chiari.
[ ] Gli endpoint hanno summary/description utili.
[ ] Sono documentati request e response.
[ ] Sono documentati status code principali.
[ ] In Production OpenAPI è disabilitato o protetto.
```

---

## Test API

```text id="qof2qi"
[ ] Esiste almeno un test per endpoint critico.
[ ] Sono testati casi validi.
[ ] Sono testati `400`, `404`, `409` dove rilevanti.
[ ] Sono testati endpoint protetti, se presenti.
[ ] I test usano database isolato o ambiente controllato.
```

---

# Checklist EF Core

## DbContext e mapping

```text id="pr9f3a"
[ ] Il `DbContext` è nello strato Infrastructure.
[ ] Il Domain non dipende da EF Core.
[ ] Le entity sono configurate con Fluent API o attributi coerenti.
[ ] I nomi tabella sono chiari.
[ ] Le chiavi primarie sono definite.
[ ] Le foreign key sono definite.
[ ] Le relazioni sono esplicite.
[ ] I comportamenti di delete sono controllati.
[ ] Le proprietà obbligatorie sono `IsRequired`.
[ ] Le lunghezze massime sono definite dove ha senso.
[ ] Gli indici sono definiti per ricerche frequenti.
```

---

## Query

```text id="zzx9f6"
[ ] Le query di sola lettura usano `AsNoTracking`.
[ ] Le query proiettano in DTO quando possibile.
[ ] Non ci sono `Include` inutili.
[ ] Non vengono caricate intere tabelle per filtrare in memoria.
[ ] Le query pesanti sono controllate con logging SQL o profiler.
[ ] Le query paginabili hanno paginazione.
[ ] I filtri sono applicati lato database.
```

Da evitare:

```csharp id="j7k8ej"
List<Libro> libri = await dbContext.Libri.ToListAsync();
List<Libro> attivi = libri.Where(x => x.Attivo).ToList();
```

Meglio:

```csharp id="ft1s76"
List<LibroListItemDto> libri = await dbContext.Libri
    .AsNoTracking()
    .Where(x => x.Attivo)
    .OrderBy(x => x.Titolo)
    .Select(x => new LibroListItemDto(
        x.Id,
        x.Titolo,
        x.Isbn,
        x.Prezzo,
        x.Attivo))
    .ToListAsync(cancellationToken);
```

---

## Change tracking

```text id="qcfx6k"
[ ] Le entity da modificare sono tracciate.
[ ] Le query read-only non sono tracciate.
[ ] Non si mantiene vivo il DbContext troppo a lungo.
[ ] Non si condivide DbContext tra thread.
[ ] I servizi scoped usano DbContext scoped.
```

---

## Migration

```text id="jsqmac"
[ ] Ogni migration è stata letta prima di applicarla.
[ ] Le migration distruttive sono controllate.
[ ] Gli script SQL sono generati per produzione, se necessario.
[ ] Le migration sono testate in ambiente simile alla produzione.
[ ] Non si applicano migration automatiche in produzione senza una scelta consapevole.
```

---

## Transazioni

```text id="kh8fzr"
[ ] `SaveChanges` singolo basta per operazioni semplici.
[ ] Operazioni multi-step critiche usano transazione esplicita.
[ ] Le transazioni sono brevi.
[ ] Non si eseguono chiamate HTTP esterne dentro transazioni lunghe.
[ ] La numerazione documenti, se progressiva, è transazionale.
```

---

## Test EF Core

```text id="v8xugd"
[ ] I test di integrazione usano SQLite in-memory o database reale di test.
[ ] Non si usa EF InMemory per testare comportamento relazionale.
[ ] Ogni test è isolato.
[ ] Il database di test viene creato e ripulito correttamente.
[ ] Mapping e query principali sono coperti.
```

---

# Checklist sicurezza

## Input e output

```text id="b3brdx"
[ ] Tutto l’input utente è validato.
[ ] Le query SQL usano parametri.
[ ] Non si concatena input utente in SQL.
[ ] L’output HTML è codificato correttamente.
[ ] I file caricati sono validati per tipo, dimensione e percorso.
[ ] Non si accettano path arbitrari dal client.
```

---

## Autenticazione

```text id="g8ya26"
[ ] Gli endpoint sensibili richiedono autenticazione.
[ ] I token sono validati correttamente.
[ ] Scadenza token e refresh sono gestiti.
[ ] Le cookie, se usate, hanno opzioni sicure.
[ ] Il logout invalida correttamente la sessione lato client/server dove necessario.
```

---

## Autorizzazione

```text id="u7r07g"
[ ] Non basta essere autenticati: servono permessi.
[ ] Le policy sono applicate agli endpoint di scrittura.
[ ] La UI può nascondere pulsanti, ma l’API autorizza davvero.
[ ] I controlli di ownership sono implementati dove necessario.
[ ] Non ci sono endpoint amministrativi pubblici per errore.
```

---

## Segreti

```text id="q8dvgb"
[ ] Connection string sensibili non sono nel repository.
[ ] API key e password non sono in appsettings committati.
[ ] I segreti arrivano da variabili ambiente o secret manager.
[ ] I log non stampano segreti.
[ ] Le pipeline CI/CD usano secret protetti.
```

---

## HTTPS e CORS

```text id="gcd4ej"
[ ] HTTPS è usato in produzione.
[ ] HTTP viene rediretto a HTTPS dove appropriato.
[ ] CORS non usa `AllowAnyOrigin` in produzione senza motivo.
[ ] Gli origin ammessi sono espliciti.
[ ] I forwarded headers sono configurati se dietro reverse proxy.
```

---

## Logging sicuro

```text id="c07c2q"
[ ] Non vengono loggate password.
[ ] Non vengono loggati token.
[ ] Non vengono loggati dati personali non necessari.
[ ] Gli errori restituiti al client non contengono stack trace.
[ ] I log tecnici sono disponibili solo a personale autorizzato.
```

---

## Dipendenze

```text id="mln3xa"
[ ] I pacchetti NuGet vulnerabili sono controllati.
[ ] I pacchetti obsoleti sono valutati.
[ ] Le immagini Docker sono aggiornate.
[ ] Le immagini base sono ufficiali o affidabili.
[ ] La pipeline può eseguire vulnerability scan.
```

Comandi utili:

```bash id="ptg9nq"
dotnet list package --vulnerable
dotnet list package --outdated
```

# Checklist performance

## Misurazione

```text id="esdcse"
[ ] Il problema è stato misurato prima di ottimizzare.
[ ] Sono disponibili log o metriche.
[ ] Le query lente sono identificate.
[ ] Le allocazioni anomale sono identificate.
[ ] Gli endpoint lenti sono tracciati.
```

Regola:

```text id="w5knl7"
Prima misura, poi ottimizza.
```

## API

```text id="xkj5ef"
[ ] Gli endpoint restituiscono solo i dati necessari.
[ ] Le liste grandi sono paginate.
[ ] I filtri vengono applicati lato database.
[ ] Le response troppo grandi sono evitate.
[ ] Le chiamate esterne hanno timeout.
[ ] Le operazioni lente sono spostate in background.
```

---

## EF Core

```text id="y9c3ku"
[ ] `AsNoTracking` è usato nelle letture.
[ ] Le proiezioni DTO evitano `Include` inutili.
[ ] Gli indici supportano le query frequenti.
[ ] Non ci sono query N+1.
[ ] Le query aggregate sono ottimizzate.
[ ] I report pesanti non bloccano richieste interattive.
```

---

## Async e concorrenza

```text id="u0rqye"
[ ] L’I/O usa async.
[ ] Non ci sono `.Wait()` o `.Result`.
[ ] I `CancellationToken` sono propagati.
[ ] Il parallelismo è usato solo per CPU-bound.
[ ] Il grado di parallelismo è controllato.
[ ] Le sezioni critiche sono ridotte.
```

---

## Caching

```text id="iz817k"
[ ] I dati letti spesso e cambiati raramente sono candidati al caching.
[ ] Il caching ha una strategia di invalidazione.
[ ] Non vengono messi in cache dati sensibili senza controllo.
[ ] Il cache key include i parametri necessari.
[ ] Il caching non sostituisce query corrette.
```

---

## Background jobs

```text id="mf3l47"
[ ] Le code interne sono bounded.
[ ] I worker rispettano il CancellationToken.
[ ] I job lunghi hanno logging e stato.
[ ] Gli errori sono gestiti per singolo job.
[ ] I retry non sono infiniti.
[ ] I job critici hanno persistenza o outbox.
```

# Checklist logging/osservabilità

## Logging applicativo

```text id="gagt6x"
[ ] Ogni operazione importante ha log di inizio/fine o evento significativo.
[ ] I log usano livelli corretti: Trace, Debug, Information, Warning, Error, Critical.
[ ] I log sono strutturati, non solo stringhe concatenate.
[ ] I log includono identificativi utili.
[ ] I log non contengono dati sensibili.
```

Esempio:

```csharp id="xpx5xj"
logger.LogInformation(
    "Documento {DocumentoId} confermato per cliente {ClienteId}",
    documento.Id,
    documento.ClienteId);
```

Meglio di:

```csharp id="sj3t0f"
logger.LogInformation(
    "Documento confermato: " + documento.Id);
```

# Riepilogo

Questa appendice raccoglie checklist per:

```text id="yk0g7y"
codice C#
API
EF Core
sicurezza
performance
deployment
logging/osservabilità
```

La regola principale è questa:

```text id="uykp23"
La qualità non arriva alla fine.
Va controllata a ogni livello:
codice, dati, API, sicurezza, prestazioni,
rilascio e osservabilità.
```

Una checklist non rende automaticamente buono un progetto, ma riduce il rischio di dimenticare aspetti importanti quando il codice cresce, il team aumenta e l’applicazione arriva in produzione.

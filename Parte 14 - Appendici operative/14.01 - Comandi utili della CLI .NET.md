# 14.01 — Comandi utili della CLI .NET

Questa appendice raccoglie i comandi più utili della **.NET CLI**, cioè l’interfaccia da riga di comando basata sul comando `dotnet`.

La CLI .NET è uno strumento cross-platform incluso nel .NET SDK e consente di creare, compilare, eseguire, testare e pubblicare applicazioni .NET da terminale. ([Microsoft Learn][1])

Il punto centrale è questo:

```text
Visual Studio è comodo, ma la CLI .NET è indispensabile
per capire davvero build, test, publish, Docker e CI/CD.
```

---

# Perché conoscere la CLI

La CLI è utile perché funziona in modo coerente su:

```text
Windows
Linux
macOS
server di build
container Docker
pipeline GitHub Actions
Azure DevOps
ambienti senza Visual Studio
```

Nel progetto finale del libro la useremo per:

```text
creare solution e progetti
aggiungere riferimenti
aggiungere pacchetti NuGet
compilare
eseguire
testare
pubblicare
creare migration EF Core
gestire workload
automatizzare comandi ripetitivi
```

# Verificare SDK installato

```bash
dotnet --version
```

Mostra la versione SDK predefinita.

Esempio:

```bash
dotnet --version
# 10.0.100
```

Per vedere tutti gli SDK installati:

```bash
dotnet --list-sdks
```

Per vedere tutti i runtime installati:

```bash
dotnet --list-runtimes
```

Informazioni complete:

```bash
dotnet --info
```

Questo comando è molto utile quando un progetto funziona su una macchina ma non su un’altra.

La documentazione Microsoft ricorda che i comandi della CLI accettano opzioni e argomenti propri e che `--help` stampa una guida sintetica del comando. ([Microsoft Learn][2])

# `dotnet new`

`dotnet new` crea progetti, solution o altri file a partire da template. La documentazione Microsoft lo descrive come il comando che usa il template engine per creare artefatti su disco. ([Microsoft Learn][3])

## Elencare i template disponibili

```bash
dotnet new list
```

Oppure cercare un template:

```bash
dotnet new search blazor
```

# `dotnet restore`

Ripristina pacchetti NuGet e dipendenze.

```bash
dotnet restore
```

Di solito non serve eseguirlo manualmente, perché diversi comandi lo eseguono implicitamente, tra cui `dotnet new`, `dotnet build`, `dotnet run`, `dotnet test` e `dotnet publish`. La documentazione Microsoft segnala che `dotnet restore` resta utile quando si vuole controllare esplicitamente il momento del restore, ad esempio in CI/CD. ([Microsoft Learn][4])

In pipeline:

```bash
dotnet restore MiniEditore.sln
dotnet build MiniEditore.sln -c Release --no-restore
```

---

# `dotnet build`

`dotnet build` compila un progetto, una solution o una file-based app e tutte le sue dipendenze. Produce binari, file `.dll`, `.deps.json`, `.runtimeconfig.json` e altri asset necessari all’esecuzione o al debug. ([Microsoft Learn][4])

## Build della solution

```bash
dotnet build MiniEditore.sln
```

## Build in Release

```bash
dotnet build MiniEditore.sln -c Release
```

## Build senza restore

```bash
dotnet build MiniEditore.sln -c Release --no-restore
```

## Build di un progetto specifico

```bash
dotnet build src/App.Api/App.Api.csproj -c Release
```

## Build con framework specifico

```bash
dotnet build src/App.Api/App.Api.csproj -f net10.0
```

## Build con proprietà MSBuild

```bash
dotnet build MiniEditore.sln -c Release -p:TreatWarningsAsErrors=true
```

# `dotnet run`

Esegue un progetto.

## Eseguire dalla cartella del progetto

```bash
cd src/App.Api
dotnet run
```

## Eseguire indicando il progetto

```bash
dotnet run --project src/App.Api/App.Api.csproj
```

## Eseguire in ambiente Development

Windows PowerShell:

```powershell
$env:ASPNETCORE_ENVIRONMENT="Development"
dotnet run --project src/App.Api/App.Api.csproj
```

Linux/macOS:

```bash
ASPNETCORE_ENVIRONMENT=Development dotnet run --project src/App.Api/App.Api.csproj
```

## Passare URL

```bash
dotnet run --project src/App.Api/App.Api.csproj --urls "http://localhost:5000"
```

Nota:

```text
dotnet run è comodo in sviluppo.
Per distribuire usa dotnet publish.
```

# `dotnet publish`

`dotnet publish` compila l’applicazione, legge le dipendenze e produce una cartella pronta per la distribuzione. Microsoft lo descrive come il modo ufficialmente supportato per preparare un’applicazione .NET alla distribuzione. ([Microsoft Learn][5])

## Publish API

```bash
dotnet publish src/App.Api/App.Api.csproj -c Release -o publish/api
```

## Publish Web

```bash
dotnet publish src/App.Web/App.Web.csproj -c Release -o publish/web
```

## Publish framework-dependent

Richiede il runtime .NET installato sul server.

```bash
dotnet publish src/App.Api/App.Api.csproj \
  -c Release \
  -o publish/api
```

## Publish self-contained

Include il runtime .NET nell’output.

```bash
dotnet publish src/App.Api/App.Api.csproj \
  -c Release \
  -r win-x64 \
  --self-contained true \
  -o publish/api-win-x64
```

## Publish single-file

```bash
dotnet publish src/App.Api/App.Api.csproj \
  -c Release \
  -r win-x64 \
  --self-contained true \
  -p:PublishSingleFile=true \
  -o publish/api-single
```

## Publish Linux x64

```bash
dotnet publish src/App.Api/App.Api.csproj \
  -c Release \
  -r linux-x64 \
  --self-contained true \
  -o publish/api-linux-x64
```

Nota:

```text
dotnet build serve per compilare.
dotnet publish serve per preparare il rilascio.
```

# `dotnet ef`

`dotnet ef` contiene i comandi CLI di Entity Framework Core per attività design-time: creare migration, applicarle al database e generare codice a partire da un database esistente. ([Microsoft Learn][6])

## Installare tool EF Core

```bash
dotnet tool install --global dotnet-ef
```

Aggiornare:

```bash
dotnet tool update --global dotnet-ef
```

Verificare:

```bash
dotnet ef --version
```

---

# Creare migration

Nel nostro progetto `AppDbContext` sta in `App.Infrastructure`, mentre `Program.cs` e `appsettings.json` stanno in `App.Api`.

Quindi:

```bash
dotnet ef migrations add InitialCreate \
  --project src/App.Infrastructure \
  --startup-project src/App.Api \
  --output-dir Data/Migrations
```

Significato:

```text
--project
  progetto dove si trova il DbContext e dove salvare la migration.

--startup-project
  progetto da usare per configurazione, DI e connection string.

--output-dir
  cartella di output delle migration.
```

---

# Applicare migration

```bash
dotnet ef database update \
  --project src/App.Infrastructure \
  --startup-project src/App.Api
```

---

# Rimuovere ultima migration

Solo se non è stata già applicata in modo definitivo.

```bash
dotnet ef migrations remove \
  --project src/App.Infrastructure \
  --startup-project src/App.Api
```

---

# Elencare migration

```bash
dotnet ef migrations list \
  --project src/App.Infrastructure \
  --startup-project src/App.Api
```

---

# Generare script SQL

```bash
dotnet ef migrations script \
  --project src/App.Infrastructure \
  --startup-project src/App.Api \
  -o artifacts/migrations.sql
```

Script idempotente:

```bash
dotnet ef migrations script \
  --idempotent \
  --project src/App.Infrastructure \
  --startup-project src/App.Api \
  -o artifacts/migrations-idempotent.sql
```

Nota:

```text
In produzione è spesso meglio controllare uno script SQL
invece di applicare migration automaticamente all’avvio.
```

# Alias CLI

Gli alias servono a rendere più rapidi i comandi ripetitivi.

## PowerShell

Puoi creare funzioni nel profilo PowerShell.

Aprire il profilo:

```powershell
notepad $PROFILE
```

Esempi:

```powershell
function dbuild {
    dotnet build MiniEditore.sln -c Release
}

function dtest {
    dotnet test MiniEditore.sln -c Release
}

function drunapi {
    dotnet run --project src/App.Api/App.Api.csproj
}

function drunweb {
    dotnet run --project src/App.Web/App.Web.csproj
}

function dwatchapi {
    dotnet watch --project src/App.Api/App.Api.csproj run
}

function dwatchweb {
    dotnet watch --project src/App.Web/App.Web.csproj run
}
```

Ricaricare il profilo:

```powershell
. $PROFILE
```

Uso:

```powershell
dbuild
dtest
drunapi
```

# Errori comuni

## Eseguire `dotnet run` dalla cartella sbagliata

Se sei fuori dal progetto:

```bash
dotnet run
```

può fallire.

Soluzione:

```bash
dotnet run --project src/App.Api/App.Api.csproj
```

---

## Confondere build e publish

```text
dotnet build
  compila per sviluppo/test.

dotnet publish
  prepara output per distribuzione.
```

---

## Usare `--output` sulla solution in modo ambiguo

Meglio pubblicare singoli progetti:

```bash
dotnet publish src/App.Api/App.Api.csproj -c Release -o publish/api
```

non:

```bash
dotnet publish MiniEditore.sln -o publish
```

La documentazione di `dotnet publish` segnala infatti che usare `--output` a livello solution ha semantica ambigua, perché più progetti possono produrre output incompatibili nella stessa cartella. ([Microsoft Learn][5])

---

## Dimenticare `--startup-project` con EF Core

Se il `DbContext` sta in `App.Infrastructure` ma la configurazione sta in `App.Api`, serve:

```bash
--project src/App.Infrastructure
--startup-project src/App.Api
```

---

## Non fissare la versione dei pacchetti

Per progetti didattici va bene installare l’ultima versione stabile.
Per progetti reali è meglio controllare e versionare con attenzione.

---

## Non distinguere Debug e Release

Durante sviluppo:

```bash
dotnet run
```

Per build/deploy:

```bash
dotnet build -c Release
dotnet publish -c Release
```

# Riepilogo

In questa appendice abbiamo raccolto i comandi fondamentali della CLI .NET:

```text
dotnet new
dotnet build
dotnet run
dotnet test
dotnet publish
dotnet add package
dotnet ef
dotnet workload
alias CLI
```

La distinzione principale è questa:

```text
dotnet new
  crea progetti e file.

dotnet build
  compila.

dotnet run
  esegue in sviluppo.

dotnet test
  esegue test automatici.

dotnet publish
  prepara il rilascio.

dotnet add package
  aggiunge dipendenze NuGet.

dotnet ef
  gestisce migration e database EF Core.

dotnet workload
  installa componenti SDK aggiuntivi.

alias/script
  rendono ripetibili i comandi più usati.
```

La regola principale è questa:

```text
Conoscere la CLI rende il progetto ripetibile:
quello che fai in locale può essere fatto anche in Docker,
in CI/CD e su un server di build.
```

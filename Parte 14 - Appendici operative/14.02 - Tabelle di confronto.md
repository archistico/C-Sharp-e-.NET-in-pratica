# `class` vs `struct` vs `record`

In C# `class`, `struct` e `record` servono tutti a definire tipi, ma hanno semantica diversa. Le classi sono reference type, gli struct sono value type, mentre i record aggiungono funzionalità orientate all’uguaglianza per valore; `record` e `record class` definiscono tipi di riferimento, mentre `record struct` definisce un tipo valore. ([Microsoft Learn][1])

| Aspetto                 | `class`                               | `struct`                                                   | `record`                                                           |
| ----------------------- | ------------------------------------- | ---------------------------------------------------------- | ------------------------------------------------------------------ |
| Tipo                    | Reference type                        | Value type                                                 | Può essere reference type o value type                             |
| Allocazione tipica      | Heap                                  | Stack o inline in altri oggetti, secondo scenario/runtime  | Dipende: `record class` reference type, `record struct` value type |
| Uguaglianza predefinita | Per riferimento                       | Per valore, campo per campo                                | Per valore, generata dal compilatore                               |
| Mutabilità              | Spesso mutabile                       | Meglio piccolo e preferibilmente immutabile                | Spesso usato per modelli immutabili                                |
| Ereditarietà            | Supportata                            | Non supportata come le classi                              | `record class` supporta ereditarietà tra record                    |
| Uso tipico              | Entity, servizi, oggetti con identità | Valori piccoli, coordinate, misure, identificatori leggeri | DTO, messaggi, value object semplici                               |
| Rischio                 | Oggetti troppo mutabili o anemici     | Copie involontarie, struct troppo grandi                   | Usarlo per entity con identità può confondere                      |

## Quando usare cosa

| Scenario                                                       | Scelta consigliata                                |
| -------------------------------------------------------------- | ------------------------------------------------- |
| Oggetto con identità e ciclo di vita                           | `class`                                           |
| Entity di dominio, per esempio `Documento`                     | `class`                                           |
| Valore piccolo senza identità, per esempio `PercentualeSconto` | `readonly record struct` oppure `readonly struct` |
| DTO API                                                        | `record`                                          |
| Command/query immutabili                                       | `record`                                          |
| Oggetto molto grande e mutabile                                | `class`                                           |
| Tipo numerico o valore leggero                                 | `struct`                                          |

Regola pratica:

```text id="jwz7kg"
Usa class per oggetti con identità.
Usa struct per valori piccoli.
Usa record per dati trasportati o confrontati per valore.
```

# `IEnumerable<T>` vs `IQueryable<T>`

| Aspetto               | `IEnumerable<T>`                        | `IQueryable<T>`                              |
| --------------------- | --------------------------------------- | -------------------------------------------- |
| Dove lavora           | In memoria o come sequenza già ottenuta | Su provider esterno, per esempio EF Core     |
| Tipo di esecuzione    | LINQ to Objects                         | Query tradotta dal provider                  |
| Esempio provider      | Liste, array, collezioni                | EF Core, provider database                   |
| Query tradotta in SQL | No                                      | Sì, se provider EF Core                      |
| Composizione query    | In memoria                              | Prima dell’esecuzione sul database           |
| Rischio               | Caricare troppi dati prima di filtrare  | Query non traducibili, errori runtime        |
| Materializzazione     | Già su oggetti .NET                     | Avviene con `ToList`, `First`, `Count`, ecc. |

## Esempio

Meglio:

```csharp id="rx1jz8"
IQueryable<Libro> query = dbContext.Libri;

query = query.Where(x => x.Attivo);

List<LibroListItemDto> libri = await query
    .OrderBy(x => x.Titolo)
    .Select(x => new LibroListItemDto(
        x.Id,
        x.Titolo,
        x.Isbn,
        x.Prezzo,
        x.Attivo))
    .ToListAsync(cancellationToken);
```

Da evitare:

```csharp id="bls5jz"
List<Libro> tutti = await dbContext.Libri.ToListAsync();

List<Libro> attivi = tutti
    .Where(x => x.Attivo)
    .ToList();
```

Nel secondo esempio hai già caricato tutti i libri dal database.

## Quando usare cosa

| Scenario                                       | Scelta consigliata                                       |
| ---------------------------------------------- | -------------------------------------------------------- |
| Query EF Core ancora da eseguire               | `IQueryable<T>`                                          |
| Metodo che lavora su collezioni già in memoria | `IEnumerable<T>`                                         |
| API pubblica di un servizio applicativo        | Spesso DTO già materializzati                            |
| Restituire query EF Core fuori dal layer dati  | Da evitare                                               |
| Composizione dinamica di query database        | `IQueryable<T>` internamente al repository/query service |

Regola pratica:

```text id="ki8tw5"
Usa IQueryable finché stai costruendo la query verso il database.
Usa IEnumerable quando stai lavorando su dati già materializzati.
```

# `Task` vs `Thread`

| Aspetto                  | `Task`                                    | `Thread`                         |
| ------------------------ | ----------------------------------------- | -------------------------------- |
| Livello                  | Astrazione ad alto livello                | Astrazione più bassa             |
| Uso tipico               | Operazioni asincrone e lavoro pianificato | Controllo esplicito di un thread |
| Relazione con ThreadPool | Spesso usa ThreadPool o completamento I/O | Thread dedicato                  |
| Ideale per I/O async     | Sì                                        | No, di solito inutile            |
| Ideale per CPU-bound     | Sì, con `Task.Run` o parallelismo         | Solo casi particolari            |
| Cancellazione            | Con `CancellationToken`                   | Manuale e più complessa          |
| Eccezioni                | Propagate tramite `await`                 | Da gestire nel thread            |
| Uso moderno comune       | Molto frequente                           | Raro nelle app business moderne  |

## Quando usare cosa

| Scenario                                         | Scelta consigliata                   |
| ------------------------------------------------ | ------------------------------------ |
| Chiamata HTTP asincrona                          | `Task` + `await`                     |
| Query database async                             | `Task` + `await`                     |
| Elaborazione CPU-bound da spostare dal thread UI | `Task.Run`, con cautela              |
| Worker ASP.NET Core                              | `BackgroundService` + `Task`         |
| Thread dedicato con controllo particolare        | `Thread`, solo se davvero necessario |

Regola pratica:

```text id="s1tluc"
Nel .NET moderno parti quasi sempre da Task.
Usa Thread solo quando hai un motivo tecnico preciso.
```

# `async/await` vs parallelismo

| Aspetto                | `async/await`                                | Parallelismo                            |
| ---------------------- | -------------------------------------------- | --------------------------------------- |
| Risolve                | Attese non bloccanti                         | Uso di più core CPU                     |
| Ideale per             | I/O-bound                                    | CPU-bound                               |
| Esempi                 | HTTP, database, file, rete                   | Calcoli, elaborazioni massive, immagini |
| Aumenta throughput I/O | Sì                                           | Non necessariamente                     |
| Aumenta uso CPU        | No, di per sé                                | Sì                                      |
| Usa più thread?        | Non necessariamente                          | Sì, spesso                              |
| Strumenti              | `Task`, `await`, `CancellationToken`         | `Parallel.For`, PLINQ, `Task.Run`       |
| Errore comune          | Pensare che async renda il codice più veloce | Usarlo per I/O invece di async          |

## Esempi

I/O-bound:

```csharp id="fj4f63"
HttpResponseMessage response =
    await httpClient.GetAsync(url, cancellationToken);
```

CPU-bound:

```csharp id="tpq9cr"
Parallel.ForEach(files, file =>
{
    ProcessImage(file);
});
```

## Quando usare cosa

| Scenario                           | Scelta consigliata                            |
| ---------------------------------- | --------------------------------------------- |
| Leggere dati da API                | `async/await`                                 |
| Salvare su database                | `async/await`                                 |
| Generare 10.000 miniature immagini | Parallelismo controllato                      |
| Importare file e scrivere DB       | Mix: async per I/O, parallelismo solo per CPU |
| UI che non deve bloccarsi          | `async/await`                                 |
| Calcolo pesante su molti elementi  | Parallelismo                                  |

Regola pratica:

```text id="btzngm"
async/await libera il thread durante l’attesa.
Il parallelismo usa più thread/core per lavorare contemporaneamente.
```

# Minimal API vs MVC vs Razor Pages vs Blazor

ASP.NET Core include più modelli applicativi: Minimal API per API HTTP con poco codice, MVC per controller e viste, Razor Pages per applicazioni page-focused e Blazor per UI web interattive con componenti Razor e C#. ([Microsoft Learn][2])

| Aspetto        | Minimal API                                       | MVC                            | Razor Pages                      | Blazor                                           |
| -------------- | ------------------------------------------------- | ------------------------------ | -------------------------------- | ------------------------------------------------ |
| Uso principale | API HTTP                                          | Web app e API strutturate      | Web app server-side page-focused | UI web interattiva a componenti                  |
| Struttura      | Endpoint in codice                                | Controller, action, view/model | Pagina `.cshtml` + PageModel     | Componenti `.razor`                              |
| Ideale per     | API leggere, microservizi, backend per SPA/Blazor | App grandi con pattern MVC     | CRUD web tradizionale            | Dashboard, gestionali, UI interattive            |
| Curva iniziale | Bassa                                             | Media                          | Bassa-media                      | Media                                            |
| Separazione    | Dipende da come organizzi endpoint                | Forte se usato bene            | Buona per pagina                 | Forte tramite componenti                         |
| Rendering HTML | No, salvo risposte custom                         | Sì                             | Sì                               | Sì                                               |
| API JSON       | Sì, ottimo                                        | Sì, ottimo                     | Non ideale come prima scelta     | Consuma API, non sostituisce necessariamente API |
| Stato UI       | Non pertinente                                    | Server-rendered                | Server-rendered                  | Component state                                  |

## Quando usare cosa

| Scenario                                        | Scelta consigliata                |
| ----------------------------------------------- | --------------------------------- |
| Backend REST per progetto finale                | Minimal API                       |
| Applicazione web classica con controller e view | MVC                               |
| CRUD server-side semplice, page-oriented        | Razor Pages                       |
| Gestionale web interattivo                      | Blazor                            |
| API pubblica complessa con molte convenzioni    | MVC o Minimal API ben organizzate |
| Dashboard e report interattivi                  | Blazor                            |

Regola pratica:

```text id="jb0tqw"
Minimal API espone dati e casi d’uso.
Blazor costruisce l’interfaccia.
MVC e Razor Pages restano ottimi per rendering server-side tradizionale.
```

# Monolite vs monolite modulare vs microservizi

| Aspetto               | Monolite        | Monolite modulare          | Microservizi                     |
| --------------------- | --------------- | -------------------------- | -------------------------------- |
| Deploy                | Uno             | Uno                        | Molti                            |
| Database              | Uno             | Uno, ma con confini logici | Idealmente database per servizio |
| Complessità operativa | Bassa           | Media-bassa                | Alta                             |
| Separazione codice    | Spesso debole   | Forte per moduli           | Forte per servizio               |
| Scalabilità           | Tutto insieme   | Tutto insieme              | Per servizio                     |
| Team autonomi         | Limitati        | Possibili entro un repo    | Più autonomi                     |
| Rete interna          | No              | No                         | Sì                               |
| Consistenza dati      | Più semplice    | Più semplice               | Spesso eventuale                 |
| Osservabilità         | Più semplice    | Più semplice               | Necessaria e più complessa       |
| Rischio principale    | Big ball of mud | Confini solo apparenti     | Monolite distribuito             |

## Quando usare cosa

| Scenario                                        | Scelta consigliata                        |
| ----------------------------------------------- | ----------------------------------------- |
| Progetto piccolo o iniziale                     | Monolite semplice                         |
| Progetto business serio ma non enorme           | Monolite modulare                         |
| Team multipli, deploy separati, dati autonomi   | Microservizi                              |
| Vuoi partire bene senza complessità distribuita | Monolite modulare                         |
| Hai solo “tanti controller”                     | Non è motivo sufficiente per microservizi |
| Reporting pesante da scalare separatamente      | Possibile servizio separato               |
| Notifiche asincrone                             | Possibile servizio separato               |

Regola pratica:

```text id="yzwcw4"
Parti da un monolite modulare.
Estrai microservizi solo quando i confini, i dati e i benefici sono chiari.
```

# SQL diretto vs EF Core

ADO.NET fornisce accesso coerente a origini dati come SQL Server, XML e sorgenti OLE DB/ODBC; EF Core invece è un ORM leggero, estensibile e cross-platform che permette di lavorare con il database usando oggetti .NET e riduce molto codice di accesso ai dati. ([Microsoft Learn][3])

| Aspetto                     | SQL diretto / ADO.NET                       | EF Core                                 |
| --------------------------- | ------------------------------------------- | --------------------------------------- |
| Controllo SQL               | Massimo                                     | Medio-alto, ma mediato da LINQ/provider |
| Produttività CRUD           | Più bassa                                   | Alta                                    |
| Mapping oggetti             | Manuale                                     | Automatico/configurabile                |
| Query complesse ottimizzate | Ottimo                                      | Possibile, ma da controllare            |
| Stored procedure legacy     | Ottimo                                      | Supportate, ma meno naturale            |
| Change tracking             | Manuale                                     | Integrato                               |
| Migration                   | Manuali/script                              | Integrate                               |
| Rischio                     | SQL injection se parametri gestiti male     | Query inefficienti se non conosci EF    |
| Ideale per                  | Legacy, performance critiche, SQL complesso | Applicazioni business moderne           |

## Quando usare cosa

| Scenario                             | Scelta consigliata          |
| ------------------------------------ | --------------------------- |
| CRUD standard                        | EF Core                     |
| Domain model con relazioni           | EF Core                     |
| Query/report SQL molto specifici     | SQL diretto o query raw     |
| Stored procedure aziendali esistenti | ADO.NET o EF Core raw SQL   |
| Massimo controllo prestazionale      | SQL diretto                 |
| Progetto didattico moderno           | EF Core                     |
| Import massivo ottimizzato           | SQL diretto o librerie bulk |

Regola pratica:

```text id="cbf6ns"
EF Core è ottimo per la maggior parte del codice applicativo.
SQL diretto resta prezioso per query specialistiche, legacy e performance critiche.
```

# WPF vs WinUI vs MAUI

WPF è una tecnologia desktop Windows basata su XAML; .NET MAUI è il framework Microsoft cross-platform per creare app native mobile e desktop con C# e XAML, con target Android, iOS, macOS e Windows. ([Microsoft Learn][4])

| Aspetto                       | WPF                          | WinUI 3                                   | .NET MAUI                                                       |
| ----------------------------- | ---------------------------- | ----------------------------------------- | --------------------------------------------------------------- |
| Target principale             | Desktop Windows              | Desktop Windows moderno / Windows App SDK | Mobile e desktop cross-platform                                 |
| Piattaforme                   | Windows                      | Windows                                   | Android, iOS, macOS, Windows                                    |
| Linguaggio UI                 | XAML                         | XAML                                      | XAML                                                            |
| Maturità enterprise           | Molto alta                   | Più moderna, ecosistema diverso           | Buona per mobile/cross-platform, diversa dal desktop enterprise |
| Ideale per                    | Gestionale Windows complesso | App Windows moderne                       | App mobile e cross-platform                                     |
| Linux                         | No                           | No                                        | No come target desktop Linux ufficiale                          |
| macOS                         | No                           | No                                        | Sì tramite Mac Catalyst                                         |
| Mobile                        | No                           | No                                        | Sì                                                              |
| DataGrid/reporting enterprise | Ecosistema molto forte       | Da valutare                               | Da valutare caso per caso                                       |
| Integrazione Windows profonda | Molto buona                  | Molto buona/moderna                       | Possibile ma non è il focus principale                          |
| Curva per chi conosce XAML    | Nota se viene da WPF         | Media                                     | Media, perché non è WPF multipiattaforma                        |

## Quando usare cosa

| Scenario                                                         | Scelta consigliata           |
| ---------------------------------------------------------------- | ---------------------------- |
| Gestionale solo Windows molto complesso                          | WPF                          |
| App Windows moderna allineata al Windows App SDK                 | WinUI 3                      |
| App Android/iOS con codice .NET condiviso                        | .NET MAUI                    |
| App desktop Windows/macOS/Linux                                  | Avalonia, non MAUI/WPF/WinUI |
| Applicazione con scanner, stampanti, lettori locali solo Windows | WPF                          |
| App per agenti commerciali su tablet e smartphone                | .NET MAUI                    |
| Dashboard gestionale web                                         | Blazor                       |

Regola pratica:

```text id="g4nsmu"
WPF è forte sul desktop Windows enterprise.
WinUI è orientato al desktop Windows moderno.
MAUI è orientato al cross-platform mobile/desktop, soprattutto quando Android e iOS contano.
```

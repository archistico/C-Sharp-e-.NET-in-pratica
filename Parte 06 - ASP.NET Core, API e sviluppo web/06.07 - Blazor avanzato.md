# Stato applicativo

Lo **stato applicativo** è l’insieme delle informazioni che l’interfaccia deve ricordare mentre l’utente usa l’applicazione.

Esempi:

```text id="s63m4h"
utente corrente
tema chiaro/scuro
filtri di ricerca
pagina corrente di una griglia
elemento selezionato
dati temporanei di un form
carrello
notifiche
preferenze UI
```

In Blazor lo stato può vivere in posti diversi:

```text id="6q3fvl"
nel componente
in un componente padre
in un servizio scoped/singleton
nel browser storage
nella query string
nel database
nella sessione server/circuito
in una cache
```

La scelta dipende dal tipo di stato.

---

## Stato locale del componente

È lo stato più semplice.

```razor id="mrxhix"
<p>Contatore: @count</p>

<button @onclick="Incrementa">
    Incrementa
</button>

@code {
    private int count;

    private void Incrementa()
    {
        count++;
    }
}
```

Questo stato appartiene solo al componente.

È adatto a:

```text id="q8gdt5"
apertura/chiusura di un pannello
valore temporaneo di un filtro
selezione locale
messaggi temporanei
stato di caricamento
```

---

## Stato nel componente padre

Quando più componenti figli devono condividere dati, spesso conviene tenere lo stato nel componente padre.

Padre:

```razor id="t0hmv7"
<ClienteSearch SearchText="searchText"
               SearchTextChanged="AggiornaSearch" />

<ClienteList Clienti="clientiFiltrati" />

@code {
    private string searchText = "";
    private List<ClienteDto> clienti = [];

    private IEnumerable<ClienteDto> clientiFiltrati =>
        string.IsNullOrWhiteSpace(searchText)
            ? clienti
            : clienti.Where(c =>
                c.Nome.Contains(searchText, StringComparison.OrdinalIgnoreCase));

    private void AggiornaSearch(string value)
    {
        searchText = value;
    }
}
```

Figlio:

```razor id="cs2jrl"
<input value="@SearchText" @oninput="OnInput" />

@code {
    [Parameter]
    public string SearchText { get; set; } = "";

    [Parameter]
    public EventCallback<string> SearchTextChanged { get; set; }

    private async Task OnInput(ChangeEventArgs e)
    {
        string value = e.Value?.ToString() ?? "";
        await SearchTextChanged.InvokeAsync(value);
    }
}
```

Questo schema mantiene chiaro il flusso:

```text id="tmu0jd"
padre possiede lo stato
figlio mostra o modifica tramite callback
```

---

## Stato in un servizio

Quando lo stato deve essere condiviso tra componenti lontani, possiamo usare un servizio.

```csharp id="jmteg4"
public class AppState
{
    public string? ClienteSelezionato { get; private set; }

    public event Action? OnChange;

    public void SelezionaCliente(string cliente)
    {
        ClienteSelezionato = cliente;
        NotifyStateChanged();
    }

    private void NotifyStateChanged()
    {
        OnChange?.Invoke();
    }
}
```

Registrazione:

```csharp id="sh5fik"
builder.Services.AddScoped<AppState>();
```

Uso in componente:

```razor id="h4e88j"
@implements IDisposable
@inject AppState AppState

<p>Cliente selezionato: @AppState.ClienteSelezionato</p>

@code {
    protected override void OnInitialized()
    {
        AppState.OnChange += StateHasChanged;
    }

    public void Dispose()
    {
        AppState.OnChange -= StateHasChanged;
    }
}
```

## Scoped, Singleton e Blazor

Attenzione ai lifetime.

In Blazor Server / Interactive Server:

```text id="n0orz0"
Scoped
  vive nel circuito dell’utente.

Singleton
  condiviso tra tutti gli utenti.
```

In Blazor WebAssembly:

```text id="l1v4v2"
Scoped
  si comporta quasi come singleton lato client,
  perché l’app gira nel browser dell’utente.
```

Regola pratica:

```text id="woq33e"
Non mettere stato utente in un singleton server-side.
Rischi di condividerlo tra utenti diversi.
```

# Persistenza stato tra prerendering e interattività

Blazor Web App può prerenderizzare un componente sul server e poi renderlo in modo interattivo.

Questo può creare un problema:

```text id="w60o6x"
1. Il componente viene eseguito durante il prerendering.
2. Carica dati.
3. Produce HTML.
4. Poi diventa interattivo.
5. Il componente può rieseguire inizializzazione e ricaricare gli stessi dati.
```

Risultato possibile:

```text id="1szs3g"
doppia chiamata al database
doppia chiamata API
flicker visivo
stato perso
form reimpostato
```

Blazor fornisce il servizio `PersistentComponentState` per persistere lo stato usato durante prerendering e riutilizzarlo quando il componente diventa interattivo. In .NET 10, la documentazione segnala anche miglioramenti nella gestione dello stato persistente durante enhanced navigation. ([Microsoft Learn][1])

---

## Esempio con `[PersistentState]`

Nelle versioni moderne, una forma comoda è usare attributi di persistenza dello stato sui componenti.

```razor id="h3yp1e"
@page "/clienti"
@rendermode InteractiveServer
@inject ClienteService ClienteService

<h1>Clienti</h1>

@if (Clienti is null)
{
    <p>Caricamento...</p>
}
else
{
    <ul>
    @foreach (ClienteDto cliente in Clienti)
    {
        <li>@cliente.Nome</li>
    }
    </ul>
}

@code {
    [PersistentState]
    public List<ClienteDto>? Clienti { get; set; }

    protected override async Task OnInitializedAsync()
    {
        Clienti ??= await ClienteService.ListaAsync();
    }
}
```

L’idea è:

```text id="706ol9"
se lo stato è già stato ripristinato
  non ricarico

se lo stato non esiste
  carico i dati
```

---

## Quando usare la persistenza dello stato

È utile quando:

```text id="9ry5e0"
il componente è prerenderizzato
il caricamento dati è costoso
vuoi evitare doppia esecuzione
vuoi evitare flicker
vuoi preservare dati iniziali tra SSR e interattività
```

Non usarla come database o come sessione generale.

Regola pratica:

```text id="qcr7sk"
PersistentComponentState serve a passare dati dal prerendering
alla fase interattiva, non a conservare stato a lungo termine.
```

# Reconnect UI

In Blazor Server e Interactive Server, il browser mantiene una connessione con il server.

Se la connessione cade, l’utente deve capire cosa sta succedendo.

Esempi di disconnessione:

```text id="f584xi"

server riavviato
timeout
problema proxy
perdita connessione SignalR
computer in sospensione
```

Le app server-side usano ASP.NET Core SignalR per comunicare con il browser, quindi valgono le considerazioni di hosting e scalabilità di SignalR. ([Microsoft Learn][3])

# JavaScript interop

Blazor permette di chiamare JavaScript da C# e C# da JavaScript.

Serve quando dobbiamo usare:

```text id="m3e2yt"
API browser
librerie JavaScript
grafici avanzati
editor visuali
mappe
focus input
dimensioni DOM
download file
integrazioni esistenti
```

Blazor non elimina JavaScript. Lo rende meno centrale nella logica applicativa.

---

## Chiamare JavaScript da C#

JavaScript:

```javascript id="g6cb5g"
window.focusElement = function (id) {
    const element = document.getElementById(id);

    if (element) {
        element.focus();
    }
};
```

Componente:

```razor id="a3mx66"
@inject IJSRuntime JS

<input id="nomeInput" />

<button @onclick="FocusNome">
    Focus
</button>

@code {
    private async Task FocusNome()
    {
        await JS.InvokeVoidAsync("focusElement", "nomeInput");
    }
}
```

---

## Moduli JavaScript

Preferibile usare moduli quando possibile.

`wwwroot/js/clienti.js`:

```javascript id="x0ollw"
export function focusElement(id) {
    const element = document.getElementById(id);

    if (element) {
        element.focus();
    }
}
```

Componente:

```razor id="o0qbb5"
@inject IJSRuntime JS
@implements IAsyncDisposable

<input id="nomeInput" />

<button @onclick="FocusNome">
    Focus
</button>

@code {
    private IJSObjectReference? module;

    protected override async Task OnAfterRenderAsync(bool firstRender)
    {
        if (firstRender)
        {
            module = await JS.InvokeAsync<IJSObjectReference>(
                "import",
                "./js/clienti.js");
        }
    }

    private async Task FocusNome()
    {
        if (module is not null)
        {
            await module.InvokeVoidAsync("focusElement", "nomeInput");
        }
    }

    public async ValueTask DisposeAsync()
    {
        if (module is not null)
        {
            await module.DisposeAsync();
        }
    }
}
```

---

## Regole pratiche per JS interop

```text id="qil6gk"
isola JavaScript in moduli
non spargere script globali ovunque
non usare JS per logica di business C#
gestisci Dispose delle risorse JS
ricorda che durante prerendering il DOM non è ancora disponibile
usa OnAfterRenderAsync per operazioni DOM
```

# Autenticazione in Blazor

L’autenticazione in Blazor dipende molto dal render mode.

Domanda fondamentale:

```text id="cgkcnx"
Il codice gira sul server o nel browser?
```

In Blazor Server / Interactive Server:

```text id="8ovcem"
l’identità utente è gestita lato server
si integrano cookie, OpenID Connect, policy ASP.NET Core
i componenti possono leggere AuthenticationState
```

In Blazor WebAssembly:

```text id="24v7oi"
il codice gira nel browser
l’app chiama API protette
i token devono essere gestiti con molta attenzione
il backend deve sempre validare autorizzazione
```

La documentazione Microsoft sulla sicurezza per Blazor Interactive Server ricorda che queste app adottano un modello stateful e richiedono mitigazioni specifiche, perché lo stato del circuito e le operazioni lato server sono parte del modello di sicurezza. ([Microsoft Learn][4])

---

## CascadingAuthenticationState

Nei componenti possiamo leggere lo stato di autenticazione.

```razor id="rw9pl2"
@using Microsoft.AspNetCore.Components.Authorization
@inject AuthenticationStateProvider AuthenticationStateProvider

@if (userName is null)
{
    <p>Utente non autenticato.</p>
}
else
{
    <p>Ciao, @userName</p>
}

@code {
    private string? userName;

    protected override async Task OnInitializedAsync()
    {
        AuthenticationState authState =
            await AuthenticationStateProvider.GetAuthenticationStateAsync();

        ClaimsPrincipal user = authState.User;

        if (user.Identity?.IsAuthenticated == true)
        {
            userName = user.Identity.Name;
        }
    }
}
```

---

## `AuthorizeView`

`AuthorizeView` mostra contenuto diverso in base all’autenticazione.

```razor id="a7b0nq"
<AuthorizeView>
    <Authorized>
        <p>Benvenuto, @context.User.Identity?.Name</p>
    </Authorized>
    <NotAuthorized>
        <p>Devi effettuare l'accesso.</p>
    </NotAuthorized>
</AuthorizeView>
```

Con ruolo:

```razor id="y265hj"
<AuthorizeView Roles="Admin">
    <Authorized>
        <button>Area amministratore</button>
    </Authorized>
</AuthorizeView>
```

---

## Proteggere una pagina

```razor id="gubmfb"
@page "/admin"
@attribute [Authorize(Roles = "Admin")]

<h1>Area admin</h1>
```

Attenzione:

```text id="8e9c8r"
Nascondere un pulsante non è sicurezza.
Anche gli endpoint/API devono essere protetti lato server.
```

# Chiamate API

Blazor WebAssembly e molti componenti interattivi chiamano API tramite `HttpClient`.

Client tipizzato:

```csharp id="winmcl"
using System.Net.Http.Json;

public class ClientiApiClient
{
    private readonly HttpClient httpClient;

    public ClientiApiClient(HttpClient httpClient)
    {
        this.httpClient = httpClient;
    }

    public async Task<List<ClienteDto>> ListaAsync()
    {
        List<ClienteDto>? clienti =
            await httpClient.GetFromJsonAsync<List<ClienteDto>>("/api/clienti");

        return clienti ?? [];
    }

    public async Task<ClienteDto?> CreaAsync(CreaClienteRequest request)
    {
        HttpResponseMessage response =
            await httpClient.PostAsJsonAsync("/api/clienti", request);

        if (!response.IsSuccessStatusCode)
        {
            return null;
        }

        return await response.Content.ReadFromJsonAsync<ClienteDto>();
    }
}
```

Registrazione:

```csharp id="p4cdmk"
builder.Services.AddHttpClient<ClientiApiClient>(client =>
{
    client.BaseAddress = new Uri("https://localhost:7001");
});
```

In WebAssembly, spesso `BaseAddress` è l’host dell’app.

---

## Gestire errori API

Non limitarti a:

```csharp id="s4mx9d"
await httpClient.GetFromJsonAsync<List<ClienteDto>>("/api/clienti");
```

perché gli errori vanno gestiti.

Esempio:

```csharp id="eb6cee"
public async Task<ApiResult<List<ClienteDto>>> ListaSicuraAsync()
{
    try
    {
        HttpResponseMessage response =
            await httpClient.GetAsync("/api/clienti");

        if (!response.IsSuccessStatusCode)
        {
            return ApiResult<List<ClienteDto>>.Failure(
                $"Errore HTTP: {response.StatusCode}");
        }

        List<ClienteDto>? clienti =
            await response.Content.ReadFromJsonAsync<List<ClienteDto>>();

        return ApiResult<List<ClienteDto>>.Success(clienti ?? []);
    }
    catch (HttpRequestException ex)
    {
        return ApiResult<List<ClienteDto>>.Failure(ex.Message);
    }
}
```

Risultato:

```csharp id="ozbgo4"
public record ApiResult<T>(
    bool IsSuccess,
    T? Value,
    string? Error)
{
    public static ApiResult<T> Success(T value)
    {
        return new ApiResult<T>(true, value, null);
    }

    public static ApiResult<T> Failure(string error)
    {
        return new ApiResult<T>(false, default, error);
    }
}
```

# Streaming rendering

Lo **streaming rendering** permette di inviare al browser aggiornamenti progressivi durante static SSR o prerendering.

Scenario:

```text id="xnn792"
la pagina parte subito
mostra contenuto iniziale
poi arrivano dati caricati più lentamente
il browser riceve aggiornamenti senza aspettare tutto
```

Nelle versioni recenti, lo streaming rendering si abilita applicando l’attributo `[StreamRendering]` al componente; la documentazione precisa che in .NET 9+ si usa `[StreamRendering]`, mentre in .NET 8 si usava `[StreamRendering(true)]`. ([Microsoft Learn][5])

---

## Esempio

```razor id="fbv5rc"
@page "/report"
@attribute [StreamRendering]

<h1>Report</h1>

@if (dati is null)
{
    <p>Caricamento dati...</p>
}
else
{
    <p>Dati caricati: @dati.Count</p>
}

@code {
    private List<string>? dati;

    protected override async Task OnInitializedAsync()
    {
        await Task.Delay(2000);

        dati =
        [
            "Dato 1",
            "Dato 2",
            "Dato 3"
        ];
    }
}
```

Vantaggio:

```text id="i0uhg0"
l'utente vede subito una pagina iniziale
e poi riceve aggiornamenti quando i dati sono pronti
```

Attenzione: gli aggiornamenti stream possono causare spostamenti del contenuto nella pagina, quindi vanno progettati bene. ([Microsoft Learn][5])

# Server-Sent Events

I **Server-Sent Events**, o SSE, permettono al server di inviare eventi al client su una connessione HTTP unidirezionale.

Sono utili per:

```text id="psbg5b"
notifiche leggere
stato elaborazione
progressi di un job
feed eventi
messaggi server-client semplici
```

Sono diversi da WebSocket:

```text id="4nxv1v"
SSE
  server → client

WebSocket
  client ↔ server
```

---

## Endpoint SSE minimale

```csharp id="l0ec54"
app.MapGet("/api/events", async (HttpContext context) =>
{
    context.Response.Headers.ContentType = "text/event-stream";

    for (int i = 1; i <= 10; i++)
    {
        await context.Response.WriteAsync($"data: Evento {i}\n\n");
        await context.Response.Body.FlushAsync();

        await Task.Delay(1000);
    }
});
```

Client JavaScript:

```javascript id="ie04vl"
const source = new EventSource("/api/events");

source.onmessage = event => {
    console.log("Evento:", event.data);
};
```

In Blazor, possiamo integrare SSE tramite JavaScript interop oppure tramite chiamate HTTP avanzate, a seconda dello scenario.

---

## Quando usare SSE in Blazor

Usalo quando:

```text id="eiz8ip"
ti basta comunicazione server → client
vuoi notifiche leggere
non ti serve bidirezionalità completa
vuoi evitare complessità WebSocket personalizzata
```

Per Blazor Server / Interactive Server, ricorda che esiste già una connessione SignalR usata dal framework. Non sempre serve aggiungere SSE.

## Evitare rendering inutili

`ShouldRender` permette di impedire rendering non necessari.

```razor id="kkbhpn"
@code {
    [Parameter]
    public ClienteDto? Cliente { get; set; }

    protected override bool ShouldRender()
    {
        return Cliente is not null;
    }
}
```

Usalo con cautela. Se blocchi rendering quando serve, la UI diventa incoerente.

---

## Virtualizzazione

Per liste grandi, usare virtualizzazione.

```razor id="95zivw"
<Virtualize Items="clienti" Context="cliente">
    <div>@cliente.Nome</div>
</Virtualize>
```

La virtualizzazione evita di renderizzare migliaia di elementi contemporaneamente.

---

## Evitare lavoro costoso nel markup

Sospetto:

```razor id="zq10af"
@foreach (var cliente in clienti.Where(c => CalcoloCostoso(c)))

}
```

Meglio calcolare prima:

```csharp id="8sf6b5"
private List<ClienteDto> clientiFiltrati = [];

private void ApplicaFiltro()

    clientiFiltrati = clienti
        .Where(c => CalcoloCostoso(c))
        .ToList();

```

# Error boundary

Un **ErrorBoundary** intercetta eccezioni non gestite nei componenti figli e permette di mostrare una UI di fallback.

Esempio:

```razor id="ykvh17"
<ErrorBoundary>
    <ChildContent>
        <ClienteDettaglio Id="10" />
    </ChildContent>
    <ErrorContent>
        <p>Si è verificato un errore nel caricamento del cliente.</p>
    </ErrorContent>
</ErrorBoundary>
```

Gli error boundaries sono uno strumento UI-level per gestire eccezioni nei componenti Blazor; la documentazione Microsoft li inserisce tra i meccanismi di gestione degli errori nelle app Blazor. ([Microsoft Learn][7])

---

## ErrorBoundary personalizzato

```razor id="1wtj9s"
<ErrorBoundary @ref="errorBoundary">
    <ChildContent>
        @ChildContent
    </ChildContent>
    <ErrorContent Context="exception">
        <div class="error-panel">
            <h3>Errore</h3>
            <p>Non è stato possibile completare l'operazione.</p>
            <button @onclick="Recover">
                Riprova
            </button>
        </div>
    </ErrorContent>
</ErrorBoundary>

@code {
    private ErrorBoundary? errorBoundary;

    [Parameter]
    public RenderFragment? ChildContent { get; set; }

    private void Recover()
    {
        errorBoundary?.Recover();
    }
}
```

---

## Cosa non fare

Non usare ErrorBoundary per nascondere bug sistematici.

Un errore catturato deve essere:

```text id="8t1tq4"
loggato
diagnosticabile
corretto
eventualmente mostrato con messaggio sicuro
```

Regola pratica:

```text id="aazrjw"
ErrorBoundary migliora la resilienza della UI,
ma non sostituisce logging, test e correzione del bug.
```

# Component library

Quando un’app Blazor cresce, conviene estrarre componenti riutilizzabili in una **Razor Class Library**.

Una component library può contenere:

```text id="d3xa7n"
componenti .razor
CSS isolato
JavaScript
immagini
servizi
modelli condivisi
helper UI
```

Microsoft documenta le Razor Class Library come meccanismo per creare pacchetti e distribuire componenti riutilizzabili con Blazor. ([Microsoft Learn][8])

---

## Creare una Razor Class Library

```bash id="i058ks"
dotnet new razorclasslib -n MiaApp.Components
```

Aggiungere riferimento:

```bash id="asy3zt"
dotnet add MiaApp/MiaApp.csproj reference MiaApp.Components/MiaApp.Components.csproj
```

Componente nella libreria:

```razor id="la8ad4"
<div class="empty-state">
    <h3>@Title</h3>
    <p>@Message</p>
</div>

@code {
    [Parameter]
    public string Title { get; set; } = "Nessun dato";

    [Parameter]
    public string Message { get; set; } = "Non ci sono elementi da mostrare.";
}
```

Uso nell’app:

```razor id="wjcv6p"
<EmptyState Title="Nessun cliente"
            Message="Non sono presenti clienti registrati." />
```

---

## CSS isolation

Componente:

```text id="qnfk6e"
EmptyState.razor
EmptyState.razor.css
```

CSS:

```css id="3odq5t"
.empty-state {
    padding: 1rem;
    border: 1px solid #ddd;
    border-radius: 0.5rem;
}
```

La CSS isolation riduce il rischio che gli stili del componente interferiscano con il resto dell’app.

---

## Quando estrarre una component library

Conviene quando:

```text id="qkl4ea"
più progetti usano gli stessi componenti
il design system è condiviso
vuoi standardizzare UI aziendale
hai componenti complessi riutilizzabili
vuoi pacchettizzare via NuGet
```

Non conviene creare una libreria troppo presto se hai ancora pochi componenti instabili.

# Esempio completo: pagina avanzata con stato, QuickGrid ed error boundary

```razor id="jv22wp"
@page "/clienti-avanzato"
@rendermode InteractiveServer
@using Microsoft.AspNetCore.Components.QuickGrid
@inject ClientiApiClient ClientiApi
@inject ILogger<ClientiAvanzato> Logger

<h1>Clienti</h1>

<ErrorBoundary>
    <ChildContent>
        <div class="toolbar">
            <input placeholder="Cerca..."
                   @bind="Search"
                   @bind:event="oninput" />

            <button @onclick="CaricaAsync">
                Aggiorna
            </button>
        </div>

        @if (isLoading)
        {
            <p>Caricamento...</p>
        }
        else
        {
            <QuickGrid Items="clientiFiltrati.AsQueryable()">
                <PropertyColumn Property="c => c.Id" Sortable="true" />
                <PropertyColumn Property="c => c.Nome" Sortable="true" />
                <PropertyColumn Property="c => c.Email" />

                <TemplateColumn Title="Azioni">
                    <button @onclick="() => Seleziona(context)">
                        Seleziona
                    </button>
                </TemplateColumn>
            </QuickGrid>
        }

        @if (clienteSelezionato is not null)
        {
            <p>Selezionato: @clienteSelezionato.Nome</p>
        }
    </ChildContent>

    <ErrorContent>
        <p>Errore durante il caricamento dei clienti.</p>
    </ErrorContent>
</ErrorBoundary>

@code {
    [PersistentState]
    public List<ClienteDto>? Clienti { get; set; }

    [PersistentState]
    public string Search { get; set; } = "";

    private bool isLoading;
    private ClienteDto? clienteSelezionato;

    private IEnumerable<ClienteDto> clientiFiltrati =>
        string.IsNullOrWhiteSpace(Search)
            ? Clienti ?? []
            : (Clienti ?? []).Where(c =>
                c.Nome.Contains(Search, StringComparison.OrdinalIgnoreCase) ||
                (c.Email?.Contains(Search, StringComparison.OrdinalIgnoreCase) ?? false));

    protected override async Task OnInitializedAsync()
    {
        if (Clienti is null)
        {
            await CaricaAsync();
        }
    }

    private async Task CaricaAsync()
    {
        try
        {
            isLoading = true;

            Clienti = await ClientiApi.ListaAsync();
        }
        catch (Exception ex)
        {
            Logger.LogError(ex, "Errore durante il caricamento clienti");
            throw;
        }
        finally
        {
            isLoading = false;
        }
    }

    private void Seleziona(ClienteDto cliente)
    {
        clienteSelezionato = cliente;
    }
}
```

Questo esempio combina:

```text id="bqtnpy"
stato persistente
QuickGrid
filtro locale
ErrorBoundary
logging
rendering interattivo server
chiamata API
```

# Errori comuni

## Stato globale gestito male

Sbagliato:

```csharp id="v950r0"
builder.Services.AddSingleton<UserState>();
```

se `UserState` contiene dati dell’utente corrente in Blazor Server.

Meglio usare `Scoped`, database, autenticazione o storage adeguato.

---

## Dimenticare il prerendering

Se un componente carica dati in `OnInitializedAsync`, può essere eseguito più volte tra prerendering e interattività.

Usa persistenza stato quando serve.

---

## Chiamare JavaScript troppo presto

Durante prerendering il DOM non è disponibile.

Operazioni DOM vanno fatte in `OnAfterRenderAsync`, non in `OnInitializedAsync`.

---

## Pensare che `AuthorizeView` basti per proteggere dati

`AuthorizeView` nasconde o mostra UI.

La sicurezza vera deve stare anche su:

```text id="ejxwzf"
endpoint API
servizi server
database query
policy
autorizzazione server-side
```

---

## Caricare migliaia di righe nella UI

Sbagliato:

```text id="hc3t5r"
carico 100.000 record in una tabella Blazor
e filtro tutto nel browser
```

Meglio:

```text id="4o7ytf"
pagination server-side
filtering server-side
virtualizzazione
QuickGrid con provider dati
```

---

## Usare ErrorBoundary come silenziatore

Catturare errori senza loggarli rende l’app apparentemente stabile ma impossibile da diagnosticare.

---

## Creare component library troppo presto

Se i componenti cambiano continuamente, estrarli in una libreria può rallentare.

Meglio stabilizzare prima pattern, naming e API dei componenti.

---

# Linee guida pratiche

```text id="hmrwk1"
Gestisci lo stato vicino a dove serve
  e spostalo in servizi solo quando deve essere condiviso.

Usa PersistentComponentState
  per evitare doppio caricamento tra prerendering e interattività.

Usa QuickGrid
  per tabelle dati semplici e performanti.

Personalizza la Reconnect UI
  nelle app Interactive Server importanti.

Usa JS interop
  solo per funzionalità browser o librerie JavaScript.

Proteggi componenti e API
  Blazor UI non sostituisce autorizzazione server-side.

Centralizza le chiamate API
  in client tipizzati.

Usa streaming rendering
  per migliorare pagine static SSR/prerendering con dati lenti.

Valuta SSE
  per notifiche server-client leggere.

Ottimizza rendering e liste grandi
  con virtualizzazione, paging e componenti ben progettati.

Usa ErrorBoundary
  per migliorare resilienza UI, non per nascondere bug.

Crea component library
  quando hai componenti stabili e riutilizzabili.
```

# Riepilogo

In questo capitolo abbiamo portato Blazor oltre gli esempi base.

Abbiamo visto:

```text id="e9kvlb"
stato applicativo
persistenza stato tra prerendering e interattività
QuickGrid
Reconnect UI
JavaScript interop
autenticazione in Blazor

```

Abbiamo chiarito che la persistenza dello stato serve a riutilizzare dati caricati durante il prerendering quando il componente diventa interattivo, evitando doppie inizializzazioni e flicker. ([Microsoft Learn][1])

Abbiamo visto che QuickGrid è una griglia dati Razor ottimizzata per scenari tabellari comuni e può essere una buona base per tabelle performanti senza introdurre subito librerie esterne. ([Microsoft Learn][2])

Abbiamo visto che lo streaming rendering permette di inviare aggiornamenti progressivi durante static SSR o prerendering, ma va abilitato consapevolmente perché può causare spostamenti del layout. ([Microsoft Learn][5])

La regola principale è questa:

```text id="6txbhq"
Blazor avanzato richiede progettare stato, render mode, sicurezza,
errori e performance con la stessa attenzione con cui si progetta il backend.
```

Blazor è molto produttivo quando i componenti restano piccoli, lo stato è esplicito, le API sono ben separate e il render mode viene scelto in base al comportamento reale dell’applicazione.

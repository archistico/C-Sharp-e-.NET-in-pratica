# 06.06 — Blazor: componenti interattivi

Nei capitoli precedenti abbiamo visto HTTP, ASP.NET Core, Minimal API, API robuste, Razor Pages e MVC. Ora possiamo affrontare **Blazor**, cioè il modello di ASP.NET Core per costruire interfacce web a componenti usando C# e Razor.

Blazor permette di creare UI interattive senza scrivere necessariamente tutta la logica in JavaScript. Il cuore di Blazor sono i **Razor components**, cioè componenti `.razor` riutilizzabili che combinano markup HTML, sintassi Razor, codice C# e gestione dello stato.

Il punto centrale è questo:

```text id="pvzb7u"
Blazor non è solo “C# nel browser”.
È un modello a componenti che può renderizzare staticamente sul server,
interagire tramite SignalR, girare in WebAssembly o combinare più modalità.
```

Microsoft distingue tra componenti Razor renderizzati staticamente e componenti renderizzati in modo interattivo; nel rendering statico il componente produce HTML ma non può gestire eventi utente tramite codice .NET, mentre nelle modalità interattive gli eventi utente vengono elaborati da .NET lato server o lato client. ([Microsoft Learn][1])

# EventCallback

`EventCallback` permette a un componente figlio di notificare un evento al componente padre.

Esempio: un componente `ClienteRow` deve dire al padre che l’utente ha cliccato “Elimina”.

`ClienteRow.razor`:

```razor id="lbykgn"
<tr>
    <td>@Cliente.Nome</td>
    <td>@Cliente.Email</td>
    <td>
        <button @onclick="Elimina">
            Elimina
        </button>
    </td>
</tr>

@code {
    [Parameter]
    public ClienteDto Cliente { get; set; } = default!;

    [Parameter]
    public EventCallback<int> OnElimina { get; set; }

    private async Task Elimina()
    {
        await OnElimina.InvokeAsync(Cliente.Id);
    }
}
```

Componente padre:

```razor id="31un6n"
<table>
    <tbody>
    @foreach (ClienteDto cliente in clienti)
    {
        <ClienteRow
            Cliente="cliente"
            OnElimina="EliminaClienteAsync" />
    }
    </tbody>
</table>

@code {
    private List<ClienteDto> clienti = [];

    private async Task EliminaClienteAsync(int clienteId)
    {
        // elimina cliente
        await Task.CompletedTask;
    }
}
```

`EventCallback` è preferibile rispetto a delegati generici quando stiamo modellando eventi tra componenti Blazor, perché si integra con il ciclo di rendering del framework.

# Data binding

Il data binding collega un valore C# a un elemento dell’interfaccia.

Blazor usa la direttiva `@bind`.

Microsoft documenta il data binding dei componenti Razor tramite l’attributo direttiva `@bind`, che può associare il valore di elementi DOM, campi, proprietà o espressioni Razor. ([Microsoft Learn][2])

Esempio:

```razor id="ehxy3r"
<input @bind="nome" />

<p>Nome inserito: @nome</p>

@code {
    private string nome = "";
}
```

Per impostazione comune, il valore viene aggiornato quando l’input perde il focus. Possiamo cambiare evento:

```razor id="q4ysux"
<input @bind="nome" @bind:event="oninput" />
```

Ora il valore viene aggiornato mentre l’utente digita.

---

## Binding bidirezionale tra componenti

Componente figlio:

```razor id="6fk5i7"
<input value="@Value" @oninput="OnInput" />

@code {
    [Parameter]
    public string Value { get; set; } = "";

    [Parameter]
    public EventCallback<string> ValueChanged { get; set; }

    private async Task OnInput(ChangeEventArgs e)
    {
        string newValue = e.Value?.ToString() ?? "";
        await ValueChanged.InvokeAsync(newValue);
    }
}
```

Uso:

```razor id="k3ixz4"
<CustomInput @bind-Value="nome" />

<p>@nome</p>

@code {
    private string nome = "";
}
```

Questo pattern è alla base dei componenti input riutilizzabili.

# Lifecycle

I componenti Blazor hanno un ciclo di vita.

I metodi principali sono:

```text id="j3auru"
OnInitialized / OnInitializedAsync
  inizializzazione del componente.

OnParametersSet / OnParametersSetAsync
  chiamati quando i parametri vengono impostati o aggiornati.

OnAfterRender / OnAfterRenderAsync
  chiamati dopo il rendering.

ShouldRender
  permette di controllare se il componente deve essere renderizzato.

Dispose / IAsyncDisposable
  pulizia risorse.
```

Microsoft spiega che i componenti Razor elaborano eventi di lifecycle tramite metodi sincroni e asincroni sovrascrivibili, utili per eseguire operazioni durante inizializzazione e rendering. ([Microsoft Learn][3])

---

## `OnInitializedAsync`

Si usa per inizializzare dati una sola volta.

```razor id="dnwidx"
@page "/clienti"
@inject ClienteApiClient ClientiApi

<h1>Clienti</h1>

@if (clienti is null)
{
    <p>Caricamento...</p>
}
else
{
    <ul>
    @foreach (ClienteDto cliente in clienti)
    {
        <li>@cliente.Nome</li>
    }
    </ul>
}

@code {
    private List<ClienteDto>? clienti;

    protected override async Task OnInitializedAsync()
    {
        clienti = await ClientiApi.ListaAsync();
    }
}
```

---

## `OnParametersSetAsync`

Si usa quando il componente dipende da parametri che possono cambiare.

```razor id="1cj2ax"
@page "/clienti/{Id:int}"

<h1>Cliente @Id</h1>

@code {
    [Parameter]
    public int Id { get; set; }

    private ClienteDto? cliente;

    protected override async Task OnParametersSetAsync()
    {
        cliente = await clienteService.TrovaAsync(Id);
    }
}
```

Se cambia `Id`, il componente può ricaricare i dati.

---

## `OnAfterRenderAsync`

Si usa quando serve interagire con il DOM o con JavaScript dopo il rendering.

```razor id="81o8c1"
@inject IJSRuntime JS

<input id="nomeInput" />

@code {
    protected override async Task OnAfterRenderAsync(bool firstRender)
    {
        if (firstRender)
        {
            await JS.InvokeVoidAsync("focusElement", "nomeInput");
        }
    }
}
```

Attenzione: questo è un caso in cui JavaScript resta utile anche in Blazor.

# Rendering

Il rendering è il processo con cui Blazor produce markup HTML aggiornato.

Quando cambia lo stato del componente, Blazor può ricalcolare il markup e aggiornare la UI.

Esempio:

```razor id="f2q6ca"
<p>@count</p>

<button @onclick="Incrementa">+</button>

@code {
    private int count;

    private void Incrementa()
    {
        count++;
    }
}
```

Quando l’utente clicca:

```text id="d0oaen"
1. L’evento viene gestito da C#.
2. Il valore count cambia.
3. Blazor renderizza di nuovo il componente.
4. La UI mostra il nuovo valore.
```

---

## `StateHasChanged`

In molti casi Blazor richiama il rendering automaticamente.

A volte, però, dobbiamo notificare manualmente un cambiamento:

```csharp id="ksogws"
StateHasChanged();
```

Esempio tipico:

```csharp id="4012oi"
private async Task CaricaAsync()
{
    isLoading = true;
    StateHasChanged();

    dati = await servizio.CaricaAsync();

    isLoading = false;
}
```

Usare `StateHasChanged` con criterio. Se devi chiamarlo continuamente, può esserci un problema di progettazione dello stato.

# Blazor Server

**Blazor Server** esegue il codice .NET sul server.

Il browser riceve HTML e mantiene una connessione interattiva con il server, normalmente tramite SignalR. Gli eventi utente vengono inviati al server, il server esegue il codice C# e invia al browser gli aggiornamenti della UI.

Schema:

```text id="zv9v0d"
Browser
  evento click
    ↓ SignalR
Server
  codice .NET
  rendering diff
    ↓ SignalR
Browser
  aggiorna UI
```

Vantaggi:

```text id="rkcpag"
download iniziale leggero
accesso diretto a servizi server
codice C# resta sul server
buono per intranet e gestionali
```

Svantaggi:

```text id="hpqo0i"
richiede connessione persistente
latenza percepibile su reti lente
stato del circuito sul server
scalabilità da progettare
offline non adatto
```

---

# Blazor WebAssembly

**Blazor WebAssembly** esegue il codice .NET nel browser tramite WebAssembly.

Schema:

```text id="4415rs"
Browser
  scarica runtime .NET WebAssembly + app
  esegue codice C# lato client
  chiama API HTTP quando serve
```

Vantaggi:

```text id="l4bncm"
interattività lato client
meno carico UI sul server
può funzionare meglio con scenari offline/PWA
nessuna connessione SignalR persistente obbligatoria
```

Svantaggi:

```text id="9xkt4f"
download iniziale più pesante
codice client scaricato nel browser
accesso al database sempre tramite API
performance dipendente dal client
sicurezza da progettare come SPA
```

---

# Blazor Web App

Nelle versioni moderne di ASP.NET Core, il template **Blazor Web App** unifica vari scenari.

Può usare:

```text id="7d30fn"
Static SSR
Interactive Server
Interactive WebAssembly
Interactive Auto
```

La documentazione Microsoft sul progetto Blazor indica che il template Blazor Web App supporta rendering statico e interattivo lato server in un singolo progetto; se si abilita anche Interactive WebAssembly, viene creato un progetto `.Client` aggiuntivo per i componenti WebAssembly. I componenti che usano Interactive WebAssembly o Interactive Auto devono trovarsi nel progetto `.Client`. ([Microsoft Learn][4])

Questo è importante perché cambia il modo in cui si organizza la soluzione.

---

# Static SSR

**Static SSR** significa rendering server-side statico.

Il server genera HTML, ma il componente non è interattivo tramite .NET dopo il caricamento.

Esempio adatto:

```text id="ejzw7z"
pagina informativa
pagina elenco semplice
pagina SEO
contenuto renderizzato dal server
form semplice gestito con POST server-side
```

Vantaggi:

```text id="36jyod"
veloce
semplice
ottimo primo caricamento
buono per SEO
non richiede circuito SignalR
non scarica runtime WebAssembly
```

Limite:

```text id="655fmy"
gli eventi Blazor come @onclick non funzionano come interazione .NET client/server
se il componente resta solo statico.
```

---

# Interactive Server

**Interactive Server** rende il componente interattivo con codice .NET eseguito sul server.

Esempio:

```razor id="ha9b8v"
@rendermode InteractiveServer

<h1>Contatore</h1>

<p>@count</p>

<button @onclick="Incrementa">
    Incrementa
</button>

@code {
    private int count;

    private void Incrementa()
    {
        count++;
    }
}
```

In questa modalità:

```text id="8pdfff"
il codice C# resta sul server
gli eventi passano tramite connessione interattiva
la UI viene aggiornata dal server
```

È una scelta naturale per gestionali interni, dashboard aziendali e applicazioni dove si vuole usare molto C# lato server.

---

# Interactive WebAssembly

**Interactive WebAssembly** rende il componente interattivo nel browser.

Esempio:

```razor id="vq98lc"
@rendermode InteractiveWebAssembly

<h1>Contatore WebAssembly</h1>

<p>@count</p>

<button @onclick="Incrementa">
    Incrementa
</button>

@code {
    private int count;

    private void Incrementa()
    {
        count++;
    }
}
```

In questa modalità:

```text id="c9zi0l"
il codice C# del componente gira nel browser
il runtime WebAssembly viene scaricato
il componente chiama API per dati server-side
```

È utile quando serve maggiore interattività client-side o ridurre dipendenza da una connessione persistente al server.

---

# Auto render mode

**Interactive Auto** combina Interactive Server e WebAssembly.

La documentazione Microsoft sulla struttura del progetto spiega che il render mode automatico usa inizialmente interactive SSR mentre il runtime e il bundle .NET vengono scaricati nel browser; dopo l’attivazione del runtime WebAssembly, il rendering passa al client-side rendering. ([Microsoft Learn][5])

Schema:

```text id="gthq9l"
Prima visita
  Interactive Server mentre WebAssembly si scarica.

Visite successive
  Interactive WebAssembly se runtime e bundle sono disponibili.
```

Vantaggi:

```text id="44eqx6"
buona prima interazione
transizione verso client-side
esperienza più fluida dopo il download
```

Svantaggi:

```text id="o41n48"
maggiore complessità
attenzione a dove vive il codice
attenzione a servizi disponibili lato client/server
attenzione al progetto .Client
```

Regola pratica:

```text id="bd0t9r"
Interactive Auto è potente,
ma non è la scelta più semplice per iniziare.
```

---

# Scegliere il render mode

```text id="fn5y5m"
Static SSR
  pagine non interattive o poco interattive.

Interactive Server
  gestionali, intranet, app aziendali con logica server.

Interactive WebAssembly
  interattività client-side, SPA, scenari offline/PWA.

Interactive Auto
  compromesso avanzato tra prima interazione server e client successivo.
```

Domanda pratica:

```text id="ke0v5w"
Dove deve vivere l'interattività?
Sul server, nel browser, o inizialmente sul server e poi nel browser?
```

# Form con `EditForm`

Blazor usa `EditForm` per gestire form, binding e validazione.

Esempio:

```razor id="t36t1j"
@page "/clienti/nuovo"

<h1>Nuovo cliente</h1>

<EditForm Model="input" OnValidSubmit="SalvaAsync">
    <DataAnnotationsValidator />
    <ValidationSummary />

    <div>
        <label>Nome</label>
        <InputText @bind-Value="input.Nome" />
        <ValidationMessage For="() => input.Nome" />
    </div>

    <div>
        <label>Email</label>
        <InputText @bind-Value="input.Email" />
        <ValidationMessage For="() => input.Email" />
    </div>

    <button type="submit">Salva</button>
</EditForm>

@code {
    private CreaClienteInput input = new();

    private async Task SalvaAsync()
    {
        // salva cliente
        await Task.CompletedTask;
    }
}
```

La documentazione sui form Blazor spiega che `EditForm` può creare un `EditContext` basato sul modello assegnato, e che l’`EditContext` tiene traccia dei metadati del processo di modifica, inclusi campi modificati e messaggi di validazione correnti. ([Microsoft Learn][6])

# Esempio completo: elenco clienti interattivo

`Clienti.razor`:

```razor id="dl4e3b"
@page "/clienti"
@rendermode InteractiveServer
@inject ClienteApiClient ClientiApi

<h1>Clienti</h1>

<div>
    <input placeholder="Cerca..."
           @bind="search"
           @bind:event="oninput" />

    <button @onclick="CaricaAsync">
        Aggiorna
    </button>
</div>

@if (isLoading)
{
    <p>Caricamento...</p>
}
else if (clienti.Count == 0)
{
    <p>Nessun cliente trovato.</p>
}
else
{
    <table>
        <thead>
            <tr>
                <th>Nome</th>
                <th>Email</th>
                <th></th>
            </tr>
        </thead>
        <tbody>
        @foreach (ClienteDto cliente in clientiFiltrati)
        {
            <ClienteRow Cliente="cliente"
                        OnSeleziona="SelezionaCliente" />
        }
        </tbody>
    </table>
}

@if (clienteSelezionato is not null)
{
    <p>Cliente selezionato: @clienteSelezionato.Nome</p>
}

@code {
    private List<ClienteDto> clienti = [];
    private string search = "";
    private bool isLoading;
    private ClienteDto? clienteSelezionato;

    private IEnumerable<ClienteDto> clientiFiltrati =>
        string.IsNullOrWhiteSpace(search)
            ? clienti
            : clienti.Where(c =>
                c.Nome.Contains(search, StringComparison.OrdinalIgnoreCase) ||
                (c.Email?.Contains(search, StringComparison.OrdinalIgnoreCase) ?? false));

    protected override async Task OnInitializedAsync()
    {
        await CaricaAsync();
    }

    private async Task CaricaAsync()
    {
        isLoading = true;

        clienti = await ClientiApi.ListaAsync();

        isLoading = false;
    }

    private void SelezionaCliente(ClienteDto cliente)
    {
        clienteSelezionato = cliente;
    }
}
```

`ClienteRow.razor`:

```razor id="ieuws5"
<tr>
    <td>@Cliente.Nome</td>
    <td>@Cliente.Email</td>
    <td>
        <button @onclick="Seleziona">
            Seleziona
        </button>
    </td>
</tr>

@code {
    [Parameter]
    public ClienteDto Cliente { get; set; } = default!;

    [Parameter]
    public EventCallback<ClienteDto> OnSeleziona { get; set; }

    private async Task Seleziona()
    {
        await OnSeleziona.InvokeAsync(Cliente);
    }
}
```

DTO:

```csharp id="d306mt"
public record ClienteDto(
    int Id,
    string Nome,
    string? Email);
```

# Blazor e JavaScript

Blazor riduce la quantità di JavaScript necessaria, ma non lo elimina.

JavaScript resta utile per:

```text id="e5tjrr"
API browser non coperte direttamente
librerie JavaScript esistenti
grafici avanzati
focus e misure DOM
interop con widget esterni
download/upload particolari
```

Blazor fornisce `IJSRuntime` per chiamare JavaScript da C#.

Esempio:

```csharp id="32dlxs"
await JS.InvokeVoidAsync("alert", "Cliente salvato");
```

Regola pratica:

```text id="lwrxv7"
Usa C# per logica applicativa e stato dei componenti.
Usa JavaScript quando devi interagire con il browser o librerie JS.
```

# Errori comuni

## Pensare che Blazor sostituisca sempre Razor Pages/MVC

Blazor è ottimo per UI interattive.
Per pagine semplici, form classici o contenuti statici, Razor Pages può essere più semplice.

---

## Mettere troppa logica nel componente

Sospetto:

```text id="l3pd0a"
componente .razor con 500 righe
query database
validazione complessa
chiamate API
mapping
stato UI
regole di business
```

Meglio separare:

```text id="xc6gul"
componenti
servizi applicativi
client API
validatori
DTO
modelli di dominio
```

---

## Non capire dove gira il codice

Domanda fondamentale:

```text id="21vvko"
Questo codice gira sul server o nel browser?
```

In Interactive Server, il codice del componente gira sul server.

In Interactive WebAssembly, gira nel browser.

Questa differenza influenza:

```text id="9hidy6"
accesso a database
segreti
file system
sicurezza
performance
latenza
dipendenze disponibili
```

Non mettere mai segreti nel codice che finisce nel browser.

---

## Usare WebAssembly per accedere direttamente al database

Blazor WebAssembly gira nel browser.
Non deve connettersi direttamente a SQL Server.

Deve chiamare API backend:

```text id="p005g5"
Blazor WebAssembly
  → API ASP.NET Core
    → database
```

---

## Abusare di `StateHasChanged`

Se devi chiamare `StateHasChanged` continuamente, probabilmente lo stato non è organizzato bene.

---

## Dimenticare la validazione server-side

La validazione Blazor migliora la UX, ma il server deve comunque validare i dati quando riceve una richiesta.

---

## Componenti troppo generici

Un componente riutilizzabile è utile se ha una responsabilità chiara.

Sospetto:

```text id="gkezlm"
SuperDynamicFormGridModalTableComponent
```

Meglio componenti piccoli e componibili.

---

# Linee guida pratiche

```text id="665v3k"
Usa componenti piccoli
  con responsabilità chiara.

Passa dati ai figli con Parameter
  e notifiche al padre con EventCallback.

Usa @bind
  per form e input semplici.

Conosci il lifecycle
  soprattutto OnInitializedAsync, OnParametersSetAsync e OnAfterRenderAsync.

Scegli bene il render mode
  Static SSR, Interactive Server, Interactive WebAssembly o Auto.

Usa EditForm
  per form strutturati.

Usa DataAnnotationsValidator
  per validazione base.

Crea componenti riutilizzabili
  quando c'è markup o comportamento ripetuto.

Tieni la logica applicativa fuori dai componenti
  quando cresce.

Ricorda dove gira il codice
  server o browser cambia tutto.

Usa JavaScript interop solo quando serve.
```

# Riepilogo

In questo capitolo abbiamo introdotto Blazor come modello a componenti per interfacce web interattive.

Abbiamo visto:

```text id="4on2ah"
componenti
parametri
EventCallback
data binding
lifecycle
rendering
Blazor Server
Blazor WebAssembly
Blazor Web App
Static SSR
Interactive Server
Interactive WebAssembly
Auto render mode
form con EditForm
validazione
componenti riutilizzabili
```

Abbiamo chiarito che Blazor Web App può combinare rendering statico e interattivo, e che i componenti possono essere renderizzati in modalità Static SSR, Interactive Server, Interactive WebAssembly o Interactive Auto. ([Microsoft Learn][8])

Abbiamo visto che Auto render mode parte usando rendering interattivo lato server mentre il runtime e il bundle WebAssembly vengono scaricati, e poi può passare al rendering client-side quando WebAssembly è disponibile. ([Microsoft Learn][5])

Abbiamo anche visto che `EditForm` e `EditContext` sono alla base dei form Blazor: tengono traccia dello stato di modifica, dei campi e dei messaggi di validazione. ([Microsoft Learn][6])

La regola principale è questa:

```text id="nphs91"
Blazor è potente quando viene usato come architettura a componenti,
non quando ogni pagina diventa un blocco enorme di markup, stato,
query e logica applicativa mescolati insieme.
```

Blazor va trattato dopo le basi web e ASP.NET Core perché solo a quel punto il lettore può capire davvero cosa cambia tra rendering statico, interattività server-side, WebAssembly e API backend.

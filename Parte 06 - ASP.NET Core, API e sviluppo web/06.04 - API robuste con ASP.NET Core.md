# 06.04 — API robuste con ASP.NET Core

Nel capitolo precedente abbiamo creato API con Minimal API: routing, binding dei parametri, validazione, `Results`, Problem Details, endpoint group, filtri, OpenAPI, Scalar/Swagger UI, test e consumo con `HttpClient`.

Ora il livello sale.

Una API che “funziona” non è automaticamente una API professionale. Una API robusta deve essere:

```text id="tcesod"
chiara nel contratto
validata
sicura
osservabile
coerente negli errori
protetta da abuso
facile da consumare
versionabile
testabile
prevedibile nelle performance
```

Il punto centrale è questo:

```text id="d8axl8"
Una API robusta non è fatta solo di endpoint.
È fatta di contratti, validazione, sicurezza, gestione errori,
logging, policy, limiti, caching e convenzioni coerenti.
```

---

# API base vs API robusta

Una API base può essere così:

```csharp id="5tnqnl"
app.MapPost("/api/clienti", async (
    Cliente cliente,
    AppDbContext dbContext) =>
{
    dbContext.Clienti.Add(cliente);
    await dbContext.SaveChangesAsync();

    return Results.Ok(cliente);
});
```

Funziona, ma ha diversi problemi:

```text id="6f285g"
espone direttamente l'entità EF Core
non valida l'input
restituisce sempre Ok invece di Created
non gestisce errori in modo coerente
non protegge l'endpoint
non logga operazioni rilevanti
non separa DTO e dominio
```

Una versione più robusta usa:

```text id="ygj2x3"
DTO
validazione
mapping esplicito
status code corretti
Problem Details
servizi applicativi
logging
autenticazione/autorizzazione
pagination/filtering/sorting
```

# DTO

Un **DTO**, Data Transfer Object, è un oggetto pensato per comunicare dati attraverso i confini dell’applicazione.

In una API, i DTO servono per:

```text id="xml50q"
request
response
contratti pubblici
versionamento
validazione input
separazione da entità EF Core
```

---

## Perché non esporre direttamente le entità

Entità EF Core:

```csharp id="lo6r9v"
public class Cliente
{
    public int Id { get; set; }
    public string Nome { get; set; } = "";
    public string? Email { get; set; }
    public bool Attivo { get; set; }

    public List<Ordine> Ordini { get; set; } = new();
}
```

Restituirla direttamente:

```csharp id="9w1vgs"
return Results.Ok(cliente);
```

può creare problemi:

```text id="1ak6vo"
esposizione di campi interni
relazioni serializzate accidentalmente
cicli di navigazione
contratto API legato al database
difficoltà di versionamento
over-posting negli input
```

Meglio usare DTO.

---

## DTO di response

```csharp id="ck0q5z"
public record ClienteDto(
    int Id,
    string Nome,
    string? Email);
```

Query:

```csharp id="0tt1xn"
ClienteDto? cliente = await dbContext.Clienti
    .AsNoTracking()
    .Where(c => c.Id == id && c.Attivo)
    .Select(c => new ClienteDto(
        c.Id,
        c.Nome,
        c.Email))
    .FirstOrDefaultAsync();
```

---

## DTO di request

```csharp id="wunssn"
public record CreaClienteRequest(
    string Nome,
    string? Email);
```

Separare request e response è importante.

La request potrebbe non contenere `Id`, mentre la response sì.

# Validazione input

La validazione impedisce di elaborare dati non validi.

Esempi:

```text id="2zdjrm"
nome obbligatorio
email valida
prezzo non negativo
quantità maggiore di zero
data iniziale prima della data finale
pageSize entro un limite massimo
```

---

## Validazione con Data annotations

```csharp id="5x5h36"
using System.ComponentModel.DataAnnotations;

public record CreaClienteRequest(
    [property: Required]
    [property: MaxLength(100)]
    string Nome,

    [property: EmailAddress]
    [property: MaxLength(200)]
    string? Email);
```

In ASP.NET Core .NET 10 le Minimal API hanno supporto integrato alla validazione; la documentazione Microsoft indica che le API top-level `AddValidation` e il filtro di validazione integrato sono stabili, mentre nelle versioni precedenti la validazione nelle Minimal API va gestita esplicitamente o con librerie dedicate. ([Microsoft Learn][1])

Configurazione:

```csharp id="0utaoj"
builder.Services.AddValidation();
builder.Services.AddProblemDetails();
```

---

## Validazione manuale per regole di business

Le Data annotations sono utili per regole semplici, ma non bastano per tutte le regole.

Esempio:

```csharp id="6n9i9c"
if (request.DataA < request.DataDa)
{
    return Results.ValidationProblem(new Dictionary<string, string[]>
    {
        ["dataA"] = ["La data finale deve essere successiva alla data iniziale."]
    });
}
```

Regola pratica:

```text id="sjsqyu"
Data annotations
  per vincoli semplici sul DTO.

Validazione applicativa
  per regole che coinvolgono più campi o dati esistenti.

Validazione di dominio
  per invarianti che devono valere sempre nell'entità.
```

# Mapping

Il mapping trasforma un oggetto in un altro.

Esempi:

```text id="63k452"
CreaClienteRequest → Cliente
Cliente → ClienteDto
AggiornaClienteRequest → modifica su Cliente esistente
```

---

## Mapping manuale

```csharp id="sxuvyu"
public static class ClienteMapping
{
    public static Cliente ToEntity(this CreaClienteRequest request)
    {
        return new Cliente
        {
            Nome = request.Nome,
            Email = request.Email,
            Attivo = true
        };
    }

    public static ClienteDto ToDto(this Cliente cliente)
    {
        return new ClienteDto(
            cliente.Id,
            cliente.Nome,
            cliente.Email);
    }
}
```

Uso:

```csharp id="ypgm1p"
Cliente cliente = request.ToEntity();

dbContext.Clienti.Add(cliente);

await dbContext.SaveChangesAsync();

return Results.Created(
    $"/api/clienti/{cliente.Id}",
    cliente.ToDto());
```

---

## Mapping in proiezione EF Core

Spesso è meglio proiettare direttamente a DTO nella query:

```csharp id="hbzrow"
List<ClienteDto> clienti = await dbContext.Clienti
    .AsNoTracking()
    .Where(c => c.Attivo)
    .OrderBy(c => c.Nome)
    .Select(c => new ClienteDto(
        c.Id,
        c.Nome,
        c.Email))
    .ToListAsync();
```

Questo evita di caricare dati inutili.

# Error handling centralizzato

Una API robusta non dovrebbe gestire ogni eccezione con `try/catch` sparsi ovunque.

Sconsigliato:

```csharp id="2cv3zj"
app.MapPost("/api/clienti", async (...) =>
{
    try
    {
        // logica
    }
    catch (Exception ex)
    {
        return Results.Problem(ex.Message);
    }
});
```

Meglio avere una gestione centralizzata.

ASP.NET Core supporta Problem Details tramite `AddProblemDetails`; il servizio implementa `IProblemDetailsService` e permette di produrre risposte di errore coerenti per le API. ([Microsoft Learn][1])

---

## Configurazione base

```csharp id="m4c6f0"
builder.Services.AddProblemDetails();

var app = builder.Build();

if (!app.Environment.IsDevelopment())
{
    app.UseExceptionHandler();
}
```

Per Minimal API, `Results.Problem` e `Results.ValidationProblem` permettono di restituire errori coerenti.

# Logging

Il logging serve a capire cosa succede nell’applicazione.

In una API robusta dovremmo loggare:

```text id="4btye3"

errori
warning
tentativi non autorizzati

```

Esempio:

```csharp id="7y0fhg"
app.MapPost("/api/clienti", async (
    CreaClienteRequest request,
    ClienteService service,
    ILogger<Program> logger) =>
{
    logger.LogInformation(
        "Creazione cliente con email {Email}",
        request.Email);

    ClienteDto cliente = await service.CreaAsync(request);

    logger.LogInformation(
        "Cliente creato con ID {ClienteId}",
        cliente.Id);

    return Results.Created($"/api/clienti/{cliente.Id}", cliente);
});
```

## Logging strutturato

```csharp id="ljn0cb"

    "Cliente creato con ID {ClienteId}",

```


Il logging strutturato permette ai sistemi di log di indicizzare `ClienteId` come campo.

# JWT

JWT significa:

```text id="r442bi"
JSON Web Token
```

Nelle API, un JWT viene spesso inviato nell’header:

```http id="lzrxf7"
Authorization: Bearer eyJhbGciOi...
```

Il server valida il token e costruisce l’identità dell’utente.

La documentazione Microsoft per JWT bearer authentication spiega che ASP.NET Core può applicare autorizzazione tramite requisiti e policy, ruoli e claim, e che il bearer token viene usato per proteggere richieste HTTP verso API. ([Microsoft Learn][2])

---

## Configurazione JWT Bearer

Esempio semplificato:

```csharp id="xfbaod"
using Microsoft.AspNetCore.Authentication.JwtBearer;
using Microsoft.IdentityModel.Tokens;
using System.Text;

builder.Services
    .AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(options =>
    {
        options.TokenValidationParameters = new TokenValidationParameters
        {
            ValidateIssuer = true,
            ValidIssuer = builder.Configuration["Jwt:Issuer"],

            ValidateAudience = true,
            ValidAudience = builder.Configuration["Jwt:Audience"],

            ValidateLifetime = true,

            ValidateIssuerSigningKey = true,
            IssuerSigningKey = new SymmetricSecurityKey(
                Encoding.UTF8.GetBytes(
                    builder.Configuration["Jwt:Key"]!))
        };
    });

builder.Services.AddAuthorization();

var app = builder.Build();

app.UseAuthentication();
app.UseAuthorization();
```

Endpoint:

```csharp id="sakgfh"
app.MapGet("/api/ordini", () =>
{
    return Results.Ok();
})
.RequireAuthorization();
```

## Nota importante sulla generazione token

# Policy

Le **policy** permettono di esprimere regole di autorizzazione più ricche dei semplici ruoli.

Esempio:

```csharp id="wi9hax"
builder.Services.AddAuthorization(options =>
{
    options.AddPolicy("SoloAdmin", policy =>
    {
        policy.RequireRole("Admin");
    });

    options.AddPolicy("ClienteWrite", policy =>
    {
        policy.RequireClaim("scope", "clienti.write");
    });
});
```

Uso:

```csharp id="fygev4"
app.MapPost("/api/clienti", () =>
{
    return Results.Ok();
})
.RequireAuthorization("ClienteWrite");
```

La documentazione Microsoft indica che ASP.NET Core supporta authorization basata su requisiti e policy, ruoli e claim. ([Microsoft Learn][2])

# Rate limiting

Il **rate limiting** limita il numero di richieste che un client può fare in un certo periodo.

Serve per:

```text id="0jxxkd"
proteggere da abuso
limitare brute force
ridurre carico
proteggere endpoint costosi
migliorare stabilità
```

ASP.NET Core include middleware di rate limiting: le app configurano policy e le applicano agli endpoint; Microsoft raccomanda di testare e revisionare attentamente le app che lo usano prima del deployment. ([Microsoft Learn][3])

---

## Configurazione base

```csharp id="3cqd14"
using System.Threading.RateLimiting;

builder.Services.AddRateLimiter(options =>
{
    options.AddFixedWindowLimiter("fixed", limiterOptions =>
    {
        limiterOptions.PermitLimit = 100;
        limiterOptions.Window = TimeSpan.FromMinutes(1);
        limiterOptions.QueueProcessingOrder =
            QueueProcessingOrder.OldestFirst;
        limiterOptions.QueueLimit = 0;
    });
});

var app = builder.Build();

app.UseRateLimiter();

app.MapGet("/api/clienti", () =>
{
    return Results.Ok();
})
.RequireRateLimiting("fixed");
```

Se il limite viene superato, il server può restituire `429 Too Many Requests`.

# CORS

CORS significa:

```text id="bf0n4u"
Cross-Origin Resource Sharing
```

Il browser applica restrizioni quando una pagina web su un’origine prova a chiamare API su un’altra origine.

Esempio:

```text id="bc91d1"
Frontend
  https://app.example.com

API
  https://api.example.com
```

Sono origini diverse.

ASP.NET Core permette di configurare CORS per abilitare richieste cross-origin; la documentazione Microsoft mostra come abilitare CORS nell’app e raccomanda di consentire solo origini attendibili. ([Microsoft Learn][4])

---

## Configurazione CORS

```csharp id="d8d8sq"
builder.Services.AddCors(options =>
{
    options.AddPolicy("Frontend", policy =>
    {
        policy
            .WithOrigins("https://app.example.com")
            .WithMethods("GET", "POST", "PUT", "DELETE")
            .WithHeaders("Content-Type", "Authorization");
    });
});

var app = builder.Build();

app.UseCors("Frontend");
```

---

## Errore comune

Sconsigliato in produzione:

```csharp id="dll68q"
policy.AllowAnyOrigin()
      .AllowAnyHeader()
      .AllowAnyMethod();
```

Può andare in prototipi locali, ma in produzione bisogna limitare le origini.

Regola pratica:

```text id="s5xwou"
CORS non è autenticazione.
CORS è una regola del browser per consentire o bloccare chiamate cross-origin.
```

## Cache-Control

Esempio header:

```http id="8v4m14"
Cache-Control: public, max-age=60
```

Significa che la risposta può essere riusata per 60 secondi.

Non usare cache pubblica per dati personali o sensibili.

---

## Caching applicativo

Esempio con `IMemoryCache`:

```csharp id="w0wxtp"
public class ClienteLookupService
{
    private readonly IMemoryCache cache;
    private readonly AppDbContext dbContext;

    public ClienteLookupService(
        IMemoryCache cache,
        AppDbContext dbContext)
    {
        this.cache = cache;
        this.dbContext = dbContext;
    }

    public async Task<ClienteDto?> TrovaAsync(int id)
    {
        string cacheKey = $"cliente:{id}";

        return await cache.GetOrCreateAsync(cacheKey, async entry =>
        {
            entry.AbsoluteExpirationRelativeToNow =
                TimeSpan.FromMinutes(5);

            return await dbContext.Clienti
                .AsNoTracking()
                .Where(c => c.Id == id)
                .Select(c => new ClienteDto(c.Id, c.Nome, c.Email))
                .FirstOrDefaultAsync();
        });
    }
}
```

# Idempotenza

Una operazione è **idempotente** quando ripeterla più volte produce lo stesso effetto finale.

Esempi:

```text id="wvms9g"
GET /api/clienti/10
  idempotente

PUT /api/clienti/10
  normalmente idempotente

DELETE /api/clienti/10
  normalmente idempotente

POST /api/pagamenti
  spesso non idempotente
```

Il problema più comune è il doppio invio.

Esempio:

```text id="1p4xdg"
Il client invia POST /api/pagamenti.
La rete cade prima della risposta.
Il client riprova.
Il pagamento rischia di essere registrato due volte.
```

---

## Idempotency key

Una soluzione è usare una chiave di idempotenza.

Request:

```http id="hu1vvc"
POST /api/pagamenti
Idempotency-Key: 8f4f6a4f-2a11-4f8d-9bb6-1e7a7f6d8c20
```

Il server salva la chiave e il risultato.

Se arriva di nuovo la stessa chiave:

```text id="j4w413"
non riesegue l’operazione
restituisce il risultato già prodotto
```

---

## Quando serve davvero

L’idempotenza è importante per:

```text id="23zej5"
pagamenti
ordini
prenotazioni
invii email critici
operazioni contabili
import massivi
chiamate retry automatiche
```

Non serve complicare ogni endpoint, ma per operazioni sensibili è fondamentale.

# Pagination

La paginazione evita di restituire migliaia o milioni di record in una sola risposta.

Request:

```text id="605o5u"
/api/clienti?page=2&pageSize=50
```

DTO query:

```csharp id="59e7oa"
public record PaginationRequest(
    int Page = 1,
    int PageSize = 50);
```

Normalizzazione:

```csharp id="ljl0aa"
int page = Math.Max(1, request.Page);
int pageSize = Math.Clamp(request.PageSize, 1, 100);
```

Query:

```csharp id="z7p1p4"
List<ClienteDto> clienti = await dbContext.Clienti
    .AsNoTracking()
    .Where(c => c.Attivo)
    .OrderBy(c => c.Nome)
    .Skip((page - 1) * pageSize)
    .Take(pageSize)
    .Select(c => new ClienteDto(c.Id, c.Nome, c.Email))
    .ToListAsync();
```

---

## Risposta paginata

```csharp id="xxfxat"
public record PagedResult<T>(

    int Page,
    int PageSize,
    int TotalCount);
```

Endpoint:

```csharp id="yv2qx2"
int totalCount = await query.CountAsync();
```

# Sorting

Il sorting permette al client di scegliere l’ordinamento.

Esempio:

```text id="ihn0ee"
/api/clienti?sortBy=nome&sortDirection=asc
```

DTO:

```csharp id="hdr2n2"
public record SortingRequest(
    string? SortBy,
    string? SortDirection);
```

Implementazione sicura:

```csharp id="xjtogl"
query = (sort.SortBy?.ToLowerInvariant(), sort.SortDirection?.ToLowerInvariant()) switch
{
    ("nome", "desc") => query.OrderByDescending(c => c.Nome),
    ("nome", _) => query.OrderBy(c => c.Nome),

    ("email", "desc") => query.OrderByDescending(c => c.Email),
    ("email", _) => query.OrderBy(c => c.Email),

    ("id", "desc") => query.OrderByDescending(c => c.Id),
    ("id", _) => query.OrderBy(c => c.Id),

    _ => query.OrderBy(c => c.Nome)
};
```

Non concatenare direttamente input utente in SQL per ordinamenti dinamici.

# Esempio completo: endpoint lista clienti robusto

Request:

```csharp id="b2okjp"
public record ListaClientiRequest(
    string? Search,
    string? Citta,
    bool? Attivo,
    int Page = 1,
    int PageSize = 50,
    string? SortBy = "nome",
    string? SortDirection = "asc");
```

Response:

```csharp id="f2a8l3"
public record ClienteDto(
    int Id,
    string Nome,
    string? Email,
    string? Citta);

public record PagedResult<T>(
    IReadOnlyList<T> Items,
    int Page,
    int PageSize,
    int TotalCount);
```

Endpoint:

```csharp id="uqxizo"
app.MapGet("/api/clienti", async (
    [AsParameters] ListaClientiRequest request,
    AppDbContext dbContext,
    ILogger<Program> logger) =>
{
    int page = Math.Max(1, request.Page);
    int pageSize = Math.Clamp(request.PageSize, 1, 100);

    IQueryable<Cliente> query = dbContext.Clienti
        .AsNoTracking();

    if (!string.IsNullOrWhiteSpace(request.Search))
    {
        string search = request.Search.Trim();

        query = query.Where(c =>
            c.Nome.Contains(search) ||
            (c.Email != null && c.Email.Contains(search)));
    }

    if (!string.IsNullOrWhiteSpace(request.Citta))
    {
        query = query.Where(c => c.Citta == request.Citta);
    }

    if (request.Attivo.HasValue)
    {
        query = query.Where(c => c.Attivo == request.Attivo.Value);
    }

    query = (request.SortBy?.ToLowerInvariant(),
             request.SortDirection?.ToLowerInvariant()) switch
    {
        ("id", "desc") => query.OrderByDescending(c => c.Id),
        ("id", _) => query.OrderBy(c => c.Id),

        ("email", "desc") => query.OrderByDescending(c => c.Email),
        ("email", _) => query.OrderBy(c => c.Email),

        ("nome", "desc") => query.OrderByDescending(c => c.Nome),
        _ => query.OrderBy(c => c.Nome)
    };

    int totalCount = await query.CountAsync();

    List<ClienteDto> items = await query
        .Skip((page - 1) * pageSize)
        .Take(pageSize)
        .Select(c => new ClienteDto(
            c.Id,
            c.Nome,
            c.Email,
            c.Citta))
        .ToListAsync();

    logger.LogInformation(
        "Lista clienti restituita. Page={Page}, PageSize={PageSize}, TotalCount={TotalCount}",
        page,
        pageSize,
        totalCount);

    return Results.Ok(new PagedResult<ClienteDto>(
        items,
        page,
        pageSize,
        totalCount));
});
```

# Errori comuni

## Esporre entità come contratto API

Sbagliato:

```csharp id="li35yz"
return Results.Ok(clienteEntity);
```

Meglio:

```csharp id="dj2a2l"
return Results.Ok(clienteDto);
```

---

## Validare solo lato client

La validazione JavaScript è utile per UX, ma non è sicurezza.

Il server deve sempre validare.

---

## Restituire messaggi di eccezione grezzi

Sbagliato:

```csharp id="rb5e46"
return Results.Problem(ex.ToString());
```

In produzione può esporre stack trace, query, path o dati sensibili.

Meglio Problem Details centralizzato e logging interno.

---

## Confondere autenticazione e autorizzazione

```text id="o8w9kn"
Autenticazione
  chi sei?

Autorizzazione
  cosa puoi fare?
```

Un utente autenticato non è automaticamente autorizzato a tutto.

---

## Accettare CORS da qualunque origine

Sconsigliato in produzione:

```csharp id="w2y97v"
AllowAnyOrigin()
```

Limitare le origini note.

---

## Pagination senza limite massimo

Sbagliato:

```text id="knx25f"
/api/clienti?pageSize=1000000
```

Meglio applicare un massimo:

```csharp id="b7ibtd"
int pageSize = Math.Clamp(request.PageSize, 1, 100);
```

---

## Sorting dinamico non controllato

Non generare SQL concatenando direttamente il nome colonna ricevuto dal client.

Usa whitelist:

```text id="xmmpih"
nome
email
id
dataCreazione
```

---

## Non pensare all’idempotenza

Per endpoint critici, i retry possono creare duplicati.

Valuta idempotency key per:

```text id="z5x8bs"
pagamenti
ordini
prenotazioni
movimenti contabili
```

# Linee guida pratiche

```text id="eekcpo"
Usa DTO
  per separare API, dominio e database.

Valida sempre input server-side
  anche se il frontend valida già.

Centralizza error handling
  con Problem Details.

Usa logging strutturato
  e non loggare dati sensibili.

Proteggi endpoint
  con autenticazione e autorizzazione.

Usa policy
  per regole di accesso esplicite.

Usa JWT
  per API stateless quando appropriato.

Applica rate limiting
  sugli endpoint pubblici o costosi.

Configura CORS
  solo per origini fidate.

Usa caching
  solo quando è coerente con sicurezza e freschezza dati.

Progetta idempotenza
  per operazioni sensibili ai retry.

Implementa pagination
  con limiti massimi.

Implementa filtering e sorting
  in modo controllato e sicuro.
```

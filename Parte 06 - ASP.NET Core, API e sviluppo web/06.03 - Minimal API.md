# 06.03 — Minimal API

Nel capitolo precedente abbiamo introdotto ASP.NET Core: `Program.cs`, middleware pipeline, routing, endpoint, file statici, configurazione, logging, Dependency Injection, ambienti, Kestrel e hosting.

Ora costruiamo le prime API.

Le **Minimal API** sono un modo diretto e compatto per creare endpoint HTTP in ASP.NET Core senza dover partire subito da controller, classi base, attributi e struttura MVC completa.

Esempio minimo:

```csharp id="b805ab"
var builder = WebApplication.CreateBuilder(args);

var app = builder.Build();

app.MapGet("/", () => "Ciao API!");

app.Run();
```

Il punto centrale è questo:

```text id="hhl343"
Minimal API permette di definire endpoint HTTP direttamente in Program.cs
o in metodi di estensione, usando route, handler, binding dei parametri,
Dependency Injection e risultati HTTP tipizzati.
```

Le Minimal API non sono un giocattolo didattico: sono un approccio reale per backend, microservizi, API interne, servizi semplici e applicazioni modulari. Microsoft le documenta come un modo per creare API HTTP con dipendenze minime, usando route handler e binding automatico dei parametri. ([Microsoft Learn][1])

Esempi:


# Binding parametri

Il **binding** è il processo con cui ASP.NET Core prende dati dalla richiesta HTTP e li converte in parametri C#.

La documentazione Microsoft definisce il parameter binding come la conversione dei dati della request in parametri fortemente tipizzati espressi dai route handler; le sorgenti possono essere route values, query string, header, body JSON, form values, servizi DI e binding personalizzato. ([Microsoft Learn][2])

---

```csharp id="ffwv9z"

{
    return Results.Ok(id);
});
```

# `Results` e `TypedResults`

`Results` è comodo e molto usato:

```csharp id="oj71lg"
return Results.Ok(cliente);
```

Nelle Minimal API moderne esiste anche `TypedResults`, utile per risultati più tipizzati.

```csharp id="2qv4wf"
return TypedResults.Ok(cliente);
```

Esempio:

```csharp id="qgw5vo"
app.MapGet("/api/clienti/{id:int}", async Task<Results<Ok<ClienteDto>, NotFound>> (
    int id,
    ClienteService service) =>
{
    ClienteDto? cliente = await service.TrovaAsync(id);

    return cliente is null
        ? TypedResults.NotFound()
        : TypedResults.Ok(cliente);
});
```

Questa forma è più verbosa, ma documenta meglio i possibili risultati e può aiutare OpenAPI e analisi statica.

Per un libro didattico, si può iniziare con `Results` e introdurre `TypedResults` come opzione più avanzata.

# Problem Details

Per gli errori HTTP moderni si usa spesso **Problem Details**, basato sul formato standard RFC 7807/RFC 9457.

In ASP.NET Core:

```csharp id="d95k5t"
builder.Services.AddProblemDetails();
```

Endpoint:

```csharp id="y5puxq"
app.MapGet("/api/clienti/{id:int}", (int id) =>
{
    if (id <= 0)
    {
        return Results.Problem(
            title: "ID non valido",
            detail: "L'ID cliente deve essere maggiore di zero.",
            statusCode: StatusCodes.Status400BadRequest);
    }

    return Results.Ok(new ClienteDto(id, "Emilie", null));
});
```

Risposta concettuale:

```json id="gj7w81"
{
  "type": "about:blank",
  "title": "ID non valido",
  "status": 400,
  "detail": "L'ID cliente deve essere maggiore di zero."
}
```

Problem Details è utile perché dà un formato coerente agli errori.

## Validazione manuale

DTO:

```csharp id="s6gx4i"
public record CreaClienteRequest(
    string Nome,
    string? Email);
```

Endpoint:

```csharp id="0ymyjp"
app.MapPost("/api/clienti", (CreaClienteRequest request) =>
{
    Dictionary<string, string[]> errors = new();

    if (string.IsNullOrWhiteSpace(request.Nome))
    {
        errors["nome"] = ["Il nome è obbligatorio."];
    }

    if (request.Email is not null &&
        !request.Email.Contains('@'))
    {
        errors["email"] = ["Email non valida."];
    }

    if (errors.Count > 0)
    {
        return Results.ValidationProblem(errors);
    }

    ClienteDto cliente = new ClienteDto(
        10,
        request.Nome,
        request.Email);

    return Results.Created($"/api/clienti/{cliente.Id}", cliente);
});
```

È esplicita, ma può diventare ripetitiva.

# Data annotations

Le **Data annotations** permettono di mettere regole di validazione sui DTO.

```csharp id="wb49g8"
using System.ComponentModel.DataAnnotations;

public record CreaClienteRequest(
    [property: Required]
    [property: MaxLength(100)]
    string Nome,

    [property: EmailAddress]
    [property: MaxLength(200)]
    string? Email);
```

Nota importante: sugli argomenti del costruttore di un `record`, gli attributi vanno spesso indicati con `property:` per applicarsi alla proprietà generata.

## Validazione Minimal API in .NET 10

In ASP.NET Core .NET 10 è stato introdotto il supporto alla validazione per Minimal API. La documentazione Microsoft indica che il supporto è stato introdotto in .NET 10 e che si possono personalizzare le risposte di errore tramite `IProblemDetailsService`; le API top-level `AddValidation` e il filtro di validazione integrato sono stabili. ([Microsoft Learn][1])

Configurazione:

```csharp id="gd2kbb"
builder.Services.AddValidation();
builder.Services.AddProblemDetails();
```

Endpoint:

```csharp id="q9ehpn"
app.MapPost("/api/clienti", (CreaClienteRequest request) =>
{
    ClienteDto cliente = new ClienteDto(
        10,
        request.Nome,
        request.Email);

    return Results.Created($"/api/clienti/{cliente.Id}", cliente);
});
```

Se il payload non rispetta le regole, ASP.NET Core può restituire una risposta di validazione senza scrivere manualmente tutti i controlli.

Attenzione: nelle versioni precedenti a .NET 10, le Data annotations nelle Minimal API contribuiscono alla metadata OpenAPI, ma la validazione va eseguita esplicitamente tramite endpoint filter, logica manuale o librerie di terze parti. La documentazione Microsoft lo chiarisce per gli scenari Minimal API e OpenAPI metadata. ([Microsoft Learn][3])

# Organizzare endpoint fuori da `Program.cs`

Per evitare un `Program.cs` enorme, possiamo usare metodi di estensione.

```csharp id="7x2idx"
public static class ClientiEndpoints
{
    public static RouteGroupBuilder MapClientiEndpoints(
        this IEndpointRouteBuilder app)
    {
        RouteGroupBuilder group = app.MapGroup("/api/clienti")
            .WithTags("Clienti");

        group.MapGet("/", GetClientiAsync);
        group.MapGet("/{id:int}", GetClienteAsync);
        group.MapPost("/", CreaClienteAsync);

        return group;
    }

    private static async Task<IResult> GetClientiAsync(
        ClienteService service)
    {
        List<ClienteDto> clienti = await service.ListaAsync();

        return Results.Ok(clienti);
    }

    private static async Task<IResult> GetClienteAsync(
        int id,
        ClienteService service)
    {
        ClienteDto? cliente = await service.TrovaAsync(id);

        return cliente is null
            ? Results.NotFound()
            : Results.Ok(cliente);
    }

    private static async Task<IResult> CreaClienteAsync(
        CreaClienteRequest request,
        ClienteService service)
    {
        ClienteDto cliente = await service.CreaAsync(request);

        return Results.Created($"/api/clienti/{cliente.Id}", cliente);
    }
}
```

In `Program.cs`:

```csharp id="c3zexv"
app.MapClientiEndpoints();
```

Questo mantiene il progetto più leggibile.

# Filtri

Gli **endpoint filters** permettono di eseguire logica prima o dopo l’endpoint.

La documentazione Microsoft spiega che i filtri Minimal API possono eseguire codice prima e dopo il route handler, ispezionare o modificare i parametri e intercettare il comportamento della risposta. ([Microsoft Learn][4])

Esempio semplice:

```csharp id="4ce5o2"
app.MapPost("/api/clienti", (CreaClienteRequest request) =>
{
    return Results.Ok();
})
.AddEndpointFilter(async (context, next) =>
{
    Console.WriteLine("Prima dell'endpoint");

    object? result = await next(context);

    Console.WriteLine("Dopo l'endpoint");

    return result;
});
```

---

## Filtro per validare ID positivo

```csharp id="3a3l2m"
public class PositiveIdFilter : IEndpointFilter
{
    public async ValueTask<object?> InvokeAsync(
        EndpointFilterInvocationContext context,
        EndpointFilterDelegate next)
    {
        int? id = context.Arguments
            .OfType<int>()
            .FirstOrDefault();

        if (id <= 0)
        {
            return Results.BadRequest("ID non valido.");
        }

        return await next(context);
    }
}
```

Uso:

```csharp id="4s0sh5"
app.MapGet("/api/clienti/{id:int}", (int id) =>
{
    return Results.Ok(new ClienteDto(id, "Emilie", null));
})
.AddEndpointFilter<PositiveIdFilter>();
```

---

## Quando usare filtri

Usali per logica trasversale leggera:

```text id="qc11id"
validazione comune
logging mirato
controlli su parametri
misurazione tempi
controlli di idempotenza
normalizzazione input
```

Non usarli per nascondere logica di business importante: quella dovrebbe stare nei servizi applicativi.

# Versionamento API

Il versionamento serve quando una API cambia ma deve continuare a supportare client esistenti.

Strategie comuni:

```text id="tqk0d5"
versione nell’URL
  /api/v1/clienti
  /api/v2/clienti

versione in query string
  /api/clienti?api-version=1.0

versione in header
  X-Api-Version: 1.0

versione nel media type
  Accept: application/vnd.miaapi.v1+json
```

Per un libro introduttivo, la versione nell’URL è la più semplice da capire.

```csharp id="x4g4n1"
RouteGroupBuilder v1 = app.MapGroup("/api/v1/clienti")
    .WithTags("Clienti v1");

RouteGroupBuilder v2 = app.MapGroup("/api/v2/clienti")
    .WithTags("Clienti v2");
```

---

## Versionamento semplice con route group

```csharp id="re6vfx"
RouteGroupBuilder apiV1 = app.MapGroup("/api/v1");

apiV1.MapGet("/clienti/{id:int}", (int id) =>
{
    return Results.Ok(new
    {
        Id = id,
        Nome = "Emilie"
    });
});

RouteGroupBuilder apiV2 = app.MapGroup("/api/v2");

apiV2.MapGet("/clienti/{id:int}", (int id) =>
{
    return Results.Ok(new
    {
        Id = id,
        NomeCompleto = "Emilie Rollandin",

        {
            Self = $"/api/v2/clienti/{id}"
        }
    });

```

Regola pratica:

```text id="ydmzbh"
Non versionare troppo presto.
Versiona quando hai client esterni o contratti che devono restare stabili.
```

# OpenAPI

OpenAPI è una specifica per descrivere API HTTP.

Un documento OpenAPI descrive:

```text id="f1epml"
endpoint
metodi HTTP
parametri
request body
response
status code
schema JSON
autenticazione
metadata
```

ASP.NET Core include supporto alla generazione di documenti OpenAPI tramite il pacchetto `Microsoft.AspNetCore.OpenApi`; Microsoft precisa però che ASP.NET Core genera documenti OpenAPI, mentre interfacce interattive come Swagger UI o Scalar non sono incluse di default e vanno aggiunte separatamente. ([Microsoft Learn][5])

---

## Configurazione OpenAPI

Pacchetto:

```bash id="ubj9dw"
dotnet add package Microsoft.AspNetCore.OpenApi
```

`Program.cs`:

```csharp id="7eukpp"
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddOpenApi();

var app = builder.Build();

if (app.Environment.IsDevelopment())
{
    app.MapOpenApi();
}

app.MapGet("/api/clienti", () =>
{
    return Results.Ok(Array.Empty<ClienteDto>());
})
.WithName("GetClienti")
.WithTags("Clienti")
.Produces<ClienteDto[]>(StatusCodes.Status200OK);

app.Run();
```

Documento disponibile, tipicamente:

```text id="vjac0m"
/openapi/v1.json
```

```csharp id="eh5j9p"
CreaClienteRequest request = new(
```

```csharp id="hmiiva"
public class ClienteStore
{
    private readonly List<ClienteDto> clienti =
    [
        new ClienteDto(1, "Emilie", "emilie@example.com"),
        new ClienteDto(2, "Mario", "mario@example.com")
    ];

    private int nextId = 3;

    public List<ClienteDto> Lista()
    {
        return clienti
            .OrderBy(c => c.Nome)
            .ToList();
    }

    public ClienteDto? Trova(int id)
    {
        return clienti.FirstOrDefault(c => c.Id == id);
    }

    public ClienteDto Crea(CreaClienteRequest request)
    {
        ClienteDto cliente = new ClienteDto(
            nextId++,
            request.Nome,
            request.Email);

        clienti.Add(cliente);

        return cliente;
    }

    public ClienteDto? Aggiorna(
        int id,
        AggiornaClienteRequest request)
    {
        int index = clienti.FindIndex(c => c.Id == id);

        if (index < 0)
        {
            return null;
        }

        ClienteDto cliente = new ClienteDto(
            id,
            request.Nome,
            request.Email);

        clienti[index] = cliente;

        return cliente;
    }

    public bool Elimina(int id)
    {
        ClienteDto? cliente = Trova(id);

        if (cliente is null)
        {
            return false;
        }

        clienti.Remove(cliente);

        return true;
    }
}
```

```csharp id="mpmqea"
using Microsoft.EntityFrameworkCore;

public static class ClientiEndpoints
{
    public static RouteGroupBuilder MapClientiEndpoints(
        this IEndpointRouteBuilder app)
    {
        RouteGroupBuilder group = app.MapGroup("/api/clienti")
            .WithTags("Clienti");

        group.MapGet("/", ListaAsync);
        group.MapGet("/{id:int}", TrovaAsync);
        group.MapPost("/", CreaAsync);
        group.MapDelete("/{id:int}", EliminaAsync);

        return group;
    }

    private static async Task<IResult> ListaAsync(
        AppDbContext dbContext)
    {
        List<ClienteDto> clienti = await dbContext.Clienti
            .AsNoTracking()
            .Where(c => c.Attivo)
            .OrderBy(c => c.Nome)
            .Select(c => new ClienteDto(
                c.Id,
                c.Nome,
                c.Email))
            .ToListAsync();

        return Results.Ok(clienti);
    }

    private static async Task<IResult> TrovaAsync(
        int id,
        AppDbContext dbContext)
    {
        ClienteDto? cliente = await dbContext.Clienti
            .AsNoTracking()
            .Where(c => c.Id == id && c.Attivo)
            .Select(c => new ClienteDto(
                c.Id,
                c.Nome,
                c.Email))
            .FirstOrDefaultAsync();

        return cliente is null
            ? Results.NotFound()
            : Results.Ok(cliente);
    }

    private static async Task<IResult> CreaAsync(
        CreaClienteRequest request,
        AppDbContext dbContext)
    {
        Cliente cliente = new Cliente
        {
            Nome = request.Nome,
            Email = request.Email,
            Attivo = true
        };

        dbContext.Clienti.Add(cliente);

        await dbContext.SaveChangesAsync();

        ClienteDto dto = new ClienteDto(
            cliente.Id,
            cliente.Nome,
            cliente.Email);

        return Results.Created($"/api/clienti/{dto.Id}", dto);
    }

    private static async Task<IResult> EliminaAsync(
        int id,
        AppDbContext dbContext)
    {
        Cliente? cliente = await dbContext.Clienti
            .FirstOrDefaultAsync(c => c.Id == id);

        if (cliente is null)
        {
            return Results.NotFound();
        }

        cliente.Attivo = false;

        await dbContext.SaveChangesAsync();

        return Results.NoContent();
    }
}
```

## Confondere endpoint filter con middleware

Middleware:

```text id="w9dthl"

```

Endpoint filter:


# Riepilogo

In questo capitolo abbiamo introdotto le Minimal API.

Abbiamo visto:

```text id="razvue"
creazione API minimale
routing
binding parametri
validazione
Data annotations
risposte HTTP
Results
Problem Details
endpoint group
filtri
versionamento API
OpenAPI
Scalar / Swagger UI
test delle API
consumo API con HttpClient
```

Abbiamo chiarito che il parameter binding delle Minimal API può leggere dati da route, query string, header, body JSON, form values, servizi DI e binding personalizzato. ([Microsoft Learn][2])

Abbiamo visto che in .NET 10 le Minimal API hanno supporto integrato alla validazione, mentre nelle versioni precedenti le Data annotations contribuiscono alla metadata OpenAPI ma la validazione deve essere gestita esplicitamente. ([Microsoft Learn][1])

Abbiamo anche distinto il documento OpenAPI dalla UI interattiva: ASP.NET Core genera il documento, mentre Scalar o Swagger UI sono strumenti separati per consultarlo e provarlo. ([Microsoft Learn][5])

La regola principale è questa:

```text id="mwksq9"
Minimal API è semplice quando ogni endpoint resta piccolo,
quando la logica applicativa viene spostata nei servizi
e quando request, response, validazione e status code sono progettati con cura.
```

Nel prossimo capitolo possiamo passare alle Web API più strutturate con controller, oppure proseguire con Razor Pages/MVC per il server-side rendering.

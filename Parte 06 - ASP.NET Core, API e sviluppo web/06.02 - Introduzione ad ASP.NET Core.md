# 06.02 — Introduzione ad ASP.NET Core

Nel capitolo precedente abbiamo costruito le basi del web: HTTP, request, response, header, status code, query string, form, JSON, cookie, sessione, HTML, CSS, JavaScript, client/server, server-side rendering e single-page application.

Ora possiamo introdurre **ASP.NET Core**.

ASP.NET Core è il framework moderno di Microsoft per creare applicazioni web, API, servizi backend, applicazioni server-side rendering e applicazioni Blazor.

Il punto centrale è questo:

```text id="4wr5ay"
ASP.NET Core riceve richieste HTTP, le fa passare in una pipeline di middleware
e le collega a endpoint che producono risposte HTTP.
```

Questa idea è la base comune di:

```text id="nxw9z6"
Minimal API
MVC
Razor Pages
Blazor
Web API
servizi backend
applicazioni web server-side
```

La documentazione Microsoft descrive ASP.NET Core come un framework che include concetti fondamentali come Dependency Injection, configurazione, middleware, routing, ambienti e hosting. ([Microsoft Learn][1])

Per creare un progetto ASP.NET Core possiamo usare Visual Studio oppure la CLI `dotnet`.

Esempio con Minimal API:

```bash id="l3zdy9"

cd DemoWeb
dotnet run
```

Esempio con Web API:

```bash id="vkk28u"

```

Esempio Razor Pages:

Esempio Blazor:


Il tipo di progetto cambia, ma la base resta ASP.NET Core:

# Middleware pipeline

La **middleware pipeline** è una catena di componenti che elaborano ogni richiesta HTTP.

Un middleware può:

```text id="wpy8ge"
leggere la request
modificare la request
decidere se passare al middleware successivo
eseguire codice prima e dopo il middleware successivo
modificare la response
interrompere la pipeline
```

Microsoft definisce il middleware come software assemblato in una pipeline per gestire richieste e risposte; ogni middleware può scegliere se passare la richiesta al successivo e può eseguire operazioni prima e dopo il successivo. ([Microsoft Learn][2])

---

## Esempio concettuale di pipeline

```text id="s5thkh"
Request
  ↓
Exception handling middleware
  ↓
HTTPS redirection middleware
  ↓
Static files middleware / static assets endpoints
  ↓
Routing
  ↓
Authentication
  ↓
Authorization
  ↓
Endpoint
  ↓
Response
```

L’ordine conta moltissimo.

Un middleware messo nel punto sbagliato può non funzionare o produrre effetti inattesi.

---

## Middleware semplice

```csharp id="8s7ju6"
app.Use(async (context, next) =>
{
    Console.WriteLine($"Request: {context.Request.Method} {context.Request.Path}");

    await next();

    Console.WriteLine($"Response: {context.Response.StatusCode}");
});
```

Questo middleware:

```text id="7rezg0"
scrive informazioni sulla request
passa al middleware successivo
scrive informazioni sulla response
```

---

## Middleware terminale

Un middleware può anche terminare la pipeline.

```csharp id="rb1fl9"
app.Run(async context =>
{
    await context.Response.WriteAsync("Risposta finale");
});
```

Dopo un middleware terminale, la richiesta non prosegue ad altri middleware.

# Routing

Il **routing** decide quale endpoint deve gestire una richiesta.

Esempio:

```csharp id="ip95z5"
app.MapGet("/", () => "Home");

app.MapGet("/clienti", () => "Elenco clienti");

app.MapGet("/clienti/{id:int}", (int id) =>
{
    return $"Cliente {id}";
});
```

Richieste:

```text id="b6718t"
GET /
  endpoint Home

GET /clienti
  endpoint elenco clienti

GET /clienti/10
  endpoint dettaglio cliente
```

---

## Route parameters

```csharp id="twmi2g"
app.MapGet("/clienti/{id:int}", (int id) =>
{
    return Results.Ok(new
    {
        Id = id,
        Nome = "Emilie"
    });
});
```

La parte:

```text id="yyub7n"
{id:int}
```

indica un parametro di route chiamato `id`, vincolato a intero.

---

## Query string

```csharp id="gjb8xe"
app.MapGet("/clienti", (string? citta, bool? attivo) =>
{
    return Results.Ok(new

        Citta = citta,

```


# `MapStaticAssets`

Da .NET 9, ASP.NET Core introduce le convenzioni `MapStaticAssets` per servire asset statici conosciuti dall’applicazione a runtime.

Esempio:

```csharp id="pc2tw6"
var builder = WebApplication.CreateBuilder(args);

var app = builder.Build();

app.MapStaticAssets();

app.MapGet("/", () => "Home");

app.Run();
```

Secondo la documentazione Microsoft, `MapStaticAssets` è usato nelle app che targettano .NET 9 o successivo; rispetto al solo Static File Middleware, fornisce ottimizzazioni come compressione a build-time e fingerprinting, quando gli asset sono noti all’applicazione. ([Microsoft Learn][4])

---

## `UseStaticFiles`

Il middleware classico per file statici è:

```csharp id="gphg1y"
app.UseStaticFiles();
```

È ancora utile quando dobbiamo servire file da posizioni particolari o scenari non coperti da `MapStaticAssets`.

Esempio:

```csharp id="yfto9i"
app.UseStaticFiles();
```

Per app precedenti a .NET 9, `UseStaticFiles` resta il meccanismo da usare. La documentazione segnala che Static File Middleware deve essere usato nelle app che targettano versioni precedenti a .NET 9, mentre nelle app .NET 9+ si usano le convenzioni `MapStaticAssets` per gli asset statici noti all’app. ([Microsoft Learn][4])

---

## Quando usare l’uno o l’altro

```text id="6erpib"
MapStaticAssets
  per asset noti all’app in .NET 9+,
  con ottimizzazioni come compressione e fingerprinting.

UseStaticFiles
  per scenari classici,
  file fuori da wwwroot,
  file aggiunti dopo il publish,
  percorsi o provider personalizzati.
```

Nel libro conviene presentare entrambi, spiegando che `MapStaticAssets` è la direzione moderna per gli asset statici nelle versioni recenti di ASP.NET Core.

# Dependency Injection

ASP.NET Core usa Dependency Injection in modo nativo.

Registrazione:

```csharp id="5zsd4z"
builder.Services.AddScoped<IClienteRepository, ClienteRepository>();
builder.Services.AddScoped<ClienteService>();
```

Uso in endpoint:

```csharp id="wqtohj"
app.MapGet("/api/clienti/{id:int}", async (
    int id,
    ClienteService clienteService) =>
{
    ClienteDto? cliente =
        await clienteService.TrovaAsync(id);

    return cliente is null
        ? Results.NotFound()
        : Results.Ok(cliente);
});
```

Uso in controller, Razor Pages, Blazor e servizi:

```text id="kb2sf3"
constructor injection
parametri endpoint
servizi registrati nel container
```

La documentazione sui fondamentali ASP.NET Core include Dependency Injection come uno dei concetti centrali del framework. ([Microsoft Learn][1])

## Lifetime dei servizi

I lifetime principali sono:

```text id="6x5v5d"
Singleton
  una istanza per tutta l’applicazione.

Scoped
  una istanza per richiesta HTTP.

Transient
  una nuova istanza a ogni richiesta dal container.
```

Esempio:

```csharp id="cgva6w"
builder.Services.AddSingleton<IClock, SystemClock>();
builder.Services.AddScoped<ClienteService>();
builder.Services.AddTransient<EmailFormatter>();
```

In ASP.NET Core, `Scoped` è particolarmente importante perché spesso coincide con la singola request HTTP.

# Ordine tipico della pipeline

Un ordine ragionevole per molte app è:

```csharp id="dnv5wv"
var app = builder.Build();

if (!app.Environment.IsDevelopment())
{
    app.UseExceptionHandler("/error");
    app.UseHsts();
}

app.MapStaticAssets();

app.UseRouting();

app.UseAuthentication();
app.UseAuthorization();

app.MapControllers();

app.Run();
```

Nota: in Minimal API moderne molte operazioni di routing sono integrate nel modello `Map...`, ma è utile capire concettualmente la pipeline.

# Errori comuni

## Ignorare l’ordine dei middleware

Sbagliato:

```csharp id="09yjhc"
app.UseAuthorization();
app.UseAuthentication();
```

L’autenticazione deve avvenire prima dell’autorizzazione.

Corretto:

```csharp id="3e4ega"
app.UseAuthentication();
app.UseAuthorization();
```

---

## Mettere dettagli tecnici in produzione

Sbagliato:

```csharp id="e4hyih"
app.UseDeveloperExceptionPage();
```

in produzione.

Meglio:

```csharp id="nyqob7"
if (app.Environment.IsDevelopment())
{
    app.UseDeveloperExceptionPage();
}
else
{
    app.UseExceptionHandler("/error");
}
```

---

## Confondere servizi e middleware

Registrare un servizio:

```csharp id="sy20dk"
builder.Services.AddScoped<ClienteService>();
```

non significa aggiungerlo alla pipeline.

Aggiungere middleware:

```csharp id="9e234c"
app.UseHttpsRedirection();
```

significa modificare il flusso delle request.

Sono due cose diverse:

```text id="9k9i9j"
Services
  oggetti disponibili tramite Dependency Injection.

Middleware
  componenti che elaborano richieste HTTP.
```

---

## Dimenticare i file statici

Se i file CSS o JS non vengono serviti, controlla:

```text id="31cq7u"
sono dentro wwwroot?
hai chiamato MapStaticAssets o UseStaticFiles?
il path richiesto è corretto?
il file viene copiato in publish?
```

---

## Mettere segreti in appsettings.json

Sbagliato:

```json id="gil0q6"
{
  "ConnectionStrings": {
    "DefaultConnection": "Server=.;User Id=sa;Password=Segreta123"
  }
}
```

se il file è versionato.

Meglio:

```text id="9xf2cu"
user secrets in sviluppo
variabili d’ambiente o secret store in produzione
```

## Usare Development in produzione

Se `ASPNETCORE_ENVIRONMENT` resta `Development` in produzione, potresti esporre dettagli di errore e comportamento diagnostico non sicuro.

Controllare sempre l’ambiente reale.

# Riepilogo

In questo capitolo abbiamo introdotto ASP.NET Core.

Abbiamo visto:

```text id="ts1gkv"
creazione progetto web
Program.cs
middleware pipeline
routing
endpoint
static files
MapStaticAssets

```

Abbiamo chiarito che ASP.NET Core elabora le richieste tramite una pipeline di middleware e le collega a endpoint tramite routing. I middleware possono decidere se passare la richiesta al componente successivo e possono eseguire codice prima e dopo il middleware successivo. ([Microsoft Learn][2])

Abbiamo visto che `MapStaticAssets` è rilevante nelle app .NET 9+ per servire asset noti con ottimizzazioni come compressione e fingerprinting, mentre `UseStaticFiles` resta importante per scenari statici classici e file serviti da posizioni non gestite da quelle convenzioni. ([Microsoft Learn][4])

La regola principale è questa:

```text id="fh2nnb"
ASP.NET Core non è solo un framework per API o pagine web.
È una piattaforma HTTP componibile basata su hosting, configurazione,
Dependency Injection, middleware, routing ed endpoint.
```

Nel prossimo capitolo possiamo costruire le prime API con Minimal API: endpoint CRUD, binding dei parametri, DTO, validazione, status code, OpenAPI e integrazione con EF Core.

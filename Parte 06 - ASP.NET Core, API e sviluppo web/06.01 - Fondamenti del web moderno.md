# 06.01 — Fondamenti del web moderno

Prima di studiare ASP.NET Core, Minimal API, MVC, Razor, Blazor o autenticazione, bisogna capire il web.

Molti sviluppatori iniziano direttamente da un framework e imparano a memoria istruzioni come:

```csharp
app.MapGet("/clienti", () => ...);
```

oppure:

```csharp
return Results.Ok(cliente);
```

ma senza avere chiaro che cosa accade davvero tra browser, server, richiesta HTTP, risposta, header, cookie, JSON e HTML.

Questo capitolo serve a costruire quella base.

Il punto centrale è questo:

```text
Il web è un dialogo tra client e server.
Il client invia una request.
Il server restituisce una response.
Tutto ASP.NET Core si costruisce sopra questo modello.
```

---

# Client e server

Nel web moderno ci sono quasi sempre due ruoli principali:

```text
Client
  Il programma che fa la richiesta.

Server
  Il programma che riceve la richiesta, la elabora e restituisce una risposta.
```

Il client può essere:

```text
browser
app mobile
app desktop
servizio backend
script
tool come Postman o curl
```

Il server può essere:

```text
app ASP.NET Core
API REST
server statico
reverse proxy
servizio cloud
microservizio
```

Esempio:

```text
Browser
  invia richiesta a https://example.com/clienti

Server
  riceve la richiesta
  legge eventuali dati
  esegue logica applicativa
  interroga il database
  restituisce HTML o JSON
```

---

# HTTP

HTTP significa:

```text
HyperText Transfer Protocol
```

È il protocollo principale usato dal web.

Quando apriamo un sito, il browser invia una richiesta HTTP.

Esempio concettuale:

```http
GET /clienti HTTP/1.1
Host: example.com
Accept: text/html
```

Il server risponde:

```http
HTTP/1.1 200 OK
Content-Type: text/html

<html>
  <body>
    <h1>Clienti</h1>
  </body>
</html>
```

Tutto il web si basa su questo scambio:

```text
request
  richiesta del client

response
  risposta del server
```

---

# Request

Una **request** HTTP è una richiesta inviata dal client al server.

Contiene:

```text
metodo HTTP
URL
header
eventuale body
cookie
query string
```

Esempio:

```http
POST /api/clienti HTTP/1.1
Host: example.com
Content-Type: application/json

{
  "nome": "Emilie",
  "email": "emilie@example.com"
}
```

Questa richiesta dice:

```text
Voglio inviare dati a /api/clienti.
I dati sono in formato JSON.
```

---

# Metodi HTTP

I metodi HTTP indicano il tipo di operazione richiesta.

I più comuni sono:

```text
GET
  leggere una risorsa.

POST
  creare una nuova risorsa o inviare dati.

PUT
  sostituire una risorsa esistente.

PATCH
  modificare parzialmente una risorsa.

DELETE
  eliminare una risorsa.
```

Esempi:

```http
GET /api/clienti
```

significa:

```text
Dammi l’elenco dei clienti.
```

```http
GET /api/clienti/10
```

significa:

```text
Dammi il cliente con ID 10.
```

```http
POST /api/clienti
```

significa:

```text
Crea un nuovo cliente.
```

```http
PUT /api/clienti/10
```

significa:

```text
Sostituisci il cliente 10 con questi nuovi dati.
```

```http
DELETE /api/clienti/10
```

significa:

```text
Elimina il cliente 10.
```

# Response

Una **response** HTTP è la risposta del server.

Contiene:

```text
status code
header
eventuale body
cookie impostati dal server
```

Esempio:

```http
HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 10,
  "nome": "Emilie",
  "email": "emilie@example.com"
}
```

Questa risposta dice:

```text
La richiesta è andata a buon fine.
Il contenuto restituito è JSON.
```

---

# Status code

Gli **status code** indicano l’esito della richiesta.

Sono numeri a tre cifre.

Le famiglie principali sono:

```text
2xx
  successo

3xx
  redirect

4xx
  errore lato client

5xx
  errore lato server
```

---

## Status code comuni

```text
200 OK
  Richiesta riuscita.

201 Created
  Risorsa creata correttamente.

204 No Content
  Richiesta riuscita, ma nessun contenuto da restituire.

301 Moved Permanently
  Risorsa spostata permanentemente.

302 Found
  Redirect temporaneo.

400 Bad Request
  Richiesta non valida.

401 Unauthorized
  Utente non autenticato.

403 Forbidden
  Utente autenticato ma non autorizzato.

404 Not Found
  Risorsa non trovata.

409 Conflict
  Conflitto, per esempio dato già esistente.

422 Unprocessable Entity
  Dati formalmente ricevuti ma semanticamente non validi.

500 Internal Server Error
  Errore interno del server.
```

---

## Esempi pratici

Richiesta:

```http
GET /api/clienti/10
```

Risposta se il cliente esiste:

```http
HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 10,
  "nome": "Emilie"
}
```

Risposta se il cliente non esiste:

```http
HTTP/1.1 404 Not Found
```

Creazione cliente:

```http
POST /api/clienti
Content-Type: application/json

{
  "nome": "Emilie"
}
```

Risposta:

```http
HTTP/1.1 201 Created
Location: /api/clienti/10
Content-Type: application/json

{
  "id": 10,
  "nome": "Emilie"
}
```

# Header

Gli **header** sono metadati della request o della response.

Esempio request:

```http
GET /api/clienti HTTP/1.1
Host: example.com
Accept: application/json
Authorization: Bearer eyJ...
```

Esempio response:

```http
HTTP/1.1 200 OK
Content-Type: application/json
Cache-Control: no-cache
```

Header comuni:

```text
Content-Type
  Indica il formato del contenuto inviato.

Accept
  Indica il formato desiderato dal client.

Authorization
  Contiene credenziali o token.

Cookie
  Invia cookie dal client al server.

Set-Cookie
  Chiede al browser di salvare un cookie.

Cache-Control
  Controlla la cache.

Location
  Indica l’URL di una risorsa creata o di un redirect.
```

# Body

Il **body** contiene i dati principali della request o della response.

Una request `GET` di solito non ha body.

Una request `POST` spesso ha body.

Esempio JSON:

```http
POST /api/clienti HTTP/1.1
Content-Type: application/json

{
  "nome": "Emilie",
  "email": "emilie@example.com"
}
```

Esempio form:

```http
POST /login HTTP/1.1
Content-Type: application/x-www-form-urlencoded

username=emilie&password=segreta
```

# Query string

La **query string** è la parte dell’URL dopo il punto interrogativo.

Esempio:

```text
/api/clienti?citta=Aosta&attivo=true
```

Qui:

```text
citta = Aosta
attivo = true
```

Una query string è utile per:

```text
filtri
ordinamenti
paginazione
ricerche
opzioni di visualizzazione
```

Esempi:

```text
/api/clienti?attivo=true
/api/clienti?citta=Aosta
/api/clienti?page=2&pageSize=50
/api/prodotti?search=monitor
```

In ASP.NET Core questi valori possono essere letti automaticamente dai parametri dell’endpoint.

Esempio concettuale:

```csharp
app.MapGet("/api/clienti", (string? citta, bool? attivo) =>
{
    // usa citta e attivo
});
```

# Form

Un **form HTML** permette all’utente di inviare dati al server.

Esempio:

```html
<form method="post" action="/clienti">
  <label>Nome</label>
  <input type="text" name="nome">

  <label>Email</label>
  <input type="email" name="email">

  <button type="submit">Salva</button>
</form>
```

Quando l’utente preme il pulsante, il browser invia una request al server.

Con:

```html
method="post"
```

la request sarà di tipo `POST`.

Con:

```html
action="/clienti"
```

i dati verranno inviati all’indirizzo `/clienti`.

---

## Form urlencoded

Formato classico:

```http
Content-Type: application/x-www-form-urlencoded

nome=Emilie&email=emilie%40example.com
```

```http
Content-Type: multipart/form-data
```

Esempio HTML:

```html
<form method="post" action="/upload" enctype="multipart/form-data">
```

# Cookie

Un **cookie** è un piccolo dato che il server può chiedere al browser di salvare.

Il server lo imposta con l’header:

```http
Set-Cookie: lingua=it; Path=/; HttpOnly
```

Il browser lo rimanda nelle richieste successive:

```http
Cookie: lingua=it
```

I cookie sono usati per:

```text
sessioni
autenticazione
preferenze utente
tracking tecnico
consensi
lingua
```

---

## Cookie e sicurezza

Attributi importanti:

```text
HttpOnly
  Il cookie non è accessibile da JavaScript.

Secure
  Il cookie viene inviato solo su HTTPS.

SameSite
  Limita l’invio del cookie in richieste cross-site.

Expires / Max-Age
  Definisce la durata del cookie.
```

Esempio:

```http
Set-Cookie: sessionId=abc123; Path=/; HttpOnly; Secure; SameSite=Lax
```

# Sessione

HTTP è stateless.

Significa che, di base, ogni request è indipendente.

Esempio:

```text
Request 1
  GET /prodotti

Request 2
  GET /carrello

Il protocollo HTTP, da solo, non ricorda chi sei.
```

La **sessione** serve a mantenere uno stato tra richieste diverse.

Di solito funziona così:

```text
1. Il server crea un identificativo di sessione.
2. Lo manda al browser in un cookie.
3. Il browser rimanda quel cookie a ogni request.
4. Il server usa l’identificativo per recuperare lo stato.
```

Esempi di dati di sessione:

```text
utente loggato
carrello
preferenze temporanee
stato di una procedura guidata
```

---

## Sessione e scalabilità

La sessione è comoda, ma va usata con attenzione.

Se lo stato resta in memoria su un singolo server, diventa più difficile scalare su più server.

In applicazioni moderne spesso si preferisce:

```text
token
database
cache distribuita
storage lato client controllato
API stateless dove possibile
```

Regola pratica:

```text
Usa la sessione per stato temporaneo e limitato.
Non usarla come database nascosto.
```

# Server-side rendering

Il **server-side rendering**, spesso abbreviato in SSR, significa che il server genera HTML completo.

Flusso:

```text
1. Browser chiede /clienti.
2. Server legge dati dal database.
3. Server genera HTML.
4. Browser riceve HTML già pronto.
5. Browser lo visualizza.
```

Esempio response:

```html
<h1>Clienti</h1>

<ul>
  <li>Emilie</li>
  <li>Mario</li>
</ul>
```

Tecnologie .NET collegate:

```text
Razor Pages
MVC Views
Blazor Static SSR
```

---

## Vantaggi SSR

```text
prima visualizzazione spesso rapida
buona compatibilità SEO
meno JavaScript necessario
modello semplice per molti gestionali
logica centrale sul server
```

---

## Svantaggi SSR

```text
ogni navigazione richiede una request al server
interattività avanzata più laboriosa
esperienza meno fluida rispetto a SPA molto dinamiche
```

Per molte applicazioni aziendali, SSR resta una scelta solida.

---

# Single-page application

Una **single-page application**, o SPA, è un’applicazione in cui il browser carica una pagina iniziale e poi aggiorna dinamicamente il contenuto con JavaScript.

Flusso:

```text
1. Browser carica HTML, CSS e JavaScript.
2. JavaScript avvia l’app nel browser.
3. L’app chiama API JSON.
4. Il contenuto cambia senza ricaricare tutta la pagina.
```

Tecnologie comuni:

```text
React
Angular
Vue
Svelte
Blazor WebAssembly
```

Esempio:

```text
GET /app
  restituisce HTML e JavaScript

GET /api/clienti
  restituisce JSON

POST /api/clienti
  crea cliente
```

---

## Vantaggi SPA

```text
interattività elevata
esperienza fluida
meno ricaricamenti pagina
buona separazione tra frontend e API
adatta ad applicazioni molto dinamiche
```

---

## Svantaggi SPA

```text
più complessità frontend
più JavaScript
SEO più delicata se non gestita
gestione stato lato client più complessa
autenticazione e sicurezza da progettare bene
```

---

# SSR e SPA a confronto

```text
Server-side rendering
  Il server produce HTML.

Single-page application
  Il client produce o aggiorna l’interfaccia usando JavaScript e API.
```

Esempio SSR:

```text
Browser → GET /clienti
Server → HTML con elenco clienti
```

Esempio SPA:

```text
Browser → GET /app
Server → HTML minimo + JavaScript

Browser → GET /api/clienti
Server → JSON

JavaScript → costruisce la UI
```

# HTTPS

HTTPS è HTTP su una connessione cifrata tramite TLS.

È essenziale perché protegge:

```text
credenziali
cookie
token
dati personali
contenuto delle richieste
contenuto delle risposte
```

Oggi un’applicazione web reale dovrebbe usare HTTPS.

Esempio:

```text
http://example.com
  non cifrato

https://example.com
  cifrato
```

In sviluppo ASP.NET Core usa spesso certificati locali per HTTPS.

# Cache

La cache permette di evitare di scaricare o ricalcolare sempre gli stessi dati.

Header comune:

```http
Cache-Control: no-cache
```

oppure:

```http
Cache-Control: public, max-age=3600
```

La cache può esistere:

```text
nel browser
in un proxy
in una CDN
nel server
in memoria applicativa
in cache distribuita
```

La cache migliora le performance, ma può creare problemi se dati sensibili o aggiornati vengono serviti nel modo sbagliato.

# Errori comuni

## Confondere URL e route applicativa

URL completo:

```text
https://example.com/api/clienti?attivo=true
```

Componenti:

```text
scheme
  https

host
  example.com

path
  /api/clienti

query string
  attivo=true
```

In ASP.NET Core spesso lavoriamo sul path e sulla query string.

---

## Usare GET per modificare dati

Sbagliato:

```http
GET /api/clienti/delete/10
```

Meglio:

```http
DELETE /api/clienti/10
```

Oppure, in form HTML tradizionale, una POST verso un’azione di cancellazione.

Regola pratica:

```text
GET deve essere usato per leggere, non per modificare stato.
```

---

## Restituire sempre 200 OK

Sbagliato:

```http
HTTP/1.1 200 OK

{
  "success": false,
  "error": "Cliente non trovato"
}
```

Meglio:

```http
HTTP/1.1 404 Not Found
```

Gli status code sono parte del contratto HTTP.

---

## Mettere dati sensibili nella query string

Sconsigliato:

```text
/login?username=emilie&password=segreta
```

La query string può finire in:

```text
log
cronologia browser
proxy
strumenti di analytics
screenshot
referrer
```

Meglio inviare dati sensibili nel body di una request sicura via HTTPS, con meccanismi di autenticazione corretti.

---

## Confondere cookie e sessione

Il cookie vive nel browser.

La sessione, normalmente, è lo stato associato a un identificativo salvato nel cookie.

```text
Cookie
  contiene un identificativo o un piccolo valore.

Sessione
  contiene stato lato server associato a quell’identificativo.
```

---

## Usare sessione per tutto

Sconsigliato:

```text
mettere nella sessione grandi quantità di dati
usare la sessione come database
dipendere dalla sessione per logiche critiche senza controllo
```

Meglio usare database, cache distribuita o token secondo il caso.

---

## Confondere HTML e JSON

HTML è per la visualizzazione.

JSON è per lo scambio dati tra programmi.

Una pagina web e una API possono rappresentare gli stessi dati, ma hanno scopi diversi.

---

# Linee guida pratiche

```text
Pensa sempre in termini di request e response.

Usa GET
  per leggere dati.

Usa POST
  per creare o inviare dati.

Usa PUT/PATCH
  per aggiornare.

Usa DELETE
  per cancellare.

Usa status code corretti
  invece di mettere tutto in 200 OK.

Usa JSON
  per API.

Usa HTML
  per pagine renderizzate.

Usa cookie con attenzione
  soprattutto per autenticazione e sessione.

Non mettere dati sensibili nella query string.

Preferisci HTTPS
  per qualunque applicazione reale.

Scegli SSR o SPA
  in base al tipo di applicazione, non per moda.
```

# Riepilogo

In questo capitolo abbiamo costruito le basi del web moderno.

Abbiamo visto:

```text
HTTP
request
response
header
status code
query string
form
JSON
cookie
sessione
HTML
CSS
JavaScript
client/server
server-side rendering
single-page application
```

Abbiamo chiarito che il web è fondato su uno scambio semplice:

```text
Client invia una request.
Server restituisce una response.
```

Abbiamo distinto:

```text
Pagina HTML
  pensata per essere visualizzata da un browser.

API JSON
  pensata per essere consumata da codice.
```

Abbiamo confrontato:

```text
Server-side rendering
  il server genera HTML.

Single-page application
  il client aggiorna l’interfaccia usando JavaScript e API.
```

La regola principale è questa:

```text
Prima di imparare ASP.NET Core, bisogna capire HTTP.
ASP.NET Core non sostituisce il web: lo implementa.
```

Nel prossimo capitolo possiamo introdurre ASP.NET Core: pipeline, middleware, routing, dependency injection, configurazione, hosting e struttura di un progetto web.

# 06.05 — Razor, MVC e rendering server-side

Nei capitoli precedenti abbiamo introdotto ASP.NET Core, le Minimal API e le API robuste. Finora ci siamo concentrati soprattutto su endpoint che restituiscono JSON.

Ma ASP.NET Core non serve solo per costruire API.

Serve anche per costruire applicazioni web in cui il server genera HTML: pagine gestionali, pannelli amministrativi, form, report, dashboard, portali interni e applicazioni aziendali classiche.

Questo approccio si chiama **server-side rendering**.

Il punto centrale è questo:

```text id="r9hmxi"
Con Razor, Razor Pages e MVC, ASP.NET Core genera HTML sul server
e lo invia già pronto al browser.
```

Questo modello resta molto importante nel mondo reale, soprattutto per applicazioni gestionali, intranet, backoffice e software aziendali.

---

# Server-side rendering in ASP.NET Core

Nel rendering server-side, il flusso è questo:

```text id="fhhvya"
1. Il browser richiede una pagina.

2. ASP.NET Core riceve la request.

3. L'applicazione esegue logica, legge dati, valida permessi.

4. Il server genera HTML usando Razor.

5. Il browser riceve HTML già pronto.

6. CSS e JavaScript completano la parte visiva e interattiva.
```

Esempio:

```http id="6x6l6m"
GET /Clienti
```

Risposta:

```html id="14ix84"
<h1>Clienti</h1>

<table>
  <tr>
    <td>Emilie</td>
    <td>emilie@example.com</td>
  </tr>
</table>
```

Il browser non deve costruire tutta la pagina da JSON. Riceve direttamente HTML.

---

# Razor syntax

**Razor** è la sintassi usata da ASP.NET Core per mescolare HTML e C#.

Microsoft definisce Razor come una sintassi di markup per incorporare codice .NET dentro pagine web; i file Razor hanno normalmente estensione `.cshtml`, mentre i componenti Razor usano `.razor`. ([Microsoft Learn][1])

Esempio:

```cshtml id="7m5vqi"
<h1>Clienti</h1>

<p>Oggi è @DateTime.Today.ToString("dd/MM/yyyy")</p>
```

Il simbolo `@` introduce codice C#.

---

## Variabili Razor

```cshtml id="5khv19"
@{
    string titolo = "Elenco clienti";
    int totale = 3;
}

<h1>@titolo</h1>

<p>Totale clienti: @totale</p>
```

---

## Condizioni

```cshtml id="jh85sc"
@if (Model.Clienti.Count == 0)
{
    <p>Nessun cliente trovato.</p>
}
else
{
    <p>Clienti trovati: @Model.Clienti.Count</p>
}
```

---

## Cicli

```cshtml id="e73jwm"
<ul>
@foreach (ClienteDto cliente in Model.Clienti)
{
    <li>@cliente.Nome - @cliente.Email</li>
}
</ul>
```

---

## Razor non dovrebbe contenere troppa logica

Razor deve servire soprattutto a presentare dati.

Sconsigliato:

```cshtml id="wvhuk2"
@{
    // query database
    // calcoli complessi
    // chiamate HTTP
    // logica di autorizzazione complicata
}
```

Meglio preparare i dati nel PageModel, nel controller o in un servizio applicativo.

Regola pratica:

```text id="7gc0m6"
Razor deve decidere come mostrare i dati,
non contenere tutta la logica dell'applicazione.
```

# Razor Pages

**Razor Pages** è un modello di sviluppo page-focused.

Ogni pagina ha normalmente:

```text id="01qajm"
un file .cshtml
  markup Razor

un file .cshtml.cs
  PageModel con logica della pagina
```

Microsoft spiega che Razor Pages può rendere più semplice e produttiva la scrittura di scenari centrati sulle pagine rispetto all’uso di controller e viste. ([Microsoft Learn][2])

# Controller

Un **controller** è una classe che raggruppa action correlate.

```csharp id="v3ditq"
using Microsoft.AspNetCore.Mvc;

public class ClientiController : Controller
{
    private readonly ClienteService clienteService;

    public ClientiController(ClienteService clienteService)
    {
        this.clienteService = clienteService;
    }

    public async Task<IActionResult> Index()
    {
        List<ClienteDto> clienti =
            await clienteService.ListaAsync();

        return View(clienti);
    }
}
```

La action `Index` restituisce una view.

# Razor Pages vs MVC

Razor Pages e MVC usano entrambi Razor, model binding, validation, layout, partial e Tag Helpers.

La differenza principale è organizzativa.

```text id="1k423g"
Razor Pages
  orientato alle pagine.
  Ogni pagina ha markup e PageModel vicini.

MVC
  orientato a controller e action.
  Le view sono separate dai controller.
```

---

# Model binding

Il **model binding** prende dati dalla request HTTP e li converte in oggetti C#.

Microsoft spiega che controller e Razor Pages lavorano con dati provenienti da request HTTP; scrivere manualmente codice per recuperare e convertire valori da route, query string e form sarebbe ripetitivo ed esposto a errori, quindi il model binding automatizza questo processo. ([Microsoft Learn][3])

Sorgenti comuni:

```text id="ya3h6m"
route values
query string
form fields
body
header
uploaded files
```

---

## Binding in Razor Pages

```csharp id="hgrwtj"
public class CreateModel : PageModel
{
    private readonly ClienteService clienteService;

    public CreateModel(ClienteService clienteService)
    {
        this.clienteService = clienteService;
    }

    [BindProperty]
    public CreaClienteInput Input { get; set; } = new();

    public void OnGet()
    {
    }

    public async Task<IActionResult> OnPostAsync()
    {
        if (!ModelState.IsValid)
        {
            return Page();
        }

        await clienteService.CreaAsync(Input);

        return RedirectToPage("./Index");
    }
}
```

`[BindProperty]` dice ad ASP.NET Core di associare i campi del form alla proprietà `Input`.

---

## Binding in MVC

```csharp id="u304ww"
[HttpPost]
public async Task<IActionResult> Create(CreaClienteInput input)
{
    if (!ModelState.IsValid)
    {
        return View(input);
    }

    await clienteService.CreaAsync(input);

    return RedirectToAction(nameof(Index));
}
```

I campi del form vengono associati al parametro `input`.

# Validation

La validazione controlla che i dati ricevuti siano validi.

Microsoft spiega che in MVC e Razor Pages il model state contiene errori provenienti sia dal model binding sia dalla validazione; per le app web è responsabilità dell’app controllare `ModelState.IsValid` e rispondere in modo appropriato, spesso mostrando di nuovo la pagina con gli errori. ([Microsoft Learn][4])

# Upload file

Per caricare file si usa un form `multipart/form-data`.

Razor:

```cshtml id="2hhzhj"
<form method="post" enctype="multipart/form-data">
    <div>
        <label asp-for="File"></label>
        <input asp-for="File" type="file">
        <span asp-validation-for="File"></span>
    </div>

    <button type="submit">Carica</button>
</form>
```

PageModel:

```csharp id="efgx5t"
public class UploadModel : PageModel
{
    private readonly IWebHostEnvironment environment;

    public UploadModel(IWebHostEnvironment environment)
    {
        this.environment = environment;
    }

    [BindProperty]
    public IFormFile? File { get; set; }

    public void OnGet()
    {
    }

    public async Task<IActionResult> OnPostAsync()
    {
        if (File is null || File.Length == 0)
        {
            ModelState.AddModelError(nameof(File), "Selezionare un file.");
            return Page();
        }

        string uploadsPath = Path.Combine(
            environment.WebRootPath,
            "uploads");

        Directory.CreateDirectory(uploadsPath);

        string safeFileName = Path.GetFileName(File.FileName);

        string filePath = Path.Combine(
            uploadsPath,
            safeFileName);

        await using FileStream stream =
            System.IO.File.Create(filePath);

        await File.CopyToAsync(stream);

        return RedirectToPage("./UploadSuccess");
    }
}
```

---

## Sicurezza upload

L’upload file è una superficie di attacco.

Attenzione a:

```text id="1kf5lf"
dimensione massima
estensioni consentite
MIME type
nome file non affidabile
path traversal
virus/malware
esecuzione di file caricati
salvataggio fuori da wwwroot se privato
autorizzazione
```

Regola pratica:

```text id="hkbw4d"
Non fidarti mai del nome file ricevuto dal browser.
Valida estensione, contenuto, dimensione e destinazione.
```

Per file privati, meglio salvarli fuori da `wwwroot` e servirli tramite endpoint autorizzato.

# Razor/MVC e antiforgery

I form server-side devono proteggersi da CSRF.

In MVC e Razor Pages, i form generati con Tag Helpers integrano normalmente token antiforgery.

Nel controller MVC è comune usare:

```csharp id="py83t0"
[ValidateAntiForgeryToken]
```

su azioni POST.

Esempio:

```csharp id="dggnv5"
[HttpPost]
[ValidateAntiForgeryToken]
public async Task<IActionResult> Create(CreaClienteInput input)
{
    // ...
}
```

Regola pratica:

```text id="4xccjl"
Form HTML autenticati
  devono usare protezione antiforgery.

API JSON con Bearer token
  hanno considerazioni diverse, spesso non basate su cookie.
```

```text id="1zze2u"
vuoi componenti interattivi ricchi
vuoi scrivere molta UI in C#
hai stato client/server più articolato
vuoi riutilizzare componenti
vuoi esperienza più app-like
hai schermate dinamiche
vuoi usare Blazor Server, WebAssembly o Web App
```

Esempi:

```text id="hrmu6s"
dashboard interattiva
configuratore
app gestionale ricca
UI con componenti riutilizzabili
form dinamici
esperienza simile a SPA
```

---

## Scelta pratica

```text id="p3ar4n"
Razor Pages
  ottimo per pagine e form semplici.

MVC
  ottimo per struttura controller/action tradizionale.

Blazor
  ottimo per componenti e interattività ricca.

Minimal API
  ottimo per backend JSON e servizi HTTP.
```

Non esiste una scelta sempre migliore.
Dipende dal tipo di applicazione.

# Errori comuni

## Mettere query database direttamente nella view

Sbagliato:

```cshtml id="15uj5d"
@{
    var clienti = dbContext.Clienti.ToList();
}
```

Meglio preparare i dati nel PageModel, controller o servizio.

---

## ViewModel troppo grande

Sospetto:

```text id="2s5af4"
una view riceve direttamente un'entità EF con decine di proprietà e relazioni
```

Meglio un ViewModel mirato.

```csharp id="54nprx"
public record ClienteListItemViewModel(
    int Id,
    string Nome,
    string? Email);
```

---

## Non controllare ModelState

Sbagliato:

```csharp id="v24fd5"
[HttpPost]
public async Task<IActionResult> Create(CreaClienteInput input)
{
    await clienteService.CreaAsync(input);
    return RedirectToAction(nameof(Index));
}
```

Meglio:

```csharp id="73wpkh"
if (!ModelState.IsValid)
{
    return View(input);
}
```

---

## Dimenticare Post/Redirect/Get

Se dopo una POST restituisci direttamente la pagina di elenco senza redirect, l’utente può reinviare il form con refresh.

Meglio:

```csharp id="j0wzx4"
return RedirectToAction(nameof(Index));
```

o:

```csharp id="2jfuqq"
return RedirectToPage("./Index");
```

---

## Disabilitare antiforgery senza motivo

I form server-side autenticati devono essere protetti da CSRF.

Non rimuovere antiforgery solo perché “dà fastidio”.

---

## Usare MVC quando Razor Pages sarebbe più semplice

Per un CRUD page-focused semplice, Razor Pages spesso è più lineare.

MVC è ottimo, ma non sempre necessario.

---

## Usare Blazor per qualunque pagina

Se una pagina è quasi statica o ha un semplice form, Razor Pages può essere più semplice da mantenere.

# Riepilogo

In questo capitolo abbiamo introdotto Razor, MVC e il rendering server-side in ASP.NET Core.

Abbiamo visto:

```text id="jbyk8e"
Razor syntax
Razor Pages
MVC
Controller
Action
View
Model binding
Validation
Tag Helpers
Layout
Partial view
Form
Upload file
Quando usare Razor/MVC rispetto a Blazor
```

Abbiamo chiarito che Razor è una sintassi per incorporare codice .NET dentro pagine web e che viene usata sia nei file `.cshtml` sia, con un modello diverso, nei componenti `.razor`. ([Microsoft Learn][1])

Abbiamo visto che Razor Pages è particolarmente adatto agli scenari centrati sulle pagine e può essere più semplice e produttivo rispetto a controller e view quando l’applicazione è composta soprattutto da form e pagine gestionali. ([Microsoft Learn][2])

Abbiamo anche visto che model binding e validation sono fondamentali: il model binding converte i dati della request in oggetti C#, mentre la validazione popola `ModelState` con eventuali errori che l’app deve controllare prima di salvare. ([Microsoft Learn][3])

La regola principale è questa:

```text id="36j92x"
Razor Pages e MVC restano scelte solide quando vuoi applicazioni web
server-rendered, semplici da distribuire, facili da indicizzare
e adatte a form, CRUD, gestionali e backoffice.
```

Blazor aggiunge un modello più ricco e componentizzato, ma non cancella il valore del rendering server-side classico.

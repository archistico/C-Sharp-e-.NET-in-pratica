# 06.08 — SignalR e comunicazione real-time

Nei capitoli precedenti abbiamo studiato ASP.NET Core, Minimal API, API robuste, Razor/MVC e Blazor. Ora affrontiamo un tema che collega molte parti già viste: **comunicazione real-time**.

Una normale API HTTP funziona così:

```text id="4li64v"
Client
  invia una richiesta

Server
  risponde

Fine della comunicazione
```

Nel real-time, invece, il server può inviare aggiornamenti ai client **senza aspettare che il client faccia continuamente polling**.

Esempi:

```text id="9m768u"
chat
notifiche live
dashboard operative
avanzamento di elaborazioni lunghe
aggiornamento stato ordini
monitoraggio sale o dispositivi
sistemi collaborativi
aggiornamenti di prezzo
messaggi amministrativi agli utenti connessi
```

Il punto centrale è questo:

```text id="d147zv"
SignalR permette al server ASP.NET Core di inviare messaggi ai client connessi,
a tutti, a un singolo client, a un utente o a un gruppo.
```

Microsoft descrive SignalR come una libreria ASP.NET Core per aggiungere funzionalità real-time alle app: gestisce automaticamente le connessioni, può inviare messaggi a tutti i client, a client specifici o a gruppi, e può scalare usando Azure SignalR Service o Redis backplane. ([Microsoft Learn][1])

---

# Concetto di real-time

Una comunicazione real-time è utile quando il client deve ricevere aggiornamenti appena accadono.

Senza SignalR, una soluzione semplice sarebbe il polling:

```text id="q5296u"
Ogni 5 secondi:
  il browser chiama GET /api/notifiche
  il server risponde
  il browser aspetta
  poi richiama
```

Funziona, ma ha problemi:

```text id="5vn9wm"
molte richieste inutili
ritardo tra evento e visualizzazione
carico maggiore sul server
codice client più rumoroso
esperienza meno fluida
```

Con SignalR:

```text id="08pu94"
1. Il client apre una connessione.
2. Il server mantiene la connessione.
3. Quando succede qualcosa, il server invia il messaggio.
4. Il client aggiorna subito la UI.
```

---

# Dove si colloca SignalR

SignalR non sostituisce tutte le API.

Di solito convivono:

```text id="84xgt6"
API HTTP
  per CRUD, query, salvataggi, autenticazione, dati iniziali.

SignalR
  per notifiche, eventi, aggiornamenti live e messaggi push.
```

Esempio:

```text id="2x3ihy"
GET /api/clienti
  carica elenco iniziale

SignalR
  notifica che un cliente è stato modificato

GET /api/clienti/10
  ricarica dettaglio aggiornato, se serve
```

Regola pratica:

```text id="k4cn3c"
Usa HTTP per operazioni richiesta/risposta.
Usa SignalR per aggiornamenti live e comunicazione event-driven.
```

# Hub

Il cuore di SignalR è l’**Hub**.

Un hub è una classe server-side che espone metodi richiamabili dai client e permette al server di inviare messaggi ai client.

Esempio:

```csharp id="ydf1pc"
using Microsoft.AspNetCore.SignalR;

public class ChatHub : Hub
{
    public async Task InviaMessaggio(string utente, string messaggio)
    {
        await Clients.All.SendAsync(
            "RiceviMessaggio",
            utente,
            messaggio);
    }
}
```

Qui:

```text id="h2l91l"
ChatHub
  è l'hub SignalR.

InviaMessaggio
  è un metodo chiamabile dal client.

Clients.All.SendAsync
  invia un messaggio a tutti i client connessi.

RiceviMessaggio
  è il nome del metodo/evento che il client deve ascoltare.
```

```javascript id="0dm5v8"
const connection = new signalR.HubConnectionBuilder()
    .withUrl("/hubs/chat")
    .withAutomaticReconnect()
```

# Connessioni

Ogni client connesso a SignalR ha un identificativo di connessione.

Dentro un hub possiamo accedere a:

```csharp id="kbj4o5"

Context.User
Context.UserIdentifier
Context.Items
```

Esempio:

```csharp id="1fcjzz"
public override async Task OnConnectedAsync()
{
    string connectionId = Context.ConnectionId;

    await Clients.Caller.SendAsync(
        "ConnessioneStabilita",
        connectionId);

    await base.OnConnectedAsync();
}
```

Disconnessione:

```csharp id="4imw7c"
public override async Task OnDisconnectedAsync(Exception? exception)
{
    string connectionId = Context.ConnectionId;

    await base.OnDisconnectedAsync(exception);
}
```

## Attenzione alla connessione

Una connessione SignalR può cadere.

Cause possibili:

```text id="5m6hv1"
rete instabile
refresh pagina
chiusura browser
timeout
server riavviato
problema proxy
deploy applicativo
```

Non progettare logica critica assumendo che la connessione resti sempre aperta.

Regola pratica:

```text id="521koi"
La connessione SignalR è un canale di comunicazione,
non un database e non una sessione affidabile permanente.
```

# `IHubContext`: inviare messaggi fuori dall’hub

Non sempre il messaggio parte da un metodo dell’hub.

Può partire da:

```text id="ydleiw"
servizio applicativo
background service
endpoint API
handler di evento
consumer RabbitMQ
job schedulato
import dati
```

Per inviare messaggi da fuori l’hub si usa `IHubContext<T>`.

```csharp id="5ic7de"
public class NotificheService
{
    private readonly IHubContext<NotificheHub> hubContext;

    public NotificheService(IHubContext<NotificheHub> hubContext)
    {
        this.hubContext = hubContext;
    }

    public async Task NotificaClienteCreatoAsync(ClienteDto cliente)
    {
        await hubContext.Clients.All.SendAsync(
            "ClienteCreato",
            cliente);
    }
}
```

Registrazione:

```csharp id="j35uic"

```

Uso da endpoint:

```csharp id="2ept4f"

    CreaClienteRequest request,
    ClienteService clienteService,
    NotificheService notificheService) =>
{
    ClienteDto cliente = await clienteService.CreaAsync(request);
```

# Dashboard live

# SignalR in Blazor

Blazor può usare SignalR in due modi diversi.

## Blazor Server / Interactive Server

Blazor Server usa già SignalR internamente per la comunicazione tra browser e server. Le app server-side usano ASP.NET Core SignalR per comunicare con il browser, quindi le considerazioni di hosting e scalabilità di SignalR si applicano anche alle app Blazor Server/Interactive Server. ([Microsoft Learn][2])

Questo non significa che non puoi usare un tuo hub SignalR, ma devi ricordare che:

```text id="pb320w"
Blazor Server ha già una connessione SignalR per la UI.
Un hub aggiuntivo è un'altra comunicazione real-time applicativa.
```

---

## Blazor WebAssembly

Blazor WebAssembly gira nel browser e può usare il client SignalR .NET.

Pacchetto:

```bash id="1z6lmz"
dotnet add package Microsoft.AspNetCore.SignalR.Client
```

Componente:

```razor id="2prjdv"
@page "/notifiche"
@using Microsoft.AspNetCore.SignalR.Client
@implements IAsyncDisposable
@inject NavigationManager Navigation

<h1>Notifiche</h1>

<ul>
@foreach (string notifica in notifiche)
{
    <li>@notifica</li>
}
</ul>

@code {
    private HubConnection? connection;
    private List<string> notifiche = [];

    protected override async Task OnInitializedAsync()
    {
        connection = new HubConnectionBuilder()
            .WithUrl(Navigation.ToAbsoluteUri("/hubs/notifiche"))
            .WithAutomaticReconnect()
            .Build();

        connection.On<NotificaDto>("NotificaRicevuta", notifica =>
        {
            notifiche.Add($"{notifica.Titolo}: {notifica.Messaggio}");

            InvokeAsync(StateHasChanged);
        });

        await connection.StartAsync();
    }

    public async ValueTask DisposeAsync()
    {
        if (connection is not null)
        {
            await connection.DisposeAsync();
        }
    }
}
```

DTO:

```csharp id="ps4nro"
public record NotificaDto(
    string Titolo,
    string Messaggio,
    DateTimeOffset Data);
```

Nota importante:

```text id="nufkl9"
Nel callback SignalR, usa InvokeAsync(StateHasChanged)
per rientrare correttamente nel contesto di rendering Blazor.
```

# Scalabilità

SignalR introduce sfide specifiche di scalabilità.

Una API HTTP classica può ricevere request indipendenti. SignalR mantiene connessioni persistenti.

Questo implica:

```text id="kmnvh6"
più memoria per connessioni
più socket aperti
gestione riconnessioni
load balancing più delicato
necessità di sticky sessions in alcuni scenari
broadcast tra server
```

La documentazione Microsoft su hosting e scaling spiega che SignalR richiede considerazioni specifiche per app ad alto traffico; per scalare, si possono usare Azure SignalR Service o Redis backplane. ([Microsoft Learn][2])

---

## Problema con più server

Scenario:

```text id="3pt179"
Server A
  ha client 1 e client 2 connessi

Server B
  ha client 3 e client 4 connessi
```

Se un evento arriva su Server A e il codice fa:

```csharp id="322lx4"
Clients.All.SendAsync(...)
```

senza meccanismo di scale-out, il messaggio raggiunge solo i client connessi a Server A.

Serve un sistema che distribuisca il messaggio agli altri server.

---

# Redis backplane

Un **Redis backplane** permette a più istanze dell’app SignalR di coordinare i messaggi.

Schema:

```text id="6013yz"
Server A
  pubblica messaggio su Redis

Redis
  distribuisce messaggio

Server B / C / D
  ricevono e inviano ai propri client connessi
```

Configurazione:

```bash id="0l51ya"
dotnet add package Microsoft.AspNetCore.SignalR.StackExchangeRedis
```

`Program.cs`:

```csharp id="5n2hld"
builder.Services
    .AddSignalR()
    .AddStackExchangeRedis(
        builder.Configuration.GetConnectionString("Redis"));
```

La documentazione Microsoft indica che per configurare Redis backplane bisogna distribuire un server Redis, installare `Microsoft.AspNetCore.SignalR.StackExchangeRedis` e chiamare `AddStackExchangeRedis` dopo `AddSignalR`. Per produzione, Microsoft raccomanda Redis backplane solo se Redis è nello stesso data center dell’app SignalR, perché la latenza di rete degrada le performance; in Azure consiglia Azure SignalR Service invece del backplane Redis. ([Microsoft Learn][3])

---

## Sticky sessions

Con ASP.NET Core SignalR e Redis scale-out, il client deve interagire con lo stesso server per la durata della connessione; quindi le sticky sessions sono richieste. Con Azure SignalR Service, invece, sticky sessions non sono richieste perché il servizio gestisce le connessioni ai client. ([Microsoft Learn][4])

Questo punto è molto importante quando l’app è dietro:

```text id="ydm1rz"
load balancer
reverse proxy
Kubernetes ingress
IIS ARR
NGINX
cloud gateway
```

---

# Azure SignalR Service

Se l’app gira in Azure o deve scalare molto, Azure SignalR Service può spostare la gestione delle connessioni fuori dall’applicazione.

Vantaggi:

```text id="3ug1vq"
meno carico sulle istanze applicative
gestione connessioni delegata
scalabilità più semplice
niente sticky sessions applicative nello stesso modo del Redis backplane
integrazione cloud
```

Per un libro generale su ASP.NET Core è sufficiente introdurlo come opzione di produzione, senza approfondire subito tutta la configurazione Azure.

## Autorizzazione nei metodi hub

Non basta proteggere l’hub intero. A volte bisogna controllare anche il gruppo o la risorsa.

Esempio:

```csharp id="2l1ic1"
public class DashboardHub : Hub
{
    private readonly DashboardAuthorizationService authorizationService;

    public DashboardHub(
        DashboardAuthorizationService authorizationService)
    {
        this.authorizationService = authorizationService;
    }

    public async Task SeguiDashboard(string dashboardId)
    {
        bool autorizzato =
            await authorizationService.PuoVedereDashboardAsync(
                Context.User,
                dashboardId);

        if (!autorizzato)
        {
            throw new HubException("Non autorizzato.");
        }

        await Groups.AddToGroupAsync(
            Context.ConnectionId,
            $"dashboard:{dashboardId}");
    }
}
```

Regola pratica:

```text id="gy5tgz"
Non fidarti del nome gruppo inviato dal client.
Verifica sempre che l’utente possa accedere a quella risorsa.
```

## Validazione input

Un metodo hub riceve input da client remoti.

Quindi va validato.

Sbagliato:

```csharp id="m3w4ft"
public async Task InviaMessaggio(string gruppo, string messaggio)
{
    await Clients.Group(gruppo).SendAsync("Messaggio", messaggio);
}
```

Meglio:

```csharp id="t1qxth"
public async Task InviaMessaggio(string gruppo, string messaggio)
{
    if (string.IsNullOrWhiteSpace(gruppo))
    {
        throw new HubException("Gruppo non valido.");
    }

    if (string.IsNullOrWhiteSpace(messaggio) ||
        messaggio.Length > 1000)
    {
        throw new HubException("Messaggio non valido.");
    }

    bool autorizzato = await UtentePuoScrivereNelGruppoAsync(gruppo);

    if (!autorizzato)
    {
        throw new HubException("Non autorizzato.");
    }

    await Clients.Group(gruppo)
        .SendAsync("Messaggio", messaggio);
}
```

## Non inviare dati sensibili a `Clients.All`

Sospetto:

```csharp id="zm0rj6"
await Clients.All.SendAsync("DatiCliente", datiCliente);
```

Se i dati sono riservati, bisogna inviarli solo a:

```text id="usitq3"
utente specifico
gruppo autorizzato
connessione specifica
```

# Pattern pratico: evento applicativo + notifica SignalR

Un buon design evita di mischiare troppo logica applicativa e SignalR.

Esempio:

```text id="e2fc2n"
ClienteService
  crea cliente

Evento applicativo
  ClienteCreato

NotificheService
  riceve evento e invia SignalR

Client Blazor/JS
  riceve notifica e aggiorna UI
```

Questo riduce accoppiamento.

---

## Esempio semplice

```csharp id="wpwogt"
public record ClienteCreatoEvent(
    int ClienteId,
    string Nome);
```

```csharp id="m95zo8"
public class ClienteService
{
    private readonly AppDbContext dbContext;
    private readonly NotificheService notificheService;

    public ClienteService(
        AppDbContext dbContext,
        NotificheService notificheService)
    {
        this.dbContext = dbContext;
        this.notificheService = notificheService;
    }

    public async Task<ClienteDto> CreaAsync(CreaClienteRequest request)
    {
        Cliente cliente = new Cliente
        {
            Nome = request.Nome,
            Email = request.Email
        };

        dbContext.Clienti.Add(cliente);

        await dbContext.SaveChangesAsync();

        await notificheService.NotificaClienteCreatoAsync(
            new ClienteCreatoEvent(cliente.Id, cliente.Nome));

        return new ClienteDto(cliente.Id, cliente.Nome, cliente.Email);
    }
}
```

# Errori comuni

## Usare SignalR al posto di API normali

Sbagliato:

```text id="g7g3od"
tutte le letture, salvataggi e query passano da SignalR
```

Meglio:

```text id="519m4h"
API HTTP per CRUD e query
SignalR per notifiche e aggiornamenti live
```

---

## Mettere logica di business negli hub

Sospetto:

```text id="8j2wmu"
Hub con query database, validazione complessa, calcoli, invio email, update stato
```

Meglio:

```text id="kq79m4"
Hub
  riceve comando/evento e delega

Service
  contiene logica applicativa
```

---

## Dimenticare autorizzazione sui gruppi

Il client non deve poter scegliere liberamente gruppi sensibili.

Sbagliato:

```csharp id="4vv88z"
await Groups.AddToGroupAsync(Context.ConnectionId, gruppoRichiestoDalClient);
```

senza verifiche.

---

## Assumere connessione sempre attiva

SignalR non garantisce che il client sia sempre connesso.

Per dati importanti:

```text id="pdxo92"
salva su database
invia notifica live
al reconnect, ricarica stato da API
```

---

## Broadcast eccessivo

Sbagliato:

```csharp id="nfs91o"
await Clients.All.SendAsync("Aggiornamento", dati);
```

per ogni piccolo evento, anche se interessa solo pochi utenti.

Meglio usare:

```text id="icz781"
Groups
User
Client
filtri applicativi
```

---

## Ignorare scalabilità

Funziona in sviluppo con un server, ma fallisce in produzione con più istanze.

Valuta:

```text id="bqzzdm"
sticky sessions
Redis backplane
Azure SignalR Service
limiti reverse proxy
timeout
numero connessioni
```

---

## Non gestire riconnessione client

Il client deve gestire:

```text id="zhgj0o"
connessione iniziale fallita
riconnessione
stato temporaneamente non aggiornato
ricaricamento dati dopo reconnect
messaggi persi
```

---

# Linee guida pratiche

```text id="3k1ykd"
Usa SignalR
  per notifiche, chat, dashboard live e aggiornamenti push.

Usa API HTTP
  per lettura iniziale, CRUD e persistenza.

Mantieni gli hub sottili
  e delega ai servizi applicativi.

Usa gruppi
  per limitare il broadcast ai client interessati.

Autorizza accesso a hub e gruppi
  non fidarti di parametri inviati dal client.

Valida input
  anche nei metodi hub.

Usa IHubContext
  per inviare messaggi da servizi, endpoint o background job.

Progetta riconnessione
  e recupero stato via API.

Scala consapevolmente
  con Redis backplane o Azure SignalR Service.

Non inviare dati sensibili a Clients.All
  scegli User, Client o Group.

Logga eventi importanti
  connessione, disconnessione, errori, invii critici.
```

# Riepilogo

In questo capitolo abbiamo introdotto SignalR e la comunicazione real-time.

Abbiamo visto:

```text id="k42uwm"
concetto di real-time
hub
client
gruppi
connessioni
notifiche
chat
dashboard live
SignalR in Blazor
SignalR con JavaScript
scalabilità
Redis backplane
sicurezza
```

Abbiamo chiarito che SignalR gestisce automaticamente le connessioni e permette di inviare messaggi a tutti i client, a client specifici o a gruppi. Può scalare con Azure SignalR Service o Redis backplane, ma la scalabilità richiede attenzione a connessioni persistenti, load balancing e sticky sessions. ([Microsoft Learn][1])

Abbiamo visto che Redis backplane consente a più istanze SignalR di coordinare messaggi, ma Microsoft raccomanda di usarlo in produzione solo quando Redis è nello stesso data center dell’app; in Azure, la raccomandazione è valutare Azure SignalR Service. ([Microsoft Learn][3])

La regola principale è questa:

```text id="t60iyr"
SignalR è ideale per eventi live,
ma non deve sostituire database, API HTTP, autorizzazione o persistenza.
```

SignalR collega web, async, eventi e architetture distribuite perché trasforma il backend da semplice risponditore HTTP a partecipante attivo della comunicazione con i client connessi.

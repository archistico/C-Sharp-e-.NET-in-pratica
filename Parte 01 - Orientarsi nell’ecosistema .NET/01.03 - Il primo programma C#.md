# 01.03 — Il primo programma C#

Nei capitoli precedenti abbiamo chiarito due aspetti fondamentali:

* che cosa sono C# e .NET;
* quali strumenti servono per sviluppare.

Ora possiamo scrivere il primo programma.

Il primo programma in un linguaggio viene spesso chiamato **Hello World**, perché normalmente stampa a video una frase semplice come:

```text id="vbbvly"
Hello, World!
```

È un esempio minimo, ma utile. Permette di verificare che:

* l’ambiente è installato correttamente;
* il progetto compila;
* il runtime esegue il programma;
* sappiamo lanciare l’applicazione;
* sappiamo modificare e rieseguire il codice.

In questo capitolo useremo una **console app**, cioè un’applicazione che legge e scrive testo nel terminale.

---

## Creare il primo progetto

Apriamo un terminale e creiamo una nuova cartella:

```bash id="83cwmy"
mkdir PrimoProgramma
cd PrimoProgramma
```

Creiamo un progetto console:

```bash id="jtvx4w"
dotnet new console
```

Il comando genera una piccola applicazione C# già pronta.

La struttura sarà simile a questa:

```text id="8h5v3q"
PrimoProgramma/
  PrimoProgramma.csproj
  Program.cs
```

Il file più importante per ora è:

```text id="yop5h1"
Program.cs
```

Apriamolo.

In un progetto console moderno potremmo trovare un codice simile:

```csharp id="f3wem8"
Console.WriteLine("Hello, World!");
```

Questo è un programma C# completo.

Può sembrare strano, perché non vediamo né una classe, né un metodo `Main`, né un namespace. Il motivo è che i progetti console moderni usano una sintassi chiamata **top-level statements**.

---

# Hello World

Il programma:

```csharp id="sga23i"
Console.WriteLine("Hello, World!");
```

fa una cosa sola: scrive una riga di testo nella console.

`Console` è una classe della libreria di base di .NET.

`WriteLine` è un metodo che scrive un testo e poi va a capo.

Possiamo modificare la frase:

```csharp id="h6s69k"
Console.WriteLine("Ciao, C#!");
```

Per eseguire il programma:

```bash id="h6tqla"
dotnet run
```

Output:

```text id="k2c12a"
Ciao, C#!
```

Questo è il nostro primo programma C#.

È semplice, ma contiene già tre elementi importanti:

```text id="p4fy4n"
Console
  classe usata per interagire con il terminale.

WriteLine
  metodo che scrive una riga di testo.

" Ciao, C#! "
  stringa, cioè testo.
```

---

# Top-level statements

Nei progetti console moderni C# permette di scrivere istruzioni direttamente nel file `Program.cs`, senza dover dichiarare esplicitamente una classe `Program` e un metodo `Main`.

Questa sintassi si chiama **top-level statements**.

Microsoft descrive i top-level statements come un modo per scrivere codice eseguibile direttamente alla radice di un file, senza la “cerimonia” di una classe `Program` e di un metodo `Main`. La documentazione consiglia di usare i top-level statements per le nuove applicazioni, soprattutto perché rendono più semplice l’avvio. ([Microsoft Learn][1])

Esempio:

```csharp id="kegg56"
Console.WriteLine("Programma avviato");

int numero = 10;
int doppio = numero * 2;

Console.WriteLine($"Il doppio di {numero} è {doppio}");
```

Questo codice è un programma completo.

---

## Che cosa genera il compilatore

Anche se noi non scriviamo esplicitamente una classe e un metodo `Main`, il compilatore li crea per noi.

In pratica, quando scriviamo:

```csharp id="jmcf05"
Console.WriteLine("Hello, World!");
```

possiamo immaginare che il compilatore lo trasformi concettualmente in qualcosa di simile:

```csharp id="5yikwi"
internal class Program
{
    private static void Main(string[] args)
    {
        Console.WriteLine("Hello, World!");
    }
}
```

Non è necessario conoscere subito tutti i dettagli, ma il concetto importante è questo:

> Il programma ha comunque un punto di ingresso. Con i top-level statements è il compilatore a generarlo per noi.

La documentazione Microsoft chiarisce infatti che, con i template console moderni, il compilatore genera gli elementi di classe e metodo necessari al programma principale. ([Microsoft Learn][2])

---

## Una sola area top-level

In un progetto può esserci **un solo file** con top-level statements.

Questo significa che non possiamo avere due file diversi che contengono istruzioni direttamente eseguibili alla radice.

Esempio non valido:

```text id="446gdx"
Program.cs
  Console.WriteLine("Programma principale");

AltroFile.cs
  Console.WriteLine("Altro codice top-level");
```

Il motivo è semplice: un’applicazione deve avere un solo punto di ingresso. La documentazione Microsoft specifica che un progetto può avere un solo file con top-level statements, pur potendo avere molti altri file contenenti classi, record, struct e altri tipi. ([Microsoft Learn][1])

Gli altri file dovranno contenere tipi normali:

```csharp id="0s6p4u"
public class Cliente
{
    public string Nome { get; set; } = "";
}
```

---

# Il metodo Main

Prima dei top-level statements, un programma console C# veniva scritto in modo più esplicito.

Esempio:

```csharp id="gss8sq"
using System;

namespace PrimoProgramma;

internal class Program
{
    static void Main(string[] args)
    {
        Console.WriteLine("Hello, World!");
    }
}
```

Qui vediamo alcuni elementi in più:

* `using System;`
* `namespace PrimoProgramma;`
* `class Program`
* `static void Main`
* `string[] args`

Il metodo `Main` è il **punto di ingresso** dell’applicazione.

Quando il programma parte, il runtime chiama `Main`.

La documentazione Microsoft lo dice in modo diretto: quando un’applicazione C# viene avviata, il runtime chiama il metodo `Main`, che rappresenta l’entry point dell’applicazione. ([Microsoft Learn][3])

---

## Forme possibili di Main

Il metodo `Main` può avere diverse firme.

Per esempio:

```csharp id="8ab1vk"
static void Main()
{
    Console.WriteLine("Ciao");
}
```

Oppure:

```csharp id="g7datb"
static void Main(string[] args)
{
    Console.WriteLine("Ciao");
}
```

Oppure può restituire un codice numerico:

```csharp id="ao9v19"
static int Main(string[] args)
{
    Console.WriteLine("Ciao");
    return 0;
}
```

Il valore restituito può essere usato dal sistema operativo o da uno script per capire se il programma è terminato correttamente.

Per convenzione:

```text id="oy756u"
0
  indica successo.

valore diverso da 0
  indica errore o terminazione anomala.
```

Da C# 7.1 è possibile avere anche un `Main` asincrono, ad esempio `async Task Main()` o `async Task<int> Main()`. Nei top-level statements, da C# 9 in poi, possiamo usare anche `await` direttamente nel file principale. ([Microsoft Learn][4])

Esempio moderno:

```csharp id="7i2zl5"
await Task.Delay(1000);
Console.WriteLine("Terminato dopo un secondo");
```

# Stile moderno e stile classico

A questo punto abbiamo due modi per scrivere lo stesso programma.

## Stile moderno

```csharp id="8js0yc"
Console.WriteLine("Hello, World!");
```

## Stile classico

```csharp id="dfrbkw"
using System;

namespace PrimoProgramma;

internal class Program
{
    static void Main(string[] args)
    {
        Console.WriteLine("Hello, World!");
    }
}
```

Entrambi sono validi.

La differenza principale è che lo stile moderno riduce il codice iniziale, mentre lo stile classico rende più esplicita la struttura.

---

## Vantaggi dello stile moderno

I top-level statements sono comodi perché:

* riducono il codice iniziale;
* rendono più semplice il primo approccio;
* sono ideali per esempi brevi;
* funzionano bene con console app e Minimal API;
* permettono di concentrarsi subito sulla logica.

Esempio:

```csharp id="9zri0k"
Console.WriteLine("Inserisci il tuo nome:");
string? nome = Console.ReadLine();

Console.WriteLine($"Ciao, {nome}!");
```

Il lettore vede subito il codice che conta.

---

## Vantaggi dello stile classico

Il metodo `Main` esplicito è utile perché:

* mostra chiaramente il punto di ingresso;
* è più familiare nei progetti legacy;
* rende evidente la classe `Program`;
* aiuta a capire codice esistente;
* può essere richiesto in alcuni contesti didattici o aziendali;
* permette di controllare esplicitamente la firma del metodo.

Esempio:

```csharp id="67hth0"
internal class Program
{
    static int Main(string[] args)
    {
        if (args.Length == 0)
        {
            Console.WriteLine("Nessun argomento specificato.");
            return 1;
        }

        Console.WriteLine($"Primo argomento: {args[0]}");
        return 0;
    }
}
```

---

## Quale stile useremo?

In questo libro useremo prevalentemente lo stile moderno quando serve mostrare esempi brevi e leggibili.

Useremo invece lo stile classico quando sarà utile rendere esplicita la struttura del programma.

Regola pratica:

```text id="uh1sst"
Esempio breve o primo progetto
  top-level statements.

Codice legacy, struttura esplicita o spiegazione dell'entry point
  metodo Main classico.
```

---

# Input e output da console

Una console app comunica con l’utente attraverso il terminale.

Per scrivere testo usiamo:

```csharp id="b6b5s4"
Console.WriteLine("Testo con ritorno a capo");
```

Oppure:

```csharp id="gmvd16"
Console.Write("Testo senza ritorno a capo");
```

La differenza è semplice:

```text id="l2o3jj"
WriteLine
  scrive e va a capo.

Write
  scrive senza andare a capo.
```

---

## Leggere testo con Console.ReadLine

Per leggere una riga inserita dall’utente usiamo:

```csharp id="l2crg4"
string? nome = Console.ReadLine();
```

Il tipo è `string?` e non semplicemente `string`.

Il punto interrogativo indica che il valore potrebbe essere `null`.

Per esempio, l’utente potrebbe chiudere l’input oppure il flusso di input potrebbe non fornire dati.

Esempio completo:

```csharp id="wmunvl"
Console.Write("Come ti chiami? ");

string? nome = Console.ReadLine();

Console.WriteLine($"Ciao, {nome}!");
```

Esecuzione:

```text id="9gjgzj"
Come ti chiami? Emilie
Ciao, Emilie!
```

---

## Gestire input vuoto

Un primo miglioramento consiste nel controllare se l’utente ha scritto qualcosa.

```csharp id="x9gb3b"
Console.Write("Come ti chiami? ");

string? nome = Console.ReadLine();

if (string.IsNullOrWhiteSpace(nome))
{
    Console.WriteLine("Non hai inserito un nome.");
}
else
{
    Console.WriteLine($"Ciao, {nome}!");
}
```

Qui usiamo:

```csharp id="xdlu7j"
string.IsNullOrWhiteSpace(nome)
```

per verificare se la stringa è:

* `null`;
* vuota;
* composta solo da spazi.

---

## Leggere numeri

`Console.ReadLine` restituisce sempre testo.

Se vogliamo leggere un numero, dobbiamo convertirlo.

Esempio ingenuo:

```csharp id="lhggds"
Console.Write("Inserisci la tua età: ");

string? testo = Console.ReadLine();
int eta = int.Parse(testo!);

Console.WriteLine($"Hai {eta} anni.");
```

Questo codice funziona se l’utente inserisce un numero valido.

Ma se l’utente scrive:

```text id="4ks9v4"
ciao
```

il programma genera un errore.

Meglio usare `TryParse`:

```csharp id="cv2r95"
Console.Write("Inserisci la tua età: ");

string? testo = Console.ReadLine();

if (int.TryParse(testo, out int eta))
{
    Console.WriteLine($"Hai {eta} anni.");
}
else
{
    Console.WriteLine("Il valore inserito non è un numero valido.");
}
```

Questo è un primo esempio di programmazione difensiva.

Non ci fidiamo ciecamente dell’input dell’utente.

---

## Un piccolo programma completo

Scriviamo ora un programma leggermente più interessante.

```csharp id="l8gq3q"
Console.WriteLine("Calcolo dell'anno di nascita");
Console.Write("Inserisci la tua età: ");

string? input = Console.ReadLine();

if (!int.TryParse(input, out int eta))
{
    Console.WriteLine("Età non valida.");
    return;
}

int annoCorrente = DateTime.Now.Year;
int annoNascitaStimato = annoCorrente - eta;

Console.WriteLine($"Se hai già compiuto gli anni, sei nato/a nel {annoNascitaStimato}.");
Console.WriteLine($"Altrimenti sei nato/a nel {annoNascitaStimato - 1}.");
```

Questo programma introduce diversi concetti:

* output;
* input;
* variabili;
* conversione;
* controllo con `if`;
* uso di `DateTime`;
* uscita anticipata con `return`.

Sono concetti che approfondiremo nei prossimi capitoli.

Per ora è sufficiente vedere che un programma è una sequenza di istruzioni che leggono dati, li elaborano e producono un risultato.

# Argomenti da riga di comando

Una console app può ricevere dati non solo tramite `Console.ReadLine`, ma anche tramite **argomenti da riga di comando**.

Quando eseguiamo un programma possiamo scrivere:

```bash id="0svafl"
dotnet run -- Emilie Aosta
```

Gli argomenti sono:

```text id="g4umrg"
Emilie
Aosta
```

Nei top-level statements possiamo accedere direttamente alla variabile speciale:

```csharp id="fqfyes"
args
```

La documentazione Microsoft specifica che, nel file che contiene top-level statements, il compilatore riconosce `args` come gli argomenti della riga di comando, di tipo `string[]`. ([Microsoft Learn][5])

Esempio:

```csharp id="cs2k0d"
Console.WriteLine($"Numero argomenti: {args.Length}");

foreach (string argomento in args)
{
    Console.WriteLine(argomento);
}
```

Esecuzione:

```bash id="bkxsvk"
dotnet run -- Emilie Aosta
```

Output:

```text id="6rzp1g"
Numero argomenti: 2
Emilie
Aosta
```

---

## Perché si usa `--`

Quando usiamo:

```bash id="t8ruhu"
dotnet run -- Emilie Aosta
```

il doppio trattino separa gli argomenti destinati al comando `dotnet run` dagli argomenti destinati alla nostra applicazione.

Senza `--`, alcuni argomenti potrebbero essere interpretati dalla CLI `dotnet` invece che dal nostro programma.

Regola pratica:

```text id="upz8xi"
dotnet run [opzioni di dotnet] -- [argomenti della tua app]
```

---

## Argomenti con Main classico

Nello stile classico gli argomenti arrivano al parametro `string[] args`:

```csharp id="7ahutt"
internal class Program
{
    static void Main(string[] args)
    {
        Console.WriteLine($"Numero argomenti: {args.Length}");

        foreach (string argomento in args)
        {
            Console.WriteLine(argomento);
        }
    }
}
```

La logica è la stessa.

La differenza è solo dove vediamo dichiarato `args`.

La documentazione Microsoft conferma che gli argomenti passati da riga di comando sono accessibili sia nei top-level statements sia tramite il parametro opzionale `Main`. ([Microsoft Learn][6])

---

## Un esempio utile: saluto da riga di comando

```csharp id="1dtw8w"
if (args.Length == 0)
{
    Console.WriteLine("Uso: dotnet run -- <nome>");
    return;
}

string nome = args[0];

Console.WriteLine($"Ciao, {nome}!");
```

Esecuzione:

```bash id="15gf19"
dotnet run -- Emilie
```

Output:

```text id="7zuakf"
Ciao, Emilie!
```

Esecuzione senza argomenti:

```bash id="x4sa61"
dotnet run
```

Output:

```text id="omvw7g"
Uso: dotnet run -- <nome>
```

# Esecuzione di un singolo file `.cs`

Tradizionalmente, per eseguire codice C# con la CLI era necessario creare un progetto con un file `.csproj`.

Con .NET 10 è stata introdotta una modalità più leggera chiamata **file-based app**, che permette di eseguire direttamente un file `.cs` usando:

```bash id="ioisaz"
dotnet run app.cs
```

Il blog ufficiale .NET ha annunciato questa funzionalità come parte di .NET 10, spiegando che permette di eseguire direttamente un file C# senza creare prima un file di progetto o una struttura completa. È pensata per apprendimento, prototipi, piccoli script e automazioni. ([Microsoft for Developers][7])

---

## Creare un file singolo

Creiamo un file chiamato:

```text id="8l3u0w"
app.cs
```

Scriviamo:

```csharp id="7rmxmt"
Console.WriteLine("Ciao da un singolo file C#!");
```

Eseguiamo:

```bash id="2kx2iu"
dotnet run app.cs
```

Output:

```text id="y6kq81"
Ciao da un singolo file C#!
```

In questo caso non abbiamo creato una solution e non abbiamo creato un `.csproj`.

Questo rende C# più immediato per esempi, esperimenti e piccoli strumenti.

---

## File singolo non significa linguaggio diverso

Una file-based app non è un “dialetto” separato di C#.

È sempre C#.

La differenza è che il progetto implicito viene gestito dagli strumenti.

Possiamo pensare a questa modalità così:

```text id="mg3c7h"
Progetto classico:
  Program.cs + .csproj

File-based app:
  app.cs
  progetto implicito generato dagli strumenti
```

Per imparare e sperimentare è molto comodo.
Per applicazioni più strutturate, il progetto `.csproj` resta la scelta migliore.

---

# Direttive file-based

Le file-based app di .NET 10 supportano alcune direttive speciali a livello di file.

Queste direttive iniziano con:

```text id="u3g9mz"
#:
```

Servono a dichiarare nel file `.cs` alcune informazioni che, in un progetto normale, sarebbero scritte nel `.csproj`.

Le principali sono:

* `#:package`
* `#:sdk`
* `#:property`

La documentazione di annuncio spiega che queste direttive permettono di esprimere nel singolo file alcune impostazioni tipiche dei progetti MSBuild, senza dover creare subito un file `.csproj`. ([Microsoft for Developers][7])

---

## Direttiva `#:package`

La direttiva `#:package` permette di aggiungere un pacchetto NuGet direttamente nel file `.cs`.

Esempio:

```csharp id="vpvkz2"
#:package Humanizer@2.14.1

using Humanizer;

var data = DateTimeOffset.Now.AddDays(-3);
var differenza = DateTimeOffset.Now - data;

Console.WriteLine(differenza.Humanize());
```

Qui stiamo usando il pacchetto NuGet `Humanizer`.

In un progetto classico avremmo aggiunto il pacchetto nel `.csproj`.

In una file-based app lo dichiariamo direttamente nel file.

Con un progetto classico:

```xml id="cwdk47"
<ItemGroup>
  <PackageReference Include="Humanizer" Version="2.14.1" />
</ItemGroup>
```

Con file-based app:

```csharp id="8q2zxi"
#:package Humanizer@2.14.1
```

Questa modalità è pratica per esempi veloci.

---

## Direttiva `#:sdk`

La direttiva `#:sdk` permette di specificare quale SDK usare.

Per impostazione predefinita, una file-based app usa:

```text id="2twfko"
Microsoft.NET.Sdk
```

Ma se vogliamo creare, per esempio, una piccola API web in un singolo file, possiamo usare:

```csharp id="z1o0fi"
#:sdk Microsoft.NET.Sdk.Web

var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

app.MapGet("/", () => "Ciao da Minimal API in un file!");

app.Run();
```

Esecuzione:

```bash id="fxr91t"
dotnet run app.cs
```

La direttiva `#:sdk Microsoft.NET.Sdk.Web` dice agli strumenti di trattare il file come se fosse parte di un progetto web. Il blog .NET usa proprio questo scenario per mostrare come abilitare funzionalità di ASP.NET Core da una file-based app. ([Microsoft for Developers][7])

---

## Direttiva `#:property`

La direttiva `#:property` permette di impostare proprietà MSBuild.

Esempio:

```csharp id="shadd6"
#:property LangVersion preview

Console.WriteLine("Uso una proprietà di build nel file.");
```

In un progetto classico scriveremmo una proprietà nel `.csproj`:

```xml id="bkv75z"
<PropertyGroup>
  <LangVersion>preview</LangVersion>
</PropertyGroup>
```

In una file-based app possiamo scriverla direttamente nel file:

```csharp id="ywd2ao"
#:property LangVersion preview
```

Questa direttiva è utile per impostazioni leggere, ma non bisogna abusarne.

Quando le proprietà diventano molte, è segno che probabilmente serve un vero progetto `.csproj`.

---

## Shebang su sistemi Unix-like

Le file-based app supportano anche lo **shebang**, cioè la riga iniziale usata nei sistemi Unix-like per rendere eseguibile uno script.

Esempio:

```csharp id="fslukz"
#!/usr/bin/dotnet run

Console.WriteLine("Ciao da uno script C# eseguibile!");
```

Su Linux o macOS possiamo rendere il file eseguibile:

```bash id="p6kutw"
chmod +x app.cs
```

Poi eseguirlo direttamente:

```bash id="8ss6yu"
./app.cs
```

Anche questa possibilità è indicata nell’annuncio ufficiale delle file-based app di .NET 10. ([Microsoft for Developers][7])

---

# Quando usare un file singolo

Una file-based app è utile quando vogliamo scrivere codice molto rapidamente.

Scenari adatti:

```text id="jmwqes"
Piccolo esperimento
Snippet didattico
Script personale
Automazione semplice
Test rapido di una libreria
Esempio da documentazione
Utility temporanea
```

Esempio:

```csharp id="ebc3v6"
Console.WriteLine(DateTime.Now);
```

Oppure:

```csharp id="lic2t5"
var file = "dati.txt";

if (File.Exists(file))
{
    Console.WriteLine(File.ReadAllText(file));
}
else
{
    Console.WriteLine("File non trovato.");
}
```

In questi casi creare una solution completa può essere eccessivo.

# Quando usare un progetto completo

Un progetto completo è preferibile quando il codice cresce.

Scenari adatti:

```text id="ly7dye"
Applicazione destinata a durare
Più file sorgente
Più classi
Test automatici
Pacchetti numerosi
Configurazione articolata
Pubblicazione
CI/CD
Team di sviluppo
Architettura a più progetti
```

Quando serve un `.csproj`, abbiamo più controllo su:

* target framework;
* pacchetti;
* proprietà di build;
* riferimenti ad altri progetti;
* configurazioni Debug/Release;
* publish;
* analizzatori;
* source generator;
* test;
* distribuzione.

---

## Regola pratica

```text id="l152a1"
File singolo
  ottimo per imparare, sperimentare e automatizzare.

Progetto completo
  migliore per applicazioni vere, manutenibili e distribuibili.
```

Oppure:

```text id="m3a3sz"
Se il codice resta in un file e fa una cosa semplice:
  file-based app.

Se il codice cresce, deve essere testato o distribuito:
  progetto .NET completo.
```

---

## Convertire una file-based app in progetto

Una caratteristica interessante delle file-based app è che possono evolvere in progetti completi.

L’annuncio ufficiale delle file-based app mostra il comando:

```bash id="lndq8l"
dotnet project convert app.cs
```

Questo comando converte il file in una struttura di progetto, creando un `.csproj`, spostando il codice in un `Program.cs` e traducendo le direttive `#:` in proprietà MSBuild o riferimenti NuGet. ([Microsoft for Developers][7])

Questo è importante perché conferma l’idea di fondo:

> una file-based app non è un vicolo cieco; può crescere e diventare un progetto normale.

# Un confronto completo

## File-based app

```text id="gycnv8"
app.cs
```

Codice:

```csharp id="sy2an8"
Console.WriteLine("Ciao");
```

Esecuzione:

```bash id="ncqaht"
dotnet run app.cs
```

Vantaggi:

* velocissima da creare;
* nessun `.csproj`;
* ottima per imparare;
* ottima per prototipi;
* ottima per script.

Svantaggi:

* meno adatta a progetti grandi;
* configurazione meno esplicita;
* meno struttura;
* può diventare disordinata se cresce troppo.

# Primo esercizio guidato

Creiamo una piccola applicazione console con progetto completo.

## 1. Creazione progetto

```bash id="gw1aus"
mkdir SalutoConsole
cd SalutoConsole
dotnet new console
```

---

## 2. Modifica di Program.cs

Sostituiamo il contenuto con:

```csharp id="kzz6cp"
Console.Write("Inserisci il tuo nome: ");

string? nome = Console.ReadLine();

if (string.IsNullOrWhiteSpace(nome))
{
    Console.WriteLine("Non hai inserito nessun nome.");
    return;
}

Console.WriteLine($"Ciao, {nome}!");
```

---

## 3. Esecuzione

```bash id="l47j34"
dotnet run
```

Output possibile:

```text id="mslfqt"
Inserisci il tuo nome: Emilie
Ciao, Emilie!
```

---

## 4. Seconda versione con argomenti

Modifichiamo il codice:

```csharp id="qwjnbx"
if (args.Length > 0)
{
    Console.WriteLine($"Ciao, {args[0]}!");
    return;
}

Console.Write("Inserisci il tuo nome: ");

string? nome = Console.ReadLine();

if (string.IsNullOrWhiteSpace(nome))
{
    Console.WriteLine("Non hai inserito nessun nome.");
    return;
}

Console.WriteLine($"Ciao, {nome}!");
```

Ora possiamo eseguire:

```bash id="eqtnnr"
dotnet run -- Emilie
```

Output:

```text id="tca1h9"
Ciao, Emilie!
```

Oppure senza argomenti:

```bash id="858neh"
dotnet run
```

In quel caso il programma chiede il nome da console.

# Cosa abbiamo imparato

In questo capitolo abbiamo scritto il primo programma C#.

Abbiamo visto che un’applicazione console può essere scritta in due stili:

```text id="pmm9bc"
Stile moderno
  top-level statements.

Stile classico
  classe Program e metodo Main.
```

Abbiamo visto come stampare testo:

```csharp id="dg02js"
Console.WriteLine("Ciao");
```

Come leggere input:

```csharp id="lnzv0s"
string? nome = Console.ReadLine();
```

Come ricevere argomenti da riga di comando:

```csharp id="9apcd8"
args
```

E come eseguire codice C# sia dentro un progetto completo sia in un singolo file `.cs`.

# Errori comuni

## Pensare che i top-level statements siano “meno veri”

I top-level statements non sono una versione semplificata o incompleta di C#.

Sono una sintassi moderna che il compilatore traduce in una struttura equivalente a un programma con `Main`.

---

## Cercare il metodo Main e non trovarlo

Nei progetti console moderni il metodo `Main` può non essere scritto esplicitamente.

Esiste comunque a livello generato dal compilatore.

---

## Usare più file con top-level statements

Un progetto può avere un solo file con top-level statements.

Gli altri file devono contenere classi, record, struct, enum, interfacce o altri tipi.

---

## Confondere input da console e argomenti

Queste due cose sono diverse.

```text id="wxjrh4"
Console.ReadLine()
  legge mentre il programma è in esecuzione.

args
  contiene valori passati quando il programma viene avviato.
```

Esempio con `ReadLine`:

```bash id="swpnbi"
dotnet run
```

Il programma chiede il nome dopo l’avvio.

Esempio con `args`:

```bash id="v1c9k3"
dotnet run -- Emilie
```

Il nome viene passato subito al programma.

---

## Usare file singoli per applicazioni troppo grandi

Le file-based app sono comode, ma non devono diventare un contenitore disordinato.

Se il codice cresce, conviene passare a un progetto completo.

---

# Conclusione

Il primo programma C# può essere scritto in una sola riga:

```csharp id="wp2kji"
Console.WriteLine("Hello, World!");
```

Dietro questa riga, però, ci sono concetti importanti:

* il programma ha un punto di ingresso;
* il runtime esegue il codice;
* la console permette input e output;
* gli argomenti da riga di comando permettono di passare dati all’avvio;
* i top-level statements semplificano la scrittura;
* il metodo `Main` resta la forma esplicita tradizionale;
* un file singolo è utile per esperimenti e script;
* un progetto completo è più adatto ad applicazioni reali.

Nei prossimi capitoli inizieremo a studiare la sintassi del linguaggio: variabili, tipi, operatori, condizioni, cicli e funzioni.

Da questo momento in poi non stiamo più solo parlando dell’ecosistema .NET: stiamo iniziando davvero a programmare in C#.

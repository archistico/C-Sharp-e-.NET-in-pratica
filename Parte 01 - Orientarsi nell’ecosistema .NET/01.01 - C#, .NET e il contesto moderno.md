# 01.01 — C#, .NET e il contesto moderno

Quando si inizia a studiare C# è molto facile confondere tra loro parole che vengono spesso usate insieme: **C#**, **.NET**, **Visual Studio**, **ASP.NET Core**, **.NET Framework**, **runtime**, **SDK**, **NuGet**, **CLR**.

Questa confusione è normale, perché nel mondo .NET il linguaggio, la piattaforma, gli strumenti e i modelli applicativi sono strettamente collegati. Tuttavia, per imparare bene è importante separarli fin da subito.

C# non è .NET.
.NET non è Visual Studio.
ASP.NET Core non è tutto .NET.
.NET Framework non è la stessa cosa di .NET moderno.

Questo capitolo serve proprio a costruire una prima mappa mentale dell’ecosistema.

---

## Che cos’è C#

**C#** è un linguaggio di programmazione moderno, fortemente tipizzato, orientato agli oggetti, ma capace anche di supportare stili funzionali, asincroni e dichiarativi.

Con C# possiamo scrivere:

* applicazioni console;
* applicazioni desktop;
* applicazioni web;
* API;
* servizi in background;
* applicazioni cloud;
* applicazioni mobile;
* giochi;
* librerie riutilizzabili;
* microservizi;
* applicazioni distribuite.

C# nasce nell’ecosistema Microsoft, ma oggi è un linguaggio usato in un ambiente molto più ampio rispetto al passato. Non è più legato soltanto a Windows e non è più limitato allo sviluppo desktop o aziendale tradizionale.

Un programma C# moderno può girare su Windows, Linux e macOS. Può essere eseguito in un container Docker, pubblicato in cloud, distribuito come servizio, compilato come applicazione desktop oppure usato per costruire backend web ad alte prestazioni.

C# è quindi il **linguaggio** con cui scriviamo il codice.

Esempio:

```csharp
Console.WriteLine("Ciao, .NET!");
```

Questa riga è codice C#.
Ma per essere eseguita ha bisogno di una piattaforma: **.NET**.

---

## Che cos’è .NET

**.NET** è la piattaforma di sviluppo che permette di compilare, eseguire e distribuire applicazioni scritte in C# e in altri linguaggi compatibili, come F# e Visual Basic.

.NET comprende diversi elementi:

* un runtime, cioè l’ambiente che esegue il codice;
* una libreria standard molto ampia;
* compilatori;
* strumenti da riga di comando;
* sistemi di packaging;
* framework applicativi;
* modelli per creare applicazioni web, desktop, mobile, cloud e servizi.

In modo semplificato possiamo dire:

> C# è il linguaggio.
> .NET è la piattaforma su cui quel linguaggio vive.

Quando scriviamo codice C#, normalmente non parliamo direttamente con il sistema operativo. Usiamo invece le API fornite da .NET.

Per esempio:

```csharp
string testo = File.ReadAllText("documento.txt");
```

In questa riga usiamo C# come linguaggio, ma `File.ReadAllText` appartiene alla libreria di .NET.

---

## Linguaggio, runtime, librerie e strumenti

Per evitare confusione, conviene distinguere quattro livelli.

## Il linguaggio

Il linguaggio è C#.

Definisce la sintassi, cioè il modo in cui scriviamo:

```csharp
if (eta >= 18)
{
    Console.WriteLine("Maggiorenne");
}
```

Il linguaggio stabilisce regole come:

* come dichiarare una variabile;
* come scrivere una classe;
* come definire un metodo;
* come usare `if`, `for`, `switch`;
* come gestire eccezioni;
* come lavorare con `async` e `await`.

---

## Il runtime

Il runtime è l’ambiente che esegue il programma.

Nel mondo .NET il runtime si occupa di attività fondamentali:

* caricamento del programma;
* esecuzione del codice;
* gestione della memoria;
* garbage collection;
* gestione delle eccezioni;
* sicurezza;
* interoperabilità con il sistema operativo;
* compilazione Just-In-Time, quando prevista.

Nel caso di .NET moderno, parliamo del runtime .NET. Nel caso storico di .NET Framework, parliamo del **CLR**, cioè **Common Language Runtime**.

Il concetto da ricordare è semplice:

> Il runtime è ciò che permette al programma compilato di funzionare davvero.

---

## Le librerie

Le librerie sono insiemi di funzionalità già pronte.

Con .NET non partiamo da zero. Abbiamo a disposizione classi per:

* manipolare stringhe;
* lavorare con numeri e date;
* leggere e scrivere file;
* usare collezioni;
* accedere a database;
* creare API web;
* serializzare JSON;
* gestire la sicurezza;
* creare log;
* comunicare via HTTP;
* usare la crittografia;
* lavorare con stream;
* creare applicazioni desktop e mobile.

Per esempio, quando usiamo:

```csharp
var oggi = DateTime.Today;
```

stiamo usando una classe della libreria di base di .NET.

---

## Gli strumenti

Gli strumenti sono i programmi che usiamo per sviluppare.

Tra questi troviamo:

* **Visual Studio**;
* **Visual Studio Code**;
* **JetBrains Rider**;
* la CLI `dotnet`;
* NuGet;
* debugger;
* profiler;
* strumenti di build;
* strumenti di test;
* strumenti di pubblicazione.

Visual Studio non è .NET.
Visual Studio è un ambiente di sviluppo che ci aiuta a lavorare con .NET.

Allo stesso modo, la CLI `dotnet` non è il linguaggio C#: è uno strumento per creare, compilare, eseguire, testare e pubblicare progetti .NET.

---

## Un primo schema mentale

Possiamo rappresentare l’ecosistema così:

```text
Codice C#
   |
   v
Compilatore C#
   |
   v
Assembly .NET
   |
   v
Runtime .NET
   |
   v
Sistema operativo
```

Accanto a questo flusso troviamo le librerie e gli strumenti:

```text
+-------------------------------+
| Strumenti                     |
| Visual Studio, VS Code, CLI   |
+-------------------------------+

+-------------------------------+
| Linguaggio                    |
| C#                            |
+-------------------------------+

+-------------------------------+
| Piattaforma                   |
| .NET Runtime + Librerie       |
+-------------------------------+

+-------------------------------+
| Modelli applicativi           |
| Console, Web, API, Desktop    |
| Mobile, Cloud, Microservizi   |
+-------------------------------+
```

Questa separazione è fondamentale. Quando qualcosa non funziona, dobbiamo chiederci: il problema è nel codice C#? Nel progetto? Nel runtime? Nella versione di .NET? Nel pacchetto NuGet? Nell’ambiente di sviluppo?

---

# Dal .NET Framework a .NET moderno

Per molti anni, quando si parlava di .NET, si intendeva soprattutto **.NET Framework**.

.NET Framework era la piattaforma principale per sviluppare applicazioni Windows:

* applicazioni desktop Windows Forms;
* applicazioni WPF;
* applicazioni ASP.NET classiche;
* servizi Windows;
* applicazioni aziendali basate su Windows e IIS.

Il .NET Framework è stato fondamentale nella storia dello sviluppo Microsoft, ma aveva un limite importante: era fortemente legato a Windows.

Con il tempo, il mondo dello sviluppo è cambiato. Le applicazioni hanno iniziato a dover girare su più sistemi operativi, in cloud, su Linux, dentro container, in ambienti distribuiti. Microsoft ha quindi creato una nuova piattaforma: **.NET Core**.

.NET Core nasce come piattaforma:

* cross-platform;
* open source;
* modulare;
* più leggera;
* più adatta al cloud;
* più adatta ai container;
* con prestazioni elevate.

Successivamente, con .NET 5, Microsoft ha unificato il percorso e ha semplificato il nome: non si parla più di “.NET Core” come nome principale, ma semplicemente di **.NET**.

Per questo oggi diciamo:

* **.NET Framework** per la piattaforma storica Windows;
* **.NET** per la piattaforma moderna e cross-platform.

---

## .NET Framework

.NET Framework è la piattaforma storica di Microsoft per Windows.

È ancora presente in moltissime aziende e in molte applicazioni esistenti. Alcuni esempi tipici:

* vecchie applicazioni Windows Forms;
* applicazioni WPF aziendali;
* applicazioni ASP.NET Web Forms;
* applicazioni ASP.NET MVC non Core;
* servizi Windows;
* software integrati con componenti COM;
* applicazioni legacy molto legate a Windows.

.NET Framework non è “scomparso”, ma non è più il centro dell’evoluzione .NET. Microsoft raccomanda, quando possibile, di migrare le applicazioni da .NET Framework a .NET moderno; quando non è possibile, la raccomandazione è almeno aggiornare a .NET Framework 4.8.1. ([Microsoft Learn][1])

Questo punto è importante: in molti ambienti aziendali .NET Framework continuerà a esistere ancora a lungo. Non va demonizzato, ma va compreso per quello che è: una piattaforma matura, stabile, legata soprattutto al mondo Windows e al software esistente.

---

## .NET Core

.NET Core è stato il grande passaggio verso il .NET moderno.

Con .NET Core, Microsoft ha cambiato direzione:

* apertura al mondo open source;
* supporto a Windows, Linux e macOS;
* forte attenzione alle performance;
* architettura più modulare;
* uso più naturale in cloud e container;
* nuova generazione di ASP.NET: ASP.NET Core.

Storicamente, .NET Core ha rappresentato il ponte tra il vecchio .NET Framework e il .NET moderno.

Oggi, però, quando creiamo un nuovo progetto, normalmente non diciamo più “uso .NET Core”, ma “uso .NET”, per esempio .NET 8, .NET 9 o .NET 10.

---

## .NET 5 e versioni successive

Con .NET 5, Microsoft ha iniziato una fase di unificazione.

Il nome “Core” è stato rimosso dal nome principale della piattaforma. Da quel momento, la piattaforma moderna viene chiamata semplicemente **.NET**.

Quindi:

```text
.NET Framework 4.x  -> piattaforma storica Windows
.NET Core 1/2/3     -> piattaforma moderna cross-platform iniziale
.NET 5+             -> piattaforma moderna unificata
```

Esempi di versioni moderne:

```text
.NET 6
.NET 7
.NET 8
.NET 9
.NET 10
```

Nel linguaggio comune può capitare di sentire ancora “.NET Core” per indicare .NET moderno. Tecnicamente, però, è più corretto parlare semplicemente di **.NET**.

---

## Mono

**Mono** è una storica implementazione open source di .NET.

Per anni ha permesso di eseguire codice .NET fuori dal mondo Windows, soprattutto quando .NET Framework era ancora strettamente legato a Windows.

Mono è stato importante per diversi scenari:

* Linux;
* macOS;
* applicazioni mobile;
* Unity;
* Xamarin.

Oggi il suo ruolo è cambiato, perché .NET moderno è già cross-platform. Tuttavia Mono rimane rilevante in alcuni contesti, soprattutto dove servono modelli di runtime particolari, come nel mondo mobile e WebAssembly.

---

## Xamarin

**Xamarin** è stata la tecnologia usata per creare applicazioni mobile native con C# e .NET, in particolare per Android e iOS.

L’idea di fondo era molto potente: scrivere logica applicativa in C# e condividerla tra più piattaforme, mantenendo la possibilità di accedere alle API native.

Xamarin ha avuto un ruolo fondamentale nell’evoluzione verso **.NET MAUI**.

---

## .NET MAUI

**.NET MAUI**, cioè **.NET Multi-platform App UI**, è il framework moderno per creare applicazioni native mobile e desktop con C# e XAML. Microsoft lo presenta come un framework multipiattaforma per creare app native per dispositivi mobili e desktop; consente di sviluppare applicazioni per Android, iOS, macOS e Windows da un’unica codebase condivisa. ([Microsoft Learn][2])

.NET MAUI è l’evoluzione di Xamarin.Forms, estesa dagli scenari mobile anche a quelli desktop. ([Microsoft Learn][2])

Con .NET MAUI possiamo quindi creare applicazioni per:

* Android;
* iOS;
* macOS;
* Windows.

Un aspetto importante è che .NET MAUI usa un singolo progetto con multi-targeting e permette di condividere la maggior parte della logica e dell’interfaccia, pur consentendo codice specifico per singola piattaforma quando serve. ([Microsoft Learn][2])

---

# Il concetto di One .NET

Per molti anni il mondo .NET è stato frammentato:

* .NET Framework per Windows;
* .NET Core per applicazioni moderne cross-platform;
* Mono per scenari multipiattaforma;
* Xamarin per mobile;
* Unity con una propria integrazione;
* .NET Standard come tentativo di definire API comuni.

Questa frammentazione rendeva difficile rispondere a una domanda apparentemente semplice:

> “Questa libreria funziona nel mio progetto?”

Il concetto di **One .NET** nasce per semplificare questa situazione.

L’idea è avere una piattaforma unificata:

* un solo ecosistema;
* un solo SDK;
* un solo modello di progetto quando possibile;
* una libreria di base condivisa;
* strumenti coerenti;
* supporto a più tipi di applicazioni.

Naturalmente, “unificato” non significa “identico in ogni scenario”. Un’applicazione web, un’app mobile e un’app desktop hanno esigenze diverse. Però condividono molto più codice, strumenti e concetti rispetto al passato.

Possiamo immaginare One .NET così:

```text
                    +-------------------+
                    |       .NET        |
                    +-------------------+
                             |
        +--------------------+--------------------+
        |                    |                    |
   Applicazioni          Applicazioni          Applicazioni
     Web/API              Desktop              Mobile
        |                    |                    |
 ASP.NET Core           WPF/WinUI          .NET MAUI
 Blazor                                     
        |
   Cloud, servizi,
   microservizi
```

Il beneficio principale è che, imparando bene C# e .NET, possiamo spostarci tra molti tipi di applicazioni senza ricominciare da zero.

---

# Versioni LTS e STS

.NET moderno ha un ciclo di rilascio regolare. Le versioni principali vengono rilasciate annualmente e sono classificate in due famiglie:

* **LTS**, Long Term Support;
* **STS**, Standard Term Support.

Secondo la policy ufficiale Microsoft, le versioni LTS ricevono supporto per tre anni, mentre le versioni STS ricevono supporto per due anni. Per rimanere supportati è necessario installare le patch più recenti della versione usata. ([Microsoft][3])

In pratica:

```text
LTS = più adatta a progetti che cercano stabilità e supporto lungo
STS = più adatta a chi vuole usare prima le novità
```

La qualità delle versioni non cambia perché una è LTS e una è STS. Cambia principalmente la durata del supporto.

Per un progetto aziendale, spesso conviene scegliere una versione **LTS**, perché riduce la frequenza obbligata degli aggiornamenti.

Per studio, sperimentazione o progetti dove si vogliono provare le funzionalità più recenti, una versione **STS** può andare benissimo.

---

## Esempio pratico di scelta

Immaginiamo di dover creare un nuovo gestionale aziendale che dovrà restare in produzione per anni.

In questo caso, una scelta ragionevole è usare una versione LTS di .NET.

Immaginiamo invece di voler provare le novità del linguaggio, del runtime o di Blazor.

In questo caso potremmo scegliere una versione STS o l’ultima versione disponibile, sapendo però che il ciclo di aggiornamento sarà più ravvicinato.

---

# Compatibilità, supporto e ciclo di vita

Quando si sceglie una versione di .NET non bisogna guardare solo alle funzionalità tecniche, ma anche al ciclo di vita.

Una versione non supportata può esporre un progetto a problemi importanti:

* assenza di patch di sicurezza;
* incompatibilità con strumenti recenti;
* difficoltà nel ricevere assistenza;
* problemi in ambienti enterprise;
* limiti nella distribuzione;
* impossibilità di aggiornare alcune dipendenze.

Per questo, quando si crea un progetto nuovo, è buona pratica scegliere una versione supportata.

In un file `.csproj`, la versione target viene indicata dal **Target Framework Moniker**, spesso abbreviato in **TFM**.

Esempio:

```xml
<TargetFramework>net10.0</TargetFramework>
```

Oppure:

```xml
<TargetFramework>net8.0</TargetFramework>
```

Per un’applicazione .NET Framework potremmo invece trovare:

```xml
<TargetFramework>net481</TargetFramework>
```

Il TFM dice al compilatore e agli strumenti quale piattaforma vogliamo usare.

# Quando usare .NET Framework e quando usare .NET moderno

Una domanda molto pratica è:

> Devo usare .NET Framework o .NET moderno?

La risposta dipende dal tipo di progetto.

---

## Nuovi progetti

Per nuovi progetti, nella maggior parte dei casi la scelta consigliata è **.NET moderno**.

È la scelta naturale per:

* nuove API;
* applicazioni ASP.NET Core;
* servizi;
* worker service;
* applicazioni cloud;
* microservizi;
* applicazioni console moderne;
* progetti containerizzati;
* applicazioni Blazor;
* librerie nuove;
* applicazioni cross-platform;
* nuovi progetti desktop quando possibile.

.NET moderno offre:

* migliori prestazioni;
* supporto cross-platform;
* evoluzione attiva;
* migliore integrazione con cloud e container;
* tooling moderno;
* supporto a C# più recente.

---

## Progetti esistenti

.NET Framework può avere ancora senso quando si lavora su applicazioni esistenti che dipendono fortemente da tecnologie storiche, per esempio:

* Web Forms;
* vecchie applicazioni WPF;
* vecchie applicazioni Windows Forms;
* componenti COM;
* librerie legacy;
* vecchi sistemi aziendali;
* dipendenze non compatibili con .NET moderno;
* applicazioni molto grandi dove la migrazione ha costi elevati.

In questi casi non sempre è realistico riscrivere tutto.

La scelta più ragionevole può essere:

1. mantenere il progetto su .NET Framework se funziona ed è supportato;
2. aggiornarlo almeno a una versione recente e supportata del .NET Framework;
3. isolare progressivamente le parti più vecchie;
4. migrare gradualmente verso .NET moderno quando possibile.

---

## Regola pratica

Una regola semplice può essere questa:

```text
Nuovo progetto?
    Usa .NET moderno.

Progetto legacy stabile e molto legato a Windows?
    Può restare su .NET Framework, ma valuta aggiornamento e migrazione.

Progetto web nuovo?
    Usa ASP.NET Core su .NET moderno.

Applicazione cloud o container?
    Usa .NET moderno.

Applicazione Web Forms esistente?
    Resta su .NET Framework oppure pianifica una migrazione architetturale.
```

# Panoramica degli ambiti applicativi

Uno dei punti di forza di .NET è la varietà degli scenari supportati. Imparando C# e .NET, possiamo lavorare in molti ambiti diversi.

---

## Applicazioni console

Le applicazioni console sono il punto di partenza più semplice.

Sono utili per:

* imparare il linguaggio;
* creare tool interni;
* scrivere script evoluti;
* elaborare file;
* automatizzare procedure;
* creare utility da riga di comando.

Esempio:

```csharp
Console.WriteLine("Elaborazione completata.");
```

Anche se sembrano semplici, le console app sono molto usate in contesti reali.

---

## Applicazioni desktop

.NET supporta diversi modelli per lo sviluppo desktop.

I principali sono:

* Windows Forms;
* WPF;
* WinUI;
* .NET MAUI per scenari multipiattaforma.

Windows Forms è più semplice e storico.
WPF è più potente per interfacce desktop Windows ricche.
WinUI è più moderno nell’ecosistema Windows.
.NET MAUI punta al multipiattaforma mobile e desktop.

Le applicazioni desktop sono ancora molto importanti in ambito aziendale, industriale e gestionale.

---

## Applicazioni web

Per il web, il framework principale è **ASP.NET Core**.

Con ASP.NET Core possiamo creare:

* siti web;
* applicazioni server-side;
* API;
* backend per app mobile;
* servizi per frontend JavaScript;
* applicazioni Blazor;
* sistemi real-time con SignalR.

ASP.NET Core è uno dei componenti più importanti del .NET moderno.

---

## API

Le API sono servizi che espongono funzionalità ad altri software.

Con .NET possiamo creare API usando:

* Minimal API;
* Controller;
* gRPC;
* SignalR per comunicazioni real-time.

Le API sono fondamentali per:

* applicazioni web moderne;
* app mobile;
* integrazioni tra sistemi;
* microservizi;
* architetture distribuite.

---

## Servizi

Un servizio è un’applicazione che lavora senza una UI diretta.

Esempi:

* servizio che elabora code;
* servizio che legge file da una cartella;
* servizio che invia email;
* servizio che sincronizza dati;
* servizio che esegue elaborazioni pianificate.

In .NET moderno possiamo creare questi scenari con:

* Worker Service;
* BackgroundService;
* servizi Windows;
* servizi Linux con systemd;
* container.

---

## Cloud

.NET è molto usato nel cloud.

Possiamo distribuire applicazioni .NET su:

* Azure;
* AWS;
* Google Cloud;
* server Linux;
* container Docker;
* Kubernetes;
* ambienti serverless.

Il cloud non cambia il linguaggio, ma cambia il modo in cui pensiamo a configurazione, scalabilità, logging, sicurezza e deployment.

---

## Mobile

Per il mobile, la tecnologia principale nell’ecosistema .NET moderno è .NET MAUI.

Con .NET MAUI possiamo creare applicazioni Android e iOS condividendo gran parte del codice. La documentazione Microsoft indica .NET MAUI come toolkit UI cross-platform per app native mobile e desktop su Android, iOS, macOS, Windows e Tizen. ([Microsoft Learn][4])

Non tutti i progetti mobile sono adatti a .NET MAUI, ma per team che conoscono già C# può essere una scelta interessante.

---

## Microservizi

I microservizi sono un modello architetturale in cui un sistema viene diviso in servizi piccoli, autonomi e distribuibili separatamente.

.NET è adatto alla creazione di microservizi grazie a:

* ASP.NET Core;
* Minimal API;
* gRPC;
* Docker;
* Kubernetes;
* configurazione moderna;
* logging;
* osservabilità;
* ottime prestazioni;
* integrazione con sistemi di messaging.

Tuttavia, è importante chiarire subito una cosa:

> I microservizi non sono il punto di partenza naturale per ogni applicazione.

Spesso, per un progetto aziendale, è meglio partire da un **monolite modulare** ben progettato e passare a microservizi solo quando ci sono motivi concreti:

* team indipendenti;
* necessità di scalare parti diverse separatamente;
* deployment indipendenti;
* domini ben separati;
* esigenze organizzative forti.

# Un possibile equivoco: “devo imparare tutto subito?”

No.

L’ecosistema .NET è molto ampio, ma non bisogna imparare tutto contemporaneamente.

Il percorso corretto è progressivo:

```text
1. C# base
2. OOP e tipi
3. Librerie fondamentali
4. LINQ
5. File e serializzazione
6. Database
7. Web/API
8. Test
9. Async e performance
10. Architettura
11. Deployment
12. Microservizi e cloud
```

Questo libro seguirà proprio questa logica: prima le fondamenta, poi gli strumenti, poi le applicazioni reali.

# Conclusione

C# e .NET formano uno degli ecosistemi più completi per lo sviluppo software moderno.

C# è il linguaggio con cui scriviamo il codice.
.NET è la piattaforma che lo compila, lo esegue e gli mette a disposizione librerie, runtime e strumenti.
ASP.NET Core, Blazor, EF Core, MAUI e gli altri framework sono specializzazioni costruite sopra questa base.

Capire questa distinzione all’inizio evita molta confusione dopo.

Quando nei prossimi capitoli scriveremo codice, creeremo progetti, useremo librerie, installeremo pacchetti NuGet o sceglieremo una versione target, sapremo collocare ogni elemento nella giusta posizione.

La mappa mentale da tenere presente è questa:

```text
C# = linguaggio
.NET = piattaforma
Runtime = esecuzione
SDK = sviluppo
Librerie = funzionalità pronte
Framework applicativi = modelli per costruire app reali
Strumenti = ambiente di lavoro
```

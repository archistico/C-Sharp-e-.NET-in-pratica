# 01.02 — Ambiente di sviluppo e strumenti

Nel capitolo precedente abbiamo costruito una prima mappa dell’ecosistema .NET. Abbiamo distinto il linguaggio C#, la piattaforma .NET, il runtime, le librerie, gli SDK e i framework applicativi.

Ora dobbiamo passare a un livello più concreto:

> quali strumenti servono per sviluppare davvero?

Prima di scrivere programmi complessi, è importante sapere come installare l’ambiente, creare una solution, aggiungere progetti, compilare, eseguire, fare debug, pubblicare un’applicazione e gestire le dipendenze.

Nel mondo .NET possiamo lavorare con diversi strumenti:

* Visual Studio;
* Visual Studio Code;
* JetBrains Rider;
* la CLI `dotnet`;
* NuGet;
* Git;
* strumenti di test;
* strumenti AI come ChatGPT e GitHub Copilot.

Non è necessario usarli tutti subito, ma è utile capire il ruolo di ciascuno.

---

## IDE, editor, SDK e CLI

Prima di iniziare, distinguiamo alcuni termini.

Un **IDE**, cioè Integrated Development Environment, è un ambiente di sviluppo completo. Include editor, compilazione, debug, gestione progetti, strumenti grafici, integrazione con Git, test e molto altro.

Esempi:

* Visual Studio;
* JetBrains Rider.

Un **editor** è uno strumento più leggero per scrivere codice. Può diventare molto potente grazie alle estensioni, ma di base è meno integrato rispetto a un IDE completo.

Esempio:

* Visual Studio Code.

L’**SDK** è il Software Development Kit. Nel caso di .NET, contiene tutto ciò che serve per creare, compilare, eseguire e pubblicare applicazioni .NET.

La **CLI** è l’interfaccia da riga di comando. Nel mondo .NET il comando principale è:

```bash id="mcvq5e"
dotnet
```

Possiamo rappresentare il rapporto tra questi elementi così:

```text id="u2py1s"
IDE / Editor
    |
    v
.NET SDK
    |
    v
CLI dotnet, compilatore, strumenti di build
    |
    v
Progetto .NET
    |
    v
Applicazione eseguibile
```

Visual Studio, VS Code e Rider sono strumenti con cui lavoriamo.
Il .NET SDK è ciò che rende possibile creare e compilare i progetti.
La CLI `dotnet` è il modo diretto per usare lo SDK da terminale.

# Installazione di Visual Studio

**Visual Studio** è l’ambiente di sviluppo più completo per C# e .NET su Windows.

È particolarmente adatto quando vogliamo lavorare con:

* applicazioni console;
* applicazioni desktop Windows Forms;
* applicazioni WPF;
* applicazioni ASP.NET Core;
* API;
* servizi;
* test;
* database;
* strumenti grafici;
* debugging avanzato.

L’installazione di Visual Studio avviene tramite **Visual Studio Installer**, che permette di scegliere le funzionalità da installare attraverso i cosiddetti **workload**. Microsoft indica Visual Studio Installer come lo strumento usato per installare e configurare Visual Studio, e alcuni workload includono già componenti .NET come ASP.NET, sviluppo web e .NET MAUI. ([Microsoft Learn][1])

---

## I workload più importanti

Quando si installa Visual Studio, non conviene selezionare tutto. È meglio installare ciò che serve.

Per un percorso C#/.NET generale, i workload più utili sono:

```text id="l7v7dx"
Sviluppo per desktop .NET
ASP.NET e sviluppo web
Sviluppo multipiattaforma .NET MAUI
Elaborazione e archiviazione dati
```

In italiano i nomi possono variare leggermente in base alla versione di Visual Studio, ma il concetto rimane lo stesso.

---

## Sviluppo per desktop .NET

Questo workload serve per creare applicazioni desktop Windows.

Include supporto per:

* Windows Forms;
* WPF;
* librerie desktop;
* strumenti di debug;
* designer visuali, quando disponibili.

È il workload da installare se vogliamo sviluppare applicazioni desktop gestionali, strumenti interni, software aziendali Windows o applicazioni WPF.

---

## ASP.NET e sviluppo web

Questo workload serve per creare applicazioni web e API.

Include supporto per:

* ASP.NET Core;
* Minimal API;
* MVC;
* Razor Pages;
* Blazor;
* strumenti web;
* IIS Express o profili di esecuzione web;
* debugging di applicazioni web.

È il workload da installare se vogliamo creare backend, API REST, siti web o applicazioni Blazor.

---

## Sviluppo multipiattaforma .NET MAUI

Questo workload serve per creare applicazioni mobile e desktop multipiattaforma con .NET MAUI.

È utile per scenari Android, iOS, macOS e Windows, ma richiede spesso componenti aggiuntivi, emulatori, SDK specifici e configurazioni più pesanti rispetto a un normale progetto console o web.

Per iniziare con C# non è indispensabile installarlo subito.

---

## Installazione minima consigliata per questo libro

Per seguire un percorso completo C# e .NET, su Windows installerei almeno:

```text id="oubd4z"
Visual Studio
  - Sviluppo per desktop .NET
  - ASP.NET e sviluppo web
  - Elaborazione e archiviazione dati, se disponibile
```

.NET MAUI può essere installato più avanti, quando si arriva alla parte sulle applicazioni client e mobile.

# Installazione del .NET SDK

Il **.NET SDK** è indispensabile.

Senza SDK possiamo forse eseguire alcune applicazioni già compilate, se abbiamo il runtime corretto, ma non possiamo sviluppare comodamente nuovi progetti.

Lo SDK contiene:

* compilatore C#;
* CLI `dotnet`;
* strumenti di build;
* template di progetto;
* strumenti per il restore dei pacchetti;
* strumenti per test e publish;
* runtime necessari allo sviluppo.

Microsoft distingue l’installazione del runtime dall’installazione dello SDK: lo SDK serve agli sviluppatori per creare applicazioni, mentre il runtime serve per eseguire applicazioni già costruite. Inoltre, alcuni workload di Visual Studio possono installare componenti .NET, ma versioni specifiche dello SDK possono essere installate anche separatamente. ([Microsoft Learn][1])

---

## Verificare se lo SDK è installato

Dopo l’installazione, apriamo un terminale e scriviamo:

```bash id="r18mgz"
dotnet --version
```

Il comando restituisce la versione dello SDK predefinito.

Esempio:

```text id="n7u1hk"
10.0.100
```

Per vedere tutti gli SDK installati:

```bash id="hle4v9"
dotnet --list-sdks
```

Per vedere tutti i runtime installati:

```bash id="vnlk2o"
dotnet --list-runtimes
```

Questi tre comandi sono molto utili quando un progetto non compila o quando Visual Studio sembra usare una versione diversa da quella attesa.

---

## SDK e runtime non sono la stessa cosa

Questa distinzione è importante.

```text id="96ehf2"
SDK
  serve per sviluppare, compilare, testare e pubblicare.

Runtime
  serve per eseguire un'applicazione già compilata.
```

Su una macchina di sviluppo installiamo lo SDK.
Su un server di produzione può bastare il runtime, se l’applicazione non è pubblicata come self-contained.

---

## Esempio pratico

Uno sviluppatore può avere installato:

```text id="5o8p95"
.NET SDK 8
.NET SDK 9
.NET SDK 10
```

Un’applicazione può invece targettare:

```xml id="dhl5rj"
<TargetFramework>net8.0</TargetFramework>
```

Questo significa che il computer può avere SDK più recenti, ma il progetto può comunque essere compilato per una versione precedente di .NET.

# Visual Studio Code e C# Dev Kit

**Visual Studio Code** è un editor leggero, multipiattaforma e molto estendibile.

Non va confuso con Visual Studio.

```text id="y5qk71"
Visual Studio
  IDE completo, soprattutto Windows, molto integrato.

Visual Studio Code
  editor leggero, multipiattaforma, basato su estensioni.
```

Per sviluppare in C# con VS Code, serve installare il supporto C#. La documentazione ufficiale di VS Code indica che il supporto C# è fornito dall’estensione **C# Dev Kit**, installabile dalla vista Extensions cercando “C# Dev Kit”; VS Code può anche proporre l’installazione quando si apre un file C#. ([code.visualstudio.com][2])

---

## Che cos’è C# Dev Kit

**C# Dev Kit** è un’estensione che migliora l’esperienza di sviluppo C# in Visual Studio Code. Microsoft la descrive come un’estensione pensata per offrire un’esperienza C# più ampia, produttiva e affidabile in VS Code; non sostituisce l’estensione C# di base, ma si appoggia alle funzionalità del language service. ([code.visualstudio.com][3])

Con C# Dev Kit possiamo avere:

* esplorazione della solution;
* IntelliSense;
* navigazione tra tipi e membri;
* esecuzione e debug;
* integrazione con test;
* supporto ai progetti `.csproj`;
* esperienza più simile a un IDE, pur restando dentro VS Code.

---

## Quando usare VS Code

VS Code è una buona scelta quando:

* vogliamo un ambiente leggero;
* lavoriamo spesso da terminale;
* sviluppiamo su Linux o macOS;
* lavoriamo con container;
* facciamo sviluppo web;
* vogliamo usare lo stesso editor per più linguaggi;
* preferiamo controllare esplicitamente la CLI.

---

## Quando preferire Visual Studio

Visual Studio è spesso preferibile quando:

* lavoriamo su WPF o Windows Forms;
* vogliamo designer visuali;
* abbiamo solution molto grandi;
* usiamo strumenti di debug avanzati;
* lavoriamo in ambiente aziendale Windows;
* vogliamo un’esperienza integrata senza configurare molte estensioni.

# JetBrains Rider

**JetBrains Rider** è un IDE completo per .NET, sviluppato da JetBrains.

È multipiattaforma e funziona su Windows, macOS e Linux. JetBrains lo presenta come un IDE cross-platform per sviluppatori .NET e game development. ([JetBrains][4])

Rider è molto apprezzato per:

* analisi del codice;
* refactoring;
* navigazione;
* integrazione con Git;
* supporto a test;
* supporto a database;
* strumenti per ASP.NET Core;
* supporto a Unity;
* esperienza coerente su più sistemi operativi.

---

## Visual Studio, VS Code o Rider?

Non esiste una risposta unica.

Possiamo ragionare così:

```text id="7k5lj6"
Visual Studio
  scelta molto forte su Windows, soprattutto per desktop, web e debugging integrato.

VS Code
  scelta leggera, flessibile, ottima per CLI, web, container e ambienti multipiattaforma.

Rider
  IDE completo multipiattaforma, molto potente per refactoring, analisi e produttività.
```

Per imparare C# e .NET, Visual Studio è spesso la scelta più semplice su Windows.
Per chi ama lavorare da terminale o su più sistemi operativi, VS Code è ottimo.
Per chi vuole un IDE professionale multipiattaforma, Rider è una scelta eccellente.

# La CLI dotnet

La CLI `dotnet` è uno degli strumenti più importanti dell’ecosistema .NET moderno.

Anche se usiamo Visual Studio, è molto utile conoscere la CLI, perché permette di capire cosa succede davvero dietro l’interfaccia grafica.

Il comando principale è:

```bash id="m0ry4n"
dotnet
```

Con questo comando possiamo:

* creare progetti;
* creare solution;
* aggiungere riferimenti;
* aggiungere pacchetti NuGet;
* compilare;
* eseguire;
* testare;
* pubblicare;
* gestire workload;
* vedere informazioni sull’ambiente.

---

## La CLI come strumento didattico

La CLI è preziosa perché mostra la struttura reale di un progetto.

* come vengono ripristinati i pacchetti;
* come si eseguono i test;
* come viene pubblicata l’applicazione;
* quali errori sono davvero errori del compilatore e non dell’IDE.

# Creazione di una solution

Una **solution** è un contenitore logico di uno o più progetti.

In Visual Studio una solution viene rappresentata da un file con estensione:

```text id="eyfopb"
.sln
```

Esempio:

```text id="36cz54"
GestionaleClienti.sln
```

Una solution può contenere un solo progetto, ma nei casi reali spesso ne contiene molti.

Esempio:

```text id="uj5x6j"
GestionaleClienti.sln
  GestionaleClienti.Domain
  GestionaleClienti.Application
  GestionaleClienti.Infrastructure
  GestionaleClienti.Api
  GestionaleClienti.Tests
```

La solution non è l’applicazione.
È il contenitore che aiuta a organizzare più progetti correlati.

---

## Creare una solution da CLI

Creiamo una cartella:

```bash id="lf8l3m"
mkdir GestionaleClienti
cd GestionaleClienti
```

Creiamo la solution:

```bash id="57avhy"
dotnet new sln -n GestionaleClienti
```

Otteniamo un file:

```text id="ga8xox"
GestionaleClienti.sln
```

A questo punto la solution esiste, ma non contiene ancora progetti.

---

# Creazione di progetti

Un **progetto** è l’unità concreta che viene compilata.

Un progetto .NET è rappresentato da un file con estensione:

```text id="ckxcc1"
.csproj
```

Esempio:

```text id="eqd43u"
GestionaleClienti.Api.csproj
```

Il progetto contiene informazioni come:

* versione target di .NET;
* pacchetti NuGet;
* riferimenti ad altri progetti;
* impostazioni di compilazione;
* tipo di output;
* proprietà specifiche.

---

## Creare un progetto console

```bash id="zasswb"
dotnet new console -n GestionaleClienti.Console
```

Aggiungiamolo alla solution:

```bash id="dw4iwv"
dotnet sln add GestionaleClienti.Console/GestionaleClienti.Console.csproj
```

Ora la solution contiene il progetto console.

---

## Creare una libreria di classi

```bash id="70pos2"
dotnet new classlib -n GestionaleClienti.Domain
```

Aggiungiamola alla solution:

```bash id="6qwwe9"
dotnet sln add GestionaleClienti.Domain/GestionaleClienti.Domain.csproj
```

---

## Creare un progetto di test

```bash id="33t8p2"
dotnet new xunit -n GestionaleClienti.Tests
```

Aggiungiamolo alla solution:

```bash id="4vo9jo"
dotnet sln add GestionaleClienti.Tests/GestionaleClienti.Tests.csproj
```

---

## Collegare i progetti tra loro

Supponiamo che il progetto di test debba testare il progetto `Domain`.

Aggiungiamo un riferimento:

```bash id="el6v7z"
dotnet add GestionaleClienti.Tests/GestionaleClienti.Tests.csproj reference GestionaleClienti.Domain/GestionaleClienti.Domain.csproj
```

Ora il progetto di test può usare le classi del progetto `Domain`.

# Struttura di un progetto .csproj

Il file `.csproj` è il file di progetto.

Un progetto console moderno può avere una struttura molto semplice:

```xml id="8w71dr"
<Project Sdk="Microsoft.NET.Sdk">

  <PropertyGroup>
    <OutputType>Exe</OutputType>
    <TargetFramework>net10.0</TargetFramework>
    <ImplicitUsings>enable</ImplicitUsings>
    <Nullable>enable</Nullable>
  </PropertyGroup>

</Project>
```

Vediamo i punti principali.

---

## Project Sdk

```xml id="buuxyg"
<Project Sdk="Microsoft.NET.Sdk">
```

Indica quale SDK viene usato per compilare il progetto.

Per una normale libreria o console app si usa:

```text id="mxndj7"
Microsoft.NET.Sdk
```

Per un progetto web si usa:

```xml id="n6jg0j"
<Project Sdk="Microsoft.NET.Sdk.Web">
```

Per altri tipi di applicazione possono esserci SDK diversi.

---

## OutputType

```xml id="ygc2pv"
<OutputType>Exe</OutputType>
```

Indica che il progetto produce un eseguibile.

Una libreria di classi, invece, normalmente non ha `OutputType` impostato a `Exe`, perché produce una DLL.

---

## TargetFramework

```xml id="y8z3r6"
<TargetFramework>net10.0</TargetFramework>
```

Indica la piattaforma target.

Altri esempi:

```xml id="l4z6ai"
<TargetFramework>net8.0</TargetFramework>
```

```xml id="8rxy6a"
<TargetFramework>net481</TargetFramework>
```

```xml id="u94rnz"
<TargetFramework>netstandard2.0</TargetFramework>
```

Questa impostazione è importantissima, perché determina quali API possiamo usare e con quali runtime il progetto sarà compatibile.

---

## ImplicitUsings

```xml id="j6r8z9"
<ImplicitUsings>enable</ImplicitUsings>
```

Abilita automaticamente alcuni `using` comuni.

Per esempio, in un progetto moderno possiamo scrivere:

```csharp id="xv3igs"
Console.WriteLine("Ciao");
```

senza dover dichiarare manualmente:

```csharp id="x3yadd"
using System;
```

Questo rende il codice iniziale più leggero.

---

## Nullable

```xml id="zu7x5p"
<Nullable>enable</Nullable>
```

Abilita i Nullable Reference Types.

Questo aiuta il compilatore a segnalarci possibili errori legati a valori `null`.

Esempio:

```csharp id="p4a237"
string nome = null; // warning se Nullable è abilitato
```

Questa impostazione è molto consigliata nei nuovi progetti.

---

## PackageReference

Quando installiamo un pacchetto NuGet, nel `.csproj` compare una sezione simile:

```xml id="6svtl2"
<ItemGroup>
  <PackageReference Include="Dapper" Version="2.1.66" />
</ItemGroup>
```

Questo significa che il progetto dipende dal pacchetto `Dapper`.

---

## ProjectReference

Quando un progetto fa riferimento a un altro progetto, compare una sezione simile:

```xml id="x3r6nf"
<ItemGroup>
  <ProjectReference Include="..\GestionaleClienti.Domain\GestionaleClienti.Domain.csproj" />
</ItemGroup>
```

Questo significa che il progetto corrente usa il progetto `GestionaleClienti.Domain`.

# Debug, build, run e publish

Nel lavoro quotidiano useremo spesso quattro azioni:

```text id="e4aqv6"
build
run
debug
publish
```

Sono simili, ma non equivalenti.

---

## Build

La **build** compila il progetto.

Da CLI:

```bash id="vz1st0"
dotnet build
```

La build verifica che il codice sia corretto e produce gli output compilati.

Durante la build possono emergere:

* errori di sintassi;
* errori di tipo;
* riferimenti mancanti;
* pacchetti non ripristinati;
* warning del compilatore.

La build non serve necessariamente a eseguire il programma. Serve a compilarlo.

---

## Run

Il comando **run** compila, se necessario, ed esegue il progetto.

```bash id="nqzsxk"
dotnet run
```

È il comando più comodo durante lo sviluppo di console app, API e piccoli progetti.

---

## Debug

Il **debug** permette di eseguire il programma controllandone il comportamento passo passo.

Durante il debug possiamo:

* mettere breakpoint;
* ispezionare variabili;
* vedere il call stack;
* eseguire una riga alla volta;
* valutare espressioni;
* capire perché il programma si comporta in un certo modo.

Il debug è uno degli strumenti più importanti per imparare davvero.

---

## Publish

Il comando **publish** prepara l’applicazione per la distribuzione.

```bash id="wktifw"
dotnet publish -c Release
```

La pubblicazione crea una cartella con i file necessari per eseguire l’applicazione.

In base alle opzioni, possiamo pubblicare:

* framework-dependent;
* self-contained;
* single file;
* trimmed;
* Native AOT, per scenari compatibili.

Esempio:

```bash id="c66rmo"
dotnet publish -c Release -r win-x64 --self-contained true
```

Questo comando pubblica per Windows a 64 bit includendo il runtime.

---

## Differenza pratica

```text id="el2xfw"
build
  compila.

run
  compila ed esegue.

debug
  esegue sotto controllo del debugger.

publish
  prepara i file per distribuire l'applicazione.
```

# Hot Reload

**Hot Reload** permette di modificare il codice mentre l’applicazione è in esecuzione e vedere gli effetti senza riavviare completamente il programma.

È molto utile in scenari come:

* applicazioni web;
* Blazor;
* UI;
* modifiche leggere alla logica;
* sviluppo iterativo.

Non tutte le modifiche possono essere applicate con Hot Reload. Alcuni cambiamenti richiedono ancora il riavvio dell’applicazione.

La cosa importante è non confondere Hot Reload con la compilazione tradizionale:

```text id="psuofy"
Build
  ricompila il progetto.

Hot Reload
  prova ad applicare modifiche al programma già in esecuzione.
```

Hot Reload è comodo, ma non sostituisce test, build pulite e debug.

# Gestione di più progetti in una solution

Nelle applicazioni reali è raro avere un solo progetto.

Una struttura più professionale separa le responsabilità.

Esempio:

```text id="lfeqej"
GestionaleClienti.sln

src/
  GestionaleClienti.Domain/
  GestionaleClienti.Application/
  GestionaleClienti.Infrastructure/
  GestionaleClienti.Api/
  GestionaleClienti.Web/

tests/
  GestionaleClienti.Domain.Tests/
  GestionaleClienti.Application.Tests/
  GestionaleClienti.Api.Tests/
```

Questa organizzazione aiuta a separare:

* logica di dominio;
* casi d’uso;
* accesso ai dati;
* API;
* interfaccia utente;
* test.

---

## Esempio di dipendenze corrette

```text id="ez8xwg"
Api
  -> Application
      -> Domain

Infrastructure
  -> Application
  -> Domain

Tests
  -> progetti da testare
```

Una regola importante:

> Il dominio non dovrebbe dipendere dall’infrastruttura.

In altre parole, le classi fondamentali del business non dovrebbero conoscere direttamente database, file system, email, API esterne o framework web.

---

## Perché separare i progetti

Separare i progetti ha diversi vantaggi:

* codice più ordinato;
* dipendenze più controllate;
* test più semplici;
* maggiore riutilizzo;
* migliore manutenibilità;
* architettura più chiara;
* possibilità di sostituire parti dell’applicazione.

Non bisogna però esagerare.

Per un progetto piccolo, troppi progetti possono creare complessità inutile.

Una regola pratica:

```text id="mvhjr1"
Progetto piccolo
  pochi progetti, struttura semplice.

Progetto medio/grande
  separazione più esplicita delle responsabilità.

Progetto enterprise
  solution modulare, test, infrastruttura, API, UI separati.
```

# Introduzione a NuGet

**NuGet** è il sistema di gestione dei pacchetti dell’ecosistema .NET.

Un pacchetto NuGet è una libreria riutilizzabile che possiamo aggiungere al nostro progetto.

Esempi di pacchetti molto usati:

```text id="dppwge"
Dapper
Newtonsoft.Json
Serilog
FluentValidation
MediatR
Microsoft.EntityFrameworkCore.SqlServer
Microsoft.EntityFrameworkCore.Sqlite
xunit
Moq
```

NuGet permette di:

* installare librerie;
* gestire versioni;
* aggiornare pacchetti;
* ripristinare dipendenze;
* pubblicare librerie riutilizzabili;
* usare feed pubblici o privati.

---

## Aggiungere un pacchetto da CLI

Esempio:

```bash id="fo6el6"
dotnet add package Newtonsoft.Json
```

Oppure specificando la versione:

```bash id="lf4yrq"
dotnet add package Newtonsoft.Json --version 13.0.3
```

Dopo l’installazione, nel file `.csproj` comparirà qualcosa del genere:

```xml id="fjx5zl"
<ItemGroup>
  <PackageReference Include="Newtonsoft.Json" Version="13.0.3" />
</ItemGroup>
```

---

## Ripristinare i pacchetti

Quando scarichiamo un progetto da Git, normalmente i pacchetti non sono salvati fisicamente nel repository.

Vengono ripristinati con:

```bash id="fxweoc"
dotnet restore
```

In molti casi `dotnet build` esegue automaticamente il restore, ma è importante sapere che questa fase esiste.

---

## Attenzione alle versioni

Le versioni dei pacchetti sono importanti.

Aggiornare un pacchetto può:

* correggere bug;
* chiudere vulnerabilità;
* migliorare prestazioni;
* introdurre breaking changes;
* cambiare comportamenti.

Non bisogna aggiornare “a caso”, soprattutto in progetti di produzione.

# Central Package Management

In una solution grande, lo stesso pacchetto può essere usato da molti progetti.

Esempio:

```text id="x63773"
GestionaleClienti.Api
  Microsoft.EntityFrameworkCore.SqlServer 10.0.0

GestionaleClienti.Infrastructure
  Microsoft.EntityFrameworkCore.SqlServer 10.0.0

GestionaleClienti.Tests
  Microsoft.EntityFrameworkCore.SqlServer 10.0.0
```

Se ogni progetto definisce la propria versione, può diventare difficile mantenere coerenza.

Il **Central Package Management** permette di gestire le versioni dei pacchetti in un file centrale.

Di solito si usa un file:

```text id="z2oi5n"
Directory.Packages.props
```

Esempio:

```xml id="mjcoag"
<Project>
  <ItemGroup>
    <PackageVersion Include="Microsoft.EntityFrameworkCore.SqlServer" Version="10.0.0" />
    <PackageVersion Include="FluentValidation" Version="12.0.0" />
    <PackageVersion Include="Serilog.AspNetCore" Version="9.0.0" />
    <PackageVersion Include="xunit" Version="2.9.3" />
  </ItemGroup>
</Project>
```

Nei singoli `.csproj`, poi, possiamo scrivere:

```xml id="fenki5"
<ItemGroup>
  <PackageReference Include="Microsoft.EntityFrameworkCore.SqlServer" />
</ItemGroup>
```

La versione viene presa dal file centrale.

---

## Perché usare Central Package Management

I vantaggi sono:

* versioni coerenti;
* aggiornamenti più semplici;
* meno duplicazione;
* migliore controllo delle dipendenze;
* solution più pulita.

È particolarmente utile in progetti medio-grandi.

Per un progetto piccolo o didattico può essere eccessivo, ma è bene conoscerlo.

# Git e controllo versione

**Git** è il sistema di controllo versione più usato nello sviluppo moderno.

Serve per tenere traccia della storia del codice.

Con Git possiamo:

* salvare modifiche;
* vedere cosa è cambiato;
* tornare a una versione precedente;
* lavorare su branch;
* collaborare con altri sviluppatori;
* fare merge;
* pubblicare il codice su piattaforme come GitHub, GitLab o Azure DevOps.

---

## Repository

Un repository Git è una cartella controllata da Git.

Per inizializzare un repository:

```bash id="ks7g4l"
git init
```

Per vedere lo stato:

```bash id="vy1lgr"
git status
```

Per aggiungere file:

```bash id="ndjyne"
git add .
```

Per creare un commit:

```bash id="15g1th"
git commit -m "Primo commit"
```

---

## Commit

Un commit è una fotografia del progetto in un certo momento.

Un buon commit dovrebbe essere:

* piccolo;
* coerente;
* descritto bene;
* legato a una modifica precisa.

Esempio di messaggio buono:

```text id="rmlhqj"
Aggiunge validazione cliente
```

Esempio meno utile:

```text id="4xiwg2"
modifiche varie
```

---

## Branch

Un branch permette di lavorare su una modifica senza toccare direttamente il ramo principale.

Esempio:

```bash id="ua37y0"
git switch -c feature/gestione-clienti
```

Quando il lavoro è pronto, il branch può essere unito al ramo principale tramite merge o pull request.

---

## Il file .gitignore

In un progetto .NET non dobbiamo salvare tutto in Git.

Cartelle come:

```text id="8xkb3j"
bin/
obj/
.vs/
```

non dovrebbero essere versionate.

Per escluderle si usa il file:

```text id="opss1q"
.gitignore
```

Per creare un `.gitignore` adatto a progetti Visual Studio/.NET, possiamo usare:

```bash id="fl0v0t"
dotnet new gitignore
```

## Git dentro Visual Studio, VS Code e Rider

Tutti gli strumenti moderni integrano Git.

Visual Studio permette di:

* vedere modifiche;
* fare commit;

* fare push e pull;
* risolvere conflitti.

VS Code offre un’integrazione Git molto leggera e immediata.

Gli strumenti AI stanno diventando parte normale del lavoro dello sviluppatore.

* documentare metodi;
* creare bozze di codice;
* produrre checklist;
* analizzare messaggi di errore.

Tuttavia, non devono essere usati in modo passivo.

---

## AI come assistente, non come sostituto

Un buon modo di usare l’AI è considerarla un assistente tecnico.

Può accelerare il lavoro, ma non deve sostituire:

* comprensione;
* revisione;
* test;
* debugging;
* responsabilità tecnica;
* controllo della sicurezza;
* conoscenza del dominio applicativo.

Il codice generato va sempre letto.

## Esempi di uso corretto

### Spiegare un errore

```text id="mh5tfg"
Questo errore compare durante dotnet build.

```

---

### Chiedere un refactoring

```text id="f8u1lx"
Questo metodo è troppo lungo.

```

```text id="voza7s"

```

### Capire una scelta architetturale

```text id="pgy3nq"

```

## Esempi di uso rischioso

### Accettare codice senza capirlo

```text id="yiwx97"
Scrivimi tutto il progetto e basta.
```

Il problema non è chiedere aiuto. Il problema è non capire ciò che si inserisce nel progetto.

---

### Copiare codice in produzione senza test

Un frammento generato può sembrare corretto, ma avere problemi:

* gestione errata dei null;
* eccezioni non considerate;
* vulnerabilità;
* prestazioni scarse;
* incompatibilità con la versione .NET usata;
* uso sbagliato di async;
* SQL injection;
* gestione errata delle transazioni.

---

### Fornire dati sensibili

Non bisogna incollare in strumenti esterni:

* password;
* connection string reali;
* token;
* dati personali;
* dati sanitari;
* dati finanziari;
* codice proprietario riservato, se non autorizzato;
* log contenenti informazioni sensibili.

# Un primo laboratorio pratico

Vediamo ora un piccolo percorso pratico da terminale.

L’obiettivo è creare:

* una solution;
* una libreria di dominio;
* una console app;
* un progetto di test;
* riferimenti tra progetti.

---

## Creiamo la cartella

```bash id="xf3ty6"
mkdir PrimoProgettoDotNet
cd PrimoProgettoDotNet
```

---

## Creiamo la solution

```bash id="fgf4yg"
dotnet new sln -n PrimoProgettoDotNet
```

---

## Creiamo una libreria

```bash id="k4cspp"
dotnet new classlib -n PrimoProgettoDotNet.Domain
```

---

## Creiamo una console app

```bash id="b2tw9r"
dotnet new console -n PrimoProgettoDotNet.Console
```

---

## Creiamo un progetto di test

```bash id="c04110"
dotnet new xunit -n PrimoProgettoDotNet.Tests
```

---

## Aggiungiamo i progetti alla solution

```bash id="i95qqd"
dotnet sln add PrimoProgettoDotNet.Domain/PrimoProgettoDotNet.Domain.csproj
dotnet sln add PrimoProgettoDotNet.Console/PrimoProgettoDotNet.Console.csproj
dotnet sln add PrimoProgettoDotNet.Tests/PrimoProgettoDotNet.Tests.csproj
```

---

## Aggiungiamo i riferimenti

La console usa il dominio:

```bash id="c87sb0"
dotnet add PrimoProgettoDotNet.Console/PrimoProgettoDotNet.Console.csproj reference PrimoProgettoDotNet.Domain/PrimoProgettoDotNet.Domain.csproj
```

I test usano il dominio:

```bash id="ucepsj"
dotnet add PrimoProgettoDotNet.Tests/PrimoProgettoDotNet.Tests.csproj reference PrimoProgettoDotNet.Domain/PrimoProgettoDotNet.Domain.csproj
```

---

## Compiliamo tutto

```bash id="g6jjiv"
dotnet build
```

---

## Eseguiamo la console

```bash id="pplivv"
dotnet run --project PrimoProgettoDotNet.Console
```

---

## Eseguiamo i test

```bash id="vzg22s"
dotnet test
```

---

## Struttura ottenuta

Alla fine avremo qualcosa del genere:

```text id="w6g65f"
PrimoProgettoDotNet/
  PrimoProgettoDotNet.sln

  PrimoProgettoDotNet.Domain/
    PrimoProgettoDotNet.Domain.csproj
    Class1.cs

  PrimoProgettoDotNet.Console/
    PrimoProgettoDotNet.Console.csproj
    Program.cs

  PrimoProgettoDotNet.Tests/
    PrimoProgettoDotNet.Tests.csproj
    UnitTest1.cs
```

Questa struttura è semplice, ma introduce già concetti importanti:

* una solution può contenere più progetti;
* ogni progetto ha un proprio `.csproj`;
* i progetti possono referenziarsi tra loro;
* il progetto di test può testare una libreria;
* la CLI permette di gestire tutto senza IDE.

# Struttura consigliata per progetti reali

Per progetti più ordinati, possiamo usare una struttura leggermente migliore:

```text id="fq9i6i"
PrimoProgettoDotNet/
  PrimoProgettoDotNet.sln
  Directory.Packages.props
  README.md
  .gitignore

  src/
    PrimoProgettoDotNet.Domain/
    PrimoProgettoDotNet.Application/
    PrimoProgettoDotNet.Infrastructure/
    PrimoProgettoDotNet.Api/

  tests/
    PrimoProgettoDotNet.Domain.Tests/
    PrimoProgettoDotNet.Application.Tests/
    PrimoProgettoDotNet.Api.Tests/
```

Questa organizzazione separa il codice applicativo dai test.

Non è obbligatoria nei progetti piccoli, ma è molto utile nei progetti destinati a crescere.

# Errori comuni all’inizio

## Confondere solution e progetto

La solution contiene progetti.
Il progetto viene compilato.

```text id="kavvvj"
.sln     = contenitore logico
.csproj  = progetto compilabile
```

---

## Installare solo il runtime

Se installiamo solo il runtime, potremmo non riuscire a creare o compilare progetti.

Per sviluppare serve lo SDK.

---

## Non controllare la versione di .NET

Quando un progetto non compila, una delle prime verifiche è:

```bash id="qtc2xq"
dotnet --info
```

---

## Salvare bin e obj in Git

Le cartelle `bin` e `obj` sono generate automaticamente.

Non vanno versionate.

---

## Aggiungere pacchetti senza criterio

Ogni pacchetto è una dipendenza.

Prima di aggiungerne uno, chiediamoci:

* mi serve davvero?
* è mantenuto?
* è compatibile con la mia versione di .NET?
* introduce dipendenze pesanti?
* ha licenza compatibile?
* è sicuro?

---

## Usare l’IDE senza capire la CLI

L’IDE è comodissimo, ma la CLI aiuta a capire.

Un buon sviluppatore .NET dovrebbe saper usare entrambi.

AI

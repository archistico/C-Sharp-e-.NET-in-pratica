# 07.03 — Sincronizzazione e dati condivisi

Nei capitoli precedenti abbiamo visto `async/await`, `Task`, `ThreadPool`, `Task.Run`, `BackgroundService` e hosted services. Ora affrontiamo uno dei temi più delicati della programmazione concorrente: **la sincronizzazione dei dati condivisi**.

Quando più thread o task accedono agli stessi dati nello stesso momento, possono nascere bug difficili da riprodurre.

Il punto centrale è questo:

```text id="uj24i5"
I bug di concorrenza non dipendono solo da cosa fa il codice,
ma anche da quando più thread eseguono certe istruzioni.
```

Questa è la parte che rende il multithreading pericoloso: due esecuzioni dello stesso programma possono produrre risultati diversi.

---

# Il problema dei dati condivisi

Un dato condiviso è una risorsa accessibile da più thread o task.

Esempi:

```text id="dfxuyu"
una variabile statica
una lista condivisa
un dizionario condiviso
una cache in memoria
un contatore globale
un file
una connessione condivisa
uno stato applicativo singleton
una coda in memoria
un oggetto modificabile usato da più servizi
```

Il problema non nasce dalla lettura in sé.

Il problema nasce quando almeno un thread modifica il dato mentre altri thread lo leggono o lo modificano.

---

# Race condition

Una **race condition** si verifica quando il risultato dipende dall’ordine temporale con cui più thread accedono a dati condivisi.

Esempio apparentemente innocuo:

```csharp id="f5fzxo"
public class Contatore
{
    private int valore;

    public void Incrementa()
    {
        valore++;
    }

    public int Valore => valore;
}
```

`valore++` sembra una sola operazione, ma concettualmente equivale a:

```text id="v9aewh"
1. leggi valore
2. aggiungi 1
3. scrivi nuovo valore
```

Se due thread eseguono `Incrementa` nello stesso momento:

```text id="p75oo8"
valore iniziale = 0

Thread A legge 0
Thread B legge 0

Thread A scrive 1
Thread B scrive 1

Risultato finale = 1
```

Ma ci aspettavamo 2.

Questo è un bug di concorrenza.

---

## Esempio riproducibile

```csharp id="bbv67h"
int contatore = 0;

List<Task> tasks = [];

for (int i = 0; i < 10; i++)
{
    tasks.Add(Task.Run(() =>
    {
        for (int j = 0; j < 100_000; j++)
        {
            contatore++;
        }
    }));
}

await Task.WhenAll(tasks);

Console.WriteLine(contatore);
```

Ci aspetteremmo:

```text id="juw98n"
1.000.000
```

Ma spesso otterremo un valore minore.

Perché?

Perché più thread aggiornano `contatore` senza sincronizzazione.

---

# Thread safety

Un codice è **thread-safe** quando può essere usato contemporaneamente da più thread senza corrompere lo stato interno e senza produrre risultati incoerenti.

Esempi di codice non thread-safe:

```csharp id="hm5qqq"
private List<string> nomi = [];

public void Aggiungi(string nome)
{
    nomi.Add(nome);
}
```

`List<T>` non è progettata per scritture concorrenti.

Se più thread chiamano `Aggiungi` contemporaneamente, possono verificarsi:

```text id="tedw1g"
dati persi
eccezioni
stato interno corrotto
risultati non deterministici
```

---

## Rendere thread-safe non significa solo aggiungere lock

Possibili strategie:

```text id="1popum"
evitare stato condiviso
rendere gli oggetti immutabili
usare lock
usare operazioni atomiche
usare collezioni concorrenti
usare code
usare attori/worker singoli
usare database come fonte di verità
usare transazioni
```

La sincronizzazione è una soluzione.
Ma spesso la soluzione migliore è ridurre o eliminare la condivisione.

---

# `lock`

La keyword `lock` permette di proteggere una sezione critica.

Una sezione critica è un blocco di codice che deve essere eseguito da un solo thread alla volta.

Esempio:

```csharp id="nh9818"
public class ContatoreThreadSafe
{
    private readonly object sync = new();
    private int valore;

    public void Incrementa()
    {
        lock (sync)
        {
            valore++;
        }
    }

    public int Valore
    {
        get
        {
            lock (sync)
            {
                return valore;
            }
        }
    }
}
```

Ora l’accesso a `valore` è protetto.

La documentazione Microsoft raccomanda di usare `lock` per sincronizzare l’accesso a risorse condivise e, nelle versioni precedenti a .NET 9/C# 13, di bloccare un oggetto dedicato non usato per altri scopi. Da .NET 9 e C# 13 è consigliato usare un’istanza dedicata di `System.Threading.Lock` per ottenere prestazioni migliori. ([Microsoft Learn][1])

---

## Su cosa fare lock

Buono:

```csharp id="3pk4hp"
private readonly object sync = new();

lock (sync)
{
    // sezione critica
}
```

Sconsigliato:

```csharp id="wjrtju"
lock (this)
{
}
```

Sconsigliato:

```csharp id="diyft4"
lock (typeof(ClienteService))
{
}
```

Sconsigliato:

```csharp id="x6hgrw"
lock ("clienti")
{
}
```

Perché?

```text id="f5rmir"
this può essere usato anche da codice esterno
typeof(...) è condiviso a livello globale per quel tipo
le stringhe possono essere internate e condivise
```

Regola pratica:

```text id="dy2lr7"
Fai lock solo su un oggetto privato, dedicato e non esposto.
```

---

# Nuovo tipo `System.Threading.Lock`

Da .NET 9 e C# 13 è disponibile un tipo dedicato:

```csharp id="rdkugv"
System.Threading.Lock
```

Esempio:

```csharp id="477lsm"
using System.Threading;

public class ContatoreThreadSafe
{
    private readonly Lock sync = new();
    private int valore;

    public void Incrementa()
    {
        lock (sync)
        {
            valore++;
        }
    }

    public int Valore
    {
        get
        {
            lock (sync)
            {
                return valore;
            }
        }
    }
}
```

C# riconosce questo tipo e usa la sua API di locking in modo specifico.

La documentazione di C# 13 indica che .NET 9 introduce `System.Threading.Lock`, un nuovo tipo per la sincronizzazione; il metodo `EnterScope()` entra in uno scope esclusivo e restituisce un `ref struct` che supporta il pattern `Dispose` per uscire dallo scope. ([Microsoft Learn][2])

Forma esplicita:

```csharp id="lz1lvx"
private readonly Lock sync = new();

public void Incrementa()
{
    using (sync.EnterScope())
    {
        valore++;
    }
}
```

Per il libro puoi presentare entrambe le forme:

```text id="almqzo"
C# moderno / .NET 9+
  System.Threading.Lock

C# precedente
  object privato dedicato
```

---

# Cosa non fare dentro un lock

Dentro un `lock` bisogna tenere il codice breve.

Sconsigliato:

```csharp id="ny9gib"
lock (sync)
{
    await httpClient.GetStringAsync(url);
}
```

In realtà non puoi usare `await` direttamente dentro un blocco `lock`: il compilatore lo impedisce. Ma il problema concettuale resta: non bisogna tenere una lock durante operazioni lente o asincrone.

Sconsigliato:

```csharp id="kyttzu"
lock (sync)
{
    Thread.Sleep(5000);
}
```

Sconsigliato:

```csharp id="kdoexv"
lock (sync)
{
    // chiamata HTTP
    // accesso database
    // invio email
    // operazione lenta
}
```

Regola pratica:

```text id="32k06q"
Dentro un lock:
  poco codice
  nessuna operazione lenta
  nessuna chiamata esterna non controllata
  nessuna attesa asincrona
```

# `Monitor`

`lock` è una sintassi più comoda costruita sopra `Monitor`.

Questo codice:

```csharp id="7dtyi3"
lock (sync)
{
    valore++;
}
```

è concettualmente simile a:

```csharp id="wgr8i2"
Monitor.Enter(sync);

try
{
    valore++;
}
finally
{
    Monitor.Exit(sync);
}
```

Microsoft spiega che l’istruzione `lock` in C# e `SyncLock` in Visual Basic sono implementate usando `Monitor.Enter` e `Monitor.Exit`, con un blocco `try/finally` per garantire il rilascio della lock. ([Microsoft Learn][3])

---

## Quando usare Monitor direttamente

Raramente.

Può servire se vuoi:

```text id="f6jx6u"
tentare di acquisire una lock con timeout
usare Wait/Pulse
gestire scenari avanzati
```

Esempio con timeout:

```csharp id="e9o9mv"
bool lockTaken = false;

try
{
    Monitor.TryEnter(sync, TimeSpan.FromSeconds(2), ref lockTaken);

    if (!lockTaken)
    {
        throw new TimeoutException("Impossibile acquisire la lock.");
    }

    // sezione critica
}
finally
{
    if (lockTaken)
    {
        Monitor.Exit(sync);
    }
}
```

Nel codice ordinario, usa `lock`.

---

# `Interlocked`

`Interlocked` fornisce operazioni atomiche su variabili semplici.

È ideale per contatori e aggiornamenti molto piccoli.

Esempio:

```csharp id="kng5y3"
private int contatore;

public void Incrementa()
{
    Interlocked.Increment(ref contatore);
}
```

Questo evita il problema di `contatore++`.

Altri metodi utili:

```csharp id="6g0wgr"
Interlocked.Decrement(ref contatore);
Interlocked.Add(ref contatore, 10);
Interlocked.Exchange(ref valore, nuovoValore);
Interlocked.CompareExchange(ref valore, nuovoValore, valoreAtteso);
```

---

## Esempio confronto

Non thread-safe:

```csharp id="8wpwfr"
contatore++;
```

Thread-safe:

```csharp id="san2ss"
Interlocked.Increment(ref contatore);
```

`Interlocked` è spesso più efficiente di `lock` per singole operazioni atomiche.

---

## Quando usare Interlocked

Usalo per:

```text id="7df9im"
contatori
flag numerici
assegnazioni atomiche
incrementi/decrementi
statistiche leggere
```

Non usarlo per proteggere strutture complesse.

Sbagliato:

```text id="lc5gjl"
uso Interlocked per coordinare modifiche a List<T>, Dictionary<TKey,TValue>,
oggetti complessi o invarianti multiple
```

Se più campi devono cambiare insieme, serve una strategia più ampia.

---

# `volatile`

`volatile` indica che un campo può essere modificato da più thread e che certe ottimizzazioni di caching/riordinamento devono essere limitate.

Esempio:

```csharp id="ecgoil"
private volatile bool shouldStop;

public void Stop()
{
    shouldStop = true;
}

public void Work()
{
    while (!shouldStop)
    {
        // lavoro
    }
}
```

Attenzione: `volatile` non rende automaticamente thread-safe qualsiasi operazione.

Questo resta non sicuro:

```csharp id="cw68tk"
private volatile int contatore;

public void Incrementa()
{
    contatore++;
}
```

Perché `++` non diventa atomico.

---

## Quando usare volatile

Raramente e con attenzione.

Può servire per:

```text id="967f0g"
flag semplici letti/scritti da più thread
scenari basso livello
loop che osservano una variabile di stop
```

Spesso è meglio usare:

```text id="hg9ffe"
CancellationToken
lock
Interlocked
Channel<T>
costrutti di sincronizzazione più espliciti
```

Regola pratica:

```text id="5owy8a"
volatile migliora la visibilità di letture/scritture semplici,
ma non sostituisce lock, Interlocked o una progettazione thread-safe.
```

# `SemaphoreSlim`

`SemaphoreSlim` limita quante operazioni possono entrare contemporaneamente in una sezione.

A differenza di `lock`, supporta l’attesa asincrona con `WaitAsync`.

Esempio: massimo 3 operazioni contemporanee.

```csharp id="699uh7"
private readonly SemaphoreSlim semaphore = new(3);

public async Task EseguiAsync(CancellationToken cancellationToken)
{
    await semaphore.WaitAsync(cancellationToken);

    try
    {
        await OperazioneAsync(cancellationToken);
    }
    finally
    {
        semaphore.Release();
    }
}
```

Questo è utile quando vuoi limitare concorrenza verso:

```text id="lq9n8w"
API esterne
database
file system
servizio lento
risorsa limitata
upload/download
```

---

## `SemaphoreSlim` come lock asincrona

Se inizializzato a 1:

```csharp id="9tjujf"
private readonly SemaphoreSlim semaphore = new(1, 1);
```

si comporta come una lock asincrona.

```csharp id="jgpkun"
await semaphore.WaitAsync(cancellationToken);

try
{
    await OperazioneCriticaAsync(cancellationToken);
}
finally
{
    semaphore.Release();
}
```

Questo è possibile perché `SemaphoreSlim` consente `await`, a differenza di `lock`.

---

## Attenzione a Release

Errore grave:

```csharp id="d1gmqo"
await semaphore.WaitAsync();

// eccezione qui
semaphore.Release();
```

Se arriva un’eccezione, `Release` non viene chiamato.

Sempre:

```csharp id="9nhbxu"
await semaphore.WaitAsync();

try
{
    // sezione protetta
}
finally
{
    semaphore.Release();
}
```

---

# `Mutex`

`Mutex` fornisce accesso esclusivo a una risorsa e può funzionare anche tra processi diversi.

Microsoft descrive `Mutex` come simile a `Monitor` per l’accesso esclusivo, ma con thread affinity: il thread che acquisisce un mutex deve rilasciarlo chiamando `ReleaseMutex`. ([Microsoft Learn][3])

Esempio: impedire più istanze dell’applicazione.

```csharp id="a7pc9c"
using Mutex mutex = new(
    initiallyOwned: true,
    name: "Global\\MiaApplicazioneUnica",
    createdNew: out bool createdNew);

if (!createdNew)
{
    Console.WriteLine("Applicazione già in esecuzione.");
    return;
}

Console.WriteLine("Applicazione avviata.");
Console.ReadLine();
```

---

## Quando usare Mutex

Usalo quando ti serve sincronizzare:

```text id="45e6ze"
più processi
più istanze dell'app
risorse condivise a livello macchina/sistema
```

Per sincronizzazione interna allo stesso processo, spesso `lock`, `SemaphoreSlim`, `Interlocked` o collezioni concorrenti sono preferibili.

---

# `ReaderWriterLockSlim`

`ReaderWriterLockSlim` permette:

```text id="5c9bc8"
più lettori contemporanei
un solo scrittore esclusivo
```

È utile quando:

```text id="m3844e"
le letture sono molte
le scritture sono poche
la risorsa supporta letture concorrenti
le scritture devono essere esclusive
```

Microsoft descrive `ReaderWriterLockSlim` come una lock per gestire accesso a una risorsa consentendo più thread in lettura o accesso esclusivo in scrittura. ([Microsoft Learn][3])

---

## Esempio

```csharp id="nppf2t"
public class ClienteCache
{
    private readonly ReaderWriterLockSlim rwLock = new();
    private readonly Dictionary<int, ClienteDto> clienti = new();

    public ClienteDto? Trova(int id)
    {
        rwLock.EnterReadLock();

        try
        {
            clienti.TryGetValue(id, out ClienteDto? cliente);
            return cliente;
        }
        finally
        {
            rwLock.ExitReadLock();
        }
    }

    public void Salva(ClienteDto cliente)
    {
        rwLock.EnterWriteLock();

        try
        {
            clienti[cliente.Id] = cliente;
        }
        finally
        {
            rwLock.ExitWriteLock();
        }
    }
}
```

---

## Quando non usarlo

Non usarlo automaticamente al posto di `lock`.

Per risorse semplici o scritture frequenti, può essere più complesso del necessario.

Scegli `ReaderWriterLockSlim` solo se:

```text id="nk1u2o"
hai misurato o previsto molte letture concorrenti
le scritture sono rare
la complessità aggiuntiva è giustificata
```

# Deadlock

Un **deadlock** si verifica quando due o più thread aspettano risorse che non verranno mai liberate.

Esempio classico:

```csharp id="kftnb3"
private readonly object lockA = new();
private readonly object lockB = new();

public void Metodo1()
{
    lock (lockA)
    {
        Thread.Sleep(100);

        lock (lockB)
        {
        }
    }
}

public void Metodo2()
{
    lock (lockB)
    {
        Thread.Sleep(100);

        lock (lockA)
        {
        }
    }
}
```

Possibile scenario:

```text id="8jd2qk"
Thread 1 acquisisce lockA
Thread 2 acquisisce lockB

Thread 1 aspetta lockB
Thread 2 aspetta lockA

Nessuno può proseguire
```

---

## Come evitare deadlock

Strategie:

```text id="dgbf7h"
acquisire lock sempre nello stesso ordine
ridurre il numero di lock
tenere le lock per poco tempo
non chiamare codice esterno dentro lock
evitare lock annidate
usare timeout quando appropriato
preferire strutture concorrenti o code
```

Esempio ordine fisso:

```csharp id="02tx1m"
lock (lockA)
{
    lock (lockB)
    {
        // sempre A poi B
    }
}
```

Tutto il codice deve rispettare lo stesso ordine.

---

# Deadlock async

Anche con `async/await` possono nascere deadlock logici.

Esempio tipico già visto:

```csharp id="gx692z"
var risultato = MetodoAsync().Result;
```

oppure:

```csharp id="ufgms7"
MetodoAsync().Wait();
```

In ambienti con `SynchronizationContext`, come UI desktop o ASP.NET classico, questo può bloccare il thread che la continuazione sta cercando di riusare.

Regola pratica:

```text id="uvtc8u"
Non bloccare codice async con Result o Wait.
Usa await fino al punto di ingresso.
```

---

# Starvation

La **starvation** si verifica quando un thread o un’attività non riesce mai, o quasi mai, a ottenere la risorsa di cui ha bisogno.

Esempi:

```text id="65fkur"
una lock è sempre occupata da altri
lettori continui impediscono allo scrittore di procedere
troppe task bloccanti saturano il ThreadPool
una coda viene riempita più velocemente di quanto venga svuotata
```

Esempio problematico:

```csharp id="zpm7ko"
Parallel.For(0, 1000, i =>
{
    Thread.Sleep(10000);
});
```

Molti worker thread restano bloccati, riducendo la capacità del ThreadPool di servire altro lavoro.

---

## ThreadPool starvation

Può accadere quando molti thread del ThreadPool vengono bloccati.

Cause comuni:

```text id="26o3qb"
Thread.Sleep
.Result
.Wait()
lock tenute troppo a lungo
I/O sincrono in server web
lavori CPU-bound eccessivi dentro ASP.NET Core
```

Sintomi:

```text id="5fznn3"
latenza alta
request lente
timeout
CPU non necessariamente al 100%
molte operazioni in attesa
```

# Strategie per ridurre la condivisione

La migliore sincronizzazione è spesso non dover sincronizzare.

---

## 1. Immutabilità

Gli oggetti immutabili possono essere condivisi più facilmente.

```csharp id="8yfyhb"
public record ClienteDto(
    int Id,
    string Nome,
    string? Email);
```

Se un oggetto non cambia, più thread possono leggerlo senza lock.

---

## 2. Variabili locali

Le variabili locali sono naturalmente isolate nel contesto del metodo.

```csharp id="wmo1wb"
public decimal CalcolaTotale(IEnumerable<Riga> righe)
{
    decimal totale = 0m;

    foreach (Riga riga in righe)
    {
        totale += riga.Importo;
    }

    return totale;
}
```

`totale` non è condiviso tra thread.

---

## 3. Scope per richiesta

In ASP.NET Core, molti servizi sono `Scoped`.

Questo significa:

```text id="aiixna"
ogni request riceve la propria istanza
```

Meno stato condiviso tra utenti.

Attenzione però ai singleton.

```csharp id="41exhj"
builder.Services.AddSingleton<AppState>();
```

Se `AppState` contiene dati modificabili per utente, è pericoloso.

---

## 4. Code e worker singolo

Invece di far modificare uno stato da molti thread, possiamo accodare comandi e farli processare da un solo worker.

Schema:

```text id="fwsjjm"
molti producer
  scrivono in una coda

un consumer
  aggiorna lo stato
```

Questo riduce la necessità di lock.

---

## 5. Collezioni concorrenti

Per alcune strutture dati, usare collezioni progettate per concorrenza.

Esempi:

```text id="des1sw"
ConcurrentDictionary<TKey,TValue>
ConcurrentQueue<T>
ConcurrentStack<T>
ConcurrentBag<T>
BlockingCollection<T>
Channel<T>
```

Le vedremo meglio nel capitolo successivo.

---

## 6. Database e transazioni

Per dati importanti, spesso la fonte di verità deve essere il database.

Esempio:

```text id="q6e4xv"
saldo
ordine
movimento contabile
prenotazione
stato pagamento
```

Qui non basta una lock in memoria se l’app gira su più istanze.

Servono:

```text id="11ar7i"
transazioni
vincoli
concorrenza ottimistica
lock database
idempotenza
outbox
```

---

# Esempio completo: cache thread-safe semplice

```csharp id="uvmz43"
public class ClienteCache
{
    private readonly Lock sync = new();
    private readonly Dictionary<int, ClienteDto> clienti = new();

    public ClienteDto? Trova(int id)
    {
        lock (sync)
        {
            clienti.TryGetValue(id, out ClienteDto? cliente);
            return cliente;
        }
    }

    public void Salva(ClienteDto cliente)
    {
        lock (sync)
        {
            clienti[cliente.Id] = cliente;
        }
    }

    public bool Rimuovi(int id)
    {
        lock (sync)
        {
            return clienti.Remove(id);
        }
    }

    public List<ClienteDto> Lista()
    {
        lock (sync)
        {
            return clienti.Values.ToList();
        }
    }
}
```

Nota importante:

```text id="bgk8es"
Lista restituisce una copia.
Non restituisce direttamente la collezione interna.
```

Se restituissimo `clienti.Values` direttamente, il chiamante potrebbe enumerare mentre un altro thread modifica il dizionario.

---

# Versione con ConcurrentDictionary

```csharp id="njel93"
public class ClienteCache
{
    private readonly ConcurrentDictionary<int, ClienteDto> clienti = new();

    public ClienteDto? Trova(int id)
    {
        clienti.TryGetValue(id, out ClienteDto? cliente);
        return cliente;
    }

    public void Salva(ClienteDto cliente)
    {
        clienti[cliente.Id] = cliente;
    }

    public bool Rimuovi(int id)
    {
        return clienti.TryRemove(id, out _);
    }

    public List<ClienteDto> Lista()
    {
        return clienti.Values.ToList();
    }
}
```

Questa versione è più semplice per operazioni indipendenti.

Ma attenzione:

```text id="r5b2xx"
ConcurrentDictionary protegge la struttura del dizionario,
non rende automaticamente atomica qualunque logica composta intorno ad essa.
```

Esempio problematico:

```csharp id="fzp3kt"
if (!clienti.ContainsKey(id))
{
    clienti[id] = cliente;
}
```

Meglio usare metodi atomici:

```csharp id="1wepq5"
clienti.TryAdd(id, cliente);
```

# Esempio: limitare chiamate concorrenti a una API esterna

```csharp id="ecags9"
public class ApiEsternaClient
{
    private readonly HttpClient httpClient;
    private readonly SemaphoreSlim semaphore = new(5, 5);

    public ApiEsternaClient(HttpClient httpClient)
    {
        this.httpClient = httpClient;
    }

    public async Task<string> ScaricaAsync(
        string url,
        CancellationToken cancellationToken)
    {
        await semaphore.WaitAsync(cancellationToken);

        try
        {
            return await httpClient.GetStringAsync(
                url,
                cancellationToken);
        }
        finally
        {
            semaphore.Release();
        }
    }
}
```

Questo codice permette massimo 5 chiamate concorrenti.

È utile quando un servizio esterno non deve essere sovraccaricato.

# Errori comuni

## Pensare che `++` sia thread-safe

Sbagliato:

```csharp id="kux9ug"
contatore++;
```

se più thread aggiornano lo stesso contatore.

Meglio:

```csharp id="gfad9m"
Interlocked.Increment(ref contatore);
```

---

## Fare lock su oggetti pubblici

Sbagliato:

```csharp id="6wrwl3"
lock (this)
{
}
```

Meglio:

```csharp id="mgmln9"
private readonly object sync = new();

lock (sync)
{
}
```

oppure, in .NET 9+:

```csharp id="bqnjeg"
private readonly Lock sync = new();

lock (sync)
{
}
```

---

## Tenere lock troppo a lungo

Sbagliato:

```csharp id="3j9f33"
lock (sync)
{
    // chiamata HTTP lenta
}
```

Meglio preparare dati fuori dalla lock e proteggere solo la modifica dello stato condiviso.

---

## Usare `lock` con `await`

Non si può usare `await` dentro `lock`.

Per scenari asincroni, valuta `SemaphoreSlim`.

---

## Usare volatile come soluzione generale

Sbagliato:

```csharp id="9r3wgp"
private volatile int contatore;

contatore++;
```

`volatile` non rende atomico l’incremento.

---

## Dimenticare `finally`

Sbagliato:

```csharp id="zubmvl"
await semaphore.WaitAsync();

await OperazioneAsync();

semaphore.Release();
```

Se `OperazioneAsync` fallisce, il semaforo non viene rilasciato.

Sempre:

```csharp id="8mivjv"
try
{
}
finally
{
    semaphore.Release();
}
```

---

## Usare lock in memoria per coordinare più server

Se l’app è distribuita su più istanze:

```text id="enobid"
Server A ha la sua lock
Server B ha un'altra lock
```

Una lock in memoria non coordina processi diversi.

Servono:

```text id="ci0w1d"
database
lock distribuite
code
Redis
transazioni
vincoli univoci
idempotenza
```

# Linee guida pratiche

```text id="aznq9b"
Evita dati condivisi
  quando puoi.

Preferisci immutabilità
  per dati letti da più thread.

Usa lock
  per proteggere sezioni critiche brevi.

In .NET 9+ valuta System.Threading.Lock
  come oggetto dedicato di lock.

Usa Interlocked
  per contatori e operazioni atomiche semplici.

Usa volatile raramente
  e solo per visibilità di flag semplici.

Usa SemaphoreSlim
  per limitare concorrenza e per scenari async.

Usa Mutex
  per sincronizzazione tra processi.

Usa ReaderWriterLockSlim
  quando hai molte letture e poche scritture.

Evita lock annidate
  o imponi un ordine fisso.

Non bloccare ThreadPool
  con Sleep, Wait o Result.

Non usare lock in memoria
  per coordinare applicazioni distribuite.

Misura sempre
  prima di introdurre primitive complesse.
```

Abbiamo chiarito che `lock` è il costrutto più comune per proteggere sezioni critiche brevi e che, da .NET 9/C# 13, Microsoft consiglia di usare un’istanza dedicata di `System.Threading.Lock` per ottenere prestazioni migliori. ([Microsoft Learn][1])

Abbiamo visto che `Monitor` è il meccanismo sottostante tradizionale di `lock`, mentre `Mutex`, `Semaphore`, `ReaderWriterLockSlim` e le altre primitive servono a scenari più specifici. ([Microsoft Learn][3])

La regola principale è questa:

```text id="xsg0sh"
La sincronizzazione migliore è quella che non devi fare.
Prima riduci la condivisione; poi scegli la primitiva più semplice
che protegge correttamente lo stato rimasto condiviso.
```

Questo capitolo prepara il prossimo passo naturale: collezioni concorrenti, producer/consumer, `Channel<T>`, pipeline asincrone e pattern per elaborare dati in modo sicuro senza riempire il codice di lock.

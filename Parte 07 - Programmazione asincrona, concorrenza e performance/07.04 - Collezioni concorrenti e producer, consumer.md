# 07.04 — Collezioni concorrenti e producer/consumer

Nei capitoli precedenti abbiamo visto `async/await`, `Task`, `ThreadPool`, `BackgroundService`, race condition, `lock`, `Interlocked`, `SemaphoreSlim`, `Mutex` e `ReaderWriterLockSlim`.

Ora passiamo alla parte più applicativa della concorrenza: **come scambiare dati tra più attività concorrenti senza riempire il codice di lock manuali**.

Il punto centrale è questo:

```text id="nr7j90"
Le collezioni concorrenti e le code producer/consumer permettono
di coordinare più thread o task in modo più sicuro rispetto
alla gestione manuale di List<T>, Queue<T> e Dictionary<TKey,TValue>.
```

Le collezioni in `System.Collections.Concurrent` sono progettate per essere thread-safe e scalabili: più thread possono aggiungere o rimuovere elementi senza richiedere sincronizzazione aggiuntiva nel codice utente. Microsoft consiglia le collezioni concorrenti quando più thread scrivono sulla stessa collezione; se invece la collezione è solo letta, possono bastare le collezioni generiche normali. ([Microsoft Learn][1])

---

# Il problema: collezioni normali e accesso concorrente

Questa classe sembra innocente:

```csharp id="g8wf3f"
public class CodaMessaggi
{
    private readonly Queue<string> messaggi = new();

    public void Aggiungi(string messaggio)
    {
        messaggi.Enqueue(messaggio);
    }

    public string? Estrai()
    {
        if (messaggi.Count == 0)
        {
            return null;
        }

        return messaggi.Dequeue();
    }
}
```

Ma non è thread-safe.

Se più thread chiamano `Aggiungi` ed `Estrai` contemporaneamente, possono verificarsi:

```text id="podxfr"
eccezioni
dati persi
stato interno corrotto
letture incoerenti
comportamenti non riproducibili
```

Potremmo aggiungere `lock`, ma spesso .NET offre già strutture progettate per questi scenari.

---

# Panoramica delle collezioni concorrenti

Le collezioni più importanti sono:

```text id="lpq9ny"
ConcurrentDictionary<TKey,TValue>
  dizionario thread-safe.

ConcurrentQueue<T>
  coda FIFO thread-safe.

ConcurrentStack<T>
  pila LIFO thread-safe.

ConcurrentBag<T>
  contenitore non ordinato thread-safe.

BlockingCollection<T>
  wrapper producer/consumer con blocking e bounding.

Channel<T>
  coda asincrona moderna per pipeline e background processing.
```

Scelta rapida:

```text id="d5wbl7"
Devo associare chiave → valore?
  ConcurrentDictionary

Devo processare in ordine di arrivo?
  ConcurrentQueue o Channel

Devo processare ultimo arrivato, primo servito?
  ConcurrentStack

Mi serve un contenitore non ordinato per lavoro parallelo?
  ConcurrentBag

Mi serve producer/consumer sincrono con blocking?
  BlockingCollection

Mi serve producer/consumer asincrono moderno?
  Channel<T>
```

# `ConcurrentDictionary<TKey,TValue>`

`ConcurrentDictionary<TKey,TValue>` è un dizionario thread-safe.

È utile per:

```text id="c1o7va"
cache condivise
lookup concorrenti
contatori per chiave
stato per utente/connessione
registri in memoria
deduplicazione
```

Esempio:

```csharp id="h2qccx"
using System.Collections.Concurrent;

ConcurrentDictionary<int, ClienteDto> clienti = new();

clienti.TryAdd(1, new ClienteDto(1, "Emilie", "emilie@example.com"));

if (clienti.TryGetValue(1, out ClienteDto? cliente))
{
    Console.WriteLine(cliente.Nome);
}
```

---

## Metodi importanti

```csharp id="k5c3c7"
TryAdd
TryGetValue
TryRemove
TryUpdate
GetOrAdd
AddOrUpdate
ContainsKey
```

---

## `GetOrAdd`

```csharp id="b2spcv"
ClienteDto cliente = clienti.GetOrAdd(
    10,
    id => new ClienteDto(id, "Nuovo cliente", null));
```

Se la chiave esiste, restituisce il valore esistente.
Se non esiste, crea e inserisce il nuovo valore.

---

## `AddOrUpdate`

```csharp id="fmorqj"
clienti.AddOrUpdate(
    10,
    id => new ClienteDto(id, "Emilie", "emilie@example.com"),
    (id, existing) => existing with
    {
        Email = "nuova@example.com"
    });
```

Se la chiave non esiste, aggiunge.
Se esiste, aggiorna.

---

## Attenzione alla logica composta

Questo è thread-safe come singole operazioni, ma non come logica complessiva:

```csharp id="mde5wb"
if (!clienti.ContainsKey(id))
{
    clienti[id] = cliente;
}
```

Tra `ContainsKey` e l’assegnazione, un altro thread potrebbe inserire la stessa chiave.

Meglio:

```csharp id="qtr9aq"
bool aggiunto = clienti.TryAdd(id, cliente);
```

Oppure:

```csharp id="al0gnt"
ClienteDto clienteFinale = clienti.GetOrAdd(id, cliente);
```

Regola pratica:

```text id="jvibm4"
Usa i metodi atomici della collezione concorrente.
Non ricostruire manualmente sequenze check-then-act.
```

---

## Esempio: contatori per chiave

```csharp id="x3vq01"
ConcurrentDictionary<string, int> contatori = new();

void Incrementa(string chiave)
{
    contatori.AddOrUpdate(
        chiave,
        addValue: 1,
        updateValueFactory: (_, valoreCorrente) => valoreCorrente + 1);
}
```

Questo è utile per statistiche semplici, ma se i contatori sono molto intensivi si può valutare anche `Interlocked` su oggetti dedicati.

# `ConcurrentQueue<T>`

`ConcurrentQueue<T>` è una coda thread-safe FIFO.

FIFO significa:

```text id="xwqtl8"
First In, First Out
il primo elemento inserito è il primo estratto
```

Esempio:

```csharp id="rm4vpd"
using System.Collections.Concurrent;

ConcurrentQueue<string> queue = new();

queue.Enqueue("Messaggio 1");
queue.Enqueue("Messaggio 2");

if (queue.TryDequeue(out string? messaggio))
{
    Console.WriteLine(messaggio);
}
```

Output:

```text id="z7sbe8"
Messaggio 1
```

---

## Quando usare ConcurrentQueue

Usala quando:

```text id="n0v38r"
più thread producono elementi
più thread consumano elementi
l'ordine di arrivo è importante
non ti serve await sul consumo
non ti serve backpressure integrata
```

Limite importante:

```text id="acl72x"
ConcurrentQueue non blocca e non attende.
Se è vuota, TryDequeue restituisce false.
```

Per code asincrone applicative, spesso `Channel<T>` è più adatto.

---

# `ConcurrentStack<T>`

`ConcurrentStack<T>` è una pila thread-safe LIFO.

LIFO significa:

```text id="yqf3as"
Last In, First Out
l'ultimo elemento inserito è il primo estratto
```

Esempio:

```csharp id="6c5p9h"
using System.Collections.Concurrent;

ConcurrentStack<string> stack = new();

stack.Push("A");
stack.Push("B");

if (stack.TryPop(out string? value))
{
    Console.WriteLine(value);
}
```

Output:

```text id="h00upm"
B
```

---

## Quando usare ConcurrentStack

Usala quando:

```text id="tdsi0a"
l'ultimo lavoro inserito ha priorità
stai modellando una pila
vuoi comportamento LIFO
```

È meno comune di `ConcurrentDictionary` e `ConcurrentQueue`, ma utile in alcuni algoritmi concorrenti.

---

# `ConcurrentBag<T>`

`ConcurrentBag<T>` è una collezione thread-safe non ordinata.

Esempio:

```csharp id="wm04kz"
using System.Collections.Concurrent;

ConcurrentBag<int> bag = new();

bag.Add(1);
bag.Add(2);
bag.Add(3);

if (bag.TryTake(out int value))
{
    Console.WriteLine(value);
}
```

Non devi assumere un ordine preciso.

---

## Quando usare ConcurrentBag

Usala quando:

```text id="cn85vg"
l'ordine non importa
più task producono risultati
vuoi raccogliere risultati da elaborazioni parallele
```

Esempio:

```csharp id="1f4cn0"
ConcurrentBag<int> risultati = new();

await Parallel.ForEachAsync(
    Enumerable.Range(1, 100),
    async (numero, cancellationToken) =>
    {
        await Task.Delay(10, cancellationToken);

        risultati.Add(numero * numero);
    });
```

---

## Quando non usarla

Non usare `ConcurrentBag` se ti serve:

```text id="vgca09"
ordine FIFO
ordine LIFO
priorità
paginazione ordinata
sequenza stabile
```

Il nome “bag” indica proprio un sacco: metti dentro e prendi fuori, senza garanzia di ordine semantico.

# `BlockingCollection<T>`

`BlockingCollection<T>` è una collezione pensata per il pattern producer/consumer.

Offre:

```text id="ej51t3"
aggiunta concorrente
rimozione concorrente
blocking quando vuota
capacità massima opzionale
CompleteAdding
integrazione con CancellationToken
```

Microsoft descrive `BlockingCollection<T>` come una classe thread-safe che implementa il pattern producer/consumer, supporta più produttori e consumatori, capacità massima opzionale, operazioni bloccanti e operazioni try con timeout. ([Microsoft Learn][2])

---

## Esempio base

```csharp id="a8ke9p"
using System.Collections.Concurrent;

BlockingCollection<string> queue = new(
    boundedCapacity: 100);

Task producer = Task.Run(() =>
{
    for (int i = 0; i < 10; i++)
    {
        queue.Add($"Messaggio {i}");
    }

    queue.CompleteAdding();
});

Task consumer = Task.Run(() =>
{
    foreach (string item in queue.GetConsumingEnumerable())
    {
        Console.WriteLine(item);
    }
});

await Task.WhenAll(producer, consumer);
```

Qui:

```text id="ak6ng5"
producer
  aggiunge elementi

consumer
  consuma elementi

CompleteAdding
  segnala che non arriveranno altri elementi

GetConsumingEnumerable
  legge fino a completamento
```

---

## Blocking e bounding

`BlockingCollection<T>` può essere bounded, cioè con capacità massima.

```csharp id="aguvz7"
BlockingCollection<string> queue = new(
    boundedCapacity: 100);
```

Se la coda raggiunge 100 elementi, i produttori vengono bloccati finché i consumatori non liberano spazio.

Questo è importante perché impedisce ai produttori di correre troppo avanti e consumare memoria senza limiti. Microsoft evidenzia che il bounding controlla la dimensione massima della collezione in memoria e impedisce ai produttori di avanzare troppo rispetto ai consumatori. ([Microsoft Learn][3])

---

## Limite di BlockingCollection

`BlockingCollection<T>` è molto utile in codice sincrono o TPL classico, ma nelle applicazioni moderne async-first spesso `Channel<T>` è più naturale.

Perché?

```text id="7aqp7l"
BlockingCollection blocca thread.
Channel<T> permette attesa asincrona.
```

In ASP.NET Core e Worker Service moderni, spesso conviene partire da `Channel<T>`.

---

# `Channel<T>`

`Channel<T>` è una struttura moderna per code asincrone producer/consumer.

È particolarmente adatta a:

```text id="yqsl5v"
background queue
pipeline asincrone
worker
eventi interni
buffer bounded
backpressure
producer e consumer async
```

La documentazione Microsoft sui channel mostra che un bounded channel ha una capacità massima e, quando è pieno, il comportamento predefinito è bloccare asincronamente il produttore finché non c’è spazio; il comportamento può essere configurato con `BoundedChannelFullMode`. ([Microsoft Learn][4])

---

## Channel non bounded

```csharp id="x9t7ly"
using System.Threading.Channels;

Channel<string> channel = Channel.CreateUnbounded<string>();

await channel.Writer.WriteAsync("Messaggio");

string messaggio = await channel.Reader.ReadAsync();
```

È semplice, ma attenzione:

```text id="mqs6gi"
se i produttori scrivono più velocemente dei consumatori,
la memoria può crescere senza controllo
```

---

## Channel bounded

Meglio spesso usare una capacità massima.

```csharp id="t160yg"
Channel<string> channel = Channel.CreateBounded<string>(
    new BoundedChannelOptions(capacity: 100)
    {
        FullMode = BoundedChannelFullMode.Wait
    });
```

Con `Wait`, quando il channel è pieno:

```text id="qlqk2f"
WriteAsync aspetta asincronicamente spazio disponibile
TryWrite restituisce false
```

Questo crea backpressure.

---

# Producer/Consumer

Il pattern **producer/consumer** separa chi produce lavoro da chi lo esegue.

Schema:

```text id="ab9z9g"
Producer
  crea elementi di lavoro

Queue / Channel
  conserva temporaneamente gli elementi

Consumer
  legge ed elabora elementi
```

Esempi:

```text id="nvfb9o"
endpoint API produce job
BackgroundService consuma job

lettore file produce righe
worker consuma righe

SignalR produce eventi
servizio notifiche consuma eventi

import produce record
validatore/processore consuma record
```

---

## Perché è utile

```text id="m4d0to"
disaccoppia chi produce da chi consuma
evita fire-and-forget incontrollato
limita concorrenza
gestisce picchi
permette backpressure
rende il lavoro background più ordinato
```

---

# Backpressure

La **backpressure** è il meccanismo che impedisce ai produttori di produrre più lavoro di quanto il sistema riesca a gestire.

Senza backpressure:

```text id="u5vwks"
producer velocissimo
consumer lento
coda cresce senza limiti
memoria aumenta
latenza cresce
app instabile
```

Con backpressure:

```text id="oe5p44"
la coda ha capacità massima
quando è piena, il producer aspetta o scarta secondo policy
il sistema resta sotto controllo
```

Con `Channel<T>`:

```csharp id="xaeklu"
Channel<Job> channel = Channel.CreateBounded<Job>(
    new BoundedChannelOptions(100)
    {
        FullMode = BoundedChannelFullMode.Wait
    });
```

---

## Full mode di Channel

Con bounded channel puoi scegliere il comportamento quando è pieno:

```text id="n497bf"
Wait
  aspetta spazio disponibile.

DropNewest
  rimuove l'elemento più nuovo per fare spazio.

DropOldest
  rimuove l'elemento più vecchio.

DropWrite
  scarta il nuovo elemento.
```

Microsoft documenta questi comportamenti tramite `BoundedChannelFullMode`: `Wait` è il valore predefinito, mentre `DropNewest`, `DropOldest` e `DropWrite` permettono strategie di perdita controllata quando il channel è pieno. ([Microsoft Learn][4])

---

## Quando scartare messaggi

Scartare può avere senso per:

```text id="u42zgx"
telemetria ad alta frequenza
aggiornamenti UI intermedi
metriche non critiche
posizione mouse
progress parziale
snapshot sostituibili
```

Non scartare per:

```text id="c4w4qp"
pagamenti
ordini
movimenti contabili
email critiche
prenotazioni
comandi di business importanti
```

Per dati critici, serve persistenza: database, outbox, message broker.

# Pipeline asincrone

Una pipeline asincrona divide l’elaborazione in fasi.

Schema:

```text id="k47zec"
Input
  ↓
Lettura
  ↓
Validazione
  ↓
Trasformazione
  ↓
Salvataggio
  ↓
Output
```

Ogni fase può avere il proprio channel.

Esempio:

```text id="k3x2he"
Channel<RigaCsv>
  lettura file

Channel<ClienteImportato>
  parsing e validazione

Channel<ClienteValido>
  salvataggio database
```

Vantaggi:

```text id="z16d09"
separazione responsabilità
concorrenza controllata
backpressure tra fasi
monitoraggio più chiaro
possibilità di parallelizzare solo alcune fasi
```

---

## Esempio concettuale di pipeline

```csharp id="xnihrx"
Channel<string> righeChannel =
    Channel.CreateBounded<string>(100);

Channel<ClienteImportato> clientiChannel =
    Channel.CreateBounded<ClienteImportato>(100);

Task lettore = LeggiRigheAsync(
    righeChannel.Writer,
    cancellationToken);

Task parser = ParsificaAsync(
    righeChannel.Reader,
    clientiChannel.Writer,
    cancellationToken);

Task salvatore = SalvaClientiAsync(
    clientiChannel.Reader,
    cancellationToken);

await Task.WhenAll(lettore, parser, salvatore);
```

---

## Lettore

```csharp id="bs62tr"
private static async Task LeggiRigheAsync(
    ChannelWriter<string> writer,
    CancellationToken cancellationToken)
{
    try
    {
        for (int i = 0; i < 1000; i++)
        {
            string riga = $"Cliente {i};cliente{i}@example.com";

            await writer.WriteAsync(riga, cancellationToken);
        }
    }
    finally
    {
        writer.Complete();
    }
}
```

---

## Parser

```csharp id="wq1jsj"
private static async Task ParsificaAsync(
    ChannelReader<string> reader,
    ChannelWriter<ClienteImportato> writer,
    CancellationToken cancellationToken)
{
    try
    {
        await foreach (string riga in reader.ReadAllAsync(cancellationToken))
        {
            string[] parti = riga.Split(';');

            if (parti.Length != 2)
            {
                continue;
            }

            ClienteImportato cliente = new(
                Nome: parti[0],
                Email: parti[1]);

            await writer.WriteAsync(cliente, cancellationToken);
        }
    }
    finally
    {
        writer.Complete();
    }
}
```

---

## Consumer finale

```csharp id="h0boq6"
private static async Task SalvaClientiAsync(
    ChannelReader<ClienteImportato> reader,
    CancellationToken cancellationToken)
{
    await foreach (ClienteImportato cliente in reader.ReadAllAsync(cancellationToken))
    {
        Console.WriteLine(
            $"Salvo {cliente.Nome} - {cliente.Email}");

        await Task.Delay(10, cancellationToken);
    }
}
```

Esempi:

## Interfaccia di coda

```csharp id="d6c7wm"
public interface IBackgroundTaskQueue<T>
{
    ValueTask EnqueueAsync(
        T item,
        CancellationToken cancellationToken);

    IAsyncEnumerable<T> DequeueAllAsync(
        CancellationToken cancellationToken);
}
```

---

## Implementazione con Channel

```csharp id="8fa84e"
using System.Threading.Channels;

public class BackgroundTaskQueue<T> : IBackgroundTaskQueue<T>
{
    private readonly Channel<T> channel;

    public BackgroundTaskQueue(int capacity)
    {
        channel = Channel.CreateBounded<T>(
            new BoundedChannelOptions(capacity)
            {
                FullMode = BoundedChannelFullMode.Wait,
                SingleReader = true,
                SingleWriter = false
            });
    }

    public async ValueTask EnqueueAsync(
        T item,
        CancellationToken cancellationToken)
    {
        await channel.Writer.WriteAsync(item, cancellationToken);
    }

    public IAsyncEnumerable<T> DequeueAllAsync(
        CancellationToken cancellationToken)
    {
        return channel.Reader.ReadAllAsync(cancellationToken);
    }
}
```

## Worker

```csharp id="p4u1p3"
public class EmailWorker : BackgroundService
{
    private readonly IBackgroundTaskQueue<EmailJob> queue;
    private readonly IServiceScopeFactory scopeFactory;
    private readonly ILogger<EmailWorker> logger;

    public EmailWorker(
        IBackgroundTaskQueue<EmailJob> queue,
        IServiceScopeFactory scopeFactory,
        ILogger<EmailWorker> logger)
    {
        this.queue = queue;
        this.scopeFactory = scopeFactory;
        this.logger = logger;
    }

    protected override async Task ExecuteAsync(
        CancellationToken stoppingToken)
    {
        await foreach (EmailJob job in queue.DequeueAllAsync(stoppingToken))
        {
            try
            {
                using IServiceScope scope =
                    scopeFactory.CreateScope();

                EmailService emailService =
                    scope.ServiceProvider
                        .GetRequiredService<EmailService>();

                await emailService.InviaAsync(
                    job.Destinatario,
                    job.Oggetto,
                    job.Corpo,
                    stoppingToken);

                logger.LogInformation(
                    "Email inviata a {Destinatario}",
                    job.Destinatario);
            }
            catch (OperationCanceledException)
            {
                throw;
            }
            catch (Exception ex)
            {
                logger.LogError(
                    ex,
                    "Errore durante invio email a {Destinatario}",
                    job.Destinatario);
            }
        }
    }
}
```

# Code in memoria: limiti

Una coda interna in memoria è semplice, ma non è adatta a tutto.

Problemi:

```text id="r5qr4i"
se il processo si riavvia, i job in memoria si perdono
non coordina più istanze dell'app
non garantisce consegna
non è ideale per operazioni critiche
non conserva storico
```

Per lavoro critico usare:

```text id="6scdwr"
database
outbox pattern
message broker
RabbitMQ
Azure Service Bus
Kafka
Hangfire/Quartz, secondo scenario
```

Regola pratica:

```text id="e2fh7h"
Coda in memoria
  per lavoro leggero, non critico, locale al processo.

Coda persistente
  per lavoro critico, distribuito o da garantire.
```

# Producer multipli e consumer multipli

Con `Channel<T>` possiamo avere:

```text id="yjtfo8"
un producer, un consumer
molti producer, un consumer
un producer, molti consumer
molti producer, molti consumer
```

Esempio con più consumer:

```csharp id="ivbwst"
List<Task> workers = [];

for (int i = 0; i < 4; i++)
{
    workers.Add(Task.Run(() =>
        ConsumaAsync(channel.Reader, cancellationToken)));
}

await Task.WhenAll(workers);
```

Consumer:

```csharp id="ek92g2"
private static async Task ConsumaAsync(
    ChannelReader<Job> reader,
    CancellationToken cancellationToken)
{
    await foreach (Job job in reader.ReadAllAsync(cancellationToken))
    {
        await ElaboraAsync(job, cancellationToken);
    }
}
```

Attenzione:

```text id="pqx3vv"
più consumer aumentano throughput
ma possono cambiare ordine di completamento
```

Se l’ordine è fondamentale, usa un solo consumer o una strategia di ordinamento esplicita.

# BlockingCollection vs Channel

```text id="23k4b8"
BlockingCollection<T>
  modello più vecchio
  molto utile in scenari sincroni
  blocca thread quando aspetta
  supporta bounding e CompleteAdding

Channel<T>
  modello moderno
  async-first
  non blocca thread durante attesa
  ideale con async/await e BackgroundService
  supporta backpressure e pipeline asincrone
```

Scelta pratica:

```text id="8hac18"
Nuovo codice async in ASP.NET Core
  Channel<T>

Codice sincrono o legacy TPL
  BlockingCollection<T>

Producer/consumer semplice ma sincrono
  BlockingCollection<T>

Pipeline asincrona moderna
  Channel<T>
```

# Esempio: pipeline con backpressure e più worker

Scenario:

```text id="pst77a"
un endpoint riceve richieste di elaborazione immagini
le accoda
4 worker elaborano
la coda è limitata a 200 job
```

Job:

```csharp id="ghp8zq"
public record ImageJob(
    Guid Id,
    string FilePath);
```

Queue:

```csharp id="od6ud6"
public class ImageJobQueue
{
    private readonly Channel<ImageJob> channel;

    public ImageJobQueue()
    {
        channel = Channel.CreateBounded<ImageJob>(
            new BoundedChannelOptions(200)
            {
                FullMode = BoundedChannelFullMode.Wait,
                SingleReader = false,
                SingleWriter = false
            });
    }

    public ValueTask EnqueueAsync(
        ImageJob job,
        CancellationToken cancellationToken)
    {
        return channel.Writer.WriteAsync(job, cancellationToken);
    }

    public ChannelReader<ImageJob> Reader => channel.Reader;
}
```

Worker:

```csharp id="d200jg"
public class ImageWorker : BackgroundService
{
    private readonly ImageJobQueue queue;
    private readonly ILogger<ImageWorker> logger;

    public ImageWorker(
        ImageJobQueue queue,
        ILogger<ImageWorker> logger)
    {
        this.queue = queue;
        this.logger = logger;
    }

    protected override async Task ExecuteAsync(
        CancellationToken stoppingToken)
    {
        Task[] workers = Enumerable
            .Range(1, 4)
            .Select(workerId => ProcessaAsync(workerId, stoppingToken))
            .ToArray();

        await Task.WhenAll(workers);
    }

    private async Task ProcessaAsync(
        int workerId,
        CancellationToken stoppingToken)
    {
        await foreach (ImageJob job in queue.Reader.ReadAllAsync(stoppingToken))
        {
            try
            {
                logger.LogInformation(
                    "Worker {WorkerId} elabora immagine {JobId}",
                    workerId,
                    job.Id);

                await ElaboraImmagineAsync(job, stoppingToken);
            }
            catch (Exception ex)
            {
                logger.LogError(
                    ex,
                    "Errore elaborazione immagine {JobId}",
                    job.Id);
            }
        }
    }

    private static async Task ElaboraImmagineAsync(
        ImageJob job,
        CancellationToken cancellationToken)
    {
        await Task.Delay(1000, cancellationToken);
    }
}
```

Endpoint:

```csharp id="wenqif"
app.MapPost("/api/images/process", async (
    string filePath,
    ImageJobQueue queue,

    return Results.Accepted(

            job.Id,
            Stato = "Accodato"
        });
});
```

# Errori comuni

## Usare `List<T>` condivisa tra thread

Sbagliato:

```csharp id="kzqk7g"
private static readonly List<string> Messaggi = [];

Messaggi.Add("ciao");
```

da più thread.

Meglio:

```csharp id="lxtygx"
ConcurrentQueue<string>
```

oppure una lista protetta da lock, oppure un `Channel<T>`.

---

## Usare ConcurrentDictionary ma fare check-then-act

Sbagliato:

```csharp id="mod8li"
if (!dictionary.ContainsKey(key))
{
    dictionary[key] = value;
}
```

Meglio:

```csharp id="jaxxde"
dictionary.TryAdd(key, value);
```

---

## Usare ConcurrentBag quando serve ordine

Sbagliato:

```text id="fa9rlm"
Uso ConcurrentBag per una coda di messaggi ordinati.
```

Meglio:

```text id="b6v8u5"
ConcurrentQueue
Channel<T>
```

---

## Usare code non bounded

Sospetto:

```csharp id="xko0n6"
Channel.CreateUnbounded<Job>();
```

per input non controllato.

Se i producer sono più veloci dei consumer, la memoria può crescere.

Meglio bounded channel quando il carico può aumentare.

---

## Non completare writer nelle pipeline

Se una pipeline ha una fine naturale, il writer deve essere completato.

```csharp id="sfysu1"
writer.Complete();
```

Altrimenti i consumer possono restare in attesa.

---

## Ignorare CancellationToken

Sbagliato:

```csharp id="kbnjek"
await foreach (Job job in reader.ReadAllAsync())
{
}
```

Meglio:

```csharp id="gj2wdt"
await foreach (Job job in reader.ReadAllAsync(cancellationToken))
{
}
```

---

## Usare coda in memoria per lavoro critico

Se non puoi perdere il lavoro, non affidarti solo alla RAM.

Usa persistenza.

---

## Troppi consumer

Aumentare i consumer non migliora sempre.

Può causare:

```text id="howpu5"
pressione sul database
rate limit API esterne
più contesa
più memoria
ordine non deterministico
```

Misura e limita.

# Riepilogo

In questo capitolo abbiamo studiato collezioni concorrenti e pattern producer/consumer.

Abbiamo visto:

```text id="dhddw5"
ConcurrentDictionary
ConcurrentQueue
ConcurrentStack
ConcurrentBag
BlockingCollection
Channel<T>
Producer/Consumer
Backpressure
Pipeline asincrone
Code interne a un’applicazione
```

Abbiamo chiarito che le collezioni concorrenti di `System.Collections.Concurrent` permettono accesso thread-safe da più thread senza sincronizzazione manuale aggiuntiva. ([Microsoft Learn][1])

Abbiamo visto che `BlockingCollection<T>` implementa il pattern producer/consumer con blocking e bounding, utile per controllare la memoria e impedire ai produttori di avanzare troppo rispetto ai consumatori. ([Microsoft Learn][2])

Abbiamo visto che `Channel<T>` è spesso la scelta migliore nel nuovo codice async-first: un bounded channel permette backpressure asincrona e può essere configurato per aspettare, scartare il più vecchio, scartare il più nuovo o rifiutare la scrittura quando è pieno. ([Microsoft Learn][4])

La regola principale è questa:

```text id="nlbi4u"
Per la concorrenza applicativa, spesso è meglio far comunicare
le parti tramite code e pipeline invece di condividere stato modificabile
protetto da lock sparse nel codice.
```

Questo capitolo è il ponte naturale verso il prossimo livello: parallelismo controllato, `Parallel.ForEachAsync`, PLINQ, CPU-bound workload, misurazione delle performance e scelta consapevole del grado di concorrenza.

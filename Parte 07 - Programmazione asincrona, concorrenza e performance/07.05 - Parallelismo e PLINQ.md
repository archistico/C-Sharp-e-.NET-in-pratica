# 07.05 — Parallelismo e PLINQ

Nei capitoli precedenti abbiamo studiato `async/await`, `Task`, `ThreadPool`, sincronizzazione, collezioni concorrenti, `Channel<T>` e producer/consumer.

Ora affrontiamo il **parallelismo**.

Il parallelismo non serve a “rendere asincrono” il codice. Serve a usare più core della CPU per eseguire contemporaneamente parti indipendenti di un lavoro.

Il punto centrale è questo:

```text id="wlozvv"
async/await serve soprattutto per non bloccare durante operazioni I/O-bound.
Parallelismo e PLINQ servono soprattutto per accelerare lavori CPU-bound
divisibili in parti indipendenti.
```

PLINQ, per esempio, nasce per velocizzare query LINQ to Objects eseguendo i delegate in parallelo su computer multi-core; Microsoft precisa che funziona meglio quando l’elaborazione di ogni elemento è indipendente e non condivide stato con gli altri elementi. ([learn.microsoft.com](https://learn.microsoft.com/en-us/dotnet/standard/parallel-programming/understanding-speedup-in-plinq?utm_source=chatgpt.com))

---

# Concorrenza, asincronia e parallelismo

Prima di scrivere codice, conviene fissare le differenze.

```text id="01wcjj"
Concorrenza
  più attività sono in corso nello stesso periodo.

Asincronia
  il codice può sospendersi mentre aspetta qualcosa,
  senza bloccare inutilmente un thread.

Parallelismo
  più attività vengono eseguite realmente nello stesso momento,
  usando più core o più thread.
```

Esempio I/O-bound:

```text id="3aqd7a"
scaricare 10 pagine web
leggere dati da database
chiamare API esterne
```

Qui spesso serve `async/await`.

Esempio CPU-bound:

```text id="k0dtxh"
ridimensionare 10.000 immagini
calcolare hash di molti file
elaborare righe di un grande CSV
fare calcoli matematici pesanti
analizzare testo in memoria
```

Qui può servire parallelismo.

# CPU-bound workload

Un workload è **CPU-bound** quando il collo di bottiglia è la CPU.

Esempio:

```csharp id="x1rf5l"
public static int ContaPrimi(int max)
{
    int count = 0;

    for (int n = 2; n <= max; n++)
    {
        if (IsPrimo(n))
        {
            count++;
        }
    }

    return count;
}

private static bool IsPrimo(int n)
{
    for (int i = 2; i * i <= n; i++)
    {
        if (n % i == 0)
        {
            return false;
        }
    }

    return true;
}
```

Questo tipo di lavoro può beneficiare del parallelismo se:

```text id="5t3jtf"
i dati sono molti
ogni elemento richiede abbastanza calcolo
le elaborazioni sono indipendenti
il costo di coordinamento è inferiore al guadagno
la macchina ha più core disponibili
```

# `Parallel.For`

`Parallel.For` è la versione parallela di un ciclo `for`.

Esempio sequenziale:

```csharp id="1m194u"
for (int i = 0; i < immagini.Count; i++)
{
    ElaboraImmagine(immagini[i]);
}
```

Versione parallela:

```csharp id="sb6mt5"
Parallel.For(0, immagini.Count, i =>
{
    ElaboraImmagine(immagini[i]);
});
```

Qui più iterazioni possono essere eseguite contemporaneamente.

Microsoft documenta `Parallel.For` come una famiglia di metodi che esegue un ciclo `for` in cui le iterazioni possono essere eseguite in parallelo. ([learn.microsoft.com](https://learn.microsoft.com/en-us/dotnet/api/system.threading.tasks.parallel.for?view=net-10.0&utm_source=chatgpt.com))

---

## Quando usare `Parallel.For`

È adatto quando:

```text id="b7a3sj"
hai un intervallo numerico
ogni iterazione è indipendente
il lavoro per iterazione è CPU-bound
non serve ordine di esecuzione
non ci sono scritture concorrenti pericolose
```

Esempio:

```csharp id="q9egj9"
int[] input = Enumerable.Range(1, 1_000_000).ToArray();
int[] output = new int[input.Length];

Parallel.For(0, input.Length, i =>
{
    output[i] = CalcoloPesante(input[i]);
});
```

Qui ogni iterazione scrive in una posizione diversa dell’array: è molto più sicuro che aggiornare una lista condivisa.

# `Parallel.ForEach`

`Parallel.ForEach` lavora su una sequenza.

Esempio:

```csharp id="b8ygll"
List<string> files = Directory
    .EnumerateFiles("C:\\Foto", "*.jpg")
    .ToList();

Parallel.ForEach(files, file =>
{
    ElaboraFile(file);
});
```

È utile quando hai una collezione di elementi da elaborare.

---

## Esempio: calcolo hash di file

```csharp id="2cs0yp"
ConcurrentDictionary<string, string> hashes = new();

Parallel.ForEach(filePaths, path =>
{
    string hash = CalcolaSha256(path);

    hashes[path] = hash;
});
```

Nota: qui usiamo `ConcurrentDictionary` perché più iterazioni scrivono nella stessa struttura condivisa.

Ancora meglio, quando possibile, evitare stato condiviso e produrre risultati locali.

---

# Controllo del grado di parallelismo

Il **grado di parallelismo** indica quante operazioni possono essere eseguite contemporaneamente.

Con `ParallelOptions` possiamo limitarlo.

```csharp id="p31fg2"
ParallelOptions options = new()
{
    MaxDegreeOfParallelism = Environment.ProcessorCount
};

Parallel.ForEach(files, options, file =>
{
    ElaboraFile(file);
});
```

La proprietà `MaxDegreeOfParallelism` limita il numero di operazioni concorrenti usate dai metodi `Parallel` che ricevono quell’istanza di `ParallelOptions`; un valore positivo impone il limite, mentre `-1` indica nessun limite, con eccezione di `ForEachAsync` dove `-1` significa `ProcessorCount`. ([learn.microsoft.com](https://learn.microsoft.com/en-us/dotnet/api/system.threading.tasks.paralleloptions.maxdegreeofparallelism?view=net-10.0&utm_source=chatgpt.com))

---

## Perché limitarlo

Limitare il parallelismo è utile quando:

```text id="8r98sw"
il lavoro usa molta CPU
il lavoro usa molta memoria
ci sono risorse esterne limitate
il database non deve essere sovraccaricato
un'API esterna ha rate limit
il disco diventa il collo di bottiglia
```

Attenzione: `Parallel.ForEach` su lavoro che fa I/O sincrono può saturare thread e peggiorare la situazione.

Per I/O asincrono, di solito è meglio usare `async/await`, `Task.WhenAll` con throttling, `SemaphoreSlim`, `Channel<T>` o `Parallel.ForEachAsync`.

---

# Esempio: limitare l’elaborazione CPU

```csharp id="bi19yt"
ParallelOptions options = new()
{
    MaxDegreeOfParallelism = Math.Max(1, Environment.ProcessorCount - 1)
};

Parallel.ForEach(immagini, options, immagine =>
{
    RidimensionaImmagine(immagine);
});
```

Qui lasciamo almeno un core logico libero per il sistema e per il resto dell’applicazione.

Non è una regola assoluta, ma è una scelta ragionevole in molte applicazioni desktop o batch.

# Break e Stop

Nei cicli paralleli, `break` e `continue` tradizionali non funzionano nello stesso modo dei cicli sequenziali.

`Parallel.For` e `Parallel.ForEach` forniscono `ParallelLoopState`.

---

## `Stop`

`Stop` chiede di interrompere il ciclo appena possibile.

```csharp id="e05hgc"
Parallel.ForEach(numeri, (numero, state) =>
{
    if (numero < 0)
    {
        state.Stop();
        return;
    }

    Elabora(numero);
});
```

`Stop` comunica:

```text id="f6yn6p"
non serve avviare altre iterazioni
fermiamoci appena possibile
```

Non garantisce che tutte le iterazioni già partite si interrompano immediatamente.

---

## `Break`

`Break` è utile quando l’ordine degli indici ha significato.

```csharp id="q14bfi"
Parallel.For(0, numeri.Length, (i, state) =>
{
    if (numeri[i] < 0)
    {
        state.Break();
        return;
    }

    Elabora(numeri[i]);
});
```

`Break` dice:

```text id="y1gwxm"
non serve elaborare indici successivi a questo
```

Ma iterazioni con indice minore o già avviate possono continuare.

---

## Differenza pratica

```text id="aqrp2h"
Stop
  fermati appena possibile, senza ragionare sull'ordine.

Break
  fermati rispetto a una posizione ordinata del ciclo.
```

Regola pratica:

```text id="wa9o5r"
Usa Stop quando cerchi una condizione globale.
Usa Break quando l'indice/ordine ha significato.
```

---

# Eccezioni aggregate

Nei cicli paralleli, più iterazioni possono fallire contemporaneamente.

Esempio:

```csharp id="cxwpbg"
try
{
    Parallel.ForEach(files, file =>
    {
        ElaboraFile(file);
    });
}
catch (AggregateException ex)
{
    foreach (Exception inner in ex.InnerExceptions)
    {
        Console.WriteLine(inner.Message);
    }
}
```

`AggregateException` raggruppa più errori.

Microsoft segnala che, quando si aggiunge logica di gestione eccezioni nei cicli paralleli, bisogna considerare che eccezioni simili possono essere generate simultaneamente su più thread e che un’eccezione in un thread può causarne altre in thread differenti; un approccio è raccoglierle in un `AggregateException`. ([learn.microsoft.com](https://learn.microsoft.com/en-us/dotnet/standard/parallel-programming/how-to-handle-exceptions-in-parallel-loops?utm_source=chatgpt.com))

---

## Gestione per elemento

Spesso è meglio gestire l’errore vicino all’elemento che fallisce.

```csharp id="kmz83n"
ConcurrentBag<Exception> errors = new();

Parallel.ForEach(files, file =>
{
    try
    {
        ElaboraFile(file);
    }
    catch (Exception ex)
    {
        errors.Add(new InvalidOperationException(
            $"Errore sul file {file}",
            ex));
    }
});

if (!errors.IsEmpty)
{
    throw new AggregateException(errors);
}
```

Questo consente di:

```text id="e9px3t"
continuare a elaborare altri elementi
sapere quali elementi sono falliti
rilanciare un errore aggregato alla fine
```

# PLINQ

PLINQ significa:

```text id="nnqzng"
Parallel LINQ
```

È una versione parallela di LINQ to Objects.

Esempio LINQ normale:

```csharp id="j5xq0u"
List<int> risultati = numeri
    .Where(n => IsPrimo(n))
    .Select(n => n * 2)
    .ToList();
```

Versione PLINQ:

```csharp id="hphsgk"
List<int> risultati = numeri
    .AsParallel()
    .Where(n => IsPrimo(n))
    .Select(n => n * 2)
    .ToList();
```

`AsParallel` dice a LINQ di usare PLINQ.

Microsoft presenta PLINQ come una modalità per ottenere miglioramenti significativi delle prestazioni su certi tipi di query, spesso aggiungendo `AsParallel`; però avverte che non tutte le query diventano più veloci e alcune possono rallentare. ([learn.microsoft.com](https://learn.microsoft.com/it-it/dotnet/standard/parallel-programming/introduction-to-plinq?utm_source=chatgpt.com))

---

# `AsParallel`

`AsParallel` trasforma una sequenza LINQ in una query parallela.

```csharp id="iux1xh"
var query = numeri
    .AsParallel()
    .Where(n => IsPrimo(n))
    .Select(n => n);
```

È adatto quando:

```text id="otcd4a"
la sequenza è in memoria
la query è LINQ to Objects
ogni elemento richiede lavoro CPU significativo
le operazioni sono indipendenti
non serve stato condiviso
```

Non è adatto per:

```text id="3z0s8e"
EF Core IQueryable verso database
chiamate HTTP per ogni elemento
operazioni I/O-bound
query piccole
operazioni banali
logica con side effect condivisi
```

---

## PLINQ e IQueryable

Attenzione:

```text id="o0dysb"
PLINQ riguarda LINQ to Objects.
Non serve a rendere parallela una query SQL di EF Core.
```

Sbagliato concettualmente:

```csharp id="tutwhs"
var clienti = dbContext.Clienti
    .AsParallel()
    .Where(c => c.Attivo)
    .ToList();
```

In pratica, prima materializzi o comunque cambi il modello di esecuzione, rischiando confusione e inefficienza.

Per database:

```text id="qwp04q"
lascia lavorare il database
ottimizza query SQL
usa indici
usa proiezioni
usa AsNoTracking
usa query asincrone
```

# `AsOrdered`

Per impostazione predefinita, PLINQ può non preservare l’ordine originale, perché mantenere l’ordine ha un costo.

Se l’ordine è importante:

```csharp id="h3pp2p"
List<int> risultati = numeri
    .AsParallel()
    .AsOrdered()
    .Where(n => IsPrimo(n))
    .ToList();
```

`AsOrdered` preserva l’ordine della sequenza originale nei risultati.

Ma attenzione:

```text id="hhfxf9"
preservare l'ordine può ridurre le prestazioni
```

Regola pratica:

```text id="tj8hgp"
Usa AsOrdered solo quando l'ordine è davvero parte del risultato.
```

---

# `WithDegreeOfParallelism`

PLINQ permette di controllare il grado di parallelismo.

```csharp id="r05k4c"
List<int> risultati = numeri
    .AsParallel()
    .WithDegreeOfParallelism(Environment.ProcessorCount)
    .Where(n => IsPrimo(n))
    .ToList();
```

Questo limita quante attività parallele PLINQ usa.

È utile quando:

```text id="ufxhih"
non vuoi saturare la macchina
il lavoro usa molta memoria
la query è parte di un’app più grande
hai più operazioni parallele contemporanee
```

# PLINQ ed eccezioni

Anche PLINQ può generare `AggregateException`.

Microsoft spiega che, quando una query PLINQ viene eseguita, più eccezioni possono essere lanciate contemporaneamente da thread diversi; PLINQ usa `AggregateException` per incapsulare tutte le eccezioni e riportarle al thread chiamante. ([learn.microsoft.com](https://learn.microsoft.com/en-us/dotnet/standard/parallel-programming/introduction-to-plinq?utm_source=chatgpt.com))

Esempio:

```csharp id="qicnhs"
try
{
    List<int> risultati = numeri
        .AsParallel()
        .Select(n =>
        {
            if (n < 0)
            {
                throw new InvalidOperationException(
                    $"Numero negativo: {n}");
            }

            return CalcoloPesante(n);
        })
        .ToList();
}
catch (AggregateException ex)
{
    foreach (Exception inner in ex.InnerExceptions)
    {
        Console.WriteLine(inner.Message);
    }
}
```

# Quando il parallelismo peggiora le prestazioni

Il parallelismo ha un costo.

Costi principali:

```text id="f3t2os"
creazione e scheduling del lavoro
partizionamento dei dati
sincronizzazione
merge dei risultati
contesa su memoria/cache CPU
contesa su lock
pressione sul garbage collector
overhead di ordinamento
accesso concorrente a risorse lente
```

Microsoft avverte esplicitamente di evitare l’over-parallelization in PLINQ: `AsParallel` introduce costi di partizionamento della sorgente e sincronizzazione dei worker thread, e il beneficio è limitato dal numero di processori disponibili. ([learn.microsoft.com](https://learn.microsoft.com/en-us/dotnet/standard/parallel-programming/potential-pitfalls-with-plinq?utm_source=chatgpt.com))

---

## Esempio in cui peggiora

```csharp id="pcxare"
List<int> risultati = numeri
    .AsParallel()
    .Where(n => n > 10)
    .Select(n => n * 2)
    .ToList();
```

Se `numeri` contiene pochi elementi o il lavoro è banale, la versione parallela può essere più lenta di:

```csharp id="eo1bd6"
List<int> risultati = numeri
    .Where(n => n > 10)
    .Select(n => n * 2)
    .ToList();
```

Perché il costo di parallelizzare supera il guadagno.

# Errori comuni con stato condiviso

Sbagliato:

```csharp id="mkws9y"
List<int> risultati = [];

Parallel.ForEach(numeri, n =>
{
    if (IsPrimo(n))
    {
        risultati.Add(n);
    }
});
```

`List<T>` non è thread-safe.

Possibili correzioni:

```csharp id="sfzunw"
ConcurrentBag<int> risultati = new();

Parallel.ForEach(numeri, n =>
{
    if (IsPrimo(n))
    {
        risultati.Add(n);
    }
});
```

Oppure usare PLINQ:

```csharp id="behebu"
List<int> risultati = numeri
    .AsParallel()
    .Where(n => IsPrimo(n))
    .ToList();
```

Meglio ancora: evitare side effect condivisi.

# Variabili locali per partizione

Per ridurre contesa, `Parallel.For` supporta stati locali.

Esempio: somma parallela.

Sbagliato:

```csharp id="nglpzw"
long totale = 0;

Parallel.ForEach(numeri, n =>
{
    Interlocked.Add(ref totale, Calcola(n));
});
```

Funziona, ma `Interlocked` viene chiamato per ogni elemento.

Meglio accumulare localmente e combinare alla fine.

```csharp id="xxyqqm"
long totale = 0;

Parallel.For<long>(
    0,
    numeri.Length,
    () => 0,
    (i, state, localTotal) =>
    {
        return localTotal + Calcola(numeri[i]);
    },
    localTotal =>
    {
        Interlocked.Add(ref totale, localTotal);
    });
```

La documentazione di `Parallel.For` mostra overload con stato locale per inizializzare computazioni locali, accumulare risultati per partizione e combinarli alla fine con `Interlocked.Add`. ([learn.microsoft.com](https://learn.microsoft.com/en-us/dotnet/api/system.threading.tasks.parallel.for?view=net-10.0&utm_source=chatgpt.com))

---

# Parallelismo e cancellazione

`Parallel.For` e `Parallel.ForEach` supportano cancellazione tramite `ParallelOptions`.

```csharp id="awh8t0"
CancellationTokenSource cts = new();

ParallelOptions options = new()
{
    CancellationToken = cts.Token,
    MaxDegreeOfParallelism = Environment.ProcessorCount
};

try
{
    Parallel.ForEach(files, options, file =>
    {
        options.CancellationToken.ThrowIfCancellationRequested();

        ElaboraFile(file);
    });
}
catch (OperationCanceledException)
{
    Console.WriteLine("Elaborazione annullata.");
}
```

Microsoft documenta che per cancellare un ciclo parallelo si passa il `CancellationToken` tramite `ParallelOptions` e si racchiude la chiamata parallela in un blocco `try/catch`. ([learn.microsoft.com](https://learn.microsoft.com/en-us/dotnet/standard/parallel-programming/how-to-cancel-a-parallel-for-or-foreach-loop?utm_source=chatgpt.com))

# Parallelismo e operazioni async

Attenzione: `Parallel.ForEach` non è pensato per lambda asincrone.

Sbagliato:

```csharp id="w6f8pd"
Parallel.ForEach(urls, async url =>
{
    await httpClient.GetStringAsync(url);
});
```

Questo è un errore concettuale: la lambda `async` restituisce una task che `Parallel.ForEach` non attende nel modo che ti aspetti.

Alternative:

```text id="ojjkgi"
Task.WhenAll
SemaphoreSlim per limitare concorrenza
Channel<T>
Parallel.ForEachAsync
```

Esempio con `Task.WhenAll` e `SemaphoreSlim`:

```csharp id="ei3myq"
SemaphoreSlim semaphore = new(10);

IEnumerable<Task> tasks = urls.Select(async url =>
{
    await semaphore.WaitAsync(cancellationToken);

    try
    {
        string html = await httpClient.GetStringAsync(
            url,
            cancellationToken);

        // usa html
    }
    finally
    {
        semaphore.Release();
    }
});

await Task.WhenAll(tasks);
```

Questo è più adatto a I/O asincrono.

# `Parallel.ForEachAsync`

Anche se non era nell’elenco iniziale, merita una nota perché spesso viene confuso con `Parallel.ForEach`.

```csharp id="6cgk0v"
await Parallel.ForEachAsync(
    urls,
    new ParallelOptions
    {
        MaxDegreeOfParallelism = 10,
        CancellationToken = cancellationToken
    },
    async (url, token) =>
    {
        string html = await httpClient.GetStringAsync(url, token);

        // usa html
    });
```

È utile per elaborazioni asincrone con grado di concorrenza controllato.

Ma non sostituisce automaticamente una buona progettazione con `Channel<T>` o code quando il flusso è complesso.

# Esempio completo: elaborazione CPU-bound con Parallel.ForEach

Scenario:

```text id="pwpzco"
abbiamo molti testi in memoria
vogliamo calcolare una metrica pesante per ciascun testo
```

DTO:

```csharp id="u7bjln"
public record DocumentoAnalizzato(
    string NomeFile,
    int Punteggio);
```

Codice:

```csharp id="4byy29"
ConcurrentBag<DocumentoAnalizzato> risultati = new();

ParallelOptions options = new()
{
    MaxDegreeOfParallelism = Environment.ProcessorCount
};

Parallel.ForEach(documenti, options, documento =>
{
    int punteggio = CalcolaPunteggioPesante(documento.Contenuto);

    risultati.Add(new DocumentoAnalizzato(
        documento.NomeFile,
        punteggio));
});

List<DocumentoAnalizzato> ordinati = risultati
    .OrderBy(r => r.NomeFile)
    .ToList();
```

Nota:

```text id="iag5dp"
ConcurrentBag non conserva ordine.
Se serve ordine, ordiniamo dopo.
```

# Esempio completo con PLINQ

```csharp id="3o517x"
List<DocumentoAnalizzato> risultati = documenti
    .AsParallel()
    .WithDegreeOfParallelism(Environment.ProcessorCount)
    .Select(documento =>
    {
        int punteggio =
            CalcolaPunteggioPesante(documento.Contenuto);

        return new DocumentoAnalizzato(
            documento.NomeFile,
            punteggio);
    })
    .OrderBy(r => r.NomeFile)
    .ToList();
```

Questa versione è più dichiarativa.

PLINQ è spesso leggibile quando la trasformazione assomiglia già a una query LINQ.

# Parallel.ForEach vs PLINQ

```text id="ipzqfc"
Parallel.ForEach
  più adatto a operazioni imperative
  utile quando hai side effect controllati
  più controllo sul corpo del ciclo
  comodo per elaborazioni batch

PLINQ
  più adatto a query su sequenze in memoria
  stile dichiarativo
  comodo per filtri, proiezioni, aggregazioni
  attenzione a ordine, merge e side effect
```

Scelta pratica:

```text id="8q53g7"
Se pensi in termini di ciclo ed elaborazione:
  Parallel.ForEach

Se pensi in termini di query LINQ:
  PLINQ
```

# Esempio: quando NON usare parallelismo

```csharp id="lzqm9i"
List<string> nomi = clienti
    .AsParallel()
    .Where(c => c.Attivo)
    .Select(c => c.Nome.Trim())
    .ToList();
```

Se `clienti` contiene 500 elementi e il lavoro per elemento è banale, questa versione può essere più lenta.

Meglio:

```csharp id="spdkp9"
List<string> nomi = clienti
    .Where(c => c.Attivo)
    .Select(c => c.Nome.Trim())
    .ToList();
```

Regola pratica:

```text id="8uzib6"
Non parallelizzare codice veloce.
Parallelizza codice costoso, indipendente e misurato.
```

```text id="t2956e"
uso disco/rete/database
```

Per misure serie in .NET si usa spesso BenchmarkDotNet, che vedremo nella parte performance.

Esempio semplice non scientifico:

```csharp id="b6orir"
Stopwatch sw = Stopwatch.StartNew();

ElaboraSequenziale();

sw.Stop();

Console.WriteLine($"Sequenziale: {sw.Elapsed}");

sw.Restart();

ElaboraParallelo();

sw.Stop();

Console.WriteLine($"Parallelo: {sw.Elapsed}");
```

Questo dà un primo segnale, ma non basta per benchmark affidabili.

# Errori comuni

## Usare parallelismo per I/O-bound

Sbagliato:

```csharp id="u6lzmx"
Parallel.ForEach(urls, url =>
{
    httpClient.GetStringAsync(url).Wait();
});
```

Meglio:

```csharp id="way3i4"
await Task.WhenAll(urls.Select(url =>
    httpClient.GetStringAsync(url)));
```

con throttling se necessario.

---

## Aggiornare collezioni non thread-safe

Sbagliato:

```csharp id="m311h0"
List<int> risultati = [];

Parallel.ForEach(numeri, n =>
{
    risultati.Add(n);
});
```

Meglio:

```csharp id="d4e4ur"
ConcurrentBag<int> risultati = new();
```

oppure PLINQ senza side effect.

---

## Parallelizzare troppo

Sbagliato:

```text id="bs4x2v"
AsParallel ovunque
Parallel.ForEach per ogni lista
MaxDegreeOfParallelism altissimo
```

Parallelizzare troppo può peggiorare prestazioni e stabilità.

---

## Preservare ordine senza necessità

```csharp id="e0v511"
.AsParallel()
.AsOrdered()
```

ha un costo.

Usalo solo quando l’ordine è parte del risultato.

---

## Usare PLINQ su query EF Core

PLINQ non ottimizza SQL.
Prima materializzare tutto e poi parallelizzare spesso è una pessima idea.

---

## Non gestire eccezioni aggregate

Se più iterazioni falliscono, devi sapere come raccogliere e diagnosticare gli errori.

---

## Ignorare risorse esterne

Anche se la CPU ha 16 core, il database o il disco potrebbero non reggere 16 operazioni pesanti contemporanee.

---

# Linee guida pratiche

```text id="mnkm9o"
Usa Parallel.For
  per cicli numerici CPU-bound.

Usa Parallel.ForEach
  per elaborare collezioni CPU-bound.

Usa ParallelOptions
  per limitare MaxDegreeOfParallelism.

Usa Stop
  quando vuoi fermare il ciclo appena possibile.

Usa Break
  quando l'ordine degli indici ha significato.

Gestisci AggregateException
  o raccogli errori per elemento.

Usa PLINQ
  per query LINQ to Objects costose e indipendenti.

Usa AsOrdered
  solo se l'ordine è necessario.

Usa WithDegreeOfParallelism
  per controllare l'uso di CPU e risorse.

Evita stato condiviso
  dentro loop paralleli.

Non usare parallelismo
  per sostituire async/await.

Non usare PLINQ
  per query database EF Core.

Misura sempre
  perché il parallelismo può peggiorare le prestazioni.
```

# Riepilogo

In questo capitolo abbiamo studiato parallelismo e PLINQ.

Abbiamo visto:

```text id="ok2uo7"
Parallel.For
Parallel.ForEach
controllo del grado di parallelismo
Break e Stop
eccezioni aggregate
PLINQ
AsParallel
AsOrdered
WithDegreeOfParallelism
quando il parallelismo peggiora le prestazioni
CPU-bound workload
```

Abbiamo chiarito che PLINQ è pensato per LINQ to Objects su workload CPU-bound, grandi e indipendenti, e che non tutte le query diventano più veloci: il parallelismo introduce overhead di partizionamento, sincronizzazione e merge dei risultati. ([learn.microsoft.com](https://learn.microsoft.com/en-us/dotnet/standard/parallel-programming/potential-pitfalls-with-plinq?utm_source=chatgpt.com))

Abbiamo visto che `MaxDegreeOfParallelism` permette di limitare quante operazioni concorrenti vengono eseguite dai metodi `Parallel`, e che questa scelta va fatta considerando CPU, memoria e risorse esterne. ([learn.microsoft.com](https://learn.microsoft.com/en-us/dotnet/api/system.threading.tasks.paralleloptions.maxdegreeofparallelism?view=net-10.0&utm_source=chatgpt.com))

Abbiamo anche visto che le eccezioni in cicli paralleli e query PLINQ possono arrivare da più thread contemporaneamente e vengono spesso gestite tramite `AggregateException`. ([learn.microsoft.com](https://learn.microsoft.com/en-us/dotnet/standard/parallel-programming/how-to-handle-exceptions-in-parallel-loops?utm_source=chatgpt.com))

La regola principale è questa:

```text id="uol1cq"
Il parallelismo è uno strumento per lavori CPU-bound indipendenti.
Non è una scorciatoia per rendere veloce qualunque codice
e non sostituisce async/await per operazioni I/O-bound.
```

Il passo successivo naturale è studiare performance e diagnostica: misurare, profilare, capire allocazioni, GC, hot path, BenchmarkDotNet e strumenti di Visual Studio.

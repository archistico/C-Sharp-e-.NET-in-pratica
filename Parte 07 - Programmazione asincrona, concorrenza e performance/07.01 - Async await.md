# 07.01 — Async/await

Fino a questo punto abbiamo studiato C#, LINQ, database, ASP.NET Core, API, Blazor e SignalR. Ora possiamo affrontare una delle parti più importanti della programmazione moderna in .NET: **la programmazione asincrona**.

`async/await` non è solo una sintassi comoda. È il modo con cui C# permette di scrivere codice non bloccante mantenendo una forma leggibile, quasi sequenziale.

Il punto centrale è questo:

```text id="kv818f"
async/await serve a non bloccare il thread mentre un'operazione è in attesa.
È fondamentale per file, rete, database, API, UI, servizi web e scalabilità.
```

Microsoft descrive la programmazione asincrona basata su `Task` come un modello che permette di scrivere codice simile a una sequenza di istruzioni, anche se l’esecuzione reale può sospendersi e riprendere quando le operazioni asincrone completano. ([Microsoft Learn][1])

---

# Perché esiste async/await

Molte operazioni non sono lente perché la CPU sta calcolando. Sono lente perché il programma sta **aspettando qualcosa**.

Esempi:

```text id="jt6fwi"
lettura da database
chiamata HTTP
lettura file
scrittura file
invio email
download
upload
attesa risposta da servizio esterno
query a un'API
```

Durante queste operazioni, bloccare un thread sarebbe inefficiente.

Esempio sincrono:

```csharp id="yusyhd"
string contenuto = File.ReadAllText("dati.txt");

Console.WriteLine(contenuto);
```

Finché la lettura non finisce, il thread resta fermo.

Versione asincrona:

```csharp id="1y6upb"
string contenuto = await File.ReadAllTextAsync("dati.txt");

Console.WriteLine(contenuto);
```

Il metodo può sospendersi durante l’attesa e riprendere quando il file è stato letto.

---

# Async non significa multithreading automatico

Errore comune:

```text id="e5bjvr"
async = nuovo thread
```

Non è corretto.

`async/await` non crea automaticamente un nuovo thread.

Serve soprattutto a liberare il thread corrente mentre aspettiamo un’operazione asincrona.

Esempio:

```csharp id="aoikbq"
await httpClient.GetStringAsync("https://example.com");
```

Durante l’attesa della rete, il thread non deve restare bloccato.

---

# I/O-bound vs CPU-bound

Per capire `async/await`, bisogna distinguere due tipi di lavoro.

---

## I/O-bound

Un lavoro è **I/O-bound** quando passa la maggior parte del tempo aspettando input/output.

Esempi:

```text id="83puna"
database
HTTP
file system
rete
code messaggi
servizi esterni
```

Qui `async/await` è ideale.

Esempio:

```csharp id="w4m4tv"
public async Task<ClienteDto?> TrovaClienteAsync(int id)
{
    ClienteDto? cliente = await httpClient
        .GetFromJsonAsync<ClienteDto>($"/api/clienti/{id}");

    return cliente;
}
```

Il programma aspetta la risposta HTTP senza bloccare inutilmente il thread.

---

## CPU-bound

Un lavoro è **CPU-bound** quando la CPU deve calcolare davvero.

Esempi:

```text id="mvvii5"
compressione pesante
elaborazione immagini
calcolo matematico intensivo
analisi di grandi file in memoria
crittografia pesante
generazione report molto complessi
```

Qui `async/await` da solo non rende il calcolo più veloce.

Esempio:

```csharp id="sblmf4"
public int Calcola()
{
    int risultato = 0;

    for (int i = 0; i < 100_000_000; i++)
    {
        risultato += i;
    }

    return risultato;
}
```

Rendere il metodo `async` non migliora nulla se dentro non c’è nulla da attendere.

Per CPU-bound, possiamo valutare:

```text id="fxuf8u"
Task.Run
Parallel.ForEach
PLINQ
algoritmi migliori
SIMD
ottimizzazione memoria
background processing
```

---

# `Task`

`Task` rappresenta un’operazione asincrona che non restituisce un valore.

Esempio:

```csharp id="ef0gmb"
public async Task SalvaAsync(Cliente cliente)
{
    dbContext.Clienti.Add(cliente);

    await dbContext.SaveChangesAsync();
}
```

Il metodo:

```text id="z6g0gl"
non restituisce un valore utile
ma rappresenta un'operazione che finirà in futuro
```

La documentazione Microsoft indica che un metodo asincrono può restituire `Task` quando esegue un’operazione senza valore di ritorno, oppure `Task<TResult>` quando restituisce un valore. ([Microsoft Learn][2])

---

# `Task<T>`

`Task<T>` rappresenta un’operazione asincrona che, al completamento, restituisce un valore di tipo `T`.

Esempio:

```csharp id="ql35br"
public async Task<ClienteDto?> TrovaAsync(int id)
{
    ClienteDto? cliente = await dbContext.Clienti
        .AsNoTracking()
        .Where(c => c.Id == id)
        .Select(c => new ClienteDto(c.Id, c.Nome, c.Email))
        .FirstOrDefaultAsync();

    return cliente;
}
```

Qui il risultato finale è:

```text id="xj0vwc"
ClienteDto?
```

ma arriva dentro:

```text id="8u8hvg"
Task<ClienteDto?>
```

perché l’operazione è asincrona.

---

# `async`

La keyword `async` abilita l’uso di `await` dentro un metodo.

```csharp id="1bhryf"
public async Task EseguiAsync()
{
    await Task.Delay(1000);
}
```

Un metodo marcato `async` può sospendere la propria esecuzione su un `await` e riprenderla quando l’operazione attesa è completata.

Attenzione: `async` da solo non rende asincrono un metodo.

Sospetto:

```csharp id="u0dmx3"
public async Task<int> SommaAsync(int a, int b)
{
    return a + b;
}
```

Questo metodo non contiene `await`. Non ha un vero lavoro asincrono.

Meglio:

```csharp id="tj3dk2"
public Task<int> SommaAsync(int a, int b)
{
    return Task.FromResult(a + b);
}
```

oppure, se non serve asincronia:

```csharp id="pbgtri"
public int Somma(int a, int b)
{
    return a + b;
}
```

---

# `await`

`await` attende il completamento di una `Task` senza bloccare il thread nel modo tradizionale.

Esempio:

```csharp id="qt9jp1"
public async Task<string> ScaricaAsync(HttpClient httpClient)
{
    string html = await httpClient.GetStringAsync("https://example.com");

    return html;
}
```

Il flusso logico è leggibile:

```text id="816xo8"
scarica
aspetta risultato
restituisci contenuto
```

Ma tecnicamente il metodo può sospendersi e riprendere più tardi.

---

# Esempio completo: API service asincrono

```csharp id="qstkk5"
public class ClienteService
{
    private readonly AppDbContext dbContext;
    private readonly ILogger<ClienteService> logger;

    public ClienteService(
        AppDbContext dbContext,
        ILogger<ClienteService> logger)
    {
        this.dbContext = dbContext;
        this.logger = logger;
    }

    public async Task<ClienteDto?> TrovaAsync(int id)
    {
        logger.LogInformation(
            "Ricerca cliente {ClienteId}",
            id);

        ClienteDto? cliente = await dbContext.Clienti
            .AsNoTracking()
            .Where(c => c.Id == id)
            .Select(c => new ClienteDto(
                c.Id,
                c.Nome,
                c.Email))
            .FirstOrDefaultAsync();

        return cliente;
    }

    public async Task<ClienteDto> CreaAsync(CreaClienteRequest request)
    {
        Cliente cliente = new Cliente
        {
            Nome = request.Nome,
            Email = request.Email,
            Attivo = true
        };

        dbContext.Clienti.Add(cliente);

        await dbContext.SaveChangesAsync();

        return new ClienteDto(
            cliente.Id,
            cliente.Nome,
            cliente.Email);
    }
}
```

Endpoint:

```csharp id="hd01yw"
app.MapGet("/api/clienti/{id:int}", async (
    int id,
    ClienteService service) =>
{
    ClienteDto? cliente = await service.TrovaAsync(id);

    return cliente is null
        ? Results.NotFound()
        : Results.Ok(cliente);
});
```

# Propagare async

Una regola importante:

```text id="du38sy"
L’asincronia tende a propagarsi verso l’alto.
```

Se un metodo chiama un metodo async, spesso anche lui deve diventare async.

Esempio:

```csharp id="hzdzjp"
public async Task<ClienteDto?> TrovaAsync(int id)
{
    return await repository.TrovaAsync(id);
}
```

Poi il controller o endpoint:

```csharp id="sh3wco"
public async Task<IResult> HandleAsync(int id)
{
    ClienteDto? cliente = await service.TrovaAsync(id);

    return cliente is null
        ? Results.NotFound()
        : Results.Ok(cliente);
}
```

Questo è normale.

Errore da evitare:

```csharp id="2oi8wj"
ClienteDto? cliente = service.TrovaAsync(id).Result;
```

---

# Eccezioni in async

Le eccezioni nei metodi async vengono catturate nella `Task` e rilanciate quando facciamo `await`.

Esempio:

```csharp id="w6jpz9"
public async Task EseguiAsync()
{
    await Task.Delay(100);

    throw new InvalidOperationException("Errore durante l'operazione.");
}
```

Gestione:

```csharp id="ruw2ow"
try
{
    await EseguiAsync();
}
catch (InvalidOperationException ex)
{
    logger.LogError(ex, "Operazione fallita");
}
```

Il `try/catch` funziona normalmente se usiamo `await`.

---

## Errore: non attendere la Task

Sbagliato:

```csharp id="bnst3q"
try
{
    EseguiAsync();
}
catch (Exception ex)
{
    logger.LogError(ex, "Errore");
}
```

Qui il metodo parte, ma non viene atteso. L’eccezione non viene catturata da quel `catch`.

Corretto:

```csharp id="s2im3d"
try
{
    await EseguiAsync();
}
catch (Exception ex)
{
    logger.LogError(ex, "Errore");
}
```

---

# `async void`

Quasi sempre da evitare.

Sbagliato:

```csharp id="cdf6b8"
public async void Salva()
{
    await dbContext.SaveChangesAsync();
}
```

Meglio:

```csharp id="wjo6ox"
public async Task SalvaAsync()
{
    await dbContext.SaveChangesAsync();
}
```

La documentazione Microsoft indica che `async void` dovrebbe essere usato solo per event handler; fuori da questi casi non segue bene il Task-based Asynchronous Pattern e rende più difficile gestione errori, composizione e testabilità. ([Microsoft Learn][3])

Eccezione accettabile:

```csharp id="x42ms4"
private async void Button_Click(object sender, EventArgs e)
{
    await SalvaAsync();
}
```

Per eventi UI storici, la firma può richiedere `void`.

# Cancellazione con `CancellationToken`

La cancellazione in .NET è cooperativa.

Significa:

```text id="pv9zjk"
chi chiama richiede la cancellazione
il metodo chiamato deve controllare il token
e fermarsi in modo ordinato
```

Microsoft spiega che il modello di cancellazione cooperativa usa `CancellationTokenSource` e `CancellationToken`: il token viene passato alle operazioni che devono ricevere notifica della cancellazione, ma non è il token stesso ad avviare la cancellazione. ([Microsoft Learn][4])

---

## Esempio base

```csharp id="uxo6i7"
public async Task ElaboraAsync(CancellationToken cancellationToken)
{
    for (int i = 0; i < 100; i++)
    {
        cancellationToken.ThrowIfCancellationRequested();

        await Task.Delay(100, cancellationToken);

        Console.WriteLine($"Passo {i}");
    }
}
```

Chiamante:

```csharp id="8p1n88"
using CancellationTokenSource cts = new();

Task task = ElaboraAsync(cts.Token);

cts.Cancel();

await task;
```

---

## CancellationToken in ASP.NET Core

Negli endpoint ASP.NET Core possiamo ricevere un `CancellationToken`.

```csharp id="do30z7"
app.MapGet("/api/clienti", async (
    AppDbContext dbContext,
    CancellationToken cancellationToken) =>
{
    List<ClienteDto> clienti = await dbContext.Clienti
        .AsNoTracking()
        .Select(c => new ClienteDto(c.Id, c.Nome, c.Email))
        .ToListAsync(cancellationToken);

    return Results.Ok(clienti);
});
```

Se il client interrompe la richiesta, ASP.NET Core può segnalare cancellazione.

Regola pratica:

```text id="d3t1cv"
Accetta CancellationToken nei metodi async pubblici
e passalo alle operazioni async interne quando disponibile.
```

# Timeout

La cancellazione può essere usata anche per implementare timeout.

Esempio:

```csharp id="tgfeq5"
using CancellationTokenSource cts =
    new(TimeSpan.FromSeconds(5));

await servizio.EseguiAsync(cts.Token);
```

Oppure combinare token esterno e timeout:

```csharp id="fbvc5a"
public async Task<ClienteDto?> TrovaConTimeoutAsync(
    int id,
    CancellationToken cancellationToken)
{
    using CancellationTokenSource timeoutCts =
        new(TimeSpan.FromSeconds(5));

    using CancellationTokenSource linkedCts =
        CancellationTokenSource.CreateLinkedTokenSource(
            cancellationToken,
            timeoutCts.Token);

    return await repository.TrovaAsync(id, linkedCts.Token);
}
```

Qui l’operazione viene cancellata se:

```text id="m39gom"
il chiamante cancella
oppure
scade il timeout
```

---

## Timeout con HttpClient

Preferibile configurare timeout in modo consapevole.

```csharp id="3ij9zm"
HttpClient client = new()
{
    Timeout = TimeSpan.FromSeconds(10)
};
```

Oppure usare `CancellationToken` nella singola chiamata:

```csharp id="l1vdk2"
HttpResponseMessage response =
    await client.GetAsync("/api/clienti", cancellationToken);
```

# `ConfigureAwait`

`ConfigureAwait(false)` dice a `await` che la continuazione non deve necessariamente tornare al contesto originale.

Esempio:

```csharp id="eyjquk"
string testo = await File.ReadAllTextAsync(path)
    .ConfigureAwait(false);
```

La documentazione del metodo `Task.ConfigureAwait` indica che attendere direttamente una `Task` può far riprendere la continuazione nel contesto originale, se presente; questo può avere costi e, in alcuni scenari UI, contribuire a deadlock. ([Microsoft Learn][5])

---

## Dove serve davvero

Regola pratica moderna:

```text id="lgd2zs"
Codice applicativo ASP.NET Core
  normalmente non serve ConfigureAwait(false).

Librerie riutilizzabili
  spesso ha senso usare ConfigureAwait(false).

Applicazioni UI desktop
  attenzione: dopo await potresti dover tornare sul thread UI.

Codice didattico/base
  non introdurlo troppo presto.
```

In ASP.NET Core non c’è lo stesso `SynchronizationContext` tipico delle vecchie UI desktop, quindi il problema classico è meno frequente.

Nel libro conviene presentarlo come argomento avanzato, non come rituale da mettere ovunque.

# `ValueTask`

`ValueTask` e `ValueTask<T>` sono alternative a `Task` e `Task<T>` pensate per scenari di performance specifici.

Esempio:

```csharp id="4yft6w"
public ValueTask<int> LeggiDaCacheAsync(string chiave)
{
    if (cache.TryGetValue(chiave, out int valore))
    {
        return ValueTask.FromResult(valore);
    }

    return new ValueTask<int>(LeggiDaDatabaseAsync(chiave));
}
```

La documentazione Microsoft avverte che `ValueTask<TResult>` può essere atteso o convertito in `Task<TResult>`, ma in generale deve essere atteso una sola volta; se queste limitazioni sono problematiche, va convertito con `AsTask`. ([Microsoft Learn][6])

Per `ValueTask` non generico, Microsoft osserva che essendo una struct con più campi può comportare copie maggiori rispetto a un riferimento `Task`; la scelta predefinita per un metodo asincrono senza risultato dovrebbe restare `Task`, usando `ValueTask` solo quando l’analisi prestazionale dimostra che conviene. ([Microsoft Learn][7])

---

## Regola pratica su ValueTask

```text id="g5jj87"
Usa Task di default.

Valuta ValueTask solo se:
  il metodo completa spesso in modo sincrono
  è chiamato moltissime volte
  hai misurato allocazioni rilevanti
  conosci bene le limitazioni
```

Non usare `ValueTask` per rendere il codice “più moderno”.
Usalo solo quando serve davvero.

# Fire-and-forget

**Fire-and-forget** significa avviare un’operazione asincrona senza attenderla.

Esempio sospetto:

```csharp id="1zilqn"
_ = InviaEmailAsync(cliente.Email);
```

Il codice prosegue senza aspettare.

Problemi:

```text id="l2uvui"
eccezioni non osservate
operazione interrotta se l'app termina
lifetime DI non valido
DbContext già disposed
difficile da testare
difficile da monitorare
perdita di dati
```

---

## Fire-and-forget in ASP.NET Core: attenzione

Sbagliato:

```csharp id="0b9ku0"
app.MapPost("/api/clienti", async (
    CreaClienteRequest request,
    ClienteService service,
    EmailService emailService) =>
{
    ClienteDto cliente = await service.CreaAsync(request);

    _ = emailService.InviaBenvenutoAsync(cliente.Email);

    return Results.Created($"/api/clienti/{cliente.Id}", cliente);
});
```

Se `EmailService` dipende da servizi scoped della request, dopo la risposta potrebbero non essere più validi.

Meglio usare:

```text id="72cv50"
background queue
BackgroundService
messaggistica
outbox pattern
job scheduler
```

## Pattern migliore: coda background

Interfaccia:

```csharp id="yalxfs"
public interface IBackgroundTaskQueue
{
    ValueTask QueueAsync(Func<CancellationToken, ValueTask> workItem);
```

```text id="oo1zi5"
l'endpoint accoda lavoro

```

# Deadlock

Un deadlock async tipico nasce quando si blocca sincronicamente aspettando una `Task`.

Sospetto:

```csharp id="tplmfu"
var risultato = MetodoAsync().Result;
```

oppure:

```csharp id="m2e4vc"
MetodoAsync().Wait();
```

In applicazioni UI o vecchi ambienti con `SynchronizationContext`, questo può causare deadlock.

Schema concettuale:

```text id="vy6iws"
1. Thread UI chiama MetodoAsync().Result e si blocca.
2. MetodoAsync arriva a await.
3. Dopo await vuole riprendere sul thread UI.
4. Ma il thread UI è bloccato su .Result.
5. Nessuno può proseguire.
```

Risultato:

```text id="7kt7rz"
deadlock
```

---

## Soluzione

Non bloccare su codice async.

Sbagliato:

```csharp id="tr54u1"
public ClienteDto? Trova(int id)
{
    return TrovaAsync(id).Result;
}
```

Corretto:

```csharp id="nlhpd0"
public async Task<ClienteDto?> TrovaAsync(int id)
{
    return await repository.TrovaAsync(id);
}
```

E propagare `async` verso l’alto.

Regola pratica:

```text id="qcxfqa"
Async all the way.
Se inizi async, continua async fino al punto di ingresso.
```

# `Task.Run`

`Task.Run` mette lavoro sul ThreadPool.

È utile soprattutto per CPU-bound quando vuoi non bloccare il thread chiamante.

Esempio in app desktop:

```csharp id="zbdtwx"
private async void Button_Click(object sender, EventArgs e)
{
    risultato = await Task.Run(() =>
    {
        return CalcoloPesante();
    });
}
```

In ASP.NET Core, usare `Task.Run` per operazioni I/O-bound è spesso inutile o dannoso.

Sbagliato:

```csharp id="ezzwxr"
await Task.Run(async () =>
{
    await dbContext.SaveChangesAsync();
});
```

Meglio:

```csharp id="pna7xn"
await dbContext.SaveChangesAsync();
```

Regola pratica:

```text id="dd01l7"
I/O-bound
  usa API async vere.

CPU-bound
  valuta Task.Run solo se ha senso spostare lavoro su ThreadPool.
```

# Parallelismo con Task.WhenAll

Se più operazioni async indipendenti possono partire insieme, possiamo usare `Task.WhenAll`.

Sequenziale:

```csharp id="mo8f9v"
ClienteDto? cliente = await clientiApi.TrovaAsync(id);
List<OrdineDto> ordini = await ordiniApi.ListaPerClienteAsync(id);
List<TicketDto> ticket = await ticketApi.ListaPerClienteAsync(id);
```

Parallelo:

```csharp id="dnf1a7"
Task<ClienteDto?> clienteTask = clientiApi.TrovaAsync(id);
Task<List<OrdineDto>> ordiniTask = ordiniApi.ListaPerClienteAsync(id);
Task<List<TicketDto>> ticketTask = ticketApi.ListaPerClienteAsync(id);

await Task.WhenAll(clienteTask, ordiniTask, ticketTask);

ClienteDto? cliente = await clienteTask;
List<OrdineDto> ordini = await ordiniTask;
List<TicketDto> ticket = await ticketTask;
```

Vantaggio:

```text id="5l2bn1"
le attese I/O indipendenti si sovrappongono
```

Attenzione:

```text id="21wdop"
non parallelizzare se le operazioni dipendono una dall'altra
non sovraccaricare servizi esterni
gestisci eccezioni e cancellazione
```

---

# Eccezioni con Task.WhenAll

Se una o più task falliscono, `await Task.WhenAll(...)` rilancia eccezione.

Esempio:

```csharp id="smgfg2"
try
{
    await Task.WhenAll(task1, task2, task3);
}
catch (Exception ex)
{
    logger.LogError(ex, "Una o più operazioni sono fallite");
}
```

Se devi analizzare tutte le eccezioni, puoi controllare le singole task dopo il fallimento.

---

# Esempio completo: metodo async robusto

```csharp id="ttgmh2"
public class ClienteImportService
{
    private readonly ClienteApiClient clienteApiClient;
    private readonly ILogger<ClienteImportService> logger;

    public ClienteImportService(
        ClienteApiClient clienteApiClient,
        ILogger<ClienteImportService> logger)
    {
        this.clienteApiClient = clienteApiClient;
        this.logger = logger;
    }

    public async Task ImportaAsync(
        IReadOnlyList<int> clientiIds,
        CancellationToken cancellationToken)
    {
        foreach (int clienteId in clientiIds)
        {
            cancellationToken.ThrowIfCancellationRequested();

            try
            {
                ClienteDto? cliente =
                    await clienteApiClient.TrovaAsync(
                        clienteId,
                        cancellationToken);

                if (cliente is null)
                {
                    logger.LogWarning(
                        "Cliente {ClienteId} non trovato",
                        clienteId);

                    continue;
                }

                await ElaboraClienteAsync(cliente, cancellationToken);
            }
            catch (OperationCanceledException)
            {
                logger.LogInformation(
                    "Import clienti annullato");

                throw;
            }
            catch (Exception ex)
            {
                logger.LogError(
                    ex,
                    "Errore durante import cliente {ClienteId}",
                    clienteId);

                // scelta progettuale:
                // continue oppure throw
            }
        }
    }

    private async Task ElaboraClienteAsync(
        ClienteDto cliente,
        CancellationToken cancellationToken)
    {
        await Task.Delay(100, cancellationToken);

        logger.LogInformation(
            "Cliente {ClienteId} elaborato",
            cliente.Id);
    }
}
```

Questo esempio mostra:

```text id="r3xhst"
Task
await
CancellationToken
OperationCanceledException
logging
gestione errori
propagazione cancellazione
```

# Errori comuni

## Usare `.Result` o `.Wait()`

Sbagliato:


# Linee guida pratiche

```text id="d5oj89"
Usa async/await
  per operazioni I/O-bound.

Non usare async
  solo per rendere moderno un metodo sincrono.

Restituisci Task
  per operazioni async senza risultato.

Restituisci Task<T>
  per operazioni async con risultato.

Evita async void
  tranne event handler.

Propaga async verso l’alto
  invece di usare Result o Wait.

Passa CancellationToken
  ai metodi che lo supportano.

Usa timeout
  per chiamate esterne e operazioni potenzialmente lente.

Usa ConfigureAwait(false)
  soprattutto in librerie, non come automatismo cieco.

Usa ValueTask
  solo dopo misurazioni e in scenari specifici.

Evita fire-and-forget
  per operazioni importanti.

Usa background service o code
  per lavoro asincrono fuori dalla request.

Distingui I/O-bound e CPU-bound
  perché richiedono strategie diverse.
```

# Riepilogo

In questo capitolo abbiamo introdotto `async/await`.

Abbiamo visto:

```text id="izrgwf"
perché esiste async/await
I/O-bound vs CPU-bound

Task<T>
async
await
eccezioni in async
cancellazione con CancellationToken

```

Abbiamo chiarito che i metodi asincroni possono restituire `Task`, `Task<TResult>`, `void` solo per event handler, tipi awaitable e anche stream asincroni come `IAsyncEnumerable<T>`. ([Microsoft Learn][2])

Abbiamo visto che la cancellazione in .NET è cooperativa: il chiamante richiede la cancellazione tramite `CancellationTokenSource`, mentre le operazioni devono osservare il `CancellationToken` e terminare in modo ordinato. ([Microsoft Learn][8])

Abbiamo anche visto che `ValueTask` è uno strumento di ottimizzazione, non la scelta predefinita: `Task` resta il tipo da usare nella maggior parte dei metodi asincroni. ([Microsoft Learn][7])

La regola principale è questa:

```text id="gih0g1"
async/await non serve a rendere il codice più veloce in assoluto.
Serve a rendere il codice non bloccante, scalabile e leggibile
quando deve attendere operazioni asincrone.
```

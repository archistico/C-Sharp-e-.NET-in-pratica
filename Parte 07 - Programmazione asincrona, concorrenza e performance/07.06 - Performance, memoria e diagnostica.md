# 07.06 — Performance, memoria e diagnostica

Nei capitoli precedenti abbiamo imparato a scrivere codice asincrono, concorrente e parallelo. Abbiamo visto `async/await`, `Task`, `ThreadPool`, sincronizzazione, collezioni concorrenti, `Channel<T>`, `Parallel.ForEach` e PLINQ.

Il punto centrale è questo:

```text id="a3k5xe"
Uno sviluppatore professionale non ottimizza a sensazione.
Misura, individua il collo di bottiglia, formula un’ipotesi,
modifica il codice e misura di nuovo.
```

Le performance non sono solo “codice veloce”. Riguardano:

```text id="l1zt72"
tempo di esecuzione

pressione sul Garbage Collector
uso CPU
uso memoria
contesa tra thread
I/O
scalabilità

```

# Misurare prima di ottimizzare

La prima regola è semplice:

```text id="z5yy8m"
Non ottimizzare ciò che non hai misurato.
```

Molti problemi di performance vengono diagnosticati male perché si parte da frasi come:

```text id="h46xwz"
secondo me è LINQ
secondo me è EF Core
secondo me è il GC
secondo me è il database
secondo me è async
```

A volte è vero.
Spesso no.

Un metodo corretto è:

```text id="cmkrjv"
1. Definisci il problema.
2. Misura il comportamento attuale.
3. Trova il collo di bottiglia.
4. Formula una modifica.
5. Misura dopo la modifica.
6. Conserva la modifica solo se migliora davvero.
```

Esempio:

```text id="78vljp"
Problema:
  La pagina report impiega 18 secondi.

Misura:
  14 secondi sono nella query SQL.
  2 secondi nella serializzazione JSON.
  1 secondo nel rendering.
  1 secondo in altro codice.

Conclusione:
  Ottimizzare una lista C# non cambierà quasi nulla.
```

# Microbenchmark vs profiling

Ci sono due attività diverse:

```text id="yh3az9"
Benchmark
  misura una funzione o un frammento specifico in modo controllato.

Profiling
  osserva un’applicazione reale o quasi reale mentre lavora.
```

Esempio benchmark:

```text id="iw425a"
È più veloce usare string.Concat o StringBuilder in questo caso?
```

Esempio profiling:

```text id="jh0jgi"
Perché questa API in produzione è lenta?
```

Entrambi sono utili, ma rispondono a domande diverse.

## Primo benchmark

```csharp id="ddevyj"
using BenchmarkDotNet.Attributes;
using BenchmarkDotNet.Running;
using System.Text;

BenchmarkRunner.Run<StringBenchmarks>();

[MemoryDiagnoser]
public class StringBenchmarks
{
    private readonly string[] parole =
    [
        "uno",
        "due",
        "tre",
        "quattro",
        "cinque"
    ];

    [Benchmark]
    public string ConcatenaConPlus()
    {
        string result = "";

        foreach (string parola in parole)
        {
            result += parola;
        }

        return result;
    }

    [Benchmark]
    public string ConcatenaConStringBuilder()
    {
        StringBuilder builder = new();

        foreach (string parola in parole)
        {
            builder.Append(parola);
        }

        return builder.ToString();
    }
}
```

Eseguire in Release:

```bash id="nmeio6"
dotnet run -c Release
```

# Allocazioni

In .NET, creare oggetti è facile e relativamente economico. Ma troppe allocazioni possono diventare un problema.

Esempio:

```csharp id="jmjv5r"
for (int i = 0; i < 1_000_000; i++)
{
    string messaggio = "Valore: " + i;
}
```

Ogni iterazione crea nuove stringhe.

Possibili effetti:

```text id="yrz5ur"
più memoria usata
più lavoro per il Garbage Collector
più pause
minore throughput
maggiore latenza
```

---

## Come ridurre allocazioni

Strategie:

```text id="of0lk0"
evitare oggetti temporanei inutili
riusare buffer quando appropriato
usare StringBuilder per concatenazioni ripetute
usare Span<T> per slicing senza copie
usare ArrayPool<T> per buffer temporanei
usare struct con attenzione
evitare closure involontarie in hot path
evitare LINQ in cicli estremamente critici se alloca troppo
```

Attenzione: non significa eliminare ogni allocazione.
Significa ridurre quelle che contano davvero.

# Garbage Collector

Il **Garbage Collector**, o GC, è il gestore automatico della memoria di .NET.

Microsoft lo descrive come il memory manager automatico del CLR: gestisce allocazione e rilascio della memoria, evitando problemi tipici della gestione manuale come dimenticare di liberare un oggetto o accedere a memoria già liberata. ([Microsoft Learn][2])

Quando creiamo un oggetto:

```csharp id="xy77gb"
Cliente cliente = new Cliente();
```

l’oggetto viene allocato sull’heap gestito.

Quando non è più raggiungibile, il GC può recuperare quella memoria.

---

## Generazioni del GC

Il GC organizza gli oggetti in generazioni:

```text id="pnfh3x"
Gen 0
  oggetti giovani, raccolti spesso.

Gen 1
  oggetti intermedi.

Gen 2
  oggetti più longevi.

LOH
  Large Object Heap, per oggetti grandi.
```

Idea pratica:

```text id="flyewr"
molti oggetti muoiono presto
pochi oggetti vivono a lungo
```

Il GC è ottimizzato su questa osservazione.

---

## Pressione sul GC

Il problema non è “il GC esiste”.

Il problema è creare pressione inutile.

Esempi:

```text id="tl7dgy"
allocare molti oggetti temporanei in hot path
creare grandi array continuamente
creare stringhe intermedie
boxare value type
produrre molte closure
materializzare collezioni intermedie inutili
```

Esempio:

```csharp id="syefgr"
List<string> nomi = clienti
    .Where(c => c.Attivo)
    .Select(c => c.Nome.Trim().ToUpperInvariant())
    .ToList();
```

Va benissimo in molti casi.
Ma in un hot path chiamato migliaia di volte al secondo può diventare da misurare.

# Stack vs heap

In modo semplificato:

```text id="p8mpic"
Stack
  memoria per chiamate di metodi, variabili locali, frame di esecuzione.
  veloce, con durata legata allo scope.

Heap
  memoria per oggetti gestiti.
  gestita dal Garbage Collector.
```

Esempio:

```csharp id="d69dgc"
int x = 10;
Cliente cliente = new Cliente();
```

In modo concettuale:

```text id="2pffqn"
x
  value type locale, spesso sullo stack o ottimizzato dal runtime.

cliente
  riferimento locale.

new Cliente()
  oggetto sull'heap gestito.
```

Attenzione: la realtà può essere più complessa per via di ottimizzazioni JIT, escape analysis, generics, inlining e cattura di variabili.

---

## Non semplificare troppo

Non tutte le struct stanno sempre sullo stack.
Non tutti i riferimenti sono costosi.
Non ogni oggetto sull’heap è un problema.

Regola pratica:

```text id="gd8utr"
Capire stack e heap serve a ragionare,
ma le decisioni di performance vanno prese con misure reali.
```

# `Span<T>`

`Span<T>` rappresenta una vista leggera su una regione contigua di memoria.

Può puntare a:

```text id="sxxbiz"
array
parte di array
stackalloc memory
memoria non gestita
stringhe, tramite ReadOnlySpan<char>
```

Microsoft descrive `Span<T>` e `ReadOnlySpan<T>` come buffer di memoria leggeri che incapsulano riferimenti a memoria gestita o non gestita; poiché possono essere archiviati solo sullo stack, non sono adatti a scenari come chiamate asincrone. ([Microsoft Learn][3])

---

## Esempio con stringa

```csharp id="ulv9se"
ReadOnlySpan<char> codice = "CLIENTE-12345".AsSpan();

ReadOnlySpan<char> prefisso = codice[..7];
ReadOnlySpan<char> numero = codice[8..];
```

Qui possiamo lavorare su porzioni della stringa senza creare nuove stringhe.

Se facessimo:

```csharp id="974e8b"
string prefisso = codiceString.Substring(0, 7);
```

creeremmo una nuova stringa.

---

## Parsing con Span

```csharp id="kyocow"
ReadOnlySpan<char> testo = "12345".AsSpan();

if (int.TryParse(testo, out int valore))
{
    Console.WriteLine(valore);
}
```

Molte API moderne accettano `ReadOnlySpan<char>` per ridurre allocazioni.

---

## Limiti di Span

Non puoi usarlo liberamente ovunque.

Esempio non valido:

```csharp id="c832ox"
public class BufferHolder
{
    public Span<byte> Buffer { get; set; }
}
```

`Span<T>` è un `ref struct`, quindi ha restrizioni precise.

Non può:

```text id="sodkb7"
essere campo di una classe
essere catturato da lambda
attraversare await
essere usato in molti scenari heap-based
```

---

# `Memory<T>`

`Memory<T>` rappresenta anch’esso una regione contigua di memoria, ma a differenza di `Span<T>` può vivere sull’heap.

Microsoft specifica che `Memory<T>` rappresenta una regione contigua come `Span<T>`, ma non è un `ref struct`; quindi può essere usato come campo in una classe e attraverso confini `await` e `yield`. ([Microsoft Learn][4])

Esempio:

```csharp id="a8w3ps"
public class BufferMessage
{
    public Memory<byte> Payload { get; set; }
}
```

Uso:

```csharp id="r8o36k"
Memory<byte> memory = new byte[1024];

await Stream.WriteAsync(memory, cancellationToken);
```

---

## Span vs Memory

```text id="wqhkxb"
Span<T>
  molto leggero
  stack-only
  ottimo per parsing e hot path sincroni

Memory<T>
  può vivere sull'heap
  può essere usato in async
  adatto a buffer mantenuti più a lungo
```

Regola pratica:

```text id="ehd0tu"
Usa Span<T> per elaborazioni locali e sincrone.
Usa Memory<T> quando il buffer deve vivere oltre lo stack corrente
o attraversare await.
```

---

# Pooling

Il pooling consiste nel riusare oggetti o buffer invece di crearli e distruggerli continuamente.

Esempio tipico:

```csharp id="qpcxwf"
ArrayPool<byte>.Shared
```

Uso:

```csharp id="lwi5le"
byte[] buffer = ArrayPool<byte>.Shared.Rent(4096);

try
{
    int bytesRead = await stream.ReadAsync(buffer, cancellationToken);

    // usa buffer
}
finally
{
    ArrayPool<byte>.Shared.Return(buffer);
}
```

Vantaggi:

```text id="as245j"
meno allocazioni
meno pressione sul GC
riuso di buffer grandi
prestazioni migliori in hot path
```

Attenzione:

```text id="tgmlr7"
il buffer affittato può essere più grande del richiesto
il contenuto precedente può essere ancora presente
devi restituirlo
non devi usarlo dopo Return
```

---

## Quando usare pooling

Usalo per:

```text id="tadqdw"
buffer grandi
operazioni ripetute ad alta frequenza
parsing intensivo
I/O ad alte prestazioni
server molto trafficati
```

Non usarlo per complicare codice normale che non ha problemi misurati.

# SIMD

SIMD significa:

```text id="m0g4qr"
Single Instruction, Multiple Data
```

Permette di eseguire la stessa operazione su più dati contemporaneamente usando istruzioni vettoriali della CPU.

Esempio concettuale:

```text id="qkopue"
somma 4, 8, 16 o più numeri in una sola istruzione vettoriale
```

In .NET si può usare tramite:

```text id="ybr4od"
System.Numerics.Vector<T>
hardware intrinsics
API ottimizzate del runtime
```

Esempio semplice:

```csharp id="dgfgjz"
using System.Numerics;

public static void SommaVettori(float[] a, float[] b, float[] result)
{
    int vectorSize = Vector<float>.Count;
    int i = 0;

    for (; i <= a.Length - vectorSize; i += vectorSize)
    {
        Vector<float> va = new(a, i);
        Vector<float> vb = new(b, i);
        Vector<float> vr = va + vb;

        vr.CopyTo(result, i);
    }

    for (; i < a.Length; i++)
    {
        result[i] = a[i] + b[i];
    }
}
```

SIMD è utile in scenari numerici, grafici, audio, elaborazione immagini e calcoli intensivi.

# JIT

JIT significa:

```text id="ommy6g"
Just-In-Time compilation
```

Normalmente il codice C# viene compilato in IL.
Quando l’app gira, il JIT compila IL in codice macchina per la piattaforma corrente.

Vantaggi:

```text id="epk4tb"
ottimizzazioni per CPU reale
compilazione solo del codice usato
ottimizzazioni runtime
buona flessibilità
```

Svantaggi:

```text id="9l0ubu"
costo di compilazione durante esecuzione
startup potenzialmente più lento
warmup
```

---

# Native AOT

Native AOT compila l’app in codice nativo al momento della pubblicazione.

Microsoft spiega che pubblicare con Native AOT produce un’app self-contained compilata ahead-of-time in codice nativo; queste app possono avere startup più rapido e footprint di memoria minore, e possono girare su macchine senza runtime .NET installato. ([Microsoft Learn][5])

Esempio publish:

```bash id="p9qtc9"
dotnet publish -c Release -r win-x64 -p:PublishAot=true
```

---

## Vantaggi Native AOT

```text id="f6svk5"
startup più rapido
minore memoria in alcuni scenari
deployment self-contained
nessun JIT a runtime
utile per CLI, microservizi piccoli, serverless, ambienti vincolati
```

Microsoft segnala inoltre che le app Native AOT non usano JIT a runtime e sono destinate a uno specifico runtime identifier, ad esempio Windows x64 o Linux x64. ([Microsoft Learn][6])

---

## Limiti Native AOT

Native AOT non è sempre compatibile con tutto.

Attenzione a:

```text id="wqti9o"
reflection dinamica
serializzazione basata su reflection
plugin caricati dinamicamente
alcune librerie non AOT-friendly
dynamic code generation
Expression.Compile in certi scenari
dimensione e configurazione trim
```

Regola pratica:

```text id="fsis4z"
Native AOT è una scelta di deployment e performance,
non una modalità da attivare ciecamente su ogni progetto.
```

# Escape analysis

L’**escape analysis** è una tecnica con cui il runtime o il compilatore cerca di capire se un oggetto “esce” dallo scope locale.

Se un oggetto non esce, in alcuni casi può essere ottimizzato, per esempio evitando allocazioni sull’heap.

Esempio concettuale:

```csharp id="btrzqj"
public int Calcola()
{
    Punto p = new Punto(10, 20);

    return p.X + p.Y;
}
```

Se `p` non viene restituito, non viene salvato in campi e non viene passato a codice che potrebbe conservarlo, il runtime può avere più possibilità di ottimizzazione.

Nel libro, questo argomento va trattato come concetto avanzato:

```text id="ym6mxq"
non scrivere codice contando ciecamente su escape analysis
ma sapere che il runtime moderno lavora per ridurre allocazioni inutili
```

# Esempio: troppe allocazioni

Codice:

```csharp id="rewf1b"
public string NormalizzaCodice(string input)
{
    return input
        .Trim()
        .ToUpperInvariant()
        .Replace("-", "")
        .Replace(" ", "");
}
```

Va bene per uso normale.

Ma se viene chiamato milioni di volte in un import, può allocare molte stringhe intermedie.

Possibili strategie:

```text id="wgusfp"
misurare con BenchmarkDotNet
usare Span<char>
usare buffer pooling
ridurre passaggi intermedi
ottimizzare solo se è davvero hot path
```

Non partire subito con codice complesso. Prima misura.

---

# Esempio: pooling controllato

```csharp id="brwawh"
public static string RimuoviSeparatore(string input)
{
    char[] buffer = ArrayPool<char>.Shared.Rent(input.Length);

    try
    {
        int index = 0;

        foreach (char c in input)
        {
            if (c != '-' && c != ' ')
            {
                buffer[index++] = c;
            }
        }

        return new string(buffer, 0, index);
    }
    finally
    {
        ArrayPool<char>.Shared.Return(buffer);
    }
}
```

Questo può ridurre allocazioni intermedie, ma è più complesso.

Regola pratica:

```text id="n7t2md"
Accetta codice più complesso solo quando la misura dimostra
che la complessità è giustificata.
```

# Errori comuni

## Ottimizzare senza misurare

Sbagliato:

```text id="g9wyyq"
Riscrivo tutto perché forse è più veloce.
```

Meglio:

Abbiamo chiarito che BenchmarkDotNet è adatto a microbenchmark ripetibili e può misurare tempo, memoria e altre metriche, mentre il profiling serve a capire il comportamento di un’applicazione reale o quasi reale. ([Microsoft Learn][1])

Abbiamo visto che il Garbage Collector gestisce automaticamente allocazione e rilascio della memoria, ma le allocazioni eccessive possono aumentare pressione sul GC e peggiorare latenza e throughput. ([Microsoft Learn][2])

Abbiamo visto che `Span<T>` e `Memory<T>` permettono di lavorare su regioni contigue di memoria riducendo copie e allocazioni, con una differenza fondamentale: `Span<T>` è stack-only, mentre `Memory<T>` può essere usato anche attraverso confini `await`. ([Microsoft Learn][3])

Abbiamo anche introdotto Native AOT come opzione di deployment che compila ahead-of-time in codice nativo, utile per startup e footprint in scenari specifici, ma da valutare rispetto a compatibilità e vincoli delle librerie. ([Microsoft Learn][5])

La regola principale è questa:

```text id="joth7g"
La performance professionale non nasce da trucchi isolati.
Nasce da misure affidabili, comprensione del runtime,
diagnostica corretta e modifiche mirate.
```

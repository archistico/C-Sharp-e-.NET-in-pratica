# 07.02 — Thread, ThreadPool e Task

Nel capitolo precedente abbiamo introdotto `async/await`: perché esiste, differenza tra I/O-bound e CPU-bound, `Task`, `Task<T>`, cancellazione, timeout, `ConfigureAwait`, `ValueTask`, fire-and-forget e deadlock.

Ora scendiamo di un livello.

Per usare bene `async/await`, non basta conoscere la sintassi. Bisogna anche capire che cosa sono:

```text id="km6n00"
thread
ThreadPool
worker thread
I/O completion thread
Task
Task scheduler
continuazioni
background service
hosted service
```

Il punto centrale è questo:

```text id="stjjoh"
async/await è il modello di scrittura del codice asincrono.
Thread, ThreadPool e Task sono parte dell'infrastruttura che permette
a .NET di eseguire lavoro concorrente e non bloccante.
```

La Task Parallel Library, o TPL, si basa sul concetto di `Task`, che rappresenta un’operazione asincrona a un livello di astrazione più alto rispetto a thread e work item del ThreadPool. Microsoft evidenzia che le task permettono un uso più efficiente e scalabile delle risorse rispetto alla gestione diretta dei thread. ([Microsoft Learn][1])

# Thread

Un **thread** è un flusso di esecuzione.

Un processo può avere più thread.

Esempio:

```text id="f7fsta"
Processo
  Thread principale
  Thread secondario
  Thread del ThreadPool
  Thread dedicati
```

Ogni thread ha:

```text id="eiqc4d"
uno stack
uno stato di esecuzione
un costo di memoria
un costo di scheduling
```

Creare thread manualmente è possibile, ma oggi è meno comune nelle applicazioni .NET moderne.

Esempio:

```csharp id="ty1vu7"
Thread thread = new Thread(() =>
{
    Console.WriteLine("Lavoro su un thread dedicato.");
});

thread.Start();
thread.Join();
```

Questo codice:

```text id="gq2rjo"
crea un thread dedicato
lo avvia
attende che finisca
```

---

## Quando usare `Thread` direttamente

Molto raramente.

Può avere senso per:

```text id="6p12ts"
integrazioni legacy
controllo molto specifico del thread
thread dedicati con requisiti particolari
interop o scenari molto specializzati
```

Nella maggior parte dei casi moderni, si preferisce:

```text id="xfm0we"
Task
Task.Run
ThreadPool
BackgroundService
Parallel
Channel<T>
servizi esterni di coda
```

Regola pratica:

```text id="rctmdu"
Non creare thread manualmente come prima scelta.
Inizia da Task, async/await e hosted services.
```

---

# ThreadPool

Il **ThreadPool** è un insieme di thread gestiti dal runtime .NET.

Serve a evitare di creare e distruggere thread continuamente.

Microsoft descrive `ThreadPool` come una classe che fornisce all’applicazione un pool di worker thread gestiti dal sistema, così lo sviluppatore può concentrarsi sui task applicativi invece che sulla gestione manuale dei thread. ([Microsoft Learn][2])

Il ThreadPool è adatto a:

```text id="luuxme"
lavori brevi
operazioni background leggere
task CPU-bound non troppo lunghi
continuazioni di task
esecuzione di molte piccole unità di lavoro
```

---

## Perché il ThreadPool è utile

Creare un thread ha un costo.

Se per ogni piccola operazione creassimo un nuovo thread, sprecheremmo risorse.

Il ThreadPool:

```text id="mcfja1"
riusa thread esistenti
regola il numero di thread
riduce overhead
migliora scalabilità
gestisce code di lavoro
```

Esempio:

```csharp id="en37jj"
ThreadPool.QueueUserWorkItem(_ =>
{
    Console.WriteLine("Lavoro eseguito dal ThreadPool.");
});
```

Questo stile è possibile, ma oggi si usa più spesso `Task.Run`.

---

# Worker thread

I **worker thread** sono thread del ThreadPool usati per eseguire lavoro gestito.

Esempi:

```text id="jl90a8"
Task.Run
continuazioni di Task
codice CPU-bound
lavori accodati al ThreadPool
```

Esempio:

```csharp id="qf8dj5"
await Task.Run(() =>
{
    CalcoloPesante();
});
```

Il lavoro viene messo sul ThreadPool e uno worker thread lo esegue.

---

# I/O completion thread

Gli **I/O completion thread** sono legati al completamento di operazioni I/O asincrone.

Quando fai:

```csharp id="g0e8ws"
await httpClient.GetStringAsync(url);
```

oppure:

```csharp id="m0kb7f"
await stream.ReadAsync(buffer);
```

il punto importante è questo:

```text id="oiogdg"
non serve tenere occupato un worker thread mentre il sistema operativo
sta aspettando rete, disco o altro I/O.
```

Quando l’operazione I/O completa, il runtime può riprendere il lavoro tramite il meccanismo di completamento asincrono.

---

## Differenza pratica

CPU-bound:

```text id="xjrl89"
serve un thread che calcola
```

I/O-bound:

```text id="yz6zbl"
serve avviare l'operazione
poi attendere il completamento senza bloccare un thread
```

Per questo è sbagliato usare `Task.Run` intorno a I/O già asincrono.

Sbagliato:

```csharp id="qap81p"
await Task.Run(async () =>
{
    await dbContext.SaveChangesAsync();
});
```

Corretto:

```csharp id="tz5ve3"
await dbContext.SaveChangesAsync();
```

---

# `Task`

Una `Task` rappresenta un’operazione.

Può essere:

```text id="qq1xgf"
non ancora iniziata
in esecuzione
completata con successo
completata con errore
cancellata
```

Esempio:

```csharp id="le4hlw"
Task task = Task.Delay(1000);

await task;
```

`Task` non significa necessariamente “thread”.

Una task può rappresentare:

```text id="cu0ocl"
lavoro su ThreadPool
operazione I/O asincrona
continuazione
operazione già completata
composizione di altre task
```

Regola chiave:

```text id="kymv6u"
Task è un'astrazione di operazione asincrona.
Thread è una risorsa di esecuzione.
Non sono la stessa cosa.
```

---

# `Task.Run`

`Task.Run` accoda lavoro al ThreadPool.

Esempio:

```csharp id="8g0zuj"
Task<int> task = Task.Run(() =>
{
    return CalcoloPesante();
});

int risultato = await task;
```

È utile quando:

```text id="o9y6pc"
hai lavoro CPU-bound
sei in una UI e non vuoi bloccare il thread principale
vuoi spostare un calcolo su ThreadPool
```

Esempio WPF/WinForms:

```csharp id="h5ix65"
private async void Button_Click(object sender, EventArgs e)
{
    button.Enabled = false;

    int risultato = await Task.Run(() =>
    {
        return CalcoloPesante();
    });

    label.Text = risultato.ToString();

    button.Enabled = true;
}
```

---

## Quando non usare `Task.Run`

Non usarlo per rendere “asincrono” codice che è già asincrono.

Sbagliato:

```csharp id="o0g7va"
await Task.Run(async () =>
{
    return await httpClient.GetStringAsync(url);
});
```

Corretto:

```csharp id="y5pska"
string contenuto = await httpClient.GetStringAsync(url);
```

Non usarlo automaticamente in ASP.NET Core per ogni operazione.

Sospetto:

```csharp id="pmtgpi"
app.MapGet("/api/report", async () =>
{
    return await Task.Run(() => GeneraReport());
});
```

Può avere senso solo se `GeneraReport` è CPU-bound, breve/controllato e non hai una soluzione migliore. Per lavori lunghi è spesso meglio un job background.

---

# Task scheduler

Il **TaskScheduler** decide dove e come una task viene eseguita.

Nella maggior parte delle applicazioni, il scheduler predefinito usa il ThreadPool.

Esempio:

```csharp id="z5mt0w"
Task task = Task.Factory.StartNew(() =>
{
    Console.WriteLine("Task eseguita.");
});
```

Questa API permette più controllo rispetto a `Task.Run`, ma è anche più facile usarla male.

---

## `TaskScheduler.Current` e `TaskScheduler.Default`

```text id="c45i91"
TaskScheduler.Default
  scheduler predefinito, normalmente basato sul ThreadPool.

TaskScheduler.Current
  scheduler associato al contesto corrente.
```

In codice applicativo moderno, raramente serve interagire direttamente con `TaskScheduler`.

---

## Scheduler UI

In applicazioni desktop, può esistere un contesto UI.

Esempio concettuale:

```csharp id="qbi08q"
TaskScheduler uiScheduler =
    TaskScheduler.FromCurrentSynchronizationContext();
```

Questo serve quando una continuazione deve tornare sul thread UI.

Con `async/await`, normalmente non devi gestire manualmente tutto questo.

---

# Continuazioni

Una **continuazione** è un’operazione che parte dopo il completamento di un’altra task.

Con `await`, le continuazioni sono quasi invisibili.

Esempio:

```csharp id="6jf05w"
string contenuto = await httpClient.GetStringAsync(url);

Console.WriteLine(contenuto.Length);
```

Il codice dopo `await` è, concettualmente, una continuazione.

Microsoft definisce una continuation task come una task creata nello stato `WaitingForActivation`, attivata automaticamente quando la task precedente completa. Una continuazione è essa stessa una `Task` e non blocca il thread su cui viene avviata. ([Microsoft Learn][3])

---

# `ContinueWith`

Prima dell’uso diffuso di `async/await`, era comune scrivere continuazioni esplicite con `ContinueWith`.

Esempio:

```csharp id="innlul"
Task<int> task = Task.Run(() =>
{
    return CalcoloPesante();
});

Task continuation = task.ContinueWith(t =>
{
    Console.WriteLine($"Risultato: {t.Result}");
});
```

Questo funziona, ma è meno leggibile di:

```csharp id="wh2hdi"
int risultato = await Task.Run(() =>
{
    return CalcoloPesante();
});

Console.WriteLine($"Risultato: {risultato}");
```

---

## Gestione errori con ContinueWith

Con `ContinueWith`, bisogna controllare lo stato della task.

```csharp id="joexh3"
Task task = Task.Run(() =>
{
    throw new InvalidOperationException("Errore.");
});

Task continuation = task.ContinueWith(t =>
{
    if (t.IsFaulted)
    {
        Console.WriteLine(t.Exception?.GetBaseException().Message);
    }
});
```

Con `await`:

```csharp id="orxbeq"
try
{
    await Task.Run(() =>
    {
        throw new InvalidOperationException("Errore.");
    });
}
catch (InvalidOperationException ex)
{
    Console.WriteLine(ex.Message);
}
```

Molto più chiaro.

---

## Quando usare ContinueWith

Oggi raramente.

Può avere senso in codice:

```text id="rmlyax"
molto basso livello
librerie che compongono task manualmente
scenari TPL avanzati
codice legacy
```

Regola pratica:

```text id="f8v6ai"
Nel codice applicativo moderno, preferisci async/await.
Usa ContinueWith solo quando hai un motivo preciso.
```

---

# Task padre/figlio

La TPL supporta il concetto di task padre e task figlio.

Esempio concettuale:

```csharp id="79ghs2"
Task parent = Task.Factory.StartNew(() =>
{
    Task child = Task.Factory.StartNew(
        () =>
        {
            Console.WriteLine("Child task");
        },
        TaskCreationOptions.AttachedToParent);
});

await parent;
```

Una child task con `AttachedToParent` può essere collegata alla task padre. Questo significa che la task padre non è considerata completata finché non completano anche le child task collegate.

---

## Perché usarle poco

Nel codice moderno, le task padre/figlio sono poco usate direttamente.

Spesso è più chiaro usare:

```text id="xb3r3k"
async/await
Task.WhenAll
Task.WhenAny
Parallel
Channel<T>
BackgroundService
```

Esempio più leggibile:

```csharp id="smq1up"
Task task1 = Operazione1Async();
Task task2 = Operazione2Async();

await Task.WhenAll(task1, task2);
```

---

## DenyChildAttach

La TPL include opzioni per impedire a task figlie esterne di attaccarsi in modo inatteso a una task padre. Questo è un dettaglio avanzato, ma aiuta a capire perché la gestione manuale di parent/child task può diventare complessa.

Regola pratica:

```text id="g0z98n"
Non progettare codice applicativo ordinario intorno a task padre/figlio.
Usa composizione esplicita con await, WhenAll e servizi dedicati.
```

---

# Long-running task

Una task lunga può essere problematica se occupa un thread del ThreadPool per molto tempo.

Esempi:

```text id="fe03w9"
loop infinito
elaborazione batch lunga
servizio che ascolta una coda
worker permanente
calcolo CPU-bound di lunga durata
```

`TaskCreationOptions.LongRunning` fornisce un suggerimento allo scheduler.

Microsoft descrive `LongRunning` come un’opzione che indica una task lunga e grossolana; è un suggerimento al `TaskScheduler` che l’oversubscription può essere giustificata, cioè creare più thread rispetto ai thread hardware disponibili. ([Microsoft Learn][4])

Esempio:

```csharp id="p9utww"
Task.Factory.StartNew(
    () =>
    {
        EseguiLavoroLungo();
    },
    CancellationToken.None,
    TaskCreationOptions.LongRunning,
    TaskScheduler.Default);
```

---

## Attenzione

`LongRunning` non è una bacchetta magica.

Non significa:

```text id="vwj4sv"
questa task è più veloce
questa task è automaticamente scalabile
questa task è adatta a stare dentro una request HTTP
```

Per lavori background persistenti, spesso è meglio usare `BackgroundService`.

---

# BackgroundService

`BackgroundService` è una classe base per implementare servizi in background nel Generic Host di .NET.

Microsoft documenta che `ExecuteAsync(CancellationToken)` viene chiamato quando l’`IHostedService` parte e deve restituire una task che rappresenta la durata delle operazioni long-running eseguite. ([Microsoft Learn][5])

Esempio:

```csharp id="vb92wk"
using Microsoft.Extensions.Hosting;
using Microsoft.Extensions.Logging;

public class ReportWorker : BackgroundService
{
    private readonly ILogger<ReportWorker> logger;

    public ReportWorker(ILogger<ReportWorker> logger)
    {
        this.logger = logger;
    }

    protected override async Task ExecuteAsync(
        CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            logger.LogInformation(
                "Worker attivo alle {Time}",
                DateTimeOffset.Now);

            await Task.Delay(
                TimeSpan.FromSeconds(10),
                stoppingToken);
        }
    }
}
```

Registrazione:

```csharp id="s7hh9v"
builder.Services.AddHostedService<ReportWorker>();
```

# Hosted services

Un **hosted service** è un servizio gestito dall’host .NET.

Implementa:

```csharp id="kklfh9"
IHostedService
```

oppure deriva da:

```csharp id="b980gl"

```

Microsoft spiega che in ASP.NET Core le attività in background possono essere implementate come hosted services; un hosted service è una classe con logica background che implementa `IHostedService`. La documentazione mostra esempi di task su timer, servizi hosted che attivano servizi scoped e code di lavoro background sequenziali. ([Microsoft Learn][6])

# BackgroundService e servizi scoped

Un `BackgroundService` è registrato come hosted service e ha un ciclo di vita diverso da una request HTTP.

Se deve usare servizi scoped, come un `DbContext`, bisogna creare uno scope.

Esempio:

```csharp id="d6sm9y"
public class EmailWorker : BackgroundService
{
    private readonly IServiceScopeFactory scopeFactory;
    private readonly ILogger<EmailWorker> logger;

    public EmailWorker(
        IServiceScopeFactory scopeFactory,
        ILogger<EmailWorker> logger)
    {
        this.scopeFactory = scopeFactory;
        this.logger = logger;
    }

    protected override async Task ExecuteAsync(
        CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            using IServiceScope scope =
                scopeFactory.CreateScope();

            EmailService emailService =
                scope.ServiceProvider
                    .GetRequiredService<EmailService>();

            await emailService.InviaEmailInCodaAsync(stoppingToken);

            await Task.Delay(
                TimeSpan.FromSeconds(30),
                stoppingToken);
        }
    }
}
```

Regola pratica:

```text id="rh0awq"
Non iniettare direttamente DbContext scoped in un BackgroundService singleton.
Crea uno scope quando ti serve usare servizi scoped.
```

---

# Timer in BackgroundService

Per lavori periodici moderni, si può usare `PeriodicTimer`.

```csharp id="h9r6re"
public class PuliziaWorker : BackgroundService
{
    private readonly ILogger<PuliziaWorker> logger;

    public PuliziaWorker(ILogger<PuliziaWorker> logger)
    {
        this.logger = logger;
    }

    protected override async Task ExecuteAsync(
        CancellationToken stoppingToken)
    {
        using PeriodicTimer timer =
            new PeriodicTimer(TimeSpan.FromMinutes(5));

        while (await timer.WaitForNextTickAsync(stoppingToken))
        {
            logger.LogInformation("Eseguo pulizia.");

            await EseguiPuliziaAsync(stoppingToken);
        }
    }

    private Task EseguiPuliziaAsync(
        CancellationToken cancellationToken)
    {
        return Task.CompletedTask;
    }
}
```

Vantaggi:

```text id="r3psvz"
codice leggibile
cancellazione integrata
niente callback sovrapposte se gestito nel loop
```

# Code background

Per evitare fire-and-forget fragile, possiamo accodare lavoro.

Interfaccia:

```csharp id="m2bhss"
public interface IBackgroundTaskQueue
{
    ValueTask QueueAsync(
        Func<CancellationToken, ValueTask> workItem);

    ValueTask<Func<CancellationToken, ValueTask>> DequeueAsync(
        CancellationToken cancellationToken);
}
```

Implementazione con `Channel<T>`:

```csharp id="5m0kfu"
using System.Threading.Channels;

public class BackgroundTaskQueue : IBackgroundTaskQueue
{
    private readonly Channel<Func<CancellationToken, ValueTask>> queue;

    public BackgroundTaskQueue(int capacity)
    {
        BoundedChannelOptions options = new(capacity)
        {
            FullMode = BoundedChannelFullMode.Wait
        };

        queue = Channel.CreateBounded<Func<CancellationToken, ValueTask>>(
            options);
    }

    public async ValueTask QueueAsync(
        Func<CancellationToken, ValueTask> workItem)
    {
        ArgumentNullException.ThrowIfNull(workItem);

        await queue.Writer.WriteAsync(workItem);
    }

    public async ValueTask<Func<CancellationToken, ValueTask>> DequeueAsync(
        CancellationToken cancellationToken)
    {
        return await queue.Reader.ReadAsync(cancellationToken);
    }
}
```

Worker:

```csharp id="4s1cp9"
public class QueuedHostedService : BackgroundService
{
    private readonly IBackgroundTaskQueue taskQueue;
    private readonly ILogger<QueuedHostedService> logger;

    public QueuedHostedService(
        IBackgroundTaskQueue taskQueue,
        ILogger<QueuedHostedService> logger)
    {
        this.taskQueue = taskQueue;
        this.logger = logger;
    }

    protected override async Task ExecuteAsync(
        CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            Func<CancellationToken, ValueTask> workItem =
                await taskQueue.DequeueAsync(stoppingToken);

            try
            {
                await workItem(stoppingToken);
            }
            catch (Exception ex)
            {
                logger.LogError(
                    ex,
                    "Errore durante esecuzione lavoro background.");
            }
        }
    }
}
```

Registrazione:

```csharp id="xi2uu2"
builder.Services.AddSingleton<IBackgroundTaskQueue>(
    new BackgroundTaskQueue(capacity: 100));

builder.Services.AddHostedService<QueuedHostedService>();
```

Questo è più robusto di un fire-and-forget disperso negli endpoint.

# Task scheduler e BackgroundService: differenza

`TaskScheduler`:

```text id="hb67vf"
decide come eseguire task
è un meccanismo TPL
di solito non lo tocchi nel codice applicativo
```

`BackgroundService`:

```text id="m8jfhb"
è un servizio gestito dall'host
ha lifecycle Start/Stop
riceve CancellationToken
è integrato con DI, logging e hosting
```

Per lavori applicativi background in ASP.NET Core o Worker Service, scegli `BackgroundService`.

# Esempio completo: generazione report in background

Scenario:

```text id="ytdbna"
L'utente chiede la generazione di un report.
L'API risponde subito 202 Accepted.
Un worker genera il report in background.
```

Request:

```csharp id="ar5x2j"
public record CreaReportRequest(
    DateOnly DataDa,
    DateOnly DataA);
```

Job:

```csharp id="n436qt"
public record ReportJob(
    Guid Id,
    DateOnly DataDa,
    DateOnly DataA);
```

Coda:

```csharp id="knq1pe"
public interface IReportQueue
{
    ValueTask EnqueueAsync(
        ReportJob job,
        CancellationToken cancellationToken);

    ValueTask<ReportJob> DequeueAsync(
        CancellationToken cancellationToken);
}
```

Implementazione:

```csharp id="poaovf"
using System.Threading.Channels;

public class ReportQueue : IReportQueue
{
    private readonly Channel<ReportJob> channel =
        Channel.CreateBounded<ReportJob>(100);

    public ValueTask EnqueueAsync(
        ReportJob job,
        CancellationToken cancellationToken)
    {
        return channel.Writer.WriteAsync(job, cancellationToken);
    }

    public ValueTask<ReportJob> DequeueAsync(
        CancellationToken cancellationToken)
    {
        return channel.Reader.ReadAsync(cancellationToken);
    }
}
```

Worker:

```csharp id="97ddzv"
public class ReportWorker : BackgroundService
{
    private readonly IReportQueue queue;
    private readonly IServiceScopeFactory scopeFactory;
    private readonly ILogger<ReportWorker> logger;

    public ReportWorker(
        IReportQueue queue,
        IServiceScopeFactory scopeFactory,
        ILogger<ReportWorker> logger)
    {
        this.queue = queue;
        this.scopeFactory = scopeFactory;
        this.logger = logger;
    }

    protected override async Task ExecuteAsync(
        CancellationToken stoppingToken)
    {
        while (!stoppingToken.IsCancellationRequested)
        {
            ReportJob job = await queue.DequeueAsync(stoppingToken);

            try
            {
                using IServiceScope scope =
                    scopeFactory.CreateScope();

                ReportService reportService =
                    scope.ServiceProvider
                        .GetRequiredService<ReportService>();

                await reportService.GeneraAsync(
                    job,
                    stoppingToken);

                logger.LogInformation(
                    "Report {ReportId} generato.",
                    job.Id);
            }
            catch (OperationCanceledException)
            {
                logger.LogInformation(
                    "Generazione report annullata.");
                throw;
            }
            catch (Exception ex)
            {
                logger.LogError(
                    ex,
                    "Errore durante generazione report {ReportId}",
                    job.Id);
            }
        }
    }
}
```

Endpoint:

```csharp id="4bnryg"
app.MapPost("/api/report", async (
    CreaReportRequest request,
    IReportQueue queue,
    CancellationToken cancellationToken) =>
{
    ReportJob job = new ReportJob(
        Guid.NewGuid(),
        request.DataDa,
        request.DataA);

    await queue.EnqueueAsync(job, cancellationToken);

    return Results.Accepted(
        $"/api/report/{job.Id}",
        new
        {
            job.Id,
            Stato = "Accodato"
        });
});
```

Registrazione:

```csharp id="3o3ffb"

builder.Services.AddScoped<ReportService>();
builder.Services.AddHostedService<ReportWorker>();
```

Questo schema evita di bloccare la request HTTP durante la generazione.

# Linee guida pratiche

```text id="u91l15"
Usa async/await
  per I/O asincrono.

Usa Task.Run
  solo per CPU-bound quando ha senso.

Non creare thread manuali
  salvo casi speciali.

Ricorda che Task non è Thread
  è un'astrazione di operazione.

Lascia il ThreadPool libero
  non bloccarlo con sleep o wait inutili.

Preferisci await
  a ContinueWith nel codice moderno.

Usa Task.WhenAll
  per operazioni asincrone indipendenti.

Usa BackgroundService
  per lavori background controllati dall'host.

Usa CancellationToken
  in worker e operazioni lunghe.

Crea scope
  quando un worker deve usare servizi scoped.

Usa code
  per spostare lavoro fuori dalla request.

Persisti i job critici
  se non puoi permetterti di perderli.
```

# Riepilogo

In questo capitolo abbiamo visto cosa succede sotto `async/await`.

Abbiamo studiato:

```text id="h5o4lx"
Thread
ThreadPool
worker thread
I/O completion thread
Task.Run
Task scheduler
continuazioni
ContinueWith
task padre/figlio
long-running task
BackgroundService
hosted services
```

Abbiamo chiarito che `Task` è un’astrazione di operazione asincrona, non un sinonimo di thread. La TPL usa task per rappresentare lavoro asincrono e concorrente a un livello più alto rispetto alla gestione diretta dei thread. ([Microsoft Learn][1])

Abbiamo visto che il ThreadPool fornisce worker thread gestiti dal runtime ed è adatto a lavori brevi e background processing leggero, mentre per attività applicative lunghe e controllate dall’host è più appropriato usare hosted services e `BackgroundService`. ([Microsoft Learn][2])

Abbiamo anche visto che `LongRunning` è solo un suggerimento al `TaskScheduler`, non una soluzione generale per job applicativi complessi. ([Microsoft Learn][4])

La regola principale è questa:

```text id="s3uqla"
Non scegliere lo strumento perché “sembra asincrono”.
Sceglilo in base al tipo di lavoro:
I/O-bound, CPU-bound, breve, lungo, critico, cancellabile o persistente.
```

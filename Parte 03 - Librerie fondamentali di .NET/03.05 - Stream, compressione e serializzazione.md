# 03.05 — Stream, compressione e serializzazione

Nel capitolo precedente abbiamo lavorato con file, directory e percorsi. Abbiamo visto come leggere e scrivere file di testo, creare cartelle, gestire encoding e intercettare errori del file system.

Ora facciamo un passo più profondo.

Quando leggiamo un file, scarichiamo dati da una rete, comprimiamo contenuti o serializziamo oggetti, spesso non lavoriamo direttamente con “file” o “stringhe”, ma con **stream**.

Uno stream rappresenta un flusso di dati.

Può arrivare da:

* un file;
* memoria;
* una connessione di rete;
* un archivio compresso;
* una richiesta HTTP;
* una risposta HTTP;
* un buffer;
* una pipeline.

Il punto centrale del capitolo è questo:

```text
Uno stream astrae la sorgente dei dati.
Il codice può leggere o scrivere byte senza sapere sempre se quei byte arrivano da un file, dalla memoria, dalla rete o da un archivio compresso.
```

---

# Concetto di stream

Uno **stream** è una sequenza di byte letta o scritta progressivamente.

In .NET la classe base è:

```csharp
System.IO.Stream
```

La documentazione Microsoft descrive `Stream` come la classe base astratta per una sequenza di byte e indica che supporta operazioni sincrone e asincrone come `ReadAsync`, `WriteAsync`, `CopyToAsync` e `FlushAsync`. ([Microsoft Learn][1])

Uno stream può essere:

```text
leggibile
  possiamo leggere dati.

scrivibile
  possiamo scrivere dati.

posizionabile
  possiamo spostarci avanti e indietro.

non posizionabile
  possiamo solo leggere o scrivere progressivamente.
```

Esempi:

```text
FileStream
  stream collegato a un file.

MemoryStream
  stream basato su memoria.

NetworkStream
  stream collegato a una connessione di rete.

BrotliStream
  stream che comprime o decomprime.

CryptoStream
  stream che cifra o decifra.
```

---

## Stream come astrazione

Immagina un metodo che copia dati da uno stream a un altro:

```csharp
async Task CopiaAsync(Stream sorgente, Stream destinazione)
{
    await sorgente.CopyToAsync(destinazione);
}
```

Questo metodo non sa se `sorgente` è:

```text
un file
un buffer in memoria
una risposta HTTP
uno stream compresso
```

E non deve saperlo.

Questa è la forza degli stream.

---

# `FileStream`

`FileStream` rappresenta uno stream collegato a un file. Microsoft lo definisce come uno stream per file che supporta lettura e scrittura sia sincrone sia asincrone. ([Microsoft Learn][2])

Esempio di scrittura:

```csharp
using FileStream stream = new FileStream(
    "dati.bin",
    FileMode.Create,
    FileAccess.Write);

byte[] dati = [1, 2, 3, 4, 5];

stream.Write(dati, 0, dati.Length);
```

Esempio di lettura:

```csharp
using FileStream stream = new FileStream(
    "dati.bin",
    FileMode.Open,
    FileAccess.Read);

byte[] buffer = new byte[stream.Length];

int letti = stream.Read(buffer, 0, buffer.Length);

Console.WriteLine($"Byte letti: {letti}");
```

---

## FileMode

`FileMode` indica come aprire o creare il file.

Valori comuni:

```text
Create
  crea un nuovo file o sovrascrive quello esistente.

CreateNew
  crea un nuovo file, ma fallisce se esiste già.

Open
  apre un file esistente.

OpenOrCreate
  apre se esiste, altrimenti crea.

Append
  apre il file e scrive in fondo.
```

Esempio:

```csharp
using FileStream stream = new FileStream(
    "log.bin",
    FileMode.Append,
    FileAccess.Write);
```

---

## FileAccess

`FileAccess` indica cosa vogliamo fare.

```text
Read
  solo lettura.

Write
  solo scrittura.

ReadWrite
  lettura e scrittura.
```

Esempio:

```csharp
using FileStream stream = new FileStream(
    "dati.bin",
    FileMode.Open,
    FileAccess.Read);
```

---

## FileShare

`FileShare` controlla cosa possono fare altri processi mentre il file è aperto.

Esempio:

```csharp
using FileStream stream = new FileStream(
    "dati.txt",
    FileMode.Open,
    FileAccess.Read,
    FileShare.Read);
```

Qui permettiamo ad altri processi di leggere il file mentre lo stiamo leggendo.

# `MemoryStream`

`MemoryStream` rappresenta uno stream basato su memoria.

È utile quando vogliamo lavorare con dati come se fossero uno stream, ma senza passare dal file system.

Esempio:

```csharp
using MemoryStream stream = new MemoryStream();

byte[] dati = [10, 20, 30];

stream.Write(dati, 0, dati.Length);

byte[] risultato = stream.ToArray();

Console.WriteLine(string.Join(", ", risultato));
```

Output:

```text
10, 20, 30
```

---

## Quando usare MemoryStream

`MemoryStream` è utile per:

* generare file in memoria;
* preparare contenuti da inviare via HTTP;
* serializzare oggetti senza scrivere subito su disco;
* comprimere dati in memoria;
* testare codice che lavora con stream;
* convertire tra byte array e API basate su stream.

Esempio:

```csharp
byte[] CreaFileInMemoria()
{
    using MemoryStream stream = new MemoryStream();
    using StreamWriter writer = new StreamWriter(stream, leaveOpen: true);

    writer.WriteLine("Report");
    writer.WriteLine("======");
    writer.Flush();

    return stream.ToArray();
}
```

Nota importante: `StreamWriter` ha un buffer interno. Per questo chiamiamo `Flush()` prima di leggere i byte dal `MemoryStream`.

---

# `StreamReader`

`StreamReader` legge caratteri da uno stream.

È utile quando lo stream contiene testo.

```csharp
using FileStream fileStream = new FileStream(
    "dati.txt",
    FileMode.Open,
    FileAccess.Read);

using StreamReader reader = new StreamReader(fileStream, Encoding.UTF8);

string contenuto = reader.ReadToEnd();

Console.WriteLine(contenuto);
```

`StreamReader` può essere creato specificando stream, encoding e opzioni di rilevamento BOM. ([Microsoft Learn][3])

---

## Leggere riga per riga

```csharp
using StreamReader reader = new StreamReader("dati.txt", Encoding.UTF8);

string? riga;

while ((riga = reader.ReadLine()) is not null)
{
    Console.WriteLine(riga);
}
```

Questa forma è utile per file grandi.

---

# `StreamWriter`

`StreamWriter` scrive caratteri su uno stream.

Esempio:

```csharp
using FileStream fileStream = new FileStream(
    "report.txt",
    FileMode.Create,
    FileAccess.Write);

using StreamWriter writer = new StreamWriter(fileStream, Encoding.UTF8);

writer.WriteLine("Report clienti");
writer.WriteLine("==============");
writer.WriteLine("Totale: 100");
```

`StreamWriter` può scrivere su file o stream e permette di specificare encoding, modalità append e altre opzioni. ([Microsoft Learn][4])

---

## `leaveOpen`

A volte vogliamo chiudere il writer senza chiudere lo stream sottostante.

Esempio:

```csharp
using MemoryStream stream = new MemoryStream();

using (StreamWriter writer = new StreamWriter(
    stream,
    Encoding.UTF8,
    leaveOpen: true))
{
    writer.WriteLine("Contenuto");
}

stream.Position = 0;

using StreamReader reader = new StreamReader(stream, Encoding.UTF8);

string testo = reader.ReadToEnd();

Console.WriteLine(testo);
```

`leaveOpen: true` impedisce a `StreamWriter` di chiudere anche il `MemoryStream`.

---

# Operazioni async su stream

Gli stream supportano operazioni asincrone.

Esempio:

```csharp
await using FileStream sorgente = new FileStream(
    "input.bin",
    FileMode.Open,
    FileAccess.Read);

await using FileStream destinazione = new FileStream(
    "output.bin",
    FileMode.Create,
    FileAccess.Write);

await sorgente.CopyToAsync(destinazione);
```

Le operazioni asincrone sono importanti quando l’I/O può richiedere tempo:

* file grandi;
* rete;
* dischi lenti;
* storage remoto;
* upload;
* download;
* web server.

---

## Lettura async testo

```csharp
using StreamReader reader = new StreamReader("dati.txt", Encoding.UTF8);

string contenuto = await reader.ReadToEndAsync();

Console.WriteLine(contenuto);
```

---

## Scrittura async testo

```csharp
using StreamWriter writer = new StreamWriter("output.txt", false, Encoding.UTF8);

await writer.WriteLineAsync("Prima riga");
await writer.WriteLineAsync("Seconda riga");
```

---

## Perché async

In un’app desktop, l’async evita di bloccare la UI.

In un’app web, l’async evita di tenere occupato un thread mentre il processo aspetta I/O.

Regola pratica:

```text
CPU-bound
  async non rende il calcolo più veloce.

I/O-bound
  async evita di bloccare thread mentre aspettiamo file, rete o database.
```

# Compressione ZIP

ZIP è un formato molto comune per archivi contenenti uno o più file.

In .NET si usa il namespace:

```csharp
System.IO.Compression
```

`ZipArchive` permette di lavorare con archivi ZIP; Microsoft documenta operazioni come creare un archivio da una directory, estrarre contenuti, aggiungere file, recuperare entry e aprire lo stream di un file dentro l’archivio. ([Microsoft Learn][5])

---

## Creare uno ZIP da una cartella

```csharp
using System.IO.Compression;

string cartellaSorgente = "Export";
string fileZip = "export.zip";

if (File.Exists(fileZip))
{
    File.Delete(fileZip);
}

ZipFile.CreateFromDirectory(cartellaSorgente, fileZip);
```

---

## Estrarre uno ZIP

```csharp
using System.IO.Compression;

string fileZip = "export.zip";
string cartellaDestinazione = "Estratto";

ZipFile.ExtractToDirectory(fileZip, cartellaDestinazione);
```

---

## Creare uno ZIP manualmente

```csharp
using System.IO.Compression;

using FileStream zipStream = new FileStream(
    "documenti.zip",
    FileMode.Create);

using ZipArchive archive = new ZipArchive(
    zipStream,
    ZipArchiveMode.Create);

ZipArchiveEntry entry = archive.CreateEntry("report.txt");

using Stream entryStream = entry.Open();
using StreamWriter writer = new StreamWriter(entryStream, Encoding.UTF8);

writer.WriteLine("Contenuto del report");
```

---

## Leggere un file dentro uno ZIP

```csharp
using System.IO.Compression;

using ZipArchive archive = ZipFile.OpenRead("documenti.zip");

ZipArchiveEntry? entry = archive.GetEntry("report.txt");

if (entry is not null)
{
    using Stream stream = entry.Open();
    using StreamReader reader = new StreamReader(stream, Encoding.UTF8);

    string contenuto = reader.ReadToEnd();

    Console.WriteLine(contenuto);
}
```

# Brotli

Brotli è un algoritmo di compressione moderno, spesso usato nel web per comprimere contenuti testuali.

In .NET si usa:

```csharp
System.IO.Compression.BrotliStream
```

`BrotliStream` fornisce metodi e proprietà per comprimere e decomprimere stream usando il formato Brotli. ([Microsoft Learn][6])

---

## Comprimere con Brotli

```csharp
using System.IO.Compression;
using System.Text;

string testo = "Questo è un testo da comprimere con Brotli.";
byte[] dati = Encoding.UTF8.GetBytes(testo);

using MemoryStream output = new MemoryStream();

using (BrotliStream brotli = new BrotliStream(
    output,
    CompressionLevel.Optimal,
    leaveOpen: true))
{
    brotli.Write(dati, 0, dati.Length);
}

byte[] compressi = output.ToArray();

Console.WriteLine($"Originale: {dati.Length} byte");
Console.WriteLine($"Compresso: {compressi.Length} byte");
```

---

## Decomprimere con Brotli

```csharp
using MemoryStream input = new MemoryStream(compressi);
using BrotliStream brotli = new BrotliStream(input, CompressionMode.Decompress);
using MemoryStream output = new MemoryStream();

brotli.CopyTo(output);

string testoDecompresso = Encoding.UTF8.GetString(output.ToArray());

Console.WriteLine(testoDecompresso);
```

---

## ZIP o Brotli?

```text
ZIP
  archivio di uno o più file, con nomi, directory e metadati.

Brotli
  compressione di uno stream di dati.
```

Esempio:

```text
Devo creare un archivio con 10 documenti
  ZIP.

Devo comprimere una risposta HTTP o un payload testuale
  Brotli.
```

# JSON con `System.Text.Json`

JSON è uno dei formati più usati per scambiare dati tra applicazioni.

Esempio JSON:

```json
{
  "id": 1,
  "nome": "Emilie",
  "email": "emilie@example.com"
}
```

In .NET moderno la libreria principale è:

```csharp
System.Text.Json
```

Microsoft descrive `System.Text.Json` come il namespace che fornisce funzionalità per serializzare e deserializzare JSON; la serializzazione converte lo stato di un oggetto in una forma memorizzabile o trasmissibile, mentre la deserializzazione ricostruisce l’oggetto dalla forma serializzata. ([Microsoft Learn][7])

---

# Serializzare oggetti

Serializzare significa trasformare un oggetto in un formato trasferibile o salvabile.

DTO:

```csharp
public record ClienteDto(
    int Id,
    string Nome,
    string? Email);
```

Serializzazione:

```csharp
using System.Text.Json;

ClienteDto cliente = new ClienteDto(
    1,
    "Emilie",
    "emilie@example.com");

string json = JsonSerializer.Serialize(cliente);

Console.WriteLine(json);
```

Output:

```json
{"Id":1,"Nome":"Emilie","Email":"emilie@example.com"}
```

---

## JSON indentato

```csharp
JsonSerializerOptions options = new JsonSerializerOptions
{
    WriteIndented = true
};

string json = JsonSerializer.Serialize(cliente, options);

Console.WriteLine(json);
```

Output:

```json
{
  "Id": 1,
  "Nome": "Emilie",
  "Email": "emilie@example.com"
}
```

---

## Nomi proprietà camelCase

Molte API JSON usano `camelCase`.

```csharp
JsonSerializerOptions options = new JsonSerializerOptions
{
    PropertyNamingPolicy = JsonNamingPolicy.CamelCase,
    WriteIndented = true
};

string json = JsonSerializer.Serialize(cliente, options);
```

Output:

```json
{
  "id": 1,
  "nome": "Emilie",
  "email": "emilie@example.com"
}
```

---

# Deserializzare oggetti

Deserializzare significa trasformare JSON in oggetti .NET.

```csharp
string json = """
{
  "id": 1,
  "nome": "Emilie",
  "email": "emilie@example.com"
}
""";

JsonSerializerOptions options = new JsonSerializerOptions
{
    PropertyNameCaseInsensitive = true
};

ClienteDto? cliente = JsonSerializer.Deserialize<ClienteDto>(
    json,
    options);

if (cliente is not null)
{
    Console.WriteLine(cliente.Nome);
}
```

Output:

```text
Emilie
```

Microsoft indica che, per deserializzare JSON, un approccio comune consiste nel creare una classe .NET con proprietà che rappresentano le proprietà JSON, poi chiamare `JsonSerializer.Deserialize`; sono disponibili overload sincroni e asincroni. ([Microsoft Learn][8])


# XML

XML è un formato più verboso di JSON, ma ancora molto presente in:

* sistemi legacy;
* integrazioni enterprise;
* SOAP;
* file di configurazione storici;
* tracciati bancari o amministrativi;
* documenti validati da XSD.

Esempio XML:

```xml
<Cliente>
  <Id>1</Id>
  <Nome>Emilie</Nome>
  <Email>emilie@example.com</Email>
</Cliente>
```

In .NET, per serializzare e deserializzare oggetti XML si può usare:

```csharp
System.Xml.Serialization.XmlSerializer
```

`XmlSerializer` serializza e deserializza oggetti in e da documenti XML e permette di controllare come gli oggetti vengono codificati in XML. ([Microsoft Learn][9])

---

# Serializzare XML

DTO XML:

```csharp
public class ClienteXmlDto
{
    public int Id { get; set; }
    public string Nome { get; set; } = "";
    public string? Email { get; set; }
}
```

Serializzazione:

```csharp
using System.Xml.Serialization;

ClienteXmlDto cliente = new ClienteXmlDto
{
    Id = 1,
    Nome = "Emilie",
    Email = "emilie@example.com"
};

XmlSerializer serializer = new XmlSerializer(typeof(ClienteXmlDto));

using FileStream stream = new FileStream(
    "cliente.xml",
    FileMode.Create,
    FileAccess.Write);

serializer.Serialize(stream, cliente);
```

---

# Deserializzare XML

```csharp
using System.Xml.Serialization;

XmlSerializer serializer = new XmlSerializer(typeof(ClienteXmlDto));

using FileStream stream = new FileStream(
    "cliente.xml",
    FileMode.Open,
    FileAccess.Read);

ClienteXmlDto? cliente = serializer.Deserialize(stream) as ClienteXmlDto;

Console.WriteLine(cliente?.Nome);
```

---

## XML e costruttore vuoto

`XmlSerializer` lavora bene con classi pubbliche che hanno proprietà pubbliche leggibili e scrivibili e un costruttore pubblico senza parametri.

Per questo, per XML, spesso si usano DTO mutabili:

```csharp
public class ClienteXmlDto
{
    public int Id { get; set; }
    public string Nome { get; set; } = "";
}
```

Invece un record posizionale immutabile può essere meno comodo con `XmlSerializer`.

# DTO

DTO significa:

```text
Data Transfer Object
```

Un DTO è un oggetto pensato per trasferire dati.

Esempio:

```csharp
public record ClienteDto(
    int Id,
    string Nome,
    string? Email);
```

Un DTO non dovrebbe contenere logica complessa.

Serve a rappresentare dati in modo semplice per:

* API;
* serializzazione;
* UI;
* messaggi;
* import/export;
* integrazioni.

---

## DTO vs modello di dominio

Modello di dominio:

```csharp
public class Cliente
{
    public int Id { get; }
    public string Nome { get; }

    public Cliente(int id, string nome)
    {
        if (id <= 0)
        {
            throw new ArgumentOutOfRangeException(nameof(id));
        }

        ArgumentException.ThrowIfNullOrWhiteSpace(nome);

        Id = id;
        Nome = nome;
    }
}
```

DTO:

```csharp
public record ClienteDto(
    int Id,
    string Nome);
```

Il dominio protegge invarianti.
Il DTO trasporta dati.

---

## Conversione tra dominio e DTO

```csharp
public static ClienteDto ToDto(Cliente cliente)
{
    return new ClienteDto(
        cliente.Id,
        cliente.Nome);
}
```

E da DTO a dominio:

```csharp
public static Cliente ToDomain(ClienteDto dto)
{
    return new Cliente(
        dto.Id,
        dto.Nome);
}
```

Durante la conversione verso il dominio, le validazioni tornano importanti.

# Versionamento dei dati serializzati

Quando salviamo o scambiamo JSON/XML, creiamo un contratto.

Quel contratto può vivere a lungo.

Esempio versione 1:

```json
{
  "id": 1,
  "nome": "Emilie"
}
```

Versione 2:

```json
{
  "id": 1,
  "nome": "Emilie",
  "email": "emilie@example.com"
}
```

Aggiungere un campo opzionale di solito è più semplice che rinominare o rimuovere campi.

---

## Problemi comuni di versionamento

```text
Rinominare proprietà
  può rompere chi legge il vecchio nome.

Cambiare tipo
  da string a number può rompere la deserializzazione.

Rimuovere proprietà
  può rompere client vecchi.

Rendere obbligatorio un campo prima opzionale
  può rompere dati già salvati.

Cambiare semantica
  il campo esiste ancora, ma significa un’altra cosa.
```

---

## Aggiungere proprietà opzionali

DTO v1:

```csharp
public record ClienteDto(
    int Id,
    string Nome);
```

DTO v2:

```csharp
public record ClienteDto(
    int Id,
    string Nome,
    string? Email);
```

Se `Email` è opzionale, i vecchi JSON possono ancora essere letti.

---

## Inserire una versione nel payload

Per dati persistiti o messaggi importanti, può essere utile includere una versione.

```csharp
public record ClienteExportDto(
    int Versione,
    int Id,
    string Nome,
    string? Email);
```

JSON:

```json
{
  "versione": 2,
  "id": 1,
  "nome": "Emilie",
  "email": "emilie@example.com"
}
```

L’importatore può decidere come comportarsi:

```csharp
if (dto.Versione == 1)
{
    // migrazione vecchio formato
}
else if (dto.Versione == 2)
{
    // formato attuale
}
else
{
    throw new NotSupportedException($"Versione non supportata: {dto.Versione}");
}
```

---

## Linee guida per il versionamento

```text
Aggiungi campi opzionali invece di rinominare campi esistenti.

Evita di cambiare il significato di un campo mantenendo lo stesso nome.

Non rendere obbligatorio un campo se esistono già dati vecchi senza quel campo.

Mantieni DTO separati dal dominio quando il contratto esterno deve restare stabile.

Valuta una proprietà Versione nei file o messaggi persistenti.
```

# Esempio completo: salvare e comprimere un export JSON

Scenario:

```text
Abbiamo un elenco di clienti.
Li serializziamo in JSON.
Comprimiamo il JSON con Brotli.
Salviamo il risultato su file.
Poi rileggiamo, decomprimiamo e deserializziamo.
```

DTO:

```csharp
public record ClienteExportDto(
    int Versione,
    int Id,
    string Nome,
    string? Email);
```

Codice:

```csharp
using System.IO.Compression;
using System.Text.Json;

List<ClienteExportDto> clienti =
[
    new ClienteExportDto(1, 1, "Emilie", "emilie@example.com"),
    new ClienteExportDto(1, 2, "Mario", null),
    new ClienteExportDto(1, 3, "Anna", "anna@example.com")
];

JsonSerializerOptions options = new JsonSerializerOptions
{
    WriteIndented = true,
    PropertyNamingPolicy = JsonNamingPolicy.CamelCase
};

await using FileStream fileStream = new FileStream(
    "clienti.json.br",
    FileMode.Create,
    FileAccess.Write);

await using (BrotliStream brotliStream = new BrotliStream(
    fileStream,
    CompressionLevel.Optimal))
{
    await JsonSerializer.SerializeAsync(
        brotliStream,
        clienti,
        options);
}
```

Lettura:

```csharp
await using FileStream fileStream = new FileStream(
    "clienti.json.br",
    FileMode.Open,
    FileAccess.Read);

await using BrotliStream brotliStream = new BrotliStream(
    fileStream,
    CompressionMode.Decompress);

List<ClienteExportDto>? clientiLetti =
    await JsonSerializer.DeserializeAsync<List<ClienteExportDto>>(
        brotliStream,
        options);

if (clientiLetti is not null)
{
    foreach (ClienteExportDto cliente in clientiLetti)
    {
        Console.WriteLine($"{cliente.Id}: {cliente.Nome}");
    }
}
```

In questo esempio:

```text
JsonSerializer
  trasforma oggetti in JSON e JSON in oggetti.

BrotliStream
  comprime e decomprime lo stream.

FileStream
  salva e legge i byte da file.

DTO
  rappresenta il contratto serializzato.

Versione
  prepara il formato a evoluzioni future.
```

# Esempio completo: creare ZIP con più JSON

Scenario:

```text
Vogliamo creare uno ZIP.
Dentro lo ZIP mettiamo un file JSON per ogni cliente.
```

```csharp
using System.IO.Compression;
using System.Text.Json;

public record ClienteDto(int Id, string Nome, string? Email);

List<ClienteDto> clienti =
[
    new ClienteDto(1, "Emilie", "emilie@example.com"),
    new ClienteDto(2, "Mario", null)
];

JsonSerializerOptions options = new JsonSerializerOptions
{
    WriteIndented = true,
    PropertyNamingPolicy = JsonNamingPolicy.CamelCase
};

using FileStream zipFile = new FileStream(
    "clienti.zip",
    FileMode.Create);

using ZipArchive archive = new ZipArchive(
    zipFile,
    ZipArchiveMode.Create);

foreach (ClienteDto cliente in clienti)
{
    ZipArchiveEntry entry = archive.CreateEntry(
        $"cliente_{cliente.Id}.json");

    await using Stream entryStream = entry.Open();

    await JsonSerializer.SerializeAsync(
        entryStream,
        cliente,
        options);
}
```

Lettura:

```csharp
using ZipArchive archive = ZipFile.OpenRead("clienti.zip");

foreach (ZipArchiveEntry entry in archive.Entries)
{
    await using Stream entryStream = entry.Open();

    ClienteDto? cliente = await JsonSerializer.DeserializeAsync<ClienteDto>(
        entryStream,
        options);

    if (cliente is not null)
    {
        Console.WriteLine($"{entry.Name}: {cliente.Nome}");
    }
}
```

# Errori comuni

## Dimenticare la posizione dello stream

Dopo aver scritto in un `MemoryStream`, la posizione è alla fine.

Sbagliato:

```csharp
using MemoryStream stream = new MemoryStream();

using StreamWriter writer = new StreamWriter(stream, leaveOpen: true);
writer.WriteLine("Testo");
writer.Flush();

using StreamReader reader = new StreamReader(stream);
string testo = reader.ReadToEnd(); // legge da fine stream
```

Corretto:

```csharp
stream.Position = 0;

using StreamReader reader = new StreamReader(stream);
string testo = reader.ReadToEnd();
```

---

## Dimenticare `Flush`

Con writer e stream compressi, i dati possono essere bufferizzati.

Sbagliato:

```csharp
writer.WriteLine("Testo");
byte[] dati = memoryStream.ToArray();
```

Meglio:

```csharp
writer.WriteLine("Testo");
writer.Flush();

byte[] dati = memoryStream.ToArray();
```

Oppure chiudere/disporre il writer prima di leggere il contenuto.

---

## Confondere testo e byte

Uno stream lavora con byte.

`StreamReader` e `StreamWriter` trasformano byte in caratteri e viceversa usando un encoding.

Regola pratica:

```text
Stream
  byte.

StreamReader / StreamWriter
  testo.

Encoding
  conversione tra byte e testo.
```

---

## Serializzare direttamente il dominio

Sconsigliato:

```csharp
JsonSerializer.Serialize(clienteDominio);
```

se `clienteDominio` contiene logica, invarianti, proprietà calcolate o relazioni complesse.

Meglio usare un DTO stabile:

```csharp
JsonSerializer.Serialize(clienteDto);
```

---

## Ignorare il versionamento

Sbagliato:

```text
Salvo JSON oggi e penso che la struttura non cambierà mai.
```

Meglio prevedere evoluzione:

```csharp
public record ExportDto(
    int Versione,
    IReadOnlyList<ClienteDto> Clienti);
```

---

## Usare XML quando JSON è più adatto, o viceversa

```text
JSON
  ottimo per API web moderne, configurazioni leggere, messaggi.

XML
  utile per sistemi legacy, standard esistenti, documenti con schema XSD, integrazioni enterprise.
```

La scelta dipende dal contratto esterno, non solo dal gusto personale.

---

## Non gestire eccezioni di deserializzazione

JSON malformato, XML non valido o dati incompatibili possono generare eccezioni.

Esempio:

```csharp
try
{
    ClienteDto? cliente = JsonSerializer.Deserialize<ClienteDto>(json);
}
catch (JsonException ex)
{
    Console.WriteLine("JSON non valido.");
    Console.Error.WriteLine(ex);
}
```

# Linee guida pratiche

```text
Usa Stream
  quando vuoi astrarre la sorgente o destinazione dei byte.

Usa FileStream
  per leggere e scrivere file a basso livello.

Usa MemoryStream
  per lavorare in memoria con API basate su stream.

Usa StreamReader / StreamWriter
  per testo.

Usa async
  per I/O potenzialmente lento, soprattutto in UI e server.

Usa ZIP
  per archivi con più file.

Usa Brotli
  per comprimere singoli payload o stream.

Usa System.Text.Json
  per JSON moderno in .NET.

Usa XmlSerializer
  per XML quando il contratto lo richiede.

Usa DTO
  per separare formato serializzato e modello di dominio.

Pensa al versionamento
  se il dato verrà salvato o scambiato nel tempo.
```

# Riepilogo

In questo capitolo abbiamo collegato file, memoria, compressione e serializzazione attraverso il concetto di stream.

Abbiamo visto:

```text
Stream
  astrazione di una sequenza di byte.

FileStream
  stream collegato a un file.

MemoryStream
  stream basato su memoria.

StreamReader
  lettura testuale da stream.

StreamWriter
  scrittura testuale su stream.

Operazioni async
  lettura e scrittura senza bloccare thread durante I/O.
```

Abbiamo introdotto la compressione:

```text
ZIP
  archivio di file e directory.

Brotli
  compressione di uno stream.
```

Abbiamo studiato la serializzazione:

```text
JSON con System.Text.Json
  formato moderno, leggero, molto usato in API e messaggi.

XML con XmlSerializer
  formato verboso ma ancora importante per integrazioni e standard legacy.

Serializzare
  trasformare oggetti in dati salvabili o trasmissibili.

Deserializzare
  ricostruire oggetti da dati serializzati.
```

Abbiamo chiarito il ruolo dei DTO:

```text
DTO
  oggetto semplice pensato per trasferire dati,
  non per contenere logica di dominio complessa.
```

Infine abbiamo introdotto il versionamento:

```text
I dati serializzati diventano un contratto.
Se quel contratto cambia, chi legge dati vecchi o messaggi esterni può rompersi.
```

La regola principale è questa:

```text
Quando lavori con stream e serializzazione, non stai solo leggendo e scrivendo dati:
stai definendo contratti tra parti del sistema, tra versioni diverse e spesso tra applicazioni diverse.
```

Questo capitolo prepara il terreno per database, API, rete, servizi web, import/export e architetture distribuite.

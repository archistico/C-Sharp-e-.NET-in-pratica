# 03.08 — Dependency Injection

Nel capitolo precedente abbiamo parlato di configurazione, logging e Options pattern. Ora affrontiamo uno dei concetti più importanti del .NET moderno: la **Dependency Injection**, spesso abbreviata in **DI**.

La Dependency Injection è ovunque:

```text
ASP.NET Core
Worker Service
console app moderne con Generic Host
Blazor
Minimal API
servizi applicativi
repository
logging
configurazione
test automatici
microservizi
```

Quando scriviamo codice reale, quasi mai una classe lavora completamente da sola. Di solito ha bisogno di altre classi:

```text
un servizio usa un repository
un controller usa un service
un service usa un logger
un repository usa un DbContext
un importer usa una configurazione
un exporter usa un file writer
```

La domanda è:

```text
Chi crea queste dipendenze?
Chi decide quale implementazione usare?
Chi gestisce il ciclo di vita degli oggetti?
```

La Dependency Injection risponde proprio a questo.

Il punto centrale del capitolo è questo:

```text
Una classe non dovrebbe costruirsi da sola tutte le proprie dipendenze.
Dovrebbe dichiarare ciò di cui ha bisogno e riceverlo dall’esterno.
```

---

# Che cos’è la Dependency Injection

Una **dipendenza** è un oggetto di cui una classe ha bisogno per funzionare.

Esempio:

```csharp
public class ClienteService
{
    private readonly ClienteRepository repository = new ClienteRepository();

    public void RegistraCliente(string nome)
    {
        repository.Salva(nome);
    }
}
```

Qui `ClienteService` dipende da `ClienteRepository`.

Il problema è che la dipendenza viene creata direttamente dentro la classe:

```csharp
new ClienteRepository()
```

Questo rende il codice:

```text
più rigido
più difficile da testare
più difficile da sostituire
più accoppiato
meno configurabile
```

Con Dependency Injection, invece, la dipendenza viene passata dall’esterno.

```csharp
public class ClienteService
{
    private readonly IClienteRepository repository;

    public ClienteService(IClienteRepository repository)
    {
        this.repository = repository;
    }

    public void RegistraCliente(string nome)
    {
        repository.Salva(nome);
    }
}
```

Ora `ClienteService` non sa quale classe concreta riceverà.

Sa solo che riceverà qualcosa che implementa `IClienteRepository`.

Microsoft descrive la Dependency Injection in .NET come un modo per risolvere il problema delle dipendenze hard-coded usando un’interfaccia o classe base per astrarre l’implementazione e registrando la dipendenza in un service container. ([Microsoft Learn][1])

---

# Il problema del `new` dentro le classi

Il problema non è `new` in assoluto.

Creare oggetti semplici è normale:

```csharp
Cliente cliente = new Cliente("Emilie");
```

Il problema nasce quando una classe crea internamente servizi, infrastruttura o componenti esterni.

Esempio rigido:

```csharp
public class OrdineService
{
    private readonly EmailSender emailSender = new EmailSender();
    private readonly OrdineRepository repository = new OrdineRepository();

    public void ConfermaOrdine(int ordineId)
    {
        repository.SalvaConferma(ordineId);
        emailSender.InviaConferma(ordineId);
    }
}
```

Qui `OrdineService` decide da solo:

```text
quale repository usare
quale sender email usare
come crearli
quando crearli
```

Se domani vogliamo usare:

```text
un repository SQL
un repository finto per test
un sender SMTP
un sender silenzioso
un sender via API esterna
```

dobbiamo modificare `OrdineService`.

Con DI, invece:

```csharp
public class OrdineService
{
    private readonly IOrdineRepository repository;
    private readonly IEmailSender emailSender;

    public OrdineService(
        IOrdineRepository repository,
        IEmailSender emailSender)
    {
        this.repository = repository;
        this.emailSender = emailSender;
    }

    public void ConfermaOrdine(int ordineId)
    {
        repository.SalvaConferma(ordineId);
        emailSender.InviaConferma(ordineId);
    }
}
```

Ora `OrdineService` dichiara ciò che gli serve.

Non decide da solo come costruirlo.

# Inversion of Control

La Dependency Injection è una forma di **Inversion of Control**, spesso abbreviata in **IoC**.

Senza IoC:

```text
La classe controlla direttamente la creazione delle sue dipendenze.
```

Con IoC:

```text
La creazione e il collegamento degli oggetti vengono spostati all’esterno.
```

Esempio senza IoC:

```csharp
public class ReportService
{
    private readonly PdfExporter exporter = new PdfExporter();
}
```

Esempio con IoC:

```csharp
public class ReportService
{
    private readonly IExporter exporter;

    public ReportService(IExporter exporter)
    {
        this.exporter = exporter;
    }
}
```

La classe non controlla più quale implementazione concreta riceve.

Questa responsabilità viene spostata nel punto di composizione dell’applicazione.

---

## Punto di composizione

Il **composition root** è il punto in cui l’applicazione registra e collega i servizi.

In una console app moderna può essere `Program.cs`.

In ASP.NET Core è normalmente sempre `Program.cs`.

Esempio:

```csharp
builder.Services.AddScoped<IClienteRepository, SqlClienteRepository>();
builder.Services.AddScoped<ClienteService>();
```

Qui diciamo:

```text
quando qualcuno chiede IClienteRepository,
usa SqlClienteRepository.

quando qualcuno chiede ClienteService,
crealo risolvendo automaticamente le sue dipendenze.
```

---

# Service container .NET

Il **service container** è il componente che conosce le registrazioni dei servizi e sa creare gli oggetti richiesti.

In .NET troviamo:

```csharp
IServiceCollection
IServiceProvider
```

`IServiceCollection` contiene le registrazioni.

`IServiceProvider` risolve i servizi.

Microsoft spiega che i servizi vengono normalmente registrati all’avvio dell’applicazione in una `IServiceCollection`; dopo la registrazione viene creato il service container, rappresentato da `IServiceProvider`. ([Microsoft Learn][1])

---

## IServiceCollection

Esempio:

```csharp
builder.Services.AddScoped<IClienteRepository, SqlClienteRepository>();
builder.Services.AddScoped<ClienteService>();
```

`builder.Services` è una `IServiceCollection`.

Serve a registrare servizi.

---

## IServiceProvider

Dopo la build dell’host, il container può risolvere servizi.

```csharp
ClienteService service = host.Services.GetRequiredService<ClienteService>();
```

In codice applicativo ordinario, però, non dovremmo usare spesso `GetRequiredService`.

Meglio ricevere le dipendenze nel costruttore.

---

# Registrazione servizi

I metodi principali sono:

```csharp
AddSingleton
AddScoped
AddTransient
```

Esempio:

```csharp
builder.Services.AddSingleton<IClock, SystemClock>();
builder.Services.AddScoped<IClienteRepository, SqlClienteRepository>();
builder.Services.AddTransient<EmailFormatter>();
```

Ogni metodo registra un servizio con un lifetime diverso.

Il lifetime indica quanto vive l’istanza creata dal container.

Microsoft documenta tre lifetime principali per i servizi registrati nel container .NET: transient, scoped e singleton. ([Microsoft Learn][2])

---

# Lifetime: Singleton

Un servizio **Singleton** viene creato una sola volta e riutilizzato per tutta la vita dell’applicazione.

```csharp
builder.Services.AddSingleton<IClock, SystemClock>();
```

Esempio:

```csharp
public interface IClock
{
    DateTimeOffset Now { get; }
}

public class SystemClock : IClock
{
    public DateTimeOffset Now => DateTimeOffset.UtcNow;
}
```

Registrazione:

```csharp
builder.Services.AddSingleton<IClock, SystemClock>();
```

Ogni classe che chiede `IClock` riceve la stessa istanza di `SystemClock`.

---

## Quando usare Singleton

Usa singleton per servizi:

```text
stateless
thread-safe
costosi da creare
globalmente condivisi
che non dipendono da dati della singola richiesta
```

Esempi ragionevoli:

```text
clock di sistema
cache globale thread-safe
provider di configurazione
servizi di lookup read-only
client HTTP gestiti correttamente
```

Attenzione: Microsoft raccomanda di usare singleton solo per servizi con stato proprio costoso da creare o globalmente condiviso, e di evitare singleton inutili per servizi senza stato quando non servono davvero. ([Microsoft Learn][3])

---

## Pericolo: singleton con stato mutabile

Sospetto:

```csharp
public class CarrelloCorrente
{
    public List<string> Articoli { get; } = new();
}
```

Registrato come singleton:

```csharp
builder.Services.AddSingleton<CarrelloCorrente>();
```

Questo sarebbe pericoloso in un’app web: tutti gli utenti condividerebbero lo stesso carrello.

Regola pratica:

```text
Singleton significa condiviso.
Se contiene stato utente o stato di richiesta, quasi sempre è sbagliato.
```

---

# Lifetime: Scoped

Un servizio **Scoped** vive per uno scope.

In ASP.NET Core, di solito uno scope corrisponde a una richiesta HTTP.

```csharp
builder.Services.AddScoped<IClienteRepository, SqlClienteRepository>();
```

Durante la stessa richiesta, chi chiede `IClienteRepository` riceve la stessa istanza.

In una richiesta diversa, ne riceve una nuova.

---

## Quando usare Scoped

Usa scoped per servizi legati a una singola operazione logica.

Esempi:

```text
repository
unit of work
DbContext EF Core
servizi applicativi per richiesta
componenti che condividono stato durante una richiesta
```

In ASP.NET Core, `DbContext` di Entity Framework Core viene normalmente registrato come scoped, perché rappresenta un’unità di lavoro legata a una richiesta o a un’operazione.

---

## Scoped fuori da ASP.NET Core

In console app e worker service non esiste automaticamente il concetto di richiesta HTTP.

Possiamo creare uno scope manualmente:

```csharp
using IServiceScope scope = host.Services.CreateScope();

ClienteService service =
    scope.ServiceProvider.GetRequiredService<ClienteService>();

service.Esegui();
```

Questo è utile nei worker quando ogni elaborazione deve avere il proprio scope.

---

# Lifetime: Transient

Un servizio **Transient** viene creato ogni volta che viene richiesto.

```csharp
builder.Services.AddTransient<EmailFormatter>();
```

Ogni classe che chiede `EmailFormatter` riceve una nuova istanza.

---

## Quando usare Transient

Usa transient per servizi:

```text
leggeri
stateless
economici da creare
senza risorse da condividere
senza stato da conservare
```

Esempi:

```text
formatter
validator
mapper semplice
builder di messaggi
calcolatori stateless
```

---

# Confronto lifetime

```text
Singleton
  una istanza per tutta l’applicazione.

Scoped
  una istanza per scope, per esempio una richiesta HTTP.

Transient
  una nuova istanza ogni volta che viene richiesta.
```

Esempio mentale:

```text
Singleton
  configurazione globale.

Scoped
  carrello/richiesta/unità di lavoro corrente.

Transient
  calcolatore temporaneo.
```

---

# Errore grave: singleton che dipende da scoped

Sbagliato:

```csharp
builder.Services.AddScoped<IClienteRepository, ClienteRepository>();
builder.Services.AddSingleton<ReportService>();
```

con:

```csharp
public class ReportService
{
    private readonly IClienteRepository repository;

    public ReportService(IClienteRepository repository)
    {
        this.repository = repository;
    }
}
```

Un singleton vive per tutta l’applicazione.
Uno scoped vive per uno scope.

Se un singleton cattura uno scoped, rischiamo di usare un oggetto oltre il suo ciclo di vita corretto.

Microsoft raccomanda di abilitare la scope validation per rilevare casi in cui un singleton cattura servizi scoped. ([Microsoft Learn][3])

Regola pratica:

```text
Un servizio con lifetime più lungo non dovrebbe dipendere direttamente
da un servizio con lifetime più breve.
```

# Constructor injection

La forma più comune di Dependency Injection è la **constructor injection**.

Le dipendenze vengono dichiarate nel costruttore.

Esempio:

```csharp
public class ClienteService
{
    private readonly IClienteRepository repository;
    private readonly ILogger<ClienteService> logger;

    public ClienteService(
        IClienteRepository repository,
        ILogger<ClienteService> logger)
    {
        this.repository = repository;
        this.logger = logger;
    }

    public void Registra(string nome)
    {
        logger.LogInformation("Registro cliente {Nome}", nome);
        repository.Salva(nome);
    }
}
```

Vantaggi:

```text
dipendenze esplicite
oggetto sempre costruito in modo valido
test più semplici
niente dipendenze nascoste
fallimento immediato se manca una registrazione
```

---

## Constructor injection e dipendenze obbligatorie

Se una classe non può funzionare senza un repository, il repository deve essere nel costruttore.

```csharp
public ClienteService(IClienteRepository repository)
{
    this.repository = repository;
}
```

Questo comunica chiaramente:

```text
ClienteService non può esistere senza IClienteRepository.
```

---

# Interfacce e implementazioni

La DI è particolarmente utile quando dipendiamo da interfacce.

```csharp
public interface IClienteRepository
{
    void Salva(string nome);
}
```

Implementazione reale:

```csharp
public class SqlClienteRepository : IClienteRepository
{
    public void Salva(string nome)
    {
        Console.WriteLine($"Salvo cliente su SQL: {nome}");
    }
}
```

Implementazione finta:

```csharp
public class FakeClienteRepository : IClienteRepository
{
    public List<string> Clienti { get; } = new();

    public void Salva(string nome)
    {
        Clienti.Add(nome);
    }
}
```

Registrazione reale:

```csharp
builder.Services.AddScoped<IClienteRepository, SqlClienteRepository>();
```

Nel test possiamo usare `FakeClienteRepository`.

# Factory

A volte non basta registrare un tipo con costruttore semplice.

Serve una factory.

Esempio:

```csharp
builder.Services.AddSingleton<IClock>(serviceProvider =>
{
    return new SystemClock();
});
```

Oppure con configurazione:

```csharp
builder.Services.AddSingleton<ExportService>(serviceProvider =>
{
    IConfiguration configuration =
        serviceProvider.GetRequiredService<IConfiguration>();

    string folder = configuration["Export:Folder"] ?? "export";

    ILogger<ExportService> logger =
        serviceProvider.GetRequiredService<ILogger<ExportService>>();

    return new ExportService(folder, logger);
});
```

---

## Factory con interfaccia

Esempio più pulito:

```csharp
public interface IReportExporterFactory
{
    IReportExporter Crea(string formato);
}
```

Implementazione:

```csharp
public class ReportExporterFactory : IReportExporterFactory
{
    private readonly IServiceProvider serviceProvider;

    public ReportExporterFactory(IServiceProvider serviceProvider)
    {
        this.serviceProvider = serviceProvider;
    }

    public IReportExporter Crea(string formato)
    {
        return formato.ToLowerInvariant() switch
        {
            "pdf" => serviceProvider.GetRequiredService<PdfReportExporter>(),
            "csv" => serviceProvider.GetRequiredService<CsvReportExporter>(),
            _ => throw new NotSupportedException($"Formato non supportato: {formato}")
        };
    }
}
```

Registrazione:

```csharp
builder.Services.AddTransient<PdfReportExporter>();
builder.Services.AddTransient<CsvReportExporter>();
builder.Services.AddSingleton<IReportExporterFactory, ReportExporterFactory>();
```

Nota: questa factory usa `IServiceProvider` internamente. Va bene se resta confinato nella factory. Non bisogna spargere `IServiceProvider` in tutto il codice applicativo.

---

## Factory senza IServiceProvider

Spesso è ancora meglio evitare `IServiceProvider` nella factory e iniettare direttamente le implementazioni.

```csharp
public class ReportExporterFactory : IReportExporterFactory
{
    private readonly PdfReportExporter pdf;
    private readonly CsvReportExporter csv;

    public ReportExporterFactory(
        PdfReportExporter pdf,
        CsvReportExporter csv)
    {
        this.pdf = pdf;
        this.csv = csv;
    }

    public IReportExporter Crea(string formato)
    {
        return formato.ToLowerInvariant() switch
        {
            "pdf" => pdf,
            "csv" => csv,
            _ => throw new NotSupportedException($"Formato non supportato: {formato}")
        };
    }
}
```

Questo rende le dipendenze più esplicite.

# Anti-pattern: Service Locator

Il **Service Locator** consiste nel chiedere direttamente al container i servizi di cui abbiamo bisogno, invece di dichiararli nel costruttore.

Sconsigliato:

```csharp
public class ClienteService
{
    private readonly IServiceProvider serviceProvider;

    public ClienteService(IServiceProvider serviceProvider)
    {
        this.serviceProvider = serviceProvider;
    }

    public void Registra(string nome)
    {
        IClienteRepository repository =
            serviceProvider.GetRequiredService<IClienteRepository>();

        repository.Salva(nome);
    }
}
```

Il problema è che le dipendenze sono nascoste.

Guardando il costruttore, sembra che `ClienteService` dipenda solo da `IServiceProvider`.

In realtà dipende anche da `IClienteRepository`.

Microsoft nelle linee guida DI raccomanda esplicitamente di evitare il pattern Service Locator, per esempio non chiamando `GetService` quando si potrebbe usare la Dependency Injection. ([Microsoft Learn][3])

---

## Perché è un problema

Il Service Locator:

```text
nasconde le dipendenze reali
rende i test più difficili
sposta errori a runtime
rende il codice meno leggibile
aumenta l’accoppiamento al container
```

Meglio:

```csharp
public class ClienteService
{
    private readonly IClienteRepository repository;

    public ClienteService(IClienteRepository repository)
    {
        this.repository = repository;
    }

    public void Registra(string nome)
    {
        repository.Salva(nome);
    }
}
```

---

# Anti-pattern: dipendenze statiche

Un altro problema comune è usare classi statiche per servizi che dovrebbero essere iniettati.

Sconsigliato:

```csharp
public static class Database
{
    public static void SalvaCliente(string nome)
    {
        // salva su database
    }
}

public class ClienteService
{
    public void Registra(string nome)
    {
        Database.SalvaCliente(nome);
    }
}
```

Questo rende difficile:

```text
sostituire implementazione
testare
configurare
gestire lifetime
isolare dipendenze
```

Meglio:

```csharp
public interface IClienteRepository
{
    void Salva(string nome);
}

public class ClienteService
{
    private readonly IClienteRepository repository;

    public ClienteService(IClienteRepository repository)
    {
        this.repository = repository;
    }
}
```

---

## Quando static va bene

Le classi statiche non sono sempre sbagliate.

Sono accettabili per funzioni pure o helper senza dipendenze.

Esempio:

```csharp
public static class PercentualeHelper
{
    public static decimal Calcola(decimal importo, decimal percentuale)
    {
        return importo * percentuale / 100m;
    }
}
```

Ma se una classe accede a:

```text
database
file system
rete
configurazione
orologio di sistema
logger
servizi esterni
```

allora spesso è meglio usare DI.

# DI in console app

Una console app moderna può usare il Generic Host.

Il Generic Host gestisce:

```text
Dependency Injection
Logging
Configuration
lifetime dell’applicazione
IHostedService
```

Microsoft descrive il Generic Host come responsabile della gestione dell’avvio e della durata dell’applicazione; può essere usato anche con console app, non solo con Worker Service. ([Microsoft Learn][4])

---

## Esempio console app con DI

`.csproj`:

```xml
<Project Sdk="Microsoft.NET.Sdk">

  <PropertyGroup>
    <OutputType>Exe</OutputType>
    <TargetFramework>net10.0</TargetFramework>
    <Nullable>enable</Nullable>
    <ImplicitUsings>enable</ImplicitUsings>
  </PropertyGroup>

  <ItemGroup>
    <PackageReference Include="Microsoft.Extensions.Hosting" Version="10.0.0" />
  </ItemGroup>

</Project>
```

Interfaccia:

```csharp
public interface IMessageWriter
{
    void Write(string message);
}
```

Implementazione:

```csharp
public class ConsoleMessageWriter : IMessageWriter
{
    public void Write(string message)
    {
        Console.WriteLine(message);
    }
}
```

Servizio applicativo:

```csharp
public class App
{
    private readonly IMessageWriter writer;
    private readonly ILogger<App> logger;

    public App(
        IMessageWriter writer,
        ILogger<App> logger)
    {
        this.writer = writer;
        this.logger = logger;
    }

    public void Run()
    {
        logger.LogInformation("Applicazione avviata");

        writer.Write("Ciao dalla console app con DI");
    }
}
```

`Program.cs`:

```csharp
using Microsoft.Extensions.DependencyInjection;
using Microsoft.Extensions.Hosting;
using Microsoft.Extensions.Logging;

HostApplicationBuilder builder = Host.CreateApplicationBuilder(args);

builder.Services.AddSingleton<IMessageWriter, ConsoleMessageWriter>();
builder.Services.AddTransient<App>();

using IHost host = builder.Build();

App app = host.Services.GetRequiredService<App>();

app.Run();
```

In questo esempio:

```text
IMessageWriter
  è il contratto.

ConsoleMessageWriter
  è l’implementazione.

App
  riceve le dipendenze nel costruttore.

Host
  configura DI, logging e lifetime.
```

# DI in ASP.NET Core

ASP.NET Core usa DI in modo nativo.

Nel file `Program.cs`:

```csharp
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddScoped<IClienteRepository, SqlClienteRepository>();
builder.Services.AddScoped<ClienteService>();

var app = builder.Build();

app.MapGet("/clienti/{id:int}", (
    int id,
    ClienteService clienteService) =>
{
    Cliente? cliente = clienteService.Trova(id);

    return cliente is null
        ? Results.NotFound()
        : Results.Ok(cliente);
});

app.Run();
```

ASP.NET Core implementa e usa Dependency Injection in modo integrato; i servizi vengono registrati nel contenitore e poi richiesti da controller, endpoint, Razor Pages, Blazor component e altri componenti framework. ([Microsoft Learn][5])

---

## Controller con constructor injection

```csharp
[ApiController]
[Route("api/clienti")]
public class ClientiController : ControllerBase
{
    private readonly ClienteService clienteService;

    public ClientiController(ClienteService clienteService)
    {
        this.clienteService = clienteService;
    }

    [HttpGet("{id:int}")]
    public ActionResult<ClienteDto> Get(int id)
    {
        ClienteDto? cliente = clienteService.Trova(id);

        if (cliente is null)
        {
            return NotFound();
        }

        return Ok(cliente);
    }
}
```

Il controller non crea `ClienteService`.

Lo riceve.

# DI e testabilità

Uno dei maggiori vantaggi della DI è la testabilità.

Senza DI:

```csharp
public class ClienteService
{
    private readonly SqlClienteRepository repository = new SqlClienteRepository();

    public void Registra(string nome)
    {
        repository.Salva(nome);
    }
}
```

Nel test siamo costretti a usare `SqlClienteRepository`.

Con DI:

```csharp
public class ClienteService
{
    private readonly IClienteRepository repository;

    public ClienteService(IClienteRepository repository)
    {
        this.repository = repository;
    }

    public void Registra(string nome)
    {
        repository.Salva(nome);
    }
}
```

Possiamo passare una fake implementation.

---

## Esempio di test senza database

Interfaccia:

```csharp
public interface IClienteRepository
{
    void Salva(string nome);
}
```

Fake:

```csharp
public class FakeClienteRepository : IClienteRepository
{
    public List<string> Salvati { get; } = new();

    public void Salva(string nome)
    {
        Salvati.Add(nome);
    }
}
```

Servizio:

```csharp
public class ClienteService
{
    private readonly IClienteRepository repository;

    public ClienteService(IClienteRepository repository)
    {
        this.repository = repository;
    }

    public void Registra(string nome)
    {
        ArgumentException.ThrowIfNullOrWhiteSpace(nome);

        repository.Salva(nome);
    }
}
```

Test:

```csharp
[Fact]
public void Registra_salva_il_cliente()
{
    FakeClienteRepository repository = new FakeClienteRepository();
    ClienteService service = new ClienteService(repository);

    service.Registra("Emilie");

    Assert.Contains("Emilie", repository.Salvati);
}
```

Il test non usa database, file, rete o container.

È veloce e isolato.

## `InMemoryClienteRepository.cs`

```csharp
public class InMemoryClienteRepository : IClienteRepository
{
    private readonly Dictionary<int, Cliente> clienti = new();

    public void Salva(Cliente cliente)
    {
        clienti[cliente.Id] = cliente;
    }

    public Cliente? Trova(int id)
    {
        clienti.TryGetValue(id, out Cliente? cliente);
        return cliente;
    }
}
```

## `ClienteService.cs`

```csharp
using Microsoft.Extensions.Logging;
using Microsoft.Extensions.Options;

public class ClienteService
{
    private readonly IClienteRepository repository;
    private readonly ILogger<ClienteService> logger;
    private readonly ApplicationOptions options;

    public ClienteService(
        IClienteRepository repository,
        ILogger<ClienteService> logger,
        IOptions<ApplicationOptions> options)
    {
        this.repository = repository;
        this.logger = logger;
        this.options = options.Value;
    }

    public void Registra(int id, string nome)
    {
        if (id <= 0)
        {
            throw new ArgumentOutOfRangeException(nameof(id));
        }

        ArgumentException.ThrowIfNullOrWhiteSpace(nome);

        logger.LogInformation(
            "Registrazione cliente {ClienteId} - {Nome}. Max clienti configurato: {MaxClienti}",
            id,
            nome,
            options.MaxClienti);

        Cliente cliente = new Cliente(id, nome);

        repository.Salva(cliente);
    }

    public Cliente? Trova(int id)
    {
        logger.LogInformation("Ricerca cliente {ClienteId}", id);

        return repository.Trova(id);
    }
}
```

using Microsoft.Extensions.Logging;

public class App
{
    private readonly ClienteService clienteService;
    private readonly ILogger<App> logger;

    public App(
        ClienteService clienteService,
        ILogger<App> logger)
    {
        this.clienteService = clienteService;
        this.logger = logger;
    }

    public void Run()
    {
        logger.LogInformation("Avvio applicazione demo DI");

        clienteService.Registra(1, "Emilie");

        Cliente? cliente = clienteService.Trova(1);

        if (cliente is not null)
        {
            Console.WriteLine($"Cliente trovato: {cliente.Nome}");
        }
    }
}

## `Program.cs`

```csharp
using Microsoft.Extensions.DependencyInjection;
using Microsoft.Extensions.Hosting;

HostApplicationBuilder builder = Host.CreateApplicationBuilder(args);

builder.Services
    .AddOptions<ApplicationOptions>()
    .Bind(builder.Configuration.GetSection(ApplicationOptions.SectionName))
    .ValidateDataAnnotations()
    .ValidateOnStart();

builder.Services.AddScoped<IClienteRepository, InMemoryClienteRepository>();
builder.Services.AddScoped<ClienteService>();
builder.Services.AddTransient<App>();

using IHost host = builder.Build();

using IServiceScope scope = host.Services.CreateScope();

App app = scope.ServiceProvider.GetRequiredService<App>();

app.Run();
```

# Errori comuni

## Registrare tutto come Singleton

Sbagliato:

```csharp
builder.Services.AddSingleton<ClienteService>();
builder.Services.AddSingleton<IClienteRepository, SqlClienteRepository>();
```

Non tutto deve vivere per tutta l’applicazione.

Servizi con stato di richiesta o unità di lavoro dovrebbero spesso essere scoped.

---

## Usare Service Locator

Sconsigliato:

```csharp
public class ClienteService
{
    public void Registra(IServiceProvider provider)
    {
        var repository = provider.GetRequiredService<IClienteRepository>();
    }
}
```

Meglio constructor injection.

---

## Iniettare troppe dipendenze

Sospetto:

```csharp
public class MegaService
{
    public MegaService(
        A a,
        B b,
        C c,
        D d,
        E e,
        F f,
        G g,
        H h)
    {
    }
}
```

Troppe dipendenze indicano spesso che la classe ha troppe responsabilità.

Possibile soluzione:

```text
dividere il servizio
creare servizi più piccoli
introdurre un oggetto che rappresenta una responsabilità coerente
rivedere il design
```

---

## Dipendere da implementazioni concrete quando basta un’interfaccia

Meno flessibile:

```csharp
public ClienteService(SqlClienteRepository repository)
{
}
```

Più flessibile:

```csharp
public ClienteService(IClienteRepository repository)
{
}
```

Non serve sempre un’interfaccia per ogni classe, ma quando vuoi sostituibilità, testabilità o isolamento, l’interfaccia è utile.

---

## Usare static per servizi infrastrutturali

Sconsigliato:

```csharp
EmailSender.Send(email, messaggio);
```

Meglio:

```csharp
public class ClienteService
{
    private readonly IEmailSender emailSender;

    public ClienteService(IEmailSender emailSender)
    {
        this.emailSender = emailSender;
    }
}
```

---

## Costruire manualmente il ServiceProvider troppo presto

Sconsigliato:

```csharp
IServiceProvider provider = builder.Services.BuildServiceProvider();
```

dentro la configurazione dei servizi.

Questo può creare container duplicati e lifetime errati.

Meglio usare overload con factory:

```csharp
builder.Services.AddSingleton<MyService>(sp =>
{
    var logger = sp.GetRequiredService<ILogger<MyService>>();
    return new MyService(logger);
});
```

Le linee guida Microsoft sconsigliano di chiamare `BuildServiceProvider` quando si configurano i servizi se lo scopo è risolvere un servizio durante la registrazione; in questi casi è preferibile usare overload che ricevono `IServiceProvider`. ([Microsoft Learn][3])

# Linee guida pratiche

```text
Usa constructor injection
  per dipendenze obbligatorie.

Registra servizi nel composition root
  di solito Program.cs.

Dipendi da interfacce
  quando vuoi sostituibilità e testabilità.

Usa Transient
  per servizi leggeri e stateless.

Usa Scoped
  per unità di lavoro, repository, DbContext e richiesta HTTP.

Usa Singleton
  per servizi condivisi, thread-safe e adatti a vivere tutta l’app.

Evita Service Locator
  perché nasconde le dipendenze.

Evita static per servizi infrastrutturali
  perché riduce testabilità e configurabilità.

Non catturare scoped dentro singleton
  perché rompe il ciclo di vita corretto.

Usa factory
  quando la creazione richiede logica o parametri runtime.

Usa DI anche nelle console app moderne
  tramite Generic Host.
```

# Riepilogo

In questo capitolo abbiamo introdotto la Dependency Injection.

Abbiamo visto:

```text
Dependency Injection
  tecnica per fornire a una classe le sue dipendenze dall’esterno.

Inversion of Control
  inversione della responsabilità di creazione e collegamento degli oggetti.

Service container
  componente che registra e risolve servizi.

IServiceCollection
  raccolta delle registrazioni.

IServiceProvider
  container che risolve i servizi.
```

Abbiamo studiato i lifetime:

```text
Singleton
  una istanza per tutta l’applicazione.

Scoped
  una istanza per scope, per esempio richiesta HTTP.

Transient
  una nuova istanza a ogni richiesta.
```

Abbiamo visto:

```text
registrazione servizi
constructor injection
factory
DI in console app
DI in ASP.NET Core
DI e testabilità
```

Abbiamo anche chiarito alcuni anti-pattern:

```text
Service Locator
  dipendenze nascoste risolte a runtime.

dipendenze statiche
  servizi globali difficili da sostituire.

BuildServiceProvider prematuro
  rischio di container duplicati e lifetime errati.
```

La regola principale è questa:

```text
Una classe dovrebbe dichiarare ciò di cui ha bisogno,
non costruirselo di nascosto.
```

La Dependency Injection è uno snodo fondamentale perché collega molti temi del .NET moderno: logging, configurazione, options, database, API, servizi applicativi, test, worker service, ASP.NET Core e architetture modulari.

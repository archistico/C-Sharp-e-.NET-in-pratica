# 03.03 — Collezioni

Nel capitolo precedente abbiamo visto numeri, stringhe, date e formattazione. Ora affrontiamo un altro blocco fondamentale di .NET: le **collezioni**.

Una collezione è una struttura che contiene più elementi.

Esempi:

```text id="3wqeq7"
elenco di clienti
righe di un documento
dizionario di configurazioni
insieme di codici validi
coda di messaggi da elaborare
pila di operazioni da annullare
```

Le collezioni sono la base di moltissimo codice C#. Prima di studiare LINQ è importante conoscerle bene, perché LINQ lavora quasi sempre su sequenze e collezioni.

Microsoft distingue tra collezioni generiche e non generiche; le collezioni generiche sono type-safe a compile time e normalmente offrono prestazioni migliori perché non richiedono cast da e verso `object`. ([Microsoft Learn][1])

Il punto centrale del capitolo è questo:

```text id="0kisyy"
Non esiste una collezione migliore in assoluto.
Esiste la collezione più adatta al problema.
```

---

# Array

Un **array** è una sequenza di elementi dello stesso tipo con dimensione fissa.

Esempio:

```csharp id="q1idmk"
int[] numeri = new int[3];

numeri[0] = 10;
numeri[1] = 20;
numeri[2] = 30;

Console.WriteLine(numeri[0]);
```

Output:

```text id="gkciev"
10
```

Gli indici partono da `0`.

```text id="pyhof2"
numeri[0]  primo elemento
numeri[1]  secondo elemento
numeri[2]  terzo elemento
```

---

## Inizializzare un array

Forma classica:

```csharp id="jtbdqz"
string[] nomi = new string[]
{
    "Emilie",
    "Mario",
    "Anna"
};
```

Forma compatta:

```csharp id="0b2f0i"
string[] nomi = { "Emilie", "Mario", "Anna" };
```

Con C# moderno:

```csharp id="ndcseo"
string[] nomi = ["Emilie", "Mario", "Anna"];
```

---

## Dimensione fissa

Un array ha dimensione fissa.

```csharp id="k91sdd"
int[] numeri = new int[3];
```

Questo array può contenere esattamente tre elementi.

Non possiamo aggiungere un quarto elemento con un metodo tipo `Add`, perché l’array non si espande automaticamente.

Se serve una collezione che cresce, normalmente si usa `List<T>`.

---

## Scorrere un array

Con `for`:

```csharp id="n54624"
string[] nomi = ["Emilie", "Mario", "Anna"];

for (int i = 0; i < nomi.Length; i++)
{
    Console.WriteLine(nomi[i]);
}
```

Con `foreach`:

```csharp id="g4xuuo"
foreach (string nome in nomi)
{
    Console.WriteLine(nome);
}
```

Usa `for` quando ti serve l’indice.
Usa `foreach` quando vuoi solo leggere gli elementi.

---

## Quando usare array

Gli array sono adatti quando:

* la dimensione è nota;
* la dimensione non cambia;
* serve accesso veloce per indice;
* lavori con API che richiedono array;
* stai gestendo dati a basso livello;
* vuoi una struttura semplice e compatta.

Esempio:

```csharp id="wo7hcn"
decimal[] aliquoteIva = [4m, 10m, 22m];
```

# `List<T>`

`List<T>` è una lista generica dinamica.

È probabilmente la collezione più usata nel codice C# applicativo.

Esempio:

```csharp id="5ygdmd"
List<string> nomi = new List<string>();

nomi.Add("Emilie");
nomi.Add("Mario");
nomi.Add("Anna");

Console.WriteLine(nomi[0]);
```

Output:

```text id="aajgix"
Emilie
```

---

## Inizializzare una lista

Forma classica:

```csharp id="4bxjt3"
List<string> nomi = new List<string>
{
    "Emilie",
    "Mario",
    "Anna"
};
```

Con collection expressions:

```csharp id="3djugf"
List<string> nomi = ["Emilie", "Mario", "Anna"];
```

---

## Operazioni comuni

Aggiungere:

```csharp id="fv7mc9"
nomi.Add("Luca");
```

Rimuovere:

```csharp id="jjkm7u"
nomi.Remove("Mario");
```

Leggere per indice:

```csharp id="fvz5t2"
string primo = nomi[0];
```

Contare:

```csharp id="tgu8eh"
int quanti = nomi.Count;
```

Verificare presenza:

```csharp id="vrc74p"
bool contiene = nomi.Contains("Emilie");
```

---

## Capacity e Count

`Count` indica quanti elementi contiene la lista.

`Capacity` indica quanti elementi può contenere prima di dover riallocare memoria.

```csharp id="iw49e5"
List<int> numeri = new List<int>();

Console.WriteLine(numeri.Count);
Console.WriteLine(numeri.Capacity);
```

Quando una `List<T>` cresce oltre la sua capacità, deve riallocare memoria e copiare gli elementi. Microsoft documenta che `List<T>.Add` è O(1) se `Count` è minore di `Capacity`, ma diventa O(n) quando serve aumentare la capacità e copiare gli elementi. ([Microsoft Learn][1])

Se conosciamo già la dimensione prevista, possiamo impostare la capacità iniziale:

```csharp id="aqmj7q"
List<Cliente> clienti = new List<Cliente>(100_000);
```

Questo può evitare riallocazioni ripetute.

---

## Quando usare List<T>

Usa `List<T>` quando:

* devi mantenere un elenco ordinato per posizione;
* devi aggiungere elementi;
* devi scorrere elementi;
* ti serve accesso per indice;
* non hai bisogno di chiavi;
* non devi garantire unicità.

Esempio:

```csharp id="fnuwxn"
List<RigaDocumento> righe = new List<RigaDocumento>();
```

# `Dictionary<TKey, TValue>`

Un `Dictionary<TKey, TValue>` memorizza coppie chiave/valore.

È adatto quando vuoi cercare rapidamente un valore partendo da una chiave.

Esempio:

```csharp id="azy1ip"
Dictionary<int, string> clienti = new Dictionary<int, string>();

clienti[1] = "Emilie";
clienti[2] = "Mario";

Console.WriteLine(clienti[1]);
```

Output:

```text id="p1euzq"
Emilie
```

Qui:

```text id="7nsl9d"
TKey
  int

TValue
  string
```

---

## Inizializzare un dizionario

```csharp id="fdmswp"
Dictionary<string, string> province = new Dictionary<string, string>
{
    ["AO"] = "Aosta",
    ["TO"] = "Torino",
    ["MI"] = "Milano"
};
```

Uso:

```csharp id="zv16k6"
Console.WriteLine(province["AO"]);
```

Output:

```text id="mxqohe"
Aosta
```

---

## Accesso sicuro con TryGetValue

Se la chiave non esiste, questo codice genera errore:

```csharp id="qzfd8z"
string città = province["XX"];
```

Meglio usare `TryGetValue`:

```csharp id="vg4m8e"
if (province.TryGetValue("AO", out string? città))
{
    Console.WriteLine(città);
}
else
{
    Console.WriteLine("Provincia non trovata.");
}
```

---

## ContainsKey

```csharp id="6xpdcg"
if (province.ContainsKey("AO"))
{
    Console.WriteLine(province["AO"]);
}
```

Funziona, ma spesso `TryGetValue` è migliore perché evita una doppia ricerca: una per controllare e una per leggere.

---

## Quando usare Dictionary

Usa `Dictionary<TKey, TValue>` quando:

* hai una chiave unica;
* devi cercare spesso per chiave;
* vuoi associare valori;
* l’ordine non è il punto principale;
* vuoi evitare ricerche lineari.

Esempio:

```csharp id="yvbddi"
Dictionary<int, Cliente> clientiPerId = new Dictionary<int, Cliente>();
```

Microsoft indica `Dictionary<TKey,TValue>` come collezione generica adatta per archiviare coppie chiave/valore con lookup rapido per chiave. ([Microsoft Learn][1])

# `HashSet<T>`

`HashSet<T>` rappresenta un insieme di elementi unici.

Esempio:

```csharp id="73cywp"
HashSet<string> codici = new HashSet<string>();

codici.Add("A");
codici.Add("B");
codici.Add("A");

Console.WriteLine(codici.Count);
```

Output:

```text id="e3k7wl"
2
```

Il valore `"A"` viene inserito una sola volta.

---

## Verificare presenza

```csharp id="lvqa2x"
if (codici.Contains("A"))
{
    Console.WriteLine("Codice presente");
}
```

`HashSet<T>` è molto efficiente quando devi verificare se un elemento appartiene a un insieme.

---

```csharp id="g5c1uh"
HashSet<string> gruppoA = ["A", "B", "C"];
HashSet<string> gruppoB = ["B", "C", "D"];

gruppoA.IntersectWith(gruppoB);

Console.WriteLine(string.Join(", ", gruppoA));
```

Output:

```text id="fqnyop"
B, C
```

Altre operazioni:

```csharp id="rt8b2r"
UnionWith
ExceptWith
IntersectWith
IsSubsetOf
IsSupersetOf
```

---

## Quando usare HashSet

Usa `HashSet<T>` quando:

* ti serve unicità;
* devi controllare presenza velocemente;
* non ti interessa l’ordine;
* devi fare operazioni insiemistiche.

Esempio:

```csharp id="5g8ew1"
HashSet<int> clientiGiaEstratti = new HashSet<int>();
```

# `Queue<T>`

`Queue<T>` rappresenta una coda FIFO:

```text id="5u48e4"
First In, First Out
```

Il primo elemento inserito è il primo a uscire.

Esempio:

```csharp id="6ld1kz"
Queue<string> coda = new Queue<string>();

coda.Enqueue("Primo");
coda.Enqueue("Secondo");
coda.Enqueue("Terzo");

Console.WriteLine(coda.Dequeue());
Console.WriteLine(coda.Dequeue());
```

Output:

```text id="b78qii"
Primo
Secondo
```

---

## Operazioni principali

Aggiungere in fondo:

```csharp id="vkaqgr"
coda.Enqueue("Messaggio");
```

Estrarre il primo:

```csharp id="10ydsu"
string messaggio = coda.Dequeue();
```

Leggere il primo senza rimuoverlo:

```csharp id="w7o4sz"
string prossimo = coda.Peek();
```

Verificare numero elementi:

```csharp id="rbctar"
int count = coda.Count;
```

---

## Quando usare Queue

Usa `Queue<T>` quando devi elaborare elementi nello stesso ordine in cui arrivano.

Esempi:

* messaggi da processare;
* richieste in attesa;
* lavori batch;
* eventi in sequenza;
* buffer FIFO.

```csharp id="wvojlq"
Queue<string> messaggi = new Queue<string>();

messaggi.Enqueue("Importa file 1");
messaggi.Enqueue("Importa file 2");

while (messaggi.Count > 0)
{
    string messaggio = messaggi.Dequeue();
    Console.WriteLine($"Elaboro: {messaggio}");
}
```

Microsoft indica `Queue<T>` come collezione generica adatta agli scenari FIFO. ([Microsoft Learn][1])

---

# `Stack<T>`

`Stack<T>` rappresenta una pila LIFO:

```text id="addnon"
Last In, First Out
```

L’ultimo elemento inserito è il primo a uscire.

Esempio:

```csharp id="nb4sgy"
Stack<string> pila = new Stack<string>();

pila.Push("Primo");
pila.Push("Secondo");
pila.Push("Terzo");

Console.WriteLine(pila.Pop());
Console.WriteLine(pila.Pop());
```

Output:

```text id="oj557x"
Terzo
Secondo
```

---

## Operazioni principali

Inserire:

```csharp id="njt0yg"
pila.Push("Operazione");
```

Estrarre:

```csharp id="6igmp7"
string operazione = pila.Pop();
```

Leggere senza rimuovere:

```csharp id="fyco6y"
string prossima = pila.Peek();
```

---

## Quando usare Stack

Usa `Stack<T>` quando l’ultima cosa inserita deve essere la prima a uscire.

Esempi:

* undo/redo;
* navigazione indietro;
* parsing;
* visite di alberi;
* gestione di operazioni annidate.

```csharp id="kbjw4v"
Stack<string> undo = new Stack<string>();

undo.Push("Inserito cliente");
undo.Push("Modificato indirizzo");
undo.Push("Cancellata riga");

string ultimaOperazione = undo.Pop();

Console.WriteLine($"Annulla: {ultimaOperazione}");
```

Microsoft indica `Stack<T>` come collezione generica adatta agli scenari LIFO. ([Microsoft Learn][1])

# `SortedList<TKey, TValue>`

`SortedList<TKey, TValue>` è una collezione chiave/valore ordinata per chiave.

Esempio:

```csharp id="x4ve66"
SortedList<int, string> clienti = new SortedList<int, string>();

clienti.Add(3, "Mario");
clienti.Add(1, "Emilie");
clienti.Add(2, "Anna");

foreach (var coppia in clienti)
{
    Console.WriteLine($"{coppia.Key}: {coppia.Value}");
}
```

Output:

```text id="0145ru"
1: Emilie
2: Anna
3: Mario
```

Gli elementi vengono enumerati in ordine di chiave.

---

## Quando usare SortedList

`SortedList<TKey,TValue>` può essere utile quando:

* ti serve una mappa chiave/valore;
* vuoi mantenere le chiavi ordinate;
* la collezione non cambia troppo spesso;
* vuoi accesso anche per indice.

Esempio:

```csharp id="jl1d7q"
SortedList<string, string> province = new SortedList<string, string>
{
    ["AO"] = "Aosta",
    ["MI"] = "Milano",
    ["TO"] = "Torino"
};
```

---

# `SortedDictionary<TKey, TValue>`

`SortedDictionary<TKey, TValue>` è anch’esso un dizionario ordinato per chiave.

Esempio:

```csharp id="5bk1su"
SortedDictionary<int, string> clienti = new SortedDictionary<int, string>();

clienti.Add(3, "Mario");
clienti.Add(1, "Emilie");
clienti.Add(2, "Anna");

foreach (var coppia in clienti)
{
    Console.WriteLine($"{coppia.Key}: {coppia.Value}");
}
```

Output:

```text id="mxk2rg"
1: Emilie
2: Anna
3: Mario
```

---

## SortedList vs SortedDictionary

Entrambe mantengono le chiavi ordinate.

La differenza pratica più importante è legata alla struttura interna e alle prestazioni in scenari diversi.

Regola semplificata:

```text id="k0pkiu"
SortedList<TKey,TValue>
  spesso efficiente in memoria e comoda per accesso per indice,
  ma inserimenti e rimozioni possono essere costosi perché gli elementi devono restare ordinati.

SortedDictionary<TKey,TValue>
  più adatto a molte modifiche progressive,
  con operazioni tipicamente logaritmiche.
```

Microsoft segnala `SortedList<TKey,TValue>` e `SortedDictionary<TKey,TValue>` come opzioni per collezioni ordinate. ([Microsoft Learn][1])

# Collezioni immutabili

Una collezione immutabile non viene modificata dopo la creazione.

Quando aggiungiamo o rimuoviamo un elemento, otteniamo una nuova collezione.

Esempio concettuale:

```csharp id="13cfd6"
using System.Collections.Immutable;

ImmutableList<string> nomi = ImmutableList.Create("Emilie", "Mario");

ImmutableList<string> nuoviNomi = nomi.Add("Anna");

Console.WriteLine(string.Join(", ", nomi));
Console.WriteLine(string.Join(", ", nuoviNomi));
```

Output:

```text id="lbjsxe"
Emilie, Mario
Emilie, Mario, Anna
```

La lista originale non cambia.

---

## Namespace e pacchetto

Le collezioni immutabili sono nel namespace:

```csharp id="7t0xq0"
System.Collections.Immutable
```

Il namespace contiene classi e interfacce che definiscono collezioni immutabili, come `ImmutableList<T>`, `ImmutableDictionary<TKey,TValue>`, `ImmutableHashSet<T>`, `ImmutableQueue<T>` e `ImmutableStack<T>`. ([Microsoft Learn][2])

A seconda del tipo di progetto, può essere necessario aggiungere il pacchetto NuGet `System.Collections.Immutable`.

---

## Quando usare collezioni immutabili

Sono utili quando:

* vuoi evitare modifiche accidentali;
* lavori con dati condivisi;
* vuoi semplificare il ragionamento sullo stato;
* vuoi oggetti thread-safe per progettazione;
* vuoi snapshot stabili;
* lavori con architetture funzionali o event-driven.

Microsoft nota che le collezioni immutabili sono intrinsecamente thread-safe perché le operazioni vengono eseguite su copie della collezione originale, che non può essere modificata. ([Microsoft Learn][1])

---

## Attenzione alle prestazioni

Le collezioni immutabili sono molto utili, ma non sempre sono più veloci.

Microsoft indica che spesso i tipi immutabili sono meno performanti delle controparti mutabili, anche se offrono il vantaggio dell’immutabilità; per esempio `ImmutableList<T>.Add` ha complessità O(log n), mentre `List<T>.Add` è O(1) ammortizzato. ([Microsoft Learn][1])

Regola pratica:

```text id="5dcx4v"
Usa collezioni mutabili quando devi costruire e modificare intensamente dati locali.

Usa collezioni immutabili quando vuoi sicurezza, snapshot e assenza di modifiche accidentali.
```

# Collezioni frozen

Le collezioni **frozen** sono collezioni immutabili ottimizzate per lookup ed enumerazione veloci.

Sono nel namespace:

```csharp id="gmf6ir"
System.Collections.Frozen
```

Il namespace contiene tipi come `FrozenDictionary<TKey,TValue>` e `FrozenSet<T>`, descritti da Microsoft come collezioni immutabili e read-only ottimizzate per lookup ed enumerazione rapidi. ([Microsoft Learn][3])

---

## Esempio con FrozenSet

```csharp id="2il8rd"
using System.Collections.Frozen;

string[] codici =
[
    "AO",
    "TO",
    "MI",
    "GE"
];

FrozenSet<string> codiciValidi = codici.ToFrozenSet();

if (codiciValidi.Contains("AO"))
{
    Console.WriteLine("Codice valido");
}
```

---

## Esempio con FrozenDictionary

```csharp id="7nnbj8"
using System.Collections.Frozen;

Dictionary<string, string> province = new Dictionary<string, string>
{
    ["AO"] = "Aosta",
    ["TO"] = "Torino",
    ["MI"] = "Milano"
};

FrozenDictionary<string, string> provinceFrozen = province.ToFrozenDictionary();

Console.WriteLine(provinceFrozen["AO"]);
```

Output:

```text id="0jxbhl"
Aosta
```

---

## Quando usare frozen

Le collezioni frozen sono adatte quando:

* costruisci la collezione una volta;
* poi la leggi molte volte;
* non devi modificarla;
* ti interessa lookup veloce;
* vuoi un dizionario o set stabile.

Esempi:

```text id="qylpeh"
codici ISO
province
stati applicativi
mapping statici
parole chiave
tabelle di lookup caricate all’avvio
```

---

## Immutable vs Frozen

```text id="vd3a1o"
Immutable collections
  adatte quando vuoi creare nuove versioni della collezione nel tempo.

Frozen collections
  adatte quando costruisci una volta e poi leggi molte volte.
```

Esempio:

```text id="832g87"
Configurazione caricata all’avvio e letta spesso
  FrozenDictionary.

Stato applicativo che produce snapshot successivi
  ImmutableDictionary.
```

# Complessità base

La **complessità algoritmica** descrive come cresce il costo di un’operazione al crescere del numero di elementi.

Le notazioni più comuni sono:

```text id="rjj5cv"
O(1)
  tempo costante.

O(log n)
  tempo logaritmico.

O(n)
  tempo lineare.

O(n log n)
  tempo quasi lineare.

O(n²)
  tempo quadratico.
```

Non serve diventare matematici per usare bene le collezioni, ma è importante avere un’intuizione.

---

## O(1)

Significa che il costo resta circa costante anche se la collezione cresce.

Esempio:

```csharp id="zdewr4"
string nome = nomi[0];
```

Accesso per indice in array o `List<T>`: tipicamente O(1).

---

## O(n)

Significa che, se raddoppiano gli elementi, il lavoro può circa raddoppiare.

Esempio:

```csharp id="msb3yi"
bool trovato = lista.Contains("Emilie");
```

In una `List<T>`, `Contains` deve potenzialmente scorrere tutti gli elementi.

---

## O(log n)

Significa che il costo cresce lentamente rispetto al numero di elementi.

È tipico di strutture ad albero bilanciate, come molte collezioni ordinate.

# Complessità delle collezioni principali

Questa tabella è volutamente semplificata. Serve come guida pratica, non come specifica completa.

| Collezione                      |         Accesso |             Ricerca |                Inserimento | Cancellazione | Note                  |
| ------------------------------- | --------------: | ------------------: | -------------------------: | ------------: | --------------------- |
| `T[]`                           | O(1) per indice |                O(n) |               non dinamico |  non dinamico | Dimensione fissa      |
| `List<T>`                       | O(1) per indice |                O(n) | O(1) ammortizzato in fondo |          O(n) | Ottima lista generale |
| `Dictionary<TKey,TValue>`       |      via chiave |          O(1) medio |                 O(1) medio |    O(1) medio | Lookup per chiave     |
| `HashSet<T>`                    |       no indice |          O(1) medio |                 O(1) medio |    O(1) medio | Unicità e presenza    |
| `Queue<T>`                      |  primo elemento |                O(n) |          O(1) ammortizzato |          O(1) | FIFO                  |
| `Stack<T>`                      | ultimo elemento |                O(n) |          O(1) ammortizzato |          O(1) | LIFO                  |
| `SortedDictionary<TKey,TValue>` |      via chiave |            O(log n) |                   O(log n) |      O(log n) | Ordinato per chiave   |
| `SortedList<TKey,TValue>`       | O(1) per indice | O(log n) per chiave |                       O(n) |          O(n) | Ordinato, compatto    |

Microsoft fornisce una tabella di complessità che conferma, tra gli altri esempi, `List<T>.Add` O(1) ammortizzato, `List<T>` indexer O(1), lookup di `HashSet<T>` O(1) medio, `SortedSet<T>.Add` O(log n), `Dictionary<T>.Add` O(1) medio e `SortedDictionary<T>.Add` O(log n). ([Microsoft Learn][1])

# Esempio completo del capitolo

Costruiamo un piccolo registro clienti che usa più collezioni in modo ragionato:

* `List<Cliente>` per mantenere l’elenco;
* `Dictionary<int, Cliente>` per ricerca rapida per ID;
* `HashSet<string>` per evitare codici fiscali duplicati;
* `Queue<string>` per lavori da eseguire;
* `Stack<string>` per undo semplificato.

```csharp id="h2mqus"
public class Cliente
{
    public int Id { get; }
    public string Nome { get; }
    public string CodiceFiscale { get; }

    public Cliente(int id, string nome, string codiceFiscale)
    {
        if (id <= 0)
        {
            throw new ArgumentOutOfRangeException(nameof(id));
        }

        ArgumentException.ThrowIfNullOrWhiteSpace(nome);
        ArgumentException.ThrowIfNullOrWhiteSpace(codiceFiscale);

        Id = id;
        Nome = nome;
        CodiceFiscale = codiceFiscale;
    }
}

public class RegistroClienti
{
    private readonly List<Cliente> clienti = new();
    private readonly Dictionary<int, Cliente> clientiPerId = new();
    private readonly HashSet<string> codiciFiscali = new(StringComparer.OrdinalIgnoreCase);
    private readonly Queue<string> lavori = new();
    private readonly Stack<string> undo = new();

    public IReadOnlyList<Cliente> Clienti => clienti;

    public void AggiungiCliente(Cliente cliente)
    {
        ArgumentNullException.ThrowIfNull(cliente);

        if (clientiPerId.ContainsKey(cliente.Id))
        {
            throw new InvalidOperationException("ID cliente già presente.");
        }

        if (!codiciFiscali.Add(cliente.CodiceFiscale))
        {
            throw new InvalidOperationException("Codice fiscale già presente.");
        }

        clienti.Add(cliente);
        clientiPerId[cliente.Id] = cliente;

        lavori.Enqueue($"Inviare email benvenuto a {cliente.Nome}");
        undo.Push($"Aggiunto cliente {cliente.Id}");
    }

    public Cliente? TrovaPerId(int id)
    {
        clientiPerId.TryGetValue(id, out Cliente? cliente);
        return cliente;
    }

    public bool EsisteCodiceFiscale(string codiceFiscale)
    {
        return codiciFiscali.Contains(codiceFiscale);
    }

    public void ElaboraLavori()
    {
        while (lavori.Count > 0)
        {
            string lavoro = lavori.Dequeue();
            Console.WriteLine($"Lavoro: {lavoro}");
        }
    }

    public void MostraUltimaOperazione()
    {
        if (undo.Count == 0)
        {
            Console.WriteLine("Nessuna operazione.");
            return;
        }

        Console.WriteLine($"Ultima operazione: {undo.Peek()}");
    }
}
```

Uso:

```csharp id="t1wncm"
RegistroClienti registro = new RegistroClienti();

registro.AggiungiCliente(new Cliente(1, "Emilie", "ABC123"));
registro.AggiungiCliente(new Cliente(2, "Mario", "DEF456"));

Cliente? cliente = registro.TrovaPerId(2);

if (cliente is not null)
{
    Console.WriteLine($"Cliente trovato: {cliente.Nome}");
}

Console.WriteLine(registro.EsisteCodiceFiscale("abc123"));

registro.MostraUltimaOperazione();
registro.ElaboraLavori();
```

Output:

```text id="ul5y1d"
Cliente trovato: Mario
True
Ultima operazione: Aggiunto cliente 2
Lavoro: Inviare email benvenuto a Emilie
Lavoro: Inviare email benvenuto a Mario
```

In questo esempio:

```text id="85g3ns"
List<Cliente>
  conserva l’elenco.

Dictionary<int, Cliente>
  consente ricerca veloce per ID.

HashSet<string>
  impedisce duplicati.

Queue<string>
  gestisce lavori FIFO.

Stack<string>
  conserva operazioni LIFO.
```

# Errori comuni

## Usare sempre List<T>

Sconsigliato:

```csharp id="h4z4yv"
List<Cliente> clienti = new();

Cliente? cliente = clienti.FirstOrDefault(c => c.Id == id);
```

Se fai questa ricerca migliaia di volte, forse serve:

```csharp id="994xkg"
Dictionary<int, Cliente> clientiPerId = new();
```

---

## Usare Dictionary quando la chiave non è davvero unica

Un dizionario richiede chiavi uniche.

Se più clienti possono avere la stessa città, questo è sbagliato:

```csharp id="oacg9t"
Dictionary<string, Cliente> clientiPerCitta = new();
```

Meglio:

```csharp id="f4xxy5"
Dictionary<string, List<Cliente>> clientiPerCitta = new();
```

Oppure, più avanti, usare `GroupBy` con LINQ.

---

## Cercare duplicati con List<T>.Contains in grandi volumi

Sospetto:

```csharp id="nm2bua"
List<int> idGiaVisti = new();

if (!idGiaVisti.Contains(id))
{
    idGiaVisti.Add(id);
}
```

Per molti elementi, meglio:

```csharp id="y8zzhd"
HashSet<int> idGiaVisti = new();

if (idGiaVisti.Add(id))
{
    // aggiunto per la prima volta
}
```

---

## Confondere Queue e Stack

```text id="6dddyq"
Queue<T>
  primo entrato, primo uscito.

Stack<T>
  ultimo entrato, primo uscito.
```

Coda di lavori:

```csharp id="frxvsk"
Queue<string> lavori = new();
```

Undo:

```csharp id="bajrae"
Stack<string> undo = new();
```

---

## Modificare una collezione mentre la si enumera

Sbagliato:

```csharp id="9jwbba"
foreach (Cliente cliente in clienti)
{
    if (cliente.Id == 10)
    {
        clienti.Remove(cliente); // errore possibile
    }
}
```

Meglio creare una lista di elementi da rimuovere, oppure usare un ciclo `for` al contrario, oppure usare metodi dedicati come `RemoveAll` su `List<T>`.

```csharp id="i5ib3c"
clienti.RemoveAll(c => c.Id == 10);
```

---

## Esporre List<T> modificabile dall’esterno

Sconsigliato:

```csharp id="pxw9du"
public List<RigaDocumento> Righe { get; set; } = new();
```

Chiunque può modificare la lista.

Meglio:

```csharp id="t20brm"
private readonly List<RigaDocumento> righe = new();

public IReadOnlyList<RigaDocumento> Righe => righe;
```

E poi esporre metodi controllati:

```csharp id="m9kwfw"
public void AggiungiRiga(RigaDocumento riga)
{
    righe.Add(riga);
}
```

# Linee guida pratiche

```text id="3u7zml"
Array
  dimensione fissa, accesso per indice, API a basso livello.

List<T>
  elenco dinamico ordinato, scelta generale più comune.

Dictionary<TKey,TValue>
  ricerca veloce per chiave.

HashSet<T>
  unicità e test di appartenenza.

Queue<T>
  elaborazione FIFO.

Stack<T>
  elaborazione LIFO.

SortedList<TKey,TValue>
  chiavi ordinate, accesso anche per indice, modifiche non troppo frequenti.

SortedDictionary<TKey,TValue>
  chiavi ordinate, adatto a modifiche frequenti.

Immutable collections
  sicurezza, snapshot, assenza di modifiche accidentali.

Frozen collections
  costruisci una volta, leggi molte volte, lookup ottimizzato.
```

# Riepilogo

In questo capitolo abbiamo studiato le collezioni fondamentali di .NET.

Abbiamo visto:

```text id="yvb0to"
array
  sequenza a dimensione fissa.

List<T>
  lista dinamica.

Dictionary<TKey,TValue>
  coppie chiave/valore.

HashSet<T>
  insieme di elementi unici.

Queue<T>
  coda FIFO.

Stack<T>
  pila LIFO.

SortedList<TKey,TValue>
  dizionario ordinato per chiave.

SortedDictionary<TKey,TValue>
  dizionario ordinato adatto a molte modifiche.

collezioni immutabili
  collezioni che producono nuove versioni senza modificare l’originale.

collezioni frozen
  collezioni read-only ottimizzate per lookup ed enumerazione.
```

Abbiamo introdotto la complessità base:

```text id="asdt3i"
O(1)
  costo costante.

O(log n)
  costo logaritmico.

O(n)
  costo lineare.
```

La regola principale è questa:

```text id="vn4f3g"
Scegli la collezione in base all’operazione più importante:
accesso per indice, ricerca per chiave, unicità, ordinamento,
inserimento, cancellazione o immutabilità.
```

Questa base è essenziale per il prossimo capitolo: LINQ. LINQ rende più espressive le operazioni su collezioni e sequenze, ma per usarlo bene bisogna prima capire su quali strutture dati sta lavorando.

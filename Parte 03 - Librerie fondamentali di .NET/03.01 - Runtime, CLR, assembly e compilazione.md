# 03.01 — Runtime, CLR, assembly e compilazione

Nella Parte 02 abbiamo studiato il linguaggio C#: variabili, tipi, classi, oggetti, generics, eventi, nullable e funzionalità moderne.

Ora passiamo dal linguaggio alla piattaforma.

Scrivere codice C# è solo la prima parte del percorso. Quando premiamo **Build**, **Run** o **Publish**, entrano in gioco molti altri elementi:

* compilatore;
* Intermediate Language;
* assembly;
* runtime;
* CLR;
* JIT;
* Garbage Collector;
* file `.dll` e `.exe`;
* framework target;
* dipendenze;
* configurazioni Debug e Release;
* pubblicazione.

Questo capitolo serve a costruire una mappa mentale di cosa succede davvero tra il file `.cs` e il programma in esecuzione.

Il punto centrale è questo:

```text
C# è il linguaggio che scriviamo.
.NET è la piattaforma che compila, organizza, esegue e distribuisce quel codice.
```

---

# Dal codice sorgente al programma in esecuzione

Quando scriviamo un file C#:

```csharp
Console.WriteLine("Ciao .NET");
```

non stiamo scrivendo direttamente istruzioni per il processore.

Il codice attraversa più fasi:

```text
Codice C#
  ↓
Compilazione
  ↓
Intermediate Language
  ↓
Assembly .dll / .exe
  ↓
Runtime .NET
  ↓
JIT oppure AOT
  ↓
Codice macchina
  ↓
Esecuzione
```

In modo semplificato:

```text
C# non viene normalmente trasformato subito nel codice macchina finale.
Viene prima compilato in un linguaggio intermedio.
Poi il runtime si occupa dell’esecuzione.
```

---

# CLR

CLR significa:

```text
Common Language Runtime
```

È il cuore dell’esecuzione gestita in .NET.

Il CLR si occupa di molte attività fondamentali:

* caricamento degli assembly;
* esecuzione del codice;
* compilazione JIT;
* gestione della memoria;
* Garbage Collection;
* gestione delle eccezioni;
* sicurezza del tipo;
* interoperabilità;
* gestione dei thread;
* supporto alla reflection.

Microsoft descrive il CLR come il runtime che gestisce allocazione e gestione della memoria, ed esegue le applicazioni anche compilando codice al volo tramite JIT. ([Microsoft Learn][1])

---

## Codice gestito

Il codice che gira sotto il controllo del runtime .NET viene chiamato **managed code**, cioè codice gestito.

Esempio:

```csharp
string nome = "Emilie";

Console.WriteLine(nome);
```

Questo codice è gestito dal runtime.

Il CLR controlla molti aspetti dell’esecuzione, per esempio:

* dove vengono allocati gli oggetti;
* quando la memoria può essere liberata;
* come vengono propagate le eccezioni;
* quando il codice IL viene compilato in codice macchina.

---

## Perché è importante

Senza questa idea, .NET può sembrare una scatola nera.

Invece è utile sapere che, quando eseguiamo un’applicazione .NET moderna, non stiamo semplicemente “lanciando un exe” nel senso tradizionale. Stiamo avviando codice che viene caricato ed eseguito da un runtime.

---

# Intermediate Language

Quando compiliamo C#, il compilatore non genera normalmente codice macchina specifico per il processore.

Genera un linguaggio intermedio chiamato:

```text
IL
Intermediate Language
```

o anche:

```text
MSIL
Microsoft Intermediate Language
```

Esempio concettuale:

```csharp
int Somma(int a, int b)
{
    return a + b;
}
```

Il compilatore C# trasforma questo metodo in IL.

Poi sarà il runtime a trasformare l’IL in codice macchina eseguibile dalla CPU.

---

## Perché usare un linguaggio intermedio

L’IL permette a .NET di separare due momenti:

```text
Compilazione del linguaggio
  C# -> IL

Esecuzione sulla macchina reale
  IL -> codice macchina
```

Questo ha alcuni vantaggi:

* più linguaggi possono compilare verso lo stesso runtime;
* il runtime può ottimizzare il codice in base alla piattaforma;
* lo stesso assembly può essere eseguito su architetture diverse, entro i limiti del target;
* strumenti come reflection, debugging e analisi possono lavorare su metadati comuni.

---

# JIT

JIT significa:

```text
Just-In-Time
```

Il compilatore JIT trasforma IL in codice macchina **durante l’esecuzione**.

Microsoft descrive la compilazione JIT come il processo che traduce MSIL in codice macchina nativo a runtime, quando i metodi vengono chiamati per la prima volta. ([Microsoft Learn][2])

Esempio concettuale:

```text
Il programma chiama CalcolaTotale()
  ↓
Il metodo è ancora in IL
  ↓
Il JIT lo compila in codice macchina
  ↓
Il processore lo esegue
```

---

## Perché il JIT è utile

Il JIT può ottimizzare il codice sapendo dove sta girando.

Per esempio:

* sistema operativo;
* architettura CPU;
* caratteristiche del processore;
* profilo di esecuzione;
* metodi realmente chiamati.

Questo permette ottimizzazioni specifiche.

---

## Svantaggio del JIT

Il JIT lavora durante l’esecuzione.

Questo significa che una parte del costo di compilazione viene pagata a runtime.

Per molte applicazioni va benissimo.

In scenari particolari, però, può essere importante ridurre:

* tempo di avvio;
* uso memoria;
* dipendenza dal runtime installato;
* costo della compilazione durante l’esecuzione.

Qui entra in gioco AOT.

---

# AOT

AOT significa:

```text
Ahead-Of-Time
```

Con AOT, la compilazione verso codice macchina avviene prima dell’esecuzione.

Invece di compilare al volo con JIT, si cerca di produrre codice nativo in anticipo.

Microsoft distingue AOT da JIT spiegando che la compilazione AOT avviene prima dell’esecuzione dell’applicazione, generalmente su una macchina diversa da quella in cui il programma gira. ([Microsoft Learn][3])

---

## JIT vs AOT

```text
JIT
  IL compilato in codice macchina durante l’esecuzione.

AOT
  codice compilato in anticipo prima dell’esecuzione.
```

---

## Quando AOT è interessante

AOT può essere utile per:

* avvio più rapido;
* minore memoria a runtime;
* distribuzione più autonoma;
* ambienti cloud/serverless;
* container leggeri;
* applicazioni console piccole;
* scenari dove il runtime JIT non è ideale.

---

## Attenzione

AOT non è una bacchetta magica.

Può introdurre limitazioni o complessità, soprattutto con:

* reflection;
* serializzazione dinamica;
* caricamento dinamico di assembly;
* librerie non compatibili;
* codice che assume la presenza del JIT.

Regola pratica:

```text
JIT
  scelta normale per molte applicazioni .NET.

AOT
  scelta mirata quando startup, memoria o distribuzione lo richiedono.
```

---

# Garbage Collector

Il **Garbage Collector**, spesso abbreviato in **GC**, è il componente del runtime che gestisce automaticamente la memoria degli oggetti.

Quando creiamo un oggetto:

```csharp
Cliente cliente = new Cliente("Emilie");
```

l’oggetto viene allocato in memoria.

Quando quell’oggetto non è più raggiungibile dal programma, il Garbage Collector può liberare la memoria.

---

## Perché serve il Garbage Collector

In linguaggi senza gestione automatica della memoria, il programmatore deve spesso liberare manualmente la memoria.

In C#, per gli oggetti gestiti, normalmente non facciamo questo:

```csharp
free(cliente);
```

Il runtime si occupa di capire quando la memoria può essere recuperata.

---

## Il GC non libera tutto subito

Un errore comune è pensare:

```text
Appena una variabile non serve più, la memoria viene liberata immediatamente.
```

Non è così.

Il Garbage Collector lavora quando il runtime decide che è opportuno.

Quindi:

```csharp
Cliente cliente = new Cliente("Emilie");
cliente = null;
```

non significa che la memoria venga liberata immediatamente.

Significa solo che quell’oggetto potrebbe diventare candidato alla raccolta, se non esistono altri riferimenti.

---

## IDisposable non è Garbage Collector

Il GC gestisce memoria gestita.

Ma alcune risorse non sono solo memoria:

* file aperti;
* connessioni database;
* socket;
* handle di sistema;
* stream;
* risorse grafiche.

Per queste risorse si usa spesso `IDisposable` e il costrutto `using`.

Esempio:

```csharp
using StreamReader reader = new StreamReader("dati.txt");

string contenuto = reader.ReadToEnd();
```

Qui `using` assicura che la risorsa venga rilasciata correttamente.

Regola pratica:

```text
Garbage Collector
  gestisce memoria degli oggetti .NET.

Dispose / using
  gestisce risorse che devono essere chiuse o rilasciate.
```

---

# Assembly

Un **assembly** è l’unità fondamentale di distribuzione e riuso in .NET.

Microsoft definisce gli assembly come unità fondamentali di deployment, versionamento, riuso e sicurezza; un assembly è una raccolta di tipi e risorse costruiti per lavorare insieme, e può avere forma `.dll` o `.exe`. ([Microsoft Learn][4])

Quando compiliamo un progetto .NET, otteniamo normalmente un assembly.

Esempi:

```text
MioProgetto.dll
MioProgetto.exe
Gestionale.Domain.dll
Gestionale.Web.dll
```

---

## Cosa contiene un assembly

Un assembly può contenere:

* IL;
* metadati;
* tipi;
* risorse;
* informazioni di versione;
* riferimenti ad altri assembly;
* manifest.

Esempio:

```text
Gestionale.Domain.dll
  contiene:
    Cliente
    Documento
    RigaDocumento
    Servizi di dominio
    metadati
    riferimenti ad altri assembly
```

---

## DLL ed EXE

In .NET possiamo avere assembly in forma:

```text
.dll
  libreria o componente usabile da altri progetti.

.exe
  applicazione eseguibile.
```

Nei progetti .NET moderni, anche molte applicazioni producono una `.dll` principale, avviata dal comando `dotnet`.

Esempio:

```bash
dotnet MioProgetto.dll
```

In altri casi la pubblicazione produce anche un eseguibile specifico per il sistema operativo.

---

# Namespace

Un **namespace** è un contenitore logico per organizzare tipi.

Esempio:

```csharp
namespace Gestionale.Clienti;

public class Cliente
{
}
```

Qui la classe `Cliente` appartiene al namespace:

```text
Gestionale.Clienti
```

La specifica C# descrive i namespace come sistema di organizzazione interna ed esterna dei programmi: servono a organizzare elementi e a presentarli ad altri programmi. ([Microsoft Learn][5])

---

## Perché servono i namespace

I namespace servono a:

* organizzare il codice;
* evitare conflitti di nomi;
* rendere più chiara la struttura logica;
* separare aree funzionali;
* semplificare l’uso dei tipi tramite `using`.

Esempio:

```csharp
using Gestionale.Clienti;

Cliente cliente = new Cliente();
```

Senza `using`, dovremmo scrivere:

```csharp
Gestionale.Clienti.Cliente cliente = new Gestionale.Clienti.Cliente();
```

---

# Differenza tra namespace e assembly

Questa è una distinzione importantissima.

```text
Namespace
  organizzazione logica dei tipi nel codice.

Assembly
  unità fisica/compilata prodotta dal build.
```

Un namespace non è un file.
Un namespace non è una DLL.
Un assembly non coincide necessariamente con un namespace.

Microsoft chiarisce nelle linee guida che nomi di assembly e DLL non devono per forza corrispondere ai namespace, anche se spesso è ragionevole scegliere nomi coerenti. ([Microsoft Learn][6])

---

## Esempio

Possiamo avere un assembly:

```text
Gestionale.dll
```

che contiene più namespace:

```text
Gestionale.Clienti
Gestionale.Documenti
Gestionale.Fatturazione
Gestionale.Sicurezza
```

Oppure possiamo avere più assembly che contengono namespace correlati:

```text
Gestionale.Domain.dll
  Gestionale.Domain.Clienti
  Gestionale.Domain.Documenti

Gestionale.Infrastructure.dll
  Gestionale.Infrastructure.Database
  Gestionale.Infrastructure.Email

Gestionale.Web.dll
  Gestionale.Web.Controllers
  Gestionale.Web.Pages
```

---

## Cartelle e namespace

Nei progetti C#, spesso le cartelle riflettono i namespace.

Esempio:

```text
/Clienti/Cliente.cs
```

namespace:

```csharp
namespace Gestionale.Clienti;
```

Ma la cartella non determina automaticamente il namespace nel linguaggio C#.

È una convenzione utile, non una regola assoluta del compilatore.

---

# Target Framework Moniker

Il **Target Framework Moniker**, abbreviato spesso in **TFM**, indica per quale framework o piattaforma viene compilato un progetto.

Nel file `.csproj` troviamo qualcosa del genere:

```xml
<PropertyGroup>
  <TargetFramework>net10.0</TargetFramework>
</PropertyGroup>
```

Oppure:

```xml
<PropertyGroup>
  <TargetFramework>net8.0</TargetFramework>
</PropertyGroup>
```

Microsoft spiega che i TFM `net8.0`, `net9.0` e `net10.0` includono tecnologie cross-platform, e che TFM specifici del sistema operativo rendono disponibili API specifiche di quel sistema. ([Microsoft Learn][7])

---

## `net8.0`

```xml
<TargetFramework>net8.0</TargetFramework>
```

Indica che il progetto targetta .NET 8.

È usato per applicazioni moderne cross-platform compatibili con .NET 8.

---

## `net9.0`

```xml
<TargetFramework>net9.0</TargetFramework>
```

Indica che il progetto targetta .NET 9.

---

## `net10.0`

```xml
<TargetFramework>net10.0</TargetFramework>
```

Indica che il progetto targetta .NET 10.

È un TFM moderno per applicazioni .NET 10.

---

## `net48`

```xml
<TargetFramework>net48</TargetFramework>
```

Indica che il progetto targetta .NET Framework 4.8.

Questo è diverso da `net8.0`, `net9.0` o `net10.0`.

`net48` appartiene alla linea storica .NET Framework, legata a Windows.

---

## Esempio di differenza

```xml
<TargetFramework>net48</TargetFramework>
```

è tipico di:

* applicazioni legacy .NET Framework;
* Windows Forms vecchio stile;
* WPF su .NET Framework;
* ASP.NET WebForms;
* librerie usate da applicazioni storiche.

```xml
<TargetFramework>net10.0</TargetFramework>
```

è tipico di:

* console app moderne;
* ASP.NET Core;
* Web API moderne;
* worker service;
* librerie moderne cross-platform.

---

## Target multipli

Un progetto può anche targettare più framework.

```xml
<PropertyGroup>
  <TargetFrameworks>net48;net8.0</TargetFrameworks>
</PropertyGroup>
```

Questo significa che il progetto viene compilato per entrambi i target.

È utile per librerie che devono essere usate sia da applicazioni legacy sia da applicazioni moderne.

Attenzione però: il codice deve essere compatibile con entrambi i framework, oppure usare compilazione condizionale.

---

# Dipendenze

Un progetto raramente vive da solo.

Ha bisogno di dipendenze:

* altri progetti della soluzione;
* pacchetti NuGet;
* librerie di .NET;
* assembly esterni;
* SDK;
* analyzer;
* strumenti di build.

Nel file `.csproj` possiamo trovare:

```xml
<ItemGroup>
  <PackageReference Include="Dapper" Version="2.1.66" />
</ItemGroup>
```

Oppure:

```xml
<ItemGroup>
  <ProjectReference Include="..\Gestionale.Domain\Gestionale.Domain.csproj" />
</ItemGroup>
```

---

## Dipendenza da pacchetto

Una dipendenza NuGet viene indicata con `PackageReference`.

```xml
<PackageReference Include="Newtonsoft.Json" Version="13.0.3" />
```

Questo dice:

```text
Il progetto usa il pacchetto Newtonsoft.Json versione 13.0.3.
```

---

## Dipendenza da progetto

Una dipendenza da un altro progetto della stessa solution viene indicata con `ProjectReference`.

```xml
<ProjectReference Include="..\Gestionale.Domain\Gestionale.Domain.csproj" />
```

Questo dice:

```text
Il progetto corrente dipende da Gestionale.Domain.
```

# Reference

Una **reference** è un riferimento a un altro assembly o progetto necessario per compilare ed eseguire il codice.

Esempio:

```csharp
using Gestionale.Domain.Clienti;
```

Questo `using` funziona solo se il progetto ha accesso all’assembly che contiene quel namespace.

Il namespace serve a trovare logicamente il tipo nel codice.
La reference serve a rendere disponibile fisicamente quell’assembly al compilatore.

---

## Namespace senza reference

Se scriviamo:

```csharp
using Gestionale.Domain.Clienti;
```

ma il progetto non ha una reference a `Gestionale.Domain`, il compilatore non trova il namespace.

L’errore tipico è:

```text
The type or namespace name could not be found
```

---

## Reference senza using

Possiamo avere la reference corretta ma non il `using`.

In quel caso possiamo usare il nome completo:

```csharp
Gestionale.Domain.Clienti.Cliente cliente = new Gestionale.Domain.Clienti.Cliente();
```

Oppure aggiungere:

```csharp
using Gestionale.Domain.Clienti;
```

Regola pratica:

```text
Reference
  rende disponibile l’assembly.

using
  rende comodo usare i namespace contenuti.
```

# Build

La **build** è il processo che trasforma il progetto sorgente in output compilato.

Con la CLI:

```bash
dotnet build
```

Durante la build avvengono varie attività:

* ripristino dipendenze, se necessario;
* compilazione del codice C#;
* generazione degli assembly;
* copia di file necessari;
* esecuzione di analyzer;
* produzione di output in `bin`;
* generazione di file intermedi in `obj`.

La documentazione del comando `dotnet build` indica che la configurazione predefinita per la maggior parte dei progetti è `Debug`, salvo override nelle impostazioni del progetto. ([Microsoft Learn][8])

---

## Cartelle `bin` e `obj`

In un progetto .NET troviamo spesso:

```text
bin
obj
```

`obj` contiene file intermedi usati durante la compilazione.

`bin` contiene l’output finale della build.

Esempio:

```text
bin/Debug/net10.0/MioProgetto.dll
```

Oppure:

```text
bin/Release/net10.0/MioProgetto.dll
```

---

# Debug

La configurazione **Debug** è pensata per lo sviluppo.

Caratteristiche tipiche:

* informazioni di debug;
* ottimizzazioni ridotte;
* migliore esperienza con breakpoint;
* più facile ispezione del codice;
* output in `bin/Debug`.

Esempio:

```bash
dotnet build -c Debug
```

Oppure:

```bash
dotnet run -c Debug
```

---

## Quando usare Debug

Usa Debug quando:

* stai sviluppando;
* devi mettere breakpoint;
* devi ispezionare variabili;
* devi usare Hot Reload;
* devi diagnosticare problemi in locale.

---

# Release

La configurazione **Release** è pensata per la distribuzione.

Caratteristiche tipiche:

* ottimizzazioni abilitate;
* output più adatto alla produzione;
* debug meno immediato;
* output in `bin/Release`.

Esempio:

```bash
dotnet build -c Release
```

---

## Debug vs Release

```text
Debug
  sviluppo, diagnostica, breakpoint.

Release
  distribuzione, ottimizzazione, produzione.
```

Regola pratica:

```text
Sviluppa in Debug.
Distribuisci in Release.
```

---

# Pubblicazione

La **pubblicazione** prepara l’applicazione per essere distribuita.

Con la CLI:

```bash
dotnet publish
```

Il comando `dotnet publish` pubblica un progetto o una soluzione .NET in una directory; inoltre chiama MSBuild e passa i parametri ricevuti a MSBuild. ([Microsoft Learn][9])

Esempio:

```bash
dotnet publish -c Release -o ./publish
```

Questo produce l’output pubblicato nella cartella:

```text
publish
```

---

## Publish non è solo build

`dotnet build` compila.

`dotnet publish` prepara l’app per essere distribuita.

La pubblicazione può includere:

* assembly compilati;
* file di configurazione;
* dipendenze;
* runtime, se self-contained;
* file statici;
* asset web;
* eseguibili specifici;
* trimming, se configurato;
* AOT, se configurato.

---

## Release predefinito da .NET 8 in poi

Da .NET 8, `dotnet publish` usa `Release` come configurazione predefinita per progetti con `TargetFramework` `net8.0` o successivo; per SDK o target precedenti il default era `Debug`, salvo override. ([Microsoft Learn][10])

Quindi con un progetto `net8.0`, `net9.0` o `net10.0`:

```bash
dotnet publish
```

usa normalmente Release.

Ma nei comandi didattici è spesso meglio essere espliciti:

```bash
dotnet publish -c Release
```

---

# Pubblicazione framework-dependent e self-contained

Quando pubblichiamo, possiamo scegliere diversi modelli.

## Framework-dependent

L’app pubblicata richiede che il runtime .NET sia installato sulla macchina di destinazione.

Esempio concettuale:

```bash
dotnet publish -c Release
```

Vantaggi:

* output più leggero;
* aggiornamenti runtime gestiti separatamente;
* adatto a server controllati.

Svantaggi:

* il runtime corretto deve essere installato.

---

## Self-contained

L’app include anche il runtime necessario.

Esempio:

```bash
dotnet publish -c Release -r win-x64 --self-contained true
```

Vantaggi:

* non richiede runtime installato separatamente;
* più prevedibile sulla macchina target.

Svantaggi:

* output più grande;
* bisogna scegliere il Runtime Identifier;
* aggiornamenti runtime da gestire ripubblicando.

---

## Runtime Identifier

Il Runtime Identifier, spesso abbreviato in **RID**, indica piattaforma e architettura.

Esempi:

```text
win-x64
linux-x64
osx-arm64
```

Esempio:

```bash
dotnet publish -c Release -r win-x64
```

```text
ImplicitUsings enable
```

# Esempio con dipendenze

```xml
<Project Sdk="Microsoft.NET.Sdk">

  <PropertyGroup>
```

# Errori comuni

## Confondere C# e .NET

C# è il linguaggio.

.NET è la piattaforma.

```text
C#
  sintassi, tipi, classi, metodi, generics.

.NET
  runtime, librerie, SDK, CLI, assembly, GC, JIT, NuGet.
```

---

## Confondere namespace e assembly

Sbagliato:

```text
Ho il namespace, quindi ho anche la DLL.
```

No.

Il namespace è logico.
L’assembly è fisico.

Serve una reference corretta all’assembly che contiene il tipo.

---

## Pensare che una cartella determini il namespace

La cartella è una convenzione.

Il namespace è dichiarato nel codice:

```csharp
namespace Gestionale.Clienti;
```

---

## Usare sempre `net48` per abitudine

`net48` è .NET Framework 4.8.

Ha senso per applicazioni legacy o vincoli specifici.

Per nuovi progetti moderni, di solito si usa un TFM moderno come:

```xml
<TargetFramework>net10.0</TargetFramework>
```

o la versione supportata richiesta dal progetto.

---

## Pubblicare in Debug

Sconsigliato:

```bash
dotnet publish -c Debug
```

Per produzione, normalmente:

```bash
dotnet publish -c Release
```

---

## Confondere build e publish

```text
Build
  compila il progetto.

Publish
  prepara l’applicazione per essere distribuita.
```

---

## Pensare che il Garbage Collector chiuda file e connessioni subito

Il GC gestisce memoria gestita.

Per file, stream, connessioni e risorse simili serve `Dispose`/`using`.

In questo capitolo abbiamo iniziato la Parte 03, passando dal linguaggio C# alla piattaforma .NET.

Abbiamo visto:

```text
CLR
  runtime che esegue codice .NET, gestisce memoria, JIT, eccezioni e molto altro.

Intermediate Language
  linguaggio intermedio prodotto dalla compilazione C#.

JIT
  compilazione da IL a codice macchina durante l’esecuzione.

AOT
  compilazione anticipata prima dell’esecuzione.

Garbage Collector
  componente che gestisce automaticamente la memoria degli oggetti gestiti.
```

Abbiamo poi introdotto:

```text
Assembly
  unità compilata e distribuibile, come DLL o EXE.

Namespace
  organizzazione logica dei tipi.

Reference
  collegamento a un assembly o progetto necessario.

Dipendenze
  pacchetti, progetti o librerie usati dal progetto.
```

Abbiamo distinto:

```text
Namespace
  nome logico nel codice.

Assembly
  file compilato fisico.

using
  comodità sintattica per usare namespace.

reference
  disponibilità reale dell’assembly al compilatore.
```

Abbiamo visto i Target Framework Moniker:

```text
net8.0
net9.0
net10.0
  .NET moderno cross-platform.

net48
  .NET Framework 4.8, linea storica Windows.
```

Infine abbiamo distinto:

```text
Build
  compila.

Debug
  configurazione per sviluppo e diagnostica.

Release
  configurazione per distribuzione.

Publish
  prepara l’applicazione per essere installata o copiata in ambiente target.
```

La regola principale è questa:

```text
Capire .NET significa capire che il codice C# è solo l’inizio.
Il runtime, gli assembly, il target framework e il processo di pubblicazione determinano come quel codice vive davvero.
```

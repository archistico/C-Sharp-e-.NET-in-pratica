# 03.07 — Logging, configurazione e opzioni

Nei capitoli precedenti abbiamo visto file, stream, serializzazione e Regex. Ora affrontiamo un tema che distingue il codice didattico dal codice da applicazione reale: **configurazione e logging**.

Un programma serio non dovrebbe avere valori importanti scritti direttamente nel codice.

Esempi da evitare:

```csharp
string connectionString = "Server=PROD-SQL;Database=Gestionale;User Id=sa;Password=123456;";
int timeoutSecondi = 30;
string cartellaExport = @"C:\Export";
```

Questi valori cambiano tra ambienti:

```text
sviluppo
test
collaudo
produzione
container
server cloud
macchina locale
```

Allo stesso modo, un’applicazione reale deve produrre log utili:

```text
quando parte
che configurazione importante ha caricato
quali operazioni sta eseguendo
quali errori incontra
quale richiesta ha generato un problema
quale utente o processo ha innescato un flusso
```

Il punto centrale del capitolo è questo:

```text
Configurazione e logging non sono dettagli accessori.
Sono infrastruttura di base per applicazioni manutenibili, diagnosticabili e distribuibili.
```

---

# Configurazione in .NET

La configurazione in .NET moderno è basata su un modello a provider.

I valori possono arrivare da sorgenti diverse:

```text
appsettings.json
appsettings.Development.json
variabili d’ambiente
argomenti da riga di comando
user secrets
Azure Key Vault
Azure App Configuration
provider personalizzati
```

Tutte queste sorgenti vengono viste attraverso un’interfaccia comune:

```csharp
IConfiguration
```

Microsoft descrive `IConfiguration` come una rappresentazione unificata delle proprietà di configurazione dell’applicazione, indipendente dai provider sottostanti. La configurazione viene letta da più sorgenti e i provider aggiunti dopo possono sovrascrivere i valori dei provider precedenti. ([Microsoft Learn][1])

---

# `appsettings.json`

Il file `appsettings.json` è il file di configurazione più comune nelle applicazioni .NET moderne.

Esempio:

```json
{
  "Application": {
    "Name": "GestionaleClienti",
    "ExportFolder": "export",
    "MaxItemsPerPage": 50
  },
  "ConnectionStrings": {
    "DefaultConnection": "Server=.;Database=Gestionale;Trusted_Connection=True;"
  },
  "Logging": {
    "LogLevel": {
      "Default": "Information",
      "Microsoft": "Warning"
    }
  }
}
```

Questo file contiene configurazioni gerarchiche.

Possiamo leggere un valore con una chiave separata da `:`:

```csharp
string? nomeApp = configuration["Application:Name"];
```

Oppure:

```csharp
string? connectionString = configuration.GetConnectionString("DefaultConnection");
```

---

## Esempio in una console app moderna

File `appsettings.json`:

```json
{
  "Application": {
    "Name": "DemoConfigurazione",
    "ExportFolder": "export"
  }
}
```

Programma:

```csharp
using Microsoft.Extensions.Configuration;

IConfiguration configuration = new ConfigurationBuilder()
    .AddJsonFile("appsettings.json", optional: false, reloadOnChange: true)
    .Build();

string? nome = configuration["Application:Name"];
string? exportFolder = configuration["Application:ExportFolder"];

Console.WriteLine(nome);
Console.WriteLine(exportFolder);
```

Nei progetti ASP.NET Core e Worker Service molta configurazione viene già predisposta dal Generic Host; nelle console app pure, invece, spesso bisogna aggiungere esplicitamente i pacchetti e costruire la configurazione. La documentazione .NET segnala proprio che le app console create con i template base non espongono automaticamente tutte le funzionalità di configurazione, mentre il Generic Host le integra più naturalmente. ([Microsoft Learn][1])

---

# Configurazione per ambiente

Di solito non vogliamo la stessa configurazione in sviluppo e produzione.

Per questo si usano file specifici:

```text
appsettings.json
appsettings.Development.json
appsettings.Staging.json
appsettings.Production.json
```

Esempio `appsettings.json`:

```json
{
  "Application": {
    "ExportFolder": "export"
  }
}
```

Esempio `appsettings.Development.json`:

```json
{
  "Application": {
    "ExportFolder": "export-dev"
  }
}
```

Se l’ambiente corrente è `Development`, il valore di `appsettings.Development.json` può sovrascrivere quello base.

ASP.NET Core carica le sorgenti di configurazione predefinite con un ordine preciso: `appsettings.json`, poi `appsettings.{Environment}.json`, poi user secrets in `Development`, poi variabili d’ambiente e infine argomenti da riga di comando con priorità più alta. ([Microsoft Learn][2])

---

# Environment

L’ambiente indica dove sta girando l’applicazione.

Valori comuni:

```text
Development
Staging
Production
```

In ASP.NET Core e nel Generic Host si usa spesso:

```csharp
IHostEnvironment
```

oppure:

```csharp
IWebHostEnvironment
```

Esempio:

```csharp
public class AmbienteService
{
    private readonly IHostEnvironment environment;

    public AmbienteService(IHostEnvironment environment)
    {
        this.environment = environment;
    }

    public void StampaAmbiente()
    {
        Console.WriteLine(environment.EnvironmentName);
    }
}
```

---

## Variabile d’ambiente

Per ASP.NET Core l’ambiente può essere impostato con:

```text
ASPNETCORE_ENVIRONMENT
```

Per Generic Host:

```text
DOTNET_ENVIRONMENT
```

Esempio PowerShell:

```powershell
$env:DOTNET_ENVIRONMENT = "Development"
dotnet run
```

Esempio Linux/macOS:

```bash
DOTNET_ENVIRONMENT=Development dotnet run
```

---

## Perché l’ambiente è importante

L’ambiente influenza:

```text
quale appsettings.{Environment}.json viene caricato
se vengono caricati gli user secrets
livello di dettaglio degli errori
logging
feature abilitate
endpoint di test o produzione
comportamenti diagnostici
```

Regola pratica:

```text
Development
  più diagnostica, configurazioni locali, user secrets.

Production
  sicurezza, logging controllato, niente dettagli sensibili all’utente.
```

# User secrets

Gli **user secrets** servono a salvare segreti locali durante lo sviluppo senza metterli nel repository Git.

Esempi di segreti:

```text
connection string con password
API key
token
password SMTP
chiavi private di sviluppo
```

Non vanno messi in `appsettings.json` se il file viene versionato.

Microsoft indica che i segreti utente vengono caricati in ambiente `Development` e che hanno precedenza sui file `appsettings.json` e `appsettings.{Environment}.json`; lo strumento Secret Manager salva i segreti in un file JSON nel profilo locale dell’utente, non nel progetto. ([Microsoft Learn][2])

---

## Inizializzare user secrets

Da terminale, dentro il progetto:

```bash
dotnet user-secrets init
```

Questo aggiunge al `.csproj` un identificativo:

```xml
<PropertyGroup>
  <UserSecretsId>...</UserSecretsId>
</PropertyGroup>
```

---

## Impostare un segreto

```bash
dotnet user-secrets set "ConnectionStrings:DefaultConnection" "Server=.;Database=Demo;User Id=demo;Password=secret;"
```

Oppure:

```bash
dotnet user-secrets set "Email:SmtpPassword" "password-di-sviluppo"
```

---

## Leggere un segreto

Nel codice il segreto si legge come qualunque configurazione:

```csharp
string? connectionString = configuration.GetConnectionString("DefaultConnection");
```

Il codice non deve sapere se il valore arriva da:

```text
appsettings.json
user secrets
variabile d’ambiente
Key Vault
```

Questa è una delle forze del sistema di configurazione.

---

## User secrets non sono per produzione

Gli user secrets sono pensati per sviluppo locale.

In produzione è meglio usare:

```text
variabili d’ambiente
secret manager della piattaforma
Azure Key Vault
AWS Secrets Manager
Kubernetes Secrets
Docker secrets
sistemi aziendali dedicati
```

Regola pratica:

```text
Mai committare segreti nel repository.
User secrets per sviluppo.
Secret store sicuro per produzione.
```

# Options pattern

Leggere valori così:

```csharp
configuration["Application:ExportFolder"]
```

funziona, ma in applicazioni grandi diventa fragile.

Problemi:

```text
chiavi scritte come stringhe sparse nel codice
nessuna tipizzazione forte
refactoring difficile
errori scoperti tardi
validazione più scomoda
```

L’**Options pattern** risolve questo problema legando una sezione di configurazione a una classe C#.

Microsoft descrive il modello di opzioni come un pattern che usa classi per fornire accesso fortemente tipizzato a gruppi di impostazioni correlate. Questo aiuta anche a separare meglio le impostazioni per scenario. ([Microsoft Learn][3])

---

## Classe options

File `ApplicationOptions.cs`:

```csharp
public class ApplicationOptions
{
    public const string SectionName = "Application";

    public string Name { get; set; } = "";
    public string ExportFolder { get; set; } = "";
    public int MaxItemsPerPage { get; set; } = 50;
}
```

File `appsettings.json`:

```json
{
  "Application": {
    "Name": "GestionaleClienti",
    "ExportFolder": "export",
    "MaxItemsPerPage": 100
  }
}
```

Registrazione:

```csharp
builder.Services.Configure<ApplicationOptions>(
    builder.Configuration.GetSection(ApplicationOptions.SectionName));
```

Uso:

```csharp
using Microsoft.Extensions.Options;

public class ExportService
{
    private readonly ApplicationOptions options;

    public ExportService(IOptions<ApplicationOptions> options)
    {
        this.options = options.Value;
    }

    public void Esporta()
    {
        Console.WriteLine(options.ExportFolder);
    }
}
```

---

# `IOptions<T>`, `IOptionsSnapshot<T>`, `IOptionsMonitor<T>`

Nel mondo .NET ci sono tre interfacce molto usate.

```text
IOptions<T>
  legge le opzioni una volta; adatto a molti servizi semplici.

IOptionsSnapshot<T>
  ricalcola le opzioni per scope; usato soprattutto in app web scoped.

IOptionsMonitor<T>
  permette di osservare cambiamenti e ottenere valori aggiornati.
```

Uso comune:

```csharp
public class MioService
{
    private readonly ApplicationOptions options;

    public MioService(IOptions<ApplicationOptions> options)
    {
        this.options = options.Value;
    }
}
```

Per la maggior parte degli esempi didattici iniziali, `IOptions<T>` è sufficiente.

---

## Validazione delle opzioni

Le opzioni possono essere validate.

Classe:

```csharp
using System.ComponentModel.DataAnnotations;

public class EmailOptions
{
    public const string SectionName = "Email";

    [Required]
    public string SmtpServer { get; set; } = "";

    [Range(1, 65535)]
    public int Port { get; set; }

    [Required]
    public string Sender { get; set; } = "";
}
```

Configurazione:

```json
{
  "Email": {
    "SmtpServer": "smtp.example.com",
    "Port": 587,
    "Sender": "noreply@example.com"
  }
}
```

Registrazione:

```csharp
builder.Services
    .AddOptions<EmailOptions>()
    .Bind(builder.Configuration.GetSection(EmailOptions.SectionName))
    .ValidateDataAnnotations()
    .ValidateOnStart();
```

`ValidateOnStart` è molto utile perché fa fallire l’applicazione all’avvio se la configurazione è sbagliata, invece di scoprire il problema più tardi.

# Logging con `ILogger`

Il logging in .NET moderno ruota attorno a:

```csharp
ILogger
ILogger<T>
```

Esempio:

```csharp
using Microsoft.Extensions.Logging;

public class ClienteService
{
    private readonly ILogger<ClienteService> logger;

    public ClienteService(ILogger<ClienteService> logger)
    {
        this.logger = logger;
    }

    public void RegistraCliente(string nome)
    {
        logger.LogInformation("Registrazione cliente {Nome}", nome);

        try
        {
            // Logica applicativa...
        }
        catch (Exception ex)
        {
            logger.LogError(ex, "Errore durante la registrazione del cliente {Nome}", nome);
            throw;
        }
    }
}
```

.NET supporta logging strutturato ad alte prestazioni tramite `ILogger`, con provider configurabili per scrivere i log su destinazioni diverse. Esistono provider integrati e provider di terze parti. ([Microsoft Learn][4])

# Livelli di log

I livelli indicano la gravità o il dettaglio del messaggio.

I principali sono:

```text
Trace
Debug
Information
Warning
Error
Critical
None
```

---

## `Trace`

Molto dettagliato.

```csharp
logger.LogTrace("Valore intermedio {Valore}", valore);
```

Uso:

```text
diagnostica molto fine
analisi profonda
quasi mai attivo in produzione ordinaria
```

---

## `Debug`

Informazioni utili in sviluppo.

```csharp
logger.LogDebug("Cliente caricato dal repository: {ClienteId}", clienteId);
```

Uso:

```text
debug locale
test
diagnostica tecnica dettagliata
```

---

## `Information`

Flusso normale dell’applicazione.

```csharp
logger.LogInformation("Ordine {OrdineId} confermato", ordineId);
```

Uso:

```text
operazioni importanti
avvio applicazione
elaborazioni completate
eventi applicativi normali
```

---

## `Warning`

Qualcosa di anomalo, ma l’app continua.

```csharp
logger.LogWarning("Cliente {ClienteId} non trovato", clienteId);
```

Uso:

```text
dati mancanti
retry
configurazioni sospette
situazioni recuperabili
```

---

## `Error`

Errore che impedisce una specifica operazione.

```csharp
logger.LogError(ex, "Errore durante il salvataggio del cliente {ClienteId}", clienteId);
```

Uso:

```text
eccezioni
operazioni fallite
problemi da indagare
```

---

## `Critical`

Errore grave dell’applicazione o del sistema.

```csharp
logger.LogCritical(ex, "Errore critico durante l'avvio dell'applicazione");
```

Uso:

```text
app non avviabile
dati corrotti
servizio non disponibile
guasto grave
```

---

Significato:


# Provider di logging

Un provider decide dove finiscono i log.

Esempi:

```text
Console
Debug
EventSource
EventLog su Windows
file tramite provider esterni
Serilog
NLog
Application Insights
OpenTelemetry

```

In una console/worker app:

```csharp

```

In ASP.NET Core, i template registrano provider predefiniti. Possiamo aggiungere o rimuovere provider in base all’ambiente.

---

## Provider multipli

La stessa chiamata:

```csharp
logger.LogInformation("Applicazione avviata");
```

può finire contemporaneamente in:

```text
console
file
sistema centralizzato
dashboard cloud
```

Il codice applicativo dipende da `ILogger`, non dal provider specifico.

Questo è importante: il servizio non deve sapere se il log finirà su console, file, Seq, Application Insights o altro.

# Logging strutturato

Il logging strutturato usa template con proprietà nominate.

Meglio:

```csharp
logger.LogInformation(
    "Ordine {OrdineId} confermato per cliente {ClienteId}",
    ordineId,
    clienteId);
```

Peggio:

```csharp
logger.LogInformation(
    $"Ordine {ordineId} confermato per cliente {clienteId}");
```

La prima forma conserva proprietà strutturate:

```text
OrdineId = 123
ClienteId = 456
```

La seconda produce solo una stringa già composta.

---

## Perché evitare interpolazione nei log

Sconsigliato:

```csharp
logger.LogInformation($"Cliente {clienteId} caricato");
```

Meglio:

```csharp
logger.LogInformation("Cliente {ClienteId} caricato", clienteId);
```

Vantaggi:

```text
migliore interrogazione nei sistemi di log
template stabile
proprietà indicizzabili
possibile minore costo se il livello è disabilitato
```

---

## Eccezioni nei log

Quando logghi un’eccezione, passa l’eccezione come primo argomento dopo il livello.

Corretto:

```csharp
logger.LogError(ex, "Errore durante l'import del file {FileName}", fileName);
```

Sconsigliato:

```csharp
logger.LogError("Errore: " + ex.Message);
```

Nel primo caso il provider può conservare stack trace e dettagli strutturati.

# Correlation ID

Un **Correlation ID** serve a collegare log appartenenti allo stesso flusso.

Esempio:

```text
Richiesta HTTP arriva
  CorrelationId = abc123

Validazione input
  CorrelationId = abc123

Query database
  CorrelationId = abc123

Errore durante salvataggio
  CorrelationId = abc123
```

Quando analizziamo i log, possiamo filtrare per `abc123` e vedere tutto il percorso.

---

## Correlation ID in ASP.NET Core

In un’app web, spesso il correlation ID arriva da un header:

```text
X-Correlation-ID
```

Oppure viene generato dal middleware se manca.

Esempio di middleware didattico:

```csharp
public class CorrelationIdMiddleware
{
    private const string HeaderName = "X-Correlation-ID";

    private readonly RequestDelegate next;
    private readonly ILogger<CorrelationIdMiddleware> logger;

    public CorrelationIdMiddleware(
        RequestDelegate next,
        ILogger<CorrelationIdMiddleware> logger)
    {
        this.next = next;
        this.logger = logger;
    }

    public async Task InvokeAsync(HttpContext context)
    {
        string correlationId;

        if (context.Request.Headers.TryGetValue(HeaderName, out var headerValue)
            && !string.IsNullOrWhiteSpace(headerValue))
        {
            correlationId = headerValue.ToString();
        }
        else
        {
            correlationId = Guid.NewGuid().ToString("N");
        }

        context.Response.Headers[HeaderName] = correlationId;

        using (logger.BeginScope(new Dictionary<string, object>
        {
            ["CorrelationId"] = correlationId
        }))
        {
            await next(context);
        }
    }
}
```

Registrazione:

```csharp
app.UseMiddleware<CorrelationIdMiddleware>();
```

---

## Logging scope

`BeginScope` permette di aggiungere proprietà comuni a tutti i log dentro un blocco.

Esempio:

```csharp
using (logger.BeginScope(new Dictionary<string, object>
{
    ["ClienteId"] = clienteId,
    ["OperazioneId"] = operazioneId
}))
{
    logger.LogInformation("Inizio operazione");
    logger.LogInformation("Fine operazione");
}
```

Entrambi i log possono includere `ClienteId` e `OperazioneId`.

# Best practice di configurazione

## 1. Non mettere segreti nel codice

Sbagliato:

```csharp
string password = "SuperPassword123";
```

Meglio:

```csharp
string? password = configuration["Email:SmtpPassword"];
```

Ancora meglio: in produzione il valore arriva da un secret store.

---

## 2. Non committare segreti in `appsettings.json`

`appsettings.json` può contenere valori non sensibili:

```json
{
  "Application": {
    "MaxItemsPerPage": 50
  }
}
```

Ma non dovrebbe contenere password, token o chiavi private se il file è versionato.

---

## 3. Usa Options pattern per configurazioni applicative

Sconsigliato:

```csharp
int maxItems = int.Parse(configuration["Application:MaxItemsPerPage"]);
```

Meglio:

```csharp
public class ApplicationOptions
{
    public int MaxItemsPerPage { get; set; }
}
```

e poi:

```csharp
IOptions<ApplicationOptions>
```

---

## 4. Valida la configurazione all’avvio

Meglio fallire subito che scoprire in produzione, al primo invio email, che manca `SmtpServer`.

```csharp
builder.Services
    .AddOptions<EmailOptions>()
    .Bind(builder.Configuration.GetSection(EmailOptions.SectionName))
    .ValidateDataAnnotations()
    .ValidateOnStart();
```

---

## 5. Usa override per ambiente

Esempio:

```text
appsettings.json
  valori comuni

appsettings.Development.json
  valori locali

variabili d’ambiente
  valori specifici della macchina/container

secret store
  segreti
```

---

## 6. Preferisci variabili d’ambiente nei container

In Docker, Kubernetes e cloud, le variabili d’ambiente sono spesso più pratiche dei file modificati a mano.

Esempio:

```text
Application__ExportFolder=/app/export
Email__SmtpServer=smtp.example.com
```

Nelle variabili d’ambiente si usa spesso `__` al posto di `:` perché `:` non è supportato o comodo in tutti gli ambienti.

La documentazione .NET segnala che le chiavi gerarchiche usano `:` nella configurazione, ma in alcuni scenari si usano delimitatori alternativi come il doppio underscore `__`. ([Microsoft Learn][5])

---

# Best practice di logging

## 1. Usa il livello corretto

Sbagliato:

```csharp
logger.LogError("Cliente non trovato");
```

Se il cliente non trovato è un caso normale, meglio:

```csharp
logger.LogInformation("Cliente {ClienteId} non trovato", clienteId);
```

Oppure `Warning`, se è anomalo.

---

## 2. Non loggare dati sensibili

Da evitare:

```text
password
token
cookie
codici fiscali completi, se non necessario
carte di credito
documenti
dati sanitari
segreti applicativi
```

Esempio sbagliato:

```csharp
logger.LogInformation("Login con password {Password}", password);
```

Meglio:

```csharp
logger.LogInformation("Tentativo di login per utente {UserName}", userName);
```

---

## 3. Usa logging strutturato

Meglio:

```csharp
logger.LogInformation(
    "Import file {FileName} completato: {RecordCount} record",
    fileName,
    recordCount);
```

Peggio:

```csharp
logger.LogInformation(
    $"Import file {fileName} completato: {recordCount} record");
```

---

## 4. Logga le eccezioni con stack trace

Corretto:

```csharp
catch (Exception ex)
{
    logger.LogError(ex, "Errore durante l'elaborazione del file {FileName}", fileName);
    throw;
}
```

---

## 5. Non ingolfare i log

Troppi log rendono difficile trovare ciò che serve.

Evita di loggare ogni singola riga elaborata se importi 1.000.000 di righe.

Meglio:

```csharp
logger.LogInformation(
    "Import completato. File {FileName}, record {RecordCount}, errori {ErrorCount}",
    fileName,
    recordCount,
    errorCount);
```

---

## 6. Aggiungi contesto

Un log utile contiene contesto.

Poco utile:

```csharp
logger.LogError("Errore durante il salvataggio");
```

Più utile:

```csharp
logger.LogError(
    ex,
    "Errore durante il salvataggio del cliente {ClienteId} nel documento {DocumentoId}",
    clienteId,
    documentoId);
```

---

## `ExportService.cs`

```csharp
using Microsoft.Extensions.Logging;
using Microsoft.Extensions.Options;

public class ExportService
{
    private readonly ApplicationOptions options;
    private readonly ILogger<ExportService> logger;

    public ExportService(
        IOptions<ApplicationOptions> options,
        ILogger<ExportService> logger)
    {
        this.options = options.Value;
        this.logger = logger;
    }

    public void EsportaClienti(int numeroClienti)
    {
        logger.LogInformation(
            "Avvio export clienti. Applicazione {ApplicationName}, cartella {ExportFolder}, clienti {ClientCount}",
            options.Name,
            options.ExportFolder,
            numeroClienti);

        if (numeroClienti == 0)
        {
            logger.LogWarning("Nessun cliente da esportare");
            return;
        }

        try
        {
            Directory.CreateDirectory(options.ExportFolder);

            string fileName = $"clienti_{DateTime.Now:yyyyMMdd_HHmmss}.txt";
            string path = Path.Combine(options.ExportFolder, fileName);

            File.WriteAllText(path, $"Clienti esportati: {numeroClienti}");

            logger.LogInformation(
                "Export completato. File {FilePath}, clienti {ClientCount}",
                path,
                numeroClienti);
        }
        catch (Exception ex)
        {
            logger.LogError(
                ex,
                "Errore durante export clienti nella cartella {ExportFolder}",
                options.ExportFolder);

            throw;
        }
    }
}
```

# Esempio: configurazione con user secrets

Supponiamo di avere:

```json
{
  "Email": {
    "SmtpServer": "smtp.example.com",
    "Port": 587,
    "Username": "dev@example.com"
  }
}
```

La password non va nel file.

La salviamo con:

```bash
dotnet user-secrets set "Email:SmtpPassword" "password-sviluppo"
```

Classe:

```csharp
public class EmailOptions
{
    public const string SectionName = "Email";

    public string SmtpServer { get; set; } = "";
    public int Port { get; set; }
    public string Username { get; set; } = "";
    public string SmtpPassword { get; set; } = "";
}
```

Uso:

```csharp
builder.Services.Configure<EmailOptions>(
    builder.Configuration.GetSection(EmailOptions.SectionName));
```

Il codice non cambia se domani la password arriva da una variabile d’ambiente o da Key Vault.

# Esempio: variabili d’ambiente

Configurazione JSON:

```json

    "ExportFolder": "export"
```

Override da variabile d’ambiente:

```powershell
$env:Application__ExportFolder = "C:\Temp\Export"
```

```csharp
configuration["Application:ExportFolder"]
```

# Errori comuni

## Mettere configurazione nel codice

Sbagliato:

```csharp
public const string ExportFolder = @"C:\Export";
```

Meglio:

```json
{
  "Application": {
    "ExportFolder": "export"
  }
}
```

---

## Mettere segreti in Git

Sbagliato:

```json
{
  "ConnectionStrings": {
    "DefaultConnection": "Server=.;User Id=sa;Password=Segreta123"
  }
}
```

Meglio usare user secrets in sviluppo e secret store in produzione.

## Usare `IConfiguration` ovunque

Possibile, ma debole:

```csharp
public class ExportService
{
    public ExportService(IConfiguration configuration)
    {
        string? folder = configuration["Application:ExportFolder"];
    }
}
```

Meglio:

```csharp
public class ExportService
{
    public ExportService(IOptions<ApplicationOptions> options)
    {
    }
}
```

`IConfiguration` va bene nei punti di composizione. Nei servizi applicativi è spesso meglio usare options tipizzate.

---

## Non validare options

Sbagliato:

```csharp
builder.Services.Configure<EmailOptions>(
    builder.Configuration.GetSection("Email"));
```

Meglio:

```csharp
builder.Services
    .AddOptions<EmailOptions>()
    .Bind(builder.Configuration.GetSection("Email"))
    .ValidateDataAnnotations()
    .ValidateOnStart();
```

---

## Loggare con interpolazione

Sconsigliato:

```csharp
logger.LogInformation($"Cliente {clienteId} salvato");
```

Meglio:

```csharp
logger.LogInformation("Cliente {ClienteId} salvato", clienteId);
```

---

## Loggare troppo o troppo poco

Troppo poco:

```text
Errore.
```

Troppo:

```text
Log per ogni variabile a ogni riga di codice.
```

Meglio log mirati, con contesto.

---

## Loggare dati sensibili

Sbagliato:

```csharp
logger.LogInformation("Token ricevuto {Token}", token);
```

Meglio evitare completamente o mascherare:

```csharp
logger.LogInformation("Token ricevuto per utente {UserId}", userId);
```

## Ignorare correlation ID

Senza correlation ID, in sistemi web o distribuiti diventa difficile seguire un flusso tra log diversi.

Meglio introdurlo presto, soprattutto per:

```text
API
worker
microservizi
integrazioni
messaggistica
operazioni batch complesse
```

# Linee guida pratiche

```text
Usa appsettings.json
  per configurazioni non sensibili e comuni.

Usa appsettings.{Environment}.json
  per differenze tra ambienti.

Usa user secrets
  per segreti locali di sviluppo.

Usa variabili d’ambiente o secret store
  per produzione e container.

Usa Options pattern
  per configurazioni fortemente tipizzate.

Valida le options all’avvio
  per fallire subito se manca qualcosa.

Usa ILogger<T>
  nei servizi applicativi.

Usa logging strutturato
  con template e proprietà nominate.

Passa le eccezioni a LogError
  per conservare stack trace.

Non loggare segreti
  mai.

Usa correlation ID
  per collegare log dello stesso flusso.
```

# Riepilogo

In questo capitolo abbiamo introdotto configurazione, options e logging.

Abbiamo visto:

```text
appsettings.json
  file base di configurazione.

appsettings.{Environment}.json
  configurazione specifica per ambiente.

Environment
  Development, Staging, Production.

User secrets
  segreti locali per sviluppo.

IConfiguration
  astrazione unificata per leggere configurazione.

Options pattern
  configurazione tipizzata tramite classi.
```

Abbiamo studiato il logging:

```text
ILogger<T>
  interfaccia principale per scrivere log.

Livelli di log
  Trace, Debug, Information, Warning, Error, Critical.

Provider
  destinazioni dei log.

Logging strutturato
  template con proprietà nominate.

Correlation ID
  identificativo per collegare log dello stesso flusso.
```

La regola principale è questa:

```text
La configurazione deve cambiare senza ricompilare il codice.
Il logging deve spiegare cosa è successo senza dover attaccare il debugger.
```

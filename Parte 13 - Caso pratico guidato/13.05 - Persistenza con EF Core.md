# 13.05 — Persistenza con EF Core

Nel capitolo precedente abbiamo implementato dominio e use case: entity, value object, command, query, handler, validazione, `Result` pattern e test unitari.

Ora dobbiamo collegare il cuore dell’applicazione a un database reale.

Useremo:

```text id="iyxzy4"
Entity Framework Core
SQLite
DbContext
Fluent API
Migrations
Repository concreto
Transazioni
Seed dati
Test di integrazione
```

Il punto centrale è questo:

```text id="mbin9v"
EF Core deve stare nello strato Infrastructure.
Il dominio non deve sapere nulla del database.
L’application layer deve usare astrazioni.
```

EF Core è l’ORM moderno di .NET per lavorare con database usando oggetti .NET e LINQ; Microsoft raccomanda anche di esaminare e testare attentamente le migration generate, perché lo schema diventa difficile da cambiare quando contiene dati di produzione. ([Microsoft Learn][1])

# AppDbContext

```csharp id="l6ju3w"
using App.Application.Abstractions;
using App.Domain.Catalogo;
using App.Domain.Clienti;
using App.Domain.Documenti;
using Microsoft.EntityFrameworkCore;

namespace App.Infrastructure.Data;

public class AppDbContext : DbContext, IAppDbContext
{
    public AppDbContext(DbContextOptions<AppDbContext> options)
        : base(options)
    {
    }

    public DbSet<Libro> Libri => Set<Libro>();

    public DbSet<Autore> Autori => Set<Autore>();

    public DbSet<Collana> Collane => Set<Collana>();

    public DbSet<LibroAutore> LibriAutori => Set<LibroAutore>();

    public DbSet<Cliente> Clienti => Set<Cliente>();

    public DbSet<Documento> Documenti => Set<Documento>();

    public DbSet<DocumentoRiga> DocumentoRighe => Set<DocumentoRiga>();

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        modelBuilder.ApplyConfigurationsFromAssembly(
            typeof(AppDbContext).Assembly);
    }
}
```

Punto importante:

```text id="b4y6hm"
AppDbContext sta in Infrastructure.
Le entity stanno in Domain.
Le configurazioni EF Core stanno in Infrastructure.
```

# Configurazione EF Core in DI

In `App.Infrastructure/DependencyInjection.cs`:

```csharp id="xupw75"
using App.Application.Abstractions;
using App.Application.Documenti;
using App.Infrastructure.Data;
using App.Infrastructure.Documenti;
using App.Infrastructure.Services;
using Microsoft.EntityFrameworkCore;
using Microsoft.Extensions.Configuration;
using Microsoft.Extensions.DependencyInjection;

namespace App.Infrastructure;

public static class DependencyInjection
{
    public static IServiceCollection AddInfrastructure(
        this IServiceCollection services,
        IConfiguration configuration)
    {
        string connectionString =
            configuration.GetConnectionString("Default")
            ?? throw new InvalidOperationException(
                "Connection string 'Default' mancante.");

        services.AddDbContext<AppDbContext>(options =>
        {
            options.UseSqlite(connectionString);
        });

        services.AddScoped<IAppDbContext>(provider =>
            provider.GetRequiredService<AppDbContext>());

        services.AddScoped<IDocumentoRepository, DocumentoRepository>();
        services.AddScoped<IDocumentoQueries, DocumentoQueries>();

        services.AddSingleton<IClock, SystemClock>();

        return services;
    }
}
```

# Mapping Libro

`LibroConfiguration.cs`:

```csharp id="yr7f4e"
using App.Domain.Catalogo;
using Microsoft.EntityFrameworkCore;
using Microsoft.EntityFrameworkCore.Metadata.Builders;

namespace App.Infrastructure.Data.Configurations;

public class LibroConfiguration : IEntityTypeConfiguration<Libro>
{
    public void Configure(EntityTypeBuilder<Libro> builder)
    {
        builder.ToTable("Libri");

        builder.HasKey(x => x.Id);

        builder.Property(x => x.Titolo)
            .HasMaxLength(200)
            .IsRequired();

        builder.Property(x => x.Isbn)
            .HasMaxLength(20);

        builder.HasIndex(x => x.Isbn)
            .IsUnique();

        builder.Property(x => x.Prezzo)
            .HasPrecision(10, 2);

        builder.Property(x => x.Attivo)
            .IsRequired();

        builder.HasOne(x => x.Collana)
            .WithMany()
            .HasForeignKey(x => x.CollanaId)
            .OnDelete(DeleteBehavior.SetNull);
    }
}
```

```text id="f5tjck"

ma il comportamento fisico dipende dal provider.
```

# Mapping Documento

`Documento` ha una lista privata `righe`.

La entity espone:

```csharp id="dtlbh4"
private readonly List<DocumentoRiga> righe = [];

public IReadOnlyCollection<DocumentoRiga> Righe =>
    righe.AsReadOnly();
```

EF Core può mappare anche campi privati, ma bisogna configurarlo bene.

```csharp id="tbcai8"
using App.Domain.Documenti;
using Microsoft.EntityFrameworkCore;
using Microsoft.EntityFrameworkCore.Metadata.Builders;

namespace App.Infrastructure.Data.Configurations;

public class DocumentoConfiguration : IEntityTypeConfiguration<Documento>
{
    public void Configure(EntityTypeBuilder<Documento> builder)
    {
        builder.ToTable("Documenti");

        builder.HasKey(x => x.Id);

        builder.Property(x => x.Numero)
            .HasMaxLength(50)
            .IsRequired();

        builder.HasIndex(x => x.Numero)
            .IsUnique();

        builder.Property(x => x.Data)
            .IsRequired();

        builder.Property(x => x.ClienteId)
            .IsRequired();

        builder.Property(x => x.TipoDocumento)
            .HasConversion<int>()
            .IsRequired();

        builder.Property(x => x.Stato)
            .HasConversion<int>()
            .IsRequired();

        builder.Property(x => x.TotaleImponibile)
            .HasPrecision(10, 2);

        builder.Property(x => x.TotaleSconto)
            .HasPrecision(10, 2);

        builder.Property(x => x.TotaleDocumento)
            .HasPrecision(10, 2);

        builder.HasMany(x => x.Righe)
            .WithOne(x => x.Documento)
            .HasForeignKey(x => x.DocumentoId)
            .OnDelete(DeleteBehavior.Cascade);

        builder.Navigation(x => x.Righe)
            .UsePropertyAccessMode(PropertyAccessMode.Field);
    }
}
```

Se EF Core non trova correttamente il campo privato, possiamo essere più espliciti:

```csharp id="vl7rft"
builder.Metadata
    .FindNavigation(nameof(Documento.Righe))!
    .SetPropertyAccessMode(PropertyAccessMode.Field);
```

# DocumentoRepository

```csharp id="zz4edb"
using App.Application.Documenti;
using App.Domain.Documenti;
using App.Infrastructure.Data;
using Microsoft.EntityFrameworkCore;

namespace App.Infrastructure.Documenti;

public class DocumentoRepository : IDocumentoRepository
{
    private readonly AppDbContext dbContext;

    public DocumentoRepository(AppDbContext dbContext)
    {
        this.dbContext = dbContext;
    }

    public Task<Documento?> GetByIdAsync(
        int id,
        CancellationToken cancellationToken)
    {
        return dbContext.Documenti
            .Include(x => x.Righe)
            .FirstOrDefaultAsync(x => x.Id == id, cancellationToken);
    }

    public async Task AddAsync(
        Documento documento,
        CancellationToken cancellationToken)
    {
        await dbContext.Documenti.AddAsync(
            documento,
            cancellationToken);
    }

    public Task<bool> ClienteExistsAsync(
        int clienteId,
        CancellationToken cancellationToken)
    {
        return dbContext.Clienti
            .AnyAsync(x => x.Id == clienteId, cancellationToken);
    }

    public Task<bool> LibroExistsAsync(
        int libroId,
        CancellationToken cancellationToken)
    {
        return dbContext.Libri
            .AnyAsync(
                x => x.Id == libroId && x.Attivo,
                cancellationToken);
    }

    public async Task<string> GetDescrizioneLibroAsync(
        int libroId,
        CancellationToken cancellationToken)
    {
        string? titolo = await dbContext.Libri
            .Where(x => x.Id == libroId)
            .Select(x => x.Titolo)
            .FirstOrDefaultAsync(cancellationToken);

        return titolo ?? "";
    }

    public async Task<decimal> GetPrezzoLibroAsync(
        int libroId,
        CancellationToken cancellationToken)
    {
        decimal? prezzo = await dbContext.Libri
            .Where(x => x.Id == libroId)
            .Select(x => (decimal?)x.Prezzo)
            .FirstOrDefaultAsync(cancellationToken);

        return prezzo ?? 0m;
    }

    public async Task SaveChangesAsync(
        CancellationToken cancellationToken)
    {
        await dbContext.SaveChangesAsync(cancellationToken);
    }
}
```

# DocumentoQueries

Le query di lettura producono DTO.

```csharp id="zdy7h4"
using App.Application.Documenti;
using App.Infrastructure.Data;
using Microsoft.EntityFrameworkCore;

namespace App.Infrastructure.Documenti;

public class DocumentoQueries : IDocumentoQueries
{
    private readonly AppDbContext dbContext;

    public DocumentoQueries(AppDbContext dbContext)
    {
        this.dbContext = dbContext;
    }

    public Task<DocumentoDetailDto?> GetDocumentoByIdAsync(
        int id,
        CancellationToken cancellationToken)
    {
        return dbContext.Documenti
            .AsNoTracking()
            .Where(x => x.Id == id)
            .Select(x => new DocumentoDetailDto(
                x.Id,
                x.Numero,
                x.Data,
                x.ClienteId,
                x.TipoDocumento,
                x.Stato,
                x.TotaleImponibile,
                x.TotaleSconto,
                x.TotaleDocumento,
                x.Righe
                    .OrderBy(r => r.Id)
                    .Select(r => new DocumentoRigaDto(
                        r.Id,
                        r.LibroId,
                        r.Descrizione,
                        r.Quantita,
                        r.PrezzoUnitario,
                        r.ScontoPercentuale,
                        r.TotaleRiga))
                    .ToList()))
            .FirstOrDefaultAsync(cancellationToken);
    }
}
```

Per le letture usiamo `AsNoTracking()` perché non dobbiamo modificare le entity.

Regola pratica:

```text id="syybva"
Query di lettura:
  AsNoTracking + proiezione DTO.

Comandi di modifica:
  carica aggregate tracciate + modifica + SaveChanges.
```

# Transazioni

EF Core usa già una transazione implicita quando chiami `SaveChanges` per più modifiche nello stesso contesto, se il provider lo supporta.

Per esempio:

```text id="avkmsg"
aggiungo Documento
aggiungo DocumentoRiga
SaveChanges
```

viene salvato come unità logica.

Ma a volte serve una transazione esplicita.

Esempi:

```text id="ocajix"
più SaveChanges nello stesso caso d’uso
operazioni su più aggregate
aggiornamenti che devono essere atomici
creazione documento + numeratore
logica complessa con più step
```

---

# Transazione esplicita

Esempio in un servizio applicativo o repository:

```csharp id="f9kk78"
await using IDbContextTransaction transaction =
    await dbContext.Database.BeginTransactionAsync(cancellationToken);

try
{
    dbContext.Documenti.Add(documento);

    // eventuali altre operazioni

    await dbContext.SaveChangesAsync(cancellationToken);

    await transaction.CommitAsync(cancellationToken);
}
catch
{
    await transaction.RollbackAsync(cancellationToken);
    throw;
}
```

Namespace:

```csharp id="z5siww"
using Microsoft.EntityFrameworkCore.Storage;
```

Nel nostro caso base, per creare un documento con righe può bastare una sola `SaveChanges`.

Però è utile introdurre il concetto perché la numerazione documento reale richiederebbe una gestione transazionale.

# Numerazione documento e transazione

Nel capitolo precedente avevamo una generazione semplificata:

```text id="g0iry4"
DOC-20260101-guid
```

In un gestionale vero, potremmo avere:

```text id="z6eiw9"
DOC-2026-000001
DOC-2026-000002
DOC-2026-000003
```

Questa numerazione deve essere:

```text id="zaag52"
univoca
progressiva
sicura in concorrenza
transazionale
```

Quindi la creazione del numero e il salvataggio del documento dovrebbero essere nello stesso confine transazionale.

Per il progetto didattico possiamo lasciare la versione semplice, ma spiegare il problema.

# Seed dati

Il seed serve a creare dati iniziali.

Esempi:

```text id="wisr6s"
collane di esempio
autori di esempio
libri di esempio
clienti di esempio
documenti di esempio
```

EF Core supporta più modalità di seeding. Nelle versioni recenti, Microsoft raccomanda `UseSeeding` e `UseAsyncSeeding`: questi metodi vengono eseguiti come parte di `EnsureCreated`, `Migrate` e `dotnet ef database update`, anche se non ci sono nuove migration. ([Microsoft Learn][2])

Per un progetto didattico con separazione `Infrastructure`, possiamo usare un initializer esplicito:

```text id="fwnleb"

```

È più facile da capire.

# Test di integrazione

I test unitari verificano dominio e handler con fake.

I test di integrazione verificano:

```text id="dwladx"

```

Per testare EF Core, Microsoft presenta due test double comuni: SQLite in-memory e il provider InMemory; raccomanda attenzione perché SQLite, pur essendo relazionale e più vicino a un database reale del provider InMemory, presenta comunque differenze rispetto al database di produzione. ([Microsoft Learn][3])

Inoltre, il provider EF Core InMemory è scoraggiato per molti test, perché non è un database relazionale, supporta meno query, non supporta transazioni e può comportarsi in modo diverso rispetto a un database reale. ([Microsoft Learn][4])

Per questo progetto useremo:

```text id="hx74j4"
SQLite in-memory
```

# Perché SQLite in-memory

Vantaggi:

```text id="uuxw7w"
è veloce
non crea file
è relazionale
supporta query SQL
supporta vincoli più realisticamente di InMemory provider
è adatto al nostro progetto, che usa SQLite anche in sviluppo
```

Attenzione:

```text id="gcc8bh"
La connessione SQLite in-memory deve restare aperta
per tutta la durata del test.
```

Se la connessione viene chiusa, il database in memoria sparisce.

# Test di integrazione API

Il problema è sostituire il database reale con SQLite in-memory.

Creiamo una factory custom.

`App.Tests/Integration/CustomWebApplicationFactory.cs`:

```csharp id="nftksc"
using App.Infrastructure.Data;
using Microsoft.AspNetCore.Hosting;
using Microsoft.AspNetCore.Mvc.Testing;
using Microsoft.Data.Sqlite;
using Microsoft.EntityFrameworkCore;
using Microsoft.Extensions.DependencyInjection;

namespace App.Tests.Integration;

public class CustomWebApplicationFactory
    : WebApplicationFactory<Program>
{
    private SqliteConnection? connection;

    protected override void ConfigureWebHost(IWebHostBuilder builder)
    {
        builder.ConfigureServices(services =>
        {
            ServiceDescriptor? descriptor = services
                .SingleOrDefault(d =>
                    d.ServiceType == typeof(DbContextOptions<AppDbContext>));

            if (descriptor is not null)
            {
                services.Remove(descriptor);
            }

            connection = new SqliteConnection("DataSource=:memory:");
            connection.Open();

            services.AddDbContext<AppDbContext>(options =>
            {
                options.UseSqlite(connection);
            });

            ServiceProvider serviceProvider =
                services.BuildServiceProvider();

            using IServiceScope scope =
                serviceProvider.CreateScope();

            AppDbContext dbContext =
                scope.ServiceProvider.GetRequiredService<AppDbContext>();

            dbContext.Database.EnsureCreated();

            Seed(dbContext);
        });
    }

    private static void Seed(AppDbContext dbContext)
    {
        if (dbContext.Clienti.Any())
        {
            return;
        }

        Cliente cliente = new(
            "Cliente API Test",
            null,
            null,
            null,
            null);

        Libro libro = new(
            "Libro API Test",
            "API-ISBN",
            20m,
            null,
            null);

        dbContext.Clienti.Add(cliente);
        dbContext.Libri.Add(libro);

        dbContext.SaveChanges();
    }

    protected override void Dispose(bool disposing)
    {
        base.Dispose(disposing);

        if (disposing)
        {
            connection?.Dispose();
        }
    }
}
```

# Errori comuni con EF Core

## Esporre entity direttamente dalle API

Sbagliato:

```csharp id="vb1pz7"
return Results.Ok(await dbContext.Libri.ToListAsync());
```

Meglio DTO:

```csharp id="pw7ljr"
.Select(x => new LibroListItemDto(...))
```

---

## Dimenticare AsNoTracking nelle query di sola lettura

Se non devi modificare entity:

```csharp id="q4ubzf"
.AsNoTracking()
```

riduce overhead del change tracker.

---

## Fare Include inutili

Sbagliato:

```csharp id="bgi53x"
.Include(x => x.Righe)
.Include(x => x.Cliente)
.Include(x => x.Altro)
```

quando poi ti servono solo due campi.

Meglio proiezione:

```csharp id="mx879f"
.Select(x => new Dto(...))
```

---

## Mettere DbContext nel Domain

Il dominio non deve conoscere EF Core.

---

## Usare InMemory provider per test relazionali

Il provider InMemory non è relazionale e può dare risultati fuorvianti, soprattutto su vincoli, transazioni e query. Microsoft avvisa che l’uso dell’InMemory provider per test è scoraggiato e che nuove funzionalità non vengono aggiunte a quel provider. ([Microsoft Learn][4])

---

## Migration non controllate

Ogni migration va letta.

Controlla:

```text id="x2eqgx"
nomi tabelle
nomi colonne
indici
foreign key
delete behavior
tipi decimal
nullable/non nullable
valori default
```

---

## Cancellazioni a cascata non volute

Attenzione a:

```csharp id="ky93g6"
.OnDelete(DeleteBehavior.Cascade)
```

Non sempre è corretto.

Esempio:

```text id="b4h2cn"
cancellare un cliente non deve cancellare documenti storici
```

Nel dubbio, preferire restrizioni o disattivazione logica.

# Linee guida pratiche

```text id="upfmi4"
Tieni EF Core in Infrastructure
  non nel Domain.

Usa Fluent API
  per mapping, vincoli, indici e relazioni.

Usa migration
  per versionare lo schema.

Leggi le migration generate
  prima di applicarle.

Usa repository concreti
  per salvare aggregate e isolare persistenza.

Usa query service
  per letture ottimizzate in DTO.

Usa AsNoTracking
  nelle query di sola lettura.

Usa transazioni esplicite
  quando un caso d’uso richiede più step atomici.

Usa seed dati
  per facilitare sviluppo e demo.

Usa SQLite in-memory
  per test di integrazione veloci e relazionali.

Evita EF InMemory provider
  quando vuoi testare comportamento relazionale.

Non esporre entity EF Core
  tramite API pubbliche.
```

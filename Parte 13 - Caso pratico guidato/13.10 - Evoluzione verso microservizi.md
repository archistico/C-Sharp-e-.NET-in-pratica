# 13.10 — Evoluzione verso microservizi

Nei capitoli precedenti abbiamo costruito un **monolite modulare**: dominio separato, use case, EF Core, API REST, Blazor, background service, Docker, CI/CD e health checks.

Ora possiamo parlare di microservizi nel modo corretto: **non come punto di partenza**, ma come possibile evoluzione.

Il punto centrale è questo:

```text id="e6p8m3"
Un microservizio non nasce perché una classe è grande
o perché vogliamo usare RabbitMQ.

Un microservizio nasce quando un modulo ha bisogno
di autonomia reale: tecnica, organizzativa, di deploy,
di scalabilità o di modello dati.
```

Questo capitolo serve a rispondere a una domanda concreta:

```text id="e5or7p"
Se il mini gestionale editoriale cresce,
quali moduli potremmo separare?
Quando avrebbe senso farlo?
E con quali costi?
```

---

# Perché parlare di microservizi solo alla fine

Nel progetto finale siamo partiti con un monolite modulare.

È stata una scelta intenzionale.

```text id="fo4hqa"
Un monolite modulare è spesso la scelta migliore
per iniziare un sistema:
un solo deploy,
un solo database,
meno complessità operativa,
confini comunque chiari.
```

Microservizi troppo presto significano spesso:

```text id="irf8t1"
più deploy
più reti
più failure mode
più log distribuiti
più sicurezza distribuita
più monitoring
più coerenza eventuale
più test complessi
più DevOps
```

Il rischio è costruire non un sistema distribuito, ma un **monolite distribuito**:

```text id="fxqdpq"
tanti servizi piccoli
fortemente accoppiati
che devono essere deployati insieme
e condividono lo stesso database.
```

Quello non è un buon sistema a microservizi.

È un monolite reso più difficile.

# Quando separare un modulo

Separare un modulo ha senso quando esiste almeno una motivazione forte.

## 1. Scalabilità diversa

Esempio:

```text id="brx6e0"
Il modulo Report diventa molto pesante.
Genera PDF, esportazioni, aggregazioni grandi.
Rallenta l’API principale.
```

Allora potremmo separare:

```text id="t5t2cm"
Report Service
```

per scalarlo indipendentemente.

---

## 2. Deploy indipendente

Esempio:

```text id="eaz5m5"
Il modulo Catalogo cambia raramente.
Il modulo Documenti cambia spesso.
```

Se ogni modifica ai documenti richiede redeploy dell’intera app, può diventare costoso.

Separare `Documenti` potrebbe avere senso.

---

## 3. Team diversi

Esempio:

```text id="ceg7pd"
Team A lavora su Catalogo.
Team B lavora su Documenti e ordini.
Team C lavora su Report e BI.
```

Se i team si bloccano a vicenda, i confini tecnici possono seguire i confini organizzativi.

---

## 4. Modello dati autonomo

Un modulo è un buon candidato se può possedere i propri dati.

Esempio:

```text id="l294f6"
Catalogo Service
  possiede libri, autori, collane.

Documenti Service
  possiede documenti e righe.

Report Service
  possiede viste/materializzazioni/report generati.
```

Regola importante:

```text id="gp3i76"
Un microservizio serio dovrebbe possedere il proprio database
o almeno il proprio schema dati.
```

Se più servizi scrivono le stesse tabelle, il confine è debole.

---

## 5. Requisiti tecnici diversi

Esempio:

```text id="i4n89f"
API gestionale:
  CRUD, bassa latenza, pochi dati.

Report:
  query pesanti, export, calcoli, job lunghi.

Notifiche:
  invio email, retry, code, template.
```

Tecnologie o deployment diversi possono giustificare la separazione.

---

# Quando NON separare

Non separare un modulo solo perché:

```text id="d15xvr"
il progetto sembra più moderno
hai letto che i microservizi sono scalabili
vuoi usare RabbitMQ
vuoi usare Kubernetes
una classe è diventata lunga
hai molti controller
vuoi evitare refactoring interno
```

Prima di separare, prova a migliorare il monolite modulare:

```text id="sv6k9y"
moduli più chiari
cartelle migliori
interfacce più pulite
test
query ottimizzate
background jobs
cache
database indicizzato
```

Molti problemi si risolvono senza microservizi.

# Strangler Fig

Lo **Strangler Fig pattern** è un modo incrementale per modernizzare o separare parti di un sistema senza riscrivere tutto. Microsoft lo descrive come un approccio graduale in cui si introduce una facciata/proxy tra client, sistema esistente e nuovo sistema; la facciata instrada alcune richieste al sistema legacy e altre al nuovo sistema, permettendo una migrazione controllata. ([Microsoft Learn][1])

Applicato al nostro progetto:

```text id="nsay3h"
App.Web
  ↓
API Gateway / Facciata
  ├── vecchia App.Api monolitica
  └── nuovo Report Service
```

All’inizio:

```text id="es4uni"
tutte le richieste vanno ad App.Api
```

Poi:

```text id="xrlg4b"
solo /api/report passa al nuovo Report Service
```

Alla fine:

```text id="r869kz"
il vecchio modulo Report dentro App.Api viene rimosso
```

Questo evita una riscrittura totale.

# Problema centrale: i dati

Il punto difficile non è creare un nuovo progetto ASP.NET Core.


Opzioni:

## Opzione A: legge dal database del monolite




# Eventi di integrazione

Un **evento di integrazione** comunica qualcosa che è già successo e che può interessare altri servizi.

Esempi:

```text id="sn0r7m"
DocumentoConfermato
DocumentoAnnullato
LibroCreato
LibroAggiornato
ClienteCreato
```

Esempio:

```csharp id="tcqzp0"
public record DocumentoConfermatoIntegrationEvent(
    Guid EventId,
    DateTimeOffset OccurredAt,
    int DocumentoId,
    string Numero,
    DateOnly Data,
    int ClienteId,
    decimal TotaleDocumento);
```

Caratteristiche:

```text id="wgvpwt"
è già successo
non chiede a un altro servizio di fare qualcosa
contiene dati utili ma non necessariamente tutto il dominio
deve essere versionabile
deve avere un identificativo
```

Differenza:

```text id="tb77z1"
Domain event
  evento interno al dominio/applicazione.

Integration event
  evento pubblicato verso altri sistemi o servizi.
```

# Il problema della doppia scrittura

Quando confermi un documento devi fare due cose:

```text id="e30joz"
1. salvare lo stato nel database
2. pubblicare un evento su RabbitMQ
```

Problema:

```text id="xv0kxw"
Se salvo il database ma fallisce RabbitMQ,
l’evento viene perso.

Se pubblico su RabbitMQ ma fallisce il database,
gli altri servizi ricevono un evento su un dato non salvato.
```

Questo è il problema della **dual write**.

Non va risolto con:

```text id="s9kfsb"
try/catch ingenuo
pubblico prima e salvo dopo
salvo prima e spero che RabbitMQ funzioni
```

Serve un pattern.

# Outbox pattern

Il **Transactional Outbox pattern** risolve il problema registrando evento e modifica dati nella stessa transazione del database. L’evento viene salvato in una tabella outbox e poi un processo separato lo pubblica sul broker; questo evita la perdita di messaggi quando bisogna aggiornare dati e inviare eventi in modo affidabile. ([Microsoft Learn][2])

Schema:

```text id="ob409w"
Transazione database:
  aggiorna Documento
  inserisce record in OutboxMessages

Worker Outbox:
  legge OutboxMessages non pubblicati
  pubblica su RabbitMQ
  marca come pubblicato
```

Così:

```text id="j80tvq"
database e evento sono salvati atomicamente
la pubblicazione può essere ritentata
gli eventi non si perdono facilmente
```

---

# Tabella OutboxMessages

Esempio entity:

```csharp id="d6xn25"
public class OutboxMessage
{
    public Guid Id { get; private set; }

    public DateTimeOffset OccurredAt { get; private set; }

    public string Type { get; private set; } = "";

    public string Payload { get; private set; } = "";

    public DateTimeOffset? ProcessedAt { get; private set; }

    public string? Error { get; private set; }

    private OutboxMessage()
    {
    }

    public OutboxMessage(
        Guid id,
        DateTimeOffset occurredAt,
        string type,
        string payload)
    {
        Id = id;
        OccurredAt = occurredAt;
        Type = type;
        Payload = payload;
    }

    public void MarkProcessed(DateTimeOffset processedAt)
    {
        ProcessedAt = processedAt;
        Error = null;
    }

    public void MarkFailed(string error)
    {
        Error = error;
    }
}
```

Mapping EF Core:

```csharp id="3cpv7k"
public class OutboxMessageConfiguration
    : IEntityTypeConfiguration<OutboxMessage>
{
    public void Configure(EntityTypeBuilder<OutboxMessage> builder)
    {
        builder.ToTable("OutboxMessages");

        builder.HasKey(x => x.Id);

        builder.Property(x => x.Type)
            .HasMaxLength(300)
            .IsRequired();

        builder.Property(x => x.Payload)
            .IsRequired();

        builder.HasIndex(x => x.ProcessedAt);
    }
}
```

# Outbox worker

Un background service pubblica gli eventi.

Schema:

```text id="k9gkf3"
ogni N secondi:
  legge messaggi non processati
  pubblica su RabbitMQ
  marca ProcessedAt
```

Esempio concettuale:

```csharp id="el60a1"
public class OutboxPublisherWorker : BackgroundService
{
    private readonly IServiceScopeFactory scopeFactory;
    private readonly ILogger<OutboxPublisherWorker> logger;

    protected override async Task ExecuteAsync(
        CancellationToken stoppingToken)
    {
        using PeriodicTimer timer =
            new(TimeSpan.FromSeconds(5));

        while (await timer.WaitForNextTickAsync(stoppingToken))
        {
            await PublishPendingMessagesAsync(stoppingToken);
        }
    }

    private async Task PublishPendingMessagesAsync(
        CancellationToken cancellationToken)
    {
        using IServiceScope scope =
            scopeFactory.CreateScope();

        AppDbContext dbContext =
            scope.ServiceProvider.GetRequiredService<AppDbContext>();

        IEventBus eventBus =
            scope.ServiceProvider.GetRequiredService<IEventBus>();

        List<OutboxMessage> messages =
            await dbContext.OutboxMessages
                .Where(x => x.ProcessedAt == null)
                .OrderBy(x => x.OccurredAt)
                .Take(20)
                .ToListAsync(cancellationToken);

        foreach (OutboxMessage message in messages)
        {
            try
            {
                await eventBus.PublishAsync(
                    message.Type,
                    message.Payload,
                    cancellationToken);

                message.MarkProcessed(DateTimeOffset.UtcNow);

                await dbContext.SaveChangesAsync(cancellationToken);
            }
            catch (Exception ex)
            {
                logger.LogError(
                    ex,
                    "Errore pubblicazione outbox message {MessageId}",
                    message.Id);

                message.MarkFailed(ex.Message);

                await dbContext.SaveChangesAsync(cancellationToken);
            }
        }
    }
}
```

# Idempotenza

Con outbox e broker, un evento può essere pubblicato più di una volta.

Il consumer deve essere idempotente.

Significa:

```text id="dctb23"
se ricevo due volte lo stesso evento,
il risultato finale deve essere corretto.
```

Esempio:

```text id="ic5ar2"
DocumentoConfermato EventId = 123

Report Service lo riceve una volta:
  aggiorna vendite

Lo riceve una seconda volta:
  non deve raddoppiare le vendite
```

Soluzione:

```text id="lhy34u"
tabella ProcessedMessages
con EventId già elaborati
```

Esempio:

```text id="4y7m7c"
se EventId esiste:
  ignora

altrimenti:
  elabora
  salva EventId come processato
```

# RabbitMQ

RabbitMQ è un message broker. Nel modello base, i produttori pubblicano messaggi e i consumatori li ricevono tramite code. La documentazione RabbitMQ descrive una queue come una collezione ordinata di messaggi, dove i messaggi vengono accodati e consegnati ai consumer; i tutorial ufficiali coprono scenari come work queue e publish/subscribe. ([rabbitmq.com][3])

Concetti base:

```text id="h2u2nd"
Producer
  pubblica messaggi.

Exchange
  riceve messaggi dal producer.

Queue
  conserva messaggi.

Consumer
  legge messaggi dalla queue.

Binding
  collega exchange e queue.
```

Schema publish/subscribe:

```text id="k9rbg0"
Documenti Service
  ↓ publish DocumentoConfermato
Exchange
  ├── Queue Report
  ├── Queue Notification
  └── Queue Dashboard
```

Ogni servizio ha la propria coda.

## Costi dei microservizi

```text id="m0yli6"
rete invece di chiamate in memoria
latenza
timeout
retry
circuit breaker
coerenza eventuale
debug distribuito
logging centralizzato
tracing
security service-to-service
deploy multipli
versionamento contratti
messaging
outbox
idempotenza
monitoring
DevOps più complesso
```

Esempio:

```text id="hvaayt"
Nel monolite:
  documento.Conferma()
  report aggiornato nella stessa transazione, se vuoi.

Nei microservizi:
  documento confermato
  evento pubblicato
  report aggiornato dopo
  possibile ritardo
  possibile retry
  possibile duplicazione evento
```

Questo è un cambiamento concettuale importante.

---

# Coerenza eventuale

Nei microservizi, non tutto è aggiornato immediatamente.

Esempio:

```text id="kk7xly"
L’utente conferma un documento alle 10:00:00.

Documenti Service:
  stato aggiornato subito.

Report Service:
  riceve evento alle 10:00:02.
  aggiorna report alle 10:00:03.
```

Per qualche secondo il report può essere vecchio.

Questo è normale in architetture event-driven.

Regola:

```text id="nt5lcy"
Se il business richiede consistenza immediata forte,
separare quel flusso può essere sbagliato.
```

# Contratti tra servizi

Quando separi un servizio, devi versionare i contratti.

Contratti HTTP:

```text id="5sn8zg"
/api/v1/catalogo/libri/{id}
```

Contratti evento:

```text id="eua0t6"
DocumentoConfermatoIntegrationEventV1
DocumentoConfermatoIntegrationEventV2
```

Regole:

```text id="xlul7q"
non rimuovere campi usati dai consumer senza coordinamento
aggiungi campi in modo compatibile
versiona eventi quando cambi significato
documenta payload e semantica
```

Aggiungiamo broker.



```text id="h6cxed"

asincrono

```

```text id="2z2419"
Report.Service
```


Aggiungiamo:

# Errori comuni

## Separare per tabella

Errore:

```text id="cyi9wz"
Libri Service
Autori Service
Collane Service
DocumentiRighe Service
```

Questo crea servizi troppo piccoli e accoppiati.

Meglio separare per capacità di business:

```text id="dc957a"
Catalogo
Documenti
Report
Notifiche
```

---

## Condividere lo stesso database

Se tutti i servizi leggono e scrivono lo stesso database:

```text id="6j4n4l"
non hai veri microservizi
hai processi distribuiti accoppiati al DB
```

Transitoriamente può servire, ma non deve essere l’obiettivo.

---

## Usare eventi come chiamate remote mascherate

Errore:

```text id="2l2enj"
pubblico evento e mi aspetto risposta immediata
```

Un evento dice:

```text id="h64035"
è successo qualcosa
```

Non dice:

```text id="j1m1aj"
fai questa cosa e rispondimi subito
```

---

## Dimenticare idempotenza

Con messaging, i duplicati sono possibili.

I consumer devono reggerli.

---

## Non avere osservabilità

Senza log centralizzati e trace, diagnosticare problemi distribuiti diventa molto difficile.

---

## Sottovalutare la sicurezza service-to-service

Tra servizi servono:

```text id="zqr319"
autenticazione
autorizzazione
segreti
TLS/mTLS se necessario
policy di rete
```

---

## Trasformare tutto in microservizi

Non tutto merita un servizio separato.

Alcune parti stanno benissimo nel monolite modulare.

# Checklist prima di separare un modulo

Prima di estrarre un microservizio, chiediti:

```text id="biji38"
Il modulo ha confini chiari?
Ha dati propri?
Ha casi d’uso autonomi?
Ha bisogno di scalare separatamente?
Ha bisogno di deploy indipendente?
Un team separato lo gestisce?
La comunicazione può essere asincrona?
Il business accetta coerenza eventuale?
Abbiamo osservabilità sufficiente?
Abbiamo CI/CD e deploy maturi?
Abbiamo gestione segreti e sicurezza?
Sappiamo gestire retry, timeout e idempotenza?
Il beneficio supera il costo?
```

Se molte risposte sono “no”, resta nel monolite modulare.

# Linee guida pratiche

```text id="mukfp3"
Parti da un monolite modulare
  non da microservizi.

Separa solo moduli con autonomia reale
  business, dati, team, deploy o scalabilità.

Usa Strangler Fig
  per migrare in modo incrementale.

Non condividere il database come destinazione finale
  ogni servizio dovrebbe possedere i propri dati.

Usa integration events
  per comunicare fatti già accaduti.

Usa Outbox
  per evitare perdita di eventi nella doppia scrittura.

Usa RabbitMQ o un broker
  quando la comunicazione asincrona diventa reale.

Rendi i consumer idempotenti
  perché i messaggi possono arrivare più di una volta.

Usa Aspire
  per orchestrazione locale e osservabilità nello sviluppo distribuito.

Aggiungi OpenTelemetry e correlation ID
  prima che il sistema diventi difficile da diagnosticare.

Valuta costi e benefici
  microservizi non sono gratuiti.
```

# Riepilogo

In questo capitolo abbiamo mostrato come il progetto finale può evolvere verso microservizi.

Abbiamo visto:

```text id="mxokoy"
quando separare un modulo
Strangler Fig
estrazione di un servizio
eventi di integrazione
Outbox
RabbitMQ
Aspire
osservabilità distribuita
costi e benefici
```

La distinzione principale è questa:

```text id="qlqhkf"
Monolite modulare
  un solo deploy, confini interni chiari.

Microservizi
  deploy separati, dati separati, comunicazione distribuita.

Eventi di integrazione
  comunicano fatti tra servizi.

Outbox
  rende affidabile la pubblicazione degli eventi.

RabbitMQ
  trasporta messaggi tra produttori e consumatori.

Aspire
  aiuta a sviluppare e osservare localmente applicazioni distribuite.

Osservabilità distribuita
  rende diagnosticabile un sistema composto da più servizi.
```

La regola principale è questa:

```text id="dyfjkf"
I microservizi non sono il punto di partenza naturale.
Sono una possibile evoluzione quando il monolite modulare
non basta più e quando il team è pronto a sostenere
i costi operativi di un sistema distribuito.
```

Per il mini gestionale editoriale, la scelta migliore resta partire dal monolite modulare. Solo dopo, se emergono necessità reali, ha senso estrarre prima notifiche o report, introdurre eventi, outbox, RabbitMQ, Aspire e osservabilità distribuita.

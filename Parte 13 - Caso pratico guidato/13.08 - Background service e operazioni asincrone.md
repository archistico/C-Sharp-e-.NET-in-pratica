# 13.08 — Background service e operazioni asincrone

Nei capitoli precedenti abbiamo costruito API REST e interfaccia Blazor.

Ora aggiungiamo una parte molto importante nelle applicazioni reali: le **operazioni in background**.

Non tutto deve essere eseguito direttamente dentro una richiesta HTTP.

Alcune operazioni sono:

```text id="b4vk7d"
lente
schedulate
ripetitive
non interattive
da accodare
da ritentare in caso di errore temporaneo
```

Il punto centrale è questo:

```text id="izg2uj"
Una richiesta HTTP dovrebbe rispondere in tempi ragionevoli.
Se un lavoro è lungo o differibile, conviene accodarlo
ed elaborarlo con un background service.
```

In ASP.NET Core, le attività in background possono essere implementate come **hosted services**: classi con logica di background che implementano `IHostedService`, spesso tramite la classe base `BackgroundService`. La documentazione Microsoft mostra scenari come attività temporizzate, servizi scoped e code di attività eseguite in sequenza. ([Microsoft Learn][1])

# Quando NON serve

Non serve un background service per qualunque cosa.

Se l’operazione è:

```text id="w2baz6"
veloce
sincrona
necessaria per dare una risposta immediata

```

allora può restare nella normale richiesta HTTP.

Esempio:

```text id="zkw7ip"

```

Non serve accodare tutto.

Regola pratica:

```text id="jp61kh"

lungo, ripetitivo o schedulato.
```

# Worker service

Un **worker service** è un servizio che gira in background dentro l’host .NET.

Può essere:

```text id="j6akrx"

in un container separato
```

```text id="df4b59"

```

```text id="h2nqrf"

```

# Channel<T>

`Channel<T>` è una struttura .NET per passare dati tra produttori e consumatori in modo asincrono. Il namespace `System.Threading.Channels` fornisce strutture di sincronizzazione per produttori/consumatori asincroni ed è disponibile nel runtime condiviso da .NET Core 3.0 in poi. ([Microsoft Learn][2])

Nel nostro caso:

```text id="gxsxdo"
produttore:
  endpoint API che accoda un job

consumatore:
  BackgroundService che legge dalla coda ed elabora
```

Schema:

```text id="ku6iyx"
API endpoint
  Writer.WriteAsync(job)

BackgroundService
  Reader.ReadAllAsync()
```

# Implementazione con Channel<T>

`App.Infrastructure/Background/BackgroundTaskQueue.cs`:

```csharp id="kcxiou"
using App.Application.Background;
using System.Threading.Channels;

namespace App.Infrastructure.Background;

public class BackgroundTaskQueue : IBackgroundTaskQueue
{
    private readonly Channel<BackgroundJob> channel;

    public BackgroundTaskQueue()
    {
        BoundedChannelOptions options = new(capacity: 100)
        {
            FullMode = BoundedChannelFullMode.Wait,
            SingleReader = true,
            SingleWriter = false
        };

        channel = Channel.CreateBounded<BackgroundJob>(options);
    }

    public async ValueTask QueueAsync(
        BackgroundJob job,
        CancellationToken cancellationToken = default)
    {
        await channel.Writer.WriteAsync(job, cancellationToken);
    }

    public IAsyncEnumerable<BackgroundJob> DequeueAllAsync(
        CancellationToken cancellationToken = default)
    {
        return channel.Reader.ReadAllAsync(cancellationToken);
    }
}
```

Perché bounded?

```text id="r3wbqk"
Una coda senza limite può consumare memoria senza controllo.
Una coda bounded applica backpressure.
```

`Channel.CreateBounded` crea un canale con capacità massima; con il comportamento `Wait`, quando la coda è piena i writer attendono spazio disponibile. ([Microsoft Learn][3])

# BackgroundService

Creiamo un worker che legge dalla coda.

`App.Api/Background/QueuedBackgroundWorker.cs`:

```csharp id="j26v9c"
using App.Application.Background;

namespace App.Api.Background;

public class QueuedBackgroundWorker : BackgroundService
{
    private readonly IBackgroundTaskQueue queue;
    private readonly IServiceScopeFactory serviceScopeFactory;
    private readonly ILogger<QueuedBackgroundWorker> logger;

    public QueuedBackgroundWorker(
        IBackgroundTaskQueue queue,
        IServiceScopeFactory serviceScopeFactory,
        ILogger<QueuedBackgroundWorker> logger)
    {
        this.queue = queue;
        this.serviceScopeFactory = serviceScopeFactory;
        this.logger = logger;
    }

    protected override async Task ExecuteAsync(
        CancellationToken stoppingToken)
    {
        logger.LogInformation("QueuedBackgroundWorker avviato.");

        await foreach (BackgroundJob job in queue.DequeueAllAsync(stoppingToken))
        {
            try
            {
                await ProcessJobAsync(job, stoppingToken);
            }
            catch (OperationCanceledException)
                when (stoppingToken.IsCancellationRequested)
            {
                logger.LogInformation(
                    "QueuedBackgroundWorker in arresto.");
            }
            catch (Exception ex)
            {
                logger.LogError(
                    ex,
                    "Errore durante l'elaborazione del job {JobType}",
                    job.GetType().Name);
            }
        }
    }

    private async Task ProcessJobAsync(
        BackgroundJob job,
        CancellationToken cancellationToken)
    {
        using IServiceScope scope =
            serviceScopeFactory.CreateScope();

        switch (job)
        {
            case GenerateSalesReportJob reportJob:
                await ProcessGenerateSalesReportAsync(
                    reportJob,
                    scope.ServiceProvider,
                    cancellationToken);
                break;

            case RecalculateDashboardJob dashboardJob:
                await ProcessRecalculateDashboardAsync(
                    dashboardJob,
                    scope.ServiceProvider,
                    cancellationToken);
                break;

            default:
                logger.LogWarning(
                    "Job non riconosciuto: {JobType}",
                    job.GetType().Name);
                break;
        }
    }

    private async Task ProcessGenerateSalesReportAsync(
        GenerateSalesReportJob job,
        IServiceProvider serviceProvider,
        CancellationToken cancellationToken)
    {
        logger.LogInformation(
            "Generazione report vendite {JobId} dal {DataDa} al {DataA}",
            job.JobId,
            job.DataDa,
            job.DataA);

        IReportJobService reportJobService =
            serviceProvider.GetRequiredService<IReportJobService>();

        await reportJobService.GenerateSalesReportAsync(
            job.JobId,
            job.DataDa,
            job.DataA,
            cancellationToken);

        logger.LogInformation(
            "Report vendite {JobId} generato correttamente",
            job.JobId);
    }

    private async Task ProcessRecalculateDashboardAsync(
        RecalculateDashboardJob job,
        IServiceProvider serviceProvider,
        CancellationToken cancellationToken)
    {
        logger.LogInformation(
            "Ricalcolo dashboard {JobId}",
            job.JobId);

        IDashboardJobService dashboardJobService =
            serviceProvider.GetRequiredService<IDashboardJobService>();

        await dashboardJobService.RecalculateAsync(
            job.JobId,
            cancellationToken);
    }
}
```

Perché `IServiceScopeFactory`?

```text id="xp119z"
Il BackgroundService è registrato come singleton.
DbContext e servizi applicativi sono spesso scoped.
Per usare servizi scoped dentro un worker singleton,
bisogna creare uno scope.
```

La documentazione Microsoft mostra proprio il caso di hosted service che attiva un servizio scoped tramite DI. ([Microsoft Learn][1])

# Elaborazioni schedulate

Non tutti i job partono da una richiesta utente.

Alcuni sono schedulati:

```text id="vulctw"
ricalcolo dashboard ogni notte
pulizia file temporanei ogni ora
invio riepilogo giornaliero
sincronizzazione periodica
```

Per job schedulati semplici puoi usare `PeriodicTimer`.

`App.Api/Background/ScheduledDashboardWorker.cs`:

```csharp id="kbwgw4"
using App.Application.Background;

namespace App.Api.Background;

public class ScheduledDashboardWorker : BackgroundService
{
    private readonly IBackgroundTaskQueue queue;
    private readonly ILogger<ScheduledDashboardWorker> logger;

    public ScheduledDashboardWorker(
        IBackgroundTaskQueue queue,
        ILogger<ScheduledDashboardWorker> logger)
    {
        this.queue = queue;
        this.logger = logger;
    }

    protected override async Task ExecuteAsync(
        CancellationToken stoppingToken)
    {
        using PeriodicTimer timer =
            new(TimeSpan.FromHours(1));

        while (await timer.WaitForNextTickAsync(stoppingToken))
        {
            Guid jobId = Guid.NewGuid();

            logger.LogInformation(
                "Accodato ricalcolo dashboard {JobId}",
                jobId);

            await queue.QueueAsync(
                new RecalculateDashboardJob(jobId),
                stoppingToken);
        }
    }
}
```

```csharp id="q4doxg"

```


Ma per il progetto base `PeriodicTimer` è sufficiente.

# Graceful shutdown

Durante l’arresto:

```text id="l3a8d8"
l’host segnala cancellazione
il worker smette di prendere nuovi job
il job corrente dovrebbe terminare o interrompersi ordinatamente
le risorse vengono rilasciate
```

Non sempre puoi completare tutto.

Per job importanti:

```text id="k6iq4j"
usa stato persistente
segna il job come Running
se l’app si spegne, al riavvio recupera job incompleti
```

Con coda in memoria:

```text id="r8em4p"
i job non elaborati vengono persi al riavvio
```

Questa è una limitazione importante.

# Retry semplice manuale

Per il progetto base possiamo implementare un retry semplice.

```csharp id="zqrxpn"
private async Task ExecuteWithRetryAsync(
    Func<CancellationToken, Task> operation,
    int maxAttempts,
    CancellationToken cancellationToken)
{
    for (int attempt = 1; attempt <= maxAttempts; attempt++)
    {
        try
        {
            await operation(cancellationToken);
            return;
        }
        catch (Exception ex) when (attempt < maxAttempts)
        {
            TimeSpan delay =
                TimeSpan.FromSeconds(Math.Pow(2, attempt));

            logger.LogWarning(
                ex,
                "Tentativo {Attempt}/{MaxAttempts} fallito. Nuovo tentativo tra {Delay}.",
                attempt,
                maxAttempts,
                delay);

            await Task.Delay(delay, cancellationToken);
        }
    }
}
```

Uso:

```csharp id="c9ebw3"
await ExecuteWithRetryAsync(
    ct => ProcessJobAsync(job, ct),
    maxAttempts: 3,
    stoppingToken);
```

# Errori nei worker

Un errore non gestito dentro un worker può fermare l’elaborazione.

Per questo il loop principale deve proteggere ogni job:

```csharp id="x0xr6a"
try
{
    await ProcessJobAsync(job, stoppingToken);
}
catch (Exception ex)
{
    logger.LogError(ex, "Errore job");
}
```

Errore comune:

```text id="gnvhqr"
un job fallisce
il worker termina
nessun job successivo viene elaborato
```

Meglio:

```text id="kdrw2x"
un job fallisce
viene loggato
eventualmente marcato come Failed
il worker continua con il job successivo
```

# Code interne vs message broker

`Channel<T>` è una coda interna al processo.

Vantaggi:

```text id="y18kxz"
semplice
veloce
nessuna infrastruttura esterna
ottima per didattica
```

Limiti:

```text id="p7voei"
non persistente
si svuota se l’app si riavvia
non funziona tra più istanze
non scala su più processi
non offre dead-letter queue
non garantisce consegna
```

Per scenari production più seri:

```text id="yvnys5"
RabbitMQ
Azure Service Bus
Kafka
MassTransit
Hangfire
Quartz
database outbox/inbox
```

Regola:

```text id="lm4jx6"
Channel<T> è ottimo per code interne leggere.
Non è un sostituto completo di un broker di messaggi.
```

# Evitare scoped service nel singleton

Errore comune:

```csharp id="k0owxp"
public class Worker : BackgroundService
{
    private readonly AppDbContext dbContext;

    public Worker(AppDbContext dbContext)
    {
        this.dbContext = dbContext;
    }
}
```

Problema:

```text id="l9qqi5"
BackgroundService è singleton.
DbContext è scoped.
Non devi iniettare direttamente scoped service in singleton.
```

Soluzione:

```csharp id="az5nq6"
private readonly IServiceScopeFactory serviceScopeFactory;
```

e poi:

```csharp id="qhnn3x"
using IServiceScope scope =
    serviceScopeFactory.CreateScope();

AppDbContext dbContext =
    scope.ServiceProvider.GetRequiredService<AppDbContext>();
```

# Linee guida pratiche

```text id="d702sm"
Usa BackgroundService
  per lavori lunghi, schedulati o differibili.

Usa Channel<T>
  per una coda interna semplice e asincrona.

Usa una coda bounded
  per evitare crescita incontrollata in memoria.

Non usare Task.Run come coda
  negli endpoint API.

Crea scope DI nel worker
  quando usi servizi scoped come DbContext.

Rispetta CancellationToken
  in tutte le operazioni asincrone.

Logga inizio, fine ed errore dei job
  con JobId e tipo job.

Usa retry solo per errori transitori
  non per errori di validazione.

Gestisci gli errori per singolo job
  senza fermare tutto il worker.

Per produzione valuta code persistenti
  o message broker esterni.

Ricorda che Channel<T> è in memoria
  e perde i job se il processo si riavvia.
```

# Riepilogo

In questo capitolo abbiamo aggiunto operazioni asincrone e background processing al progetto finale.

Abbiamo visto:

```text id="hjxilw"
Worker service
Code interne con Channel<T>
Elaborazioni schedulate
Logging
Cancellazione
Retry
Errori
```

La distinzione principale è questa:

```text id="eqge74"
Endpoint API
  riceve la richiesta e accoda il lavoro.

Channel<T>
  conserva temporaneamente i job in memoria.

BackgroundService
  consuma la coda ed esegue i job.

Servizi applicativi
  contengono la logica del lavoro.

Logging
  rende osservabile quello che accade.

CancellationToken
  permette arresto ordinato.
```

La regola principale è questa:

```text id="k55sab"
Non trasformare una richiesta HTTP in un contenitore di lavori lunghi.
Accoda il lavoro, restituisci una risposta chiara,
ed elabora in background con logging, cancellazione e gestione errori.
```

Nel progetto finale questa infrastruttura resta volutamente semplice, ma introduce un concetto fondamentale per applicazioni .NET reali: separare il tempo della richiesta utente dal tempo dell’elaborazione applicativa.

# 13.02 — Analisi del dominio

Nel capitolo precedente abbiamo presentato il progetto finale: un **mini gestionale editoriale** con API, database, dashboard e report.

Prima di scrivere codice, però, serve una fase spesso trascurata: l’**analisi del dominio**.

Il punto centrale è questo:

```text id="rlw44p"
Un progetto non parte dalle tabelle.
Parte dai concetti del dominio, dai casi d’uso,
dalle regole e dai confini funzionali.
```

Se cominciamo subito da database, endpoint e pagine, rischiamo di costruire un’applicazione tecnicamente funzionante ma confusa.

In questo capitolo definiamo:

```text id="rx7alc"
requisiti
casi d’uso
entità
value object
DTO
regole di business
validazioni
errori
confini dei moduli
```

---

# Obiettivo dell’analisi

Il nostro progetto è un gestionale editoriale semplificato.

Non vogliamo modellare tutta la complessità reale di una casa editrice.

Vogliamo costruire un sistema abbastanza realistico per mostrare:

```text id="xm6e0q"
modellazione dati
API
business logic
validazione
EF Core
Blazor
test
report
architettura modulare
```

La regola didattica sarà:

```text id="c240yu"
semplice, ma non banale
```

# Value object

Un **value object** rappresenta un valore senza identità propria.

Si confronta per valore, non per Id.

Nel progetto base possiamo introdurre pochi value object per non complicare troppo il codice.

Possibili value object:

```text id="iwl2sh"
Money
PercentualeSconto
Isbn
Periodo
```

---

# Money

Un importo monetario deve usare `decimal`, non `double`.

Esempio semplice:

```csharp id="ht2z9f"
public readonly record struct Money(decimal Value)
{
    public static Money Zero => new(0m);

    public Money
    {
        if (Value < 0)
        {
            throw new ArgumentOutOfRangeException(
                nameof(Value),
                "L'importo non può essere negativo.");
        }
    }

    public static Money operator +(Money left, Money right)
    {
        return new Money(left.Value + right.Value);
    }
}
```

Per il progetto base, però, possiamo anche usare direttamente `decimal` nelle entity e spiegare che `Money` sarebbe un’evoluzione più rigorosa.

Scelta didattica consigliata:

```text id="m4u7qh"
usare decimal nel modello base
citare Money come evoluzione
```

---

# PercentualeSconto

Uno sconto deve essere compreso tra 0 e 100.

Value object:

```csharp id="m6j611"
public readonly record struct PercentualeSconto(decimal Value)
{
    public PercentualeSconto
    {
        if (Value < 0 || Value > 100)
        {
            throw new ArgumentOutOfRangeException(
                nameof(Value),
                "Lo sconto deve essere compreso tra 0 e 100.");
        }
    }
}
```

Anche qui, nella prima versione possiamo usare `decimal` e validare nel servizio.

---

# Isbn

L’ISBN potrebbe essere modellato come value object.

```csharp id="c15w38"
public readonly record struct Isbn(string Value)
{
    public Isbn
    {
        if (string.IsNullOrWhiteSpace(Value))
        {
            throw new ArgumentException(
                "ISBN non valido.",
                nameof(Value));
        }
    }
}
```

Validare davvero ISBN-10 e ISBN-13 richiede regole specifiche.
Per il progetto base, possiamo limitarci a:

```text id="x8087u"
ISBN opzionale
ISBN non vuoto se presente
ISBN unico se presente
```

# Periodo

Per report e dashboard è utile un value object `Periodo`.

```csharp id="nyihc4"
public readonly record struct Periodo(
    DateOnly DataDa,
    DateOnly DataA)
{
    public Periodo
    {
        if (DataDa > DataA)
        {
            throw new ArgumentException(
                "La data iniziale non può essere successiva alla data finale.");
        }
    }
}
```

Questo è un buon esempio perché evita di passare due date non validate in giro per il codice.

# Regole di business

Le regole di business sono le condizioni che mantengono valido il sistema.

---

## Regole catalogo

```text id="ejhvzg"
Un libro deve avere un titolo.

Un libro non può avere prezzo negativo.

Un ISBN, se presente, deve essere unico.

Un libro può essere disattivato.

Un libro usato in documenti non dovrebbe essere cancellato fisicamente.

Una collana disattivata non dovrebbe essere proposta come scelta predefinita
per nuovi libri.
```

---

## Regole clienti

```text id="t2tp88"
Un cliente deve avere una ragione sociale.

L’email, se presente, deve avere formato plausibile.

Un cliente con documenti associati non dovrebbe essere cancellato fisicamente.
```

---

## Regole documenti

```text id="yx2ek7"
Un documento deve avere un cliente.

Un documento deve avere una data.

Un documento confermato deve avere almeno una riga.

Una riga deve riferirsi a un libro esistente.

La quantità deve essere maggiore di zero.

Il prezzo unitario non può essere negativo.

Lo sconto deve essere compreso tra 0 e 100.

Il totale riga è calcolato dal sistema.

Il totale documento è la somma delle righe.

Un documento in bozza può essere modificato.

Un documento confermato non dovrebbe essere modificato liberamente.

Un documento annullato non dovrebbe essere modificato.
```

---

## Regole report

```text id="bh4wnj"
La data iniziale deve essere minore o uguale alla data finale.

I report vendite considerano solo documenti confermati.

I documenti annullati non entrano nei totali.
```

# Validazioni

Le validazioni servono a intercettare dati non validi.

Le divideremo in tre livelli.

---

## Validazione di input

Serve a controllare le request API.

Esempio:

```text id="m8u61h"
Titolo obbligatorio
Prezzo non negativo
Quantità maggiore di zero
Sconto tra 0 e 100
```

Questa validazione può stare negli endpoint o in classi dedicate.

---

## Validazione applicativa

Serve a verificare condizioni che richiedono accesso ai dati.

Esempi:

```text id="fai42l"
il cliente esiste
il libro esiste
l’ISBN non è già usato
la collana indicata esiste
gli autori indicati esistono
```

Queste regole richiedono database.

---

## Validazione di dominio

Serve a proteggere invarianti fondamentali.

Esempi:

```text id="zi3m6e"
un documento confermato non torna in bozza
una riga non può avere quantità zero
uno sconto non può essere maggiore di 100
```

In un modello più ricco, queste regole vivrebbero direttamente nelle entity o nei value object.

Nel progetto base possiamo usare servizi applicativi, ma spiegando il concetto.

Esempio semplice:

```csharp id="fiznc9"
public class Result
{
    private Result(bool success, string? error)
    {
        Success = success;
        Error = error;
    }

    public bool Success { get; }

    public string? Error { get; }

    public static Result Ok()
    {
        return new Result(true, null);
    }

    public static Result Fail(string error)
    {
        return new Result(false, error);
    }
}
```

Versione generica:

```csharp id="brc6c3"
public class Result<T>
{
    private Result(bool success, T? value, string? error)
    {
        Success = success;
        Value = value;
        Error = error;
    }

    public bool Success { get; }

    public T? Value { get; }

    public string? Error { get; }

    public static Result<T> Ok(T value)
    {
        return new Result<T>(true, value, null);
    }

    public static Result<T> Fail(string error)
    {
        return new Result<T>(false, default, error);
    }
}
```

Per il progetto finale, possiamo partire semplice e migliorare gradualmente.

# Decisioni prese

In questa analisi decidiamo che:

```text id="jrtwdd"
Il progetto è un monolite modulare.

SQLite è il database iniziale.

EF Core gestisce persistenza e relazioni.

Le API espongono DTO, non entity.

I documenti hanno stati: Bozza, Confermato, Annullato.

I totali documento sono calcolati dal sistema.

I documenti annullati non entrano nei report.

I report vendite considerano documenti confermati.

Le cancellazioni fisiche saranno limitate.

Libri e clienti usati nei documenti non verranno cancellati fisicamente.

La UI Blazor consumerà API HTTP.

La logica di calcolo sarà testata con xUnit.
```

# Riepilogo

In questo capitolo abbiamo fatto l’analisi del dominio del progetto finale.

Abbiamo definito:

```text id="ip2ppm"
requisiti
casi d’uso
entità
value object
DTO
regole di business
validazioni
errori
confini dei moduli
```

La distinzione più importante è questa:

```text id="wwtu5u"
Entità
  oggetti con identità e ciclo di vita.

Value object
  valori senza identità, confrontati per valore.

DTO
  oggetti per trasferire dati tra API e client.

Regole di business
  condizioni che mantengono valido il sistema.

Validazioni
  controlli su input, dati e invarianti.

Moduli
  confini funzionali del sistema.
```

La regola principale è questa:

```text id="btyb6b"
Prima di scrivere endpoint, tabelle e pagine,
devi sapere quali concetti stai modellando,
quali regole vuoi proteggere
e dove finiscono le responsabilità di ogni modulo.
```

Questa analisi non è burocrazia: è ciò che permette al progetto finale di essere un’applicazione ragionata, non una somma casuale di CRUD.

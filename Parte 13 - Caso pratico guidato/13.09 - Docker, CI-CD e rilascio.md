# Dockerfile

Un `Dockerfile` descrive come costruire un’immagine container.

Per applicazioni .NET useremo quasi sempre un **multi-stage build**:

```text id="iwgduh"
stage build
  usa immagine SDK
  restore
  build
  test, se necessario
  publish

stage runtime
  usa immagine ASP.NET Runtime
  copia output pubblicato
  avvia l’app
```

Vantaggio:

```text id="dy1gxz"
l’immagine finale contiene solo ciò che serve per eseguire l’app,
non tutto l’SDK.
```

Le immagini ufficiali .NET sono ottimizzate da Microsoft e sono disponibili nei registry ufficiali; l’immagine `dotnet/aspnet` contiene runtime ASP.NET Core e runtime .NET, mentre `dotnet/sdk` serve per sviluppo, build e test. ([Docker Hub][2])

# Dockerfile per App.Api

Percorso consigliato:

```text id="mbfhzk"
src/App.Api/Dockerfile
```

Esempio:

```dockerfile id="vrv2cp"
FROM mcr.microsoft.com/dotnet/sdk:10.0 AS build
WORKDIR /src

COPY ["src/App.Domain/App.Domain.csproj", "src/App.Domain/"]
COPY ["src/App.Application/App.Application.csproj", "src/App.Application/"]
COPY ["src/App.Infrastructure/App.Infrastructure.csproj", "src/App.Infrastructure/"]
COPY ["src/App.Api/App.Api.csproj", "src/App.Api/"]

RUN dotnet restore "src/App.Api/App.Api.csproj"

COPY . .

RUN dotnet publish "src/App.Api/App.Api.csproj" \
    -c Release \
    -o /app/publish \
    /p:UseAppHost=false

FROM mcr.microsoft.com/dotnet/aspnet:10.0 AS final
WORKDIR /app

COPY --from=build /app/publish .

ENV ASPNETCORE_URLS=http://+:8080
ENV ASPNETCORE_ENVIRONMENT=Production

EXPOSE 8080

ENTRYPOINT ["dotnet", "App.Api.dll"]
```

Nota:

```text id="y7q3vk"
Il container espone la porta 8080.
La porta pubblica verrà decisa da Docker Compose o dal sistema di deploy.
```

# Docker Compose

Docker Compose serve a eseguire più servizi insieme.

Nel nostro caso:

```text id="di2l49"
api
web
volume SQLite
eventuale reverse proxy in evoluzione
```

File:

```text id="mye9rz"
docker-compose.yml
```

Esempio:

```yaml id="k6mbn2"
services:
  api:
    build:
      context: .
      dockerfile: src/App.Api/Dockerfile
    image: minieditore-api:local
    environment:
      ASPNETCORE_ENVIRONMENT: Production
      ASPNETCORE_URLS: http://+:8080
      ConnectionStrings__Default: Data Source=/app/data/minieditore.db
    ports:
      - "8080:8080"
    volumes:
      - api-data:/app/data
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/health/live"]
      interval: 30s
      timeout: 5s
      retries: 3
      start_period: 20s

  web:
    build:
      context: .
      dockerfile: src/App.Web/Dockerfile
    image: minieditore-web:local
    environment:
      ASPNETCORE_ENVIRONMENT: Production
      ASPNETCORE_URLS: http://+:8080
      Api__BaseUrl: http://api:8080
    ports:
      - "8081:8080"
    depends_on:
      api:
        condition: service_healthy

volumes:
  api-data:
```

Nota importante:

```text id="66wkmi"
Dentro Docker Compose, App.Web chiama App.Api usando il nome del servizio:
http://api:8080

Dal browser invece accedi a:
http://localhost:8081
```

# Attenzione a curl nei container

Il `healthcheck` di Compose usa `curl`.

Alcune immagini runtime potrebbero non includere `curl`.

Alternative:

```text id="2mcy65"
installare curl nell’immagine
usare wget se presente
creare un piccolo healthcheck custom
lasciare l’healthcheck a orchestratore esterno
```

Per esempio, se vuoi installare `curl` nell’immagine finale Debian-based:

```dockerfile id="y2xc5k"
RUN apt-get update \
    && apt-get install -y --no-install-recommends curl \
    && rm -rf /var/lib/apt/lists/*
```

Per un progetto didattico, puoi mostrare il concetto e specificare che il comando va adattato all’immagine runtime effettiva.

# SQLite in container

SQLite va bene per il progetto didattico, ma in container bisogna capire dove viene salvato il file.

Se la connection string è:

```text id="mpids3"
Data Source=/app/data/minieditore.db
```

allora `/app/data` deve essere persistente.

Con Compose:

```yaml id="uyb7w4"
volumes:
  - api-data:/app/data
```

Senza volume:

```text id="wt929h"
il database viene perso quando il container viene eliminato
```

Per produzione reale, spesso conviene usare:

```text id="ax92m5"

database gestito
```

SQLite può essere adatto per:


# Workflow CI base

File:

```text id="wv06ah"
.github/workflows/ci.yml
```

```yaml id="cjkxt7"
name: CI

on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main

permissions:
  contents: read

jobs:
  build-test:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup .NET
        uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '10.0.x'

      - name: Restore
        run: dotnet restore MiniEditore.sln

      - name: Build
        run: dotnet build MiniEditore.sln -c Release --no-restore

      - name: Test
        run: dotnet test MiniEditore.sln -c Release --no-build
```

Questa è la base minima.

# Pipeline con publish artifact

Aggiungiamo publish di API e Web.

```yaml id="v7r60o"
name: CI

on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main

permissions:
  contents: read

jobs:
  build-test-publish:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup .NET
        uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '10.0.x'

      - name: Restore
        run: dotnet restore MiniEditore.sln

      - name: Build
        run: dotnet build MiniEditore.sln -c Release --no-restore

      - name: Test
        run: dotnet test MiniEditore.sln -c Release --no-build

      - name: Publish API
        run: >
          dotnet publish src/App.Api/App.Api.csproj
          -c Release
          --no-build
          -o ./artifacts/api

      - name: Publish Web
        run: >
          dotnet publish src/App.Web/App.Web.csproj
          -c Release
          --no-build
          -o ./artifacts/web

      - name: Upload API artifact
        uses: actions/upload-artifact@v4
        with:
          name: app-api
          path: ./artifacts/api

      - name: Upload Web artifact
        uses: actions/upload-artifact@v4
        with:
          name: app-web
          path: ./artifacts/web
```

Questo produce artifact scaricabili dalla run.

# Push immagini su registry

Per un deploy reale, serve pubblicare immagini su un registry:

```text id="e8qbo9"

Docker Hub
Azure Container Registry
Amazon ECR
Google Artifact Registry
```

Esempio GitHub Container Registry:

```yaml id="cjz1yq"
permissions:
  contents: read
  packages: write

jobs:
  docker-publish:
    runs-on: ubuntu-latest
    needs: build-test-publish

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Login to GitHub Container Registry
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build and push API image
        run: |
          docker build \
            -f src/App.Api/Dockerfile \
            -t ghcr.io/${{ github.repository_owner }}/minieditore-api:${{ github.sha }} \
            -t ghcr.io/${{ github.repository_owner }}/minieditore-api:latest \
            .
          docker push ghcr.io/${{ github.repository_owner }}/minieditore-api:${{ github.sha }}
          docker push ghcr.io/${{ github.repository_owner }}/minieditore-api:latest

      - name: Build and push Web image
        run: |
          docker build \
            -f src/App.Web/Dockerfile \
            -t ghcr.io/${{ github.repository_owner }}/minieditore-web:${{ github.sha }} \
            -t ghcr.io/${{ github.repository_owner }}/minieditore-web:latest \
            .
          docker push ghcr.io/${{ github.repository_owner }}/minieditore-web:${{ github.sha }}
          docker push ghcr.io/${{ github.repository_owner }}/minieditore-web:latest
```

# Health checks

Gli health checks servono a capire se l’applicazione è viva e pronta.

Tipi comuni:

```text id="ceecbm"
liveness
  il processo è vivo?

readiness
  l’app è pronta a ricevere traffico?

database
  il database risponde?

dependencies
  servizi esterni disponibili?
```

ASP.NET Core include middleware per health checks; per impostazione predefinita il middleware esegue tutti i controlli registrati, ma è possibile filtrare i controlli con un predicate. ([Microsoft Learn][5])

---

# Configurare health checks in App.Api

In `Program.cs`:

```csharp id="y3d8lx"
builder.Services.AddHealthChecks()
    .AddCheck("self", () => HealthCheckResult.Healthy());
```

Servono using:

```csharp id="fqz9dy"
using Microsoft.Extensions.Diagnostics.HealthChecks;
```

Mapping endpoint:

```csharp id="gmxs0j"
app.MapHealthChecks("/health/live");
app.MapHealthChecks("/health/ready");
```

Versione più utile:

```csharp id="x9d98v"
builder.Services.AddHealthChecks()
    .AddCheck("self", () => HealthCheckResult.Healthy(), tags: ["live"])
    .AddDbContextCheck<AppDbContext>("database", tags: ["ready"]);
```

Mapping filtrato:

```csharp id="a3z2k6"
app.MapHealthChecks("/health/live", new HealthCheckOptions
{
    Predicate = check => check.Tags.Contains("live")
});

app.MapHealthChecks("/health/ready", new HealthCheckOptions
{
    Predicate = check => check.Tags.Contains("ready")
});
```

Using:

```csharp id="etij3w"
using Microsoft.AspNetCore.Diagnostics.HealthChecks;
```

# Migration in produzione

Domanda importante:

```text id="te37sq"
Chi applica le migration?
```

Opzioni:

```text id="vxaenr"
applica automaticamente all’avvio
applica da pipeline
applica manualmente con script controllato
applica con job separato
```

Per sviluppo:

```text id="dg6lm0"
Database.Migrate() all’avvio può andare bene
```

Per produzione:

```text id="cnbngq"
meglio maggiore controllo
```

Motivi:

```text id="w4zyjm"
migration distruttive
downtime
rollback difficile
compatibilità tra versioni app e database
più istanze che partono insieme
```

Scelta didattica consigliata:

```text id="rmwcyo"
Development:
  DbInitializer con Migrate e seed

Production:
  migration applicate da pipeline o comando dedicato
```

# Deploy automatico da GitHub Actions

Un deploy semplice via SSH può essere:

```yaml id="myrve5"
name: Deploy

on:
  workflow_dispatch:

jobs:
  deploy:
    runs-on: ubuntu-latest

    steps:
      - name: Deploy via SSH
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.DEPLOY_HOST }}
          username: ${{ secrets.DEPLOY_USER }}
          key: ${{ secrets.DEPLOY_SSH_KEY }}
          script: |
            cd /opt/minieditore
            docker compose pull
            docker compose up -d
            docker image prune -f
```

Nota:

```text id="jy5d6z"
Questo è un esempio pratico, non l’unico modo.
In ambienti aziendali si può usare Azure DevOps,
GitHub Environments, approval manuali, registry privati,
runner self-hosted o strumenti di deploy dedicati.
```

# Rollback

Il rollback è fondamentale.

Se usi tag con SHA:

```text id="doz1dy"
minieditore-api:abc123
minieditore-web:abc123
```

puoi tornare a una versione precedente modificando `.env`:

```env id="nbndko"
API_IMAGE=ghcr.io/ORG/minieditore-api:oldsha
WEB_IMAGE=ghcr.io/ORG/minieditore-web:oldsha
```

Poi:

```bash id="tw14ft"
docker compose pull
docker compose up -d
```

Attenzione:

```text id="k0v3t2"
Il rollback dell’app è facile.
Il rollback del database può essere difficile.
```

Quindi le migration devono essere compatibili quando possibile.

# Errori comuni

## Usare Development in produzione

Sbagliato:

```text id="xd2gos"
ASPNETCORE_ENVIRONMENT=Development
```

Può esporre dettagli tecnici e configurazioni non adatte.

---

## Non usare volumi con SQLite

Senza volume, perdi il database quando elimini il container.

---

## Usare `latest` senza tag tracciabile

`latest` non dice quale commit è in produzione.

Meglio anche:

```text id="gqs03z"
github.sha
versione semantica
numero build
```

---

## Non testare Dockerfile in CI

Il codice può compilare, ma l’immagine può non costruirsi.

---

## Applicare migration automaticamente senza controllo

In produzione può essere rischioso, soprattutto con più istanze.

---

## Health check troppo superficiale

`/health/live` può dire che il processo è vivo.

`/health/ready` dovrebbe dire se l’app è pronta.

Sono due cose diverse.

---

## Mettere segreti in docker-compose.yml

Meglio usare:

```text id="d0b0eq"
.env fuori dal repository
secret manager
variabili ambiente del sistema di deploy
```

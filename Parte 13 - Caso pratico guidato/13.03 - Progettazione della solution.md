# 13.03 — Progettazione della solution

Nel capitolo precedente abbiamo analizzato il dominio del mini gestionale editoriale: requisiti, casi d’uso, entità, DTO, regole di business, validazioni ed errori.

Ora dobbiamo decidere **come organizzare la solution**.

Il punto centrale è questo:

```text
Una buona solution non serve solo a “mettere ordine”.
Serve a rendere chiaro dove vive ogni responsabilità:
dominio, casi d’uso, persistenza, API, UI e test.
```

Per il progetto finale useremo una **Clean Architecture leggera**.

Non vogliamo una struttura accademica o troppo astratta, ma nemmeno un progetto unico dove tutto finisce in `Program.cs`, `Controllers`, `Models` e `Services`.

---

# Obiettivo della progettazione

La solution deve permettere di separare:

```text
regole di dominio
casi d’uso applicativi
accesso ai dati
API HTTP
interfaccia Blazor
test
```

La struttura proposta sarà:

```text
MiniEditore.sln

src/
  App.Domain/
  App.Application/
  App.Infrastructure/
  App.Api/
  App.Web/

tests/
  App.Tests/
```

Useremo nomi generici `App.*` nella spiegazione, ma nel progetto reale potremmo chiamarli:

```text
MiniEditore.Domain
MiniEditore.Application
MiniEditore.Infrastructure
MiniEditore.Api
MiniEditore.Web
MiniEditore.Tests
```

Esempio `StatoDocumento`:

{

    Confermato = 2,
    Annullato = 3

Esempio `Documento`:

```csharp
namespace App.Domain.Documenti;

public class Documento
{
    public int Id { get; private set; }

    public string Numero { get; private set; } = "";

    public DateOnly Data { get; private set; }

    public int ClienteId { get; private set; }

    public StatoDocumento Stato { get; private set; } = StatoDocumento.Bozza;

    private readonly List<DocumentoRiga> righe = [];

    public IReadOnlyCollection<DocumentoRiga> Righe => righe.AsReadOnly();

    public decimal TotaleImponibile { get; private set; }

    public decimal TotaleSconto { get; private set; }

    public decimal TotaleDocumento { get; private set; }

    private Documento()
    {
    }

    public Documento(
        string numero,
        DateOnly data,
        int clienteId)
    {
        if (string.IsNullOrWhiteSpace(numero))
        {
            throw new DomainException("Il numero documento è obbligatorio.");
        }

        if (clienteId <= 0)
        {
            throw new DomainException("Il cliente è obbligatorio.");
        }

        Numero = numero;
        Data = data;
        ClienteId = clienteId;
        Stato = StatoDocumento.Bozza;
    }

    public void AggiungiRiga(
        int libroId,
        string descrizione,
        int quantita,
        decimal prezzoUnitario,
        decimal scontoPercentuale)
    {
        if (Stato != StatoDocumento.Bozza)
        {
            throw new DomainException(
                "È possibile modificare solo documenti in bozza.");
        }

        DocumentoRiga riga = new(
            libroId,
            descrizione,
            quantita,
            prezzoUnitario,
            scontoPercentuale);

        righe.Add(riga);

        RicalcolaTotali();
    }

    public void Conferma()
    {
        if (Stato != StatoDocumento.Bozza)
        {
            throw new DomainException(
                "Solo un documento in bozza può essere confermato.");
        }

        if (righe.Count == 0)
        {
            throw new DomainException(
                "Non è possibile confermare un documento senza righe.");
        }

        Stato = StatoDocumento.Confermato;
    }

    public void Annulla()
    {
        if (Stato == StatoDocumento.Annullato)
        {
            return;
        }

        Stato = StatoDocumento.Annullato;
    }

    private void RicalcolaTotali()
    {
        decimal totaleLordo = righe.Sum(x => x.TotaleLordo);
        decimal totaleNetto = righe.Sum(x => x.TotaleRiga);

        TotaleImponibile = totaleLordo;
        TotaleDocumento = totaleNetto;
        TotaleSconto = totaleLordo - totaleNetto;
    }
}
```

Questa classe protegge regole importanti:

```text
un documento non valido non dovrebbe esistere
un documento confermato non dovrebbe essere modificato liberamente
un documento senza righe non può essere confermato
i totali sono calcolati dal sistema
```

# AppDbContext

```csharp
using App.Application.Abstractions;
using App.Domain.Catalogo;
using App.Domain.Clienti;
using App.Domain.Documenti;
using Microsoft.EntityFrameworkCore;

namespace App.Infrastructure.Data;

public class AppDbContext : DbContext, IAppDbContext
{
    public AppDbContext(DbContextOptions<AppDbContext> options)
        : base(options)
    {
    }

    public DbSet<Libro> Libri => Set<Libro>();

    public DbSet<Autore> Autori => Set<Autore>();

    public DbSet<Collana> Collane => Set<Collana>();

    public DbSet<Cliente> Clienti => Set<Cliente>();

    public DbSet<Documento> Documenti => Set<Documento>();

    public DbSet<DocumentoRiga> DocumentoRighe => Set<DocumentoRiga>();

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        modelBuilder.ApplyConfigurationsFromAssembly(
            typeof(AppDbContext).Assembly);
    }
}
```

Interfaccia in `Application`:

```csharp
using App.Domain.Catalogo;
using App.Domain.Clienti;
using App.Domain.Documenti;
using Microsoft.EntityFrameworkCore;

namespace App.Application.Abstractions;

public interface IAppDbContext
{
    DbSet<Libro> Libri { get; }

    DbSet<Autore> Autori { get; }

    DbSet<Collana> Collane { get; }

    DbSet<Cliente> Clienti { get; }

    DbSet<Documento> Documenti { get; }

    DbSet<DocumentoRiga> DocumentoRighe { get; }

    Task<int> SaveChangesAsync(
        CancellationToken cancellationToken = default);
}
```

Nota didattica:

```text
Questa scelta mantiene Application indipendente da AppDbContext concreto,
ma accetta comunque una dipendenza da EF Core tramite DbSet.

È una Clean Architecture leggera, non purista.
```

In una versione più rigida, `Application` non dipenderebbe nemmeno da EF Core, ma per un progetto didattico questa soluzione è un buon compromesso.

# Configurazioni EF Core

Esempio:

```csharp
using App.Domain.Catalogo;
using Microsoft.EntityFrameworkCore;
using Microsoft.EntityFrameworkCore.Metadata.Builders;

namespace App.Infrastructure.Data.Configurations;

public class LibroConfiguration : IEntityTypeConfiguration<Libro>
{
    public void Configure(EntityTypeBuilder<Libro> builder)
    {
        builder.ToTable("Libri");

        builder.HasKey(x => x.Id);

        builder.Property(x => x.Titolo)
            .HasMaxLength(200)
            .IsRequired();

        builder.Property(x => x.Isbn)
            .HasMaxLength(20);

        builder.HasIndex(x => x.Isbn)
            .IsUnique()
            .HasFilter("Isbn IS NOT NULL");

        builder.Property(x => x.Prezzo)
            .HasPrecision(10, 2);
    }
}
```

Così il modello di persistenza resta ordinato e non sporca `OnModelCreating`.

# Esempio endpoint

```csharp
using App.Application.Catalogo.Libri;

namespace App.Api.Endpoints;

public static class CatalogoEndpoints
{
    public static IEndpointRouteBuilder MapCatalogoEndpoints(
        this IEndpointRouteBuilder app)
    {
        RouteGroupBuilder group = app
            .MapGroup("/api/catalogo")
            .WithTags("Catalogo");

        group.MapGet("/libri", GetLibriAsync);
        group.MapGet("/libri/{id:int}", GetLibroAsync);
        group.MapPost("/libri", CreateLibroAsync);
        group.MapPut("/libri/{id:int}", UpdateLibroAsync);
        group.MapDelete("/libri/{id:int}", DisattivaLibroAsync);

        return app;
    }

    private static async Task<IResult> GetLibriAsync(
        ILibroService libroService,
        CancellationToken cancellationToken)
    {
        IReadOnlyList<LibroListItemDto> libri =
            await libroService.GetLibriAsync(cancellationToken);

        return Results.Ok(libri);
    }

    private static async Task<IResult> GetLibroAsync(
        int id,
        ILibroService libroService,
        CancellationToken cancellationToken)
    {
        Result<LibroDetailDto> result =
            await libroService.GetLibroAsync(id, cancellationToken);

        return result.ToHttpResult();
    }

    private static async Task<IResult> CreateLibroAsync(
        CreateLibroRequest request,
        ILibroService libroService,
        CancellationToken cancellationToken)
    {
        Result<int> result =
            await libroService.CreateLibroAsync(request, cancellationToken);

        return result.Success
            ? Results.Created($"/api/catalogo/libri/{result.Value}", result.Value)
            : result.ToHttpResult();
    }

    private static async Task<IResult> UpdateLibroAsync(
        int id,
        UpdateLibroRequest request,
        ILibroService libroService,
        CancellationToken cancellationToken)
    {
        Result result =
            await libroService.UpdateLibroAsync(id, request, cancellationToken);

        return result.ToHttpResult();
    }

    private static async Task<IResult> DisattivaLibroAsync(
        int id,
        ILibroService libroService,
        CancellationToken cancellationToken)
    {
        Result result =
            await libroService.DisattivaLibroAsync(id, cancellationToken);

        return result.ToHttpResult();
    }
}
```

# Possibile progetto `App.Contracts`

Non era nella lista iniziale, ma vale la pena citarlo.

Per evitare di duplicare DTO tra API e Web, si potrebbe creare:

```text
App.Contracts
```

Con:

```text
DTO pubblici
request API
response API
```

Dipendenze:

```text
App.Application → App.Contracts
App.Api → App.Contracts
App.Web → App.Contracts
```

Però, per non aumentare troppo i progetti, nella prima versione possiamo tenere i DTO in `Application` e far referenziare `App.Web` ad `App.Application`, oppure duplicare solo i contratti necessari.

Scelta didattica consigliata:

```text
Per semplicità:
  App.Web può referenziare App.Application solo per i DTO.

Nota architetturale:
  in un progetto reale si può estrarre App.Contracts.
```

# Dipendenze tra progetti

Le dipendenze devono seguire una direzione chiara.

Schema:

```text
App.Domain
  non dipende da nessuno

App.Application
  dipende da App.Domain

App.Infrastructure
  dipende da App.Application
  dipende da App.Domain

App.Api
  dipende da App.Application
  dipende da App.Infrastructure

App.Web
  dipende da App.Application, oppure da App.Contracts se creato

App.Tests
  dipende dai progetti da testare
```

Schema visuale:

```text
              ┌────────────────┐
              │   App.Domain   │
              └───────▲────────┘
                      │
              ┌───────┴────────┐
              │ App.Application │
              └───────▲────────┘
                      │
              ┌───────┴────────┐
              │App.Infrastructure│
              └───────▲────────┘
                      │
              ┌───────┴────────┐
              │    App.Api      │
              └────────────────┘

              ┌────────────────┐
              │    App.Web      │
              └────────────────┘

              ┌────────────────┐
              │   App.Tests     │
              └────────────────┘
```

Più precisamente:

```text
App.Api usa Application e Infrastructure.

Application non conosce Api.

Domain non conosce Application.

Infrastructure conosce Application perché implementa le sue astrazioni.

Web chiama Api via HTTP.
```

---

# Regola delle dipendenze

La regola principale:

```text
Le dipendenze devono puntare verso il centro.
```

Centro:

```text
Domain
Application
```

Esterno:

```text
Infrastructure
Api
Web
```

Quindi:

```text
Domain non deve dipendere da Infrastructure.

Application non deve dipendere da Api.

Application non deve dipendere da Web.

Domain non deve sapere che esiste SQLite.

Domain non deve sapere che esiste Blazor.
```

---

# Clean Architecture leggera

Useremo una Clean Architecture semplificata.

Non introdurremo subito:

```text
MediatR obbligatorio
CQRS completo
repository generici
unit of work custom
mapping framework obbligatorio
domain events
pipeline behavior
Specification pattern
```

Questi concetti sono utili in progetti grandi, ma qui potrebbero appesantire.

La nostra versione leggera prevede:

```text
Domain:
  regole e modelli principali

Application:
  casi d’uso e DTO

Infrastructure:
  EF Core e servizi tecnici

Api:
  endpoint HTTP

Web:
  UI Blazor

Tests:
  test
```

Regola:

```text
Pulito non significa complicato.
Pulito significa che ogni cosa sta nel posto giusto.
```

public static class DependencyInjection
{
    public static IServiceCollection AddInfrastructure(
        this IServiceCollection services,
        IConfiguration configuration)
    {
        services.AddDbContext<AppDbContext>(options =>
        {
            options.UseSqlite(
                configuration.GetConnectionString("Default"));
        });

        services.AddScoped<IAppDbContext>(provider =>
            provider.GetRequiredService<AppDbContext>());

        services.AddSingleton<IClock, SystemClock>();

        return services;
    }
}

# Result pattern

```csharp
namespace App.Application.Common;

public class Result
{
    protected Result(
        bool success,
        string? error,
        ErrorType errorType)
    {
        Success = success;
        Error = error;
        ErrorType = errorType;
    }

    public bool Success { get; }

    public string? Error { get; }

    public ErrorType ErrorType { get; }

    public static Result Ok()
    {
        return new Result(true, null, ErrorType.None);
    }

    public static Result Fail(
        string error,
        ErrorType errorType = ErrorType.Validation)
    {
        return new Result(false, error, errorType);
    }
}
```

```csharp
namespace App.Application.Common;

public class Result<T> : Result
{
    private Result(
        bool success,
        T? value,
        string? error,
        ErrorType errorType)
        : base(success, error, errorType)
    {
        Value = value;
    }

    public T? Value { get; }

    public static Result<T> Ok(T value)
    {
        return new Result<T>(true, value, null, ErrorType.None);
    }

    public static new Result<T> Fail(
        string error,
        ErrorType errorType = ErrorType.Validation)
    {
        return new Result<T>(false, default, error, errorType);
    }
}
```

```csharp
namespace App.Application.Common;

public enum ErrorType
{
    None = 0,
    Validation = 1,
    NotFound = 2,
    Conflict = 3,
    Business = 4
}
```

# Conversione Result → HTTP

In `App.Api`:

```csharp
using App.Application.Common;

namespace App.Api.Extensions;

public static class ResultExtensions
{
    public static IResult ToHttpResult(this Result result)
    {
        if (result.Success)
        {
            return Results.NoContent();
        }

        return result.ErrorType switch
        {
            ErrorType.NotFound =>
                Results.NotFound(result.Error),

            ErrorType.Conflict =>
                Results.Conflict(result.Error),

            ErrorType.Validation =>
                Results.BadRequest(result.Error),

            ErrorType.Business =>
                Results.BadRequest(result.Error),

            _ =>
                Results.BadRequest(result.Error)
        };
    }

    public static IResult ToHttpResult<T>(this Result<T> result)
    {
        if (result.Success)
        {
            return Results.Ok(result.Value);
        }

        return result.ErrorType switch
        {
            ErrorType.NotFound =>
                Results.NotFound(result.Error),

            ErrorType.Conflict =>
                Results.Conflict(result.Error),

            ErrorType.Validation =>
                Results.BadRequest(result.Error),

            ErrorType.Business =>
                Results.BadRequest(result.Error),

            _ =>
                Results.BadRequest(result.Error)
        };
    }
}
```

Questa estensione evita di duplicare `if/else` in ogni endpoint.

# Configurazione per EF Core migrations

Poiché `AppDbContext` sta in `App.Infrastructure`, ma l’app eseguibile è `App.Api`, i comandi EF Core saranno di questo tipo:

```bash
dotnet ef migrations add InitialCreate \
  --project src/App.Infrastructure \
  --startup-project src/App.Api \
  --output-dir Data/Migrations
```

Aggiornamento database:

```bash
dotnet ef database update \
  --project src/App.Infrastructure \
  --startup-project src/App.Api
```

Questo è un punto importante:

```text
Il progetto che contiene il DbContext è Infrastructure.
Il progetto che contiene appsettings e host è Api.
```

# Web e API: scelta architetturale

Nel progetto finale possiamo scegliere due modalità:

## Modalità 1: API e Web separati

```text
App.Web
  chiama App.Api via HTTP
```

Vantaggi:

```text
separazione chiara
modello realistico
UI e API deployabili separatamente
prepara scenari mobile/desktop futuri
```

Svantaggi:

```text
configurazione CORS se domini diversi
più complessità rispetto a una sola app
```

## Modalità 2: Blazor dentro la stessa app API

```text
App.Api
  ospita API e UI
```

Vantaggi:

```text
più semplice
un solo deploy
meno configurazione
```

Svantaggi:

```text
meno separazione
meno chiaro per architetture client/server
```

Scelta consigliata per il libro:

```text
API e Web separati,
per mostrare bene il rapporto client/server.
```

# Dipendenza `App.Web → App.Application`

Questa dipendenza può sembrare strana perché `Application` contiene servizi applicativi.

Nel nostro caso la useremo solo per riusare DTO e request.

Alternative:

```text
creare App.Contracts
duplicare i DTO nel Web
generare client OpenAPI
```

Per semplicità didattica:

```text
teniamo App.Web → App.Application
ma spieghiamo che in un progetto reale App.Contracts
potrebbe essere una scelta più pulita.
```

Vantaggio:

# Errori da evitare

## Mettere tutto in `App.Api`

Sbagliato:

```text
entity
DbContext
endpoint
servizi
DTO
logica business
tutto nello stesso progetto
```

Funziona all’inizio, ma cresce male.

---

## Far dipendere `Domain` da EF Core

Sbagliato:

```csharp
using Microsoft.EntityFrameworkCore;
```

dentro `App.Domain`.

Il dominio non deve conoscere EF Core.

---

## Mettere query SQL o EF Core nella UI

Sbagliato:

```text
Blazor page → DbContext → SQLite
```

La UI deve chiamare API o servizi dedicati.

---

## Creare astrazioni inutili

Non serve introdurre subito:

```text
IRepository<T>
IUnitOfWork
IServiceManager
IManagerFactory
```

solo per “fare architettura”.

Le astrazioni devono risolvere problemi reali.

---

## Duplicare mapping ovunque

Se un mapping viene ripetuto in molti punti, meglio centralizzarlo o usare metodi di proiezione.

# Linee guida pratiche

```text
Tieni il dominio pulito
  niente EF Core, niente HTTP, niente Blazor.

Metti i casi d’uso in Application
  servizi applicativi, DTO, validazioni e Result.

Metti EF Core in Infrastructure
  DbContext, configurazioni, migration e seed.

Tieni Api sottile
  endpoint piccoli che chiamano Application.

Tieni Web separato
  UI Blazor che consuma API.

Usa DI
  per collegare servizi e implementazioni.

Usa configurazione esterna
  connection string e URL non hard-coded.

Usa logging strutturato
  nei servizi applicativi e negli endpoint importanti.

Organizza per modulo
  Catalogo, Clienti, Documenti, Dashboard, Report.

Non complicare troppo
  Clean Architecture leggera, non architettura cerimoniale.
```

# Riepilogo

In questo capitolo abbiamo progettato la solution del progetto finale.

Useremo questi progetti:

```text
App.Domain
App.Application
App.Infrastructure
App.Api
App.Web
App.Tests
```

La distinzione principale è questa:

```text
App.Domain
  regole e concetti del dominio.

App.Application
  casi d’uso, DTO, contratti e orchestrazione applicativa.

App.Infrastructure
  EF Core, SQLite e dettagli tecnici.

App.Api
  Minimal API e configurazione HTTP.

App.Web
  interfaccia Blazor.

App.Tests
  test unitari e di integrazione.
```

La regola principale è questa:

```text
Le dipendenze devono andare verso il centro:
Domain e Application non devono conoscere i dettagli esterni.
```

Questa struttura consente di costruire un gestionale piccolo ma realistico: abbastanza semplice da seguire, ma abbastanza ordinato da mostrare come si progetta una vera applicazione .NET moderna.

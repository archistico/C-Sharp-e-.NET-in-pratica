# 12.04 — Avalonia e lo sviluppo cross-platform

Nel capitolo precedente abbiamo visto .NET MAUI, orientato soprattutto allo sviluppo mobile e multipiattaforma.

Ora parliamo di **Avalonia**, una tecnologia molto interessante per applicazioni desktop cross-platform in .NET.

Il punto centrale è questo:

```text id="s8j9ta"
Avalonia è un framework UI cross-platform per .NET
che permette di creare applicazioni desktop moderne
per Windows, macOS e Linux usando C#, XAML e MVVM.
```

La documentazione ufficiale descrive Avalonia come un framework UI open source e cross-platform per .NET. Avalonia usa un proprio motore di rendering per disegnare i controlli, così che l’applicazione abbia aspetto e comportamento coerenti sulle diverse piattaforme. Supporta UI scritte in C# o F# con XAML e può essere distribuito su Windows, macOS, Linux desktop, Linux embedded, iOS, Android e WebAssembly, con livelli di supporto diversi. ([Documentazione Avalonia][1])

---

# Che cos’è Avalonia

Avalonia è un framework per costruire interfacce grafiche in .NET.

È particolarmente interessante per chi conosce WPF perché condivide molti concetti:

```text id="qk99pj"
XAML
binding
MVVM
controlli
layout
data template
resource
stili
property system
eventi
```

Ma Avalonia non è WPF.

È un framework diverso, progettato per funzionare su più piattaforme.

```text id="m1k3ap"
WPF
  solo Windows.

Avalonia
  Windows, macOS, Linux e altri target.
```

Avalonia ha anche una documentazione specifica per chi arriva da WPF, dove viene chiarito che molti concetti sono trasferibili, ma che le differenze principali riguardano styling, template, property system ed eventi. ([Documentazione Avalonia][2])

# Perché Avalonia è importante nell’ecosistema .NET

Per molto tempo, chi sviluppava desktop in .NET aveva una scelta abbastanza netta:

```text id="xnikop"
WinForms o WPF
  se l’app doveva girare su Windows.

Tecnologie diverse
  se servivano macOS o Linux.
```

Avalonia cambia questo scenario perché permette di rimanere nell’ecosistema .NET anche per desktop cross-platform.

Scenari tipici:

```text id="k3npxb"
tool tecnico per Windows, macOS e Linux
app gestionale desktop multipiattaforma
software per ambienti Linux
applicazioni per dispositivi embedded Linux
strumenti interni per sviluppatori
app con UI ricca e logica .NET condivisa
```

# Differenza tra Avalonia, WPF e .NET MAUI

È importante posizionare bene Avalonia.

```text id="mswy8x"
WPF
  desktop Windows enterprise.
  Ottimo per applicazioni Windows complesse.

.NET MAUI
  mobile e desktop cross-platform.
  Forte quando Android e iOS sono importanti.

Avalonia
  desktop cross-platform.
  Forte quando Windows, macOS e Linux sono importanti.
```

In pratica:

```text id="j90rsn"
Se devi fare una app Windows molto complessa:
  WPF resta spesso una scelta pragmatica.

Se devi fare una app mobile Android/iOS:
  .NET MAUI è più naturale.

Se devi fare una app desktop Windows/macOS/Linux:
  Avalonia è spesso una delle scelte più interessanti.
```

Regola pratica:

```text id="m4ncwn"
Avalonia non è “MAUI alternativo”.
Avalonia è soprattutto una risposta forte al problema:
voglio fare desktop cross-platform in .NET.
```

Per un libro generale su .NET, però, il focus più utile è:

```text id="rz1pam"
desktop cross-platform:

```

# Rendering cross-platform

Una delle differenze più importanti rispetto a .NET MAUI è il modello di rendering.

MAUI usa controlli nativi della piattaforma tramite handler.

Avalonia invece disegna direttamente i controlli tramite il proprio motore di rendering.

La documentazione Avalonia evidenzia che il framework usa un proprio rendering engine e che Skia è il backend predefinito; l’obiettivo è produrre output coerente tra piattaforme, senza wrapper di controlli nativi e senza molte differenze specifiche di piattaforma. ([Documentazione Avalonia][1])

Questo significa:

```text id="fz5ona"
la UI è più coerente tra Windows, macOS e Linux
il comportamento visuale dipende meno dai controlli nativi
lo styling può essere più uniforme
l’app ha un aspetto più controllabile
```

Ma significa anche:

```text id="c8qc4s"
non sempre avrai automaticamente il look nativo perfetto
di ogni sistema operativo
```

Regola pratica:

```text id="ya6cnc"
Avalonia punta alla coerenza cross-platform.
MAUI punta di più all’integrazione con controlli nativi.
```

# XAML e file `.axaml`

Avalonia usa XAML, ma i file hanno estensione `.axaml`.

La documentazione del tutorial iniziale spiega che `.axaml` significa Avalonia XAML ed è un’estensione specifica di Avalonia, usata per distinguere i file Avalonia dai normali file XAML. ([Documentazione Avalonia][4])

Esempio:

```text id="g0o3pb"
MainWindow.axaml
MainWindow.axaml.cs
```

Esempio XAML Avalonia:

```xml id="jutmfs"
<Window xmlns="https://github.com/avaloniaui"
        xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
        x:Class="MiaApp.Views.MainWindow"
        Title="MiaApp"
        Width="900"
        Height="600">

    <StackPanel Margin="20" Spacing="12">
        <TextBlock Text="Benvenuta in Avalonia"
                   FontSize="24"
                   FontWeight="Bold" />

        <TextBox Text="{Binding NomeCliente}" />

        <Button Content="Salva"
                Command="{Binding SalvaCommand}" />
    </StackPanel>

</Window>
```

Chi conosce WPF si sentirà abbastanza a casa, ma deve imparare alcune differenze di sintassi e comportamento.

# Binding

Avalonia supporta data binding e MVVM.

La documentazione ufficiale evidenzia supporto a compiled bindings controllati a build-time, pieno supporto MVVM e integrazione con CommunityToolkit.Mvvm. ([Documentazione Avalonia][1])

Esempio:

```xml id="ukn37k"
<TextBox Text="{Binding NomeCliente}" />
<TextBlock Text="{Binding NomeCliente}" />
```

ViewModel:

```csharp id="sucl25"
public class ClienteViewModel : INotifyPropertyChanged
{
    private string nomeCliente = "";

    public string NomeCliente
    {
        get => nomeCliente;
        set
        {
            if (nomeCliente == value)
            {
                return;
            }

            nomeCliente = value;
            OnPropertyChanged(nameof(NomeCliente));
        }
    }

    public event PropertyChangedEventHandler? PropertyChanged;

    private void OnPropertyChanged(string propertyName)
    {
        PropertyChanged?.Invoke(
            this,
            new PropertyChangedEventArgs(propertyName));
    }
}
```

# Comandi

Avalonia supporta il pattern `ICommand`.

XAML:

```xml id="rua8ox"
<Button Content="Carica"
        Command="{Binding CaricaCommand}" />
```

ViewModel:

```csharp id="h8sl82"
public ICommand CaricaCommand { get; }
```

Implementazione semplice:

```csharp id="q8czjg"
public class RelayCommand : ICommand
{
    private readonly Action execute;
    private readonly Func<bool>? canExecute;

    public RelayCommand(
        Action execute,
        Func<bool>? canExecute = null)
    {
        this.execute = execute;
        this.canExecute = canExecute;
    }

    public event EventHandler? CanExecuteChanged;

    public bool CanExecute(object? parameter)
    {
        return canExecute?.Invoke() ?? true;
    }

    public void Execute(object? parameter)
    {
        execute();
    }

    public void RaiseCanExecuteChanged()
    {
        CanExecuteChanged?.Invoke(this, EventArgs.Empty);
    }
}
```

Per progetti reali, spesso conviene usare `CommunityToolkit.Mvvm` per ridurre il codice boilerplate.

# Styling

Lo styling è una delle differenze più importanti rispetto a WPF.

Avalonia usa un sistema di styling ispirato a CSS, con selettori, classi, pseudoclassi e control themes. La documentazione ufficiale spiega che il meccanismo di styling con classi è analogo a CSS e che gli stili vengono cercati risalendo il logical tree; gli stili definiti a livello applicazione possono essere usati ovunque, ma possono essere sovrascritti più vicino al controllo. ([Documentazione Avalonia][5])

Esempio:

```xml id="snqgq6"
<Window.Styles>
    <Style Selector="TextBlock.h1">
        <Setter Property="FontSize" Value="24" />
        <Setter Property="FontWeight" Value="Bold" />
    </Style>

    <Style Selector="Button.primary">
        <Setter Property="Background" Value="#2563EB" />
        <Setter Property="Foreground" Value="White" />
        <Setter Property="Padding" Value="12 8" />
    </Style>
</Window.Styles>

<StackPanel Margin="20" Spacing="12">
    <TextBlock Classes="h1"
               Text="Titolo" />

    <Button Classes="primary"
            Content="Salva" />
</StackPanel>
```

Rispetto a WPF, questa è una differenza concettuale importante.

In WPF penseresti spesso a `Style` come risorsa con `TargetType` o `x:Key`.

In Avalonia pensi molto a:

```text id="ianu45"
selector
classi
pseudoclassi
control themes
```

# DataTemplate

I `DataTemplate` collegano un tipo di dato o ViewModel a una rappresentazione visuale.

Esempio lista:

```xml id="baos5q"
<ListBox ItemsSource="{Binding Clienti}">
    <ListBox.ItemTemplate>
        <DataTemplate>
            <StackPanel>
                <TextBlock Text="{Binding Nome}"
                           FontWeight="Bold" />
                <TextBlock Text="{Binding Email}" />
            </StackPanel>
        </DataTemplate>
    </ListBox.ItemTemplate>
</ListBox>
```

Per navigazione MVVM puoi usare `DataTemplate` per associare ViewModel e View.

Esempio concettuale:

```xml id="o58cmj"
<Application.DataTemplates>
    <DataTemplate DataType="{x:Type vm:ClientiViewModel}">
        <views:ClientiView />
    </DataTemplate>

    <DataTemplate DataType="{x:Type vm:OrdiniViewModel}">
        <views:OrdiniView />
    </DataTemplate>
</Application.DataTemplates>
```

La guida di migrazione da WPF segnala che i data template in Avalonia sono simili a WPF, ma sono conservati in una collezione `DataTemplates` invece che nelle risorse e supportano matching su interfacce e tipi derivati. ([Documentazione Avalonia][2])

# Property system

Avalonia ha un property system concettualmente simile a quello WPF, ma non identico.

In WPF si parla di `DependencyProperty`.

In Avalonia si usano proprietà tipizzate come:

```text id="zaqzre"
StyledProperty
DirectProperty
AttachedProperty
```

La documentazione di migrazione da WPF evidenzia che Avalonia usa generics fortemente tipizzati come `StyledProperty`, `DirectProperty` e `AttachedProperty`, invece di una singola classe `DependencyProperty`. ([Documentazione Avalonia][2])

Per un capitolo generale non serve entrare troppo nel dettaglio, ma il lettore deve sapere che:

```text id="ywymg8"
il property system esiste
serve per binding, styling, animazioni e proprietà dei controlli
non è una copia 1:1 di WPF
```

# Navigazione tra view

Avalonia non impone un unico modello di navigazione.

Per applicazioni desktop è comune usare:

```text id="jk63b5"
MainWindow
  menu laterale
  ContentControl centrale
  CurrentViewModel nel MainViewModel
  DataTemplate per mostrare la View corretta
```

Esempio:

```xml id="fsqvdi"
<Grid ColumnDefinitions="220,*">
    <StackPanel Background="#111827"
                Spacing="8"
                Padding="12">
        <Button Content="Dashboard"
                Command="{Binding ShowDashboardCommand}" />

        <Button Content="Clienti"
                Command="{Binding ShowClientiCommand}" />

        <Button Content="Ordini"
                Command="{Binding ShowOrdiniCommand}" />
    </StackPanel>

    <ContentControl Grid.Column="1"
                    Content="{Binding CurrentViewModel}" />
</Grid>
```

ViewModel:

```csharp id="h6k34n"
public class MainWindowViewModel : ViewModelBase
{
    private object? currentViewModel;

    public MainWindowViewModel(
        DashboardViewModel dashboardViewModel,
        ClientiViewModel clientiViewModel,
        OrdiniViewModel ordiniViewModel)
    {
        ShowDashboardCommand = new RelayCommand(
            () => CurrentViewModel = dashboardViewModel);

        ShowClientiCommand = new RelayCommand(
            () => CurrentViewModel = clientiViewModel);

        ShowOrdiniCommand = new RelayCommand(
            () => CurrentViewModel = ordiniViewModel);

        CurrentViewModel = dashboardViewModel;
    }

    public object? CurrentViewModel
    {
        get => currentViewModel;
        set => SetProperty(ref currentViewModel, value);
    }

    public ICommand ShowDashboardCommand { get; }
    public ICommand ShowClientiCommand { get; }
    public ICommand ShowOrdiniCommand { get; }
}
```

Questo modello è molto simile a quello usabile in WPF.

```csharp id="nj6odf"
public partial class App : Application
{
    public IServiceProvider Services { get; private set; } = null!;

    public override void OnFrameworkInitializationCompleted()
    {
        ServiceCollection services = new();

        services.AddSingleton<MainWindow>();
        services.AddSingleton<MainWindowViewModel>();

        services.AddTransient<ClientiViewModel>();
        services.AddTransient<IClientiService, ClientiService>();

        services.AddHttpClient<IClientiApiClient, ClientiApiClient>(
            client =>
            {
                client.BaseAddress =
                    new Uri("https://api.example.com");
            });

        Services = services.BuildServiceProvider();

        if (ApplicationLifetime is IClassicDesktopStyleApplicationLifetime desktop)
        {
            desktop.MainWindow =
                Services.GetRequiredService<MainWindow>();
        }

        base.OnFrameworkInitializationCompleted();
    }
}
```

# Integrazione con API

Avalonia è spesso usata come client desktop che consuma API ASP.NET Core.

Schema:

```text id="pbq52e"
Avalonia App
  ↓ HTTPS
ASP.NET Core API
  ↓
Database / servizi
```

Servizio:

```csharp id="d3whc3"
public interface IClientiApiClient
{
    Task<IReadOnlyList<ClienteDto>> GetClientiAsync(
        CancellationToken cancellationToken);
}
```

Implementazione:

```csharp id="s4gaf9"
public class ClientiApiClient : IClientiApiClient
{
    private readonly HttpClient httpClient;

    public ClientiApiClient(HttpClient httpClient)
    {
        this.httpClient = httpClient;
    }

    public async Task<IReadOnlyList<ClienteDto>> GetClientiAsync(
        CancellationToken cancellationToken)
    {
        IReadOnlyList<ClienteDto>? clienti =
            await httpClient.GetFromJsonAsync<IReadOnlyList<ClienteDto>>(
                "/api/clienti",
                cancellationToken);

        return clienti ?? [];
    }
}
```

ViewModel:

```csharp id="on397x"
public class ClientiViewModel : ViewModelBase
{
    private readonly IClientiApiClient clientiApiClient;

    public ClientiViewModel(IClientiApiClient clientiApiClient)
    {
        this.clientiApiClient = clientiApiClient;
        CaricaCommand = new AsyncRelayCommand(CaricaAsync);
    }

    public ObservableCollection<ClienteDto> Clienti { get; } = [];

    public ICommand CaricaCommand { get; }

    private async Task CaricaAsync()
    {
        IReadOnlyList<ClienteDto> clienti =
            await clientiApiClient.GetClientiAsync(CancellationToken.None);

        Clienti.Clear();

        foreach (ClienteDto cliente in clienti)
        {
            Clienti.Add(cliente);
        }
    }
}
```

# Migrazione da WPF

Avalonia è spesso interessante per team che conoscono WPF.

La documentazione ufficiale dice che Avalonia condivide molti concetti con WPF: controlli, layout, data binding e MVVM funzionano nello stesso modo o hanno equivalenti chiari. Tuttavia, le differenze principali sono styling, data template, property system ed eventi. ([Documentazione Avalonia][2])

Concetti trasferibili:

```text id="bgph4b"
XAML
Grid
StackPanel
binding
ViewModel
ICommand
ObservableCollection
DataTemplate
UserControl
Window
```

Concetti da reimparare:

```text id="r09md0"
sistema di styling
selector e classi
pseudoclassi
property system
alcuni nomi evento
alcuni controlli
deployment cross-platform
```

Regola pratica:

```text id="ygn4l2"
Se conosci WPF, Avalonia sarà familiare.
Ma non fare copia-incolla mentale totale:
è simile, non identico.
```

# Avalonia XPF

La documentazione cita anche **Avalonia XPF**, una soluzione pensata per eseguire applicazioni WPF esistenti cross-platform con modifiche minime, sopra il motore di rendering Avalonia. ([Documentazione Avalonia][1])

Per un libro generale è sufficiente dire:

```text id="kgm31y"
Avalonia classico
  serve per creare app Avalonia.

Avalonia XPF
  è un prodotto specifico per portare app WPF esistenti
  verso più piattaforme con meno riscrittura.
```

Non va confuso con Avalonia standard.

# Limiti pratici

Avalonia è potente, ma va valutata con realismo.

## Non è WPF identico

Molte conoscenze si trasferiscono, ma non tutto.

Differenze importanti:

```text id="z65yo9"
styling
template
property system
eventi
alcuni controlli
alcuni comportamenti
```

---

## Ecosistema componenti

WPF ha un ecosistema storico enorme di componenti enterprise.

Avalonia ha un ecosistema più giovane.

Prima di scegliere Avalonia per un gestionale complesso, verifica:

```text id="d0g46a"
DataGrid avanzate
chart
reporting
controlli terze parti
temi
editor complessi
componenti specializzati
```

---

## Desktop cross-platform richiede test reali

Non basta testare su Windows.

Devi testare su:

```text id="eu69d4"
Windows
macOS
Linux
diverse risoluzioni
diversi DPI
font diversi
window manager diversi su Linux
```

## Mobile e WebAssembly

Avalonia supporta anche mobile e WebAssembly, ma per un progetto business bisogna verificare maturità, requisiti, prestazioni e modello di distribuzione.

Per mobile, .NET MAUI può essere una scelta più naturale se Android/iOS sono il focus principale.

Se devi usare API native molto specifiche, devi prevedere codice per piattaforma.


# Errori comuni

## Pensare che Avalonia sia WPF multipiattaforma perfetto

Avalonia è simile a WPF, ma non è WPF.

---

## Ignorare lo styling diverso

Lo styling Avalonia è più CSS-like.
Chi arriva da WPF deve impararlo bene.

---

## Non testare su Linux/macOS

Se il requisito è cross-platform, bisogna testare su ogni piattaforma reale.

---

## Mettere logica nel code-behind

Come in WPF, è meglio usare MVVM e servizi.

---

## Pensare che cross-platform elimini il deployment per piattaforma

Ogni piattaforma ha pacchetti, firme, permessi e convenzioni proprie.

---

## Scegliere Avalonia solo perché “è moderno”

La scelta deve dipendere da target, UI, team, distribuzione e componenti richiesti.

# Linee guida pratiche

```text id="an8rm1"
Usa Avalonia
  per desktop cross-platform in .NET.

Usa XAML e MVVM
  per separare UI, stato e logica.

Usa Window e UserControl
  per organizzare la struttura visuale.

Usa binding e comandi
  per collegare View e ViewModel.

Usa ObservableCollection
  per liste dinamiche.

Studia bene lo styling
  perché è una delle differenze più importanti rispetto a WPF.

Usa DataTemplate
  per visualizzare dati e ViewModel.

Integra servizi e DI
  come nelle altre app moderne .NET.

Isola il codice platform-specific
  in servizi dedicati.

Testa su tutte le piattaforme target
  non solo su Windows.

Valuta l’ecosistema componenti
  prima di impegnarti su app enterprise complesse.

Non confondere Avalonia con MAUI
  Avalonia è forte sul desktop cross-platform;
  MAUI è più orientato a mobile e app native multipiattaforma.
```

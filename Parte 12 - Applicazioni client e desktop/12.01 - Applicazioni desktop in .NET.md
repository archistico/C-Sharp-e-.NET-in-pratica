# 12.01 — Applicazioni desktop in .NET

Finora abbiamo parlato molto di web, API, servizi, cloud, microservizi e DevOps. Però .NET non è solo backend e web.

Una parte importante dell’ecosistema .NET riguarda ancora le **applicazioni client e desktop**, soprattutto in ambito aziendale, industriale, gestionale, tecnico e amministrativo.

Il punto centrale è questo:

```text id="al9qi5"
Il desktop .NET non è una tecnologia unica.
È un ecosistema di tecnologie diverse, con obiettivi diversi:
WinForms, WPF, WinUI, .NET MAUI, Avalonia.
```

Microsoft mantiene una guida desktop per .NET dedicata alle tecnologie Windows UI basate su .NET, in particolare Windows Forms e WPF. ([Microsoft Learn][1])

---

# Perché parlare ancora di desktop

Negli ultimi anni molte applicazioni si sono spostate verso il web, ma il desktop resta importante quando servono:

```text id="y166gu"
integrazione forte con Windows
accesso a dispositivi locali
scanner
stampanti
lettori smart card
lettori documenti
porte seriali
file system locale
app offline
interfacce molto reattive
applicazioni gestionali interne
strumenti tecnici
software industriale
software medicale
software di laboratorio
applicazioni legacy da mantenere
```

Esempio:

```text id="agzhv5"
Un gestionale web è ottimo per inserimento dati standard.
Ma se devi pilotare scanner, lettori documenti, stampanti locali
o hardware specifico, il desktop può essere molto più naturale.
```

---

# Desktop moderno vs web

Prima di scegliere una tecnologia, bisogna decidere se l’app deve essere desktop o web.

## Web

Vantaggi:

```text id="qk7wh4"
distribuzione centralizzata
accesso da browser
aggiornamenti immediati
multi-piattaforma naturale
facile integrazione con API
adatto a molti utenti distribuiti
```

Svantaggi:

```text id="h36e7w"
integrazione hardware locale più difficile
dipendenza da rete/browser
UI molto ricche più complesse
offline più difficile
accesso limitato al sistema operativo
```

## Desktop

Vantaggi:

```text id="dldgte"
integrazione locale forte
prestazioni UI elevate
accesso a filesystem e dispositivi
offline più semplice
esperienza utente molto controllata
adatto a postazioni operative
```

Svantaggi:

```text id="y5pb06"
distribuzione più complessa
aggiornamenti client da gestire
dipendenza dal sistema operativo
installazione e permessi
supporto versioni diverse sulle postazioni
```

Regola pratica:

```text id="dgzllr"
Web quando vuoi accesso semplice e distribuzione centralizzata.
Desktop quando l’integrazione locale e l’esperienza operativa
sono più importanti.
```

# WinForms

**Windows Forms**, o WinForms, è una tecnologia storica per applicazioni desktop Windows.

È basata su un modello visuale semplice:

```text id="cr8ysh6"
form
controlli
eventi
designer visuale
codice code-behind
```

Esempio mentale:

```text id="u1442f"
Button.Click
TextBox.Text
DataGridView
Form.ShowDialog()
```

È molto usata in applicazioni aziendali legacy, strumenti interni e gestionali Windows.

---

# Quando usare WinForms

WinForms può avere senso quando:

```text id="w9pd0x"
devi mantenere applicazioni esistenti
devi creare tool Windows semplici
il team conosce bene il modello eventi
serve produttività rapida con designer
l’interfaccia non richiede grande personalizzazione grafica
hai molte librerie o componenti WinForms già pronti
```

Esempi:

```text id="dk9d9g"
tool amministrativo interno
utility di import/export
piccolo gestionale Windows
app legacy da modernizzare gradualmente
```

---

# Punti forti di WinForms

```text id="dcflvc"
semplice da imparare
designer maturo
ottimo per CRUD e form tradizionali
molta documentazione
molti componenti di terze parti
facile per app interne Windows
```

---

# Limiti di WinForms

```text id="fpsjic"
solo Windows
modello UI meno moderno
layout complessi meno naturali
binding meno potente di WPF
personalizzazione grafica più limitata
separazione UI/logica spesso debole se non disciplinata
```

Regola pratica:

```text id="yoqj9d"
WinForms è ottimo per semplicità e manutenzione legacy.
Non è la scelta migliore per UI moderne, molto personalizzate
o fortemente basate su binding.
```

# WPF

**WPF**, Windows Presentation Foundation, è una tecnologia desktop Windows basata su XAML, binding, template, risorse, stili e un modello grafico molto potente.

Microsoft descrive WPF come una tecnologia di interfaccia grafica open source per Windows in .NET. ([Microsoft Learn][2])

WPF introduce concetti importanti:

```text id="z2ko46"
XAML
binding
dependency properties
styles
templates
resources
commands
data templates
MVVM
```

È ancora molto usato in applicazioni desktop aziendali complesse.

---

# Quando usare WPF

WPF è una buona scelta quando:

```text id="x4w43u"
l’app è Windows-only
serve UI ricca e personalizzabile
serve binding potente
serve MVVM
hai griglie, dashboard, form complessi
devi integrare dispositivi locali Windows
vuoi separare UI e logica in modo serio
hai app WPF esistenti da mantenere o modernizzare
```

Esempi:

```text id="yvyuwv"
gestionale desktop avanzato
software di sportello
dashboard operativa
app con scanner e lettori documenti
strumenti tecnici con UI complessa
software industriale
```

---

# Punti forti di WPF

```text id="w6e5yg"
binding molto potente
MVVM naturale
stili e template avanzati
UI altamente personalizzabile
separazione logica/UI migliore di WinForms
ecosistema maturo
ottimo per applicazioni enterprise Windows
```

---

# Limiti di WPF

```text id="rspaod"
solo Windows
curva di apprendimento più alta

```

Regola pratica:


# WinUI

**WinUI 3** è il framework Microsoft moderno per creare applicazioni desktop Windows con interfaccia nativa, Fluent Design e Windows App SDK.

Microsoft descrive WinUI 3 come il framework UI nativo moderno per applicazioni desktop Windows, fornito tramite Windows App SDK, per creare esperienze moderne e reattive su Windows 10 versione 1809 e successive, incluso Windows 11. ([Microsoft Learn][3])

WinUI usa XAML, ma non è WPF.

È orientato al mondo Windows moderno.

---

# Quando usare WinUI

WinUI può avere senso quando:

```text id="kvevs5"
vuoi una nuova app Windows moderna
vuoi Fluent Design
vuoi integrazione con Windows App SDK
l’app è Windows 10/11
vuoi usare controlli moderni nativi
vuoi guardare al futuro dell’ecosistema Windows UI Microsoft
```

Esempi:

```text id="j6med2"
nuova app desktop Windows moderna
client aziendale con look Windows 11
app distribuita via MSIX/Store o canali aziendali
```

---

# Punti forti di WinUI

```text id="sv6wko"
UI moderna
Fluent Design
integrazione Windows App SDK
buona direzione per nuove app Windows native
supporto a controlli moderni
```

La Windows App SDK può anche essere usata con app esistenti WPF, Windows Forms, Win32 o altri framework, per aggiungere funzionalità moderne dello stack Windows. ([Microsoft Learn][4])

---

# Limiti di WinUI

```text id="mtz0dt"
solo Windows
ecosistema meno storico di WPF/WinForms
migrazione da WPF non automatica
alcune abitudini WPF non si trasferiscono direttamente
richiede conoscenza Windows App SDK
```

Regola pratica:

```text id="axp4b1"
WinUI è interessante per nuove app Windows moderne.
WPF resta spesso più pragmatico per molte applicazioni enterprise
già mature o molto complesse.
```

# .NET MAUI

**.NET MAUI** significa .NET Multi-platform App UI.

È il framework Microsoft cross-platform per creare app native mobile e desktop con C# e XAML.

Microsoft lo descrive come un framework cross-platform per creare app native mobile e desktop con C# e XAML, da una singola codebase, per Android, iOS, macOS e Windows. ([Microsoft Learn][5])

Piattaforme principali:

```text id="wcnses"
Android
iOS
macOS tramite Mac Catalyst
Windows tramite WinUI
```

La documentazione Microsoft specifica che le app .NET MAUI per Android, iOS e Windows possono essere compilate in Visual Studio, ma per lo sviluppo iOS è necessario un Mac in rete. ([Microsoft Learn][6])

---

.NET MAUI ha senso quando:

```text id="egv5ab"
vuoi mobile e desktop con codebase condivisa
devi supportare Android/iOS
vuoi usare C# e XAML
hai app companion mobile
vuoi riusare logica .NET tra client e backend
l’esperienza mobile è importante
```

Esempi:

```text id="nxo9b0"
app mobile aziendale
app per operatori sul campo
app mobile + desktop leggero
client multipiattaforma con UI relativamente uniforme
```

# Avalonia

**Avalonia** è un framework UI .NET cross-platform open source, molto interessante per applicazioni desktop multipiattaforma.

La documentazione ufficiale lo descrive come un framework UI cross-platform per .NET che può targetizzare Windows, macOS, Linux, iOS, Android e WebAssembly, con livelli di supporto diversi per piattaforma. ([docs.avaloniaui.net][7])

Avalonia usa XAML, ma non è WPF.

È molto apprezzato quando si vuole portare l’esperienza “simile a WPF” oltre Windows.

# MVVM

**MVVM** significa Model-View-ViewModel.

È un pattern molto usato nelle applicazioni XAML, soprattutto WPF, Avalonia, MAUI e in parte WinUI.

Schema:

```text id="qrmw5m"
View
  interfaccia grafica: XAML, controlli, layout.

ViewModel
  stato e comportamento presentazionale.

Model
  dati e logica di dominio/applicazione.
```

Obiettivo:

```text id="xlyiod"
separare la UI dalla logica
rendere il codice testabile
ridurre code-behind
favorire binding e comandi
```

---

# Esempio ViewModel

```csharp id="ezfzob"
public class ClienteViewModel : INotifyPropertyChanged
{
    private string nome = "";

    public string Nome
    {
        get => nome;
        set
        {
            if (nome == value)
            {
                return;
            }

            nome = value;
            OnPropertyChanged(nameof(Nome));
        }
    }

    public event PropertyChangedEventHandler? PropertyChanged;

    protected virtual void OnPropertyChanged(string propertyName)
    {
        PropertyChanged?.Invoke(
            this,
            new PropertyChangedEventArgs(propertyName));
    }
}
```

View XAML concettuale:

```xml id="sd3bql"
<TextBox Text="{Binding Nome, UpdateSourceTrigger=PropertyChanged}" />
<TextBlock Text="{Binding Nome}" />
```

Quando `Nome` cambia, la UI si aggiorna.

---

# Binding

Il **binding** collega proprietà del ViewModel e controlli UI.

Esempio:

```xml id="aup4l4"
<TextBox Text="{Binding Nome}" />
```

Significa:

```text id="qewv9h"
il TextBox mostra e/o aggiorna la proprietà Nome
del DataContext corrente
```

Tipi comuni:

```text id="lyd2v2"
OneWay
  ViewModel → UI

TwoWay
  ViewModel ↔ UI

OneTime
  valore letto una volta

OneWayToSource
  UI → ViewModel
```

Il binding è uno dei motivi principali per cui WPF, MAUI, Avalonia e WinUI favoriscono MVVM.

---

# Comandi

I **comandi** permettono alla View di invocare azioni del ViewModel senza gestire direttamente eventi nel code-behind.

Esempio:

```csharp id="c8jb12"
public ICommand SalvaCommand { get; }
```

XAML:

```xml id="c0i4qm"
<Button Content="Salva"
        Command="{Binding SalvaCommand}" />
```

Vantaggi:

```text id="j5si7m"
meno code-behind
azioni testabili
abilitazione/disabilitazione centralizzata
separazione UI/logica
```

Esempio concettuale:

```csharp id="an2zgg"
public class ClienteViewModel
{
    public ClienteViewModel()
    {
        SalvaCommand = new RelayCommand(
            execute: Salva,
            canExecute: PuoSalvare);
    }

    public ICommand SalvaCommand { get; }

    private void Salva()
    {
        // salva cliente
    }

    private bool PuoSalvare()
    {
        return !string.IsNullOrWhiteSpace(Nome);
    }

    public string Nome { get; set; } = "";
}
```

# Validazione

La validazione desktop può avvenire a più livelli.

```text id="jzplgm"
validazione UI
  campo obbligatorio, formato email, numeri validi

validazione ViewModel
  regole presentazionali

  invarianti vere del business

  vincoli finali e concorrenza
```

Esempio:

```text id="n3t0uh"

  validazione UI/ViewModel

Un ordine spedito non può essere annullato
  regola di dominio

Il codice cliente deve essere unico
  vincolo database/API
```

In WPF, spesso si usano approcci come:

```text id="n9s2vu"

ValidationRules
DataAnnotations

```

Regola pratica:

```text id="vw0ocj"

Ma le regole di dominio devono vivere anche fuori dalla UI.
```

# Framework-dependent vs self-contained desktop

## Framework-dependent

```text id="a4483y"
l’app richiede .NET installato sulla macchina
pacchetto più piccolo
runtime aggiornabile centralmente
```

## Self-contained

```text id="lcxubk"
l’app include il runtime .NET
pacchetto più grande
meno prerequisiti sulla macchina
ogni app porta la sua versione runtime
```

Esempio publish WPF:

```bash id="ayj8nv"
dotnet publish -c Release -r win-x64 --self-contained true
```

Single-file:

```bash id="fjkd5m"
dotnet publish -c Release -r win-x64 \
  --self-contained true \
  -p:PublishSingleFile=true
```

# Desktop e API

Una scelta architetturale moderna è evitare che il client desktop contenga tutta la logica e acceda direttamente al database.

Scenario tradizionale:

```text id="o2yfu9"
Desktop App
  ↓
Database
```

Scenario più moderno:

```text id="uqcpqo"
Desktop App
  ↓
API
  ↓
Database
```

Vantaggi:

```text id="yi6vp2"
sicurezza migliore
regole business centralizzate
database non esposto ai client
aggiornamenti server-side
possibile riuso da web/mobile
logging e audit più controllati
```

Svantaggi:

```text id="wdt26g"
serve rete
serve backend
serve gestione autenticazione
offline più complesso
```

Regola pratica:

```text id="eqa1dk"
Per nuove applicazioni aziendali,
valuta seriamente desktop come client di API,
non come applicazione che parla direttamente col database.
```

# Errori comuni

## Scegliere desktop quando basta web

Se l’app è solo CRUD gestionale e deve essere usata da molte sedi, spesso web è più semplice.

---

## Scegliere web quando serve integrazione hardware forte

Scanner, smart card, stampanti locali e periferiche speciali possono rendere il desktop più adatto.

---

## Usare WPF senza MVVM

Si può fare, ma si rischia code-behind enorme e poco testabile.

---

## Mettere logica business nella View

La View deve gestire UI.
Le regole devono stare nel ViewModel, application layer o domain layer.

---

## Accesso diretto al database dal client senza valutazione

In molte architetture moderne è preferibile usare API.

---

## Ignorare la distribuzione

Un’app desktop non è finita quando compila.
Bisogna pensare ad aggiornamenti, installer, runtime, permessi e rollback.

---

## Pensare che cross-platform significhi zero differenze

Anche con MAUI o Avalonia, le piattaforme hanno differenze reali:

```text id="m56xxo"
filesystem
permessi
font
notifiche
dimensioni schermo
packaging
input
accessibilità
comportamento finestre
```

# Linee guida pratiche

```text id="jrl6aa"
Usa WinForms
  per legacy, tool rapidi e applicazioni Windows semplici.

Usa WPF
  per desktop Windows enterprise complesso, MVVM e UI ricche.

Usa WinUI
  per nuove app Windows moderne basate su Fluent Design.

Usa .NET MAUI
  quando mobile e multipiattaforma sono centrali.

Usa Avalonia
  quando vuoi desktop .NET cross-platform serio,
  soprattutto Windows/macOS/Linux.

Valuta web
  quando la distribuzione centralizzata è più importante
  dell’integrazione locale.

Usa MVVM
  per separare UI e logica nelle tecnologie XAML.

Usa binding e comandi
  per ridurre code-behind e aumentare testabilità.

Valida a più livelli
  UI, ViewModel, dominio, backend/database.

Pensa presto alla distribuzione
  installer, update, runtime, firma, permessi.

Evita segreti nel client
  preferisci API backend per accesso a dati sensibili.
```

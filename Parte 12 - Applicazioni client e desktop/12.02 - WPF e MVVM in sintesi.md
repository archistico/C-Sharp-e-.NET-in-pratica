# 12.02 — WPF e MVVM in sintesi

Nel capitolo precedente abbiamo visto le principali tecnologie desktop dell’ecosistema .NET: WinForms, WPF, WinUI, .NET MAUI e Avalonia.

Ora facciamo un focus sintetico su **WPF e MVVM**, senza trasformare questo capitolo in un libro nel libro.

L’obiettivo non è spiegare tutto WPF nel dettaglio, ma dare al lettore una mappa chiara dei concetti fondamentali:

```text id="kdy7la"
XAML
Window
UserControl
Binding
DataContext
INotifyPropertyChanged
ICommand
ObservableCollection
ResourceDictionary
Styling
navigazione tra view
integrazione con servizi e Dependency Injection
```

Il punto centrale è questo:

```text id="avbby3"
WPF è potente perché separa bene interfaccia, stato e comportamento:
la View descrive la UI in XAML,
il ViewModel espone dati e comandi,
il Model e i servizi gestiscono logica e dati.
```

Microsoft descrive WPF come un framework UI per applicazioni desktop Windows basato su .NET, con un modello dichiarativo tramite XAML, data binding, controlli, layout, risorse e stili. ([learn.microsoft.com](https://learn.microsoft.com/en-us/dotnet/desktop/wpf/overview/?view=netdesktop-10.0))

# Perché WPF

WPF è particolarmente adatto quando devi costruire applicazioni desktop Windows con:

```text id="exv0ae"
interfacce ricche
binding dati complesso
griglie e form articolati
dashboard operative
stili personalizzati
template grafici
validazione lato client
integrazione con dispositivi locali
separazione UI/logica tramite MVVM
```

Esempi:

```text id="p2zjct"
gestionale desktop enterprise
software di sportello
applicazione con scanner o lettori documenti
dashboard interna
strumento tecnico per operatori
client desktop che consuma API
```

WPF non è la scelta migliore quando vuoi:

```text id="lvzds6"
una web app accessibile da browser
un’app mobile Android/iOS
un desktop cross-platform Windows/macOS/Linux
```

In quei casi, valuta ASP.NET Core/Blazor, .NET MAUI o Avalonia.

# XAML

**XAML** è il linguaggio dichiarativo usato da WPF per descrivere l’interfaccia.

Esempio:

```xml id="lyol6m"
<Window x:Class="MiaApp.MainWindow"
        xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
        xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
        Title="MiaApp"
        Width="800"
        Height="450">

    <Grid Margin="20">
        <StackPanel>
            <TextBlock Text="Nome cliente" />
            <TextBox Text="{Binding NomeCliente}" />
            <Button Content="Salva"
                    Command="{Binding SalvaCommand}" />
        </StackPanel>
    </Grid>
</Window>
```

XAML descrive:

```text id="r9k8ra"
layout
controlli
binding
stili
risorse
template
eventuali comportamenti UI
```

Il codice C# dovrebbe contenere la logica applicativa, non la struttura visuale.

# Code-behind

Ogni `Window` o `UserControl` ha spesso un file code-behind.

Esempio:

```text id="jh22pi"
MainWindow.xaml
MainWindow.xaml.cs
```

Il code-behind non è vietato.

È adatto per:

```text id="jx8246"

inizializzazione della view
gestione di eventi strettamente UI
interazioni difficili da esprimere in XAML
```

Ma non dovrebbe contenere:

```text id="qxc96j"

```

Regola pratica:

```text id="v2dobg"
Il code-behind deve conoscere la UI.

I servizi devono conoscere dati, API e infrastruttura.
```

# View e ViewModel

Nel pattern MVVM:

```text id="gk9kxn"
View
  XAML: controlli, layout, stili.

ViewModel
  proprietà, stato, comandi, validazione presentazionale.

Model
  dati, entità, DTO, logica di dominio.
```

Esempio:

```text id="cj5h96"
ClientiView.xaml
  mostra lista clienti.

ClientiViewModel.cs
  espone ObservableCollection<ClienteDto>,
  CaricaCommand,
  SalvaCommand,
  ClienteSelezionato.

ClienteDto.cs / Cliente.cs
  rappresenta i dati.
```

La View non dovrebbe sapere come caricare dati dal database o da una API.
Chiama il ViewModel tramite binding e comandi.

# Binding

Il **binding** collega proprietà del ViewModel a proprietà della View.

Esempio:

```xml id="yty1go"
<TextBox Text="{Binding NomeCliente}" />
```

Se il `DataContext` della View è un ViewModel con proprietà `NomeCliente`, il TextBox legge o aggiorna quella proprietà.

Esempio ViewModel:

```csharp id="j6mzfv"
public class ClienteViewModel : INotifyPropertyChanged
{
    private string nomeCliente = "";

    public string NomeCliente
    {
        get => nomeCliente;
        set
        {
            if (nomeCliente == value)
            {
                return;
            }

            nomeCliente = value;
            OnPropertyChanged(nameof(NomeCliente));
        }
    }

    public event PropertyChangedEventHandler? PropertyChanged;

    protected void OnPropertyChanged(string propertyName)
    {
        PropertyChanged?.Invoke(
            this,
            new PropertyChangedEventArgs(propertyName));
    }
}
```

---

# Modalità di binding

Le modalità più comuni sono:

```text id="n1ewbj"
OneWay
  ViewModel → View

TwoWay
  ViewModel ↔ View

OneTime
  letto una volta

OneWayToSource
  View → ViewModel
```

Esempio:

```xml id="tt3m8w"
<TextBox Text="{Binding NomeCliente,
                        Mode=TwoWay,
                        UpdateSourceTrigger=PropertyChanged}" />
```

`UpdateSourceTrigger=PropertyChanged` aggiorna il ViewModel mentre l’utente digita.

Senza specificarlo, alcuni controlli aggiornano la sorgente solo alla perdita del focus.

# INotifyPropertyChanged

`INotifyPropertyChanged` è l’interfaccia che permette al ViewModel di notificare alla View che una proprietà è cambiata.

Senza notifica:

```text id="k0g67h"
il valore cambia nel ViewModel,
ma la UI potrebbe non aggiornarsi.
```

Implementazione base:

```csharp id="igg35p"
public abstract class ViewModelBase : INotifyPropertyChanged
{
    public event PropertyChangedEventHandler? PropertyChanged;

    protected void OnPropertyChanged(
        [CallerMemberName] string? propertyName = null)
    {
        PropertyChanged?.Invoke(
            this,
            new PropertyChangedEventArgs(propertyName));
    }

    protected bool SetProperty<T>(
        ref T field,
        T value,
        [CallerMemberName] string? propertyName = null)
    {
        if (EqualityComparer<T>.Default.Equals(field, value))
        {
            return false;
        }

        field = value;
        OnPropertyChanged(propertyName);
        return true;
    }
}
```

Uso:

```csharp id="wweflj"
public class ClienteViewModel : ViewModelBase
{
    private string nomeCliente = "";

    public string NomeCliente
    {
        get => nomeCliente;
        set => SetProperty(ref nomeCliente, value);
    }
}
```

Questa classe base evita molto codice ripetitivo.

# ICommand

`ICommand` rappresenta un’azione invocabile dalla UI.

Esempio XAML:

```xml id="f7b561"
<Button Content="Salva"
        Command="{Binding SalvaCommand}" />
```

ViewModel:

```csharp id="l3c1ed"
public ICommand SalvaCommand { get; }
```

Implementazione semplice di `RelayCommand`:

```csharp id="v26fj3"
public class RelayCommand : ICommand
{
    private readonly Action execute;
    private readonly Func<bool>? canExecute;

    public RelayCommand(
        Action execute,
        Func<bool>? canExecute = null)
    {
        this.execute = execute;
        this.canExecute = canExecute;
    }

    public event EventHandler? CanExecuteChanged;

    public bool CanExecute(object? parameter)
    {
        return canExecute?.Invoke() ?? true;
    }

    public void Execute(object? parameter)
    {
        execute();
    }

    public void RaiseCanExecuteChanged()
    {
        CanExecuteChanged?.Invoke(this, EventArgs.Empty);
    }
}
```

Uso:

```csharp id="c6vse7"
public class ClienteViewModel : ViewModelBase
{
    private string nomeCliente = "";

    public ClienteViewModel()
    {
        SalvaCommand = new RelayCommand(
            execute: Salva,
            canExecute: PuoSalvare);
    }

    public string NomeCliente
    {
        get => nomeCliente;
        set
        {
            if (SetProperty(ref nomeCliente, value))
            {
                ((RelayCommand)SalvaCommand).RaiseCanExecuteChanged();
            }
        }
    }

    public ICommand SalvaCommand { get; }

    private bool PuoSalvare()
    {
        return !string.IsNullOrWhiteSpace(NomeCliente);
    }

    private void Salva()
    {
        // logica di salvataggio
    }
}
```

Con `CanExecute`, il bottone può abilitarsi o disabilitarsi automaticamente.

# Comandi asincroni

Molte azioni reali sono asincrone:

```text id="a3w575"
chiamare API
salvare su database
leggere file
caricare dati
```

Un comando asincrono deve evitare doppi click e gestire errori.

Esempio concettuale:

```csharp id="zp2fzx"
public class AsyncRelayCommand : ICommand
{
    private readonly Func<Task> execute;
    private readonly Func<bool>? canExecute;
    private bool isExecuting;

    public AsyncRelayCommand(
        Func<Task> execute,
        Func<bool>? canExecute = null)
    {
        this.execute = execute;
        this.canExecute = canExecute;
    }

    public event EventHandler? CanExecuteChanged;

    public bool CanExecute(object? parameter)
    {
        return !isExecuting && (canExecute?.Invoke() ?? true);
    }

    public async void Execute(object? parameter)
    {
        if (!CanExecute(parameter))
        {
            return;
        }

        try
        {
            isExecuting = true;
            RaiseCanExecuteChanged();

            await execute();
        }
        finally
        {
            isExecuting = false;
            RaiseCanExecuteChanged();
        }
    }

    public void RaiseCanExecuteChanged()
    {
        CanExecuteChanged?.Invoke(this, EventArgs.Empty);
    }
}
```

Uso:

```csharp id="j0cwxt"
CaricaClientiCommand = new AsyncRelayCommand(CaricaClientiAsync);
```

Regola pratica:

```text id="mg7kk3"
La UI deve restare reattiva.
Le operazioni lente devono essere asincrone.
```

# ObservableCollection

`ObservableCollection<T>` è una collezione che notifica la UI quando vengono aggiunti, rimossi o modificati elementi nella lista.

Esempio:

```csharp id="e2h1pr"
public ObservableCollection<ClienteDto> Clienti { get; } = [];
```

XAML:

```xml id="x3zfsd"
<ListBox ItemsSource="{Binding Clienti}"
         DisplayMemberPath="Nome" />
```

Aggiunta:

```csharp id="c8cw01"
Clienti.Add(new ClienteDto
{
    Id = 1,
    Nome = "Mario Rossi"
});
```

La `ListBox` si aggiorna automaticamente.

---

# ObservableCollection: attenzione ai thread

La `ObservableCollection` deve essere modificata dal thread UI.

Errore tipico:

```text id="q3emxk"
CollectionView does not support changes to its SourceCollection
from a thread different from the Dispatcher thread.
```

Se carichi dati in background, aggiorna la collezione sul Dispatcher/UI thread.

Esempio:

```csharp id="j55ru2"
Application.Current.Dispatcher.Invoke(() =>
{
    Clienti.Add(cliente);
});
```

Meglio ancora: carica i dati in una lista temporanea e poi aggiorna la collezione in UI thread.

# Navigazione tra view

In WPF non esiste un unico modo obbligatorio per navigare.

Per applicazioni enterprise è comune usare:

```text id="dceryi"
MainWindow
  contiene menu laterale/header/footer

ContentControl
  mostra la view corrente

NavigationService custom
  cambia il ViewModel corrente
```

Esempio MainWindow:

```xml id="s2245t"
<Grid>
    <Grid.ColumnDefinitions>
        <ColumnDefinition Width="220" />
        <ColumnDefinition Width="*" />
    </Grid.ColumnDefinitions>

    <StackPanel Grid.Column="0">
        <Button Content="Dashboard"
                Command="{Binding ShowDashboardCommand}" />
        <Button Content="Clienti"
                Command="{Binding ShowClientiCommand}" />
        <Button Content="Ordini"
                Command="{Binding ShowOrdiniCommand}" />
    </StackPanel>

    <ContentControl Grid.Column="1"
                    Content="{Binding CurrentViewModel}" />
</Grid>
```

MainViewModel:

```csharp id="qxz84i"
public class MainViewModel : ViewModelBase
{
    private object? currentViewModel;

    public MainViewModel(
        DashboardViewModel dashboardViewModel,
        ClientiViewModel clientiViewModel,
        OrdiniViewModel ordiniViewModel)
    {
        ShowDashboardCommand = new RelayCommand(
            () => CurrentViewModel = dashboardViewModel);

        ShowClientiCommand = new RelayCommand(
            () => CurrentViewModel = clientiViewModel);

        ShowOrdiniCommand = new RelayCommand(
            () => CurrentViewModel = ordiniViewModel);

        CurrentViewModel = dashboardViewModel;
    }

    public object? CurrentViewModel
    {
        get => currentViewModel;
        set => SetProperty(ref currentViewModel, value);
    }

    public ICommand ShowDashboardCommand { get; }
    public ICommand ShowClientiCommand { get; }
    public ICommand ShowOrdiniCommand { get; }
}
```

# DataTemplate per collegare ViewModel e View

Per far sì che WPF sappia quale View mostrare per ogni ViewModel, puoi usare `DataTemplate`.

In `App.xaml`:

```xml id="qbfhbd"
<Application.Resources>
    <DataTemplate DataType="{x:Type vm:DashboardViewModel}">
        <views:DashboardView />
    </DataTemplate>

    <DataTemplate DataType="{x:Type vm:ClientiViewModel}">
        <views:ClientiView />
    </DataTemplate>

    <DataTemplate DataType="{x:Type vm:OrdiniViewModel}">
        <views:OrdiniView />
    </DataTemplate>
</Application.Resources>
```

Quando il `ContentControl` riceve un `ClientiViewModel`, WPF usa automaticamente `ClientiView`.

Questo è uno dei modi più puliti per fare navigazione MVVM.

# Integrazione con servizi

Il ViewModel non dovrebbe fare direttamente tutto.

Esempio sbagliato:

```csharp id="yb9r8e"
public async Task CaricaClientiAsync()
{
    using SqlConnection connection = new(...);
    // query SQL direttamente nel ViewModel
}
```

Meglio:

```csharp id="tb7t1x"
public interface IClientiService
{
    Task<IReadOnlyList<ClienteDto>> GetClientiAsync(
        CancellationToken cancellationToken);
}
```

Implementazione:

```csharp id="k66771"
public class ClientiService : IClientiService
{
    private readonly HttpClient httpClient;

    public ClientiService(HttpClient httpClient)
    {
        this.httpClient = httpClient;
    }

    public async Task<IReadOnlyList<ClienteDto>> GetClientiAsync(
        CancellationToken cancellationToken)
    {
        IReadOnlyList<ClienteDto>? clienti =
            await httpClient.GetFromJsonAsync<IReadOnlyList<ClienteDto>>(
                "/api/clienti",
                cancellationToken);

        return clienti ?? [];
    }
}
```

ViewModel:

```csharp id="xgggdi"
public class ClientiViewModel : ViewModelBase
{
    private readonly IClientiService clientiService;

    public ClientiViewModel(IClientiService clientiService)
    {
        this.clientiService = clientiService;
        Clienti = [];
        CaricaCommand = new AsyncRelayCommand(CaricaAsync);
    }

    public ObservableCollection<ClienteDto> Clienti { get; }

    public ICommand CaricaCommand { get; }

    private async Task CaricaAsync()
    {
        IReadOnlyList<ClienteDto> clienti =
            await clientiService.GetClientiAsync(CancellationToken.None);

        Clienti.Clear();

        foreach (ClienteDto cliente in clienti)
        {
            Clienti.Add(cliente);
        }
    }
}
```

Così il ViewModel è testabile e non dipende direttamente da HTTP, SQL o file.

# Dependency Injection in WPF

WPF non nasce con una struttura host come ASP.NET Core, ma puoi usare `Microsoft.Extensions.Hosting`.

Pacchetti:

```bash id="u30jt8"
dotnet add package Microsoft.Extensions.Hosting
dotnet add package Microsoft.Extensions.Http
```

`App.xaml`:

```xml id="cj8glg"
<Application x:Class="MiaApp.App"
             xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
             xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml">
</Application>
```

`App.xaml.cs`:

```csharp id="n0zywq"
public partial class App : Application
{
    private readonly IHost host;

    public App()
    {
        host = Host.CreateDefaultBuilder()
            .ConfigureServices((context, services) =>
            {
                services.AddSingleton<MainWindow>();

                services.AddSingleton<MainViewModel>();
                services.AddSingleton<DashboardViewModel>();
                services.AddTransient<ClientiViewModel>();
                services.AddTransient<OrdiniViewModel>();

                services.AddTransient<DashboardView>();
                services.AddTransient<ClientiView>();
                services.AddTransient<OrdiniView>();

                services.AddHttpClient<IClientiService, ClientiService>(
                    client =>
                    {
                        client.BaseAddress =
                            new Uri("https://localhost:5001");
                    });
            })
            .Build();
    }

    protected override async void OnStartup(
        StartupEventArgs e)
    {
        await host.StartAsync();

        MainWindow mainWindow =
            host.Services.GetRequiredService<MainWindow>();

        mainWindow.Show();

        base.OnStartup(e);
    }

    protected override async void OnExit(
        ExitEventArgs e)
    {
        await host.StopAsync();
        host.Dispose();

        base.OnExit(e);
    }
}
```

`MainWindow.xaml.cs`:

```csharp id="owcvxo"
public partial class MainWindow : Window
{
    public MainWindow(MainViewModel viewModel)
    {
        InitializeComponent();
        DataContext = viewModel;
    }
}
```

Questo avvicina una app WPF moderna al modello usato in ASP.NET Core, console app e worker service.

# Validazione nel ViewModel

Esempio con `INotifyDataErrorInfo` semplificato:

```csharp id="lqxo5c"
public class ClienteEditViewModel :
    ViewModelBase,
    INotifyDataErrorInfo
{
    private readonly Dictionary<string, List<string>> errors = [];

    private string nome = "";

    public string Nome
    {
        get => nome;
        set
        {
            if (SetProperty(ref nome, value))
            {
                ValidateNome();
            }
        }
    }

    public bool HasErrors => errors.Count > 0;

    public event EventHandler<DataErrorsChangedEventArgs>? ErrorsChanged;

    public IEnumerable GetErrors(string? propertyName)
    {
        if (propertyName is null)
        {
            return Enumerable.Empty<string>();
        }

        return errors.TryGetValue(propertyName, out List<string>? value)
            ? value
            : Enumerable.Empty<string>();
    }

    private void ValidateNome()
    {
        errors.Remove(nameof(Nome));

        if (string.IsNullOrWhiteSpace(Nome))
        {
            errors[nameof(Nome)] =
            [
                "Il nome è obbligatorio."
            ];
        }

        ErrorsChanged?.Invoke(
            this,
            new DataErrorsChangedEventArgs(nameof(Nome)));
    }
}
```

XAML:

```xml id="ctph7b"
<TextBox Text="{Binding Nome,
                        UpdateSourceTrigger=PropertyChanged,
                        ValidatesOnNotifyDataErrors=True}" />
```

Regola pratica:

```text id="sz838t"
La validazione UI aiuta l’utente.
Ma le regole di dominio devono essere validate anche nel backend
o nel domain model.
```

```text id="c68n55"
  API client
  File system
  Database locale
  Servizi esterni
```

Questo evita che WPF diventi il contenitore di tutta la logica.

---

# Errori comuni

## Troppa logica nel code-behind

Sospetto:

```text id="wpj0vs"
Button_Click contiene query SQL,
validazione business,
salvataggio e navigazione.
```

Meglio spostare la logica in ViewModel e servizi.

---

## ViewModel che conosce controlli WPF

Sbagliato:

```csharp id="qz8gvl"
public TextBox NomeTextBox { get; set; }
```

Il ViewModel non dovrebbe dipendere da `TextBox`, `Button`, `Window`.

---

## Non usare INotifyPropertyChanged

Se il ViewModel cambia ma la UI non si aggiorna, spesso manca notifica.

---

## Usare List<T> per liste dinamiche

Per liste che cambiano a runtime, usa `ObservableCollection<T>`.

---

## Modificare ObservableCollection da thread non UI

Causa errori di threading.

---

## Comandi async gestiti male

Un comando asincrono deve prevenire doppi click e gestire stato di caricamento/errori.

---

## Stili duplicati ovunque

Meglio usare ResourceDictionary e stili centralizzati.

---

## Navigazione fatta creando View ovunque

Meglio una strategia coerente: navigation service, CurrentViewModel, DataTemplate, DI.

---

# Linee guida pratiche

```text id="rdf96b"
Usa XAML
  per layout, controlli, binding e stili.

Usa Window
  per la finestra principale o dialoghi principali.

Usa UserControl
  per dividere schermate e componenti.

Usa Binding
  per collegare View e ViewModel.

Imposta bene il DataContext
  preferibilmente tramite Dependency Injection.

Implementa INotifyPropertyChanged
  per aggiornare la UI quando cambia lo stato.

Usa ICommand
  per azioni invocate dalla UI.

Usa ObservableCollection
  per liste che devono aggiornare la UI.

Usa ResourceDictionary
  per colori, stili e risorse comuni.

Usa MVVM
  per evitare code-behind pieno di logica.

Usa servizi
  per API, database, file system e integrazioni.

Usa Dependency Injection
  per costruire View, ViewModel e servizi in modo testabile.

Mantieni il dominio fuori dalla UI
  WPF è il livello di presentazione, non tutta l’applicazione.
```

# Riepilogo

In questo capitolo abbiamo fatto una sintesi di WPF e MVVM.

Abbiamo visto:

```text id="tf5jzi"
XAML
Window
UserControl
Binding
DataContext
INotifyPropertyChanged
ICommand
ObservableCollection
ResourceDictionary
Styling
Navigazione tra view
Integrazione con servizi e DI
```

La distinzione principale è questa:

```text id="n7gk3y"
View
  definisce la UI in XAML.

ViewModel
  espone stato, proprietà e comandi.

Model/servizi
  gestiscono dati, regole e integrazioni.
```

La regola principale è questa:

```text id="icaz2w"
Una buona app WPF non è una collezione di eventi Button_Click.
È una composizione ordinata di View, ViewModel, servizi,
binding, comandi, risorse e validazioni.
```

Questo capitolo deve restare una sintesi: per approfondire davvero WPF servirebbe un percorso dedicato su XAML, binding avanzato, dependency properties, template, risorse, controlli, performance, threading, MVVM, validazione, navigazione e distribuzione.

# 12.03 — .NET MAUI e sviluppo cross-platform

Nel capitolo precedente abbiamo sintetizzato WPF e MVVM, cioè un modello molto forte per applicazioni desktop Windows.

Ora allarghiamo lo sguardo allo sviluppo **cross-platform** con **.NET MAUI**.

L’obiettivo non è trasformare il libro in un manuale completo su MAUI, ma dare al lettore una mappa realistica:

```text id="t6mil4"
che cos’è .NET MAUI
quando ha senso usarlo
come si struttura una UI
come funziona la navigazione
come si integrano API e servizi
quali sono i limiti pratici
```

Il punto centrale è questo:

```text id="d81q0l"
.NET MAUI è utile quando vuoi costruire app native mobile e desktop
con C# e XAML, condividendo una parte importante del codice.

Non è semplicemente “WPF multipiattaforma”.
È un framework diverso, con obiettivi diversi e vincoli diversi.
```

Microsoft definisce .NET MAUI come un framework cross-platform per creare app native mobile e desktop con C# e XAML, da una singola codebase, per Android, iOS, macOS e Windows. È l’evoluzione di Xamarin.Forms, estesa anche a scenari desktop. ([Microsoft Learn][1])

# Quando usare .NET MAUI

MAUI ha senso quando:

```text id="i7ob1w"

vuoi usare C# e .NET
vuoi condividere codice tra mobile e desktop
hai già backend .NET

```

Esempi:

```text id="d3iem0"

```

---

# Quando non è la scelta migliore

MAUI può non essere la scelta migliore quando:

```text id="c6a0jj"
vuoi una app desktop Windows enterprise molto complessa
vuoi un’esperienza WPF avanzata con griglie, template e binding sofisticati
devi supportare Linux desktop come target principale
vuoi una UI altamente personalizzata e identica al pixel su ogni piattaforma
il team non ha esperienza mobile
hai bisogno di molte integrazioni native particolari
```

Per desktop Windows complesso, WPF può essere più pragmatico.

Per desktop cross-platform Windows/macOS/Linux, Avalonia può essere più adatta.

Per app puramente web, ASP.NET Core/Blazor può essere più semplice.

# Piattaforme e strumenti

.NET MAUI supporta Android, iOS, macOS tramite Mac Catalyst e Windows. Su Windows, le app Android, iOS e Windows possono essere compilate con Visual Studio; per lo sviluppo iOS è richiesto un Mac in rete. Questo ha implicazioni pratiche:

```text id="exodkh"

```

Regola pratica:


# BindingContext

In WPF si parla di `DataContext`.

In MAUI il concetto equivalente è `BindingContext`.

Esempio code-behind semplice:

```csharp id="zbxz9f"
public partial class ClientiPage : ContentPage
{
    public ClientiPage()
    {
        InitializeComponent();
        BindingContext = new ClientiViewModel();
    }
}
```

In app reali è meglio usare Dependency Injection:

```csharp id="o710me"
public partial class ClientiPage : ContentPage
{
    public ClientiPage(ClientiViewModel viewModel)
    {
        InitializeComponent();
        BindingContext = viewModel;
    }
}
```

# MVVM in MAUI

MAUI funziona molto bene con MVVM.

ViewModel:

```csharp id="ttf8eo"
public class ClienteViewModel : INotifyPropertyChanged
{
    private string nome = "";

    public string Nome
    {
        get => nome;
        set
        {
            if (nome == value)
            {
                return;
            }

            nome = value;
            OnPropertyChanged(nameof(Nome));
        }
    }

    public ICommand SalvaCommand { get; }

    public ClienteViewModel()
    {
        SalvaCommand = new Command(Salva);
    }

    private void Salva()
    {
        // logica di salvataggio
    }

    public event PropertyChangedEventHandler? PropertyChanged;

    private void OnPropertyChanged(string propertyName)
    {
        PropertyChanged?.Invoke(
            this,
            new PropertyChangedEventArgs(propertyName));
    }
}
```

In progetti reali conviene usare una base `ViewModelBase` o librerie come CommunityToolkit.Mvvm, ma il concetto rimane lo stesso:

```text id="ya0spw"
la pagina mostra
il ViewModel espone stato e comandi
i servizi fanno il lavoro esterno
```

# Shell

**Shell** è il modello di navigazione e struttura applicativa più comune in MAUI.

Permette di definire:

```text id="zhz7cw"
struttura dell’app
rotte
Flyout menu
tab
navigazione URI-based
gerarchie di pagine
```

La documentazione Microsoft spiega che MAUI Shell include un’esperienza di navigazione URI-based basata su route, permettendo di navigare verso pagine senza seguire necessariamente una gerarchia rigida e con supporto alla navigazione all’indietro. ([Microsoft Learn][3])

Esempio `AppShell.xaml`:

```xml id="kwyhc9"
<Shell
    x:Class="MiaApp.AppShell"
    xmlns="http://schemas.microsoft.com/dotnet/2021/maui"
    xmlns:x="http://schemas.microsoft.com/winfx/2009/xaml"
    xmlns:pages="clr-namespace:MiaApp.Pages">

    <FlyoutItem Title="Home">
        <ShellContent
            Title="Home"
            ContentTemplate="{DataTemplate pages:HomePage}"
            Route="home" />
    </FlyoutItem>

    <FlyoutItem Title="Ordini">
        <ShellContent
            Title="Ordini"
            ContentTemplate="{DataTemplate pages:OrdiniPage}"
            Route="ordini" />
    </FlyoutItem>

</Shell>
```

# Navigazione

Con Shell puoi navigare usando route.

Esempio:

```csharp id="el5w74"
await Shell.Current.GoToAsync("ordini");
```

Navigazione con parametro:

```csharp id="zf3830"
await Shell.Current.GoToAsync(
    $"dettaglio-ordine?id={ordine.Id}");
```

Registrazione route:

```csharp id="xtzhhl"
Routing.RegisterRoute(
    "dettaglio-ordine",
    typeof(DettaglioOrdinePage));
```

Ricezione parametro con query property:

```csharp id="xlkimr"
[QueryProperty(nameof(OrdineId), "id")]
public partial class DettaglioOrdinePage : ContentPage
{
    public string OrdineId { get; set; } = "";
}
```

In app strutturate con MVVM, spesso si preferisce passare la navigazione attraverso un servizio:

```csharp id="xs2q0z"
public interface INavigationService
{
    Task GoToOrdineAsync(int ordineId);
}
```

per evitare che i ViewModel dipendano direttamente da `Shell.Current`.

# Integrazione con API

Molte app MAUI sono client di API ASP.NET Core.

Schema:

```text id="n58ips"
MAUI App
  ↓ HTTPS
ASP.NET Core API
  ↓
Database / servizi
```

Servizio API:

```csharp id="l3f2q4"
public interface IOrdiniApiClient
{
    Task<IReadOnlyList<OrdineDto>> GetOrdiniAsync(
        CancellationToken cancellationToken);
}
```

Implementazione:

```csharp id="r0jzmr"
public class OrdiniApiClient : IOrdiniApiClient
{
    private readonly HttpClient httpClient;

    public OrdiniApiClient(HttpClient httpClient)
    {
        this.httpClient = httpClient;
    }

    public async Task<IReadOnlyList<OrdineDto>> GetOrdiniAsync(
        CancellationToken cancellationToken)
    {
        IReadOnlyList<OrdineDto>? ordini =
            await httpClient.GetFromJsonAsync<IReadOnlyList<OrdineDto>>(
                "/api/ordini",
                cancellationToken);

        return ordini ?? [];
    }
}
```

Registrazione in `MauiProgram.cs`:

```csharp id="ikzkur"
builder.Services.AddHttpClient<IOrdiniApiClient, OrdiniApiClient>(
    client =>
    {
        client.BaseAddress =
            new Uri("https://api.example.com");
    });
```

ViewModel:

```csharp id="zjufps"
public class OrdiniViewModel : ViewModelBase
{
    private readonly IOrdiniApiClient ordiniApiClient;

    public ObservableCollection<OrdineDto> Ordini { get; } = [];

    public ICommand CaricaCommand { get; }

    public OrdiniViewModel(IOrdiniApiClient ordiniApiClient)
    {
        this.ordiniApiClient = ordiniApiClient;
        CaricaCommand = new AsyncRelayCommand(CaricaAsync);
    }

    private async Task CaricaAsync()
    {
        IReadOnlyList<OrdineDto> ordini =
            await ordiniApiClient.GetOrdiniAsync(
                CancellationToken.None);

        Ordini.Clear();

        foreach (OrdineDto ordine in ordini)
        {
            Ordini.Add(ordine);
        }
    }
}
```

# Dependency Injection in MAUI

MAUI usa un modello di configurazione in `MauiProgram.cs`.

Esempio:

```csharp id="y3t3hi"
public static class MauiProgram
{
    public static MauiApp CreateMauiApp()
    {
        MauiAppBuilder builder = MauiApp.CreateBuilder();

        builder
            .UseMauiApp<App>()
            .ConfigureFonts(fonts =>
            {
                fonts.AddFont("OpenSans-Regular.ttf", "OpenSansRegular");
            });

        builder.Services.AddSingleton<HomePage>();
        builder.Services.AddSingleton<HomeViewModel>();

        builder.Services.AddTransient<OrdiniPage>();
        builder.Services.AddTransient<OrdiniViewModel>();

        builder.Services.AddHttpClient<IOrdiniApiClient, OrdiniApiClient>(
            client =>
            {
                client.BaseAddress =
                    new Uri("https://api.example.com");
            });

        return builder.Build();
    }
}
```

Questo consente di usare lo stesso stile moderno già visto in ASP.NET Core, Worker Service e WPF con hosting/DI.

# Limiti pratici

MAUI è potente, ma va scelto con consapevolezza.

Ogni piattaforma ha particolarità:

```text id="trzn4e"

lifecycle
```

## 2. Testing su dispositivi reali

Gli emulatori non bastano.

Serve testare su:

```text id="q2d22y"
Android reali
iPhone/iPad reali
Windows

diverse dimensioni schermo

```

---

## 3. UI desktop complessa

MAUI supporta desktop, ma se il tuo obiettivo principale è un gestionale Windows molto complesso, WPF può essere più adatto.

Esempi dove WPF può vincere:

```text id="vuzebl"

```

---

## 4. Linux desktop

MAUI non è la scelta naturale per Linux desktop.

Se Linux desktop è un requisito forte, valuta Avalonia.

# Linee guida pratiche

```text id="xsqogf"
Usa .NET MAUI
  quando mobile e cross-platform sono requisiti reali.

Non usarlo solo perché “così faccio tutto con una codebase”
  se il tuo target principale è desktop Windows complesso.

Usa XAML e MVVM
  per separare UI, stato e logica.

Usa Shell
  per struttura e navigazione dell’app.

Usa CollectionView
  per liste moderne e performanti.

Isola codice platform-specific
  in servizi dedicati o cartelle Platforms.

Usa HttpClient e servizi
  per integrare API ASP.NET Core.

Gestisci bene autenticazione e token
  soprattutto su mobile.

Prevedi rete instabile e offline
  se l’app è usata fuori ufficio.

Testa su dispositivi reali
  non solo emulatori.

Valuta HybridWebView o Blazor Hybrid
  se hai UI web da riusare o componenti Razor utili.

Considera i limiti pratici
  desktop complesso, Linux, SDK nativi, store, lifecycle.
```

La distinzione principale è questa:

```text id="nyjwqc"
WPF
  desktop Windows enterprise.

.NET MAUI
  app native mobile e desktop cross-platform,
  soprattutto quando Android/iOS sono importanti.

Avalonia
  desktop cross-platform forte, incluso Linux.
```

La regola principale è questa:


# 02.09 — Generics e tipi riutilizzabili

Nei capitoli precedenti abbiamo imparato a creare classi, record, struct, enum, interfacce e gerarchie polimorfiche. Ora affrontiamo uno degli strumenti più importanti del C# moderno: i **generics**.

I generics permettono di scrivere codice riutilizzabile senza perdere la sicurezza dei tipi.

Sono ovunque nell’ecosistema .NET:

```csharp id="w7v2pz"
List<string>
Dictionary<int, string>
IEnumerable<Cliente>
Task<Documento>
Nullable<int>
IRepository<Cliente>
Result<Ordine>
```

Se non si capiscono i generics, diventa difficile capire:

* collezioni;
* LINQ;
* async/await;
* dependency injection;
* repository;
* DTO;
* result pattern;
* interfacce riutilizzabili;
* molte API di .NET.

Il punto centrale è questo:

> un tipo generico è un tipo scritto una volta sola, ma utilizzabile con tipi diversi.

---

# Il problema che risolvono i generics

Immaginiamo di voler creare una classe che contiene un valore.

Per un intero potremmo scrivere:

```csharp id="lc0pey"
public class ContenitoreInt
{
    public int Valore { get; set; }
}
```

Per una stringa:

```csharp id="q458rl"
public class ContenitoreString
{
    public string Valore { get; set; } = "";
}
```

Per un cliente:

```csharp id="e25ybi"
public class ContenitoreCliente
{
    public Cliente? Valore { get; set; }
}
```

Questo approccio non scala.

La struttura è sempre la stessa. Cambia solo il tipo del valore contenuto.

Con i generics possiamo scrivere una sola classe:

```csharp id="qr2ir7"
public class Contenitore<T>
{
    public T? Valore { get; set; }
}
```

Ora possiamo usarla con tipi diversi:

```csharp id="o7hf1z"
Contenitore<int> numero = new Contenitore<int>();
Contenitore<string> testo = new Contenitore<string>();
Contenitore<Cliente> cliente = new Contenitore<Cliente>();
```

`T` è un **parametro di tipo**.

Non rappresenta un valore, ma un tipo che verrà specificato quando useremo la classe.

---

# Introduzione ai generics

Un generic è un tipo o un metodo che contiene uno o più parametri di tipo.

Esempio:

```csharp id="cknjzm"
public class Scatola<T>
{
    public T Contenuto { get; }

    public Scatola(T contenuto)
    {
        Contenuto = contenuto;
    }
}
```

Qui `T` è un segnaposto.

Quando scriviamo:

```csharp id="t436z3"
Scatola<string> scatolaTesto = new Scatola<string>("Ciao");
```

`T` diventa `string`.

Quando scriviamo:

```csharp id="dft3le"
Scatola<int> scatolaNumero = new Scatola<int>(42);
```

`T` diventa `int`.

---

## Perché non usare `object`?

Prima dei generics si poteva scrivere una classe basata su `object`.

```csharp id="1c7r6c"
public class Scatola
{
    public object? Contenuto { get; set; }
}
```

Uso:

```csharp id="8ksxy0"
Scatola scatola = new Scatola();

scatola.Contenuto = "Ciao";

string testo = (string)scatola.Contenuto;
```

Il problema è che perdiamo sicurezza di tipo.

Questo codice compila:

```csharp id="2oc0ji"
scatola.Contenuto = 123;
```

E poi può fallire a runtime:

```csharp id="q4ws2f"
string testo = (string)scatola.Contenuto; // errore a runtime
```

Con i generics:

```csharp id="i7l2ph"
Scatola<string> scatola = new Scatola<string>("Ciao");

string testo = scatola.Contenuto;
```

Il compilatore sa che `Contenuto` è una stringa.

Questo non compila:

```csharp id="du4dj5"
Scatola<string> scatola = new Scatola<string>("Ciao");

scatola.Contenuto = 123; // errore
```

I generics evitano molti cast e molti errori a runtime.

La specifica C# descrive i tipi generici come tipi con parametri di tipo; i parametri di tipo possono rappresentare sia value type sia reference type. La stessa specifica distingue anche value type e reference type: una variabile value type contiene direttamente i dati, mentre una variabile reference type contiene un riferimento a un oggetto. ([Microsoft Learn][1])

# Classi generiche

Una **classe generica** è una classe che usa uno o più parametri di tipo.

Esempio:

```csharp id="etef30"
public class RisultatoOperazione<T>
{
    public bool Successo { get; }
    public T? Valore { get; }
    public string? Errore { get; }

    private RisultatoOperazione(bool successo, T? valore, string? errore)
    {
        Successo = successo;
        Valore = valore;
        Errore = errore;
    }

    public static RisultatoOperazione<T> Ok(T valore)
    {
        return new RisultatoOperazione<T>(true, valore, null);
    }

    public static RisultatoOperazione<T> Fallito(string errore)
    {
        return new RisultatoOperazione<T>(false, default, errore);
    }
}
```

Uso:

```csharp id="qpl6fn"
RisultatoOperazione<int> risultatoNumero = RisultatoOperazione<int>.Ok(42);

RisultatoOperazione<string> risultatoTesto = RisultatoOperazione<string>.Ok("Operazione completata");
```

Questa classe può rappresentare il risultato di un’operazione che restituisce un valore di qualunque tipo.

---

## Esempio con Cliente

```csharp id="wnm3v2"
public class Cliente
{
    public int Id { get; }
    public string Nome { get; }

    public Cliente(int id, string nome)
    {
        Id = id;
        Nome = nome;
    }
}
```

Uso:

```csharp id="j52w86"
Cliente cliente = new Cliente(1, "Emilie");

RisultatoOperazione<Cliente> risultato = RisultatoOperazione<Cliente>.Ok(cliente);

if (risultato.Successo)
{
    Console.WriteLine(risultato.Valore?.Nome);
}
```

## Più parametri di tipo

Una classe generica può avere più parametri.

```csharp id="xy35la"
    public TPrimo Primo { get; }
    public TSecondo Secondo { get; }

    public Coppia(TPrimo primo, TSecondo secondo)
    {
        Primo = primo;
        Secondo = secondo;
    }
}
```

Uso:

```csharp id="9tba9l"

Console.WriteLine(coppia.Primo);
Console.WriteLine(coppia.Secondo);
```

Output:

```text id="74kmz5"
1
Uno
```

# Metodi generici

Un metodo generico è un metodo che dichiara uno o più parametri di tipo.

Esempio:

```csharp id="hpt2l3"
T Identita<T>(T valore)
{
    return valore;
}
```

Uso:

```csharp id="hwda3k"
int numero = Identita<int>(42);
string testo = Identita<string>("Ciao");
```

Spesso il compilatore riesce a dedurre il tipo:

```csharp id="4wikbe"
int numero = Identita(42);
string testo = Identita("Ciao");
```

---

## Metodo generico utile: scambio di valori

```csharp id="gyurw1"
void Scambia<T>(ref T primo, ref T secondo)
{
    T temporaneo = primo;
    primo = secondo;
    secondo = temporaneo;
}
```

Uso con interi:

```csharp id="l5vrde"
int a = 10;
int b = 20;

Scambia(ref a, ref b);

Console.WriteLine(a);
Console.WriteLine(b);
```

Output:

```text id="rxcdvs"
20
10
```

Uso con stringhe:

```csharp id="1fp6x5"
string nome1 = "Emilie";
string nome2 = "Mario";

Scambia(ref nome1, ref nome2);

Console.WriteLine(nome1);
Console.WriteLine(nome2);
```

Output:

```text id="jyay94"
Mario
Emilie
```

Il metodo è scritto una volta sola, ma funziona con tipi diversi.

---

## Metodo generico con risultato

```csharp id="dqi3fo"
T? PrimoODefault<T>(IEnumerable<T> elementi)
{
    foreach (T elemento in elementi)
    {
        return elemento;
    }

    return default;
}
```

Uso:

```csharp id="29wo10"
List<int> numeri = new List<int> { 10, 20, 30 };

int primo = PrimoODefault(numeri);

Console.WriteLine(primo);
```

Output:

```text id="wv4ypw"
10
```

Con stringhe:

```csharp id="zdp06n"
List<string> nomi = new List<string> { "Emilie", "Mario" };

string? primoNome = PrimoODefault(nomi);

Console.WriteLine(primoNome);
```

Output:

```text id="2ygpoe"
Emilie
```

# Interfacce generiche

Un’interfaccia generica definisce un contratto che dipende da un tipo.

Esempio:

```csharp id="d4o4ac"
public interface IRepository<T>
{
    T? TrovaPerId(int id);
    void Salva(T elemento);
}
```

Questa interfaccia dice:

> un repository lavora con elementi di tipo `T`.

Implementazione per `Cliente`:

```csharp id="s4puv4"
public class ClienteRepository : IRepository<Cliente>
{
    private readonly List<Cliente> clienti = new();

    public Cliente? TrovaPerId(int id)
    {
        return clienti.FirstOrDefault(cliente => cliente.Id == id);
    }

    public void Salva(Cliente elemento)
    {
        clienti.Add(elemento);
    }
}
```

Uso:

```csharp id="a89bjo"
IRepository<Cliente> repository = new ClienteRepository();

repository.Salva(new Cliente(1, "Emilie"));

Cliente? cliente = repository.TrovaPerId(1);

Console.WriteLine(cliente?.Nome);
```

# Vincoli generici

A volte un parametro di tipo `T` è troppo generico.

Esempio:

```csharp id="xs8ss2"
void StampaNome<T>(T elemento)
{
    Console.WriteLine(elemento.Nome); // errore
}
```

Il compilatore non sa se `T` ha una proprietà `Nome`.

I **vincoli** permettono di dire al compilatore quali caratteristiche deve avere `T`.

La sintassi usa `where`.

---

# `where T : class`

Il vincolo:

```csharp id="hcnnz9"
where T : class
```

dice che `T` deve essere un reference type.

Esempio:

```csharp id="nltqpk"
public class Cache<T> where T : class
{
    private T? elemento;

    public void Imposta(T valore)
    {
        elemento = valore;
    }

    public T? Ottieni()
    {
        return elemento;
    }
}
```

Uso valido:

```csharp id="x151xz"
Cache<Cliente> cacheCliente = new Cache<Cliente>();
Cache<string> cacheStringa = new Cache<string>();
```

Uso non valido:

```csharp id="zfnld8"
Cache<int> cacheNumero = new Cache<int>(); // errore
```

`int` è un value type.

---

# `where T : struct`

Il vincolo:

```csharp id="tt0cq8"
where T : struct
```

dice che `T` deve essere un value type non nullable.

Esempio:

```csharp id="vo28qz"
public class Intervallo<T> where T : struct
{
    public T Da { get; }
    public T A { get; }

    public Intervallo(T da, T a)
    {
        Da = da;
        A = a;
    }
}
```

Uso valido:

```csharp id="5qdx8v"
Intervallo<int> intervalloNumeri = new Intervallo<int>(1, 10);
Intervallo<DateTime> intervalloDate = new Intervallo<DateTime>(DateTime.Today, DateTime.Today.AddDays(7));
```

Uso non valido:

```csharp id="hvszne"
Intervallo<string> intervalloTesto = new Intervallo<string>("A", "Z"); // errore
```

`string` è un reference type.

---

# `where T : new()`

Il vincolo:

```csharp id="ozedjx"
where T : new()
```

dice che `T` deve avere un costruttore pubblico senza parametri.

Esempio:

```csharp id="x0uxnv"
T CreaIstanza<T>() where T : new()
{
    return new T();
}
```

Uso:

```csharp id="pifh28"
public class Cliente
{
    public string Nome { get; set; } = "";
}

Cliente cliente = CreaIstanza<Cliente>();
```

Il compilatore permette `new T()` solo se sa che `T` ha un costruttore senza parametri.

---

## Attenzione a `new()`

Il vincolo `new()` può essere comodo, ma non sempre è una buona scelta progettuale.

Se un oggetto ha bisogno di parametri obbligatori per essere valido, un costruttore vuoto può portare a oggetti inconsistenti.

Esempio sospetto:

```csharp id="7f1p8n"
public class Cliente
{
    public string Nome { get; set; } = "";
}
```

Permette:

```csharp id="n8u1bn"
Cliente cliente = new Cliente();
```

ma il cliente nasce senza nome.

Nel domain model, spesso è meglio usare costruttori espliciti. Nei DTO, invece, `new()` può essere più accettabile.

---

# `where T : interface`

Più precisamente, si scrive il nome dell’interfaccia come vincolo.

Esempio:

```csharp id="hqg1qz"
public interface IIdentificabile
{
    int Id { get; }
}
```

Metodo generico:

```csharp id="pkhyxa"
T? TrovaPerId<T>(IEnumerable<T> elementi, int id)
    where T : IIdentificabile
{
    foreach (T elemento in elementi)
    {
        if (elemento.Id == id)
        {
            return elemento;
        }
    }

    return default;
}
```

Ora il compilatore sa che `T` ha una proprietà `Id`.

Implementazione:

```csharp id="3coyvx"
public class Cliente : IIdentificabile
{
    public int Id { get; }
    public string Nome { get; }

    public Cliente(int id, string nome)
    {
        Id = id;
        Nome = nome;
    }
}
```

Uso:

```csharp id="mlvup6"
List<Cliente> clienti = new List<Cliente>
{
    new Cliente(1, "Emilie"),
    new Cliente(2, "Mario")
};

Cliente? cliente = TrovaPerId(clienti, 2);

Console.WriteLine(cliente?.Nome);
```

Output:

```text id="wyx2ja"
Mario
```

---

## Più vincoli insieme

Possiamo combinare più vincoli.

```csharp id="nvf99v"
public class Repository<T>
    where T : class, IIdentificabile, new()
{
    public T Crea()
    {
        return new T();
    }
}
```

Qui `T` deve essere:

```text id="s9qsv5"
un reference type
implementare IIdentificabile
avere un costruttore pubblico senza parametri
```

La documentazione C# elenca i vincoli generici tra cui `class`, `struct`, `new()` e vincoli basati su classi o interfacce; il vincolo `new()`, quando usato con altri vincoli, deve essere specificato per ultimo. ([Microsoft Learn][1])

# Generics e collezioni

Le collezioni generiche sono probabilmente il primo punto in cui uno sviluppatore incontra davvero i generics.

Esempio:

```csharp id="nm0wk3"
List<string> nomi = new List<string>();

nomi.Add("Emilie");
nomi.Add("Mario");
```

`List<string>` significa:

> lista di stringhe.

Il compilatore impedisce di inserire un tipo sbagliato:

```csharp id="zol6bz"
nomi.Add(123); // errore
```

---

## List<T>

```csharp id="c1hb3q"
List<int> numeri = new List<int>
{
    1,
    2,
    3
};

foreach (int numero in numeri)
{
    Console.WriteLine(numero);
}
```

`T` è `int`.

---

## Dictionary<TKey, TValue>

`Dictionary<TKey, TValue>` ha due parametri di tipo:

```csharp id="yf5elj"
Dictionary<int, string> clienti = new Dictionary<int, string>();

clienti[1] = "Emilie";
clienti[2] = "Mario";

Console.WriteLine(clienti[1]);
```

Qui:

```text id="j8tvp4"
TKey
  int

TValue
  string
```

Il dizionario associa chiavi intere a valori stringa.

---

## IEnumerable<T>

`IEnumerable<T>` rappresenta una sequenza di elementi di tipo `T`.

```csharp id="v7smwz"
IEnumerable<string> nomi = new List<string>
{
    "Emilie",
    "Mario",
    "Anna"
};
```

Possiamo scorrerla:

```csharp id="tnfmgl"
foreach (string nome in nomi)
{
    Console.WriteLine(nome);
}
```

Molti metodi LINQ lavorano su `IEnumerable<T>`:

```csharp id="cwnzg6"
IEnumerable<string> nomiConA = nomi.Where(nome => nome.StartsWith("A"));
```

---

# `Nullable<T>`

Nel capitolo sui tipi abbiamo visto i nullable value types:

```csharp id="sjwdzj"
int? eta = null;
```

Questa sintassi è una forma abbreviata per:

```csharp id="04vjdd"
Nullable<int> eta = null;
```

`Nullable<T>` è un tipo generico.

Esempi:

```csharp id="eqd6wj"
Nullable<int> numero = 10;
Nullable<DateTime> data = null;
Nullable<decimal> importo = 19.90m;
```

Normalmente si usa la sintassi abbreviata:

```csharp id="y8ewkc"
int? numero = 10;
DateTime? data = null;
decimal? importo = 19.90m;
```

---

## Perché Nullable<T> richiede value type

`Nullable<T>` funziona solo con value type.

Questo è valido:

```csharp id="bd78ai"
Nullable<int> numero = null;
```

Questo non è valido:

```csharp id="cbg3ew"
Nullable<string> testo = null; // errore
```

Per i reference type si usa la sintassi nullable reference type:

```csharp id="q89lig"
string? testo = null;
```

Sono concetti collegati, ma non identici.

```text id="n3cdtg"
int?
  Nullable<int>, cioè nullable value type.

string?
  nullable reference type, gestito dal compilatore con analisi dei null.
```

# Covarianza e controvarianza

Covarianza e controvarianza sono concetti avanzati, ma è utile averne una prima idea.

Riguardano la possibilità di usare tipi generici con relazioni di ereditarietà.

Partiamo da queste classi:

```csharp id="td1kma"
public class Animale
{
    public string Nome { get; set; } = "";
}

public class Cane : Animale
{
}
```

Ogni `Cane` è un `Animale`.

Quindi questo è valido:

```csharp id="0q3nfo"
Cane cane = new Cane();
Animale animale = cane;
```

Ma con tipi generici la questione è più delicata.

---

## Il problema

Potremmo pensare che questo sia valido:

```csharp id="f3pbdo"
List<Cane> cani = new List<Cane>();

List<Animale> animali = cani; // errore
```

Ma non è consentito.

Perché?

Se fosse consentito, potremmo fare:

```csharp id="o24uh3"
animali.Add(new Animale());
```

Ma `animali` in realtà punterebbe a una `List<Cane>`. Avremmo inserito un `Animale` generico in una lista che dovrebbe contenere solo cani.

Per proteggere la sicurezza dei tipi, `List<Cane>` non è assegnabile a `List<Animale>`.

---

## Covarianza

La covarianza permette una conversione generica quando il tipo viene solo prodotto, cioè restituito in uscita.

Esempio con `IEnumerable<T>`:

```csharp id="o2rduv"
IEnumerable<Cane> cani = new List<Cane>
{
    new Cane { Nome = "Fido" }
};

IEnumerable<Animale> animali = cani;
```

Questo è consentito.

Perché `IEnumerable<T>` permette solo di leggere elementi. Se leggiamo da una sequenza di cani, possiamo considerarli animali.

Non possiamo aggiungere un `Animale` dentro `IEnumerable<Animale>`, perché `IEnumerable<T>` non ha un metodo `Add`.

---

## Controvarianza

La controvarianza funziona nella direzione opposta, quando il tipo viene consumato in ingresso.

Esempio con `Action<T>`:

```csharp id="xb44i6"
void StampaAnimale(Animale animale)
{
    Console.WriteLine(animale.Nome);
}

Action<Animale> azioneAnimale = StampaAnimale;

Action<Cane> azioneCane = azioneAnimale;
```

Questo è valido perché un metodo capace di gestire qualsiasi `Animale` è sicuramente capace di gestire un `Cane`.

---

## `out` e `in` nei generics

Nelle interfacce e nei delegate generici, C# usa:

```text id="vapldn"
out
  per covarianza.

in
  per controvarianza.
```

Esempio semplificato:

```csharp id="h4y1vn"
public interface IProduttore<out T>
{
    T Produci();
}
```

`out T` indica che `T` viene prodotto in uscita.

Esempio:

```csharp id="b77v76"
public interface IConsumatore<in T>
{
    void Consuma(T elemento);
}
```

`in T` indica che `T` viene consumato in ingresso.

La documentazione Microsoft spiega che la covarianza consente di usare un tipo più derivato rispetto a quello specificato, mentre la controvarianza consente di usare un tipo più generale; C# supporta questi concetti per interfacce e delegate generici tramite `out` e `in`. ([learn.microsoft.com](https://learn.microsoft.com/en-us/dotnet/csharp/programming-guide/concepts/covariance-contravariance/?utm_source=chatgpt.com))

---

## Regola pratica

Per ora basta ricordare:

```text id="ptb615"
List<Cane> non è List<Animale>.

IEnumerable<Cane> può essere usato come IEnumerable<Animale>.

out T
  tipo prodotto in uscita.

in T
  tipo consumato in ingresso.
```

Covarianza e controvarianza diventeranno più chiare quando useremo delegate, LINQ e interfacce come `IEnumerable<T>`.

# Generics e repository

Un esempio classico nei progetti enterprise è il repository generico.

Definiamo un contratto:

```csharp id="q4isw0"
public interface IEntity
{
    int Id { get; }
}
```

Repository:

```csharp id="l68fd2"
public interface IRepository<T>
    where T : class, IEntity
{
    T? TrovaPerId(int id);
    void Aggiungi(T entity);
    IEnumerable<T> Tutti();
}
```

Implementazione in memoria:

```csharp id="e6idq8"
public class InMemoryRepository<T> : IRepository<T>
    where T : class, IEntity
{
    private readonly List<T> elementi = new();

    public T? TrovaPerId(int id)
    {
        return elementi.FirstOrDefault(elemento => elemento.Id == id);
    }

    public void Aggiungi(T entity)
    {
        elementi.Add(entity);
    }

    public IEnumerable<T> Tutti()
    {
        return elementi;
    }
}
```

Classe `Cliente`:

```csharp id="urrbte"
public class Cliente : IEntity
{
```

# Generics e Result Pattern

Un altro uso moderno dei generics è il **Result Pattern**.

Invece di restituire solo un valore o lanciare eccezioni per casi previsti, possiamo restituire un oggetto che rappresenta successo o fallimento.

```csharp id="dfo9ao"
public record Result<T>
{
    public bool Successo { get; }
    public T? Valore { get; }
    public string? Errore { get; }

    private Result(bool successo, T? valore, string? errore)
    {
        Successo = successo;
        Valore = valore;
        Errore = errore;
    }

    public static Result<T> Ok(T valore)
    {
        return new Result<T>(true, valore, null);
    }

    public static Result<T> Fail(string errore)
    {
        return new Result<T>(false, default, errore);
    }
}
```

Uso:

```csharp id="lc60z8"
Result<Cliente> CercaCliente(int id)
{
    if (id <= 0)
    {
        return Result<Cliente>.Fail("ID non valido.");
    }

    Cliente cliente = new Cliente(id, "Emilie");

    return Result<Cliente>.Ok(cliente);
}
```

Chiamata:

```csharp id="de0cx3"
Result<Cliente> risultato = CercaCliente(1);

if (risultato.Successo)
{
    Console.WriteLine(risultato.Valore?.Nome);
}
else
{
    Console.WriteLine(risultato.Errore);
}
```

Questo pattern è utile quando il fallimento è previsto e fa parte del flusso applicativo.

# `nameof` con tipi generici non associati

Nelle versioni precedenti di C#, se volevamo ottenere il nome di un tipo generico con `nameof`, dovevamo specificare un argomento di tipo concreto:

```csharp id="tu03mc"
string nome = nameof(List<int>);
```

Il risultato è:

```text id="hk7ihn"
List
```

Il tipo `int` non influisce sul risultato, ma doveva comunque essere scritto.

Con C# 14, `nameof` supporta i tipi generici non associati, cioè senza argomenti concreti:

```csharp id="q6na63"
string nomeLista = nameof(List<>);
string nomeDizionario = nameof(Dictionary<,>);
```

Risultati:

```text id="i2epkk"
List
Dictionary
```

Microsoft indica questa novità tra le funzionalità di C# 14: `nameof(List<>)` restituisce `List`, mentre nelle versioni precedenti era necessario usare un tipo generico chiuso come `List<int>`. ([Microsoft Learn][2])

---

## Perché è utile

È utile in logging, diagnostica, messaggi di errore e analyzer.

Esempio:

```csharp id="va4dqu"
Console.WriteLine($"Tipo generico: {nameof(Result<>)}");
```

Output:

```text id="e2ctnj"
Tipo generico: Result
```

Con tipi a due parametri:

```csharp id="o2i6vp"
Console.WriteLine(nameof(Dictionary<,>));
```

Output:

```text id="xkyy2x"
Dictionary
```

La proposta del linguaggio specifica che sono supportati anche casi come `nameof(A<>.B)`, mentre non sono supportati tipi parzialmente non associati come `Dictionary<int,>`. ([Microsoft Learn][3])

## Repository generico

```csharp id="cotj7q"
public class RepositoryInMemoria<T>
    where T : class, IIdentificabile
{
    private readonly List<T> elementi = new();

    public void Aggiungi(T elemento)
    {
        elementi.Add(elemento);
    }

    public Result<T> TrovaPerId(int id)
    {
        if (id <= 0)
        {
            return Result<T>.Fail("ID non valido.");
        }

        T? elemento = elementi.FirstOrDefault(e => e.Id == id);

        if (elemento is null)
        {
            return Result<T>.Fail($"{typeof(T).Name} non trovato.");
        }

        return Result<T>.Ok(elemento);
    }

    public IReadOnlyList<T> Tutti()
    {
        return elementi;
    }
}
```

# Errori comuni

## Usare object invece dei generics

Sconsigliato:

```csharp id="idhz30"
public class Scatola
{
    public object? Valore { get; set; }
}
```

Meglio:

```csharp id="7u31qw"
public class Scatola<T>
{
    public T? Valore { get; set; }
}
```

I generics evitano cast e aumentano la sicurezza di tipo.

---

## Creare vincoli troppo forti

Sospetto:

```csharp id="mq9yh6"
public class Repository<T>
    where T : class, IIdentificabile, new()
{
}
```

Il vincolo `new()` obbliga tutte le entità ad avere un costruttore vuoto. Questo può indebolire il domain model.

Usare vincoli solo quando servono davvero.

---

## Pensare che List<Cane> sia List<Animale>

Non è valido:

```csharp id="xf9zdx"
List<Cane> cani = new List<Cane>();
List<Animale> animali = cani; // errore
```

Se serve solo leggere, usare `IEnumerable<Animale>`:

```csharp id="mepyez"
IEnumerable<Animale> animali = cani;
```

---

## Usare generics quando non aggiungono valore

Non serve rendere tutto generico.

Sospetto:

```csharp id="kldnfr"
public class ClienteService<T>
{
}
```

Se il servizio lavora solo con `Cliente`, il parametro `T` è inutile.

---

## Ignorare i nullable con T

Con `T?` bisogna capire se `T` è reference type o value type.

Esempio:

```csharp id="fa0wop"
public T? Valore { get; set; }
```

Il significato dipende dai vincoli e dal contesto nullable.

Nei generics avanzati, la gestione dei null richiede attenzione.

# Riepilogo

In questo capitolo abbiamo introdotto i generics, uno degli strumenti più importanti del C# moderno.

Abbiamo visto che un tipo generico usa parametri di tipo:

```csharp id="vj5w3h"

{
    public T Contenuto { get; }
}
```

Abbiamo studiato:

```text id="dm25et"
classi generiche
  classi riutilizzabili con tipi diversi.

metodi generici
  metodi parametrizzati per tipo.

interfacce generiche
  contratti riutilizzabili e tipizzati.

vincoli generici
  regole che limitano quali tipi possono essere usati.
```

Abbiamo visto vincoli comuni:

```text id="o124be"
where T : class
  T deve essere un reference type.

where T : struct
  T deve essere un value type non nullable.

where T : new()
  T deve avere un costruttore pubblico senza parametri.

where T : IInterfaccia
  T deve implementare una certa interfaccia.
```

Abbiamo introdotto:

```text id="qeu1fh"
Nullable<T>
  base generica di int?, decimal?, DateTime?.

collezioni generiche
  List<T>, Dictionary<TKey, TValue>, IEnumerable<T>.

covarianza
  usare tipi più specifici quando il tipo viene prodotto.

controvarianza
  usare tipi più generali quando il tipo viene consumato.

nameof con generici non associati
  novità moderna di C# 14, per esempio nameof(List<>).
```

La regola principale è questa:

```text id="yw38ys"
Usa i generics quando vuoi riutilizzare una struttura o un comportamento
mantenendo la sicurezza dei tipi.
```

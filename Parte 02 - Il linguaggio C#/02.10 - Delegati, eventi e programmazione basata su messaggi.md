# 02.10 — Delegati, eventi e programmazione basata su messaggi

Nei capitoli precedenti abbiamo visto metodi, classi, interfacce, ereditarietà e generics. Ora introduciamo un concetto che a prima vista può sembrare più astratto, ma che è centrale in moltissime parti di .NET: i **delegati** e gli **eventi**.

Li ritroveremo in molti scenari:

* interfacce grafiche;
* pulsanti e controlli UI;
* callback;
* LINQ;
* validazione;
* programmazione asincrona;
* observer pattern;
* sistemi real-time;
* SignalR;
* code di messaggi;
* microservizi;
* architetture event-driven.

Il punto centrale è questo:

> un delegato permette di trattare un metodo come un valore.
> un evento permette a un oggetto di notificare ad altri oggetti che qualcosa è accaduto.

Con questi strumenti, il codice non è più soltanto una sequenza di chiamate dirette. Diventa un sistema in cui una parte può dire:

```text
È successo qualcosa.
Chi è interessato può reagire.
```

---

# Delegati

Un **delegato** è un tipo che rappresenta un riferimento a un metodo con una certa firma.

La firma comprende:

* tipo di ritorno;
* numero dei parametri;
* tipo dei parametri.

Esempio:

```csharp
public delegate void Operazione();
```

Questo delegato può contenere un riferimento a un metodo che:

* non restituisce nulla;
* non riceve parametri.

Esempio:

```csharp
void Saluta()
{
    Console.WriteLine("Ciao!");
}

Operazione operazione = Saluta;

operazione();
```

Output:

```text
Ciao!
```

Qui `operazione` è una variabile che contiene un riferimento al metodo `Saluta`.

Quando scriviamo:

```csharp
operazione();
```

stiamo invocando il metodo contenuto nel delegato.

---

## Delegato con parametri

Possiamo definire un delegato che riceve parametri.

```csharp
public delegate void OperazioneSuNumero(int numero);
```

Metodo compatibile:

```csharp
void StampaDoppio(int numero)
{
    Console.WriteLine(numero * 2);
}
```

Uso:

```csharp
OperazioneSuNumero operazione = StampaDoppio;

operazione(5);
```

Output:

```text
10
```

Il metodo `StampaDoppio` è compatibile con il delegato perché:

```text
riceve un int
restituisce void
```

---

## Delegato con valore di ritorno

Un delegato può anche rappresentare un metodo che restituisce un valore.

```csharp
public delegate int Calcolo(int a, int b);
```

Metodo compatibile:

```csharp
int Somma(int a, int b)
{
    return a + b;
}
```

Uso:

```csharp
Calcolo calcolo = Somma;

int risultato = calcolo(3, 4);

Console.WriteLine(risultato);
```

Output:

```text
7
```

---

## Perché usare un delegato?

I delegati sono utili quando vogliamo passare un comportamento a un altro metodo.

Esempio:

```csharp
void EseguiOperazione(int numero, OperazioneSuNumero operazione)
{
    operazione(numero);
}
```

Uso:

```csharp
void StampaQuadrato(int numero)
{
    Console.WriteLine(numero * numero);
}

EseguiOperazione(5, StampaQuadrato);
```

Output:

```text
25
```

Qui `EseguiOperazione` non sa quale operazione verrà eseguita. Riceve il comportamento dall’esterno.

Questo è un passaggio importante: il codice diventa più flessibile perché una parte del programma può ricevere una logica da eseguire.

---

# Delegati e lambda

I delegati si usano spesso con le lambda expressions.

Invece di scrivere:

```csharp
void StampaQuadrato(int numero)
{
    Console.WriteLine(numero * numero);
}

EseguiOperazione(5, StampaQuadrato);
```

possiamo scrivere:

```csharp
EseguiOperazione(5, numero =>
{
    Console.WriteLine(numero * numero);
});
```

Oppure, se il corpo è semplice:

```csharp
EseguiOperazione(5, numero => Console.WriteLine(numero * numero));
```

La lambda:

```csharp
numero => Console.WriteLine(numero * numero)
```

è compatibile con un delegato che riceve un `int` e restituisce `void`.

# `Action`

In molti casi non serve definire un delegato personalizzato, perché .NET fornisce delegati generici già pronti.

Uno dei più usati è `Action`.

`Action` rappresenta un metodo che non restituisce un valore.

Esempio senza parametri:

```csharp
Action saluta = () => Console.WriteLine("Ciao!");

saluta();
```

Output:

```text
Ciao!
```

---

## Action con parametri

`Action<T>` rappresenta un metodo che riceve un parametro e restituisce `void`.

```csharp
Action<string> stampaMessaggio = messaggio =>
{
    Console.WriteLine(messaggio);
};

stampaMessaggio("Operazione completata");
```

Output:

```text
Operazione completata
```

`Action<int>` rappresenta un metodo che riceve un `int`.

```csharp
Action<int> stampaDoppio = numero =>
{
    Console.WriteLine(numero * 2);
};

stampaDoppio(10);
```

Output:

```text
20
```

---

## Action con più parametri

Esistono versioni di `Action` con più parametri.

```csharp
Action<string, int> stampaCliente = (nome, eta) =>
{
    Console.WriteLine($"{nome}, {eta} anni");
};

stampaCliente("Emilie", 42);
```

Output:

```text
Emilie, 42 anni
```

La regola è:

```text
Action
  nessun parametro, nessun valore di ritorno.

Action<T>
  un parametro, nessun valore di ritorno.

Action<T1, T2>
  due parametri, nessun valore di ritorno.
```

---

# `Func`

`Func` rappresenta un metodo che restituisce un valore.

Esempio:

```csharp
Func<int> ottieniNumero = () => 42;

int numero = ottieniNumero();

Console.WriteLine(numero);
```

Output:

```text
42
```

---

## Func con parametri

`Func<T, TResult>` rappresenta un metodo che riceve un parametro di tipo `T` e restituisce un valore di tipo `TResult`.

```csharp
Func<int, int> raddoppia = numero => numero * 2;

Console.WriteLine(raddoppia(5));
```

Output:

```text
10
```

Qui:

```text
int
  tipo del parametro.

int finale
  tipo del valore restituito.
```

Quindi:

```csharp
Func<int, int>
```

significa:

```text
riceve int
restituisce int
```

---

## Func con più parametri

```csharp
Func<int, int, int> somma = (a, b) => a + b;

Console.WriteLine(somma(3, 4));
```

Output:

```text
7
```

La regola è:

```text
Func<T1, TResult>
  un parametro e un valore di ritorno.

Func<T1, T2, TResult>
  due parametri e un valore di ritorno.

Func<T1, T2, T3, TResult>
  tre parametri e un valore di ritorno.
```

L’ultimo tipo è sempre il tipo restituito.

---

## Esempio pratico con Func

Supponiamo di voler applicare un calcolo a una lista di importi.

```csharp
List<decimal> importi = new List<decimal>
{
    100m,
    200m,
    300m
};

decimal ApplicaCalcolo(decimal importo, Func<decimal, decimal> calcolo)
{
    return calcolo(importo);
}
```

Uso:

```csharp
decimal conIva = ApplicaCalcolo(100m, importo => importo * 1.22m);

Console.WriteLine(conIva);
```

Output:

```text
122
```

Il metodo `ApplicaCalcolo` non conosce la regola. La riceve dall’esterno.

# `Predicate`

`Predicate<T>` rappresenta un metodo che riceve un valore di tipo `T` e restituisce `bool`.

Serve a esprimere una condizione.

Esempio:

```csharp
Predicate<int> èPari = numero => numero % 2 == 0;

Console.WriteLine(èPari(4));
Console.WriteLine(èPari(5));
```

Output:

```text
True
False
```

`Predicate<T>` è concettualmente simile a:

```csharp
Func<T, bool>
```

Quindi:

```csharp
Predicate<int>
```

e:

```csharp
Func<int, bool>
```

possono rappresentare entrambi una funzione che riceve un `int` e restituisce `bool`.

---

## Esempio con Predicate

```csharp
bool TuttiValidi<T>(IEnumerable<T> elementi, Predicate<T> criterio)
{
    foreach (T elemento in elementi)
    {
        if (!criterio(elemento))
        {
            return false;
        }
    }

    return true;
}
```

Uso:

```csharp
List<int> numeri = new List<int> { 2, 4, 6 };

bool tuttiPari = TuttiValidi(numeri, numero => numero % 2 == 0);

Console.WriteLine(tuttiPari);
```

Output:

```text
True
```

# Delegati multicast

Un delegato può contenere più metodi.

Esempio:

```csharp
Action operazioni = () => Console.WriteLine("Prima operazione");

operazioni += () => Console.WriteLine("Seconda operazione");
operazioni += () => Console.WriteLine("Terza operazione");

operazioni();
```

Output:

```text
Prima operazione
Seconda operazione
Terza operazione
```

Questo comportamento è importante per capire gli eventi: più oggetti possono registrarsi e reagire allo stesso evento.

Possiamo anche rimuovere un metodo:

```csharp
void Prima()
{
    Console.WriteLine("Prima");
}

void Seconda()
{
    Console.WriteLine("Seconda");
}

Action azione = Prima;
azione += Seconda;

azione -= Prima;

azione();
```

Output:

```text
Seconda
```

# Eventi

Un **evento** è un meccanismo che permette a un oggetto di notificare che qualcosa è accaduto.

Esempi:

```text
Un pulsante è stato cliccato.
Un ordine è stato confermato.
Un download è terminato.
Un valore è cambiato.
Un messaggio è arrivato.
Un cliente è stato registrato.
```

L’oggetto che genera l’evento viene spesso chiamato **publisher**.
Gli oggetti che si registrano per riceverlo vengono chiamati **subscriber**.

---

## Dichiarare un evento

In C#, un evento si dichiara usando la keyword `event`.

```csharp
public class Contatore
{
    public event Action? SogliaRaggiunta;

    private int valore;

    public void Incrementa()
    {
        valore++;

        if (valore == 3)
        {
            SogliaRaggiunta?.Invoke();
        }
    }
}
```

Uso:

```csharp
Contatore contatore = new Contatore();

contatore.SogliaRaggiunta += () =>
{
    Console.WriteLine("Soglia raggiunta!");
};

contatore.Incrementa();
contatore.Incrementa();
contatore.Incrementa();
```

Output:

```text
Soglia raggiunta!
```

---

## Che cosa significa `?.Invoke()`

Questa riga:

```csharp
SogliaRaggiunta?.Invoke();
```

significa:

> se qualcuno si è registrato all’evento, invocalo; altrimenti non fare nulla.

Senza `?.`, se non ci fossero subscriber, l’evento sarebbe `null` e otterremmo un errore.

---

## Registrarsi a un evento

Per registrarsi a un evento si usa `+=`.

```csharp
contatore.SogliaRaggiunta += GestisciSoglia;
```

Dove:

```csharp
void GestisciSoglia()
{
    Console.WriteLine("La soglia è stata raggiunta.");
}
```

Per rimuovere la registrazione si usa `-=`.

```csharp
contatore.SogliaRaggiunta -= GestisciSoglia;
```

# Event handler

In .NET esiste una convenzione molto diffusa per gli eventi:

```csharp
event EventHandler NomeEvento;
```

Oppure:

```csharp
event EventHandler<TEventArgs> NomeEvento;
```

La firma classica di un event handler è:

```csharp
void Metodo(object? sender, EventArgs e)
```

Dove:

```text
sender
  oggetto che ha generato l’evento.

e
  dati dell’evento.
```

---

## Evento senza dati aggiuntivi

```csharp
public class PulsanteSimulato
{
    public event EventHandler? Click;

    public void SimulaClick()
    {
        Click?.Invoke(this, EventArgs.Empty);
    }
}
```

Uso:

```csharp
PulsanteSimulato pulsante = new PulsanteSimulato();

pulsante.Click += Pulsante_Click;

pulsante.SimulaClick();

void Pulsante_Click(object? sender, EventArgs e)
{
    Console.WriteLine("Pulsante cliccato.");
}
```

Output:

```text
Pulsante cliccato.
```

---

## Evento con dati personalizzati

Per passare dati con un evento, possiamo creare una classe che eredita da `EventArgs`.

```csharp
public class ClienteRegistratoEventArgs : EventArgs
{
    public string NomeCliente { get; }

    public ClienteRegistratoEventArgs(string nomeCliente)
    {
        NomeCliente = nomeCliente;
    }
}
```

Classe publisher:

```csharp
public class RegistroClienti
{
    public event EventHandler<ClienteRegistratoEventArgs>? ClienteRegistrato;

    public void Registra(string nome)
    {
        if (string.IsNullOrWhiteSpace(nome))
        {
            throw new ArgumentException("Il nome è obbligatorio.", nameof(nome));
        }

        ClienteRegistrato?.Invoke(
            this,
            new ClienteRegistratoEventArgs(nome));
    }
}
```

Uso:

```csharp
RegistroClienti registro = new RegistroClienti();

registro.ClienteRegistrato += Registro_ClienteRegistrato;

registro.Registra("Emilie");

void Registro_ClienteRegistrato(object? sender, ClienteRegistratoEventArgs e)
{
    Console.WriteLine($"Cliente registrato: {e.NomeCliente}");
}
```

Output:

```text
Cliente registrato: Emilie
```

# Publisher/subscriber

Il modello **publisher/subscriber** descrive una relazione in cui:

* un oggetto pubblica un evento;
* uno o più oggetti si iscrivono;
* il publisher non conosce direttamente i subscriber.

Esempio:

```text
RegistroClienti
  pubblica ClienteRegistrato.

InvioEmailService
  si iscrive e invia una mail.

AuditService
  si iscrive e scrive un log.

StatisticheService
  si iscrive e aggiorna statistiche.
```

Il publisher non deve sapere quanti subscriber ci sono.

---

## Esempio

Publisher:

```csharp
public class Ordine
{
    public event EventHandler? Confermato;

    public bool ÈConfermato { get; private set; }

    public void Conferma()
    {
        if (ÈConfermato)
        {
            throw new InvalidOperationException("Ordine già confermato.");
        }

        ÈConfermato = true;

        Confermato?.Invoke(this, EventArgs.Empty);
    }
}
```

Subscriber:

```csharp
void InviaEmailConferma(object? sender, EventArgs e)
{
    Console.WriteLine("Invio email di conferma.");
}

void ScriviLogConferma(object? sender, EventArgs e)
{
    Console.WriteLine("Scrivo log di conferma.");
}
```

Uso:

```csharp
Ordine ordine = new Ordine();

ordine.Confermato += InviaEmailConferma;
ordine.Confermato += ScriviLogConferma;

ordine.Conferma();
```

Output:

```text
Invio email di conferma.
Scrivo log di conferma.
```

L’ordine non sa nulla di email o log. Si limita a comunicare che è stato confermato.

---

# Observer pattern

Il modello publisher/subscriber è molto vicino all’**Observer pattern**.

L’Observer pattern descrive una relazione in cui un oggetto, chiamato **subject**, mantiene un elenco di osservatori e li avvisa quando il suo stato cambia.

In C# gli eventi sono un modo naturale per implementare questo pattern.

Esempio concettuale:

```text
Subject
  oggetto osservato.

Observer
  oggetto interessato ai cambiamenti.

Evento
  notifica del cambiamento.
```

Nel nostro esempio:

```text
Ordine
  subject.

InviaEmailConferma
ScriviLogConferma
  observer/subscriber.

Confermato
  evento.
```

---

## Esempio con cambio valore

```csharp
public class SensoreTemperatura
{
    public event EventHandler<TemperaturaCambiataEventArgs>? TemperaturaCambiata;

    private decimal temperatura;

    public decimal Temperatura
    {
        get => temperatura;
        set
        {
            if (temperatura == value)
            {
                return;
            }

            temperatura = value;

            TemperaturaCambiata?.Invoke(
                this,
                new TemperaturaCambiataEventArgs(value));
        }
    }
}

public class TemperaturaCambiataEventArgs : EventArgs
{
    public decimal NuovaTemperatura { get; }

    public TemperaturaCambiataEventArgs(decimal nuovaTemperatura)
    {
        NuovaTemperatura = nuovaTemperatura;
    }
}
```

Uso:

```csharp
SensoreTemperatura sensore = new SensoreTemperatura();

sensore.TemperaturaCambiata += (sender, e) =>
{
    Console.WriteLine($"Nuova temperatura: {e.NuovaTemperatura}");
};

sensore.Temperatura = 22.5m;
sensore.Temperatura = 23.0m;
```

Output:

```text
Nuova temperatura: 22,5
Nuova temperatura: 23,0
```

# Eventi nei componenti UI

Gli eventi sono fondamentali nelle interfacce grafiche.

Un’applicazione UI è piena di eventi:

```text
Click
TextChanged
SelectionChanged
Loaded
Closing
MouseMove
KeyDown
```

Esempio concettuale in una UI:

```csharp
button.Click += Button_Click;

void Button_Click(object? sender, EventArgs e)
{
    Console.WriteLine("Hai premuto il pulsante.");
}
```

Il pulsante è il publisher.
Il metodo `Button_Click` è il subscriber.

Quando l’utente clicca, il controllo genera l’evento.

---

## Perché gli eventi sono naturali nella UI

In una UI il flusso non è lineare.

Non possiamo scrivere:

```text
prima l’utente clicca qui
poi scrive lì
poi seleziona questo
poi chiude la finestra
```

È l’utente a decidere l’ordine delle azioni.

Il programma deve reagire.

Per questo le UI sono naturalmente event-driven:

```text
evento dell’utente
  -> codice di risposta
```

---

## Esempio didattico

```csharp
public class TextBoxSimulata
{
    public event EventHandler? TestoCambiato;

    private string testo = "";

    public string Testo
    {
        get => testo;
        set
        {
            if (testo == value)
            {
                return;
            }

            testo = value;
            TestoCambiato?.Invoke(this, EventArgs.Empty);
        }
    }
}
```

Uso:

```csharp
TextBoxSimulata textBox = new TextBoxSimulata();

textBox.TestoCambiato += (sender, e) =>
{
    Console.WriteLine("Il testo è cambiato.");
};

textBox.Testo = "Ciao";
textBox.Testo = "Ciao mondo";
```

Output:

```text
Il testo è cambiato.
Il testo è cambiato.
```

Questo esempio imita il comportamento di molti controlli UI reali.

# Eventi nelle architetture applicative

Gli eventi non servono solo nelle interfacce grafiche.

Sono molto utili anche nelle architetture applicative.

Esempi:

```text
ClienteRegistrato
OrdineConfermato
PagamentoRicevuto
DocumentoAnnullato
EmailInviata
FileImportato
ReportGenerato
```

Questi eventi possono essere usati per separare la logica principale da effetti secondari.

---

## Esempio: registrazione cliente

Senza eventi:

```csharp
public class ClienteService
{
    public void RegistraCliente(string nome, string email)
    {
        Console.WriteLine("Salvo cliente nel database.");
        Console.WriteLine("Invio email di benvenuto.");
        Console.WriteLine("Scrivo log.");
        Console.WriteLine("Aggiorno statistiche.");
    }
}
```

Il metodo fa troppe cose.

Con eventi:

```csharp
public class ClienteService
{
    public event EventHandler<ClienteRegistratoEventArgs>? ClienteRegistrato;

    public void RegistraCliente(string nome, string email)
    {
        Console.WriteLine("Salvo cliente nel database.");

        ClienteRegistrato?.Invoke(
            this,
            new ClienteRegistratoEventArgs(nome, email));
    }
}

public class ClienteRegistratoEventArgs : EventArgs
{
    public string Nome { get; }
    public string Email { get; }

    public ClienteRegistratoEventArgs(string nome, string email)
    {
        Nome = nome;
        Email = email;
    }
}
```

Subscriber:

```csharp
void InviaEmailBenvenuto(object? sender, ClienteRegistratoEventArgs e)
{
    Console.WriteLine($"Invio email a {e.Email}");
}

void ScriviAudit(object? sender, ClienteRegistratoEventArgs e)
{
    Console.WriteLine($"Audit: registrato cliente {e.Nome}");
}
```

Uso:

```csharp
ClienteService service = new ClienteService();

service.ClienteRegistrato += InviaEmailBenvenuto;
service.ClienteRegistrato += ScriviAudit;

service.RegistraCliente("Emilie", "emilie@example.com");
```

Output:

```text
Salvo cliente nel database.
Invio email a emilie@example.com
Audit: registrato cliente Emilie
```

Il servizio principale non contiene direttamente la logica di email e audit.

## Eventi di dominio

In architetture più avanzate si parla spesso di **domain events**.

Un domain event rappresenta qualcosa di significativo accaduto nel dominio.

Esempi:

```text
OrdineConfermato
PagamentoAutorizzato
ClienteSospeso
DocumentoFirmato
SpedizionePartita
```

Non è semplicemente un evento tecnico. È un fatto di business.

Esempio:

```csharp
public record OrdineConfermato(int OrdineId, DateTime DataConferma);
```

Questo record rappresenta un messaggio interno al dominio.

Potrebbe essere gestito da più handler:

```text
Invia email
Aggiorna magazzino
Genera movimento contabile
Scrivi audit
Notifica altri servizi
```

Nel capitolo sull’architettura vedremo che questa idea è alla base di molti sistemi modulari ed event-driven.

# Differenza tra evento, callback e messaggio

Queste parole vengono spesso usate insieme, ma non indicano esattamente la stessa cosa.

---

## Callback

Una **callback** è una funzione passata a un’altra funzione affinché venga chiamata in un secondo momento.

Esempio:

```csharp
void EseguiConCallback(Action callback)
{
    Console.WriteLine("Prima della callback");

    callback();

    Console.WriteLine("Dopo la callback");
}
```

Uso:

```csharp
EseguiConCallback(() =>
{
    Console.WriteLine("Sono la callback");
});
```

Output:

```text
Prima della callback
Sono la callback
Dopo la callback
```

Qui passiamo un comportamento da eseguire.

Una callback è spesso uno-a-uno:

```text
chiami un metodo
gli passi una funzione
quel metodo la richiama
```

---

## Evento

Un **evento** è una notifica emessa da un oggetto.

Più subscriber possono registrarsi.

Esempio:

```csharp
public event EventHandler? Completato;
```

L’oggetto non sa quanti subscriber ci sono.

Un evento è spesso uno-a-molti:

```text
un publisher
molti subscriber
```

---

## Messaggio

Un **messaggio** è un dato che rappresenta una comunicazione.

Può essere inviato:

* in memoria;
* tramite una coda;
* tramite un bus;
* tramite rete;
* tra microservizi;
* tra processi diversi.

Esempio:

```csharp
public record ClienteRegistratoMessage(
    int ClienteId,
    string Nome,
    string Email);
```

Un messaggio non è necessariamente legato agli eventi C#.

Può essere serializzato in JSON e inviato a un altro servizio.

---

## Confronto

```text
Callback
  passo una funzione da richiamare.

Evento
  notifico che qualcosa è accaduto.

Messaggio
  invio dati che rappresentano una comunicazione o un fatto.
```

Esempio pratico:

```text
Callback
  Quando hai finito, chiama questa funzione.

Evento
  Il cliente è stato registrato; chi è interessato reagisca.

Messaggio
  Pubblico ClienteRegistrato su una coda RabbitMQ.
```

# Eventi C# e messaging distribuito

È importante non confondere gli eventi del linguaggio C# con i messaggi di un sistema distribuito.

Un evento C#:

```csharp
public event EventHandler ClienteRegistrato;
```

vive dentro lo stesso processo applicativo.

Un messaggio distribuito:

```csharp
public record ClienteRegistrato(int ClienteId, string Email);
```

può viaggiare tra servizi diversi usando:

* code;
* broker;
* HTTP;
* gRPC;
* SignalR;
* RabbitMQ;
* Azure Service Bus;
* Kafka.

Gli eventi C# sono uno strumento linguistico.
I messaggi distribuiti sono uno strumento architetturale.

Il concetto è simile, ma il contesto è diverso.

---

# Eventi e memoria

Quando ci si iscrive a un evento, il publisher conserva un riferimento al subscriber.

Questo può avere effetti importanti.

Esempio:

```csharp
publisher.Evento += subscriber.GestisciEvento;
```

Se il publisher vive molto più a lungo del subscriber, il subscriber potrebbe non essere liberato dalla memoria finché resta iscritto.

Per questo, in certi scenari, è importante disiscriversi:

```csharp
publisher.Evento -= subscriber.GestisciEvento;
```

Questo è particolarmente importante in applicazioni UI, componenti lunghi, servizi singleton e oggetti che hanno cicli di vita diversi.

Regola pratica:

```text
Se ti iscrivi a un evento di un oggetto che vive più a lungo di te,
valuta sempre quando disiscriverti.
```

# Esempio completo del capitolo

Costruiamo un piccolo esempio con:

* delegati;
* `Func`;
* eventi;
* publisher/subscriber;
* messaggio applicativo.

Scenario:

```text
Un servizio registra ordini.
Quando un ordine viene confermato, genera un evento.
Più subscriber reagiscono: log e notifica.
Il calcolo dello sconto viene passato come funzione.
```

---

## Modello ordine

```csharp
public class Ordine
{
    public int Id { get; }
    public decimal TotaleLordo { get; }
    public decimal TotaleNetto { get; private set; }
    public bool Confermato { get; private set; }

    public Ordine(int id, decimal totaleLordo)
    {
        if (id <= 0)
        {
            throw new ArgumentOutOfRangeException(nameof(id));
        }

        if (totaleLordo < 0)
        {
            throw new ArgumentOutOfRangeException(nameof(totaleLordo));
        }

        Id = id;
        TotaleLordo = totaleLordo;
        TotaleNetto = totaleLordo;
    }

    public void ApplicaSconto(Func<decimal, decimal> calcolaSconto)
    {
        decimal sconto = calcolaSconto(TotaleLordo);

        if (sconto < 0 || sconto > TotaleLordo)
        {
            throw new InvalidOperationException("Sconto non valido.");
        }

        TotaleNetto = TotaleLordo - sconto;
    }

    public void Conferma()
    {
        if (Confermato)
        {
            throw new InvalidOperationException("Ordine già confermato.");
        }

        Confermato = true;
    }
}
```

---

## EventArgs

```csharp
public class OrdineConfermatoEventArgs : EventArgs
{
    public int OrdineId { get; }
    public decimal TotaleNetto { get; }

    public OrdineConfermatoEventArgs(int ordineId, decimal totaleNetto)
    {
        OrdineId = ordineId;
        TotaleNetto = totaleNetto;
    }
}
```

---

## Servizio ordini

```csharp
public class OrdineService
{
    public event EventHandler<OrdineConfermatoEventArgs>? OrdineConfermato;

    public Ordine CreaEConfermaOrdine(
        int id,
        decimal totaleLordo,
        Func<decimal, decimal> calcolaSconto)
    {
        Ordine ordine = new Ordine(id, totaleLordo);

        ordine.ApplicaSconto(calcolaSconto);
        ordine.Conferma();

        OrdineConfermato?.Invoke(
            this,
            new OrdineConfermatoEventArgs(ordine.Id, ordine.TotaleNetto));

        return ordine;
    }
}
```

---

## Subscriber

```csharp
public class AuditService
{
    public void GestisciOrdineConfermato(object? sender, OrdineConfermatoEventArgs e)
    {
        Console.WriteLine($"AUDIT - Ordine {e.OrdineId} confermato.");
    }
}

public class NotificaService
{
    public void GestisciOrdineConfermato(object? sender, OrdineConfermatoEventArgs e)
    {
        Console.WriteLine($"NOTIFICA - Ordine {e.OrdineId}, totale {e.TotaleNetto}.");
    }
}
```

---

## Uso

```csharp
OrdineService ordineService = new OrdineService();

AuditService auditService = new AuditService();
NotificaService notificaService = new NotificaService();

ordineService.OrdineConfermato += auditService.GestisciOrdineConfermato;
ordineService.OrdineConfermato += notificaService.GestisciOrdineConfermato;

Ordine ordine = ordineService.CreaEConfermaOrdine(
    id: 1,
    totaleLordo: 100m,
    calcolaSconto: totale => totale * 10m / 100m);

Console.WriteLine($"Totale netto ordine: {ordine.TotaleNetto}");
```

Output:

```text
AUDIT - Ordine 1 confermato.
NOTIFICA - Ordine 1, totale 90.
Totale netto ordine: 90
```

In questo esempio:

```text
Func<decimal, decimal>
  rappresenta la regola di calcolo dello sconto.

OrdineConfermato
  è un evento.

OrdineService
  è il publisher.

AuditService e NotificaService
  sono subscriber.

OrdineConfermatoEventArgs
  contiene i dati dell’evento.
```

# Errori comuni

## Creare delegati personalizzati quando bastano Action o Func

Possibile, ma spesso inutile:

```csharp
public delegate int Calcolo(int a, int b);
```

Spesso basta:

```csharp
Func<int, int, int> calcolo;
```

Crea un delegato personalizzato quando il nome aggiunge significato al dominio.

Esempio utile:

```csharp
public delegate decimal CalcolaSconto(decimal totale);
```

Qui il nome `CalcolaSconto` comunica l’intenzione.

---

## Usare eventi per logica che dovrebbe essere diretta

Non tutto deve diventare evento.

Sospetto:

```csharp
public event EventHandler? PrimaDiCalcolareTotale;
public event EventHandler? DopoCalcoloTotale;
```

Se la logica è parte essenziale dell’operazione, spesso è meglio chiamare metodi espliciti.

Gli eventi sono utili per notificare qualcosa che è accaduto, non per rendere oscuro il flusso principale.

---

## Dimenticare il controllo null sull’evento

Rischioso:

```csharp
OrdineConfermato(this, args);
```

Se nessuno è iscritto, genera errore.

Meglio:

```csharp
OrdineConfermato?.Invoke(this, args);
```

---

## Non disiscriversi dagli eventi

In certi scenari, dimenticare:

```csharp
publisher.Evento -= Handler;
```

può trattenere oggetti in memoria più a lungo del necessario.

Questo è particolarmente importante in UI e oggetti con cicli di vita diversi.

---

## Mettere troppa logica negli event handler UI

Sconsigliato:

```csharp
void Button_Click(object? sender, EventArgs e)
{
    // 200 righe di logica
}
```

Meglio:

```csharp
void Button_Click(object? sender, EventArgs e)
{
    RegistraCliente();
}
```

E poi spostare la logica in metodi o servizi separati.

---

## Confondere eventi C# e messaggi distribuiti

Un evento C# non attraversa automaticamente processi, server o microservizi.

Questo:

```csharp
public event EventHandler ClienteRegistrato;
```

vale dentro lo stesso processo.

Per comunicare tra servizi servono messaggi e infrastrutture dedicate.

# Riepilogo

In questo capitolo abbiamo introdotto delegati, eventi e programmazione basata su notifiche.

Abbiamo visto:

```text
delegato
  tipo che rappresenta un riferimento a un metodo.

Action
  delegato predefinito per metodi senza valore di ritorno.

Func
  delegato predefinito per metodi con valore di ritorno.

Predicate
  delegato predefinito per condizioni che restituiscono bool.
```

Abbiamo poi introdotto gli eventi:

```text
event
  meccanismo per notificare che qualcosa è accaduto.

EventHandler
  convenzione .NET per eventi.

EventArgs
  dati associati all’evento.

publisher
  oggetto che pubblica l’evento.

subscriber
  oggetto che si iscrive e reagisce.
```

Abbiamo collegato questi concetti a modelli più ampi:

```text
Observer pattern
  oggetti osservatori reagiscono ai cambiamenti di un subject.

UI event-driven
  l’interfaccia reagisce agli eventi dell’utente.

Eventi applicativi
  parti del sistema reagiscono a fatti significativi.

Messaggi
  dati che rappresentano comunicazioni, anche tra processi o servizi.
```

Infine abbiamo distinto:

```text
callback
  funzione passata e richiamata.

evento
  notifica uno-a-molti dentro il processo.

messaggio
  dato comunicato, anche in architetture distribuite.
```

La regola principale è questa:

```text
Usa delegati e callback quando vuoi passare comportamento.
Usa eventi quando vuoi notificare che qualcosa è accaduto.
Usa messaggi quando vuoi comunicare fatti o comandi tra parti più disaccoppiate del sistema.
```

Questo capitolo è un ponte verso molti temi futuri: UI desktop, Blazor, async/await, SignalR, domain events, code di messaggi, microservizi e architetture event-driven.

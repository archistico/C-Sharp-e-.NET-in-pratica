# 02.05 — Gestione degli errori ed eccezioni

Finora abbiamo scritto codice assumendo che tutto vada bene:

* l’utente inserisce un numero quando chiediamo un numero;
* una variabile contiene un valore valido;
* un metodo riceve parametri corretti;
* una conversione riesce;
* un oggetto non è `null`;
* un calcolo produce un risultato sensato.

Nelle applicazioni reali, però, le cose non vanno sempre così.

Un utente può inserire un testo al posto di un numero.
Un file può non esistere.
Una connessione può fallire.
Un valore può essere `null`.
Un metodo può ricevere un parametro non valido.
Un calcolo può superare i limiti numerici.
Un servizio esterno può non rispondere.

Per questo C# e .NET mettono a disposizione un meccanismo strutturato per rappresentare e gestire gli errori: le **eccezioni**.

---

## Che cos’è un’eccezione

Un’**eccezione** è un oggetto che rappresenta una condizione anomala avvenuta durante l’esecuzione del programma.

Quando qualcosa va storto, il codice può **lanciare** un’eccezione. Se nessuno la gestisce, il programma termina con errore.

Esempio:

```csharp
int numero = int.Parse("ciao");

Console.WriteLine(numero);
```

Questo codice prova a convertire la stringa `"ciao"` in un numero intero.

La conversione non è possibile, quindi viene generata un’eccezione.

Il programma non può proseguire normalmente, perché `numero` non può essere calcolato.

---

## Eccezioni e flusso normale

Un’eccezione interrompe il flusso normale del programma.

Esempio:

```csharp
Console.WriteLine("Prima");

int numero = int.Parse("ciao");

Console.WriteLine("Dopo");
```

L’output sarà:

```text
Prima
```

La riga:

```csharp
Console.WriteLine("Dopo");
```

non viene eseguita, perché l’eccezione interrompe il flusso.

---

## Un’eccezione è un oggetto

In .NET le eccezioni sono oggetti che derivano, direttamente o indirettamente, dalla classe:

```csharp
Exception
```

Un’eccezione contiene informazioni utili, per esempio:

* tipo dell’errore;
* messaggio;
* stack trace;
* eccezione interna;
* dati aggiuntivi.

Esempio:

```csharp
try
{
    int numero = int.Parse("ciao");
}
catch (Exception ex)
{
    Console.WriteLine(ex.GetType().Name);
    Console.WriteLine(ex.Message);
}
```

Output possibile:

```text
FormatException
The input string 'ciao' was not in a correct format.
```

Il tipo dell’eccezione è `FormatException`.

---

# `try`

Il blocco `try` contiene codice che potrebbe generare un’eccezione.

Sintassi:

```csharp
try
{
    // codice potenzialmente rischioso
}
```

Da solo, però, `try` non basta. Deve essere accompagnato almeno da `catch` o `finally`.

Esempio:

```csharp
try
{
    int numero = int.Parse("123");
    Console.WriteLine(numero);
}
catch
{
    Console.WriteLine("Errore durante la conversione.");
}
```

In questo caso la conversione riesce, quindi il `catch` non viene eseguito.

Output:

```text
123
```

Se cambiamo il valore:

```csharp
try
{
    int numero = int.Parse("ciao");
    Console.WriteLine(numero);
}
catch
{
    Console.WriteLine("Errore durante la conversione.");
}
```

Output:

```text
Errore durante la conversione.
```

---

# `catch`

Il blocco `catch` intercetta un’eccezione.

Forma semplice:

```csharp
try
{
    int numero = int.Parse("ciao");
}
catch
{
    Console.WriteLine("Errore.");
}
```

Questa forma intercetta qualunque eccezione, ma non ci dà dettagli.

Meglio specificare il tipo:

```csharp
try
{
    int numero = int.Parse("ciao");
}
catch (FormatException)
{
    Console.WriteLine("Il testo non contiene un numero valido.");
}
```

Oppure catturare l’oggetto eccezione:

```csharp
try
{
    int numero = int.Parse("ciao");
}
catch (FormatException ex)
{
    Console.WriteLine("Errore di formato.");
    Console.WriteLine(ex.Message);
}
```

---

## Più blocchi catch

Possiamo gestire eccezioni diverse in modi diversi.

```csharp
try
{
    Console.Write("Inserisci un numero: ");
    string? input = Console.ReadLine();

    int numero = int.Parse(input!);

    int risultato = 100 / numero;

    Console.WriteLine($"Risultato: {risultato}");
}
catch (FormatException)
{
    Console.WriteLine("Il valore inserito non è un numero valido.");
}
catch (DivideByZeroException)
{
    Console.WriteLine("Non puoi dividere per zero.");
}
catch (Exception ex)
{
    Console.WriteLine("Errore imprevisto.");
    Console.WriteLine(ex.Message);
}
```

Qui gestiamo:

```text
FormatException
  input non numerico.

DivideByZeroException
  divisione per zero.

Exception
  qualunque altro errore non previsto.
```

---

## Ordine dei catch

L’ordine dei `catch` è importante.

Bisogna mettere prima le eccezioni più specifiche e poi quelle più generiche.

Corretto:

```csharp
try
{
    int numero = int.Parse("ciao");
}
catch (FormatException)
{
    Console.WriteLine("Formato non valido.");
}
catch (Exception)
{
    Console.WriteLine("Errore generico.");
}
```

Sbagliato:

```csharp
try
{
    int numero = int.Parse("ciao");
}
catch (Exception)
{
    Console.WriteLine("Errore generico.");
}
catch (FormatException)
{
    Console.WriteLine("Formato non valido.");
}
```

Il secondo `catch` non verrebbe mai raggiunto, perché `Exception` cattura già tutto.

---

## Catch con filtro `when`

C# permette anche di aggiungere una condizione al `catch` usando `when`.

```csharp
try
{
    int numero = int.Parse("-1");

    if (numero < 0)
    {
        throw new ArgumentOutOfRangeException(nameof(numero), "Il numero non può essere negativo.");
    }
}
catch (ArgumentOutOfRangeException ex) when (ex.ParamName == "numero")
{
    Console.WriteLine("Errore sul parametro numero.");
}
```

Il filtro `when` è utile quando vogliamo gestire un’eccezione solo in alcune condizioni.

---

# `finally`

Il blocco `finally` contiene codice che deve essere eseguito comunque, sia che ci sia un’eccezione sia che non ci sia.

Sintassi:

```csharp
try
{
    // codice
}
catch
{
    // gestione errore
}
finally
{
    // codice eseguito sempre
}
```

Esempio:

```csharp
try
{
    Console.WriteLine("Apro una risorsa.");
    int numero = int.Parse("ciao");
}
catch
{
    Console.WriteLine("Errore durante l'elaborazione.");
}
finally
{
    Console.WriteLine("Chiudo la risorsa.");
}
```

Output:

```text
Apro una risorsa.
Errore durante l'elaborazione.
Chiudo la risorsa.
```

Il blocco `finally` viene eseguito anche se si verifica un errore.

---

## Quando usare finally

`finally` è utile per operazioni di pulizia:

* chiudere file;
* chiudere connessioni;
* rilasciare risorse;
* ripristinare stato temporaneo;
* scrivere log di chiusura.

Esempio concettuale:

```csharp
StreamReader? reader = null;

try
{
    reader = new StreamReader("dati.txt");
    string contenuto = reader.ReadToEnd();
    Console.WriteLine(contenuto);
}
catch (Exception ex)
{
    Console.WriteLine(ex.Message);
}
finally
{
    reader?.Dispose();
}
```

Nella pratica moderna, per molte risorse useremo `using`, che vedremo nel capitolo sui file e sugli stream.

---

# `throw`

La keyword `throw` serve a lanciare un’eccezione.

Esempio:

```csharp
throw new InvalidOperationException("Operazione non consentita.");
```

Possiamo usarla quando il nostro codice rileva una situazione non valida.

Esempio:

```csharp
decimal CalcolaSconto(decimal importo, decimal percentuale)
{
    if (importo < 0)
    {
        throw new ArgumentOutOfRangeException(nameof(importo), "L'importo non può essere negativo.");
    }

    if (percentuale < 0 || percentuale > 100)
    {
        throw new ArgumentOutOfRangeException(nameof(percentuale), "La percentuale deve essere tra 0 e 100.");
    }

    return importo * percentuale / 100m;
}
```

Qui il metodo rifiuta parametri non validi.

Questo è meglio che restituire un risultato senza senso.

---

## Rilanciare un’eccezione

Dentro un `catch` possiamo rilanciare l’eccezione.

Corretto:

```csharp
try
{
    EseguiOperazione();
}
catch (Exception ex)
{
    Console.WriteLine(ex.Message);
    throw;
}
```

`throw;` rilancia la stessa eccezione mantenendo lo stack trace originale.

Sbagliato:

```csharp
try
{
    EseguiOperazione();
}
catch (Exception ex)
{
    Console.WriteLine(ex.Message);
    throw ex;
}
```

`throw ex;` è sconsigliato perché può alterare lo stack trace e rendere più difficile capire dove sia nato davvero l’errore.

Regola pratica:

```text
Per rilanciare la stessa eccezione:
  usa throw;

Non usare:
  throw ex;
```

---

## Lanciare una nuova eccezione con inner exception

A volte vogliamo aggiungere contesto a un errore.

```csharp
try
{
    EseguiOperazioneCritica();
}
catch (Exception ex)
{
    throw new InvalidOperationException("Errore durante l'operazione critica.", ex);
}
```

Il secondo parametro `ex` diventa la **inner exception**, cioè l’eccezione originale.

Questo è utile perché aggiungiamo un messaggio più significativo senza perdere la causa iniziale.

---

# Eccezioni built-in comuni

.NET offre molte eccezioni già pronte. Conoscerne alcune aiuta a scrivere codice più chiaro.

---

## `Exception`

È la classe base delle eccezioni.

```csharp
catch (Exception ex)
{
    Console.WriteLine(ex.Message);
}
```

Va bene come ultimo livello di gestione, ma non è ideale catturarla sempre senza distinzione.

---

## `ArgumentException`

Indica che un argomento passato a un metodo non è valido.

```csharp
void ImpostaNome(string nome)
{
    if (string.IsNullOrWhiteSpace(nome))
    {
        throw new ArgumentException("Il nome è obbligatorio.", nameof(nome));
    }
}
```

---

## `ArgumentNullException`

Indica che un argomento obbligatorio è `null`.

```csharp
void StampaNome(string nome)
{
    if (nome is null)
    {
        throw new ArgumentNullException(nameof(nome));
    }

    Console.WriteLine(nome);
}
```

Con C# moderno possiamo spesso evitare molti `null`, ma nei confini pubblici di un metodo la validazione resta importante.

---

## `ArgumentOutOfRangeException`

Indica che un argomento è fuori dall’intervallo consentito.

```csharp
void ImpostaEta(int eta)
{
    if (eta < 0 || eta > 130)
    {
        throw new ArgumentOutOfRangeException(nameof(eta), "Età non valida.");
    }
}
```

---

## `InvalidOperationException`

Indica che l’operazione non è valida nello stato corrente dell’oggetto o del programma.

Esempio:

```csharp
void ConfermaOrdine(bool ordineGiaConfermato)
{
    if (ordineGiaConfermato)
    {
        throw new InvalidOperationException("L'ordine è già stato confermato.");
    }

    Console.WriteLine("Ordine confermato.");
}
```

Il problema non è il parametro in sé. Il problema è lo stato in cui si trova l’operazione.

---

## `FormatException`

Indica che un valore ha un formato non valido.

```csharp
int numero = int.Parse("ciao");
```

Genera una `FormatException`.

---

## `DivideByZeroException`

Indica una divisione per zero.

```csharp
int risultato = 10 / 0;
```

---

## `NullReferenceException`

Indica che stiamo provando a usare un riferimento `null`.

```csharp
Cliente? cliente = null;

Console.WriteLine(cliente.Nome);
```

Questo errore è molto comune e spesso indica un bug.

Meglio prevenire:

```csharp
if (cliente is null)
{
    Console.WriteLine("Cliente non disponibile.");
    return;
}

Console.WriteLine(cliente.Nome);
```

Oppure, se `null` è previsto:

```csharp
Console.WriteLine(cliente?.Nome ?? "Cliente non disponibile");
```

---

## `OverflowException`

Indica che un’operazione numerica ha superato i limiti del tipo.

```csharp
checked
{
    int massimo = int.MaxValue;
    int risultato = massimo + 1;
}
```

---

## `NotImplementedException`

Indica che una funzionalità non è ancora stata implementata.

```csharp
void EsportaPdf()
{
    throw new NotImplementedException();
}
```

Va bene durante lo sviluppo, ma non dovrebbe restare in codice di produzione se quel metodo può essere chiamato.

---

## `NotSupportedException`

Indica che un’operazione non è supportata.

```csharp
void Esporta(string formato)
{
    if (formato == "DOCX")
    {
        throw new NotSupportedException("L'esportazione DOCX non è supportata.");
    }
}
```

---

## Riepilogo eccezioni comuni

```text
ArgumentException
  argomento non valido.

ArgumentNullException
  argomento null non consentito.

ArgumentOutOfRangeException
  argomento fuori intervallo.

InvalidOperationException
  operazione non valida nello stato corrente.

FormatException
  formato non valido.

DivideByZeroException
  divisione per zero.

NullReferenceException
  uso di un riferimento null.

OverflowException
  superamento dei limiti numerici.

NotImplementedException
  funzionalità non ancora implementata.

NotSupportedException
  funzionalità non supportata.
```

# Creare eccezioni personalizzate

Nella maggior parte dei casi conviene usare le eccezioni built-in di .NET.

Tuttavia, a volte può avere senso creare un’eccezione personalizzata, soprattutto quando vogliamo rappresentare un errore specifico del dominio applicativo.

Esempio:

```csharp
public class DocumentoScadutoException : Exception
{
    public DocumentoScadutoException()
    {
    }

    public DocumentoScadutoException(string message)
        : base(message)
    {
    }

    public DocumentoScadutoException(string message, Exception innerException)
        : base(message, innerException)
    {
    }
}
```

Uso:

```csharp
void VerificaDocumento(DateOnly dataScadenza)
{
    DateOnly oggi = DateOnly.FromDateTime(DateTime.Today);

    if (dataScadenza < oggi)
    {
        throw new DocumentoScadutoException("Il documento è scaduto.");
    }
}
```

---

## Quando creare un’eccezione personalizzata

Ha senso quando:

* l’errore ha un significato specifico nel dominio;
* vogliamo intercettarlo in modo distinto;
* vogliamo aggiungere proprietà specifiche;
* l’eccezione migliora la leggibilità del codice.

Esempio:

```csharp
try
{
    VerificaDocumento(dataScadenza);
}
catch (DocumentoScadutoException)
{
    Console.WriteLine("Impossibile procedere: documento scaduto.");
}
```

---

## Quando non crearla

Non serve creare una nuova eccezione per ogni errore.

Inutile:

```csharp
public class NomeVuotoException : Exception
{
}
```

In molti casi è più chiaro usare:

```csharp
throw new ArgumentException("Il nome è obbligatorio.", nameof(nome));
```

Regola pratica:

```text
Usa eccezioni built-in quando descrivono bene il problema.
Crea eccezioni personalizzate solo quando aggiungono significato reale.
```

---

# Errore atteso o bug?

Non tutti gli errori sono uguali.

Una distinzione importantissima è tra:

```text
errore atteso
bug
```

---

## Errore atteso

Un errore atteso è una situazione prevedibile nel normale uso del programma.

Esempi:

* l’utente inserisce un numero non valido;
* una ricerca non trova risultati;
* una password è sbagliata;
* un file scelto dall’utente non esiste;
* un documento è scaduto;
* un campo obbligatorio è vuoto.

Questi casi vanno gestiti in modo controllato.

Esempio:

```csharp
Console.Write("Età: ");
string? input = Console.ReadLine();

if (!int.TryParse(input, out int eta))
{
    Console.WriteLine("Età non valida.");
    return;
}
```

Qui non usiamo eccezioni, perché l’input non valido è prevedibile.

---

## Bug

Un bug è un errore nel programma.

Esempi:

* un oggetto che non dovrebbe essere `null` è `null`;
* una lista viene usata con un indice fuori range;
* una regola interna produce uno stato impossibile;
* un metodo viene chiamato con parametri che il codice non dovrebbe mai produrre;
* una dipendenza non è stata configurata.

Esempio:

```csharp
if (ordine is null)
{
    throw new InvalidOperationException("Ordine non inizializzato.");
}
```

Se `ordine` non dovrebbe mai essere null in quel punto, non conviene nascondere il problema.

---

## Differenza pratica

```text
Errore atteso
  va gestito e comunicato all’utente o al chiamante.

Bug
  va fatto emergere, registrato e corretto.
```

Esempio:

```csharp
// Errore atteso
if (!int.TryParse(input, out int eta))
{
    Console.WriteLine("Inserisci un numero valido.");
    return;
}

// Bug o stato non valido
if (servizioClienti is null)
{
    throw new InvalidOperationException("Servizio clienti non configurato.");
}
```

---

# Validazione input

La validazione input serve a controllare i dati prima di usarli.

Gli input possono arrivare da:

* console;
* file;
* database;
* API;
* form web;
* parametri di metodo;
* configurazione;
* messaggi esterni.

Non bisogna fidarsi ciecamente di nessun input esterno.

---

## Validazione semplice

```csharp
Console.Write("Nome: ");
string? nome = Console.ReadLine();

if (string.IsNullOrWhiteSpace(nome))
{
    Console.WriteLine("Il nome è obbligatorio.");
    return;
}
```

---

## Validazione numerica

```csharp
Console.Write("Età: ");
string? input = Console.ReadLine();

if (!int.TryParse(input, out int eta))
{
    Console.WriteLine("Età non valida.");
    return;
}

if (eta < 0 || eta > 130)
{
    Console.WriteLine("Età fuori intervallo.");
    return;
}
```

---

## Validazione nei metodi

Quando scriviamo un metodo, dobbiamo decidere se validare i parametri.

Esempio:

```csharp
decimal CalcolaSconto(decimal importo, decimal percentuale)
{
    if (importo < 0)
    {
        throw new ArgumentOutOfRangeException(nameof(importo), "L'importo non può essere negativo.");
    }

    if (percentuale < 0 || percentuale > 100)
    {
        throw new ArgumentOutOfRangeException(nameof(percentuale), "La percentuale deve essere tra 0 e 100.");
    }

    return importo * percentuale / 100m;
}
```

Qui l’eccezione è corretta perché il metodo non può lavorare con parametri non validi.

---

# Guard clauses

Le **guard clauses** sono controlli iniziali che proteggono il metodo da input o stati non validi.

Esempio senza guard clauses:

```csharp
decimal CalcolaTotale(decimal importo, decimal iva)
{
    if (importo >= 0)
    {
        if (iva >= 0)
        {
            return importo + (importo * iva / 100m);
        }
        else
        {
            throw new ArgumentOutOfRangeException(nameof(iva));
        }
    }
    else
    {
        throw new ArgumentOutOfRangeException(nameof(importo));
    }
}
```

Funziona, ma è inutilmente annidato.

Versione con guard clauses:

```csharp
decimal CalcolaTotale(decimal importo, decimal iva)
{
    if (importo < 0)
    {
        throw new ArgumentOutOfRangeException(nameof(importo), "L'importo non può essere negativo.");
    }

    if (iva < 0)
    {
        throw new ArgumentOutOfRangeException(nameof(iva), "L'IVA non può essere negativa.");
    }

    return importo + (importo * iva / 100m);
}
```

Molto più leggibile.

---

## Guard clauses con `ArgumentNullException.ThrowIfNull`

Nelle versioni moderne di .NET possiamo usare anche metodi helper.

Esempio:

```csharp
void StampaCliente(Cliente cliente)
{
    ArgumentNullException.ThrowIfNull(cliente);

    Console.WriteLine(cliente.Nome);
}
```

Equivale concettualmente a:

```csharp
void StampaCliente(Cliente cliente)
{
    if (cliente is null)
    {
        throw new ArgumentNullException(nameof(cliente));
    }

    Console.WriteLine(cliente.Nome);
}
```

Questa forma è compatta e leggibile per controlli comuni.

# Logging base

Gestire un errore non significa solo mostrarlo all’utente.

Spesso dobbiamo anche **registrarlo**.

Il logging serve a conservare informazioni utili per capire cosa è successo.

In una console app semplice possiamo iniziare con:

```csharp
try
{
    int numero = int.Parse("ciao");
}
catch (Exception ex)
{
    Console.WriteLine("Errore durante la conversione.");
    Console.Error.WriteLine(ex);
}
```

`Console.Error` scrive sullo stream di errore standard, separato dallo stream normale di output.

---

## Non mostrare dettagli tecnici all’utente finale

In un programma destinato a utenti finali, non è sempre opportuno mostrare dettagli tecnici.

Esempio poco adatto all’utente:

```text
System.NullReferenceException: Object reference not set to an instance of an object
   at Program.Main() in C:\Progetti\App\Program.cs:line 12
```

Meglio mostrare un messaggio comprensibile:

```text
Si è verificato un errore imprevisto. Contattare l’assistenza.
```

Ma nel log tecnico dobbiamo conservare il dettaglio.

---

## Logging minimo

Per ora possiamo immaginare una funzione semplice:

```csharp
void LogErrore(Exception ex)
{
    Console.Error.WriteLine("ERRORE");
    Console.Error.WriteLine(DateTime.Now);
    Console.Error.WriteLine(ex);
}
```

Uso:

```csharp
try
{
    EseguiOperazione();
}
catch (Exception ex)
{
    LogErrore(ex);
    Console.WriteLine("Operazione non riuscita.");
}
```

Più avanti, in applicazioni reali, useremo strumenti come:

* `ILogger`;
* logging strutturato;
* file di log;
* sistemi centralizzati;
* Application Insights;
* Serilog;
* NLog.

Per ora è importante il principio:

```text
Messaggio per l’utente
  chiaro e non tecnico.

Log per lo sviluppatore
  dettagliato e utile alla diagnosi.
```

# Eccezioni e async/await

Le eccezioni funzionano anche con i metodi asincroni.

Esempio:

```csharp
async Task EseguiAsync()
{
    await Task.Delay(100);
    throw new InvalidOperationException("Errore asincrono.");
}
```

Possiamo gestire l’eccezione con `try/catch` usando `await`.

```csharp
try
{
    await EseguiAsync();
}
catch (InvalidOperationException ex)
{
    Console.WriteLine(ex.Message);
}
```

Output:

```text
Errore asincrono.
```

Il punto importante è questo:

> Se un metodo asincrono restituisce `Task` o `Task<T>`, l’eccezione viene catturata nel task e rilanciata quando usiamo `await`.

---

## Eccezioni senza await

Attenzione a questo caso:

```csharp
try
{
    EseguiAsync();
}
catch (Exception ex)
{
    Console.WriteLine(ex.Message);
}
```

Qui manca `await`.

Il metodo asincrono viene avviato, ma il `try/catch` non intercetta correttamente l’eccezione nel modo atteso.

Forma corretta:

```csharp
try
{
    await EseguiAsync();
}
catch (Exception ex)
{
    Console.WriteLine(ex.Message);
}
```

Regola pratica:

```text
Se chiami un metodo async, quasi sempre devi usare await.
```

---

## Evitare `async void`

I metodi asincroni dovrebbero restituire:

```text
Task
Task<T>
ValueTask
ValueTask<T>
```

Evitare:

```csharp
async void MetodoAsync()
{
    await Task.Delay(100);
    throw new Exception("Errore");
}
```

`async void` è difficile da gestire perché le eccezioni non vengono propagate al chiamante nello stesso modo dei metodi `Task`.

Eccezione importante: gli event handler nelle UI spesso devono essere `async void`.

Esempio accettabile:

```csharp
private async void Button_Click(object sender, EventArgs e)
{
    await SalvaAsync();
}
```

Nel codice applicativo normale, però, preferire:

```csharp
async Task SalvaAsync()
{
    await Task.Delay(100);
}
```

---

## Eccezioni con Task.WhenAll

Quando eseguiamo più task insieme:

```csharp
await Task.WhenAll(task1, task2, task3);
```

se uno o più task falliscono, viene generata un’eccezione.

Esempio:

```csharp
Task t1 = Operazione1Async();
Task t2 = Operazione2Async();

try
{
    await Task.WhenAll(t1, t2);
}
catch (Exception ex)
{
    Console.WriteLine("Almeno una operazione è fallita.");
    Console.WriteLine(ex.Message);
}
```

In scenari avanzati può essere necessario ispezionare i singoli task per capire quali sono falliti.

Approfondiremo async/await in un capitolo dedicato. Per ora basta ricordare:

```text
try/catch funziona con async/await,
ma bisogna usare await.
```

---

# Errori da non nascondere

Uno degli errori più gravi nella gestione delle eccezioni è nasconderle.

Esempio pessimo:

```csharp
try
{
    EseguiOperazione();
}
catch
{
}
```

Questo codice cattura qualunque errore e non fa nulla.

Il programma continua come se niente fosse, ma qualcosa è andato storto.

Questo tipo di codice rende i bug difficili da individuare.

---

## Catch vuoto

Da evitare quasi sempre:

```csharp
catch
{
}
```

Se proprio dobbiamo ignorare un errore, deve esserci una ragione chiara e possibilmente un commento.

Esempio accettabile in casi particolari:

```csharp
try
{
    File.Delete(fileTemporaneo);
}
catch (IOException ex)
{
    LogErrore(ex);
    // Non blocchiamo l'applicazione se la cancellazione del file temporaneo fallisce.
}
```

Qui l’errore non viene completamente nascosto: viene almeno registrato.

---

## Catturare Exception troppo presto

Anche questo può essere problematico:

```csharp
try
{
    EseguiOperazione();
}
catch (Exception)
{
    Console.WriteLine("Errore.");
}
```

Se non facciamo log e non sappiamo cosa è successo, abbiamo perso informazioni importanti.

Meglio:

```csharp
try
{
    EseguiOperazione();
}
catch (Exception ex)
{
    LogErrore(ex);
    Console.WriteLine("Operazione non riuscita.");
}
```

---

## Trasformare bug in valori innocui

Esempio pericoloso:

```csharp
decimal CalcolaTotale()
{
    try
    {
        return EseguiCalcolo();
    }
    catch
    {
        return 0m;
    }
}
```

Se il calcolo fallisce, restituire `0` potrebbe produrre dati sbagliati.

In un gestionale, un totale sbagliato è spesso peggio di un errore visibile.

Meglio far emergere il problema o gestirlo esplicitamente.

---

# Quando non usare le eccezioni

Le eccezioni non dovrebbero essere usate per controllare il flusso normale del programma.

Sono pensate per situazioni anomale, non per casi ordinari e prevedibili.

---

## Non usare eccezioni per validare input comune

Sconsigliato:

```csharp
try
{
    int eta = int.Parse(input);
    Console.WriteLine(eta);
}
catch
{
    Console.WriteLine("Età non valida.");
}
```

Meglio:

```csharp
if (int.TryParse(input, out int eta))
{
    Console.WriteLine(eta);
}
else
{
    Console.WriteLine("Età non valida.");
}
```

Perché l’input non valido dell’utente è prevedibile. Non è davvero eccezionale.

---

## Non usare eccezioni per cercare valori

Sconsigliato:

```csharp
Cliente TrovaCliente(int id)
{
    Cliente? cliente = CercaCliente(id);

    if (cliente is null)
    {
        throw new Exception("Cliente non trovato.");
    }

    return cliente;
}
```

Dipende dal contesto.

Se “cliente non trovato” è un caso normale, meglio:

```csharp
Cliente? TrovaCliente(int id)
{
    return CercaCliente(id);
}
```

Oppure, con pattern più espliciti che vedremo in seguito:

```csharp
bool ProvaTrovaCliente(int id, out Cliente? cliente)
{
    cliente = CercaCliente(id);
    return cliente is not null;
}
```

Se invece il cliente doveva assolutamente esistere e la sua assenza indica uno stato incoerente, allora un’eccezione può avere senso.

---

## Non usare eccezioni per condizioni normali

Esempio sconsigliato:

```csharp
try
{
    decimal sconto = CalcolaSconto(cliente);
}
catch
{
    sconto = 0;
}
```

Meglio esprimere chiaramente la regola:

```csharp
decimal sconto = cliente.HaDirittoASconto
    ? CalcolaSconto(cliente)
    : 0m;
```

---

## Regola pratica

```text
Usa eccezioni per situazioni anomale.

Usa condizioni, TryParse, valori nullable o risultati espliciti
per casi previsti nel normale flusso del programma.
```

---

# Esempio completo del capitolo

Costruiamo un piccolo programma che legge un importo e una percentuale di sconto.

Vogliamo gestire:

* input non numerico;
* importo negativo;
* percentuale fuori intervallo;
* calcolo corretto;
* logging base in caso di errore imprevisto.

```csharp
try
{
    decimal importo = LeggiImporto();
    decimal percentualeSconto = LeggiPercentualeSconto();

    decimal sconto = CalcolaSconto(importo, percentualeSconto);
    decimal totale = importo - sconto;

    Console.WriteLine($"Importo: {importo}");
    Console.WriteLine($"Sconto: {sconto}");
    Console.WriteLine($"Totale: {totale}");
}
catch (ArgumentOutOfRangeException ex)
{
    Console.WriteLine(ex.Message);
}
catch (Exception ex)
{
    LogErrore(ex);
    Console.WriteLine("Si è verificato un errore imprevisto.");
}

decimal LeggiImporto()
{
    Console.Write("Importo: ");
    string? input = Console.ReadLine();

    if (!decimal.TryParse(input, out decimal importo))
    {
        throw new ArgumentException("L'importo inserito non è valido.");
    }

    if (importo < 0)
    {
        throw new ArgumentOutOfRangeException(nameof(importo), "L'importo non può essere negativo.");
    }

    return importo;
}

decimal LeggiPercentualeSconto()
{
    Console.Write("Sconto percentuale: ");
    string? input = Console.ReadLine();

    if (!decimal.TryParse(input, out decimal percentuale))
    {
        throw new ArgumentException("La percentuale inserita non è valida.");
    }

    if (percentuale < 0 || percentuale > 100)
    {
        throw new ArgumentOutOfRangeException(nameof(percentuale), "La percentuale deve essere tra 0 e 100.");
    }

    return percentuale;
}

decimal CalcolaSconto(decimal importo, decimal percentuale)
{
    if (importo < 0)
    {
        throw new ArgumentOutOfRangeException(nameof(importo), "L'importo non può essere negativo.");
    }

    if (percentuale < 0 || percentuale > 100)
    {
        throw new ArgumentOutOfRangeException(nameof(percentuale), "La percentuale deve essere tra 0 e 100.");
    }

    return importo * percentuale / 100m;
}

void LogErrore(Exception ex)
{
    Console.Error.WriteLine("ERRORE");
    Console.Error.WriteLine(DateTime.Now);
    Console.Error.WriteLine(ex);
}
```

Questo esempio mostra diversi concetti:

* `try/catch`;
* eccezioni specifiche;
* `throw`;
* guard clauses;
* validazione input;
* logging base;
* separazione dei metodi.

---

## Nota didattica sull’esempio

Nel programma precedente abbiamo scelto di lanciare eccezioni anche dentro `LeggiImporto` e `LeggiPercentualeSconto`.

In un’app console semplice, si potrebbe anche gestire tutto con `TryParse` e `return`.

Per esempio:

```csharp
if (!decimal.TryParse(input, out decimal importo))
{
    Console.WriteLine("Importo non valido.");
    return;
}
```

Entrambi gli approcci possono avere senso.

La differenza è questa:

```text
Codice interattivo vicino all’utente
  spesso conviene gestire l’errore con if, TryParse e messaggi chiari.

Metodo di dominio o metodo riutilizzabile
  spesso conviene rifiutare parametri non validi con eccezioni.
```

# Errori comuni

## Usare `catch` vuoto

Sbagliato:

```csharp
try
{
    EseguiOperazione();
}
catch
{
}
```

Il programma nasconde l’errore.

Meglio:

```csharp
try
{
    EseguiOperazione();
}
catch (Exception ex)
{
    LogErrore(ex);
    Console.WriteLine("Operazione non riuscita.");
}
```

---

## Catturare tutto senza motivo

Sospetto:

```csharp
try
{
    int risultato = 10 + 20;
}
catch (Exception ex)
{
    Console.WriteLine(ex.Message);
}
```

Questo codice non ha bisogno di un `try/catch`.

Non bisogna circondare ogni riga con gestione eccezioni.

---

## Usare eccezioni al posto di `TryParse`

Sconsigliato:

```csharp
try
{
    int numero = int.Parse(input);
}
catch
{
    Console.WriteLine("Numero non valido.");
}
```

Meglio:

```csharp
if (!int.TryParse(input, out int numero))
{
    Console.WriteLine("Numero non valido.");
}
```

---

## Rilanciare con `throw ex`

Sbagliato:

```csharp
catch (Exception ex)
{
    throw ex;
}
```

Meglio:

```csharp
catch
{
    throw;
}
```

---

## Mostrare stack trace all’utente finale

Sconsigliato:

```csharp
catch (Exception ex)
{
    Console.WriteLine(ex);
}
```

In un’app per utenti finali, meglio:

```csharp
catch (Exception ex)
{
    LogErrore(ex);
    Console.WriteLine("Si è verificato un errore imprevisto.");
}
```

---

## Restituire valori falsi in caso di errore

Pericoloso:

```csharp
decimal CalcolaTotale()
{
    try
    {
        return EseguiCalcolo();
    }
    catch
    {
        return 0m;
    }
}
```

Se il calcolo fallisce, `0` potrebbe essere un dato sbagliato e pericoloso.

---

## Dimenticare `await`

Sbagliato:

```csharp
try
{
    SalvaAsync();
}
catch (Exception ex)
{
    Console.WriteLine(ex.Message);
}
```

Corretto:

```csharp
try
{
    await SalvaAsync();
}
catch (Exception ex)
{
    Console.WriteLine(ex.Message);
}
```

# Riepilogo

In questo capitolo abbiamo introdotto la gestione degli errori in C#.

Abbiamo visto che un’eccezione è un oggetto che rappresenta una condizione anomala durante l’esecuzione.

Abbiamo studiato:

```text
try
  contiene codice che può fallire.

catch
  intercetta e gestisce eccezioni.

finally
  esegue codice di pulizia sempre.

throw
  lancia un’eccezione.
```

Abbiamo visto alcune eccezioni built-in comuni:

```text
ArgumentException
ArgumentNullException
ArgumentOutOfRangeException
InvalidOperationException
FormatException
DivideByZeroException
NullReferenceException
OverflowException
NotImplementedException
NotSupportedException
```

Abbiamo distinto:

```text
errore atteso
  situazione prevedibile nel normale uso del programma.

bug
  errore logico o stato incoerente del programma.
```

Abbiamo introdotto pratiche importanti:

```text
validazione input
  controllare i dati prima di usarli.

guard clauses
  uscire subito quando i dati non sono validi.

logging
  registrare informazioni tecniche sugli errori.

async/await
  usare await per intercettare correttamente eccezioni asincrone.

non nascondere errori
  evitare catch vuoti e valori fittizi.
```

La regola più importante è questa:

```text
Le eccezioni servono per condizioni anomale.
Non devono sostituire il normale controllo del flusso.
```

Prima di lavorare con file, database, rete, API e applicazioni web, è essenziale avere questo modello mentale. Ogni tecnologia che useremo nei prossimi capitoli potrà fallire in modi diversi, ma il principio resterà lo stesso: validare ciò che è prevedibile, gestire ciò che è recuperabile, registrare ciò che è diagnostico e non nascondere mai i bug.

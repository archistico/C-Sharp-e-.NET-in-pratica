# 02.06 — Classi, oggetti e incapsulamento

Fino a questo punto abbiamo lavorato con variabili, tipi fondamentali, condizioni, cicli, metodi ed eccezioni. Sono tutti elementi indispensabili, ma non bastano per costruire applicazioni reali.

Un programma gestionale, una Web API, un’app desktop o un sistema di servizi non lavora solo con numeri e stringhe isolate. Lavora con concetti del dominio:

* cliente;
* documento;
* ordine;
* prodotto;
* fattura;
* prenotazione;
* movimento;
* utente;
* pagamento.

In C# questi concetti vengono spesso rappresentati attraverso **classi** e **oggetti**.

Questo è il primo vero capitolo di programmazione orientata agli oggetti.

L’obiettivo non è solo imparare la sintassi di una classe, ma capire una cosa molto più importante:

> una classe serve a rappresentare un concetto, proteggere i suoi dati e impedire che l’oggetto finisca in uno stato non valido.

---

# Classi

Una **classe** è un modello, uno schema, una definizione.

Descrive quali dati e quali comportamenti avranno gli oggetti creati a partire da essa.

Esempio:

```csharp id="hwzwh5"
public class Cliente
{
    public string Nome { get; set; } = "";
    public string Cognome { get; set; } = "";
    public int Eta { get; set; }
}
```

Questa classe descrive il concetto di `Cliente`.

Dice che un cliente ha:

```text id="y4hsxv"
Nome
Cognome
Eta
```

La classe non è ancora un cliente concreto. È il modello da cui possiamo creare clienti concreti.

---

## Classe come tipo personalizzato

Quando definiamo una classe, stiamo creando un nuovo tipo.

Dopo aver scritto:

```csharp id="g6gz5z"
public class Cliente
{
    public string Nome { get; set; } = "";
    public string Cognome { get; set; } = "";
}
```

possiamo dichiarare variabili di tipo `Cliente`:

```csharp id="usgrvw"
Cliente cliente;
```

Esattamente come possiamo dichiarare:

```csharp id="cpptml"
int numero;
string nome;
DateTime data;
```

La differenza è che `Cliente` è un tipo definito da noi.

---

# Oggetti

Un **oggetto** è un’istanza concreta di una classe.

Se la classe è il modello, l’oggetto è l’elemento reale creato a partire da quel modello.

Esempio:

```csharp id="jg8ezg"
Cliente cliente = new Cliente();

cliente.Nome = "Emilie";
cliente.Cognome = "Rollandin";
cliente.Eta = 42;

Console.WriteLine($"{cliente.Nome} {cliente.Cognome}");
```

Qui:

```text id="sjn9qm"
Cliente
  è la classe, cioè il tipo.

cliente
  è una variabile che contiene un riferimento a un oggetto Cliente.

new Cliente()
  crea un nuovo oggetto.
```

---

## Creare oggetti con object initializer

Possiamo creare e inizializzare un oggetto in modo più compatto:

```csharp id="xhubnj"
Cliente cliente = new Cliente
{
    Nome = "Emilie",
    Cognome = "Rollandin",
    Eta = 42
};
```

Questa sintassi si chiama **object initializer**.

È molto usata quando una classe espone proprietà pubbliche impostabili.

---

## Più oggetti della stessa classe

Da una classe possiamo creare molti oggetti.

```csharp id="y57m47"
Cliente cliente1 = new Cliente
{
    Nome = "Emilie",
    Cognome = "Rollandin",
    Eta = 42
};

Cliente cliente2 = new Cliente
{
    Nome = "Mario",
    Cognome = "Rossi",
    Eta = 35
};
```

`cliente1` e `cliente2` sono due oggetti diversi, ma entrambi sono istanze della classe `Cliente`.

---

# Campi

Un **campo** è una variabile dichiarata dentro una classe.

Esempio:

```csharp id="7w0323"
public class Contatore
{
    private int valore;
}
```

Qui `valore` è un campo.

I campi rappresentano lo stato interno dell’oggetto.

---

## Campi privati

Molto spesso i campi sono `private`.

```csharp id="wxg3y9"
public class Contatore
{
    private int valore;

    public void Incrementa()
    {
        valore++;
    }

    public int OttieniValore()
    {
        return valore;
    }
}
```

Uso:

```csharp id="yn8jga"
Contatore contatore = new Contatore();

contatore.Incrementa();
contatore.Incrementa();

Console.WriteLine(contatore.OttieniValore());
```

Output:

```text id="1pnwv0"
2
```

Il campo `valore` non è accessibile direttamente dall’esterno.

Questo non è consentito:

```csharp id="mf95zn"
contatore.valore = 10; // errore
```

Il campo è privato.

---

## Perché non rendere tutto pubblico?

Potremmo scrivere:

```csharp id="nvt4sd"
public class Contatore
{
    public int Valore;
}
```

Uso:

```csharp id="wn3exq"
Contatore contatore = new Contatore();
contatore.Valore = -100;
```

Il problema è che chiunque può mettere l’oggetto in uno stato non valido.

Un contatore negativo potrebbe non avere senso.

Meglio proteggere lo stato:

```csharp id="ha4nzh"
public class Contatore
{
    private int valore;

    public int Valore => valore;

    public void Incrementa()
    {
        valore++;
    }
}
```

Ora il valore può essere letto, ma non modificato arbitrariamente.

---

# Proprietà

Le **proprietà** sono il modo più comune in C# per esporre dati di un oggetto.

Esempio:

```csharp id="71go48"
public class Cliente
{
    public string Nome { get; set; } = "";
}
```

Questa è una proprietà automatica.

Contiene:

```text id="czqz73"
get
  permette di leggere il valore.

set
  permette di assegnare il valore.
```

Uso:

```csharp id="ovc4lo"
Cliente cliente = new Cliente();

cliente.Nome = "Emilie";

Console.WriteLine(cliente.Nome);
```

---

## Proprietà automatica

Una proprietà automatica è compatta:

```csharp id="4ii0tf"
public string Nome { get; set; } = "";
```

Il compilatore crea internamente un campo nascosto, chiamato spesso **backing field**.

Concettualmente, possiamo immaginare qualcosa di simile:

```csharp id="6e657e"
private string nome = "";

public string Nome
{
    get
    {
        return nome;
    }
    set
    {
        nome = value;
    }
}
```

La proprietà automatica evita di scrivere codice ripetitivo quando non serve logica particolare.

---

## Proprietà con logica

Quando vogliamo controllare il valore assegnato, possiamo scrivere il `set` manualmente.

```csharp id="f3ed9h"
public class Cliente
{
    private string nome = "";

    public string Nome
    {
        get
        {
            return nome;
        }
        set
        {
            if (string.IsNullOrWhiteSpace(value))
            {
                throw new ArgumentException("Il nome è obbligatorio.", nameof(value));
            }

            nome = value;
        }
    }
}
```

Qui impediamo di assegnare un nome vuoto.

Uso:

```csharp id="b5hocj"
Cliente cliente = new Cliente();

cliente.Nome = "Emilie"; // valido

cliente.Nome = ""; // eccezione
```

---

## `value`

Dentro il `set`, la parola `value` rappresenta il valore che stiamo assegnando.

```csharp id="9vx673"
set
{
    nome = value;
}
```

Se scriviamo:

```csharp id="vg5d54"
cliente.Nome = "Emilie";
```

all’interno del `set`, `value` vale `"Emilie"`.

---

## Proprietà in sola lettura

Una proprietà può avere solo `get`.

```csharp id="pwl7yz"
public class Cliente
{
    public string NomeCompleto { get; }

    public Cliente(string nome, string cognome)
    {
        NomeCompleto = $"{nome} {cognome}";
    }
}
```

Dall’esterno possiamo leggere:

```csharp id="u01fmd"
Console.WriteLine(cliente.NomeCompleto);
```

ma non assegnare:

```csharp id="rbb0or"
cliente.NomeCompleto = "Altro"; // errore
```

---

## Proprietà calcolata

Una proprietà può calcolare il valore al momento della lettura.

```csharp id="1ry67s"
public class Cliente
{
    public string Nome { get; set; } = "";
    public string Cognome { get; set; } = "";

    public string NomeCompleto
    {
        get
        {
            return $"{Nome} {Cognome}";
        }
    }
}
```

Uso:

```csharp id="5zxtkw"
Cliente cliente = new Cliente
{
    Nome = "Emilie",
    Cognome = "Rollandin"
};

Console.WriteLine(cliente.NomeCompleto);
```

Output:

```text id="d7gjj5"
Emilie Rollandin
```

La proprietà `NomeCompleto` non memorizza un dato separato. Lo calcola usando `Nome` e `Cognome`.

Forma compatta:

```csharp id="1g35ym"
public string NomeCompleto => $"{Nome} {Cognome}";
```

---

# Metodi

I **metodi** rappresentano i comportamenti di un oggetto.

Esempio:

```csharp id="rawy2v"
public class Cliente
{
    public string Nome { get; set; } = "";
    public string Cognome { get; set; } = "";
    public bool Attivo { get; private set; } = true;

    public void Disattiva()
    {
        Attivo = false;
    }

    public string OttieniNomeCompleto()
    {
        return $"{Nome} {Cognome}";
    }
}
```

Uso:

```csharp id="owxjif"
Cliente cliente = new Cliente
{
    Nome = "Emilie",
    Cognome = "Rollandin"
};

Console.WriteLine(cliente.OttieniNomeCompleto());

cliente.Disattiva();

Console.WriteLine(cliente.Attivo);
```

Output:

```text id="xi8qz5"
Emilie Rollandin
False
```

I metodi permettono di esprimere azioni:

```text id="zo5lgz"
Disattiva
Attiva
Conferma
Annulla
CalcolaTotale
AggiungiRiga
RimuoviRiga
```

---

## Proprietà o metodo?

A volte non è immediato capire se usare una proprietà o un metodo.

Regola pratica:

```text id="9wjt5e"
Proprietà
  espone uno stato o un valore semplice.

Metodo
  esegue un’azione o un calcolo significativo.
```

Esempio proprietà:

```csharp id="756eyu"
public string NomeCompleto => $"{Nome} {Cognome}";
```

Esempio metodo:

```csharp id="himw43"
public decimal CalcolaTotaleConIva(decimal aliquota)
{
    return Totale * (1 + aliquota / 100m);
}
```

Se l’operazione richiede parametri, ha effetti collaterali o può fallire, spesso è meglio un metodo.

---

# Costruttori

Un **costruttore** è un metodo speciale chiamato quando creiamo un oggetto con `new`.

Serve a inizializzare l’oggetto.

Esempio:

```csharp id="1w6svq"
public class Cliente
{
    public string Nome { get; }
    public string Cognome { get; }

    public Cliente(string nome, string cognome)
    {
        Nome = nome;
        Cognome = cognome;
    }
}
```

Uso:

```csharp id="6f2ibz"
Cliente cliente = new Cliente("Emilie", "Rollandin");

Console.WriteLine(cliente.Nome);
Console.WriteLine(cliente.Cognome);
```

Il costruttore ha lo stesso nome della classe e non ha tipo di ritorno. Microsoft descrive i costruttori proprio come membri chiamati quando viene creata un’istanza di una classe o struct, utili per impostare valori iniziali e garantire che l’oggetto nasca in uno stato valido. ([Microsoft Learn][1])

---

## Costruttore con validazione

Un costruttore è il posto ideale per controllare che i dati iniziali siano validi.

```csharp id="kj42g0"
public class Cliente
{
    public string Nome { get; }
    public string Cognome { get; }

    public Cliente(string nome, string cognome)
    {
        if (string.IsNullOrWhiteSpace(nome))
        {
            throw new ArgumentException("Il nome è obbligatorio.", nameof(nome));
        }

        if (string.IsNullOrWhiteSpace(cognome))
        {
            throw new ArgumentException("Il cognome è obbligatorio.", nameof(cognome));
        }

        Nome = nome;
        Cognome = cognome;
    }
}
```

Ora non possiamo creare un cliente senza nome:

```csharp id="lkv0n4"
Cliente cliente = new Cliente("", "Rollandin"); // eccezione
```

Questo è un primo esempio di oggetto che protegge la propria validità.

---

## Costruttore vuoto

Se non dichiariamo nessun costruttore, C# fornisce un costruttore vuoto implicito.

```csharp id="yzdipf"
public class Cliente
{
    public string Nome { get; set; } = "";
}
```

Possiamo scrivere:

```csharp id="ap7wxe"
Cliente cliente = new Cliente();
```

Ma se dichiariamo un costruttore con parametri:

```csharp id="6rzlwd"
public class Cliente
{
    public string Nome { get; }

    public Cliente(string nome)
    {
        Nome = nome;
    }
}
```

il costruttore vuoto non viene più creato automaticamente.

Questo non compila:

```csharp id="nsgbbh"
Cliente cliente = new Cliente(); // errore
```

Serve passare il nome:

```csharp id="txwxex"
Cliente cliente = new Cliente("Emilie");
```

---

## Più costruttori

Una classe può avere più costruttori.

```csharp id="o8egdu"
public class Cliente
{
    public string Nome { get; }
    public string Cognome { get; }
    public bool Attivo { get; }

    public Cliente(string nome, string cognome)
        : this(nome, cognome, true)
    {
    }

    public Cliente(string nome, string cognome, bool attivo)
    {
        Nome = nome;
        Cognome = cognome;
        Attivo = attivo;
    }
}
```

Il primo costruttore richiama il secondo usando:

```csharp id="dtu4t0"
: this(nome, cognome, true)
```

Questo evita duplicazione.

---

# Costruttori primari

I **costruttori primari** permettono di dichiarare i parametri del costruttore direttamente nella dichiarazione della classe.

Esempio:

```csharp id="ng6c1a"
public class Cliente(string nome, string cognome)
{
    public string Nome { get; } = nome;
    public string Cognome { get; } = cognome;

    public string NomeCompleto => $"{Nome} {Cognome}";
}
```

Uso:

```csharp id="z9r3dk"
Cliente cliente = new Cliente("Emilie", "Rollandin");

Console.WriteLine(cliente.NomeCompleto);
```

I costruttori primari sono stati introdotti per classi e struct in C# 12 e permettono di rendere disponibili i parametri del costruttore nel corpo del tipo. ([Microsoft Learn][2])

---

## Costruttore primario con validazione

Possiamo comunque validare i valori.

Un approccio è usare metodi statici privati:

```csharp id="xyw3ft"
public class Cliente(string nome, string cognome)
{
    public string Nome { get; } = ValidaTesto(nome, nameof(nome));
    public string Cognome { get; } = ValidaTesto(cognome, nameof(cognome));

    public string NomeCompleto => $"{Nome} {Cognome}";

    private static string ValidaTesto(string valore, string nomeParametro)
    {
        if (string.IsNullOrWhiteSpace(valore))
        {
            throw new ArgumentException("Il valore è obbligatorio.", nomeParametro);
        }

        return valore;
    }
}
```

I costruttori primari sono comodi, ma non sempre rendono il codice più chiaro. Se la logica di costruzione è articolata, un costruttore tradizionale può essere più leggibile.

---

## Quando usare i costruttori primari

Sono adatti per classi semplici, dove i parametri iniziali vengono usati direttamente per inizializzare proprietà o campi.

Esempio buono:

```csharp id="48631v"
public class Punto(decimal x, decimal y)
{
    public decimal X { get; } = x;
    public decimal Y { get; } = y;
}
```

Meno adatto:

```csharp id="8itga4"
public class Ordine(/* molti parametri */)
{
    // molte validazioni
    // molte trasformazioni
    // molte regole
}
```

Regola pratica:

```text id="hp4mxv"
Costruttore primario
  ottimo per inizializzazioni semplici.

Costruttore tradizionale
  spesso migliore quando ci sono molte regole.
```

# Proprietà `required`

Le proprietà `required` obbligano chi crea un oggetto a inizializzare determinati membri.

Esempio:

```csharp id="v1z5kz"
public class Cliente
{
    public required string Nome { get; set; }
    public required string Cognome { get; set; }
    public int Eta { get; set; }
}
```

Uso corretto:

```csharp id="i8ronc"
Cliente cliente = new Cliente
{
    Nome = "Emilie",
    Cognome = "Rollandin",
    Eta = 42
};
```

Uso non corretto:

```csharp id="f2kk8w"
Cliente cliente = new Cliente
{
    Nome = "Emilie"
};
```

Il compilatore segnala che `Cognome` deve essere valorizzato.

---

## Che cosa risolve required

Senza `required`, possiamo scrivere:

```csharp id="r0ziat"
public class Cliente
{
    public string Nome { get; set; } = "";
    public string Cognome { get; set; } = "";
}
```

E poi:

```csharp id="hotdf4"
Cliente cliente = new Cliente();
```

L’oggetto esiste, ma forse non ha senso: un cliente senza nome e cognome potrebbe essere inconsistente.

Con `required` diciamo al compilatore:

> chi crea l’oggetto deve fornire questi valori.

---

## `required` non sostituisce la validazione

Attenzione: `required` obbliga a inizializzare una proprietà, ma non garantisce che il valore sia corretto.

Questo compila:

```csharp id="ddnqgp"
Cliente cliente = new Cliente
{
    Nome = "",
    Cognome = ""
};
```

Le proprietà sono state valorizzate, ma i valori sono vuoti.

Quindi, per regole forti, servono ancora validazione e incapsulamento.

Regola pratica:

```text id="ogwjl3"
required
  garantisce che un valore venga fornito.

validazione
  garantisce che il valore abbia senso.
```

---

# Campo `field` nelle proprietà

Nelle versioni moderne di C#, il linguaggio introduce la keyword contestuale `field` per accedere al backing field generato dal compilatore dentro gli accessor di una proprietà. In C# 14, `field` permette di aggiungere logica a `get` o `set` senza dichiarare manualmente un campo privato separato. ([Microsoft Learn][3])

Prima di questa funzionalità, se volevamo validare una proprietà dovevamo scrivere un campo esplicito.

Esempio classico:

```csharp id="y2mpyz"
public class Cliente
{
    private string nome = "";

    public string Nome
    {
        get => nome;
        set
        {
            if (string.IsNullOrWhiteSpace(value))
            {
                throw new ArgumentException("Il nome è obbligatorio.", nameof(value));
            }

            nome = value;
        }
    }
}
```

Con `field`, il codice può diventare più compatto:

```csharp id="4kk9oj"
public class Cliente
{
    public string Nome
    {
        get;
        set
        {
            if (string.IsNullOrWhiteSpace(value))
            {
                throw new ArgumentException("Il nome è obbligatorio.", nameof(value));
            }

            field = value;
        }
    } = "";
}
```

Qui `field` rappresenta il backing field generato dal compilatore.

---

## Esempio con normalizzazione

```csharp id="jqzcn5"
public class Cliente
{
    public string Nome
    {
        get;
        set => field = value.Trim();
    } = "";
}
```

Se scriviamo:

```csharp id="pi5n18"
cliente.Nome = "  Emilie  ";
```

la proprietà salverà:

```text id="t1ut1o"
Emilie
```

La documentazione Microsoft mostra proprio `field` come modo per scrivere proprietà con logica negli accessor senza dichiarare manualmente un backing field. ([Microsoft Learn][4])

---

## Quando usare `field`

`field` è utile quando:

* serve una validazione semplice;
* serve normalizzare un valore;
* serve logica nel `set`;
* non vogliamo scrivere manualmente un campo privato.

Per codice che deve essere compatibile con versioni C# precedenti alla 14, è meglio usare il campo privato esplicito.

# Modificatori di accesso

I modificatori di accesso controllano da dove un membro può essere usato.

I più importanti sono:

```text id="e3996t"
public
private
protected
internal
protected internal
```

Sono fondamentali per l’incapsulamento.

---

# `public`

Un membro `public` è accessibile da qualsiasi codice che possa vedere il tipo.

```csharp id="by8ypy"
public class Cliente
{
    public string Nome { get; set; } = "";
}
```

Uso:

```csharp id="vjipow"
Cliente cliente = new Cliente();
cliente.Nome = "Emilie";
```

`public` va usato per ciò che vogliamo esporre davvero.

---

# `private`

Un membro `private` è accessibile solo dentro la stessa classe.

```csharp id="yelr0g"
public class Contatore
{
    private int valore;

    public void Incrementa()
    {
        valore++;
    }

    public int Valore => valore;
}
```

Dall’esterno possiamo fare:

```csharp id="b6nxh3"
contatore.Incrementa();
Console.WriteLine(contatore.Valore);
```

ma non:

```csharp id="nhc56b"
contatore.valore = 10; // errore
```

`private` è il modificatore più importante per proteggere lo stato interno.

---

# `protected`

Un membro `protected` è accessibile nella classe stessa e nelle classi derivate.

```csharp id="n068u0"
public class Documento
{
    protected string Stato { get; set; } = "Bozza";
}

public class Fattura : Documento
{
    public void Conferma()
    {
        Stato = "Confermata";
    }
}
```

`protected` ha senso quando si lavora con ereditarietà.

Lo approfondiremo nel capitolo dedicato a classi derivate e polimorfismo.

---

# `internal`

Un membro o tipo `internal` è accessibile solo all’interno dello stesso assembly, cioè dello stesso progetto compilato.

```csharp id="bslobk"
internal class ServizioCalcoloInterno
{
}
```

È utile per classi che devono essere usate dentro una libreria, ma non esposte all’esterno.

Esempio:

```text id="727l5h"
Gestionale.Domain
  contiene classi pubbliche usate dagli altri progetti.

Gestionale.Domain
  contiene anche classi internal usate solo internamente.
```

---

# `protected internal`

`protected internal` combina due regole:

> accessibile da classi derivate oppure da codice nello stesso assembly.

Esempio:

```csharp id="jfz5kq"
public class BaseEntity
{
    protected internal void MarcaComeModificata()
    {
        // ...
    }
}
```

È meno comune nei primi progetti, ma può comparire in librerie e framework.

---

## Riepilogo modificatori

```text id="qh8h12"
public
  accessibile ovunque.

private
  accessibile solo nella stessa classe.

protected
  accessibile nella stessa classe e nelle classi derivate.

internal
  accessibile nello stesso assembly/progetto.

protected internal
  accessibile nello stesso assembly oppure da classi derivate.
```

Regola pratica:

```text id="7c9z1r"
Esponi il meno possibile.
Rendi public solo ciò che deve davvero essere usato dall’esterno.
```

# Incapsulamento

L’**incapsulamento** è uno dei concetti più importanti della programmazione orientata agli oggetti.

Significa:

> nascondere i dettagli interni di un oggetto ed esporre solo operazioni controllate.

Un oggetto non dovrebbe essere un sacchetto di dati modificabili liberamente.

Dovrebbe proteggere le proprie regole.

---

## Esempio senza incapsulamento

```csharp id="r9s1vs"
public class ContoCorrente
{
    public decimal Saldo;
}
```

Uso:

```csharp id="c0dgxv"
ContoCorrente conto = new ContoCorrente();

conto.Saldo = -1000000m;
```

Questo oggetto permette qualunque valore.

Se un saldo negativo non è consentito, la classe non sta proteggendo la propria regola.

---

## Esempio con incapsulamento

```csharp id="57z776"
public class ContoCorrente
{
    public decimal Saldo { get; private set; }

    public void Versa(decimal importo)
    {
        if (importo <= 0)
        {
            throw new ArgumentOutOfRangeException(nameof(importo), "L'importo deve essere positivo.");
        }

        Saldo += importo;
    }

    public void Preleva(decimal importo)
    {
        if (importo <= 0)
        {
            throw new ArgumentOutOfRangeException(nameof(importo), "L'importo deve essere positivo.");
        }

        if (importo > Saldo)
        {
            throw new InvalidOperationException("Saldo insufficiente.");
        }

        Saldo -= importo;
    }
}
```

Uso:

```csharp id="okmm06"
ContoCorrente conto = new ContoCorrente();

conto.Versa(100m);
conto.Preleva(30m);

Console.WriteLine(conto.Saldo);
```

Output:

```text id="x21lxc"
70
```

Questo non è consentito:

```csharp id="xwt2c8"
conto.Saldo = -1000m; // errore
```

Il saldo può cambiare solo tramite `Versa` e `Preleva`.

Questo è incapsulamento.

---

# Invarianti

Un’**invariante** è una regola che deve essere sempre vera per un oggetto valido.

Esempi:

```text id="3w6vz8"
Un conto corrente non può avere saldo negativo.
Un ordine confermato deve avere almeno una riga.
Una percentuale di sconto deve essere tra 0 e 100.
Una data di fine non può precedere la data di inizio.
Un cliente deve avere un nome.
```

L’incapsulamento serve a proteggere le invarianti.

---

## Esempio: periodo valido

```csharp id="w8lzco"
public class Periodo
{
    public DateOnly DataInizio { get; }
    public DateOnly DataFine { get; }

    public Periodo(DateOnly dataInizio, DateOnly dataFine)
    {
        if (dataFine < dataInizio)
        {
            throw new ArgumentException("La data di fine non può precedere la data di inizio.");
        }

        DataInizio = dataInizio;
        DataFine = dataFine;
    }

    public int DurataInGiorni()
    {
        return DataFine.DayNumber - DataInizio.DayNumber + 1;
    }
}
```

Uso:

```csharp id="32098g"
Periodo periodo = new Periodo(
    new DateOnly(2026, 4, 1),
    new DateOnly(2026, 4, 30));

Console.WriteLine(periodo.DurataInGiorni());
```

Questo non è consentito:

```csharp id="lw54og"
Periodo periodo = new Periodo(
    new DateOnly(2026, 4, 30),
    new DateOnly(2026, 4, 1));
```

La classe impedisce di creare un periodo inconsistente.

# Oggetti validi e oggetti inconsistenti

Un oggetto è **valido** quando rispetta tutte le regole necessarie al suo corretto funzionamento.

Un oggetto è **inconsistente** quando contiene dati incompatibili con quelle regole.

---

## Oggetto inconsistente

```csharp id="jyo48q"
public class RigaDocumento
{
    public string Descrizione { get; set; } = "";
    public int Quantita { get; set; }
    public decimal PrezzoUnitario { get; set; }
}
```

Uso:

```csharp id="0ieewq"
RigaDocumento riga = new RigaDocumento
{
    Descrizione = "",
    Quantita = -5,
    PrezzoUnitario = -10m
};
```

Questo oggetto è tecnicamente creato, ma non ha senso.

---

## Oggetto più protetto

```csharp id="9747tb"
public class RigaDocumento
{
    public string Descrizione { get; }
    public int Quantita { get; }
    public decimal PrezzoUnitario { get; }

    public decimal Subtotale => Quantita * PrezzoUnitario;

    public RigaDocumento(string descrizione, int quantita, decimal prezzoUnitario)
    {
        if (string.IsNullOrWhiteSpace(descrizione))
        {
            throw new ArgumentException("La descrizione è obbligatoria.", nameof(descrizione));
        }

        if (quantita <= 0)
        {
            throw new ArgumentOutOfRangeException(nameof(quantita), "La quantità deve essere maggiore di zero.");
        }

        if (prezzoUnitario < 0)
        {
            throw new ArgumentOutOfRangeException(nameof(prezzoUnitario), "Il prezzo non può essere negativo.");
        }

        Descrizione = descrizione;
        Quantita = quantita;
        PrezzoUnitario = prezzoUnitario;
    }
}
```

Ora una riga documento nasce già valida.

Uso:

```csharp id="v1o6zl"
RigaDocumento riga = new RigaDocumento("Libro C#", 2, 25m);

Console.WriteLine(riga.Subtotale);
```

Output:

```text id="vi5d6a"
50
```

---

## Oggetti anemici

Nel mondo enterprise si parla spesso di **anemic domain model**, cioè modelli fatti solo di proprietà pubbliche senza comportamento.

Esempio:

```csharp id="5hjd83"
public class Ordine
{
    public List<RigaDocumento> Righe { get; set; } = new();
    public bool Confermato { get; set; }
}
```

Chiunque può modificare tutto.

Un modello più ricco espone comportamenti:

```csharp id="h2c24x"
public class Ordine
{
    private readonly List<RigaDocumento> righe = new();

    public IReadOnlyList<RigaDocumento> Righe => righe;
    public bool Confermato { get; private set; }

    public void AggiungiRiga(RigaDocumento riga)
    {
        if (Confermato)
        {
            throw new InvalidOperationException("Non puoi modificare un ordine confermato.");
        }

        righe.Add(riga);
    }

    public void Conferma()
    {
        if (righe.Count == 0)
        {
            throw new InvalidOperationException("Non puoi confermare un ordine senza righe.");
        }

        Confermato = true;
    }
}
```

Questo oggetto protegge meglio le proprie regole.

# Classi parziali

Una **classe parziale** è una classe divisa in più file usando la keyword `partial`.

Esempio file `Cliente.cs`:

```csharp id="f7tyfu"
public partial class Cliente
{
    public string Nome { get; set; } = "";
    public string Cognome { get; set; } = "";
}
```

File `Cliente.Validazione.cs`:

```csharp id="sbdclk"
public partial class Cliente
{
    public bool ÈValido()
    {
        return !string.IsNullOrWhiteSpace(Nome)
            && !string.IsNullOrWhiteSpace(Cognome);
    }
}
```

Per il compilatore è come se fosse una sola classe.

---

## Quando usare classi parziali

Le classi parziali sono utili quando:

* codice generato e codice scritto a mano devono convivere;
* una classe molto grande viene separata per responsabilità tecniche;
* strumenti visuali generano parte della classe;
* framework richiedono estensioni parziali.

Esempi tipici:

* Windows Forms;
* WPF;
* source generators;
* codice generato da strumenti;
* modelli generati.

---

## Quando evitarle

Non bisogna usare `partial` per nascondere classi troppo grandi.

Se una classe è enorme, il problema non è che sta in un solo file. Il problema è che probabilmente ha troppe responsabilità.

Regola pratica:

```text id="uk1gbr"
Usa partial per separare codice generato o parti tecniche.
Non usarlo come scusa per classi troppo grandi.
```

---

# Classi statiche

Una **classe statica** non può essere istanziata.

Contiene solo membri statici.

Esempio:

```csharp id="uo6h5i"
public static class CalcolatoreIva
{
    public static decimal CalcolaIva(decimal imponibile, decimal aliquota)
    {
        return imponibile * aliquota / 100m;
    }
}
```

Uso:

```csharp id="bzixua"
decimal iva = CalcolatoreIva.CalcolaIva(100m, 22m);

Console.WriteLine(iva);
```

Output:

```text id="dd0gzi"
22
```

Non possiamo scrivere:

```csharp id="t4uv4w"
CalcolatoreIva calcolatore = new CalcolatoreIva(); // errore
```

---

## Quando usare classi statiche

Le classi statiche sono utili per funzioni pure o helper senza stato.

Esempi:

```text id="lr8iu0"
Math
File
Path
Convert
```

Esempio:

```csharp id="vdqm9m"
public static class Validazione
{
    public static bool ÈEmailValida(string email)
    {
        return !string.IsNullOrWhiteSpace(email)
            && email.Contains('@');
    }
}
```

Uso:

```csharp id="1apcap"
if (Validazione.ÈEmailValida("test@example.com"))
{
    Console.WriteLine("Email valida");
}
```

---

## Attenzione alle classi statiche

Le classi statiche possono diventare un problema se contengono stato globale modificabile.

Esempio pericoloso:

```csharp id="u6rn16"
public static class StatoApplicazione
{
    public static string UtenteCorrente { get; set; } = "";
}
```

Questo stato può essere modificato da qualsiasi punto del programma.

Nei progetti grandi, stato globale e dipendenze statiche rendono il codice più difficile da testare e mantenere.

Regola pratica:

```text id="we00j5"
Classe statica per funzioni senza stato
  spesso va bene.

Classe statica con stato globale modificabile
  da usare con molta cautela.
```

# Esempio completo del capitolo

Costruiamo una piccola classe `Documento` con righe.

L’obiettivo è proteggere alcune regole:

```text id="b64s93"
Un documento deve avere un cliente.
Una riga deve avere descrizione, quantità positiva e prezzo non negativo.
Un documento confermato non può essere modificato.
Un documento senza righe non può essere confermato.
```

---

## Classe RigaDocumento

```csharp id="x0ubos"
public class RigaDocumento
{
    public string Descrizione { get; }
    public int Quantita { get; }
    public decimal PrezzoUnitario { get; }

    public decimal Subtotale => Quantita * PrezzoUnitario;

    public RigaDocumento(string descrizione, int quantita, decimal prezzoUnitario)
    {
        if (string.IsNullOrWhiteSpace(descrizione))
        {
            throw new ArgumentException("La descrizione è obbligatoria.", nameof(descrizione));
        }

        if (quantita <= 0)
        {
            throw new ArgumentOutOfRangeException(nameof(quantita), "La quantità deve essere maggiore di zero.");
        }

        if (prezzoUnitario < 0)
        {
            throw new ArgumentOutOfRangeException(nameof(prezzoUnitario), "Il prezzo non può essere negativo.");
        }

        Descrizione = descrizione;
        Quantita = quantita;
        PrezzoUnitario = prezzoUnitario;
    }
}
```

---

## Classe Documento

```csharp id="ihgipz"
public class Documento
{
    private readonly List<RigaDocumento> righe = new();

    public string Cliente { get; }
    public bool Confermato { get; private set; }
    public IReadOnlyList<RigaDocumento> Righe => righe;
    public decimal Totale => righe.Sum(riga => riga.Subtotale);

    public Documento(string cliente)
    {
        if (string.IsNullOrWhiteSpace(cliente))
        {
            throw new ArgumentException("Il cliente è obbligatorio.", nameof(cliente));
        }

        Cliente = cliente;
    }

    public void AggiungiRiga(RigaDocumento riga)
    {
        ArgumentNullException.ThrowIfNull(riga);

        if (Confermato)
        {
            throw new InvalidOperationException("Non puoi modificare un documento confermato.");
        }

        righe.Add(riga);
    }

    public void Conferma()
    {
        if (righe.Count == 0)
        {
            throw new InvalidOperationException("Non puoi confermare un documento senza righe.");
        }

        Confermato = true;
    }
}
```

---

## Uso

```csharp id="9fth3w"
Documento documento = new Documento("Emilie Rollandin");

documento.AggiungiRiga(new RigaDocumento("Libro C#", 2, 25m));
documento.AggiungiRiga(new RigaDocumento("Corso .NET", 1, 100m));

Console.WriteLine($"Cliente: {documento.Cliente}");
Console.WriteLine($"Totale: {documento.Totale}");

documento.Conferma();

Console.WriteLine($"Confermato: {documento.Confermato}");
```

Output:

```text id="bqpjtd"
Cliente: Emilie Rollandin
Totale: 150
Confermato: True
```

Se proviamo ad aggiungere una riga dopo la conferma:

```csharp id="hh3xjs"
documento.AggiungiRiga(new RigaDocumento("Altro", 1, 10m));
```

otteniamo un’eccezione.

Questo è il punto centrale dell’incapsulamento:

> l’oggetto espone operazioni sensate e impedisce modifiche arbitrarie al proprio stato.

# Errori comuni

## Rendere pubblici tutti i campi

Sconsigliato:

```csharp id="xruf97"
public class Cliente
{
    public string Nome;
    public int Eta;
}
```

Meglio usare proprietà e validazione quando serve:

```csharp id="kwgzxx"
public class Cliente
{
    public string Nome { get; private set; }

    public Cliente(string nome)
    {
        if (string.IsNullOrWhiteSpace(nome))
        {
            throw new ArgumentException("Il nome è obbligatorio.", nameof(nome));
        }

        Nome = nome;
    }
}
```

---

## Usare proprietà pubbliche modificabili per tutto

Semplice, ma rischioso:

```csharp id="u151v6"
public class Ordine
{
    public bool Confermato { get; set; }
    public List<RigaDocumento> Righe { get; set; } = new();
}
```

Chiunque può modificare righe e stato.

Meglio:

```csharp id="v5pu48"
public class Ordine
{
    private readonly List<RigaDocumento> righe = new();

    public IReadOnlyList<RigaDocumento> Righe => righe;
    public bool Confermato { get; private set; }
}
```

---

## Confondere classe e oggetto

```text id="19socm"
Classe
  definizione, modello, tipo.

Oggetto
  istanza concreta creata con new.
```

Esempio:

```csharp id="oz2e4j"
Cliente cliente = new Cliente();
```

`Cliente` è la classe.
`cliente` è l’oggetto, o più precisamente una variabile che fa riferimento all’oggetto.

---

## Dimenticare la validazione nel costruttore

Sconsigliato:

```csharp id="xm1ccv"
public Cliente(string nome)
{
    Nome = nome;
}
```

Se `nome` non può essere vuoto, meglio:

```csharp id="wzll62"
public Cliente(string nome)
{
    if (string.IsNullOrWhiteSpace(nome))
    {
        throw new ArgumentException("Il nome è obbligatorio.", nameof(nome));
    }

    Nome = nome;
}
```

---

## Usare `required` pensando che validi il contenuto

```csharp id="k79s7o"
public required string Nome { get; set; }
```

Obbliga a fornire un valore, ma non impedisce:

```csharp id="wbks1x"
Nome = "";
```

Per quello serve validazione.

---

## Usare classi statiche per tutto

Sconsigliato:

```csharp id="uls8sm"
public static class GestioneClienti
{
    public static void SalvaCliente()
    {
    }

    public static void CancellaCliente()
    {
    }

    public static void CalcolaSconto()
    {
    }
}
```

Una classe statica generica può diventare un contenitore disordinato.

Meglio modellare responsabilità più precise.

# Riepilogo

In questo capitolo abbiamo introdotto il primo blocco fondamentale della programmazione orientata agli oggetti.

Abbiamo visto:

```text id="cq7g0f"
classe
  modello che descrive dati e comportamenti.

oggetto
  istanza concreta di una classe.

campo
  variabile interna di una classe.

proprietà
  modo controllato per esporre dati.

metodo
  comportamento dell’oggetto.

costruttore
  inizializza un nuovo oggetto.
```

Abbiamo introdotto anche strumenti moderni:

```text id="e4hbrd"
costruttori primari
  sintassi compatta per dichiarare parametri del costruttore nella classe.

required
  obbliga a inizializzare certe proprietà.

field
  consente di accedere al backing field generato dal compilatore negli accessor di proprietà.
```

Abbiamo studiato i modificatori di accesso:

```text id="wmicre"
public
private
protected
internal
protected internal
```

E soprattutto abbiamo introdotto i concetti più importanti del capitolo:

```text id="spc18g"
incapsulamento
  proteggere lo stato interno ed esporre solo operazioni controllate.

invariante
  regola che deve essere sempre vera per un oggetto valido.

oggetto valido
  oggetto che rispetta le proprie regole.

oggetto inconsistente
  oggetto che contiene dati incompatibili con il proprio significato.
```

La regola principale è questa:

```text id="2fr5l5"
Una classe non dovrebbe limitarsi a contenere dati.
Dovrebbe proteggere le regole del concetto che rappresenta.
```

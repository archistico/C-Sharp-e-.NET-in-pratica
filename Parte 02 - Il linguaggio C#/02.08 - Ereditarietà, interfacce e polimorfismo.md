# 02.08 — Ereditarietà, interfacce e polimorfismo

Nei capitoli precedenti abbiamo imparato a creare tipi personalizzati usando classi, struct, record ed enum. Abbiamo visto come proteggere lo stato interno di un oggetto e come scegliere il tipo più adatto per rappresentare dati, valori e concetti del dominio.

Ora entriamo in una parte fondamentale della progettazione orientata agli oggetti: la capacità di scrivere codice **estensibile**.

In C# possiamo progettare tipi che:

* riutilizzano comportamento esistente;
* specializzano una classe base;
* espongono contratti comuni;
* possono essere usati in modo polimorfico;
* permettono di sostituire un’implementazione con un’altra;
* riducono l’accoppiamento tra parti del codice.

Gli strumenti principali sono:

* ereditarietà;
* classi base;
* metodi `virtual`;
* metodi `override`;
* classi e membri `abstract`;
* classi e membri `sealed`;
* interfacce;
* implementazione esplicita di interfaccia;
* composizione.

Il punto centrale del capitolo è questo:

> ereditarietà e interfacce servono a progettare codice estendibile, ma vanno usate con criterio. Non ogni relazione tra tipi deve diventare ereditarietà.

---

# Ereditarietà

L’**ereditarietà** permette di creare una classe nuova partendo da una classe esistente.

La classe esistente si chiama **classe base**.
La nuova classe si chiama **classe derivata**.

Microsoft descrive l’ereditarietà come il meccanismo che consente di creare nuove classi che riusano, estendono e modificano il comportamento definito da altre classi. Una classe derivata può avere una sola classe base diretta. ([Microsoft Learn][1])

Esempio:

```csharp id="4nrboc"
public class Persona
{
    public string Nome { get; set; } = "";
    public string Cognome { get; set; } = "";

    public string NomeCompleto => $"{Nome} {Cognome}";
}

public class Cliente : Persona
{
    public int CodiceCliente { get; set; }
}
```

La classe `Cliente` eredita da `Persona`.

Questo significa che `Cliente` possiede anche i membri pubblici di `Persona`.

Uso:

```csharp id="bfxj1c"
Cliente cliente = new Cliente
{
    Nome = "Emilie",
    Cognome = "Rollandin",
    CodiceCliente = 123
};

Console.WriteLine(cliente.NomeCompleto);
Console.WriteLine(cliente.CodiceCliente);
```

Output:

```text id="zjoj86"
Emilie Rollandin
123
```

---

## Sintassi dell’ereditarietà

La sintassi è:

```csharp id="se7iyg"
public class ClasseDerivata : ClasseBase
{
}
```

Esempio:

```csharp id="j11dds"
public class Dipendente : Persona
{
    public string Matricola { get; set; } = "";
}
```

Qui `Dipendente` eredita da `Persona`.

---

## Relazione “è un”

L’ereditarietà dovrebbe rappresentare una relazione del tipo:

```text id="kkvnbc"
Cliente è una Persona.
Dipendente è una Persona.
Fattura è un Documento.
```

Esempio ragionevole:

```csharp id="6yqd2g"
public class Documento
{
    public string Numero { get; set; } = "";
    public DateOnly Data { get; set; }
}

public class Fattura : Documento
{
    public decimal Totale { get; set; }
}
```

Una `Fattura` è un tipo di `Documento`.

---

## Quando l’ereditarietà è sbagliata

Non bisogna usare l’ereditarietà solo per riusare codice.

Esempio sospetto:

```csharp id="asf8lv"
public class DatabaseLogger
{
    public void ScriviLog(string messaggio)
    {
        Console.WriteLine(messaggio);
    }
}

public class ClienteService : DatabaseLogger
{
}
```

`ClienteService` non è un `DatabaseLogger`.

In questo caso l’ereditarietà è concettualmente sbagliata. Se `ClienteService` ha bisogno di scrivere log, dovrebbe **usare** un logger, non ereditarlo.

Meglio:

```csharp id="e5ukxg"
public class ClienteService
{
    private readonly DatabaseLogger logger;

    public ClienteService(DatabaseLogger logger)
    {
        this.logger = logger;
    }

    public void RegistraCliente(string nome)
    {
        logger.ScriviLog($"Registro cliente {nome}");
    }
}
```

Questa è composizione, che vedremo più avanti.

# Classi base

Una **classe base** contiene dati e comportamenti comuni a più classi derivate.

Esempio:

```csharp id="wvntq3"
public class Documento
{
    public string Numero { get; protected set; }
    public DateOnly Data { get; protected set; }

    public Documento(string numero, DateOnly data)
    {
        if (string.IsNullOrWhiteSpace(numero))
        {
            throw new ArgumentException("Il numero è obbligatorio.", nameof(numero));
        }

        Numero = numero;
        Data = data;
    }

    public string DescrizioneBase()
    {
        return $"{Numero} del {Data}";
    }
}
```

Classe derivata:

```csharp id="nqw8qn"
public class Fattura : Documento
{
    public decimal Totale { get; }

    public Fattura(string numero, DateOnly data, decimal totale)
        : base(numero, data)
    {
        if (totale < 0)
        {
            throw new ArgumentOutOfRangeException(nameof(totale));
        }

        Totale = totale;
    }
}
```

Uso:

```csharp id="ik15ni"
Fattura fattura = new Fattura(
    "FAT-001",
    new DateOnly(2026, 4, 24),
    150m);

Console.WriteLine(fattura.DescrizioneBase());
Console.WriteLine(fattura.Totale);
```

Output:

```text id="tk9xca"
FAT-001 del 24/04/2026
150
```

---

## Chiamare il costruttore base

Quando la classe base ha un costruttore con parametri, la classe derivata deve chiamarlo.

```csharp id="nadnnl"
public Fattura(string numero, DateOnly data, decimal totale)
    : base(numero, data)
{
    Totale = totale;
}
```

La parte:

```csharp id="c0agso"
: base(numero, data)
```

chiama il costruttore della classe base.

---

## `protected`

Nel capitolo precedente abbiamo visto `protected`.

È molto usato con l’ereditarietà.

```csharp id="6mqk5s"
public class Documento
{
    protected string Stato { get; set; } = "Bozza";
}

public class Fattura : Documento
{
    public void Conferma()
    {
        Stato = "Confermata";
    }
}
```

`Stato` non è accessibile dall’esterno, ma è accessibile dalle classi derivate.

Attenzione però: abusare di `protected` può rendere le classi derivate troppo dipendenti dai dettagli interni della classe base.

# `virtual`

Un membro `virtual` può essere ridefinito da una classe derivata.

Esempio:

```csharp id="owqc39"
public class Documento
{
    public virtual string Descrivi()
    {
        return "Documento generico";
    }
}
```

Classe derivata:

```csharp id="qi2svy"
public class Fattura : Documento
{
    public override string Descrivi()
    {
        return "Fattura";
    }
}
```

Uso:

```csharp id="hxia23"
Fattura fattura = new Fattura();

Console.WriteLine(fattura.Descrivi());
```

Output:

```text id="58680i"
Fattura
```

---

## Metodo virtuale con comportamento base

Un metodo virtuale può avere già un comportamento nella classe base.

```csharp id="zbdu9t"
public class Documento
{
    public string Numero { get; }

    public Documento(string numero)
    {
        Numero = numero;
    }

    public virtual string Descrivi()
    {
        return $"Documento {Numero}";
    }
}
```

La classe derivata può riusarlo:

```csharp id="1fp1k4"
public class Fattura : Documento
{
    public decimal Totale { get; }

    public Fattura(string numero, decimal totale)
        : base(numero)
    {
        Totale = totale;
    }

    public override string Descrivi()
    {
        return $"{base.Descrivi()} - Totale: {Totale}";
    }
}
```

Qui:

```csharp id="q6n4ro"
base.Descrivi()
```

chiama il metodo della classe base.

---

# `override`

`override` serve a ridefinire un membro virtuale o astratto ereditato dalla classe base.

Esempio completo:

```csharp id="j8fjx5"
public class Documento
{
    public virtual decimal CalcolaTotale()
    {
        return 0m;
    }
}

public class Fattura : Documento
{
    public decimal Imponibile { get; set; }
    public decimal Iva { get; set; }

    public override decimal CalcolaTotale()
    {
        return Imponibile + Iva;
    }
}
```

Uso:

```csharp id="xybavo"
Fattura fattura = new Fattura
{
    Imponibile = 100m,
    Iva = 22m
};

Console.WriteLine(fattura.CalcolaTotale());
```

Output:

```text id="kizkc9"
122
```

---

## override deve corrispondere a un membro virtual o abstract

Questo non è valido:

```csharp id="6dizcl"
public class Documento
{
    public string Descrivi()
    {
        return "Documento";
    }
}

public class Fattura : Documento
{
    public override string Descrivi()
    {
        return "Fattura";
    }
}
```

Il metodo base non è `virtual`, quindi non può essere sovrascritto.

Versione corretta:

```csharp id="isvht1"
public class Documento
{
    public virtual string Descrivi()
    {
        return "Documento";
    }
}
```

---

# `abstract`

Una classe `abstract` non può essere istanziata direttamente.

Serve come base per classi derivate.

```csharp id="dzcp8r"
public abstract class Documento
{
    public string Numero { get; }

    protected Documento(string numero)
    {
        Numero = numero;
    }

    public abstract decimal CalcolaTotale();
}
```

Qui `CalcolaTotale` è un metodo astratto: non ha corpo.

Le classi derivate devono implementarlo.

```csharp id="v4v213"
public class Fattura : Documento
{
    public decimal Imponibile { get; }
    public decimal Iva { get; }

    public Fattura(string numero, decimal imponibile, decimal iva)
        : base(numero)
    {
        Imponibile = imponibile;
        Iva = iva;
    }

    public override decimal CalcolaTotale()
    {
        return Imponibile + Iva;
    }
}
```

Uso:

```csharp id="c5hzkm"
Documento documento = new Fattura("FAT-001", 100m, 22m);

Console.WriteLine(documento.CalcolaTotale());
```

Output:

```text id="tjqwht"
122
```

La documentazione Microsoft specifica che una classe astratta è pensata come classe base e non può essere istanziata direttamente; i membri astratti dichiarati in essa devono essere implementati dalle classi derivate non astratte. ([Microsoft Learn][2])

---

## Classe astratta con membri concreti

Una classe astratta può contenere sia membri astratti sia membri concreti.

```csharp id="83r8hq"
public abstract class Documento
{
    public string Numero { get; }

    protected Documento(string numero)
    {
        Numero = numero;
    }

    public string Descrivi()
    {
        return $"Documento {Numero}";
    }

    public abstract decimal CalcolaTotale();
}
```

Questo è utile quando vogliamo condividere comportamento comune, ma lasciare alcune parti obbligatoriamente specializzate.

---

# `sealed`

`sealed` impedisce che una classe venga ereditata.

```csharp id="qpqgzu"
public sealed class CodiceFiscale
{
    public string Valore { get; }

    public CodiceFiscale(string valore)
    {
        Valore = valore;
    }
}
```

Questo non è consentito:

```csharp id="w0g9e7"
public class CodiceFiscaleSpeciale : CodiceFiscale
{
}
```

---

## sealed su override

Possiamo anche bloccare ulteriori override.

```csharp id="u6xys5"
public class Documento
{
    public virtual string Descrivi()
    {
        return "Documento";
    }
}

public class Fattura : Documento
{
    public sealed override string Descrivi()
    {
        return "Fattura";
    }
}
```

Una classe che eredita da `Fattura` non potrà più sovrascrivere `Descrivi`.

---

## Quando usare sealed

`sealed` è utile quando:

* non vogliamo permettere estensioni non previste;
* vogliamo proteggere invarianti;
* il tipo rappresenta un concetto finale;
* vogliamo semplificare il modello;
* vogliamo evitare ereditarietà impropria.

Regola pratica:

```text id="b9yaj3"
Rendi una classe ereditabile solo se l’hai progettata per esserlo.
```

Non tutte le classi devono essere estensibili.

# Polimorfismo

Il **polimorfismo** permette di trattare oggetti diversi attraverso un tipo comune.

Esempio:

```csharp id="axh0ud"
public abstract class Documento
{
    public string Numero { get; }

    protected Documento(string numero)
    {
        Numero = numero;
    }

    public abstract decimal CalcolaTotale();
}

public class Fattura : Documento
{
    public decimal Totale { get; }

    public Fattura(string numero, decimal totale)
        : base(numero)
    {
        Totale = totale;
    }

    public override decimal CalcolaTotale()
    {
        return Totale;
    }
}

public class NotaCredito : Documento
{
    public decimal Importo { get; }

    public NotaCredito(string numero, decimal importo)
        : base(numero)
    {
        Importo = importo;
    }

    public override decimal CalcolaTotale()
    {
        return -Importo;
    }
}
```

Uso polimorfico:

```csharp id="f9swkm"
List<Documento> documenti = new List<Documento>
{
    new Fattura("FAT-001", 100m),
    new NotaCredito("NC-001", 30m)
};

foreach (Documento documento in documenti)
{
    Console.WriteLine($"{documento.Numero}: {documento.CalcolaTotale()}");
}
```

Output:

```text id="f2mtea"
FAT-001: 100
NC-001: -30
```

La variabile è di tipo `Documento`, ma il metodo eseguito è quello della classe concreta.

Questo è polimorfismo.

---

## Perché il polimorfismo è utile

Senza polimorfismo potremmo scrivere:

```csharp id="lrd8xa"
foreach (Documento documento in documenti)
{
    if (documento is Fattura fattura)
    {
        Console.WriteLine(fattura.Totale);
    }
    else if (documento is NotaCredito notaCredito)
    {
        Console.WriteLine(-notaCredito.Importo);
    }
}
```

Con il polimorfismo:

```csharp id="v45xyp"
foreach (Documento documento in documenti)
{
    Console.WriteLine(documento.CalcolaTotale());
}
```

Ogni oggetto sa come calcolare il proprio totale.

Il codice chiamante non deve conoscere tutti i casi specifici.

---

# Cast tra tipi

Con l’ereditarietà possiamo assegnare un oggetto derivato a una variabile del tipo base.

```csharp id="2jyojg"
Fattura fattura = new Fattura("FAT-001", 100m);

Documento documento = fattura;
```

Questo si chiama **upcast**.

È sicuro, perché ogni `Fattura` è un `Documento`.

---

## Upcast

```csharp id="801b83"
Fattura fattura = new Fattura("FAT-001", 100m);

Documento documento = fattura;
```

Non serve cast esplicito.

---

## Downcast

Il contrario si chiama **downcast**.

```csharp id="8nuvld"
Documento documento = new Fattura("FAT-001", 100m);

Fattura fattura = (Fattura)documento;
```

Questo funziona solo se l’oggetto reale è davvero una `Fattura`.

Se non lo è, otteniamo errore a runtime.

---

## Downcast sicuro con `is`

Meglio usare pattern matching:

```csharp id="ieaock"
if (documento is Fattura fattura)
{
    Console.WriteLine(fattura.Totale);
}
```

Qui la variabile `fattura` viene creata solo se `documento` è davvero una `Fattura`.

---

## Downcast con `as`

Possiamo anche usare `as`.

```csharp id="mopju3"
Fattura? fattura = documento as Fattura;

if (fattura is not null)
{
    Console.WriteLine(fattura.Totale);
}
```

Oggi, nella maggior parte dei casi, `is` con pattern matching è più diretto.

# Interfacce

Un’interfaccia definisce un **contratto**.

Dice quali membri un tipo deve fornire, senza necessariamente dire come devono essere implementati.

Esempio:

```csharp id="kr0alw"
public interface INotificatore
{
    void Invia(string destinatario, string messaggio);
}
```

Questa interfaccia dice:

> un notificatore deve avere un metodo `Invia`.

Non dice se il messaggio verrà inviato via email, SMS, log, API o altro.

---

## Implementare un’interfaccia

```csharp id="ng6gmx"
public class NotificatoreEmail : INotificatore
{
    public void Invia(string destinatario, string messaggio)
    {
        Console.WriteLine($"Email a {destinatario}: {messaggio}");
    }
}
```

Uso:

```csharp id="ol32vq"
INotificatore notificatore = new NotificatoreEmail();

notificatore.Invia("test@example.com", "Ciao!");
```

Output:

```text id="klfnjm"
Email a test@example.com: Ciao!
```

La variabile è di tipo `INotificatore`, ma l’oggetto reale è `NotificatoreEmail`.

---

## Più implementazioni

```csharp id="uqf7sp"
public class NotificatoreSms : INotificatore
{
    public void Invia(string destinatario, string messaggio)
    {
        Console.WriteLine($"SMS a {destinatario}: {messaggio}");
    }
}

public class NotificatoreConsole : INotificatore
{
    public void Invia(string destinatario, string messaggio)
    {
        Console.WriteLine($"Console: {destinatario} - {messaggio}");
    }
}
```

Ora possiamo scrivere codice che dipende dall’interfaccia, non dalla classe concreta.

```csharp id="1b9i8o"
void InviaMessaggio(INotificatore notificatore)
{
    notificatore.Invia("utente", "Messaggio di prova");
}
```

Chiamate:

```csharp id="snja7v"
InviaMessaggio(new NotificatoreEmail());
InviaMessaggio(new NotificatoreSms());
InviaMessaggio(new NotificatoreConsole());
```

---

## Interfacce e polimorfismo

Le interfacce permettono polimorfismo senza ereditarietà di classe.

```csharp id="230fc6"
List<INotificatore> notificatori = new List<INotificatore>
{
    new NotificatoreEmail(),
    new NotificatoreSms(),
    new NotificatoreConsole()
};

foreach (INotificatore notificatore in notificatori)
{
    notificatore.Invia("cliente", "Promemoria");
}
```

Ogni classe implementa il contratto a modo suo.

---

## Una classe può implementare più interfacce

In C#, una classe può ereditare da una sola classe base, ma può implementare molte interfacce.

```csharp id="o9dxxd"
public interface IStampabile
{
    void Stampa();
}

public interface IEsportabile
{
    void Esporta();
}

public class Report : IStampabile, IEsportabile
{
    public void Stampa()
    {
        Console.WriteLine("Stampa report");
    }

    public void Esporta()
    {
        Console.WriteLine("Esporta report");
    }
}
```

Questo rende le interfacce molto flessibili.

---

# Implementazione esplicita di interfaccia

A volte vogliamo implementare un membro di interfaccia senza esporlo direttamente come metodo pubblico della classe.

Oppure vogliamo risolvere conflitti tra interfacce diverse.

In questi casi possiamo usare l’**implementazione esplicita**.

Esempio:

```csharp id="xcl8my"
public interface IStampabile
{
    void Esegui();
}

public interface IEsportabile
{
    void Esegui();
}

public class Report : IStampabile, IEsportabile
{
    void IStampabile.Esegui()
    {
        Console.WriteLine("Stampa report");
    }

    void IEsportabile.Esegui()
    {
        Console.WriteLine("Esporta report");
    }
}
```

Uso:

```csharp id="r3g5rm"
Report report = new Report();

IStampabile stampabile = report;
IEsportabile esportabile = report;

stampabile.Esegui();
esportabile.Esegui();
```

Output:

```text id="tp4bzy"
Stampa report
Esporta report
```

La specifica C# chiarisce che un membro implementato esplicitamente è accessibile tramite un’istanza dell’interfaccia, non direttamente tramite l’istanza della classe concreta. ([Microsoft Learn][3])

---

## Perché usare implementazione esplicita

È utile quando:

* due interfacce hanno membri con lo stesso nome;
* vogliamo nascondere un membro dal pubblico principale della classe;
* il nome dell’interfaccia non è ideale per l’API pubblica della classe;
* implementiamo interfacce tecniche come `IDisposable` o `IEnumerable`.

Esempio concettuale:

```csharp id="qohifm"
public class Risorsa : IDisposable
{
    void IDisposable.Dispose()
    {
        Console.WriteLine("Rilascio risorsa");
    }

    public void Chiudi()
    {
        Console.WriteLine("Chiusura esplicita");
    }
}
```

---

# Default interface methods

Tradizionalmente, un’interfaccia conteneva solo il contratto: firme di metodi, proprietà, eventi o indicizzatori.

Con C# 8 sono stati introdotti i **default interface methods**, cioè membri di interfaccia con implementazione predefinita.

Esempio:

```csharp id="ld34b7"
public interface ILogger
{
    void Log(string messaggio);

    void LogErrore(string messaggio)
    {
        Log($"ERRORE: {messaggio}");
    }
}
```

Una classe può implementare solo il metodo principale:

```csharp id="9fsbe6"
public class ConsoleLogger : ILogger
{
    public void Log(string messaggio)
    {
        Console.WriteLine(messaggio);
    }
}
```

Uso:

```csharp id="54tloq"
ILogger logger = new ConsoleLogger();

logger.Log("Operazione completata");
logger.LogErrore("Operazione fallita");
```

Output:

```text id="h8bbd3"
Operazione completata
ERRORE: Operazione fallita
```

La documentazione Microsoft indica che i default interface members permettono a un’interfaccia di fornire un corpo di metodo; i tipi che implementano l’interfaccia ereditano l’implementazione predefinita e possono eventualmente ridefinirla. ([Microsoft Learn][4])

---

## Quando usare default interface methods

Sono utili soprattutto per:

* evolvere interfacce esistenti senza rompere implementazioni già presenti;
* fornire comportamento comune minimo;
* evitare duplicazione in scenari controllati;
* creare contratti con funzionalità opzionali.

Attenzione però: non bisogna trasformare le interfacce in classi base mascherate.

Regola pratica:

```text id="8y1d6p"
Default interface methods
  utili per evoluzione e compatibilità.

Classi base astratte
  più adatte quando serve condividere stato o logica sostanziale.
```

---

# Membri statici nelle interfacce

Nelle versioni moderne di C#, le interfacce possono dichiarare anche membri statici, inclusi membri `static abstract` o `static virtual`.

La documentazione Microsoft indica che un’interfaccia può definire membri statici astratti o virtuali per richiedere che i tipi implementanti forniscano determinati membri statici. Questo è particolarmente utile per operatori, factory e generic math. ([Microsoft Learn][5])

Esempio semplificato:

```csharp id="1kyjsf"
public interface ICreabile<T>
{
    static abstract T CreaDefault();
}
```

Implementazione:

```csharp id="f4im5w"
public class Cliente : ICreabile<Cliente>
{
    public string Nome { get; set; } = "";

    public static Cliente CreaDefault()
    {
        return new Cliente { Nome = "Nuovo cliente" };
    }
}
```

Uso in metodo generico:

```csharp id="edah0a"
T Crea<T>() where T : ICreabile<T>
{
    return T.CreaDefault();
}
```

Chiamata:

```csharp id="b5x9lh"
Cliente cliente = Crea<Cliente>();

Console.WriteLine(cliente.Nome);
```

Output:

```text id="u1kcxw"
Nuovo cliente
```

---

## Quando parlarne davvero

I membri statici nelle interfacce sono una funzionalità potente, ma avanzata.

Nel codice quotidiano di un principiante, le interfacce servono soprattutto per:

* definire contratti;
* disaccoppiare implementazioni;
* rendere il codice testabile;
* usare Dependency Injection;
* progettare servizi sostituibili.

I membri statici nelle interfacce sono importanti soprattutto in:

* librerie generiche;
* tipi matematici;
* operatori;
* factory generiche;
* framework.

La documentazione Microsoft segnala che gli static virtual interface members permettono di definire interfacce con operatori o altri membri statici e usarle come vincoli generici. ([Microsoft Learn][6])

Per questo, in un percorso didattico, è corretto presentarli come nota moderna, senza renderli centrali.

# Composizione vs ereditarietà

L’ereditarietà rappresenta una relazione:

```text id="fvhra3"
è un
```

La composizione rappresenta una relazione:

```text id="z9oufp"
ha un
usa un
è composto da
```

Esempio ereditarietà:

```text id="7hjqbq"
Fattura è un Documento.
```

Esempio composizione:

```text id="1mxer5"
Ordine ha righe.
Cliente ha indirizzo.
ServizioClienti usa un repository.
```

---

## Esempio di composizione

```csharp id="deq3gm"
public class Indirizzo
{
    public string Via { get; }
    public string Citta { get; }

    public Indirizzo(string via, string citta)
    {
        Via = via;
        Citta = citta;
    }
}

public class Cliente
{
    public string Nome { get; }
    public Indirizzo Indirizzo { get; }

    public Cliente(string nome, Indirizzo indirizzo)
    {
        Nome = nome;
        Indirizzo = indirizzo;
    }
}
```

Qui `Cliente` non eredita da `Indirizzo`.

Un cliente **ha un** indirizzo.

Uso:

```csharp id="rz8uqb"
Indirizzo indirizzo = new Indirizzo("Via Roma", "Aosta");

Cliente cliente = new Cliente("Emilie", indirizzo);
```

---

## Composizione con servizi

```csharp id="m4z5r4"
public interface IClienteRepository
{
    void Salva(string nome);
}

public class ClienteService
{
    private readonly IClienteRepository repository;

    public ClienteService(IClienteRepository repository)
    {
        this.repository = repository;
    }

    public void Registra(string nome)
    {
        repository.Salva(nome);
    }
}
```

`ClienteService` non eredita da `IClienteRepository`.

Lo usa.

Questa scelta rende il servizio più flessibile e testabile.

---

## Perché spesso preferire la composizione

L’ereditarietà crea un legame forte tra classe base e classe derivata.

La classe derivata dipende dalla struttura e dalle decisioni della classe base.

La composizione è spesso più flessibile perché possiamo sostituire il componente usato.

Esempio:

```csharp id="jp9es9"
public class ClienteService
{
    private readonly INotificatore notificatore;

    public ClienteService(INotificatore notificatore)
    {
        this.notificatore = notificatore;
    }

    public void RegistraCliente(string email)
    {
        notificatore.Invia(email, "Registrazione completata");
    }
}
```

Possiamo passare:

```csharp id="yiorgh"
new NotificatoreEmail()
new NotificatoreSms()
new NotificatoreConsole()
```

senza modificare `ClienteService`.

---

# Quando preferire interfacce e composizione

Spesso conviene preferire interfacce e composizione quando vogliamo rappresentare una capacità.

Esempi:

```text id="bdnxt3"
stampabile
esportabile
serializzabile
notificatore
repository
validatore
calcolatore
provider
servizio
```

Questi concetti non descrivono necessariamente “che cos’è” un oggetto, ma “che cosa sa fare”.

---

## Esempio: esportazione

```csharp id="ajwnx3"
public interface IEsportatore
{
    void Esporta(string contenuto);
}

public class EsportatorePdf : IEsportatore
{
    public void Esporta(string contenuto)
    {
        Console.WriteLine($"Esporto PDF: {contenuto}");
    }
}

public class EsportatoreCsv : IEsportatore
{
    public void Esporta(string contenuto)
    {
        Console.WriteLine($"Esporto CSV: {contenuto}");
    }
}
```

Un servizio può usare l’interfaccia:

```csharp id="80hm69"
public class ReportService
{
    private readonly IEsportatore esportatore;

    public ReportService(IEsportatore esportatore)
    {
        this.esportatore = esportatore;
    }

    public void GeneraReport()
    {
        string contenuto = "Dati report";
        esportatore.Esporta(contenuto);
    }
}
```

Ora `ReportService` non dipende da PDF o CSV.

Dipende da un contratto.

---

## Vantaggi

Questo approccio offre:

* minore accoppiamento;
* maggiore testabilità;
* sostituzione facile delle implementazioni;
* migliore integrazione con Dependency Injection;
* codice più modulare;
* meno gerarchie rigide.

---

# Quando usare ereditarietà

L’ereditarietà resta utile quando:

* esiste una vera relazione “è un”;
* c’è comportamento comune reale;
* la gerarchia è stabile;
* le classi derivate sono specializzazioni naturali;
* vogliamo usare polimorfismo basato su una classe base;
* la classe base protegge regole comuni.

Esempio ragionevole:

```csharp id="pqs7by"
public abstract class DocumentoContabile
{
    public string Numero { get; }

    protected DocumentoContabile(string numero)
    {
        Numero = numero;
    }

    public abstract decimal CalcolaImpattoContabile();
}

public class Fattura : DocumentoContabile
{
    public decimal Totale { get; }

    public Fattura(string numero, decimal totale)
        : base(numero)
    {
        Totale = totale;
    }

    public override decimal CalcolaImpattoContabile()
    {
        return Totale;
    }
}

public class NotaCredito : DocumentoContabile
{
    public decimal Totale { get; }

    public NotaCredito(string numero, decimal totale)
        : base(numero)
    {
        Totale = totale;
    }

    public override decimal CalcolaImpattoContabile()
    {
        return -Totale;
    }
}
```

Qui `Fattura` e `NotaCredito` sono specializzazioni di `DocumentoContabile`.

---

# Quando evitare ereditarietà

Evita l’ereditarietà quando:

* vuoi solo riusare codice;
* la relazione “è un” non è vera;
* la classe base diventa troppo generica;
* le classi derivate devono disattivare comportamenti base;
* la gerarchia cambia spesso;
* vuoi solo sostituire una strategia;
* vuoi testare facilmente implementazioni diverse.

Esempio sbagliato:

```csharp id="yfq60p"
public class EmailSender
{
    public void SendEmail(string email, string messaggio)
    {
    }
}

public class OrdineService : EmailSender
{
}
```

`OrdineService` non è un `EmailSender`.

Meglio:

```csharp id="f40zn4"
public class OrdineService
{
    private readonly INotificatore notificatore;

    public OrdineService(INotificatore notificatore)
    {
        this.notificatore = notificatore;
    }
}
```

# Esempio completo del capitolo

Costruiamo un piccolo esempio con:

* classe astratta;
* ereditarietà;
* override;
* polimorfismo;
* interfaccia;
* composizione.

Scenario:

```text id="808zlv"
Abbiamo documenti contabili diversi.
Ogni documento ha un impatto economico.
Alla fine vogliamo notificare il totale usando un notificatore sostituibile.
```

---

## Classe base astratta

```csharp id="ax773d"
public abstract class DocumentoContabile
{
    public string Numero { get; }
    public DateOnly Data { get; }

    protected DocumentoContabile(string numero, DateOnly data)
    {
        if (string.IsNullOrWhiteSpace(numero))
        {
            throw new ArgumentException("Il numero è obbligatorio.", nameof(numero));
        }

        Numero = numero;
        Data = data;
    }

    public abstract decimal CalcolaImpatto();

    public virtual string Descrivi()
    {
        return $"{Numero} del {Data}";
    }
}
```

---

## Classi derivate

```csharp id="26r30a"
public sealed class Fattura : DocumentoContabile
{
    public decimal Totale { get; }

    public Fattura(string numero, DateOnly data, decimal totale)
        : base(numero, data)
    {
        if (totale < 0)
        {
            throw new ArgumentOutOfRangeException(nameof(totale));
        }

        Totale = totale;
    }

    public override decimal CalcolaImpatto()
    {
        return Totale;
    }

    public override string Descrivi()
    {
        return $"Fattura {base.Descrivi()} - Totale {Totale}";
    }
}

public sealed class NotaCredito : DocumentoContabile
{
    public decimal Totale { get; }

    public NotaCredito(string numero, DateOnly data, decimal totale)
        : base(numero, data)
    {
        if (totale < 0)
        {
            throw new ArgumentOutOfRangeException(nameof(totale));
        }

        Totale = totale;
    }

    public override decimal CalcolaImpatto()
    {
        return -Totale;
    }

    public override string Descrivi()
    {
        return $"Nota credito {base.Descrivi()} - Totale {Totale}";
    }
}
```

---

## Interfaccia notificatore

```csharp id="ws5s1j"
public interface INotificatore
{
    void Invia(string messaggio);
}
```

Implementazioni:

```csharp id="av5qh1"
public class NotificatoreConsole : INotificatore
{
    public void Invia(string messaggio)
    {
        Console.WriteLine($"NOTIFICA: {messaggio}");
    }
}

public class NotificatoreSilenzioso : INotificatore
{
    public void Invia(string messaggio)
    {
        // Non fa nulla.
        // Utile per test o ambienti dove non vogliamo notifiche reali.
    }
}
```

---

## Servizio che usa composizione

```csharp id="pmkhc8"
public class RiepilogoDocumentiService
{
    private readonly INotificatore notificatore;

    public RiepilogoDocumentiService(INotificatore notificatore)
    {
        this.notificatore = notificatore;
    }

    public decimal CalcolaTotale(IEnumerable<DocumentoContabile> documenti)
    {
        decimal totale = 0m;

        foreach (DocumentoContabile documento in documenti)
        {
            totale += documento.CalcolaImpatto();
        }

        notificatore.Invia($"Totale documenti: {totale}");

        return totale;
    }
}
```

## Uso

```csharp id="dq48no"
List<DocumentoContabile> documenti = new List<DocumentoContabile>
{
    new Fattura("FAT-001", new DateOnly(2026, 4, 24), 100m),
    new Fattura("FAT-002", new DateOnly(2026, 4, 25), 200m),
    new NotaCredito("NC-001", new DateOnly(2026, 4, 26), 50m)

INotificatore notificatore = new NotificatoreConsole();

  è un contratto.

```

# Errori comuni

## Usare ereditarietà solo per riusare codice

Sbagliato:

```csharp id="d8oksg"
public class UtilityEmail
{
    public void InviaEmail()
    {
    }
}

public class ClienteService : UtilityEmail
{
}
```

`ClienteService` non è una `UtilityEmail`.

Meglio composizione.

---

## Dimenticare `virtual`

Sbagliato:

```csharp id="1uexk5"
public class Documento
{
    public string Descrivi()
    {
        return "Documento";
    }
}

public class Fattura : Documento
{
    public override string Descrivi()
    {
        return "Fattura";
    }
}
```

Il metodo base deve essere `virtual` oppure `abstract`.

---

## Usare classi base troppo generiche

Sospetto:

```csharp id="kr24a4"
public class EntitaBase
{
    public int Id { get; set; }
    public string Nome { get; set; } = "";
    public DateTime DataCreazione { get; set; }
    public void Salva() { }
    public void Cancella() { }
    public void InviaEmail() { }
}
```

Una classe base troppo generica tende a diventare un contenitore confuso.

---

## Interfacce troppo grandi

Sbagliato:

```csharp id="6rs6xt"
public interface IServizioCompleto
{
    void Salva();
    void Cancella();
    void Stampa();
    void Esporta();
    void InviaEmail();
    void CalcolaStatistiche();
}
```

Meglio interfacce piccole e specifiche:

```csharp id="ua1orq"
public interface ISalvabile
{
    void Salva();
}

public interface IEsportabile
{
    void Esporta();
}
```

---

## Dipendere da classi concrete quando basta un’interfaccia

Meno flessibile:

```csharp id="slp5y3"
public class ClienteService
{
    private readonly NotificatoreEmail notificatore;
}
```

Più flessibile:

```csharp id="46j45e"
public class ClienteService
{
    private readonly INotificatore notificatore;
}
```

---

## Usare interfacce inutili

Non bisogna creare interfacce per ogni classe automaticamente.

Esempio sospetto:

```csharp id="78bikr"
public interface ICliente
{
    string Nome { get; }
}

public class Cliente : ICliente
{
    public string Nome { get; set; } = "";
}
```

Se esiste una sola implementazione e non serve polimorfismo, testabilità o disaccoppiamento, l’interfaccia può essere inutile.

Regola pratica:

```text id="3v0rlq"
Crea un’interfaccia quando ti serve un contratto sostituibile.
Non crearla solo per abitudine.
```

# Riepilogo

In questo capitolo abbiamo introdotto strumenti fondamentali per progettare codice estendibile.

Abbiamo visto:

```text id="l99v01"
ereditarietà
  una classe derivata riusa ed estende una classe base.

classe base
  classe da cui altre classi ereditano.

virtual
  membro ridefinibile.

override
  ridefinizione di un membro ereditato.

abstract
  membro o classe incompleta, da completare nelle derivate.

sealed
  impedisce ulteriore ereditarietà o override.

polimorfismo
  usare tipi diversi attraverso un tipo comune.

cast
  conversione tra tipo base e tipo derivato.
```

Abbiamo poi introdotto le interfacce:

```text id="m6a3n6"
interfaccia
  contratto che un tipo deve rispettare.

implementazione esplicita
  implementazione accessibile tramite il tipo interfaccia.

default interface methods
  membri di interfaccia con implementazione predefinita.

membri statici nelle interfacce
  funzionalità moderna utile in scenari avanzati e generici.
```

Infine abbiamo confrontato ereditarietà e composizione:

```text id="kgrq78"
ereditarietà
  relazione “è un”.

composizione
  relazione “ha un” o “usa un”.
```

La regola più importante è questa:

```text id="1wxx3z"
Preferisci composizione e interfacce quando vuoi flessibilità.
Usa ereditarietà solo quando la relazione “è un” è reale e stabile.
```

Da questo punto in avanti il codice può diventare più modulare: non lavoriamo più solo con singoli tipi isolati, ma con contratti, sostituzioni, gerarchie e comportamenti polimorfici.

Nel prossimo capitolo affronteremo i generics, che permettono di scrivere tipi e metodi riutilizzabili senza perdere sicurezza di tipo.

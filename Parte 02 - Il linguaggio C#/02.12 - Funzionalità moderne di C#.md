# 02.12 — Funzionalità moderne di C#

In questo capitolo raccogliamo alcune funzionalità moderne di C# che sono già comparse nei capitoli precedenti o che incontreremo spesso nel resto del libro.

L’obiettivo non è elencare tutte le novità del linguaggio in modo enciclopedico. L’obiettivo è capire quali funzionalità migliorano davvero il codice quotidiano:

* riducono boilerplate;
* rendono più chiari i modelli;
* migliorano la sicurezza;
* semplificano la scrittura di dati strutturati;
* aiutano a lavorare con collezioni;
* migliorano performance in scenari specifici;
* rendono il linguaggio più espressivo.

C# è nato come linguaggio orientato agli oggetti fortemente tipizzato. Nel tempo è diventato un linguaggio molto più ricco, capace di supportare stili diversi: imperativo, object-oriented, funzionale, dichiarativo e orientato ai dati.

Il punto centrale del capitolo è questo:

```text id="ldmk8z"
Le funzionalità moderne non servono a scrivere codice più “furbo”.
Servono a scrivere codice più chiaro, più sicuro e più intenzionale.
```

# Global using

In molti file C# troviamo direttive `using` come queste:

```csharp id="pcnugk"
using System;
using System.Collections.Generic;
using System.Linq;
using System.Threading.Tasks;
```

Servono a rendere disponibili tipi definiti in namespace diversi.

Per esempio, per usare `List<T>` normalmente serve:

```csharp id="r65tqy"
using System.Collections.Generic;
```

Con C# moderno possiamo dichiarare alcuni `using` come **globali**.

Esempio:

```csharp id="chfyku"
global using System;
global using System.Collections.Generic;
global using System.Linq;
```

Un `global using` vale per tutti i file del progetto.

Quindi non dobbiamo ripetere gli stessi `using` in ogni file.

---

## Dove mettere i global using

Spesso si crea un file dedicato:

```text id="vq4p7n"
GlobalUsings.cs
```

Contenuto:

```csharp id="pr5rxk"
global using System;
global using System.Collections.Generic;
global using System.Linq;
```

In questo modo i namespace più comuni sono disponibili in tutto il progetto.

---

## Implicit usings

Nei progetti .NET moderni possiamo trovare anche questa impostazione nel `.csproj`:

```xml id="khqikh"
<PropertyGroup>
  <ImplicitUsings>enable</ImplicitUsings>
</PropertyGroup>
```

Con `ImplicitUsings` abilitato, l’SDK aggiunge automaticamente alcuni `using` comuni in base al tipo di progetto.

Per esempio, una console app moderna ha già disponibili molti namespace frequenti.

---

## Quando usare global using

Sono utili per namespace davvero comuni in tutto il progetto.

Esempi ragionevoli:

```csharp id="17q85e"
global using System;
global using System.Collections.Generic;
global using System.Linq;
```

In un progetto web:

```csharp id="cc02vv"
global using Microsoft.AspNetCore.Mvc;
```

In un progetto di test:

```csharp id="49bhfd"
global using Xunit;
```

---

## Quando evitarli

Non conviene mettere tutto nei global using.

Sconsigliato:

```csharp id="6gm2ay"
global using Progetto.Clienti.Servizi.Validazione.Avanzata.Interna;
global using Progetto.Fatturazione.Dettagli.Calcoli.Specializzati;
```

Se un namespace è usato solo in pochi file, meglio lasciarlo come `using` locale.

Regola pratica:

```text id="2w34ri"
Global using
  per dipendenze comuni e trasversali.

Using locale
  per dipendenze specifiche di un file.
```

# File-scoped namespace

Prima di C# 10, un namespace veniva normalmente scritto così:

```csharp id="xrwwcs"
namespace Gestionale.Clienti
{
    public class Cliente
    {
        public string Nome { get; set; } = "";
    }
}
```

Con i **file-scoped namespace** possiamo scrivere:

```csharp id="62azoz"
namespace Gestionale.Clienti;

public class Cliente
{
    public string Nome { get; set; } = "";
}
```

Il namespace vale per tutto il file.

La documentazione Microsoft indica global using e file-scoped namespace tra le funzionalità introdotte nell’evoluzione moderna di C#, pensate per ridurre rumore e rendere più chiara l’organizzazione del codice. ([Microsoft Learn][1])

---

## Vantaggio principale

Il vantaggio è ridurre un livello di indentazione.

Prima:

```csharp id="kjxm33"
namespace Gestionale.Clienti
{
    public class Cliente
    {
        public string Nome { get; set; } = "";
    }
}
```

Dopo:

```csharp id="y6o0vx"
namespace Gestionale.Clienti;

public class Cliente
{
    public string Nome { get; set; } = "";
}
```

Il secondo codice è più pulito.

---

## Regola pratica

Nei progetti nuovi, il file-scoped namespace è spesso la scelta migliore.

```csharp id="9hqp2e"
namespace MioProgetto.Domain;
```

Usa il namespace classico solo quando hai una ragione specifica, per esempio più namespace nello stesso file. Ma nei progetti ordinati, normalmente un file contiene tipi appartenenti allo stesso namespace.

# Raw string literals

Le **raw string literals** semplificano la scrittura di stringhe lunghe o contenenti caratteri speciali.

Sono molto utili per:

* JSON;
* XML;
* HTML;
* SQL;
* testo multilinea;
* esempi di codice;
* template;
* espressioni regolari.

Prima potevamo scrivere:

```csharp id="0tn0pq"
string json = "{ \"nome\": \"Emilie\", \"citta\": \"Aosta\" }";
```

Le virgolette interne devono essere precedute da `\`.

Con raw string literal:

```csharp id="0vp81t"
string json = """
{
  "nome": "Emilie",
  "citta": "Aosta"
}
""";
```

Molto più leggibile.

---

## Raw string con interpolazione

Possiamo combinare raw string e interpolazione.

```csharp id="4jxjhc"
string nome = "Emilie";
string citta = "Aosta";

string json = $$"""
{
  "nome": "{{nome}}",
  "citta": "{{citta}}"
}
""";
```

Output:

```json id="5m16l2"
{
  "nome": "Emilie",
  "citta": "Aosta"
}
```

Con `$$"""` usiamo doppie graffe `{{ }}` per interpolare.

Questo è utile quando il testo contiene già molte parentesi graffe, come JSON o template.

---

## Esempio con SQL

```csharp id="8xcgnb"
string sql = """
SELECT
    CLIE_CODI,
    CLIE_NOME,
    CLIE_COGN
FROM CLIENTI_GEN
WHERE CLIE_CODI = @CodiceCliente
""";
```

La query resta leggibile.

---

## Quando usarle

Regola pratica:

```text id="dqll7t"
Stringa breve
  usa la stringa normale.

Testo multilinea o con molte virgolette
  usa raw string literal.
```

# List patterns

I **list patterns** permettono di riconoscere la forma di una sequenza.

Li abbiamo già incontrati nel capitolo sul controllo del flusso.

Esempio:

```csharp id="7u4z4p"
int[] numeri = { 1, 2, 3 };

string descrizione = numeri switch
{
    [] => "Sequenza vuota",
    [1, 2, 3] => "Sequenza esatta 1, 2, 3",
    [var primo, ..] => $"Inizia con {primo}"
};
```

Output:

```text id="03g7gu"
Sequenza esatta 1, 2, 3
```

I list patterns sono stati introdotti in C# 11 insieme ad altre funzionalità come raw string literals e required members. ([Knowledge Continuum][2])

---

## Esempio con comandi

```csharp id="9v19mt"
string[] args = ["saluta", "Emilie"];

string risultato = args switch
{
    [] => "Nessun comando",
    ["help"] => "Mostra aiuto",
    ["saluta", var nome] => $"Ciao, {nome}",
    _ => "Comando non riconosciuto"
};

Console.WriteLine(risultato);
```

Output:

```text id="kgdwhk"
Ciao, Emilie
```

---

## Quando usarli

Sono molto utili quando vuoi interpretare una sequenza strutturata:

* argomenti da riga di comando;
* piccoli array;
* percorsi logici;
* token;
* comandi;
* pattern semplici su liste.

Non usarli per sostituire sempre LINQ o cicli normali.

---

# Required members

I **required members** obbligano chi crea un oggetto a valorizzare certe proprietà.

Esempio:

```csharp id="kgymwq"
public class Cliente
{
    public required string Nome { get; init; }
    public required string Cognome { get; init; }
    public string? Email { get; init; }
}
```

Uso corretto:

```csharp id="t8zdas"
Cliente cliente = new Cliente
{
    Nome = "Emilie",
    Cognome = "Rollandin",
    Email = "emilie@example.com"
};
```

Uso non corretto:

```csharp id="f0gj4h"
Cliente cliente = new Cliente
{
    Nome = "Emilie"
};
```

Il compilatore segnala che `Cognome` è obbligatorio.

---

## Required non valida il contenuto

Attenzione:

```csharp id="ovfqe0"
Cliente cliente = new Cliente
{
    Nome = "",
    Cognome = ""
};
```

Questo soddisfa `required`, ma non è detto che sia valido.

`required` garantisce che una proprietà venga inizializzata.
Non garantisce che il valore abbia senso.

Per quello servono costruttori, guard clauses e validazione.

---

## Quando usarlo

È utile soprattutto per:

* DTO;
* opzioni;
* modelli di configurazione;
* oggetti inizializzati con object initializer;
* modelli dove vuoi evitare proprietà non inizializzate.

Esempio:

```csharp id="pmvrpo"
public class EmailOptions
{
    public required string SmtpServer { get; init; }
    public required int Port { get; init; }
    public string? Username { get; init; }
}
```

---

# Primary constructors

I **primary constructors** permettono di dichiarare i parametri del costruttore direttamente nella dichiarazione della classe o struct.

Esempio:

```csharp id="nt28qw"
public class Cliente(string nome, string cognome)
{
    public string Nome { get; } = nome;
    public string Cognome { get; } = cognome;

    public string NomeCompleto => $"{Nome} {Cognome}";
}
```

Uso:

```csharp id="4m7a2l"
Cliente cliente = new Cliente("Emilie", "Rollandin");

Console.WriteLine(cliente.NomeCompleto);
```

Output:

```text id="9tonb2"
Emilie Rollandin
```

I primary constructors sono una delle funzionalità principali di C# 12, insieme alle collection expressions. ([Microsoft Learn][3])

---

## Quando sono utili

Sono utili per tipi semplici:

```csharp id="zxfq7w"
public class Punto(decimal x, decimal y)
{
    public decimal X { get; } = x;
    public decimal Y { get; } = y;
}
```

Oppure per servizi con dependency injection:

```csharp id="6xcu2o"
public class ClienteService(IClienteRepository repository, ILogger logger)
{
    public void Registra(string nome)
    {
        logger.Log($"Registro cliente {nome}");
        repository.Salva(nome);
    }
}
```

---

## Quando preferire un costruttore tradizionale

Se la costruzione richiede molte regole, il costruttore tradizionale può essere più leggibile.

```csharp id="lmny10"
public class Cliente
{
    public string Nome { get; }
    public string Cognome { get; }

    public Cliente(string nome, string cognome)
    {
        ArgumentException.ThrowIfNullOrWhiteSpace(nome);
        ArgumentException.ThrowIfNullOrWhiteSpace(cognome);

        Nome = nome;
        Cognome = cognome;
    }
}
```

Regola pratica:

```text id="9ri102"
Primary constructor
  ottimo per inizializzazioni semplici.

Costruttore tradizionale
  spesso migliore con molte validazioni o regole.
```

# Collection expressions

Le **collection expressions** permettono di inizializzare collezioni con una sintassi più compatta.

Esempio:

```csharp id="qeviyt"
int[] numeri = [1, 2, 3, 4, 5];
```

Con liste:

```csharp id="4vn6r6"
List<string> nomi = ["Emilie", "Mario", "Anna"];
```

Con `IEnumerable<T>`:

```csharp id="inqkew"
IEnumerable<int> valori = [10, 20, 30];
```

Le collection expressions sono state introdotte con C# 12. ([Microsoft Learn][3])

---

## Spread element

Possiamo usare `..` per inserire gli elementi di un’altra collezione.

```csharp id="jzg4wn"
int[] primi = [1, 2, 3];
int[] secondi = [4, 5, 6];

int[] tutti = [.. primi, .. secondi, 7, 8];

Console.WriteLine(string.Join(", ", tutti));
```

Output:

```text id="ze5fcp"
1, 2, 3, 4, 5, 6, 7, 8
```

---

## Prima e dopo

Prima:

```csharp id="ad6n9n"
List<string> nomi = new List<string>
{
    "Emilie",
    "Mario",
    "Anna"
};
```

Dopo:

```csharp id="4d8o1q"
List<string> nomi = ["Emilie", "Mario", "Anna"];
```

La seconda forma è più compatta.

---

## Quando usarle

Sono utili per collezioni piccole o inizializzazioni chiare.

```csharp id="jvddfp"
string[] codiciValidi = ["A", "B", "C"];
```

Per inizializzazioni complesse, la sintassi classica può restare più leggibile.

# Extension methods

Gli **extension methods** permettono di aggiungere metodi a un tipo esistente senza modificarlo.

Esempio:

```csharp id="nngkm5"
public static class StringExtensions
{
    public static bool ÈVuotaONulla(this string? testo)
    {
        return string.IsNullOrWhiteSpace(testo);
    }
}
```

Uso:

```csharp id="5g8wq4"
string? nome = "";

if (nome.ÈVuotaONulla())
{
    Console.WriteLine("Nome non valido.");
}
```

Il primo parametro preceduto da `this` indica il tipo esteso:

```csharp id="d7rs4s"
this string? testo
```

---

## Perché sono utili

Gli extension methods sono utili per:

* migliorare la leggibilità;
* aggiungere helper a tipi esistenti;
* organizzare funzioni di supporto;
* creare API fluide;
* lavorare con LINQ.

LINQ è basato in larga parte su extension methods.

Esempio:

```csharp id="r9o608"
List<int> numeri = [1, 2, 3, 4, 5];

List<int> pari = numeri
    .Where(n => n % 2 == 0)
    .ToList();
```

`Where` e `ToList` sono extension methods.

---

## Quando evitarli

Non bisogna abusarne.

Sconsigliato:

```csharp id="l92r1w"
public static void SalvaSuDatabase(this string testo)
{
    // ...
}
```

Una stringa non dovrebbe “sapere” come salvarsi su database.

Regola pratica:

```text id="mhdabt"
Extension method
  buono per operazioni naturali sul tipo.

Servizio separato
  migliore per logica applicativa o infrastrutturale.
```

---

# Extension members moderni

C# 14 introduce una forma più moderna per dichiarare membri di estensione tramite **extension blocks**. Questa evoluzione permette di raggruppare membri di estensione e, oltre ai metodi, supportare anche altre forme come proprietà o operatori di estensione. Microsoft include gli extension members tra le novità principali di C# 14. ([Microsoft Learn][4])

Esempio concettuale:

```csharp id="7kxhe8"
public static class StringExtensions
{
    extension(string testo)
    {
        public bool ÈLunga => testo.Length > 100;

        public string Tronca(int lunghezzaMassima)
        {
            if (testo.Length <= lunghezzaMassima)
            {
                return testo;
            }

            return testo[..lunghezzaMassima];
        }
    }
}
```

Uso:

```csharp id="zcv58f"
string descrizione = "Testo molto lungo...";

if (descrizione.ÈLunga)
{
    Console.WriteLine(descrizione.Tronca(100));
}
```

---

## Come trattarli nel libro

Per un percorso didattico conviene presentarli come funzionalità moderna, ma non sostituire subito gli extension methods classici.

Gli extension methods tradizionali sono ancora fondamentali e molto presenti nel codice esistente.

Regola pratica:

```text id="rpnuwd"
Extension methods classici
  indispensabili da conoscere.

Extension members moderni
  utili da conoscere se il libro tratta C# 14 o successivo.
```

# `Span<T>` e `ReadOnlySpan<T>`

`Span<T>` e `ReadOnlySpan<T>` sono tipi pensati per lavorare con porzioni contigue di memoria in modo efficiente e sicuro.

Sono usati soprattutto per:

* performance;
* parsing;
* elaborazione di stringhe;
* buffer;
* I/O;
* riduzione delle allocazioni;
* codice a basso livello ma sicuro.

Esempio introduttivo:

```csharp id="ssx4ov"
int[] numeri = [1, 2, 3, 4, 5];

Span<int> parte = numeri.AsSpan(1, 3);

parte[0] = 99;

Console.WriteLine(string.Join(", ", numeri));
```

Output:

```text id="unacyf"
1, 99, 3, 4, 5
```

`parte` non è una nuova copia dell’array. È una vista su una porzione dell’array.

---

## ReadOnlySpan<T>

`ReadOnlySpan<T>` permette di leggere una porzione di memoria senza modificarla.

```csharp id="xz5bky"
string testo = "ABCDEF";

ReadOnlySpan<char> parte = testo.AsSpan(1, 3);

Console.WriteLine(parte.ToString());
```

Output:

```text id="czmwsx"
BCD
```

---

## Perché sono importanti

Senza `Span<T>`, molte operazioni su stringhe e array creano nuove allocazioni.

Con `Span<T>` possiamo lavorare su porzioni dei dati esistenti.

Esempio concettuale:

```csharp id="9uidk1"
ReadOnlySpan<char> codice = "CLIENTE-12345".AsSpan();

ReadOnlySpan<char> prefisso = codice[..7];
ReadOnlySpan<char> numero = codice[8..];
```

Questo può essere molto efficiente.

---

## C# moderno e span

C# 14 migliora l’integrazione tra array, `Span<T>` e `ReadOnlySpan<T>`, riconoscendo meglio le conversioni tra questi tipi e permettendo scenari più naturali con metodi di estensione, conversioni e inferenza generica. ([Microsoft Learn][4])

Per ora basta ricordare:

```text id="v9pl5d"
Span<T>
  vista modificabile su memoria contigua.

ReadOnlySpan<T>
  vista in sola lettura su memoria contigua.
```

Sono strumenti importanti, ma non vanno usati ovunque.

Nel codice gestionale ordinario, spesso `string`, array, liste e LINQ sono sufficienti. `Span<T>` diventa rilevante quando le performance e le allocazioni sono davvero importanti.

# Miglioramenti alle stringhe

Il C# moderno ha introdotto diverse funzionalità che migliorano il lavoro con le stringhe.

---

## Interpolazione

Già vista nei capitoli precedenti:

```csharp id="6w3ph3"
string nome = "Emilie";
int eta = 42;

string messaggio = $"Ciao, {nome}. Hai {eta} anni.";
```

Molto meglio di:

```csharp id="nf9su2"
string messaggio = "Ciao, " + nome + ". Hai " + eta + " anni.";
```

---

## Raw string literals

Utili per stringhe multilinea:

```csharp id="gw4w18"
string html = """
<div>
    <h1>Titolo</h1>
    <p>Contenuto</p>
</div>
""";
```

---

## Stringhe interpolate multilinea

Possiamo scrivere testi complessi in modo leggibile:

```csharp id="tzk05u"
string nome = "Emilie";
string email = "emilie@example.com";

string testo = $$"""
Cliente:
  Nome: {{nome}}
  Email: {{email}}
""";
```

---

## Confronti espliciti

Per confrontare stringhe in modo robusto:

```csharp id="544akp"
bool uguali = string.Equals(
    "emilie",
    "Emilie",
    StringComparison.OrdinalIgnoreCase);
```

È meglio essere espliciti sul tipo di confronto, soprattutto quando il risultato non deve dipendere dalla cultura corrente.

---

## Miglioramenti in .NET

Oltre al linguaggio C#, anche .NET evolve. Per esempio, .NET 10 introduce l’ordinamento numerico per il confronto delle stringhe, utile in casi in cui valori come `"file2"` e `"file10"` devono essere ordinati in modo naturale. Questo è un miglioramento della libreria .NET, non della sintassi C# in senso stretto. ([Microsoft Learn][4])

# Null-conditional assignment

Tra le funzionalità moderne di C# 14 c’è anche la **null-conditional assignment**.

Permette di assegnare un valore a una proprietà o a un campo solo se l’oggetto a sinistra non è `null`.

Esempio:

```csharp id="ros6a5"
cliente?.Nome = "Emilie";
```

Significa:

```text id="4crhbb"
se cliente non è null
  assegna Nome = "Emilie"

se cliente è null
  non fare nulla
```

La funzionalità è indicata tra le novità di C# 14 insieme a extension members e `nameof` con generici non associati. ([Microsoft for Developers][5])

---

## Prima di C# 14

Prima si scriveva:

```csharp id="9c8r26"
if (cliente is not null)
{
    cliente.Nome = "Emilie";
}
```

Con la nuova sintassi:

```csharp id="6bguki"
cliente?.Nome = "Emilie";
```

---

## Attenzione

Non bisogna usarla per nascondere bug.

Se `cliente` non dovrebbe mai essere `null`, meglio non scrivere:

```csharp id="w6w3iy"
cliente?.Nome = "Emilie";
```

Meglio far emergere il problema:

```csharp id="g7b4l9"
ArgumentNullException.ThrowIfNull(cliente);

cliente.Nome = "Emilie";
```

Regola pratica:

```text id="g1ctom"
Usa ?.=
quando null è una possibilità prevista e innocua.

Evitalo
quando null indica uno stato sbagliato del programma.
```

---

# `nameof` con generici non associati

Abbiamo già visto `nameof`.

```csharp id="b03m2p"
string nome = nameof(Cliente);
```

Con C# 14, `nameof` può essere usato anche con tipi generici non associati:

```csharp id="w1imny"
string nomeLista = nameof(List<>);
string nomeDizionario = nameof(Dictionary<,>);
```

Output:

```text id="wjff96"
List
Dictionary
```

Prima era necessario usare un tipo generico chiuso, per esempio:

```csharp id="rgqw6p"
nameof(List<int>)
```

La proposta ufficiale specifica che `nameof(List<>)` restituisce `"List"` e che non è necessario fornire un tipo generico fittizio solo per ottenere il nome. ([Microsoft Learn][6])

---

## Quando serve

È utile soprattutto in:

* diagnostica;
* logging;
* analyzer;
* generatori di codice;
* messaggi di errore;
* librerie generiche.

Nel codice quotidiano è una comodità piccola, ma coerente con l’evoluzione del linguaggio.

# Evoluzione del linguaggio da C# 1 a C# moderno

C# non è rimasto fermo.

È nato come linguaggio object-oriented fortemente tipizzato, vicino all’ecosistema .NET Framework. Nel tempo ha incorporato molte idee moderne: generics, LINQ, async/await, pattern matching, record, nullable reference types, top-level statements, file-scoped namespace, raw string literals, collection expressions e molto altro.

La documentazione Microsoft mantiene una cronologia delle versioni del linguaggio e mostra come ogni fase abbia aggiunto strumenti per ridurre boilerplate, migliorare sicurezza, supportare nuovi stili di programmazione e rendere più espressivo il codice. ([Microsoft Learn][1])

---

## C# 1: base object-oriented

C# 1 includeva già:

```text id="u1wly4"
classi
struct
interfacce
delegati
eventi
proprietà
eccezioni
```

# Come usare le funzionalità moderne senza abusarne

Le nuove funzionalità non devono essere usate solo perché esistono.

Un buon criterio è chiedersi:

```text id="vn2soa"
Il codice diventa più chiaro?
Il contratto diventa più esplicito?
Riduco boilerplate inutile?
Evito errori comuni?
Miglioro performance dove serve davvero?
```

Se la risposta è sì, la funzionalità moderna è probabilmente utile.

Se invece il codice diventa più difficile da leggere, meglio restare con una forma più semplice.

---

## Esempio di buon uso

```csharp id="43fdcc"
public record ClienteDto(
    int Id,
    string Nome,
    string Cognome);
```

Qui il record è perfetto: rappresenta dati.

---

## Esempio di abuso

```csharp id="lkqa8w"
var risultato = dati switch
{
    [_, [_, var x, ..], ..] when x?.Trim().Length > 3 => x,
    _ => null
};
```

Magari è corretto, ma non è leggibile.

Il codice moderno non deve diventare un indovinello.

# Esempio completo del capitolo

Costruiamo un piccolo esempio che usa diverse funzionalità moderne in modo ragionato.

Scenario:

```text id="u6vs6r"
Vogliamo leggere una lista di comandi.
Ogni comando può creare un cliente.
Usiamo record, required, collection expressions, list patterns, raw string e file-scoped namespace.
```

File:

```text id="d47n06"
ClienteDto.cs
```

```csharp id="b6dpjo"
namespace Gestionale.Clienti;

public record ClienteDto(
    int Id,
    string Nome,
    string Cognome);
```

File:

```text id="pcn2ag"
ClienteOptions.cs
```

```csharp id="jzejur"
namespace Gestionale.Clienti;

public class ClienteOptions
{
    public required string PrefissoCodice { get; init; }
    public int LunghezzaMinimaNome { get; init; } = 2;
}
```

File:

```text id="rrbvzv"
Program.cs
```

```csharp id="g5x6bb"
using Gestionale.Clienti;

ClienteOptions options = new ClienteOptions
{
    PrefissoCodice = "CLI"
};

string[] comandi =
[
    "crea 1 Emilie Rollandin",
    "crea 2 Mario Rossi",
    "help"
];

foreach (string comando in comandi)
{
    string[] parti = comando.Split(' ', StringSplitOptions.RemoveEmptyEntries);

    string risposta = parti switch
    {
        ["help"] => """
                    Comandi disponibili:
                      crea <id> <nome> <cognome>
                      help
                    """,

        ["crea", var idTesto, var nome, var cognome]
            when int.TryParse(idTesto, out int id)
              && nome.Length >= options.LunghezzaMinimaNome =>
            CreaMessaggioCliente(new ClienteDto(id, nome, cognome), options),

        ["crea", ..] =>
            "Uso corretto: crea <id> <nome> <cognome>",

        _ =>
            "Comando non riconosciuto"
    };

    Console.WriteLine(risposta);
}

static string CreaMessaggioCliente(ClienteDto cliente, ClienteOptions options)
{
    string codice = $"{options.PrefissoCodice}-{cliente.Id:0000}";

    return $$"""
             Cliente creato:
               Codice: {{codice}}
               Nome: {{cliente.Nome}}
               Cognome: {{cliente.Cognome}}
             """;
}
```

Output:

```text id="oib6k2"
Cliente creato:
  Codice: CLI-0001
  Nome: Emilie
  Cognome: Rollandin

Cliente creato:
  Codice: CLI-0002
  Nome: Mario
  Cognome: Rossi

Comandi disponibili:
  crea <id> <nome> <cognome>
  help
```

In questo esempio abbiamo usato:

```text id="0q3zfl"
file-scoped namespace
  per ridurre indentazione.

record
  per rappresentare ClienteDto.

required
  per obbligare PrefissoCodice.

collection expressions
  per inizializzare array.

list patterns
  per interpretare comandi.

raw string literals
  per messaggi multilinea leggibili.

interpolazione
  per costruire output.
```

# Errori comuni

## Usare funzionalità moderne solo per moda

Non tutto deve diventare pattern matching, record o expression-bodied member.

Il codice deve essere prima leggibile.

---

## Mettere troppi global using

Sconsigliato:

```csharp id="6ei97a"
global using Tutto.Il.Mio.Progetto.Ovunque;
```

Meglio limitarsi ai namespace davvero trasversali.

---

## Usare primary constructors con troppa logica nascosta

Se il tipo richiede molte validazioni, un costruttore tradizionale può essere più chiaro.

---

## Usare required al posto della validazione

```csharp id="3trssj"
public required string Nome { get; init; }
```

non impedisce:

```csharp id="hh0mub"
Nome = "";
```

`required` e validazione risolvono problemi diversi.

---

## Usare raw string per testi brevi

Inutile:

```csharp id="tfgqvx"
string nome = """
Emilie
""";
```

Meglio:

```csharp id="y6fycs"
string nome = "Emilie";
```

---

## Usare Span<T> troppo presto

`Span<T>` è potente, ma non deve complicare codice normale.

Prima scrivi codice chiaro.
Ottimizza con `Span<T>` quando hai un motivo misurabile.

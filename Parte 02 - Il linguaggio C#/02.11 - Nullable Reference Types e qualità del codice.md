# 02.11 — Nullable Reference Types e qualità del codice

Il valore `null` è una delle cause più comuni di errori nei programmi C#.

Chi ha sviluppato in .NET per un po’ di tempo ha quasi certamente incontrato un errore di questo tipo:

```text id="2z3kqm"
System.NullReferenceException: Object reference not set to an instance of an object.
```

Questo errore significa, in sostanza:

> stai provando a usare un oggetto, ma in realtà la variabile non contiene nessun oggetto: contiene `null`.

Per molti anni, in C#, ogni reference type poteva essere `null` senza che il compilatore potesse aiutare davvero.

Esempio:

```csharp id="1h5zxk"
string nome = null;

Console.WriteLine(nome.Length);
```

Questo codice poteva compilare, ma fallire a runtime.

Con il C# moderno sono stati introdotti i **Nullable Reference Types**, uno strumento che aiuta a rendere esplicito se una variabile, proprietà, parametro o valore di ritorno può essere `null`.

Il punto centrale del capitolo è questo:

> `string` e `string?` non comunicano la stessa intenzione.
> Il primo dice “mi aspetto sempre un valore”.
> Il secondo dice “qui il valore può mancare”.

---

# Il problema dei `null`

`null` significa assenza di riferimento.

Esempio:

```csharp id="cgzpw8"
Cliente? cliente = null;
```

La variabile `cliente` esiste, ma non punta a nessun oggetto.

Se proviamo a usare un membro dell’oggetto:

```csharp id="ieuuog"
Console.WriteLine(cliente.Nome);
```

il programma fallisce, perché non c’è nessun oggetto da cui leggere `Nome`.

---

## Esempio classico

```csharp id="a4851x"
public class Cliente
{
    public string Nome { get; set; }
}
```

In versioni tradizionali di C#, questo codice era molto comune.

Il problema è che `Nome` poteva essere `null`.

```csharp id="nbzdmn"
Cliente cliente = new Cliente();

Console.WriteLine(cliente.Nome.Length);
```

`Nome` non è stato inizializzato, quindi può essere `null`.

Il codice può generare una `NullReferenceException`.

---

## Il problema non è `null` in sé

`null` non è sempre sbagliato.

A volte l’assenza di valore è significativa.

Esempi:

```text id="vhl12t"
Data di cancellazione non presente
Secondo nome non indicato
Cliente non trovato
Documento senza data di scadenza
Risultato opzionale
```

Il problema nasce quando non è chiaro se `null` sia ammesso oppure no.

Esempio ambiguo:

```csharp id="olgm7e"
public string TrovaNomeCliente(int id)
{
    // può restituire null?
}
```

Chi usa questo metodo non sa se deve gestire `null`.

Meglio:

```csharp id="gdnfej"
public string? TrovaNomeCliente(int id)
{
    // null significa cliente non trovato
}
```

oppure:

```csharp id="8arvt7"
public string TrovaNomeCliente(int id)
{
    // se non trovato, lancia eccezione o restituisce un valore valido
}
```

Il contratto deve essere chiaro.

---

# Nullable Reference Types

I **Nullable Reference Types** permettono di distinguere tra reference type che dovrebbero essere sempre valorizzati e reference type che possono essere `null`.

Con questa funzionalità abilitata:

```csharp id="7chmva"
string nome = "Emilie";
```

significa:

> `nome` non dovrebbe essere `null`.

Invece:

```csharp id="l7vwbr"
string? soprannome = null;
```

significa:

> `soprannome` può essere una stringa oppure `null`.

---

## Abilitare Nullable Reference Types

Nei progetti moderni troviamo spesso questa impostazione nel file `.csproj`:

```xml id="gw88gl"
<PropertyGroup>
  <Nullable>enable</Nullable>
</PropertyGroup>
```

Con questa opzione il compilatore analizza il codice e segnala possibili problemi di null.

Esempio:

```csharp id="budlsc"
string nome = null;
```

Il compilatore produce un warning.

Non è necessariamente un errore bloccante, ma è un segnale importante:

> stai assegnando `null` a qualcosa che hai dichiarato come non nullable.

---

## Warning, non magia

I Nullable Reference Types non eliminano automaticamente tutti i problemi di null.

Sono un sistema di analisi statica del compilatore.

Significa che il compilatore cerca di capire dove potresti avere un valore `null` non gestito.

Esempio:

```csharp id="20vetf"
string? nome = TrovaNome();

Console.WriteLine(nome.Length);
```

Il compilatore segnala un warning, perché `nome` potrebbe essere `null`.

Versione corretta:

```csharp id="t23f8o"
string? nome = TrovaNome();

if (nome is not null)
{
    Console.WriteLine(nome.Length);
}
```

Oppure:

```csharp id="59ulmv"
Console.WriteLine(nome?.Length);
```

Oppure:

```csharp id="28g5dn"
Console.WriteLine(nome?.Length ?? 0);
```

---

# `string` vs `string?`

La distinzione tra `string` e `string?` è uno degli esempi più importanti.

---

## `string`

```csharp id="ypniqd"
string nome = "Emilie";
```

Comunica:

> questa variabile deve contenere sempre una stringa valida.

Questo codice non dovrebbe essere accettato senza warning:

```csharp id="83h0xd"
string nome = null;
```

Se un metodo riceve:

```csharp id="r4a31x"
void Saluta(string nome)
{
    Console.WriteLine($"Ciao, {nome}");
}
```

il contratto implicito è:

> chi chiama `Saluta` deve passare un nome non nullo.

---

## `string?`

```csharp id="hmk2hk"
string? soprannome = null;
```

Comunica:

> qui il testo può mancare.

Un metodo che accetta `string?` deve gestire il caso `null`.

```csharp id="j39qlq"
void StampaSoprannome(string? soprannome)
{
    if (string.IsNullOrWhiteSpace(soprannome))
    {
        Console.WriteLine("Nessun soprannome disponibile.");
        return;
    }

    Console.WriteLine(soprannome);
}
```

Il punto è importante:

```text id="pkjeix"
string
  chi chiama deve fornire un valore.

string?
  chi implementa deve gestire l’assenza di valore.
```

---

## Esempio con proprietà

Sconsigliato:

```csharp id="7c6n54"
public class Cliente
{
    public string Nome { get; set; }
    public string? Soprannome { get; set; }
}
```

Con nullable abilitato, `Nome` produce warning perché non viene inizializzato.

Soluzioni possibili.

### Valore predefinito

```csharp id="ahx4m4"
public class Cliente
{
    public string Nome { get; set; } = "";
    public string? Soprannome { get; set; }
}
```

### Costruttore

```csharp id="b3l8mx"
public class Cliente
{
    public string Nome { get; }
    public string? Soprannome { get; }

    public Cliente(string nome, string? soprannome = null)
    {
        if (string.IsNullOrWhiteSpace(nome))
        {
            throw new ArgumentException("Il nome è obbligatorio.", nameof(nome));
        }

        Nome = nome;
        Soprannome = soprannome;
    }
}
```

### Proprietà required

```csharp id="rplx2n"
public class Cliente
{
    public required string Nome { get; init; }
    public string? Soprannome { get; init; }
}
```

Attenzione: `required` obbliga a inizializzare `Nome`, ma non impedisce di passare una stringa vuota. Per quello serve validazione.

---

# Nullable value types e nullable reference types

È importante non confondere:

```csharp id="sondcp"
int?
```

con:

```csharp id="0z95w4"
string?
```

Sembrano simili, ma non sono esattamente la stessa cosa.

---

## `int?`

`int` è un value type.

```csharp id="qaehaf"
int? eta = null;
```

`int?` è una forma abbreviata di:

```csharp id="gfbqas"
Nullable<int>
```

Qui il valore può essere un intero oppure `null`.

---

## `string?`

`string` è un reference type.

```csharp id="q9bvzx"
string? nome = null;
```

Qui il `?` non crea un `Nullable<string>`.
Serve al compilatore per sapere che questa variabile può essere `null`.

Regola pratica:

```text id="zgi51j"
int?
  vero tipo generico Nullable<int>.

string?
  annotazione nullable per reference type.
```

---

# Operatore `!`

L’operatore `!` dopo un’espressione si chiama **null-forgiving operator**.

Serve a dire al compilatore:

> so che questo valore sembra nullable, ma in questo punto ti garantisco che non è null.

Esempio:

```csharp id="b86bng"
string? nome = TrovaNome();

Console.WriteLine(nome!.Length);
```

Il `!` elimina il warning.

Ma attenzione: non elimina il rischio reale.

Se `nome` è davvero `null`, il programma fallirà comunque a runtime.

---

## Esempio pericoloso

```csharp id="yzg3vj"
string? nome = null;

Console.WriteLine(nome!.Length);
```

Questo compila senza warning, ma genera una `NullReferenceException`.

L’operatore `!` non controlla nulla.
Dice solo al compilatore di fidarsi.

---

## Quando usare `!`

Va usato raramente.

Può essere utile quando:

* il compilatore non riesce a capire una garanzia che noi sappiamo essere vera;
* una proprietà viene inizializzata da un framework;
* un test costruisce intenzionalmente uno scenario;
* un controllo avviene in un metodo che il compilatore non riesce a interpretare.

Esempio:

```csharp id="590h63"
public class ClienteController
{
    private ClienteService service = null!;

    public void Inizializza(ClienteService clienteService)
    {
        service = clienteService;
    }
}
```

Questo può comparire in scenari con framework, anche se spesso Dependency Injection o costruttori espliciti sono preferibili.

---

## Quando evitarlo

Sconsigliato:

```csharp id="ifdgee"
Console.WriteLine(cliente!.Nome);
```

Se non sei davvero sicuro che `cliente` non sia `null`, meglio controllare:

```csharp id="qbsf9c"
if (cliente is null)
{
    Console.WriteLine("Cliente non disponibile.");
    return;
}

Console.WriteLine(cliente.Nome);
```

Regola pratica:

```text id="arx43u"
Usa ! solo quando il compilatore non capisce una garanzia reale.
Non usarlo per zittire warning che non vuoi affrontare.
```

---

# Annotazioni nullable

Le annotazioni nullable servono a dichiarare il contratto di una API.

Esempio:

```csharp id="pmpbfn"
public string? TrovaNomeCliente(int id)
```

Questa firma comunica:

> il metodo può restituire una stringa oppure `null`.

Chi chiama il metodo deve gestire il caso `null`.

---

## Parametro nullable

```csharp id="91sjj6"
void StampaNome(string? nome)
{
    if (string.IsNullOrWhiteSpace(nome))
    {
        Console.WriteLine("Nome non disponibile.");
        return;
    }

    Console.WriteLine(nome);
}
```

Il metodo accetta `null`.

---

## Parametro non nullable

```csharp id="eodxdl"
void Saluta(string nome)
{
    ArgumentException.ThrowIfNullOrWhiteSpace(nome);

    Console.WriteLine($"Ciao, {nome}");
}
```

Il metodo non vuole ricevere `null`.

Con nullable abilitato, chi prova a passare un `string?` riceverà un warning.

Esempio:

```csharp id="4zaef7"
string? nome = TrovaNome();

Saluta(nome); // warning: nome potrebbe essere null
```

Versione corretta:

```csharp id="zohg9g"
string? nome = TrovaNome();

if (nome is not null)
{
    Saluta(nome);
}
```

---

## Proprietà nullable

```csharp id="6jwg28"
public class Cliente
{
    public string Nome { get; }
    public string? Email { get; }

    public Cliente(string nome, string? email)
    {
        ArgumentException.ThrowIfNullOrWhiteSpace(nome);

        Nome = nome;
        Email = email;
    }
}
```

Qui il contratto è chiaro:

```text id="gw8gwp"
Nome
  obbligatorio.

Email
  opzionale.
```

---

# API più sicure

I Nullable Reference Types aiutano a progettare API più sicure.

Una buona API comunica chiaramente:

* quali parametri sono obbligatori;
* quali parametri sono opzionali;
* quali proprietà possono mancare;
* quali metodi possono non trovare un risultato;
* quali valori devono essere sempre presenti.

---

## API ambigua

```csharp id="4iq9cn"
public Cliente TrovaCliente(int id)
{
    // può restituire null?
}
```

Questa firma dice che restituisce sempre un `Cliente`, ma se internamente può tornare `null`, il contratto è falso.

---

## API esplicita

```csharp id="405dzx"
public Cliente? TrovaCliente(int id)
{
    // null se non trovato
}
```

Ora chi chiama sa che il cliente potrebbe mancare.

Uso:

```csharp id="x5qixm"
Cliente? cliente = repository.TrovaCliente(10);

if (cliente is null)
{
    Console.WriteLine("Cliente non trovato.");
    return;
}

Console.WriteLine(cliente.Nome);
```

---

## API che non ammette assenza

Se l’assenza del cliente è un errore, meglio comunicarlo con un nome diverso o con un’eccezione.

```csharp id="xcl3gm"
public Cliente OttieniClienteObbligatorio(int id)
{
    Cliente? cliente = TrovaCliente(id);

    if (cliente is null)
    {
        throw new InvalidOperationException($"Cliente {id} non trovato.");
    }

    return cliente;
}
```

Il nome `OttieniClienteObbligatorio` è più chiaro di un generico `GetCliente`.

---

## Try pattern

Un altro approccio è il pattern `Try`.

```csharp id="g8z1p3"
public bool TryTrovaCliente(int id, out Cliente? cliente)
{
    cliente = CercaNelDatabase(id);

    return cliente is not null;
}
```

Uso:

```csharp id="g6w0ol"
if (repository.TryTrovaCliente(10, out Cliente? cliente))
{
    Console.WriteLine(cliente.Nome);
}
else
{
    Console.WriteLine("Cliente non trovato.");
}
```

Il pattern `Try` è familiare perché lo abbiamo già visto con:

```csharp id="bu7z5p"
int.TryParse(...)
decimal.TryParse(...)
DateTime.TryParse(...)
```

---

# Contratti impliciti ed espliciti

Ogni metodo comunica un contratto.

A volte il contratto è implicito, cioè non dichiarato chiaramente.

A volte è esplicito, cioè visibile nella firma.

---

## Contratto implicito debole

```csharp id="ta7kxj"
public void InviaEmail(string email)
{
    // email non deve essere null o vuota
}
```

Il contratto esiste, ma non è del tutto visibile.

Possiamo rafforzarlo con validazione:

```csharp id="nxpylo"
public void InviaEmail(string email)
{
    ArgumentException.ThrowIfNullOrWhiteSpace(email);

    Console.WriteLine($"Invio email a {email}");
}
```

---

## Contratto esplicito con nullable

```csharp id="7n6up1"
public void ImpostaSecondaEmail(string? emailSecondaria)
{
    if (string.IsNullOrWhiteSpace(emailSecondaria))
    {
        Console.WriteLine("Nessuna seconda email impostata.");
        return;
    }

    Console.WriteLine($"Seconda email: {emailSecondaria}");
}
```

Qui `string?` dice chiaramente che l’assenza è ammessa.

---

## Contratto esplicito con tipo dedicato

A volte il miglior contratto non è `string`, ma un tipo dedicato.

Meno espressivo:

```csharp id="0f6nei"
public void InviaEmail(string email)
```

Più espressivo:

```csharp id="e1xeju"
public void InviaEmail(EmailAddress email)
```

Con un tipo dedicato:

```csharp id="ajlq6n"
public sealed class EmailAddress
{
    public string Valore { get; }

    public EmailAddress(string valore)
    {
        ArgumentException.ThrowIfNullOrWhiteSpace(valore);

        if (!valore.Contains('@'))
        {
            throw new ArgumentException("Email non valida.", nameof(valore));
        }

        Valore = valore;
    }

    public override string ToString()
    {
        return Valore;
    }
}
```

Ora il metodo non deve più ricevere una stringa qualunque. Riceve un oggetto già validato.

---

# Null Object pattern

Il **Null Object pattern** è una tecnica che consiste nel sostituire `null` con un oggetto reale che rappresenta un comportamento “vuoto” o neutro.

Esempio classico: logging.

Interfaccia:

```csharp id="sahecv"
public interface ILogger
{
    void Log(string messaggio);
}
```

Implementazione reale:

```csharp id="gsm3ne"
public class ConsoleLogger : ILogger
{
    public void Log(string messaggio)
    {
        Console.WriteLine(messaggio);
    }
}
```

Implementazione nulla:

```csharp id="3bnfc6"
public class NullLogger : ILogger
{
    public void Log(string messaggio)
    {
        // Non fa nulla.
    }
}
```

Uso:

```csharp id="kbd495"
public class ClienteService
{
    private readonly ILogger logger;

    public ClienteService(ILogger? logger)
    {
        this.logger = logger ?? new NullLogger();
    }

    public void RegistraCliente(string nome)
    {
        logger.Log($"Registro cliente {nome}");

        Console.WriteLine("Cliente registrato.");
    }
}
```

In questo modo il resto della classe non deve controllare ogni volta:

```csharp id="l0it7b"
logger?.Log(...)
```

perché `logger` non sarà mai `null`.

---

## Quando usare Null Object

È utile quando:

* un comportamento opzionale può essere rappresentato da una versione neutra;
* vogliamo evitare controlli `null` ripetuti;
* l’assenza di comportamento è accettabile;
* vogliamo semplificare il codice chiamante.

Esempi:

```text id="hr8lbm"
NullLogger
NullNotifier
NullCache
NullMetricsCollector
```

---

## Quando non usarlo

Non usare Null Object se l’assenza dell’oggetto è un errore.

Esempio:

```csharp id="ucfzdl"
public class PagamentoService
{
    private readonly IPaymentGateway gateway;
}
```

Se il gateway di pagamento manca, probabilmente non vogliamo un `NullPaymentGateway` che finge di pagare senza fare nulla.

In quel caso meglio fallire subito.

Regola pratica:

```text id="wb7e31"
Null Object
  buono per comportamenti opzionali e neutri.

Eccezione
  migliore quando l’assenza è un errore di configurazione o di logica.
```

---

# Optional values

Un valore opzionale può essere rappresentato in modi diversi.

In C# le scelte più comuni sono:

```text id="w3jtdd"
null
Nullable<T>
Try pattern
Result<T>
Option<T> custom
Null Object
```

Non esiste una sola soluzione corretta per tutti i casi.

---

## Usare `null`

Va bene quando l’assenza è semplice e comprensibile.

```csharp id="yt4x1v"
public string? Soprannome { get; init; }
```

Qui `null` significa:

> nessun soprannome.

È ragionevole.

---

## Usare nullable value type

```csharp id="7sl2rj"
public DateOnly? DataCancellazione { get; private set; }
```

Qui `null` significa:

> non cancellato.

Anche questo è chiaro.

---

## Usare Try pattern

```csharp id="z42nza"
public bool TryTrovaCliente(int id, out Cliente? cliente)
```

Adatto quando:

* vogliamo evitare eccezioni;
* il fallimento è normale;
* il chiamante deve gestire successo/fallimento.

---

## Usare Result<T>

```csharp id="5almg6"
public Result<Cliente> CercaCliente(int id)
```

Adatto quando vogliamo restituire:

* successo/fallimento;
* valore;
* messaggio di errore;
* eventualmente codici o dettagli.

Esempio:

```csharp id="xwfu5n"
public record Result<T>(bool Successo, T? Valore, string? Errore);
```

---

## Usare Null Object

```csharp id="ipmskz"
ILogger logger = loggerOpzionale ?? new NullLogger();
```

Adatto quando l’assenza può essere rappresentata da comportamento neutro.

---

# Errori comuni

## Mettere `?` ovunque

Sbagliato:

```csharp id="cdtcfw"
public class Cliente
{
    public string? Nome { get; set; }
    public string? Cognome { get; set; }
    public string? CodiceFiscale { get; set; }
}
```

Se `Nome` e `Cognome` sono obbligatori, non devono essere nullable.

Meglio:

```csharp id="yeyt5c"
public class Cliente
{
    public string Nome { get; }
    public string Cognome { get; }
    public string? CodiceFiscale { get; }

    public Cliente(string nome, string cognome, string? codiceFiscale)
    {
        ArgumentException.ThrowIfNullOrWhiteSpace(nome);
        ArgumentException.ThrowIfNullOrWhiteSpace(cognome);

        Nome = nome;
        Cognome = cognome;
        CodiceFiscale = codiceFiscale;
    }
}
```

---

## Usare `!` per zittire il compilatore

Sospetto:

```csharp id="eghl7a"
Console.WriteLine(cliente!.Nome);
```

Spesso significa che stiamo ignorando un problema invece di risolverlo.

Meglio:

```csharp id="ww8wpo"
if (cliente is null)
{
    Console.WriteLine("Cliente non trovato.");
    return;
}

Console.WriteLine(cliente.Nome);
```

---

## Restituire `null` da un metodo non nullable

Sbagliato:

```csharp id="pq1wqv"
public Cliente TrovaCliente(int id)
{
    return null;
}
```

La firma promette un `Cliente`, ma restituisce `null`.

Corretto:

```csharp id="lrdxej"
public Cliente? TrovaCliente(int id)
{
    return null;
}
```

Oppure:

```csharp id="jbj0ug"
public Cliente OttieniCliente(int id)
{
    throw new InvalidOperationException("Cliente non trovato.");
}
```

A seconda del contratto desiderato.

---

## Non inizializzare proprietà non nullable

Sbagliato:

```csharp id="fy6fdl"
public class Cliente
{
    public string Nome { get; set; }
}
```

Meglio:

```csharp id="3mxsbk"
public class Cliente
{
    public string Nome { get; set; } = "";
}
```

Ancora meglio, se il nome è obbligatorio:

```csharp id="waiff2"
public class Cliente
{
    public string Nome { get; }

    public Cliente(string nome)
    {
        ArgumentException.ThrowIfNullOrWhiteSpace(nome);
        Nome = nome;
    }
}
```

---

## Confondere stringa vuota e null

```csharp id="qhp13s"
string? nome = "";
```

`null` e stringa vuota non sono la stessa cosa.

```text id="3gmv9f"
null
  il valore manca.

""
  il valore esiste, ma è vuoto.
```

Spesso, per input testuale, conviene controllare entrambi:

```csharp id="dsj5lg"
if (string.IsNullOrWhiteSpace(nome))
{
    Console.WriteLine("Nome mancante o vuoto.");
}
```

---

## Usare null quando un valore di dominio sarebbe più chiaro

Meno espressivo:

```csharp id="0vaqxp"
string? stato = null;
```

Più chiaro:

```csharp id="63hsg9"
public enum StatoCliente
{
    NonSpecificato,
    Attivo,
    Sospeso,
    Cancellato
}
```

Non sempre `null` è il modo migliore per rappresentare l’assenza o uno stato speciale.

# Esempio completo del capitolo

Costruiamo una piccola API interna per cercare clienti.

Vogliamo distinguere chiaramente:

* cliente trovato;

## Modello Cliente

## Repository

```csharp id="dce0cg"
public class ClienteRepository
{
    private readonly List<Cliente> clienti = new();
    private readonly ILogger logger;

    public ClienteRepository(ILogger? logger = null)
    {
        this.logger = logger ?? new NullLogger();
    }

    public void Aggiungi(Cliente cliente)
    {
        ArgumentNullException.ThrowIfNull(cliente);

        clienti.Add(cliente);

        logger.Log($"Cliente aggiunto: {cliente.Nome}");
    }

    public Cliente? TrovaPerId(int id)
    {
        if (id <= 0)
        {
            return null;
        }

        Cliente? cliente = clienti.FirstOrDefault(c => c.Id == id);

        if (cliente is null)
        {
            logger.Log($"Cliente {id} non trovato.");
        }

        return cliente;
    }

    public bool TryTrovaPerId(int id, out Cliente? cliente)
    {
        cliente = TrovaPerId(id);
        return cliente is not null;
    }
}
```

## Uso

```csharp id="s79ryd"
ClienteRepository repository = new ClienteRepository(new ConsoleLogger());

repository.Aggiungi(new Cliente(1, "Emilie", "emilie@example.com"));
repository.Aggiungi(new Cliente(2, "Mario", null));

Cliente? cliente = repository.TrovaPerId(1);

if (cliente is not null)
{
    Console.WriteLine($"Cliente trovato: {cliente.Nome}");

    if (cliente.Email is not null)
    {
        Console.WriteLine($"Email: {cliente.Email}");
    }
}
else
{
    Console.WriteLine("Cliente non trovato.");
}
```

```csharp id="1f9t4z"

}
else
{
    Console.WriteLine("Cliente non trovato.");
}
```

  per ricerca sicura.

# Linee guida pratiche

## 1. Abilita Nullable nei nuovi progetti

Nel `.csproj`:

```xml id="lvojqn"
<Nullable>enable</Nullable>
```

È una delle impostazioni più utili per migliorare la qualità del codice C# moderno.

---

## 2. Usa `?` solo quando `null` è davvero ammesso

```csharp id="irw63h"
public string? Soprannome { get; init; }
```

Va bene se il soprannome è opzionale.

Non usare `?` solo per eliminare warning.

---

## 3. Inizializza sempre i membri non nullable

Con costruttore:

```csharp id="rq0rtc"
public Cliente(string nome)
{
    Nome = nome;
}
```

Con valore predefinito:

```csharp id="keb61w"
public string Nome { get; set; } = "";
```

Con `required`:

```csharp id="yvktso"
public required string Nome { get; init; }
```

---

## 4. Non abusare di `!`

Il null-forgiving operator è uno strumento di ultima istanza.

Se lo usi spesso, probabilmente il modello nullable non è progettato bene.

---

## 5. Distingui assenza, errore e valore vuoto

```text id="x854vj"
null
  valore assente.

""
  stringa vuota.

errore
  stato non valido o operazione fallita.
```

Non usare sempre `null` per tutto.

---

## 6. Progetta firme chiare

Ambiguo:

```csharp id="qqs5ep"
Cliente GetCliente(int id)
```

Più chiaro:

```csharp id="40vkkz"
Cliente? TrovaCliente(int id)
```

oppure:

```csharp id="5l4t3s"
Cliente OttieniClienteObbligatorio(int id)
```

oppure:

```csharp id="4ndhoa"
bool TryTrovaCliente(int id, out Cliente? cliente)
```

# Riepilogo

In questo capitolo abbiamo affrontato uno dei temi più importanti del C# moderno: la gestione esplicita dei `null`.

Abbiamo visto che `null` rappresenta l’assenza di riferimento e che, se usato senza disciplina, può portare a `NullReferenceException`.

Abbiamo introdotto:

```text id="4899s0"
Nullable Reference Types
  sistema di annotazioni e warning del compilatore per gestire meglio i null.

string
  riferimento che non dovrebbe essere null.

string?
  riferimento che può essere null.

int?
  Nullable<int>, nullable value type.

operatore !
  null-forgiving operator: zittisce il compilatore, ma non controlla nulla a runtime.
```

Abbiamo poi visto come usare le annotazioni nullable per progettare API più sicure:

```text id="d46us5"
Parametro non nullable
  il chiamante deve fornire un valore.

Parametro nullable
  il metodo deve gestire l’assenza.

Valore di ritorno nullable
  il chiamante deve gestire il caso mancante.

Valore di ritorno non nullable
  il metodo promette di restituire sempre un valore.
```

Abbiamo distinto diversi modi per rappresentare valori opzionali:

```text id="z7icgo"
null
Nullable<T>
Try pattern
Result<T>
Null Object pattern
tipo dedicato
```

E abbiamo chiarito alcune regole pratiche:

```text id="s9746j"
Non mettere ? ovunque.
Non usare ! per nascondere problemi.
Non restituire null da metodi non nullable.
Inizializza sempre i membri non nullable.
Distingui null, stringa vuota, errore e valore assente.
```

La regola centrale è questa:

```text id="6vrj4x"
Il nullable non è solo una funzione del compilatore.
È un modo per rendere espliciti i contratti del codice.
```

Un buon uso dei Nullable Reference Types rende il codice più leggibile, più sicuro e più onesto: chi scrive una API dichiara chiaramente cosa può mancare, e chi la usa viene guidato dal compilatore a gestire quei casi.

# 02.04 — Funzioni, metodi e organizzazione del codice

Nei capitoli precedenti abbiamo imparato a scrivere istruzioni, usare variabili, combinare espressioni, convertire valori e controllare il flusso con `if`, `switch` e cicli.

Ora dobbiamo affrontare un passaggio fondamentale: **organizzare il codice**.

Un programma scritto tutto in sequenza dentro `Program.cs` può andare bene per esempi molto piccoli, ma diventa presto difficile da leggere, modificare e testare.

Esempio:

```csharp
Console.Write("Prezzo: ");
string? inputPrezzo = Console.ReadLine();

if (!decimal.TryParse(inputPrezzo, out decimal prezzo))
{
    Console.WriteLine("Prezzo non valido.");
    return;
}

Console.Write("Sconto: ");
string? inputSconto = Console.ReadLine();

if (!decimal.TryParse(inputSconto, out decimal sconto))
{
    Console.WriteLine("Sconto non valido.");
    return;
}

decimal totale = prezzo - (prezzo * sconto / 100m);

Console.WriteLine($"Totale: {totale}");
```

Il codice funziona, ma contiene già più responsabilità:

* leggere input;
* validare numeri;
* calcolare uno sconto;
* stampare il risultato.

Quando il programma cresce, è meglio dividere il codice in blocchi più piccoli e riutilizzabili.

Questi blocchi si chiamano **metodi**.

---

# Funzioni e metodi

Nel linguaggio comune molti programmatori usano le parole **funzione** e **metodo** in modo simile.

In C#, tecnicamente, parliamo soprattutto di **metodi**, perché una funzione è quasi sempre definita dentro un tipo, per esempio dentro una classe.

Esempio:

```csharp
decimal CalcolaTotale(decimal prezzo, decimal percentualeSconto)
{
    return prezzo - (prezzo * percentualeSconto / 100m);
}
```

Questo è un metodo.

Riceve due valori:

```text
prezzo
percentualeSconto
```

e restituisce un valore:

```text
totale scontato
```

Possiamo chiamarlo così:

```csharp
decimal totale = CalcolaTotale(100m, 20m);

Console.WriteLine(totale);
```

Output:

```text
80
```

Un metodo serve a dare un nome a una porzione di logica.

Invece di scrivere ogni volta:

```csharp
prezzo - (prezzo * percentualeSconto / 100m)
```

possiamo scrivere:

```csharp
CalcolaTotale(prezzo, percentualeSconto)
```

Il codice diventa più leggibile.

---

# Definizione di metodo

Un metodo ha una struttura generale:

```csharp
tipoDiRitorno NomeMetodo(parametri)
{
    // corpo del metodo
}
```

Esempio:

```csharp
int Somma(int a, int b)
{
    return a + b;
}
```

Analizziamo le parti:

```text
int
  tipo di ritorno.

Somma
  nome del metodo.

int a, int b
  parametri.

return a + b;
  valore restituito.
```

Il metodo `Somma` riceve due interi e restituisce un intero.

Uso:

```csharp
int risultato = Somma(3, 4);

Console.WriteLine(risultato);
```

Output:

```text
7
```

---

## Metodo senza valore di ritorno

Un metodo può anche non restituire nulla.

In questo caso il tipo di ritorno è `void`.

```csharp
void StampaTitolo()
{
    Console.WriteLine("Gestionale Clienti");
}
```

Chiamata:

```csharp
StampaTitolo();
```

Output:

```text
Gestionale Clienti
```

`void` significa:

> questo metodo esegue un’azione, ma non produce un valore da usare dopo.

---

## Metodo con valore di ritorno

Un metodo che restituisce un valore usa `return`.

```csharp
decimal CalcolaIva(decimal imponibile, decimal aliquota)
{
    return imponibile * aliquota / 100m;
}
```

Chiamata:

```csharp
decimal iva = CalcolaIva(100m, 22m);

Console.WriteLine(iva);
```

Output:

```text
22
```

Il tipo del valore restituito deve essere compatibile con il tipo dichiarato.

Questo è corretto:

```csharp
int OttieniNumero()
{
    return 10;
}
```

Questo no:

```csharp
int OttieniNumero()
{
    return "dieci"; // errore
}
```

Il metodo dichiara di restituire `int`, quindi non può restituire una stringa.

---

# Parametri

I **parametri** sono valori che il metodo riceve dall’esterno.

Esempio:

```csharp
void Saluta(string nome)
{
    Console.WriteLine($"Ciao, {nome}!");
}
```

`nome` è un parametro.

Chiamata:

```csharp
Saluta("Emilie");
```

Output:

```text
Ciao, Emilie!
```

Quando chiamiamo il metodo, il valore `"Emilie"` viene passato al parametro `nome`.

---

## Più parametri

Un metodo può avere più parametri.

```csharp
void StampaCliente(string nome, string cognome, int eta)
{
    Console.WriteLine($"{nome} {cognome}, {eta} anni");
}
```

Chiamata:

```csharp
StampaCliente("Emilie", "Rollandin", 42);
```

Output:

```text
Emilie Rollandin, 42 anni
```

L’ordine dei parametri è importante.

```csharp
StampaCliente("Rollandin", "Emilie", 42);
```

compila, ma produce un risultato logicamente sbagliato:

```text
Rollandin Emilie, 42 anni
```

---

## Parametri e argomenti

È utile distinguere due parole:

```text
Parametro
  nome dichiarato nella definizione del metodo.

Argomento
  valore passato quando il metodo viene chiamato.
```

Esempio:

```csharp
void Saluta(string nome)
{
    Console.WriteLine($"Ciao, {nome}");
}

Saluta("Emilie");
```

Qui:

```text
nome
  è il parametro.

"Emilie"
  è l’argomento.
```

---

# Valori di ritorno

Un metodo può restituire un valore al chiamante.

```csharp
int Raddoppia(int numero)
{
    return numero * 2;
}
```

Uso:

```csharp
int risultato = Raddoppia(5);

Console.WriteLine(risultato);
```

Output:

```text
10
```

Il valore restituito può essere:

* assegnato a una variabile;
* passato a un altro metodo;
* usato in un’espressione;
* restituito da un altro metodo.

Esempio:

```csharp
Console.WriteLine(Raddoppia(5));
```

Esempio in un’espressione:

```csharp
int totale = Raddoppia(5) + 3;

Console.WriteLine(totale);
```

Output:

```text
13
```

---

## return interrompe il metodo

Quando viene eseguito `return`, il metodo termina.

```csharp
string ClassificaEta(int eta)
{
    if (eta < 0)
    {
        return "Età non valida";
    }

    if (eta < 18)
    {
        return "Minorenne";
    }

    return "Maggiorenne";
}
```

Se `eta` è minore di zero, il primo `return` termina subito il metodo.

Uso:

```csharp
Console.WriteLine(ClassificaEta(20));
Console.WriteLine(ClassificaEta(15));
Console.WriteLine(ClassificaEta(-1));
```

Output:

```text
Maggiorenne
Minorenne
Età non valida
```

---

# Overload

L’**overload** permette di definire più metodi con lo stesso nome, ma con parametri diversi.

Esempio:

```csharp
void Stampa(string testo)
{
    Console.WriteLine(testo);
}

void Stampa(int numero)
{
    Console.WriteLine($"Numero: {numero}");
}

void Stampa(decimal importo)
{
    Console.WriteLine($"Importo: {importo}");
}
```

Chiamate:

```csharp
Stampa("Ciao");
Stampa(42);
Stampa(19.90m);
```

Output:

```text
Ciao
Numero: 42
Importo: 19,90
```

Il compilatore sceglie quale metodo chiamare in base agli argomenti.

---

## Overload con numero diverso di parametri

```csharp
decimal CalcolaTotale(decimal prezzo)
{
    return prezzo;
}

decimal CalcolaTotale(decimal prezzo, decimal scontoPercentuale)
{
    return prezzo - (prezzo * scontoPercentuale / 100m);
}
```

Uso:

```csharp
Console.WriteLine(CalcolaTotale(100m));
Console.WriteLine(CalcolaTotale(100m, 20m));
```

Output:

```text
100
80
```

---

## Overload utile e overload confuso

L’overload è utile quando i metodi rappresentano la stessa operazione con input diversi.

Buono:

```csharp
void Saluta(string nome)
{
    Console.WriteLine($"Ciao, {nome}");
}

void Saluta(string nome, string cognome)
{
    Console.WriteLine($"Ciao, {nome} {cognome}");
}
```

Meno buono:

```csharp
void Elabora(string testo)
{
    // invia email
}

void Elabora(int numero)
{
    // cancella cliente
}
```

Qui il nome `Elabora` è troppo generico e i comportamenti sono troppo diversi.

Regola pratica:

```text
Usa overload quando il significato del metodo resta lo stesso.
Evitalo quando lo stesso nome nasconde operazioni diverse.
```

---

# Parametri opzionali

Un parametro può avere un valore predefinito.

In questo caso diventa **opzionale**.

Esempio:

```csharp
void Saluta(string nome, string prefisso = "Ciao")
{
    Console.WriteLine($"{prefisso}, {nome}!");
}
```

Chiamate:

```csharp
Saluta("Emilie");
Saluta("Emilie", "Buongiorno");
```

Output:

```text
Ciao, Emilie!
Buongiorno, Emilie!
```

Nel primo caso `prefisso` assume il valore predefinito `"Ciao"`.

---

## Parametri opzionali alla fine

I parametri opzionali devono normalmente stare dopo quelli obbligatori.

Corretto:

```csharp
void StampaOrdine(int numeroOrdine, bool dettagliato = false)
{
    // ...
}
```

Sbagliato:

```csharp
void StampaOrdine(bool dettagliato = false, int numeroOrdine)
{
    // errore
}
```

Il motivo è che il compilatore deve sapere come associare gli argomenti ai parametri.

---

## Quando usare parametri opzionali

I parametri opzionali sono utili per opzioni semplici.

Esempio:

```csharp
void StampaMessaggio(string messaggio, bool maiuscolo = false)
{
    if (maiuscolo)
    {
        messaggio = messaggio.ToUpper();
    }

    Console.WriteLine(messaggio);
}
```

Uso:

```csharp
StampaMessaggio("ciao");
StampaMessaggio("ciao", true);
```

Output:

```text
ciao
CIAO
```

Se però i parametri opzionali diventano molti, il metodo può diventare poco chiaro.

```csharp
CreaReport(true, false, true, false, 10, "PDF");
```

Meglio usare named arguments o un oggetto di opzioni, che vedremo più avanti.

---

# Named arguments

I **named arguments** permettono di specificare il nome del parametro durante la chiamata.

Esempio:

```csharp
void CreaUtente(string nome, string cognome, bool attivo)
{
    Console.WriteLine($"{nome} {cognome} - Attivo: {attivo}");
}
```

Chiamata normale:

```csharp
CreaUtente("Emilie", "Rollandin", true);
```

Chiamata con named arguments:

```csharp
CreaUtente(nome: "Emilie", cognome: "Rollandin", attivo: true);
```

Il risultato è lo stesso, ma la seconda forma è più leggibile.

---

## Named arguments con parametri opzionali

Sono molto utili quando ci sono parametri opzionali.

```csharp
void StampaReport(string titolo, bool includiDettagli = false, bool includiTotali = true)
{
    Console.WriteLine(titolo);
    Console.WriteLine($"Dettagli: {includiDettagli}");
    Console.WriteLine($"Totali: {includiTotali}");
}
```

Chiamata:

```csharp
StampaReport("Vendite", includiDettagli: true);
```

Qui è chiaro quale parametro stiamo impostando.

Senza named arguments:

```csharp
StampaReport("Vendite", true);
```

funziona, ma è meno esplicito.

---

## Cambiare l’ordine degli argomenti

Con i named arguments possiamo anche cambiare l’ordine.

```csharp
CreaUtente(attivo: true, cognome: "Rollandin", nome: "Emilie");
```

È valido, ma non sempre consigliabile.

Di solito è meglio mantenere l’ordine naturale dei parametri e usare i nomi per chiarezza, non per creare ordine arbitrario.

---

# `ref`

Normalmente, quando passiamo una variabile a un metodo, il metodo riceve il valore della variabile.

Esempio:

```csharp
void Incrementa(int numero)
{
    numero++;
}

int valore = 10;

Incrementa(valore);

Console.WriteLine(valore);
```

Output:

```text
10
```

Il metodo modifica la propria copia locale, non la variabile originale.

Con `ref`, invece, il metodo può modificare la variabile originale.

```csharp
void Incrementa(ref int numero)
{
    numero++;
}

int valore = 10;

Incrementa(ref valore);

Console.WriteLine(valore);
```

Output:

```text
11
```

---

## Uso di ref

`ref` deve essere indicato sia nella definizione del metodo sia nella chiamata.

Definizione:

```csharp
void Incrementa(ref int numero)
{
    numero++;
}
```

Chiamata:

```csharp
Incrementa(ref valore);
```

Questo rende esplicito che il metodo può modificare la variabile passata.

---

## Quando usare ref

`ref` va usato con attenzione.

È utile in casi specifici:

* ottimizzazioni particolari;
* codice a basso livello;
* metodi che devono modificare più valori;
* scenari di performance;
* compatibilità con API esistenti.

Nel codice applicativo normale è meglio preferire metodi che restituiscono valori.

Meno chiaro:

```csharp
void Calcola(ref decimal totale)
{
    totale = totale * 1.22m;
}
```

Più chiaro:

```csharp
decimal CalcolaTotaleConIva(decimal imponibile)
{
    return imponibile * 1.22m;
}
```

---

# `out`

`out` permette a un metodo di restituire un valore attraverso un parametro.

Esempio classico:

```csharp
bool riuscito = int.TryParse("123", out int numero);
```

`TryParse` restituisce:

```text
true/false
```

e produce anche il valore convertito dentro `numero`.

---

## Definire un metodo con out

```csharp
bool ProvaDividi(decimal dividendo, decimal divisore, out decimal risultato)
{
    if (divisore == 0)
    {
        risultato = 0;
        return false;
    }

    risultato = dividendo / divisore;
    return true;
}
```

Uso:

```csharp
if (ProvaDividi(10m, 2m, out decimal risultato))
{
    Console.WriteLine($"Risultato: {risultato}");
}
else
{
    Console.WriteLine("Divisione non valida.");
}
```

Output:

```text
Risultato: 5
```

Se il divisore è zero:

```csharp
if (ProvaDividi(10m, 0m, out decimal risultato))
{
    Console.WriteLine($"Risultato: {risultato}");
}
else
{
    Console.WriteLine("Divisione non valida.");
}
```

Output:

```text
Divisione non valida.
```

---

## Differenza tra ref e out

```text
ref
  il valore deve essere già inizializzato.
  il metodo può leggerlo e modificarlo.

out
  il valore può non essere inizializzato.
  il metodo deve assegnarlo prima di terminare.
```

Esempio con `ref`:

```csharp
int valore = 10;
Incrementa(ref valore);
```

Esempio con `out`:

```csharp
int.TryParse("123", out int numero);
```

Regola pratica:

```text
Usa out per il pattern “provo a fare qualcosa e, se riesco, ti restituisco anche il risultato”.
```

---

# `in`

`in` passa un argomento per riferimento, ma impedisce al metodo di modificarlo.

Esempio:

```csharp
void StampaImporto(in decimal importo)
{
    Console.WriteLine(importo);

    // importo = 10m; // errore
}
```

Uso:

```csharp
decimal totale = 100m;

StampaImporto(in totale);
```

`in` è utile soprattutto per evitare copie di strutture grandi, mantenendo però la sicurezza di non modificare il valore.

Nel codice quotidiano si usa meno di `ref` e `out`.

---

## Quando usare in

`in` può essere utile con struct grandi o codice orientato alle performance.

Esempio concettuale:

```csharp
void Analizza(in DatiPesanti dati)
{
    // legge dati senza modificarli
}
```

Per tipi piccoli come `int`, `bool`, `decimal` o `DateTime`, nella maggior parte dei casi non serve preoccuparsene.

Regola pratica:

```text
Nel codice applicativo ordinario, usa in raramente.
È più importante scrivere metodi chiari che micro-ottimizzare troppo presto.
```

---

# Tuple

Le tuple permettono di restituire più valori da un metodo in modo semplice.

Esempio:

```csharp
(string Nome, string Cognome) DividiNomeCompleto(string nomeCompleto)
{
    string[] parti = nomeCompleto.Split(' ', StringSplitOptions.RemoveEmptyEntries);

    if (parti.Length < 2)
    {
        return (nomeCompleto, "");
    }

    return (parti[0], parti[1]);
}
```

Uso:

```csharp
var risultato = DividiNomeCompleto("Emilie Rollandin");

Console.WriteLine(risultato.Nome);
Console.WriteLine(risultato.Cognome);
```

Output:

```text
Emilie
Rollandin
```

---

## Deconstruction

Possiamo anche destrutturare la tupla:

```csharp
(string nome, string cognome) = DividiNomeCompleto("Emilie Rollandin");

Console.WriteLine(nome);
Console.WriteLine(cognome);
```

Output:

```text
Emilie
Rollandin
```

Oppure usando `var`:

```csharp
var (nome, cognome) = DividiNomeCompleto("Emilie Rollandin");
```

---

## Tuple o classe?

Le tuple sono comode per risultati piccoli e locali.

Esempio buono:

```csharp
(decimal Imponibile, decimal Iva, decimal Totale) CalcolaTotale(decimal importo)
{
    decimal iva = importo * 22m / 100m;
    decimal totale = importo + iva;

    return (importo, iva, totale);
}
```

Uso:

```csharp
var risultato = CalcolaTotale(100m);

Console.WriteLine(risultato.Imponibile);
Console.WriteLine(risultato.Iva);
Console.WriteLine(risultato.Totale);
```

Se però il risultato ha un significato importante nel dominio, è meglio creare un tipo dedicato.

Meglio una classe o un record:

```csharp
public record TotaleDocumento(decimal Imponibile, decimal Iva, decimal Totale);
```

Regola pratica:

```text
Tuple
  ottime per risultati piccoli e locali.

Classe o record
  meglio per concetti importanti del dominio.
```

# Local functions

Una **local function** è un metodo definito dentro un altro metodo o dentro il codice top-level.

Esempio:

```csharp
decimal CalcolaTotale(decimal prezzo, decimal sconto)
{
    ValidaPercentuale(sconto);

    return prezzo - (prezzo * sconto / 100m);

    void ValidaPercentuale(decimal percentuale)
    {
        if (percentuale < 0 || percentuale > 100)
        {
            throw new ArgumentException("Percentuale non valida.", nameof(percentuale));
        }
    }
}
```

La funzione `ValidaPercentuale` esiste solo dentro `CalcolaTotale`.

---

## Quando usare local functions

Le local functions sono utili quando:

* una logica serve solo dentro un metodo;
* vogliamo evitare duplicazione interna;
* vogliamo dare un nome a un passaggio;
* non vogliamo esporre un metodo alla classe intera.

Esempio:

```csharp
decimal CalcolaPrezzoFinale(decimal prezzo, decimal sconto, decimal iva)
{
    decimal prezzoScontato = ApplicaPercentuale(prezzo, sconto);
    decimal prezzoConIva = prezzoScontato + ApplicaPercentuale(prezzoScontato, iva);

    return prezzoConIva;

    decimal ApplicaPercentuale(decimal valore, decimal percentuale)
    {
        return valore * percentuale / 100m;
    }
}
```

Qui `ApplicaPercentuale` è un dettaglio interno.

# Lambda expressions

Le **lambda expressions** sono funzioni anonime, cioè funzioni senza nome, spesso usate come valori.

Esempio semplice:

```csharp
Func<int, int> raddoppia = x => x * 2;

Console.WriteLine(raddoppia(5));
```

Output:

```text
10
```

La parte:

```csharp
x => x * 2
```

significa:

> preso `x`, restituisci `x * 2`.

---

## Lambda con più parametri

```csharp
Func<int, int, int> somma = (a, b) => a + b;

Console.WriteLine(somma(3, 4));
```

Output:

```text
7
```

---

## Lambda con blocco

Se la logica è più lunga, possiamo usare un blocco:

```csharp
Func<int, string> classifica = eta =>
{
    if (eta < 18)
    {
        return "Minorenne";
    }

    return "Maggiorenne";
};
```

Uso:

```csharp
Console.WriteLine(classifica(20));
```

Output:

```text
Maggiorenne
```

---

## Lambda e LINQ

Le lambda sono molto usate con LINQ.

Esempio:

```csharp
List<int> numeri = new List<int> { 1, 2, 3, 4, 5, 6 };

List<int> pari = numeri
    .Where(numero => numero % 2 == 0)
    .ToList();

foreach (int numero in pari)
{
    Console.WriteLine(numero);
}
```

Output:

```text
2
4
6
```

Qui:

```csharp
numero => numero % 2 == 0
```

è una lambda che dice:

> tieni solo i numeri pari.

Approfondiremo LINQ più avanti. Per ora basta capire che le lambda permettono di passare logica come argomento.

# Expression-bodied members

Quando un metodo è molto semplice, possiamo scriverlo in forma compatta usando `=>`.

Forma normale:

```csharp
int Raddoppia(int numero)
{
    return numero * 2;
}
```

Forma expression-bodied:

```csharp
int Raddoppia(int numero) => numero * 2;
```

Le due forme sono equivalenti.

---

## Esempi

```csharp
decimal CalcolaIva(decimal imponibile) => imponibile * 22m / 100m;

bool ÈMaggiorenne(int eta) => eta >= 18;

string Saluta(string nome) => $"Ciao, {nome}!";
```

Sono metodi brevi, leggibili e con una sola espressione.

---

## Quando usarli

Buon uso:

```csharp
int Quadrato(int numero) => numero * numero;
```

Meno buono:

```csharp
string ClassificaCliente(Cliente cliente) =>
    cliente.Attivo && cliente.Eta >= 18 && cliente.TotaleAcquisti > 1000m
        ? "VIP"
        : cliente.Attivo
            ? "Attivo"
            : "Non attivo";
```

È compatto, ma difficile da leggere.

Regola pratica:

```text
Usa expression-bodied members per metodi molto semplici.
Se la logica cresce, usa il blocco normale.
```

# Principio DRY

DRY significa:

```text
Don't Repeat Yourself
```

Cioè:

> non ripetere la stessa logica in più punti del codice.

Supponiamo di avere questo codice:

```csharp
decimal prezzo1 = 100m;
decimal sconto1 = 20m;
decimal totale1 = prezzo1 - (prezzo1 * sconto1 / 100m);

decimal prezzo2 = 250m;
decimal sconto2 = 10m;
decimal totale2 = prezzo2 - (prezzo2 * sconto2 / 100m);

decimal prezzo3 = 80m;
decimal sconto3 = 5m;
decimal totale3 = prezzo3 - (prezzo3 * sconto3 / 100m);
```

La formula dello sconto è ripetuta tre volte.

Meglio estrarla in un metodo:

```csharp
decimal ApplicaSconto(decimal prezzo, decimal percentualeSconto)
{
    return prezzo - (prezzo * percentualeSconto / 100m);
}
```

Uso:

```csharp
decimal totale1 = ApplicaSconto(100m, 20m);
decimal totale2 = ApplicaSconto(250m, 10m);
decimal totale3 = ApplicaSconto(80m, 5m);
```

Ora la formula esiste in un solo punto.

Se domani la regola cambia, modifichiamo un solo metodo.

---

## DRY non significa estrarre tutto

Non bisogna però esagerare.

A volte due pezzi di codice sembrano uguali, ma rappresentano concetti diversi.

Esempio:

```csharp
decimal scontoCliente = totale * 10m / 100m;
decimal penaleRitardo = totale * 10m / 100m;
```

La formula è identica, ma il significato è diverso.

Estrarre troppo presto può creare metodi generici e poco chiari.

Regola pratica:

```text
Non duplicare logica reale.
Non astrarre solo perché due righe si somigliano.
```

---

# Codice imperativo e codice funzionale

C# supporta diversi stili di programmazione.

Due stili importanti sono:

* stile imperativo;
* stile funzionale.

---

## Stile imperativo

Nel codice imperativo descriviamo passo per passo cosa deve fare il programma.

Esempio:

```csharp
List<int> numeri = new List<int> { 1, 2, 3, 4, 5, 6 };
List<int> pari = new List<int>();

foreach (int numero in numeri)
{
    if (numero % 2 == 0)
    {
        pari.Add(numero);
    }
}

foreach (int numero in pari)
{
    Console.WriteLine(numero);
}
```

Qui diciamo esplicitamente:

1. crea una lista vuota;
2. scorri i numeri;
3. se il numero è pari, aggiungilo;
4. poi stampa i risultati.

---

## Stile funzionale

Nel codice più funzionale descriviamo il risultato che vogliamo ottenere.

```csharp
List<int> numeri = new List<int> { 1, 2, 3, 4, 5, 6 };

List<int> pari = numeri
    .Where(numero => numero % 2 == 0)
    .ToList();

foreach (int numero in pari)
{
    Console.WriteLine(numero);
}
```

Qui diciamo:

> prendi i numeri pari.

Non descriviamo manualmente come costruire la lista.

---

## Quale stile usare?

Non c’è un vincitore assoluto.

Lo stile imperativo è spesso più chiaro quando:

* la logica è sequenziale;
* ci sono molti passaggi;
* servono effetti collaterali;
* bisogna gestire errori;
* il lettore è principiante.

Lo stile funzionale è spesso più chiaro quando:

* trasformiamo collezioni;
* filtriamo dati;
* componiamo operazioni;
* vogliamo ridurre codice ripetitivo.

Regola pratica:

```text
Usa lo stile che rende più chiara l’intenzione.
Non usare uno stile solo perché sembra più moderno.
```

---

# Quando estrarre un metodo

Estrarre un metodo significa prendere una porzione di codice e darle un nome.

Prima:

```csharp
decimal imponibile = 100m;
decimal aliquota = 22m;
decimal iva = imponibile * aliquota / 100m;
decimal totale = imponibile + iva;
```

Dopo:

```csharp
decimal CalcolaTotaleConIva(decimal imponibile, decimal aliquota)
{
    decimal iva = imponibile * aliquota / 100m;
    return imponibile + iva;
}

decimal totale = CalcolaTotaleConIva(100m, 22m);
```

---

## Segnali che indicano che un metodo va estratto

### Il codice è duplicato

Se la stessa logica compare in più punti, probabilmente va estratta.

```csharp
decimal CalcolaSconto(decimal importo, decimal percentuale)
{
    return importo * percentuale / 100m;
}
```

---

### Il metodo è troppo lungo

Se un metodo supera molte righe e fa molte cose, può essere utile dividerlo.

Esempio:

```text
Leggi input
Valida input
Calcola totale
Salva dati
Stampa riepilogo
```

Sono responsabilità diverse.

---

### Un blocco ha un’intenzione precisa

Esempio:

```csharp
if (cliente.Attivo && cliente.Eta >= 18 && cliente.DocumentoValido)
{
    Console.WriteLine("Accesso consentito");
}
```

Possiamo estrarre:

```csharp
bool PuòAccedere(Cliente cliente)
{
    return cliente.Attivo && cliente.Eta >= 18 && cliente.DocumentoValido;
}
```

Uso:

```csharp
if (PuòAccedere(cliente))
{
    Console.WriteLine("Accesso consentito");
}
```

Il codice legge quasi come una frase.

---

### Serve testare una logica

Se una regola è importante, conviene isolarla in un metodo testabile.

```csharp
bool ÈScontoValido(decimal percentuale)
{
    return percentuale >= 0 && percentuale <= 100;
}
```

Questo metodo può essere testato facilmente.

---

## Quando non estrarre

Non bisogna estrarre metodi inutilmente.

Prima:

```csharp
Console.WriteLine("Ciao");
```

Dopo, inutile:

```csharp
void StampaCiao()
{
    Console.WriteLine("Ciao");
}
```

Se il metodo non aggiunge significato, non serve.

Regola pratica:

```text
Estrai un metodo quando il nome del metodo rende il codice più chiaro.
```

---

# Come dare nomi chiari ai metodi

Il nome di un metodo è importantissimo.

Un buon nome spiega cosa fa il metodo.

---

## Metodi che eseguono azioni

Di solito iniziano con un verbo:

```csharp
StampaReport()
InviaEmail()
SalvaCliente()
CancellaDocumento()
AggiornaTotale()
CaricaConfigurazione()
```

Esempio:

```csharp
void SalvaCliente()
{
    // salva cliente
}
```

---

## Metodi che calcolano valori

Spesso iniziano con:

```text
Calcola
Ottieni
Crea
Genera
Determina
```

Esempi:

```csharp
decimal CalcolaTotale()
{
    return 100m;
}

string OttieniDescrizioneStato()
{
    return "Attivo";
}

Guid GeneraIdentificativo()
{
    return Guid.NewGuid();
}
```

---

## Metodi che verificano condizioni

Spesso restituiscono `bool` e hanno nomi come:

```text
È...
Ha...
Può...
Deve...
Contiene...
```

Esempi:

```csharp
bool ÈMaggiorenne(int eta)
{
    return eta >= 18;
}

bool HaDocumentoValido(Cliente cliente)
{
    return cliente.DocumentoValido;
}

bool PuòAccedere(Cliente cliente)
{
    return cliente.Attivo && cliente.DocumentoValido;
}
```

Questi nomi rendono il codice molto leggibile:

```csharp
if (PuòAccedere(cliente))
{
    Console.WriteLine("Accesso consentito");
}
```

---

## Evitare nomi generici

Nomi poco utili:

```csharp
Gestisci()
Elabora()
Processa()
Fai()
Esegui()
Controlla()
```

Non sono sempre sbagliati, ma spesso sono troppo vaghi.

Meglio:

```csharp
CalcolaTotaleDocumento()
ValidaCodiceFiscale()
InviaEmailConferma()
CaricaClientiAttivi()
GeneraReportMensile()
```

---

## Il nome deve dire il livello giusto di dettaglio

Troppo generico:

```csharp
Calcola()
```

Troppo specifico:

```csharp
CalcolaTotaleDocumentoConIvaAlVentiduePercentoEScontoClienteVip()
```

Meglio:

```csharp
CalcolaTotaleDocumento()
```

I dettagli possono stare nei parametri o nel corpo del metodo.

---

# Esempio completo del capitolo

Partiamo da un programma scritto in modo lineare.

```csharp
Console.Write("Prezzo: ");
string? inputPrezzo = Console.ReadLine();

if (!decimal.TryParse(inputPrezzo, out decimal prezzo))
{
    Console.WriteLine("Prezzo non valido.");
    return;
}

Console.Write("Sconto percentuale: ");
string? inputSconto = Console.ReadLine();

if (!decimal.TryParse(inputSconto, out decimal sconto))
{
    Console.WriteLine("Sconto non valido.");
    return;
}

if (sconto < 0 || sconto > 100)
{
    Console.WriteLine("Lo sconto deve essere tra 0 e 100.");
    return;
}

decimal valoreSconto = prezzo * sconto / 100m;
decimal totale = prezzo - valoreSconto;

Console.WriteLine($"Totale: {totale}");
```

Funziona, ma possiamo organizzarlo meglio.

---

## Versione organizzata con metodi

```csharp
decimal? prezzo = LeggiDecimal("Prezzo: ");

if (prezzo is null)
{
    Console.WriteLine("Prezzo non valido.");
    return;
}

decimal? sconto = LeggiDecimal("Sconto percentuale: ");

if (sconto is null)
{
    Console.WriteLine("Sconto non valido.");
    return;
}

if (!ÈPercentualeValida(sconto.Value))
{
    Console.WriteLine("Lo sconto deve essere tra 0 e 100.");
    return;
}

decimal totale = CalcolaTotaleScontato(prezzo.Value, sconto.Value);

Console.WriteLine($"Totale: {totale}");

decimal? LeggiDecimal(string messaggio)
{
    Console.Write(messaggio);

    string? input = Console.ReadLine();

    if (decimal.TryParse(input, out decimal valore))
    {
        return valore;
    }

    return null;
}

bool ÈPercentualeValida(decimal percentuale)
{
    return percentuale >= 0 && percentuale <= 100;
}

decimal CalcolaTotaleScontato(decimal prezzo, decimal percentualeSconto)
{
    decimal valoreSconto = prezzo * percentualeSconto / 100m;
    return prezzo - valoreSconto;
}
```

Ora il codice principale è più leggibile:

```csharp
decimal? prezzo = LeggiDecimal("Prezzo: ");
decimal? sconto = LeggiDecimal("Sconto percentuale: ");
bool valido = ÈPercentualeValida(sconto.Value);
decimal totale = CalcolaTotaleScontato(prezzo.Value, sconto.Value);
```

Ogni metodo ha una responsabilità chiara:

```text
LeggiDecimal
  legge e converte un numero decimale.

ÈPercentualeValida
  verifica che la percentuale sia tra 0 e 100.

CalcolaTotaleScontato
  calcola il prezzo finale.
```

Questo è il vantaggio principale dei metodi: rendono il programma più leggibile e più facile da modificare.

---

# Errori comuni

## Scrivere metodi troppo lunghi

Un metodo che fa troppe cose diventa difficile da capire.

Segnale negativo:

```text
Legge input
Valida dati
Calcola importi
Salva su database
Invia email
Stampa report
```

Sono troppe responsabilità per un solo metodo.

---

## Usare nomi troppo generici

Poco chiaro:

```csharp
void Gestisci()
{
}
```

Meglio:

```csharp
void GestisciRegistrazioneCliente()
{
}
```

Ancora meglio, se possibile:

```csharp
void RegistraCliente()
{
}
```

---

## Usare ref e out quando basta return

Meno chiaro:

```csharp
void CalcolaTotale(decimal imponibile, out decimal totale)
{
    totale = imponibile * 1.22m;
}
```

Più chiaro:

```csharp
decimal CalcolaTotale(decimal imponibile)
{
    return imponibile * 1.22m;
}
```

`out` è utile, ma non deve sostituire sempre `return`.

---

## Abusare dei parametri opzionali

Poco leggibile:

```csharp
GeneraReport(true, false, true, 10, "PDF");
```

Meglio:

```csharp
GeneraReport(
    includiDettagli: true,
    includiGrafici: false,
    includiTotali: true,
    numeroMassimoRighe: 10,
    formato: "PDF");
```

Se i parametri diventano troppi, meglio valutare un oggetto dedicato.

---

## Estrarre metodi senza migliorare il significato

Inutile:

```csharp
void ScriviRiga()
{
    Console.WriteLine();
}
```

Se il metodo non comunica un’intenzione, non aggiunge valore.

# Riepilogo

In questo capitolo abbiamo imparato a organizzare il codice usando metodi e funzioni.

Abbiamo visto la struttura di un metodo:

```csharp
tipoDiRitorno NomeMetodo(parametri)
{
    corpo del metodo
}
```

Abbiamo distinto:

```text
parametri
  valori dichiarati dal metodo.

argomenti
  valori passati alla chiamata.
```

Abbiamo introdotto:

```text
void
  metodo senza valore di ritorno.

return
  restituisce un valore o termina il metodo.

overload
  più metodi con lo stesso nome e parametri diversi.

parametri opzionali
  parametri con valore predefinito.

named arguments
  argomenti passati specificando il nome del parametro.

ref
  passa una variabile modificabile dal metodo.

out
  permette al metodo di produrre un valore aggiuntivo.

in
  passa per riferimento senza permettere modifiche.

tuple
  permettono di restituire più valori.

local functions
  funzioni definite dentro altre funzioni.

lambda expressions
  funzioni anonime, spesso usate con LINQ.

expression-bodied members
  forma compatta per metodi semplici.
```

Abbiamo poi visto alcuni principi di organizzazione:

```text
DRY
  evita duplicazione reale di logica.

Codice imperativo
  descrive passo per passo cosa fare.

Codice funzionale
  descrive trasformazioni e risultati.

Estrarre un metodo
  serve quando un blocco ha una responsabilità chiara.

Nomi chiari
  rendono il codice leggibile quasi come una frase.
```

Il punto centrale è questo:

> Un metodo non serve solo a evitare duplicazione. Serve soprattutto a dare un nome a un’idea.

Nel prossimo capitolo faremo un passo ulteriore: useremo metodi, dati e comportamenti per costruire i nostri tipi, entrando nella programmazione orientata agli oggetti.

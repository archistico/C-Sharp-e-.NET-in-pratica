# 04.01 — LINQ: interrogare collezioni e sequenze

Nella Parte 03 abbiamo studiato molte librerie fondamentali di .NET: numeri, stringhe, date, collezioni, file, stream, serializzazione, logging, configurazione e Dependency Injection.

Ora entriamo in una parte molto concreta: **interrogare, trasformare e salvare dati**.

Il primo strumento da conoscere è **LINQ**.

LINQ significa:

```text
Language Integrated Query
```

cioè query integrate nel linguaggio.

Con LINQ possiamo interrogare sequenze di dati usando C#:

```csharp
List<Cliente> clienti = [
    new Cliente(1, "Emilie", "Aosta", true),
    new Cliente(2, "Mario", "Torino", false),
    new Cliente(3, "Anna", "Aosta", true)
];

var clientiAttivi = clienti
    .Where(c => c.Attivo)
    .OrderBy(c => c.Nome)
    .ToList();
```

LINQ è uno dei pilastri del C# moderno perché permette di lavorare in modo espressivo con:

```text
collezioni in memoria
array
liste
dizionari
file letti riga per riga
risultati di API
query database tramite EF Core
XML
sequenze generate dinamicamente
```

Il punto centrale del capitolo è questo:

```text
LINQ permette di descrivere cosa vogliamo ottenere da una sequenza,
senza scrivere sempre manualmente cicli, condizioni e accumuli.
```

---

# Che cos’è LINQ

LINQ è un insieme di funzionalità del linguaggio e della libreria .NET che permette di interrogare dati.

Senza LINQ:

```csharp
List<Cliente> clientiAttivi = new List<Cliente>();

foreach (Cliente cliente in clienti)
{
    if (cliente.Attivo)
    {
        clientiAttivi.Add(cliente);
    }
}
```

Con LINQ:

```csharp
List<Cliente> clientiAttivi = clienti
    .Where(cliente => cliente.Attivo)
    .ToList();
```

Il secondo codice è più dichiarativo.

Non diciamo passo per passo:

```text
crea lista
scorri clienti
controlla condizione
aggiungi alla lista
```

Diciamo:

```text
dammi i clienti attivi
```

---

# Modello di esempio

Per gli esempi useremo queste classi:

```csharp
public class Cliente
{
    public int Id { get; }
    public string Nome { get; }
    public string Citta { get; }
    public bool Attivo { get; }

    public Cliente(int id, string nome, string citta, bool attivo)
    {
        Id = id;
        Nome = nome;
        Citta = citta;
        Attivo = attivo;
    }
}
```

E questa lista:

```csharp
List<Cliente> clienti = [
    new Cliente(1, "Emilie", "Aosta", true),
    new Cliente(2, "Mario", "Torino", false),
    new Cliente(3, "Anna", "Aosta", true),
    new Cliente(4, "Luca", "Milano", true),
    new Cliente(5, "Giulia", "Torino", false)
];
```

---

# Query syntax

LINQ può essere scritto in due stili principali:

```text
query syntax
method syntax
```

La **query syntax** assomiglia a SQL.

Esempio:

```csharp
var risultato =
    from cliente in clienti
    where cliente.Attivo
    orderby cliente.Nome
    select cliente;
```

Uso:

```csharp
foreach (Cliente cliente in risultato)
{
    Console.WriteLine(cliente.Nome);
}
```

Output:

```text
Anna
Emilie
Luca
```

---

## Struttura della query syntax

```csharp
var risultato =
    from elemento in sequenza
    where condizione
    orderby ordinamento
    select proiezione;
```

Esempio:

```csharp
var nomiClientiAttivi =
    from cliente in clienti
    where cliente.Attivo
    orderby cliente.Nome
    select cliente.Nome;
```

Qui il risultato non è più una sequenza di `Cliente`, ma una sequenza di `string`.

---

## Quando usare query syntax

La query syntax può essere leggibile quando ci sono:

```text
join
group by
più from
query simili a SQL
```

Esempio con raggruppamento:

```csharp
var clientiPerCitta =
    from cliente in clienti
    group cliente by cliente.Citta;
```

È una sintassi comoda, ma nella pratica quotidiana molti sviluppatori C# usano più spesso la method syntax.

---

# Method syntax

La **method syntax** usa metodi di estensione come:

```text
Where
Select
OrderBy
GroupBy
Join
Any
Count
ToList
```

Esempio:

```csharp
var risultato = clienti
    .Where(cliente => cliente.Attivo)
    .OrderBy(cliente => cliente.Nome)
    .Select(cliente => cliente.Nome);
```

La method syntax è molto diffusa perché:

```text
si combina bene con lambda
è compatta
è coerente con molte API .NET
rende esplicita la pipeline di trasformazione
```

---

## Query syntax e method syntax sono collegate

Questa query:

```csharp
var risultato =
    from cliente in clienti
    where cliente.Attivo
    orderby cliente.Nome
    select cliente.Nome;
```

è sostanzialmente equivalente a:

```csharp
var risultato = clienti
    .Where(cliente => cliente.Attivo)
    .OrderBy(cliente => cliente.Nome)
    .Select(cliente => cliente.Nome);
```

Il compilatore traduce la query syntax in chiamate a metodi LINQ.

# `Where`

`Where` filtra una sequenza.

Esempio:

```csharp
IEnumerable<Cliente> clientiAttivi = clienti
    .Where(cliente => cliente.Attivo);
```

Output:

```csharp
foreach (Cliente cliente in clientiAttivi)
{
    Console.WriteLine(cliente.Nome);
}
```

```text
Emilie
Anna
Luca
```

---

## Condizioni più complesse

```csharp
var clientiAttiviDiAosta = clienti
    .Where(cliente => cliente.Attivo && cliente.Citta == "Aosta");
```

Output:

```text
Emilie
Anna
```

---

## Where con metodo separato

Invece di una lambda inline:

```csharp
var risultato = clienti.Where(cliente => cliente.Attivo);
```

possiamo usare un metodo:

```csharp
bool ÈClienteAttivo(Cliente cliente)
{
    return cliente.Attivo;
}

var risultato = clienti.Where(ÈClienteAttivo);
```

Questo è utile quando la condizione ha un nome importante o è riutilizzata.

---

# `Select`

`Select` trasforma ogni elemento della sequenza.

Esempio:

```csharp
IEnumerable<string> nomi = clienti
    .Select(cliente => cliente.Nome);
```

Output:

```csharp
foreach (string nome in nomi)
{
    Console.WriteLine(nome);
}
```

```text
Emilie
Mario
Anna
Luca
Giulia
```

---

## Select verso DTO

`Select` è molto usato per creare DTO.

```csharp
public record ClienteDto(
    int Id,
    string Nome,
    string Citta);
```

Query:

```csharp
List<ClienteDto> dto = clienti
    .Where(cliente => cliente.Attivo)
    .Select(cliente => new ClienteDto(
        cliente.Id,
        cliente.Nome,
        cliente.Citta))
    .ToList();
```

Qui partiamo da `Cliente` e otteniamo `ClienteDto`.

---

## Select con tipo anonimo

Per proiezioni temporanee possiamo usare tipi anonimi.

```csharp
var risultato = clienti
    .Select(cliente => new
    {
        cliente.Nome,
        cliente.Citta
    });
```

Uso:

```csharp
foreach (var item in risultato)
{
    Console.WriteLine($"{item.Nome} - {item.Citta}");
}
```

# `OrderBy`

`OrderBy` ordina una sequenza in modo crescente.

```csharp
var ordinatiPerNome = clienti
    .OrderBy(cliente => cliente.Nome);
```

Output:

```text
Anna
Emilie
Giulia
Luca
Mario
```

## OrderByDescending

Per ordinare in modo decrescente:

```csharp
var ordinatiPerNomeDesc = clienti
    .OrderByDescending(cliente => cliente.Nome);
```

---

# `ThenBy`

`ThenBy` aggiunge un secondo criterio di ordinamento.

Esempio:

```csharp
var ordinati = clienti
    .OrderBy(cliente => cliente.Citta)
    .ThenBy(cliente => cliente.Nome);
```

Prima ordiniamo per città, poi per nome.

Output:

```text
Aosta - Anna
Aosta - Emilie
Milano - Luca
Torino - Giulia
Torino - Mario
```

---

## ThenByDescending

```csharp
var ordinati = clienti
    .OrderBy(cliente => cliente.Citta)
    .ThenByDescending(cliente => cliente.Nome);
```

---

## Errore comune con OrderBy ripetuto

Sbagliato:

```csharp
var ordinati = clienti
    .OrderBy(cliente => cliente.Citta)
    .OrderBy(cliente => cliente.Nome);
```

Il secondo `OrderBy` sostituisce l’ordinamento precedente.

Corretto:

```csharp
var ordinati = clienti
    .OrderBy(cliente => cliente.Citta)
    .ThenBy(cliente => cliente.Nome);
```

# `GroupBy`

`GroupBy` raggruppa gli elementi in base a una chiave.

Esempio:

```csharp
var gruppi = clienti
    .GroupBy(cliente => cliente.Citta);
```

Uso:

```csharp
foreach (var gruppo in gruppi)
{
    Console.WriteLine($"Città: {gruppo.Key}");

    foreach (Cliente cliente in gruppo)
    {
        Console.WriteLine($"  {cliente.Nome}");
    }
}
```

Output:

```text
Città: Aosta
  Emilie
  Anna
Città: Torino
  Mario
  Giulia
Città: Milano
  Luca
```

---

## GroupBy con conteggio

```csharp
var conteggiPerCitta = clienti
    .GroupBy(cliente => cliente.Citta)
    .Select(gruppo => new
    {
        Citta = gruppo.Key,
        NumeroClienti = gruppo.Count()
    });
```

Output:

```csharp
foreach (var item in conteggiPerCitta)
{
    Console.WriteLine($"{item.Citta}: {item.NumeroClienti}");
}
```

```text
Aosta: 2
Torino: 2
Milano: 1
```

## GroupBy con somma

Aggiungiamo un modello ordine:

```csharp
public class Ordine
{
    public int Id { get; }
    public int ClienteId { get; }
    public decimal Totale { get; }

    public Ordine(int id, int clienteId, decimal totale)
    {
        Id = id;
        ClienteId = clienteId;
        Totale = totale;
    }
}
```

Lista:

```csharp
List<Ordine> ordini = [
    new Ordine(1, 1, 100m),
    new Ordine(2, 1, 50m),
    new Ordine(3, 3, 200m),
    new Ordine(4, 4, 80m)
];
```

Somma per cliente:

```csharp
var totaliPerCliente = ordini
    .GroupBy(ordine => ordine.ClienteId)
    .Select(gruppo => new
    {
        ClienteId = gruppo.Key,
        Totale = gruppo.Sum(ordine => ordine.Totale)
    });
```

# `Join`

`Join` collega due sequenze in base a una chiave.

Abbiamo:

```text
clienti
ordini
```

Vogliamo ottenere ordini con nome cliente.

```csharp
var ordiniConCliente = clienti.Join(
    ordini,
    cliente => cliente.Id,
    ordine => ordine.ClienteId,
    (cliente, ordine) => new
    {
        Cliente = cliente.Nome,
        OrdineId = ordine.Id,
        ordine.Totale
    });
```

Uso:

```csharp
foreach (var item in ordiniConCliente)
{
    Console.WriteLine($"{item.Cliente} - Ordine {item.OrdineId}: {item.Totale}");
}
```

Output:

```text
Emilie - Ordine 1: 100
Emilie - Ordine 2: 50
Anna - Ordine 3: 200
Luca - Ordine 4: 80
```

---

## Join con query syntax

Con query syntax il join è spesso più leggibile:

```csharp
var ordiniConCliente =
    from cliente in clienti
    join ordine in ordini
        on cliente.Id equals ordine.ClienteId
    select new
    {
        Cliente = cliente.Nome,
        OrdineId = ordine.Id,
        ordine.Totale
    };
```

# `GroupJoin`

`GroupJoin` collega due sequenze producendo gruppi di elementi collegati.

È utile per rappresentare relazioni uno-a-molti.

Esempio:

```csharp
var clientiConOrdini = clienti.GroupJoin(
    ordini,
    cliente => cliente.Id,
    ordine => ordine.ClienteId,
    (cliente, ordiniCliente) => new
    {
        Cliente = cliente.Nome,
        Ordini = ordiniCliente
    });
```

Uso:

```csharp
foreach (var item in clientiConOrdini)
{
    Console.WriteLine(item.Cliente);

    foreach (Ordine ordine in item.Ordini)
    {
        Console.WriteLine($"  Ordine {ordine.Id}: {ordine.Totale}");
    }
}
```

Output:

```text
Emilie
  Ordine 1: 100
  Ordine 2: 50
Mario
Anna
  Ordine 3: 200
Luca
  Ordine 4: 80
Giulia
```

I clienti senza ordini restano presenti, con un gruppo vuoto.

## GroupJoin con query syntax

```csharp
var clientiConOrdini =
    from cliente in clienti
    join ordine in ordini
        on cliente.Id equals ordine.ClienteId
        into ordiniCliente
    select new
    {
        Cliente = cliente.Nome,
        Ordini = ordiniCliente
    };
```

# `Any`

`Any` verifica se almeno un elemento soddisfa una condizione.

```csharp
bool esistonoClientiAttivi = clienti
    .Any(cliente => cliente.Attivo);
```

Output:

```text
True
```

Senza condizione:

```csharp
bool ciSonoClienti = clienti.Any();
```

---

## Any è utile per esistenza

Sconsigliato:

```csharp
if (clienti.Count() > 0)
{
}
```

Meglio:

```csharp
if (clienti.Any())
{
}
```

`Any` può fermarsi appena trova un elemento.

---

# `All`

`All` verifica se tutti gli elementi soddisfano una condizione.

```csharp
bool tuttiAttivi = clienti
    .All(cliente => cliente.Attivo);
```

Output:

```text
False
```

Esempio:

```csharp
bool tuttiHannoNome = clienti
    .All(cliente => !string.IsNullOrWhiteSpace(cliente.Nome));
```

---

## Attenzione a All su sequenza vuota

Su una sequenza vuota, `All` restituisce `true`.

Esempio:

```csharp
List<Cliente> vuota = [];

bool risultato = vuota.All(c => c.Attivo);

Console.WriteLine(risultato);
```

Output:

```text
True
```

Perché non esiste nessun elemento che violi la condizione.

Se vuoi verificare che la lista non sia vuota e che tutti rispettino la condizione:

```csharp
bool valido = clienti.Any() && clienti.All(c => c.Attivo);
```

---

# `Count`

`Count` conta gli elementi.

```csharp
int numeroClienti = clienti.Count();
```

Con condizione:

```csharp
int attivi = clienti.Count(cliente => cliente.Attivo);
```

---

## Proprietà Count vs metodo Count()

Per `List<T>` esiste la proprietà:

```csharp
int count = clienti.Count;
```

Per una sequenza generica `IEnumerable<T>` si usa:

```csharp
int count = sequenza.Count();
```

Attenzione: su una sequenza generata dinamicamente, `Count()` potrebbe enumerare tutta la sequenza.

# `Min` e `Max`

`Min` restituisce il valore minimo.

```csharp
decimal minimo = ordini.Min(ordine => ordine.Totale);
```

`Max` restituisce il valore massimo.

```csharp
decimal massimo = ordini.Max(ordine => ordine.Totale);
```

## MinBy e MaxBy

Nelle versioni moderne di .NET abbiamo anche `MinBy` e `MaxBy`.

```csharp
Ordine? ordineMinimo = ordini.MinBy(ordine => ordine.Totale);
Ordine? ordineMassimo = ordini.MaxBy(ordine => ordine.Totale);
```

Questi restituiscono l’oggetto, non solo il valore.

# `First`

`First` restituisce il primo elemento.

```csharp
Cliente primo = clienti.First();
```

Con condizione:

```csharp
Cliente primoAttivo = clienti.First(cliente => cliente.Attivo);
```

Se non trova nulla, genera eccezione.

---

# `FirstOrDefault`

`FirstOrDefault` restituisce il primo elemento oppure il valore predefinito.

Per reference type, il valore predefinito è `null`.

```csharp
Cliente? cliente = clienti
    .FirstOrDefault(cliente => cliente.Id == 10);

if (cliente is null)
{
    Console.WriteLine("Cliente non trovato.");
}
```

---

## Quando usare FirstOrDefault

Usalo quando l’assenza del risultato è prevista.

```csharp
Cliente? cliente = clienti.FirstOrDefault(c => c.Id == id);
```

Poi gestisci `null`.

---

# `Single`

`Single` restituisce l’unico elemento della sequenza.

```csharp
Cliente cliente = clienti.Single(c => c.Id == 1);
```

`Single` genera eccezione se:

```text
non trova nessun elemento
trova più di un elemento
```

Questo lo rende utile quando vuoi dichiarare una regola forte:

```text
deve esistere esattamente un cliente con questo ID
```

---

# `SingleOrDefault`

Anche se non era nell’elenco iniziale, è utile citarlo.

```csharp
Cliente? cliente = clienti.SingleOrDefault(c => c.Id == 1);
```

Genera eccezione se trova più di un elemento.
Restituisce `null` se non trova nulla.

Regola pratica:

```text
First
  voglio il primo e se manca è errore.

FirstOrDefault
  voglio il primo, ma può mancare.

Single
  deve essercene esattamente uno.

SingleOrDefault
  può non esserci, ma se c’è deve essere unico.
```

# Deferred execution

Una delle idee più importanti di LINQ è la **deferred execution**, cioè esecuzione differita.

Molte query LINQ non vengono eseguite quando vengono definite.

Vengono eseguite quando vengono enumerate.

Esempio:

```csharp
var query = clienti
    .Where(cliente => cliente.Attivo);

Console.WriteLine("Query creata");
```

A questo punto la query non ha ancora filtrato davvero tutti i clienti.

Viene eseguita qui:

```csharp
foreach (Cliente cliente in query)
{
    Console.WriteLine(cliente.Nome);
}
```

---

## Esempio dimostrativo

```csharp
var query = clienti
    .Where(cliente =>
    {
        Console.WriteLine($"Controllo {cliente.Nome}");
        return cliente.Attivo;
    });

Console.WriteLine("Prima del foreach");

foreach (Cliente cliente in query)
{
    Console.WriteLine($"Cliente attivo: {cliente.Nome}");
}
```

Output:

```text
Prima del foreach
Controllo Emilie
Cliente attivo: Emilie
Controllo Mario
Controllo Anna
Cliente attivo: Anna
Controllo Luca
Cliente attivo: Luca
Controllo Giulia
```

Questo mostra che `Where` viene eseguito durante l’enumerazione.

---

## Perché è utile

La deferred execution permette:

```text
composizione progressiva delle query
esecuzione solo quando serve
pipeline efficienti
possibilità di lavorare con sequenze grandi
integrazione con IQueryable ed EF Core
```

Esempio:

```csharp
IEnumerable<Cliente> query = clienti;

if (soloAttivi)
{
    query = query.Where(c => c.Attivo);
}

if (!string.IsNullOrWhiteSpace(citta))
{
    query = query.Where(c => c.Citta == citta);
}

List<Cliente> risultato = query.ToList();
```

La query viene costruita a pezzi e materializzata solo alla fine.

---

# Immediate execution

L’**immediate execution** avviene quando chiediamo subito un risultato concreto.

Esempi:

```csharp
ToList()
ToArray()
Count()
Sum()
Average()
Min()
Max()
First()
Single()
Any()
All()
```

Esempio:

```csharp
List<Cliente> clientiAttivi = clienti
    .Where(c => c.Attivo)
    .ToList();
```

`ToList()` forza l’esecuzione e crea una lista.

---

## ToList e ToArray

```csharp
List<Cliente> lista = clienti
    .Where(c => c.Attivo)
    .ToList();

Cliente[] array = clienti
    .Where(c => c.Attivo)
    .ToArray();
```

Usa `ToList()` quando vuoi:

```text
materializzare il risultato
enumerare più volte senza rieseguire la query
passare una lista modificabile
evitare che la sorgente cambi sotto la query
```

---

## Attenzione a ToList troppo presto

Sconsigliato:

```csharp
var risultato = clienti
    .ToList()
    .Where(c => c.Attivo)
    .ToList();
```

Se `clienti` è già una lista, il primo `ToList()` è inutile.

Con EF Core, fare `ToList()` troppo presto può essere ancora peggio: potresti caricare troppi dati dal database e poi filtrare in memoria.

Regola pratica:

```text
Componi la query prima.
Materializza con ToList solo quando ti serve davvero il risultato.
```

# `IEnumerable<T>`

`IEnumerable<T>` rappresenta una sequenza di elementi enumerabile.

È l’interfaccia base su cui lavorano molti metodi LINQ in memoria.

Esempio:

```csharp
IEnumerable<Cliente> query = clienti
    .Where(c => c.Attivo);
```

`IEnumerable<T>` dice:

```text
posso scorrere elementi di tipo T
```

Non dice necessariamente:

```text
quanti sono
se sono in memoria
se posso modificarli
se sono una lista
se posso accedere per indice
```

---

## IEnumerable come parametro

Spesso è bene accettare `IEnumerable<T>` quando un metodo deve solo leggere una sequenza.

```csharp
decimal CalcolaTotale(IEnumerable<Ordine> ordini)
{
    return ordini.Sum(o => o.Totale);
}
```

Il metodo funziona con:

```text
array
List<T>
risultati LINQ
sequenze generate
```

---

## Attenzione a enumerare più volte

Esempio:

```csharp
void StampaStatistiche(IEnumerable<Ordine> ordini)
{
    Console.WriteLine(ordini.Count());
    Console.WriteLine(ordini.Sum(o => o.Totale));
}
```

Se `ordini` è una query differita, viene enumerata due volte.

Se questo è un problema:

```csharp
void StampaStatistiche(IEnumerable<Ordine> ordini)
{
    List<Ordine> lista = ordini.ToList();

    Console.WriteLine(lista.Count);
    Console.WriteLine(lista.Sum(o => o.Totale));
}
```

---

# `IQueryable<T>`

`IQueryable<T>` rappresenta una query che può essere tradotta da un provider.

È fondamentale con Entity Framework Core.

Esempio concettuale:

```csharp
IQueryable<Cliente> query = dbContext.Clienti
    .Where(c => c.Attivo);
```

Qui la query non è semplicemente una sequenza in memoria.

È un’espressione che EF Core può tradurre in SQL.

Esempio:

```csharp
var clientiAttivi = await dbContext.Clienti
    .Where(c => c.Attivo)
    .OrderBy(c => c.Nome)
    .ToListAsync();
```

Può diventare una query SQL simile a:

```sql
SELECT *
FROM Clienti
WHERE Attivo = 1
ORDER BY Nome
```

---

## IEnumerable vs IQueryable

Differenza fondamentale:

```text
IEnumerable<T>
  lavora su oggetti .NET, normalmente in memoria.

IQueryable<T>
  rappresenta una query traducibile da un provider, per esempio EF Core verso SQL.
```

Esempio importante:

```csharp
IQueryable<Cliente> query = dbContext.Clienti;

query = query.Where(c => c.Attivo);

List<Cliente> risultato = await query.ToListAsync();
```

Il filtro può essere eseguito dal database.

Ma se facciamo:

```csharp
IEnumerable<Cliente> clienti = dbContext.Clienti.ToList();

var risultato = clienti.Where(c => c.Attivo);
```

abbiamo già caricato tutto in memoria prima di filtrare.

---

## Regola pratica

```text
Con collezioni in memoria
  usa LINQ su IEnumerable<T>.

Con database EF Core
  componi IQueryable<T> finché possibile,
  poi materializza con ToListAsync, FirstOrDefaultAsync, CountAsync, ecc.
```

# Esempio completo: report clienti

Scenario:

```text
Abbiamo clienti e ordini.
Vogliamo produrre un report:
- solo clienti attivi
- con totale ordini
- ordinati per città e nome
- raggruppati per città
```

Modelli:

```csharp
public class Cliente
{
    public int Id { get; }
    public string Nome { get; }
    public string Citta { get; }
    public bool Attivo { get; }

    public Cliente(int id, string nome, string citta, bool attivo)
    {
        Id = id;
        Nome = nome;
        Citta = citta;
        Attivo = attivo;
    }
}

public class Ordine
{
    public int Id { get; }
    public int ClienteId { get; }
    public decimal Totale { get; }

    public Ordine(int id, int clienteId, decimal totale)
    {
        Id = id;
        ClienteId = clienteId;
        Totale = totale;
    }
}

public record ClienteReportDto(
    int ClienteId,
    string Nome,
    string Citta,
    decimal TotaleOrdini);
```

Dati:

```csharp
List<Cliente> clienti = [
    new Cliente(1, "Emilie", "Aosta", true),
    new Cliente(2, "Mario", "Torino", false),
    new Cliente(3, "Anna", "Aosta", true),
    new Cliente(4, "Luca", "Milano", true),
    new Cliente(5, "Giulia", "Torino", false)
];

List<Ordine> ordini = [
    new Ordine(1, 1, 100m),
    new Ordine(2, 1, 50m),
    new Ordine(3, 3, 200m),
    new Ordine(4, 4, 80m)
];
```

Query:

```csharp
List<ClienteReportDto> report = clienti
    .Where(cliente => cliente.Attivo)
    .GroupJoin(
        ordini,
        cliente => cliente.Id,
        ordine => ordine.ClienteId,
        (cliente, ordiniCliente) => new ClienteReportDto(
            cliente.Id,
            cliente.Nome,
            cliente.Citta,
            ordiniCliente.Sum(ordine => ordine.Totale)))
    .OrderBy(item => item.Citta)
    .ThenBy(item => item.Nome)
    .ToList();
```

Stampa:

```csharp
foreach (ClienteReportDto item in report)
{
    Console.WriteLine(
        $"{item.Citta} - {item.Nome}: {item.TotaleOrdini:C}");
}
```

Output possibile:

```text
Aosta - Anna: 200,00 €
Aosta - Emilie: 150,00 €
Milano - Luca: 80,00 €
```

Raggruppamento finale:

```csharp
var reportPerCitta = report
    .GroupBy(item => item.Citta)
    .Select(gruppo => new
    {
        Citta = gruppo.Key,
        NumeroClienti = gruppo.Count(),
        Totale = gruppo.Sum(item => item.TotaleOrdini)
    })
    .OrderBy(item => item.Citta)
    .ToList();
```

Stampa:

```csharp
foreach (var item in reportPerCitta)
{
    Console.WriteLine(
        $"{item.Citta}: {item.NumeroClienti} clienti, totale {item.Totale:C}");
}
```

Output:

```text
Aosta: 2 clienti, totale 350,00 €
Milano: 1 clienti, totale 80,00 €
```

# Errori comuni

## Usare LINQ quando un ciclo è più chiaro

LINQ è potente, ma non sempre è la scelta migliore.

Sospetto:

```csharp
var risultato = clienti
    .Where(c => c.Attivo)
    .Select(c => new
    {
        c.Nome,
        Totale = ordini
            .Where(o => o.ClienteId == c.Id)
            .Select(o => o.Totale)
            .DefaultIfEmpty(0)
            .Aggregate((a, b) => a + b)
    });
```

A volte un ciclo chiaro è meglio di una query troppo densa.

---

## Fare ToList troppo presto

Sconsigliato:

```csharp
var risultato = clienti
    .ToList()
    .Where(c => c.Attivo)
    .ToList();
```

Meglio:

```csharp
var risultato = clienti
    .Where(c => c.Attivo)
    .ToList();
```

Con EF Core questo errore può diventare grave perché sposta filtri dal database alla memoria.

---

## Non capire deferred execution

Esempio:

```csharp
var query = clienti.Where(c => c.Attivo);

clienti.Add(new Cliente(6, "Paolo", "Roma", true));

foreach (Cliente cliente in query)
{
    Console.WriteLine(cliente.Nome);
}
```

La query include anche `Paolo`, perché viene eseguita al momento del `foreach`.

Se volevamo congelare il risultato prima:

```csharp
var lista = clienti
    .Where(c => c.Attivo)
    .ToList();
```

---

## Usare First quando il risultato può mancare

Sbagliato:

```csharp
Cliente cliente = clienti.First(c => c.Id == 999);
```

Se il cliente può mancare:

```csharp
Cliente? cliente = clienti.FirstOrDefault(c => c.Id == 999);
```

---

## Usare Single senza voler davvero unicità

`Single` è corretto solo se vuoi imporre che il risultato sia unico.

Se ti basta il primo:

```csharp
FirstOrDefault
```

Se deve essere esattamente uno:

```csharp
Single
```

---

## Usare Count() > 0 invece di Any()

Sconsigliato:

```csharp
if (clienti.Count() > 0)
{
}
```

Meglio:

```csharp
if (clienti.Any())
{
}
```

---

## Ignorare il costo delle query annidate

Sospetto:

```csharp
var report = clienti.Select(cliente => new
{
    cliente.Nome,
    Totale = ordini
        .Where(ordine => ordine.ClienteId == cliente.Id)
        .Sum(ordine => ordine.Totale)
});
```

Su liste piccole va bene.

Su grandi volumi può essere inefficiente perché per ogni cliente scorre gli ordini.

Meglio raggruppare prima:

```csharp
Dictionary<int, decimal> totaliPerCliente = ordini
    .GroupBy(o => o.ClienteId)
    .ToDictionary(
        gruppo => gruppo.Key,
        gruppo => gruppo.Sum(o => o.Totale));

var report = clienti.Select(cliente => new
{
    cliente.Nome,
    Totale = totaliPerCliente.TryGetValue(cliente.Id, out decimal totale)
        ? totale
        : 0m
});
```

# Linee guida pratiche

```text
Usa Where
  per filtrare.

Usa Select
  per trasformare.

Usa OrderBy e ThenBy
  per ordinare con uno o più criteri.

Usa GroupBy
  per aggregare per chiave.

Usa Join
  per collegare sequenze.

Usa GroupJoin
  per relazioni uno-a-molti.

Usa Any
  per verificare esistenza.

Usa All
  per verificare che tutti rispettino una condizione.

Usa Count, Sum, Average, Min, Max
  per aggregazioni.

Usa FirstOrDefault
  quando un elemento può mancare.

Usa Single
  solo quando deve esistere esattamente un elemento.

Ricorda deferred execution
  le query vengono spesso eseguite solo quando enumerate.

Usa ToList o ToArray
  quando vuoi materializzare il risultato.

Distingui IEnumerable<T> e IQueryable<T>
  soprattutto quando lavori con EF Core.
```

# Riepilogo

In questo capitolo abbiamo introdotto LINQ, uno degli strumenti più importanti del C# moderno.

Abbiamo visto che LINQ permette di interrogare sequenze usando due stili:

```text
query syntax
  simile a SQL.

method syntax
  basata su metodi di estensione e lambda.
```

Abbiamo studiato i principali operatori:

```text
Where
Select
OrderBy
ThenBy
GroupBy
Join
GroupJoin
Any
All
Count
Sum
Average
Min
Max
First
Single
FirstOrDefault
```

Abbiamo poi chiarito due concetti fondamentali:

```text
Deferred execution
  la query viene eseguita quando viene enumerata.

Immediate execution
  la query viene eseguita subito per produrre un risultato concreto.
```

Infine abbiamo distinto:

```text
IEnumerable<T>
  sequenza .NET, spesso in memoria.

IQueryable<T>
  query traducibile da un provider, per esempio EF Core verso SQL.
```

La regola principale è questa:

```text
LINQ rende il codice più espressivo quando descrive chiaramente una trasformazione di dati.
Ma va usato conoscendo esecuzione differita, materializzazione e costo delle operazioni.
```

Questo capitolo prepara direttamente il terreno per EF Core: molte query che scriveremo contro un database useranno la stessa forma LINQ, ma verranno tradotte in SQL.

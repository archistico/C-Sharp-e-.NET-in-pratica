# 04.06 — EF Core avanzato e performance

Nei capitoli precedenti abbiamo introdotto EF Core e poi abbiamo approfondito la modellazione: relazioni, chiavi, indici, owned types, value objects, shadow properties, concurrency token, soft delete e global query filters.

Ora affrontiamo un tema decisivo: **usare EF Core senza usarlo in modo ingenuo**.

EF Core è comodo, ma proprio per questo può nascondere costi importanti:

```text
query generate male
troppe query al database
caricamento di dati inutili
tracking non necessario
lazy loading incontrollato
N+1 query
update inefficienti
transazioni gestite male
repository pattern usato senza criterio
```

Il punto centrale del capitolo è questo:

```text
EF Core non è lento per definizione.
Diventa lento quando non controlliamo caricamento, tracking, query generate e ciclo di vita del DbContext.
```

---

# Loading delle relazioni

Quando un’entità ha relazioni con altre entità, dobbiamo decidere **come** caricarle.

EF Core offre tre strategie principali:

```text
Eager loading
  carico subito i dati collegati.

Lazy loading
  carico i dati collegati quando accedo alla proprietà.

Explicit loading
  carico i dati collegati manualmente quando decido io.
```

Queste strategie sembrano simili, ma hanno conseguenze molto diverse sulle performance.

---

# Eager loading

L’**eager loading** carica dati collegati insieme alla query principale.

Esempio:

```csharp
List<Ordine> ordini = await dbContext.Ordini
    .Include(o => o.Cliente)
    .Include(o => o.Righe)
    .ToListAsync();
```

Qui EF Core carica:

```text
Ordini
Clienti collegati
Righe collegate
```

L’eager loading si usa quando sappiamo già che ci serviranno i dati correlati.

Microsoft documenta `Include` come metodo per specificare dati correlati da includere nei risultati di una query. ([Microsoft Learn][1])

---

## `Include`

`Include` carica una navigation property.

```csharp
List<Ordine> ordini = await dbContext.Ordini
    .Include(o => o.Cliente)
    .ToListAsync();
```

Se poi accediamo a:

```csharp
ordine.Cliente.Nome
```

il cliente è già stato caricato.

---

## `ThenInclude`

`ThenInclude` permette di includere livelli successivi.

Esempio:

```csharp
List<Ordine> ordini = await dbContext.Ordini
    .Include(o => o.Righe)
        .ThenInclude(r => r.Prodotto)
    .ToListAsync();
```

Qui carichiamo:

```text
Ordini
  Righe
    Prodotto di ogni riga
```

---

## Include multipli

```csharp
List<Ordine> ordini = await dbContext.Ordini
    .Include(o => o.Cliente)
    .Include(o => o.Righe)
        .ThenInclude(r => r.Prodotto)
    .ToListAsync();
```

È potente, ma bisogna fare attenzione: includere troppe relazioni può produrre query pesanti, molte righe duplicate o una quantità eccessiva di dati trasferiti.

---

# Problema delle query singole troppo grandi

Quando EF Core carica più collezioni correlate usando JOIN in una query singola, il risultato SQL può produrre molte righe duplicate. Questo fenomeno è spesso chiamato **cartesian explosion**.

Esempio concettuale:

```text
Ordine
  10 righe
  5 note
```

Una query con più collection include può produrre combinazioni tra righe e note, aumentando il numero di righe restituite.

EF Core documenta il problema delle single query con JOIN e propone le split query come alternativa in scenari in cui i JOIN generano problemi di performance. ([Microsoft Learn][1])

---

## `AsSplitQuery`

```csharp
List<Ordine> ordini = await dbContext.Ordini
    .Include(o => o.Righe)
    .Include(o => o.Note)
    .AsSplitQuery()
    .ToListAsync();
```

Con `AsSplitQuery`, EF Core divide il caricamento in più query SQL.

Vantaggio:

```text
evita esplosioni di righe dovute a JOIN multipli
```

Svantaggio:

```text
può aumentare il numero di roundtrip verso il database
```

Regola pratica:

```text
Include semplice
  spesso va bene come single query.

Più collection Include
  valuta AsSplitQuery.

Query critiche
  guarda sempre il SQL generato.
```

# Lazy loading

Il **lazy loading** carica una relazione solo quando accediamo alla navigation property.

Esempio concettuale:

```csharp
List<Cliente> clienti = await dbContext.Clienti.ToListAsync();

foreach (Cliente cliente in clienti)
{
    Console.WriteLine(cliente.Ordini.Count);
}
```

Se `Ordini` è configurato per lazy loading, ogni accesso a `cliente.Ordini` può generare una query al database.

---

## Configurazione lazy loading con proxy

Per usare i proxy serve il pacchetto:

```text
Microsoft.EntityFrameworkCore.Proxies
```

Configurazione:

```csharp
builder.Services.AddDbContext<AppDbContext>(options =>
{
    options.UseSqlServer(connectionString);
    options.UseLazyLoadingProxies();
});
```

Le navigation devono essere `virtual`:

```csharp
public class Cliente
{
    public int Id { get; set; }
    public string Nome { get; set; } = "";

    public virtual List<Ordine> Ordini { get; set; } = new();
}
```

---

## Perché il lazy loading è rischioso

Il lazy loading è comodo, ma rende meno visibile quando partono query al database.

Questo è il classico problema:

```csharp
List<Cliente> clienti = await dbContext.Clienti.ToListAsync();

foreach (Cliente cliente in clienti)
{
    Console.WriteLine($"{cliente.Nome}: {cliente.Ordini.Count}");
}
```

Se ci sono 100 clienti, possiamo avere:

```text
1 query per caricare i clienti
100 query per caricare gli ordini dei singoli clienti
```

Totale:

```text
101 query
```

Questo è il problema **N+1**.

Microsoft raccomanda di evitare il lazy loading quando può attivare involontariamente il problema N+1, preferendo eager loading o explicit loading perché rendono evidente nel codice quando avvengono accessi al database. ([Microsoft Learn][2])

---

# N+1 problem

Il problema N+1 si verifica quando:

```text
1 query carica la lista principale
N query caricano dati correlati, una per ogni elemento
```

Esempio:

```csharp
List<Cliente> clienti = await dbContext.Clienti.ToListAsync();

foreach (Cliente cliente in clienti)
{
    decimal totale = cliente.Ordini.Sum(o => o.Totale);
    Console.WriteLine($"{cliente.Nome}: {totale}");
}
```

Se `Ordini` viene caricato in lazy loading, la query è molto inefficiente.

---

## Soluzione con Include

```csharp
List<Cliente> clienti = await dbContext.Clienti
    .Include(c => c.Ordini)
    .ToListAsync();

foreach (Cliente cliente in clienti)
{
    decimal totale = cliente.Ordini.Sum(o => o.Totale);
    Console.WriteLine($"{cliente.Nome}: {totale}");
}
```

Meglio ancora, se serve solo un report, usare una proiezione.

---

## Soluzione con proiezione

```csharp
var report = await dbContext.Clienti
    .Select(c => new
    {
        c.Id,
        c.Nome,
        TotaleOrdini = c.Ordini.Sum(o => o.Totale)
    })
    .ToListAsync();
```

Questa query chiede direttamente al database il risultato desiderato.

Regola pratica:

```text
Per visualizzare dati:
  preferisci proiezioni verso DTO.

Per modificare entità:
  carica entità tracciate.

Per navigare un grafo:
  usa Include con attenzione.
```

---

# Explicit loading

L’**explicit loading** carica una relazione manualmente.

Esempio:

```csharp
Cliente? cliente = await dbContext.Clienti
    .FirstOrDefaultAsync(c => c.Id == id);

if (cliente is not null)
{
    await dbContext.Entry(cliente)
        .Collection(c => c.Ordini)
        .LoadAsync();
}
```

Qui il caricamento degli ordini è esplicito.

Per una reference navigation:

```csharp
await dbContext.Entry(ordine)
    .Reference(o => o.Cliente)
    .LoadAsync();
```

EF Core 8 supporta anche explicit loading su entità non tracciate; diversamente dal lazy loading, questo può essere fatto in modo asincrono. ([Microsoft Learn][3])

---

## Quando usare explicit loading

Usalo quando:

```text
non sai in anticipo se i dati correlati serviranno
vuoi decidere caso per caso
vuoi rendere visibile il roundtrip al database
vuoi caricare una relazione dopo una decisione applicativa
```

Esempio:

```csharp
Cliente? cliente = await dbContext.Clienti
    .FirstOrDefaultAsync(c => c.Id == id);

if (cliente is null)
{
    return;
}

if (devoMostrareStorico)
{
    await dbContext.Entry(cliente)
        .Collection(c => c.Ordini)
        .LoadAsync();
}
```

# `AsNoTracking`

Per impostazione predefinita, EF Core traccia le entità caricate.

Questo è utile se vogliamo modificarle:

```csharp
Cliente? cliente = await dbContext.Clienti
    .FirstOrDefaultAsync(c => c.Id == id);

cliente!.Nome = "Nuovo nome";

await dbContext.SaveChangesAsync();
```

Ma se dobbiamo solo leggere dati, il tracking può essere inutile.

```csharp
List<ClienteListaDto> clienti = await dbContext.Clienti
    .AsNoTracking()
    .Where(c => c.Attivo)
    .Select(c => new ClienteListaDto(
        c.Id,
        c.Nome,
        c.Email))
    .ToListAsync();
```

`AsNoTracking` disabilita il tracking per quella query, riducendo memoria e lavoro del Change Tracker nelle query di sola lettura.

---

## Quando usare AsNoTracking

```text
Query di sola lettura
  usa AsNoTracking.

Query per modificare entità
  lascia il tracking.

Query verso DTO
  spesso il tracking non serve comunque,
  ma AsNoTracking comunica chiaramente l’intenzione.
```

---

## Attenzione

Questo codice non salva la modifica:

```csharp
Cliente? cliente = await dbContext.Clienti
    .AsNoTracking()
    .FirstOrDefaultAsync(c => c.Id == id);

cliente!.Nome = "Nuovo nome";

await dbContext.SaveChangesAsync();
```

L’entità non è tracciata.

Per aggiornare, meglio caricare senza `AsNoTracking` oppure riattaccare l’entità in modo esplicito, con attenzione.

---

# Query compilate

EF Core deve trasformare una query LINQ in una query eseguibile. In molti casi EF Core riusa automaticamente parti del lavoro, ma per query molto frequenti e critiche possiamo usare query compilate.

Esempio:

```csharp
private static readonly Func<AppDbContext, int, Task<Cliente?>> TrovaClientePerId =
    EF.CompileAsyncQuery(
        (AppDbContext dbContext, int id) =>
            dbContext.Clienti.FirstOrDefault(c => c.Id == id));
```

Uso:

```csharp
Cliente? cliente = await TrovaClientePerId(dbContext, id);
```

---

## Quando usare query compilate

Usale solo quando:

```text
la query viene eseguita moltissime volte
hai misurato che la compilazione query è un costo rilevante
la query è stabile
sei in un punto realmente critico
```

Non servono per ogni query.

Regola pratica:

```text
Prima scrivi query chiare.
Poi misura.
Solo dopo valuta query compilate.
```

---

# Modelli compilati

EF Core costruisce un modello interno che descrive mapping, entità, proprietà e relazioni. Nei modelli grandi, la costruzione del modello può incidere sul tempo di avvio.

I **compiled models** permettono di generare il modello a compile-time invece che costruirlo interamente a runtime.

Microsoft segnala che i compiled models, introdotti in EF Core 6, possono migliorare il tempo di avvio per modelli grandi perché il modello viene generato a compile-time. ([Microsoft Learn][4])

---

## Generare un modello compilato

Comando tipico:

```bash
dotnet ef dbcontext optimize
```

Questo genera codice C# per il modello compilato.

Poi si configura il contesto, per esempio:

```csharp
options.UseModel(AppDbContextModel.Instance);
```

Il nome esatto del modello generato dipende dal progetto e dal namespace.

---

## Quando usare modelli compilati

Sono utili quando:

```text
il modello EF Core è grande
il tempo di avvio è importante
l’app è serverless o avviata spesso
hai molte entità e configurazioni
hai misurato startup lento legato al model building
```

Non sono necessari nella maggior parte dei piccoli progetti.

# Batch update

Prima di EF Core 7, per aggiornare più righe era comune fare:

```csharp
List<Cliente> clienti = await dbContext.Clienti
    .Where(c => !c.Attivo)
    .ToListAsync();

foreach (Cliente cliente in clienti)
{
    cliente.Attivo = true;
}

await dbContext.SaveChangesAsync();
```

Questo comporta:

```text
lettura delle righe
materializzazione degli oggetti
tracking
un update per ogni entità o batch gestito da EF
```

Per aggiornamenti massivi, è più efficiente eseguire direttamente un comando SQL.

---

# `ExecuteUpdate`

`ExecuteUpdate` permette di aggiornare righe direttamente nel database senza caricare entità in memoria.

Esempio:

```csharp
int righeAggiornate = await dbContext.Clienti
    .Where(c => !c.Attivo)
    .ExecuteUpdateAsync(setters => setters
        .SetProperty(c => c.Attivo, true));
```

Questo genera un update diretto.

Esempio concettuale SQL:

```sql
UPDATE Clienti
SET Attivo = 1
WHERE Attivo = 0
```

EF Core 7 ha introdotto `ExecuteUpdate` e `ExecuteDelete`; queste API permettono di esprimere update/delete usando LINQ, sono più efficienti per scenari bulk e bypassano completamente il Change Tracker, eseguendo immediatamente invece di attendere `SaveChanges`. ([Microsoft Learn][4])

---

## ExecuteUpdate con più proprietà

```csharp
int righe = await dbContext.Clienti
    .Where(c => c.IsDeleted)
    .ExecuteUpdateAsync(setters => setters
        .SetProperty(c => c.Attivo, false)
        .SetProperty(c => c.DataModifica, DateTimeOffset.UtcNow));
```

---

## Attenzione al Change Tracker

`ExecuteUpdate` bypassa il Change Tracker.

Se nello stesso `DbContext` hai già entità caricate, queste non vengono aggiornate automaticamente in memoria.

Esempio problematico:

```csharp
Cliente cliente = await dbContext.Clienti.FirstAsync(c => c.Id == 1);

await dbContext.Clienti
    .Where(c => c.Id == 1)
    .ExecuteUpdateAsync(setters => setters
        .SetProperty(c => c.Nome, "Nuovo nome"));

Console.WriteLine(cliente.Nome); // potrebbe essere ancora il vecchio valore
```

Regola pratica:

```text
Usa ExecuteUpdate per operazioni bulk.
Evita di mescolarlo con entità già tracciate nello stesso contesto,
oppure ricarica i dati.
```

---

# `ExecuteDelete`

`ExecuteDelete` cancella righe direttamente nel database.

```csharp
int righeCancellate = await dbContext.Clienti
    .Where(c => c.IsDeleted)
    .ExecuteDeleteAsync();
```

Esempio concettuale SQL:

```sql
DELETE FROM Clienti
WHERE IsDeleted = 1
```

Anche `ExecuteDelete` bypassa il Change Tracker ed esegue subito. ([Microsoft Learn][4])

---

## Soft delete ed ExecuteDelete

Se usi soft delete, attenzione:

```csharp
await dbContext.Clienti
    .Where(c => c.Id == id)
    .ExecuteDeleteAsync();
```

Questo fa una cancellazione fisica.

Per soft delete, meglio:

```csharp
await dbContext.Clienti
    .Where(c => c.Id == id)
    .ExecuteUpdateAsync(setters => setters
        .SetProperty(c => c.IsDeleted, true)
        .SetProperty(c => c.DeletedAt, DateTimeOffset.UtcNow));
```

# Transazioni

EF Core usa automaticamente una transazione per `SaveChanges` quando il provider la supporta e quando ci sono più operazioni da salvare.

Esempio:

```csharp
dbContext.Ordini.Add(ordine);
dbContext.RigheOrdine.AddRange(righe);

await dbContext.SaveChangesAsync();
```

Se una parte fallisce, le modifiche vengono annullate.

---

## Transazione esplicita

A volte serve controllare manualmente la transazione.

```csharp
await using var transaction =
    await dbContext.Database.BeginTransactionAsync();

try
{
    dbContext.Ordini.Add(ordine);
    await dbContext.SaveChangesAsync();

    await servizioEsterno.RegistraMovimentoAsync(ordine.Id);

    ordine.Stato = "Confermato";
    await dbContext.SaveChangesAsync();

    await transaction.CommitAsync();
}
catch
{
    await transaction.RollbackAsync();
    throw;
}
```

Attenzione: se nella transazione includiamo chiamate esterne, dobbiamo valutare bene cosa succede se il servizio esterno riesce ma il database fallisce, o viceversa.

---

## Quando usare transazioni esplicite

```text
più SaveChanges nella stessa unità atomica
ADO.NET + EF Core insieme
operazioni su più repository nello stesso DbContext
comandi SQL manuali e operazioni EF Core insieme
```

# Concorrenza ottimistica

La concorrenza ottimistica parte da un’idea:

```text
Non blocco il record mentre l’utente lavora.
Controllo al salvataggio se qualcuno lo ha modificato nel frattempo.
```

In EF Core si configura un concurrency token.

Con SQL Server, spesso si usa `rowversion`.

```csharp
public class Cliente
{
    public int Id { get; set; }
    public string Nome { get; set; } = "";

    public byte[] RowVersion { get; set; } = [];
}
```

Configurazione:

```csharp
modelBuilder.Entity<Cliente>()
    .Property(c => c.RowVersion)
    .IsRowVersion();
```

Quando EF Core aggiorna la riga, include il valore originale della `RowVersion` nella condizione. Se nessuna riga viene modificata, EF Core capisce che c’è stato un conflitto e genera `DbUpdateConcurrencyException`. La documentazione EF Core descrive proprio questo meccanismo come concorrenza ottimistica basata su concurrency token. ([Microsoft Learn][5])

---

## Gestire il conflitto

```csharp
try
{
    await dbContext.SaveChangesAsync();
}
catch (DbUpdateConcurrencyException ex)
{
    foreach (var entry in ex.Entries)
    {
        await entry.ReloadAsync();
    }

    throw new InvalidOperationException(
        "Il record è stato modificato da un altro utente. Ricaricare i dati e riprovare.",
        ex);
}
```

Strategie possibili:

```text
client wins
  salvo comunque i valori dell’utente corrente.

database wins
  scarto le modifiche locali e ricarico dal database.

merge manuale
  mostro differenze e faccio decidere l’utente.

retry
  ricalcolo e riprovo, se l’operazione è automatica.
```

# Logging SQL

Vedere il SQL generato è essenziale per capire cosa sta facendo EF Core.

In sviluppo possiamo configurare logging:

```csharp
builder.Services.AddDbContext<AppDbContext>(options =>
{
    options.UseSqlServer(connectionString);
    options.LogTo(Console.WriteLine, LogLevel.Information);
});
```

Per vedere parametri e valori sensibili:

```csharp
options.EnableSensitiveDataLogging();
```

Attenzione: `EnableSensitiveDataLogging` può scrivere nei log dati sensibili. Va usato solo in sviluppo o in ambienti controllati.

---

## Query tag

Possiamo aggiungere tag alle query per riconoscerle nei log SQL.

```csharp
List<ClienteListaDto> clienti = await dbContext.Clienti
    .TagWith("Query lista clienti attivi per dashboard")
    .Where(c => c.Attivo)
    .Select(c => new ClienteListaDto(c.Id, c.Nome))
    .ToListAsync();
```

Nel SQL generato comparirà un commento.

Questo è utile per collegare codice C# e query nei log.

# Interceptor

Per scenari più avanzati, EF Core supporta gli interceptor.

Gli interceptor permettono di intercettare, modificare o sopprimere operazioni EF Core, incluse operazioni di basso livello come comandi database e operazioni di alto livello come `SaveChanges`. Per semplice logging, però, Microsoft indica che logging o `Microsoft.Extensions.Logging` sono scelte più adatte. ([Microsoft Learn][5])

Esempi d’uso:

```text
audit automatico
multi-tenancy
diagnostica avanzata
modifica comandi
misurazione tempi query
```

# Repository pattern: pro e contro

Il **Repository pattern** incapsula l’accesso ai dati dietro un’interfaccia.

Esempio:

```csharp
public interface IClienteRepository
{
    Task<Cliente?> TrovaAsync(int id);
    Task<List<Cliente>> ListaAttiviAsync();
    void Aggiungi(Cliente cliente);
}
```

Implementazione EF Core:

```csharp
public class ClienteRepository : IClienteRepository
{
    private readonly AppDbContext dbContext;

    public ClienteRepository(AppDbContext dbContext)
    {
        this.dbContext = dbContext;
    }

    public Task<Cliente?> TrovaAsync(int id)
    {
        return dbContext.Clienti
            .FirstOrDefaultAsync(c => c.Id == id);
    }

    public Task<List<Cliente>> ListaAttiviAsync()
    {
        return dbContext.Clienti
            .Where(c => c.Attivo)
            .OrderBy(c => c.Nome)
            .ToListAsync();
    }

    public void Aggiungi(Cliente cliente)
    {
        dbContext.Clienti.Add(cliente);
    }
}
```

---

## Vantaggi del Repository pattern

```text
isola query ricorrenti
centralizza accesso ai dati
nasconde dettagli EF Core al layer applicativo
può aiutare nei test
può rendere più chiara l’intenzione del dominio
può evitare IQueryable sparso ovunque
```

---

## Svantaggi del Repository pattern

EF Core stesso ha già caratteristiche simili a repository e unit of work:

```text
DbSet
  rappresenta una raccolta di entità.

DbContext
  rappresenta una unità di lavoro.
```

Un repository troppo generico può peggiorare il design.

Sospetto:

```csharp
public interface IRepository<T>
{
    Task<T?> GetByIdAsync(int id);
    Task<List<T>> GetAllAsync();
    void Add(T entity);
    void Update(T entity);
    void Delete(T entity);
}
```

Questo spesso aggiunge poco valore e nasconde funzionalità importanti di EF Core.

---

## Repository utile

Un repository è utile quando espone query di dominio significative.

```csharp
public interface IOrdineRepository
{
    Task<Ordine?> TrovaConRigheAsync(int ordineId);
    Task<List<Ordine>> ListaDaFatturareAsync(DateOnly data);
    Task<bool> EsisteNumeroDocumentoAsync(string numeroDocumento);
}
```

Qui il repository parla il linguaggio dell’applicazione.

---

## Repository poco utile

```csharp
public interface IRepository<T>
{
    IQueryable<T> Query();
    void Add(T entity);
    void Remove(T entity);
}
```

Se esponi `IQueryable<T>` da ogni repository, stai spesso riportando EF Core fuori dall’astrazione.

Regola pratica:

```text
Repository utile
  incapsula query significative e protegge il modello dati.

Repository inutile
  replica DbSet senza aggiungere semantica.
```

# Unit of Work: pro e contro

Il pattern **Unit of Work** rappresenta un insieme di modifiche da salvare insieme.

In EF Core, `DbContext` è già una Unit of Work.

Esempio:

```csharp
dbContext.Ordini.Add(ordine);
dbContext.Movimenti.Add(movimento);
dbContext.Notifiche.Add(notifica);

await dbContext.SaveChangesAsync();
```

Tutte le modifiche vengono salvate insieme.

---

## UnitOfWork wrapper

A volte si vede:

```csharp
public interface IUnitOfWork
{
    Task<int> SaveChangesAsync();
}
```

Implementazione:

```csharp
public class EfUnitOfWork : IUnitOfWork
{
    private readonly AppDbContext dbContext;

    public EfUnitOfWork(AppDbContext dbContext)
    {
        this.dbContext = dbContext;
    }

    public Task<int> SaveChangesAsync()
    {
        return dbContext.SaveChangesAsync();
    }
}
```

Questo può essere utile se vuoi impedire ai servizi applicativi di dipendere direttamente dal `DbContext`.

Ma può anche essere un wrapper povero che aggiunge poco.

---

## Quando ha senso

```text
architettura a layer rigidi
dominio separato dall’infrastruttura
più repository coordinati
necessità di mock nei test
regole applicative intorno al commit
outbox pattern
audit centralizzato
```

---

## Quando non serve

```text
applicazione CRUD semplice
Minimal API piccola
servizi che usano già direttamente DbContext
wrapper che contiene solo SaveChangesAsync
```

Regola pratica:

```text
DbContext è già Unit of Work.
Aggiungi un tuo UnitOfWork solo se porta una regola architetturale reale.
```

# Esempio completo: query efficiente per dashboard

Scenario:

```text
Dashboard clienti:
- solo clienti attivi
- numero ordini
- totale ordini
- ultimo ordine
- nessun tracking
- DTO finale
```

DTO:

```csharp
public record ClienteDashboardDto(
    int ClienteId,
    string Nome,
    int NumeroOrdini,
    decimal TotaleOrdini,
    DateOnly? UltimoOrdine);
```

Query:

```csharp
List<ClienteDashboardDto> dashboard = await dbContext.Clienti
    .AsNoTracking()
    .Where(c => c.Attivo)
    .Select(c => new ClienteDashboardDto(
        c.Id,
        c.Nome,
        c.Ordini.Count,
        c.Ordini.Sum(o => o.Totale),
        c.Ordini
            .OrderByDescending(o => o.Data)
            .Select(o => (DateOnly?)o.Data)
            .FirstOrDefault()))
    .OrderByDescending(x => x.TotaleOrdini)
    .Take(50)
    .ToListAsync();
```

Questa query evita:

```text
lazy loading
N+1
caricamento di entità complete
tracking inutile
Include non necessario
```

Per una dashboard, spesso questa forma è migliore di:

```csharp
List<Cliente> clienti = await dbContext.Clienti
    .Include(c => c.Ordini)
    .ToListAsync();
```

# Esempio completo: operazione bulk

Scenario:

```text
Disattivare clienti che non hanno ordini da più di 3 anni.
```

```csharp
DateOnly limite = DateOnly.FromDateTime(
    DateTime.Today.AddYears(-3));

int righeAggiornate = await dbContext.Clienti
    .Where(c => c.Attivo)
    .Where(c => !c.Ordini.Any(o => o.Data >= limite))
    .ExecuteUpdateAsync(setters => setters
        .SetProperty(c => c.Attivo, false)
        .SetProperty(c => c.DataModifica, DateTimeOffset.UtcNow));
```

Vantaggi:

```text
non carica clienti in memoria
non attiva tracking
genera un update diretto
è adatto a operazioni massive
```

# Esempio completo: transazione esplicita

Scenario:

```text
Creare ordine.
Creare movimento contabile.
Aggiornare stato cliente.
Salvare tutto insieme.
```

```csharp
await using var transaction =
    await dbContext.Database.BeginTransactionAsync();

try
{
    dbContext.Ordini.Add(ordine);
    dbContext.MovimentiContabili.Add(movimento);

    cliente.DataUltimoOrdine = ordine.Data;

    await dbContext.SaveChangesAsync();

    await transaction.CommitAsync();
}
catch
{
    await transaction.RollbackAsync();
    throw;
}
```

Se tutte le operazioni usano lo stesso `DbContext`, spesso una singola `SaveChangesAsync` è già transazionale. La transazione esplicita diventa utile quando l’unità di lavoro è più articolata.

# Errori comuni

## Usare lazy loading senza rendersene conto

Sospetto:

```csharp
foreach (Cliente cliente in clienti)
{
    Console.WriteLine(cliente.Ordini.Count);
}
```

Se `Ordini` è lazy loaded, potresti avere N+1 query.

Meglio usare proiezione o `Include`.

---

## Usare Include per tutto

Sospetto:

```csharp
var clienti = await dbContext.Clienti
    .Include(c => c.Ordini)
    .ThenInclude(o => o.Righe)
    .ThenInclude(r => r.Prodotto)
    .ToListAsync();
```

Se ti servono solo nome cliente e totale, è meglio una proiezione.

---

## Dimenticare AsNoTracking nelle query di sola lettura

Sospetto:

```csharp
var lista = await dbContext.Clienti
    .Where(c => c.Attivo)
    .ToListAsync();
```

Per sola lettura:

```csharp
var lista = await dbContext.Clienti
    .AsNoTracking()
    .Where(c => c.Attivo)
    .ToListAsync();
```

---

## Usare ToListAsync troppo presto

Sbagliato:

```csharp
var clienti = await dbContext.Clienti.ToListAsync();

var risultato = clienti
    .Where(c => c.Attivo)
    .Select(c => new ClienteDto(c.Id, c.Nome))
    .ToList();
```

Meglio:

```csharp
var risultato = await dbContext.Clienti
    .Where(c => c.Attivo)
    .Select(c => new ClienteDto(c.Id, c.Nome))
    .ToListAsync();
```

---

## Usare ExecuteUpdate pensando che aggiorni entità tracciate

`ExecuteUpdate` aggiorna il database direttamente. Non aggiorna automaticamente gli oggetti già caricati nel Change Tracker.

---

## Repository generico senza valore

Sospetto:

```csharp
repository.GetAll()
```

che internamente fa:

```csharp
dbContext.Set<T>().ToListAsync()
```

Questo spesso nasconde EF Core senza migliorare il design.

---

## UnitOfWork wrapper inutile

Sospetto:

```csharp
await unitOfWork.SaveChangesAsync();
```

se `unitOfWork` è solo un wrapper intorno a `dbContext.SaveChangesAsync()` e non aggiunge alcuna regola architetturale.

---

# Linee guida pratiche

```text
Preferisci proiezioni verso DTO
  per schermate, API e dashboard.

Usa Include
  quando devi davvero navigare il grafo di entità.

Evita lazy loading
  se non hai una ragione forte e controllo sui roundtrip.

Usa explicit loading
  quando il caricamento deve essere deciso a runtime.

Usa AsNoTracking
  per query di sola lettura.

Controlla il SQL generato
  soprattutto nelle query importanti.

Usa ExecuteUpdate / ExecuteDelete
  per operazioni massive che non richiedono tracking.

Non mescolare alla leggera bulk update e entità già tracciate
  nello stesso DbContext.

Usa transazioni esplicite
  solo quando SaveChanges non basta.

Gestisci la concorrenza ottimistica
  nei dati modificabili da più utenti.

Usa repository e unit of work
  solo se portano valore architetturale reale.
```

# Riepilogo

In questo capitolo abbiamo visto EF Core da una prospettiva più professionale.

Abbiamo studiato le strategie di caricamento:

```text
Eager loading
  carica subito con Include.

Lazy loading
  carica al primo accesso, ma rischia N+1.

Explicit loading
  carica manualmente quando serve.
```

Abbiamo visto strumenti di performance:

```text
AsNoTracking
  evita tracking inutile.

Query compilate
  utili per query ripetute e critiche.

Modelli compilati
  utili per ridurre startup in modelli grandi.

ExecuteUpdate
  update diretto senza tracking.

ExecuteDelete
  delete diretto senza tracking.

Logging SQL
  fondamentale per capire cosa viene eseguito.
```

Abbiamo affrontato:

```text
N+1 problem
transazioni
concorrenza ottimistica
repository pattern
unit of work
```

La regola principale è questa:

```text
EF Core va usato intenzionalmente:
decidi cosa caricare, quando tracciarlo, quando materializzarlo
e quale SQL vuoi ragionevolmente ottenere.
```

Un uso maturo di EF Core non consiste nel mettere `Include` ovunque o nel trasformare ogni query in repository generici. Consiste nel progettare query chiare, misurabili e coerenti con il caso d’uso.

# 04.02 — LINQ avanzato e trasformazione dati

Nel capitolo precedente abbiamo introdotto LINQ come strumento per interrogare collezioni e sequenze. Abbiamo visto `Where`, `Select`, `OrderBy`, `GroupBy`, `Join`, aggregazioni e la differenza tra `IEnumerable<T>` e `IQueryable<T>`.

Ora facciamo un passo avanti.

LINQ non serve solo a filtrare liste. In un’applicazione reale LINQ viene usato per:

```text id="r4pttp"
costruire DTO
trasformare strutture annidate
produrre report
raggruppare dati
calcolare statistiche
unire sequenze
preparare dati per UI e API
convertire modelli di dominio in modelli di lettura
```

Il punto centrale del capitolo è questo:

```text id="66rzgc"
LINQ diventa davvero potente quando lo usiamo per trasformare dati,
non solo per cercarli.
```

```csharp id="evvcei"

{

    public string Nome { get; }
    public string Citta { get; }
    public bool Attivo { get; }

    public Cliente(int id, string nome, string citta, bool attivo)

        Attivo = attivo;

{

    public int ClienteId { get; }
    public DateOnly Data { get; }
    public List<RigaOrdine> Righe { get; }

    {
        Id = id;
        ClienteId = clienteId;
        Data = data;

        CodiceProdotto = codiceProdotto;
```

# Proiezioni complesse

Una **proiezione** trasforma un elemento in un altro tipo di risultato.

La proiezione base usa `Select`.

Esempio semplice:

```csharp id="ja4f1i"
var nomi = clienti
    .Select(cliente => cliente.Nome);
```

Una proiezione complessa costruisce invece un oggetto più ricco.

```csharp id="2wi7ac"
var riepilogoClienti = clienti
    .Select(cliente => new
    {
        cliente.Id,
        cliente.Nome,
        cliente.Citta,
        Stato = cliente.Attivo ? "Attivo" : "Non attivo"
    });
```

Uso:

```csharp id="160e64"
foreach (var cliente in riepilogoClienti)
{
    Console.WriteLine($"{cliente.Nome} - {cliente.Citta} - {cliente.Stato}");
}
```

---

## Proiezione con calcolo

```csharp id="wbazw6"
var riepilogoOrdini = ordini
    .Select(ordine => new
    {
        ordine.Id,
        ordine.ClienteId,
        ordine.Data,
        NumeroRighe = ordine.Righe.Count,
        Totale = ordine.Righe.Sum(riga => riga.Subtotale)
    });
```

Output:

```text id="99jkos"
Ordine 1 - Totale 179,80
Ordine 2 - Totale 136,50
Ordine 3 - Totale 29,90
Ordine 4 - Totale 274,90
```

Questa proiezione crea un modello di lettura più adatto a un report.

# Oggetti anonimi

Gli **oggetti anonimi** sono tipi creati dal compilatore senza dichiarare una classe esplicita.

Esempio:

```csharp id="omr6ak"
var cliente = new
{
    Id = 1,
    Nome = "Emilie"
};
```

Con LINQ:

```csharp id="l8zmj1"
var risultato = clienti
    .Select(c => new
    {
        c.Id,
        c.Nome,
        c.Citta
    });
```

Sono utili quando:

```text id="328sxa"
il risultato è locale a un metodo
non deve uscire dall’ambito corrente
serve solo per una trasformazione intermedia
non vuoi creare un DTO formale
```

---

## Limite degli oggetti anonimi

Gli oggetti anonimi sono comodi, ma non sono ideali come contratto pubblico.

Sconsigliato:

```text id="jqsp68"
Restituire oggetti anonimi da API pubbliche.
Usarli come modello condiviso tra layer.
Basare su di essi contratti di serializzazione importanti.
```

Per questi casi meglio usare DTO espliciti.

---

# DTO

Un **DTO** è un Data Transfer Object: un oggetto pensato per trasferire dati.

Esempio:

```csharp id="5dn1yb"
public record ClienteRiepilogoDto(
    int Id,
    string Nome,
    string Citta,
    bool Attivo);
```

Proiezione:

```csharp id="2h4r6p"
List<ClienteRiepilogoDto> dto = clienti
    .Select(c => new ClienteRiepilogoDto(
        c.Id,
        c.Nome,
        c.Citta,
        c.Attivo))
    .ToList();
```

---

## DTO con dati calcolati

```csharp id="koz94h"
public record OrdineRiepilogoDto(
    int OrdineId,
    int ClienteId,
    DateOnly Data,
    int NumeroRighe,
    decimal Totale);
```

Proiezione:

```csharp id="u8z13k"
List<OrdineRiepilogoDto> riepiloghi = ordini
    .Select(o => new OrdineRiepilogoDto(
        o.Id,
        o.ClienteId,
        o.Data,
        o.Righe.Count,
        o.Righe.Sum(r => r.Subtotale)))
    .ToList();
```

---

## Perché usare DTO

I DTO sono utili perché:

```text id="hflnop"
separano il modello interno dal contratto esterno
evitano di esporre entità complete
semplificano serializzazione e API
contengono solo i dati necessari
permettono versionamento più controllato
```

Regola pratica:

```text id="83fvep"
Oggetto anonimo
  per trasformazioni locali.

DTO
  per dati che escono dal metodo, dal layer o dall’applicazione.
```

# Flattening con `SelectMany`

`SelectMany` serve ad appiattire sequenze annidate.

Nel nostro modello:

```text id="j24pyx"
ordini
  ogni ordine contiene molte righe
```

Quindi abbiamo:

```text id="co8qm9"
List<Ordine>
  List<RigaOrdine>
```

Se vogliamo tutte le righe di tutti gli ordini, usiamo `SelectMany`.

```csharp id="d0qhea"
IEnumerable<RigaOrdine> tutteLeRighe = ordini
    .SelectMany(ordine => ordine.Righe);
```

Stampa:

```csharp id="ohj5hp"
foreach (RigaOrdine riga in tutteLeRighe)
{
    Console.WriteLine($"{riga.Descrizione} - {riga.Subtotale}");
}
```

---

## SelectMany mantenendo il padre

Spesso non vogliamo solo la riga, ma anche informazioni dell’ordine.

```csharp id="h4ya37"
var righeConOrdine = ordini
    .SelectMany(
        ordine => ordine.Righe,
        (ordine, riga) => new
        {
            OrdineId = ordine.Id,
            ordine.ClienteId,
            ordine.Data,
            riga.CodiceProdotto,
            riga.Descrizione,
            riga.Quantita,
            riga.PrezzoUnitario,
            riga.Subtotale
        });
```

Qui `SelectMany` produce una riga per ogni riga d’ordine, mantenendo i dati dell’ordine padre.

Output concettuale:

```text id="y5s1wf"
Ordine 1 - Libro C# - 59,80
Ordine 1 - Corso .NET - 120,00
Ordine 2 - Supporto tecnico - 136,50
Ordine 3 - Libro C# - 29,90
Ordine 4 - Corso .NET - 240,00
Ordine 4 - Libro WPF - 34,90
```

---

## Quando usare SelectMany

Usalo quando hai:

```text id="zhttqi"
lista di liste
cliente con ordini
ordine con righe
categoria con prodotti
cartella con file
documento con dettagli
```

e vuoi passare da una struttura annidata a una sequenza piatta.

Regola pratica:

```text id="puh2u0"
Select
  un elemento produce un risultato.

SelectMany
  un elemento produce più risultati, poi vengono appiattiti.
```

## Totale venduto per prodotto

Partiamo dalle righe appiattite:

```csharp id="a2iijq"
var venditePerProdotto = ordini
```

```csharp id="a3dsmn"
var totaliPerCliente = ordini
```

# Join esterni

Nel capitolo precedente abbiamo visto `Join`, che produce un inner join.

Un inner join restituisce solo gli elementi che hanno corrispondenza da entrambe le parti.

Esempio:

```csharp id="4mrdoz"
var clientiConOrdini = clienti.Join(
    ordini,
    cliente => cliente.Id,
    ordine => ordine.ClienteId,
    (cliente, ordine) => new
    {
        Cliente = cliente.Nome,
        OrdineId = ordine.Id
    });
```

Questo esclude i clienti senza ordini.

Un **join esterno sinistro** mantiene invece tutti gli elementi della sequenza di sinistra, anche quando non esiste una corrispondenza a destra.

---

## Left join prima di .NET 10

Prima dei metodi `LeftJoin` e `RightJoin`, un left join in LINQ si scriveva con:

```text id="do3te4"
GroupJoin
SelectMany
DefaultIfEmpty
```

Esempio:

```csharp id="d45d4c"
var leftJoin = clienti
    .GroupJoin(
        ordini,
        cliente => cliente.Id,
        ordine => ordine.ClienteId,
        (cliente, ordiniCliente) => new
        {
            cliente,
            ordiniCliente
        })
    .SelectMany(
        x => x.ordiniCliente.DefaultIfEmpty(),
        (x, ordine) => new
        {
            Cliente = x.cliente.Nome,
            OrdineId = ordine?.Id,
            Totale = ordine is null
                ? 0m
                : ordine.Righe.Sum(r => r.Subtotale)
        });
```

Output concettuale:

```text id="f2i45z"
Emilie - Ordine 1 - 179,80
Emilie - Ordine 2 - 136,50
Mario - nessun ordine - 0
Anna - Ordine 3 - 29,90
Luca - Ordine 4 - 274,90
Giulia - nessun ordine - 0
```

È corretto, ma abbastanza verboso.

# `LeftJoin` e `RightJoin`

In .NET 10 sono disponibili `LeftJoin` e `RightJoin` in `System.Linq.Enumerable` e `System.Linq.Queryable`. Microsoft li descrive come metodi che eseguono rispettivamente outer left equijoin e outer right equijoin; nel left join ogni elemento della prima sequenza resta nel risultato anche se la seconda non contiene una corrispondenza. ([Microsoft Learn][1])

---

## LeftJoin in .NET 10

Con .NET 10 possiamo scrivere:

```csharp id="8qgzkc"
var leftJoin = clienti.LeftJoin(
    ordini,
    cliente => cliente.Id,
    ordine => ordine.ClienteId,
    (cliente, ordine) => new
    {
        Cliente = cliente.Nome,
        OrdineId = ordine?.Id,
        Totale = ordine is null
            ? 0m
            : ordine.Righe.Sum(r => r.Subtotale)
    });
```

Questo è molto più leggibile del pattern `GroupJoin` + `DefaultIfEmpty`.

---

## RightJoin in .NET 10

`RightJoin` mantiene tutti gli elementi della sequenza di destra.

Esempio concettuale:

```csharp id="d7xmwk"
var rightJoin = clienti.RightJoin(
    ordini,
    cliente => cliente.Id,
    ordine => ordine.ClienteId,
    (cliente, ordine) => new
    {
        Cliente = cliente?.Nome ?? "Cliente mancante",
        OrdineId = ordine.Id,
        Totale = ordine.Righe.Sum(r => r.Subtotale)
    });
```

È utile quando la sequenza “obbligatoria” è quella di destra.

---

## Attenzione con EF Core

Con `IQueryable<T>`, i metodi disponibili devono essere supportati dal provider. .NET 10 espone `LeftJoin` e `RightJoin` anche per `Queryable`, ma quando si lavora con database bisogna sempre verificare la traduzione effettiva del provider EF Core usato. La documentazione Microsoft indica che i metodi sono presenti sia in `Enumerable` sia in `Queryable`, ma l’esecuzione finale dipende dal provider quando si usa `IQueryable<T>`. ([Microsoft Learn][1])

Regola pratica:

```text id="ti9qwn"
Con IEnumerable<T>
  LeftJoin e RightJoin lavorano in memoria.

Con IQueryable<T>
  devono essere tradotti dal provider, per esempio EF Core verso SQL.
```

# `CountBy`

`CountBy` conta gli elementi raggruppandoli per chiave.

È stato introdotto in .NET 9 insieme ad `AggregateBy`; Microsoft spiega che questi metodi permettono di aggregare stato per chiave senza allocare raggruppamenti intermedi tramite `GroupBy`. ([Microsoft Learn][2])

Prima:

```csharp id="erka4k"
var conteggiPerCitta = clienti
    .GroupBy(c => c.Citta)
    .Select(g => new
    {
        Citta = g.Key,
        Conteggio = g.Count()
    });
```

Con `CountBy`:

```csharp id="fbijgg"
var conteggiPerCitta = clienti
    .CountBy(c => c.Citta);
```

Uso:

```csharp id="qk3sje"
foreach (var item in conteggiPerCitta)
{
    Console.WriteLine($"{item.Key}: {item.Value}");
}
```

Output:

```text id="wbk4d2"
Aosta: 2
Torino: 2
Milano: 1
```

---

## Quando usare CountBy

Usa `CountBy` quando devi calcolare frequenze.

Esempi:

```text id="qqu7vm"
numero clienti per città
numero ordini per cliente
numero prodotti per categoria
numero accessi per giorno
frequenza parole in un testo
```

Esempio:

```csharp id="swmwgp"
var frequenzeProdotti = ordini
    .SelectMany(o => o.Righe)
    .CountBy(r => r.CodiceProdotto);
```

# `AggregateBy`

`AggregateBy` permette di aggregare valori per chiave senza passare da `GroupBy`.

Microsoft documenta `AggregateBy` come un metodo che applica una funzione accumulatrice su una sequenza raggruppando i risultati per chiave. ([Microsoft Learn][3])

Esempio: totale venduto per prodotto.

Con `GroupBy`:

```csharp id="8t9e8a"
var totali = ordini
    .SelectMany(o => o.Righe)
    .GroupBy(r => r.CodiceProdotto)
    .Select(g => new
    {
        Codice = g.Key,
        Totale = g.Sum(r => r.Subtotale)
    });
```

Con `AggregateBy`:

```csharp id="s68h40"
var totali = ordini
    .SelectMany(o => o.Righe)
    .AggregateBy(
        keySelector: r => r.CodiceProdotto,
        seed: 0m,
        func: (totale, riga) => totale + riga.Subtotale);
```

Uso:

```csharp id="9m6kxq"
foreach (var item in totali)
{
    Console.WriteLine($"{item.Key}: {item.Value}");
}
```

---

## AggregateBy con seed per chiave

Alcuni overload permettono di generare il seed a partire dalla chiave.

Esempio concettuale:

```csharp id="5ihoq2"
var statistiche = ordini
    .SelectMany(o => o.Righe)
    .AggregateBy(
        keySelector: r => r.CodiceProdotto,
        seedSelector: codice => new StatisticaProdotto(codice),
        func: (statistica, riga) => statistica.Aggiungi(riga));
```

Dove `StatisticaProdotto` potrebbe essere un tipo dedicato all’accumulo.

# `Aggregate`

`Aggregate` riduce una sequenza a un unico valore.

Esempio semplice: somma.

```csharp id="0bc1jz"
int[] numeri = [1, 2, 3, 4];

int somma = numeri.Aggregate(
    seed: 0,
    func: (accumulatore, numero) => accumulatore + numero);

Console.WriteLine(somma);
```

Output:

```text id="h369ht"
10
```

---

## Aggregate per costruire testo

```csharp id="huq5q5"
string[] parole = ["Ciao", "mondo", ".NET"];

string frase = parole.Aggregate(
    seed: "",
    func: (testo, parola) =>
        string.IsNullOrEmpty(testo)
            ? parola
            : testo + " " + parola);

Console.WriteLine(frase);
```

Output:

```text id="6sak9x"
Ciao mondo .NET
```

In questo caso, però, per concatenare molto testo è spesso meglio `string.Join` o `StringBuilder`.

```csharp id="xk9y36"
string frase = string.Join(" ", parole);
```

---

## Aggregate per oggetti statistici

```csharp id="xnnt0m"
public record StatisticaOrdini(
    int NumeroOrdini,
    decimal Totale)
{
    public decimal Media => NumeroOrdini == 0
        ? 0m
        : Totale / NumeroOrdini;
}
```

Uso:

```csharp id="cbbmls"
StatisticaOrdini statistica = ordini.Aggregate(
    seed: new StatisticaOrdini(0, 0m),
    func: (acc, ordine) => acc with
    {
        NumeroOrdini = acc.NumeroOrdini + 1,
        Totale = acc.Totale + ordine.Righe.Sum(r => r.Subtotale)
    });
```

---

## Quando usare Aggregate

Usa `Aggregate` quando:

```text id="gzk92o"
devi costruire un valore finale da una sequenza
non esiste già un metodo più semplice
l’accumulo ha una logica specifica
vuoi esprimere una riduzione funzionale
```

Non usarlo quando esiste già un metodo più chiaro.

Sconsigliato:

```csharp id="scaz09"
var totale = ordini.Aggregate(0m, (acc, o) => acc + o.Righe.Sum(r => r.Subtotale));
```

Più chiaro:

```csharp id="hwlig8"
var totale = ordini.Sum(o => o.Righe.Sum(r => r.Subtotale));
```

# Comparer personalizzati

Molti operatori LINQ accettano un comparer.

Esempi:

```text id="whiv5t"

GroupBy
Join
ToDictionary
OrderBy

```

I comparer servono a definire come confrontare valori.

## Confronto case-insensitive

```csharp id="w4xf6j"
List<string> citta = [

    "Torino",
```

Output:

```text id="hdoxyb"
Aosta
Torino
Milano
```

---

## GroupBy con comparer

```csharp id="10ga4y"
var gruppi = citta

{
    Console.WriteLine($"{gruppo.Key}: {gruppo.Count()}");
```

Output:

```text id="9bvbgs"

```

## Comparer personalizzato per oggetti

Supponiamo di voler confrontare clienti per città ignorando maiuscole/minuscole.

```csharp id="eeui2j"
public class ClienteCittaComparer : IEqualityComparer<Cliente>
{
    public bool Equals(Cliente? x, Cliente? y)
    {
        if (x is null || y is null)
        {
            return x is y;
        }

        return string.Equals(
            x.Citta,
            y.Citta,
            StringComparison.OrdinalIgnoreCase);
    }

    public int GetHashCode(Cliente obj)
    {
        return StringComparer.OrdinalIgnoreCase.GetHashCode(obj.Citta);
    }
}
```

Uso:

```csharp id="r1i8zl"
var clientiConCittaDistinta = clienti
    .Distinct(new ClienteCittaComparer())
    .ToList();
```

---

## Meglio DistinctBy quando basta una chiave

Se vuoi distinguere per una proprietà, spesso basta `DistinctBy`.

```csharp id="bbg1mk"
var clientiConCittaDistinta = clienti
    .DistinctBy(c => c.Citta, StringComparer.OrdinalIgnoreCase)
    .ToList();
```

Questo è più semplice di un comparer completo.

# LINQ e performance

LINQ rende il codice più leggibile, ma non elimina il costo delle operazioni.

Ogni query ha un costo.

---

## Where + Select è una pipeline

```csharp id="ydkc6z"
var query = clienti
    .Where(c => c.Attivo)
    .Select(c => c.Nome);
```

Questa pipeline viene eseguita durante l’enumerazione.

Su sequenze in memoria, ogni elemento viene attraversato e trasformato.

---

## Attenzione alle enumerazioni multiple

Esempio:

```csharp id="wnux22"
IEnumerable<Ordine> ordiniCliente = ordini
    .Where(o => o.ClienteId == 1);

int numero = ordiniCliente.Count();
decimal totale = ordiniCliente.Sum(o => o.Righe.Sum(r => r.Subtotale));
```

La sequenza viene enumerata due volte.

Se la sorgente è costosa, meglio materializzare:

```csharp id="kk3kba"
List<Ordine> ordiniCliente = ordini
    .Where(o => o.ClienteId == 1)
    .ToList();

int numero = ordiniCliente.Count;
decimal totale = ordiniCliente.Sum(o => o.Righe.Sum(r => r.Subtotale));
```

## Attenzione alle query annidate

Sospetto:

```csharp id="tsm5q9"
var report = clienti
    .Select(c => new
    {
        c.Nome,
        Totale = ordini
            .Where(o => o.ClienteId == c.Id)
            .Sum(o => o.Righe.Sum(r => r.Subtotale))
    })
    .ToList();
```

Per ogni cliente scorre tutti gli ordini.

Su pochi dati va bene.
Su molti dati può diventare inefficiente.

Meglio pre-aggregare:

```csharp id="6f4yq6"
Dictionary<int, decimal> totaliPerCliente = ordini
    .GroupBy(o => o.ClienteId)
    .ToDictionary(
        g => g.Key,
        g => g.Sum(o => o.Righe.Sum(r => r.Subtotale)));

var report = clienti
    .Select(c => new
    {
        c.Nome,
        Totale = totaliPerCliente.TryGetValue(c.Id, out decimal totale)
            ? totale
            : 0m
    })
    .ToList();
```

## ToList non è gratis

`ToList()` materializza una sequenza in memoria.

È utile quando serve, ma non va messo ovunque.

Sospetto:

```csharp id="2zr9x1"
var risultato = clienti
    .Where(c => c.Attivo)
    .ToList()
    .Select(c => c.Nome)
    .ToList();
```

Meglio:

```csharp id="lj2jtz"
var risultato = clienti
    .Where(c => c.Attivo)
    .Select(c => c.Nome)
    .ToList();
```

## LINQ con EF Core

Con EF Core, la performance dipende anche dalla traduzione SQL.

Esempio buono:

```csharp id="rcc4nq"
var clienti = await dbContext.Clienti
    .Where(c => c.Attivo)
    .Select(c => new ClienteDto(c.Id, c.Nome))
    .ToListAsync();
```

Il filtro e la proiezione possono essere tradotti in SQL.

Esempio rischioso:

```csharp id="mqvfqy"
var clienti = await dbContext.Clienti.ToListAsync();

var attivi = clienti
    .Where(c => c.Attivo)
    .ToList();
```

Qui abbiamo caricato tutti i clienti e filtrato in memoria.

Regola pratica:

```text id="zs1iv8"
Con EF Core:
  filtra e proietta prima di ToListAsync.
```

# Errori comuni con LINQ

## Query troppo dense

Sospetto:

```csharp id="ey2av5"
var risultato = clienti.Where(c => c.Attivo).GroupJoin(ordini, c => c.Id, o => o.ClienteId, (c, os) => new { c, os }).SelectMany(x => x.os.DefaultIfEmpty(), (x, o) => new { x.c.Nome, Totale = o?.Righe.Sum(r => r.Subtotale) ?? 0 }).GroupBy(x => x.Nome).Select(g => new { Nome = g.Key, Totale = g.Sum(x => x.Totale) }).OrderByDescending(x => x.Totale).ToList();
```

Meglio spezzare:

```csharp id="x1eoeb"
var clientiAttivi = clienti
    .Where(c => c.Attivo);

var clientiConOrdini = clientiAttivi.LeftJoin(
    ordini,
    c => c.Id,
    o => o.ClienteId,
    (cliente, ordine) => new
    {
        cliente.Nome,
        Totale = ordine?.Righe.Sum(r => r.Subtotale) ?? 0m
    });

var risultato = clientiConOrdini
    .GroupBy(x => x.Nome)
    .Select(g => new
    {
        Nome = g.Key,
        Totale = g.Sum(x => x.Totale)
    })
    .OrderByDescending(x => x.Totale)
    .ToList();
```

La leggibilità conta.

## Usare LINQ per effetti collaterali

Sconsigliato:

```csharp id="ao5x89"
clienti.Select(c =>
{
    Console.WriteLine(c.Nome);
    return c;
}).ToList();
```

LINQ dovrebbe trasformare dati, non sostituire `foreach` per effetti collaterali.

Meglio:

```csharp id="vgv0zh"
foreach (Cliente cliente in clienti)
{
    Console.WriteLine(cliente.Nome);
}
```

## Non gestire sequenze vuote

```csharp id="o7m901"
decimal media = ordini.Average(o => o.Righe.Sum(r => r.Subtotale));
```

Se `ordini` è vuota, genera eccezione.

Meglio:

```csharp id="rb4noy"
decimal media = ordini.Any()
    ? ordini.Average(o => o.Righe.Sum(r => r.Subtotale))
    : 0m;
```

## Confondere GroupBy con ToDictionary

`GroupBy` produce gruppi.

```csharp id="c4b2yf"
var gruppi = clienti.GroupBy(c => c.Citta);
```

`ToDictionary` produce chiavi uniche.

```csharp id="ll7r7y"
var perId = clienti.ToDictionary(c => c.Id);
```

Se la chiave non è unica, `ToDictionary` genera eccezione.

Sbagliato:

```csharp id="bswcyg"
var perCitta = clienti.ToDictionary(c => c.Citta);
```

Se più clienti hanno la stessa città, fallisce.

Meglio:

```csharp id="dg88vd"
var perCitta = clienti
    .GroupBy(c => c.Citta)
    .ToDictionary(
        g => g.Key,
        g => g.ToList());
```

# Esempio completo: report professionale

Obiettivo:

```text id="y50m0i"
Produrre un report clienti:
- tutti i clienti attivi
- anche se non hanno ordini
- totale ordini
- numero ordini
- città
- ordinamento per città e totale decrescente
```

DTO:

```csharp id="ryrzzy"
public record ClienteReportDto(
    int ClienteId,
    string Nome,
    string Citta,
    int NumeroOrdini,
    decimal TotaleOrdini);
```

Versione efficiente in memoria:

```csharp id="0u2jm0"
Dictionary<int, List<Ordine>> ordiniPerCliente = ordini
    .GroupBy(o => o.ClienteId)
    .ToDictionary(
        g => g.Key,
        g => g.ToList());

List<ClienteReportDto> report = clienti
    .Where(c => c.Attivo)
    .Select(c =>
    {
        ordiniPerCliente.TryGetValue(c.Id, out List<Ordine>? ordiniCliente);

        ordiniCliente ??= [];

        return new ClienteReportDto(
            c.Id,
            c.Nome,
            c.Citta,
            ordiniCliente.Count,
            ordiniCliente.Sum(o => o.Righe.Sum(r => r.Subtotale)));
    })
    .OrderBy(r => r.Citta)
    .ThenByDescending(r => r.TotaleOrdini)
    .ThenBy(r => r.Nome)
    .ToList();
```

Stampa:

```csharp id="4z11ck"
foreach (ClienteReportDto item in report)
{
    Console.WriteLine(
        $"{item.Citta} - {item.Nome}: " +
        $"{item.NumeroOrdini} ordini, totale {item.TotaleOrdini:C}");
}
```

Output concettuale:

```text id="gzi42e"
Aosta - Emilie: 2 ordini, totale 316,30 €
Aosta - Anna: 1 ordini, totale 29,90 €
Milano - Luca: 1 ordini, totale 274,90 €
Torino - Giulia: 0 ordini, totale 0,00 €
```

# Versione con LeftJoin

Se vogliamo una riga per ogni coppia cliente/ordine, compresi i clienti senza ordine:

```csharp id="clbgfl"
var righeClienteOrdine = clienti
    .Where(c => c.Attivo)
    .LeftJoin(
        ordini,
        cliente => cliente.Id,
        ordine => ordine.ClienteId,
        (cliente, ordine) => new
        {
            cliente.Id,
            cliente.Nome,
            cliente.Citta,
            OrdineId = ordine?.Id,
            TotaleOrdine = ordine is null
                ? 0m
                : ordine.Righe.Sum(r => r.Subtotale)
        });
```

Poi possiamo raggruppare:

```csharp id="rcbhwb"
List<ClienteReportDto> report = righeClienteOrdine
    .GroupBy(x => new
    {
        x.Id,
        x.Nome,
        x.Citta
    })
    .Select(g => new ClienteReportDto(
        g.Key.Id,
        g.Key.Nome,
        g.Key.Citta,
        g.Count(x => x.OrdineId is not null),
        g.Sum(x => x.TotaleOrdine)))
    .OrderBy(x => x.Citta)
    .ThenByDescending(x => x.TotaleOrdini)
    .ToList();
```

# Linee guida pratiche

```text id="ac1jj1"
Usa Select
  per proiezioni semplici e complesse.

Usa oggetti anonimi
  per risultati locali e temporanei.

Usa DTO
  per risultati che escono dal metodo, dal layer o dall’app.

Usa SelectMany
  per appiattire collezioni annidate.

Usa GroupBy
  per report e aggregazioni leggibili.

Usa CountBy
  per frequenze semplici per chiave.

Usa AggregateBy
  per accumuli per chiave più efficienti o specifici.

Usa Aggregate
  quando devi ridurre una sequenza a un solo risultato custom.

Usa LeftJoin / RightJoin
  con .NET 10 o successivo quando vuoi join esterni leggibili.

Usa comparer personalizzati
  quando uguaglianza o ordinamento non sono quelli predefiniti.

Materializza con ToList
  solo quando serve davvero.

Con EF Core
  lascia la query in IQueryable finché possibile.
```

# Riepilogo

In questo capitolo abbiamo portato LINQ da uso base a uso professionale.

Abbiamo visto:

```text id="e6tny3"
proiezioni complesse
oggetti anonimi
DTO
flattening con SelectMany
raggruppamenti avanzati
join esterni
LeftJoin e RightJoin
CountBy
AggregateBy
Aggregate
comparer personalizzati
```

Abbiamo chiarito che `CountBy` e `AggregateBy` sono stati introdotti in .NET 9 per aggregare per chiave senza dover creare raggruppamenti intermedi con `GroupBy`. ([Microsoft Learn][2])

Abbiamo anche visto che .NET 10 introduce `LeftJoin` e `RightJoin`, rendendo più leggibili i join esterni rispetto al vecchio pattern basato su `GroupJoin`, `SelectMany` e `DefaultIfEmpty`. ([Microsoft Learn][1])

La regola principale è questa:

```text id="v06nbc"
LINQ professionale non significa scrivere query più corte.
Significa scrivere trasformazioni dati chiare, corrette e proporzionate al volume dei dati.
```

Nel prossimo capitolo possiamo passare alla persistenza vera e propria con database relazionali, ADO.NET, SQLite o direttamente Entity Framework Core, a seconda della scaletta scelta.

# 04.03 — Accesso ai database con ADO.NET

Entity Framework Core è oggi lo strumento principale per lavorare con database relazionali in molte applicazioni .NET moderne. Tuttavia, ADO.NET resta una conoscenza fondamentale.

ADO.NET è lo strato storico e ancora attuale su cui si basano molte tecnologie di accesso ai dati. È più vicino al database rispetto a EF Core: si lavora direttamente con connessioni, comandi SQL, parametri, reader, transazioni, stored procedure e tabelle in memoria.

Il punto centrale del capitolo è questo:

```text
ADO.NET offre controllo esplicito sull’accesso al database.
EF Core offre un modello più alto, orientato agli oggetti.
Conoscere ADO.NET aiuta a capire cosa succede davvero sotto molti strumenti moderni.
```

---

# Perché conoscere ancora ADO.NET

ADO.NET è importante per diversi motivi.

```text
Applicazioni legacy
  Molte applicazioni aziendali usano ancora DataSet, DataTable, TableAdapter,
  stored procedure e accesso dati manuale.

Controllo fine
  Puoi controllare SQL, parametri, transazioni, reader e tempi di apertura connessione.

Stored procedure
  In molti ambienti enterprise la logica dati è ancora concentrata in stored procedure.

Performance
  In scenari specifici, leggere dati con DbDataReader può essere molto efficiente.

Debug
  Capire ADO.NET aiuta a diagnosticare problemi di connessione, timeout,
  parametri, transazioni e mapping dati.

Interop con codice esistente
  Molte librerie interne o vecchi progetti espongono ancora DataTable o DataSet.
```

Non significa che ADO.NET debba sostituire EF Core in ogni progetto nuovo. Significa che uno sviluppatore .NET completo dovrebbe conoscerne almeno i concetti fondamentali.

---

# Il modello generale di ADO.NET

ADO.NET ruota attorno ad alcuni oggetti principali:

```text
DbConnection
  rappresenta una connessione al database.

DbCommand
  rappresenta un comando SQL o una stored procedure.

DbParameter
  rappresenta un parametro del comando.

DbDataReader
  legge righe in avanti, una alla volta.

DataTable
  rappresenta una tabella in memoria.

DataSet
  rappresenta un insieme di tabelle in memoria.

DbTransaction
  rappresenta una transazione.
```

Schema semplificato:

```text
Connection
  ↓
Command
  ↓
Parameters
  ↓
ExecuteReader / ExecuteScalar / ExecuteNonQuery
  ↓
Reader / valore / numero righe
```

# `DbConnection`

`DbConnection` rappresenta una connessione a un database.

Esempio concettuale:

```csharp
DbConnection connection = ...
connection.Open();
```

Per SQL Server useremo normalmente:

```csharp
SqlConnection
```

Per SQLite:

```csharp
SqliteConnection
```

---

## Connessione a SQL Server

Con il provider moderno `Microsoft.Data.SqlClient`:

```csharp
using Microsoft.Data.SqlClient;

string connectionString =
    "Server=.;Database=Gestionale;Trusted_Connection=True;TrustServerCertificate=True;";

using SqlConnection connection = new SqlConnection(connectionString);

connection.Open();

Console.WriteLine("Connessione aperta.");
```

La connection string indica:

```text
Server
  server SQL Server.

Database
  database da usare.

Trusted_Connection=True
  autenticazione Windows.

TrustServerCertificate=True
  utile in ambienti di sviluppo se il certificato non è configurato correttamente.
```

In produzione la configurazione TLS/certificati va gestita con attenzione.

---

## Connessione SQL Server con utente e password

```csharp
string connectionString =
    "Server=sqlserver01;Database=Gestionale;User Id=utente;Password=password;";
```

Attenzione: non mettere mai password direttamente nel codice.

Meglio leggere la connection string da:

```text
appsettings.json

secret store
```

Esempio da configurazione:

```csharp
string? connectionString =
    configuration.GetConnectionString("DefaultConnection");
```

```csharp
using Microsoft.Data.Sqlite;
```

SQLite usa un file come database.


```csharp
string dbPath = Path.Combine(AppContext.BaseDirectory, "app.db");
```

# `DbCommand`

`DbCommand` rappresenta un comando da eseguire sul database.

Può essere:

```text
SELECT
INSERT
UPDATE
DELETE
stored procedure
comando DDL come CREATE TABLE
```

Esempio SQL Server:

```csharp
using Microsoft.Data.SqlClient;

using SqlConnection connection = new SqlConnection(connectionString);

connection.Open();

using SqlCommand command = connection.CreateCommand();

command.CommandText = """
SELECT CLIE_CODI, CLIE_NOME
FROM CLIENTI_GEN
WHERE CLIE_CODI = @Id
""";

command.Parameters.AddWithValue("@Id", 123);

using SqlDataReader reader = command.ExecuteReader();

while (reader.Read())
{
    int id = reader.GetInt32(0);
    string nome = reader.GetString(1);

    Console.WriteLine($"{id} - {nome}");
}
```

---

# Tipi principali di esecuzione

Un comando ADO.NET può essere eseguito in modi diversi.

```text
ExecuteReader
  quando la query restituisce righe.

ExecuteScalar
  quando serve un solo valore.

ExecuteNonQuery
  per INSERT, UPDATE, DELETE o comandi senza result set.
```

---

## `ExecuteReader`

Usato per leggere righe.

```csharp
using SqlCommand command = connection.CreateCommand();

command.CommandText = """
SELECT Id, Nome
FROM Clienti
WHERE Attivo = 1
""";

using SqlDataReader reader = command.ExecuteReader();

while (reader.Read())
{
    int id = reader.GetInt32(0);
    string nome = reader.GetString(1);

    Console.WriteLine($"{id}: {nome}");
}
```

---

## `ExecuteScalar`

Usato per ottenere un solo valore.

Esempio: conteggio clienti.

```csharp
using SqlCommand command = connection.CreateCommand();

command.CommandText = """
SELECT COUNT(*)
FROM Clienti
WHERE Attivo = 1
""";

int count = Convert.ToInt32(command.ExecuteScalar());

Console.WriteLine($"Clienti attivi: {count}");
```

Esempio: massimo ID.

```csharp
command.CommandText = "SELECT MAX(Id) FROM Clienti";

object? result = command.ExecuteScalar();

int? maxId = result == DBNull.Value || result is null
    ? null
    : Convert.ToInt32(result);
```

---

## `ExecuteNonQuery`

Usato per comandi che non restituiscono righe.

```csharp
using SqlCommand command = connection.CreateCommand();

command.CommandText = """
UPDATE Clienti
SET Attivo = 0
WHERE Id = @Id
""";

command.Parameters.AddWithValue("@Id", 10);

int righeModificate = command.ExecuteNonQuery();

Console.WriteLine($"Righe modificate: {righeModificate}");
```

# `DbDataReader`

`DbDataReader` legge dati in modo veloce, in avanti, una riga alla volta.

È adatto quando:

```text
devi leggere molte righe
non devi modificare dati in memoria
non ti serve una DataTable
vuoi evitare overhead
vuoi elaborare progressivamente
```

Esempio:

```csharp
using SqlDataReader reader = command.ExecuteReader();

while (reader.Read())
{
    int id = reader.GetInt32(reader.GetOrdinal("Id"));
    string nome = reader.GetString(reader.GetOrdinal("Nome"));

    Console.WriteLine($"{id}: {nome}");
}
```

---

## Accesso per indice

```csharp
int id = reader.GetInt32(0);
string nome = reader.GetString(1);
```

È veloce, ma fragile se cambia l’ordine delle colonne.

---

## Accesso per nome colonna

```csharp
int idIndex = reader.GetOrdinal("Id");
int nomeIndex = reader.GetOrdinal("Nome");

int id = reader.GetInt32(idIndex);
string nome = reader.GetString(nomeIndex);
```

È più robusto.

---

## Gestire `NULL`

Nel database un campo può essere `NULL`.

Prima di leggere con `GetString`, `GetInt32`, ecc., bisogna controllare.

```csharp
int emailIndex = reader.GetOrdinal("Email");

string? email = reader.IsDBNull(emailIndex)
    ? null
    : reader.GetString(emailIndex);
```

Errore comune:

```csharp
string email = reader.GetString(emailIndex);
```

Se il valore è `NULL`, genera eccezione.

---

# Parametri SQL

I parametri SQL sono fondamentali.

Servono a:

```text
evitare SQL injection
gestire correttamente tipi e conversioni
separare codice SQL dai valori
migliorare robustezza
```

Sbagliato:

```csharp
string sql = $"SELECT * FROM Clienti WHERE Nome = '{nome}'";
```

Corretto:

```csharp
using SqlCommand command = connection.CreateCommand();

command.CommandText = """
SELECT Id, Nome
FROM Clienti
WHERE Nome = @Nome
""";

command.Parameters.AddWithValue("@Nome", nome);
```

---

# SQL injection

La SQL injection è una vulnerabilità in cui input dell’utente viene concatenato dentro SQL, modificando il significato della query.

Esempio pericoloso:

```csharp
string nome = "' OR 1=1 --";

string sql = $"""
SELECT Id, Nome
FROM Clienti
WHERE Nome = '{nome}'
""";
```

Il risultato diventa concettualmente:

```sql
SELECT Id, Nome
FROM Clienti
WHERE Nome = '' OR 1=1 --'
```

Questa query può restituire tutti i clienti.

---

## Protezione con parametri

```csharp
command.CommandText = """
SELECT Id, Nome
FROM Clienti
WHERE Nome = @Nome
""";

command.Parameters.AddWithValue("@Nome", nome);
```

Ora il valore viene trattato come valore, non come codice SQL.

Regola fondamentale:

```text
Non concatenare mai input utente dentro una query SQL.
Usa sempre parametri.
```

---

# `AddWithValue`: comodo ma da usare con attenzione

`AddWithValue` è comodo:

```csharp
command.Parameters.AddWithValue("@Nome", nome);
```

Ma in SQL Server può creare problemi di tipo o lunghezza, perché il provider deduce il tipo dal valore.

Per codice più controllato:

```csharp
SqlParameter parameter = new SqlParameter("@Nome", SqlDbType.NVarChar, 100)
{
    Value = nome
};

command.Parameters.Add(parameter);
```

Per importi:

```csharp
SqlParameter parameter = new SqlParameter("@Importo", SqlDbType.Decimal)
{
    Precision = 18,
    Scale = 2,
    Value = importo
};

command.Parameters.Add(parameter);
```

Regola pratica:

```text
AddWithValue
  accettabile per esempi e codice semplice.

Parametri tipizzati
  preferibili in codice enterprise, stored procedure e query sensibili alle performance.
```

---

# Inserimento dati con parametri

Esempio SQL Server:

```csharp
using SqlCommand command = connection.CreateCommand();

command.CommandText = """
INSERT INTO Clienti (Nome, Email, Attivo)
VALUES (@Nome, @Email, @Attivo)
""";

command.Parameters.Add(new SqlParameter("@Nome", SqlDbType.NVarChar, 100)
{
    Value = nome
});

command.Parameters.Add(new SqlParameter("@Email", SqlDbType.NVarChar, 200)
{
    Value = email is null ? DBNull.Value : email
});

command.Parameters.Add(new SqlParameter("@Attivo", SqlDbType.Bit)
{
    Value = true
});

int righeInserite = command.ExecuteNonQuery();
```

Nota importante:

```text
null in C#
  assenza di riferimento.

DBNull.Value
  valore NULL per il database.
```

Per passare `NULL` al database bisogna usare `DBNull.Value`.

# `DataTable`

`DataTable` rappresenta una tabella in memoria.

È uno strumento classico di ADO.NET, molto usato in applicazioni legacy, WinForms, WebForms, reportistica e codice enterprise.

Una `DataTable` contiene:

```text
DataColumn
  definizione delle colonne.

DataRow
  righe di dati.

Rows
  collezione di righe.

Columns
  collezione di colonne.
```

Esempio:

```csharp
DataTable table = new DataTable("Clienti");

table.Columns.Add("Id", typeof(int));
table.Columns.Add("Nome", typeof(string));

DataRow row = table.NewRow();

row["Id"] = 1;
row["Nome"] = "Emilie";

table.Rows.Add(row);
```

Lettura:

```csharp
foreach (DataRow riga in table.Rows)
{
    int id = (int)riga["Id"];
    string nome = (string)riga["Nome"];

    Console.WriteLine($"{id}: {nome}");
}
```

---

# `DataColumn`

`DataColumn` rappresenta una colonna di una `DataTable`.

```csharp
DataColumn idColumn = new DataColumn("Id", typeof(int));
DataColumn nomeColumn = new DataColumn("Nome", typeof(string));

table.Columns.Add(idColumn);
table.Columns.Add(nomeColumn);
```

Possiamo impostare proprietà:

```csharp
DataColumn idColumn = new DataColumn("Id", typeof(int))
{
    AllowDBNull = false,
    Unique = true
};

table.Columns.Add(idColumn);
```

---

# `DataRow`

`DataRow` rappresenta una riga di una `DataTable`.

```csharp
DataRow row = table.NewRow();

row["Id"] = 1;
row["Nome"] = "Emilie";

table.Rows.Add(row);
```

Gestione `DBNull`:

```csharp
object emailValue = row["Email"];

string? email = emailValue == DBNull.Value
    ? null
    : (string)emailValue;
```

---

# Caricare una `DataTable` da database

Con `SqlDataAdapter`:

```csharp
using SqlConnection connection = new SqlConnection(connectionString);

using SqlCommand command = connection.CreateCommand();

command.CommandText = """
SELECT Id, Nome, Email
FROM Clienti
WHERE Attivo = @Attivo
""";

command.Parameters.Add(new SqlParameter("@Attivo", SqlDbType.Bit)
{
    Value = true
});

using SqlDataAdapter adapter = new SqlDataAdapter(command);

DataTable table = new DataTable();

adapter.Fill(table);
```

Lettura:

```csharp
foreach (DataRow row in table.Rows)
{
    int id = (int)row["Id"];
    string nome = (string)row["Nome"];

    string? email = row["Email"] == DBNull.Value
        ? null
        : (string)row["Email"];

    Console.WriteLine($"{id}: {nome} - {email}");
}
```

# `DataSet`

`DataSet` rappresenta un insieme di `DataTable` in memoria.

È come un piccolo database disconnesso.

Può contenere:

```text

vincoli
dati disconnessi dalla connessione
```

Esempio:

```csharp
DataSet dataSet = new DataSet("Gestionale");

DataTable clientiTable = new DataTable("Clienti");
clientiTable.Columns.Add("Id", typeof(int));
clientiTable.Columns.Add("Nome", typeof(string));

ordiniTable.Columns.Add("Id", typeof(int));
ordiniTable.Columns.Add("ClienteId", typeof(int));
ordiniTable.Columns.Add("Totale", typeof(decimal));

dataSet.Tables.Add(clientiTable);
dataSet.Tables.Add(ordiniTable);
```

# Transazioni

Una transazione permette di eseguire più operazioni come un’unica unità.

O tutto va a buon fine, oppure tutto viene annullato.

Esempio:

```text
Inserisco documento
Inserisco righe documento
Aggiorno saldo cliente

Se fallisce una parte, annullo tutto.
```

---

## Transazione SQL Server

```csharp
using SqlConnection connection = new SqlConnection(connectionString);

connection.Open();

using SqlTransaction transaction = connection.BeginTransaction();

try
{
    using SqlCommand command = connection.CreateCommand();

    command.Transaction = transaction;

    command.CommandText = """
    INSERT INTO Documenti (ClienteId, DataDocumento)
    VALUES (@ClienteId, @DataDocumento)
    """;

    command.Parameters.Add(new SqlParameter("@ClienteId", SqlDbType.Int)
    {
        Value = clienteId
    });

    command.Parameters.Add(new SqlParameter("@DataDocumento", SqlDbType.DateTime)
    {
        Value = DateTime.Now
    });

    command.ExecuteNonQuery();

    command.Parameters.Clear();

    command.CommandText = """
    UPDATE Clienti
    SET UltimaOperazione = @DataDocumento
    WHERE Id = @ClienteId
    """;

    command.Parameters.Add(new SqlParameter("@DataDocumento", SqlDbType.DateTime)
    {
        Value = DateTime.Now
    });

    command.Parameters.Add(new SqlParameter("@ClienteId", SqlDbType.Int)
    {
        Value = clienteId
    });

    command.ExecuteNonQuery();

    transaction.Commit();
}
catch
{
    transaction.Rollback();
    throw;
}
```

Regola fondamentale:

```text
Tutti i comandi che fanno parte della transazione devono usare:
- la stessa connessione
- la stessa transaction
```

# Stored procedure

Una stored procedure è codice SQL salvato nel database.

In SQL Server:

```sql
CREATE PROCEDURE dbo.Clienti_SelectById
    @Id INT
AS
BEGIN
    SELECT Id, Nome, Email
    FROM Clienti
    WHERE Id = @Id;
END
```

Chiamata da ADO.NET:

```csharp
using SqlCommand command = connection.CreateCommand();

command.CommandType = CommandType.StoredProcedure;
command.CommandText = "dbo.Clienti_SelectById";

command.Parameters.Add(new SqlParameter("@Id", SqlDbType.Int)
{
    Value = id
});

using SqlDataReader reader = command.ExecuteReader();

if (reader.Read())
{
    string nome = reader.GetString(reader.GetOrdinal("Nome"));
    Console.WriteLine(nome);
}
```

---

## Parametri output

Stored procedure:

```sql
CREATE PROCEDURE dbo.Clienti_CountByCitta
    @Citta NVARCHAR(100),
    @Totale INT OUTPUT
AS
BEGIN
    SELECT @Totale = COUNT(*)
    FROM Clienti
    WHERE Citta = @Citta;
END
```

ADO.NET:

```csharp
using SqlCommand command = connection.CreateCommand();

command.CommandType = CommandType.StoredProcedure;
command.CommandText = "dbo.Clienti_CountByCitta";

command.Parameters.Add(new SqlParameter("@Citta", SqlDbType.NVarChar, 100)
{
    Value = "Aosta"
});

SqlParameter totaleParameter = new SqlParameter("@Totale", SqlDbType.Int)
{
    Direction = ParameterDirection.Output
};

command.Parameters.Add(totaleParameter);

command.ExecuteNonQuery();

int totale = (int)totaleParameter.Value;

Console.WriteLine(totale);
```

---

# Stored procedure con più result set

Una stored procedure può restituire più result set.

Esempio:

```sql
CREATE PROCEDURE dbo.Report_ClientiOrdini
AS
BEGIN
    SELECT Id, Nome
    FROM Clienti;

    SELECT Id, ClienteId, Totale
    FROM Ordini;
END
```

ADO.NET:

```csharp
using SqlDataReader reader = command.ExecuteReader();

while (reader.Read())
{
    Console.WriteLine($"Cliente: {reader["Nome"]}");
}

if (reader.NextResult())
{
    while (reader.Read())
    {
        Console.WriteLine($"Ordine: {reader["Id"]}");
    }
}
```

`NextResult()` passa al result set successivo.

# Timeout

In ADO.NET è importante controllare i timeout.

```csharp
command.CommandTimeout = 120;
```

Il timeout è espresso in secondi.

È utile per:

```text
query lunghe
stored procedure complesse
report
import/export
operazioni batch
```

Attenzione: aumentare il timeout non risolve una query inefficiente. Può essere necessario, ma bisogna anche analizzare indici, piano di esecuzione e volume dati.

# ADO.NET async

Molti metodi hanno versioni asincrone.

```csharp
await connection.OpenAsync();

using SqlDataReader reader = await command.ExecuteReaderAsync();

while (await reader.ReadAsync())
{
    int id = reader.GetInt32(0);
    string nome = reader.GetString(1);
}
```

Esempio `ExecuteNonQueryAsync`:

```csharp
int righe = await command.ExecuteNonQueryAsync();
```

Esempio `ExecuteScalarAsync`:

```csharp
object? result = await command.ExecuteScalarAsync();
```

Regola pratica:

```text
In applicazioni web e servizi, preferisci async per operazioni I/O.
In tool console semplici, il sincrono può essere sufficiente.
```

# Esempio completo SQL Server: repository ADO.NET

Modello:

```csharp
public record ClienteDto(
    int Id,
    string Nome,
    string? Email,
    bool Attivo);
```

Repository:

```csharp
using Microsoft.Data.SqlClient;
using System.Data;

public class ClienteAdoRepository
{
    private readonly string connectionString;

    public ClienteAdoRepository(string connectionString)
    {
        ArgumentException.ThrowIfNullOrWhiteSpace(connectionString);

        this.connectionString = connectionString;
    }

    public async Task<ClienteDto?> TrovaPerIdAsync(int id)
    {
        await using SqlConnection connection = new SqlConnection(connectionString);

        await connection.OpenAsync();

        await using SqlCommand command = connection.CreateCommand();

        command.CommandText = """
        SELECT Id, Nome, Email, Attivo
        FROM Clienti
        WHERE Id = @Id
        """;

        command.Parameters.Add(new SqlParameter("@Id", SqlDbType.Int)
        {
            Value = id
        });

        await using SqlDataReader reader = await command.ExecuteReaderAsync();

        if (!await reader.ReadAsync())
        {
            return null;
        }

        int idIndex = reader.GetOrdinal("Id");
        int nomeIndex = reader.GetOrdinal("Nome");
        int emailIndex = reader.GetOrdinal("Email");
        int attivoIndex = reader.GetOrdinal("Attivo");

        return new ClienteDto(
            reader.GetInt32(idIndex),
            reader.GetString(nomeIndex),
            reader.IsDBNull(emailIndex) ? null : reader.GetString(emailIndex),
            reader.GetBoolean(attivoIndex));
    }

    public async Task<int> InserisciAsync(string nome, string? email)
    {
        await using SqlConnection connection = new SqlConnection(connectionString);

        await connection.OpenAsync();

        await using SqlCommand command = connection.CreateCommand();

        command.CommandText = """
        INSERT INTO Clienti (Nome, Email, Attivo)
        OUTPUT INSERTED.Id
        VALUES (@Nome, @Email, @Attivo)
        """;

        command.Parameters.Add(new SqlParameter("@Nome", SqlDbType.NVarChar, 100)
        {
            Value = nome
        });

        command.Parameters.Add(new SqlParameter("@Email", SqlDbType.NVarChar, 200)
        {
            Value = email is null ? DBNull.Value : email
        });

        command.Parameters.Add(new SqlParameter("@Attivo", SqlDbType.Bit)
        {
            Value = true
        });

        object? result = await command.ExecuteScalarAsync();

        return Convert.ToInt32(result);
    }

    public async Task DisattivaAsync(int id)
    {
        await using SqlConnection connection = new SqlConnection(connectionString);

        await connection.OpenAsync();

        await using SqlCommand command = connection.CreateCommand();

        command.CommandText = """
        UPDATE Clienti
        SET Attivo = 0
        WHERE Id = @Id
        """;

        command.Parameters.Add(new SqlParameter("@Id", SqlDbType.Int)
        {
            Value = id
        });

        await command.ExecuteNonQueryAsync();
    }
}
```

# Esempio completo SQLite

Creazione tabella:

```csharp
using Microsoft.Data.Sqlite;

string dbPath = Path.Combine(AppContext.BaseDirectory, "clienti.db");
string connectionString = $"Data Source={dbPath}";

await using SqliteConnection connection = new SqliteConnection(connectionString);

await connection.OpenAsync();

await using SqliteCommand createCommand = connection.CreateCommand();

createCommand.CommandText = """
CREATE TABLE IF NOT EXISTS Clienti
(
    Id INTEGER PRIMARY KEY AUTOINCREMENT,
    Nome TEXT NOT NULL,
    Email TEXT NULL,
    Attivo INTEGER NOT NULL
);
""";

await createCommand.ExecuteNonQueryAsync();
```

Inserimento:

```csharp
await using SqliteCommand insertCommand = connection.CreateCommand();

insertCommand.CommandText = """
INSERT INTO Clienti (Nome, Email, Attivo)
VALUES ($nome, $email, $attivo);

SELECT last_insert_rowid();
""";

insertCommand.Parameters.AddWithValue("$nome", "Emilie");
insertCommand.Parameters.AddWithValue(
    "$email",
    "emilie@example.com");
insertCommand.Parameters.AddWithValue("$attivo", 1);

long id = (long)(await insertCommand.ExecuteScalarAsync() ?? 0L);
```

Lettura:

```csharp
await using SqliteCommand selectCommand = connection.CreateCommand();

selectCommand.CommandText = """
SELECT Id, Nome, Email, Attivo
FROM Clienti
WHERE Id = $id
""";

selectCommand.Parameters.AddWithValue("$id", id);

await using SqliteDataReader reader = await selectCommand.ExecuteReaderAsync();

if (await reader.ReadAsync())
{
    long clienteId = reader.GetInt64(reader.GetOrdinal("Id"));
    string nome = reader.GetString(reader.GetOrdinal("Nome"));

    int emailIndex = reader.GetOrdinal("Email");

    string? email = reader.IsDBNull(emailIndex)
        ? null
        : reader.GetString(emailIndex);

    bool attivo = reader.GetInt64(reader.GetOrdinal("Attivo")) == 1;

    Console.WriteLine($"{clienteId}: {nome} - {email} - {attivo}");
}
```

Nota: SQLite è più flessibile sui tipi rispetto a SQL Server. Spesso i booleani vengono rappresentati come `0`/`1`.

# Quando preferire ADO.NET a EF Core

ADO.NET può essere preferibile quando:

```text
Devi chiamare molte stored procedure esistenti
  soprattutto in sistemi legacy.

Vuoi controllo completo sul SQL
  senza mapping ORM.

Hai query molto specifiche
  ottimizzate a mano.

Devi leggere grandi volumi in streaming
  con DbDataReader.

Devi integrare codice legacy
  basato su DataTable, DataSet o TableAdapter.

Vuoi evitare tracking e overhead ORM
  in scenari molto mirati.

Devi fare operazioni particolari
  non espressive comodamente in EF Core.
```

---

# Quando preferire EF Core

EF Core è spesso preferibile quando:

```text
Hai un modello dati applicativo ricco
  con entità e relazioni.

Vuoi lavorare con LINQ
  invece che scrivere SQL manuale ovunque.

Vuoi migrations
  per evolvere lo schema.

Vuoi change tracking
  per aggiornare entità.

Vuoi navigazioni tra entità
  e mapping più strutturato.

Vuoi produttività
  nei CRUD e nelle API moderne.
```

---

# ADO.NET, Dapper ed EF Core

È utile collocare gli strumenti:

```text
ADO.NET
  massimo controllo, più codice manuale.

Dapper
  micro-ORM leggero sopra ADO.NET, ottimo per SQL esplicito e mapping rapido.

EF Core
  ORM completo, LINQ, tracking, migrations, mapping avanzato.
```

Dapper non sostituisce ADO.NET concettualmente: lo semplifica.

EF Core non elimina il bisogno di conoscere SQL: lo astrae, ma le query restano database.

# Errori comuni

## Concatenare SQL con input utente

Sbagliato:

```csharp
command.CommandText =
    "SELECT * FROM Clienti WHERE Nome = '" + nome + "'";
```

Corretto:

```csharp
command.CommandText =
    "SELECT * FROM Clienti WHERE Nome = @Nome";

command.Parameters.Add(new SqlParameter("@Nome", SqlDbType.NVarChar, 100)
{
    Value = nome
});
```

---

## Non chiudere connessioni, command e reader

Sbagliato:

```csharp
SqlConnection connection = new SqlConnection(connectionString);
connection.Open();
```

Meglio:

```csharp
using SqlConnection connection = new SqlConnection(connectionString);
connection.Open();
```

Oppure async:

```csharp
await using SqlConnection connection = new SqlConnection(connectionString);
await connection.OpenAsync();
```

---

## Dimenticare `DBNull.Value`

Sbagliato:

```csharp
command.Parameters.AddWithValue("@Email", null);
```

Meglio:

```csharp
command.Parameters.AddWithValue(
    "@Email",
    email is null ? DBNull.Value : email);
```

---

## Leggere NULL senza controllare

Sbagliato:

```csharp
string email = reader.GetString(emailIndex);
```

Corretto:

```csharp
string? email = reader.IsDBNull(emailIndex)
    ? null
    : reader.GetString(emailIndex);
```

---

## Usare `SELECT *`

Sconsigliato:

```sql
SELECT *
FROM Clienti
```

Meglio:

```sql
SELECT Id, Nome, Email, Attivo
FROM Clienti
```

Vantaggi:

```text
meno dati trasferiti
mapping più chiaro
meno fragilità
query più leggibile
```

---

## Non impostare timeout per operazioni lunghe

Sospetto:

```csharp
command.ExecuteReader();
```

Per report o stored procedure pesanti:

```csharp
command.CommandTimeout = 120;
```

---

## Aprire una connessione troppo presto

Sconsigliato:

```csharp
connection.Open();

// molte operazioni in memoria

command.ExecuteReader();
```

Meglio aprire la connessione il più tardi possibile e chiuderla il prima possibile.

ADO.NET usa connection pooling, quindi aprire e chiudere connessioni correttamente è normale.

---

## Usare DataTable per tutto

`DataTable` è comoda in legacy e reportistica, ma nei nuovi progetti spesso DTO e reader espliciti sono più chiari.

Sospetto:

```text
Uso DataTable anche quando mi servono solo 3 colonne fortemente tipizzate.
```

Meglio:

```csharp
public record ClienteDto(int Id, string Nome, string? Email);
```

# Linee guida pratiche

```text
Usa sempre parametri SQL
  per evitare SQL injection e problemi di conversione.

Usa using / await using
  per connessioni, command, reader e transazioni.

Apri la connessione tardi
  e chiudila presto.

Usa DbDataReader
  per letture efficienti e progressive.

Usa DataTable / DataSet
  quando devi integrare codice legacy, binding classico o reportistica.

Usa DTO
  per codice moderno più tipizzato.

Gestisci DBNull.Value
  quando lavori con valori NULL del database.

Usa transazioni
  quando più operazioni devono riuscire o fallire insieme.

Usa stored procedure
  quando il sistema le richiede o quando centralizzare SQL nel database è una scelta architetturale.

Valuta EF Core
  per CRUD, LINQ, mapping e produttività.

Valuta ADO.NET
  per controllo fine, legacy, stored procedure e performance mirata.
```

# Riepilogo

In questo capitolo abbiamo introdotto ADO.NET, lo strato fondamentale di accesso ai database in .NET.

Abbiamo visto:

```text
DbConnection
  connessione al database.

DbCommand
  comando SQL o stored procedure.

DbDataReader
  lettura efficiente riga per riga.

DataTable
  tabella in memoria.

DataRow
  riga di una DataTable.

DataColumn
  colonna di una DataTable.

DataSet
  insieme di tabelle in memoria.

DataView
  vista filtrata e ordinata su DataTable.

TableAdapter
  componente legacy generato da Visual Studio per DataSet tipizzati.
```

Abbiamo studiato:

```text
parametri SQL
SQL injection
transazioni
stored procedure
connessioni SQL Server
connessioni SQLite
lettura sincrona e asincrona
gestione dei NULL
```

La regola principale è questa:

```text
ADO.NET richiede più codice manuale,
ma offre controllo diretto su SQL, connessioni, transazioni e lettura dati.
```

EF Core sarà più produttivo in molti scenari moderni, ma ADO.NET resta essenziale per capire il funzionamento dell’accesso dati, mantenere applicazioni legacy e gestire casi in cui serve controllo preciso.

# 04.05 — Modellazione con EF Core

Nel capitolo precedente abbiamo introdotto Entity Framework Core: `DbContext`, `DbSet`, provider, query base, insert, update, delete, migrations e change tracking.

Ora affrontiamo un tema ancora più importante: **modellare bene**.

Con EF Core non basta creare classi e sperare che il database venga fuori correttamente. Una buona modellazione deve tenere insieme:

```text id="431177"
modello C#
schema relazionale
regole di dominio
vincoli del database
performance delle query
evoluzione futura
```

Il punto centrale del capitolo è questo:

```text id="ee1e5b"
EF Core funziona per convenzione, ma un progetto reale richiede configurazione esplicita nei punti importanti.
```

EF Core può mappare oggetti .NET su database relazionali e supporta relazioni, chiavi, indici, change tracking, concurrency token, owned types e query filters. Le convenzioni sono utili per partire, ma la Fluent API diventa fondamentale quando il modello cresce. ([Microsoft Learn][1])

---

# Data annotations

Le **data annotations** sono attributi applicati direttamente alle classi e alle proprietà.

Esempio:

```csharp id="9ecb7b"
using System.ComponentModel.DataAnnotations;
using System.ComponentModel.DataAnnotations.Schema;

public class Cliente
{
    public int Id { get; set; }

    [Required]
    [MaxLength(100)]
    public string Nome { get; set; } = "";

    [MaxLength(200)]
    public string? Email { get; set; }

    public bool Attivo { get; set; } = true;
}
```

Le annotation più comuni sono:

```text id="f3b71c"
[Key]
[Required]
[MaxLength]
[StringLength]
[Column]
[Table]
[ForeignKey]
[Timestamp]
[ConcurrencyCheck]
```

---

## Vantaggi delle data annotations

```text id="87d39a"
sono semplici
sono visibili vicino alla proprietà
sono facili da capire
funzionano bene per regole basilari
```

Esempio:

```csharp id="1f305e"
[MaxLength(50)]
public string Codice { get; set; } = "";
```

## Limiti delle data annotations

Le data annotations diventano insufficienti quando il mapping è complesso.

Esempi:

```text id="b9bd65"
relazioni avanzate
indici composti
owned types
shadow properties
global query filters
configurazioni condizionali
mapping più pulito separato dal dominio
```

In questi casi si usa la Fluent API.

# Fluent API

La **Fluent API** configura il modello nel metodo `OnModelCreating`.

```csharp id="d47d41"
public class AppDbContext : DbContext
{
    public DbSet<Cliente> Clienti => Set<Cliente>();
    public DbSet<Ordine> Ordini => Set<Ordine>();
    public DbSet<RigaOrdine> RigheOrdine => Set<RigaOrdine>();
    public DbSet<Prodotto> Prodotti => Set<Prodotto>();
    public DbSet<Categoria> Categorie => Set<Categoria>();

    public AppDbContext(DbContextOptions<AppDbContext> options)
        : base(options)
    {
    }

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        modelBuilder.Entity<Cliente>(entity =>
        {
            entity.ToTable("Clienti");

            entity.HasKey(c => c.Id);

            entity.Property(c => c.Nome)
                .IsRequired()
                .HasMaxLength(100);

            entity.Property(c => c.Email)
                .HasMaxLength(200);

            entity.HasIndex(c => c.Email);
        });
    }
}
```

---

## Perché usare Fluent API

La Fluent API è spesso preferibile in applicazioni strutturate perché:

```text id="675e74"
mantiene le entità più pulite
centralizza il mapping
permette configurazioni più complete
evita di mescolare dominio e persistenza
funziona meglio con configurazioni complesse
```

Regola pratica:

```text id="bc298c"
Data annotations
  buone per esempi e regole semplici.

Fluent API
  migliore per progetti reali, mapping complesso e configurazione esplicita.
```

# Configurazioni separate

Quando il modello cresce, `OnModelCreating` può diventare enorme.

Meglio usare classi di configurazione separate.

```csharp id="802508"
using Microsoft.EntityFrameworkCore;
using Microsoft.EntityFrameworkCore.Metadata.Builders;

public class ClienteConfiguration : IEntityTypeConfiguration<Cliente>
{
    public void Configure(EntityTypeBuilder<Cliente> entity)
    {
        entity.ToTable("Clienti");

        entity.HasKey(c => c.Id);

        entity.Property(c => c.Nome)
            .IsRequired()
            .HasMaxLength(100);

        entity.Property(c => c.Email)
            .HasMaxLength(200);

        entity.HasIndex(c => c.Email);
    }
}
```

Nel `DbContext`:

```csharp id="43e70a"
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.ApplyConfiguration(new ClienteConfiguration());
}
```

Oppure automaticamente:

```csharp id="b748db"
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.ApplyConfigurationsFromAssembly(
        typeof(AppDbContext).Assembly);
}
```

Questa forma è molto pulita per progetti grandi.

# Chiavi primarie

La chiave primaria identifica univocamente una riga.

Per convenzione, EF Core riconosce proprietà chiamate:

```text id="2384fb"
Id
NomeEntitaId
```

Esempio:

```csharp id="5b9e1d"
public class Cliente
{
    public int Id { get; set; }
}
```

Configurazione esplicita:

```csharp id="1cf85e"
modelBuilder.Entity<Cliente>()
    .HasKey(c => c.Id);
```

---

## Chiave primaria composta

A volte una chiave è composta da più colonne.

Esempio: tabella ponte manuale tra prodotto e categoria.

```csharp id="f897d7"
public class ProdottoCategoria
{
    public int ProdottoId { get; set; }
    public int CategoriaId { get; set; }

    public Prodotto Prodotto { get; set; } = null!;
    public Categoria Categoria { get; set; } = null!;
}
```

Configurazione:

```csharp id="c14d88"
modelBuilder.Entity<ProdottoCategoria>()
    .HasKey(pc => new
    {
        pc.ProdottoId,
        pc.CategoriaId
    });
```

# Chiavi esterne

Una chiave esterna rappresenta una relazione verso un’altra tabella.

Esempio:

```csharp id="e26b97"
public class Ordine
{
    public int Id { get; set; }
    public int ClienteId { get; set; }

    public Cliente Cliente { get; set; } = null!;
}
```

`ClienteId` è la foreign key.
`Cliente` è la navigation property.

EF Core può creare foreign key anche come shadow property, cioè proprietà presente nel modello EF ma non nella classe .NET. Le shadow properties sono mantenute nel Change Tracker e sono utili quando si vuole evitare di esporre dettagli relazionali nel modello di dominio. ([Microsoft Learn][2])

# Relazione uno-a-molti

Una relazione uno-a-molti è la più comune.

```text id="1135a7"
Un Cliente
  ha molti Ordini.

Un Ordine
  appartiene a un Cliente.
```

Classi:

```csharp id="4fa3c9"
public class Cliente
{
    public int Id { get; set; }
    public string Nome { get; set; } = "";

    public List<Ordine> Ordini { get; set; } = new();
}
```

```csharp id="f5ea14"
public class Ordine
{
    public int Id { get; set; }
    public int ClienteId { get; set; }

    public Cliente Cliente { get; set; } = null!;
}
```

Fluent API:

```csharp id="4b720a"
modelBuilder.Entity<Ordine>(entity =>
{
    entity.HasOne(o => o.Cliente)
        .WithMany(c => c.Ordini)
        .HasForeignKey(o => o.ClienteId)
        .OnDelete(DeleteBehavior.Restrict);
});
```

Significato:

```text id="975275"
Ordine ha un Cliente.
Cliente ha molti Ordini.
La foreign key è Ordine.ClienteId.
La cancellazione del cliente è ristretta se esistono ordini.
```

---

# Relazione uno-a-uno

Una relazione uno-a-uno collega un’entità a una sola altra entità.

Esempio:

```text id="288e8b"
Cliente
  ha una SchedaCliente.

SchedaCliente
  appartiene a un Cliente.
```

Classi:

```csharp id="d0af20"
public class Cliente
{
    public int Id { get; set; }
    public string Nome { get; set; } = "";

    public SchedaCliente? Scheda { get; set; }
}
```

```csharp id="d23ba9"
public class SchedaCliente
{
    public int Id { get; set; }
    public int ClienteId { get; set; }

    public string? Note { get; set; }

    public Cliente Cliente { get; set; } = null!;
}
```

Fluent API:

```csharp id="6e6e8b"
modelBuilder.Entity<Cliente>()
    .HasOne(c => c.Scheda)
    .WithOne(s => s.Cliente)
    .HasForeignKey<SchedaCliente>(s => s.ClienteId);
```

In una relazione uno-a-uno bisogna essere chiari su quale tabella contiene la foreign key.

# Relazione molti-a-molti

Una relazione molti-a-molti indica che molti elementi di un lato sono collegati a molti elementi dell’altro.

Esempio:

```text id="4f27d7"
Un Prodotto
  può appartenere a molte Categorie.

Una Categoria
  può contenere molti Prodotti.
```

EF Core supporta relazioni many-to-many tramite skip navigations, dove una tabella di join può essere creata senza dichiarare esplicitamente una classe ponte, oppure configurata in modo esplicito quando servono colonne aggiuntive. ([Microsoft Learn][3])

Classi:

```csharp id="7c4d07"
public class Prodotto
{
    public int Id { get; set; }
    public string Descrizione { get; set; } = "";

    public List<Categoria> Categorie { get; set; } = new();
}
```

```csharp id="5bc45d"
public class Categoria
{
    public int Id { get; set; }
    public string Nome { get; set; } = "";

    public List<Prodotto> Prodotti { get; set; } = new();
}
```

Fluent API:

```csharp id="ce6d45"
modelBuilder.Entity<Prodotto>()
    .HasMany(p => p.Categorie)
    .WithMany(c => c.Prodotti)
    .UsingEntity("ProdottiCategorie");
```

---

## Molti-a-molti con entità ponte

Se la relazione contiene dati aggiuntivi, serve una classe esplicita.

Esempio:

```text id="72a222"
ProdottoCategoria
  ProdottoId
  CategoriaId
  DataAssegnazione
```

Classe:

```csharp id="8d48c1"
public class ProdottoCategoria
{
    public int ProdottoId { get; set; }
    public int CategoriaId { get; set; }
    public DateTimeOffset DataAssegnazione { get; set; }

    public Prodotto Prodotto { get; set; } = null!;
    public Categoria Categoria { get; set; } = null!;
}
```

Configurazione:

```csharp id="c0e50b"
modelBuilder.Entity<ProdottoCategoria>(entity =>
{
    entity.ToTable("ProdottiCategorie");

    entity.HasKey(pc => new
    {
        pc.ProdottoId,
        pc.CategoriaId
    });

    entity.HasOne(pc => pc.Prodotto)
        .WithMany()
        .HasForeignKey(pc => pc.ProdottoId);

    entity.HasOne(pc => pc.Categoria)
        .WithMany()
        .HasForeignKey(pc => pc.CategoriaId);

    entity.Property(pc => pc.DataAssegnazione)
        .IsRequired();
});
```

Regola pratica:

```text id="e50e1c"
Many-to-many semplice
  usa skip navigations.

Many-to-many con dati sulla relazione
  usa entità ponte esplicita.
```

# Indici

Gli indici migliorano le performance di ricerca, ordinamento e join.

Esempio:

```csharp id="dd5605"
modelBuilder.Entity<Cliente>()
    .HasIndex(c => c.Email);
```

Indice univoco:

```csharp id="d9824b"
modelBuilder.Entity<Prodotto>()
    .HasIndex(p => p.Codice)
    .IsUnique();
```

Indice composto:

```csharp id="a998d8"
modelBuilder.Entity<Cliente>()
    .HasIndex(c => new
    {
        c.Citta,
        c.Nome
    });
```

---

## Quando creare indici

Valuta un indice su colonne usate spesso in:

```text id="bdba82"
WHERE
JOIN
ORDER BY
ricerche per codice
vincoli di unicità
filtri globali
```

Esempio:

```csharp id="736b92"
entity.HasIndex(c => c.Attivo);
entity.HasIndex(c => c.Email).IsUnique();
```

Attenzione: gli indici migliorano le letture, ma hanno un costo su insert, update e spazio disco.

# Owned types

Gli **owned types** rappresentano tipi posseduti da un’entità.

Sono utili per modellare value object.

Esempio:

```csharp id="8284db"
public class Indirizzo
{
    public string Via { get; set; } = "";
    public string Citta { get; set; } = "";
    public string Cap { get; set; } = "";
    public string Nazione { get; set; } = "";
}
```

Usato dentro `Ordine`:

```csharp id="148abf"
public class Ordine
{
    public int Id { get; set; }

    public Indirizzo IndirizzoSpedizione { get; set; } = new();
}
```

Configurazione:

```csharp id="dff5a3"
modelBuilder.Entity<Ordine>(entity =>
{
    entity.OwnsOne(o => o.IndirizzoSpedizione, indirizzo =>
    {
        indirizzo.Property(i => i.Via)
            .HasColumnName("SpedizioneVia")
            .HasMaxLength(200);

        indirizzo.Property(i => i.Citta)
            .HasColumnName("SpedizioneCitta")
            .HasMaxLength(100);

        indirizzo.Property(i => i.Cap)
            .HasColumnName("SpedizioneCap")
            .HasMaxLength(10);

        indirizzo.Property(i => i.Nazione)
            .HasColumnName("SpedizioneNazione")
            .HasMaxLength(100);
    });
});
```

Il valore può essere salvato nella stessa tabella dell’ordine, con colonne dedicate.

# Value objects

Un **value object** rappresenta un concetto identificato dai suoi valori, non da un ID.

Esempi:

```text id="57ab5f"
Indirizzo
Money
EmailAddress
Periodo
Coordinate
```

Un value object dovrebbe essere:

```text id="191830"
immutabile o quasi immutabile
senza identità propria
validato internamente
con uguaglianza per valore
```

Esempio più orientato al dominio:

```csharp id="4bc106"
public class EmailAddress
{
    public string Valore { get; private set; } = "";

    private EmailAddress()
    {
        // Per EF Core
    }

    public EmailAddress(string valore)
    {
        ArgumentException.ThrowIfNullOrWhiteSpace(valore);

        if (!valore.Contains('@'))
        {
            throw new ArgumentException("Email non valida.", nameof(valore));
        }

        Valore = valore;
    }

    public override string ToString()
    {
        return Valore;
    }
}
```

Per tipi così semplici, si può usare una conversione:

```csharp id="d5f4ac"
modelBuilder.Entity<Cliente>()
    .Property<EmailAddress>("Email")
    .HasConversion(
        email => email.Valore,
        valore => new EmailAddress(valore));
```

In pratica, per iniziare è spesso più semplice usare owned types per value object composti, e value converters per piccoli wrapper di singolo valore.

# Shadow properties

Una **shadow property** è una proprietà presente nel modello EF Core ma non nella classe C#.

Esempio:

```csharp id="547abc"
modelBuilder.Entity<Cliente>()
    .Property<DateTimeOffset>("CreatedAt");
```

La classe `Cliente` non contiene `CreatedAt`, ma EF Core la mappa comunque.

Impostare valore:

```csharp id="e3d864"
dbContext.Entry(cliente)
    .Property("CreatedAt")
    .CurrentValue = DateTimeOffset.UtcNow;
```

Shadow properties sono utili per dati tecnici che non vogliamo esporre nel modello applicativo, per esempio colonne di audit o foreign key nascoste. EF Core mantiene valore e stato di queste proprietà nel Change Tracker. ([Microsoft Learn][2])

---

## Audit con shadow properties

Configurazione:

```csharp id="af3ca7"
modelBuilder.Entity<Cliente>(entity =>
{
    entity.Property<DateTimeOffset>("CreatedAt");
    entity.Property<DateTimeOffset?>("UpdatedAt");
});
```

In `SaveChanges` o `SaveChangesAsync` possiamo valorizzarle.

```csharp id="ce1db3"
public override Task<int> SaveChangesAsync(
    CancellationToken cancellationToken = default)
{
    DateTimeOffset now = DateTimeOffset.UtcNow;

    foreach (var entry in ChangeTracker.Entries())
    {
        if (entry.State == EntityState.Added)
        {
            entry.Property("CreatedAt").CurrentValue = now;
        }

        if (entry.State == EntityState.Modified)
        {
            entry.Property("UpdatedAt").CurrentValue = now;
        }
    }

    return base.SaveChangesAsync(cancellationToken);
}
```

# Concurrency token

Un **concurrency token** serve a gestire conflitti quando due utenti o processi modificano lo stesso dato.

EF Core implementa la concorrenza ottimistica configurando una proprietà come concurrency token: quando l’entità viene aggiornata o cancellata, EF confronta nel database il valore originale del token letto con quello corrente; se non coincide, l’operazione fallisce con un conflitto. ([Microsoft Learn][4])

---

## RowVersion su SQL Server

In SQL Server si usa spesso una colonna `rowversion`.

Classe:

```csharp id="e77d99"
public class Cliente
{
    public int Id { get; set; }
    public string Nome { get; set; } = "";

    public byte[] RowVersion { get; set; } = [];
}
```

Configurazione:

```csharp id="508d73"
modelBuilder.Entity<Cliente>()
    .Property(c => c.RowVersion)
    .IsRowVersion();
```

Quando due utenti modificano lo stesso cliente, il secondo salvataggio può generare:

```csharp id="2c0dc3"
DbUpdateConcurrencyException
```

---

## Gestire un conflitto

```csharp id="51fb89"
try
{
    await dbContext.SaveChangesAsync();
}
catch (DbUpdateConcurrencyException ex)
{
    // Qui possiamo:
    // - informare l'utente
    // - ricaricare i dati dal database
    // - confrontare valori originali e correnti
    // - decidere una strategia di merge

    throw;
}
```

Regola pratica:

```text id="733cae"
Concurrency token
  non impedisce che due utenti modifichino lo stesso dato.
  Permette di accorgersi del conflitto al salvataggio.
```

# Soft delete

Il **soft delete** consiste nel non cancellare fisicamente una riga, ma marcarla come eliminata.

Esempio:

```csharp id="be690a"
public class Cliente
{
    public int Id { get; set; }
    public string Nome { get; set; } = "";
    public bool IsDeleted { get; set; }
}
```

Invece di:

```csharp id="404a27"
dbContext.Clienti.Remove(cliente);
```

facciamo:

```csharp id="f84416"
cliente.IsDeleted = true;

await dbContext.SaveChangesAsync();
```

---

## Intercettare delete e trasformarla in soft delete

Esempio semplificato nel `DbContext`:

```csharp id="1db373"
public override Task<int> SaveChangesAsync(
    CancellationToken cancellationToken = default)
{
    foreach (var entry in ChangeTracker.Entries<Cliente>())
    {
        if (entry.State == EntityState.Deleted)
        {
            entry.State = EntityState.Modified;
            entry.Entity.IsDeleted = true;
        }
    }

    return base.SaveChangesAsync(cancellationToken);
}
```

Ora se qualcuno chiama:

```csharp id="d672a2"
dbContext.Clienti.Remove(cliente);
```

l’entità non viene cancellata fisicamente: viene aggiornata con `IsDeleted = true`.

# Global query filters

I **global query filters** applicano automaticamente una condizione a tutte le query su una certa entità.

Sono molto utili per:

```text id="e350ec"
soft delete
multi-tenancy
filtri per azienda/sede
dati attivi/non cancellati
```

EF Core permette di associare un filtro a un tipo di entità; il filtro viene applicato ogni volta che quel tipo viene interrogato, come se EF aggiungesse automaticamente un `Where` alla query. ([Microsoft Learn][5])

Configurazione soft delete:

```csharp id="29d7f5"
modelBuilder.Entity<Cliente>()
    .HasQueryFilter(c => !c.IsDeleted);
```

Da questo momento:

```csharp id="0be520"
List<Cliente> clienti = await dbContext.Clienti.ToListAsync();
```

restituisce solo clienti non cancellati.

---

## Ignorare un query filter

A volte vogliamo vedere anche i record cancellati.

```csharp id="3493e5"
List<Cliente> tutti = await dbContext.Clienti
    .IgnoreQueryFilters()
    .ToListAsync();
```

Attenzione: `IgnoreQueryFilters()` disabilita i filtri globali per quella query.

---

## Filtri multipli e EF Core 10

Prima di EF Core 10, più condizioni sullo stesso tipo venivano di solito combinate in un singolo `HasQueryFilter`, per esempio soft delete più tenant:

```csharp id="8d6c8c"
modelBuilder.Entity<Cliente>()
    .HasQueryFilter(c => !c.IsDeleted && c.TenantId == tenantId);
```

EF Core 10 introduce i **named query filters**, che permettono di definire più filtri nominati sullo stesso tipo e disabilitarli selettivamente. Questa novità risolve un limite storico dei global query filters quando si vogliono combinare scenari come soft delete e multi-tenancy. ([Microsoft Learn][6])

Esempio concettuale EF Core 10:

```csharp id="5751d4"
modelBuilder.Entity<Cliente>()
    .HasQueryFilter("SoftDeleteFilter", c => !c.IsDeleted)
    .HasQueryFilter("TenantFilter", c => c.TenantId == tenantId);
```

Disabilitazione selettiva:

```csharp id="3da225"
var clienti = await dbContext.Clienti
    .IgnoreQueryFilters(["SoftDeleteFilter"])
    .ToListAsync();
```

# Esempio completo di configurazione

Ecco una configurazione più realistica.

```csharp id="78caf1"
using Microsoft.EntityFrameworkCore;
using Microsoft.EntityFrameworkCore.Metadata.Builders;

public class ClienteConfiguration : IEntityTypeConfiguration<Cliente>
{
    public void Configure(EntityTypeBuilder<Cliente> entity)
    {
        entity.ToTable("Clienti");

        entity.HasKey(c => c.Id);

        entity.Property(c => c.Nome)
            .IsRequired()
            .HasMaxLength(100);

        entity.Property(c => c.Email)
            .HasMaxLength(200);

        entity.HasIndex(c => c.Email)
            .IsUnique()
            .HasFilter("[Email] IS NOT NULL");

        entity.Property(c => c.RowVersion)
            .IsRowVersion();

        entity.HasQueryFilter(c => !c.IsDeleted);

        entity.HasMany(c => c.Ordini)
            .WithOne(o => o.Cliente)
            .HasForeignKey(o => o.ClienteId)
            .OnDelete(DeleteBehavior.Restrict);
    }
}
```

Questa configurazione fa molte cose:

```text id="e0b2c0"
definisce tabella
definisce chiave primaria
configura lunghezze
crea indice univoco su Email
configura concurrency token
applica soft delete globale
configura relazione uno-a-molti
impedisce cancellazione a cascata automatica del cliente con ordini
```

# Soft delete con interfaccia

Invece di ripetere `IsDeleted` manualmente, possiamo introdurre un’interfaccia.

```csharp id="b92634"
public interface ISoftDelete
{
    bool IsDeleted { get; set; }
    DateTimeOffset? DeletedAt { get; set; }
}
```

Entità:

```csharp id="c5cafb"
public class Cliente : ISoftDelete
{
    public int Id { get; set; }
    public string Nome { get; set; } = "";

    public bool IsDeleted { get; set; }
    public DateTimeOffset? DeletedAt { get; set; }

    public List<Ordine> Ordini { get; set; } = new();
}
```

Override:

```csharp id="2b2ce1"
public override Task<int> SaveChangesAsync(
    CancellationToken cancellationToken = default)
{
    DateTimeOffset now = DateTimeOffset.UtcNow;

    foreach (var entry in ChangeTracker.Entries<ISoftDelete>())
    {
        if (entry.State == EntityState.Deleted)
        {
            entry.State = EntityState.Modified;
            entry.Entity.IsDeleted = true;
            entry.Entity.DeletedAt = now;
        }
    }

    return base.SaveChangesAsync(cancellationToken);
}
```

# Errori comuni

## Affidarsi solo alle convenzioni

Le convenzioni sono comode, ma non sempre sufficienti.

Sospetto:

```csharp id="d3815e"
public decimal Prezzo { get; set; }
```

Meglio specificare precisione:

```csharp id="f83bf9"
entity.Property(p => p.Prezzo)
    .HasPrecision(18, 2);
```

Questo è fondamentale per importi monetari.

---

## Dimenticare gli indici

Se cerchi spesso per email:

```csharp id="8f021b"
await dbContext.Clienti
    .FirstOrDefaultAsync(c => c.Email == email);
```

valuta:

```csharp id="e76b42"
entity.HasIndex(c => c.Email);
```

---

## Cancellazione a cascata non voluta

Una relazione può generare cancellazioni a cascata.

Meglio essere espliciti:

```csharp id="54456b"
.OnDelete(DeleteBehavior.Restrict);
```

oppure scegliere consapevolmente `Cascade`, `SetNull`, `NoAction`.

---

## Esporre foreign key oppure nasconderle senza criterio

Modello pragmatico:

```csharp id="5310da"
public int ClienteId { get; set; }
public Cliente Cliente { get; set; } = null!;
```

Modello più domain-oriented:

```csharp id="2b56b4"
public Cliente Cliente { get; private set; } = null!;
```

con foreign key shadow.

Entrambi sono validi. La scelta dipende dall’architettura.

---

## Usare owned types per entità con identità propria

`Indirizzo` può essere owned type se vive e muore con l’ordine.

Ma se l’indirizzo ha vita autonoma, storico, audit, condivisione tra clienti, allora forse è un’entità.

Regola pratica:

```text id="aa1f1f"
Owned type
  non ha identità propria.

Entity
  ha identità propria e ciclo di vita autonomo.
```

---

## Soft delete senza filtro globale

Se usi `IsDeleted` ma dimentichi il filtro:

```csharp id="2e5363"
await dbContext.Clienti.ToListAsync();
```

restituirà anche record cancellati logicamente.

Meglio:

```csharp id="1f94cd"
entity.HasQueryFilter(c => !c.IsDeleted);
```

---

## Global query filters dimenticati nelle query amministrative

Con il filtro attivo, potresti non vedere dati cancellati o di altri tenant.

Per funzioni amministrative:

```csharp id="0cd552"
dbContext.Clienti
    .IgnoreQueryFilters()
```

ma va usato con molta attenzione.

---

## Non gestire la concorrenza

Senza concurrency token, l’ultimo salvataggio può sovrascrivere modifiche fatte da un altro utente.

Con `RowVersion`, EF Core può accorgersi del conflitto.

---

# Linee guida pratiche

```text id="dd63af"
Usa Fluent API
  per mapping importante e relazioni complesse.

Usa configurazioni separate
  per mantenere pulito il DbContext.

Configura precisione dei decimal
  soprattutto per importi monetari.

Configura indici
  sulle colonne usate per ricerca, join e vincoli univoci.

Rendi esplicite le relazioni
  con HasOne, WithMany, HasForeignKey e OnDelete.

Usa owned types
  per value object senza identità propria.

Usa shadow properties
  per dati tecnici non esposti nel dominio.

Usa concurrency token
  quando più utenti possono modificare gli stessi dati.

Usa soft delete
  quando devi conservare storico o evitare cancellazioni fisiche.

Usa global query filters
  per soft delete e multi-tenancy.

Non ignorare il database
  vincoli, indici e relazioni sono parte del modello.
```

# 04.04 — Entity Framework Core: introduzione

Nel capitolo precedente abbiamo visto ADO.NET: connessioni, comandi, reader, parametri, transazioni, `DataTable`, stored procedure e accesso diretto al database.

Ora saliamo di livello.

**Entity Framework Core**, spesso abbreviato in **EF Core**, è lo strumento principale per la persistenza moderna in molte applicazioni .NET.

EF Core permette di lavorare con il database usando oggetti C#:

```csharp id="647fpa"
Cliente cliente = new Cliente
{
    Nome = "Emilie",
    Email = "emilie@example.com"
};

dbContext.Clienti.Add(cliente);

await dbContext.SaveChangesAsync();
```

Invece di scrivere manualmente:

```sql id="lrzp2p"
INSERT INTO Clienti (Nome, Email)
VALUES (@Nome, @Email)
```

EF Core genera ed esegue il SQL necessario.

Microsoft descrive EF Core come un mapper object-relational moderno, open source e multipiattaforma, che consente agli sviluppatori .NET di lavorare con database usando oggetti .NET ed eliminare gran parte del codice di accesso dati scritto manualmente. ([Microsoft Learn][1])

Il punto centrale del capitolo è questo:

```text id="vg4k82"
EF Core non elimina il database.
Lo mappa su oggetti .NET, permettendo di interrogare e modificare dati
con LINQ, entità, DbContext e change tracking.
```

---

# Che cos’è un ORM

ORM significa:

```text id="qfblwx"
Object-Relational Mapper
```

Un ORM mette in relazione due mondi diversi:

```text id="zyyuqi"
mondo object-oriented
  classi, oggetti, proprietà, relazioni, metodi

mondo relazionale
  tabelle, colonne, righe, chiavi primarie, chiavi esterne
```

Esempio nel database:

```text id="fwrsfh"
Tabella Clienti
  Id
  Nome
  Email
  Attivo
```

Esempio in C#:

```csharp id="wzmxmy"
public class Cliente
{
    public int Id { get; set; }
    public string Nome { get; set; } = "";
    public string? Email { get; set; }
    public bool Attivo { get; set; }
}
```

EF Core collega questi due mondi.

---

## Cosa fa un ORM

Un ORM come EF Core può occuparsi di:

```text id="db1vfw"
mappare classi su tabelle
mappare proprietà su colonne
generare query SQL da LINQ
materializzare righe database in oggetti
tracciare modifiche agli oggetti
generare INSERT, UPDATE e DELETE
gestire relazioni tra entità
creare ed evolvere lo schema con migrations
```

Esempio LINQ:

```csharp id="x71fto"
List<Cliente> clientiAttivi = await dbContext.Clienti
    .Where(c => c.Attivo)
    .OrderBy(c => c.Nome)
    .ToListAsync();
```

EF Core può tradurre questa query in SQL simile a:

```sql id="88qctx"
SELECT Id, Nome, Email, Attivo
FROM Clienti
WHERE Attivo = 1
ORDER BY Nome
```

---

## ORM non significa ignorare SQL

EF Core aumenta la produttività, ma non elimina la necessità di capire il database.

Uno sviluppatore deve comunque conoscere:

```text id="jjieze"
chiavi primarie
chiavi esterne
indici
transazioni
vincoli
tipi dati
performance query
N+1 query
migrazioni
concorrenza
```

Regola pratica:

```text id="o35bju"
EF Core ti permette di scrivere meno codice SQL manuale,
ma non ti autorizza a ignorare come funziona il database.
```

# Entity

Una **entity** è una classe C# che EF Core mappa su una tabella o vista del database.

Esempio:

```csharp id="k9i69z"
public class Cliente
{
    public int Id { get; set; }
    public string Nome { get; set; } = "";
    public string? Email { get; set; }
    public bool Attivo { get; set; } = true;
}
```

Questa classe può essere mappata su una tabella `Clienti`.

Ogni oggetto `Cliente` rappresenta una riga.

```text id="8h2i6s"
Classe Cliente
  tabella Clienti

Proprietà Id
  colonna Id

Proprietà Nome
  colonna Nome

Oggetto Cliente
  riga della tabella
```

---

## Entità semplice

```csharp id="kwffqa"
public class Prodotto
{
    public int Id { get; set; }
    public string Codice { get; set; } = "";
    public string Descrizione { get; set; } = "";
    public decimal Prezzo { get; set; }
}
```

---

## Entità con relazione

```csharp id="da7gkg"
public class Ordine
{
    public int Id { get; set; }
    public int ClienteId { get; set; }
    public DateOnly Data { get; set; }

    public Cliente Cliente { get; set; } = null!;
    public List<RigaOrdine> Righe { get; set; } = new();
}
```

```csharp id="30zj38"
public class RigaOrdine
{
    public int Id { get; set; }
    public int OrdineId { get; set; }
    public string Descrizione { get; set; } = "";
    public int Quantita { get; set; }
    public decimal PrezzoUnitario { get; set; }

    public Ordine Ordine { get; set; } = null!;
}
```

Qui:

```text id="a7xe3m"
Ordine.ClienteId
  chiave esterna verso Cliente.

Ordine.Cliente
  navigation property verso il cliente.

Ordine.Righe
  collection navigation verso le righe.
```

# `DbContext`

`DbContext` è il cuore di EF Core.

Rappresenta una sessione di lavoro con il database.

Un `DbContext` si occupa di:

```text id="hv0kkj"
connessione
modello dati
query
change tracking
salvataggio modifiche
transazioni implicite
mapping tra entità e database
```

Microsoft spiega che `DbContext` rappresenta una sessione con il database e può essere usato per interrogare e salvare istanze delle entità; con Dependency Injection, `AddDbContext` registra normalmente il contesto con lifetime scoped. ([Microsoft Learn][2])

Esempio:

```csharp id="j1hjml"
using Microsoft.EntityFrameworkCore;

public class AppDbContext : DbContext
{
    public DbSet<Cliente> Clienti => Set<Cliente>();
    public DbSet<Ordine> Ordini => Set<Ordine>();
    public DbSet<RigaOrdine> RigheOrdine => Set<RigaOrdine>();

    public AppDbContext(DbContextOptions<AppDbContext> options)
        : base(options)
    {
    }
}
```

---

## DbContext come unità di lavoro

Un `DbContext` dovrebbe vivere per una singola unità di lavoro.

Esempi:

```text id="ytub0d"
una richiesta HTTP
una schermata di modifica
una singola operazione batch
un comando applicativo
```

Non dovrebbe essere tenuto aperto per tutta la vita dell’applicazione.

Regola pratica:

```text id="pggwa8"
DbContext non è un singleton.
In ASP.NET Core si registra normalmente come scoped.
```

---

# `DbSet`

Un `DbSet<TEntity>` rappresenta una collezione di entità di un certo tipo.

```csharp id="0v7gym"
public DbSet<Cliente> Clienti => Set<Cliente>();
```

Con `DbSet` possiamo:

```text id="kpdigv"
interrogare dati
aggiungere entità
rimuovere entità
aggiornare entità tracciate
comporre query LINQ
```

Esempi:

```csharp id="q7y92m"
List<Cliente> clienti = await dbContext.Clienti.ToListAsync();
```

```csharp id="tfylcc"
dbContext.Clienti.Add(new Cliente
{
    Nome = "Emilie",
    Email = "emilie@example.com"
});
```

```csharp id="x6lxwp"
Cliente? cliente = await dbContext.Clienti
    .FirstOrDefaultAsync(c => c.Id == 1);
```

---

## DbSet non è una List

Anche se sembra una collezione, `DbSet` non è una `List<T>`.

```text id="cj1xf0"
List<T>
  dati già in memoria.

DbSet<T>
  punto di partenza per costruire query verso il database.
```

Questa query:

```csharp id="yxu2el"
var query = dbContext.Clienti
    .Where(c => c.Attivo);
```

non ha ancora caricato i clienti in memoria.

Il caricamento avviene con:

```csharp id="e4mejn"
List<Cliente> clienti = await query.ToListAsync();
```

# Provider

EF Core usa provider specifici per ogni database.

Un provider sa tradurre le operazioni EF Core nel linguaggio e nei tipi del database concreto.

Esempi:

```text id="wd073j"

PostgreSQL
MySQL/MariaDB
Oracle

```

Microsoft segnala che i provider EF Core possono provenire da origini diverse e raccomanda di valutarne qualità, licenze, supporto e compatibilità di versione. ([Microsoft Learn][3])

# Configurazione connessione

In un’app .NET moderna, la connection string non dovrebbe stare nel codice.

Sconsigliato:

```csharp id="waf5kh"
options.UseSqlServer("Server=.;Database=Demo;Trusted_Connection=True;");
```

Meglio:

```json id="18l8zi"
{
  "ConnectionStrings": {
    "DefaultConnection": "Server=.;Database=Demo;Trusted_Connection=True;TrustServerCertificate=True;"
  }
}
```

E nel codice:

```csharp id="vtj1r9"
builder.Services.AddDbContext<AppDbContext>(options =>
{
    string? connectionString =
        builder.Configuration.GetConnectionString("DefaultConnection");

    options.UseSqlServer(connectionString);
});
```

## Convenzioni

EF Core riconosce molte cose automaticamente.

Esempio:

```csharp id="iqjs68"
public class Cliente

    public string Nome { get; set; } = "";

```

Per convenzione:

```text id="wdn0sp"
Id
  diventa chiave primaria.

  viene mappata su una tabella.

Nome
  diventa una colonna.
```

## Data annotations

```csharp id="ki8o3h"
using System.ComponentModel.DataAnnotations;
using System.ComponentModel.DataAnnotations.Schema;

public class Cliente
{
    public int Id { get; set; }

    [Required]
    [MaxLength(100)]
    public string Nome { get; set; } = "";

    [MaxLength(200)]

    [Column(TypeName = "decimal(18,2)")]

}
```

## Fluent API

La Fluent API si configura in `OnModelCreating`.

```csharp id="q9gu6j"
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.Entity<Cliente>(entity =>
    {
        entity.ToTable("Clienti");

        entity.HasKey(c => c.Id);

        entity.Property(c => c.Nome)
            .IsRequired()
            .HasMaxLength(100);

        entity.Property(c => c.Email)
            .HasMaxLength(200);
    });
}
```

Regola pratica:

```text id="lpdptx"
Convenzioni
  ottime per iniziare.

Data annotations
  comode per regole semplici vicine al modello.

Fluent API
  preferibile per configurazioni complete, relazioni complesse e mapping più pulito.
```

# Migrations

Le **migrations** servono a far evolvere lo schema del database insieme al modello C#.

Esempio:

```text id="2xjqlx"
Aggiungo proprietà Email a Cliente
  ↓
Creo una migration
  ↓
EF Core genera ALTER TABLE
  ↓
Aggiorno il database
```

Microsoft spiega che, una volta creato il modello, le migrations permettono di creare un database dal modello e di evolvere il database mentre il modello cambia. ([Microsoft Learn][5])

---

## Installare gli strumenti EF Core

In genere serve il tool:

```bash id="cij4pw"
dotnet tool install --global dotnet-ef
```

Oppure aggiornare:

```bash id="68yj5d"
dotnet tool update --global dotnet-ef
```

Nel progetto servono i pacchetti provider e design, per esempio:

```bash id="syd4yr"
dotnet add package Microsoft.EntityFrameworkCore.SqlServer
dotnet add package Microsoft.EntityFrameworkCore.Design
```

Per SQLite:

```bash id="k82ebl"
dotnet add package Microsoft.EntityFrameworkCore.Sqlite
dotnet add package Microsoft.EntityFrameworkCore.Design
```

---

## Creare una migration

```bash id="m6rzps"
dotnet ef migrations add InitialCreate
```

Questo genera file C# nella cartella `Migrations`.

Esempio concettuale:

```csharp id="as7rpo"
migrationBuilder.CreateTable(
    name: "Clienti",
    columns: table => new
    {
        Id = table.Column<int>(nullable: false),
        Nome = table.Column<string>(maxLength: 100, nullable: false),
        Email = table.Column<string>(maxLength: 200, nullable: true)
    },
    constraints: table =>
    {
        table.PrimaryKey("PK_Clienti", x => x.Id);
    });
```

---

## Applicare migration al database

```bash id="e8nmkx"
dotnet ef database update
```

Questo crea o aggiorna il database in base alle migrations.

---

## Migrazioni e provider multipli

Se la stessa applicazione supporta provider diversi, per esempio SQL Server e SQLite, bisogna prestare attenzione: EF Core genera migrations per il provider attivo, e Microsoft indica che in scenari multi-provider è spesso necessario mantenere set di migrations separati per provider. ([Microsoft Learn][6])

Regola pratica:

```text id="p7cmxa"
Un progetto didattico può usare SQLite per semplicità.
Un progetto reale deve scegliere bene provider e strategia migrations.
```

# Creazione database

Ci sono due approcci principali:

```text id="nkg57k"
EnsureCreated
  crea il database direttamente dal modello, senza migrations.

Migrations
  crea ed evolve il database tramite versioni controllate.
```

---

## `EnsureCreated`

```csharp id="pvdfah"
await dbContext.Database.EnsureCreatedAsync();
```

È comodo per:

```text id="z6ngfu"
demo
test semplici
prototipi
database temporanei
```

Ma non è adatto a gestire evoluzioni di schema nel tempo.

---

## `Migrate`

```csharp id="a8a7fi"
await dbContext.Database.MigrateAsync();
```

Applica le migrations pendenti.

È più vicino a uno scenario reale, ma va usato con criterio in produzione.

Regola pratica:

```text id="9r38v6"
EnsureCreated
  demo e prototipi.

Migrations
  sviluppo reale ed evoluzione schema.

Script SQL delle migrations
  spesso preferibili per ambienti di produzione controllati.
```

# Query base

Con EF Core si interrogano i dati usando LINQ.

Esempio:

```csharp id="bzl0gr"
List<Cliente> clienti = await dbContext.Clienti
    .OrderBy(c => c.Nome)
    .ToListAsync();
```

Filtrare:

```csharp id="cdmrm4"
List<Cliente> clientiAttivi = await dbContext.Clienti
    .Where(c => c.Attivo)
    .OrderBy(c => c.Nome)
    .ToListAsync();
```

Cercare per chiave:

```csharp id="h2f8m7"
Cliente? cliente = await dbContext.Clienti
    .FirstOrDefaultAsync(c => c.Id == id);
```

Oppure:

```csharp id="owd75y"
Cliente? cliente = await dbContext.Clienti.FindAsync(id);
```

---

## Proiezione verso DTO

Spesso non conviene caricare tutta l’entità se servono solo poche colonne.

DTO:

```csharp id="tnpmp0"
public record ClienteListaDto(
    int Id,
    string Nome,
    string? Email);
```

Query:

```csharp id="w38zcs"
List<ClienteListaDto> clienti = await dbContext.Clienti
    .Where(c => c.Attivo)
    .OrderBy(c => c.Nome)
    .Select(c => new ClienteListaDto(
        c.Id,
        c.Nome,
        c.Email))
    .ToListAsync();
```

Questa è una pratica importante: filtrare e proiettare prima di `ToListAsync`.

# Update

Per aggiornare un’entità, di solito:

```text id="ca3qu6"
carichi l’entità
modifichi le proprietà
chiami SaveChangesAsync
```

Esempio:

```csharp id="x4327k"
Cliente? cliente = await dbContext.Clienti
    .FirstOrDefaultAsync(c => c.Id == id);

if (cliente is null)
{
    return;
}

cliente.Email = "nuova@example.com";

await dbContext.SaveChangesAsync();
```

EF Core rileva la modifica tramite change tracking e genera l’`UPDATE`.

---

## Update senza caricare tutto

EF Core ha anche API per aggiornamenti massivi, come `ExecuteUpdateAsync`, ma conviene trattarle in un capitolo più avanzato.

Per l’introduzione è meglio partire dal modello classico:

```text id="dtu2u9"
load
modify
save
```

# Delete

Per cancellare:

```csharp id="xdr74t"
Cliente? cliente = await dbContext.Clienti
    .FirstOrDefaultAsync(c => c.Id == id);

if (cliente is null)
{
    return;
}

dbContext.Clienti.Remove(cliente);

await dbContext.SaveChangesAsync();
```

EF Core genera il `DELETE`.

---

## Cancellazione logica

In molte applicazioni aziendali non si cancella davvero la riga.

Si usa una cancellazione logica:

```csharp id="95z63t"
cliente.Attivo = false;

await dbContext.SaveChangesAsync();
```

Oppure:

```csharp id="x2rusr"
cliente.DataCancellazione = DateTimeOffset.UtcNow;
```

Regola pratica:

```text id="k8v2so"
Delete fisico
  rimuove la riga.

Delete logico
  mantiene la riga ma la marca come non attiva/cancellata.
```

# Change tracking

Il **change tracking** è il meccanismo con cui EF Core tiene traccia degli oggetti caricati e delle modifiche fatte.

Microsoft spiega che ogni istanza di `DbContext` tiene traccia delle modifiche fatte alle entità, e che queste modifiche guidano gli aggiornamenti al database quando viene chiamato `SaveChanges`. ([Microsoft Learn][7])

Esempio:

```csharp id="zjtvxl"
Cliente? cliente = await dbContext.Clienti
    .FirstOrDefaultAsync(c => c.Id == 1);

cliente!.Nome = "Nuovo nome";

await dbContext.SaveChangesAsync();
```

EF Core sa che `Nome` è cambiato perché l’entità è tracciata dal `DbContext`.

---

## Stati delle entità

Gli stati principali sono:

```text id="frgzps"
Added
  nuova entità da inserire.

Unchanged
  entità caricata e non modificata.

Modified
  entità modificata.

Deleted
  entità da eliminare.

Detached
  entità non tracciata dal DbContext.
```

Esempio:

```csharp id="l6sf79"
var entry = dbContext.Entry(cliente);

Console.WriteLine(entry.State);
```

---

## AsNoTracking

Se dobbiamo solo leggere dati, senza modificarli, possiamo evitare il tracking.

```csharp id="jemer0"
List<ClienteListaDto> clienti = await dbContext.Clienti
    .AsNoTracking()
    .Where(c => c.Attivo)
    .Select(c => new ClienteListaDto(c.Id, c.Nome, c.Email))
    .ToListAsync();
```

`AsNoTracking` può migliorare performance e ridurre memoria nelle query di sola lettura.

Regola pratica:

```text id="4m19d8"
Query per modificare dati
  tracking utile.

Query di sola lettura
  valuta AsNoTracking.
```

# Esempio completo con SQLite

Costruiamo una console app minima con EF Core e SQLite.

---

## Creazione progetto

## `Cliente.cs`

```csharp id="gwt8y8"
public class Cliente
{
    public int Id { get; set; }
    public string Nome { get; set; } = "";
    public string? Email { get; set; }
    public bool Attivo { get; set; } = true;
}
```

---

## `AppDbContext.cs`

```csharp id="5141bv"
using Microsoft.EntityFrameworkCore;

public class AppDbContext : DbContext
{
    public DbSet<Cliente> Clienti => Set<Cliente>();

    protected override void OnConfiguring(DbContextOptionsBuilder optionsBuilder)
    {
        optionsBuilder.UseSqlite("Data Source=clienti.db");
    }

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        modelBuilder.Entity<Cliente>(entity =>
        {
            entity.ToTable("Clienti");

            entity.HasKey(c => c.Id);

            entity.Property(c => c.Nome)
                .IsRequired()
                .HasMaxLength(100);

            entity.Property(c => c.Email)
                .HasMaxLength(200);
        });
    }
}
```

Per un esempio introduttivo `OnConfiguring` è accettabile. In applicazioni reali, però, è meglio configurare il `DbContext` tramite Dependency Injection e connection string esterna.

---

## `Program.cs`

```csharp id="jdl8tl"
using Microsoft.EntityFrameworkCore;

await using AppDbContext dbContext = new AppDbContext();

await dbContext.Database.EnsureCreatedAsync();

Cliente cliente = new Cliente
{
    Nome = "Emilie",
    Email = "emilie@example.com",
    Attivo = true
};

dbContext.Clienti.Add(cliente);

await dbContext.SaveChangesAsync();

Console.WriteLine($"Cliente inserito con ID {cliente.Id}");

List<Cliente> clientiAttivi = await dbContext.Clienti
    .AsNoTracking()
    .Where(c => c.Attivo)
    .OrderBy(c => c.Nome)
    .ToListAsync();

foreach (Cliente c in clientiAttivi)
{
    Console.WriteLine($"{c.Id}: {c.Nome} - {c.Email}");
}

Cliente? clienteDaModificare = await dbContext.Clienti
    .FirstOrDefaultAsync(c => c.Id == cliente.Id);

if (clienteDaModificare is not null)
{
    clienteDaModificare.Email = "nuova@example.com";
    await dbContext.SaveChangesAsync();
}

Cliente? clienteDaCancellare = await dbContext.Clienti
    .FirstOrDefaultAsync(c => c.Id == cliente.Id);

if (clienteDaCancellare is not null)
{
    dbContext.Clienti.Remove(clienteDaCancellare);
    await dbContext.SaveChangesAsync();
}
```

# Esempio con DI e SQL Server

In un’app ASP.NET Core o Worker Service useremo normalmente DI.

`appsettings.json`:

```json id="xzpoec"
{
  "ConnectionStrings": {
    "DefaultConnection": "Server=.;Database=DemoEfCore;Trusted_Connection=True;TrustServerCertificate=True;"
  }
}
```

`Program.cs`:

```csharp id="uwdfw6"
using Microsoft.EntityFrameworkCore;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddDbContext<AppDbContext>(options =>
{
    options.UseSqlServer(
        builder.Configuration.GetConnectionString("DefaultConnection"));
});

var app = builder.Build();

app.MapGet("/clienti", async (AppDbContext dbContext) =>
{
    List<ClienteListaDto> clienti = await dbContext.Clienti
        .AsNoTracking()
        .Where(c => c.Attivo)
        .OrderBy(c => c.Nome)
        .Select(c => new ClienteListaDto(c.Id, c.Nome, c.Email))
        .ToListAsync();

    return Results.Ok(clienti);
});

app.Run();

public record ClienteListaDto(
    int Id,
    string Nome,
    string? Email);
```

# Errori comuni

## Trattare DbContext come singleton

Sbagliato:

```csharp id="aqskau"
builder.Services.AddSingleton<AppDbContext>();
```

Meglio:

```csharp id="c0bij6"
builder.Services.AddDbContext<AppDbContext>(options =>
{
    options.UseSqlServer(connectionString);
});
```

`AddDbContext` registra normalmente il contesto come scoped, che è il comportamento adatto alla maggior parte delle applicazioni ASP.NET Core. ([Microsoft Learn][2])

---

## Fare `ToListAsync` troppo presto

Sbagliato:

```csharp id="ju6hy7"
List<Cliente> tutti = await dbContext.Clienti.ToListAsync();

List<Cliente> attivi = tutti
    .Where(c => c.Attivo)
    .ToList();
```

Meglio:

```csharp id="jr179j"
List<Cliente> attivi = await dbContext.Clienti
    .Where(c => c.Attivo)
    .ToListAsync();
```

Nel secondo caso il filtro viene eseguito dal database.

---

## Esporre direttamente entità nelle API

Sconsigliato:

```csharp id="thsql0"
return Results.Ok(cliente);
```

se `Cliente` è un’entità di dominio/persistenza ricca.

Meglio:

```csharp id="vrp1nl"
return Results.Ok(new ClienteDto(
    cliente.Id,
    cliente.Nome,
    cliente.Email));
```

I DTO proteggono il contratto esterno.

---

## Dimenticare `SaveChangesAsync`

Questo codice non salva nulla nel database:

```csharp id="r879to"
dbContext.Clienti.Add(cliente);
```

Serve:

```csharp id="xuvn8d"
await dbContext.SaveChangesAsync();
```

---

## Usare `Update` su entità scollegate senza criterio

Sospetto:

```csharp id="bjd8l6"
dbContext.Clienti.Update(cliente);
await dbContext.SaveChangesAsync();
```

`Update` marca spesso l’intera entità come modificata. Può essere rischioso con DTO parziali o dati provenienti dall’esterno.

Per iniziare è più sicuro:

```text id="m24y7p"
caricare l’entità
modificare le proprietà desiderate
salvare
```

---

## Ignorare il SQL generato

EF Core genera SQL. Non bisogna ignorarlo.

In sviluppo è utile attivare logging SQL per capire:

```text id="idg8mh"
query generate
parametri
numero di query
join
filtri
N+1 query
performance
```

---

## Usare migrations senza controllo in produzione

In sviluppo:

```bash id="wgmv03"
dotnet ef database update
```

è comodo.

In produzione spesso è meglio generare script SQL, farli revisionare e applicarli con procedure controllate.

# Quando fare attenzione

EF Core va usato con più cautela quando:

```text id="mxdr9k"

tutta la logica passa da stored procedure complesse
le query sono molto ottimizzate a mano
serve controllo SQL riga per riga
ci sono requisiti estremi di performance
il mapping relazionale è molto distante dal modello a oggetti
```

In questi casi possono essere più adatti:

```text id="l1xr9z"
ADO.NET
Dapper
stored procedure
SQL manuale
approccio ibrido
```

# Linee guida pratiche

```text id="kk8ubw"
Usa DbContext
  come unità di lavoro breve.

Registra DbContext come scoped
  nelle app con Dependency Injection.

Usa DbSet
  come punto di accesso alle entità.

Usa LINQ
  per filtrare, ordinare e proiettare.

Filtra prima di ToListAsync
  per far lavorare il database.

Usa DTO
  per query di lettura e API.

Usa AsNoTracking
  per query di sola lettura.

Chiama SaveChangesAsync
  per rendere persistenti insert, update e delete.

Usa migrations
  per evolvere lo schema nel tempo.

Non ignorare il SQL generato
  soprattutto su query importanti.
```
